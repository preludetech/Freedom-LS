# Spec: Unstarted-course click behaviour ‚Äî dedicated course detail page

## 1. Summary

When a student clicks an **unstarted** course (one they have not registered for), take them to a
dedicated, mobile-responsive **course detail page** that helps them decide to enrol. This replaces the
current desktop **modal** preview entirely ‚Äî the modal approach is removed and will not return.

The existing standalone preview page (`course_preview` view, served at `courses/<slug>/preview/`) is
**promoted and renamed** to become the single canonical detail page for every screen size, then enhanced
to match the design references. No brand-new route is introduced.

Three new **optional** `Course` fields back the new page content: learning outcomes ("what you'll
learn"), difficulty level, and estimated duration. All degrade gracefully ‚Äî when a value is unset, the
corresponding UI element is **omitted entirely** (no `‚Äî`, "N/A", "TBD", or default placeholder).

## 2. Why this matters

- **Conversion + trust.** A dedicated detail page is the de-facto standard across course platforms
  (Udemy, Coursera, edX, Teachable). A two-column layout (main content + sticky enrolment panel) with a
  single clear CTA is the highest-converting pattern (see `research_course_landing_ux.md`,
  `research_enrolment_panel_patterns.md`).
- **Consistency over novelty.** A real page (not a modal) renders identically on mobile and desktop,
  removes the desktop/mobile branch divergence, and lets us reuse the breadcrumbs, course-icon, accent
  colour, and table-of-contents components the rest of the app already uses.
- **Honesty.** The design mocks contain fabricated data (PROBATION / OPEN COURSE tags, a 4.8‚òÖ rating,
  "Save for later" / wishlist, "Certificate on completion", invented stats). FLS has no backing data for
  any of these, so rendering them would be deceptive. The spec renders **only real, derived or
  author-supplied data** and omits everything else.

## 3. Scope

### In scope

- Promote + rename the standalone preview page to a canonical **course detail page**; rebuild its body to
  match the design (hero, main column, sticky sign-up panel).
- Point the not-registered course **cards** (dashboard) and **rows** (all-courses list) at the detail
  page on **all** screen sizes via a plain link.
- Remove the desktop **modal** branch and the now-dead supporting code.
- Add three optional `Course` fields, a migration, admin entries, the Pydantic content schema entries,
  and demo-content values for each.

### Out of scope (unchanged from the idea)

- **Course content preview** ‚Äî opening/viewing course items before enrolling. The detail page shows
  structure only (table of contents), not item content.
- **Certificate on completion** ‚Äî FLS issues no certificate; a certificate claim would be fake data.
  Explicitly dropped. (A `# TODO`/`@claude` comment may mark where it would live if a certificate model
  is added later ‚Äî do not fabricate it now.)
- Ratings / reviews, enrolment counts, "save for later" / wishlist, any payment or pricing concept ‚Äî
  no backing data exists.
- Changes to the enrolment backend. The existing one-click `register_for_course` flow is reused
  unchanged.

## 4. Naming decision (confirmed with user)

Rename `preview` ‚Üí `detail` throughout, since the page is now the canonical detail page, not a preview:

| Thing | Before | After |
|---|---|---|
| View function | `course_preview` | `course_detail` |
| URL path | `courses/<slug>/preview/` | `courses/<slug>/detail/` |
| URL name | `student_interface:course_preview` | `student_interface:course_detail` |
| Page template | `course_preview.html` | `course_detail.html` |
| Start-url helper | `_preview_start_url()` | `_detail_start_url()` |

`course_preview_content.html` (the partial shared today by the modal body and the standalone page) is
**removed**: with the modal gone, its only consumer is the page itself, so its content is folded into the
new `course_detail.html` (and any new detail partials). Update all references ‚Äî links, tests, reverse()
calls ‚Äî to the new names.

## 5. Data model changes (`Course`, app `content_engine`)

All three fields are **optional** and added to `freedom_ls/content_engine/models.py`'s `Course`
(`models.py:149`). Each is omitted from the student-facing page when empty.

### 5.1 Learning outcomes ‚Äî "What you'll learn"

- **Field:** `learning_outcomes` ‚Äî `ArrayField(models.CharField(max_length=255), blank=True,
  default=list)` (Postgres `ArrayField` from `django.contrib.postgres.fields`; the stack is PostgreSQL
  17).
- **Why a list field, not a related model** (confirmed with user): outcomes are authored as a YAML list
  in course frontmatter; a plain ordered list of short strings needs no separate model, second migration,
  or demo-loader change. Order is the list order.
- **Rendering:** "What you'll learn" section renders one tick-marked item per entry. Omit the **whole
  section** when the list is empty.

### 5.2 Difficulty level

- **Field:** `difficulty` ‚Äî `CharField(max_length=20, blank=True, default="", choices=DifficultyLevel.choices)`.
- **Choices** (confirmed with user ‚Äî includes "All levels"): a `models.TextChoices` enum
  `DifficultyLevel` with `BEGINNER = "beginner", "Beginner"`, `INTERMEDIATE = "intermediate",
  "Intermediate"`, `ADVANCED = "advanced", "Advanced"`, `ALL_LEVELS = "all_levels", "All levels"`.
- **Rendering:** hero stat showing `get_difficulty_display()`. Omit when blank. Never default to
  "Beginner" as a placeholder.

### 5.3 Estimated duration

- **Field:** `estimated_duration` ‚Äî `DurationField(null=True, blank=True)`.
- **Why `DurationField`** (user asked us to research and choose something flexible): native Django,
  validates, sorts, represents any granularity, stored as a PostgreSQL `interval`, and Pydantic coerces a
  YAML scalar (`"2:00:00"`, ISO-8601 `"PT2H"`, or integer seconds) straight to `timedelta`. Preferred
  over integer-minutes (locks granularity + needs a unit convention in the name) and free-text (can't
  sort/validate, invites inconsistent formatting ‚Äî the very "sloppy data" the honesty rule warns
  against).
- **Display:** a `display_estimated_duration` helper (method on `Course` or a template filter) formats
  the `timedelta` into a coarse, honest string, e.g. `~2 hours`, `~45 minutes`, `~1 hour 30 min`. Omit
  the stat when the field is null. Keep the formatter small and dependency-free (no new humanize
  package).

### 5.4 Migration, admin, schema, demo content

- **Migration:** one new migration in `freedom_ls/content_engine/migrations/` (next is `0010_*`). Plain
  `AddField` for all three (all nullable/blank with sane defaults ‚Äî no data migration needed). Never edit
  existing migrations.
- **Admin:** add `learning_outcomes`, `difficulty`, `estimated_duration` to the first fieldset of
  `CourseAdmin` (`content_engine/admin.py:186`). `CourseAdmin` uses `SiteAwareModelAdmin` ‚Äî follow that
  pattern.
- **Content schema:** add the three fields to the Pydantic `Course` model in
  `freedom_ls/content_engine/schema.py`. That model is `extra="forbid"`, so fields absent from the schema
  are silently dropped by `content_save`. Use `list[str]` for outcomes, an optional enum/str for
  difficulty, and `timedelta | None` for duration.
- **Demo content:** add real values to the frontmatter of at least one demo course under `demo_content/`
  (e.g. `functionality_demo_standard_markdown/course.md`) so the new sections are visible in QA ‚Äî a
  `learning_outcomes:` list, a `difficulty:` value, and an `estimated_duration:` value. Leave at least
  one demo course **without** the new fields so the graceful-omission path is also demonstrable.

## 6. Page structure

The page extends the standard student page chrome. Because the current `course_preview.html` extends
`_base.html` (not `_base_interface.html`), breadcrumbs must be included explicitly (see 6.1).

### 6.1 Breadcrumbs

- Reuse `panel_framework/partials/breadcrumbs.html`. Pass `breadcrumbs` as a list of `{label, url}` dicts;
  the last crumb has no `url` (renders as the current page).
- First crumb: "‚Üê All courses" linking to the all-courses list
  (`reverse("student_interface:courses")` ‚Äî confirm the exact list URL name during implementation).
  Final crumb: the course title (no url).

### 6.2 Hero

- Background uses the course accent gradient: apply `course-accent-{{ course.accent_slot_key }}` (existing
  system; classes in `tailwind.components.css`, tokens per theme). No new colour data.
- Course icon rendered large and semi-transparent, anchored bottom-right, via the existing `course_icon`
  template tag (`{% course_icon course.icon course.icon_fallback css_class="size-..." %}`). Keep opacity
  low (~0.1‚Äì0.2) so it doesn't compete with the title.
- Title (`course.title`) and brief description (`course.subtitle`, omit if blank).
- Optional category eyebrow: if `course.category` is set, show it as a small tag. Do **not** invent tags
  like "PROBATION" / "OPEN COURSE".
- **Stats strip** ‚Äî render each stat only when genuinely available; omit individually when empty:
  - **Modules / lessons** ‚Äî derived from content structure (always available). Use
    `len(course.viewable_items())` for lesson/assessment count and/or the `CoursePart` count for modules
    (decide exact wording during implementation; show a real count).
  - **Difficulty level** ‚Äî `get_difficulty_display()`, omit when blank (¬ß5.2).
  - **Estimated duration** ‚Äî `display_estimated_duration`, omit when null (¬ß5.3).
- An honest **"Free ¬∑ open enrolment"** label is allowed (no payment exists, so always true). Do **not**
  show "self-paced" in the hero (cohort deadlines mean it isn't universally true).
- Do **not** render a "Certificate on completion" stat (no certificate exists ‚Äî it would be fake data).

### 6.3 Main content area (left / primary column)

- **About this course** ‚Äî `course.rendered_content()` (sanitized markdown HTML) wrapped in
  `<c-markdown-container>`. Omit the section when `content` is empty.
- **What you'll learn** ‚Äî tick-listed `learning_outcomes` (¬ß5.1). Omit the whole section when empty.
- **Course content** ‚Äî reuse `student_interface/partials/course_minimal_toc.html` (the `course-children`
  partialdef). It requires `children` as the rich dicts from `get_course_index()` (title, type, url,
  status, deadlines, nested children) plus `course` and `is_registered`. The view already calls
  `get_course_index()`, so keep that. Do **not** reinvent the ToC, and do **not** add a content
  **preview** (opening items) ‚Äî structure only.

### 6.4 Right-hand sign-up panel

A surface that lets the user enrol. Contents:

- Honest **"Free / open to everyone"** label + short supporting microcopy (e.g. "One click. No credit
  card.").
- A single, high-contrast **primary CTA**. Label is **state-aware** (the panel currently always says
  "Start"; this is the gap to close):
  - Not registered (the primary case for this page): **"Enrol & start"**, links to the
    `register_for_course` URL.
  - Registered, not started: **"Start course"**, links to the first item.
  - Registered, in progress / complete: a sensible label ("Continue" / "Review course") ‚Äî reuse the
    existing `_detail_start_url()` branch targets; wording refinement only.
  - The start-url branching stays in `_detail_start_url()` (renamed from `_preview_start_url()`),
    unchanged in logic.
- **"This course includes"** ‚Äî real, derived data only:
  - lesson/module count (as in the hero stats);
  - "Includes assessments" **only when** the course has `Form` children
    (`any(isinstance(c, Form) for c in course.viewable_items())`).
  - No star rating, no wishlist/"save for later", no fabricated includes. A certificate item may be
    marked with a `# TODO`/`@claude` comment for when a certificate model exists ‚Äî not rendered now.
- **Layout:** sticky on desktop (Tailwind `sticky top-...` within a two-column grid); on mobile it stacks
  into the single column. Keep the CTA reachable without scrolling to the very bottom (panel near the top
  on mobile, immediately after the hero). A mobile sticky bottom CTA bar is **optional/nice-to-have**, not
  required for this spec.

### 6.5 Responsive layout

- Desktop (‚â• lg): two-column grid ‚Äî main content (~2/3) + sticky sign-up panel (~1/3).
- Mobile / narrow: single column ‚Äî hero ‚Üí sign-up panel (or its CTA) ‚Üí main content (outcomes ‚Üí about ‚Üí
  content ToC). No desktop-only modal branch anywhere.
- Implementation should consult the `brand-guidelines`, `frontend-styling`, and `template` skills for
  exact Tailwind/cotton usage; reuse existing components rather than introducing new primitives.

## 7. Wiring the course list cards/rows

- **`course_card_not_registered.html`** and **`course_row_not_registered.html`**: remove the
  `hidden md:block` desktop **modal** branch and the `md:hidden` mobile branch; replace with a **single**
  plain link (all screen sizes) to `student_interface:course_detail` for the course. The card/row shell
  components (`course-card-shell`, `course-row-shell`) are reused.
- These templates currently consume `course.preview_children` and `course.preview_start_url` (set by
  `_annotate_preview_context`). After the modal is gone they need only the course slug for the link.

## 9. Success criteria

1. Clicking an unstarted course from **both** the dashboard (card) and the all-courses list (row), on
   **any** screen size, performs a navigation to `courses/<slug>/detail/` ‚Äî no modal is opened anywhere.
2. The detail page renders: breadcrumbs (first crumb ‚Üí all courses), an accent-coloured hero with large
   semi-transparent course icon, title, subtitle (when set), a stats strip, an "About this course"
   markdown block (when set), a "What you'll learn" list (when outcomes set), the reused course-content
   ToC, and a sticky sign-up panel with a state-aware CTA.
3. **Honesty:** for a course with none of the new fields set, the page shows **no** difficulty,
   duration, or outcomes elements at all ‚Äî no placeholders, dashes, or "N/A". Verified against a demo
   course left without the new fields.
4. The new CTA reads "Enrol & start" for an unregistered user and links to `register_for_course`;
   enrolling still uses the unchanged one-click flow and lands the user in the course.
5. The three new `Course` fields exist with a migration, appear in the admin, are accepted by the
   content schema, and have real demo-content values visible on at least one demo course.
6. The `c-modal` component still works for the educator delete-confirmation and modal-form flows
   (not broken by cleanup).
7. `course_preview_content.html` is gone and no code references the old `preview` view/URL/template
   names; tests are updated to the new `detail` names and pass.
8. `uv run pytest` passes (existing preview/card/row tests updated to the renamed page and new
   not-registered link behaviour; new tests cover the three model fields' presence and graceful
   omission, and the state-aware CTA label).
