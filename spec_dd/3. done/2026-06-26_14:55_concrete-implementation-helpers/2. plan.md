# Plan: Concrete Implementation Helpers

This plan implements `1. spec.md`. Inside the **FLS repo** the work is documentation and
Claude-plugin command authoring plus a small file-deletion cleanup — **no Django application code,
no models, no migrations, no frontend templates** here. The **template repo** (which *does* contain a
Django `config/`) already exists as an empty git repo and is **populated as part of this work**
(Task A).

**Template repo location (load-bearing).** The external template repo lives at
`/home/sheena/workspace/lms/freedom-ls-concrete-template` (remote
`git@github.com:preludetech/freedom-ls-concrete-template.git`, branch `main`, currently empty — no
commits). **All template-repo file changes are made in that directory and committed in that repo —
never in the FLS worktree.** This supersedes the spec's "creating the external repo is a follow-up"
note (§2): the repo now exists, so we build it out here. (Self-correction to the earlier reading of
this plan, which treated the template repo as not-created-here.)

Because there is no FLS application code, this plan adds **no pytest tests** to FLS. The FLS-side test
obligation is success criterion 7: run the existing suite after the Caprover cleanup to confirm
nothing breaks. The template repo is validated separately by booting it (Task G).

## Task E — `update_fls` improvements (spec §5.3; success criterion 6)

Edit `fls-claude-plugin/commands/concrete/update_fls.md`. Keep the existing correct core (advance
spec-by-spec, one commit per spec `Update FLS: <spec-name>`, test gate). Add:

- **Read `upgrade_notes.md`**: for each completed spec, read
  `submodules/Freedom-LS/spec_dd/3. done/<spec-dir>/upgrade_notes.md` and use its frontmatter flags
  to decide which integration steps to run. **Backward compatibility**: if absent, **warn** and fall
  back to the current prose-inference behaviour.
- **Dry-run / preview step**: before any pointer moves, list all pending completed specs and, per
  spec, its `upgrade_notes.md` flags (or "no notes — will infer"), so the operator sees the full plan
  first.
- **Migration safety**: run `uv run python manage.py migrate --check` **before** applying each spec
  and `uv run python manage.py makemigrations --check` **after**, to catch migration-state issues and
  silent conflicts.
- **Template-drift detection**: for specs whose notes flag `requires_template_review`, diff the listed
  changed FLS template paths against the concrete project's overrides and **flag** (do not auto-merge)
  any override whose upstream source changed.
- **Always `uv sync`** after a pointer move (catches new FLS deps). Note that concrete-project CI uses
  `uv sync --locked`.
- **Documented rollback procedure**: how to reset the submodule pointer and discard the in-progress
  integration commit, returning to the last good state, when a spec fails mid-integration.

Pseudocode for the per-spec loop (illustrative, to fold into the command prose):

```
for spec in pending_completed_specs (chronological):
    notes = read frontmatter of submodules/Freedom-LS/spec_dd/3. done/<spec>/upgrade_notes.md
    if notes missing: warn("no upgrade_notes.md — falling back to current prose-inference"); use existing prose-inference path (no new inference code)
    migrate --check          # fail early on dirty migration state
    move submodule pointer to the commit that completed <spec>
    uv sync                  # pick up new deps
    if notes.requires_migrations: makemigrations && migrate
    if notes.requires_template_review: flag drift for notes.changed_template_paths (no auto-merge)
    if notes.requires_tailwind_rebuild: npm run tailwind_build
    makemigrations --check   # catch conflicts introduced by this spec
    uv run pytest            # test gate
    commit "Update FLS: <spec>"
# on failure mid-spec: follow documented rollback (reset pointer, discard integration commit)
```
