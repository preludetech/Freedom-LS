# Spec: Make the FLS test suite portable to concrete implementations ‚Äî Part 1 ("quiet the noise")

> **Part 1 of a two-part effort.** Part 1 (this spec) is the defensive
> bleed-stopper: it stops collection aborts and brand-noise that a downstream
> concrete project hits today when it runs `uv run pytest`, and updates the FLS
> claude-plugin docs so the new conventions hold in future development. Part 2
> ("positive signal" ‚Äî the conformance suite, system checks, upgrade-notes
> tie-in) is a separate spec (`fls-test-portability-part-2/idea.md`) and is out
> of scope here.

## 1. Background & problem

FLS ships its entire test suite **inside the installed `freedom_ls` package**
(`freedom_ls/<app>/tests/`), not in a top-level `tests/` dir that a downstream
could exclude. A concrete implementation runs `uv run pytest` with its own
`DJANGO_SETTINGS_MODULE` and ‚Äî finding no project-local `tests/` matching its
`testpaths` ‚Äî pytest silently falls back to full recursive collection from the
repo root and vacuums up every FLS test in the vendored subtree, running them
against the concrete project's own settings, theme, branding, icon set and app
list.

This was observed during the real `home_page` FLS update and produces three
failure modes:

1. **Collection-hard-failures (aborts the whole run).** Test modules import an
   optional app's factories/models at module scope. When a concrete project has
   legitimately not installed that app, the import raises Django's
   `RuntimeError: Model class ... isn't in an application in INSTALLED_APPS` at
   *collection* time, which aborts the entire session
   (`Interrupted: N errors during collection`, exit code 2) ‚Äî the operator sees
   nothing green at all.
2. **Branding / config-coupled assertion failures.** Many FLS tests assert
   FLS-default literals (icon `viewBox`, logo dimensions, demo-content strings,
   theme markup) that a concrete project is *expected* to override ‚Äî so they
   fail even though nothing is wrong.
3. **Browser / e2e tests that can't run headless-plain.** `@pytest.mark.playwright`
   tests need a running server + browser; a plain `uv run pytest` (with
   `pytest-socket` blocking the network) cannot run them.

### The deeper issue

"Run `uv run pytest` in the concrete project" conflates two goals:
**(a) FLS regression testing** ‚Äî belongs in the FLS repo against FLS's own
settings, not the downstream's job; and **(b) concrete integration
verification** ‚Äî *"did I wire FLS up correctly?"*, which is what a downstream
actually needs. Today a downstream accidentally does (a) and drowns in noise.
Part 1 attacks (a): it stops the noise and the aborts so that whatever real
signal is present becomes visible. Making (b) a first-class thing is Part 2.

### Current state (verified in this repo)

- `pyproject.toml` `[tool.pytest.ini_options]`:
  `addopts = "--strict-markers -m 'not ci_only' --disable-socket --allow-hosts=127.0.0.1,::1 --cov ..."`,
  `testpaths = ["freedom_ls", "tests", "fls-content-plugin"]`, and only two
  registered markers: `playwright`, `ci_only`. `--strict-markers` is on, so any
  unregistered marker is a hard collection error.
- Six test modules import `freedom_ls.course_applications` factories/models at
  module scope with **no guard**:
  - `freedom_ls/course_applications/tests/test_backends.py:12`
  - `freedom_ls/course_applications/tests/test_models.py:11`
  - `freedom_ls/course_applications/tests/test_queries.py:11`
  - `freedom_ls/course_applications/tests/test_views.py:12-13`
  - `freedom_ls/student_interface/tests/test_course_access_integration.py:27`
  - `freedom_ls/course_access/tests/test_visibility_enforcing_backend.py:19`
  `course_applications` is a real optional app (it is FLS's default
  `COURSE_ACCESS_BACKEND`, `config/settings_base.py:413-415`).
- No `@pytest.mark.fls_internal` marker exists anywhere; no `importorskip` /
  `apps.is_installed` / `collect_ignore` collection guards exist anywhere in
  `freedom_ls/`.
- De-branding surface partially mapped (see ¬ß5 for the rule and the
  representative table; the plan enumerates the rest): `icons/tests/test_renderer.py`
  (`test_returns_svg_with_viewbox` hardcodes `viewBox="0 0 24 24"` on the *ambient

  `::test_phosphor_icon_set` siblings *pin* the set via `@override_settings` and
  are already portable); further ambient-default-set coupling in
  `icons/tests/test_render.py` and `student_interface/tests/test_course_icon.py`;
  `accounts/tests/test_email_utils.py`
  (`test_email_logo_dimensions_scales_to_display_height` **and** the sibling
  `test_image_dimensions_reads_png_intrinsic_size` hardcode the shipped `512x248`
  logo); `content_engine/tests/test_demo_content_picture_titles.py` (reads a
  shipped `demo_content/` file that is **excluded from the packaged
  distribution**); `panel_framework/tests/test_htmx_navigation.py`
  (`test_non_htmx_returns_full_page`, theme/markup assertions ‚Äî cause to be
  pinned down).

## 2. Goal

Make an FLS upgrade **low-noise** for a concrete implementation:

1. Running the concrete project's tests surfaces **real integration problems**
   and nothing else ‚Äî no brand-mismatch or browser-setup noise.
2. FLS's own brand/demo/e2e tests are **cleanly excludable** by downstreams and
   **degrade to skips, never collection aborts**, when an optional app is absent.
3. The new conventions are documented in the FLS claude-plugin so future
   development is held to them.

(The positive "am I wired correctly?" signal is Part 2 and out of scope.)

## 3. Scope

**In scope**

- **Layer 0 ‚Äî Test-dependency hygiene (done FIRST):** audit *every* test-only
  cross-app dependency (the dashed edges in `docs/app_structure.md`) against the
  rule that an app whose runtime does not depend on another app should not have
  its tests depend on it either, and record a keep/remove verdict for each. In
  this codebase the audit lands as **all eight edges kept in Part 1**: seven are
  inherent, and the single incidental one (`markdown_rendering ‚Üí content_engine`)
  is a test-relocation refactor with no portability payoff, so it is carved off
  into its own debt spec rather than done here. Layer 0 runs first so the
  keep/guard decisions are settled before Layer 1 writes any collection guard.
- **Layer 1 ‚Äî Marker taxonomy + collection safety:** register `fls_internal`;
  guard optional-app module-scope imports so an absent app skips, never aborts.
- **Layer 2 ‚Äî De-brand FLS's own tests:** rewrite brand-literal assertions to
  test the contract; mark `fls_internal` only where a test genuinely depends on
  FLS's own brand/demo/repo state.
- **Layer 6 (Part-1 portions) ‚Äî Make it stick:** update the plugin skills,
  resources, code-reviewer agent, and the downstream `update_fls.md` command.

**Out of scope (Part 2 / elsewhere)**

- The conformance module, Django system checks for `COURSE_ACCESS_BACKEND`, and
  the upgrade-notes tie-in (Part 2 ‚Äî `fls-test-portability-part-2/idea.md`).
- The concrete project's own pytest config (`testpaths`, `--ignore`, shipping a
  `tests/` dir) ‚Äî that is template-repo / downstream work.
- No `e2e` marker is introduced (see ¬ß4a decision) and FLS's browser suite is
  **not** made to run from a concrete project ‚Äî only cleanly excludable.

## 4. Design decisions

### 4a. Marker taxonomy ‚Äî keep `playwright`, add `fls_internal`, **no `e2e`**

The taxonomy after this spec:

- **default (unmarked) = portable** contract/unit tests ‚Äî the
  downstream-valuable set;
- **`playwright`** = browser-dependent, needs a running server *(exists,
  unchanged)*;
- **`fls_internal`** = **new**; only valid under FLS's own
  settings/theme/branding/demo content;
- **`ci_only`** = existing slow/real-time tests *(unchanged)*.

**Decision ‚Äî no `e2e` marker.** The earlier research doc recommended adding an
`e2e` marker, but the refined idea overrides that: `playwright` already marks
the ~20 browser tests (all under per-app `tests/e2e/` dirs) and is the accurate
name. Introducing `e2e` as an alias/rename would be churn across every existing
Playwright test and every plugin doc code sample for no gain. The single stable
name a downstream excludes for browser tests is `playwright`.

`fls_internal` is registered in `pyproject.toml`'s `markers = [...]` ‚Äî mandatory
because `--strict-markers` is on. Prefer module-level
`pytestmark = pytest.mark.fls_internal` only for files that are *entirely*
brand-coupled; mixed files mark individual tests (a blanket `pytestmark` on a
file with portable contract tests would wrongly de-scope real integration
signal).

### 4b. Collection safety ‚Äî module-top `INSTALLED_APPS` skip guard

The guard must run **before** the offending import, in the same module, keyed on
`settings.INSTALLED_APPS` (a plain string check, safe to read without app-registry
readiness):

```python
from __future__ import annotations

import pytest
from django.conf import settings

if "freedom_ls.course_applications" not in settings.INSTALLED_APPS:
    pytest.skip("course_applications not installed", allow_module_level=True)

from freedom_ls.course_applications.factories import CourseApplicationFactory  # now safe
```

**Why this exact form** (from research):

- A bare `pytest.importorskip("freedom_ls.course_applications")` does **not**
  fix it ‚Äî the package *is* importable; it is the model-registry `RuntimeError`
  that fires when the model class is defined outside `INSTALLED_APPS`, which
  `importorskip` won't catch without an explicit `exc_type`. The
  `INSTALLED_APPS` string check is robust and needs no app-registry readiness.
- `allow_module_level=True` is required because `pytest.skip()` at module scope
  is otherwise a usage error.
- `@pytest.mark.skipif` does **not** work here ‚Äî the module-scope import raises
  before Python reaches the decorator.

**Belt-and-braces `collect_ignore_glob` (decision: include).** Add a colocated
`conftest.py` that inspects `settings.INSTALLED_APPS` and, when the app is
absent, populates `collect_ignore_glob` so the offending files are never even
imported:

```python
# freedom_ls/course_applications/tests/conftest.py
from django.conf import settings

collect_ignore_glob: list[str] = []
if "freedom_ls.course_applications" not in settings.INSTALLED_APPS:
    collect_ignore_glob = ["test_*.py"]
```

Rationale: this is the only mechanism that avoids the import *entirely* rather
than catching its failure, and it composes cleanly with the per-module guard as
defense-in-depth (belt: conftest never imports the file; braces: even if it did,
the module-level guard skips cleanly). `collect_ignore`/`collect_ignore_glob`
are scoped to the directory of the `conftest.py` that defines them, so the
conftest must live **in the same directory** as the guarded files ‚Äî hence a
`conftest.py` in `course_applications/tests/`. The two cross-app modules that
live in *other* apps' test dirs (`student_interface/tests/test_course_access_integration.py`,
`course_access/tests/test_visibility_enforcing_backend.py`) rely on the
per-module guard (their sibling tests must still collect), unless Layer 0
removes the dependency entirely.

### 4c. `testpaths` is not a safety net ‚Äî markers + `collect_ignore` are

`testpaths` only scopes discovery when pytest is given no path args, and
silently falls back to full recursive collection when its globs match nothing ‚Äî
which is the original bug. Command-line `-m` **replaces** (does not merge with)
`addopts`'s `-m`. The real controls are the `fls_internal`/`playwright` markers
plus `collect_ignore`. This is documented, not relied upon.

### 4d. FLS's own config stays full-suite; downstream exclusion is documented

**FLS's own `addopts` keeps running the full suite** (only `-m 'not ci_only'` as
today, plus the new marker *registered*) ‚Äî FLS regression testing must exercise
`fls_internal` and `playwright` tests against FLS's own settings. The
recommended downstream selection is
`-m "not playwright and not fls_internal and not ci_only"`. Because a
downstream's own `pyproject.toml` is its config source, this is **not** inherited
from the vendored FLS subtree ‚Äî it must be copy-pasted into the
downstream/template repo's `addopts`, and FLS's `markers =` block copied too
(`--strict-markers` requires referenced markers to be registered wherever config
resolves). Part 1 therefore documents the selection and applies it at the
downstream call sites in `commands/concrete/update_fls.md`; it does not change
FLS's own default exclusion. (Shipping that config into the template/downstream
repo itself is out of scope ‚Äî template-repo work.)

### 4e. Layer 0 ‚Äî remove test-only dependencies where genuinely unnecessary (runs FIRST)

**Rule:** if an app does not depend on another app for its functionality (no
runtime import ‚Äî a *solid* arrow in `docs/app_structure.md`), its tests ideally
should not depend on that other app either. A test-only cross-app import (a
*dashed* arrow) is a portability liability: it makes an otherwise-independent
app's test module unrunnable ‚Äî or, if the imported app is optional, abortable ‚Äî
in a downstream that has legitimately not installed the other app.

**Layer 0 is done before every other layer.** Every test-only edge removed is one
fewer guard to write in Layer 1 and one fewer coupling to reason about in Layer 2.
The hygiene pass therefore *gates* the collection-safety layer: Layer 1 only needs
to guard the `course_applications` importers that Layer 0 leaves in place.

**Audit surface ‚Äî every dashed (test-only) edge in `docs/app_structure.md`, not
just the `course_applications` ones.** The map can drift, so grep is
authoritative and `/app_map` should be re-run: the map currently *omits* the live
`course_access ‚Üí course_applications` dashed edge
(`test_visibility_enforcing_backend.py:19` imports `CourseApplicationFactory`) ‚Äî
a concrete demonstration that grepping the tree, not trusting the diagram, is how
the surface is enumerated. The eight test-only edges, each **read and judged**
(the plan's Task 0 carries the working):

| Test-only edge | What the test imports & why | Verdict |
|---|---|---|
| `student_interface ‚Üí course_applications` | `CourseApplicationFactory` + models (module scope) ‚Äî integration test whose purpose *is* the two apps together | **KEEP + guard** (optional app ‚Üí collection-abort risk) |
| `course_access ‚Üí course_applications` | `CourseApplicationFactory` (module scope) ‚Äî parametrizes both backends and proves delegation to the application backend | **KEEP + guard** (optional app; not incidental ‚Äî see plan Task 3) |
| `accounts ‚Üí student_management` | `UserCourseRegistration` as the outcome oracle of the deferred-login ‚Üí enrolment flow (`test_deferred_login.py` drives `student_interface`'s access view) | **KEEP** ‚Äî inherent to the flow under test; core app, no abort risk |
| `course_interest ‚Üí student_management` | `UserCourseRegistrationFactory` to materialise the *registered* state the `express_interest` view branches on (422 vs 404) | **KEEP** ‚Äî the factory creates the exact behavioural branch; inherent |
| `role_based_permissions ‚Üí student_management` | `CohortFactory`/`Cohort` ‚Äî the shipped role config **string-references** `student_management.view_cohort`/`add_cohort` perms; tests validate that config against the real `Cohort` | **KEEP** ‚Äî a stub model wouldn't carry the `view_cohort` permission; inherent |
| `webhooks ‚Üí accounts` | `UserFactory` (the `AUTH_USER_MODEL`) and `SiteFactory` (a thin wrapper for the Django builtin `Site`, housed in `accounts` by suite-wide convention) | **KEEP** ‚Äî `User` is inherent; `SiteFactory` coupling is a convention, not per-test laziness |
| `site_aware_models ‚Üí accounts` | `User` as the concrete `SiteAwareFactory` subclass under test; `SiteFactory` for site-cache tests | **KEEP** ‚Äî testing the abstract base needs a concrete site-aware model; a throwaway test model is the refactor this spec declines to force |

No portability risk (core app; function-local imports). Split into its own debt spec: `spec_dd/1. next/debt_markdown_rendering_package_isolation/`. |

**Outcome of the audit: all eight edges KEEP in Part 1; one is carved off into a
future debt spec.** Six of the eight edges import from *always-installed core
apps* (`accounts`, `student_management`, `content_engine`), so they carry **no
collection-abort risk** ‚Äî they are pure architectural hygiene, and all are
inherent functional coupling except one. That one ‚Äî `markdown_rendering ‚Üí
content_engine` (a lower-layer app's tests reaching *up* into a higher-layer app's
filters/components) ‚Äî **is** genuinely incidental, but its removal is a
test-relocation refactor with no portability payoff, so Part 1 records it and
hands it to a dedicated debt spec
(`spec_dd/1. next/debt_markdown_rendering_package_isolation/`) rather than doing
it under the "quiet the noise" banner. The two `course_applications` edges are
inherent *and* the sole collection-abort risk, so they are kept and handled by the
Layer 1 guard.

**Systemic note (not fixed here):** `accounts` is the de-facto test-support hub ‚Äî
`SiteFactory` wraps the Django builtin `Site` but lives in `accounts`, so most
apps import it from there. Relocating it (e.g. to `site_aware_models`) is a
suite-wide refactor with no portability payoff (`accounts` is always installed);
it is recorded as a future-cleanup candidate, out of scope for Part 1.

**Guardrail:** prefer removal, but "inherent" is a legitimate verdict ‚Äî the goal
is to delete *unnecessary* coupling, not to purify the graph at any cost, and not
to invent throwaway models or undertake large refactors. The one genuinely
incidental edge here is a test-relocation refactor with no portability payoff, so
it is deferred to its own debt spec rather than done under Part 1. Record each
verdict in the Task 0 commit message (criterion 5). Part 1 removes no edge, so it
makes **no test-code change** in Layer 0 ‚Äî the audit's product is the recorded
verdicts plus the carved-off debt spec; regenerating `docs/app_structure.md` via
`/app_map` is optional here (it would only capture the previously-missing
`course_access ‚Üí course_applications` edge).

## 5. De-branding the assertion surface (Layer 2)

Shrink the `fls_internal` set by making assertions test the **contract**, not
FLS-default literals. **De-brand where the test is really a contract test in
disguise; reach for `fls_internal` only when the test genuinely depends on FLS's
own brand/demo/repo state.** Every test moved from `fls_internal` ‚Üí portable
becomes real integration signal for every downstream.

### The rule (this is what the plan enumerates against)

The dominant portability failure mode is **ambient-config coupling**: a test that
silently assumes an FLS-default *ambient* value ‚Äî the default `FREEDOM_LS_ICON_SET`
(`heroicons`), the shipped logo asset, the demo content on disk ‚Äî and asserts a
literal derived from it. Under a downstream that has legitimately overridden that
value (a different icon set, a re-branded logo), the literal no longer matches and
the test fails though nothing is wrong. The fix, in priority order:

1. **Pin the input, keep the value.** If the test's subject is a *specific*
   configuration (a named icon set, an explicit logo), pin that input with
   `@override_settings` / an explicit argument and keep the exact assertion. This
   is the "test with a controlled input" pattern the testing skill endorses
   (¬ß6) ‚Äî a pinned-set test is already portable and must be **left as-is**, not
   loosened.
2. **Assert the contract, not the literal.** If the test's subject is the
   *ambient* default, assert structurally (a `viewBox="0 0 \d+ \d+"` regex, a
   scaling ratio) or derive the expected value from the configured asset rather
   than a constant.
3. **Mark `fls_internal`** only when the test genuinely cannot run outside FLS's
   own repo/demo/brand state (see the demo-content row).

**The table below is representative, not exhaustive.** It anchors the treatments;
**the plan must enumerate the full ambient-coupled surface** ‚Äî at minimum the
additional idea-flagged candidates `icons/tests/test_render.py` and
`student_interface/tests/test_course_icon.py` (both stub only `heroicons` and
assume it is the active set, e.g. `test_course_icon.py:70`), and the same-file
sibling `accounts/tests/test_email_utils.py::test_image_dimensions_reads_png_intrinsic_size`
(hardcodes `512x248` and reads the shipped repo logo path ‚Äî the same coupling as
the scaling test below). Enumerate by grepping for FLS-default literals and for
tests that read `settings`/shipped assets without pinning them, not by trusting
this table to be complete.

| File / test | Current literal | Part-1 treatment |
|---|---|---|
| `icons/tests/test_renderer.py::test_returns_svg_with_viewbox` | hardcoded `viewBox="0 0 24 24"` on the **ambient default** set | **De-brand.** Assert structurally (regex `viewBox="0 0 \d+ \d+"`, as `icons/tests/test_no_font_awesome.py:35` already does) or derive the box from the active `FREEDOM_LS_ICON_SET`. |
| `icons/tests/test_renderer.py::test_lucide_icon_set` (and the already-structural `::test_tabler_icon_set`, `::test_phosphor_icon_set`, `test_course_icon.py::test_active_set_respects_settings`) | `test_lucide_icon_set` asserts `viewBox="0 0 24 24"` under `@override_settings(FREEDOM_LS_ICON_SET="lucide")`; the tabler/phosphor siblings pin the set and assert only `<svg` | **Leave as-is ‚Äî already portable.** The set is *pinned*, so any literal is a controlled-input assertion, not brand coupling. Do **not** loosen `test_lucide_icon_set` to a regex; that would weaken a good test. These are the reference style for de-branding the ambient-default tests. |
| `accounts/tests/test_email_utils.py::test_email_logo_dimensions_scales_to_display_height` | hardcoded shipped logo `512x248` | **De-brand.** Read expected native dimensions from the *configured* logo asset, not a constant, then assert the scaling contract. (Apply the same treatment to the sibling `::test_image_dimensions_reads_png_intrinsic_size` ‚Äî see the enumeration note above.) |
| `content_engine/tests/test_demo_content_picture_titles.py` | reads a shipped `demo_content/` file off disk | **Mark `fls_internal`.** `demo_content*` is excluded from the packaged distribution, so a downstream hits `FileNotFoundError`, not a mismatch ‚Äî inherently repo-dependent. |
| `panel_framework/tests/test_htmx_navigation.py::test_non_htmx_returns_full_page` | theme/markup/ARIA strings (`id="sidebar-nav"`, `aria-label="Breadcrumb"`, ‚Ä¶) | **Pin down the actual First-Class failure cause at plan time**, then de-brand if it is a brand literal, else mark `fls_internal`. Visible assertions look structural, not brand-literal ‚Äî needs a closer look before deciding. |

## 7. Success criteria

1. `fls_internal` is registered in `pyproject.toml` `markers`; FLS's own
   `uv run pytest` still runs the full suite green (no change to its default
   `-m 'not ci_only'` exclusion).
2. With `course_applications` **removed** from `INSTALLED_APPS` (simulating a
   downstream that omits the optional app), `uv run pytest` **collects without
   aborting** ‚Äî the six affected modules degrade to skips (or are ignored via
   `collect_ignore_glob`), not `Interrupted: N errors during collection`.
3. Running with `-m "not playwright and not fls_internal and not ci_only"`
   deselects the browser suite and the genuinely brand/demo-coupled tests,
   leaving the portable contract set.
4. The de-branded icon and email-logo tests pass under a **non-default**
   `FREEDOM_LS_ICON_SET` / a differently-sized configured logo (they assert the
   contract, not FLS literals). Only genuinely repo-state-dependent tests
   (`test_demo_content_picture_titles.py`, and any `panel_framework` test that
   turns out brand-coupled) carry `fls_internal`.
5. Layer 0 (done first): **every** test-only cross-app dependency ‚Äî all eight
   dashed edges enumerated by grep (the map omits one), not just the
   `course_applications` ones ‚Äî is assessed and its verdict recorded in the Task 0

## 8. Why this matters

FLS is explicitly built to be embedded in downstream projects. Today a routine
FLS upgrade lands a downstream in a suite that is partly broken by design (brand
tests), partly abortive (optional-app collection errors), and partly un-runnable
(browser tests). Part 1 turns off the noise ‚Äî quiet where it should be quiet ‚Äî so
that when Part 2 adds the positive signal, a real wiring bug is the only thing
that shows up loud. Layer 6 keeps it that way as FLS grows.
