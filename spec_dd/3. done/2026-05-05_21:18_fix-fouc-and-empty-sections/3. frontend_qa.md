# Frontend QA: Fix FOUC on Alpine modals and hide empty dashboard sections

This QA plan verifies both parts of the spec end-to-end in a real browser via Playwright MCP. Use the `fls:use-playwright` skill for browser interaction and the `fls:sdd:do_qa` workflow to drive the run.

## Setup

1. Pick a port:

   ```
   PORT=$(.claude/fls/scripts/find_available_port.sh)
   ```

2. Start the dev server (background task):

   ```
   uv run python manage.py runserver $PORT
   ```

3. Use the **DemoDev** site for all QA test data (per project convention). All URLs below are relative to `http://127.0.0.1:$PORT`.

4. Make sure Tailwind has been rebuilt at least once after the implementation:

   ```
   npm run tailwind_build
   ```

5. The QA agent should use `qa-data-helper` to provision any missing test users / data described below.

## Test users

| User | Purpose | Required data shape |
|---|---|---|
| `flash_user` | exercises Part A (modal flash) | active student account on DemoDev with access to the `functionality-demo-show-end-with-quiz` course |
| `empty_user` | exercises Part B empty path | active student account on DemoDev with **no** registrations, **no** completed courses, **no** recommendations |
| `populated_user` | exercises Part B populated path | active student account on DemoDev with **at least one** recommendation and **at least one** completed course |

The exact email/password pairs should be created or reused via the `qa-data-helper` agent. If passwords are not known, log in via Django shell `force_login` or via a cookie-bearing client — record the method used in the QA report.

## Test 1 — Picture-zoom modal does not flash on first paint

**Purpose:** verify the global `[x-cloak]` rule prevents the `c-picture` overlay from appearing during the window between HTML parse and Alpine boot.

**Steps:**

1. Log in as `flash_user`.
2. Open Chrome DevTools (Playwright `browser_navigate` then `browser_evaluate` to set network throttling — or use Playwright's built-in throttling). Apply slow 3G throttling so the FOUC window is widened.
3. Hard-refresh the page `http://127.0.0.1:$PORT/courses/functionality-demo-show-end-with-quiz/1/`. Take a Playwright screenshot **immediately** (within ~200ms of navigation start). Then take another after the page has fully settled.
4. Confirm: at no point does the fullscreen modal overlay (the `c-picture` zoom modal) appear over the page content. The thumbnail image is visible, but the dimmed/blurred fullscreen layer must never render before the user clicks the thumbnail.
5. Click the picture thumbnail. The modal should open normally, with the image enlarged and the page dimmed. Close the modal (escape key or close button). The page returns to normal.
6. Repeat the hard-refresh capture at three viewport widths. Take a Playwright screenshot at each:
   - Desktop: `1280 x 800`
   - Tablet: `768 x 1024`
   - Mobile: `375 x 812`

**Pass criteria:**

- Initial screenshots at all three widths show no modal overlay before user interaction.
- Post-click screenshots show the modal opens correctly at all three widths.
- After closing, the modal is fully hidden again.

## Test 2 — Other Alpine `x-show` elements do not flash

**Purpose:** the audit identified two templates that needed an `x-cloak` patch (`course_minimal_toc.html` nested expand div, and `cotton/dropdown-menu.html` panel). Verify they no longer flash.

### 2a. Course minimal TOC nested expand

**Steps:**

1. Log in as `flash_user` and navigate to a course that uses minimal TOC and contains at least one Course Part with children — the `functionality-demo-show-end-with-quiz` course is fine if it has Course Parts; otherwise pick another DemoDev course that does.
2. Hard-refresh the course home / minimal TOC page with slow throttling.
3. Take an immediate screenshot. The nested children list under each Course Part must be **collapsed** at first paint — no flash of the expanded list.
4. Click the toggle button on a Course Part. The nested list should expand smoothly via `x-collapse`.
5. Capture screenshots at desktop, tablet, mobile widths.

**Pass criteria:** No visible flash of an expanded children list before Alpine boots. Toggle still works after Alpine boots.

### 2b. Dropdown menu

**Steps:**

1. Find a page in the educator or student interface that uses `c-dropdown-menu` (3-dots menu). Common locations: cohort detail, student detail, or any list with row-level actions. Use the educator interface on DemoDev if no student-facing dropdown is reachable.
2. Hard-refresh with throttling.
3. Take an immediate screenshot. No dropdown panel should be visible anywhere on the page.
4. Click the dropdown trigger. The panel should open with the existing transition.
5. Click outside the panel — it should close.

**Pass criteria:** Closed state is enforced by CSS at parse time. Open / close behaviour unaffected.

## Test 3 — Modal cotton component still works

**Purpose:** sanity check that adding the global `[x-cloak]` rule did not regress the existing `cotton/modal.html` whose nested `x-show` elements rely on the cloaked outer wrapper (no individual cloak).

**Steps:**

1. Log in and navigate to any page that uses `c-modal` — the course-card "Description" button on the dashboard opens one (`{% partial view-course-button %}` in `course_list.html`).
2. Visit `http://127.0.0.1:$PORT/` as a user who has registered courses (or a course with a Description). Hard-refresh.
3. Confirm no modal is visible at first paint.
4. Click "Description" on a course card. The modal should open with the title, body, and footer buttons.
5. Click "Close" or the backdrop. The modal closes.

**Pass criteria:** No flash at parse time, modal opens and closes normally.

## Test 4 — Sidebar / mobile drawer

**Purpose:** confirm the existing `_base_interface.html` and `panel_framework/partials/sidebar_nav.html` `x-show` elements remain non-flashing.

**Steps:**

1. Resize to mobile width (`375 x 812`) and visit any authenticated page that uses `_base_interface.html`.
2. Hard-refresh. Take an immediate screenshot. The mobile sidebar drawer and its backdrop must not appear before Alpine boots.
3. Tap the menu button. Drawer slides open with backdrop. Tap the backdrop or the close control. Drawer closes.

**Pass criteria:** No flash; behaviour unchanged.

## Test 5 — Empty dashboard hides "Recommended Courses" and "Learning History"

**Purpose:** verify Part B for a user with no recommendations and no completed courses.

**Steps:**

1. Log in as `empty_user`.
2. Navigate to `http://127.0.0.1:$PORT/`.
3. Wait for the HTMX-loaded course list partial to populate (the dashboard fetches `student_interface:partial_list_courses` on load).
4. Take a screenshot at desktop width.
5. Inspect the DOM (Playwright `browser_evaluate` running `document.body.innerHTML` or a targeted `querySelector`).

**Pass criteria:**

- The string `"Recommended Courses"` is not present anywhere in the rendered HTML (as either heading text or `<h2>` content).
- The string `"Learning History"` is not present anywhere in the rendered HTML.
- The string `"No recommended courses yet"` is not present.
- The string `"No completed courses yet"` is not present.
- The "Your Courses" section *is* visible — either with its registered courses or with the existing "You haven't signed up for any courses yet" empty-state line. (Out of scope for this change.)
- The "All Courses" CTA button is still present at the bottom.
- Screenshots taken at desktop (`1280 x 800`), tablet (`768 x 1024`), and mobile (`375 x 812`) widths look clean — no orphan headings, no awkward whitespace where the hidden sections used to be.

## Test 6 — Populated dashboard shows both sections

**Purpose:** verify Part B does not over-hide for a user with data.

**Steps:**

1. Log in as `populated_user`.
2. Navigate to `http://127.0.0.1:$PORT/` and wait for the partial to load.
3. Take a screenshot at desktop width.
4. Inspect the DOM.

**Pass criteria:**

- The `"Recommended Courses"` heading is rendered, with its grid of course cards underneath.
- The `"Learning History"` heading is rendered, with its grid of completed-course cards underneath.
- The "Your Courses" section is visible if the user has any registrations.
- The "All Courses" CTA button is still present.
- Screenshots at desktop, tablet, and mobile widths render cleanly with all three sections stacked correctly.

## Test 7 — Mixed populated state

**Purpose:** confirm the two `{% if %}` guards are independent — one section hidden does not affect the other.

**Steps:**

1. Pick or create a user with **recommendations but no completed courses** (call them `recs_only_user`).
2. Log in, visit `/`, take a screenshot.
3. Inspect the DOM.

**Pass criteria:**

- `"Recommended Courses"` is present.
- `"Learning History"` is **not** present.

Then repeat with a user who has **completed courses but no recommendations** (`history_only_user`):

- `"Learning History"` is present.
- `"Recommended Courses"` is **not** present.

## Failure handling

If any test fails, the QA report should:

1. State which test failed and at which step.
2. Attach the screenshot showing the failure.
3. Note whether the failure is a flash regression (Part A) or a section-hiding regression (Part B).
4. Per `feedback_tdd_for_bugfixes`, any subsequent fix must start with a failing test that reproduces the issue.
