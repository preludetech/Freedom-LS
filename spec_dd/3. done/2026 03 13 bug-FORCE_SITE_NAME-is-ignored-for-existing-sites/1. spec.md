# Bug Fix: FORCE_SITE_NAME is ignored for existing sites

## Problem

When `FORCE_SITE_NAME` is set in dev settings, some parts of the application ignore it and resolve the site by request domain instead. This causes two user-visible bugs:

### BUG 1: Cannot log in as existing users

With `FORCE_SITE_NAME=DemoDev`, running `python manage.py runserver 8010` and visiting `http://127.0.0.1:8010/accounts/login/`:

- The page renders the DemoDev site (templates/branding resolve correctly via `get_cached_site`)
- But logging in as `demodev@email.com` fails with "The email address and/or password you specified are not correct."
- Attempting to sign up as `demodev@email.com` gives an IntegrityError because the user already exists in the database

**Cause:** `UserManager.get_queryset()` calls `get_current_site(request)` which resolves by domain (`127.0.0.1:8010`). No site matches that domain, so the queryset filters out all users. The authentication backend then cannot find the user and reports invalid credentials.

### BUG 2: FORCE_SITE_NAME ignored when a matching domain site exists

With `FORCE_SITE_NAME=DemoDev`, running `python manage.py runserver 8001` and visiting the home page shows the **Bloom** site instead of DemoDev.

**Cause:** The Bloom site is configured with domain `127.0.0.1:8001`. Because `UserManager.get_queryset()` uses `get_current_site(request)` instead of `get_cached_site(request)`, it resolves to Bloom by domain match, completely ignoring the `FORCE_SITE_NAME` setting. This means the user queryset filters by the Bloom site, making DemoDev users invisible.

## Root Cause

The codebase has two functions for resolving the current site:

- **`get_cached_site(request)`** - Custom function that respects `FORCE_SITE_NAME` and caches the result on the request object.
- **`get_current_site(request)`** - Django's built-in function that resolves by request domain only.

The following locations incorrectly use `get_current_site()` instead of `get_cached_site()`:

| Location | Current (buggy) | Should be |
|---|---|---|
| `freedom_ls/accounts/models.py` - `UserManager.get_queryset()` | `get_current_site(request)` | `get_cached_site(request)` |
| `freedom_ls/accounts/allauth_account_adapter.py` - `AccountAdapter.is_open_for_signup()` | `get_current_site(request)` | `get_cached_site(request)` |
| `freedom_ls/xapi_learning_record_store/api.py` - `hello()` endpoint | `get_current_site(request)` | `get_cached_site(request)` |

Meanwhile, `SiteAwareManager.get_queryset()` and `SiteAwareModelBase.save()` correctly use `get_cached_site()`.

**Note:** `site_aware_models/factories.py` also calls `get_current_site()` inside `_get_current_site()`, but this is acceptable because factories are only used in tests where `mock_site_context` patches `get_current_site` directly. Similarly, `get_cached_site()` itself calls `get_current_site()` as a fallback when `FORCE_SITE_NAME` is not set — this is correct by design.

## Why this matters

The `FORCE_SITE_NAME` feature was introduced to support per-branch development databases with worktrees. Without it working correctly, developers cannot run the dev server on any arbitrary port and have site resolution work reliably. The dev experience breaks because:

- Authentication fails silently (misleading "wrong password" error)
- Signup gives a confusing IntegrityError instead of recognising the existing user

## Fix

Follow TDD: write a failing test for each bug, then fix the code to make it pass.

### 1. Tests (write first, expect failures)

- Add a test for `UserManager.get_queryset()` that sets `FORCE_SITE_NAME` and verifies the queryset filters by the forced site, not the request domain.
- Add a test for `AccountAdapter.is_open_for_signup()` that sets `FORCE_SITE_NAME` and verifies it checks the signup policy for the forced site, not the request domain.
- Ensure existing tests for `get_cached_site()` still pass.

### 2. `UserManager.get_queryset()` (`freedom_ls/accounts/models.py`)

Replace `get_current_site(request)` with `get_cached_site(request)`.

### 3. `AccountAdapter.is_open_for_signup()` (`freedom_ls/accounts/allauth_account_adapter.py`)

Replace `get_current_site(request)` with `get_cached_site(request)`.

### 4. `hello()` endpoint (`freedom_ls/xapi_learning_record_store/api.py`)

Replace `get_current_site(request)` with `get_cached_site(request)`.

### 5. Codebase audit

Grep the entire codebase for any other calls to `get_current_site(request)` that should be using `get_cached_site(request)` instead. Every place that resolves the current site in application code should use `get_cached_site()` to ensure consistent behaviour when `FORCE_SITE_NAME` is set.

## Success Criteria

1. With `FORCE_SITE_NAME=DemoDev`, running `python manage.py runserver 8010` (or any port) resolves to the DemoDev site.
2. Existing DemoDev users can log in successfully regardless of the port used.
3. No calls to `get_current_site(request)` remain in application code (only `get_cached_site(request)` should be used). Acceptable exceptions: inside `get_cached_site()` itself (as a fallback), inside test fixtures/factories (where `mock_site_context` patches it), and in `conftest.py`.
4. All new and existing tests pass.
