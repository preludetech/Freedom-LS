# Spec: more deploy preparation

Small, high-leverage deploy-readiness fixes on the FLS side, discovered while wiring up the
concrete-project deployment against the separate Ansible Infrastructure repo. Two threads:
make a misconfigured Sentry **release** loud and belong to the image, and close the
app↔infra `.env` gaps the infra `app_env` role depends on.

**Framing (load-bearing, repeated in the idea): efficiency and ease, no scope creep.** Every
choice below prefers the smallest change that removes a real footgun. The only behavioural
change to running code is the new fail-fast on a missing `WEBHOOK_ENCRYPTION_SALT` in
production; everything else is a check, a `.env`/doc parity fix, or a template-repo handoff.

Research backing: [`research_sentry_release.md`](./research_sentry_release.md),
[`research_app_infra_coupling.md`](./research_app_infra_coupling.md).

---

## Problem statement

1. **A blank `SENTRY_RELEASE` while `SENTRY_DSN` is set is invisible.** `init_sentry()`
   (`freedom_ls/deployment/sentry.py`) passes `release=config.SENTRY_RELEASE`, which is `None`
   when the env var is unset or blank. Sentry keeps working, but every event is untagged —
   regressions can't be tied to a deploy and "resolved in next release" silently degrades. The
   misconfiguration should be visible quickly, without breaking a working deploy.
2. **The release identifier belongs to the image, not to per-service runtime env.** Both the
   `web` (Gunicorn) and `db_worker` processes go through the same
   `DeploymentAppConfig.ready() → init_sentry()` path and read `config.SENTRY_RELEASE` from the
   process environment. If each Compose service sets it independently, the two can drift and
   split one deploy's events across two release tags. Baking it into the image at build time
   makes it identical for every container from that image, with no per-service sync and no
   drift on a plain `docker compose restart`. (This lands in the template repo — see Handoffs.)
3. **The app↔infra coupling has `.env.example` gaps.** The Ansible Infrastructure repo's
   `app_env` role fills each stack's `.env` from vault using **`.env.example` as the canonical
   key list**. Three prod settings are read by FLS but absent from `.env.example`. The
   highest-signal one is `WEBHOOK_ENCRYPTION_SALT`, which in production **silently falls back to
   a hardcoded insecure dev salt** (`config/settings_base.py:443-450`) — unlike `SECRET_KEY`,
   which hard-fails via `require_secret_key()`.

## Scope — this FLS repo

### A. Sentry release warning — Django system check (`Warning`)

Add a Django system check in a new `freedom_ls/deployment/checks.py`, wired the same way
`course_access` wires its checks. This mirrors the established repo pattern for
"config is internally inconsistent, should be visible on every boot, must not crash a working
deploy" — a conditional `Warning`, not a runtime raise.

- **Shape:** modelled on `freedom_ls/course_access/checks.py`'s
  `check_preview_overrides_disabled_in_production` (W001) — a conditional Warning with an
  early-return guard. (That reference check is registered `@register(Tags.security)`; this new
  check deliberately registers **untagged** — see **Registration** below — because a missing
  telemetry tag is not a security boundary. Neither tagging affects whether it runs: both fire
  on a plain `manage.py check` and under `check --deploy`.)
- **Condition:** fires when `config.SENTRY_DSN` is truthy **and** `config.SENTRY_RELEASE` is
  falsy. `AppSettings.__getattr__` already strips whitespace and normalises `""`→`None`
  (`freedom_ls/base/app_settings.py:36-39`), so `config.SENTRY_RELEASE` is `None` for both an
  unset and a blank (`SENTRY_RELEASE=`) var — the check tests plain falsiness and needs no
  blank-string handling of its own.
- **Severity:** `Warning`, not `Error`. A wrong/blank release only degrades Sentry's
  release-based features; it never breaks the running app. Consistent with
  `course_access.W001`.
- **Registration:** plain `@register()` (untagged). Untagged checks run on every
  `manage.py check` / `migrate` / `runserver` / `test`, so the warning surfaces at the exact
  moment the deploy pipeline's one-off `manage.py migrate` runs against real prod env vars, and
  the existing CI job `django-check-deploy` (`.github/workflows/security.yml`,
  `check --deploy --fail-level WARNING`) picks it up for free. **Not** `deploy=True` (that would
  exclude it from the default run). **Not** `Tags.security` (a missing telemetry tag is not a
  security boundary).
- **Check ID:** `freedom_ls_deployment.W001`. **Decision:** add an explicit
  `label = "freedom_ls_deployment"` to `DeploymentAppConfig` (`freedom_ls/deployment/apps.py`).
  The app has no models and no migrations, and nothing references it by label (INSTALLED_APPS
  uses the full name `freedom_ls.deployment`), so this is safe and matches the
  `freedom_ls_course_access.W001` namespacing precedent — chosen over the bare Django-default
  `deployment.W001` for consistency and unambiguous `SILENCED_SYSTEM_CHECKS` entries.
- **Wiring:** register the check in `DeploymentAppConfig.ready()` by importing
  `freedom_ls.deployment.checks` (same pattern as `CourseAccessConfig.ready()`), alongside the
  existing `init_sentry()` call.
- **Silenceable:** a deployment that deliberately doesn't want release tracking silences
  `freedom_ls_deployment.W001` via `SILENCED_SYSTEM_CHECKS` rather than living with a permanent
  nag. Document the ID wherever the deployment security checklist already documents silenced
  check IDs.

**Why a check and not the alternatives** (from `research_sentry_release.md` §1):
- *A logged warning in `init_sentry()`* — weaker (only visible in log aggregation, not a
  deploy/CI exit code) and against the "don't add logging unless asked" convention. Recorded as
  a documented **non-goal for V1**, not rejected forever: an easy follow-up if operators later
  want a boot-time stdout signal for a bare `docker compose up` with no `check`/`migrate` in
  front of it.
- *Fail-fast (`raise`)* — wrong altitude. `require_secret_key()`'s crash-loop is justified
  because a blank `SECRET_KEY` disables the session/CSRF security boundary; a blank
  `SENTRY_RELEASE` degrades one feature of an already-optional integration. Raising would take
  down a deploy for a strictly less severe problem — against the "efficiency and ease" framing.

### B. `.env.example` gap closures (the infra `app_env` key list)

- **`WEBHOOK_ENCRYPTION_SALT` — the one gap with real security bite.**
  - Add it to `.env.example`, marked **secret**, with a one-line comment that an unset value in
    production silently weakens webhook-secret Fernet encryption (so the reason it matters is on
    the shape file, not just a bare var).
  - **Make production fail-fast when unset.** Add a `require_webhook_encryption_salt()`
    primitive to `freedom_ls/deployment/settings_defaults.py`, mirroring `require_secret_key()`
    (read env, `.strip()`, raise `ImproperlyConfigured` with a clear message when blank/unset).
    In `config/settings_prod.py`, reassign `SALT_KEY = fls_defaults.require_webhook_encryption_salt()`
    **after** the `from .settings_base import *` line (the same place `SECRET_KEY` is
    reassigned). This leaves the deterministic dev fallback in `settings_base.py` intact for
    dev/test, and closes the "insecure dev salt silently reaches prod" path — the fallback can
    no longer survive into production.
  - **Two existing `settings_prod` loaders must be given a dummy salt, or this change breaks
    them.** Both currently load `config.settings_prod` with no `WEBHOOK_ENCRYPTION_SALT`, so the
    new fail-fast would raise `ImproperlyConfigured` at settings-import time — before either does
    anything useful. Each already supplies a dummy `SECRET_KEY` for exactly this reason; a dummy
    `WEBHOOK_ENCRYPTION_SALT` must be added the same way:
    - The CI `django-check-deploy` job (`.github/workflows/security.yml`) runs
      `check --deploy --settings=config.settings_prod`; add `WEBHOOK_ENCRYPTION_SALT` to its
      `env:` block alongside the existing dummy `SECRET_KEY`. Without it the job errors on
      settings import and never reaches the Sentry check.
    - The existing test `test_prod_settings_uses_database_task_backend`
      (`freedom_ls/deployment/tests/test_settings_defaults.py`) reloads `config.settings_prod`
      with only `HOST_DOMAIN`/`SECRET_KEY` monkeypatched; it must also set
      `WEBHOOK_ENCRYPTION_SALT`, or the reload raises.
  - **Rollout note:** this is the only behavioural change to running code. Any environment
    currently running without `WEBHOOK_ENCRYPTION_SALT` set will crash-loop on boot until the
    var is supplied — that is the intended fix (it was silently insecure before). This is
    deploy-*preparation* ahead of the first real deployment, so there is no existing prod data
    encrypted under the dev-fallback salt to migrate; the upgrade-notes step should still call
    out the required var and the crash-on-missing behaviour for any downstream project.
- **`DJANGO_ADMIN_URL`** — add to `.env.example` (config, safe default `admin/`, read at
  `config/urls.py:31`). Documentation-parity fix: already documented in
  `docs/deployment-security-checklist.md` and `docs/product/admin-interface.md`.
- **`HSTS_SECONDS` / `HSTS_INCLUDE_SUBDOMAINS` / `HSTS_PRELOAD`** — add to `.env.example`
  (config; safe defaults `3600` / `False` / `False` already in `config/settings_prod.py:28-32`).
  Documentation-parity fix so `.env.example` and the security checklist stop drifting. No code
  change — the defaults already exist.

## Handoff to the template repo (`freedom-ls-concrete-template`)

Delivered later via the existing SDD `/update_template_repo` sync step — **not this repo's
code**. Recorded here so nothing is lost:

- **Bake `SENTRY_RELEASE` at build time.** `ARG SENTRY_RELEASE` / `ENV SENTRY_RELEASE=…` in the
  Dockerfile's **final runtime stage only** (an `ARG` used inside a stage must be re-declared
  inside that stage; the build-only stages don't need it), sourced from `github.sha` via
  `docker/build-push-action` `build-args:`.
- **Value convention:** raw full git SHA for V1 (no new semver scheme — inventing one is scope
  creep the idea's framing argues against; a plain SHA satisfies all Sentry release-name
  restrictions).
- **GHCR tag correspondence — decided: loose.** Keep the GHCR tag at `docker/metadata-action`'s
  default `type=sha` (`sha-<short>`) and set `SENTRY_RELEASE` to the full `github.sha`
  independently. They are not byte-identical but both trace to the same commit, and the full SHA
  is what Sentry recognises for any future commit-association work. No metadata-action config
  change. (The byte-identical option — `type=sha,format=long,prefix=` — was considered and not
  chosen: more config for a lookup-free hop that isn't needed for V1.)
- **Note:** baking the release as an `ENV` makes it visible in `docker inspect`/`docker
  history` — fine, it's a git SHA, not a secret. Contrast `SENTRY_AUTH_TOKEN`, which must never
  be baked into a pushed image (only relevant if/when commit-association is adopted — see
  out-of-scope).
- **Pin a stable per-stack Postgres `container_name`** in the template's Compose file
  (`lms-staging-db` / `lms-prod-db`) matching the infra inventory's `pg_container_name`, which
  the `backups` role `docker exec`s. Pure naming handshake between two files, neither in this
  repo.
- **Mirror the `.env.example` additions** (Section B) into the template's copy once they land
  here.

## Out of scope / already handled (no action)

- **`sentry-cli` release finalize / commit-association / deploy markers** — documented *future*
  step only. Simply passing the `release` string already solves problem #1 for V1. Commit
  association needs a new `SENTRY_AUTH_TOKEN` secret + a Sentry↔GitHub integration; source maps
  are irrelevant to a server-rendered Django/HTMX app. Do **not** add a `SENTRY_AUTH_TOKEN` var,
  CI step, or integration setup in this V1 — a one-line "future step" note where the Sentry
  service is documented is enough.
- **`SENTRY_ENVIRONMENT`** has an analogous "set-it-whenever-DSN-is-set" note in `.env.example`
  but is **not** part of this idea. Do **not** let the new check quietly grow to cover it.
- **Health probes, `db_worker` + `prune_db_task_results`, backup restore-drill query
  (`SELECT count(*) FROM django_migrations`)** — all already shipped/valid (health endpoints
  installed and mounted; DB task backend is the prod default). Template-repo/infra concerns, not
  new app code.
- **`LEGAL_DOCS_MANIFEST_PATH`, `FLS_THEME`** — correctly *not* `.env.example`/vault-fill
  concerns (build-time / image-baked, not per-stack secrets). No action.
- **No `docker-compose.yml` in this repo** — production Compose (and `container_name`) is owned
  entirely by the template repo. Confirmed; nothing to build here.

## Success criteria

**Sentry release check**
- A new `freedom_ls/deployment/checks.py` defines a `@register()`ed check returning a single
  `Warning(id="freedom_ls_deployment.W001")` **iff** `SENTRY_DSN` is set and `SENTRY_RELEASE`
  is falsy; it returns `[]` in every other combination (both set, DSN unset, both unset).
- `DeploymentAppConfig` sets `label = "freedom_ls_deployment"` and imports the checks module in
  `ready()`; `manage.py check` emits `freedom_ls_deployment.W001` in the DSN-set/release-blank
  case and is silent otherwise.
- The check does not block `migrate`/`runserver` (it is a `Warning`) and is silenceable via
  `SILENCED_SYSTEM_CHECKS`. The existing CI `django-check-deploy` job stays green (its env sets
  no `SENTRY_DSN`, so the check stays silent there — no false positive).
- Tests cover all four `DSN × RELEASE` truthiness combinations (including a blank
  `SENTRY_RELEASE=` normalising to falsy).

**Webhook salt fail-fast**
- `require_webhook_encryption_salt()` exists in `settings_defaults.py`, returns the stripped env
  value when set, and raises `ImproperlyConfigured` when unset/blank/whitespace-only.
- `config/settings_prod.py` reassigns `SALT_KEY` from it; a prod settings load with
  `WEBHOOK_ENCRYPTION_SALT` unset raises `ImproperlyConfigured`, and dev/test (using
  `settings_base`'s deterministic fallback) is unaffected.
- Unit tests mirror the existing `require_secret_key()` tests (set / unset / whitespace-only).
- The CI `django-check-deploy` job's `env:` and the existing
  `test_prod_settings_uses_database_task_backend` test both supply a dummy
  `WEBHOOK_ENCRYPTION_SALT`, so the new fail-fast does not break the `settings_prod` load each
  performs — `uv run pytest` and that CI job stay green.

**`.env.example` parity**
- `WEBHOOK_ENCRYPTION_SALT` (secret, with security comment), `DJANGO_ADMIN_URL` (config,
  default `admin/`), and `HSTS_SECONDS` / `HSTS_INCLUDE_SUBDOMAINS` / `HSTS_PRELOAD` (config,
  documented defaults) all appear in `.env.example`, marked secret/config consistently with the
  existing entries.
- No settings key read by `config/settings_prod.py` that the infra `app_env` role must fill
  remains absent from `.env.example`.

**Overall**
- `uv run pytest` passes. The only runtime-behaviour change is the webhook-salt fail-fast;
  Sentry with a valid config behaves exactly as before.
- The template-repo and future-`sentry-cli` items are recorded (this spec + the SDD handoff
  steps) and explicitly not implemented in this repo.
