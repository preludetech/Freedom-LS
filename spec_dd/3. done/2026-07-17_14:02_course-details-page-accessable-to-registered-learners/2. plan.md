# Plan: Make the course details page reachable for registered learners

Implementation plan for `1. spec.md`. This feature is **template-only**: it adds a
low-emphasis "Details" text link to eight course-card / course-row leaf partials and
repoints one breadcrumb link. No models, URLs, views, or access rules change.

Work is done **TDD-first** (see `fls:testing`): for each unit, write the failing
`href`-assertion test, then edit the template to make it pass.

---

## 1. Approach at a glance

| # | Unit | Files | What changes |
|---|---|---|---|
| A | Shared link partial | new `partials/course_details_link.html` | The `<a>Details</a>` markup, extracted once |
| B | Card leaves (×4) | `course_card_registered/complete/not_registered/coming_soon.html` | `{% include %}` the partial on its own right-aligned footer line |
| C | Row leaves (×4) | `course_row_registered/complete/not_registered/coming_soon.html` | `{% include %}` the partial inline, right-aligned beside the title |
| D | Breadcrumb repoint | `partials/player_breadcrumbs.html` | First crumb → `course_detail`; fix stale comment |

The destination `student_interface:course_detail`
(`courses/<slug:course_slug>/detail/`, `urls.py:10-14`) already exists and is already
reachable — **no backend work** (spec §4).

## 2. Shared partial vs inline (spec §11 judgement call): extract a partial

The link markup is **byte-identical** across all eight leaves (same `href`, same label
"Details", same chevron, same styling). Only the *positioning wrapper* differs per
surface (card footer line vs row inline). Therefore extract the link body **once** into
a Django `{% include %}` partial and let each call site own its wrapper. This is the DRY
choice and is explicitly permitted by spec §7.

Note on `templates_and_cotton.md`: its rule *"Don't create cotton components for one-off
use — use partials"* forbids a **Cotton `<c-*>` component**, not a Django `{% include %}`
partial. A shared `{% include %}` fragment is exactly the recommended tool here, so this
plan does **not** create a Cotton component. `<c-button>` is likewise not used (spec §7 —
too heavy for a low-emphasis secondary link).

### Unit A — `partials/course_details_link.html`

Create `freedom_ls/student_interface/templates/student_interface/partials/course_details_link.html`.

The link reuses the existing "Browse all courses" affordance
(`course_list.html:92-98`): `text-sm font-medium no-underline hover:text-primary-hover`,
`inline-flex items-center gap-1`, trailing `next` icon in an `aria-hidden="true"` span.
Add `relative z-10` so the anchor paints **above** the title anchor's `::before` overlay
(spec §3.2 — the seam is *established here*, it does not pre-exist), and `p-3` for a
~44×44px touch target (spec §6; note `p-2` = ~36px total against the `text-sm` ~20px
line-height, which misses the 44px figure — use `p-3` = 12px/side → ~44px). Carry
`whitespace-nowrap` from the reused pattern so the label + chevron never wrap on a narrow
row. `course` is available via inherited include context.

The shipped `{% comment %}` block below carries no spec/section citations (`code-comments`
skill: keep the reasoning, drop the citation):

```django
{% comment %}
"Details" entry point to the course overview page (student_interface:course_detail).
Low-emphasis secondary link; reuses the "Browse all courses" text-link affordance.

`relative z-10` lifts this anchor above the sibling title link's ::before stretched
overlay so it is independently clickable and keyboard-focusable. `p-3` gives an
adequate (~44px) touch target. The chevron is decorative (aria-hidden).

Context: course (needs course.slug). Placement/alignment is the caller's job:
cards wrap this on its own right-aligned footer line; rows render it inline,
right-aligned beside the title.
{% endcomment %}
<a href="{% url 'student_interface:course_detail' course_slug=course.slug %}"
   class="relative z-10 inline-flex items-center gap-1 whitespace-nowrap p-3 text-sm font-medium no-underline hover:text-primary-hover">
    Details
    <span aria-hidden="true">
        <c-icon name="next" class="size-4" />
    </span>
</a>
```

Skills: `fls:template` (partial conventions, `{% url %}`), `fls:frontend-styling`
(Tailwind text-link classes), `fls:icon-usage` (`next` is an existing semantic icon —
no new icon registration).

## 3. Unit B — Dashboard card leaves (spec §5.1, §3.4)

Placement per §3.4: **content footer, bottom-right, own line**, inside the card body
below the hero and **clear of the progress bar / completed badge**. The card body slot
renders `eyebrow → default slot → footer`; the progress bar / badge live in the `footer`
slot. So the Details line goes at the **end of the default slot** (after the title, and
after the "Next up" line where present) — it renders above the footer, on its own
right-aligned line.

Wrapper at each call site:

```django
<div class="flex justify-end">
    {% include "student_interface/partials/course_details_link.html" %}
</div>
```

Per-leaf insertion points:

- **`course_card_registered.html`** — after the optional `Next up:` `<p>` (line ~44),
  still inside the default slot, before `<c-slot name="footer">`.
- **`course_card_complete.html`** — after the `card_title_link.html` include (line ~27),
  before `<c-slot name="footer">`.
- **`course_card_not_registered.html`** — after the title `<h3>` (line ~35). The
  whole-card stretched link already targets `course_detail`; the explicit Details link is
  intentionally redundant and must resolve to the **identical** URL (spec §3.3).
- **`course_card_coming_soon.html`** — after the `flex items-start justify-between`
  title/chip block (closes line ~33).

The stretched title link keeps its current whole-card behaviour into the player (spec
§5.1); the not-registered card's whole-card link already points at the details page and
is unchanged.

Skills: `fls:template`, `fls:frontend-styling`, `fls:icon-usage`.

## 4. Unit C — All-courses row leaves (spec §5.2)

Rows have more horizontal room (`flex-1` details column), so the Details link renders
**inline, right-aligned beside the title** rather than on a dedicated line — following the
existing coming-soon-row precedent of an inline right-aligned element beside the title
(`course_row_coming_soon.html:29-37`).

Standard wrapper (registered / complete / not-registered rows): put the title and the
Details link in a single `justify-between` flex row. Concretely, wrap the **entire**
existing title block — for the registered/complete rows that is the whole
`{% with url=... title=... %}{% include "…/card_title_link.html" %}{% endwith %}` block
(all three lines, not just the `{% include %}` line); for the not-registered row it is the
bare title `<h3>` — in the flex `<div>`, then add the Details include beside it:

```django
<div class="flex items-center justify-between gap-2">
    [ existing title block — the {% with %}…{% include card_title_link %}…{% endwith %},
      or the bare <h3> — placed here unchanged ]
    {% include "student_interface/partials/course_details_link.html" %}
</div>
```

(The bracketed line above is a plan-only marker, not template syntax — do not paste it
into the file; `fls:template` forbids Jinja-style `{# … #}` comments in templates.)

Per-leaf insertion points:

- **`course_row_registered.html`** — wrap the `card_title_link.html` include (line ~35).
- **`course_row_complete.html`** — wrap the `card_title_link.html` include (line ~30).
- **`course_row_not_registered.html`** — wrap the title `<h3>` (lines ~33-38). Same
  intentional-redundancy note as the not-registered card — both links go to the identical
  `course_detail` URL.
- **`course_row_coming_soon.html`** — this row already has a
  `flex items-start justify-between gap-2` block holding the title on the left and a
  "Coming soon" chip on the right (lines ~29-37). Add the Details link into the right
  cluster so the chip and Details sit together, e.g. wrap the chip + include in a
  `flex items-center gap-2` on the right side of the existing justify-between. Keep the
  title on the left untouched. Note the outer block stays `items-start` (unchanged), so
  when eyeballing the four rows side by side the coming-soon Details may sit fractionally
  higher than the vertically-centred Details on the other three rows — cosmetic only, and
  changing the pre-existing `items-start` is out of scope.

The all-courses page is public (spec §5.2): the link must render for anonymous,
registered, and authenticated-unregistered visitors. The destination is public for all
non-`HIDDEN` courses, so no visibility branch is needed — the same markup renders for all
three audiences.

Skills: `fls:template`, `fls:frontend-styling`, `fls:icon-usage`.

## 6. Testing (TDD — spec §8)

Write the assertion **before** each template edit. Per `fls:testing` and project
conventions: assert **behaviour and rendered `href`s only — no CSS-class / style
assertions**. Use `reverse("student_interface:course_detail", kwargs={"course_slug": ...})`
for the expected URL and `factory_boy` factories for data (`UserFactory`,
`CourseFactory`, `TopicFactory`, `UserCourseRegistrationFactory`, `CourseProgressFactory`).

### 6.1 Card affordance — extend `tests/test_course_cards.py`

For **each of the four card states**, render the dashboard
(`reverse("student_interface:dashboard")`) with a course in that state and assert the
response body contains a "Details" link whose `href` resolves to `course_detail` for that
course.

- Registered (0%) and In-progress (>0%): use `UserCourseRegistrationFactory`
  (+ `CourseProgressFactory` / `TopicProgressFactory` for in-progress), following the
  existing fixtures in this file (`course_with_topics`, `logged_in_client`).
- Complete: registration + `CourseProgressFactory(progress_percentage=100,
  completed_time=timezone.now())`.
- Not-registered: appears in the dashboard's Recommended / Available sections — follow the
  existing not-registered setup already present in this file.
- Coming-soon: build a `CourseVisibility.COMING_SOON` course. **Neither
  `test_course_cards.py` nor `test_dashboard_view.py` has a coming-soon fixture** — copy the
  real precedent in `tests/test_listing_visibility.py` (its local `_course(visibility=…)`
  helper and the `test_dashboard_coming_soon_*` tests), so the coming-soon "Details"
  assertion is actually written, not skipped for lack of a template.

Suggested assertion shape (per state):

```python
detail_url = reverse("student_interface:course_detail",
                     kwargs={"course_slug": course.slug})
body = response.content.decode()
assert "Details" in body
assert f'href="{detail_url}"' in body
```

For the not-registered card the detail URL already appears (stretched link); assert the
**"Details"** visible text is present in addition, so the explicit affordance is proven,
and that both occurrences resolve to the same URL.

### 6.2 Row affordance — extend `tests/test_all_courses_rows.py`

Same per-state assertion on the all-courses page
(`reverse("student_interface:courses")`) for all four row states. **Additionally** assert
the Details link renders for:

- an **anonymous** visitor (use the plain public client, per `test_all_courses_public.py`
  conventions), and
- an **authenticated-but-unregistered** visitor (`logged_in_client` with no
  registration).

Reuse the file's existing `courses` fixture and state-setup helpers for the
registered / in-progress / complete / not-registered rows. The `courses` fixture
(`conftest.py`) only creates default-visibility courses, so for the **coming-soon** row
build a `CourseVisibility.COMING_SOON` course explicitly, copying the precedent in
`tests/test_listing_visibility.py` (`_course(visibility=…)` + `test_all_courses_coming_soon_*`).

### 6.3 Breadcrumb — new test file

No breadcrumb test exists. Add a small plain Django `Client` test (e.g.
`tests/test_player_breadcrumbs.py`) that requests a `view_course_item` page for a
registered learner and asserts the **first breadcrumb anchor's `href`** resolves to
`course_detail` (and **not** to `view_course_item` index 1). Parse with a targeted regex /
substring on the breadcrumb `<nav aria-label="Breadcrumb">` block, asserting the
`course_detail` URL is the first crumb's href.

### 6.4 Optional E2E (spec §8, §11 — judgement call): **skip**

Do **not** add a Playwright E2E test. Rationale: the three pytest layers above already
assert the exact rendered `href`s for every state and the breadcrumb, which is the
behaviour that matters; navigation from an `<a href>` needs no JS and adds no
HTMX/interaction risk worth a slow browser test (`fls:testing`: prefer pytest;
`fls:playwright-tests` reserved for real interaction/HTMX flows). Independent
clickability of the link over the `::before` overlay is guaranteed structurally by the
`relative z-10` treatment (Unit A) and is not asserted in an automated test.

### 6.5 Regression

Run the full suite (`uv run pytest`) — the existing card/row/breadcrumb tests must stay
green (spec §9.6). The breadcrumb change may touch existing assertions in
`test_course_item_navigation.py` / `test_resume_and_redirect.py` that check the first
crumb's href; update any that assumed index-1 to expect `course_detail`.

**Explicitly handle this now-stale test:**
`test_resume_and_redirect.py::test_breadcrumb_first_crumb_links_to_item_one_not_course_home`.
Its assertion is a bare `item_one_url in html` substring check over the whole response
body, **not** scoped to the breadcrumb `<nav>`. Because the item under test is the last
item, `course_topic.html`'s "Home"/"Finish Course" button also links to `view_course_item`
index 1, so this test will keep **passing** after Unit D even though its intent is now
false — a green suite gives no signal to find it. Either **delete** it (superseded by the
new §6.3 breadcrumb test) or **rewrite** it to scope its assertion to the breadcrumb `<nav
aria-label="Breadcrumb">` block the same way §6.3 does, now expecting `course_detail`.
