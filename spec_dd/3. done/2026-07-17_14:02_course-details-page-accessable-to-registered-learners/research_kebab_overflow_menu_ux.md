# Research: Kebab / Overflow Menu UX for "Course Details" Link on Cards

## Summary / Key Recommendation

The proposed design diverges from convention on two counts, and both divergences should be reconsidered. First, **overflow menus are meant for hiding two or more secondary actions** — every major design system (Material, Carbon, GOV.UK's underlying heuristics) treats them as a compression mechanism for a *list* of options; when there's only one action, authorities consistently say to show that action directly (as a visible icon/link) rather than hiding it behind a "..." tap. Second, **bottom-left is not a recognised placement** for card overflow controls — top-right (occasionally top/bottom-right) is the near-universal convention, and NN/g's research shows even *conventionally placed* overflow content is discovered by only ~27% of users vs ~48% for visible controls, so an unconventional position compounds an already weak discoverability pattern. Recommendation: replace the single-item kebab with a small, always-visible, labelled icon button (e.g. an info/share icon with visible or icon+tooltip "Details" or "Share" text) placed in a corner of the card (top-right is the safer, more conventional pick even though it's a bit tighter against the stretched-link card body), sized ≥44×44px, sitting above the stretched-link z-index, with a proper accessible name. Keep the "kebab with a Details item" pattern in reserve only for the day there are ≥2 secondary actions on the card — at that point the existing groundwork (stacking, ARIA menu-button pattern, focus handling) documented below will already apply.

## 1. Overflow menu for a single action

- NN/g and general UI-pattern guidance treat the "meatball"/kebab icon as a container for **multiple** contextual actions, not a wrapper for one: "Reserve meatball and kebab menu icons for revealing additional actions and options" — implying there are, in fact, multiple options to reveal ([NN/g, Menu-Design Checklist](https://www.nngroup.com/articles/menu-design/)).
- Overflow/hidden navigation is measurably worse for discoverability than visible controls: in NN/g's research, hidden navigation (including overflow menus) was used by only 27% of participants vs 48% for the equivalent visible navigation ([NN/g, Hamburger Menus and Hidden Navigation Hurt UX Metrics](https://www.nngroup.com/articles/hamburger-menus/)). A single item hidden this way pays the full discoverability tax for the "convenience" of hiding just one link.
- Material Design 3 frames overflow menus explicitly as an overflow-*from-a-set*: least-used actions from a row of actions spill into the "more" menu as space runs out; it is not a general-purpose action-hiding widget ([Material Design 3, App bars guidelines](https://m3.material.io/components/app-bars/guidelines); [m2 App bars: top](https://m2.material.io/components/app-bars-top/)).
- IBM Carbon's overflow-menu component guidance is oriented the same way — used "when additional options are available to the user but there is a space constraint," and recommends keeping the list scannable (up to ~12 items), again presuming a genuine list ([Carbon Design System, Overflow menu](https://carbondesignsystem.com/components/overflow-menu/usage/)).
- Apple's HIG frames the parallel "context menu" as a way to "provide access to functionality that's directly related to an onscreen item, without cluttering the interface" and explicitly warns against using such menus for rarely-needed items — the implication is that a menu earns its existence when it declutters multiple actions, not when it's substituting for a single, already-simple action ([Apple HIG, Context menus](https://developer.apple.com/design/human-interface-guidelines/menus-and-actions)).
- **Divergence flag**: a kebab with exactly one item ("Details") has no compression benefit and only downside (extra click, lower discoverability, extra ARIA/keyboard work). **Recommendation**: show "Details" (or "Share"/"Copy link" — see Q6) as a direct, always-visible icon button. Only convert to a kebab once there are genuinely ≥2 secondary actions to hide.

## 2. Placement conventions

- The conventional position for a card-level overflow control is the **top-right corner**: "An overflow menu is typically placed in the upper-right corner of a card, but can be placed in the lower-right corner if doing so improves content layout and legibility... consistent positioning of menu icons across cards contributes to a cleaner and more predictable interface" (general card/dashboard UI pattern guidance, echoing Material's card and PatternFly's card guidance) ([PatternFly, Card design guidelines](https://www.patternfly.org/components/card/design-guidelines/); [Material Design, Cards](https://m1.material.io/components/cards.html)).
- Note that documented alternatives are top-right or **bottom-right** — never bottom-left. Left-side placement for a "more options" affordance is not attested in any of the design systems reviewed (Material, Carbon, PatternFly, Apple HIG). Bottom-left is also where users' eyes/thumbs least expect a "more" control, since reading order (LTR locales) and typical F-pattern scanning put attention top-left/top-right first, and the de-facto "more actions" corner is consistently on the right (mirrored for RTL) ([Material Design 3, App bars](https://m3.material.io/components/app-bars/guidelines) — "for RTL languages, placement positions should be flipped," implying right-alignment is the anchor concept, not a specific side in isolation).
- **Divergence flag**: bottom-left is unconventional. **Recommendation**: use top-right (matching the course title/thumbnail composition) or, if the top-right is visually busy (e.g. occupied by a progress badge), bottom-right — not bottom-left. Whichever corner is chosen, use the *same* corner on every card so the position becomes a learned habit for users ("consistent positioning... contributes to a cleaner and more predictable interface").

## 3. Interaction with a stretched-link card

- The core technical/accessibility problem: **an interactive element must not contain another interactive/focusable descendant** — this rules out literally nesting a `<button>` inside the stretched `<a>`. Screen reader/keyboard users encounter overlapping hit areas and unpredictable tab order if this rule is violated (per accessible-cards guidance summarized from [Kitty Giraudel, Accessible Cards](https://kittygiraudel.com/2022/04/02/accessible-cards/) and [Inclusive Components, Cards](https://inclusive-components.design/cards/)).
- The standard fix is the **stretched-link pattern**: the card's primary link gets `position: relative` on the container and a `::after` pseudo-element that is `position: absolute; inset: 0` to make the whole card clickable, while the card's *other* interactive controls (the kebab button, etc.) are pulled to a higher stacking order with `position: relative; z-index: 2` (or similar) so they remain independently clickable/tappable above the stretched overlay (Bootstrap's `.stretched-link` utility documents this exact mechanism: [Bootstrap, Stretched link](https://getbootstrap.com/docs/4.3/utilities/stretched-link/); summarized also in [Medium, Nested Links in Cards](https://medium.com/@pratva.bsta/nested-links-in-cards-a-solution-for-clickability-issues-98b063cb347c)).
- Adrian Roselli — one of the most-cited authorities on clickable-card patterns — recommends explicitly **carving out dead space** around any secondary control rather than just relying on z-index stacking, e.g. shrinking the stretched-link's clickable area with `bottom: 3.75em` (or similar) so the area around the button/menu isn't also fighting for the same click, and removing the overlay `content: none` for any genuinely nested links. His guidance: "leave space for [additional] controls" and use dead space above/around the control "to help prevent misclicks," while cautioning "context matters, test with your users" ([Adrian Roselli, Block Links, Cards, Clickable Regions, Rows, Etc.](https://adrianroselli.com/2020/02/block-links-cards-clickable-regions-etc.html)).
- A related, more conservative school of thought (dev.to's "Clickable Card Patterns and Anti-Patterns") argues that instead of stretching the *whole card*, it's often better to link only the primary distinguishing text (e.g. the course title) and enlarge its hit area via padding — this sidesteps the whole nested-interactive-element problem, and coexists more naturally with a secondary icon button that has its own generous tap target ([DEV Community, Clickable Card Patterns and Anti-Patterns](https://dev.to/micmath/clickable-card-patterns-and-anti-patterns-2hl2)).
- **Recommendation for this feature**: keep the stretched-link card, but (a) exclude the corner containing the "Details"/kebab button from the stretched-link hit area (via reduced inset or Roselli's dead-space technique), and (b) give the button `z-index` above the stretched overlay, `position: relative`, and its own real DOM button element that is a *sibling* of the stretched link, not a descendant.

## 4. Accessibility (keyboard, ARIA, touch targets)

- If a menu is used (once there are ≥2 items), follow the **W3C WAI-ARIA APG Menu Button pattern**: the trigger is a real `<button>` with `aria-haspopup="menu"` and `aria-expanded` toggled `true`/`false` as the menu opens/closes; Enter, Space, or Down Arrow on the button opens the menu and moves focus to the first item; **Escape** (and optionally Shift+Tab) closes the menu and returns focus to the trigger button; while open, arrow keys roam focus between menu items (roving tabindex), so only one item is ever in the natural Tab order at a time ([W3C WAI-ARIA APG, Menu Button Pattern](https://www.w3.org/WAI/ARIA/apg/patterns/menu-button/)).
- **Icon-only buttons need an accessible name.** Best practice is either `aria-label` on the button (concise, good for simple one-word labels like "Details" or "Course options") or visually-hidden text inside the button if you also want the label available for translation tooling; a bare icon with no text alternative fails for screen reader users ([summarized from icon-accessibility guidance surveyed via web search, incl. Iconly's Icon Accessibility overview](https://iconly.ai/blog/icon-accessibility/); general pattern also endorsed by the WAI-ARIA APG examples above).
- **Touch target size**: WCAG 2.2 sets a **minimum of 24×24 CSS px** (Level AA, with a spacing exception for smaller targets), but the safer, widely-cited targets are **44×44pt** (Apple HIG) and **48×48dp** (Material Design) — i.e. build the button to at least 44×44px regardless of the icon's visual size, using padding to enlarge the invisible hit area ([W3C, Understanding SC 2.5.8 Target Size (Minimum)](https://www.w3.org/WAI/WCAG21/Understanding/target-size.html); cross-referenced touch-target guidance summarized from [Smashing Magazine, Accessible Target Sizes Cheatsheet](https://www.smashingmagazine.com/2023/04/accessible-tap-target-sizes-rage-taps-clicks/)).
- If we ship a **single direct "Details" icon/link** instead of a menu (per the Q1/Q2 recommendation), most of this ARIA-menu complexity disappears — it becomes a plain link/button with an accessible name and adequate target size, no `aria-haspopup`/roving-focus/Escape handling required. This is a meaningful implementation-simplicity argument in favor of not using a kebab for one item.

## 5. Consistency across surfaces (card grid vs. list row vs. breadcrumb)

- The general design-systems principle (GOV.UK Design System's component/pattern model, and Nielsen's heuristic #4 "Consistency and Standards") is that **the same action should behave predictably**, but it does not require pixel-identical widgets across contexts — it requires that the *label, meaning, and outcome* stay constant: "Users should not have to wonder whether different words, situations, or actions mean the same thing... consistency... ensures that actions and functionalities are located in predictable places and behave consistently at all times" ([NN/g, Consistency and Standards](https://www.nngroup.com/articles/consistency-and-standards/)). GOV.UK's design system similarly treats "components" as reusable building blocks that get adapted per "pattern"/page type, explicitly allowing different components to serve one user task across different page types as long as the underlying pattern and terminology are consistent ([GOV.UK Design System](https://design-system.service.gov.uk/); [GOV.UK Design System, Patterns](https://design-system.service.gov.uk/patterns/)).
- Applied here: it's fine — and arguably better — for a **card** (space-constrained, visual-first) to use a compact icon affordance, a **list row** (more horizontal room, already has inline metadata/actions) to show "Details" as a plain inline text link, and a **breadcrumb** to show it as a normal breadcrumb crumb/link — *provided* the label ("Details") and destination are identical across all three, and the icon (if used) is the same icon everywhere. What must **not** vary is the meaning of the affordance or its label wording (don't say "Details" on the card and "More info" on the list row).
- Practical implication for this feature: keep "Details" as the fixed label/name across all three surfaces; let the *presentation* differ (icon button on card, inline link on row, breadcrumb crumb on player) without treating that as an inconsistency.

## 6. "Share" framing — is "Details" the right label?

- The underlying job here is partly "let me get a link to this course to send to someone," which is a different mental model from "let me read more about this course myself." The literature on link-sharing UX suggests that for **serious/targeted content** (which a course link is), many users simply want to **copy the URL** rather than trigger a native/social share sheet: "For targeted, serious, and commercial content, a large proportion of users preferred to share by copying and pasting the URL" ([CSS-Tricks, UX Considerations for Web Sharing](https://css-tricks.com/ux-considerations-for-web-sharing/)).
- The same source notes desktop browsers (other than Safari) have no built-in share functionality, so a **"Copy link" affordance is a concretely useful, low-effort addition** on desktop-heavy LMS contexts, whereas a generic "Share" button mostly acts as a discoverability nudge rather than doing anything users can't already do via the browser/URL bar ("a custom share button can still provide a cue... to elicit the share") ([CSS-Tricks, UX Considerations for Web Sharing](https://css-tricks.com/ux-considerations-for-web-sharing/)).
- Recommendation, kept intentionally lightweight (do not over-engineer): **"Details" is a fine label** for the primary destination-navigation use case (a registered learner clicking through to read the overview page), and is the correct label if the immediate action is "go to the details page." If sharing is a first-class goal, consider adding a distinct, secondary **"Copy link"** action (e.g., a small copy icon next to/within the same affordance) rather than renaming "Details" to "Share" — the destination page *is* the shareable page, so "Details" (navigate) and "Copy link" (get the URL without navigating) can coexist as two clearly-labelled, low-cost actions if there's real demand for pre-navigation sharing. Do not build a full native Web Share API integration or social-network buttons unless there's a specific request for it — that would be scope creep beyond a single small card affordance.
