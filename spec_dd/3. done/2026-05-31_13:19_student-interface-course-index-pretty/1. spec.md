# Spec: Student dashboard (course index) visual refresh

## Summary

A **visual / layout refresh only** of the student dashboard (course index landing
page) — the page rendered by `student_interface.views.dashboard`
(`student_interface:dashboard`, template `student_interface/dashboard.html`,
sections in `partials/course_list.html`).

We are making the current implementation look and feel closer to the Claude Design
prototype (`Learner Experience v3.html`, "dashboard" screen) **without adding new
functionality or faked data**. The only genuinely new piece of content is an
"Available courses" preview section, built entirely from data we already have.

**Out of scope (explicitly):** continue-learning hero, stats/streak panel,
card-level restyling, faked placeholder content, list-vs-card rework, all-courses
page restyle. Where the prototype implies heavier infrastructure, it is excluded.

## Why this matters

The current dashboard centres its greeting and wraps every section in a boxed
`.surface` container. Per the supporting research
(`research-dashboard-ux-best-practices.md`, `research-dashboard-ux-patterns.md`):

- **Borderless, whitespace-grouped sections** are the modern, research-backed
  default. Boxing whole sections applies Gestalt "Common Region" — a strong signal
  — everywhere, which adds visual noise without communicating a real relationship,
  and creates false "floors" that discourage scrolling. Visual weight belongs on
  the interactive **cards**, not on section containers.
- **Left-aligned headings** match F-pattern / layer-cake scanning: users run their
  eyes down the left margin, and a consistent left edge lets them scan stacked
  sections in a straight line. Centred headings break that line.
- A clear **typographic hierarchy** (H1 greeting → section heading → card title)
  lets users instantly tell page title from section from item.
- A small, muted **date eyebrow** is calm orientation chrome — kept subordinate so
  the greeting stays primary.
- The **"show a few + Browse all"** pattern (Netflix / Spotify / LinkedIn) lets
  users learn more exists *before* they scan, and keeps the dashboard scannable.

## Scope of changes

### 1. Greeting / heading area

- **Left-align** the "Welcome back, {first name}" greeting (currently centred via
  `text-center` on the wrapper `div` in `dashboard.html`).
- It **remains the page's single H1** and the visually largest/boldest text on the
  page.
- Add a small **date eyebrow** line *above* the greeting.
  - Format: weekday + day + month + year, e.g. **`Friday, 30 May 2026`** (en-GB)
    or **`Friday, May 30, 2026`** (en-US). The exact word order is **determined by
    the browser's locale** (see below), not pinned — the example is illustrative.
    The intent is an unambiguous, spelled-out date (long weekday + long month +
    full year), never a numeric `30/05/2026`-style string.
  - **Render the date client-side, in the browser's own timezone and locale**, not
    on the server. Rationale:
    - **Timezone:** the server's local date can differ from the learner's. The
      browser knows the user's actual timezone; a server-rendered date can read as
      "yesterday"/"tomorrow" for them.
    - **Staleness:** a server-rendered date is baked in at page load. If the page
      is left open overnight it silently goes stale. Client-side JS can keep it
      correct.
  - **Implementation (use the `fls:alpine-js` skill — Alpine is the project's
    client-side tool):** a small Alpine component formats `new Date()` with
    `toLocaleDateString(undefined, { weekday: 'long', day: 'numeric', month:
    'long', year: 'numeric' })` (`undefined` locale = the browser's locale; no
    timezone arg = the browser's timezone) and writes it into the eyebrow. To
    avoid overnight staleness, recompute when the tab regains focus / visibility
    (`visibilitychange`); a per-minute tick is unnecessary.
  - **No-JS / pre-hydration fallback:** the eyebrow is calm orientation chrome,
    not load-bearing, so it may simply be empty until Alpine hydrates (use
    `x-cloak` to avoid a flash). Do **not** server-render a fallback date — a
    wrong-timezone date is worse than no date, and the element carries no
    essential information. The view does **not** need to pass `today`.
  - It must be **visually subordinate** to the greeting: smaller, lighter/muted
    (e.g. an eyebrow/`text-muted`-style token), not bold, not competing with the
    name. It is orientation chrome, not load-bearing.

### 2. Section styling (Your Courses / Recommended / Available / History)

- **Remove the boxed `.surface` containers / borders** around each section in
  `partials/course_list.html` (the `Your Courses`, `Recommended Courses`,
  `Learning History`, and new `Available courses` sections). Group sections with
  **whitespace + typographic hierarchy** instead.
- Section headings become **smaller and left-aligned**, with a clear size/weight
  **step down from the H1 greeting** (H1 greeting → section heading → card title).
  Headings front-load the meaningful keyword (existing labels already do this:
  "Your Courses", "Recommended Courses", "Available courses", "Learning History").
- **Semantics:** section headings stay as `<h2>` (the greeting is the single
  `<h1>`); the prototype's `h3` is a styling artefact, not a semantic instruction.
  Match the prototype's *visual* size, keep the correct heading level for
  accessibility.
- Course **cards themselves are not restyled** — the card shell
  (`cotton/course-card-shell.html`) keeps its current surface/shadow/hover
  treatment. Visual weight stays on the cards.

### 3. Available courses section (new)

- Add a **new "Available courses" section** to the dashboard. The existing
  "Recommended Courses" section stays as-is alongside it.
- **Page order:** Your Courses → Recommended Courses → Available courses →
  Learning History.
- **Content:** a capped preview of **max 3 courses** — the first 3 available
  courses the student is **not** registered for, using the **existing default
  ordering** from the all-courses page (`get_all_courses()`, i.e. the same order
  `student_interface:courses` shows). No new ordering is introduced.
- **Exclusions (applied before the cap):**
  - Exclude **every** course the student is registered for — **in-progress *and*
    completed** — not just in-progress. Use the full registration set
    (`get_course_registrations(user)` ids), **not** the in-progress-only
    `registered_ids` the all-courses view derives from `get_current_courses()`.
    Rationale: a completed course already appears under **Learning History** on
    this same page; including it here would both duplicate it and invite the
    learner to "register" for a course they have already finished.
  - Exclude courses already shown in the **Recommended Courses** section, so the
    same course never appears in both. Exclusion happens **before** the max-3 cap
    (i.e. the first 3 *non-recommended, unregistered* courses in default order).
- **Cards:** reuse the **existing unregistered "preview" card** treatment used by
  Recommended (desktop opens the preview modal; mobile links to the course preview
  page) via `_annotate_preview_context(course, is_registered=False)`. No new card
  variant.
- **"Browse all courses" affordance:** add a **right-aligned link in the section
  header**, next to the "Available courses" heading (Netflix/Spotify/LinkedIn
  convention).
  - It links to the existing all-courses page (`student_interface:courses`).
  - It must be a **real, keyboard-focusable `<a>`** with **self-describing text**
    for screen-reader users — visible text **"Browse all courses"** (a trailing
    arrow is decorative only).
- **Remove the bottom "All Courses" button** (`<c-button-group variant="centered">`
  / `<c-button>All Courses</c-button>` at the foot of `course_list.html`) — the
  header-right "Browse all courses" link replaces it.
  - **Keep** the existing empty-state "Browse courses" button shown in the "Your
    Courses" section when the student has no registered courses.

## Confirmed behaviour / edge cases

- **Fewer than 3** unregistered, non-recommended available courses → show whatever
  is available (1–2 cards).
- **None** available (after exclusions) → the **whole Available courses section
  hides gracefully** — heading *and* "Browse all courses" link both hidden. No
  empty box, no orphan link.
- The Available courses section is **independent** of whether the student has
  registered courses: it can appear even when "Your Courses" is showing its
  empty state (it aids discovery), and hides only when there are no qualifying
  courses.
- **No new functionality, models, migrations, or faked content.** All data comes
  from existing helpers (`get_all_courses`, `get_current_courses`,
  `get_recommended_courses`) and existing annotation helpers.

## Theming

- The prototype was drawn for the **first_class** theme. Implement each change in
  the **default** theme using **standard role tokens** (`--color-primary`,
  `text-on-surface`, `bg-surface`, status tokens, `--fls-radius-*`, eyebrow/muted
  tokens, etc.), then override in **first_class** only where the brand look needs
  it.
- **Do not hardcode hex values** — brand colours/shapes flow from theme tokens.
- Consult the **`brand-guidelines`** and **`fls:frontend-styling`** skills for
  tokens, the eyebrow/heading scale, and Tailwind conventions; use the
  **`fls:template`** skill for cotton/partial conventions.

## Implementation pointers (non-binding)

- **View** (`student_interface/views.py::dashboard`): compute `available_courses`
  from `get_all_courses()`, excluding **all registered course ids** (the full
  `get_course_registrations(user)` set — in-progress *and* completed, **not** the
  in-progress-only `registered_ids` used by `all_courses`) and the recommended
  collection ids, capped at 3, each annotated via
  `_annotate_preview_context(..., is_registered=False)`. Reuse existing helpers.
  Note the two views use *different* registration sets by design (`all_courses`
  treats completed courses as unregistered previews; the dashboard excludes them
  because it already lists them under Learning History), so the registered-id
  derivations are not identical and should not be force-merged into one helper.
- **Templates:** `dashboard.html` (greeting + client-side Alpine date eyebrow,
  left-align) and
  `partials/course_list.html` (remove `.surface` boxes, restyle headings, add
  Available section with header-right link, remove bottom All Courses button).
- Use `select_related`/`prefetch_related` as the existing helpers already do; no
  new N+1 queries.

## Success criteria

1. Greeting is left-aligned, remains the single visually-dominant `<h1>`, with a
   muted, spelled-out date eyebrow (e.g. `Friday, 30 May 2026`; exact word order
   follows the browser locale) above it, rendered client-side in the browser's
   timezone/locale and kept fresh on tab re-focus (empty until JS hydrates; no
   server-rendered fallback date).
2. No section is wrapped in a boxed `.surface`/bordered container; sections are
   grouped by whitespace with left-aligned headings showing a clear H1 → section →
   card-title size/weight step. Section headings remain `<h2>`.
3. A new "Available courses" section shows ≤3 unregistered courses — excluding
   every course the student is registered for (in-progress *and* completed) and
   any shown under Recommended — in the all-courses default order, using the
   existing preview-card behaviour. A completed course never appears in both
   Available courses and Learning History.
4. The section header has a right-aligned, keyboard-focusable `<a>` reading
   "Browse all courses" linking to `student_interface:courses`; the old bottom
   "All Courses" button is gone; the empty-state "Browse courses" button remains.
5. When no qualifying available courses exist, the entire Available courses
   section (heading + link) is hidden.
6. Styling uses theme role tokens (no hardcoded hex); default theme implemented
   first, first_class overridden only where needed.
7. No new models/migrations/functionality; no faked or placeholder content; cards
   are not restyled; existing tests still pass.
