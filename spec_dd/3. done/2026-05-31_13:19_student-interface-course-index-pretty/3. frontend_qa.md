# Frontend QA: Student dashboard visual refresh

Manual / Playwright browser checks for the dashboard refresh. The dashboard is
the student course-index landing page (`student_interface:dashboard`).

## Setup

1. Pick a random free port and start the dev server:
   ```
   PORT=$(.claude/fls/scripts/find_available_port.sh)
   uv run python manage.py runserver $PORT
   ```
   Base URL: `http://127.0.0.1:$PORT`
2. Build Tailwind if styling looks stale: `npm run tailwind_build`.
3. **Test data:** use the **DemoDev** site. You need a student account on DemoDev
   and a catalogue with enough courses to exercise the scenarios below
   (≥4 courses that the student is *not* registered for and *not* recommended, so
   the Available section can show its 3-card cap with extras left over). If the
   data isn't present, create it via the `qa-data-helper` agent:
   - A student user (note the login email/password).
   - Several courses (≥6 total recommended).
   - For the test student: register 1–2 courses (in progress), complete 1
     registered course, and add 1 recommended course. Leave ≥4 courses
     untouched (neither registered nor recommended).

## Log in

1. Visit `http://127.0.0.1:$PORT/` (or the login page) and sign in as the test
   student on the **DemoDev** site.
2. Land on the dashboard (`/` for students, or follow the dashboard link).

---

## Test 1 — Greeting is left-aligned + date eyebrow

1. On the dashboard, look at the top heading area.
2. **Expect:**
   - "Welcome back, <first name>" is **left-aligned** (not centred), and is the
     largest/boldest text on the page (single H1).
   - **Above** the greeting sits a small, muted **date eyebrow** showing a
     spelled-out date — long weekday + day + long month + full year, e.g.
     `Friday, 30 May 2026` (word order follows your browser locale). It must
     **not** be a numeric `30/05/2026` style and must be visually subordinate to
     the greeting (smaller, lighter, not bold).
3. **Locale check:** the date's word order must follow the browser locale and be
   spelled out — confirm a long weekday and long month (e.g. `Friday` / `May`),
   never a numeric format. (Optional deeper check: switch the browser/OS locale
   between en-GB and en-US and reload — the word order should change accordingly,
   e.g. `Friday, 30 May 2026` vs `Friday, May 30, 2026`.)
4. **Timezone check:** the date must match *your browser's* local date, not the
   server's. (Optional: set the browser timezone — e.g. via DevTools sensors — to
   one a calendar day ahead/behind the server and confirm the eyebrow shows the
   browser's local date.)
5. **Freshness check:** switch to another tab, wait, then return to the dashboard
   tab — the eyebrow should still show the correct current date (it recomputes on
   tab re-focus).
6. **No-flash check:** on a hard reload, the eyebrow should not flash a wrong/
   placeholder date — it stays empty until it populates.

---

## Test 2 — Sections are borderless, grouped by whitespace, left-aligned

1. Scroll the dashboard.
2. **Expect:**
   - No section ("Your Courses", "Recommended Courses", "Available courses",
     "Learning History") is wrapped in a bordered/boxed card container — sections
     are separated by whitespace only.
   - Section headings are **left-aligned** and clearly **smaller than the H1
     greeting** but larger than the course-card titles (H1 → section → card-title
     hierarchy is visible).
   - The course **cards themselves** keep their surface/shadow/hover look
     (cards are not flattened).

---

## Test 3 — "Available courses" section content & order

1. Find the **"Available courses"** section.
2. **Expect:**
   - It appears **after** "Recommended Courses" and **before** "Learning
     History". Full order: Your Courses → Recommended Courses → Available courses
     → Learning History.
   - It shows **at most 3** course cards.
   - The cards are courses the student is **not** registered for and that are
     **not** already in "Recommended Courses" (cross-check: no course appears in
     both Recommended and Available).
   - A course shown under **Learning History** (completed) must **not** also
     appear under Available courses.
   - **Ordering:** the cards are the *first* eligible courses in the all-courses
     default order. Open the all-courses page (the "Browse all courses" link),
     note the order courses are listed there, then confirm the Available section
     shows those same courses in that same relative order (skipping any that are
     registered or recommended) — no re-sorting is introduced.
   - Cards behave like the Recommended preview cards: on **desktop** clicking a
     card opens the **preview modal**; on **mobile** (narrow the window < md /
     use device emulation) it navigates to the course preview page.

---

## Test 4 — "Browse all courses" link

1. In the **"Available courses"** section header, look to the right of the
   heading.
2. **Expect:**
   - A link reading **"Browse all courses"** (a trailing arrow is decorative).
   - It is keyboard-focusable: press `Tab` to it and `Enter` — it navigates to
     the all-courses page (`/.../courses`).
   - Clicking it also navigates to the all-courses page.
3. **Expect the OLD bottom button is gone:** there is no centred "All Courses"
   button at the very bottom of the dashboard anymore.

---

## Test 5 — Empty Available section hides gracefully

1. Use a student who is registered for / has recommended **every** available
   course (so nothing qualifies for Available). Create via `qa-data-helper` if
   needed.
2. Reload the dashboard.
3. **Expect:** the **entire** "Available courses" section is gone — no heading,
   no "Browse all courses" link, no empty box.

---

## Test 6 — Empty-state "Browse courses" button still present

1. Log in as a student with **no registered courses**.
2. View the dashboard.
3. **Expect:** the "Your Courses" area shows the empty-state message and a
   **"Browse courses"** button (this button is retained). The "Available courses"
   section may still appear (it aids discovery and is independent of whether the
   student has registered courses).

---

## Test 7 — Both themes

1. Repeat a quick pass of Tests 1–4 under the **default** theme and the
   **first_class** theme.
2. **Expect:** the layout/hierarchy reads correctly in both; brand colours/shapes
   come from theme tokens (nothing looks like an un-themed hardcoded colour).

---

## Regression

- No JavaScript console errors on the dashboard (check the browser console).
- Existing dashboard behaviour (registered course progress bars + "Next up",
  recommended preview cards, learning-history cards) still works.
