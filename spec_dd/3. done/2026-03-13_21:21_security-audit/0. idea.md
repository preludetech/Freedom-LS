# Security Audit & Hardening

Harden the FLS codebase against OWASP Top 10:2025, ISO 27001 Annex A technical controls, and common Django security misconfigurations. This spec focuses on securing existing code and adding automated security tooling.

Privacy/compliance features (data export, consent, deletion) are in a separate spec: `spec_dd/1. next/00. privacy-compliance/`.
CSP enforcement is in a separate spec: `spec_dd/1. next/CSP-rollout/`.

## Research

Look over the research documents in this directory:
- `research.md` - OWASP Top 10:2025 through a Django lens, AI-generated code risks, tool recommendations, working configs
- `research_iso27001.md` - ISO 27001 Annex A controls relevant to application code
- `research_current_codebase.md` - Current security posture and identified gaps
- `research_security_tools.md` - Extended tool research with ratings

## In scope

### 1. Production settings hardening
- HTTPS enforcement: `SECURE_SSL_REDIRECT`, `SECURE_HSTS_SECONDS`, `SECURE_HSTS_INCLUDE_SUBDOMAINS`, `SECURE_HSTS_PRELOAD`
- Secure cookies: `SESSION_COOKIE_SECURE`, `CSRF_COOKIE_SECURE`, `SESSION_COOKIE_HTTPONLY`, `CSRF_COOKIE_HTTPONLY`, `SESSION_COOKIE_SAMESITE`
- Security headers: `SECURE_CONTENT_TYPE_NOSNIFF`, `SECURE_REFERRER_POLICY`, `SECURE_CROSS_ORIGIN_OPENER_POLICY`
- Session timeout: `SESSION_COOKIE_AGE`
- Upload limits: `DATA_UPLOAD_MAX_MEMORY_SIZE`, `FILE_UPLOAD_MAX_MEMORY_SIZE`
- These should only apply to production. Dev settings must remain usable

### 2. Password & authentication hardening
- Switch to Argon2 as primary password hasher (install `argon2-cffi`)
- Install `django-axes` for brute-force protection on login
- Rate limiting on password reset and registration endpoints

### 3. Per-app security review
- Review every app for: missing `@login_required`, IDOR vulnerabilities, raw SQL, `mark_safe()` on user content, `@csrf_exempt`, mass assignment
- Fix any issues found
- Don't worry about educator interface access permissions (separate spec exists)

### 4. Security automation - CI pipeline
- GitHub Actions workflow: Bandit (SARIF), pip-audit, Semgrep (`p/django`), Gitleaks, `manage.py check --deploy`
- Dependabot configuration for pip and GitHub Actions
- Enable GitHub secret scanning and Dependabot alerts
- All free/open-source tools

### 5. Security automation - pre-commit hooks
- Bandit in pre-commit
- detect-secrets with baseline
- Gitleaks
- Ruff security rules (`S` prefix)
- shellcheck for `.claude/hooks/` scripts

### 6. Security automation - Claude Code
- PreToolUse security guard hook (blocks dangerous patterns like `raw()`, `mark_safe()`, `@csrf_exempt`, `eval()`, migration tampering)
- PostToolUse Bandit scan on modified Python files
- Extended deny rules for secrets and production config
- `/security-review` slash command
- `/threat-model` slash command

### 7. Infrastructure deployment checklist
- Static markdown document covering: server hardening, database security, TLS configuration, firewall rules, backup encryption, log management, monitoring
- Reference `manage.py check --deploy` output
- Not automated - just a checklist for deployers

### 8. Admin URL obfuscation
- Change default `/admin/` path to something unpredictable

## Out of scope
- Educator interface access permissions (not implemented yet, separate spec)
- Privacy/compliance features: data export, consent management, account deletion, cookie consent (separate spec: `00. privacy-compliance`)
- Webhook PII control (privacy-compliance spec)
- CSP enforcement (separate spec: `CSP-rollout`)
- DAST scanning setup (separate spec: `DAST-scanning`)
- Encryption at rest for individual database fields (separate spec: `encryption-at-rest`)
- CORS configuration (separate spec: `CORS-configuration`)
- Multi-factor authentication (separate spec: `MFA`)

## Preferences
- Strongly prefer free and open-source tools
- LLM-based tools (Claude Code hooks, skills, commands) are encouraged
- Minimal new dependencies - prefer Django built-ins and configuration over third-party packages where possible
