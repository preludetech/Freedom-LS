# Security Audit & Hardening — Specification

## Overview

Harden the FLS codebase against OWASP Top 10:2025, ISO 27001 Annex A technical controls, and common Django security misconfigurations. This spec focuses on securing existing code and adding automated security tooling — no new user-facing features.

Privacy/compliance features are in a separate spec (`spec_dd/1. next/00. privacy-compliance/`).
CSP enforcement is in a separate spec (`spec_dd/1. next/CSP-rollout/`).

---

## 1. Production Settings Hardening

**Why:** The current production settings are missing critical HTTPS enforcement, cookie security, and security headers. Without these, traffic can be intercepted, sessions hijacked, and browsers cannot enforce security policies. `manage.py check --deploy` would flag multiple warnings. Security Misconfiguration is OWASP #2 for 2025.

### 1.1 HTTPS Enforcement

Add to `config/settings_prod.py`:

```python
SECURE_SSL_REDIRECT = True
```

**Infrastructure-dependent setting:** `SECURE_PROXY_SSL_HEADER` is NOT included by default. It is only needed when running behind a reverse proxy (e.g. Cloudflare, Nginx, AWS ALB) that terminates SSL. Without it, redirect loops occur behind such proxies.

> **Post-deployment review:** Once infrastructure has been decided, review whether `SECURE_PROXY_SSL_HEADER` is needed:
> - **Behind a reverse proxy (Cloudflare, Nginx, etc.):** Add `SECURE_PROXY_SSL_HEADER = ("HTTP_X_FORWARDED_PROTO", "https")` to production settings. Verify the proxy sets the `X-Forwarded-Proto` header.
> - **Direct SSL termination (no proxy):** Do not set `SECURE_PROXY_SSL_HEADER` — it is not needed and trusting the header without a proxy is a security risk.
> - **Test by visiting HTTP URLs** and confirming they redirect to HTTPS without looping.

### 1.2 HSTS Rollout Strategy

HSTS tells browsers to always use HTTPS. Once set, it cannot be easily undone — browsers will refuse HTTP connections for the specified duration. Therefore, roll out in stages:

**Stage 1 — Testing (1 hour):**
```python
SECURE_HSTS_SECONDS = 3600
SECURE_HSTS_INCLUDE_SUBDOMAINS = False
SECURE_HSTS_PRELOAD = False
```

**Stage 2 — Confidence (1 week):**
```python
SECURE_HSTS_SECONDS = 604800
SECURE_HSTS_INCLUDE_SUBDOMAINS = True
SECURE_HSTS_PRELOAD = False
```

**Stage 3 — Full enforcement (1 year):**
```python
SECURE_HSTS_SECONDS = 31536000
SECURE_HSTS_INCLUDE_SUBDOMAINS = True
SECURE_HSTS_PRELOAD = True
```

Implement as configurable environment variables so deployers control the rollout:
```python
SECURE_HSTS_SECONDS = int(os.environ.get("HSTS_SECONDS", "3600"))
SECURE_HSTS_INCLUDE_SUBDOMAINS = os.environ.get("HSTS_INCLUDE_SUBDOMAINS", "False") == "True"
SECURE_HSTS_PRELOAD = os.environ.get("HSTS_PRELOAD", "False") == "True"
```

### 1.3 Secure Cookies

```python
SESSION_COOKIE_SECURE = True
SESSION_COOKIE_HTTPONLY = True      # Django default, make explicit
SESSION_COOKIE_SAMESITE = "Lax"
CSRF_COOKIE_SECURE = True
CSRF_COOKIE_HTTPONLY = True
CSRF_COOKIE_SAMESITE = "Lax"
```

**Decision:** `SameSite=Lax` (not `Strict`) because `Strict` breaks top-level navigations from external links — students clicking course links from emails would lose their session.

### 1.4 Security Headers

```python
SECURE_CONTENT_TYPE_NOSNIFF = True
SECURE_REFERRER_POLICY = "strict-origin-when-cross-origin"
SECURE_CROSS_ORIGIN_OPENER_POLICY = "same-origin"
```

`X_FRAME_OPTIONS = "SAMEORIGIN"` is already set in base settings (needed for PDF display).

### 1.5 Session Timeout

```python
SESSION_COOKIE_AGE = 1209600  # 2 weeks (Django default, make explicit)
```

**Decision:** 2 weeks is acceptable for an LMS where students have long-running sessions. Re-authentication for sensitive actions is tracked in a separate spec (`spec_dd/1. next/session-timeout-strategy/`).

### 1.6 Upload Limits

```python
DATA_UPLOAD_MAX_MEMORY_SIZE = 5_242_880   # 5 MB
FILE_UPLOAD_MAX_MEMORY_SIZE = 5_242_880   # 5 MB
```

### 1.7 Dev Settings Must Remain Usable

None of the above settings should be added to `settings_base.py` or `settings_dev.py`. Production hardening applies only to `settings_prod.py`. Dev settings already correctly have `DEBUG = True` and relaxed password validators.

### Success Criteria

- `uv run python manage.py check --deploy` produces zero warnings when run against production settings
- All settings are configurable via environment variables where deployment-specific
- Dev environment remains fully functional with no TLS requirements

## 2. Password & Authentication Hardening

**Why:** FLS currently uses PBKDF2 (Django default), which is vulnerable to GPU-based cracking. There is no brute-force protection beyond allauth's rate limits. Cryptographic Failures is OWASP #4; Authentication Failures is OWASP #7.

### 2.1 Argon2 Password Hashing

Install `argon2-cffi` and configure `PASSWORD_HASHERS` in `settings_base.py`:

```python
PASSWORD_HASHERS = [
    "django.contrib.auth.hashers.Argon2PasswordHasher",
    "django.contrib.auth.hashers.PBKDF2PasswordHasher",
    "django.contrib.auth.hashers.PBKDF2SHA1PasswordHasher",
    "django.contrib.auth.hashers.BCryptSHA256PasswordHasher",
]
```

**Why Argon2:** It is memory-hard, making GPU/ASIC attacks orders of magnitude more expensive than PBKDF2. Keeping PBKDF2 as a fallback ensures existing passwords continue to work and are transparently re-hashed to Argon2 on next login.

This goes in `settings_base.py` (not prod-only) because password hashing should be consistent across environments — otherwise dev passwords won't work if a database is shared or migrated.

### 2.2 Brute-Force Protection with django-axes

Install `django-axes` and configure:

```python
# settings_base.py
INSTALLED_APPS += ["axes"]

AUTHENTICATION_BACKENDS = [
    "axes.backends.AxesStandaloneBackend",  # Must be first
    "django.contrib.auth.backends.ModelBackend",
    "guardian.backends.ObjectPermissionBackend",
    "allauth.account.auth_backends.AuthenticationBackend",
]

AXES_FAILURE_LIMIT = 5
AXES_COOLOFF_TIME = 1  # 1 hour cooloff after lockout
AXES_LOCKOUT_PARAMETERS = ["ip_address", "username"]
AXES_RESET_ON_SUCCESS = True
```

Add `"axes.middleware.AxesMiddleware"` at the end of `MIDDLEWARE`.

**Why django-axes over allauth rate limits alone:** allauth's `login_failed` rate limit (`10/m/ip, 5/5m/key`) provides basic throttling, but django-axes provides persistent lockout tracking, admin visibility into blocked IPs, and is the standard Django solution for brute-force protection recommended by OWASP. They complement each other — allauth handles the rate limiting, axes handles the lockout and audit trail.

### 2.3 Allauth Rate Limits Verification

Verify that production settings do NOT set `ACCOUNT_RATE_LIMITS = False`. The dev settings correctly disable rate limits for development convenience, but production must use the defaults:

- `login`: 30/m/ip
- `login_failed`: 10/m/ip, 5/5m/key
- `signup`: 20/m/ip
- `reset_password`: 20/m/ip, 5/m/key
- `change_password`: 5/m/user
- `confirm_email`: 1/3m/key

No additional `django-ratelimit` dependency is needed.

### 2.4 Password Validator Strengthening

Update `settings_base.py` (currently configured but with default min_length):

```python
AUTH_PASSWORD_VALIDATORS = [
    {
        "NAME": "django.contrib.auth.password_validation.UserAttributeSimilarityValidator",
        "OPTIONS": {"max_similarity": 0.7},
    },
    {
        "NAME": "django.contrib.auth.password_validation.MinimumLengthValidator",
        "OPTIONS": {"min_length": 10},
    },
    {"NAME": "django.contrib.auth.password_validation.CommonPasswordValidator"},
    {"NAME": "django.contrib.auth.password_validation.NumericPasswordValidator"},
]
```

**Decision:** Minimum length 10 (not 12) balances security with usability for an LMS. The similarity validator threshold is tightened to 0.7. Dev settings will continue to override these with an empty list.

### Success Criteria

- `Argon2PasswordHasher` is the primary hasher
- Existing PBKDF2 passwords work and are re-hashed on login
- After 5 failed login attempts, the account is locked for 1 hour
- Allauth rate limits are active in production
- Password validators enforce minimum length of 10

## 3. Per-App Security Review

**Why:** AI-generated code produces insecure patterns ~45% of the time. A systematic review catches IDOR vulnerabilities, missing auth decorators, and unsafe patterns before they are exploited. Broken Access Control is OWASP #1.

### 3.1 Scope

Review every app for:
- Missing `@login_required` or `LoginRequiredMixin` on views that require authentication
- IDOR vulnerabilities — objects fetched by ID without ownership/permission checks
- `raw()`, `extra()`, or `RawSQL` usage
- `mark_safe()` on user-controlled content
- `@csrf_exempt` on state-changing views
- Mass assignment patterns

### 3.2 Known Issues to Fix

The following issues were identified during research:

1. **mark_safe() usage in markdown_utils.py** — currently safe because content is sanitized with `nh3.clean()` first. **No fix needed**, but add a comment documenting why this is safe.

### 3.3 Out of Scope

- XAPI endpoints — not yet implemented, authentication will be addressed when the feature is built
- Educator interface IDOR vulnerabilities (separate spec: `spec_dd/1. next/educator-idor-fixes/`)
- Educator interface role-based access permissions (separate spec exists)

### Success Criteria

- All views serving authenticated content have `@login_required` or equivalent
- No `raw()`, `extra()`, `RawSQL`, `@csrf_exempt`, or unsafe `mark_safe()` found, unless justified
- No mass assignment patterns found

## 4. Security Automation — CI Pipeline

**Why:** Manual security review doesn't scale. Automated scanning catches vulnerable dependencies, insecure code patterns, and leaked secrets on every PR. ISO 27001 A.8.29 requires security testing during development.

### 4.1 GitHub Actions Security Workflow

Create `.github/workflows/security.yml` with these jobs:

**Bandit SAST** — Static analysis for Python security issues, outputs SARIF for GitHub Code Scanning:
- Run `bandit -r . -x ./tests -f sarif -o bandit.sarif --severity-level medium`
- Upload SARIF to GitHub Code Scanning

**pip-audit** — Dependency vulnerability scanning against the Python Packaging Advisory Database:
- Use `pypa/gh-action-pip-audit@v1.1.0`
- No account or API key required

**Semgrep** — Framework-aware SAST with Django-specific taint tracking:
- Run `semgrep scan --config auto --sarif -o semgrep.sarif`
- Upload SARIF to GitHub Code Scanning

**Django deployment check** — Validates security settings against best practices:
- Run `manage.py check --deploy --fail-level WARNING --settings=config.settings_prod`
- Provide dummy environment variables in the CI step (e.g., `SECRET_KEY=ci-dummy`, `HOST_DOMAIN=example.com`) so the production settings module can import without errors

**Triggers:**
- On push to `main`
- On pull requests to `main`
- Weekly scheduled scan (Monday 06:00 UTC)

### 4.2 Dependabot Configuration

Create `.github/dependabot.yml`:

```yaml
version: 2
updates:
  - package-ecosystem: "pip"
    directory: "/"
    schedule:
      interval: "weekly"
    open-pull-requests-limit: 10
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
```

### 4.3 GitHub Security Features

Enable in repository settings (manual step, documented in deployment checklist):
- Dependabot alerts
- Dependabot security updates
- Secret scanning
- Secret push protection (if public repo)

### Success Criteria

- Security workflow runs on every PR and blocks merge on high/critical findings
- Dependabot creates PRs for vulnerable dependencies automatically
- SARIF results visible in GitHub Security tab
- Weekly scans catch newly disclosed vulnerabilities

## 5. Security Automation — Pre-commit Hooks

**Why:** Catching security issues before they reach CI is faster feedback. Pre-commit hooks run locally in seconds, preventing insecure code from ever being committed.

### 5.1 Additions to Existing Pre-commit Config

The codebase already has: detect-secrets (with baseline), ruff (with S rules), detect-private-key, debug-statements.

Add:

**Bandit** — Authoritative Python security scanner (Ruff reimplements the rules but Bandit is the reference):
```yaml
- repo: https://github.com/PyCQA/bandit
  rev: "1.8.6"
  hooks:
    - id: bandit
      args: ["-c", "pyproject.toml", "-ll", "-x", "./tests"]
      additional_dependencies: ["bandit[toml]"]
```

**shellcheck** — Static analysis for shell scripts (we have `.claude/hooks/` scripts):
```yaml
- repo: https://github.com/shellcheck-py/shellcheck-py
  rev: v0.10.0.1
  hooks:
    - id: shellcheck
```

Add Bandit configuration to `pyproject.toml`:
```toml
[tool.bandit]
exclude_dirs = ["tests", "*/tests/*"]
skips = []
```

### 5.2 Not Adding

- **Gitleaks** — Requires Go binary installation. detect-secrets already covers secret scanning with baseline management.
- **pygrep-hooks (python-no-eval)** — Ruff S rules already cover `eval()` detection.

### Success Criteria

- `pre-commit run --all-files` passes cleanly
- Bandit catches insecure patterns before commit
- Shell scripts in `.claude/hooks/` are validated by shellcheck
- No new dependencies beyond `bandit[toml]` and `shellcheck-py`

## 6. Security Automation — Claude Code

**Why:** Claude Code generates code throughout development. Deterministic hooks enforce security rules that the LLM cannot bypass, and slash commands standardize security review workflows. This is the highest-leverage security configuration for an AI-assisted development workflow.

### 6.1 PreToolUse Security Guard Hook

Create `.claude/hooks/security-guard.sh` that blocks dangerous pattern **introductions** (not existing patterns) in Bash, Write, and Edit tool calls:

**Blocked patterns in code edits:**
- `raw(` , `extra(`, `RawSQL` — raw SQL injection risk
- `mark_safe(` — XSS risk
- `@csrf_exempt` — CSRF bypass
- `eval(`, `exec(` — code injection
- `pickle.loads(`, `yaml.load(` — insecure deserialization
- `**request.POST.dict()`, `**request.GET.dict()` — mass assignment

**Blocked patterns in Bash:**
- `rm -rf` — destructive operations
- Access to `.env`, `id_rsa`, `*.pem`, `*.key` files

**Blocked file writes:**
- Modifications to `.env*` files (`.env.example` edits are allowed)
- Modifications to existing migration files (creating new ones is fine)

The hook reads JSON from stdin containing `tool_name` and `tool_input`, and exits with code 2 to block. It only checks the *new content* being introduced, not existing file content, to avoid false positives on files that legitimately use these patterns (e.g., `markdown_utils.py` using `mark_safe()` after sanitization).

Add a new entry to the existing `PreToolUse` array in `.claude/settings.json` (there is already a `Bash(git commit:*)` matcher — add this alongside it, do not replace it):
```json
{
  "matcher": "Bash|Write|Edit",
  "hooks": [
    {
      "type": "command",
      "command": ".claude/hooks/security-guard.sh"
    }
  ]
}
```

### 6.2 PostToolUse Bandit Scan

Create `.claude/hooks/post-edit-bandit.sh` that runs Bandit on every modified Python file after edit. Add a new entry to the existing `PostToolUse` array in `.claude/settings.json` (there is already an `Edit|Write` matcher for `ruff_fix.sh` — add this as a separate entry alongside it):

```bash
#!/bin/bash
set -euo pipefail
INPUT=$(cat)
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // .tool_input.path // ""')
if [[ "$FILE_PATH" =~ \.py$ ]]; then
    RESULTS=$(uv run bandit -q -ll "$FILE_PATH" 2>&1) || true
    if [[ -n "$RESULTS" ]]; then
        echo "Bandit found issues in $FILE_PATH:"
        echo "$RESULTS"
    fi
fi
exit 0
```

This is advisory (exit 0), not blocking — it surfaces findings for Claude to address but doesn't prevent the edit.

### 6.3 Extended Deny Rules

The existing `.claude/settings.json` already denies `Read(.env)`, `Read(.env.*)`, `Write(config/settings_prod.py)`, `Bash(git commit --no-verify:*)`, and `Bash(git push --force:*)`.

Add these new rules to the existing deny list:

```json
"Read(**/*.pem)",
"Read(**/*.key)"
```

### 6.4 `/security-review` Slash Command

Create `.claude/commands/security-review.md`:

```markdown
Perform a security review of the current changes:
1. Run `uv run bandit -r . -x ./tests` and analyze results
2. Run `uvx pip-audit` for dependency vulnerabilities
3. Run `detect-secrets scan` for hardcoded secrets
4. Review all new/modified views for auth decorators and ownership checks
5. Check all ORM queries for raw SQL or injection risks
6. Verify CSRF protection on all state-changing views
7. Run `uv run python manage.py check --deploy`
Focus areas: $ARGUMENTS
```

### 6.5 `/threat-model` Slash Command

Create `.claude/commands/threat-model.md`:

```markdown
Perform threat modeling for: $ARGUMENTS

1. Identify assets at risk (user data, credentials, sessions, etc.)
2. List threat actors (unauthenticated users, authenticated users, admins, external attackers)
3. Map attack vectors relevant to this feature against OWASP Top 10:2025
4. Define required security controls for each vector
5. Check if existing code implements those controls
6. List any gaps that need to be addressed
```

### 6.6 Integration with Spec-Driven Development Workflow

Update `.claude/commands/spec_driven_dev/README.md` to incorporate these commands into the standard workflow:

- **`/threat-model`** — Add as a step after the spec is created but before planning (between Step 2 and the current Step 3). Run `/threat-model <feature name>` to identify security concerns before any code is written.
- Update the plan command so it takes the threat model into account
- **`/security-review`** — Add as a step after QA but before making a PR. Run `/security-review` to scan for vulnerabilities before committing.

### Success Criteria

- Security guard hook blocks all listed dangerous patterns when introduced
- Security guard hook does NOT false-positive on existing legitimate uses
- Bandit runs automatically after every Python file edit
- `/security-review` and `/threat-model` commands work correctly
- Deny rules prevent reading secrets and modifying production settings
- Spec-driven dev README includes `/threat-model` and `/security-review` at the correct workflow stages

## 7. Infrastructure Deployment Checklist

**Why:** Application-level security is insufficient without server-level hardening. Deployers need a reference document covering the infrastructure concerns that code alone cannot address. ISO 27001 A.8.27 requires secure system architecture documentation.

### 7.1 Checklist Document

Create `docs/deployment-security-checklist.md` as a static markdown document covering:

1. **Server hardening** — SSH key-only access, disable root login, automatic security updates, firewall (ufw/iptables)
2. **Database security** — PostgreSQL listen_address restrictions, SSL connections (`sslmode=require`), separate database user with minimal privileges, regular backups
3. **TLS configuration** — TLS 1.2+ only, strong cipher suites, certificate renewal automation (Let's Encrypt/Cloudflare)
4. **HSTS rollout instructions** — Reference the three-stage rollout from section 1.2
5. **Firewall rules** — Allow only 80/443 inbound, restrict database port to application server
6. **Backup encryption** — Encrypted database backups, secure offsite storage, tested restore procedure
7. **Log management** — Centralised log collection, log rotation, access controls on log files
8. **Monitoring** — Uptime monitoring, error rate alerting, security event alerting
9. **Django deployment check** — Run `manage.py check --deploy` and resolve all warnings
10. **Environment variables** — Checklist of all required environment variables with descriptions
11. **GitHub security features** — Enable Dependabot alerts, secret scanning, security advisories

This is a reference document for deployers — not automated.

### Success Criteria

- Checklist covers all items listed above
- References `manage.py check --deploy` output
- Lists all required environment variables

## 8. Admin URL Obfuscation

**Why:** `/admin/` is the most predictable target for automated scanners. Changing it eliminates a massive volume of automated probes. This is security through obscurity — it won't stop a determined attacker — but it significantly reduces noise and automated attack surface.

### 8.1 Configurable Admin Path

Update `config/urls.py`:

```python
import os

ADMIN_URL = os.environ.get("DJANGO_ADMIN_URL", "admin/")

urlpatterns = [
    path(ADMIN_URL, admin.site.urls),
    ...
]
```

**Default:** `admin/` in development (no environment variable needed). Production deployments set `DJANGO_ADMIN_URL` to an unpredictable path.

Document the environment variable in the deployment checklist.

### Success Criteria

- Admin URL is configurable via `DJANGO_ADMIN_URL` environment variable
- Default remains `admin/` for development convenience
- Deployment checklist documents this setting
