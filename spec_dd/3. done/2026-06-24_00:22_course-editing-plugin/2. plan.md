# Implementation Plan: Course Editing Plugin (`fls-content`)

This plan implements the spec in `1. spec.md`. It builds a new, self-contained Claude
Code plugin (`fls-content-plugin/`) that documents FLS course-authoring conventions,
converts messy Markdown into valid FLS content, ships a Django-free validator, and
wires an SDD sync step that keeps the plugin current with FLS.

The plugin is **author-facing artifacts** (Claude Code skills/commands/agents â€” Markdown)
plus **one piece of real Python** (the bundled validator). Only the validator is unit
testable; the skills/commands/agents are authored reference material verified against the
spec's success criteria. Tests are TDD only where there is executable code (D3).

> Source-of-truth note: the literal authoring facts (per-type frontmatter, the full
> auto/propose/never conversion table, escaping quirks) live in the four `research_*.md`
> files in this directory and in the FLS source. This plan cites where each fact comes
> from; implementers must copy the **verbatim** values captured below (and re-read the
> research files) rather than reconstruct them from memory.

## 0. Ground rules (apply to every task)

- **Mirror the existing plugin exactly.** `fls-content-plugin/` copies the file shapes of
  `fls-claude-plugin/` â€” `plugin.json` schema, `SKILL.md`/command/agent frontmatter, and
  directory layout. Templates are captured per-task below.
- **New namespace `fls-content`** â€” never reuse `fls:`. Skills/agents resolve as
  `fls-content:<name>`; command is `/fls-content:format-content`.
- **Self-contained & literal** (spec Â§9.3): every authoring fact is stated in the plugin,
  never by reference to FLS source the author cannot see.
- **Right altitude** (spec Â§9.4): author-facing, task-oriented, shallow. `SKILL.md` bodies
  â‰¤ ~80 lines; exhaustive tables go to `resources/`. No pipeline/schema internals.
- **DRY:** each authoring fact lives in exactly one `resources/` file; other skills refer to
  it by namespaced prose mention (`See the \`fls-content:widget-reference\` skill`). Every
  `resources/` file opens with an HTML comment recording the FLS source file(s) it was
  distilled from, so D4 re-sync is mechanical (spec Â§6, research_claude_plugin_structure Â§6).
- **Resource references use relative paths** (`resources/<file>.md`) because each skill owns
  an embedded `resources/` subdir â€” matching `skills/claude-code-authoring/` and
  `skills/icon-usage/` (not the `${CLAUDE_PLUGIN_ROOT}/resources/...` form used for the
  developer plugin's top-level resources).
- **Never invent/edit UUIDs; never alter substantive prose** (spec Â§9.1â€“9.2). These are
  load-bearing rules repeated in the `conventions` skill, the `markdown-conversion` skill,
  and the converter agent's prompt.

**The plugin reproduces only the type *names* (the dict keys), never the
`{...}` render config (`label`/`icon`/`color`)** â€” that styling lives in each
deployment's Django settings and is none of the plugin's business. Authoring
correctness needs only the set of valid names.

**These keys are a fully overridable default, not a fixed floor.** Admonition
types are deployment-configurable: a content repo's `.fls-content.yaml`
`admonition_types` list is the **complete, authoritative** valid set for that
repo and may add to, remove from, *or* rename the base keys above. The plugin
ships this base set purely as (a) the documented default, (b) the fallback when
no `.fls-content.yaml` is found, and (c) the `init` starting template â€” it must
**never** present these keys as exhaustive or assume any of them survives in a
given deployment (spec Â§5; success criterion 10).

Notes the skills must capture:

## 5. Task D2 â€” Conversion command + agent (spec Â§7)

### D2.1 `agents/content-formatter.md`
```markdown
---
name: content-formatter
description: |-
  Converts exactly one messy Markdown or YAML source file into valid FLS content
  structure, in place. Conservative: auto-applies only lossless transforms, proposes
  everything semantic, never invents prose or UUIDs. Spawned one-per-file by
  /fls-content:format-content. Returns a structured status + any author-attention items.
tools: Read, Edit, Write, Glob, Grep, Skill
model: sonnet
---
```
Body instructs the agent to:
1. `Skill`-load `fls-content:markdown-conversion`, `widget-reference`, `content-types`,
   `conventions` for its rules.
2. Read the **resolved `.fls-content.yaml`** path passed in its prompt (the orchestrator
   discovers it; agent does not re-search) to know the active `admonition_types`; fall back
   to the base set if none was provided, and say so.
3. Convert its single file per spec Â§7 "Conversion behaviour":
   - auto-apply: local `![alt](path)` â†’ `<c-picture>`; Obsidian `![[image]]` /
     `![[image | title]]` â†’ `<c-picture>` (**expand here** â€” `content_save` drops alt text);
     YouTube watch URL â†’ `<c-youtube video_id>`.
   - propose-only (into the review return, never applied): blockquote â†’ admonition/pull-quote,
     Q&A â†’ flashcard, "optional reading" â†’ accordion, math â†’ equation, image sequence â†’
     image-grid.
   - heading handling: lift the `title` (and `subtitle`) heading out of the body; re-base
     remaining body headings so the top section is `#`; never alter heading **text**;
     normalise setext â†’ ATX first; **flag** skipped levels.
   - existing widgets: leave correct ones; normalise unambiguously-fixable ones (stripped
     attrs, `c-image-grid` closed-`c-picture` form, missing blank lines, bare `accordion open`,
     unescaped code/equation bodies); **flag** unknown-name widgets and out-of-set admonition
     types â€” never emit them as-is. Never touch prose inside a widget.
   - naming/layout: rename mis-named files/dirs to the correct `NN. name` / role-file form.
   - idempotent: check everything; fix only what is broken; never touch existing `uuid:` or
     already-valid prose; flag possible prose discrepancies.
   - frontmatter safety: quote generated strings; merge (don't clobber) pre-existing YAML.
4. **Return** (as its final text) a structured object: `status: ok|failed|blocked`, and a
   list of author-attention items (with source-line refs) for the orchestrator to collate â€”
   it does **not** write the review file itself (keeps one writer for the merged file).

### D2.2 `commands/format-content.md`

---
