# Webhook Translation Layer — Implementation Plan

## Overview

This plan adds Jinja2-based payload transformation to the existing webhook system: new fields on `WebhookEndpoint`, a `WebhookSecret` model for encrypted secrets, hardcoded presets, template rendering in the delivery pipeline, SSRF protection, and a richer Send Test admin feature.

**Skills to use:** `testing` (TDD for all steps), `admin-interface` (admin configuration), `multi-tenant` (SiteAwareModel patterns), `request-code-review` (after completing each major step).

## Step 4: Jinja2 template rendering engine

Create `freedom_ls/webhooks/rendering.py`:

```python
def get_jinja2_env() -> jinja2.SandboxedEnvironment:
    """Return a SandboxedEnvironment with StrictUndefined."""
    return SandboxedEnvironment(undefined=StrictUndefined)

def build_template_context(event: WebhookEvent, site_id: int) -> dict:
    """Build the template context with event data and resolved secrets."""
    secrets_qs = WebhookSecret.objects.filter(site_id=site_id)
    secrets_dict = {s.name: s.encrypted_value for s in secrets_qs}
    return {
        "event": {
            "id": str(event.pk),
            "type": event.event_type,
            "timestamp": event.created_at.isoformat().replace("+00:00", "Z"),
            "data": event.payload,
        },
        "secrets": secrets_dict,
    }

def render_template(template_str: str, context: dict) -> str:
    """Render a Jinja2 template string with the given context."""
    env = get_jinja2_env()
    template = env.from_string(template_str)
    return template.render(**context)
```

**TDD approach:** Write tests for:
- `build_template_context` includes event data and secrets
- `render_template` renders variables correctly
- `render_template` raises on undefined variables (StrictUndefined)
- `render_template` blocks access to Python internals (SandboxedEnvironment)
- Jinja2 `tojson` filter works in templates
- `replace` filter works (needed for Brevo preset)

**Skill:** `testing`

## Step 5: Modify delivery pipeline

Edit `freedom_ls/webhooks/delivery.py` — modify `attempt_delivery()`:

If `endpoint.has_transformation`:
1. Call `build_template_context(event, endpoint.site_id)` to get context
2. Render `body_template` using `render_template()` — if rendering raises, mark delivery as failed with error in `last_response_error_message` and return early (no HTTP request made)
3. Render `headers_template` (if set) using `render_template()`, parse as JSON dict — if fails, mark as failed
4. Use `endpoint.http_method` (or `"POST"` if null) for the HTTP method
5. Set `Content-Type` from `endpoint.content_type` (or `"application/json"` if null)
6. If `endpoint.auth_type == "signing"`: add HMAC signature headers (using the rendered body)
7. If `endpoint.auth_type == "none"`: skip HMAC headers
8. Merge rendered headers into final headers (rendered headers override defaults)
9. Send request using `httpx.request(method=..., ...)` instead of `httpx.post(...)`

If not `endpoint.has_transformation`: existing behaviour unchanged.

Extract the HTTP request portion into a helper to avoid deep nesting:

```python
def _build_transformed_request(endpoint, event) -> tuple[str, str, dict[str, str]]:
    """Returns (method, body, headers) for a transformed endpoint.
    Raises TemplateRenderError on failure."""
    ...

def _build_standard_request(endpoint, event) -> tuple[str, str, dict[str, str]]:
    """Returns (method, body, headers) for standard webhook format."""
    ...
```

**TDD approach:** Write tests for:
- Endpoint without transformation: delivery works as before (regression)
- Endpoint with transformation: uses rendered body/headers
- Template rendering failure: delivery marked failed, error stored in `last_response_error_message`
- Custom HTTP method used when set
- Custom content type set in headers
- `auth_type="signing"`: HMAC headers present even with transformation
- `auth_type="none"`: no HMAC headers
- Rendered headers override defaults
- JSON content type with invalid JSON body: delivery still proceeds

**Skill:** `testing`

## Step 6: Webhook presets

Create `freedom_ls/webhooks/presets.py`:

```python
from dataclasses import dataclass

@dataclass(frozen=True)
class WebhookPreset:
    name: str
    slug: str
    description: str
    default_url: str
    http_method: str
    content_type: str
    headers_template: str
    body_template: str

WEBHOOK_PRESETS: dict[str, WebhookPreset] = {}

def _register(preset: WebhookPreset) -> None:
    WEBHOOK_PRESETS[preset.slug] = preset

def get_preset_choices() -> list[tuple[str, str]]:
    return [(slug, p.name) for slug, p in WEBHOOK_PRESETS.items()]

# --- Brevo Track Event (v3) ---
_register(WebhookPreset(
    name="Brevo — Track Event",
    slug="brevo-track-event",
    description="Calls Brevo's Events API to trigger automations. Required secrets: brevo_api_key",
    default_url="https://api.brevo.com/v3/events",
    http_method="POST",
    content_type="application/json",
    headers_template='{\n  "api-key": "{{ secrets.brevo_api_key }}",\n  "accept": "application/json"\n}',
    body_template='{\n  "event_name": "{{ event.type | replace(\'.\', \'_\') }}",\n  "identifiers": {\n    "email_id": "{{ event.data.user_email }}"\n  },\n  "event_properties": {{ event.data | tojson }}\n}',
))
```

**TDD approach:** Write tests for:

- Brevo preset templates render with sample data without errors and produce valid JSON
