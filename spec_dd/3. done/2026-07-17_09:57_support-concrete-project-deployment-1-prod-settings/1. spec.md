# Spec 1 (this repo): P0 production settings defaults + importable `settings_defaults` module

> **Part of the `support-concrete-project-deployment` effort.** See `spec-order.md` for the full
> ordering, parallelism, and cross-spec dependencies. This is the **foundational** spec вЂ” every
> other spec imports its primitives or pairs with its proxy-header change. Do it **first**.
>
> **Target repo:** this repo (`freedom_ls`) вЂ” reference `config/` and the existing
> `freedom_ls/deployment/` app (a new `settings_defaults.py` module is added to it).

## Terminology

`ConcreteFlsImplementation` = the specific downstream production repo that raised this handoff
(installs `freedom_ls` as a read-only git submodule, owns its own `config/`, apps, and
`tailwind.input.css`). It always means that downstream project, never a theme. "Every concrete
project" = concrete FLS projects in general.

**Framing premise (load-bearing):** FLS is never deployed standalone вЂ” only concretely, as a
submodule of a downstream project. The standalone repo exists to be *consumed*, not *deployed*.

## 1. Problem

A concrete FLS project installs `freedom_ls` as a read-only submodule and authors its own
production deployment. Most of the deployment friction is **not project-specific**: it lives in
FLS's reference `config/settings_prod.py`. Several of these gaps are **silent production-breakers**
every concrete project inherits.

### 1.1 The propagation problem (the structural root cause)

FLS has **no importable base-settings module**: each concrete project owns a full copy of
`config/settings_base.py` + `settings_prod.py`, so a settings fix in one project does not
propagate. A fix must currently land in **three** surfaces or it drifts:

1. **The concrete-project template repo** (`freedom-ls-concrete-template`) вЂ” the scaffold new
   projects are generated from. Source of prod-settings defaults for **future** projects.
2. **FLS's own reference `config/`** (this repo's `config/settings_*.py`) вЂ” the documented
   *authority*; the same bug exists in both.
3. **Existing downstream projects, via `/fls:concrete:update_fls`.** `ConcreteFlsImplementation`
   already exists, generated from an older template, so a fix landed only in (1) reaches *new*
   projects and **silently misses the flagship consumer** until `/update_fls` is run against it.

This spec's durable answer is to make the P0 defaults **importable from `freedom_ls`** (see В§5),
collapsing "three surfaces" to "one surface + a submodule SHA bump."

## 2. Goals

- Remove the silent production-breakers in the reference prod settings.
- Make the P0 defaults **importable primitives** in `freedom_ls`, so future fixes propagate on a
  submodule SHA bump rather than three copied edits.

## 3. Non-goals

- Kubernetes / horizontal scaling.
- Forcing async/out-of-process background work as the default (see Spec 3).
- The deployment *artifacts* (Dockerfile, compose, Caddyfile) вЂ” those are Spec 5, in the template repo.

### 4.1 Correction to the idea вЂ” `check --deploy` already runs

The idea's P0 "release gate" claim (*"Nothing runs `manage.py check --deploy` anywhere in the
stack"*) is **false**: it runs in CI (`.github/workflows/security.yml:61вЂ“86`, job
`django-check-deploy`, on push/PR to `main` and weekly) and is documented as a manual pre-release
step (`deployment-security-checklist.md:106вЂ“121`). **Decision (confirmed with the team): drop the
"wire `check --deploy` as a release gate" work.** Keep only the genuinely-missing empty-`SECRET_KEY`
hard-fail (item F). Note that Django's `check --deploy` does **not** flag a missing
`SECURE_PROXY_SSL_HEADER`, so CI is not currently catching item A вЂ” item A stands on its own.

## 5. Requirements

### 5.1 вЂ” P0 production settings defaults

Each removes a silent production failure every concrete project inherits. **All P0 values are also
delivered as importable primitives (В§6);** the reference `config/` and template `settings_prod.py`
become **import-and-assign** consumers of those primitives. *(Scope note: this spec migrates only
**this repo's reference `config/settings_prod.py`** to the import-and-assign form. The **template
repo's** `settings_prod.py` migration is delivered by Spec 5 вЂ” see В§6 and `spec-order.md`.)*

**A. `SECURE_PROXY_SSL_HEADER` вЂ” kill the redirect loop.**
Set `SECURE_PROXY_SSL_HEADER = ("HTTP_X_FORWARDED_PROTO", "https")` as a **hard default** (not
env-guarded вЂ” `settings_prod.py` is only ever reached behind a TLS-terminating proxy). Add it to
the template manifest contract. The settings comment must enumerate the five preconditions that
keep the header trustworthy behind Cloudflare + Caddy (origin firewalled to Cloudflare ranges;
Cloudflare SSL **Full (strict)**, never Flexible; `trusted_proxies` scoped exactly to Cloudflare's
ranges, never `0.0.0.0/0`; app container publishes no port except through Caddy; no custom
`header_up` overriding the default). Detail: `research_prod_settings_security_defaults.md`.
*(Do not confuse this with `TRUSTED_PROXY_IP_HEADER` at `settings_base.py:285` вЂ” that is the
**client-IP** header (`X-Forwarded-For`); this item is the **scheme** header (`X-Forwarded-Proto`)
that drives `request.is_secure()`. Two separate proxy-trust surfaces.)*

**B. Container-friendly logging вЂ” stdout/stderr by default.**
Default prod logging to stdout/stderr only. Drop the `RotatingFileHandler`s (or make them opt-in
via an env flag) and stop bind-mounting `logs/`. **This change is only safe alongside the Spec 5
`json-file` log caps** вЂ” moving to stdout relocates the disk-fill risk to uncapped Docker
`json-file` logs. Delivered as `build_logging_config()` (В§6). *(Do not land this ahead of Spec 5's
per-service `max-size`/`max-file` caps вЂ” see `spec-order.md`.)*

**C. Tasks: delete the misleading re-declaration, keep `ImmediateBackend` as the deliberate
default.**
Remove the redundant `TASKS` re-declaration + `# TODO @claude` from `settings_prod.py:192вЂ“199`
(prod already inherits `ImmediateBackend` from `settings_base.py:406вЂ“409`). This makes the
`ImmediateBackend` default a *deliberate* choice, not a stray TODO. **Removing this TODO is
permitted under CLAUDE.md** (do not remove any *other* TODO): the decision it flagged is now
deliberately made вЂ” `ImmediateBackend` is the shipped default and the durable opt-in backend is
Spec 3 вЂ” and its guidance is **not lost**, the same caveat survives as the plain comment on
`settings_base.py:404вЂ“405` ("A production backend вЂ¦ is needed before using tasks in production").
So the removal is lossless.

**D. DB SSL via `DB_SSLMODE` env var.**
Support `DB_SSLMODE` в†’ `DATABASES["default"]["OPTIONS"] = {"sslmode": вЂ¦}`. **Default to `prefer`,
not `require`**, and ship `DB_SSLMODE=disable` in the template `.env.example`. The shipped topology
is same-host containerised `postgres:17` (no TLS by default); a `require` default makes the app
fail to connect on first boot. Reserve `require`/`verify-full` for external/managed DBs and say so
in `.env.example`. Honest nuance: for same-host containerised Postgres, `sslmode` is close to
security-theatre (the real control is **not publishing `5432`** вЂ” Spec 5). **Rescope
`docs/deployment-security-checklist.md:22`** ("DB connections use SSL/TLS") to external/managed DBs
only. Delivered as `database_ssl_options(sslmode)` (В§6).

**E. `CONN_MAX_AGE` + `CONN_HEALTH_CHECKS`.**
Add an env-guarded `CONN_MAX_AGE` (recommended **60вЂ“300s**, **not** `None`/unlimited) and ship
`CONN_HEALTH_CHECKS = True` alongside it (persistent connections leave stale sockets after a `db`
restart/redeploy that 500 the next request without health checks). PgBouncer is a later scaling
step, out of scope. Delivered as importable constants (В§6).

**F. Hard-fail on missing/empty `SECRET_KEY`.**
Change `os.getenv("SECRET_KEY", "")` в†’ `os.environ["SECRET_KEY"]` so a missing key raises
`KeyError` at **settings-import** time (Gunicorn boot в†’ visible crash-loop) rather than lazily on
the first request. This only catches an *empty/missing* key; a *weak-but-present* key is Django's
`security.W009`, already covered by the existing CI `check --deploy` (В§4.1). Delivered as
`require_secret_key()` (В§6).

> **Dropped from the idea (see В§4.1):** wiring `check --deploy` as a new release gate вЂ” it already
> runs in CI and is documented as a manual step.

## 6. Importable prod-settings module (core of the effort)

The propagation root cause (В§1.1) is only truly fixed by making the P0 defaults **importable from
`freedom_ls`**, so a fix propagates by a submodule SHA bump instead of three edits.

**Design (decided by research вЂ” `research_importable_base_settings.md`):**

- **Do not adopt a settings framework.** `django-configurations` requires rewriting every project's
  `manage.py`/`wsgi.py`/`asgi.py` and is stale (last release 2024-03, no declared Django 6 / Python
  3.13 support). Rejected.
- **Extend the pattern FLS already uses.** The closest precedent already lives in the deployment
  app: `freedom_ls/deployment/storage.py` exposes `build_s3_media_storage()` вЂ” a pure, Django-free,
  unit-tested helper (`freedom_ls/deployment/tests/test_storage.py`) that `config/settings_prod.py`
  imports **at settings-load time** (`settings_prod.py:173`). `config/settings_base.py:23вЂ“25` uses
  the same shape for `configure_theme` (a pure function) and `FLS_WEBHOOK_EVENT_TYPES` (a flat
  constant). Ship a new module in exactly this spirit вЂ”
  **`freedom_ls/deployment/settings_defaults.py`**, a new pure module added to the **existing**
  `freedom_ls/deployment/` app (the app already exists and is registered in `INSTALLED_APPS` at
  `settings_base.py:100`). It mirrors `storage.py`: no Django import at module top, so it is safe to
  import before the app registry is ready. It exposes the P0 items as:
  - **flat constants:** `SECURE_PROXY_SSL_HEADER`, `CONN_MAX_AGE`, `CONN_HEALTH_CHECKS`
  - **small pure functions:** `build_logging_config()` в†’ stdout by default;
    `database_ssl_options(sslmode)`; `require_secret_key()`
- **No `globals()`-mutating `apply_fls_defaults(globals())`** вЂ” it hides final values from a plain
  read of `settings_prod.py` and makes override-ordering bugs silent. Named symbols a project
  assigns explicitly keep `settings_prod.py` thin, greppable, diffable, and **fail loudly**
  (`ImportError`/`AttributeError`) on drift from the pinned submodule.
- **What this changes:** the template's `settings_prod.py` changes **once** (literals в†’
  import-and-assign) вЂ” that template edit is delivered by **Spec 5**; this spec delivers the
  importable module itself and migrates **this repo's reference `config/settings_prod.py`**. Future
  P0-class fixes then land only in `freedom_ls` and reach every project on a SHA bump.
  `settings_prod.py` stays project-owned and thin вЂ” it *chooses* which FLS defaults to use and
  supplies genuinely project-specific values (`HOST_DOMAIN`, S3 bucket, `DB_SSLMODE` source of
  truth). Existing downstream projects need a **one-time** `settings_prod.py` migration to the
  import form the first time `/update_fls` runs after this lands.

### 6.1 Relationship to the per-app `config.py` convention

A separate, recently-merged convention (`freedom_ls/base/app_settings.py` вЂ” `AppSettings` /
`Setting` / `required_settings_errors()`, documented in the `fls:app-settings` skill) also concerns
settings. **It does not apply to this spec**, and the two must not be conflated. The distinction is
now concrete inside the deployment app itself, which already carries **both** patterns side by side:
`freedom_ls/deployment/config.py` (`DeploymentSettings(AppSettings)` for PostHog/Sentry вЂ” app-owned
settings) **and** `freedom_ls/deployment/storage.py` (a pure helper for a Django built-in,
`STORAGES`, imported straight into `settings_prod.py`, *not* routed through `config.py`).
`settings_defaults.py` joins the `storage.py` category. The two conventions run in opposite
directions:

| Convention | Direction | Covers |
|---|---|---|
| `config.py` / `AppSettings` | **read** вЂ” an FLS app reads a project-configurable, *app-owned* setting through its `config`, with declared defaults + required-enforcement | App-owned settings. **Explicitly excludes** Django built-ins (`DATABASES`, `LOGGING`, `MIDDLEWARE`, вЂ¦) and env-derived secrets, which stay as direct `settings.X`. |
| `settings_defaults.py` (this spec, В§6) | **assign** вЂ” `freedom_ls` provides deployment default *values/functions* that `settings_prod.py` imports and assigns into Django settings | The P0 deployment defaults (В§5 AвЂ“F). |

Every В§5 setting (`SECURE_PROXY_SSL_HEADER`, `LOGGING`, `TASKS`, `DATABASES["default"]["OPTIONS"]`,
`CONN_MAX_AGE`, `CONN_HEALTH_CHECKS`, `SECRET_KEY`) is a Django built-in or an env-derived secret вЂ”
exactly the category `AppSettings` excludes. So they correctly stay as import-and-assign via
`settings_defaults.py` and are **not** routed through `AppSettings`; the two conventions are
complementary, not alternatives. **Implementer:** consult `fls:app-settings` for the boundary, but do
not put these deployment defaults behind `AppSettings`.

## 7. Propagation requirement (every P0 landing)

Landing a P0 fix in the reference `config/` only patches *future* projects. **Every P0 landing has
a further step: run `/fls:concrete:update_fls` against `ConcreteFlsImplementation`** (and any other
existing project) so the flagship consumer is actually patched. A triage that stops at reference
config leaves the project that raised this idea unpatched.

## 8. Success criteria

1. **No redirect loop:** behind a TLS-terminating proxy, `SECURE_SSL_REDIRECT = True` +
   `SECURE_PROXY_SSL_HEADER` resolves `request.is_secure()` correctly; HSTS becomes active (verify
   in the same pass вЂ” HSTS/redirect are inert until the proxy header lands).
2. **Logging is stdout/stderr-only** in the reference config with no `logs/` bind mount (the
   Docker-side `json-file` caps that make this safe are Spec 5 вЂ” do not ship B before them).
3. **No tasks lie in settings:** `settings_prod.py` has no redundant `TASKS`/`# TODO @claude`;
   `ImmediateBackend` is the deliberate default.
4. **DB connects on first boot** with the shipped same-host Postgres and `DB_SSLMODE=disable`;
   `require` reserved for external DBs; checklist item 22 rescoped honestly.
5. **`CONN_MAX_AGE` + `CONN_HEALTH_CHECKS`** set; no stale-socket 500 after a `db` restart.
6. **Missing `SECRET_KEY` crash-loops at boot** (visible), not a green boot that 500s the first request.
7. **Propagation collapsed:** the P0 defaults are importable from `freedom_ls`; the reference
   `settings_prod.py` is import-and-assign, failing loudly on drift; a future P0-class settings fix
   is a one-surface change in `freedom_ls` + a submodule SHA bump; `ConcreteFlsImplementation` is
   patched via `/fls:concrete:update_fls`.
