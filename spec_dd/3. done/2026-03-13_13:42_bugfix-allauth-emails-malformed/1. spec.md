# Spec: Fix Malformed Allauth Email Confirmation Links

## Problem

After upgrading to Django 6.x (from Django 5.2), allauth email confirmation links are broken. The confirmation URLs contain quoted-printable soft line breaks (`=\n`) that split the URL across multiple lines, making the links unusable.

### Example of the bug

The raw email contains:

```
Confirm Email Address: http://127.0.0.1:8010/accounts/confirm-email/MQ:1w0wOR=
:x4RrcIzx-WXFfiT2Clspj7ZC22lr592x1T-goy8V44I/
```

The URL should be a single unbroken line:

```
http://127.0.0.1:8010/accounts/confirm-email/MQ:1w0wOR:x4RrcIzx-WXFfiT2Clspj7ZC22lr592x1T-goy8V44I/
```

### Root cause

Django 6 changed the default signing algorithm from SHA-1 to SHA-256, producing longer HMAC signatures. These longer tokens push the confirmation URL past the 76-character limit imposed by MIME quoted-printable encoding (RFC 2045). Python's `email` library automatically wraps lines at this limit by inserting `=\n` (a soft line break), which corrupts the URL.

This affects:
- Plain text email clients that don't decode quoted-printable
- Developers reading raw email files from the filebased email backend (`gitignore/emails/`)
- Any email client or renderer that treats the `=\n` as literal characters

### Why this matters

- Users cannot confirm their email addresses, blocking registration entirely (since `ACCOUNT_EMAIL_VERIFICATION = "mandatory"`)
- Password reset and other allauth flows with long URLs may also be affected
- The dev workflow for testing emails is broken

## Solution

Fix the email Content-Transfer-Encoding to avoid quoted-printable line wrapping that breaks URLs.

### Approach: Override email encoding

The Python `email` library chooses quoted-printable encoding by default for text content. We need to ensure emails use an encoding that preserves long URLs intact. Options within this approach:

1. **Use `8bit` Content-Transfer-Encoding** - Modern mail servers support 8bit encoding, which doesn't wrap lines. This is the simplest fix.
2. **Use `base64` Content-Transfer-Encoding** - Always safe but makes raw email files unreadable (less relevant since the bug exists precisely because we read raw files).

The recommended sub-approach is `8bit` encoding, applied by overriding the email sending in the custom `AccountAdapter` to set the encoding on the email message before sending.

### Why not shorten the tokens?

Shortening the signing tokens would require overriding Django's signing mechanism or reverting to SHA-1, which would be a security regression. The encoding fix is the correct layer to address this — the email transport should handle long URLs without corruption.

## Scope

### In scope

- Fix email encoding for all allauth-generated emails to prevent URL line wrapping
- Automated pytest tests that verify URLs in generated emails are not broken by encoding
- Test coverage for new functionality. Dont write tests for all allauth flows, just one will be sufficient. Eg: password reset (password reset is not something we will ever turn off, but we might turn user registration off)
- Manual QA test plan covering all allauth flows (register, validate pass, password reset)

## Success criteria

1. Email confirmation links in generated emails are valid, unbroken URLs — no `=\n` splits
2. This holds true for both HTML and plain text email parts
3. Automated tests verify that all allauth email flows produce emails with intact URLs
4. Password reset and email change URLs are also verified as intact
5. Existing email styling and content (premailer, custom templates) continue to work correctly
