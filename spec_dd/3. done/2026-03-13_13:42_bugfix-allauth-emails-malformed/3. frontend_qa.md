# Frontend QA: Fix Malformed Allauth Email Confirmation Links

## Prerequisites

- Dev server running: `uv run python manage.py runserver`
- Email backend is file-based in dev (emails saved to `gitignore/emails/`)
- A site exists in the database (default dev setup)
- Signups are enabled for the site

## Test 1: Email Confirmation on Registration

1. Open browser to `http://127.0.0.1:8010/accounts/signup/`
2. Fill in the registration form with a new email and password
3. Submit the form
4. Check `gitignore/emails/` for the latest email file
5. Open the email file in a text editor
6. **Verify:**
   - The confirmation URL (containing `/accounts/confirm-email/`) is on a single line with no `=\n` breaks
   - The `Content-Transfer-Encoding` header shows `8bit` (not `quoted-printable`)
   - The URL is clickable/copyable as a complete URL
7. Copy the URL from the email and paste it into the browser
8. **Verify:** The email confirmation page loads successfully

## Test 2: Password Reset

1. Open browser to `http://127.0.0.1:8010/accounts/password/reset/`
2. Enter the email of an existing user
3. Submit the form
4. Check `gitignore/emails/` for the latest email file
5. Open the email file in a text editor
6. **Verify:**
   - The password reset URL (containing `/accounts/password/reset/key/`) is on a single line with no `=\n` breaks
   - The `Content-Transfer-Encoding` header shows `8bit`
7. Copy the URL from the email and paste it into the browser
8. **Verify:** The password reset form loads and allows setting a new password

## Test 3: Email Change (if user is logged in)

1. Log in as an existing user
2. Navigate to the email management page (usually `/accounts/email/`)
3. Add a new email address
4. Check `gitignore/emails/` for the confirmation email
5. **Verify:**
   - The confirmation URL is unbroken (no `=\n`)
   - The `Content-Transfer-Encoding` header shows `8bit`
6. Copy and visit the URL
7. **Verify:** The new email is confirmed successfully

## How to inspect email files

The email files in `gitignore/emails/` are raw MIME messages. Each email is a multipart message containing two parts:

1. **text/plain** — the plain text version
2. **text/html** — the HTML version

### Checklist for each MIME part

For **every** test above, verify the following in the raw email file:

- [ ] Search for `Content-Transfer-Encoding` — it should appear as `8bit` for both the text/plain and text/html parts
- [ ] `quoted-printable` should NOT appear anywhere in the file
- [ ] Search for `=\n` (equals sign followed by newline) — should NOT appear in the email body
- [ ] The full URL is present as a single unbroken string in **both** the text/plain and text/html sections
- [ ] Copying the URL from the file and pasting into a browser loads the correct page

### How to identify the two MIME parts

Look for MIME boundary markers (lines starting with `--`). Between each pair of boundaries you'll find a `Content-Type` header (`text/plain` or `text/html`) followed by a `Content-Transfer-Encoding` header. Check both sections independently.

### Bad (before fix):
```
Content-Transfer-Encoding: quoted-printable

... http://127.0.0.1:8010/accounts/confirm-email/MQ:1w0wOR=
:x4RrcIzx-WXFfiT2Clspj7ZC22lr592x1T-goy8V44I/
```

### Good (after fix):
```
Content-Transfer-Encoding: 8bit

... http://127.0.0.1:8010/accounts/confirm-email/MQ:1w0wOR:x4RrcIzx-WXFfiT2Clspj7ZC22lr592x1T-goy8V44I/
```
