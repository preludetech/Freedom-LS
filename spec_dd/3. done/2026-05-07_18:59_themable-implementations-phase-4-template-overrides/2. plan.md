# Implementation Plan: First Class — Tier 3 template overrides (Phase 4 of 5)

This plan turns `1. spec.md` (in this folder) into shippable code. The phase ships three small cotton overrides under the `first_class` theme directory and adds an integration test that proves the resolver picks up theme `templates/` dirs.

For the holistic five-phase context, see `../themable-implementations-master-decomposed-into-phases/2. plan.md`.

**Phases 1–3 are hard prerequisites.**

---

## Architectural principles inherited

1. **Themes are sparse.** First Class adds three cotton overrides — not the full base/cotton tree.
2. **Defaults stay in the main project.** Generic primitives stay in `freedom_ls/base/templates/cotton/*`; the override mechanism shadows them selectively.
3. **This phase ends in a green build.**

---

## Pre-existing functionality to reuse (DRY)

- The Phase-1 resolver (`freedom_ls/base/theming.py::configure_theme`) — already prepends the active theme's `templates/` dir to `TEMPLATES[0]["DIRS"]`. No new wiring needed; Phase 4 just exercises that codepath.
- Cotton's existing template loader chain — a same-named file in a higher-priority `DIRS` entry shadows the FLS default automatically. No new loader logic, no cache-key plumbing.
- Existing FLS cotton primitives (`freedom_ls/base/templates/cotton/header-button.html`, `chip.html`, `loading-indicator.html`) — read the originals to capture the full `<c-vars>` declarations and slot/attrs forwarding, then write the overrides.

---

## End-state checklist

- `freedom_ls/themes/first_class/templates/cotton/header-button.html`, `chip.html`, `loading-indicator.html` exist.
- Each override clearly looks different from the default and honours the original's full `<c-vars>` declaration plus `{{ attrs }}` and `{{ slot }}` forwarding.
- Pages that render those cotton primitives use the overridden versions when `FLS_THEME=first_class`.
- Pages render unchanged when `FLS_THEME=default`.
- New integration test in `freedom_ls/base/tests/test_theme_template_override.py` passes.
- Phase 1–3 deliverables remain green.

---

## Task 1 — Verify resolver wiring picks up theme templates

The resolver (Phase 1) already prepends the active theme's `templates/` dir to `TEMPLATES[0]["DIRS"]`. Until Phase 4 the active-theme dir has no `templates/`, so this codepath is untested in practice.

Add an integration test `freedom_ls/base/tests/test_theme_template_override.py`:

- Render a template that uses `<c-loading-indicator>` (or whichever primitive is overridden).
- Under default settings, assert the rendered HTML contains a marker from the FLS default cotton component.
- Under `RESOLVED_THEME_DIR` pointed at `first_class`, assert the rendered HTML contains a marker from the override.

If `RESOLVED_THEME_DIR` is hard to swap mid-test, use a fixture that reconfigures `TEMPLATES[0]["DIRS"]` via `override_settings` and clears the template cache (Django's `template.utils.get_app_template_dirs` cache + cotton's loader cache).

---

## Task 2 — Author the three overrides

Each override is a small, deliberately-different copy of the FLS default. Read the original first to capture its full `<c-vars>` block, then ship a version that:

- **`templates/cotton/header-button.html`** — labels in the `font-display` (Outfit) family with a brand-coloured underline accent.
- **`templates/cotton/chip.html`** — pill-shaped (`rounded-full`) with a left-border colour stripe instead of a background tint.
- **`templates/cotton/loading-indicator.html`** — replace the spinning icon with a brand-coloured pulsing dot row, message rendered in the `font-display` (Outfit) family.

These are demos, not polished design. The bar is "obviously different from default; still functional".

Add brand names (`Outfit`, `DM Sans`) to `cspell.json` if any token usage trips the spellcheck.

---

## Task 3 — Phase verification

- `FLS_THEME=first_class npm run tailwind_build` — clean (the `@source "./freedom_ls/themes/*/templates/**/*.html"` from Phase 1 picks up any new utility classes).
- Visual check: pages using `<c-loading-indicator>`, `<c-chip>`, `<c-header-button>` render the override variants under `first_class`.
- Switching to `FLS_THEME=default` restores the original templates.
- All previous tests green; new integration test green.

## Skills and MCPs to use

- **`fls:template`** — cotton-override resolution, `<c-vars>` contract for overridden cotton primitives. Refer to `fls-claude-plugin/resources/templates_and_cotton.md`.
- **`fls:testing`** — integration test for template-override pickup; use `override_settings` and template-cache invalidation. Refer to `fls-claude-plugin/resources/testing.md`.
- **`fls:icon-usage`** — only relevant if any of the overrides touches icons; otherwise skip.
- **`fls:request-code-review`** — invoke at the end of the phase before commit.
- **MCP `playwright`** — visual verification of the three override surfaces under both themes.
