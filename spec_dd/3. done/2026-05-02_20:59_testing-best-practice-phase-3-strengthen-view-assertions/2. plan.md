This plan covers the work described in `1. spec.md`. We are following TDD. Don't write all the new assertions up front: for each test, run the existing test, confirm it passes, add the stronger assertion, run it again, ensure it still passes (or expose a real bug).

def test_invalid_event_type_returns_400(
    self, admin_user, mock_site_context, client
) -> None:
    """Unknown event_type should be rejected by validate_event_type."""
    endpoint = WebhookEndpointFactory(event_types=["user.registered"])
    client.force_login(admin_user)
    response = client.post(
        _send_test_result_url(endpoint.pk),
        data={"event_type": "definitely.not.a.real.event"},
    )
    assert response.status_code == 400
    # No event/delivery should be created on rejection.
    assert WebhookEvent.objects.count() == 0
    assert WebhookDelivery.objects.count() == 0
```

## TDD execution order

For each file, in this order:

1. Read the file end-to-end (one Read per file is fine — they're all <400 lines).
2. Run only that file's tests: `uv run pytest <path-to-file> -x` to confirm green baseline.
3. Make the strengthening edits in one batch per file (Edit tool, multiple `old_string`/`new_string` pairs as separate calls).
4. Re-run the file. If a strengthened assertion **fails**, that's a real-world finding:
   - First decide: is this test now wrong, or did it expose a real bug in the view?
   - If a real bug: file the bug (a TODO comment in the test plus a `pytest.skip("blocked on TODO/issue")` is fine per the spec) and **do not** fix the production code in this spec.
   - If the test was wrong: adjust the assertion (the goal is "stronger and correct", not "stronger and wrong").
5. After all per-file work is done, run `uv run pytest` (full suite) under the Phase 2 gates (`-p no:cacheprovider --create-db -p socket` etc. — whatever `pyproject.toml` / `pytest.ini` already configures). The Phase 2 spec is in `spec_dd/3. done/` and the gates are already wired into the repo config; we don't need to invoke them manually.

## Pseudocode for the new negative tests

### `test_invalid_event_type_returns_400` (webhooks)

```
Arrange: factory endpoint with event_types=["user.registered"]; force_login(admin_user)
Act:     POST to send_test_result_url with event_type="not.a.real.event"
Assert:  status_code == 400
         WebhookEvent.objects.count() == 0     # no event created on rejection
         WebhookDelivery.objects.count() == 0  # no delivery attempted
```

No mock needed — the view should reject before reaching `attempt_delivery`.

Arrange: course_with_nested_structure fixture (creates course + items)

Assert:  status_code == 302
         "/login" in response.url
```

Same shape for `form_start` (use the `Form`-parametrized fixture).
