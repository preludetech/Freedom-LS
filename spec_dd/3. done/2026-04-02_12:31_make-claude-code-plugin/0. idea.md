# Idea: Extract .claude/ Configuration into a Claude Code Plugin

## Goal

Turn the current `.claude/` directory contents into a reusable Claude Code plugin called `fls`. The plugin will live inside the FreedomLS repository. Concrete implementations of FreedomLS will install the plugin in a standalone, self-contained way.

## Why

The current `.claude/` directory contains ~50 files (skills, commands, agents, hooks, docs) that define FreedomLS development conventions. These are generic to any FreedomLS implementation, but currently they're only available in this repo. A plugin makes them portable and reusable.

## Plugin Structure

The plugin will be named `fls` and follow the Claude Code plugin conventions:

```
fls-claude-plugin/               # Lives inside the FreedomLS repo
├── .claude-plugin/
│   └── plugin.json       # {"name": "fls", ...}
├── commands/             # All current commands (commit, tdd, spec_driven_dev/, etc.)
├── agents/               # code-reviewer, qa-data-helper
├── skills/               # All current skills (testing, htmx, template, etc.)
│   └── <skill>/
│       └── SKILL.md
├── docs/                 # Shared reference docs (factory_boy.md, testing.md, etc.)
├── hooks/
│   ├── hooks.json        # Hook event configuration
│   └── scripts/          # ruff_fix.sh, security-guard.sh, post-edit-bandit.sh
├── .mcp.json             # Playwright MCP config
└── README.md
```

## Key Design Decisions

### No `@` path references

The plugin MUST NOT use `@` syntax for file references. In plugins, `@` resolves to the project root, not the plugin root. Since different FreedomLS implementations have different project roots, `@` references would break.

**Solution:** Move all `.claude/docs/*.md` files into the plugin's top-level `docs/` directory. Skills, agents, and commands reference them via `${CLAUDE_PLUGIN_ROOT}/docs/filename.md`. This is consistent — all reference docs live in one place, avoiding duplication and making reuse straightforward (e.g. `factory_boy.md` is referenced by the testing skill, playwright-tests skill, and qa-data-helper agent).

### No hardcoded absolute paths

All paths within the plugin must use `${CLAUDE_PLUGIN_ROOT}` (for hooks, MCP config) or be relative within the plugin structure.

Files currently with hardcoded paths that need fixing:
- `git-worktree-setup/SKILL.md` — hardcoded `/home/sheena/workspace/...`
- `qa-data-helper.md` — hardcoded agent memory paths

### Hook ownership moves to plugin

The plugin will own all hook definitions (ruff_fix, security-guard, post-edit-bandit). The project's `settings.json` should become minimal — just `enabledPlugins` and any project-specific overrides. This avoids hooks running twice.

### CLAUDE.md strategy

The plugin can provide foundational FreedomLS conventions. The project's root `CLAUDE.md` should only contain project-specific details (commands, local paths, etc.) that vary between implementations.

## What Gets Extracted (42+ files)

- **Settings:** Base permissions, deny rules (from settings.json)
- **Hooks:** ruff_fix.sh, post-edit-bandit.sh, security-guard.sh
- **Skills:** 12 generic skills (testing, htmx, multi-tenant, template, frontend-styling, icon-usage, admin-interface, markdown-content, alpine-js, playwright-tests, request-code-review, use-playwright)
- **Commands:** All 13 commands (commit, tdd_implement, do_todos, security-review, catchup, make_github_issue, placeholder_page, threat-model, address_pr_review, spec_driven_dev/*)
- **Agents:** code-reviewer, qa-data-helper
- **Docs:** All 9 docs (moved into skill references/)
- **MCP:** Playwright config

## What Stays Project-Local

- `settings.local.json` — user-specific overrides
- `agent-memory/` — runtime state accumulated per-project
- `fls.md` — committed plugin settings file (dev credentials, project-scoped config)
- `fls.local.md` — gitignored, machine-specific overrides
- `brand-guidelines` skill — branding is implementation-specific, not generic to FreedomLS

## Installation

Concrete FreedomLS implementations install the plugin via:
1. Project `settings.json`: enable the `fls` plugin with a path to the plugin directory
2. Or use `--plugin-dir` flag during development

The plugin should be installed in the current project too, so it gets used immediately.

## Resolved Decisions

1. **Dev credentials** (`demodev@email.com` in do_qa.md and use-playwright): Make configurable via plugin settings (`.claude/fls.md`). This file is committed to git since dev credentials are project-scoped — all dev machines share the same test users set up by shared dev utilities. Commands/skills read credentials from there at runtime. `.claude/fls.local.md` can override if needed for machine-specific config.
2. **git-worktree-setup skill**: Generalize it for the plugin — remove hardcoded paths, make it work for any FreedomLS bare-repo + worktree setup.
3. **Stale files** (oldSKILL.md): Delete during extraction — they reference outdated icon libraries and are no longer relevant. Note: brand-guidelines stays project-local (each implementation has its own branding).
4. **Project scripts** (fetch_pr_comments.sh, find_available_port.sh, dev_db_*.sh, install_dev.sh): Move into the plugin's `scripts/` directory so the plugin is fully self-contained. Commands reference them via `${CLAUDE_PLUGIN_ROOT}/scripts/`.
