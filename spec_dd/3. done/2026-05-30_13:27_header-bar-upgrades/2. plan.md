## Stream 1 ŌĆö `User.initials` property

### 1.1 Decide property type

The spec defers `@property` vs `@cached_property`. Recommendation: plain `@property`. The header renders a single avatar per request and the computation is O(length of name) with no DB access. `@cached_property` adds an instance cache that survives mutation and complicates tests; not worth it for a derivation this cheap. Document the decision in the property's docstring so future callers (educator student lists, comments) inherit the contract.

### 1.2 TDD: add the property + tests

Order: write the failing tests first, then the implementation, per `feedback_tdd_for_bugfixes.md` and the `fls:testing` skill.

### 1.3 Implementation sketch

```python
class User(...):
    @property
    def initials(self) -> str | None:
        """Return one-or-two-character initials for avatar display, or None.

        Cascade (first match wins):
          1. first_name + last_name ŌåÆ first letter of each.
          2. first_name only ŌåÆ if it splits on whitespace into Ōēź2 tokens, first
             letter of the first two; otherwise first two letters.
          3. Else: first two alphabetic chars of the email local-part, skipping
             any leading non-alphabetic characters.
          4. Otherwise: None ŌĆö caller renders a fallback icon.

        Diacritics are preserved (no ASCII folding). Non-Latin scripts return a
        single grapheme rather than two characters.
        """
        first = (self.first_name or "").strip()
        last = (self.last_name or "").strip()

        if first and last:
            return _two_or_one(first[0], last[0])

        name = first or last
        if name:
            tokens = name.split()
            if len(tokens) >= 2:
                return _two_or_one(tokens[0][0], tokens[1][0])
            # Single token ŌĆö first two characters if both Latin, else single grapheme.
            return _two_or_one(name[0], name[1] if len(name) > 1 else "")

        # Email local-part fallback ŌĆö skip leading non-alphabetic chars.
        local = (self.email or "").split("@", 1)[0]
        alphas = [ch for ch in local if ch.isalpha()]
        if not alphas:
            return None
        return _two_or_one(alphas[0], alphas[1] if len(alphas) > 1 else "")
```

`_two_or_one` is a private module-level helper that:
- Uppercases via `str.upper()` (Unicode-aware).
- Returns the two characters concatenated if both are Latin, otherwise just the first character (matches the "single grapheme for non-Latin" rule). "Latin" here means the base character (after `unicodedata.normalize("NFD", ch)[0]`) lies in the basic-Latin / Latin-Extended block ŌĆö see the sketch in `research_avatar.md` ┬¦2 for the reference test. Implementer can copy that helper verbatim or inline the check; either is fine.

Type hints required (`str | None` per CLAUDE.md modern-syntax rule). No `Any`. No logging.

## Stream 3 ŌĆö Sticky header + on-scroll treatment

### 3.1 Sticky positioning

Edit `partials/header_bar.html`. Add `sticky top-0 z-30` to the `<header>` element's class list. Keep `bg-header` (Stream 2) and `shadow-md`. The element stays in document flow.

`z-30` sits above general content and `z-20` (the existing course-progress-table sticky rows in `educator_interface/partials/course_progress_panel.html` use `z-20`/`z-30`, but they are scoped to a scrollable `<div>` and don't compete with the page header) but below modals (the cotton modal uses `z-50`).

### 3.2 Audit ancestors for sticky-breaking properties

Read-only audit of `_base.html` and `_base_interface.html` ancestors of `<header>`. Confirm none of them have `overflow-hidden`, `overflow-auto`, `overflow-scroll`, `transform`, `filter`, or `contain: paint` on any wrapper between `<body>` and `<header>`. Initial inspection: `_base.html` puts `<header>` directly inside `<body>` via `{% include %}`, so there is no wrapping element between them ŌĆö sticky should work unmodified. Re-confirm on the actual rendered DOM during QA.

### 3.3 Scroll-padding for anchor links and focus

Edit `tailwind.components.css` (or add a tiny `@layer base` block somewhere already loaded after `tailwind.input.css`). Add:

```css
@layer base {
    html {
        scroll-padding-top: 5rem; /* tuned to actual header height during QA */
    }
}
```

Pick a single value rather than a CSS custom property keyed off `--header-h` (we don't have a measured header-height variable, and the header is `p-3 sm:p-4 + content` which is ~56ŌĆō64px ŌĆö `5rem` (80px) gives generous clearance and is fine for both breakpoints). Tune during QA on both themes.

Spec defers the choice between `scroll-padding-top` on `html` vs the scroll container vs per-target `scroll-margin-top`. Going with `html` because (a) it's the simplest single-source rule, (b) it works for both anchor links and `:focus-visible` autoscroll, and (c) there is no inner scroll container in the current FLS layout that would override it.

### 3.4 Scrolled-state mechanism ŌĆö pick CSS-only

The spec offers two mechanisms: Alpine + `scroll.window.passive` listener, or CSS `@container scroll-state(stuck: top)`.

**Decision: use the Alpine path.** Reasons:

- `@container scroll-state(stuck: top)` is a 2025/2026 spec; browser support is still Chromium-leaning and not yet broad enough to ship without a JS fallback. Adding the JS fallback then making the CSS the "happy path" is more code than just the JS.
- We already have an Alpine ecosystem in the same template family (`sidebarComponent`, `dropdownMenu`). One more small Alpine `data` reads naturally.
- The listener is a passive `scroll` on `window` ŌĆö cheap, idempotent, no layout reads on every fire (we toggle a class only when the boolean state changes).
- The change-detection guard means we don't thrash the class ŌĆö the listener checks `const stuck = window.scrollY > 0; if (stuck !== this.scrolled) this.scrolled = stuck;`.

Implementation:

1. Add to `freedom_ls/base/static/base/js/alpine-components.js` a new `Alpine.data("headerScroll", ...)` registration:

   ```js
   Alpine.data("headerScroll", () => ({
       scrolled: false,
       _onScroll: null,
       init() {
           this._onScroll = () => {
               const next = window.scrollY > 0;
               if (next !== this.scrolled) this.scrolled = next;
           };
           // Set initial state in case the page is loaded mid-scroll (back-button restore).
           this._onScroll();
           window.addEventListener("scroll", this._onScroll, { passive: true });
       },
       destroy() {
           if (this._onScroll) {
               window.removeEventListener("scroll", this._onScroll);
               this._onScroll = null;
           }
       },
   }));
   ```

2. In `partials/header_bar.html` add `x-data="headerScroll"` and `x-bind:data-scrolled="scrolled"` to the `<header>`. The class swap is then driven by CSS attribute selectors ŌĆö no `x-bind:class` string concatenation needed.

### 3.5 Default-theme on-scroll: stronger shadow

The default theme keeps the header opaque. Per spec ┬¦3 it gains a stronger shadow / hairline once scrolled. Two options:

- Tailwind `shadow-md` baseline + `data-[scrolled=true]:shadow-lg`.
- Or a `border-b border-on-header/15` once scrolled.

Going with the shadow approach because the existing class is `shadow-md` and stepping it up to `shadow-lg` matches NN/g's "subtle separation cue" guidance. If QA finds `shadow-lg` too soft against page content, fall back to `data-[scrolled=true]:border-b data-[scrolled=true]:border-on-header/15`.

```html
<header class="bg-header text-on-header sticky top-0 z-30
               shadow-md data-[scrolled=true]:shadow-lg
               motion-reduce:transition-none
               transition-shadow duration-200
               p-3 sm:p-4 flex justify-between items-center"
        x-data="headerScroll"
        x-bind:data-scrolled="scrolled">
```

### 3.6 first_class on-scroll: translucent + blur

`first_class` renders the same `<header>` markup but the *visual* swap to translucent + backdrop-blur is keyed off the `data-scrolled` attribute. Because Tailwind utilities are global, we add the variant utilities to the same class list and let them apply in any theme ŌĆö they will only *look different* under `first_class` (where the header background is `--color-surface`-on-white) because the alpha actually reveals the page content beneath.

Recommended class additions on the same `<header>`:

```
data-[scrolled=true]:bg-header/85
data-[scrolled=true]:backdrop-blur-md
data-[scrolled=true]:saturate-150
```

Why this works in both themes simultaneously:

- In default the header sits on its own opaque `bg-header` (which equals `--color-primary`). `bg-header/85` over an unblurred opaque page top makes essentially no visible difference on the default theme ŌĆö the header is the topmost paint anyway. `backdrop-blur-md` does nothing because `bg-header/85` of the brand colour is still effectively opaque enough that there's nothing meaningful "behind" to soften. We accept the GPU cost of a minor compositor layer in default, OR we scope the blur with a Tailwind theme/data variant. **Decision: accept the no-op cost rather than complicate the markup.** The spec's success criterion 6 ("no `backdrop-blur` is applied" in default) is satisfied via *value* ŌĆö `bg-header` in default has full alpha at the role-colour level ŌĆö but to be strictly correct we instead emit the blur classes only under first_class.

To be strictly compliant with success criterion 6 (no blur in default), the simplest path is to put the translucent+blur utilities behind a theme attribute on `<html>`/`<body>`, OR override them inside `first_class/theme.css`. The simplest mechanism that doesn't add new templating gymnastics:

> **Decided approach:** put the translucent-and-blur rules inside a `first_class/theme.css` `@layer utilities` block so they only apply when first_class is the active theme. The default theme just gets the shadow change.

Concretely, in `freedom_ls/themes/first_class/static/themes/first_class/theme.css` add:

```css
@layer utilities {
    /* When the header is scrolled in first_class, switch to a frosted-glass
     * surface. The base header otherwise stays opaque/flush at the page top. */
    header[data-scrolled="true"] {
        background-color: color-mix(in oklch, var(--color-header) 85%, transparent);
        backdrop-filter: blur(12px) saturate(1.5);
        -webkit-backdrop-filter: blur(12px) saturate(1.5);
    }

    @media (prefers-reduced-motion: reduce) {
        header {
            transition: none;
        }
    }
}
```

This way the default theme's compiled CSS contains no blur at all (success criterion 6 met literally), and first_class adds the rules on top via the active-theme cascade. The exact alpha (`85%` vs `90%`) is tuned during QA for contrast.

### 3.7 Reduced-motion respect

The `<header>` carries `motion-reduce:transition-none` (Stream 3.5). `motion-reduce:` resolves to `@media (prefers-reduced-motion: reduce)`. Combined with the explicit `@media (prefers-reduced-motion: reduce)` block in 3.6, both the shadow change in default and the colour/blur change in first_class resolve instantly under reduced motion. The static stuck state is not motion and is fine.

### 3.8 Existing dropdown-close-on-scroll

The shared `cotton/dropdown-menu.html` uses `x-on:scroll.window="close"`. Per the spec we leave this alone. If QA finds the auto-close too aggressive on the now-sticky header, the follow-up is to scope the rule to non-sticky triggers (out of scope here). Add a `# @claude` comment in `cotton/dropdown-menu.html` next to the listener noting the deferred decision so the next reader has context.
