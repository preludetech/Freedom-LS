# Spec: Token-efficient, structurally sound SDD workflow

## Overview

This spec tunes up the FLS Spec-Driven Development (SDD) workflow — the command and helper
files under `fls-claude-plugin/commands/sdd/` (and the top-level `commands/*.md` they invoke) —
so it is **more token-efficient, faster, and more robust**, **without changing what the workflow
produces** (same artifacts, same user-facing step sequence).

It is a **focused tune-up**, not a re-architecture. Migrating commands to skills and rethinking the
fresh-agent-per-step orchestration are explicitly **deferred to a separate follow-up idea** (see
Out of scope). Everything here is achievable on the current Claude Code 2.1.x line using only
stable, documented features.

Alongside tuning the command/helper files, this round also **extracts the verified Claude Code
behaviour into a durable skill** (`claude-code-authoring`) so the load-bearing constraints live in
one referenceable place instead of being re-stated in every command. The four research docs are the
scratch source material; the skill is the keeper (see D6 / §7).

Source of the changes: the idea file (`idea.md`) plus four verified research docs in this directory
(`research_subagent_capabilities.md`, `research_model_selection.md`, `research_cc_features.md`,
`research_resilience_patterns.md`), which checked current Claude Code behaviour against the live docs.

## Why this matters

The SDD command files are read into context — and inlined into subagent prompts — on **every run**.
Padding, duplicated instructions, and impossible asks are paid for repeatedly. Four concrete classes
of defect exist today:

1. **Instructions the executing context physically cannot perform.** Because subagents cannot nest,
   the single available fan-out level (main thread → subagent) is consumed by `next.md`'s
   fresh-subagent wrapper. When the wrapped command then tries to "create subagents to…"
   (`spec_from_idea.md` Step 1, `plan_from_spec.md` Steps 4 & 6, `improve_idea.md` Step 2), it is
   already at depth 1 and the fan-out **silently cannot run**. Subagents also cannot ask the user
   questions (`AskUserQuestion` is orchestrator-only) — yet several commands, while running inside
   `next.md`'s subagent wrapper, instruct that context to ask the user (`spec_from_idea` Step 2,
   `plan_from_spec`, `improve_idea` Step 3, `implement_plan`). (Subagents likewise cannot type slash
   commands; no command tells a subagent to do so today, but that constraint still bounds the redesign.)
2. **No model tiering.** Every step runs on the session model. Mechanical steps (running tests,
   commits, moving files, ticking the todo) pay Opus/Sonnet rates for Haiku-class work.
3. **Batch fragility and missing input contracts.** Where the workflow fans out work, one unit's
   failure can lose the whole batch, and subagents that discover they need user input mid-run have
   no way to get it.
4. **Hard-won Claude Code knowledge has no home.** The constraints that drive this whole redesign
   (subagents can't nest, can't type slash commands, can't ask the user; model tiering caveats;
   file-based hand-off) live only in disposable research docs. Without a durable skill, each new
   command/skill/agent author rediscovers them by trial and error, and the trimmed command files
   have nowhere to point when they stop explaining the constraints inline.

Plus correctness clean-ups surfaced while reading the files (see §6).

## Background facts that drive the design (from research)

These are verified and are the load-bearing constraints:

- **Subagents cannot nest.** Depth is capped at 1 (main → subagent). The `Agent`/`Task` tool is not
  exposed inside a subagent. All fan-out must be owned by whatever runs at depth 0.
- **Subagents cannot type slash commands.** There is no `SlashCommand` tool inside a subagent. Only
  the main session can invoke slash commands.
- **Subagents cannot prompt the user.** `AskUserQuestion` is not available inside a Task-spawned
  subagent; only the orchestrator/main session can ask. Input to a subagent is **spawn-time only** —
  there is no supported mid-run messaging (experimental Agent Teams `SendMessage` is out of scope).
- **Model is controllable per step.** Via per-agent `model:` frontmatter (`haiku`/`sonnet`/`opus`/
  full ID/`inherit`), per-command `model:` frontmatter, and the `CLAUDE_CODE_SUBAGENT_MODEL` env var.
  The env var **overrides all per-agent/per-command settings for every subagent** — so it flattens
  tiering and must not be set globally if per-step tiering is wanted. The Task-tool per-spawn `model`
  parameter only accepts the bare enum `sonnet|opus|haiku`.
- **`model:` frontmatter only switches the model when the file is *invoked* as a slash command or
  *spawned* as a subagent.** A command or helper whose text is run **inline** (read-and-followed by
  another context — which is how `next.md` runs `(cmd)` steps after D1, and how the `protected/`
  helpers are invoked) keeps the running context's model; its own `model:` frontmatter is inert. This
  is why model tiering is placed on **dedicated subagent files**, not on the command/helper files
  (see D2).
- **Structured JSON output is not enforced** for interactive Task subagents (SDK-only). Robust hand-off
  stays **file-based**, with a short agreed return string per unit.

## Goals / success criteria

A run is "done" when all of these hold:

1. **No impossible instructions.** No command instructs its executing context to do something it
   cannot: no nested-subagent fan-out asks, no subagent told to type a slash command, no subagent told
   to ask the user. Every such site is moved to the depth-0/orchestrator context or reworded.
2. **Model tiering by default, overridable.** Mechanical work (tests, commits, file moves,
   todo-ticking, setup) is delegated to a cheaper/faster model via dedicated subagents; non-interactive
   fan-out workers run on a mid-tier model; the interactive authoring/review commands run at depth 0 on
   the user's session model. The mechanism (per-agent `model:` frontmatter on dedicated subagent files)
   is documented and easy to override, and does **not** rely on the global env var.
3. **Batch resilience.** A single failed unit in any fan-out no longer forces the whole batch to
   restart: one subagent per unit, each writing its own output file, resume = skip units whose output
   already exists, retry only the failed units.
4. **Input contracts.** Each fan-out phase gathers the user input it needs **before** spawning;
   subagents are non-interactive and fail fast with a structured `blocked: needs X` result that the
   orchestrator can act on by re-spawning.
5. **Lower token overhead.** Command/helper files are trimmed of padding and duplication; hand-offs
   stay compact (file paths + short return summaries, not dumped file contents); no re-reading files
   just to "verify".
6. **Correctness.** Every `(cmd)` item in a generated `todo.md` maps to a command file that actually
   exists and is resolvable by `next.md`; the `/implement_plan` naming is consistent across files.
7. **A canonical Claude Code authoring skill exists.** A single skill (`claude-code-authoring`)
   documents the verified "works / doesn't work" patterns (no subagent nesting/fan-out, no slash
   commands from subagents, no `AskUserQuestion` in subagents, prefer skills over slash commands for
   subagent-invokable logic, per-step model tiering, file-based hand-off, fan-out resilience,
   input-contract + re-spawn). It is auto-discoverable via its `description`, and the SDD
   command/README files **reference it** rather than restating these rules inline.

Critically: **the produced artifacts (`1. spec.md`, `2. plan.md`, `3. frontend_qa.md`, `qa_report.md`,
`todo.md`) and the user-facing step order are unchanged.** This is a refactor of *how* the workflow
runs, not *what* it emits.

## Decisions (resolving the idea's open questions)

### D1 — Orchestration / nesting fix: run `(cmd)` steps at depth 0

`next.md` **stops spawning a fresh subagent** to wrap `(cmd)` steps. Instead it runs the next command
**on the main thread (depth 0)**. The clean-context guarantee the wrapper used to provide becomes the
**user's responsibility: run `/clear` before each `/sdd:next`.**

Concretely, `next.md` **reads the next command file and follows its steps inline** at depth 0, rather than
spawning a subagent to run it.

*Why:* subagents can't nest, so the wrapper was consuming the one fan-out level and silently breaking
each command's own research/review fan-out. Running the command inline at depth 0 — rather than one level
down in a wrapper subagent — keeps the single fan-out level free, so each command's own fan-out is legal
again, and it is simpler than tagging "leaf" vs "orchestrating" commands. The trade-off — losing automatic
context isolation between steps — is accepted and mitigated by the documented `/clear` step.

*Note on `next.md`'s tools:* because `next.md` now **executes** the inlined command's steps itself, it must
**keep the `Agent` tool** — it performs the worker/mechanic spawns the inlined command calls for — and
broaden its `allowed-tools` to the union those commands need (e.g. `Read, Glob, Bash, Write, Edit, Skill,
Agent`). (An earlier draft of this decision said `next.md` could *drop* `Agent`; that rested on the mistaken
assumption that `next.md` stops spawning entirely. Under inline execution it does the spawning on the
inlined command's behalf.) A corollary of inline execution: a command's own `allowed-tools`/`model:`
frontmatter is **inert** when the command is reached via `/sdd:next` (only `next.md`'s grants and session
model are live); those per-command frontmatter settings take effect only on **direct** invocation. `next.md`
must be updated so its description and prose no longer describe wrapping a command in a fresh subagent, and
it must instruct the user to `/clear` first.

### D2 — Model-selection mechanism: tier via dedicated subagent files, with a documented override

Tier defaults live in **`model:` frontmatter on dedicated subagent definitions** (`.claude/agents/*.md`),
where per-agent frontmatter is the robust, research-confirmed mechanism (already in use — both
`agents/qa-data-helper.md` and `agents/code-reviewer.md` set `model: opus`). Work that should run on a
cheaper or different tier is **delegated to one of these agents**. We do **not** put tier `model:`
frontmatter on the command/helper files themselves, because a command or helper whose text is run
*inline* (read-and-followed by another context — which is how `next.md` runs `(cmd)` steps after D1, and
how the `protected/` helpers are invoked) does **not** switch the model.

- **Haiku agent(s)** — mechanical work: running tests, git commits, file moves, todo-ticking, and the
  worktree/todo setup helpers' work (`update_todo`, `setup_todo_list`, `start_worktree`). The depth-0
  command delegates these to a Haiku agent rather than doing them itself. Optionally pair with
  `effort: low|medium`.
- **Sonnet agent(s)** — non-interactive fan-out workers: the research workers in
  `improve_idea`/`spec_from_idea`, the review/skill-scan workers in `plan_from_spec`, and the
  security/structure review workers.
- **Session model (no frontmatter tiering)** — the orchestrating commands themselves
  (`spec_from_idea`, `plan_from_spec` authoring, the review commands, and `next.md`'s routing). These
  run at depth 0 because they are interactive (they gather user input via `AskUserQuestion`, which
  subagents cannot use) and are run inline, so their model is the user's session model. The heavy
  authoring/reasoning therefore inherits the session model — the user selects a strong model (e.g.
  Opus/Sonnet) for the session — while cheap mechanical sub-work is pushed *down* into Haiku agents
  (the orchestrator-worker cost pattern). `next.md` is **not** given a Haiku `model:` of its own, since
  that would force the authoring/review commands it runs inline onto Haiku.

**The override knob:** a single documented note (in `commands/sdd/README.md`) explaining that a step's
model is changed by editing the relevant agent file's `model:` frontmatter, and that
`CLAUDE_CODE_SUBAGENT_MODEL` can force one model for *all* subagents in a pinch — with the explicit
warning that **it overrides all per-agent frontmatter** and so must be left unset (or `inherit`) for
normal tiered operation.

*Why this layer:* per-agent frontmatter is portable, version-controlled, already in use, and — unlike
command frontmatter run inline, or the Task per-spawn `model` parameter (which only accepts the three
bare aliases) — reliably switches the model for the delegated work. It is the mechanism Anthropic's own
cost guidance points to ("for simple subagent tasks, specify `model: haiku`").

*Open sub-decision for the plan/user:* whether to pin **dated full model IDs** (e.g. `claude-haiku-4-5`)
for reproducibility vs **aliases** (`haiku`) that track the recommended version. Default recommendation:
use aliases for readability, since this plugin is maintained alongside Claude Code and not a frozen
artifact — but call this out so the user can choose pinning if they want stable automation.

### D3 — Commands vs skills: do NOT migrate this round

Custom commands have largely merged into skills, and the robust way for a *subagent* to run reusable
workflow logic is a skill (subagents can invoke skills but not slash commands). **We do not migrate in
this round** — it is deferred to the follow-up idea. The impossible-instruction problems are still fixed
now (D1), by moving fan-out and slash-command invocation to depth 0/orchestrator, not by converting
commands to skills. The merger is noted as the key option for the follow-up.

### D4 — Mid-run input: input-contract + re-spawn, stable features only

Each fan-out phase declares, up front, the user input it needs. The **orchestrator gathers that input
before spawning** (it can use `AskUserQuestion`; subagents cannot) and bakes it into the subagent prompt.
A subagent that still discovers a gap returns a structured `blocked: needs [...]` result (with a pointer
to any partial output it wrote) instead of stalling; the orchestrator collects the input and **re-spawns
a fresh subagent** with the original brief plus the new answers. No experimental Agent Teams / live
resume.

### D5 — Target Claude Code version

Target the **current 2.1.x line** (the version the research verified against; per-agent/per-command
`model:` frontmatter, `AskUserQuestion`, non-nesting subagents, commands-merged-into-skills all hold
here). State this assumption in `commands/sdd/README.md` so future maintainers know which behaviour the
files depend on. We do **not** add runtime version checks (out of scope); this is a documented target,
not enforced.

### D6 — Capture Claude Code mechanics as a skill (source of truth)

The verified Claude Code behaviour that drives this redesign is distilled into **one skill**, not a
set: `claude-code-authoring`, living at `fls-claude-plugin/skills/claude-code-authoring/SKILL.md`
(one-dir-per-skill convention). A concise `SKILL.md` holds the do/don't patterns; deeper detail goes
in supporting files under that skill's `resources/` dir (precedent:
`skills/icon-usage/resources/*.md`).

- **One skill, not several.** A single trigger-rich `description` is easier to discover and maintain
  than several overlapping skill descriptions competing to auto-trigger.
- **Scope = generic mechanics + FLS authoring application.** It documents the platform constraints
  *and* how they shape authoring this plugin's commands/skills/agents (depth-0 orchestration, where
  model tiering lives, file-based hand-off, the input-contract pattern).
- **Source of truth.** The four `research_*.md` docs are the input; the skill is the distilled,
  maintained output. Trimmed SDD command/helper files and `commands/sdd/README.md` **link to the
  skill** instead of re-explaining constraints — this is part of the token-hygiene work (§5), not
  separate from it.
- **Frontmatter follows existing skills:** `name`, a trigger-rich `description` (so it surfaces when
  anyone authors/edits a command, skill, or agent, or asks how Claude Code subagents/skills/models
  behave), and a read-only `allowed-tools` (`Read, Grep, Glob`) like the other doc-style skills.

*Why a skill (not a command or a `resources/` doc):* a skill auto-triggers from its description
exactly when someone is authoring this kind of file, and — unlike a slash command — a **subagent can
invoke it** too. That makes it the right home for "how Claude Code works" knowledge.

## Scope — the changes

### 1. Fix impossible instructions (the nesting / wrong-context bug)

- **`next.md`:** stop wrapping `(cmd)` steps in a fresh subagent; **read each command file and follow its
  steps inline** at depth 0 on the main thread. This applies to **both** Agent-spawn sites in `next.md`: the
  `(cmd)` branch (Step 3) **and** the `/sdd:start` bootstrap in Step 1 (which today spawns a fresh agent to
  run `start.md` when no `todo.md` exists) — both must be reworked to run inline at depth 0 instead of in a
  wrapper subagent. Add a step telling the user to `/clear` before `/sdd:next`. Remove the "fresh agent per
  command" rationale, but **keep the `Agent` tool** (next.md now performs the inlined command's
  worker/mechanic spawns itself) and broaden `allowed-tools` to the union those commands need. (Note
  interaction with §6: command resolution must search the right directories.)
- **`spec_from_idea.md` (Step 1), `plan_from_spec.md` (Steps 4 & 6), `improve_idea.md` (Step 2),
  `implement_plan.md` (Steps 2–3):** these now run at depth 0, so their "create subagents to…" /
  "delegate to a sub-agent" instructions are legal. Rework each so the fan-out follows the resilience
  pattern (§3) and the input-contract pattern (§4): the command (at depth 0) is the orchestrator that
  gathers input, spawns one subagent per unit, and re-spawns on `blocked`. Note `implement_plan.md`'s
  `allowed-tools` is currently `Read, Glob, Write, Skill` — it omits `Agent` despite delegating to
  sub-agents, so its `allowed-tools` must be corrected to include the tools it actually needs at
  depth 0.
- **Audit pass:** check every command/helper for any remaining instruction the executing context can't
  perform (a subagent typing a slash command, a subagent asking the user). Move each to the
  depth-0/orchestrator context or reword it. Produce, as part of implementation, the full list of
  "impossible instruction" sites and how each was reworked (this is the idea's deferred per-command audit).

### 2. Model tiering

Apply D2: tiering lives on **dedicated subagent definitions** (`.claude/agents/*.md`) with `model:`
(and optional `effort:`) frontmatter. Extract the mechanical work (test-runner, commit, file moves,
todo-ticking, worktree/todo setup) into Haiku agent(s) that the depth-0 commands delegate to, and the
non-interactive fan-out research/review workers into Sonnet agent(s). Do **not** put tier `model:`
frontmatter on the command/helper files themselves (it does not switch the model when their text is run
inline); the interactive authoring/review commands run at depth 0 on the session model. Document the
override and the `CLAUDE_CODE_SUBAGENT_MODEL` warning in `README.md`.

### 3. Batch resilience (fan-out hardening)

Anywhere the workflow fans out (research in `improve_idea`; review/skill-scan in `plan_from_spec`;
batch implementation in `implement_plan`; any multi-dimension review), apply:

- **One subagent per unit**, never one subagent looping over the whole batch.
- **Each unit writes its own output file** at a deterministic path (write to a temp name then rename, so
  a crashed mid-write unit never leaves a half-file that looks complete).
- **Resume = skip units whose output already exists** and is marked ok; the orchestrator scans before
  spawning and only spawns missing/not-ok units.
- **Structured per-unit return:** each unit ends with a short fixed result the orchestrator can parse,
  e.g. `status: ok|failed|blocked`, `output_path`, `reason`.
- **Retry only failed units**, bounded (e.g. 2 attempts), including the prior error in the retry brief.
- Keep **work and synthesis separate** where it applies (workers write files; a later step reads them),
  so synthesis can be retried without re-running workers.

### 4. Input contracts

Apply D4 to each fan-out phase: declare required user input up front; orchestrator gathers it before
spawning; subagents are non-interactive and fail-fast with `blocked: needs [...]`; orchestrator
re-spawns with the answers.

### 5. Token hygiene

- Trim padding, repetition, and duplicated lines from all command/helper files (e.g. repeated
  "IMPORTANT" blocks, restated rules already in `CLAUDE.md`). Preserve every behavioural instruction and
  every TODO/`@claude` comment — trimming is about wording, not dropping requirements.
- Keep hand-offs compact: phases communicate through on-disk artifacts plus a short return summary; do
  not dump file contents into the orchestrator's context, and do not re-read files just to "verify".
- Prefer passing **file paths** over file contents between steps.
- Where command/helper files currently restate Claude Code constraints inline, replace the restatement
  with a one-line pointer to the `claude-code-authoring` skill (§7). The constraint explanations live
  once — in the skill — not duplicated across files.

### 6. Correctness clean-ups

- **`next.md` command resolution is broken for non-`sdd/` commands.** It resolves commands only at
  `fls-claude-plugin/commands/sdd/<name>.md`, but the generated `todo.md` references `/threat-model`,
  `/security-review`, and `/address_pr_review`, which live at `fls-claude-plugin/commands/<name>.md`
  (top level). Fix the resolution logic to search both the `sdd/` directory and the top-level
  `commands/` directory (and to fail clearly if a referenced command exists in neither).
- **`/implement_plan` naming.** `implement_plan.md` has `name: implement-plan` and no `description`; the
  todo and `next.md` resolve it by file name `implement_plan`. Reconcile so the checklist, the command's
  frontmatter, and `next.md`'s resolver all agree (give it a `description`, and ensure the resolvable
  name matches what the todo emits).
- **Verify the full `todo.md` template:** every `(cmd)` item must map to an existing, resolvable command
  file. Confirm `/improve_idea`, `/spec_from_idea`, `/spec_review`, `/threat-model`, `/plan_from_spec`,
  `/plan_security_review`, `/plan_structure_review`, `/implement_plan`, `/security-review`, `/do_qa`,
  `/address_pr_review`, `/finish_worktree` each resolve. Note: the README's "TODO: command for reacting
  to QA-reported bugs" stays a TODO — the QA bug-fix item in the todo is a `(user)`/`(user + cmd)` manual
  step, not a missing command, so it is not a resolution failure. Do not delete that TODO.

### 7. Author the `claude-code-authoring` skill

Apply D6: create one skill at `fls-claude-plugin/skills/claude-code-authoring/SKILL.md` (concise
do/don't patterns) plus `resources/*.md` for deeper detail. Distil it from the four `research_*.md`
docs already in this directory — **every documented behaviour must trace to a claim in those docs**;
do not invent capabilities. Required content:

- **Subagent capabilities & limits** (from `research_subagent_capabilities.md`): no nesting / no
  fan-out from a subagent (the `Agent` tool is unavailable at depth 1); no `SlashCommand` tool in
  subagents → subagents **cannot type slash commands**; input is **spawn-time-only**, no supported
  mid-run messaging (Agent Teams `SendMessage` flagged experimental / out of scope); subagents *can*
  auto-trigger and **preload** skills (`skills:` frontmatter); agent `.md` frontmatter fields
  (`name`, `description`, `tools`/`disallowedTools`, `model`, `skills`, `effort`, …) and the
  **model resolution order**.
- **Prefer skills to slash commands** for any logic a subagent must run — the headline rule — with
  the why (no `SlashCommand` tool inside subagents; skills are the only reusable-logic primitive a
  subagent can invoke).
- **Model tiering** (from `research_model_selection.md` + D2): per-agent `model:` frontmatter as the
  reliable knob; Haiku for mechanical work, Sonnet for non-interactive fan-out, the session model
  for interactive authoring at depth 0; the **inline-execution caveat** (a file whose text is run
  inline keeps the caller's model — its own `model:` frontmatter is inert); the
  `CLAUDE_CODE_SUBAGENT_MODEL` override-everything warning; the aliases-vs-pinned-IDs trade-off.
- **Orchestration / fan-out at depth 0** (from `research_resilience_patterns.md` + D1/D3/D4): the one
  fan-out level lives at depth 0; the resilience pattern (one subagent per unit, each writes its own
  output file, resume = skip units already done, structured pass/fail/blocked return, retry only
  failures); the input-contract + re-spawn pattern for work that needs `AskUserQuestion`.
- **Interactive-CLI feature notes** (from `research_cc_features.md`): `AskUserQuestion` is
  orchestrator-only (not in subagents); structured-JSON output is SDK-only → use **file-based
  hand-off** in the slash-command flow; brief pointers to hooks / plan mode / permission modes as
  **"exists, deferred"** reference so authors know they exist — do **not** prescribe adding them
  (that stays in the follow-up; see Out of scope).
- **Cross-references:** the skill links back to `commands/sdd/README.md` and the agent files
  (`agents/code-reviewer.md`, `agents/qa-data-helper.md`) as living examples; the README and the
  trimmed commands link forward to the skill.
- **Anchor to the target version:** state the Claude Code 2.1.x target (per D5) inside the skill so
  its claims are pinned to verified behaviour, mirroring the README note.

## Out of scope

- Migrating commands to skills (deferred to the follow-up idea).
- Experimental Agent Teams / `SendMessage` / resumable live subagents.
- Changing the *outputs* of the workflow (spec/plan/QA artifacts) or the user-facing step sequence.
- Adding new enforcement infrastructure (hooks for test-running / artifact validation, `settings.json`
  permission modes, structured-output schemas). The research surfaces these as high-value, but they
  change behaviour beyond a tune-up and belong to the follow-up re-architecture. Do not add them here.
- Runtime Claude Code version checks (the target version is documented, not enforced).
- Implementing the advanced features the new skill *documents*. The `claude-code-authoring` skill
  **describes** hooks, plan/permission modes, and structured outputs as "exists, deferred" reference —
  documenting them does not mean we adopt them this round; that stays in the follow-up re-architecture.

## Risks / trade-offs

- **`/clear` is now manual.** If the user forgets to `/clear` before `/sdd:next`, context from the
  previous step leaks into the next. Mitigation: the `/clear` reminder is part of `next.md`'s report and
  the README; the cost is degraded efficiency, not incorrectness.
- **The `claude-code-authoring` skill can drift from Claude Code's behaviour.** It captures verified
  2.1.x behaviour; platform changes can age it. Mitigation: anchor every claim to the documented target
  version (D5), and keep the skill the single place to update — so drift is fixed in one file rather
  than across many command/helper files.
- **Model aliases drift.** Using aliases (`haiku`/`sonnet`/`opus`) means the resolved model can change as
  Anthropic advances the pointers. Mitigation: documented; users who want frozen behaviour can pin dated
  IDs (D2 sub-decision).
- **Authoring quality tracks the session model.** Because the interactive authoring/review commands run
  at depth 0 on the user's session model (not a pinned Opus), running the SDD workflow on a weak session
  model degrades spec/plan quality. This is a deliberate trade-off of the dedicated-agent-files approach
  (D2): authoring must stay at depth 0 to use `AskUserQuestion`, so it cannot be pinned via a subagent's
  `model:` field. Mitigation: document that the session should run on a strong model; only mechanical and
  fan-out work is delegated to cheaper agents.
- **Trimming could drop a requirement.** Mitigation: trimming is wording-only; every behavioural
  instruction and TODO/`@claude` comment is preserved, and `spec_review`/the existing review steps still
  run against the unchanged success criteria.
