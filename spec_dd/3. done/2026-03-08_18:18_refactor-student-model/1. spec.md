# Spec: Remove the Student Model

## Overview

Remove the `Student` model entirely from the codebase. All models that currently have a ForeignKey to `Student` should instead FK to `User`. The `User` model must not be changed in any way.

The `Student` model (`student_management/models.py`) is a thin profile wrapper around `User` with three extra fields (`id_number`, `date_of_birth`, `cellphone`) and a ForeignKey to `User`. These fields are no longer needed and will be dropped. Progress models (`CourseProgress`, `TopicProgress`, `FormProgress`) and `RecommendedCourse` already point directly to `User`, making `Student` an unnecessary intermediary.

---

## Decisions Made

### Models will be renamed

Models that reference "Student" in their name will be renamed for clarity:

| Current Name | New Name | Reason |
|---|---|---|
| `StudentCourseRegistration` | `UserCourseRegistration` | FK changes from Student to User; name should reflect this |
| `StudentCohortDeadlineOverride` | `UserCohortDeadlineOverride` | Same reason |
| `CohortMembership` | `CohortMembership` | No change needed; does not reference Student in its name |

Table renames will be handled via `db_table` Meta option or `RenameModel` migration operations.

### Educator interface will list Users, not Students

The educator interface currently uses `Student` as the list/detail model. After removal, it will list all `User` objects (regardless of registration status). The UI language will change from "students" to "users". URL paths will change from `/students/` to `/users/`.

### App names stay the same

The `student_management` and `student_interface` app names will not change. Renaming Django apps is extremely complex (table prefixes, migration history, imports) and the benefit doesn't justify the risk. The `student_interface` name still makes conceptual sense as it serves users in their learner role.

### Student-specific fields are dropped entirely

The `id_number`, `date_of_birth`, and `cellphone` fields on `Student` are not migrated anywhere. They are simply removed.

### Student methods become standalone functions

Student model methods will be relocated to standalone utility functions that accept a `User` parameter. This follows the principle of preferring simple functions over mixins or model method bloat.

---

## What Needs to Change

### 1. Models with FK to Student

These models must have their `student` FK changed to a `user` FK pointing to `User`:

- `CohortMembership.student` -> `CohortMembership.user`
- `StudentCourseRegistration.student` -> `UserCourseRegistration.user` (model also renamed)
- `StudentCohortDeadlineOverride.student` -> `UserCohortDeadlineOverride.user` (model also renamed)

### 2. Unique Constraints

These constraints reference `student` and must be updated to reference `user`:

- `unique_student_course_registration` (fields: `site_id`, `collection`, `student`) -> update to use `user`
- `unique_student_cohort_override_per_item` (fields: `cohort_course_registration`, `student`, `content_type`, `object_id`) -> update to use `user`

Constraint names should also be updated to replace "student" with "user".

### 3. Student Model Methods

The following methods on `Student` need to become standalone functions (in `student_management/utils.py` or `student_interface/utils.py` as appropriate):

| Method | New Function | Location |
|---|---|---|
| `get_course_registrations()` | `get_course_registrations(user: User)` | `student_management/utils.py` |
| `completed_courses()` | `get_completed_courses(user: User)` | `student_interface/utils.py` |
| `current_courses()` | `get_current_courses(user: User)` | `student_interface/utils.py` |

These methods already use `self.user` internally to query progress, so the conversion is straightforward.

### 4. Code References to Update

#### `student_interface/utils.py`
- `get_student(user)` — remove entirely; callers should use `User` directly
- `get_is_registered(user, course)` — update to query `UserCourseRegistration` and `CohortMembership` directly via `user`
- `get_completed_courses(user)` — update to query `CourseProgress` directly via `user`
- `get_current_courses(user)` — update to query progress directly via `user`
- `get_course_index()` — update to use `user` for deadline lookups

#### `student_management/deadline_utils.py`
- All deadline functions currently accept `student: Student`. Change to accept `user: User`.
- Internal queries use `student` FK on `CohortMembership`, `UserCourseRegistration`, and `UserCohortDeadlineOverride` — update to use `user` FK.

#### `educator_interface/views.py`
- `StudentConfig` -> rename to `UserConfig`, change model from `Student` to `User`
- `StudentDataTable` -> rename to `UserDataTable`, update search fields (remove `user__` prefix since fields are now directly on `User`)
- `StudentInstanceView` -> rename to `UserInstanceView`
- Update all panels/views that reference student relationships
- Change URL paths from `/students/` to `/users/`
- Update URL names from `student_*` to `user_*`

#### Admin
- Remove `StudentAdmin` entirely
- Update `StudentCourseRegistrationAdmin` -> `UserCourseRegistrationAdmin`: change `student__user` references to `user`
- Update `StudentCohortDeadlineOverrideAdmin` -> `UserCohortDeadlineOverrideAdmin`: change student references to `user`
- Update any inline admin classes that reference `Student`

#### Factories
- Remove `StudentFactory` entirely
- Update `StudentCourseRegistrationFactory` -> `UserCourseRegistrationFactory`: use `UserFactory` directly for the `user` field
- Update `StudentCohortDeadlineOverrideFactory` -> `UserCohortDeadlineOverrideFactory`: same
- Update `CohortMembershipFactory`: use `UserFactory` directly for the `user` field
- Any other factories that reference `StudentFactory` should use `UserFactory`

#### Templates
- Update any template references from `student.user.email`, `student.user.first_name`, etc. to `user.email`, `user.first_name`, etc.
- Update template references from "student" to "user" in display text

#### Tests
- Remove all tests that test the `Student` model directly
- Update all tests that create `Student` objects to create `User` objects instead
- Update all assertions that reference `student` fields/relationships

## Migration Strategy

Use a multi-step migration approach. At every step, the database must be in a valid, deployable state.

### Pre-migration check

Before running any migrations, verify no `User` has multiple `Student` records:

```sql
SELECT user_id, COUNT(*) FROM freedom_ls_student_management_student GROUP BY user_id HAVING COUNT(*) > 1;
```

If duplicates exist, deduplication must happen first.

### Step 1: Add nullable `user` FK (schema migration)

Add a new nullable `user` ForeignKey to `CohortMembership`, `StudentCourseRegistration`, and `StudentCohortDeadlineOverride`. This is a pure additive schema change.

```python
user = models.ForeignKey(
    settings.AUTH_USER_MODEL,
    on_delete=models.CASCADE,
    null=True,
    blank=True,
)
```

### Step 2: Data migration

Create a data migration that populates `user_id` from `student.user_id` for all rows in the three affected models. Use raw SQL for performance:

```python
def forward(apps, schema_editor):
    schema_editor.execute("""
        UPDATE table_name SET user_id = s.user_id
        FROM freedom_ls_student_management_student AS s
        WHERE table_name.student_id = s.id
    """)
```

### Step 3: Make `user` FK non-nullable, remove `student` FK (schema migration)

- Alter the `user` field to be non-nullable (`null=False`)
- Remove the `student` FK field from all three models
- Update unique constraints to reference `user` instead of `student`

### Step 4: Delete the Student model (schema migration)

- Remove the `Student` model class from `models.py`
- Run `makemigrations` to generate the `DeleteModel` operation

### Step 5: Rename models (schema migration)

- Rename `StudentCourseRegistration` to `UserCourseRegistration`
- Rename `StudentCohortDeadlineOverride` to `UserCohortDeadlineOverride`
- Use `RenameModel` migration operations (Django handles table renames automatically)

### Step 6: Code cleanup

- Update all Python code (views, utils, admin, factories, tests)
- Update all templates
- Update URL paths and names
- Remove orphaned imports

## Success Criteria

1. **No `Student` model exists** — the model class is gone, the database table is dropped
2. **All FK relationships point to User** — `CohortMembership.user`, `UserCourseRegistration.user`, `UserCohortDeadlineOverride.user` all FK to `User`
3. **All tests pass** — the full test suite passes with `--create-db` to verify migration history consistency
4. **No stale references** — grepping for `Student` (as a model reference), `student_id`, `student__`, `.student` finds no hits outside of migration files and app/module names (`student_management`, `student_interface`)
5. **Educator interface works** — lists all Users, URL paths use `/users/`, UI text says "users" not "students"
6. **Admin interface works** — no errors when loading admin pages for renamed models
7. **Migrations are reversible** — steps 1-2 can be reversed without data loss
8. **No data loss** — all existing `CohortMembership`, registration, and deadline override records are preserved with correct `user_id` values
9. **Models are renamed** — `StudentCourseRegistration` is now `UserCourseRegistration`, `StudentCohortDeadlineOverride` is now `UserCohortDeadlineOverride`
