def viewable_items(self) -> list:
    """Return ordered list of all viewable child content items (no CoursePart sentinels)."""
    return [item for item in self.children_flat() if not isinstance(item, CoursePart)]

## Step 6 — Update TOC index builder (GREEN)

In `freedom_ls/student_interface/utils.py`:

`create_child_dict_with_flattened_index` and `get_course_index` change as follows. The `start_index` parameter becomes the **viewable** running index, and the function returns `items_added` = number of **viewable items** consumed.

Inside the `CoursePart` branch:

- The CoursePart itself does not increment the counter (`items_added` starts at 0, not 1).
- Each viewable child increments the counter; the child's URL uses `start_index + items_added + 1` (1-based).
- `nested_index` is removed; just use `start_index + items_added + 1` while iterating children.
- The CoursePart row's `url` is set from the URL of its **first viewable child** (`part_children_dicts[0]["url"]`), per the spec — not from the IN_PROGRESS or READY child. Replace today's three-way pick (`in_progress_child["url"]` / `ready_child["url"]` / `part_children_dicts[0]["url"]`) with a single source: the first viewable child.
- Status calculation is unchanged (still based on children's IN_PROGRESS / READY / all-COMPLETE).
- Preserve today's BLOCKED-row behaviour: the existing `child_dict["url"] if status != BLOCKED else None` guard at the end keeps a fully-BLOCKED part's `url` as `None`.
- If a CoursePart has no viewable children, `part_children_dicts` is empty, status falls through to `BLOCKED`, and `url` is `None` — matching the spec's "if a CoursePart has no viewable children, its `url` is `None`."

Inside the non-CoursePart (regular item) branch:

- `items_added = 1`.
- URL uses `start_index + 1` (1-based) — unchanged in form, but now `start_index` is in the viewable-only space.

In `get_course_index`:

- `global_index` is now the running viewable-item count.
- The existing `global_index += items_added` line is unchanged in shape but `items_added` no longer counts the `CoursePart` itself.

Pseudocode for the CoursePart branch:

```python
part_children = content_item.children()
part_children_dicts = []
items_added = 0  # CoursePart itself consumes no URL slot

for part_child in part_children:
    if isinstance(part_child, CoursePart):
        # Defensive: today's data model does not nest parts; skip URL allocation
        # for any unexpected nested CoursePart and let status logic ignore it.
        continue
    if is_registered:
        child_status, part_next_status = get_content_status(part_child, user, part_next_status)
        child_url = reverse(
            "student_interface:view_course_item",
            kwargs={"course_slug": course.slug, "index": start_index + items_added + 1},
        )
    else:
        child_status, child_url = BLOCKED, ""
    ...build part_child_dict...
    part_children_dicts.append(part_child_dict)
    items_added += 1
```

Then the CoursePart's own `url`, when not BLOCKED, is `part_children_dicts[0]["url"]` (which is the first viewable child's URL).

The "skip nested CoursePart" branch is a defensive comment, not new behaviour — `children_flat()` flattens only one level today and the data model does not nest parts.

## Notes / non-obvious points

- `viewable_items()` is implemented as a filter over `children_flat()` rather than a parallel walk, so any future changes to course structure traversal stay in one place.
- The deadline-check guard `not isinstance(current_item, CoursePart)` in `view_course_item` is no longer needed and should be removed at the same time as the redirect-forward branch — leaving it in would be dead code.
- `course_form_complete` does not render a Previous button, so its only use of `children` is to compute `is_last_item`. The semantics of `is_last_item` are preserved by `len(viewable_items)` because the bound was always "is this the last item the user can navigate to," and a `CoursePart` was never a destination in practice (the redirect papered over it).
- The `educator_interface/views.py:_paginate_course_items` walker stays on `children_flat()`. It groups items under part headers and genuinely needs the sentinels — explicitly out of scope.
- Out-of-range index handling (today's `IndexError` → 500) is unchanged, per the spec's "Out-of-range indices" section.
