# Spec — Fix course part navigation bugs by re-indexing to viewable items

## Background

Courses in FLS can be organised into `CoursePart`s, each holding a sequence of viewable items (topics, forms, activities). The student-facing item view is `view_course_item` in `freedom_ls/student_interface/views.py`, addressed by an `index` into the flattened item list returned by `Course.children_flat()` in `freedom_ls/content_engine/models.py`.

`children_flat()` interleaves `CoursePart` markers with their child items in the flattened sequence, so `CoursePart`s occupy real URL indices. They are not directly viewable — when `view_course_item` resolves to a `CoursePart`, it redirects forward to the next item.

The current navigation logic computes:

- `previous_url` = `index - 1` (when `index > 1`)
- `next_url` = `index + 1` (when `index < total`)

Because `CoursePart` markers occupy real indices, both URLs can land on a `CoursePart`, producing the two documented bugs and one near-bug:

1. **Bug 1 — Previous on the very first item.** At `index=2` (first viewable item of a course whose `index=1` is a `CoursePart` header), the Previous button is rendered and points at `index=1`. Visiting `index=1` redirects forward to `index=2`, so Previous silently bounces to the same page.
2. **Bug 2 — Previous on the first item of a non-first part.** At the first item of any subsequent part, Previous points at the preceding `CoursePart` header, which redirects forward to the same item the user was already on. Net effect: nothing happens.
3. **Latent symmetric issue on Next.** The last item of a part has `next_url = index + 1`, which is the next part's `CoursePart` header. The redirect-forward rule papers this over visually, but it costs the user an extra HTTP round trip and reuses fragile redirect plumbing for what should be a single direct link.

The root cause is the index scheme itself: URL indices count `CoursePart` sentinels as if they were viewable items. The fix removes that mismatch.

## Goals

- Course item URL indices count **viewable items only**. Indices `1..N` map directly to the N viewable items of the course in order.
- Previous and Next buttons always link to a directly viewable item, or are not rendered at all when no such item exists in that direction. They become trivial `index ± 1`.
- The redirect-forward branch in `view_course_item` (for landing on a `CoursePart`) goes away — `CoursePart`s no longer have URL slots in this scheme.
- The Table of Contents continues to show `CoursePart` headers and their children, with each `CoursePart` linking to its first viewable child.
- Bug fix is TDD-driven.

## Non-goals

- Preserving previously-issued URLs that encoded `CoursePart`-inclusive indices. URL-stability is not a constraint for this change.
- Removing `Course.children_flat()`. It is still needed by code that genuinely iterates over the structure (educator pagination groups items under part headers).
- Changing how courses are authored, how `CoursePart`s are stored, or any model/migration changes.
- Adding new navigation surfaces (breadcrumbs, part jumpers, etc.).

## Design

### Add `Course.viewable_items()` returning the flat list of viewable items only

In `freedom_ls/content_engine/models.py`, add:

```python
def viewable_items(self) -> list:
    """Return ordered list of all viewable child content items (no CoursePart sentinels)."""
    return [item for item in self.children_flat() if not isinstance(item, CoursePart)]
```

This is the new index space for course item URLs: position `i` (1-based) in this list maps to URL `index=i`.

`children_flat()` is unchanged.

### Switch URL-indexed call sites to `viewable_items()`

The four view-layer sites that currently resolve a URL `index` to a content item via `course.children_flat()[index - 1]` switch to `course.viewable_items()[index - 1]`:

- `student_interface/views.py:view_course_item`
- `student_interface/views.py:form_start`
- `student_interface/views.py:form_fill_page`
- `student_interface/views.py:course_form_complete`

The TOC index builder in `student_interface/utils.py` is a separate kind of change (it builds URLs by counting, not by indexing into a list) and is covered in its own section below.

In `view_course_item`:

- `previous_url` becomes simply `index - 1` (when `index > 1`).
- `next_url` becomes simply `index + 1` (when `index < len(viewable_items)`).
- The `isinstance(current_item, CoursePart)` redirect-forward branch is **removed** — there is no longer any URL that resolves to a `CoursePart`.

### Update the TOC index builder

`student_interface/utils.py:create_child_dict_with_flattened_index` and its caller `get_course_index` currently increment a `global_index` across `CoursePart` sentinels so that URLs line up with `children_flat()`. After this change:

- The index counter only increments for viewable items.
- A `CoursePart` itself does **not** consume an index slot.
- A `CoursePart`'s TOC entry gets its `url` from its **first viewable child** (i.e. the same destination today's redirect-forward lands on, just without the redirect hop). If a `CoursePart` has no viewable children, its `url` is `None`.

This preserves the existing TOC behaviour (CoursePart rows are clickable and take you to the first item under them) while removing the redirect-forward mechanism.

### Educator pagination keeps `children_flat()`

`educator_interface/views.py:_paginate_course_items` walks the interleaved list to build part-header spans for the educator gradebook. It does not use URL indices and is unaffected. It continues to call `children_flat()`.

### QA helper commands

- `qa_helpers/management/commands/qa_add_course_items_for_pagination.py:39` currently does `sum(1 for c in course.children_flat() if isinstance(c, Topic | Form))`. Replace with `len(course.viewable_items())`.
- `qa_helpers/management/commands/qa_create_cohort_progress.py:138` iterates `children_flat()`. If the iteration only acts on viewable items, switch to `viewable_items()`; otherwise leave as-is. The implementer should make the smaller of the two changes based on what the code actually does.

### Why this approach

- **Removes the bug class, doesn't paper over it.** The two bugs and the latent Next round-trip are all symptoms of the same modelling mistake: URL indices counting non-viewable sentinels. Fixing the index scheme removes the symptoms structurally.
- **Simplifies `view_course_item`.** Previous/Next become `index ± 1` again; the CoursePart-redirect branch disappears.
- **Keeps `children_flat()` for the one place that needs it.** Educator pagination genuinely operates on the interleaved structure; that code does not change.
- **TOC behaviour preserved end-to-end.** CoursePart rows still link to "the first thing under this part," just via a direct URL instead of a redirect.

### Why not the alternative (skip-in-view)

A smaller fix would leave the index scheme alone and have `view_course_item` walk past `CoursePart` slots when computing prev/next URLs. That keeps the bugs' root cause (mismatched index space) in place: every consumer of `children_flat()` that uses indices still has to know about CoursePart sentinels, the redirect-forward branch stays, and the latent extra-redirect issue on Next remains unless duplicated logic is added. The user has confirmed URL stability is not a constraint, so the cleaner re-index is preferred.

## Behaviour summary

| Scenario | Today | After fix |
|---|---|---|
| URL index space | Counts `CoursePart` sentinels as items | Counts viewable items only |
| First viewable item of course (course starts with a part header) | URL `index=2`, Previous shown and bounces | URL `index=1`, no Previous button (`previous_url is None`) |
| First viewable item of a non-first part | Previous points at part header, redirects to same item | Previous points directly at last viewable item of previous part |
| Last viewable item of a part (not last in course) | Next points at next part header, redirects forward | Next points directly at first viewable item of next part |
| Last viewable item of the course | Next button hidden | Unchanged |
| Direct visit to a `CoursePart` URL (today: `index=1`, `index=4`, etc.) | Redirects forward to first viewable child | No such URL exists; visiting an out-of-range index is a 404 from `IndexError` handling (see below) |
| TOC row for a `CoursePart` | Clickable; redirects forward to first child | Clickable; links directly to first child (one fewer redirect) |

### Out-of-range indices

`view_course_item` and the form views currently do `children[index - 1]` with no bounds check, so a too-large `index` raises `IndexError` (HTTP 500). This bug exists today and is **out of scope** for this spec; the implementer should not introduce a fix or change behaviour here. After the re-index, the largest valid index drops (since CoursePart slots are gone), but the failure mode is identical.

## Testing (TDD)

Write failing tests first, then implement. Per project convention, test functionality (URL values, view responses, model output), not styling. The existing `demo_content/functionality_demo_course_parts/` fixture (3 parts) is suitable test data.

Required test cases:

1. **`Course.viewable_items()` excludes CoursePart sentinels.** Returns the same items as `children_flat()` filtered to non-`CoursePart` instances, in the same relative order.
2. **First viewable item — no Previous.** At `index=1` of a course that begins with a part, the response context has `previous_url is None` and the rendered template does not contain the Previous button.
3. **First viewable item of a non-first part — Previous links to last item of previous part.** `previous_url` is `index - 1`, which resolves to the last viewable item of the preceding part. Following the URL returns 200 and renders the expected topic/form.
4. **Middle of a part — regression guard.** Previous and Next at a middle viewable index point to the immediately adjacent viewable items (`index - 1` and `index + 1`).
5. **Last item of a part (not last of course) — Next links to first item of next part.** `next_url` is `index + 1`, which resolves to the first viewable item of the next part. Following the URL returns 200 (no redirect chain).
6. **Last item of course — regression guard.** `next_url is None` at the final viewable index.
7. **TOC: CoursePart row links to first viewable child.** `get_course_index` returns a `CoursePart` dict whose `url` resolves (via the URL resolver) to the same `index` as the part's first viewable child.
8. **TOC: viewable indices are dense.** The URLs of consecutive viewable items in the TOC differ by exactly 1 in their `index` kwarg.

The existing tests in `freedom_ls/student_interface/tests/test_view_course_item.py` all hit `index=2` to reach the first item inside a leading `CoursePart` (the structure under test is: `CoursePart` at slot 1, first item at slot 2, second item at slot 3, third item at slot 4). After the re-index, those items shift to indices 1, 2, 3 respectively, so the affected tests (`test_accessing_topic_inside_course_part_should_display_topic`, `test_starting_form_inside_course_part_should_work`, `test_view_course_item_anonymous_redirects_to_login`, `test_form_start_anonymous_redirects_to_login`, `test_form_complete_inside_course_part_should_work`) must have their `index=2` updated to `index=1` to match the new viewable-only index space. No test asserts the obsolete CoursePart redirect-forward behaviour, so nothing needs deletion — only renumbering.

## Success criteria

- All test cases above pass.
- Manual reproduction of the two reported bug URLs (mapped to their new viewable-only indices in `functionality_demo_course_parts`) shows the corrected behaviour: Previous absent on the first viewable item, Previous landing on the previous part's last item on the first item of a subsequent part.
- No new pre-commit, type, or existing test failures (after removing/rewriting the obsolete CoursePart-redirect test).
- Changes confined to:
  - `freedom_ls/content_engine/models.py` (add `viewable_items`)
  - `freedom_ls/student_interface/views.py` (4 call sites + simplified prev/next)
  - `freedom_ls/student_interface/utils.py` (TOC index builder)
  - `freedom_ls/qa_helpers/management/commands/qa_add_course_items_for_pagination.py` (1 line)
  - `freedom_ls/qa_helpers/management/commands/qa_create_cohort_progress.py` (only if iteration is purely over viewables)
  - Tests for the above.
- No template, URL config, or migration edits.
