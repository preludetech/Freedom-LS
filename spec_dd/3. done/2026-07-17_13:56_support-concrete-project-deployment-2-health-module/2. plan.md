# Plan: P1 importable health module

Implementation plan for `1. spec.md` — an importable `freedom_ls.health` package giving downstream
concrete projects liveness (no deps) + readiness (DB) probes with zero `HEALTH_CHECK` config, plus
the paired importable `SECURE_REDIRECT_EXEMPT` default.

Follow **TDD**: for each unit, write the failing test first, then the implementation. Do not
pre-write every test up front.

## Design summary

`django-health-check` (revsys/codingjoe, Django 6 compatible) is the check-execution **engine**;
FLS ships a thin, opinionated wrapper `freedom_ls.health` around it.

> **Engine-API note (v4):** `django-health-check` **v4.0.0 removed** the plugin-registry API
> (`health_check.plugins.plugin_dir`, `health_check.views.MainView`,
> `health_check.db.backends.DatabaseBackend`, and the per-service opt-in apps `health_check.db` /
> `health_check.cache` / `health_check.storages`). v4 replaces it with a declarative
> `health_check.views.HealthCheckView` whose `checks` attribute is a tuple of dotted-path check
> classes (defaults: `Cache`, `Database`, `DNS`, `Mail`, `Storage`). This plan targets **v4** (pinned
> `>=4`); the redesign below owns the FLS default check set on the view class / an FLS setting instead
> of an `AppConfig.ready()` registry call — which is *simpler* and removes the app-label footgun the
> old design guarded against, since there is no registry to forget to populate.

- **liveness** — a plain FLS view that always returns 200 and touches **no** dependencies. It is a
  hand-written view (not a `django-health-check` subset) so that liveness is *structurally* incapable
  of checking the DB or any backend — this hardcodes the "liveness must never touch a dependency"
  rule rather than leaving it as a settings footgun (research §2).
- **readiness** — an FLS `ReadinessView(HealthCheckView)`, which runs the checks named in its `checks`
  attribute. FLS **ships its own default check set** by reading that list from an app-owned setting
  `HEALTH_READINESS_CHECKS` (declared via `freedom_ls.base.app_settings`, default
  `["health_check.checks.Database"]`), so out of the box readiness == a DB connectivity check and
  nothing else — with **no** `HEALTH_CHECK` config and no reliance on the downstream's
  `INSTALLED_APPS`. Owning the default in FLS code is what makes readiness the "FLS-chosen default
  check set" the spec asks for. A downstream that wants cache/storage checks opts in **in its own
  settings** by setting `HEALTH_READINESS_CHECKS` (e.g.
  `["health_check.checks.Database", "health_check.checks.Cache", "health_check.checks.Storage"]`) —
  the intended opt-in, matching the spec's "add S3/cache checks in their own settings"; liveness
  stays check-free because it is a separate view.

This gives both probes with **zero** `HEALTH_CHECK` settings block downstream — a downstream only adds
`health_check` (core machinery/templates) and `freedom_ls.health` to `INSTALLED_APPS` and
`include()`s the URLconf — plus a clean upgrade path (override `HEALTH_READINESS_CHECKS` in the
project's own settings). No `HEALTH_CHECK["SUBSETS"]` config is needed.

Applied-migrations is deliberately **not** a readiness check — it belongs in the deploy smoke test as
`manage.py migrate --check` (spec §4; research §2). Nothing to build here for it.

## Tasks

Ordered so each builds on the last. Each code unit is test-first.

### 1. `SECURE_REDIRECT_EXEMPT` importable default

- **Test** (`freedom_ls/deployment/tests/test_settings_defaults.py`): assert
  `settings_defaults.SECURE_REDIRECT_EXEMPT == [r"^health/"]` (mirror the style of
  `test_conn_health_checks_constant_is_true`).
- **Implement** in `freedom_ls/deployment/settings_defaults.py` — a flat module-level constant beside
  `SECURE_PROXY_SSL_HEADER`:

  ```python
  # Path-prefix regexes exempt from SECURE_SSL_REDIRECT's 301→https. Internal probes
  # hit /health/… over plain HTTP (Docker healthcheck / smoke test); paired with the
  # proxy-header setting, a naive probe would otherwise read the 301 as unhealthy.
  # Exempting the prefix serves those paths plainly. Assigned in config/settings_prod.py
  # and kept greppable there rather than mutated invisibly from an AppConfig.
  SECURE_REDIRECT_EXEMPT: list[str] = [r"^health/"]
  ```

  (The comment keeps the *reasoning* but no spec-section citation, per `fls:code-comments`.)

- **Assign** in `config/settings_prod.py`, in the `--- HTTPS Enforcement ---` block:

  ```python
  SECURE_REDIRECT_EXEMPT = fls_defaults.SECURE_REDIRECT_EXEMPT
  ```

### 2. Register the engine in `INSTALLED_APPS`

In `config/settings_base.py` `INSTALLED_APPS`, add the `django-health-check` **core** app (its
`HealthCheckView`, check classes, and dashboard templates) — the FLS wrapper app `freedom_ls.health`
is added in Task 3:

```python
"health_check",
```

That single app is all the engine needs in v4 (it ships every check class — `Database`, `Cache`,
`Storage`, … — inside `health_check.checks`; there are no per-service sub-apps to install). FLS owns
the *default check set* via the `HEALTH_READINESS_CHECKS` setting (Task 3), not via this app list.

### 3. New `freedom_ls/health/` package

Create the package mirroring the `freedom_ls/deployment` shape. It has **no models**, so no
migrations. Use the **`fls:app-settings`** skill for the `config.py` below.

Files:

- `freedom_ls/health/__init__.py` — empty.
- `freedom_ls/health/apps.py` — a minimal `AppConfig`; **no `ready()` registration** (v4 has no
  plugin registry to populate):

  ```python
  from django.apps import AppConfig

  class HealthAppConfig(AppConfig):
      name = "freedom_ls.health"
  ```

- `freedom_ls/health/config.py` — the FLS-owned default readiness check set, declared as an app
  setting (mirrors `freedom_ls/course_access/config.py`). Default is DB-only; downstream overrides
  `HEALTH_READINESS_CHECKS` in its own Django settings to add cache/storage checks:

  ```python
  from __future__ import annotations

  from freedom_ls.base.app_settings import AppSettings, Setting

  class HealthConfig(AppSettings):
      HEALTH_READINESS_CHECKS: list[str]

      declared_settings = {
          "HEALTH_READINESS_CHECKS": Setting(
              default=["health_check.checks.Database"]
          ),
      }

  config = HealthConfig()
  ```

- `freedom_ls/health/views.py` — the check-free liveness view plus the readiness view whose check set
  is the FLS default (read at request time so a downstream override takes effect):

  ```python
  from django.http import HttpRequest, JsonResponse
  from health_check.views import HealthCheckView

  from freedom_ls.health.config import config

  def liveness(request: HttpRequest) -> JsonResponse:
      """Shallow liveness probe. Returns 200 whenever the process can serve a
      request; touches no database, cache, or other dependency by design."""
      return JsonResponse({"status": "alive"})

  class ReadinessView(HealthCheckView):
      """Readiness probe: runs the FLS default check set (DB-only unless a project
      overrides HEALTH_READINESS_CHECKS). Non-200 when a checked dependency is down."""

      @property
      def checks(self) -> list[str]:
          return config.HEALTH_READINESS_CHECKS
  ```

  (Confirm the exact `HealthCheckView` import path against the installed `django-health-check` during
  implementation — verified `health_check.views.HealthCheckView` in 4.4.3; the TDD tests below pin the
  observable behaviour regardless. `checks` is a `@property` overriding the base class attribute so
  the list is resolved per request from the setting.)

- `freedom_ls/health/urls.py` — fixed two-path URLconf, namespaced:

  ```python
  from django.urls import path

  from freedom_ls.health import views

  app_name = "health"

  urlpatterns = [
      path("liveness/", views.liveness, name="liveness"),
      path("readiness/", views.ReadinessView.as_view(), name="readiness"),
  ]
  ```

- `freedom_ls/health/tests/__init__.py` — empty.
- `freedom_ls/health/tests/test_health_endpoints.py` — see test coverage below.

Then add `"freedom_ls.health"` to `INSTALLED_APPS` (near the other `freedom_ls.*` apps).

**Test coverage for the endpoints** (write first). Because `config/urls.py` does not `include` the
health URLconf until Task 4, mark the endpoint tests with `@pytest.mark.urls("freedom_ls.health.urls")`
so `reverse("health:…")` resolves in isolation:

- `GET /health/liveness/` → 200 and does not hit the DB. Use `reverse("health:liveness")`. Assert 200
  and the JSON body `{"status": "alive"}`. Run it **without** `@pytest.mark.django_db` to prove
  liveness needs no DB; optionally wrap the request in `django_assert_num_queries(0)` to assert zero
  queries explicitly.
- `GET /health/readiness/` with a healthy DB (`@pytest.mark.django_db`) → 200; use
  `reverse("health:readiness")`.
- **Readiness-actually-checks-the-DB contract test** — guards the FLS default check set so readiness
  can never silently regress to a check-free always-200. Assert that `ReadinessView().get_checks()`
  yields a `Database` check by default (e.g. `"Database"` appears in
  `[type(c).__name__ for c in ReadinessView().get_checks()]`), i.e. readiness runs exactly the DB
  check by default. Do not re-test `django-health-check`'s own DB-down internals.

> **Async-view test note:** v4's `HealthCheckView` is an **async** CBV (`async def get/dispatch`,
> `await asyncio.gather(...)`). Under the sync Django test `client`, Django runs it via
> `AsyncToSync`, which raises `RuntimeError: You cannot use AsyncToSync in the same thread as an async
> event loop` if the calling thread already has a running event loop — which another test in the suite
> can leave behind, making the readiness endpoint test order-dependently flaky. Drive the readiness
> request from a fresh worker thread (`ThreadPoolExecutor(max_workers=1).submit(client.get, …)`): a
> new thread has no running loop, matching how a production WSGI worker serves it. The v4 `Database`
> check uses `connection.temporary_connection()` (its own `SELECT 1` connection, independent of the
> test transaction), so running the probe off the main thread still returns 200. Production is
> unaffected — gunicorn WSGI workers have no running loop.
