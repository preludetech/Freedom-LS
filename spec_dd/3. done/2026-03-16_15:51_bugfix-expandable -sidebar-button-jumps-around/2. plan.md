# Plan: Fix Expandable Sidebar — Button Jumps Around on Mobile

## Overview

Restructure `_base_interface.html` to introduce a **dedicated header bar** between the site header and the content area. The toggle button moves from inside the sidebar column into this fixed header bar, eliminating the position jump. The sidebar panel and header bar work together for both mobile overlay and desktop push behaviors.

## Task 1: Add computed class methods to `sidebarComponent`

**File:** `freedom_ls/base/static/base/js/alpine-components.js`

The Alpine CSP build forbids inline expressions. Add computed methods to `sidebarComponent` that return the appropriate CSS classes based on state. These will be referenced via `x-bind:class` in templates.

Add the following methods:

```javascript
// Returns classes for the header bar left section (sidebar heading + close button)
headerBarLeftClass() {
    // When open on desktop: show as w-64 section
    // When open on mobile: hidden (sidebar overlays everything)
    // When closed: hidden
    if (!this.sidebarOpen) return "hidden";
    if (this.isMobile) return "hidden";
    return "w-64 shrink-0 flex items-center justify-between px-4";
},

// Returns classes for the header bar toggle section
headerBarToggleClass() {
    // When sidebar is open on desktop, hide the toggle (it's replaced by the close button in the left section)
    // When sidebar is open on mobile, hide (sidebar overlays header bar)
    // When closed: show
    if (this.sidebarOpen) return "hidden";
    return "";
},

// Returns classes for the sidebar column (desktop push)
sidebarColumnClass() {
    if (this.sidebarOpen && !this.isMobile) return "w-64 shrink-0";
    return "";
},

// Returns classes for the mobile sidebar panel
sidebarPanelClass() {
    if (this.isMobile) return "fixed left-0 z-40 w-64 bg-surface shadow-lg overflow-y-auto";
    return "";
},

// Returns classes for the mobile backdrop
backdropVisible() {
    return this.sidebarOpen && this.isMobile;
},
```

**Note:** The exact class lists will be refined during TDD. The key point is that all conditional classes must use computed methods — no inline ternaries. All `x-transition` directives in templates must use only Tailwind transition utility classes (e.g., `x-transition:enter="transition ease-out duration-300"`), never inline JavaScript expressions, to maintain CSP compatibility.

## Task 2: Restructure `_base_interface.html` with header bar

**File:** `freedom_ls/base/templates/_base_interface.html`

Replace the current layout with the new header bar structure. The `x-data` scope wraps everything.

### New template structure (pseudocode):

```
{% block body %}
<div x-data="sidebarComponent" data-storage-key="...">

  <!-- HEADER BAR: always between site header and content -->
  <div class="flex items-center border-b border-border bg-surface px-2 h-12">

    <!-- Toggle button (visible when sidebar closed) -->
    <div x-bind:class="headerBarToggleClass">
      <button x-on:click="toggle" ...>
        <c-icon name="menu_open" />
      </button>
    </div>

    <!-- Left section: sidebar heading + close button (visible when sidebar open on desktop) -->
    <div x-bind:class="headerBarLeftClass">
      <span class="font-semibold text-sm truncate">{% block sidebar_heading %}{% endblock %}</span>
      <button x-on:click="toggle" ...>
        <c-icon name="close" />
      </button>
    </div>

    <!-- Centered contextual text (always visible, centered in remaining space) -->
    <div class="flex-1 text-center text-sm font-medium truncate">
      {% block header_bar_text %}{% endblock %}
    </div>
  </div>

  <!-- MOBILE BACKDROP -->
  <div x-cloak x-show="backdropVisible" x-on:click="closeSidebar"
       x-transition:enter="..." x-transition:leave="..."
       class="fixed inset-0 z-30 bg-black/50" ...>
  </div>

  <!-- CONTENT AREA: sidebar + main -->
  <div class="flex relative {% block sidebar_class %}{% endblock %}">

    <!-- Sidebar column (desktop push) -->
    <div x-bind:class="sidebarColumnClass">
      <!-- Sidebar panel -->
      <div x-cloak x-show="sidebarOpen" x-on:click="handleSidebarClick"
           x-bind:class="sidebarPanelClass"
           x-transition:enter="..." x-transition:leave="...">

        <!-- Mobile sidebar header (heading + close button, only on mobile) -->
        <div x-show="isMobile" class="flex items-center justify-between p-4 border-b border-border">
          <span class="font-semibold text-sm">{% block sidebar_heading_mobile %}{% endblock %}</span>
          <button x-on:click="closeSidebar" ...>
            <c-icon name="close" />
          </button>
        </div>

        {% block sidebar_content %}{% endblock %}
      </div>
    </div>

    <!-- Main content -->
    <div class="flex-1 min-w-0">
      {% block content %}{% endblock %}
    </div>
  </div>

</div>
{% endblock body %}
```

### Key design decisions:

1. **Header bar** is a new `<div>` **outside** the flex sidebar/content container, positioned between the site header and the sidebar+content area.
2. The **toggle button** is in the header bar and only shows when the sidebar is closed — its screen position never changes.
3. On **desktop open**, the header bar shows the sidebar heading + close button on the left (w-64) and centered text on the right.
4. On **mobile open**, the sidebar overlays from below the site header. The sidebar panel itself contains a heading + close button at its top.
5. The **mobile sidebar** must use a `top` offset to sit below the site header (not `inset-y-0`). Compute or hardcode based on the header_bar.html height. Use `top-0` relative to the `_base_interface` content area if the sidebar panel is positioned within a relative container below the site header.
6. **Two sidebar heading blocks** (`sidebar_heading` for desktop header bar, `sidebar_heading_mobile` for mobile panel top) can be DRYed up — either use a single block name referenced twice, or use a template variable. Since Django blocks can only appear once, consider using `{{ block.super }}` or a template `{% with %}` pattern. Alternatively, just duplicate the block with matching defaults in child templates.

**Refinement note:** Since Django template blocks can't be rendered in two places, the simplest approach is to have a single `sidebar_heading` block and use `{% block sidebar_heading %}{% endblock %}` in both locations. Django will raise an error for duplicate blocks. Instead, use **one block** for the heading text, and render it once in the header bar (desktop) and once in the mobile sidebar panel. The solution: define a single `sidebar_heading` block in the desktop header bar section, and for the mobile panel, render a separate `sidebar_heading_mobile` block. Child templates set both to the same value. This is a small duplication but clean and explicit.

## Task 5: Adjust mobile sidebar positioning in Alpine component

**File:** `freedom_ls/base/static/base/js/alpine-components.js`

The `sidebarPanelClass()` method must position the mobile sidebar panel so it starts **below** the `_base.html` site header. The mobile sidebar should overlay both the new header bar and the main content, but NOT the site header.

Options:
- Use a CSS class like `top-[var(--header-height)]` and set `--header-height` on the site header via a style attribute.
- Or use a fixed value that matches the site header's natural height (the header uses `p-3 sm:p-4` which is roughly `48-56px`). Since the header height can vary, prefer a CSS custom property approach or position the fixed panel inside a container that's already below the header.

Since `_base_interface.html` renders inside `{% block body %}` which is inside `<main>` which is after the header, the simplest approach: make the mobile sidebar `fixed` with `top` calculated from the header. Or better: wrap the entire `_base_interface` output in a `relative` container and use `absolute inset-0` for the mobile sidebar (it will be contained within the body block area). Actually, `fixed` positioning ignores parent containers — so we need an explicit top offset.

**Recommended approach:** Set `top: 0` and use `h-screen` minus header, OR dynamically calculate the offset in `init()`. The cleanest solution for CSP compatibility: in `init()`, measure the distance from the component root to the viewport top and store it, then apply it as a CSS variable or inline style.

Actually, the simplest clean approach: in the template, the mobile sidebar overlay sits inside the `x-data` div. Use JavaScript in `init()` to get `this.$el.getBoundingClientRect().top` and set a CSS custom property `--sidebar-top` on the element. Then the mobile sidebar panel uses `top: var(--sidebar-top)` with `fixed left-0 bottom-0`.

Add to `sidebarComponent.init()`:
```javascript
// Set CSS custom property for mobile sidebar top offset
const top = this.$el.getBoundingClientRect().top;
this.$el.style.setProperty('--sidebar-top', top + 'px');
```

And the mobile sidebar panel class includes a style for `top: var(--sidebar-top)` and `bottom: 0` instead of `inset-y-0`.

Similarly, the mobile backdrop should use the same top offset.
