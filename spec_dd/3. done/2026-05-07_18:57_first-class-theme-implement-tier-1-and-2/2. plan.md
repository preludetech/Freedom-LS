# Implementation Plan: first_class theme — Tier 1 + Tier 2

This plan turns `1. spec.md` into ordered, TDD-friendly tasks. Concrete
`@apply` recipes already exist in `research_tier2_overrides.md`; this plan
references them rather than re-printing them. Read the spec alongside this
plan — the spec is the source of truth on *what* and *why*, this plan
covers *how* and *in what order*.

The work touches three files plus a small handful of templates. There is no
Python, no Django models, no migrations, and no admin work.

---

## Files in scope (the universe of edits)

Token / contract layer:

- `freedom_ls/themes/default/static/themes/default/theme.css` — Tier-1
  token additions (mono font + alias, `*-light` and `on-*-light` tokens).
- `freedom_ls/themes/first_class/static/themes/first_class/theme.css` —
  Tier-1 token additions, radii correction, plus the new
  `@layer components { … }` and `@layer base { … }` Tier-2 override
  blocks.
- `tailwind.components.css` — Tier-2 default-contract additions
  (`.btn-secondary`, `.btn-ghost`, `.btn-accent`, `.chip-info`,
  `.chip-secondary`, `.chip-muted`, `.alert*`); removal of `.btn-outline`.

Template / call-site migrations:

- `freedom_ls/base/templates/cotton/button.html` — usage example in
  the docstring (`<c-button variant="outline">` → `variant="secondary"`).
- `freedom_ls/base/templates/cotton/data-table.html` — single
  `class="btn btn-outline"` → `class="btn btn-secondary"`.
- `freedom_ls/base/templates/cotton/pagination.html` — eight
  `class="btn btn-outline …"` occurrences → `btn btn-secondary`.
- `freedom_ls/student_interface/templates/student_interface/partials/course_list.html`
  — two `<c-button variant="outline">` → `variant="secondary"`.
- `freedom_ls/student_interface/templates/student_interface/course_topic.html`
  — two `<c-button … variant="outline" …>` → `variant="secondary"`.
- `freedom_ls/panel_framework/templates/panel_framework/partials/delete_confirmation.html`
  — single `<c-button variant="outline" @click="close">` →
  `variant="secondary"`.

Out of scope (do **not** touch):

- `freedom_ls/icons/templates/cotton/icon.html` — its `variant="outline"`
  is the icon component's *icon style* variant (outline vs solid icons),
  not a button variant. The grep audit must filter this out.
- `freedom_ls/icons/tests/test_renderer.py` — same reason: `variant="outline"`
  here refers to icon rendering, not buttons.

Existing functionality being relied on (not edited, just exercised):

- `freedom_ls/base/management/commands/write_active_theme_css.py` — picks
  up the active theme via `FLS_THEME` env var, writes
  `tailwind.active_theme.css`. Re-run after any theme.css change.
- `tailwind.input.css` — already imports default tokens, then
  `tailwind.components.css`, then the active theme. Cascade ordering is
  already correct for the Tier-2 first_class overrides to win — no edits
  needed.
- `npm run tailwind_build` — single command builds the CSS pipeline.
- `freedom_ls/base/templates/cotton/button.html` — the cotton component
  composes `btn btn-{{ variant }}`, so changing `variant="outline"` to
  `variant="secondary"` everywhere is enough; no logic changes in
  `button.html` itself beyond the docstring example.
- `freedom_ls/base/templates/cotton/chip.html` — same pattern for chips
  (referenced in `research_component_class_usage.md`); not edited here
  but the new `.chip-info`, `.chip-secondary`, `.chip-muted` classes will
  automatically be available via `<c-chip variant="info">` etc.

## Implementation order

This is the dependency-safe order. Tier-1 tokens land first because the
Tier-2 overrides reference them; the `.btn-outline` removal lands in the
same commit as its template-call-site migrations so the build is never
broken in the middle.

### Step 1 — Tier-1 tokens in default theme

Edit `freedom_ls/themes/default/static/themes/default/theme.css`:

1. Inside the existing `@theme { … }` block, add (per spec §1.1):

    ```css
    --fls-font-mono: ui-monospace, SFMono-Regular, Menlo, Consolas,
        "Liberation Mono", "Courier New", monospace;
    ```

2. Inside the same `@theme { … }` block, add the four `*-light` tokens
   (spec §1.2) and the four `on-*-light` tokens (spec §1.3). Use the
   exact hex values from the spec tables.

3. Inside the existing `@theme inline { … }` block, add:

    ```css
    --font-mono: var(--fls-font-mono);
    ```

   (Aliasing `--font-mono` once in default is the whole point — first_class
   only redeclares `--fls-font-mono`.)

### Step 2 — Tier-1 tokens + radii correction in first_class theme

Edit `freedom_ls/themes/first_class/static/themes/first_class/theme.css`:

1. Add to the existing `@theme { … }` block:

    ```css
    --fls-font-mono: "IBM Plex Mono", ui-monospace, Menlo, Consolas, monospace;
    ```

   No `@theme inline` aliasing — inherited from default.

2. Add the four `--color-*-light` tokens and the four `--color-on-*-light`
   tokens with the same hex values as default (spec §1.2 / §1.3).

3. Replace the current radii values with the corrected ramp (spec §1.4):

    ```css
    --fls-radius-sm: 0.375rem;  /* 6px  — chips, badges */
    --fls-radius-md: 0.5rem;    /* 8px  — buttons, inputs */
    --fls-radius-lg: 0.75rem;   /* 12px — cards, modals */
    /* --fls-radius-pill stays inherited from default (9999px) */
    ```

4. Replace the existing radii comment ("bumped one step for the brand's
   rounder card treatment") with a one-line note that radii match
   `design-system/radii.html`.

### Step 3 — Tier-2 default contract additions in `tailwind.components.css`

Edit `tailwind.components.css` (additions go inside the existing
`@layer components { … }` block — do not add a second `@layer components`
block):

1. Add `.btn-secondary` (spec §2.1):

    ```css
    .btn-secondary {
        @apply bg-transparent text-primary border border-primary
               hover:bg-primary/10 focus:ring-primary;
    }
    ```

2. Add `.btn-ghost` (spec §2.1):

    ```css
    .btn-ghost {
        @apply bg-transparent text-primary
               hover:bg-primary/10 focus:ring-primary;
    }
    ```

3. Add `.btn-accent` (spec §2.1):

    ```css
    .btn-accent {
        @apply bg-accent text-on-accent
               hover:bg-accent-hover focus:ring-accent;
    }
    ```

4. Add `.chip-info`, `.chip-secondary`, `.chip-muted` (spec §2.2):

    ```css
    .chip-info       { @apply bg-info/15 text-info; }
    .chip-secondary  { @apply bg-secondary/15 text-secondary; }
    .chip-muted      { @apply bg-surface-2 text-muted; }
    ```

5. Add the alert family (spec §2.3). Concrete recipe (final `@apply` list
   to be tuned at implementation time within the spec's stated shape):

    ```css
    .alert {
        @apply block px-4 py-3 rounded-md border text-on-surface;
    }
    .alert-success {
        @apply bg-success/10 border-success/30 text-on-surface;
    }
    .alert-error {
        @apply bg-error/10 border-error/30 text-on-surface;
    }
    .alert-info {
        @apply bg-info/10 border-info/30 text-on-surface;
    }
    ```

6. **Remove** the existing `.btn-outline` rule from
   `tailwind.components.css`. It must be deleted in the same commit as
   the template-call-site migrations (Step 4) so no intermediate state
   leaves the build referencing an undefined class.

### Step 4 — Migrate `.btn-outline` / `variant="outline"` call sites

Edit each of the six template files listed under "Files in scope" (do
not touch `freedom_ls/icons/templates/cotton/icon.html` or
`freedom_ls/icons/tests/test_renderer.py`). Replacement rule:

- `class="btn btn-outline …"` → `class="btn btn-secondary …"`
- `<c-button variant="outline" …>` → `<c-button variant="secondary" …>`

After editing, run a final audit grep:

```bash
grep -rn '\bbtn-outline\b\|variant="outline"' freedom_ls/
```

The only remaining hits must be inside `freedom_ls/icons/` (icon-style
variant, intentionally untouched). Anything else is a missed call site.

### Step 5 — first_class Tier-2 component overrides

Append a new `@layer components { … }` block to
`freedom_ls/themes/first_class/static/themes/first_class/theme.css`. Inside
it, add the override rules listed in spec §3.1 and §3.2. The `@apply`
recipes are pre-drafted in `research_tier2_overrides.md`, with the two
radius corrections noted in spec §3.1:

- `.btn` override uses `rounded-md` (not `rounded-sm` as the research file
  pre-correction draft says) — after spec §1.4, 8px lives at
  `--fls-radius-md`.
- `.surface` override uses `rounded-lg` (not "no shape change" as the
  research file says) — after spec §1.4, 12px lives at
  `--fls-radius-lg`.

Pseudocode for the block (final values per spec §3.1 / §3.2):

```css
@layer components {
    .btn {
        @apply px-6 py-3 text-sm font-semibold rounded-md;
        transition: background-color 200ms cubic-bezier(0.22, 1, 0.36, 1),
                    color 200ms cubic-bezier(0.22, 1, 0.36, 1);
    }
    .btn-secondary { @apply border-2; }

    .chip            { @apply gap-1.5 px-3 py-1.5 text-xs font-semibold rounded-pill; }
    .chip-primary    { @apply bg-primary/10 text-primary; }
    .chip-warning    { @apply bg-warning-light text-on-warning-light; }
    .chip-success    { @apply bg-success-light text-on-success-light; }
    .chip-error      { @apply bg-error-light text-on-error-light; }
    .chip-info       { @apply bg-info-light text-on-info-light; }
    .chip-secondary  { @apply bg-secondary/10 text-secondary; }
    /* .chip-muted: only override if the default reads poorly on first_class
       surface-2 (tune at implementation time per spec §3.2). */

    .surface { @apply bg-surface rounded-lg; }

    .alert-success { @apply bg-success-light border-success/30 text-on-success-light; }
    .alert-error   { @apply bg-error-light   border-error/30   text-on-error-light;   }
    .alert-info    { @apply bg-info-light    border-info/30    text-on-info-light;    }
}
```

Do **not** override `.btn-primary`, `.btn-success`, `.btn-error`,
`.btn-accent`, `.btn-ghost`, `.chip-xs`, or `.alert` (the base shape) —
spec §3.3 lists these as picking up the brand look via tokens + the
reopened `.btn` / `.chip` base.

### Step 6 — first_class `@layer base` heading override

Append (also to `freedom_ls/themes/first_class/static/themes/first_class/theme.css`,
below the `@layer components` block) the heading-weight override (spec §4):

```css
@layer base {
    h1 { @apply tracking-tight; }
    h2 { @apply font-semibold tracking-tight; }
    h3 { @apply font-semibold; }
    h4 { @apply font-semibold; }
}
```

### Step 7 — Rebuild Tailwind, sanity-build everywhere

After all CSS edits:

1. `npm run tailwind_build` — regenerates `static/vendor/tailwind.output.css`
   and triggers `_write_active_theme` so the active theme picks up the
   first_class overrides when `FLS_THEME=first_class`.
2. Build for **both** themes to check both compile cleanly:

    ```bash
    FLS_THEME=default npm run tailwind_build
    FLS_THEME=first_class npm run tailwind_build
    ```

3. The build must complete without "unknown utility" warnings (which is
   how Tailwind v4 surfaces an `@apply` that references a missing token —
   e.g. a typo on `text-on-warning-light`).

### Repo-wide grep assertion

```python
def test_no_template_uses_btn_outline_or_button_variant_outline():
    """Audit-grep equivalent. Excludes freedom_ls/icons/ where
    variant='outline' is the icon-style variant, not a button variant."""
    import subprocess
    result = subprocess.run(
        ["grep", "-rln",
         r"\bbtn-outline\b\|variant=\"outline\"",
         "freedom_ls/"],
        capture_output=True, text=True,
    )
    hits = [
        line for line in result.stdout.splitlines()
        if not line.startswith("freedom_ls/icons/")
    ]
    assert hits == [], f"Unexpected btn-outline / variant=\"outline\" usage: {hits}"
```

This last test is the contract guard for the migration: it stays in the
suite so future work can't reintroduce `.btn-outline` by accident.

### What we are NOT testing

Per CLAUDE.md and the user's `feedback_no_style_assertions` memory:

- Do **not** assert that any rendered template contains specific
  Tailwind classes like `bg-primary` or `rounded-md`.
- Do **not** assert pixel widths, colours, or computed styles.
- Visual fidelity is a manual / Playwright concern, not a unit-test
  concern.

## Existing functionality to reuse (DRY check)

- `c-button` cotton component (`freedom_ls/base/templates/cotton/button.html`)
  composes the class list as `"btn btn-{{ variant }}"`. The migration is
  driven entirely by changing the `variant` attribute on each call site —
  no changes to `button.html` itself except the docstring example.
- `c-chip` cotton component (`freedom_ls/base/templates/cotton/chip.html`,
  per `research_component_class_usage.md`) follows the same pattern, so
  `<c-chip variant="info">`, `<c-chip variant="secondary">`,
  `<c-chip variant="muted">` will Just Work once `.chip-info`,
  `.chip-secondary`, `.chip-muted` exist in `tailwind.components.css`.
- `write_active_theme_css` management command + `tailwind_build` npm
  script already handle the theme build pipeline. No new tooling.
- The cascade order (default tokens → components → active theme) is
  already established in `tailwind.input.css`. No new `@import` lines.
- The hover-mix machinery in default's `@theme` block (`color-mix(...)`)
  derives `*-hover` tokens automatically, so `.btn-accent` inherits a
  workable hover via `--color-accent-hover` without needing a per-token
  override in either theme.

## Risks and rollback notes

- **Cascade ordering**: spec §3 calls for one devtools spot-check on
  first deploy that `first_class/theme.css` `@layer components` rules
  win over `tailwind.components.css`. This is currently established by
  the `@import` order in `tailwind.input.css` (default tokens →
  components → active theme); no edit needed, but worth a one-time
  inspect-element on a `.btn-primary` after the first build to confirm.
- **`.btn-outline` removal is breaking**: the changed semantic (neutral
  grey → primary outline) is acknowledged in spec §"Success criteria".
  Anywhere `.btn-outline` was used as a "tertiary cancel" affordance
  (e.g. `delete_confirmation.html` Cancel button) will read as a
  secondary CTA after migration. This is intentional per the spec — no
  rollback path; do not preserve a `.btn-outline` alias.
- **`.btn` padding bump in first_class** (`py-2` → `py-3`): can push
  tight nav rows to a second line. Check the dashboard / pagination
  rows at first_class build time.
- **`.chip-success` / `-warning` / `-error` text colour change in
  first_class**: switches from `text-success` (or `text-on-warning`) to
  `text-on-success-light` etc. Contrast is fine on the new pastel
  backgrounds (per `research_first_class_token_gaps.md`), but eyeball
  the educator cohort screen (`course_progress_panel.html`) where these
  chips are densest.
