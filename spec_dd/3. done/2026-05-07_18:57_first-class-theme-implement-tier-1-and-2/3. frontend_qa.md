# Frontend QA: first_class theme — Tier 1 + Tier 2

This QA plan walks the rendered output under both themes to confirm the
spec's visual targets land. The work has zero new pages and zero new
flows — every change is a token, a class, or a `variant=` rename — so
the QA is a structured visual sweep across pages where the affected
buttons, chips, alerts, and surfaces appear.

Use the `fls:use-playwright` skill for the browser interactions.

---

## Setup (run once before any test below)

All tests use the **DemoDev** site (per `feedback_use_demodev_site`
memory and `FORCE_SITE_NAME=DemoDev` in `config/settings_dev.py`).

1. Pick a random port:
    ```bash
    PORT=$(.claude/fls/scripts/find_available_port.sh)
    ```

2. Build CSS for the **first_class** theme (Tier-1 + Tier-2 visual
   target):
    ```bash
    FLS_THEME=first_class npm run tailwind_build
    ```

3. Start the dev server with the first_class theme active:
    ```bash
    FLS_THEME=first_class uv run python manage.py runserver $PORT
    ```

4. Base URL for all the tests below: `http://127.0.0.1:$PORT`.

5. Demo credentials (created by `uv run manage.py create_demo_data`):
    - Educator login: `demodev@email.com` / password `demodev@email.com`
    - Student login: `demodev_s1@email.com` / password
      `demodev_s1@email.com` (any `demodev_sN@email.com` up to `s50`).

If the demo data is missing, run
`uv run manage.py create_demo_data` first. If a specific test below
needs cohort progress data, the `qa-data-helper` agent (or
`uv run manage.py qa_create_cohort_progress DemoDev`) creates it.

After all the first_class tests complete, repeat the **default theme
sweep** (test 7 below) to confirm no regression.

## Test 1 — Tier-1 token bundle is live (devtools)

Goal: confirm the Tier-1 tokens declared in spec §1 actually resolve in
the browser, and the cascade order is correct.

1. Navigate to `http://127.0.0.1:$PORT/` (any page; tokens are
   document-scoped via `:root`).
2. Open browser devtools → Console. Evaluate:
    ```js
    const s = getComputedStyle(document.documentElement);
    [
      "--fls-font-mono",
      "--color-success-light",
      "--color-warning-light",
      "--color-error-light",
      "--color-info-light",
      "--color-on-success-light",
      "--color-on-warning-light",
      "--color-on-error-light",
      "--color-on-info-light",
      "--fls-radius-sm",
      "--fls-radius-md",
      "--fls-radius-lg",
    ].map(k => [k, s.getPropertyValue(k).trim()]);
    ```
3. Expected:
    - `--fls-font-mono` starts with `"IBM Plex Mono"`.
    - The four `--color-*-light` values are
      `#F0FFF4` / `#FFFFF0` / `#FFF5F5` / `#EBF8FF` (or the equivalent
      computed-form Tailwind emits).
    - The four `--color-on-*-light` values are
      `#22543D` / `#744210` / `#742A2A` / `#2A4365`.
    - Radii are `0.375rem` / `0.5rem` / `0.75rem` (the spec §1.4
      corrected ramp), **not** the old `0.5 / 0.75 / 1rem`.
4. Cascade spot-check: in the Elements panel, inspect any `.btn-primary`
   on the page. Its computed `background-color` should resolve to
   `rgb(40, 53, 147)` (Deep Indigo, `#283593`). If it resolves to the
   default Ocean blue, the active-theme `@import` order is wrong.

## Test 2 — `.btn` family in first_class

Goal: confirm the new `.btn-secondary`, `.btn-ghost`, `.btn-accent` look
right and that no `.btn-outline` survives.

1. Log in as the educator (`demodev@email.com`).
2. Navigate to a delete-confirmation flow. Easiest path: open any
   instance with a delete action in the educator interface (e.g. a
   cohort detail panel) and trigger the delete confirmation modal.
3. Expected on the modal:
    - The "Cancel" button uses `.btn-secondary` (transparent background,
      primary-indigo text and border, `border-2` thicker outline per
      spec §3.2). It should **not** look neutral grey.
    - The "Delete" / destructive button uses `.btn-error` (solid red).
4. Navigate to the courses list under the student interface
   (`/courses/`). Any "Browse" / secondary CTA buttons should now read
   as `.btn-secondary` (indigo outline) rather than the old neutral
   grey.
5. Pagination spot-check: navigate to a long list (e.g. educator cohort
   list, or any data table that paginates). Confirm pagination
   prev/next/page-number buttons render as `.btn-secondary` with the
   primary-indigo outline.
6. View page source / inspect element on any page that previously used
   `.btn-outline`. There should be no `btn-outline` token in the class
   list anywhere.
7. (Optional) Find a page using `.btn-accent` (currently zero callers in
   templates — if added incidentally, confirm Altitude-Orange fill +
   white text). For `.btn-ghost`, same: no callers today, so this is
   only a "the class exists in CSS" check via devtools (Test 1 covers
   that the CSS file rebuilt).

## Test 3 — Chip family in first_class

Goal: confirm `.chip-primary`, `.chip-warning`, `.chip-success`,
`.chip-error`, plus the new `.chip-info`, `.chip-secondary`,
`.chip-muted` render with the brand-correct pastel-tint look.

1. Log in as the educator.
2. Navigate to the educator's course-progress panel (the one that
   renders `course_progress_panel.html`). Path: educator interface →
   pick a cohort → view progress.
3. Expected:
    - "Hard" chip: `.chip-error` — pale pink background
      (`#FFF5F5`-ish), deep red text (`#742A2A`-ish).
    - "Soft" chip: `.chip-warning` — pale cream background
      (`#FFFFF0`-ish), deep amber text (`#744210`-ish).
    - "Pass" chip: `.chip-success` — pale green background
      (`#F0FFF4`-ish), deep green text (`#22543D`-ish).
    - "Fail" chip: `.chip-error` — same as Hard.
    - All chips are full pills (`rounded-pill`, semibold, `text-xs`).
4. For chips not currently used in templates (`.chip-info`,
   `.chip-secondary`, `.chip-muted`), confirm the CSS exists by reading
   the rendered `tailwind.output.css` or via devtools. They become
   visible once a future template wires them in (or if you spot-test
   with `<c-chip variant="info">` in a temp view).
5. Contrast spot-check: the chip text must be legible on its pastel
   background. If any of the four `text-on-*-light` foregrounds reads as
   thin or washed-out, flag it for spec re-tune (the hex values are
   pulled from the brand doc and should pass WCAG AA at chip size, per
   `research_first_class_token_gaps.md`).

## Test 4 — `.surface` and card radii

Goal: confirm `.surface` cards lift off the page background and land at
the corrected 12px (`--fls-radius-lg`) radius.

1. Log in as a student (`demodev_s1@email.com`).
2. Navigate to the courses list (`/courses/`) — the course cards are
   wrapped in `.surface`.
3. Expected:
    - Each card has an explicit `bg-surface` background
      (`#F8F9FC`-ish), visibly distinct from the page background.
    - Card corners read as ~12px rounded (the `--fls-radius-lg`-after-
      correction value), **not** the old 16px or the default-theme 8px.
4. Spot-check the panel framework (`panel_container.html` and
   `educator_interface/.../panel_container.html`) — same `.surface`
   class is used there.

## Test 5 — Mono font

Goal: confirm `--font-mono` is wired and renders IBM Plex Mono in
first_class.

1. Find any rendered `<code>` block in the demo content (e.g. open a
   course topic that contains markdown with inline code or a code
   fence). The student interface course topic pages are the easiest
   surface — pick one of the `functionality_demo_*` courses.
2. Inspect the rendered `<code>` / `<pre>` element. Confirm the computed
   `font-family` resolves to a string starting with `"IBM Plex Mono"`.
3. If IBM Plex Mono is not loaded as a webfont in the page, the fallback
   stack (`ui-monospace, Menlo, Consolas, monospace`) takes over — that's
   acceptable for this QA pass; webfont loading is a separate concern.

## Test 6 — Heading weights / tracking (first_class `@layer base` override)

Goal: confirm the brand H2-H4 weight (`font-semibold`, not `font-bold`)
and tracking (`tracking-tight` on H1-H2) override is live.

1. Navigate to any page with all four heading levels — the student
   course-home page (`/courses/<slug>/`) is a good candidate, or pick
   one of the demo content topics that includes nested headings.
2. Expected at the `lg:` breakpoint (i.e. on a desktop-width browser
   ≥ 1024px):
    - H1 renders at ~36px, bold (`font-bold`), with negative letter
      spacing (`tracking-tight`).
    - H2 renders at ~30px, **semibold** (`font-semibold`, not
      `font-bold`), `tracking-tight`.
    - H3 renders at ~24px, **semibold**, no tracking change.
    - H4 renders at ~20px, **semibold**.
3. Inspect each heading in devtools and confirm `font-weight: 600` on
   H2-H4 (semibold). If any heading still computes 700, the
   `@layer base` block in `first_class/theme.css` lost the cascade
   battle — re-check that it's loaded *after* the default theme's base
   layer.

## Test 7 — Default-theme regression sweep

Goal: confirm the new `tailwind.components.css` classes render
coherently on the **default** theme too, and that the `.btn-outline`
removal didn't blow anything up.

1. Stop the runserver.
2. Rebuild and restart under the default theme:
    ```bash
    FLS_THEME=default npm run tailwind_build
    FLS_THEME=default uv run python manage.py runserver $PORT
    ```
3. Re-walk Tests 2 and 3 above on default theme:
    - Every previous `.btn-outline` call site (delete-confirm Cancel,
      pagination, course list secondary buttons) should now render as
      `.btn-secondary`. On the default theme this is a transparent
      background with the default `--color-primary` (Ocean blue) outline
      — visibly different from the old neutral grey, but coherent.
      Acknowledge this is the spec's intentional semantic change (from
      "neutral grey cancel" to "primary outline secondary CTA").
    - Chips on `course_progress_panel.html` should render with the
      default theme's softer (`/15` opacity) palette — flatter than the
      first_class pastel-tint look but coherent with `.chip-primary`'s
      existing treatment.
4. Verify no console errors in devtools. Specifically: no "unknown
   utility" warnings from Tailwind v4 and no missing-token warnings.
5. Verify no visual regressions on pages that previously rendered
   correctly (educator dashboards, student course list, panel framework
   instances).

## Test 8 — Audit grep

Final guard: from the repo root, run:

```bash
grep -rn '\bbtn-outline\b\|variant="outline"' freedom_ls/
```

Expected: the only matches are inside `freedom_ls/icons/` (icon-style
variant — intentionally untouched). Any other hit is a missed migration
and must be fixed before the QA is considered passed.

---

## Pass criteria

The QA passes when:

- All eight tests above complete without unexpected differences.
- No `.btn-outline` reaches the rendered DOM under either theme (per
  Test 8).
- No console / devtools warnings about missing CSS variables or unknown
  utilities under either theme.
- The educator cohort progress panel renders correctly with the new
  pastel chip palette under first_class.
- The visual diff against `./design-system/components-buttons.html`,
  `components-chips.html`, `components-feedback.html`, `radii.html`,
  `type-headings.html`, `type-mono.html` is within the deliberate
  trade-offs called out in spec §"Success criteria" (no
  `chip-success` darker-text token, no `*-light` for secondary).

If any test fails, file the bug and follow `feedback_tdd_for_bugfixes`:
write a failing test first, then fix.
