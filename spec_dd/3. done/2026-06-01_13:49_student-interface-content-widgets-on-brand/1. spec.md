# Spec: On-brand course content widgets

## Overview

Add a richer, accessible, on-brand set of **content widgets** for students inside
course content. Widgets are authored as cotton components (`<c-widget …>`) inside
markdown. The rendering pipeline is:

```
markdown → render → nh3 sanitise (allowlist) → cotton compile → Django render → safe HTML
```

(`freedom_ls/markdown_rendering/markdown_utils.py`; allowlist in
`config/settings_base.py` → `MARKDOWN_ALLOWED_TAGS`.)

Every new cotton tag must be registered in `MARKDOWN_ALLOWED_TAGS` with its allowed
attributes. Interactivity comes only from **Alpine.js (CSP build)** or **HTMX** — no
arbitrary JS. The one exception this spec introduces is **KaTeX**, a client-side
math *rendering* library (not application JS); see §3.2.

This spec implements widgets in the **default** theme using **role tokens**
(`--color-primary`, `text-on-surface`, `bg-surface`, status tokens, `--fls-radius-*`,
`font-mono`, …) and overrides in the **first_class** theme only where the brand look
needs it. **No hardcoded hex** — colours/shapes flow from the active theme tokens
(`freedom_ls/themes/default/…/theme.css`, `freedom_ls/themes/first_class/…/theme.css`).

### Why this matters

Course content today has a handful of simple widgets (`c-callout`, `c-picture`,
`c-youtube`, `c-pdf-embed`, `c-file-download`, `c-content-link`). Authors need a
broader, consistent, **accessible** vocabulary to express real teaching content —
quotes, equations, captioned figures, comparison tables, code/log output, captioned
video — without reaching for raw HTML (which the sanitiser would strip) or
inaccessible patterns (carousels, hover-only tooltips). Doing this **on-brand** and
**theme-token-driven** means the same content looks right in any FLS install's theme.

## 1. Scope

### In scope (this spec)

| # | Widget | Action |
|---|--------|--------|
| 1 | **Pull quote** | New `c-pull-quote` (figure/blockquote/figcaption + attribution) |
| 2 | **Equation / formula block** | New `c-equation` — client-side **KaTeX** typeset math |
| 3 | **Video** | Extend `c-youtube` — optional caption/title |
| 4 | **Figure / photo** | Extend `c-picture` — figure semantics, optional caption number, **keyboard-accessible** lightbox trigger + dialog a11y |
| 5 | **Annotated diagram** | **No new widget** — reuse the figure/photo widget; author the annotated SVG/image as an asset + a text legend (`<dl>`/list) for lettered call-outs |
| 6 | **Image gallery (no carousel)** | New `c-image-grid` — static thumbnail grid opening the existing picture lightbox; (stacked `c-picture` figures is the other supported pattern) |
| 7 | **Data table** | New `c-table` — purpose-built for content; semantic `<table>` (caption, `scope`) in a **focusable horizontally-scrollable** wrapper. **Does not reuse the app's existing `c-data-table` / `scrollTableLabels`.** |
| 8 | **Code / log block** | New `c-code-block` — brand monospace `<pre><code>` with filename/title chrome; **no** syntax highlighting, **no** copy button; focusable scroll or wrap |

### Split out into their own specs (deferred — see `spec_dd/1. next/`)

- **Callouts & admonitions** → `content-configurable-admonitions/`. The requirement
  grew from "extend `c-callout` with six tones" into a **configurable** content-
  admonition system (builders define types → colour + icon + label). `c-callout`
  stays as-is for application-level notifications and is **not** touched here.
- **Glossary term + definition list** → `content-glossary-widget/`. The accessible
  inline-reveal interaction (WCAG 1.4.13) is non-trivial and gets its own pass.

### Explicitly out of scope (unchanged from idea)

- **Learning objective** widget — skipped.
- **Audio** — skipped this pass.
- **Carousel** — not built (UX-discouraged, hard to make accessible); replaced by the
  image grid / stacked figures.
- **Code syntax highlighting** and **copy button** — deferred (each adds a dependency
  and/or sanitiser surface; a separate scoped decision).
- **Custom video player** (seeking, caption toggle, language switcher) — not built;
  we keep the native YouTube iframe.
- Whole categories: **04 Assessment**, **05 Interactive content**, **07 Reference &
  social**.

## 2. Cross-cutting requirements

### 2.1 Accessibility (applies to every widget)

- **Native semantics first**, ARIA only to fill real gaps (`figure/figcaption`,
  `blockquote`, `table`+`caption`+`scope`, `pre/code`, real `<button>`s).
- **Colour is never the sole signal** (WCAG 1.4.1) — e.g. diagram call-outs carry a
  text legend.
- **Everything interactive is keyboard operable** with a **visible focus ring** and a
  sensible focus order; never trap or lose focus.
- **Respect `prefers-reduced-motion`** for any animation/transition (Tailwind
  `motion-reduce:` variants, as already used in `partials/_toast.html`). This includes
  the lightbox backdrop transition.
- **Wide content** (tables, code, equations, dense diagrams) lives in a **focusable**
  (`tabindex="0"`) scroll container with an accessible label — never clipped with
  `overflow:hidden` (WCAG 1.4.10 reflow).
- **Media** leans on the platform player for captions/controls; never autoplay with
  sound; the iframe always has a descriptive `title`.
- Contrast: body text/labels meet 4.5:1; non-text UI (pill borders, focus rings)
  meet 3:1 — verified in **both** themes (check the lightest first_class tints).

### 2.2 Theming

- Build each widget in the **default** theme with **role tokens** only. Verify it in
  **first_class** (most of the brand look should flow automatically through tokens).
- Add **first_class** overrides only where the brand genuinely needs it (e.g. a
  decorative pull-quote mark, code-block chrome treatment). Put theme-specific tweaks
  in the theme CSS, not inline hex.
- Reuse existing component classes/tokens before inventing new ones.

### 2.3 Pipeline / registration

- Each new tag (`c-pull-quote`, `c-equation`, `c-image-grid`, `c-table`,
  `c-code-block`) and each new attribute on extended tags
  (`c-youtube`, `c-picture`) is added to `MARKDOWN_ALLOWED_TAGS` in
  `config/settings_base.py`. Exact attribute lists are finalised in the plan.
- Cotton template filenames map to tag names; templates live in
  `freedom_ls/content_engine/templates/cotton/`.
- New Alpine components (CSP build) for these widgets belong to the **content_engine
  app**, not `base`. Create
  `freedom_ls/content_engine/static/content_engine/js/alpine-components.js` and load it
  via the `{% block extra_alpine_components %}` hook (the same pattern
  `student_interface` and `panel_framework` use in `_base.html`) on the page that
  renders content. Register components CSP-style — named methods/properties only, no
  inline expressions. The existing `pictureModal` / `scrollTableLabels` components in
  `base` are reference patterns for *how* to write a CSP Alpine component; do not add
  content-widget code to the `base` app.

## 3. Widget specifications

### 3.1 Pull quote — `c-pull-quote`

- **Markup**: `<figure>` wrapping `<blockquote>` for the quote, attribution in
  `<figcaption>` **outside** the blockquote. Optional `cite` attribute (source URL)
  on the `<blockquote>`. `<cite>` only wraps a work's title, never a person's name.
- **Attributes** (final set in plan): `attribution` (person/source text), `cite`
  (URL, optional), `source` (work title, optional).
- **Body**: slot rendered as markdown.
- **A11y**: decorative quote-mark glyphs (if any) `aria-hidden`. Visual treatment +
  attribution carry the meaning.
- **Theming**: role tokens; oversized decorative quote mark / accent is a candidate
  for a first_class-only flourish, scaled down on small screens.
- **Why**: attributed quotes are common in teaching content; a semantic
  figure/blockquote is the correct, accessible unit.

### 3.2 Equation / formula block — `c-equation` (client-side KaTeX)

**Decision: client-side KaTeX.** KaTeX ships as a static asset and renders LaTeX in
the browser **after** nh3 sanitisation. Consequences:

- **No `MARKDOWN_ALLOWED_TAGS` change for KaTeX output** — rendering happens
  client-side, post-sanitisation, so nh3 never sees the MathML/HTML KaTeX emits. The
  only allowlist change is the `c-equation` tag itself. (This is simpler and a smaller
  sanitiser surface than the server-side option originally floated in the idea.)
- KaTeX is a **rendering library**, the one allowed exception to "no arbitrary JS".

**Behaviour**

- `c-equation` wraps author-supplied **LaTeX** (as text content / attribute) and a
  container that KaTeX renders into. Core target is **display/block** equations.
- The rendered equation sits in a **focusable, horizontally-scrollable** container
  (`tabindex="0"`, `overflow-x:auto`, accessible label) so long equations don't clip
  and keyboard users can scroll (WCAG 1.4.10).
- Optional `label`/number attribute for an equation reference (e.g. "(3)").
- KaTeX outputs MathML for screen-reader semantics; render with `throwOnError:false`
  so a malformed equation degrades to readable source rather than breaking the page.

**Loading**

- Vendor KaTeX CSS + JS (+ fonts) into static assets and load them on student content
  pages that render markdown (scope the include so non-content pages don't pay for
  it). Confirm load mechanism in the plan.
- Decide in the plan whether a tiny Alpine/init hook renders only `c-equation`
  containers (preferred — explicit, avoids `$`-delimiter clashes with currency like
  "R1000"/"$5") versus a global `$$` auto-render. **Lean: render only the widget's
  containers**, not global delimiters.

**Security (flag for plan security review)**

- Configure KaTeX with **`trust: false`** (and a strict/safe macro set) so commands
  like `\href`, `\url`, `\includegraphics` cannot inject `javascript:` URLs or
  external resources. The author's LaTeX is untrusted content.
- Since KaTeX renders client-side from text the sanitiser already passed, ensure the
  LaTeX is delivered as **text**, not pre-parsed HTML.

### 3.3 Video — extend `c-youtube`

- Keep the existing simple, responsive **YouTube iframe** embed (native captions,
  speed, fullscreen). Iframe keeps its descriptive `title` (existing `video_title`).
- **Add**: optional `caption`/title shown with the video (figure/figcaption). **No**
  custom seeking player, caption toggle, or language switcher.
- **A11y**: aspect-ratio wrapper (already present) so it scales without overflow.

### 3.4 Figure / photo — extend `c-picture`

The existing `c-picture` already uses `figure/figcaption` + an Alpine lightbox
(`pictureModal`). Extend it **additively**:

- **A11y fixes (important)**: the current thumbnail is an `<img @click="show">` —
  **not keyboard-operable**. Make the lightbox **trigger a real `<button>`** (or
  add `role="button"` + `tabindex="0"` + key handlers) so keyboard users can open it.
  Give the lightbox dialog proper semantics (`role="dialog"`/`aria-modal`,
  accessible name, move focus in on open and restore on close, Escape already wired).
- Respect `prefers-reduced-motion` on the backdrop-blur/transition.
- **App ownership note**: `pictureModal` currently lives in the **`base`** app
  (pre-existing) while `c-picture` is a content_engine template — a cross-app coupling
  that predates this work. New content-widget Alpine logic should land in
  content_engine (§2.3). Whether to leave, move, or duplicate `pictureModal` for the
  a11y fix is a question for the **plan structure review**; do not silently grow the
  `base`-app surface for content concerns.
- **Add**: optional figure **number/label** (e.g. "Figure 3").
- **Alt vs caption guidance** documented for authors: `alt` describes the image for
  those who can't see it; `figcaption` is the visible caption; avoid duplicating one
  verbatim in the other; decorative images get `alt=""`.
- Keep the file lookup (`get_file_by_path`) and error fallback.

### 3.5 Annotated diagram — reuse the figure/photo widget (no new widget)

- The annotated SVG/image is authored as an **asset** and dropped into `c-picture`.
  Do **not** build an interactive SVG-annotation engine.
- Lettered call-outs (A/B/C) **require a text legend** — a `<dl>` or list pairing each
  letter with its label, in/near the figcaption, so the letters mean something to
  screen-reader users. (Plain `<dl>` markup is already allowed by the sanitiser; this
  is an authoring pattern + demo, not new code.)
- Documented and demonstrated in the demo course.

### 3.6 Image gallery — `c-image-grid` (no carousel)

- **No carousel** (UX-discouraged; hard to make accessible). For multiple images,
  support two patterns:
  1. a vertical **stack of captioned `c-picture` figures** (authoring pattern), and
  2. a static **thumbnail grid** (`c-image-grid`) where each thumbnail opens the
     **existing picture lightbox** modal.
- **A11y**: each thumbnail is a real keyboard-operable trigger with an accessible
  name; each image has proper `alt`; grid reflows on small screens; no auto-advance,
  no motion.
- The grid's Alpine component lives in the **content_engine** app (§2.3). It can
  follow the `pictureModal` pattern / share the same lightbox approach as `c-picture`
  rather than inventing a new modal system — subject to the same app-ownership note in
  §3.4 (resolve in the structure review).

### 3.7 Data table — `c-table`

> **Name**: use `c-table`. **Do not** call it `c-data-table` — that name is already
> taken by a different, unrelated grid component in the `base` app (the
> educator/panel list-view table). Cotton resolves templates by name globally, so the
> content table needs its own free name.

- **Markup**: semantic `<table>` with `<caption>` (first child), `<th scope="col|row">`,
  `<thead>/<tbody>`. Authors write the table as a markdown table (the pipeline already
  enables the `tables` extension) or HTML; the widget supplies the caption + the
  responsive wrapper + brand styling.
- **Responsive**: wrap in a **focusable horizontally-scrollable** container
  (`overflow-x:auto`, `tabindex="0"`, labelled via the caption). **No**
  automatic row-stacking (it strips semantics and breaks comparison tables). Sticky
  header / scroll cue as feasible.
- **Build this purpose-specific for content — do NOT reuse the existing
  `c-data-table` component or the `scrollTableLabels` Alpine component /
  `scroll-table-labels.html`.** Those belong to another part of the app and carry
  their own assumptions (sortable/paginated grids; mobile floating row-label overlays
  driven from the first column). The content table is its own widget with its own
  (new) template and, if interactivity is needed, its own small Alpine component. Keep
  the two independent.
- **Scope**: the table does table things — caption, headers/`scope`, body, brand
  styling, and the responsive wrapper. No status-pill widget in this pass (status
  pills are likely a later addition).
- **Attributes** (final in plan): `c-table` → `caption` (plus any minimal options
  the responsive wrapper needs).

### 3.8 Code / log block — `c-code-block`

- **Markup**: brand-styled monospace `<pre><code>` with optional **filename/title
  chrome** (a header bar). Uses `font-mono` token + brand "terminal/log" styling.
- **No syntax highlighting, no copy button** (both deferred — each adds a dependency
  and/or sanitiser surface). These remain candidates for a later scoped decision; do
  not implement now.
- **Responsive / reflow**: long lines use a **focusable** (`tabindex="0"`) scroll
  container (`overflow-x:auto`) **or** soft-wrap (`white-space:pre-wrap`) — choose per
  attribute; logs/commands often prefer scroll, prose-like output prefers wrap. Plan
  decides the default + optional `wrap` attribute.
- **Attributes** (final in plan): `title`/`filename` (optional), optional `language`
  label (display only, no highlighting), optional `wrap`.
- Note: the pipeline **disables indented code blocks** but supports fenced code; the
  widget styles/【wraps】 a code slot. Confirm in plan whether the slot is a fenced
  block or raw text.

## 5. Security considerations (for the plan security review)

1. **KaTeX (`c-equation`)** — client-side rendering of untrusted author LaTeX. Must
   set `trust: false` and a safe macro configuration to block `\href`/`\url`/
   `\includegraphics` `javascript:`/resource injection. New static assets (JS/CSS/
   fonts) are added. No nh3 allowlist change for KaTeX *output* (renders post-
   sanitise), only the `c-equation` tag is allowlisted.
2. **New allowlist entries** — every new tag/attribute widens
   `MARKDOWN_ALLOWED_TAGS`. Keep attribute sets minimal; author input that influences
   styling must map to **fixed role tokens / fixed classes**, never raw CSS or hex
   passed through to the DOM.
3. **Iframe embeds** (`c-youtube`) — keep the existing constrained YouTube embed URL
   and `referrerpolicy`.
4. **File lookups** (`c-picture`, image grid) — continue using `get_file_by_path`
   against the `content_instance`; no path traversal beyond the content root.

## 6. Success criteria

- All eight in-scope widgets render correctly through the markdown→sanitise→cotton
  pipeline and are registered in `MARKDOWN_ALLOWED_TAGS`.
- Each widget works and looks correct in **both** the default and first_class themes,
  using role tokens only (no hardcoded hex).
- **Accessibility**: native semantics used; colour never the sole signal; all
  interactive elements (lightbox triggers, image grid, scroll
  containers) are keyboard-operable with visible focus; wide content is in focusable
  scroll containers; `prefers-reduced-motion` respected; iframe titled; tables have
  caption + `scope`.
- `c-equation` typesets real math via KaTeX with `trust:false`; long equations scroll
  in a focusable container; malformed LaTeX degrades gracefully.
- The existing `c-picture` lightbox becomes **keyboard-openable** with proper dialog
  semantics (regression-fix for current `<img @click>` trigger).
- `c-callout` and all existing demo content continue to work unchanged.
- The new `functionality_demo_content_widgets` course loads via `content_save` and
  demonstrates every widget and every option; it serves as the QA fixture set.
- Tests cover rendering/sanitisation of each new tag (functionality, not CSS classes).
