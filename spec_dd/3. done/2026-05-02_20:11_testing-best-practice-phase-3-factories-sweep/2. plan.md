## Phase B — Add the missing factories

Each new factory follows the conventions in `fls:testing` and `fls-claude-plugin/resources/factory_boy.md`. Bring up the `fls:testing` skill once before starting Phase B and refer back to it for any pattern question (Sequence, SubFactory, RelatedFactory-via-Trait, etc.).

### B1. `SiteFactory` in `freedom_ls/accounts/factories.py`

Rationale for app placement: `Site` is the django.contrib.sites model — there is no FLS app that "owns" it. `accounts` already depends on `Site` (the `SiteSignupPolicy` model lives there) and is a low-level app most other apps already import from, so adding `SiteFactory` there avoids new cross-app dependencies. Confirm no `SiteFactory` exists anywhere first (`grep -rn "class SiteFactory" freedom_ls`).

**Important — base class:** `Site` is *not* a site-aware model; it *is* the site dimension. Subclass `factory.django.DjangoModelFactory` directly (the spec calls this out explicitly under "Site is the exception").

Pseudocode:

```python
# freedom_ls/accounts/factories.py
from django.contrib.sites.models import Site

class SiteFactory(factory.django.DjangoModelFactory):
    """Factory for django.contrib.sites.models.Site.

    NOT a SiteAwareFactory — Site is the site dimension, not a site-aware model.
    The Site this factory creates is the kind of object you might pass into
    `mock_site_context`, not one created under it.
    """

    class Meta:
        model = Site
        django_get_or_create = ("name",)  # avoid UniqueConstraint clashes across tests

    name = factory.Sequence(lambda n: f"TestSite{n}")
    domain = factory.LazyAttribute(lambda obj: f"{obj.name.lower()}.example.com")
```

Notes:
- `django_get_or_create = ("name",)` is defensive: several existing tests use stable site names like `"ForcedSite"` / `"DomainSite"` / `"SiteB"` and assert on them. With get-or-create, calling `SiteFactory(name="ForcedSite")` twice in different test setups won't blow up on the unique constraint.
- Do **not** import `_thread_locals` or anything from `site_aware_models` here — `Site` is upstream of FLS site-awareness.

### B2. `SiteSignupPolicyFactory` in `freedom_ls/accounts/factories.py`

`SiteSignupPolicy` is a `SiteAwareModel` (verified at `freedom_ls/accounts/models.py:89`). Subclass `SiteAwareFactory`.

There is a `UniqueConstraint(fields=["site"])` so we either use `django_get_or_create` on `site` or accept that callers must pass distinct `site` values. The existing test patterns use `update_or_create`, which suggests get-or-create semantics fit the codebase's intent.

Pseudocode:

```python
# freedom_ls/accounts/factories.py
from freedom_ls.accounts.models import SiteSignupPolicy
from freedom_ls.site_aware_models.factories import SiteAwareFactory

class SiteSignupPolicyFactory(SiteAwareFactory):
    """Factory for SiteSignupPolicy.

    Site-aware: by default the policy is created under the current
    `mock_site_context` site. Pass `site=other_site` only when the test's
    purpose is cross-site behaviour (e.g. policy for a non-current Site).
    """

    class Meta:
        model = SiteSignupPolicy
        django_get_or_create = ("site",)  # mirrors UniqueConstraint(fields=["site"])

    allow_signups = True
```

### B3. `panel_framework.tests` — `_make_stub` helper

Spec rule: `StubModel` is a test-only model defined inside a conftest via `editor.create_model(...)`. It must NOT live in a regular `factories.py`. Add a small helper in `freedom_ls/panel_framework/tests/conftest.py`, alongside the existing `make_staff_user`:

```python
# freedom_ls/panel_framework/tests/conftest.py
def _make_stub(name: str | None = None, **kwargs) -> StubModel:
    """Create a StubModel for tests. `name` defaults to a unique value."""
    if name is None:
        name = f"stub-{next(_counter)}"
    return StubModel.objects.create(name=name, **kwargs)


def _make_stub_child(parent: StubModel, **kwargs) -> StubChild:
    """Create a StubChild parented to `parent`."""
    return StubChild.objects.create(parent=parent, **kwargs)
```

The existing `_counter = itertools.count(1)` already in the conftest is the source of unique values — reuse it; don't add another counter.

Export both helpers from the conftest module so tests can `from .conftest import _make_stub, _make_stub_child`.

Note: the call site `panel_framework/tests/test_panel_actions.py:133` does `StubModel.objects.get(name="New Item")` — that's a *read*, not a create, and stays as-is. Also the assertions like `StubModel.objects.filter(name="...").exists()` are reads and stay.

## Phase C — Sweep call sites

For each call site, replace `.objects.create(...)` with the appropriate factory call **without changing the meaning of the test**:

- If the original passed a specific name/value used by an assertion, the factory call passes the same value.
- If the original relied only on defaults, use the bare `Factory()` form.
- Site-aware factories: rely on `mock_site_context` for the `site` field. Pass `site=...` explicitly only where the test's purpose is cross-site behaviour.

### C1. accounts tests

**`freedom_ls/accounts/tests/test_user_manager.py:17-18`** — currently uses bare `Site.objects.create(...)`. The test forces a non-current site via `FORCE_SITE_NAME` and explicitly works *outside* `mock_site_context`. Use `SiteFactory(name="ForcedSite", domain="forced.example.com")` and `SiteFactory(name="testserver", domain="testserver")`. The user creation lines that follow (`User(email=..., site=...)` directly) also bypass `UserFactory` deliberately to set explicit `site`; leave those as-is — they're not `.objects.create(...)` and the spec doesn't sweep `Model(...)`+`.save()` patterns.

**`freedom_ls/accounts/tests/test_allauth_signup_policy.py:53-56`** — same pattern, in `test_is_open_for_signup_respects_force_site_name`. Replace:
- `Site.objects.create(domain="forced.example.com", name="ForcedSite")` → `SiteFactory(name="ForcedSite", domain="forced.example.com")`
- `Site.objects.create(domain="testserver", name="DomainSite")` → `SiteFactory(name="DomainSite", domain="testserver")`
- `SiteSignupPolicy.objects.create(site=forced_site, allow_signups=False)` → `SiteSignupPolicyFactory(site=forced_site, allow_signups=False)` — explicit `site=` is correct here per the spec ("explicit override is reserved for tests whose purpose is cross-site behaviour").

The sibling tests in the same file at lines 27/41 use `SiteSignupPolicy.objects.update_or_create(...)`. Those aren't `.objects.create(...)` — leave them alone.

### C2. role_based_permissions tests

- **`tests/test_models.py:94`** — `Site.objects.create(domain="other.example.com", name="Other")` → `SiteFactory(name="Other", domain="other.example.com")`.
- **`tests/test_utils.py:98-104`** — `ObjectRoleAssignment.objects.create(user=..., content_type=..., object_id=..., role=..., site=...)`. The factory's GFK convenience is `target_object=...`. Prefer that:

      user=user, target_object=mock_site_context, role="instructor",
