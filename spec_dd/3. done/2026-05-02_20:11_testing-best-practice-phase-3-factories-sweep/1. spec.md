# Testing best practice — Phase 3: factories sweep

## Summary

Inventory the models that don't yet have a factory, add the missing factories, then replace the remaining `.objects.create()` call sites in the test suite (audit baseline: ~16; re-inventory at implementation time) with the new and existing factories.

Parent context: see `spec_dd/3. done/2026-04-28_07:25_testing-best-practice/1. spec.md` (Phase 1) for shared decisions, skip list, and parallelism map. Citations live in `spec_dd/3. done/2026-04-28_07:25_testing-best-practice/research_audit.md` §4 and §9.

**Depends on:** Phase 2 (tooling quick wins).

**Can run in parallel with:** every other Phase 3 spec and the optional Phase 4 E2E spec — *with the caveat that this spec and `testing-best-practice-phase-3-cleanup-flaky-and-redundant-tests` may share files; treat as parallel-with-merge-care*. Whichever lands second rebases.

---

## Why this matters

`.objects.create(...)` in tests has two persistent costs:

- **Drift.** When a model gains a `NOT NULL` field, every direct-create call site has to be updated by hand. A factory absorbs the change in one place.
- **Fixture explosion in graphs.** Cohort → Course → Topic → Form → Activity → Progress is a real graph in this codebase. Without `SubFactory` / `RelatedFactory` discipline (Phase 1 documented this in `fls:testing`), each test rebuilds the graph from scratch.

The audit identified ~16 sites that should be factories. Several of those sites exist *because the factory doesn't exist yet* — Site, SiteSignupPolicy, role-assignment models, and the panel framework's `StubModel` are the recurring missing pieces.

## Scope

### Inventory missing factories

Run a fresh inventory at implementation time (the audit is a snapshot — some entries below have already been resolved by sibling work, and new `.objects.create()` sites may have appeared). Starting points from `research_audit.md` §4 and §9:

- `Site` (note: this is `django.contrib.sites.models.Site`, not an FLS site-aware model — see "Site is the exception" under Decisions made)
- `SiteSignupPolicy`
- `ObjectRoleAssignment` — verify whether `ObjectRoleAssignmentFactory` already exists in `freedom_ls/role_based_permissions/factories.py` before adding
- `SystemRoleAssignment` — verify whether `SystemRoleAssignmentFactory` already exists in `freedom_ls/role_based_permissions/factories.py` before adding
- `panel_framework.StubModel` (note: `StubModel` is defined inside `freedom_ls/panel_framework/tests/conftest.py` as a test-only stub created via `editor.create_model()`; treat its "factory" as a small helper colocated with the conftest rather than an entry in a regular `factories.py`)
- Any other model that appears in `.objects.create(...)` more than once across `freedom_ls/**/tests/`.

For each: confirm a factory genuinely doesn't exist (grep `class.*Factory` near the model), then add one in the app's `factories.py` (create the file if it doesn't exist).

### Add the factories

Apply the patterns documented in the Phase 1 update to `fls:testing` and the reference in `${CLAUDE_PLUGIN_ROOT}/resources/factory_boy.md`:

- Subclass `SiteAwareFactory` (from `freedom_ls/site_aware_models/factories.py`) for any FLS site-aware model. Plain `factory.django.DjangoModelFactory` is only correct for non-site-aware models.
- `SubFactory` for required FKs.
- `RelatedFactory` for reverse-side relationships built as part of "this object plus its dependents". Default to wrapping `RelatedFactory` in a `Trait` so the related row is opt-in (per `factory_boy.md`).
- `Trait` for variants — invoked as a boolean kwarg, e.g. `SiteFactory(with_signup_policy=True)`. (Traits are declared inside an inner `class Params:` and toggled by name; there is no `trait="…"` kwarg.)
- `post_generation` for many-to-many wiring or post-save side effects.
- Use `factory.Faker` for realistic strings; use `factory.Sequence` for unique-constrained fields where deterministic values matter.
- Site-aware models: factories must respect `mock_site_context` (do not bypass site filtering). The `SiteAwareFactory` base class wires the `site` field automatically from the request context, so the routine call is `MyFactory(...)` with no `site=`. Explicit override (`MyFactory(site=other_site)`) is reserved for tests whose purpose is cross-site behavior — e.g. `freedom_ls/accounts/tests/test_allauth_signup_policy.py`, where `SiteSignupPolicy` is deliberately created for a non-current Site. Don't set `site=` as a workaround for a missing `mock_site_context` fixture.

### Sweep call sites

Replace the `.objects.create()` call sites identified by `research_audit.md` §4 — and any others surfaced by a fresh grep at implementation time — with the new or existing factories. The audit cited ~16 sites; the actual current count will differ as the suite has churned, so use the fresh inventory as the source of truth. Where a strengthen-view-assertions PR has left a `# TODO: factory needed — Phase 3 factories sweep` comment, resolve it.

Legitimate exceptions (do not sweep):

- `freedom_ls/site_aware_models/tests/test_factories.py` and similar tests whose subject under test is the factory machinery itself.
- `StubModel.objects.create(...)` calls where `StubModel` is the test-internal stub model from `freedom_ls/panel_framework/tests/conftest.py` — these become a thin `_make_stub(...)` helper rather than a `factory_boy` factory in an app's `factories.py`.

When sweeping:

- Don't change the *meaning* of the test. If the original `.objects.create(name="Foo")` set a specific name for assertion purposes, the factory call should set the same name (`MyFactory(name="Foo")`).
- If a factory's defaults are sufficient, use the bare `MyFactory()` form — explicit is better than implicit, but only where the value matters.

### Explicitly out of scope

- No new tests written. This spec is a substitution sweep + factory additions, not coverage work.
- No deletion of brittle tests. The cleanup spec already ran.
- No view-assertion strengthening, no parametrize, no tautology fixes.
- No production model changes. If a model is genuinely awkward to factory (cycles, required-on-creation side effects), document the awkwardness in the PR and file a follow-up; don't refactor the model in this spec.
- No skill edits.

## Decisions made

- **Add factories to the app that owns the model**, not a central `factories/` package. Matches existing convention.
- **`mock_site_context` is mandatory for site-aware factories.** Tests that build a site-aware object outside a site context were latent bugs anyway; this is the right time to catch them.
- **Site is the exception.** `django.contrib.sites.models.Site` is itself the site dimension, not a site-aware model. A `SiteFactory` for it should subclass `factory.django.DjangoModelFactory` directly, not `SiteAwareFactory`, and the resulting `Site` row is the kind of object you might pass *into* `mock_site_context`, not one created under it.
- **Don't rewrite tests beyond the substitution.** Mid-sweep urge to "while I'm here, also …" must be resisted. Other Phase 3 specs cover those concerns.
- **Merge-care with cleanup spec.** Both edit test files. Whichever lands second rebases — no special tooling, just attention.

## Success criteria

1. A factory exists for every model that previously had ≥1 `.objects.create()` call in tests (per the fresh inventory), with the explicit exception of test-internal stubs like `panel_framework.tests.conftest.StubModel`, which use a colocated helper instead.
2. The `.objects.create()` call sites identified by the fresh inventory (seeded by `research_audit.md` §4) are replaced with factory calls, except for the legitimate exceptions listed under "Sweep call sites". Any `# TODO: factory needed — Phase 3 factories sweep` markers from sibling specs are resolved.
3. New factories follow the patterns documented in `fls:testing` and `${CLAUDE_PLUGIN_ROOT}/resources/factory_boy.md` (`SiteAwareFactory` base class for site-aware models, `SubFactory`, `RelatedFactory` (typically opt-in via a `Trait`), `Trait`, `post_generation` where appropriate).
4. Site-aware factories respect `mock_site_context` by default. Explicit `site=` override is used only in tests whose purpose is cross-site behavior (e.g. `SiteSignupPolicy` for a non-current site).
5. `uv run pytest` passes under the Phase 2 gates.
6. No production code changes. No skill edits. No new tests.
