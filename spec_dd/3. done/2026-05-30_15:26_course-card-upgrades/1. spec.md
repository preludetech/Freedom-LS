# Course-card upgrades & dashboard cleanup

## Summary

The learner dashboard (currently `student_interface.home`) is the entry point for every signed-in learner. Today it is bare: a generic course-list partial loaded over HTMX, no per-course recognition cues, no clear state indication, and an anonymous-user branch that never fires in practice (`LOGIN_REDIRECT_URL = "/"`).

This spec covers two related but independently shippable pieces of work:

1. **Course cards** — give every course a stable visual identity (semantic icon + a stable per-course accent slot) and render three visually distinct cards for the *In progress* / *Not started* / *Complete* states.
2. **Dashboard cleanup** — rename Home → Dashboard, require login, drop the HTMX-on-load round trip, and delete the now-dead `partial_list_courses` view and its tests.

Both pieces land on the same template (`home.html` → `dashboard.html` after rename), so they share a worktree even though they could merge separately.

The idea file (`idea.md`) and three research notes (`research_palette_mapping.md`, `research_reference_implementations.md`, `research_ux_best_practices.md`) are the canonical inputs and explain *why* most of the decisions below were made; this spec is the contract the implementation has to meet.

## 1. Course icon — data & defaults

### Why

Learners scan dashboards by recognition, not reading. Canvas-style colour blocks plus a small icon are the cheapest way to make a six-card grid look like six distinct things. The icon is content authored alongside the course (in markdown frontmatter), not a file upload — uploaded images are explicitly out of scope.

### Decisions

- **Add a new `"course"` entry to `SEMANTIC_ICON_NAMES`** and to all four icon-set mappings (heroicons, lucide, tabler, phosphor). It is the default when no icon is specified. Reasonable glyphs:
  - heroicons: `academic-cap`
  - lucide: `graduation-cap`
  - tabler: `school`
  - phosphor: `graduation-cap`
  Each must pass the existing `freedom_ls.E003`/`E004` system checks (verified at `manage.py check`).
- **Add `icon` and `icon_fallback` fields to the Course pydantic schema** (`freedom_ls/content_engine/schema.py`):
  - `icon: str | None = Field(None, description="Semantic icon name from SEMANTIC_ICON_NAMES, or a literal glyph name (e.g. 'drone') resolved against the active icon set.")`.
  - `icon_fallback: str | None = Field(None, description="Optional explicit '<iconset>:<name>' reference (e.g. 'phosphor:drone') used only when 'icon' fails to resolve in the active icon set.")`.
  - Both default to `None` in the schema. The Django model resolves the ultimate fallback to `"course"` at read time so theme-level overrides still apply.
- **Add `icon` and `icon_fallback` fields to the Course Django model** (`freedom_ls/content_engine/models.py`):
  - `icon = models.CharField(max_length=64, blank=True, default="")`.
  - `icon_fallback = models.CharField(max_length=80, blank=True, default="")` — wider than `icon` to accommodate the `<iconset>:<glyph>` form.
  - Empty string == "not set". Storing the default explicitly would force a data migration when the default changes; storing empty keeps the seam.
  - Templates and helpers pass the **raw** `course.icon` and `course.icon_fallback` values to the resolver. The empty-string sentinel is meaningful — step 1 of the resolution order treats it as "use the default semantic name". No `icon_name` property; the resolver owns defaulting.
- **Resolution order at render time** (encoded once in the `student_interface` course-icon helper, see "Rendering" below):
  1. If `icon` is empty → semantic `"course"` (default).
  2. If `icon` is in `SEMANTIC_ICON_NAMES` → semantic path. Always resolves — semantic names are mapped per-set. `icon_fallback` is unused in this branch.
  3. Else if `icon` is a literal glyph present in the active icon set's iconify JSON → render that glyph. `icon_fallback` is unused.
  4. Else if `icon_fallback` is set and parses as `<iconset>:<name>` → load that set's iconify JSON directly and render `<name>`. This is the "use phosphor's drone glyph even on a heroicons site" path.
  5. Else → semantic `"course"`. Documented graceful degradation, not a misconfiguration.
- **Validate at save time** (in `save_course` / on the model `clean()`):
  - `icon` empty + `icon_fallback` empty: OK.
  - `icon` empty + `icon_fallback` set: `ValidationError` — a fallback without a primary makes no sense.
  - `icon` set:
    - In `SEMANTIC_ICON_NAMES`: OK. Pairing with `icon_fallback` is allowed — the fallback simply goes unused because the semantic step resolves first. Authors don't have to know whether a name is semantic when they author content; the resolver works it out.
    - Else: must be a literal glyph in **at least one** registered icon set. Catches typos like `dron`. Pairing with `icon_fallback` is allowed and recommended for cross-set robustness.
    - **Variant pairing check (literal glyphs only)**: for every registered icon set whose JSON contains the unsuffixed `icon`, every variant in that set's `variants` dict that has a non-`None` suffix must also have its suffixed glyph (`<icon><suffix>`) present in the JSON. If any are missing, raise `ValidationError` listing the offending `<set>:<variant>` pairs. Rationale: the active set is theme-driven and unknown at save time, so we validate against every set that could be active. Catches half-shipped glyph pairs at authoring time rather than at first render.
  - `icon_fallback` set: must match `^[a-z0-9_-]+:[a-z0-9_-]+$`, the `<iconset>` part must be a key in `ICON_SETS`, and the `<name>` part must be present in `load_iconify_data(<iconset>)["icons"]`. Validation runs against the named set's JSON, not the site's active set. The same variant pairing check applies to the named set: every variant with a non-`None` suffix must have its suffixed glyph present, else `ValidationError`.
  - Whitespace, empty-after-strip, multiple colons in `icon_fallback`: `ValidationError`.
  - Error messages should call out the resolution chain and what each part needs to look like, so authors can fix the value without re-reading the spec.
- **Persist both fields in the `content_save` command**: no special handling required — `save_with_uuid` already iterates fields from the pydantic dump, both fields are plain strings, the existing `model_dump` flow picks them up.

### Rendering

The `c-icon`/`{% icon %}` rendering path currently takes a single `name` argument and treats it as a semantic name. The course-card resolver lives **outside** the icons app — `IconBackend` must stay portable and reusable in other Django projects, and "course icon resolution" is a `student_interface` concern.

- New module-level helper in `freedom_ls.student_interface` (e.g. `freedom_ls/student_interface/course_icon.py`): `render_course_icon(icon: str, icon_fallback: str = "", *, variant: str = "outline", css_class: str = "size-5", aria_label: str = "") -> str`. It walks the resolution order above by composing existing primitives — `get_icon_backend().render(...)` for the semantic step, and `load_iconify_data(<set>)` plus a small literal-lookup helper (or direct JSON walk) for the literal-active and explicit-fallback steps.
- The existing semantic-only `render(name, ...)` entrypoint and the `IconBackend` base class are unchanged. Every existing caller keeps working. No new abstract method on `IconBackend`.
- Variant handling: semantic step (2) honours the requested variant via the existing mapping. Literal-active step (3) and explicit-fallback step (4) render the named glyph directly; the variants dict consulted is the one for the **set being read** (active set for step 3, the explicitly-named set for step 4). If that set's `variants` dict has a non-`None` suffix for the requested variant, the suffixed glyph (`<name><suffix>`) **must** be present in the JSON — save-time validation guarantees this for any course that loaded successfully. If at render time the suffixed glyph is missing (e.g. an icon set was modified post-save), raise an `IconResolutionError` rather than silently rendering the unsuffixed name; silent fallback hides regressions in the icon set.
- The template wires both values via a `student_interface` template tag (or a thin cotton wrapper): `{% course_icon course.icon course.icon_fallback %}`. Author writes one or both in the markdown frontmatter; the template never branches.

### Demo content

Update **one** course in `demo_content/` to set `icon: star` (or another distinctive existing semantic name) in frontmatter. The other two courses keep no icon, so the demo exercises both the explicit and default paths. Suggested target: `demo_content/functionality_demo_standard_markdown/course.md` already has the most polished frontmatter — picking `icon: notes` would suit "Standard Markdown".

### Migration

A schema migration is required to add `Course.icon`. Default is empty string, no data migration needed.

### Out of scope

- Per-course image uploads.
- Editor-controlled icon picker UI in the admin (the field is editable via plain admin text input — that's enough for now).
- Adding new semantic names beyond `"course"`.

## 2. Per-course accent colour

### Why

A course's identity has to survive theme switches, dark mode, and brand recolours. Hashing a slug to an arbitrary HSL hex — like `branch_name_to_color` does for the dev-debug branch banner — would lock us out of the theme system. The fix is to map each course to a slot in a curated, **separately themeable** colour series and let the theme resolve the actual colour at render time.

The palette is deliberately narrow: five vibrant slots is enough variety to make a six-card grid read as distinct things without looking like a toy — see `research_palette_mapping.md`. The series is kept **separate from the semantic UI role tokens** (`primary`/`secondary`/…) so card vibrancy can be tuned without recolouring buttons or badges.

### Decisions

- **Palette is a separate, neutral slot series**: `freedom_ls/content_engine/course_accent.py` defines `PALETTE: tuple[str, ...] = ("1", "2", "3", "4", "5")`.
  - The slot keys are **neutral identifiers, not role names**. Each slot `N` maps to a `--fls-course-accent-<N>*` token group in the theme and to `.course-accent-<N>` / `.course-progress-<N>` component classes. The three stay in lockstep.
  - Order is fixed — never reorder; reordering would remap every existing course's accent.
- **Persist the slot on the model.** `Course.accent_slot = models.PositiveSmallIntegerField(default=0, editable=False)`.
  - Assigned **once at row creation** in `Course.save()`: when the row is new, `accent_slot = Course.objects.filter(site_id=self.site_id).count() % len(PALETTE)`. This cycles slots by per-site creation order, so the catalogue stays **evenly distributed** across the five slots (no stochastic clustering) and a course keeps the same colour everywhere it appears. Updates never reshuffle the slot.
  - Why store rather than recompute: balanced distribution is only achievable with knowledge of the existing catalogue, which a pure per-title function can't see. Distribution is deterministic and stable; there is no per-request hashing.
- **Expose the slot via a model property**: `Course.accent_role` returns `PALETTE[self.accent_slot]`. Templates read `course.accent_role` directly — no view-side annotation needed, unlike `progress_percentage` / `next_up` which remain view annotations.
- **Templates render** the icon tile and progress bar with the per-slot component classes:
  - icon tile background: `class="course-accent-{{ course.accent_role|default:'1' }}"` — a vibrant gradient with a contrasting icon colour baked in.
  - progress bar: `class="course-progress-{{ course.accent_role|default:'1' }}"` — soft track + solid fill.
  - card title: stays on `text-on-surface` (via `text-inherit` on the title link). Accent does not bleed into typography.
- **Reserve a `Course.accent_override` seam, but do not build it.** If/when a per-course override field is added, prefer it over the computed slot — i.e. `course.accent_override or PALETTE[course.accent_slot]`. The seam is one `or` away; don't pre-implement.

### Accent token series & soft tints

The card accent series and a general set of role soft-tints both live in `freedom_ls/themes/default/static/themes/default/theme.css`.

- **Course-accent gradient series** — five slots, each with a `-from` / `-to` / `-icon` triple, a composite gradient, and a soft tint for the progress track:
  ```
  --fls-course-accent-1-from: #4F46E5;  --fls-course-accent-1-to: #2563EB;  --fls-course-accent-1-icon: #FFFFFF;  /* indigo → blue   */
  /* …slots 2–5 (violet→fuchsia, teal→cyan, emerald→green, amber→orange)… */
  --fls-course-accent-1: linear-gradient(135deg, var(--fls-course-accent-1-from) 0%, var(--fls-course-accent-1-to) 100%);
  --fls-course-accent-1-soft: color-mix(in oklab, var(--fls-course-accent-1-to) 14%, var(--color-surface));
  ```
  Themes override the per-slot `-from` / `-to` / `-icon` values only (e.g. `freedom_ls/themes/first_class/...`); the composite gradient and soft-tint formulas are inherited. The matching `.course-accent-<N>` and `.course-progress-<N>` component classes are defined in `tailwind.components.css`.
- **Role soft-tints** — additionally declare `--color-{role}-soft` tokens for the five role tokens (12% `color-mix` of the role into `--color-surface`). These are general-purpose (e.g. the completed-card badge uses `bg-success-soft text-success`); they are **not** what the accent tiles/progress bars use. Tailwind v4 picks them up as `bg-primary-soft`, `text-primary-soft`, etc.
- The mix percentages (14% for accent soft tints, 12% for role soft tints) are starting points from `research_palette_mapping.md`; dark themes may bump them. A later theme-tuning concern, not a blocker.

### Verification

`test_course_accent.py` covers the whole accent feature in one file:

- the `PALETTE` contract: exactly five entries, and the slot keys are **decoupled from the UI role names** (recolouring cards never recolours buttons/badges);
- slot assignment: slots cycle `0→1→2→3→4→0` in per-site creation order (keeping the catalogue even), `accent_role` exposes the matching `PALETTE` keys, per-site slot sequences are independent, and updating a course does not reshuffle its slot.

### Out of scope

- A per-course `accent_override` field (mentioned only as a future seam).
- Auto-contrast checking at render time. The (slot, surface) pairs are fixed per theme — verify in theme review, not at every render.
- Avoiding adjacent-card colour collisions in the grid. Even-distribution-by-creation-order makes runs of identical slots unlikely but does not guarantee a specific grid arrangement; solving that exactly would break the "same course always looks the same" contract.

## 3. Dashboard course card

### Why

Today the dashboard renders a single `course-card` partial that handles all three states (In progress / Not started / Completed) by branching on `course_status`. The branches share too much markup — every change has to be made three ways and the visual differences end up muddled. Reference implementations (Coursera, Canvas, edX, LinkedIn Learning) all converge on three visually distinct cards with a stable silhouette — see `research_reference_implementations.md`.

### Card silhouette

All three states share the same outer structure so the grid stays visually calm:

- A top-aligned accent tile (the course icon centred on a vibrant `course-accent-{slot}` gradient).
- An eyebrow label.
- The course title.
- A state-specific content slot (progress bar / preview affordance / completion badge).
- The whole card is the click target via the **stretched-link pattern** (single semantic `<a>` on the title + `::before` overlay extending the hit zone). Never wrap the card in `<a>`.

### State 1 — In progress

- **Eyebrow**: `In progress`
- **Body**: course title (linked); `Next up: <next-item-title>` in muted small type.
- **Progress**: `<progress role="progressbar" aria-valuemin="0" aria-valuemax="100" aria-valuenow="{{ percent }}" aria-label="Progress for {{ course.title }}">` + numeric percent next to the bar (e.g. `28%`). The bar uses the `course-progress-{accent_slot}` component class (soft track + solid fill), keeping the per-course identity coherent.
- **Click destination**: the same URL the existing `Continue` button uses — `student_interface:view_course_item` for the next ready/in-progress item. Compute it server-side using the existing `get_course_index` helper or a small helper layered on top; do not re-implement progress traversal.
- **"Next up" data**: derived from `get_course_index` — pick the first child whose `status` is `IN_PROGRESS`, falling back to the first `READY` child. Display its `title`. If none exists (rare: registered but no ready item), omit the line — never render `Next up: ` with a blank.

### State 2 — Not started

- **Eyebrow**: `Not started`
- **Body**: course title; a `Next up:` line when one exists (shares the in-progress markup); and a **0% progress bar** rendered as a deliberate visual anchor so not-started cards line up cleanly beside in-progress cards in a mixed grid. No deadline, no description on the card itself.
- **Click destination**: the **course preview** (title, subtitle, description, table-of-contents, and a `Start` button when the learner is allowed to start).
- **Presentation split**:
  - **Desktop (≥ md breakpoint)**: open the preview as a modal using the existing `<c-modal>` component (already used in the current `course_list.html`). The modal trigger is the stretched link; clicking the card opens the modal.
  - **Mobile (< md breakpoint)**: the card link routes to a dedicated preview page at `/courses/<slug>/preview/`. The same content is rendered server-side.
- **The modal and the page render identical content** — a single shared template partial drives both. CSS controls which presentation is used (the modal markup hides itself below `md:`; the page route is what runs there).
- **Why split**: modal-on-mobile fights the back gesture and overflows the viewport once a real ToC is present (see `research_ux_best_practices.md`). Going to a dedicated route on mobile keeps everything deep-linkable and refresh-safe.
- **Hard requirement**: the preview content (title, subtitle, description, ToC, start button) must come from a **single template partial** that both the modal and the page include. No copy-paste.

### State 3 — Complete

- **Eyebrow**: `Completed`
- **Body**: course title + a small **check-shape badge** (an icon, not just a coloured pill — state must be conveyed by *text + shape*, not colour alone, per WCAG 1.4.1).
- **Progress region**: replaced by the `Completed` eyebrow + check badge — never render a 100%-filled bar (Carbon explicitly warns against this).
- **Click destination**: the existing `student_interface:course_finish` page.

### Click target & accessibility

- **Stretched-link pattern**: a single `<a>` on the title; a `::before` pseudo-element on the card extends the hit zone; `:focus-within` puts the focus ring on the card outer.
- **One primary action per card.** Future secondary actions (unenrol, share, view certificate) are out of scope; when added, they will sit above the `::before` overlay (`position: relative; z-index: 1`) with their own focus rings.
- **Status conveyed three ways** for every state: eyebrow text + icon/badge shape + colour. Colour is decorative.
- **Focus ring** wraps the whole card via `:focus-within`, not just the title.
- **Title** stays on `text-on-surface`. It is *not* recoloured to the accent — see palette research.
- **Card text selection** must still work. The stretched-link pattern preserves it; wrapping the card in `<a>` does not.

### Empty / fallback states

- **No enrolments**: render a true empty state ("You haven't signed up for any courses yet.") with a clear "Browse courses" CTA. The existing message in `course_list.html` is fine to reuse but should be visually pulled out into an empty-state region rather than buried in a card grid.
- **0% progress on an in-progress card**: use the *Not started* state instead. The Not-started card carries its own 0% progress bar as a visual anchor (see State 2), so the grid stays aligned — what we avoid is the *in-progress* variant showing a zero-filled bar.
- **`get_current_courses` annotates `progress_percentage = 0` when `CourseProgress` exists but no items are completed.** That row is technically "in progress" by registration but visually "not started". Resolve as: if `progress_percentage == 0`, render the *Not started* card variant.

### Template structure

- New partial: `student_interface/templates/student_interface/partials/course_card_in_progress.html`
- New partial: `student_interface/templates/student_interface/partials/course_card_not_started.html`
- New partial: `student_interface/templates/student_interface/partials/course_card_complete.html`
- New shared partial: `student_interface/templates/student_interface/partials/course_preview_content.html` (used by both modal and dedicated mobile page).
- The existing `course_list.html` partials (`current-courses`, `recommended-courses`, `learning-history`, `all-courses`) are updated to dispatch to the right card partial based on state. The old `course-card` partial is **deleted** once nothing references it.
- The `all_courses` page (`/courses/`) reuses the same three card partials with the same state derivation logic. No bespoke layout for that page.

### Out of scope (this section)

- Card secondary actions (unenrol, share, view certificate).
- A "Continue where you left off" hero block above the grid (research recommends it; ship it separately).
- An author-controlled colour override.
- Photographic course thumbnails.

## 4. Dashboard cleanup

### Rename Home → Dashboard

- URL **path** stays at `/`. URL **name** changes from `home` to `dashboard` in `freedom_ls/student_interface/urls.py`. No `LOGIN_REDIRECT_URL` change — `/` already works.
- View function `home` → `dashboard` in `freedom_ls/student_interface/views.py`.
- Template `home.html` → `dashboard.html`.
- Update every `{% url 'student_interface:home' %}` reference. Today: only `course_finish.html` uses it. Update the visible label there ("Return to Home" → "Return to Dashboard").
- Update `course_topic.html` breadcrumb ("Home" → "Dashboard").
- Update `dashboard.html` `<title>` block from `Home` to `Dashboard`.
- The `c-icon name="home"` on the Return-to-Dashboard button can stay — `home` is the icon's *semantic* meaning (go to root); the label change is what matters.

### Require login

- Add `@login_required` to the `dashboard` view.
- Anonymous users hitting `/` are redirected to login and bounced back via `LOGIN_REDIRECT_URL = "/"`.
- The anonymous branch in the dashboard template (the welcome / login-prompt block) is **deleted** — it is unreachable once `@login_required` is in place.

### Drop the HTMX-on-load round trip

- Today: `dashboard.html` renders an empty `<div hx-get="…partial_list_courses" hx-trigger="load">` and the user sees a spinner before content paints. The partial computes everything in-process — there's no remote call or expensive query that justifies the round-trip.
- Change: the `dashboard` view computes `registered_courses`, `completed_courses`, `recommended_courses` directly (using the existing helpers in `student_interface/utils.py`), annotates **every course** (current, completed, and `recommendation.collection`) with `accent_role`, and additionally annotates each in-progress course with its `progress_percentage` (already present) and `next_up` title, then passes them to the template. The course-list partials render directly, no HTMX.

### Delete the dead view & tests

- Remove the `partial_list_courses` view from `views.py`.
- Remove its URL pattern from `urls.py`.
- Remove the four `test_partial_list_courses_*` tests (and any helpers used only by them) in `freedom_ls/student_interface/tests/test_course_list_views.py`.
- The course-list partials in `partials/course_list.html` are *not* deleted — they're reused directly by the dashboard view (and by the all-courses page).
- Default for this spec: **delete**. If we later need a public partial endpoint we can reintroduce it surgically — dead code rots.

### Out of scope

- A separate marketing/landing page at `/`. Anonymous users get the login page; that's enough.
- Any change to `all_courses` (the `/courses/` page) beyond reusing the new card partials.
- Any change to other student-interface views.

## Migration & backward compatibility

- **Schema/model migration**: add `Course.icon` and `Course.icon_fallback`. Both default to empty string; no data migration needed.
- **URL name rename `home` → `dashboard`**: path is unchanged, so external bookmarks and `LOGIN_REDIRECT_URL` continue to work. Internal call sites are limited (one template).
- **Removed view `partial_list_courses`**: the URL `/partials/courses/` returns 404 after the change. No external consumer is expected — it was only used by the dashboard's own HTMX trigger. Tests are updated in the same commit.
- **Model migration `Course.accent_slot`**: add the field (`PositiveSmallIntegerField`, `editable=False`, default 0) and a data migration that backfills existing rows in slug order per site for an even slot distribution. New rows are assigned in `Course.save()`.
- **Theme CSS additions**: adding the `--fls-course-accent-*` slot series and the `--color-{role}-soft` tokens is additive; existing themes that don't override them inherit the defaults. No breakage.
- **Icon registry addition**: adding `"course"` to `SEMANTIC_ICON_NAMES` and to all four mappings is additive. The existing `freedom_ls.E007` system check enforces that all four mappings stay in lockstep — implementation is forced to update all four or `manage.py check` fails.
- **Literal-glyph icons + `icon_fallback`**: both are new; nothing in the codebase produces these today, so adding them is purely additive. Existing semantic-only callers (`{% icon %}` etc.) keep working because they go through `IconBackend.render`, which is unchanged. Only the new course-icon resolver branches on the literal-lookup / fallback paths.

## Success criteria

- A registered learner who logs in lands on `/`, sees an authenticated dashboard immediately (no spinner), and the page is titled "Dashboard".
- An anonymous user who visits `/` is redirected to login, then back to `/` after authenticating.
- The dashboard renders three visually distinct card variants for in-progress / not-started / completed courses, with stable per-course icon and accent role across page reloads and browser sessions.
- A course definition that specifies `icon: notes` in its markdown frontmatter is persisted via `content_save`, and the rendered card uses that icon. A course with no `icon` falls back to the new default `"course"` icon.
- A course definition that specifies a literal glyph name not in `SEMANTIC_ICON_NAMES` (e.g. `icon: drone`) renders that glyph from the active icon set when present. If `icon_fallback: phosphor:drone` is also specified and the active set has no `drone`, the rendered card uses phosphor's `drone` glyph. If neither resolves, the card renders the default `"course"` icon — no broken `<svg>`, no error. A primary `icon` value not present in any registered icon set, or a malformed `icon_fallback`, raises `ValidationError` at `content_save` time.
- A course's accent slot is assigned once at creation and stable thereafter — identical across page reloads, browser sessions, and separate Python processes — and the catalogue is evenly distributed across the five slots.
- Whole-card click works on every state via keyboard and mouse; the focus ring wraps the card; text selection inside the card still works.
- The Not-started state opens a modal preview on desktop and navigates to a dedicated `/courses/<slug>/preview/` page on mobile; both render the same content from a single shared partial.
- `manage.py check` passes — no new icon mapping errors; all four icon sets include the `course` semantic name.
- All existing tests in `freedom_ls/student_interface/tests/` pass after the rename and the dead-test removal. New tests cover: the `PALETTE` contract and `accent_slot` assignment behaviour, icon field round-trip via `content_save`, `@login_required` on the dashboard, dashboard view renders without HTMX, dead URL `/partials/courses/` returns 404.
- No CSS class assertions in tests (per project conventions).

## Open questions for spec review

> **Resolved:** semantic + `icon_fallback` is allowed. The resolution chain is: (1) is `icon` a semantic name? use semantic mapping; (2) else, is `icon` a literal glyph in the active icon set? render it; (3) else, fall back to `<iconset>:<name>` from `icon_fallback`. If `icon` happens to be semantic, the fallback simply goes unused — harmless. Authors don't have to know whether a given name is semantic at authoring time.

> **Resolved:** variant fallback for literal glyphs is loud — never silent. Save-time validation rejects any literal `icon` (or `icon_fallback` glyph) whose registered set has a non-`None` variant suffix without the matching suffixed glyph in the JSON. Render time raises `IconResolutionError` if a suffixed glyph is missing despite validation.
