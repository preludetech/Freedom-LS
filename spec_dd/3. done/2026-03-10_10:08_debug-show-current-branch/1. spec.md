# Spec: Debug Branch Indicator Panel

## Overview

When `DEBUG=True`, all pages display a small fixed-position panel in the bottom-left corner showing the current git branch name. This helps developers quickly identify which branch/worktree they are working on, reducing the risk of making changes on the wrong branch.

## Why This Matters

- Developers working across multiple worktrees or branches can lose track of which branch is serving the current page
- The branch name is not visible in the browser — only in the terminal — so a persistent visual indicator prevents mistakes
- Color-coding by branch name provides instant visual differentiation without needing to read text

## Features

### 1. Branch Detection

Read the current git branch name at server startup.

- Read `.git/HEAD` directly (no subprocess, no third-party dependencies)
- Handle git worktrees: `.git` can be a file containing `gitdir: /path/...` rather than a directory. Follow the gitdir path to find the actual HEAD. This project uses worktrees so this is essential
- Handle detached HEAD: show the short SHA (first 7 characters)
- Return `None` if no git repo is detected (graceful degradation)
- Cache the branch name at first read (lazy singleton). The branch does not change while the server is running, so there is no reason to re-read on every request

### 2. Color Generation from Branch Name

Generate a deterministic color from the branch name so the same branch always produces the same color.

- Hash the branch name using MD5 (`hashlib.md5`)
- Map the hash to HSL values:
  - Hue: 0-360 (full range)
  - Saturation: 0.45-0.75 (constrained for readability)
  - Lightness: 0.40-0.60 (constrained for readability)
- Convert HSL to hex for use as badge background color
- Pick black or white text color based on relative luminance (WCAG contrast algorithm)
- No third-party dependencies: Python's `hashlib` and `colorsys` stdlib modules are sufficient

**Why these constraints:** Unconstrained HSL values can produce colors that are too pale, too dark, or too saturated to read text against. The chosen ranges ensure all generated colors are mid-tone and reasonably saturated.

### 3. Django Context Processor

A context processor that provides branch name and color data to templates.

- Lives in `freedom_ls/base/context_processors.py` (alongside the existing `posthog_config` processor)
- Only registered in `config/settings_dev.py`, NOT in `config/settings_base.py`
  - Follow the same pattern used for `debug_toolbar` and `django_browser_reload`: append to the existing context processors list
  - This means zero overhead in production — the processor is never called when `DEBUG=False`
- Two layers of safety:
  1. The context processor is not registered in production settings
  2. The template uses `{% if debug_branch_name %}` guard so even if somehow loaded, it renders nothing without the context variable
- Returns a dict with keys:
  - `debug_branch_name`: the branch name string (or `None`)
  - `debug_branch_color`: hex background color
  - `debug_branch_text_color`: `#000000` or `#ffffff` for text contrast

### 4. UI Badge in Base Template

A small pill/badge fixed to the bottom-left corner of every page.

- Added to `freedom_ls/base/templates/_base.html` just before `</body>`
- Guarded with `{% if debug_branch_name %}`
- **Collapsed state:** A small colored dot (~40px) showing just the color
- **Expanded state:** Shows the full branch name in `font-mono` text
- **Default state:** Expanded (so branch is immediately visible)
- Click to toggle between expanded and collapsed (not hover — unreliable on touch devices)
- Expand/collapse state persisted in `localStorage` so it survives page reloads
- Scoped inline styles to avoid conflicts with the rest of the app

**Why click instead of hover:** Touch devices don't have hover, and accidental hover interactions are frustrating. Click is intentional and works everywhere.

**Why localStorage:** Developers who prefer the panel collapsed shouldn't have to re-collapse it on every page load.

### 5. Styling

- `position: fixed; bottom: 8-12px; left: 8-12px` — bottom-left is the least contested screen area (avoids chat widgets, scroll-to-top buttons, etc.)
- `z-index: 50` (Tailwind's `z-50`) — high enough to sit above page content but below modals
- Smooth CSS transition for expand/collapse animation
- `font-mono` for the branch name text
- On very small screens (< 480px), hide or make extra compact to avoid obstructing content

## Implementation Location

| Component | File |
|---|---|
| Branch detection + color logic | `freedom_ls/base/context_processors.py` |
| Context processor registration | `config/settings_dev.py` |
| Badge template markup | `freedom_ls/base/templates/_base.html` |

No new files, apps, or dependencies are needed.

## Success Criteria

1. When `DEBUG=True`, every page shows a colored badge in the bottom-left corner displaying the current git branch name
2. The badge color is deterministic — same branch name always produces the same color
3. Text on the badge is readable (sufficient contrast against the background color)
4. Clicking the badge toggles between expanded (full branch name) and collapsed (dot only) states
5. The expand/collapse state persists across page loads via `localStorage`
6. The badge defaults to expanded on first visit
7. When `DEBUG=False` (production), no badge is rendered and no context processor runs
8. Git worktrees are handled correctly (`.git` as a file pointing to `gitdir:`)
9. Detached HEAD shows a short SHA instead of a branch name
10. No third-party dependencies are added
11. The badge does not interfere with existing page content or other fixed-position elements
