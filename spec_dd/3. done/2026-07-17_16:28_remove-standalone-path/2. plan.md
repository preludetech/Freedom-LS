# Plan: remove the standalone deployment path, document concrete-only

Implementation plan for `1. spec.md`. This spec has **no application-code changes**: it deletes six
standalone-only deployment artifacts and rewrites one product doc so the concrete (submodule) path is
the only documented deployment path. There are no models, migrations, views, or Python changes, and
therefore no automated tests to write (TDD does not apply — success is verified by file-absence and a
dangling-reference grep, see §5).

## 0. Context & verified state

- **Target repo:** this repo (`freedom_ls`).
- **Artifacts to delete** (all exist, confirmed present):
  - `docker-compose.yml` — standalone prod stack, three services (`db`, `web`, `nginx`). Sets
    `DJANGO_SETTINGS_MODULE=config.settings_prod`; carries the DB **bind mount**
    (`${DB_DATA_PATH:-~/.fls_postges_prod_data}:/var/lib/postgresql/data`, `:10`) and the log **bind
    mount** (`./logs:/app/logs`, `:26`) landmines.
  - `nginx.conf` — the reverse-proxy config consumed only by the deleted `nginx` service.
  - `Dockerfile` — the standalone image build (superseded by the Spec 5 template Dockerfile).
  - `docker-entrypoint.sh` — build-only companion of the `Dockerfile` (chowns media/logs, `gosu`).
  - `.dockerignore` — build-only companion of the `Dockerfile`; no consumer once it is gone.
  - `docs/how tos/DOCKER_DEPLOY.md` — the stale standalone nginx how-to.
- **Explicitly kept, do not touch:**
  - `dev_db/` (the local dev stack: `dev_db/docker-compose.yaml` + init/helper scripts). It is a
    separate stack and is unrelated to the standalone prod artifacts.
  - `docs/deployment-security-checklist.md` (DB-TLS `:22`, reverse-proxy-header `:36` — provider
    -agnostic, stay correct under Caddy; out of scope per spec §3.2).
- **Doc to rewrite:** `docs/product/deployment.md`.
- **Cross-references to deleted artifacts (repo-wide grep, excluding `spec_dd/`):** the **only** live
  in-repo link to any deleted artifact is `docs/product/deployment.md:122` (the `DOCKER_DEPLOY.md`

  standalone artifacts — leave them.
- **Canonical concrete deployment target:** the template repo
  `git@github.com:preludetech/freedom-ls-concrete-template.git` (a GitHub *template* repository).
  Its in-repo manifest is `fls-claude-plugin/resources/template_repo_manifest.md`. **Spec 5** ships

## 1. Delete the six standalone artifacts (spec §3.1)

Delete these files with `git rm` (so the removal is staged and the deletion is unambiguous):


## 2. Rewrite `docs/product/deployment.md` (spec §3.2)

Goal: the doc stops presenting the in-repo **nginx Compose stack** as the current working setup, and
instead describes the **concrete (submodule) path via the template repo (Caddy)** as *the* — and only
— deployment path. No dangling references to any deleted artifact may remain.

**Approach — fold the concrete-deployment guidance into `deployment.md` itself** (do not create a new
how-to file). The deleted `DOCKER_DEPLOY.md` was the standalone step-by-step; its replacement
step-by-step now lives in the **template repo** (Spec 5 + that repo's own README), so `deployment.md`
becomes the single narrative guide that *points at* the template repo rather than duplicating steps.
This keeps the doc set DRY, creates no new file to link, and lets the "Deployment Guides" section be
removed cleanly (§2c) with nothing left dangling.

**Give the guidance the spec's literal name.** Spec §3.2 calls for "a single **'Deploying a concrete
(submodule-based) project'** guide." Satisfy the literal wording *without* a new file: add a named
`## Deploying a concrete (submodule-based) project` section **inside `deployment.md`** (in place of the
removed "Deployment Guides" section, §2c) that carries the pointer-to-template-repo content — the
canonical, only deployment path. This resolves the "is it a heading or just narrative?" ambiguity
decisively: it is a real, named section, but it lives in the existing product doc, not a new
`docs/how tos/` file.

### 2a. Remove the "CI/CD Pipeline" section (`:39–50`)

Delete the entire section (aspirational GitHub Actions → GHCR → SSH-pull pipeline that is not built in
this repo). The GHCR/CI scaffolding is Spec 5 template-repo work; do not describe it as in-repo. The
"CI itself does run: tests and security scanning" fact is already stated in the Summary (`:9`) and
under ISO 27001 Vulnerability management (`:107`), so removing this section loses nothing factual.

### 2b. Reframe nginx-current / Caddy-planned → concrete-via-template-repo (Caddy)

The doc currently frames nginx as the current in-repo reverse proxy and Caddy as planned-but-unbuilt.
That framing goes false the moment §1 deletes the nginx artifacts. Reframe every such statement so the
**deployment lives in the template repo** and **Caddy is the reverse proxy there**, with no claim that
this (`freedom_ls`) repo ships a runnable prod stack. Touch points:

- **Summary (`:7`)** — replace "Caddy is the planned reverse proxy; the Docker Compose configuration
  currently in this repository uses nginx …" with a statement that concrete projects deploy from the
  template repo, whose Compose stack uses Caddy for automatic HTTPS. Drop the parenthetical
  cross-link to a "Target Architecture" nginx caveat.
- **Target Architecture (`:15`, `:20`, `:28`)** — remove the "planned target, not current state /
  nginx today" caveats and the "(planned; nginx today)" annotations. Present Caddy as the reverse
  proxy in the concrete (template-repo) stack. Keep the architecture diagram; drop the
  "(planned)"/"nginx today" qualifiers on the proxy line.
- **PostgreSQL note (`:30`)** — the "named Docker volume (never a bind mount)" guidance is correct and
  worth keeping; it now describes the template-repo stack. Retain it (it is the antidote to the
  bind-mount landmine we just deleted).
- **ISO 27001 Shared Responsibility → TLS row (`:100`)** — change "TLS encryption (planned via Caddy;
  the current Compose config terminates TLS at nginx)" to describe TLS terminating at Caddy in the
  template-repo stack; drop the nginx clause.
- **Application-Level Facts → "Container-friendly logging" (`:69`)** — this bullet currently ends
  "…it still writes rotating log files under `logs/`, and **the existing `logs/` bind mount is
  unchanged**." That `logs/` bind mount is defined **only** in the `docker-compose.yml` (`:26`) that
  §1 deletes, so the clause becomes a dangling reference to a just-removed landmine. **Fix the
  bind-mount clause only** — drop or reword "the existing `logs/` bind mount is unchanged" so it no
  longer asserts a bind mount that no longer exists in this repo (e.g. end the sentence at "…still
  writes rotating log files under `logs/`."). **Do not** rewrite the rest of the bullet: the
  stdout-only-logging deferral is paired with Spec 1 (stdout logging) and Spec 5 (log-size caps) per
  `spec-order.md`, so leave that narrative intact — this is a one-clause dangling-reference fix, not a
  logging-story rewrite.

Keep the "planned / not-yet-built" language for the pieces that genuinely remain unbuilt and are *not*
tied to the deleted nginx artifacts (Ansible provisioning, Cloudflare front, backups automation,
worker container). Those are unaffected by this spec — do not over-edit them.
