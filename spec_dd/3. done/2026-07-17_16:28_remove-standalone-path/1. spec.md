# Spec 4 (this repo): remove the standalone deployment path, document concrete-only

> **Part of the `support-concrete-project-deployment` effort.** See `spec-order.md` for ordering and
> cross-spec dependencies. The artifact **removals** can run in parallel with Specs 2 and 3 once
> Spec 1 lands; the **concrete-only guide** points at Spec 5's shipped scaffolding, so its doc rewrite
> is best written **alongside/after Spec 5**.
>
> **Target repo:** this repo (`freedom_ls`) — `docker-compose.yml`, `nginx.conf`, standalone
> `Dockerfile`, and `docs/`.

## Terminology

`ConcreteFlsImplementation` = the specific downstream production repo that raised this handoff
(installs `freedom_ls` as a read-only git submodule). FLS is never deployed standalone — only
concretely, as a submodule of a downstream project. The standalone deployment artifacts and docs are
therefore **legacy to remove**, not a parallel path to maintain.

## 1. Problem

FLS ships standalone-only deployment artifacts and docs for a mode **no one runs**. They describe a
deployment that does not match what actually ships and carry landmines (a DB **bind mount** and a
log **bind mount**). Docs and shipped code disagree; the fix is to delete the artifacts that
describe the mode no one runs and make the concrete (submodule) path the only documented one.

## 3. Requirements

### 3.1 — Remove superseded standalone artifacts

**Remove** (do not repair) the standalone-only artifacts, since there is no standalone deployment for
them to serve and repairing them would create a second, unused deployment path:

- The **entire** standalone production compose stack — the whole `docker-compose.yml` (all three
  services: `db`, `web`, `nginx`) plus `nginx.conf`. The file is standalone-only (it sets
  `DJANGO_SETTINGS_MODULE=config.settings_prod`) and is **not** the local dev stack — dev uses the
  separate `dev_db/docker-compose.yaml`, which stays. Remove the **whole file**, not just the
  `nginx` service: the DB bind-mount (`:10`) and log bind-mount (`:26`) landmines live in the `db`
  and `web` services, and leaving the `web` service would strand a `build:` pointing at the deleted
  `Dockerfile`.
- The broken standalone `Dockerfile` (superseded by the Spec 5 template Dockerfile), together with
  its build-only companions `docker-entrypoint.sh` and `.dockerignore`, which have no consumer once
  the `Dockerfile` is gone.
- The stale `docs/how tos/DOCKER_DEPLOY.md`.

Docs and shipped code must stop disagreeing — by deleting the artifacts that describe a mode no one
runs. Removing the whole `docker-compose.yml` is what makes the DB bind-mount and log bind-mount
landmines disappear.

### 3.2 — Concrete (submodule) deployment is the only documented path

Replace the standalone deployment docs with a single **"Deploying a concrete (submodule-based)
project"** guide that points at the Spec 5 template-repo scaffolding as the canonical, and only,
deployment path. Update `docs/product/deployment.md` so its narrative stops presenting the in-repo
nginx Compose stack as the current working setup and instead describes the concrete (submodule) path
via the template repo (Caddy) as the deployment path, and remove the sections that describe artifacts
this spec deletes or that never shipped: the aspirational CI/CD-pipeline section (`:39–50`) and the
**"Deployment Guides"** section (`:118–122`), whose only entry links the deleted `DOCKER_DEPLOY.md`.

**Not in scope — `docs/deployment-security-checklist.md`.** Its DB-TLS (`:22`) and reverse-proxy
header (`:36`) items are provider-agnostic (they list Nginx/Cloudflare/ALB as examples and apply
equally to Caddy) and stay correct once the standalone path is gone, so this spec leaves them
untouched. The `SECURE_PROXY_SSL_HEADER` wiring itself is Spec 1's concern, not this one's.

## 4. Success criteria

- **Standalone artifacts gone:** the whole `docker-compose.yml`, `nginx.conf`, standalone
