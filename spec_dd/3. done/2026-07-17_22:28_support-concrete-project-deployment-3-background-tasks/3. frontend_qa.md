# Background Tasks (durable backend) — Frontend QA Plan

This change makes **production** default to the durable `django-tasks-db` `DatabaseBackend` (with an
out-of-process `db_worker`) instead of the synchronous `ImmediateBackend`, and adds a
`WebhookDelivery` idempotency guard. Webhook dispatch is the only task, and it is triggered by three
real user-facing flows, so the QA below re-checks the webhooks admin surfaces the original webhooks
QA covered **and** the new async/worker/idempotency behaviour.

There is no student/educator-facing webhook UI — all webhook management is Django admin (Unfold).

## Setup

- **Server / port** (do **not** use the default 8000):
  - `PORT=$(.claude/fls/scripts/find_available_port.sh)`
  - `uv run python manage.py runserver $PORT`
  - Base URL: `http://127.0.0.1:$PORT`
- **Site:** dev is forced to the **DemoDev** site (`FORCE_SITE_NAME = "DemoDev"`), so site selection
  does not depend on the port.
- **Admin login:** superuser `demodev@email.com` / `demodev@email.com` (per the existing webhooks
  QA). If it does not exist, seed with `uv run python manage.py create_demo_data`.
- **Admin URL:** `http://127.0.0.1:$PORT/admin/`
- **Student:** `uv run python manage.py qa_create_course_player_student` — creates
  `demodev_s1@email.com` (password == email), email pre-verified, registered on a course with
  progress, on DemoDev. (This runs without a request, so it does **not** itself fire a webhook — it
  is setup only.)
- **Email inbox (for signup verification):** Mailpit at `http://localhost:8025`.
- **Webhook capture target:** in dev `DEBUG=True`, so plain `http://` and localhost URLs are allowed
  (the SSRF/HTTPS gate is production-only). Use a real capture (webhook.site) to see payloads, or the
  dummy `http://localhost:9999/webhook` (nothing listens — deliveries record as `failed`, which is
  fine for verifying rows are created).

Register one **WebhookEndpoint** for the flows below:

1. Go to `http://127.0.0.1:$PORT/admin/webhooks/webhookendpoint/` → **Add**.
2. Description `QA durable-backend`, URL = your capture target, tick **all three** event types
   (User registered, Course completed, Course registered), leave **Is active** checked. Save.

# Part A — Default dev (`ImmediateBackend`): regression + flows fire synchronously

Confirms the model constraint + dispatch rewrite did not regress the existing webhooks admin, and
that each user flow still produces an event + a single delivery. In this config delivery happens
**in-request** (synchronously).

## Test A1: Webhook admin surfaces still work (regression)

1. On the **Add endpoint** form, verify event types render as **checkboxes** (no free-text) and the
   **Secret** field is NOT shown; after saving, Secret is populated and read-only, Failure count 0.
2. `http://127.0.0.1:$PORT/admin/webhooks/webhookevent/` — read-only, **no Add** button, payload JSON
   visible.
3. `http://127.0.0.1:$PORT/admin/webhooks/webhookdelivery/` — columns include **Status** and
   **Attempt count**; filters for Status / Endpoint / Event type.
4. Select the endpoint → action **"Send test ping"** → Go. Verify a `webhook.test` event appears and
   a delivery row is created (status `failed` if the target isn't listening).
5. Select a failed delivery → action **"Retry"** → Go. Verify the row updates (its retry
   fields/last-attempt change) and no *duplicate* delivery row is created.
6. Endpoint list → **Disable endpoints** then **Enable endpoints** bulk actions toggle Is active.

**Expected:** All existing admin behaviour is intact; the new unique constraint does not block
send-test (fresh event each time) or retry (reuses the row).

## Test A2: Signup → `user.registered`

1. Log out. Go to `http://127.0.0.1:$PORT/accounts/signup/`.
2. Fill email, password + confirm, first name; submit. Open Mailpit (`http://localhost:8025`) and
   click the confirmation link (verification is mandatory).
3. Log back into admin → `.../webhookevent/` shows a `user.registered` event; `.../webhookdelivery/`
   shows exactly **one** delivery for that event against the endpoint.

**Adversarial:** sign up **again with the same email** — allauth must not create a second usable
account, and you should not get a duplicate `user.registered` for the same registration action.

## Test A3: Course registration → `course.registered`

1. As student `demodev_s1@email.com`, open a **free / self-serve** course detail page:
   `http://127.0.0.1:$PORT/courses/<slug>/detail/`, and click the enrol CTA (routes to
   `/courses/<slug>/access/`).
2. In admin, verify one `course.registered` event + one delivery.

**Adversarial / side-effects:**
- Click enrol **again** on the same course → no second `course.registered` (registration is
  `get_or_create`; the event fires only on create).
- Try an **application-gated** course → you are redirected to the apply page and **no**
  `course.registered` fires (default `ApplicationCourseAccessBackend`). Confirm no event is created.

## Test A4: Course completion → `course.completed`

1. As the student, open the course player and advance to the **last** item; on the final topic click
   **"Finish Course"** (submits `mark_complete`). You land on `/courses/<slug>/finish/`.
2. In admin, verify one `course.completed` event + one delivery.

**Adversarial:** revisit and "finish" the **already-completed** course → it is a no-op and fires
**no** new `course.completed` event.

# Part B — Durable `DatabaseBackend` + `db_worker` (the change under test)

Dev defaults to `ImmediateBackend` by design, so temporarily switch it to the durable backend to
exercise the new behaviour. **Revert afterwards.**

**Enable (temporary local override):**
1. In `config/settings_dev.py`, temporarily append:
   `from freedom_ls.deployment import settings_defaults as fls_defaults` and
   `TASKS = fls_defaults.DATABASE_TASKS` (confirm `django_tasks_db` is in `INSTALLED_APPS` — it is,
   from base).
2. `uv run python manage.py migrate` (creates the `django_tasks_db` task tables).
3. Restart the server; in a **second shell** run the worker: `uv run python manage.py db_worker`.

## Test B1: New Tasks admin surface exists

1. Go to `http://127.0.0.1:$PORT/admin/` and confirm a new **Tasks** (`django_tasks_db`) section is
   listed.
2. Open it — it lists task rows with state (scheduled / running / completed / errored).

**Expected:** the durable backend registers a browser-visible Tasks admin that did not exist under
`ImmediateBackend`.

## Test B2: Delivery is now asynchronous (worker-driven)

1. With `db_worker` running, drive **course completion** (as in A4) for a fresh registration.
2. Immediately after the page returns, in admin the new `WebhookDelivery` row is **`pending`** and a
   task row appears in the Tasks admin.
3. Within a moment the worker drains it: the Tasks row moves to completed and the delivery
   transitions to `success` (or `failed` if the target isn't reachable).

**Expected:** unlike Part A, the user's request returns before delivery runs; the worker completes it
out-of-process.

## Test B3: Single delivery per (event, endpoint) — idempotency

1. Trigger a flow, let the worker process it, and confirm **exactly one** `WebhookDelivery` row for
   that `(event, endpoint)` pair — never duplicates.

(The redelivery/at-least-once duplicate-prevention path itself is verified by the automated test
`test_dispatch_event_is_idempotent`; the browser check confirms the normal single-delivery outcome.)

## Test B4: Regression under the durable backend

1. Repeat the **send-test ping** and **retry** admin actions (Test A1 steps 4–5) with the worker
   running. Send-test still creates a fresh event + delivery; retry still updates the existing row —
   neither trips the unique constraint.

## Test B5 (failure mode): worker not running → tasks queue, don't drop

1. **Stop** `db_worker`. Drive a flow that fires a webhook.
2. In admin, the `WebhookDelivery` stays **`pending`** and the task sits **scheduled/unprocessed** in
   the Tasks admin — nothing is delivered.
3. **Start** `db_worker` again → the queued task drains and the delivery transitions.

**Expected:** this demonstrates the hard operational dependency introduced by the change — with the
durable backend the worker **must** run or user-triggered webhooks never leave the queue. (Under
`ImmediateBackend` there was no such queue.)

**Teardown:** revert the `config/settings_dev.py` override so dev returns to `ImmediateBackend`.
