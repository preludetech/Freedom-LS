# Plan: Layout Spacing Cleanup

This plan turns `1. spec.md` (D1–D6) into a minimal, ordered set of code
changes. The work is layout / CSS only — no models, views, URLs, permissions,
or data changes. Existing functionality must keep working.

## Approach in one paragraph

Introduce a single `c-page` cotton component that owns horizontal padding,
max-width, and top/bottom padding for the content well. Move the current
`max-w-7xl mx-auto px-* py-*` work out of `_base.html`'s `<main>` and out of
its inner content `<div>`, so `_base.html` only owns page chrome. Update
`_base_interface.html` to own its own outer horizontal padding (because the
sidebar/header-bar shell sits at the page edge there) and switch the inner
sidebar+content container from `flex` (no gap) to `grid` with `lg:gap-12`.
Apply `c-page` (or its `width="narrow"` variant) inside every learner-facing
content block. Remove now-redundant per-template `mt-*`/`mb-*`/`px-*`/`py-*`
that duplicates the wrapper's job, and consolidate inter-section spacing onto
a single `space-y-*` per parent. Add a small `@layer base` rule that gives
heading elements with an `id` a comfortable `scroll-margin-top` so anchor
links don't land flush.

## Pre-flight: agree the spacing values

Before touching templates, agree the final padding values against
`brand-guidelines`. The spec gives indicative starting values; the final
numbers are tunable but must satisfy non-functional requirements 2 and 3
(at most two breakpoints from `sm:`/`lg:`, default Tailwind scale only).

Starting values (from spec D2):

| Axis | Mobile | `sm:` | `lg:` |
|---|---|---|---|
| `c-page` horizontal padding | `px-4` | `sm:px-6` | `lg:px-8` |
| `c-page` vertical padding | `py-8` | — | `lg:py-12` |
| Sidebar/main grid gap | (stacked `gap-8`) | — | `lg:gap-12` |
| Heading `scroll-margin-top` | base CSS rule (`scroll-mt-24` equivalent) | | |

Confirm with the `brand-guidelines` skill before implementing. If the brand
guidelines contradict these numbers, update the table and proceed — but stay
on Tailwind's default scale.

## Task 1 — Create the `c-page` cotton component

- New file: `freedom_ls/base/templates/cotton/page.html`
- Implement a `width` attribute (default = wide, learner standard;
  `narrow` for the form-complete results / course-finish pages — see Task 8).
- The wrapper is a single `<div>` that owns `mx-auto`, the chosen `max-w-*`,
  the horizontal `px-*`, and the vertical `py-*` ramps.
- It does **not** own inter-section spacing. Pages put a `space-y-*` on the
  parent inside the wrapper if they have multiple sibling sections.

Pseudocode:

```django
{# freedom_ls/base/templates/cotton/page.html #}
{# Cotton variants: width="wide" (default) | width="narrow" #}
<c-vars width="wide" />

{% if width == "narrow" %}
    <div class="mx-auto w-full max-w-4xl px-4 py-8 sm:px-6 lg:px-8 lg:py-12 {{ class }}">
        {{ slot }}
    </div>
{% else %}
    <div class="mx-auto w-full max-w-7xl px-4 py-8 sm:px-6 lg:px-8 lg:py-12 {{ class }}">
        {{ slot }}
    </div>
{% endif %}
```

Notes:

- `max-w-7xl` matches the current `_base.html` width.
- `max-w-4xl` matches the existing readability cap on
  `course_form_complete.html` and `course_finish.html` (D4).
- `class` passthrough lets a page add an extra margin on the wrapper if it
  ever needs to (we expect it not to need to).
- Keep the variant list small. If a page ever needs a different width, add a
  variant — do not reintroduce hand-rolled `<div class="max-w-*">` in pages.

## Task 2 — Strip outer padding from `_base.html`

Goal: `_base.html` keeps the page chrome (header bar, messages, debug badge,
`<main>` shell) but stops owning content-well width / padding. The wrapper
takes that over.

Edit `freedom_ls/base/templates/_base.html`:

- `<main class="max-w-7xl mx-auto px-2 sm:px-6 lg:px-8">` →
  `<main>` (no classes, or `<main class="">` — keep it minimal).
- Remove the inner default-body `<div class="py-6 sm:py-12 space-y-6 sm:space-y-12">`
  that wraps `{% block content %}`. The default `{% block body %}` should
  render `{% block pre-content %}` and `{% block content %}` directly inside
  `<main>`.

Pseudocode for the new `<main>`:

```django
<main>
    {% block body %}
        {% block pre-content %}{% endblock pre-content %}
        {% block content %}
            Replace this in child templates
        {% endblock content %}
    {% endblock body %}

    {% block extra_body %}{% endblock extra_body %}
</main>
```

Implication: every page that extends `_base.html` and uses `{% block content %}`
must now provide its own `<c-page>` wrapper around its content (Task 8).
Pages that override `{% block body %}` (`_base_interface.html`,
`course_home.html`, `allauth/layouts/entrance.html`, etc.) opt out of this
default and own their own outer layout — covered separately in Tasks 3–8.

## Task 3 — Update `_base_interface.html` to own outer horizontal padding and switch sidebar+content to grid

Edit `freedom_ls/base/templates/_base_interface.html`:

1. **Outer horizontal padding.** Because `_base.html`'s `<main>` no longer
   owns horizontal padding, the sidebar shell's outermost element must own
   it (so the header bar and overlay backdrop don't push to screen edge).
   Add `px-4 sm:px-6 lg:px-8` to the top-level `<div x-data="sidebarComponent">`
   wrapper. This is the **single exception** to the "wrapper owns horizontal
   padding" rule, called out in spec success criterion 4.

2. **Header bar.** No change to its internal flex layout, but it now sits
   inside the padded outer div, so `pt-4 pb-4` stays.

3. **Sidebar + content container (was flex).** Change:

   ```
   <div class="flex relative {% block sidebar_class %}{% endblock %}">
   ```

   to a responsive grid:

   ```
   <div class="grid grid-cols-1 gap-8
               lg:grid-cols-[16rem_minmax(0,1fr)] lg:gap-12
               relative {% block sidebar_class %}{% endblock %}">
   ```

   Notes:
   - `16rem` matches the existing `w-64` track that the alpine
     `sidebarColumnClass()` already uses. If the track ever needs to change,
     it changes here and in the alpine helper together.
   - `minmax(0, 1fr)` on the content column prevents long unbreakable words
     / wide code blocks from blowing out the grid (research §3).
   - `lg:gap-12` is the single source of truth for sidebar/content spacing
     at desktop. No `ml-*` on the main column. No `mr-*` on the sidebar.
   - `gap-8` at the stacked breakpoint provides vertical breathing room if
     the sidebar ever stacks rather than overlays. Current behaviour on
     mobile is overlay-only, so this `gap-8` doesn't fire visually — it's
     a guard for future-proofing per spec functional req 4.
   - `min-w-0` stays on the main content column.

4. **Sidebar column markup.** The Alpine `sidebarColumnClass()` returns
   `"w-64 shrink-0"` when open on desktop and `""` otherwise. With grid we
   no longer need `shrink-0` (grid columns don't shrink), and `w-64` is
   redundant because the column track is `16rem` already. Simplest option:
   leave the alpine helper alone — `w-64 shrink-0` on a grid item is
   harmless (the track wins). Document this in a code comment so future
   maintainers don't get confused. (If we want to clean it up, remove the
   `w-64 shrink-0` from `sidebarColumnClass()` — but that's a stretch goal,
   not required by the spec.)

5. **Alpine functionality must keep working.** No changes to:
   `toggle()`, `closeSidebar()`, `handleSidebarClick()`, the `localStorage`
   storage key, the `matchMedia` mobile detection, the backdrop, or the
   transition classes. Verified by spec functional req 7.

## Task 4 — Apply `c-page` inside `_base_interface.html`-derived pages

Pages that extend `_base_interface.html` (via
`student_interface/_course_base.html` or `educator_interface/interface.html`)
override `{% block content %}`, which is rendered in the right-hand grid
column. The `c-page` wrapper goes **inside that column's content**, not
around the sidebar shell.

Concretely: in every `student_interface` page that extends
`_course_base.html` (course_topic, course_form, course_form_page,
course_form_complete, course_finish), wrap the existing `{% block content %}`
contents with `<c-page>…</c-page>`. The wrapper is what now owns the
top/bottom breathing room and any width cap; the surrounding shell already
owns horizontal padding (Task 3).

Visually: the right-hand column ends up with `c-page`'s `py-8 lg:py-12` and
its `max-w-*` centring. There is no double horizontal padding because
`c-page`'s `px-*` runs inside the shell's `px-*` only via the wrapper's
`mx-auto max-w-*` reflow — but to be safe, on this template family the
inner `<c-page>` should not add additional horizontal padding. Either:

- Option A (preferred, simpler): keep `c-page` as one component with its
  default `px-*`. The double padding is small (e.g. 16px + 16px = 32px on
  mobile) and only happens inside the right-hand column, which is already
  visually narrower than full-width — extra inset is acceptable. Verify in
  QA, tune if needed.
- Option B (if A looks wrong): pass a `class="px-0"` override on the
  inner `c-page`, or expose a `padding="none"` variant. Don't choose this
  speculatively — only if A actually looks bad.

Pick Option A for the first pass. Re-check during QA at all three widths.

## Task 5 — Add anchor-link `scroll-margin-top` via base CSS

Per spec D1: do **not** add `scroll-padding-top` on `<html>`. Instead, add a
`@layer base` rule to `tailwind.components.css` that gives heading elements
with an `id` a comfortable scroll margin. This means anchor links land below
the top of the viewport with breathing room — and it works automatically for
markdown-rendered headings in `markdown-container`.

Edit `tailwind.components.css`, inside the existing `@layer base` block, add
(near the existing heading rules):

```css
h1[id], h2[id], h3[id], h4[id], h5[id], h6[id] {
    scroll-margin-top: calc(var(--spacing) * 24);
}
```

Notes:

- `calc(var(--spacing) * 24)` matches Tailwind v4's `scroll-mt-24` (24 *
  0.25rem = 6rem = 96px). If Tailwind v4 spacing variable name differs in
  this project, use the literal `6rem`. The exact value is tunable; the
  goal is "a comfortable buffer below the page header in case it ever
  becomes sticky, and a clear gap from the viewport top either way."
- The rule is keyed on `[id]` so it only affects anchor targets, not every
  heading.
- After editing the CSS source, run `npm run tailwind_build` so the
  compiled `static/vendor/tailwind.output.css` includes the rule.

Verify the heading list of files / styles: all current `h*` rules in
`tailwind.components.css` already exist in the same `@layer base` block —
add the new rule next to them, don't fork into a new block.
