# Spec: First Class theme — Tier 2 component-class overrides (Phase 3 of 5)

## Why

This is **Phase 3** of the multi-phase Themable Freedom LS effort. It validates the Tier-2 architecture (semantic component classes) by demonstrating that a theme can selectively re-open component classes from `tailwind.components.css` **without owning the full set**. Defaults stay in the main project; the theme just appends overrides.

After Phase 3, the `first_class` canary theme has rounder buttons, flatter chips, and a thicker outline-button border — all driven by an `@layer components` block in `first_class/theme.css`. The default theme's `theme.css` stays tokens-only, proving the sparse principle.

This phase is the canary that proves Tier 2 of the master spec: **a downstream project can tweak component shapes by re-opening `.btn-primary` etc. in their own `@layer components` block — no template edits, no FLS fork.**

For the holistic five-phase context, see `../themable-implementations-master-decomposed-into-phases/`.

## Scope of this phase

**In:**

- An `@layer components` block appended to `freedom_ls/themes/first_class/static/themes/first_class/theme.css` overriding 2–3 component classes from `tailwind.components.css`.
- Confirmation that the cascade order in `tailwind.input.css` (set up in Phase 2) actually causes the theme's overrides to win.
- An optional inline comment in `tailwind.input.css` explaining the load-bearing import order.

**Out of this phase (covered later):**

- Tier-3 cotton-template overrides for `first_class` — Phase 4.
- Documentation that explains Tier-2 to downstream theme authors — Phase 5.

**Phases 1 + 2 are hard prerequisites.** This phase assumes the resolver, the default theme, the `first_class` Tier-1 token theme, and the `tailwind.input.css` active-theme import are all in place.

## Goals & success criteria

1. **`tailwind.components.css` (in main project) is unchanged structurally** — it remains the source of truth for component classes.
2. **`freedom_ls/themes/first_class/static/themes/first_class/theme.css` adds an `@layer components` block** overriding exactly the demo classes:
   - `.btn-primary` rounder + slightly stronger shadow (e.g. `@apply rounded-2xl shadow-md;`).
   - `.chip` flatter (no shadow) to match the brand's data-forward look.
   - `.btn-outline` bolder border (e.g. `@apply border-2;`).
3. **The default theme's `theme.css` stays tokens-only** (sparse principle) — a grep for `@layer components` in `freedom_ls/themes/default/` returns nothing.
4. **Under `FLS_THEME=first_class`, those overridden components render with the tweaks** on the dev server.
5. **Under `FLS_THEME=default`, components render unchanged** — the tweaks are scoped to the active theme.
6. **Phase 1 + 2 deliverables remain green.**

## Architecture used by this phase

### Tier 2 — semantic component classes (recap)

FLS keeps `btn`, `btn-primary`, `chip`, `chip-success`, `surface`, etc. as the public visual contract. **Class names are a documented public API** with deprecation rules — same caution as a Python signature.

The default definitions of these classes live in **`tailwind.components.css` in the FLS repo root**. They are not moved into the default theme. The default theme's `theme.css` stays tokens-only. This keeps themes sparse — a downstream theme that just wants new colours doesn't have to think about component classes at all.

A theme extends a Tier-2 class by re-opening it inside its own `theme.css`:

```css
@layer components {
  .btn-primary { @apply rounded-2xl shadow-md; }
}
```

Both rule sets survive; later declarations win per the cascade (see master `research_tailwind_v4_theming.md` §5). Because the active theme's `theme.css` is `@import`-ed *after* `tailwind.components.css`, theme Tier-2 changes automatically win without needing `!important`.

### Required cascade order in `tailwind.input.css`

Set up in Phase 2 but load-bearing for this phase to work:

1. Default theme `theme.css` is imported first (tokens baseline).
2. `tailwind.components.css` imported next (defines `.btn-primary` etc., reading tokens).
3. Active theme `theme.css` imported last (can re-declare both tokens and component classes).

If this order is wrong from Phase 2, fix it in this phase before adding the overrides.

### First Class Tier-2 demo overrides

Pick exactly the classes the master spec calls out so the canary stays focused:

- `.btn-primary` rounder + slightly stronger shadow.
- `.chip` flatter (no shadow) to match the brand's data-forward look.
- `.btn-outline` bolder border.

These are demos, not polished design. The bar is "obviously different from default; still functional".

## Out of scope (deferred)

- Theming-guide content that explains Tier-2 to downstream authors — Phase 5.
- Adding First Class IBM Plex Mono / type scale / Light tints — noted in `first-class-theme.md`, not required for the Tier-2 canary.
