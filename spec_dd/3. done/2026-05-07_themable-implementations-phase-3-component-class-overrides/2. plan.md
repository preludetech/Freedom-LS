# Implementation Plan: First Class — Tier 2 component-class overrides (Phase 3 of 5)

This plan turns `1. spec.md` (in this folder) into shippable code. The phase appends a small `@layer components` block to the existing `first_class/theme.css` and confirms the cascade order in `tailwind.input.css` causes the overrides to win.

For the holistic five-phase context, see `../themable-implementations-master-decomposed-into-phases/2. plan.md`.

**Phases 1 + 2 are hard prerequisites.**

---

## Architectural principles inherited

1. **Themes are sparse.** The default theme's `theme.css` stays tokens-only. The first_class theme adds component classes only because it wants to override a few — it does not own the full set.
2. **Defaults stay in the main project.** `tailwind.components.css` remains the source of truth for component class definitions.
3. **This phase ends in a green build.**

---

## Pre-existing functionality to reuse (DRY)

- `tailwind.components.css` — defines `.btn-primary`, `.chip`, `.btn-outline`, etc. Phase 3 overrides these without modifying the source-of-truth file.
- The Phase-2 `tailwind.input.css` active-theme import — already imports `first_class/theme.css` after `tailwind.components.css`, which is exactly the cascade order Tier-2 overrides need.

---

## End-state checklist

- `tailwind.components.css` (in main project) is unchanged structurally — it remains the source of truth for component classes.
- `freedom_ls/themes/first_class/static/themes/first_class/theme.css` adds an `@layer components` block overriding 2–3 classes.
- The default theme's `theme.css` stays tokens-only (sparse principle).
- Visual: under `FLS_THEME=first_class`, those overridden components render with the tweaks.
- Phase 1 + 2 deliverables remain green.

---

## Task 1 — Confirm cascade order

Read the current `tailwind.input.css` and confirm:

1. Default theme `theme.css` is imported first (tokens baseline).
2. `tailwind.components.css` imported next (defines `.btn-primary` etc., reading tokens).
3. Active theme `theme.css` imported last (can re-declare both tokens and component classes).

If the order is wrong from Phase 2, fix it now. Add a short comment in `tailwind.input.css` explaining the order is load-bearing.

---

## Task 2 — Add `@layer components` block to `first_class/theme.css`

Append to `freedom_ls/themes/first_class/static/themes/first_class/theme.css`:

```css
@layer components {
    .btn-primary {
        @apply rounded-2xl shadow-md;        /* rounder + heavier shadow */
    }

    .chip {
        @apply shadow-none;                  /* flatter to match brand */
    }

    .btn-outline {
        @apply border-2;                     /* bolder border */
    }
}
```

Pick exactly the classes the spec calls out so the canary stays focused.

---

## Task 3 — Phase verification

- `FLS_THEME=first_class npm run tailwind_build` — clean.
- Visual check on a page that renders all three classes (e.g. cohort detail or content topic): buttons rounder + shadowed, chips flatter, outlined buttons have a thicker border.
- `FLS_THEME=default npm run tailwind_build` — defaults unchanged.
- Default theme's `theme.css` remains tokens-only — grep to confirm no `@layer components` block has crept in.
