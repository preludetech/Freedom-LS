# Spec: Educator Interface Better Navigation

## Problem

When navigating the educator interface, users lose track of where they are. There is no breadcrumb trail, no visual indication of the active sidebar item, and no way to see which specific instance (e.g. which cohort) is currently selected in the sidebar. These three gaps combine to make orientation difficult, especially when switching between sections or drilling into instance detail pages.

## Overview

Three complementary navigation improvements, all built into the panel framework so any interface using `panel_framework_view` benefits automatically:

1. **Breadcrumbs** -- hierarchy-based breadcrumbs at the top of the main content area
2. **Sidebar active highlighting** -- visual treatment of the active sidebar link
3. **Sidebar instance dropdown** -- expand the active section to show the selected instance

Additionally, **all navigation is converted to HTMX partial swaps** so the page never fully reloads when navigating between sections or instances. Breadcrumbs and sidebar state update via `hx-swap-oob`.

---

## Feature 1: HTMX-Powered Navigation

### Why this matters

Currently, clicking a sidebar link or a data table row link triggers a full page load. Converting to HTMX partial swaps enables the SPA-like experience needed for OOB updates to breadcrumbs and sidebar. It also makes navigation feel instant.

### Behaviour

- Sidebar navigation links use `hx-get` targeting the main content area, with `hx-push-url="true"` so the browser URL stays in sync
- Data table row links (e.g. clicking a cohort name) also use HTMX to swap main content + push URL
- The server response for HTMX navigation requests includes:
  - The main content HTML (swapped into the content target)
  - An OOB breadcrumb fragment (`hx-swap-oob="true"`)
  - An OOB sidebar fragment (`hx-swap-oob="true"`)
- Full page loads (direct URL visit, browser refresh) continue to work -- the server renders the complete page with correct breadcrumb and sidebar state
- Browser back/forward buttons work correctly via HTMX history (DOM snapshot restore)

### Key decisions

- **Why HTMX navigation instead of full page loads?** Full page loads would make breadcrumbs and sidebar state trivial (just render them server-side). But the idea explicitly calls for OOB swaps, and the SPA-like feel is a better experience for educators who frequently switch between cohorts, users, and courses. The cost is additional complexity in the response format.
- **Where do OOB fragments come from?** The `panel_framework_view` function assembles the response. For HTMX requests, it renders the main content plus OOB fragments for breadcrumb and sidebar. This keeps the logic centralised rather than scattered across individual views.

### Success criteria

- Clicking any sidebar link or data table instance link swaps content without a full page reload
- Browser URL updates to reflect the current page
- Browser back/forward navigates correctly
- Direct URL visits and page refreshes render the full page correctly
- No flash of unstyled/stale content during navigation

## Feature 2: Breadcrumbs

### Why this matters

Breadcrumbs give the user a persistent, glanceable indication of where they are in the site hierarchy. They also provide quick navigation back to parent pages without relying on the sidebar.

### Behaviour

- Breadcrumbs appear at the top of the main content area, below the header bar and above the page heading
- Hierarchy-based (not path/history-based) -- reflects site structure, not click path
- All items except the current page are clickable links (using HTMX, consistent with Feature 1)
- The current page item is not a link
- Breadcrumbs appear consistently on every page within the interface
- Maximum 2-4 levels

### Breadcrumb levels

| Page | Breadcrumb trail |
|------|-----------------|
| Educator root (no section selected) | `Educator` (current) |
| Cohorts list | `Educator` > `Cohorts` (current) |
| Specific cohort | `Educator` > `Cohorts` > `Data Science 2026` (current) |
| Users list | `Educator` > `Users` (current) |
| Specific user | `Educator` > `Users` > `Jane Doe` (current) |
| Courses list | `Educator` > `Courses` (current) |
| Specific course | `Educator` > `Courses` > `Introduction to Python` (current) |

The root label ("Educator") comes from the interface's `header_bar_text` block or an equivalent config value, so other interfaces using the panel framework get their own root label.

### Accessibility

- `<nav aria-label="Breadcrumb">` wrapping an `<ol>`
- `aria-current="page"` on the last item
- Separators rendered via CSS `::before` pseudo-elements (not text content) to avoid screen reader noise
- Visible focus indicators on breadcrumb links

### HTMX integration

- Breadcrumb container has a stable `id` (e.g. `id="breadcrumbs"`)
- On HTMX navigation, the server includes an OOB breadcrumb fragment: `<nav id="breadcrumbs" hx-swap-oob="true" ...>`
- Breadcrumb links themselves use HTMX (`hx-get`, `hx-target`, `hx-push-url`) for consistent SPA navigation

### Implementation approach

- Breadcrumb data is generated server-side in `panel_framework_view` based on the resolved path
- A new breadcrumb block in `_base_interface.html` renders the breadcrumb `<nav>`
- A cotton component (`<c-breadcrumbs>`) or template partial renders the breadcrumb list from a context variable

### Success criteria

- Breadcrumbs visible on every page in the educator interface
- Breadcrumb hierarchy matches the table above
- All breadcrumb items except the current page are clickable and navigate via HTMX
- Screen readers can navigate breadcrumbs as a landmark and understand the hierarchy
- Breadcrumbs update correctly when navigating via HTMX (no stale state)

## Feature 3: Sidebar Active Highlighting

### Why this matters

Without highlighting, the user has to read each sidebar item and mentally match it to the page content to know which section they're in. A visual active state makes this instant.

### Behaviour

- The active sidebar link gets a distinct visual treatment combining 2-3 of:
  - Background fill (muted/subtle)
  - Left accent bar (2-4px, primary colour)
  - Semi-bold text weight
- Hover state on non-active items is subtle (e.g. slight background shift) and visually distinct from the active state
- Only one item is active at a time
- Active state is determined server-side (not client-side)

### Accessibility

- `aria-current="page"` attribute on the active link -- this is the source of truth
- Visual styling uses the CSS attribute selector `[aria-current="page"]` so accessibility and visual state cannot go out of sync
- Hover and active states meet WCAG 2.1 AA contrast requirements (4.5:1 for text, 3:1 for UI components)

### HTMX integration

- The sidebar `<nav>` has a stable `id` (e.g. `id="sidebar-nav"`)
- On HTMX navigation, the server includes an OOB sidebar fragment that re-renders the full sidebar nav with the correct `aria-current="page"` on the new active item

### Implementation approach

- `_build_menu_items` in the panel framework gains awareness of the current path to mark the active item
- The sidebar template renders `aria-current="page"` on the matching item
- CSS handles the visual treatment via attribute selector

### Success criteria

- Active sidebar item is visually distinct from inactive items at a glance
- Hover state is visually distinct from (and subtler than) active state
- Active state updates correctly when navigating via HTMX
- `aria-current="page"` is present on the active link and only the active link

## Feature 4: Sidebar Instance Dropdown

### Why this matters

When viewing a specific instance (e.g. a particular cohort), the sidebar currently gives no indication of which instance is selected -- it only highlights the section. The dropdown shows the selected instance name directly in the sidebar, reinforcing orientation.

### Behaviour

- When the user navigates to a specific instance (e.g. `/educator/cohorts/42/`), the parent section in the sidebar expands to show the instance name below it
- The expanded section shows only the currently selected instance (not a full list of all instances)
- The instance name is styled as the active item (with `aria-current="page"`), and the parent section label retains its highlight as well (but subtler, indicating "you're in this section")
- When viewing a list page (e.g. `/educator/cohorts/`), sections are not expanded -- the section link itself has `aria-current="page"`
- Only one section can be expanded at a time
- Expand/collapse is animated using Alpine.js `x-collapse`

### Example

When viewing the "Data Science 2026" cohort:
```
  Cohorts            v
    Data Science 2026    <-- active (aria-current="page")
  Users
  Courses
```

When viewing the Cohorts list:
```
  Cohorts                <-- active (aria-current="page")
  Users
  Courses
```

### Accessibility

- Uses the W3C disclosure pattern: `<button aria-expanded="true|false">` controlling a nested `<ul>`
- Keyboard support: Tab to navigate between items, Enter/Space to toggle expand/collapse, Escape to close
- The toggle button and nested list are properly associated

### HTMX integration

- Server determines which section is expanded and which item is active in the OOB sidebar fragment
- Alpine.js handles the expand/collapse animation client-side
- Server renders `aria-expanded="true"` on the toggle button and the nested `<ul>` visible when the section should be open; Alpine.js picks up this initial state

### Implementation approach

- `_build_menu_items` returns richer data: each menu item includes whether it's active, whether it has an expanded child, and the child's label/URL if applicable
- The sidebar template uses the disclosure pattern with Alpine.js for animation
- Alpine component registered via `Alpine.data()` (CSP-safe), consistent with existing components (`sidebarComponent`, `tabContainer`)

### Success criteria

- Sidebar expands to show the selected instance when viewing an instance page
- Sidebar collapses (no expanded section) when viewing a list page
- Expand/collapse is animated smoothly
- Keyboard users can toggle sections and navigate to items
- Only the currently selected instance appears in the expanded section (not a full list)
- State updates correctly when navigating via HTMX

---

## Scope boundaries

- These features apply to the panel framework and any interface using `panel_framework_view` (currently the educator interface)
- Tab navigation within an instance page (e.g. "Course Progress" / "Details" tabs) is unchanged -- tabs already work via HTMX
- Data table internal interactions (search, sort, pagination) are unchanged -- they already use HTMX within their own scope
- The student interface is not in scope
