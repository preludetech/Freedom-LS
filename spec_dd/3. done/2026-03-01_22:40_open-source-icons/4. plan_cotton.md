# Plan: Convert icon system to `<c-icon />` cotton component

## Context

The Font Awesome → Heroicons migration is complete. All templates currently use `{% icon "name" %}` with `{% load icon_tags %}`. This plan replaces that with a single `<c-icon />` cotton component as the **only** icon API, using the `heroicons` package's built-in template tags directly.

### Key design decision

The `heroicons` package already provides template tags (`{% heroicon_outline %}`, `{% heroicon_solid %}`, etc.) that handle all SVG rendering. There's no need for custom SVG manipulation code. The cotton component uses these tags directly.

The only Python needed is a tiny template filter to resolve semantic names (`"next"` → `"arrow-right"`) from the `ICONS` registry.

**Accessibility simplification:** Instead of custom XML manipulation to inject `<title>` elements, we use `aria-label` attribute on the SVG (set via the heroicons tag's kwargs). This is well-supported and eliminates all custom `ElementTree` code.

## Phase 1: Simplify the icon system

### Task 1.1: Slim down `icons.py` to just the registry

**File**: `freedom_ls/base/icons.py`

Keep only the `ICONS` dict. Delete any rendering logic if present (currently it's only the dict, so no change needed).

### Task 1.2: Replace `icon_tags.py` with a single filter

**File**: `freedom_ls/base/templatetags/icon_tags.py`

Delete all the existing code (the `{% icon %}` simple tag with `ElementTree`, `deepcopy`, `_load_icon`, UUID generation, etc.). Replace with a single template filter:

```python
from django import template

from freedom_ls.base.icons import ICONS

register = template.Library()


@register.filter
def icon_name(name: str, force: str = "") -> str:
    """Resolve a semantic icon name to a heroicon name.

    Used internally by the <c-icon /> cotton component.
    If force is truthy, bypasses the registry and returns name as-is
    (for one-off heroicon names not in the registry).
    """
    if str(force).lower() in ("true", "1", "yes"):
        return name
    return ICONS[name]  # KeyError if not in registry
```

That's the entire file. All SVG rendering is handled by the `heroicons` package.

### Task 1.3: Create `cotton/icon.html`

**File**: `freedom_ls/base/templates/cotton/icon.html`

```html
<c-vars name="" variant="outline" class="size-5" force="" aria_label="" />

{% load icon_tags heroicons %}

{% with resolved=name|icon_name:force label=aria_label|default:name %}
    {% if variant == "solid" %}
        {% heroicon_solid resolved size=None class=class aria_label=label role="img" %}
    {% elif variant == "mini" %}
        {% heroicon_mini resolved size=None class=class aria_label=label role="img" %}
    {% elif variant == "micro" %}
        {% heroicon_micro resolved size=None class=class aria_label=label role="img" %}
    {% else %}
        {% heroicon_outline resolved size=None class=class aria_label=label role="img" %}
    {% endif %}
{% endwith %}

{% comment %}
Usage:

  <c-icon name="next" />
  <c-icon name="success" class="size-6 text-success" />
  <c-icon name="close" variant="solid" />
  <c-icon name="success" aria_label="Completed" />
  <c-icon name="arrow-right" force="true" />
{% endcomment %}
```

- `size=None` removes the heroicons package's default `width`/`height` attributes so Tailwind classes (`size-5`, etc.) control sizing.
- `aria_label` defaults to the semantic icon name via `|default:name`, so every icon is accessible by default. Pass a custom `aria_label` to override (e.g., `aria_label="Completed"` instead of `"success"`).

### Task 1.4: Update tests

**File**: `freedom_ls/base/tests/test_icon_tag.py`

Rewrite tests to match the new architecture:

1. **`icon_name` filter**: Test that `"next"` resolves to `"arrow-right"`, unknown name raises `KeyError`
2. **`icon_name` filter with force**: Test that `"arrow-right"` with force=`"true"` returns `"arrow-right"` unchanged
3. **Registry validation**: Test that every value in `ICONS` maps to a valid heroicon (use `heroicons._load_icon` to verify)
4. **Cotton component rendering**: Test that `<c-icon name="next" />` renders an `<svg>` element with `class="size-5"` and `role="img"`
5. **Default aria_label**: Test that `<c-icon name="next" />` renders `aria-label="next"` (defaults to the semantic name)
6. **Custom aria_label**: Test that `<c-icon name="success" aria_label="Done" />` renders `aria-label="Done"`
7. **Variant**: Test `<c-icon name="next" variant="solid" />` renders the solid variant
8. **Custom class**: Test `<c-icon name="next" class="size-6 text-red-500" />` applies the class
9. **Force bypass**: Test that `<c-icon name="arrow-right" force="true" />` renders the heroicon directly without registry lookup
10. **Force equivalence**: Test that `<c-icon name="next" />` and `<c-icon name="arrow-right" force="true" />` render the same SVG paths

Delete old tests that tested the removed `{% icon %}` tag directly.
