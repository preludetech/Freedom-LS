# Spec: Replace Font Awesome with Heroicons

## Overview

Replace Font Awesome (loaded via CDN) with Heroicons (by Tailwind Labs) using Adam Johnson's `heroicons` Python package, with a `<c-icon>` cotton component providing a clean, consistent API backed by a central icon registry.

### Why this matters

- **Privacy & reliability**: Removes third-party CDN dependency (cdnjs.cloudflare.com)
- **Performance**: Eliminates ~100KB CSS download for only 28 icons used
- **Consistency**: Unifies three different icon systems (Font Awesome `<i>` tags, inline SVGs, Unicode characters) into one
- **Maintainability**: Central registry means changing an icon's meaning requires editing one line, not hunting through templates

## Architecture

### Icon package: `heroicons` by Adam Johnson

- **PyPI**: `heroicons[django]`
- **Why**: Made by Tailwind Labs (visual harmony with TailwindCSS stack), MIT license, excellent Django support via Adam Johnson's package (supports Django up to 6.0, Python up to 3.14), renders inline SVGs styled with Tailwind classes

### `<c-icon>` cotton component

A cotton component that provides semantic icon lookup and renders heroicons:

```html
<c-icon name="next" class="size-5 text-blue-500" />
<c-icon name="success" variant="solid" class="size-6" />
```

**How it works internally:**

1. The component is defined at `freedom_ls/base/templates/cotton/icon.html`
2. It loads the `icon_tags` template tag library
3. It delegates to an internal `{% icon %}` template tag which:
   - Receives a name (e.g., `"next"`)
   - Looks up the name in the central `ICONS` registry dict
   - If found, uses the mapped heroicon name (e.g., `"arrow-right"`)
   - If not found, raises a `KeyError` — all icons must be defined in the registry to prevent typos and ensure consistency
   - If `force` is passed as `"true"`, skips the registry lookup and uses the name as a literal heroicon name (escape hatch for one-off cases)
   - Renders the SVG using the `heroicons` package's rendering functions

**Component props:**

| Prop | Default | Description |
|------|---------|-------------|
| `name` | (required) | Semantic name from registry (or direct heroicon name when `force="true"`) |
| `variant` | `"outline"` | Heroicon variant: `outline`, `solid`, `mini`, `micro` |
| `class` | `"size-5"` | CSS classes (Tailwind sizing, color, animation) |
| `force` | `""` | If `"true"`, skip registry lookup and use name as a literal heroicon name |
| `aria_label` | `""` | If set, renders `role="img"` + `<title>` instead of `aria-hidden="true"` |

**Component file** (`freedom_ls/base/templates/cotton/icon.html`):

```html
<c-vars name="" variant="outline" class="size-5" force="" aria_label="" />

{% load icon_tags %}

{% icon name variant=variant class=class force=force aria_label=aria_label %}
```

The component is a thin wrapper around the `{% icon %}` template tag. The template tag contains the actual Python logic for loading and rendering SVGs from the heroicons package. This separation is necessary because cotton components are pure HTML templates — the SVG manipulation (XML parsing, attribute setting, accessibility attributes) requires Python code in the template tag.

**Why a cotton component as the primary API:**
- No `{% load icon_tags %}` boilerplate in every template
- Consistent `<c-*>` syntax matching all other components in the project
- HTML-like syntax is more readable than template tag syntax
- The internal `{% icon %}` template tag still exists for use inside other cotton components (like `<c-chip>`, `<c-button>`) that need to render icons programmatically

**Why a Python dict (not Enum, not YAML)**:
- Type-checkable and IDE-friendly
- Easy to validate in tests (ensure every value maps to a real heroicon)
- Fits the project's Python-centric approach
- No extra parsing dependencies

### Variant strategy

- **Default variant**: `outline` (stroke-based, 24px design)
- Templates override per-use with `variant="solid"` etc. when needed
- The registry maps names to icon names only, not variants
- **Why**: Keeps the registry simple. Variant choice is a presentation concern that varies by context (e.g., a status icon might be `outline` in a list but `solid` in a badge).

Replace all `<i class="fa-solid fa-...">` / `<i class="fas fa-...">` / `<i class="fa fa-...">` tags with `<c-icon name="semantic_name" />` calls.

**Key abstraction points that reduce work:**

**Note on internal cotton component usage:** Cotton components that render icons internally (like `<c-button>`, `<c-chip>`) should use the `{% icon %}` template tag directly rather than nesting `<c-icon>`. This is the standard pattern — the cotton component wrapping is for end-user templates.

## Component updates

### `<c-button>` (`cotton/button.html`)

**Current**: Renders `<i class="fa-solid fa-{{ icon_left }} mr-2"></i>`
**New**: Renders `{% icon icon_left class="size-5 mr-2" %}` (using the template tag internally)

The `icon_left` and `icon_right` parameters keep the same API. Templates continue using `<c-button icon-left="check">` — but now icon names must be semantic names from the registry.

### `<c-chip>` (`cotton/chip.html`)

**Current**: Renders `<i class="fa-solid fa-{{ icon }}"></i>`
**New**: Renders `{% icon icon class="size-4" %}` (using the template tag internally)

### `<c-callout>` (`cotton/callout.html`)

**Current**: Maps callout level to FA class names (`fa-info-circle`, `fa-exclamation-triangle`, etc.)
**New**: Maps callout level to semantic icon names (`info`, `warning`, `error`, `success`) and renders via `{% icon %}`

### `<c-dropdown-menu>` (`cotton/dropdown-menu.html`)

**Current**: Default trigger uses `<i class="fa fa-ellipsis-v" aria-hidden="true"></i>`
**New**: Default trigger uses `{% icon "more_options" class="size-5" %}`

### `<c-loading-indicator>` (`cotton/loading-indicator.html`)

**Current**: `<i class="fas fa-spinner fa-spin text-3xl text-muted/50"></i>`
**New**: `{% icon "loading" class="size-8 animate-spin text-muted/50" %}`

The `arrow-path` heroicon (circular arrows) with Tailwind's `animate-spin` class replaces FA's `fa-spinner fa-spin`.

### `<c-header-button>` (`cotton/header-button.html`)

The component itself renders a bare `<button>` with `{{ slot }}` — all FA usages are in `{% comment %}` documentation blocks only. Update the example icons in the comment blocks from FA `<i>` tags to `<c-icon />` calls so the documentation reflects the new approach.

### Toast messages (`partials/messages.html`)

Replace the 5 inline SVGs with `<c-icon />` calls using semantic names (`success`, `error`, `warning`, `info`, `close`).

### Modal close button (`cotton/modal.html`)

Replace inline SVG close icon with `<c-icon name="close" />`.

### Data table boolean cells (`cotton/data-table-cells/boolean.html`)

Replace inline SVG checkmark/X with `<c-icon name="boolean_true" />` / `<c-icon name="boolean_false" />`.

### Data table sorting (`cotton/data-table.html`)

Replace FA sort icons with `<c-icon name="sort_asc" />`, `<c-icon name="sort_desc" />`, `<c-icon name="sort_neutral" />`.

## Accessibility

All icons default to **informative** — `aria-label` is set to the semantic icon name and `role="img"` is applied. This means every icon is accessible by default without any extra work.

The `aria_label` prop can be overridden to provide a more descriptive label:

```html
<!-- Default — aria-label is the semantic name ("check") -->
<button><c-icon name="check" class="size-5 mr-2" /> Mark Complete</button>
<!-- Renders: <svg role="img" aria-label="check" ...> -->

<!-- Custom label — override the default -->
<c-icon name="success" aria_label="Completed" class="size-5" />
<!-- Renders: <svg role="img" aria-label="Completed" ...> -->
```

## CI regression test

Write a test that greps the entire template directory for leftover Font Awesome patterns to prevent regression:

**Patterns to check for:**
- `fa-` (FA icon class prefix)
- `fa ` (old FA class prefix)
- `fas ` / `far ` / `fab ` / `fal ` (FA style prefixes)
- `fa-solid` / `fa-regular` / `fa-brands` (FA 6 style prefixes)
- `font-awesome` (CDN URL or references)
- `cdnjs.cloudflare.com` (CDN URL)

Also check for leftover inline SVG icons:
- `<svg` elements in templates that contain icon paths (e.g., `viewBox="0 0 24 24"` with `<path` children) — all standard icons should use `<c-icon />` instead

**Why**: The most common icon migration failure is missed replacements. An automated grep catches any regressions when new templates are added.

- Uses the `{% icon %}` template tag inside other cotton components that render icons internally

## Testing

### Unit tests for the icon system

1. **Registry validation**: Test that every value in `ICONS` dict corresponds to a valid heroicon name
2. **Template tag rendering**: Test that `{% icon "next" %}` renders an `<svg>` element
3. **Semantic lookup**: Test that `{% icon "next" %}` renders the same SVG as `{% icon "arrow-right" force=True %}`
4. **Variant override**: Test that `variant="solid"` renders solid variant
5. **Accessibility defaults**: Test that rendered SVG includes `aria-hidden="true"` by default
6. **Accessibility label**: Test that `aria_label="Close"` renders `role="img"` and `<title>`
7. **Unknown icon**: Test that an unregistered icon name raises `KeyError`
8. **Force passthrough**: Test that `{% icon "arrow-right" force=True %}` renders the heroicon directly without registry lookup
9. **Cotton component rendering**: Test that `<c-icon name="next" />` renders the same SVG output as the template tag equivalent

### Regression test

10. **No leftover FA patterns**: Grep all `.html` templates for Font Awesome class patterns — test fails if any are found
11. **No leftover inline SVG icons**: Grep all `.html` templates for hand-coded `<svg` elements that represent icons (exclude legitimate non-icon SVGs if any). All standard icons should use `<c-icon />` instead of inline SVG paths
