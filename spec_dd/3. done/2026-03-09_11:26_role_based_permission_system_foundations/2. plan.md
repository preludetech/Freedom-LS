## Task 2: Implement `types.py` — Role and SiteRolesConfig

**Depends on:** Task 1

**Skill:** `tdd_implement`

**File:** `freedom_ls/role_based_permissions/types.py`

Implement:
- `Role` — frozen dataclass with fields: `display_name: str`, `permissions: frozenset[str]`, `lti_role: str | None = None`, `ui_hint: str = "standalone"`, `description: str = ""`
- `SiteRolesConfig` — container class with:
  - `__init__(self, roles: dict[str, Role])`
  - `extend(self, overrides: dict) -> SiteRolesConfig` — three override forms: full replacement (Role), modification (dict with add_permissions/remove_permissions), inheritance (dict with "inherits" key)
  - `__getitem__`, `__contains__`, `__iter__`, `items()`, `keys()`, `all_permission_strings()`

See spec Section 2.3 for the exact pseudocode.

**Tests** (`freedom_ls/role_based_permissions/tests/test_types.py`):
- `Role` is frozen (immutable)
- `Role` defaults: `lti_role=None`, `ui_hint="standalone"`, `description=""`
- `SiteRolesConfig` basic access: `__getitem__`, `__contains__`, `__iter__`, `items()`, `keys()`
- `SiteRolesConfig.extend()` with full Role replacement
- `SiteRolesConfig.extend()` with `add_permissions` dict
- `SiteRolesConfig.extend()` with `remove_permissions` dict
- `SiteRolesConfig.extend()` with both `add_permissions` and `remove_permissions`
- `SiteRolesConfig.extend()` with `inherits` key — copies parent permissions and applies modifications
- `SiteRolesConfig.extend()` with `inherits` from unknown role raises `ValueError`
- `SiteRolesConfig.extend()` preserves parent fields (`display_name`, `lti_role`, `ui_hint`, `description`) when not overridden
- `SiteRolesConfig.extend()` overrides parent fields when specified
- `SiteRolesConfig.all_permission_strings()` returns union of all role permissions
- Single-level inheritance only (a role inheriting from an inherited role should still work since the parent is already resolved in `new_roles`)

## Task 6: Implement models — Role Assignment Models

**Depends on:** Task 1
**Skill:** `tdd_implement`

**File:** `freedom_ls/role_based_permissions/models.py`

Implement three models as described in spec Sections 3.3.1–3.3.3:

### `SystemRoleAssignment`
- `user` — FK to `User`, `on_delete=CASCADE`, `related_name="system_roles"`
- `role` — `CharField(max_length=50)`
- `is_active` — `BooleanField(default=True)`
- `assigned_by` — FK to `User`, `null=True`, `on_delete=SET_NULL`, `related_name="+"`
- `assigned_at` — `DateTimeField(auto_now_add=True)`
- `Meta`: `unique_together = ["user", "role"]`
- Does NOT extend `SiteAwareModel` (plain `models.Model`)

### `SiteRoleAssignment`
- Extends `SiteAwareModel` (gets UUID PK + site FK + auto-filtering)
- `user` — FK to `User`, `on_delete=CASCADE`, `related_name="site_roles"`
- `role` — `CharField(max_length=50)`
- `is_active` — `BooleanField(default=True)`
- `assigned_by` — FK to `User`, `null=True`, `on_delete=SET_NULL`, `related_name="+"`
- `assigned_at` — `DateTimeField(auto_now_add=True)`
- `Meta`: `unique_together = ["user", "site", "role"]`, indexes on `["user", "is_active"]` and `["site", "role"]`

### `ObjectRoleAssignment`
- Extends `SiteAwareModel`
- `user` — FK to `User`, `on_delete=CASCADE`, `related_name="object_role_assignments"`
- `content_type` — FK to `ContentType`, `on_delete=CASCADE`
- `object_id` — `CharField(max_length=255)` (NOT UUIDField — accommodates UUID, BigAutoField, IntegerField PKs)
- `target` — `GenericForeignKey("content_type", "object_id")`
- `role` — `CharField(max_length=50)`
- `is_active` — `BooleanField(default=True)`
- `assigned_by` — FK to `User`, `null=True`, `on_delete=SET_NULL`, `related_name="+"`
- `assigned_at` — `DateTimeField(auto_now_add=True)`
- `Meta`: `unique_together = ["user", "content_type", "object_id", "role"]`, indexes on `["user", "is_active"]` and `["content_type", "object_id", "role", "is_active"]`

After implementing, run `uv run manage.py makemigrations` and `uv run manage.py migrate`.

**Factories** (`freedom_ls/role_based_permissions/factories.py`):
- `SystemRoleAssignmentFactory` — plain `DjangoModelFactory` (not SiteAwareFactory since SystemRoleAssignment is not site-aware)
- `SiteRoleAssignmentFactory` — extends `SiteAwareFactory`
- `ObjectRoleAssignmentFactory` — extends `SiteAwareFactory`, with helpers for setting content_type/object_id from a target object

**Tests** (`freedom_ls/role_based_permissions/tests/test_models.py`):
- `SystemRoleAssignment`: create, unique_together constraint enforced
- `SiteRoleAssignment`: create with site, unique_together constraint, site-aware filtering works
- `ObjectRoleAssignment`: create with Course target (UUID PK), create with User target (BigAutoField PK), create with Site target (IntegerField PK), unique_together constraint, `target` GenericFK resolves correctly
- `is_active` defaults to `True`
- `assigned_at` is auto-set

## Task 7: Implement `utils.py` — Role Assignment and Permission Sync

**Depends on:** Tasks 5, 6 (needs loader, models, and all config)
**Skill:** `tdd_implement`

**File:** `freedom_ls/role_based_permissions/utils.py`

Implement the following functions (see spec Sections 3.4 and 4.2):

### `validate_role(role_name: str) -> None`
- Load config via `get_role_config()`
- Raise `ValueError` if `role_name` not in config

### Internal helpers

#### `_get_active_roles_for_user_on_object(user, obj) -> set[str]`
- Query `ObjectRoleAssignment` for the user + object (via ContentType lookup) where `is_active=True`
- Return set of role name strings

#### `_get_active_roles_for_user_on_site(user, site) -> set[str]`
- Query `SiteRoleAssignment` for the user + site where `is_active=True`
- Return set of role name strings
- Needed by `assign_site_role` / `remove_site_role` to compute permissions on the Site object

#### `_get_guardian_perms_as_full_strings(user, obj) -> set[str]`
- Query guardian's `UserObjectPermission` model directly to get `"app_label.codename"` format strings
- Cannot use `get_perms()` because it returns bare codenames only
- Join through `Permission` and `ContentType` to build full strings

### `sync_user_object_permissions(user, obj, dry_run=False) -> dict`
- Get all active roles for user on object (use `_get_active_roles_for_user_on_object` for generic objects, `_get_active_roles_for_user_on_site` when obj is a Site)
- Compute desired permissions (union of all role permissions from config)
- Get current guardian permissions as full strings
- Compute diff (to_add, to_remove)
- If not dry_run, apply changes in a transaction using `assign_perm`/`remove_perm`
- Return dict with user, object, roles, added, removed

### Object-level role functions

#### `assign_object_role(user, target, role, assigned_by=None) -> ObjectRoleAssignment`
- Validate role
- Get or create `ObjectRoleAssignment` (reactivate if inactive)
- Call `sync_user_object_permissions`
- Keep TODO comment for AuditLog
- Return assignment

#### `remove_object_role(user, target, role, removed_by=None) -> None`
- Set matching assignment's `is_active=False`
- Call `sync_user_object_permissions` (recomputes — other roles may share permissions)
- Keep TODO comment for AuditLog

### Site-level role functions
- `assign_site_role(user, role, assigned_by=None) -> SiteRoleAssignment` — creates `SiteRoleAssignment`, then syncs guardian permissions on the current Site object via `sync_user_object_permissions(user, site)`
- `remove_site_role(user, role, removed_by=None) -> None` — deactivates assignment, resyncs guardian on Site object

### System-level role functions
- `assign_system_role(user, role, assigned_by=None) -> SystemRoleAssignment` — creates `SystemRoleAssignment`. No guardian sync (system roles have no object to scope to — they're checked directly from assignment records)
- `remove_system_role(user, role, removed_by=None) -> None` — deactivates assignment

### Convenience query methods (spec Section 3.3.3 note)

#### `get_object_roles(user, obj) -> set[str]`
- Generic: returns set of active role names for user on any object (via ContentType)

#### `get_course_roles(user, course) -> set[str]`
- Convenience wrapper around `get_object_roles` for Course objects

#### `get_cohort_roles(user, cohort) -> set[str]`
- Convenience wrapper around `get_object_roles` for Cohort objects

**Tests** (`freedom_ls/role_based_permissions/tests/test_utils.py`):
- `validate_role`: valid role passes, unknown role raises `ValueError`
- `assign_object_role`: creates assignment, sets guardian permissions on the object
- `assign_object_role`: reactivates inactive assignment
- `assign_object_role`: invalid role raises `ValueError` and creates no assignment
- `remove_object_role`: deactivates assignment, removes guardian permissions
- `remove_object_role`: when user has multiple roles sharing a permission, only removes permissions not covered by remaining roles
- `sync_user_object_permissions`: computes correct diff (added/removed sets)
- `sync_user_object_permissions`: dry_run=True reports changes without applying them
- `sync_user_object_permissions`: no roles means all permissions removed
- `assign_site_role` / `remove_site_role`: creates site assignment and syncs guardian permissions on Site object
- `remove_site_role`: deactivates and resyncs, preserves perms from other active site roles
- `assign_system_role` / `remove_system_role`: creates/deactivates system assignments (no guardian sync)
- `get_object_roles`: returns correct active roles for user on object
- `get_course_roles` / `get_cohort_roles`: convenience wrappers return correct results
- Integration test: assign instructor role on a Cohort, verify `get_objects_for_user()` returns that cohort (proves educator_interface views will work)

## Task 8: Implement `sync_role_permissions` management command

**Depends on:** Task 7
**Skill:** `tdd_implement`

**File:** `freedom_ls/role_based_permissions/management/commands/sync_role_permissions.py`

Implement the deployment sync command (spec Section 4.3):

- `--dry-run` flag: report changes without writing
- `--report-orphans` flag: detect guardian perms with no matching role assignment
- Phase 1: Ensure permission objects exist in Django's `auth_permission` table
- Phase 2: Iterate all active `ObjectRoleAssignment` records, sync each user+object pair
- Phase 3: Iterate all active `SiteRoleAssignment` records, sync each
- Phase 4: Iterate all active `SystemRoleAssignment` records (no guardian sync needed — just validate roles exist in config)
- Phase 5: If `--report-orphans`, scan guardian `UserObjectPermission` for rows not traceable to any active assignment
- Output: count of drifted assignments, orphan report

**Tests** (`freedom_ls/role_based_permissions/tests/test_management_commands.py`):
- Sync command with no assignments: runs cleanly, reports 0 drift
- Sync command detects and fixes drift (manually add a guardian perm, verify command removes it)
- Sync command with `--dry-run` reports drift but doesn't change guardian state
- Sync command handles orphaned assignments (target object deleted)
- `--report-orphans` detects manually-granted guardian permissions
