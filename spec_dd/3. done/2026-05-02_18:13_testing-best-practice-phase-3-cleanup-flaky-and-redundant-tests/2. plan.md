# Implementation plan — Phase 3: cleanup flaky and redundant tests

This plan executes the test-only cleanup described in `1. spec.md`. The work is in three independent buckets that should land in a single PR but can be implemented sequentially:

1. **Delete / rewrite brittle assertions** at the audited sites (CSS classes, hardcoded config values, trivial model defaults).
2. **Audit `@pytest.mark.django_db(transaction=True)` usages** — strip where unjustified, add rationale comments where kept, document the rule in `fls-claude-plugin/resources/testing.md`.
3. **Fix flaky tests surfaced by `pytest-randomly`** by addressing the actual shared-state leak (no `@pytest.mark.order`, no silent skips).

There are **no production code changes**, **no new factories**, **no view-assertion strengthening**, and **no `SKILL.md` edits**. The single permitted documentation edit is the new `transaction=True` rule in `fls-claude-plugin/resources/testing.md`.

The full audit citations live in `spec_dd/3. done/2026-04-28_07:25_testing-best-practice/research_audit.md` (§1, §6, §7). Re-verify counts at implementation time — the audit is a snapshot.

## Tasks

The order is: docs first (so the rule exists before tests reference it), then deletions/rewrites (the bulk of the diff), then `transaction=True` audit, then random-order sweep.

### 1. Update `resources/testing.md` with the `transaction=True` rule

Edit `fls-claude-plugin/resources/testing.md`. Add a new subsection under "Test order independence" titled **"`transaction=True` is expensive — justify it"**.

Pseudocontent:

```markdown
### `transaction=True` is expensive — justify it

The default `@pytest.mark.django_db` runs each test inside a transaction that is rolled back at teardown. `@pytest.mark.django_db(transaction=True)` instead runs each test in its own transaction and **truncates tables** between tests, which is dramatically slower *and* exposes signal handlers and `transaction.on_commit(...)` callbacks that the rollback path was hiding.

Use `transaction=True` **only** when the test genuinely needs:
- `transaction.on_commit(...)` callbacks to fire (e.g. webhook event delivery).
- `select_for_update` semantics inside the test.
- Cross-connection visibility (e.g. Playwright e2e tests where the browser runs in a separate process and must see committed data).

When `transaction=True` is used, **add a one-line comment** on the marker explaining the reason. Example:

# transaction=True so that on_commit hooks for webhook delivery fire under test
@pytest.mark.django_db(transaction=True)

If you are tempted to add `transaction=True` for any other reason (it makes a flaky test pass, you don't know why the rollback path is failing), do not — it is masking a real bug in the test or the production code.
```

This is the **only** documentation edit allowed by the spec. Do not touch `SKILL.md` files. Do not edit `playwright-testing.md`.

### 3. Audit `@pytest.mark.django_db(transaction=True)` usages

For each of the seven sites listed above (and any new ones that appear at implementation time — `git grep 'transaction=True'`):

1. Read the test. Does it depend on `transaction.on_commit`, `select_for_update`, or cross-connection visibility?
2. **If yes**: add a one-line comment immediately above the marker stating the reason. Match the tone of the existing comment in `freedom_ls/panel_framework/tests/conftest.py:90`. Example:

   ```python
   # transaction=True so on_commit webhook-delivery callbacks fire under test
   @pytest.mark.django_db(transaction=True)
   class TestWebhookEventsOnAccountCreation:
       ...
   ```

3. **If no**: strip the marker (replace with `@pytest.mark.django_db`) and run the test under the new marker to confirm it still passes. **If the test starts failing after the strip, the answer was actually "yes" — restore the marker and add the rationale comment.**

Threat-model §4 control C5 covers this: "kept-with-rationale-comment everywhere else." When in doubt, **keep the marker and add the comment** — stripping a marker that was secretly load-bearing turns a real test into a silently-passing-for-the-wrong-reason test.

Per the spec's success-criterion #7, this audit is the **only** doc-style change beyond the `resources/testing.md` rule edit. Do not edit `SKILL.md`.

### 4. Random-order flake sweep

Phase 2 has `pytest-randomly` live with a deterministic seed per CI run. Surface flakes with several seeds and against the `last` seed:

- Run `uv run pytest -m "not playwright" -p randomly --randomly-seed=last` immediately to capture the current ordering. Note the seed.
- Then run `uv run pytest -m "not playwright"` 5–10 times, each picking up a fresh random seed. Capture the failing test ids and seeds.
- For each unique failure:
  1. Run the failing test alone (`uv run pytest <node-id>`) — does it still fail? If yes, it's not order-dependence; investigate as a regular bug.
  2. Run with `-p no:randomly` and the same node id — does it pass? That confirms order-dependence.
  3. Bisect: run the failing test with each preceding test id from the random-order log, narrowing until you find the *one* prior test that mutates the shared state. The bisect target is something like `uv run pytest <suspect-prior> <failing-test> -p no:randomly`.
  4. Fix the **leak**, not the symptom. Common leak shapes:
     - Module-level mutable (a class attribute, a module dict): reset in fixture or move to instance scope.
     - Factory sequence collision: ensure the factory's `Sequence` is reset or use `LazyFunction(uuid)`.
     - `Site.objects.get_current()` cache: clear `Site.objects.clear_cache()` in fixture teardown.
     - Signal re-attached without cleanup: detach in fixture teardown via `signal.disconnect(...)`.
     - `clear_caches()` not called: many `freedom_ls/role_based_permissions/tests/*` already use `_clear_caches` as `autouse`; check whether the failing test is missing that fixture or running in a module that doesn't import it.
  5. Re-run with the failing seed to confirm the fix. Re-run 5 more random seeds to confirm no new failures appear.

#### Forbidden remediations

- **No `@pytest.mark.order(N)`** — this hides ordering coupling instead of fixing it; defeats the gate.
- **No `pytest.mark.skip` without a tracking ticket** — file the issue first, then `pytest.skip("blocked on #NNN")` with the ticket id in the message; never silently skip.
- **No fixture that papers over the leak** (e.g. an `autouse` fixture that simply re-runs the cleanup the leaking test should have done). The fix must be in the test that causes the leak, or in a clearly-named teardown that addresses the specific shared state.

#### If a flake reveals a real production bug

Per the spec: **do not fix production code in this PR**. File a tracking issue describing the bug, mark the test `pytest.skip("blocked on #NNN")` with the ticket reference, and link the ticket in the PR description. The production fix lands in its own PR with its own threat model.

### 5. Verify net LoC negative

Before opening the PR, run `git diff --stat main...` and confirm `-N` lines is greater than `+N` lines for the test files. This is success criterion #4 and the threat-model's control C6 — a positive net LoC means the cleanup turned into a rewrite, which is out of scope.

The `resources/testing.md` addition (~15 lines) and any rationale comments on `transaction=True` markers (~7 × 1 line) will add a small positive count outside the test files; the test-file diff alone must be net-negative. Calculate from `git diff --stat main... -- 'freedom_ls/**/test_*.py' 'freedom_ls/**/tests/**/*.py'`.

Run:

```
git diff --name-only main... | grep -v -E '(test_|/tests/|conftest\.py$|resources/testing\.md$)'
```

## Risks and mitigations

- **Bulk-deleting tests removes a behavioural assertion by accident.** Mitigation: per-deletion checklist in spec step 1 ("read surrounding assertions"). Reviewer-side mitigation in threat-model §7.2.
- **A rewrite is too weak (e.g. asserts only on type, not on the security-relevant invariant).** Mitigation: threat-model §7.3 review check — "if someone widened the permission set, would the rewritten assertion fail?"
- **Stripping `transaction=True` from a marker that was actually needed turns a passing test into a silently-passing-for-the-wrong-reason test.** Mitigation: keep-by-default; only strip if the test obviously does no `on_commit` / `select_for_update` / cross-connection work; restore the marker if the test fails after the strip (per Task 3).
- **A flake fix appears to work but only because the new fixture happens to evaluate before the leaking test on the next run.** Mitigation: re-run with multiple seeds (Task 4 step 5); also run with `-p no:randomly --randomly-seed=last` to confirm the original failing seed still passes after the fix.
- **`pytest-randomly` reveals a flake whose root cause is in production code (e.g. a signal handler with hidden state).** Mitigation: the spec's file-and-skip rule applies — file a ticket, skip the test referencing the ticket, document in the PR description, do not fix in this PR.
- **Net LoC drifts positive because the rewrites add more lines than the deletions remove.** Mitigation: prefer pure-deletion to rewrite for the cited config-count tests (most are redundant with already-existing structural tests); the rewrites should be one-liners, not multi-line replacements.
