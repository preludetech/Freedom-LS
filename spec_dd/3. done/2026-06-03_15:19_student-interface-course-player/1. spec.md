# Specification: Student Course Player

## Summary

Rework the student course-reading experience into a focused two-pane **course
player**: a persistent left-hand Table of Contents (TOC) panel and the current
item's content on the right. The intermediate course start page is retired so
enrolled learners go straight into content; the player gains breadcrumbs,
per-item browser page titles, an outline panel with a pinned progress header,
and a mobile bottom-sheet treatment for the TOC.

This spec is centred on the `student_interface` app, but also unifies the app's
sidebar mechanism: it introduces one shared `<dialog>`-based side panel in `base`
and migrates the `educator_interface` and `panel_framework` sidebars onto it,
retiring the legacy `sidebarComponent` (see Section 6). It draws on three research
documents in this directory (`research_desktop_toc_sidebar.md`,
`research_mobile_bottom_sheet_nav.md`, `research_breadcrumbs_resume_titles.md`)
and incorporates product decisions captured during research review (see
"Decisions made" below).

---

## Why this matters

- **Less friction to learning.** Every screen between a returning learner and
  the content they came for is a drop-off point. Resolving the bare course URL
  straight to a resume target removes an intermediate landing screen and matches
  the "Continue where you left off" expectation users have from every other LMS
  and reading app.
- **Orientation when parachuted in.** Dropping learners directly into content
  creates "where am I?" moments. Breadcrumbs, an always-on outline with the
  current item highlighted, and accurate per-item page titles answer that
  question visually and for assistive tech.
- **Learners first.** The always-open desktop outline and the thumb-reachable
  mobile bottom sheet put navigation where learners can use it without losing
  their place in the content.

---

## Decisions made (during research review)

These were open questions resolved by the product owner; the spec applies them
directly.

1. **`course_home` is legacy.** Courses no longer have a "home page." The old
   start-page template (`course_home.html`) and its rendering view are retired.
   The bare course URL becomes a resume redirect (see Section 1).
2. **Unify every sidebar on a native `<dialog>`.** The mobile TOC bottom sheet
   and — crucially — the shared interface **side panel** are rebuilt on the native
   HTML `<dialog>` element. The legacy Alpine `sidebarComponent` (hosted by
   `_base_interface.html` and inherited by the student course base, the **educator
   interface**, and the **panel_framework**) is **replaced** by one shared
   side-panel mechanism so that every sidebar in the app behaves identically. The
   triggers that open a side panel differ per location; the mechanism is the same
   everywhere (see Section 4 and Section 6).
   - **Implementation note:** the unified side panel is the same `<dialog>` driven
     by one `sidePanel` controller (docked via `dialog.show()` on desktop, modal
     via `dialog.showModal()` below `lg`). The **picture lightbox was NOT migrated**
     onto this in this pass — it keeps a dedicated focus-managed overlay
     (`contentLightbox`, replacing the old `pictureModal`); see Section 6. A
     generic `<c-overlay>` + `overlayDialog` `<dialog>` primitive was prototyped as
     a shared foundation but **removed as dead code** once it was clear nothing
     consumed it (the side panel needs its own dual-mode controller and the
     lightbox stayed bespoke).
3. **No item/module counter in the TOC header.** The TOC header shows only the
   progress bar and the percentage — the `{N} / {M} mods` counter from the
   mockup is removed entirely (see Section 3).

## Section 1: Retire the course start page; resume redirect on the bare URL

### Behaviour

- The bare course URL `/courses/{slug}/` (URL name `course_home`) **no longer
  renders the old start-page template**. The `course_home.html` template and the
  `course_home` view body are retired (see "Migration / cleanup" for what
  replaces the URL name and its inbound references).
- For an **enrolled** learner, the bare course URL resolves to a resume target
  and **302-redirects** to the canonical per-item URL
  `/courses/{slug}/{N}/` (URL name `view_course_item`):
  - **No progress yet → the first viewable item** (`index = 1`). A true start.
  - **Has progress → the last-accessed viewable item** (resume).
- The per-item URL `/courses/{slug}/{N}/` is the **canonical, bookmarkable** URL.
  The bare course URL is a stable "take me to where I am" entry point that always
  resolves to the learner's current spot.
- For an **unenrolled / anonymous** visitor, the player is off-limits. The bare
  course URL must not render the player or leak per-user state. Such a visitor
  belongs on the existing **course preview page**
  (`/courses/{slug}/preview/`), which keeps the description / overview /
  register CTA. The bare URL should send unenrolled users there (or to login for
  anonymous users, consistent with current auth behaviour). Nothing in the
  player should ever link an unenrolled user into the player.

### Determining the last-accessed item

> **Approach taken (plan + implementation):** rather than *derive* the resume
> target from per-item progress timestamps, the plan stored it **explicitly** —
> `CourseProgress` gained a nullable `last_accessed_item` `GenericForeignKey`
> (content-type FK + UUID `object_id`), stamped on every item view in
> `view_course_item` for both topics and forms. This is the refinement this spec
> anticipated (see Migration / cleanup), and it makes the per-item-timestamp
> derivation and the `view_form`-on-GET change below **unnecessary**. The text
> below is retained for context but is superseded by the explicit field.

The resume target is read from `CourseProgress.last_accessed_item`:

- It is set at the single canonical write point (`view_course_item`), so a form
  the learner was last reading participates in resume identically to a topic — no
  `view_form`-on-GET touch is needed and **no spurious `FormProgress` attempt** is
  ever minted on a bare GET.
- The resume helper resolves the stored item's 1-based index in
  `course.viewable_items()` and redirects there.
- If a learner has a `CourseProgress` row but no `last_accessed_item` yet (edge
  case), or the stored item is no longer viewable, fall back to the first item.
- This lookup must be a single, testable helper (in `utils.py`): one row fetch +
  one FK resolve, no per-item query loop.

### Back-button safety (no redirect loop)

A 302 from the bare URL to the per-item URL risks a Back-button loop: Back
returns to the bare URL, which re-redirects forward. Guard against this:

- The redirect must be a plain 302 to a **different** canonical URL, and the
  player item pages must **not** link back to the bare course URL anywhere
  (breadcrumb course-root crumb and the "Home"/finish buttons must point at item
  URLs or the preview, never at `course_home`). With nothing in the player
  pointing back at the bare URL, normal Back navigation lands on the previously
  viewed item, not the redirector.
  - **Dashboard cards are exempt.** The registered-course card on the dashboard
    may keep `course_home` as its link fallback (used only when there is no
    actionable "next up" item). The dashboard is **not** a player page, and a 302
    redirect from a non-player navigation does not create a Back-button loop (the
    redirecting URL is not retained in session history), so the card simply lands
    the learner on their resume item — which is more correct than hard-coding
    item 1 for, e.g., a completed course.
- The QA plan must explicitly exercise the Back button from a resolved item and
  confirm no loop and no skipped history.

### Why

Research (`research_breadcrumbs_resume_titles.md` §B) is consistent: reduce
intermediate steps, default to resume, keep the bookmarked course URL resolving
to a *current* item, and test the Back-button journey because course→player
redirects are a known loop hazard.

## Section 2: Breadcrumbs (desktop)

At the top of the main content area, above the item content, show a breadcrumb
trail:

```
{course title} > {course-part if there is one} > {current item title}
```

### Requirements

- Wrap in a breadcrumb navigation landmark: `<nav aria-label="Breadcrumb">`
  containing an ordered list (`<ol>`).
- **Ancestor crumbs are links** (course root → the course's **first item**,
  `/courses/{slug}/1/` — a concrete, stable item URL, never the retired
  `course_home` / bare course URL; course-part → its first viewable item). The
  **current item is plain text, not a link**, and carries `aria-current="page"`.
- When the current item **has no course-part**, drop that crumb entirely — no
  dangling separator, no padding.
- Render the `>` separators as **CSS decoration** (e.g. `::before`/`::after`),
  not as real text or list items, so screen readers don't announce them.
- **Truncate** an over-long course or item title with an ellipsis rather than
  wrapping or breaking the layout; keep the full text available to assistive
  tech (e.g. `title` attribute / not removed from the accessible name).

### Course-part lookup

The breadcrumb needs the current item's parent course-part. `viewable_items()`
flattens parts away, and there is currently **no item→parent-part accessor**.
This lookup must be added (a helper that, given the course and the current item,
returns the containing `CoursePart` or `None`) and reused by the breadcrumb, the
mobile header, and the page title. It must not introduce N+1 queries.

### Why

`research_breadcrumbs_resume_titles.md` §A: breadcrumbs are high-value secondary
navigation for deep-linked / parachuted-in users (exactly what Section 1
creates). The current page must not be a link; separators must be CSS; drop the
empty part crumb; truncate gracefully.

## Section 3: Left-hand TOC panel (desktop)

The player's left column is a persistent outline panel, replacing the generic
`_base_interface.html` sidebar usage for player pages.

### Header (pinned to the top of the panel)

```
COURSE OUTLINE          ← small eyebrow label
{Course Title}
[####......] progress bar
{X}% complete
```

- **No `{N} / {M} mods` counter.** (Removed per decision 3.) The header shows
  only the eyebrow, the course title, the determinate progress bar, and the
  `{X}% complete` text.
- The progress bar and the `% complete` text **derive from the same progress
  data** so they never disagree — use `CourseProgress.progress_percentage`
  (the single stored source of truth, already maintained by
  `update_course_progress_on_completion`). Do not recompute the percentage a
  second way in the template.
- The progress bar exposes its value to assistive tech (`role="progressbar"`
  with `aria-valuenow/valuemin/valuemax`, or an equivalent accessible text).
- The header is **pinned**; only the TOC body below it scrolls.

### Body (scrolls independently of the content)

- Render the table of contents reusing the existing `course_minimal_toc.html`
  structure (counters, status icons, deadline badges, `coursePart`
  expand/collapse).
- **Two levels only**: course part → item. No third tier. (Matches the existing
  data model and `get_course_index`.)
- **Course parts are expandable** disclosure controls. Each toggle is a
  `<button>` carrying `aria-expanded`, with a chevron affordance (already the
  case via the `coursePart` Alpine component + `expand`/`collapse` icons).
- **Highlight the current item** with a visual treatment (fill / accent bar) AND
  `aria-current="page"`. Auto-expand the part containing the current item, and
  scroll the current item into view within the TOC after navigation.
- **Preserve manual expand/collapse** the learner makes during the session —
  don't reset on every navigation. The existing `coursePart` component already
  persists per-part expand state in `localStorage` keyed by course slug + part
  index; auto-expanding the current part must not clobber a part the learner
  manually collapsed beyond the current one. (Auto-expand applies to the part
  containing the current item; other parts keep their stored state.)
- **Per-item status icons** (complete / current / not-started / locked / failed)
  must not rely on colour alone — pair each with an icon **and** accessible text
  (visually-hidden label or `aria-label`). The current icons (`complete`,
  `in_progress`, `locked`, `retry`, plus type icons) need accessible text added;
  today they have none.
- Wrap the panel in a `<nav aria-label="Course outline">` landmark with real
  `<ul>`/`<li>` structure (disclosure pattern, **not** an ARIA `tree`).
- The panel and the content area **scroll independently** (panel is full
  viewport height with its own scroll region; content scrolls separately and
  resets to top on navigation).

### Focus on navigation

When the learner activates an item and new content loads, move focus to the new
content's heading (`<h1>`) rather than the top of the page, and keep the TOC's
scroll/expansion context.

### Why

`research_desktop_toc_sidebar.md`: two-pane player is the dominant pattern;
pinned progress header; shallow nesting; disclosure (not `tree`); preserve
context and current-item visibility; single source of truth for progress; no
colour-only status; accessible status text.

## Section 4: Responsive behaviour

### Large screens (laptops, `lg` and up)

- The TOC panel is **always open**. No close control on the player.
- Content area keeps a comfortable max-width so the persistent panel doesn't
  cramp reading.

### Phones (below `lg`)

- The TOC panel is **collapsed by default** (not expanded).
- It opens as a **modal bottom sheet that rises from the bottom** — a bottom-sheet
  *presentation* of the shared side-panel mechanism (Section 6). **All three
  interfaces use the bottom sheet** below `lg` (the player TOC, the educator nav,
  and the panel_framework nav) — the bottom sheet is the shared default
  presentation, not a player-only treatment. The background behaviour (scrim,
  inertness, dismiss routes, focus) is identical across all of them.
- Modal sheet behaviour:
  - A scrim dims the background and the background is **inert** while open.
  - Opens to a **partial height** showing current context, with the TOC list
    scrolling **inside** the sheet (`overscroll-behavior: contain` to prevent
    scroll chaining to the page behind).
  - Animate with `transform`/`opacity` (not `height`/`top`) to avoid jank /
    layout shift.
- **Multiple dismiss routes, same outcome:** tap on the scrim, the **Escape**
  key, and the device **Back** button. (Swipe-down / drag-to-dismiss is optional
  polish.)
  - **Implementation note:** the shared panel deliberately carries **no chrome of
    its own** — no title row and **no visible × close control**. On mobile it is a
    native modal sheet dismissed by the scrim / Escape / device Back; on desktop
    it is docked and always open. (This applies to all three interfaces — see
    Section 6 — so the educator/panel bottom sheets also dismiss via scrim/Escape/
    Back rather than a × button.)
- Focus management: move focus into the sheet on open; **return focus to the TOC
  toggle** on close.

### Implementation baseline

Build the bottom sheet on the native `<dialog>` element opened with
`showModal()` (gives focus trapping, background inertness, Escape-to-close, and a
`::backdrop` scrim for free), with Alpine handling only open/close wiring and
`aria-expanded` on the toggle, and Tailwind transitions for the slide-up. See
Section 6 for the shared side-panel + overlay foundation this is built on. Touch
targets ≥ 48×48px; keep most-tapped controls slightly up from the very bottom
edge.

### Mobile header

The header chrome is **unified across breakpoints** — the same breadcrumb trail
renders on every viewport rather than a separate phone-only "compact bar". On
phones it is augmented with the controls a small screen needs:

- The **same breadcrumb trail** as desktop (`course > [part] > current item`,
  ancestor crumbs linked, current item plain `aria-current="page"` text). Long
  crumbs **truncate with an ellipsis at the ends** (each crumb has a `max-width`);
  there is no middle-truncated condensed variant.
- A **clearly labelled TOC toggle** sits **beside the breadcrumbs** (icon **plus**
  `aria-label` and `aria-expanded` — not a bare hamburger) that opens the bottom
  sheet. It is mobile-only (`lg:hidden`); on desktop the panel is always docked,
  so there is nothing to toggle.
- A **thin progress bar** with a `%`, rendered **below the breadcrumbs on phones
  only** (on desktop the docked outline panel carries its own progress header, so
  the content header stays uncluttered).
- **No back arrow** and **no bookmark icon.** (The course-root breadcrumb crumb
  plus the per-item "Previous" button and the browser Back button cover backward
  navigation; the bookmark icon was out of scope.)

### Why

`research_mobile_bottom_sheet_nav.md`: bottom sheet (not side drawer) for thumb
reach; modal with scrim + inert background; partial height + internal scroll;
all dismiss routes consistent; labelled toggle (hidden nav hurts
discoverability); native `<dialog>` baseline; animate transforms not layout.

## Section 5: Browser page title

The browser page `<title>` reflects the current item, **most-specific first**:

```
{Item title} — {Course title} — {Site name}
```

- The item title is front-loaded (visible in a crowded tab strip, announced
  first by screen readers).
- If a **course-part** exists it may sit between item and course
  (`{Item} — {Part} — {Course} — {Site}`), but the part is the **first
  droppable** element if length is a concern; the site/brand is trimmed before
  the item or course.
- Use a single consistent separator (en-dash `—`).
- If items ever swap **without a full page load** (e.g. an HTMX content swap),
  the title must still update on navigation. (Current navigation is full-page,
  so a server-rendered `{% block title %}` satisfies this today; the requirement
  stands for any future HTMX-driven item swaps.)

### Why

`research_breadcrumbs_resume_titles.md` §C: WCAG 2.4.2 requires descriptive,
page-specific titles that update on navigation; front-load the distinctive part
because tabs/announcements truncate.

## Section 6: Native `<dialog>` overlays + unified side panel

Every **sidebar** in the app is built on the native `<dialog>` element, sharing
one mechanism. This replaces the hand-rolled Alpine `sidebarComponent` with a
single accessible side-panel so that **all sidebars behave the same**, with
location-specific triggers. The picture lightbox is also `<dialog>`-flavoured but,
in this pass, kept its own bespoke overlay rather than the shared shell (see
Consumer 2).

### The shared foundation

The accessible **background behaviour** is provided by the native `<dialog>`
element opened with `showModal()`:

- scrim / background blur (`::backdrop`),
- background inertness,
- dismiss routes (scrim tap, Escape, device Back — **no visible × close
  control**; the panel carries no chrome of its own), and
- focus management (focus in on open, return focus to the trigger on close).

Native `<dialog>` provides focus trapping, inertness, Escape, and `::backdrop` for
free — exactly what hand-rolled overlays get wrong. Alpine wires only open/close
and `aria-expanded` on the trigger; Tailwind handles transitions.

- **Implementation note:** the unified side panel is a single `<dialog>` driven by
  the `sidePanel` controller directly (it needs a dual desktop-docked / mobile-modal
  mode a generic shell does not model). A generic `<c-overlay>` cotton shell +
  `overlayDialog` controller was prototyped as a shared centred/sheet/drawer
  primitive but **removed as dead code** — nothing consumed it (the side panel uses
  its own controller; the picture lightbox stayed bespoke).

### Consumer 1 — the unified side panel (replaces `sidebarComponent`)

**Every sidebar in the app shares one mechanism.** The legacy Alpine
`sidebarComponent` — hosted by `_base_interface.html` and inherited by the
**student course base**, the **educator interface**, and the **panel_framework**
test interface — is **replaced** by a single shared side-panel component. Its
contract:

- **Desktop (`lg`+):** a persistent, docked column. It coexists with the page, so
  it is opened non-modally with `dialog.show()` (a modal `<dialog>` would inert the
  page, which is wrong for a docked sidebar). The panel is **always docked open on
  desktop across every interface** (`data-desktop-lock` is set for all of them) and
  there is **no desktop collapse/toggle control**. (This is a deliberate
  simplification of the old per-interface desktop collapse + persisted open-state;
  the `localStorage` keys still exist on the controller but the desktop is locked
  open, so they are effectively dormant.)
- **Below `lg`:** an overlay opened with `<dialog>.showModal()` — scrim, inert
  background, focus trap, Escape, and device-Back dismissal (no × control).
- **Presentation:** all three interfaces present as a **bottom sheet** below `lg`
  (the shared default). The player TOC, the educator nav, and the panel_framework
  nav all rise from the bottom — same mechanism, same dismiss/focus behaviour.
  (The `<dialog>` mechanism *can* do a side-drawer variant, but the side panel
  does not use it; the bottom sheet is used everywhere.)
- **Triggers differ per location** (course-player TOC toggle, educator nav
  toggle, panel_framework toggle) but all open the same component the same way.

Migrating means: introduce the shared side-panel component, repoint
`_base_interface.html` onto it, and **retire `sidebarComponent`** from
`base/static/base/js/alpine-components.js`. All three host interfaces must keep
working with no user-visible regression (see Migration / cleanup and the QA
plan). Content that lives *inside* the panel — the course TOC body, the educator
`sidebar_nav.html` / `sidebarMenuItem` expand-collapse — is unchanged; only the
panel **shell** mechanism changes.

### Consumer 2 — the picture / diagram lightbox

A **centred lightbox** (not a bottom sheet) for tap-to-zoom figures in
`content_engine`'s `cotton/picture.html`. The legacy `pictureModal` Alpine
component is **retired and replaced by `contentLightbox`**.

- **Implementation note — not on a native `<dialog>` foundation.** In this pass
  the lightbox was **not** migrated onto native `<dialog>`. It stays an Alpine
  `x-show` overlay, but a **focus-managed** one: `contentLightbox` captures the
  trigger, moves focus to the close button on open, restores focus to the trigger
  on close, closes on Escape and on click-outside, and keeps a minimal focus trap
  on the close button. It has a backdrop **blur** and a visible **× close** button
  (unlike the side panel). It does **not** make the background inert (an `x-show`
  `<div>` is not a top-layer modal). Migrating it onto a native `<dialog>` is a
  reasonable future cleanup, not part of this spec.

### Not part of this migration — `c-modal`

The existing `c-modal` cotton component is a **form / dialog modal**, not a
sidebar, and is **out of scope** for this change. It keeps working as-is. (It
could adopt the same `<dialog>` foundation in a future pass; not now.)

### Why

The idea's implementation notes call for a native-`<dialog>` (low-JS, accessible)
baseline with shared scrim/blur/inertness/dismiss reused across overlay types.
The product owner additionally directed that **all sidebars behave identically**
and reuse one mechanism across the student player, educator interface, and panels
app, with location-specific triggers.

## Migration / cleanup

- **`course_home` URL name and view.** The URL name is referenced by several
  templates and views. Either remove the name and update all references, or
  repurpose the URL/view as the resume redirector (Section 1). The old
  start-page template `course_home.html` is **retired** either way. Inbound
  references to update (do not leave any pointing at a rendered start page):
  - `views.py`: `_preview_start_url` 0-progress fallback, `register_for_course`
    redirect, deadline-locked redirect in `view_course_item`.
  - templates: `course_topic.html` (Home / Finish buttons),
    `course_form_complete.html` ("Return to Course") → repointed to item 1. The
    dashboard registered-card fallback (`partials/course_card_registered.html`)
    **intentionally keeps `course_home`** — it is loop-safe there (see Section 1,
    "Dashboard cards are exempt").
  - tests: `tests/e2e/test_course_toc.py` references `course_home`.
- **`register_for_course`** currently redirects to `course_home`; after this
  change it should redirect into the player (resume redirect / first item).
- **Deadline-locked redirect in `view_course_item`.** It currently redirects a
  deadline-locked item to `course_home`. Since `course_home` becomes the resume
  redirector, leaving this pointed at it risks an **infinite loop** (locked item
  → `course_home` → resume → the same locked item → …). Redirect deadline-locked
  items to the existing **course preview page** (`course_preview`) instead: it
  already exists, is loop-free, and needs no new view. This is the minimal
  loop-safe choice for now — deadlines get a dedicated future specification, so
  this is deliberately the easiest stop-gap, not the final deadline UX.
- **~~`view_form` must record access on GET.~~ Superseded by the explicit field.**
  The plan elected the refinement below (an explicit `last_accessed_item` pointer),
  written once at the single canonical write point in `view_course_item` for both
  topics and forms. `view_form` therefore needs **no** change and never mints a
  spurious `FormProgress` on GET — the old topic/form asymmetry is irrelevant.
- **One new model field was added (the anticipated refinement).** Resume is **not**
  derived from per-item timestamps. The plan took the escape hatch this spec
  flagged: `CourseProgress` gains a nullable `last_accessed_item`
  `GenericForeignKey` (content-type FK + UUID `object_id`, mirroring
  `CohortDeadline`), stamped on every item view in `view_course_item`. `get_resume_index`
  reads it (one row + one FK resolve) and falls back to item 1 when it is null or
  no longer viewable. (The progress percentage still lives on `CourseProgress`.)
- **`course_minimal_toc.html`** is reused for the player TOC body; its status
  icons need accessible text added (Section 3) and the current-item highlight /
  `aria-current` must be threaded through (it currently has no notion of "current
  item").
- **Unify all sidebars onto the shared side-panel (Section 6).** Introduce the
  shared `<dialog>`-based side-panel component, repoint
  `base/templates/_base_interface.html` onto it, and **retire `sidebarComponent`**
  from `base/static/base/js/alpine-components.js`. The three inheriting interfaces
  must all be migrated and verified:
  - `student_interface/templates/student_interface/_course_base.html` (TOC,
    `localStorage` key `sidebar-course-toc`, bottom-sheet presentation),
  - `educator_interface/templates/educator_interface/interface.html` (educator
    nav, bottom-sheet presentation — the shared default),
  - `panel_framework/templates/panel_framework/test_interface.html`.
  Panel **contents** (`panel_framework`'s `sidebar_nav.html` + `sidebarMenuItem`,
  the course TOC body) are unchanged — only the panel shell mechanism is replaced.
- **Replace `pictureModal`** with `contentLightbox`: the `pictureModal` Alpine
  component is removed and `content_engine/templates/cotton/picture.html` is
  reworked onto a focus-managed `contentLightbox` overlay. Note: in this pass the
  lightbox was **not** migrated onto the native `<dialog>` / `<c-overlay>` shell —
  it stays an `x-show` overlay with explicit focus management (Section 6,
  Consumer 2).

## Out of scope

- **Bookmarking** items (the bookmark icon in the mobile mockup is decoration —
  omitted).
- **Migrating `c-modal`** onto the shared `<dialog>` foundation
  (`base/templates/cotton/modal.html` and its `modal_form` /
  `delete_confirmation` / `course_card` consumers). It is a form/dialog modal,
  not a sidebar; it keeps working as-is and a future pass may converge it.
- Changing the underlying progress-calculation logic or scoring strategies.
- SEO / `BreadcrumbList` structured data (low priority for authenticated
  in-course views).

## Success criteria

1. **Bare URL resolves correctly.** `/courses/{slug}/` 302-redirects an enrolled
   learner to `/courses/{slug}/{N}/` — first item with no progress, last-accessed
   item with progress. An unenrolled/anonymous visitor is sent to the preview
   page / login, never into the player, and no per-user state leaks.
2. **No Back-button loop.** From a resolved item, the browser Back button behaves
   sanely (lands on the prior item, no redirect loop, no skipped history).
3. **Old start page is gone.** `course_home.html` is no longer rendered anywhere;
   all inbound references resolve to player item URLs or the preview page.
4. **Breadcrumbs.** Desktop shows `course > [part] > current item` with the part
   crumb dropped when absent, the current item as non-linked `aria-current`,
   CSS separators, a `<nav aria-label="Breadcrumb">` + `<ol>`, and graceful
   truncation of long titles.
5. **TOC header.** Pinned header shows eyebrow + course title + determinate
   progress bar + `% complete`, all from `CourseProgress.progress_percentage`,
   with the progress value exposed to assistive tech. No `N / M` counter.
6. **TOC body.** Two-level disclosure outline; current item highlighted with
   `aria-current="page"`; the containing part auto-expands and the current item
   is scrolled into view; manual expansions persist across navigation; every
   status icon has accessible text; panel and content scroll independently.
7. **Desktop layout.** Panel always open at `lg`+ (every interface — no desktop
   collapse/toggle control), content has a comfortable max-width.
8. **Mobile.** TOC collapsed by default; opens as a modal bottom sheet (scrim,
   inert background, partial height, internal scroll); dismissible via scrim tap,
   Escape, and device Back (and optionally swipe-down) all with the same outcome
   — there is **no visible × control**; focus moves into the sheet on open and
   back to the toggle on close.
9. **Mobile header.** Unified with desktop: the same breadcrumb trail (end-
   truncated), a labelled TOC toggle with `aria-expanded` beside it, and a thin
   progress bar with `%` below it on phones; no back arrow, no bookmark icon.
10. **Page title.** `<title>` is `{Item} — {Course} — {Site}` (part optional and
    first-droppable), updates on navigation.
11. **Unified sidebars on native `<dialog>`.** The side panel + mobile bottom
    sheet are built on a native `<dialog>` (the `sidePanel` controller) sharing
    scrim/inertness/dismiss/focus, with per-interface presentation. The picture
    lightbox keeps a bespoke focus-managed `contentLightbox` overlay (not migrated
    to native `<dialog>` this pass). `sidebarComponent` and `pictureModal` are
    retired. The shared side panel works in all three host interfaces — student
    course base, educator interface, and panel_framework — each with its own
    trigger. Educator / panels are now **always docked on desktop** (no collapse)
    and dismiss the mobile bottom sheet via scrim/Escape/Back (no ×); these are
    the intentional consequences of the unified shell.
12. **Tests** verify behaviour (redirect targets, auth branching, Back-button,
    breadcrumb part-dropping, title content, modal open/dismiss/focus) — not CSS
    classes or styling, per project testing conventions.
