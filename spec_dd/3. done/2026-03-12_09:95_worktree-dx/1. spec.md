# Spec: Worktree Developer Experience

## Problem

When running multiple Claude Code instances across git worktrees, developers hit bottlenecks:

1. **Shared database:** All worktrees use the same `db` database (hardcoded in `settings_dev.py`). Concurrent migrations and test runs collide.
2. **Site/domain conflicts:** Django Sites framework resolves the current site by request domain. All worktrees serve on `127.0.0.1:8000`, so they all resolve to the same site. Content loaded via `content_save` is tied to a single Site, causing cross-worktree interference.

## Solution Overview

Give each worktree its own database, derived from the git branch name. Add a `FORCE_SITE_NAME` setting so site resolution doesn't depend on the request domain. Update dev tooling to make setup automatic.

## Detailed Design

### 1. Extract `get_current_branch()` to shared utility

**Why:** The function already exists in `freedom_ls/base/context_processors.py` with full worktree support (file-based `.git` parsing). It needs to be importable from `settings_dev.py`, which runs before Django is initialized. Moving it to a plain Python module makes it reusable without Django dependencies.

**What:**

- Create `freedom_ls/base/git_utils.py` containing:
  - `get_current_branch(base_dir: Path | None = None) -> str | None` — extracted from `context_processors.py`. Accepts an optional `base_dir` parameter. When `None`, falls back to `django.conf.settings.BASE_DIR` (lazy import inside the function body to avoid import-time Django dependency).
  - `branch_to_db_name(branch: str) -> str` — sanitizes a branch name for use as a PostgreSQL database name:
    - Lowercase
    - Replace `/`, `-`, `.` with `_`
    - Prefix with `db_`
    - Truncate to 50 characters (within PostgreSQL's 63-char limit, leaving room for `test_` prefix)
  - Keep `@functools.lru_cache(maxsize=1)` on `get_current_branch()`
- Update `freedom_ls/base/context_processors.py` to import `get_current_branch` from `git_utils` instead of defining it locally.
- Existing tests for `get_current_branch()` should continue to pass with minimal changes (updated import path).

### 2. Dynamic database name in `settings_dev.py`

**Why:** Each worktree gets its own database, eliminating contention during migrations and test runs.

**What:**

- Import `get_current_branch` and `branch_to_db_name` from `freedom_ls.base.git_utils`
- Call `get_current_branch(base_dir=BASE_DIR)` to detect the branch
- If a branch is detected, use `branch_to_db_name(branch)` as the database name
- If branch detection fails, fall back to `"db"` (current behavior)
- Django automatically prepends `test_` for the test database (e.g., `db_main` -> `test_db_main`)

**Example mappings:**

| Branch | Database name |
|---|---|
| `main` | `db_main` |
| `feature/auth-flow` | `db_feature_auth_flow` |
| `hotfix-2.1` | `db_hotfix_2_1` |
| (detached/unknown) | `db` |

### 3. `FORCE_SITE_NAME` setting

**Why:** With per-branch databases, each branch has its own Site records. All worktrees serve on `127.0.0.1:8000`, but since each has its own database, domain-based resolution works fine in practice. However, `FORCE_SITE_NAME` provides an explicit, reliable override that doesn't depend on matching domains — it ensures the correct site is always selected regardless of how the dev server is accessed.

**What:**

- Modify `get_cached_site(request)` in `freedom_ls/site_aware_models/models.py`:
  - Before calling `get_current_site(request)`, check for `settings.FORCE_SITE_NAME`
  - If set, do `Site.objects.get(name=settings.FORCE_SITE_NAME)` and cache on the request
  - If not set (or the setting doesn't exist), fall back to existing `get_current_site(request)` behavior
- Add `FORCE_SITE_NAME = "DemoDev"` to `settings_dev.py`
- Management commands like `content_save` already accept site name as an argument — no changes needed there.

### 4. `dev_db_init.sh` script

**Why:** Developers need a simple way to create the per-branch database in the shared PostgreSQL container.

**What:**

- New script at project root: `dev_db_init.sh`
- Uses the same branch-name sanitization logic as `settings_dev.py` (implemented in shell: lowercase, replace `/-.` with `_`, prefix `db_`, truncate to 50 chars)
- Connects to PostgreSQL via `psql` on host port 6543 (the existing docker-compose exposed port)
- Creates the branch database and grants privileges to `pguser`
- Creates the test database (`test_` prefix) and grants privileges
- Idempotent: uses `CREATE DATABASE IF NOT EXISTS` pattern (check `pg_database` catalog, skip if exists)
- Does NOT start or manage the docker container — assumes it's already running

### 5. `dev_db_delete.sh` script

**Why:** Clean up databases for branches that are no longer needed.

**What:**

- New script at project root: `dev_db_delete.sh`
- Same branch detection and sanitization as `dev_db_init.sh`
- Drops both the branch database and its test database
- No confirmation prompt — data is trivially recreatable via `install_dev.sh`
- Terminates active connections before dropping

### 6. Update `install_dev.sh`

**Why:** New developers (or new worktrees) should get a fully working environment with one command.

**What:**

Current script only runs `uv sync`, `npm i`, and adds Playwright MCP. Extend it to also:

1. Call `./dev_db_init.sh` (create per-branch databases)
2. Run `uv run manage.py migrate` (apply migrations)
3. Run `uv run manage.py create_demo_data` (create all demo sites/users — left as-is, not simplified)
4. Run `uv run manage.py content_save ./demo_content DemoDev` (load demo content)

## Out of Scope

- **Port conflicts:** Multiple worktrees binding port 8000 is not addressed. Developers can use `runserver 0.0.0.0:800X` when running multiple servers simultaneously. Typically only one worktree runs a dev server at a time.
- **Simplifying `create_demo_data`:** The command still creates all 5 demo sites. With per-branch databases the overhead is negligible and not worth optimizing separately.

## Decisions Made

| Decision | Rationale |
|---|---|
| Reuse existing `get_current_branch()` (file-based parsing) instead of subprocess | Already tested, handles worktrees, no subprocess overhead. Avoid reinventing. |
| Put shared utility in `freedom_ls/base/git_utils.py` | Projects built on top of freedom_ls can import and use it. Plain Python module with no Django imports at module level. |
| `base_dir` parameter with lazy `settings.BASE_DIR` fallback | Allows use from `settings_dev.py` (before Django init) and from Django code (no arg needed). |
| Keep `lru_cache` on `get_current_branch()` | Harmless for settings (called once), beneficial for context processor (called per request). Cache key includes `base_dir` so works correctly. |
| Fall back to `"db"` if branch detection fails | Preserves current behavior for non-git or detached HEAD scenarios. |
| No confirmation prompt on `dev_db_delete.sh` | Dev data is trivially recreatable. Minimizes friction. |
| Leave `create_demo_data` unchanged | Not slow enough to matter. Separate concern from worktree DX. |

## Success Criteria

1. A developer can run `install_dev.sh` in a new worktree and get a fully working environment with its own database
2. Two worktrees can run `pytest` simultaneously without database conflicts
3. Two worktrees can run migrations simultaneously without conflicts
4. `get_cached_site()` returns the correct site via `FORCE_SITE_NAME` regardless of request domain
5. Existing tests pass with the refactored `get_current_branch()` import
6. `dev_db_delete.sh` cleanly removes per-branch databases
7. The `main` branch database is named `db_main`, not `db` (clean break from hardcoded name)
