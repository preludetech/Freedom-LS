When running claude instances on multiple worktrees, the dev experience is not great. There is a lot of waiting due to:

- All worktrees making use of the same development database (specified in dev_db). This means there is a bottleneck around testing
- Sites being associated with different urls, stored in the database. By default `python manage.py create_demo_data` sets up a specific site. When we call `content_save` on the demo content then it is loaded into the database. Content can only be associated with one Site at a time.

# Proposed solution

## settings_dev.py — dynamic database name

Derive the database name from the current git branch using `git branch --show-current` (via `subprocess`). This command handles worktrees transparently — no special `.git` file parsing needed.

- Sanitize branch names for PostgreSQL: replace `/`, `-`, `.` with `_`, lowercase, truncate to 50 chars
- Prefix with `db_` so the name always starts with a letter (e.g. `main` → `db_main`, `feature/auth-flow` → `db_feature_auth_flow`)
- Fall back to `db` if branch detection fails (not in a git repo, detached HEAD, etc.)
- Django automatically prepends `test_` for the test database, so `db_main` → `test_db_main`

## dev_db_init.sh

New script. Creates a database and test database for the current branch in the existing postgres container (port 6543).

Connect via `psql` on the host (the docker composition exposes port 6543). Use the same branch-name sanitization as `settings_dev.py`.

```
PGPASSWORD=password psql -h 127.0.0.1 -p 6543 -U pguser -d postgres -c "CREATE DATABASE ${DB_NAME};"
PGPASSWORD=password psql -h 127.0.0.1 -p 6543 -U pguser -d postgres -c "GRANT ALL PRIVILEGES ON DATABASE ${DB_NAME} TO pguser;"

PGPASSWORD=password psql -h 127.0.0.1 -p 6543 -U pguser -d postgres -c "CREATE DATABASE test_${DB_NAME};"
PGPASSWORD=password psql -h 127.0.0.1 -p 6543 -U pguser -d postgres -c "GRANT ALL PRIVILEGES ON DATABASE test_${DB_NAME} TO pguser;"
```

If the docker composition in dev_db/ is running (even in a separate branch) then `dev_db_init.sh` should just work. And then `pytest` should connect correctly.

We don't want to have to run a separate database composition per branch. We just want to make multiple databases in the same postgres container.

Also create a `dev_db_delete.sh` script to completely remove the databases associated with the current branch.

## install_dev.sh

Update this script to also set up the branch database and load demo data:

- Call `dev_db_init.sh`
- Call `python manage.py migrate`
- Call `python manage.py create_demo_data`
- Call `python manage.py content_save ./demo_content DemoDev`

## Add FORCE_SITE_NAME to settings_dev

Currently the current site is derived from the request url via `get_cached_site()` in `freedom_ls/site_aware_models/models.py`.

Modify `get_cached_site()` to check for a `FORCE_SITE_NAME` setting first. If set, look up the site by name instead of by request domain. This means all worktrees can serve on `127.0.0.1:8000` without conflicting — each has its own database with its own site record.

In `settings_dev.py` add `FORCE_SITE_NAME = "DemoDev"`

Management commands like `content_save` already take the site name as an argument, so no changes needed there.

# Open questions

1. **Port conflicts:** Multiple worktrees running `runserver` will all try to bind port 8000. Should we address this (e.g., derive port from branch name, or just document using `runserver 800X`)? Or is this out of scope since typically only one worktree runs a dev server at a time?
2. **`create_demo_data` scope:** Currently creates 5 sites (Demo, DemoDev, Bloom, etc.). With per-branch DBs, each branch only needs DemoDev. Should we simplify this for dev, or leave as-is?
