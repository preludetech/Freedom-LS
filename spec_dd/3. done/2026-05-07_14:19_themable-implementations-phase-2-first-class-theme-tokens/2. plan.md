# Implementation Plan: First Class theme — Tier 1 tokens (Phase 2 of 5)

This plan turns `1. spec.md` (in this folder) into shippable code. The phase delivers a complete second theme using only Tier-1 token redeclarations: a new theme directory plus a small wiring change to `tailwind.input.css` so the active theme's `theme.css` cascades on top of the baseline.

For the holistic five-phase context, see `../themable-implementations-master-decomposed-into-phases/2. plan.md`.

**Phase 1 is a hard prerequisite** — this phase assumes the theme resolver, the default theme on disk, the rewritten `tailwind.input.css`, and the renamed token classes are all in place.

---

## Architectural principles inherited from Phase 1

1. **Themes are sparse.** A theme contains only what it overrides. The First Class theme contains only `theme.css` + `theme.md` — no templates, no component classes.
2. **Defaults stay in the main project.** Component classes stay in `tailwind.components.css`; nothing is migrated wholesale.
3. **This phase ends in a green build.** No half-implemented features.

---

## Pre-existing functionality to reuse (DRY)

- The Phase-1 theme resolver (`freedom_ls/base/theming.py::resolve_theme_dir` and `configure_theme`) — picks up `first_class` automatically once the directory exists in `freedom_ls/themes/`.
- Phase-1 `STATICFILES_DIRS` prepend — surfaces `static/themes/first_class/theme.css` once the directory exists.
- The Phase-1 token role contract — `first_class/theme.css` redeclares the same `--color-*` / radius / font surface, just with brand values.

---

## End-state checklist

- `freedom_ls/themes/first_class/static/themes/first_class/theme.css` exists with First Class brand tokens.
- Setting `FLS_THEME=first_class` and rebuilding Tailwind produces a visually rebranded site (indigo/teal/orange brand, DM Sans / Outfit fonts, brand radii).
- `FLS_THEME=default` still produces the unaltered baseline.
- Phase 1 deliverables remain green.

## Task 1 — Create `first_class` theme directory

```
freedom_ls/themes/first_class/
    static/
        themes/first_class/
            theme.css
    theme.md
```

`theme.css` redeclares only the values that differ from the default theme.
It does **not** re-declare the `--radius-*` / `--font-*` / `--color-focus-ring`
aliases — those live in the default theme's `@theme inline` block (Phase 1)
and remain in effect under the cascade. Overriding the FLS-namespaced
`--fls-*` and `--color-*` tokens is enough:

```css
@theme {
    /* Brand */
    --color-primary: #283593;        /* Deep Indigo */
    --color-on-primary: #FFFFFF;
    --color-secondary: #00CEC9;      /* Electric Teal */
    --color-on-secondary: #1A1A2E;
    --color-accent: #FF6B35;         /* Altitude Orange */
    --color-on-accent: #FFFFFF;

    /* Status */
    --color-success: #38A169;
    --color-on-success: #FFFFFF;
    --color-warning: #D69E2E;
    --color-on-warning: #1A1A2E;
    --color-error: #E53E3E;
    --color-on-error: #FFFFFF;
    --color-info: #3182CE;
    --color-on-info: #FFFFFF;

    /* Surfaces */
    --color-surface: #F8F9FC;
    --color-surface-2: #EDF2F7;
    --color-on-surface: #1A1A2E;
    --color-border: #E2E8F0;
    --color-muted: #718096;

    /* Shape — bumped one step. `pill` stays at the inherited 9999px. */
    --fls-radius-sm: 0.5rem;
    --fls-radius-md: 0.75rem;
    --fls-radius-lg: 1rem;

    /* Type — DM Sans body, Outfit display. */
    --fls-font-sans: "DM Sans", system-ui, sans-serif;
    --fls-font-display: "Outfit", system-ui, sans-serif;
}
```

`theme.md`: short manifest noting the brand source and that this is the canary for all three tiers.

## Task 2 — Wire active-theme `theme.css` import into `tailwind.input.css`

The dev `tailwind.input.css` must import the active theme's `theme.css` after `tailwind.components.css` so future Tier-2 overrides (Phase 3) cascade correctly. The `@import` URL has to be deterministic at build time (Tailwind has no env-var interpolation), so the active theme is resolved through Django and written to a small generated file that `tailwind.input.css` imports unconditionally.

**Files:**

1. `freedom_ls/base/management/commands/write_active_theme_css.py` — new Django management command.
   - Reads `settings.FLS_THEME` and `settings.RESOLVED_THEME_DIR`.
   - Builds the path to the active theme's `theme.css`
     (`<resolved>/static/themes/<slug>/theme.css`); raises `CommandError`
     if the file is missing.
   - Writes `<BASE_DIR>/tailwind.active_theme.css` with a single
     `@import "./freedom_ls/themes/<slug>/static/themes/<slug>/theme.css";`
     line. Uses a `BASE_DIR`-relative path when possible; falls back to an
     absolute path for themes that live outside `BASE_DIR`.

2. `package.json` — wire the command into both build scripts:

   ```json
   "scripts": {
     "_write_active_theme": "uv run manage.py write_active_theme_css",
     "tailwind_watch": "npm run _write_active_theme && npx @tailwindcss/cli -i ./tailwind.input.css -o ./static/vendor/tailwind.output.css --watch",
     "tailwind_build": "npm run _write_active_theme && npx @tailwindcss/cli -i ./tailwind.input.css -o ./static/vendor/tailwind.output.css"
   }
   ```

3. `tailwind.input.css` — append the import after `tailwind.components.css`:

   ```css
   /* Active theme — generated each build by `manage.py write_active_theme_css`,
      keyed on the `FLS_THEME` env var. When FLS_THEME=default this re-imports
      the same baseline (no-op on cascade); for any other slug the active
      theme's token redeclarations win. */
   @import "./tailwind.active_theme.css";
   ```

4. `.gitignore` — exclude `tailwind.active_theme.css`. The file is regenerated each build.

This mechanism keeps the Tailwind build and the Django runtime in lockstep
(both go through `settings.FLS_THEME`) and works identically for downstream
consumers — they get the same `package.json` script and the same
`tailwind.input.css` import.
