# Spec: First Class theme — Tier 1 tokens (Phase 2 of 5)

## Why

This is **Phase 2** of the multi-phase Themable Freedom LS effort. It validates the Tier-1 architecture (design tokens) shipped in Phase 1 by adding a second theme — `first_class` — that demonstrates a **complete brand rebrand using only token redeclarations**. No templates, no component-class changes, no Python.

After Phase 2, switching `FLS_THEME=first_class` and rebuilding Tailwind produces a visually rebranded site (indigo/teal/orange brand, DM Sans / Outfit fonts, brand radii). Switching back to `FLS_THEME=default` produces the unaltered baseline.

This phase is the canary that proves the Tier-1 promise from the master spec: **a downstream project can rebrand FLS by writing one `theme.css` file and pointing `FLS_THEME` at their theme directory — no template edits, no Python.**

For the holistic five-phase context, see `../themable-implementations-master-decomposed-into-phases/`.

## Scope of this phase

**In:**

- A new `freedom_ls/themes/first_class/` theme directory containing `theme.css` (Tier-1 token overrides only) and `theme.md` (manifest).
- A small mechanism to import the active-theme `theme.css` into `tailwind.input.css` after `tailwind.components.css` so the cascade works (this is the foundation Phase 3 builds on).
- Verification that switching between `default` and `first_class` produces the expected visual diff and back.

**Out of this phase (covered later):**

- Tier-2 component-class re-opens (rounder buttons, flatter chips) — Phase 3.
- Tier-3 cotton-template overrides (header-button, chip, button) — Phase 4.
- Documentation updates that reference `first_class` — Phase 5.

**Phase 1 is a hard prerequisite.** This phase assumes the resolver, the default theme, the rewritten `tailwind.input.css`, and the renamed token classes are all in place.

## Goals & success criteria

1. **`FLS_THEME=first_class npm run tailwind_build` exits cleanly** and the dev server, when started with `FLS_THEME=first_class`, shows:
   - Header / primary CTAs in deep indigo (`#283593`).
   - Body in DM Sans, headings in Outfit (visibly different from the default system font).
   - Cards / inputs slightly rounder (radii bumped one step).
   - Surfaces a faintly cooler off-white (`#F8F9FC`) than the default pure white.
2. **`FLS_THEME=default npm run tailwind_build` still produces the unaltered baseline** — Phase 1 deliverables are not regressed.
3. **A downstream rebrand needs no template edits and no Python edits** — the entire rebrand is achieved by writing one `theme.css` and pointing `FLS_THEME` at it. (The FLS dev repo *itself* may add small build-side plumbing — e.g. a management command that resolves the active theme path for Tailwind — to make the active-theme import work; see §Active-theme import wiring below. That plumbing is invisible to downstream consumers.)
4. **Email colour parser** picks up the first_class brand colours when `FLS_THEME=first_class` (`EMAIL_COLOR_PRIMARY` returns `#283593`, etc.).
5. **Both themes' static `theme.css` files remain individually resolvable** via `/static/themes/<slug>/theme.css` regardless of which theme is active.

## Architecture used by this phase

### Tier 1 — design tokens (recap from Phase 1)

Token contract lives at `:root`. Tailwind's `@theme inline` block maps the role-named tokens onto Tailwind colour utilities, so `bg-primary` etc. compile to `var(--color-primary)` and follow the cascade. A static rebrand (downstream redeclares `:root`) and a future runtime swap (set `data-theme`) work the same way without further code changes (see master `research_tailwind_v4_theming.md` §2).

The default theme's `theme.css` is always imported as the baseline. A non-default theme's `theme.css` is imported afterwards and re-declares only the tokens it wants to change. Cascade order means the later declaration wins.

### First Class brand tokens

From `../themable-implementations-master-decomposed-into-phases/first-class-theme.md` ("Modern Altitude"):

- `primary` = Deep Indigo `#283593`, `on-primary` = `#FFFFFF`
- `secondary` = Electric Teal `#00CEC9`, `on-secondary` = Cockpit Dark `#1A1A2E`
- `accent` = Altitude Orange `#FF6B35`, `on-accent` = `#FFFFFF`
- `success` = `#38A169`, `warning` = `#D69E2E`, `error` = `#E53E3E`, `info` = `#3182CE` (all with `#FFFFFF` `on-*` pairs except `warning` which uses Cockpit Dark)
- `surface` = Stratosphere `#F8F9FC`, `surface-2` = `#EDF2F7`, `on-surface` = Cockpit Dark `#1A1A2E`
- `border` = `#E2E8F0`, `muted` = `#718096`
- `focus-ring` = alias of `primary` (live)
- `--fls-font-sans` = `"DM Sans", system-ui, sans-serif`
- `--fls-font-display` = `"Outfit", system-ui, sans-serif`
- Radii bumped one step (`--fls-radius-{sm,md,lg}`, e.g. `--fls-radius-md` ≈ `0.75rem`) to match the brand's softer card treatment. `--fls-radius-pill` is unchanged (pill is by definition `9999px` and does not bump with the rest of the scale).

### Active-theme import wiring

Phase 1 left `tailwind.input.css` with the default theme's `theme.css` imported as the always-on baseline, then `tailwind.components.css`. This phase adds the active-theme import so non-default themes win on cascade.

The mechanism: a small Django management command,
`manage.py write_active_theme_css`, resolves the active theme through Django
settings (`FLS_THEME` + `FLS_THEMES_DIRS` + `RESOLVED_THEME_DIR`) and writes a
single-line `tailwind.active_theme.css` containing
`@import "<active-theme>/theme.css";`. The `tailwind_build` and
`tailwind_watch` `npm` scripts run the command immediately before invoking
Tailwind, and `tailwind.input.css` `@import`s the generated file *after*
`tailwind.components.css` so the active theme wins on cascade. The generated
file is gitignored.

When `FLS_THEME=default`, the generated import re-imports the baseline file
(no-op on cascade). For any other slug, the active theme's redeclarations
win.

This is preferred over a static "hard-code two import lines" approach
because it (a) ensures the Tailwind build and the Django runtime cannot
disagree about which file is the active theme — both resolve through the
same settings — and (b) extends to downstream projects without per-theme
edits to `tailwind.input.css`. Phase 5 documents the same wiring for
downstream consumers.

## Out of scope (deferred)

- Any First Class component-class re-opens (rounder buttons, flatter chips, bolder outline) — Phase 3.
- Any First Class cotton-template overrides (`header-button`, `chip`, `button`) — Phase 4.
- IBM Plex Mono for code blocks, Success/Warning/Error *Light* tints, full type scale — noted in `first-class-theme.md` and may be added later if the canary needs them, but are not requirements for this phase.
- Per-site theming on a single deployment, full 50–950 colour ramps, etc. — deferred per master spec.
