# Frontend QA: Phase 2 — First Class theme (Tier 1 tokens)

## What we're verifying

1. Switching `FLS_THEME=first_class` rebrands every page via Tier-1 tokens only — colours, fonts, radii. No structural change, no component-class shape change, no template change (those land in later phases).
2. Switching back to `FLS_THEME=default` produces the unchanged Phase-1 baseline.
3. Email colour parser picks up first_class brand values when active.
4. Both themes' static `theme.css` files are individually resolvable regardless of which theme is active.

We use the **DemoDev** site for all test data.

## Test 2 — `first_class` theme: end-to-end Tier-1 rebrand

### 2.1 Restart with the canary theme

Stop the server. Then:

```bash
FLS_THEME=first_class npm run tailwind_build
FLS_THEME=first_class uv run python manage.py runserver $PORT
```

Expected: server starts cleanly. Tailwind output regenerated with First Class tokens.

### 2.2 Walk the major pages

Visit the same URL list as Test 1:

- `/` — student home / course list.
- `/courses/`.
- A course home page.
- A course topic page.
- A course form page.
- Educator interface (if applicable).
- Account pages.
- Login (sign out then revisit `/`).

For each, take a screenshot via Playwright MCP for the QA artifact.

**Tier 1 (tokens) — should be visible everywhere:**

- Primary brand colour: deep indigo (`#283593`) instead of ocean blue. Header bar, primary CTAs, links — all indigo.
- Secondary colour: electric teal (`#00CEC9`) on any element using `bg-secondary` / `text-secondary`.
- Accent: altitude orange (`#FF6B35`) — on whatever `bg-accent` / `text-accent` references exist.
- Surface: stratosphere off-white (`#F8F9FC`) — body background should look very faintly cooler than pure white.
- Body text: cockpit dark (`#1A1A2E`) — slightly bluer-black than the default.
- Border colour: lighter slate (`#E2E8F0`).
- Body font is **DM Sans** (visibly different from system sans). Headings (`h1`–`h4`) are **Outfit**.
- Radii are bumped one step — cards and inputs read slightly rounder.

**Things that must NOT have changed (these belong to Phase 3 / Phase 4):**

- Button shape — `btn-primary` should still have the same radius / shadow as default; the *colour* changes (indigo) but the *shape* is unchanged this phase.
- Chip elevation — chips still have whatever shadow they had under default; flatness is Phase 3.
- Cotton component templates (header buttons, chips, buttons) — same markup as default; template overrides are Phase 4.
- Layout / structure — same elements in the same places.
- All routes that worked under default still work (no template-resolution failure).

### 2.3 Quick interactive flow

Sign out, then sign back in, then visit a course and complete a form (any short activity in the demo content). The point is to exercise success / error / messages partials under the canary theme so we see the semantic-role rebrand applied to alerts.

## Test 4 — Email colour fallbacks under both themes

```bash
FLS_THEME=default uv run python manage.py shell -c "from django.conf import settings; print(settings.EMAIL_COLOR_PRIMARY, settings.EMAIL_COLOR_FOREGROUND)"
FLS_THEME=first_class uv run python manage.py shell -c "from django.conf import settings; print(settings.EMAIL_COLOR_PRIMARY, settings.EMAIL_COLOR_FOREGROUND)"
```

Expected:

- default → `#2B6CB0 #1A2332`
- first_class → `#283593 #1A1A2E`

If a key returns the hard-coded fallback (e.g. `#2B6CB0` under `first_class`), `parse_tailwind_colors` did not find the new key in the resolved theme.css — investigate the path or token name.

---

## Test 5 — Static asset isolation

Confirm that brand assets do not collide between themes. With each theme active, request:

- `http://127.0.0.1:$PORT/static/themes/default/theme.css`
- `http://127.0.0.1:$PORT/static/themes/first_class/theme.css`

Expected: 200 OK on both regardless of which theme is active (because both `static/themes/<slug>/` paths are served via `STATICFILES_DIRS` once collected). The namespacing prevents collision.
