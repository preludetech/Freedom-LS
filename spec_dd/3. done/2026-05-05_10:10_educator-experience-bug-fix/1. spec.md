# Specification: Educator Experience Bug Fix

This spec addresses two related defects in the educator-facing data tables. Both are frontend / template-layer issues ŌĆö no model, view, URL, or business-logic changes are required. Detailed root-cause investigation lives in `research_recursive_table.md` and `research_pagination.md`.

## Problem 1: Recursive table nesting on filter/sort

### Why it matters

Educators rely on the cohort details page (`/educator/cohorts/<uuid>/__tabs/details`) and several other panel-based views to inspect students, registrations, and progress. Filtering or sorting any of the five affected tables visibly breaks the layout ŌĆö a duplicate panel header, search bar, and surrounding chrome get nested inside the table div on every interaction. After two clicks the page is unusable until reloaded.

Because the bug is rooted in the shared `DataTablePanel` base class, every consumer of that base class is affected, not just the originally reported `CohortStudentsPanel`.

### Root cause (summary)

`DataTablePanel.render()` always returns the full panel container (`section.surface` with title, actions, and search bar) regardless of request type. HTMX filter/sort requests target the inner `#table-<panel_name>` div with `hx-swap="outerHTML"`, so the entire panel chrome ends up replacing the table div, producing the nested-panel artefact. The list view at `/educator/users` is unaffected because it renders `DataTable` directly without a panel wrapper.

The correct pattern already exists in `CohortCourseProgressPanel.render()` (`educator_interface/views.py` ~lines 706ŌĆō710): detect `HX-Request` and return only `get_content(...)` for HTMX, fall back to the wrapped `super().render(...)` for full-page loads.

### Decision: fix at the base class

Override `render()` on `DataTablePanel` itself rather than on each of the five concrete subclasses. Reasoning:

- One place to fix means one place to regress later. Five subclasses with identical override boilerplate would invite divergence.
- The bug is a property of "data-table inside a panel container with an inner HTMX target" ŌĆö that property lives on the base class, so the fix belongs there.
- `CohortCourseProgressPanel` is not a `DataTablePanel` subclass (it has its own custom paginators), so its existing override stays untouched.

### Scope

Fix all five panels in one change by editing the base class:

1. `CohortStudentsPanel` ŌĆö `/educator/cohorts/<uuid>/__tabs/details` (the reported bug)
2. `CourseRegistrationsPanel` ŌĆö same details tab
3. `UserCohortsPanel` ŌĆö `/educator/users/<user_pk>`
4. `CourseCohortRegistrationsPanel` ŌĆö `/educator/courses/<uuid>`
5. `CourseStudentRegistrationsPanel` ŌĆö `/educator/courses/<uuid>`

### Approach

In `freedom_ls/panel_framework/panels.py`, change `DataTablePanel.render()` to:

- Check `request.headers.get("HX-Request") == "true"`.
- If HTMX: return `self.get_content(request, base_url=base_url, panel_name=panel_name)` directly (no panel wrapper).
- Otherwise: return `super().render(...)` so initial page loads still get the full surface, header, and actions.

This mirrors the existing `CohortCourseProgressPanel` override exactly.

### Non-regression requirements

- `/educator/users`, `/educator/cohorts`, `/educator/courses` (list views using `DataTable` directly) must continue to filter/sort cleanly. They do not flow through `DataTablePanel.render()`, so no behaviour change is expected.
- Initial full-page loads of any panel must still render with the panel surface, header, and action buttons (edit, etc.).
- HTMX filter/sort/search/pagination on each of the five affected panels must update the inner table only, with no nested chrome.

### Success criteria

- On `/educator/cohorts/<uuid>/__tabs/details`, sorting any column or typing in the search box updates the table without nesting a second panel.
- The same is true for the four other affected panels.
- Initial page loads of every affected panel still show their headers, search bars, and action buttons.
- A regression test (Playwright or pytest+HTMX assertion) covers the HTMX response shape: filter request returns just the table content, full-page request returns the wrapped panel.

## Problem 2: Inconsistent pagination across data tables

### Why it matters

Pagination has two distinct visual styles in the educator interface:

- **Numbered buttons** (First / Previous / 1 2 ŌĆ” N / Next / Last) on `/educator/users`, `/educator/cohorts`, `/educator/courses`, and all panel data tables ŌĆö rendered inline inside `cotton/data-table.html`.
- **Minimal "┬½ Prev / Next ┬╗"** with an "Items XŌĆōY of Z" range indicator ŌĆö used only by the course progress panel (`course_progress_panel.html`), which has two independent paginators (course items and students).

Educators see different pagination shapes on adjacent tabs of the same cohort, which is jarring and harder to learn. The numbered-button style is the desired standard.

A second problem is duplication: the numbered-button markup is roughly 90 lines of conditional template logic embedded inside `data-table.html`. Copy-pasting it into the course progress panel would be the wrong fix ŌĆö the right fix is to extract it once.

### Decision: extract a single `<c-pagination>` cotton component

Create a reusable component at `freedom_ls/base/templates/cotton/pagination.html`. Both `data-table.html` and `course_progress_panel.html` will delegate to it.

Reasoning:

- One source of truth for pagination markup. Future styling tweaks happen in one file.
- `course_progress_panel.html` has two independent paginators sharing one URL ŌĆö a component that accepts `extra_params` handles this case naturally.
- Cotton components are the existing conventional reuse mechanism in this codebase (`c-data-table`, `c-button`, `c-button-group`, etc.).

### Component inputs

The component must cover all current call sites, so it accepts:

- `page_obj` (required) ŌĆö a Django `Page` object.
- `base_url` (required) ŌĆö base URL for pagination links.
- `table_id` (required) ŌĆö HTMX swap target id.
- `sort_by`, `sort_order`, `search_query` (optional) ŌĆö preserved across pagination clicks for data tables.
- `extra_params` (optional) ŌĆö additional query parameters for cases where multiple paginators share a URL (course progress panel: e.g. when paginating course items, the student page number must be preserved, and vice versa).

### Approach

1. Create `freedom_ls/base/templates/cotton/pagination.html` containing the existing numbered-button markup from `data-table.html` (lines ~106ŌĆō197). Preserve the responsive split: prev/next + "Page X of Y" on mobile, full numbered buttons on desktop. HTMX-driven via `hx-get` / `hx-target` / `hx-swap`, with a fallback `href` for progressive enhancement.
2. Refactor `cotton/data-table.html` to replace its inline pagination block with a single `<c-pagination ... />` call. No visual change for users.
3. Replace the two prev/next blocks in `course_progress_panel.html` with two `<c-pagination ... />` calls ŌĆö one for course items, one for students. Each call passes the other paginator's current page number via `extra_params` so independence is preserved.

### Non-regression requirements

- All existing data tables (`/educator/users`, `/educator/cohorts`, `/educator/courses`, and the five panel tables) keep their current sort, search, filter, and HTMX behaviour. Visual output of `data-table.html` pagination must be unchanged.
- Query parameters (`sort`, `order`, `search`, `registration`, `col_page`, `page`) are preserved across pagination clicks.
- On the course progress panel, clicking a page button on the course-items paginator must not reset the students paginator, and vice versa.
- Initial page loads (no JS / no HTMX) still navigate via the fallback `href`.

### Success criteria

- A single `<c-pagination>` cotton component exists and is the only place that renders pagination markup in the educator interface.
- The course progress panel uses the numbered-button style consistent with all other educator tables.
- All affected pages render and behave identically to before, except the course progress panel now shows numbered buttons instead of prev/next.
- Independent paginator state on the course progress panel is verified (clicking page 2 of course items leaves student pagination on its current page, and vice versa).

## Out of scope

- Student-interface form page navigation (`c-form-page-link`) ŌĆö different use case, not table pagination.
- Backend changes (no view, model, or URL changes needed).
- Page-size changes or other styling tweaks not required to address these two bugs.
- Changes to `CohortCourseProgressPanel.render()` HTMX detection ŌĆö it already implements the correct pattern.
