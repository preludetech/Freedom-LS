# Factory Boy Implementation — Specification

## Problem Statement

The FLS test suite (~4,135 lines across 27 test files) relies heavily on manual `Model.objects.create()` calls and bespoke pytest fixtures to build test data. This creates two problems:

1. **Repetition**: The same object-creation logic is duplicated across conftest files and individual tests (~340 `.create()` calls). When a model's required fields change, every test that creates that model must be updated manually.

2. **Fragile hierarchies**: Creating a valid object graph (e.g. a `FormProgress` requires a `User`, a `Form`, a `FormPage`, a `FormQuestion`, and a `Site`) is error-prone. If the definition of "valid" changes (new required field, new relationship), tests may silently construct invalid data rather than surfacing the bug.

Factory Boy solves both problems by centralising object-creation logic into declarative factory classes. Faker provides realistic, randomised default values so tests don't share magic strings.

## Goals

- Replace all manual ORM object creation in tests with factory_boy factories
- Replace all object-creation pytest fixtures with direct factory calls
- Update QA helper management commands to use factories
- Update documentation and Claude skills/commands so factory_boy is the standard going forward

## Decisions

### Factory location

**Decision**: One `factories.py` file per app, co-located with the app's models.

```
freedom_ls/<app_name>/factories.py
```

**Why**: Keeps factory definitions close to the models they create. When a model changes, the factory is right there to update. Avoids a single monolithic factories file that grows unwieldy.

### Site-awareness

**Decision**: Factories pull the site from the thread-local request context (via `get_current_site()`), matching how the models themselves behave. No `SiteFactory` is needed.

**Why**: The `mock_site_context` fixture already sets up the thread-local site for every test that touches site-aware models. Having factories mirror this behaviour keeps them consistent with production code. Tests that need a specific site can still pass `site=some_site` explicitly.

**Implication**: Every test that calls a factory for a site-aware model must have `mock_site_context` active. This is already the case for all existing tests.

### Fixture removal

**Decision**: Remove conftest.py fixtures that exist solely to create model instances (`user`, `form`, `activity`, etc.). Tests call factories directly. Non-object fixtures (`mock_site_context`, `site_aware_request`, `logged_in_page`, `live_server_site`, `add_item_to_collection`) remain.

**Why**: Factory calls are explicit and self-documenting at the call site. A test that says `user = UserFactory(is_staff=True)` is clearer than one that injects a `user` fixture whose configuration lives in a different file. Removing wrapper fixtures eliminates a layer of indirection.

### mock_site_context remains explicit

**Decision**: `mock_site_context` stays as an explicitly-requested fixture (not `autouse=True`).

**Why**: Explicit is better than implicit. Tests that don't touch the database don't need site context, and requiring it in the test signature makes dependencies visible.

### Migration scope

**Decision**: Migrate all existing tests to use factories (not just new tests).

**Why**: A partial migration leaves two parallel systems for creating test data, making the codebase harder to understand and maintain. A complete migration ensures consistency and means there's one way to create test objects.

## Factory Design

### Base pattern

All factories for site-aware models follow this pattern:

```python
import factory
from django.contrib.sites.shortcuts import get_current_site
from freedom_ls.site_aware_models.middleware import _thread_local


class SiteAwareFactory(factory.django.DjangoModelFactory):
    """Base factory for all site-aware models."""

    site = factory.LazyFunction(
        lambda: getattr(_thread_local, "request", None)
        and get_current_site(_thread_local.request)
    )
```

App-specific factories inherit from this base.

### SubFactory usage

Factories use `factory.SubFactory` for required ForeignKey relationships. This means calling `StudentFactory()` automatically creates a `User` (via `UserFactory`) unless one is passed explicitly.

Example:
```python
# Creates a Student AND a User
student = StudentFactory()

# Creates a Student linked to an existing User
student = StudentFactory(user=existing_user)
```

### Sequence and Faker usage

- **Sequences** for fields with uniqueness constraints (slugs, emails, names)
- **Faker** for realistic but non-unique text (descriptions, question text, content)
- **LazyAttribute** to derive one field from another (slug from title)

### Traits

Use factory_boy `Trait` for common variations:

```python
class UserFactory(SiteAwareFactory):
    class Meta:
        model = User

    class Params:
        staff = factory.Trait(is_staff=True)
        superuser = factory.Trait(is_staff=True, is_superuser=True)
```

Usage: `UserFactory(staff=True)`, `UserFactory(superuser=True)`

### GenericForeignKey handling

`ContentCollectionItem` and deadline models use Django's `GenericForeignKey`. Factories for these models accept the related object directly and resolve `content_type` and `object_id` via `LazyAttribute`:

```python
class ContentCollectionItemFactory(SiteAwareFactory):
    collection_object = None  # Pass the parent (Course/CoursePart)
    child_object = None       # Pass the child (Topic/Activity/Form/etc.)

    collection_content_type = factory.LazyAttribute(
        lambda o: ContentType.objects.get_for_model(o.collection_object)
    )
    collection_object_id = factory.LazyAttribute(lambda o: o.collection_object.pk)
    # ... same for child
```
