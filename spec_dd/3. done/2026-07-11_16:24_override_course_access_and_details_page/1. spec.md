# Spec: Course details polish + system-wide visibility/access overrides

## Summary

Two related, independent pieces of work that both make in-development courses
look and demo better. They share the `course_access` / content pipeline surface
area but are otherwise decoupled and could ship separately.

1. **Problem 1 — TOC-in-development flag.** A new course frontmatter flag,
   `table_of_contents_in_development`, that hides the three table-of-contents
   surfaces on the course detail page so a course still being built does not
   render empty shells ("0 lessons", a heading with nothing under it).
2. **Problem 2 — global visibility/access overrides.** Two dev/staging-only
   global settings that let every course be previewed as fully visible and/or
   freely accessible, without editing the database, so `coming_soon` / `hidden`
   / access-gated courses can be demoed live.

Both problems are specced together here because they land in adjacent code and
share this worktree; they carry no runtime dependency on each other and can be
implemented and reviewed as two independent slices.

---

## Problem 1 — `table_of_contents_in_development` course flag

### Why this matters

When a course has no content yet (still being authored), the course detail page
renders empty shells that look broken in a demo:

- The **"Course content"** `<section>` renders its `<h2>` heading with an empty
  `<ul>` under it — the heading has no conditional guard today.
- The **"This course includes"** panel shows "0 lessons".
- The hero stats strip shows a **"Lessons"** stat card reading "0 lessons".

We want authors to suppress everything to do with the table of contents for a
specific course while it is being built, without changing enrolment, access, or
whether the course is listed.

### Behaviour

Add a new boolean course frontmatter flag `table_of_contents_in_development`
(default `false`). When `true`, the course detail page **omits entirely** all
three TOC-related surfaces — no empty shells, no placeholders, the areas simply
do not render:

1. **Hero "Lessons" stat card** — omitted (not blanked/zeroed). The stats strip
   already conditionally omits other cards (difficulty, duration,
   enrolment-summary are each wrapped in `{% if %}`), so this follows the
   existing pattern in the same block.
2. **"This course includes" lesson-count line** — the `<li>{{ lesson_count }}
   lesson…</li>` is omitted. **Edge case (decided):** when omitting the
   lesson-count line would leave the panel with no items at all (no
   `includes_assessments`, and no certificate item — the certificate `@claude
   TODO` is not implemented yet), the **entire "This course includes"
   `<h3>` + `<ul>` panel is omitted** so no orphan heading sits above an empty
   list. If the panel still has at least one item (e.g. "Includes assessments"),
   the panel renders with the lesson-count line dropped.
3. **"Course content" `<section>`** — the whole `<section><h2>Course
   content</h2>…</section>` block is omitted (heading included), not just its
   inner `<div>`, otherwise the heading-with-nothing-under-it bug persists.

The flag **only** suppresses TOC UI. It does not change enrolment, access,
listing, or the `visibility` lifecycle. It is orthogonal to `visibility`
(`published` / `coming_soon` / `hidden`); the typical use is a `coming_soon`
course whose lessons are not written yet — it stays listed and demoable, but its
empty TOC is hidden.

Do **not** remove the existing `@claude TODO` certificate comment in the "This
course includes" block when editing it (per CLAUDE.md — it stays until the
certificate feature ships).

### Validation rule

A **published** course may not have `table_of_contents_in_development: true` — a
live course should always show its contents. This is a pure intra-`Course`
schema invariant (no backend-owned interpretation), enforced at content
load/authoring time so a bad combination fails loudly during `content_validate`
/ `content_save` rather than rendering a half-broken page.

Implement it as a new `@model_validator(mode="after")` on the `Course` pydantic
model in `content_engine/schema.py`, sitting alongside the existing
`_validate_icon_fields`, raising `ValueError` with `self.file_path` in the
message (same shape as `_validate_icon_fields` and `Form.validate_quiz_fields`).
This inherits the file-located error reporting that `validate_yaml_section()`
already provides. Illustrative message:

> `table_of_contents_in_development must be false for a published course (in <file_path>)`

A defence-in-depth re-check inside `save_course()` (mirroring how it re-runs
`validate_course_icon_fields`) is **optional** and only warranted if a code path
calls `save_course` directly without going through schema validation; follow the
existing icon-field precedent if included, but it is not required for
correctness.

### Data model — schema + model + migration (all three required)

This is **not** a schema-only change. `save_with_uuid()` in
`content_engine/management/commands/content_save.py` reconciles every
frontmatter field name against the Django model's field names and hard-fails
(`ValueError`) on any frontmatter field with no matching model field. So a
complete implementation needs, following the exact shape of the existing
`visibility` field / `0013_course_visibility.py` migration:

1. A new field on the pydantic `Course` schema
   (`content_engine/schema.py`): `table_of_contents_in_development: bool = False`
   (with a `Field(...)` description, matching the surrounding style).
2. A matching Django model field on `Course` (`content_engine/models.py`):
   `table_of_contents_in_development = models.BooleanField(default=False)`.
3. A new migration via `uv run manage.py makemigrations` then `migrate` — one
   `AddField`, following `0013_course_visibility.py`. **Never** edit an existing
   migration.

The view (`student_interface/views.py::course_detail`) passes the flag into the
template context (or the template reads `course.table_of_contents_in_development`
directly, consistent with how other `course.*` fields are used in the template).

### Success criteria (Problem 1)

- A course with `table_of_contents_in_development: true` renders the detail page
  with **none** of the three TOC surfaces (no Lessons stat card, no lesson-count
  line, no "Course content" section), and no empty heading/list remnants.
- When the flag is `true`, no assessments, and no certificate item, the whole
  "This course includes" panel is absent. When assessments exist, the panel
  shows only the "Includes assessments" item.
- A course with the flag `false`/omitted renders exactly as today.
- `content_validate` / `content_save` reject a course that is both `published`
  and `table_of_contents_in_development: true`, with a clear, file-located error.
- `content_save` succeeds for a valid course carrying the new field (schema ⇄
  model reconciliation passes).
- The `@claude TODO` certificate comment is preserved.

## Problem 2 — global visibility/access overrides (dev/staging only)

### Why this matters

We want to preview courses that are `coming_soon` or `hidden`, and courses that
are access-gated, in dev/staging environments — **without editing the
database** — so they demo exactly as they will once launched. This is a global,
system-wide override applied to every course across every site.

Neither override writes to the database. `Course.visibility` and
`Course.access_config` stay exactly as loaded from content YAML; only what the
access/visibility layer *reports* to callers changes.

### The two settings

Both are new global settings owned by the `course_access` app (which already
owns `COURSE_ACCESS_BACKEND` and wraps both visibility and access enforcement),
both booleans defaulting to `False`, declared in
`freedom_ls/course_access/config.py` alongside `COURSE_ACCESS_BACKEND`:

| Setting | Type | Default | Effect when `True` |
|---|---|---|---|
| `OVERRIDE_COURSE_VISIBILITY_TO_VISIBLE` | `bool` | `False` | Every course is treated as visible. `coming_soon` / `hidden` courses become reachable **and look fully published** — no "Coming soon" badge, no "I'm interested" CTA. |
| `OVERRIDE_COURSE_ACCESS_TO_FREE` | `bool` | `False` | Every course is treated as freely accessible — shown and enterable as though its access type were "free". |

The long, explicit names are deliberate: grepping `settings.py` / `config.py`
for `OVERRIDE_` immediately shows *what* is overridden and *to what*, which
matters for a setting whose accidental production use is the exact risk being
guarded against. Declared as optional booleans (`Setting(default=False)`), never
`required=True`, so a project that never sets them behaves exactly as today in
every environment. Reads at every chokepoint use the standard one-line
`config.OVERRIDE_...` idiom (import the `course_access.config` singleton) — never
`settings.OVERRIDE_...` directly.

### Where the overrides hook in

The access override and the backend half of the visibility override funnel
through the backend, but the visibility override also has to reach a few seams
that bypass the backend (see below). Because `get_course_access_backend()` is
`functools.cache`-d, the override must be **read fresh at each seam** on every
call (not baked into which backend class gets cached); this also keeps the
existing test `cache_clear()` pattern working under `override_settings`.

**Visibility override (`OVERRIDE_COURSE_VISIBILITY_TO_VISIBLE`)** — must cover
every seam that today special-cases `coming_soon` / `hidden`:

- `VisibilityEnforcingBackend.get_access()` — when the override is on, skip the
  `COMING_SOON` / `HIDDEN` interception and delegate straight to the inner
  backend's normal decision (course behaves as `published`).
- `VisibilityEnforcingBackend.filter_visible()` — when on, do not exclude
  `HIDDEN` courses; return the inner backend's queryset unfiltered by visibility.
- `raise_404_if_hidden_unregistered()` in `course_access/visibility.py` — this
  standalone function is called directly by the detail / apply / express-interest
  views (`student_interface/views.py`, `course_applications/views.py`,
  `course_interest/views.py`) and **bypasses the backend entirely**, so it must
  become override-aware too: when the override is on, do not raise `Http404` for
  a hidden course.
- `initiate_course_access()` in `student_interface/views.py` — the self-service
  enrol chokepoint short-circuits a `coming_soon` course with a
  `== CourseVisibility.COMING_SOON` redirect back to the detail page **before**
  consulting the backend, so a demoed `coming_soon` course's "Enrol for free" CTA
  would bounce instead of self-registering. When the override is on this
  short-circuit must be skipped so the request flows through to the backend's
  normal free self-registration. (The `raise_404_if_hidden_unregistered` call in
  the same view is already covered above.)
- **"Look fully published" cosmetics** — the direct `== CourseVisibility.COMING_SOON`
  reads that stamp "Coming soon" status/affordances must be suppressed when the
  visibility override is on, so overridden courses demo without any "Coming soon"
  indicator or "I'm interested" CTA. These reads are the cosmetic layer
  downstream of the reachability gate; when the override is on they should behave
  as if `visibility == PUBLISHED`. They live in three places and **all** must be
  covered (the success criteria name the detail page explicitly):
    - Listing / dashboard cards — `student_interface/utils.py::get_course_listing`,
      `views.py::_annotate_recommendations`, `views.py::_available_courses`.
    - The **detail page itself** — `course_detail()` computes
      `is_coming_soon = course.visibility == CourseVisibility.COMING_SOON`
      (`student_interface/views.py`), which drives the template's
      `{% if is_coming_soon and not is_registered %}` branch (the express-interest
      "I'm interested" CTA and the `stamp_interest` call). Suppressing only the
      card reads would still leave the detail page showing "I'm interested"; this
      computation must be override-aware too so the detail page shows the normal
      enrol CTA.

Because the flag is now read at roughly half a dozen `coming_soon` /
`hidden` sites (backend, `raise_404_if_hidden_unregistered`,
`initiate_course_access`, and the cosmetic reads), route them all through a
single shared `config.OVERRIDE_COURSE_VISIBILITY_TO_VISIBLE`-reading helper in
`course_access` (rather than repeating the `config` read inline) so the seams
cannot drift.

**Access override (`OVERRIDE_COURSE_ACCESS_TO_FREE`)** — must make every course
report as free. These three reads must be applied in the
`VisibilityEnforcingBackend` **wrapper** (the backend `get_course_access_backend()`
returns and caches), **not** in `FreeOnlyCourseAccessBackend`: gated inner
backends such as `ApplicationCourseAccessBackend` override these very methods, so
a read placed only on the free backend would miss exactly the gated courses the
override exists to demo. The wrapper today delegates `is_accessible_for_free` /
`get_access_badge` to its inner backend and delegates `get_access` after its
visibility checks, so the override short-circuits each to the free answer before
delegating:

- `is_accessible_for_free(course)` → `True` when the override is on (the
  config-only, no-DB, no-per-user free/gated signal used for badges/JSON-LD).
- `get_access_badge(course)` → the "Free" badge when the override is on (used by
  listings and dashboard discovery).
- `get_access()` → the free-course CTA / `can_self_register` /
  `can_access_content` decision when the override is on (a `CourseAccessDecision`
  whose `is_accessible_for_free` is `True`, so the detail page's JSON-LD agrees),
  so gated courses become enterable as free.

There is no queryset filter to override for access (application-gating never
removes a course from a listing — it only changes the CTA/badge), so the access
override only needs to affect the access-type decision methods above.

Prefer implementing the override reads inside the existing backend methods /
`raise_404_if_hidden_unregistered` (the read-fresh approach) over swapping which
backend the loader caches, so the `functools.cache` on `get_course_access_backend()`
does not need to be invalidated at runtime and the existing test caveat continues
to hold.

### Production safety guardrail

These overrides are for dev/staging only and would be damaging if left on in
production. Three layers of safety:

1. **Both default to `False`** — a project that never sets them behaves exactly
   as today, in every environment including production.
2. **A Django system-check `Warning`** (not an `Error` — it must not block
   `check` / `migrate` / `test` / deploy) in `freedom_ls/course_access/checks.py`,
   registered with `@register(Tags.security)` and modelled on
   `freedom_ls/accounts/checks.py::check_legal_docs_present_when_required`. It
   fires (id `freedom_ls_course_access.W001` — the app's first warning; its only
   existing check is the `E001` error) whenever either override is
   `True` **and** `settings.DEBUG` is `False`. `DEBUG` is read directly via
   `settings.DEBUG` (a Django built-in, per the app-settings skill's carve-out),
   not routed through `config.py`. The message names both settings and hints to
   unset them for the environment.
3. **Documented as dev/staging-only** with an inline comment in
   `course_access/config.py`, and set to `True` only in a settings module that
   also keeps `DEBUG = True` (i.e. alongside `config/settings_dev.py`'s existing
   `DEBUG = True`) — **never** in `config/settings_base.py` or
   `config/settings_prod.py`. This spec does not require adding the override to
   any settings module by default; enabling it is a per-environment choice.

### Multi-site note

Both overrides operate at the backend/function layer (`get_course_access_backend()`
is a process-wide singleton with no site parameter;
`raise_404_if_hidden_unregistered()` takes only `(user, course)`), so they are
naturally site-agnostic and apply uniformly across every site's courses with no
site-specific plumbing. "Global" already matches the shape of the existing seams.

### Success criteria (Problem 2)

- With both overrides `False` (the default), behaviour is byte-for-byte
  identical to today in every environment.
- With `OVERRIDE_COURSE_VISIBILITY_TO_VISIBLE = True`: a `hidden` course's detail
  page returns 200 (no 404) for an unregistered visitor; `hidden` courses appear
  in listings; `coming_soon` courses show the normal enrol CTA and no "Coming
  soon" badge / "I'm interested" affordance anywhere (detail, listings,
  dashboard, recommendations); and clicking that enrol CTA actually
  self-registers the learner (the `initiate_course_access` `coming_soon`
  short-circuit no longer bounces them back to the detail page).
- With `OVERRIDE_COURSE_ACCESS_TO_FREE = True`: every course shows a "Free"
  badge, is self-registerable, and its content is enterable as free, regardless
  of `access_config`.
- `Course.visibility` and `Course.access_config` are never written; the override
  is purely at the read/report layer.
- `manage.py check` emits `freedom_ls_course_access.W001` (a Warning, not an
  Error) when either override is `True` while `DEBUG` is `False`, and is silent
  otherwise.
- The overrides are read via `config.OVERRIDE_...` (never `settings.` directly),
  and read fresh inside the backend methods so the `functools.cache` on the
  loader does not need invalidating.

## Out of scope

- Implementing a course certificate model / the certificate line in "This course
  includes" (the `@claude TODO` stays untouched).
- Any per-course (rather than global) override mechanism.
- Neutralizing the `coming_soon` gates in the apply (`course_applications/views.py`)
  and express-interest (`course_interest/views.py`) views. Those endpoints stay
  gated on real `visibility`; the visibility override reaches only the detail
  page, listings, dashboard/recommendations, and the self-service enrol
  chokepoint (`initiate_course_access`). The apply / express-interest views are
  reachable only via affordances the override already hides, so leaving them
  gated does not affect the demo.
- A dedicated `settings_staging.py` (none exists today; staging is a
  `DEBUG`-controlled deploy of the prod-style settings).
- Adding the overrides to any shipped settings module by default.

## Notes on decisions made

- **Field name `table_of_contents_in_development`** (over `contents_in_development`
  / `toc_hidden` / `hide_toc`): explicit that only the TOC is suppressed — a
  course's content can be under development while its table of contents is
  complete enough to display. Matches the codebase's no-prefix boolean style
  (`submit_on_exit`, `required`, `quiz_show_incorrect`); names the cause, letting
  the template infer the effect, consistent with how `visibility` names a
  lifecycle state rather than an effect.
- **"This course includes" empty panel is hidden entirely** (not left as an
  orphan heading over an empty list) — chosen for a clean demo, consistent with
  omitting the other two TOC surfaces outright.
- **Override reads live inside the backend methods / `raise_404_if_hidden_unregistered`**
  rather than swapping the cached backend, to avoid `functools.cache`
  invalidation and preserve the existing test `cache_clear()` contract.
- **The visibility override reaches the self-service enrol chokepoint
  (`initiate_course_access`)** so a demoed `coming_soon` course's "Enrol for free"
  CTA actually self-registers, not just displays — the point of the override is to
  demo a course exactly as it will behave once launched. The apply /
  express-interest views are left gated (see Out of scope) because they are only
  reachable via affordances the override already suppresses.
- **The access override is applied in the `VisibilityEnforcingBackend` wrapper**,
  not the free backend, so gated inner backends (which override the same methods)
  are still forced to a free answer.
- **Warning, not Error**, for the production guardrail — a project might run a
  `DEBUG=False` staging environment intentionally, so the check must surface the
  risk without blocking deploys.
