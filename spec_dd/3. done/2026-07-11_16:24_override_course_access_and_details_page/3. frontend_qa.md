# Frontend QA: TOC-in-development flag + visibility/access preview overrides

Manual / browser QA. All checks run in a browser against the dev server. Dev
settings pin `FORCE_SITE_NAME = "DemoDev"`, so the running site is always
**DemoDev**. Create all test data on **DemoDev** (see `feedback_use_demodev_site`).
The `do_qa` flow auto-creates missing data via the `fls:qa-data-helper` agent.

## Setup

### 1. Test data

```bash
# Base demo site + verified learner (demodev@email.com / demodev@email.com) + courses
uv run python manage.py create_demo_data

# Free + application-gated courses on DemoDev, plus a learner enrolled in nothing:
uv run python manage.py qa_create_course_access_types --site DemoDev
#   FREE course slug:  qa-free-course-access-types
#   GATED course slug: qa-application-gated-course-access-types
```

Courses needed (ask `fls:qa-data-helper` to create any that are missing — a
**published** course may NOT have `table_of_contents_in_development: true`, so the
TOC-flag courses must be `coming_soon` or `hidden`):

| Purpose | Slug (suggested) | visibility | table_of_contents_in_development | access |
|---|---|---|---|---|
| Coming-soon demo (already in demo content) | `content-widgets-demo-reference` | `coming_soon` | false | free |
| **TOC hidden, no assessments** | `qa-toc-in-development` | `coming_soon` | **true** | free — and its child content should contain **no Form/assessment** children so the "This course includes" panel would be empty |
| **TOC hidden, with an assessment** | `qa-toc-in-development-with-assessment` | `coming_soon` | **true** | free — include **at least one Form (assessment)** child |
| Hidden course | `qa-hidden-course` | `hidden` | false | free |

Logins:
- **Existing learner:** `demodev@email.com` / `demodev@email.com` (verified,
  enrolled in nothing).

### 2. Run the dev server on a random port

```bash
PORT=$(.claude/fls/scripts/find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL below: **`http://127.0.0.1:$PORT`** (substitute the real port). Build
Tailwind first if styling looks wrong: `npm run tailwind_build`.

### 3. Enabling the preview overrides (Part B only)

The two overrides default to `False`. To exercise them in the browser, temporarily
enable them in `config/settings_dev.py` (which keeps `DEBUG = True`), then restart
the dev server. **Revert these edits when done** — never commit them.

```python
# config/settings_dev.py — TEMPORARY QA ONLY, revert after testing
OVERRIDE_COURSE_VISIBILITY_TO_VISIBLE = True   # Workflows B1–B3
OVERRIDE_COURSE_ACCESS_TO_FREE = True          # Workflow B4
```

Enable only the override a workflow calls for, restart the server, test, then set
it back to `False` (or remove the line) and restart before the next part.

# Part A — `table_of_contents_in_development` flag (overrides OFF)

Keep both overrides `False` (default) for all of Part A.

## Workflow A1 — Flag `true`, no assessments: all TOC surfaces gone

1. Logged out (or logged in as the learner), go to
   `http://127.0.0.1:$PORT/courses/qa-toc-in-development/detail/`.
2. **Expect:** page loads (200).
3. In the **hero stats strip**, expect **no "Lessons" stat card** (Level /
   Duration / Enrolment cards may still appear if the course has them).
4. In the sign-up panel, **expect no "This course includes" heading at all** —
   the whole panel is absent (no assessments, no certificate → nothing to list).
5. **Expect no "Course content" section** anywhere on the page — no "Course
   content" heading sitting above an empty list.
6. **Expect NOT to see** any "0 lessons" text or empty list remnants.

## Workflow A2 — Flag `true`, with an assessment: panel keeps only assessments

1. Go to
   `http://127.0.0.1:$PORT/courses/qa-toc-in-development-with-assessment/detail/`.
2. **Expect:** no "Lessons" stat card, and no "Course content" section (as A1).
3. **Expect:** the "This course includes" panel **is present**, showing only
   **"Includes assessments"** — and **no** "N lessons" line above it.

## Workflow A3 — Flag `false`/omitted: page unchanged (regression)

1. Go to `http://127.0.0.1:$PORT/courses/qa-free-course-access-types/detail/`
   (a normal published/free course, flag not set).
2. **Expect:** everything renders as before — the **"Lessons"** stat card (e.g.
   "3 lessons"), a "This course includes" panel with an **"N lessons"** line, and
   the **"Course content"** section with its table of contents. Nothing is hidden.

# Part B — preview overrides

## Workflow B1 — Visibility override: hidden course reachable

Enable only `OVERRIDE_COURSE_VISIBILITY_TO_VISIBLE = True`, restart the server.

1. **Logged out** (incognito), go to
   `http://127.0.0.1:$PORT/courses/qa-hidden-course/detail/`.
2. **Expect:** page loads **200** (with the override OFF this returns **404** —
   verify that too by reverting, restarting, and re-visiting: it 404s).
3. Go to `http://127.0.0.1:$PORT/courses/` — **expect the hidden course now
   appears** in the catalogue listing.

## Workflow B2 — Visibility override: coming-soon looks published

Override still `True`.

1. Go to
   `http://127.0.0.1:$PORT/courses/content-widgets-demo-reference/detail/`
   (a `coming_soon` course).
2. **Expect NOT to see** any "Coming soon" badge/eyebrow and **no "I'm
   interested"** control. Instead the normal enrol CTA (**"Enrol for free"**)
   appears.
3. Go to `http://127.0.0.1:$PORT/courses/` and to the dashboard `/`
   (logged in as the learner): the coming-soon course shows **no "Coming soon"**
   status anywhere (listings, recommendations, dashboard cards) — it presents
   like a normal course.

## Workflow B3 — Visibility override: enrol actually self-registers

Override still `True`. Log in as `demodev@email.com` / `demodev@email.com`.

1. Open the coming-soon course detail
   (`.../content-widgets-demo-reference/detail/`) and click **"Enrol for free"**.
2. **Expect:** you are **self-registered and land inside the course** (the course
   player / first item) — you are **not** bounced back to the detail page.
   (With the override OFF, this CTA is the express-interest control instead.)

## Workflow B4 — Access override: every course is free

Set `OVERRIDE_COURSE_VISIBILITY_TO_VISIBLE = False` and
`OVERRIDE_COURSE_ACCESS_TO_FREE = True`, restart.

1. Go to `http://127.0.0.1:$PORT/courses/` and find the application-gated course
   (`qa-application-gated-course-access-types`).
2. **Expect:** its badge now reads **"Free"** (not "By application").
3. Open `.../qa-application-gated-course-access-types/detail/`.
4. **Expect:** the CTA reads **"Enrol for free"** (not "Apply now"); the enrolment
   signal near the CTA reads the free copy; and (logged in) clicking it
   **self-registers** you into the course as free.
5. **View source** of the gated detail page and confirm the Course **JSON-LD** has
   `"isAccessibleForFree": true`.

## Workflow B5 — Data is never written (spot check)

After Parts B1–B4, set both overrides back to `False` and restart.

1. Re-visit `qa-hidden-course` logged out → **expect 404 again** (hidden), and the
   gated course badge reads **"By application"** again, and the coming-soon course
   shows "Coming soon" again.
2. This confirms the overrides only changed what was *reported*, never the stored
   `visibility` / `access_config`.

## Workflow B6 — Production-safety warning (terminal, not browser)

1. In `config/settings_dev.py`, temporarily set `DEBUG = False` **and**
   `OVERRIDE_COURSE_VISIBILITY_TO_VISIBLE = True`.
2. Run `uv run python manage.py check`.
3. **Expect:** a **Warning** `freedom_ls_course_access.W001` naming the enabled
   override(s), and the command still exits successfully (it is a Warning, not an
   Error — it must not block).
4. Set `DEBUG = True` again (or unset the override) and re-run `check` → **expect
   the warning is gone**.
5. **Revert all `settings_dev.py` edits** when finished.
