# Implementation plan — Dev email viewer (Mailpit)

## Summary

Replace the file-based dev email backend with [Mailpit](https://mailpit.axllent.org/),
an email-testing tool that captures SMTP mail and exposes a web inbox. Mailpit is
added to the shared `dev_db/docker-compose.yaml` so one instance serves every
worktree, mirroring the existing shared-Postgres pattern. Development Django is
pointed at Mailpit's SMTP listener (`localhost:1025`); the web inbox lives at
`http://localhost:8025`. The remaining work is documentation cleanup so nothing
still tells developers to look in `gitignore/emails`.

This is an infrastructure + configuration + docs change. There is no application
code with testable behaviour (see "Testing note" below).

## Tasks

### Task 1 — Add the Mailpit service to the dev composition

Edit `dev_db/docker-compose.yaml`. Add a `mailpit` service under `services:`,
alongside `postgres`, following the same plain style (no restart policy, no named
volume — captured mail is intentionally ephemeral per spec):

```yaml
  mailpit:
    image: axllent/mailpit:v1.21   # pin — verify/bump to current stable before committing
    ports:
      - "1025:1025"   # SMTP listener Django sends to
      - "8025:8025"   # web inbox
```

Notes:
- **Pin the image tag** (do not use `:latest`), matching the existing `postgres:17`
  pin in this file so the composition behaves identically across machines. Before
  committing, confirm the exact current stable tag on
  https://github.com/axllent/mailpit/releases and use that (the `v1.21` above is a
  placeholder to replace with the verified tag).
- No `volumes:` entry — mail clears on container restart (spec: ephemeral).
- Default Mailpit ports `1025`/`8025` are used directly (spec decision).
- Keep it ordered/indented consistently with the existing `postgres` service so the
  file stays clean.

### Task 2 — Point development email at Mailpit

Edit `config/settings_dev.py`. Replace the file-based backend block (lines 56-59):

```python
###
# EMAIL

EMAIL_BACKEND = "django.core.mail.backends.filebased.EmailBackend"
EMAIL_FILE_PATH = "gitignore/emails"
```

with a minimal SMTP backend aimed at Mailpit:

```python
###
# EMAIL
# Development email is captured by Mailpit (see dev_db/docker-compose.yaml).
# Browse the inbox at http://localhost:8025

EMAIL_BACKEND = "django.core.mail.backends.smtp.EmailBackend"
EMAIL_HOST = "localhost"
EMAIL_PORT = 1025
```

Notes:
- Keep the block to these three settings. Django's SMTP backend already defaults
  `EMAIL_HOST_USER`/`EMAIL_HOST_PASSWORD` to `""` and `EMAIL_USE_TLS` to `False`,
  which is exactly what unauthenticated Mailpit needs — adding those lines is noise
  (and avoids a needless `# noqa: S105` for an empty-password literal).
- Leave `config/settings_prod.py` untouched — production keeps its env-driven SMTP
  config. This satisfies the "production email behaviour is unchanged" criterion.

## Testing note (TDD scope)

The spec's behaviour is delivered entirely through docker-compose, dev-only Django
settings, and documentation. None of these have application-level behaviour that a
`pytest` unit test can meaningfully or stably assert:

- `config/settings_dev.py` is a development-only settings module; the test suite
  runs with `TESTING=True` and does not exercise the SMTP backend. Asserting on the
  literal backend string would be a tautological test of configuration, not
  behaviour.
- The docker-compose service and docs are not Python.

Therefore no automated test is added. Verification is performed through the QA flow
(browser-based: trigger an email, confirm it lands in the Mailpit inbox with
HTML / plain-text / raw views). Run `uv run pytest` after the change only to confirm
nothing regressed — the existing suite must stay green.
