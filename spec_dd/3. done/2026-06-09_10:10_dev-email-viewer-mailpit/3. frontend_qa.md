# Frontend / browser QA — Dev email viewer (Mailpit)

This feature has no FLS-built UI, but its acceptance is entirely browser-verified:
an email triggered in development must appear in **Mailpit's** web inbox, and its
HTML / plain-text / raw views must be viewable. These tests confirm that.

## Prerequisites / setup

1. **Start the dev composition (now includes Mailpit):**
   ```
   cd dev_db
   docker compose up
   ```
   Wait until both `postgres` and `mailpit` containers are running. Mailpit's web
   inbox is served at `http://localhost:8025` and its SMTP listener at
   `localhost:1025`.

2. **Migrate (if needed):** in the worktree root, `uv run python manage.py migrate`.

3. **Pick a free port and run the dev server** (do NOT use port 8000):
   ```
   PORT=$(.claude/fls/scripts/find_available_port.sh)
   uv run python manage.py runserver $PORT
   ```
   The app base URL is `http://127.0.0.1:$PORT`.

4. **Test data:** these flows need a verified user on the **DemoDev** site (the dev
   `FORCE_SITE_NAME`). If no suitable user exists, the `fls:qa-data-helper` agent
   should create one on the DemoDev site. Note the user's email address — call it
   `TEST_EMAIL` below.

5. **Clear the inbox first (optional but cleaner):** open
   `http://localhost:8025`, and if there are old messages, use Mailpit's
   "Delete all" so the new message is unambiguous. (Mail is ephemeral and also
   clears on container restart.)

## Test 1 — A triggered email lands in the Mailpit inbox

**Goal:** verify the core acceptance criterion — triggering an email-sending flow
in development produces a message in the Mailpit web inbox.

1. In the browser, go to `http://127.0.0.1:$PORT/accounts/password/reset/`
   (the allauth password-reset request page). If the path differs in this build,
   navigate from the login page's "Forgot password?" link instead.
2. Enter `TEST_EMAIL` and submit.
   - **Expect:** the app shows its "password reset sent" confirmation page (no SMTP
     error). A connection error here means Django could not reach Mailpit on
     `localhost:1025` — check the `mailpit` container is up.
3. Open `http://localhost:8025` (Mailpit inbox), refreshing if needed.
   - **Expect:** a new message appears at the top of the list, addressed to
     `TEST_EMAIL`, with a recognisable password-reset subject and the FLS branded
     sender.

## Test 2 — HTML, plain-text, and raw views are all viewable

**Goal:** verify the spec criterion that all three views render in Mailpit.

1. In `http://localhost:8025`, click the message from Test 1 to open it.
2. **HTML view (default):**
   - **Expect:** the branded FLS HTML email renders (header/logo, body, a
     reset-link call to action), styled — confirming CSS inlining survived delivery.
3. Switch to the **Text** view (Mailpit's view toggle / tab).
   - **Expect:** the plain-text multipart alternative renders, containing the reset
     URL as readable text.
4. Switch to the **Raw** (source) view.
   - **Expect:** the raw RFC822 source is shown, including the `Content-Type:
     multipart/alternative` boundary and both the HTML and text parts — confirming a
     proper multipart message was delivered over SMTP (not a file dump).

## Test 3 — (Optional) the captured link actually works

**Goal:** sanity-check that the captured email is functional, not just visible.

1. From the email's HTML or text view in Mailpit, copy the password-reset URL.
2. Paste it into the browser against `http://127.0.0.1:$PORT`.
   - **Expect:** the allauth "set new password" page loads for `TEST_EMAIL`,
     confirming the link Mailpit captured is the real, working link a developer
     would use during a demo.

## What this does NOT need to cover

- Production email (out of scope; `settings_prod.py` is untouched).
- Per-branch inbox isolation (intentionally shared; one Mailpit serves all
  worktrees).
- Persistence across restarts (intentionally ephemeral).
