## Tasks

### Task 1 — Parametrize `test_score_quiz_single_correct_answer` / `test_score_quiz_single_incorrect_answer`

**File:** `freedom_ls/student_progress/tests/test_form_progress_score_quiz.py`

The two single-question tests differ only in: (a) which option the answer selects (correct vs incorrect) and (b) the expected `scores["score"]` (1 vs 0). `max_score` is 1 in both. Same fixtures, same shape — textbook parametrize candidate.

Leave `test_score_quiz_multiple_questions_mixed_answers` and `test_score_quiz_includes_unanswered_questions_in_max_score` alone — they have meaningfully different setup (3 questions, different answer patterns) and per the spec, "If the variants need different fixtures or different setup beyond a single value swap, **don't parametrize**".

**Pseudocode:**

```python
@pytest.mark.parametrize(
    "select_correct, expected_score",
    [
        (True,  1),
        (False, 0),
    ],
    ids=["correct_answer_scores_one", "incorrect_answer_scores_zero"],
)
@pytest.mark.django_db
def test_score_quiz_single_question(mock_site_context, select_correct, expected_score):
    user = UserFactory()
    form = FormFactory()
    page = FormPageFactory(form=form, title="Quiz Page 1", order=0)
    question = FormQuestionFactory(form_page=page, question="What is 2 + 2?", type="multiple_choice", order=0)
    correct = QuestionOptionFactory(question=question, text="4", value="4", order=0, correct=True)
    wrong   = QuestionOptionFactory(question=question, text="3", value="3", order=1, correct=False)
    QuestionOptionFactory(question=question, text="5", value="5", order=2, correct=False)

    form_progress = FormProgressFactory(user=user, form=form)
    answer = QuestionAnswerFactory(form_progress=form_progress, question=question)
    answer.selected_options.add(correct if select_correct else wrong)

    form_progress.score_quiz()

    form_progress.refresh_from_db()
    assert form_progress.scores is not None
    assert form_progress.scores["score"] == expected_score
    assert form_progress.scores["max_score"] == 1
```

Delete the two original tests. The two-test count drops to one parametrized test with two cases. The other two tests in the file remain unchanged.

### Task 2 — Parametrize `TestGetTextColor`

**File:** `freedom_ls/base/tests/test_context_processors.py`

Current state (verified with a fresh read): four tests in `TestGetTextColor`, each asserting `get_text_color(<hex>) == <expected>`. All four are pure value-in/value-out — perfect parametrize candidates.

Leave `TestBranchNameToColor`, `TestDebugBranchNotInProdSettings`, and `TestDebugBranchInfo` alone — different shapes, different fixtures.

class TestGetTextColor:

Replace the four existing methods inside the class with this single parametrized method.

### Task 3 — Evaluate `partial_list_courses` cluster (`test_course_list_views.py`)

**File:** `freedom_ls/student_interface/tests/test_course_list_views.py`

The audit cited this file as a copy-paste cluster. A fresh read shows five `partial_list_courses` tests, each exercising a **different behavioural axis**:

- `_anonymous_sees_empty` — anonymous client, asserts both lists empty (no factories beyond fixture)
- `_current_courses` — registers a course, asserts it appears in `registered_courses`
- `_current_courses_have_progress_percentage` — asserts the `progress_percentage` attribute exists on registered courses
- `_completed_courses` — registers + creates `CourseProgress(completed_time=…)`, asserts membership in `completed_courses` and absence from `registered_courses`
- `_includes_recommended_courses` — uses `RecommendedCourseFactory`, asserts `recommended_courses` context

Per the spec: "If the variants need different fixtures or different setup beyond a single value swap, **don't parametrize** — the resulting test is harder to read than the duplicates." These tests have different setup (different factories, different post-conditions, different context keys). They are a duplication-of-shell, not a duplication-of-behaviour.

**Decision:** Do **not** parametrize. Apply two surgical readability cleanups instead:

1. Extract a tiny local helper `_logged_in_client(user)` to remove the repeated three-line `Client(); client.force_login(user)` block. Keep it inside this test file — do not lift across files.
2. Add a comment near the cluster header recording the rationale ("Each test asserts a distinct context-key behaviour; parametrize would obscure intent").

The PR description must reference the audit's note that "the audit said three; verify what's actually there" and explain the decision. The spec explicitly allows "a written PR-description rationale for why it stayed duplicated" as an alternative outcome (Success criterion 1).

If, on a fresh look during implementation, two of the five tests turn out to be a strict input/output pair with identical setup, parametrize **just those two** and leave the rest alone. Decide at implementation time, not now.

### Task 4 — Replace formula-derived oracles in `test_course_progress_calculation.py`

**File:** `freedom_ls/student_management/tests/test_course_progress_calculation.py`

The audit-cited tautology pattern is `assert percentage == 67  # round(66.666...)` at lines 74 and 123. A fresh `grep round(` confirms exactly those two sites — both already use a hard-coded literal `67` (good); the tautology is the **comment**, which signals the author derived the oracle by re-running the production rounding.

Plan: convert the existing two-and-three-item ad-hoc tests into a single parametrized test whose oracles are written down up-front, arithmetically distinct, and **not** annotated with rounding formulae. Keep the existing "course parts" and "mixed direct/part children" tests as they exercise structural behaviour distinct from the percentage-rounding axis — but where they incidentally check `67`, replace the `# round(66.666...)` comment with a written-down "(2 of 3) → 67" rationale.

**Pseudocode for the new parametrized test:**

```python
@pytest.mark.parametrize(
    "completed, total, expected",
    [
        (0, 1, 0),     # nothing done, single item
        (1, 1, 100),   # everything done, single item
        (1, 2, 50),    # half done
        (1, 3, 33),    # one of three (round 33.333)
        (2, 3, 67),    # two of three (round 66.666)
        (3, 4, 75),    # three of four — round number, no rounding
        (1, 4, 25),    # one of four — round number
    ],
    ids=[
        "0_of_1_is_0",
        "1_of_1_is_100",
        "1_of_2_is_50",
        "1_of_3_is_33",
        "2_of_3_is_67",
        "3_of_4_is_75",
        "1_of_4_is_25",
    ],
)
@pytest.mark.django_db
def test_progress_percentage_for_n_of_m(mock_site_context, completed, total, expected):
    course: Course = CourseFactory()
    topics = [TopicFactory(title=f"Topic {i}") for i in range(total)]
    for i, topic in enumerate(topics):
        course.items.create(child=topic, order=i)
    completed_topic_ids = {t.id for t in topics[:completed]}

    percentage = calculate_course_progress_percentage(course, completed_topic_ids, set())

    assert percentage == expected
```

Then:

- The existing tests `test_course_with_no_children_returns_zero_percent`, `test_course_with_one_topic_none_completed`, `test_course_with_one_topic_completed`, `test_course_with_mixed_content`, `test_course_with_multiple_items_partial_completion` overlap with cases in the new parametrized test. Delete the ones whose behaviour is covered by the parametrized form **only** when the deletion does not lose behavioural coverage (e.g. `mixed_content` exercises a `Topic` + `Form` mix and stays).
- Keep `test_course_with_no_children_returns_zero_percent` (zero-children edge case has no items at all — not just zero completed of N).
- Keep `test_course_with_course_part_children`, `test_course_with_mixed_direct_and_part_children`, `test_course_with_forms_in_course_part` — they exercise structural axes (CoursePart traversal, mixed direct+part, Forms inside parts) the parametrized test deliberately does not cover.
- In `test_course_with_mixed_direct_and_part_children` (line 123) replace the `# round(66.666...)` comment with `# (2 of 3) → 67`.

**Bug-surfacing protocol (per spec decision):** if any oracle disagrees with `calculate_course_progress_percentage`, do **not** silently adjust the literal. File a ticket, mark the offending case `pytest.skip("blocked on #NNN")`, and proceed.

### Task 5 — Rewrite `test_later_retries_use_correct_base_delays`

**File:** `freedom_ls/webhooks/tests/test_delivery.py` (lines 116-139, inside `TestCalculateNextRetry`)

Current loop iterates `RETRY_DELAYS` and re-derives both ends of the jitter window from the same constant the production code uses. Re-running the implementation: textbook tautology.

Plan: replace the loop with one parametrized test whose expected windows are written down literally (`60..72`, `300..360`, `1800..2160`, `7200..8640`, `43200..51840` — 20% jitter on each `RETRY_DELAYS` entry). Keep the existing `test_first_retry_within_expected_range` and `test_jitter_adds_variability` tests unchanged — they cover different axes.

**Pseudocode:**

```python
@pytest.mark.parametrize(
    "attempt_count, base_delay_seconds, max_delay_seconds",
    [
        (1, 60,    72),     # 60 + 20% = 72
        (2, 300,   360),    # 300 + 20% = 360
        (3, 1800,  2160),   # 1800 + 20% = 2160
        (4, 7200,  8640),   # 7200 + 20% = 8640
        (5, 43200, 51840),  # 43200 + 20% = 51840
    ],
    ids=[
        "attempt_1_uses_60s_base",
        "attempt_2_uses_300s_base",
        "attempt_3_uses_1800s_base",
        "attempt_4_uses_7200s_base",
        "attempt_5_uses_43200s_base",
    ],
)
def test_retry_delay_window(self, attempt_count, base_delay_seconds, max_delay_seconds):
    result = calculate_next_retry(attempt_count)
    now = timezone.now()
    min_expected = now + timedelta(seconds=base_delay_seconds)
    max_expected = now + timedelta(seconds=max_delay_seconds)
    # 2-second slop for clock drift between calculate_next_retry() and timezone.now() above.
    assert min_expected - timedelta(seconds=2) <= result <= max_expected + timedelta(seconds=2)
```

Notes:

- The expected `max_delay_seconds` literals are written down from the spec ("20% jitter on RETRY_DELAYS"), not derived from `RETRY_DELAYS` at test time. That is the whole point.
- Do not import `RETRY_DELAYS` for this new test — that re-introduces the tautology.
- The `_first_retry_within_expected_range` test is now logically a strict subset of `attempt_1_uses_60s_base`. Delete it to avoid duplication.

If `RETRY_DELAYS` ever changes, this test must be edited to match — that is the **correct** behaviour for an oracle test: a constant-change forces the human to write down the new oracle.
