# Spec: Move Django messages out of the page heading area into bottom-anchored toasts

## Background

The current `partials/messages.html` renders all Django messages as a single
fixed overlay anchored at `top-24 left-1/2 -translate-x-1/2` (centre-top), which
overlaps page headings and sticky chrome on most screens. There is one
`aria-live="assertive"` region for *all* severities (so success / info messages
announce as if they were urgent), no `aria-atomic`, and no auto-dismiss
behaviour matched to severity (every message disappears after 8s including
errors). There is also no HTMX delivery path: messages only render on full-page
loads via `{% include "partials/messages.html" %}` in `_base.html`.

This work moves the toast region out of the heading area, splits the live
regions by severity, ties auto-dismiss to severity, and adds a single HTMX
delivery mechanism for messages produced by HTMX requests.

Reference research is in `research_toasts.md`. Decisions in this spec are
informed by that research; this document only restates findings where they
shape a decision.

## Scope

### In scope

- Replace `freedom_ls/base/templates/partials/messages.html` with a new
  bottom-anchored toast region.
- Update the Alpine `message` component (or replace it with a `toasts` store)
  to support per-severity timing, pause-on-hover, pause-on-focus, and
  pause-on-window-blur.
- Add two ARIA live regions in the base layout (one polite, one assertive)
  rendered once on page load; toast HTML is appended into the appropriate
  region.
- Add an HTMX delivery mechanism for toasts produced by HTMX requests, so
  toasts work after non-full-page interactions (form submits via HTMX, panel
  actions, etc.). This work picks **out-of-band swap** (rationale below).
- Respect `prefers-reduced-motion`.
- Use `env(safe-area-inset-bottom)` / `env(safe-area-inset-right)` in offsets
  so toasts clear iOS home indicators and notches.
- Update `<meta name="viewport">` to `viewport-fit=cover` so safe-area-inset
  values resolve to non-zero on iOS.
- Capture before/after Playwright screenshots at desktop, tablet, and mobile
  widths for QA.

### Out of scope (explicit non-goals)

- Swipe-to-dismiss. Research flags this as a polish item with non-trivial
  pointer-event work; defer to a follow-up.
- Stack-shrink / hover-to-expand visual stack effect (Sonner-style). Use a
  plain vertical column with a visible cap.
- Length-aware (read-time-based) auto-dismiss durations. Use fixed durations
  per severity instead.
- Visual progress indicator (countdown bar). Nice-to-have, deferred.
- Toast sound, vibration, push-style notifications.
- A general-purpose JS API for *components* to fire toasts on the client side
  without a server round trip. Toasts are driven by Django messages only.
  Out-of-band swaps and full-page loads are sufficient for everything
  in-scope.
- Migration of the Unfold admin's own messages. The admin uses
  `unfold/helpers/messages.html` from the package and is unaffected.

## Decisions and rationale

### Position

- **Desktop (>=md, 768px+):** bottom-right corner, vertical stack, fixed
  offset of `1rem` plus `env(safe-area-inset-right)` and
  `env(safe-area-inset-bottom)`. Max toast width ~`24rem` (384px).
  - *Why:* OS-aligned, matches Sonner / HeroUI / Bootstrap convention. Bottom
    avoids the heading-overlap class of bug entirely. Right edge keeps the
    toast region away from sidebars on educator/student panels (which sit on
    the left).
- **Mobile (<md, <768px):** anchored to viewport bottom, full width minus
  `0.5rem` margin each side, capped to `100%`, plus
  `env(safe-area-inset-bottom)` offset.
  - *Why:* WCAG reflow (1.4.10) requires legibility at 320px without
    horizontal scroll; corner offsets are meaningless at that width. Matches
    Material's snackbar pattern.
- The container is `position: fixed` with `pointer-events: none`; each
  individual toast re-enables `pointer-events: auto` so clicks between toasts
  pass through to the page underneath.

### Severity model and ARIA

- Two pre-rendered live regions in `_base.html`, both empty on first paint:
  - `<div role="status" aria-live="polite" aria-atomic="false">` — receives
    success, info, warning, debug toasts.
  - `<div role="alert" aria-live="assertive" aria-atomic="false">` — receives
    error toasts only.
- `aria-atomic="true"` is set on each *individual toast element*, so the
  toast text is announced as a unit but existing toasts in the region are
  not re-announced when a new one arrives. (Setting `aria-atomic` on the
  *container* is the well-documented Bootstrap pitfall — research §5.)
- Severity is the existing Django messages tag (`debug`, `info`, `success`,
  `warning`, `error`).
- Icons remain `aria-hidden`; meaning is carried in the toast text. The
  visual style continues to pair colour with an icon so colour is not the
  only signal (WCAG 1.4.1).

### Auto-dismiss

- Per the idea file:
  - `success`, `info`, `debug`: auto-dismiss after **5000 ms**.
  - `warning`: auto-dismiss after **7000 ms**.
  - `error`: **persistent** — no auto-dismiss; user must click the close
    button (or press `Esc` while the close button is focused).
- The dismiss timer pauses while:
  - The pointer is over the toast (`@mouseenter` / `@mouseleave`).
  - The keyboard focus is inside the toast (`@focusin` / `@focusout`) — so
    keyboard users tabbing through the close button do not have the toast
    vanish out from under them.
  - The window has lost focus (`window` `blur` / `focus`) — so a user
    returning to the tab still sees the message.
- "Pause" means: stop the timer at its current remaining time and resume from
  that remaining time, not restart from zero. Implementation note:
  `setTimeout` + a stored `remaining` value, recomputed on pause.

### Stack behaviour

- Vertical column. Newest at the **bottom** (closest to the viewport edge),
  older toasts pushed up — consistent with bottom-anchored conventions
  (Sonner, Material desktop). Insert via DOM append; visual order matches
  source order.
- **Soft cap: 5 toasts visible by default.** When a sixth arrives, the oldest
  non-error toast is dismissed early. Errors are never auto-evicted (they are
  persistent by design and may carry information the user needs). If the
  visible stack already contains 5 errors, additional incoming non-error
  toasts are dropped silently; additional incoming errors are still appended
  (the cap is intentionally soft for errors). (Educator flows can in
  principle emit many error messages from a single form validation; this
  rule keeps them all visible.) The cap is "soft" only for errors — for all
  non-error severities it is a hard cap of 5.
- 8px gap between adjacent toasts.
- When a mid-stack toast is dismissed, surviving toasts slide to fill the
  gap via `transform` transition (a transform-only FLIP-style animation, or
  the simpler grid-gap collapse — implementation choice deferred to the
  plan).

### Manual dismiss

- Every toast has a close button in its top-right corner, visible at all
  times (not only on hover). `aria-label="Dismiss notification"`. Visible
  hit target ~16x16, padded to a minimum 24x24 hit area. (Note: this is
  smaller than the WCAG 2.5.8 target-size minimum of 24x24 CSS px which is
  met by the padded hit area, not the icon glyph itself; that's
  intentional and consistent with common toast UI.)
- Pressing **Esc** while keyboard focus is anywhere inside the toast (most
  commonly on the close button) dismisses that toast.
- Clicking the close button on a toast clears its dismiss timer immediately.

### Animation

- Enter: `transform` (translateX from `+100%` for desktop bottom-right;
  translateY from `+100%` for mobile bottom-anchored) plus `opacity` 0→1.
  Duration **200 ms**, easing `ease-out`.
- Exit: same axis, reversed. Duration **150 ms**, easing `ease-in`.
- Stack-shift (when a neighbour is dismissed): `transform` only, **150 ms**
  `ease-out`.
- Wrap motion in `@media (prefers-reduced-motion: no-preference)`. Under
  `prefers-reduced-motion: reduce`, only `opacity` animates and translate
  distance is `0`.
- No animation of `top`, `left`, `height`, `width`, or `margin` — research
  §6 (compositor-only).

### HTMX delivery — choose OOB swap (single mechanism)

The idea explicitly requires picking one mechanism. **This spec picks
out-of-band swap.**

- *Why OOB swap and not `HX-Trigger`:*
  - Server-side rendering is the project default (cotton + Django templates
    + minimal Alpine). OOB swap keeps message rendering server-side and
    consistent with the full-page-load path: the same partial renders both.
  - `HX-Trigger` requires duplicating the message HTML logic in JS (or
    serving a fragment endpoint), which adds a second source of truth.
  - The full-page-load path *must* render messages server-side anyway (login
    redirects, non-HTMX requests). With OOB swap, that same path serves
    HTMX responses with no extra client wiring — the response just contains
    an additional fragment with `hx-swap-oob`.
  - Mixing both mechanisms causes duplicate announcements (idea file and
    research §8). Sticking to one is mandatory.

- *Mechanism:*
  - The toast region in `_base.html` has stable IDs on the two live region
    `<div>`s: `id="toast-region-polite"` and `id="toast-region-assertive"`.
  - For full-page loads: `partials/messages.html` is included in `_base.html`
    as today and renders queued messages directly into those two regions on
    first paint (split by severity).
  - For HTMX responses: a Django middleware (or response post-processor)
    appends an OOB fragment to *every* HTMX response that has queued
    messages. The fragment uses `hx-swap-oob="beforeend:#toast-region-polite"`
    (or `...-assertive`) so each new toast is appended into the matching
    region without replacing existing toasts.
  - The same partial template is shared by full-page and HTMX paths to
    avoid duplicate rendering logic.
  - HTMX error responses (4xx/5xx) need to surface error messages too. The
    middleware path covers this for any response that runs through the
    Django response cycle. Network failures (no response) are not in scope —
    HTMX's own `htmx:responseError` listener could be added later if needed.
  - **HTMX swap-on-error caveat:** by default HTMX does not swap content for
    non-2xx responses. The project already permits 422 swaps via the
    existing `htmx:beforeSwap` handler in `alpine-components.js`. To make the
    OOB messages fragment process on other 4xx/5xx responses, that handler
    must be extended (or a sibling handler added) so the OOB fragment for
    `#toast-region-polite` / `#toast-region-assertive` is honoured even when
    the primary swap is suppressed. The plan must explicitly cover this —
    otherwise toasts will silently fail to appear on errors that aren't
    422. (One viable approach: in `htmx:beforeSwap`, set `shouldSwap = true`
    when the response contains an OOB messages fragment, regardless of
    status code, while keeping `isError` truthful.)

### Pointer events on the container

- The toast container is `pointer-events: none`; each toast within it is
  `pointer-events: auto`. This prevents the (often invisible) container from
  blocking clicks on page content between toasts. Research §7.

### Stable element identity

- Each toast gets a unique DOM id generated server-side
  (`uuid.uuid4().hex` rendered into `id="toast-{uuid}"`). The Alpine dismiss
  logic targets the toast root by `this.$root` rather than by id, but unique
  ids future-proof us for `aria-labelledby` and any future stack
  manipulation.

### Tailwind / styling notes

- Severity colour mapping continues to use the existing brand tokens:
  `border-success`, `border-warning`, `border-danger`, `border-primary`,
  `border-muted`. The toast surface remains `bg-surface`. Body text remains
  the default foreground token. Contrast against `bg-surface` must meet
  4.5:1 for body text and 3:1 for the close icon — verify during QA.
- Container Tailwind classes are *not* prescribed in this spec; the
  implementation plan picks them. The constraints are: fixed positioning,
  bottom anchor, responsive switch at `md`, safe-area-inset offsets,
  pointer-events handled per above.

## Affected files (indicative — not a plan)

- `freedom_ls/base/templates/_base.html` — pre-render the two live region
  containers; update `viewport-fit=cover`.
- `freedom_ls/base/templates/partials/messages.html` — rewrite to render
  messages into the right live region with the new structure; reuse for both
  full-page and HTMX OOB paths.
- `freedom_ls/base/static/base/js/alpine-components.js` — replace the
  current `message` component with a per-toast component supporting
  pause-on-hover / focus / window-blur and severity-aware timing. The
  project uses the `@alpinejs/csp` build, which forbids inline JS
  expressions in templates: all logic must live in `Alpine.data()`
  registrations, and template directives are limited to method/property
  references (e.g. `x-on:click="dismiss"`, `x-show="show"`) and class lists.
  Alpine processes `x-data` on dynamically inserted DOM (so toasts appended
  via `hx-swap-oob="beforeend"` are initialised correctly).
- `freedom_ls/base/static/base/js/alpine-components.js` — extend the
  existing `htmx:beforeSwap` handler to honour OOB messages fragments on
  non-2xx responses so toasts surface on HTMX error paths (see HTMX
  delivery section).
- A new Django middleware (likely in `freedom_ls/base/middleware.py` or a
  new module) that injects an OOB messages fragment into HTMX responses.
- Settings: register the middleware in `config/settings.py`.
- Tests: pytest unit tests for the middleware (HTMX detection, OOB fragment
  inclusion, severity routing); Playwright E2E for the full toast flow at
  three viewport widths.

## Success criteria

A reviewer can sign this off when:

1. **Position.** Toasts render bottom-right on desktop (>=md) and bottom-edge
   full-width on mobile (<md). At no breakpoint do toasts overlap any page
   heading, sticky header, or the iOS home indicator (visible at
   `viewport-fit=cover` on a notched device).
2. **Live regions.** The base layout contains exactly one polite and one
   assertive live region; both are present at first paint with empty bodies;
   each toast carries `aria-atomic="true"` on itself, never on the container.
3. **Severity routing.** Success / info / warning / debug toasts append into
   the polite region; error toasts append into the assertive region. A
   screen reader test (NVDA + Firefox or VoiceOver + Safari) confirms a
   success message announces without interrupting and an error message
   interrupts.
4. **Auto-dismiss timing.** Success/info/debug dismiss after 5s, warnings
   after 7s, errors do not auto-dismiss. Hovering a toast pauses its timer;
   moving the pointer away resumes from the *remaining* time, not zero.
   Same for keyboard focus and window blur.
5. **Manual dismiss.** Every toast has a visible close button with
   `aria-label="Dismiss notification"`. Clicking it removes the toast and
   clears its timer. Pressing Esc while keyboard focus is inside the toast
   dismisses that toast.
6. **HTMX delivery.** A view that calls `messages.success(...)` and returns
   an HTMX partial response causes a toast to appear without a full-page
   reload. A view that calls `messages.error(...)` with a 4xx response also
   surfaces the toast. No duplicate toast announcements (no double-ARIA
   announcements). No `HX-Trigger`-based message path is in use anywhere.
7. **Full-page delivery.** Login redirects and non-HTMX form posts continue
   to render messages on the next page load with the same visual treatment
   and the same live-region routing.
8. **Stacking.** Five concurrent toasts display simultaneously without
   overlap. A sixth incoming non-error toast evicts the oldest non-error
   toast. Mid-stack dismissal animates neighbours into the gap via
   `transform` only.
9. **Animation.** Enter/exit animations use only `transform` and `opacity`.
   Under `prefers-reduced-motion: reduce`, no translate distance is applied.
10. **Pointer events.** Clicks on page content located *between* visible
    toasts (i.e. through the gaps) reach the underlying page; only the
    toast bodies themselves capture pointer events.
11. **Visual regression.** Before/after Playwright screenshots are captured
    at three widths (e.g. 1440px desktop, 834px tablet, 390px mobile) and
    attached to the QA artifacts. No unintended regressions on adjacent
    page chrome.
12. **No new style assertions in tests.** Per project conventions: tests
    cover behaviour (which region a toast lands in, whether the timer
    pauses, OOB fragment presence), not Tailwind class names.

## Open questions / explicit deferrals for spec review

These are recorded so the review step can either approve the chosen default
or override it:

- **Stack cap of 5 with error-priority eviction.** The cap and eviction rule
  are a guess that errs toward never losing error context. If the typical
  flow only ever shows 1–2 toasts, drop the eviction logic and just use a
  hard cap.
- **Pause on focus.** Included by analogy with research recommendations;
  the idea file only mentions hover and window blur. Pausing on focus is
  required for keyboard-accessible close buttons not to be ripped out from
  under the user, so it's hard to drop.
- **OOB swap chosen over `HX-Trigger`.** Documented above. If the project
  later wants client-fired toasts (e.g. WebSocket push, optimistic UI), an
  additive `HX-Trigger`/event-based path could be layered on without
  disturbing the OOB fragment used here — but only if the duplicate
  announcement pitfall is solved (probably by routing all paths through a
  single `addToast(payload)` helper).
- **5s for success/info/debug, 7s for warning, persistent for error.**
  Taken from the idea file directly.
- **Eviction policy for full-page loads with >5 queued messages.** Django
  can technically queue arbitrarily many messages across requests. If a
  full-page load arrives with 12 queued messages, render all 12 (don't drop
  any during the server-side render), but the same client-side cap evicts
  the oldest non-error ones until 5 visible remain. Errors stay visible.
