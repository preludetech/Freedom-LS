# Spec: Theme & site branding in emails (hardening + unification)

> **Note on this revision.** This spec was updated to match the design as
> implemented. During implementation the failure-handling decision was
> deliberately reversed: instead of *warn + hardcoded hex fallback*, email theme
> values are resolved **fail-loud** from a token map that merges the **default
> theme** under the active theme. The default theme is the single source of
> fallback values, so a token merely omitted by the active theme falls through to
> the default and never errors; an unresolvable required token (e.g. an active
> theme that defines a token with a malformed value, or a missing default
> `theme.css`) raises rather than silently rendering a wrong colour. As part of
> this, the `EMAIL_COLOR_*` / `EMAIL_FONT_FAMILY` settings and the
> `email_settings` context processor were removed in favour of a lazily-cached
> `get_email_theme()` injected per-send. Sections below reflect the shipped
> design; the *Background* section is preserved as the verified pre-work state.

## Summary

The Freedom Learning System already ships a branded transactional-email system
(allauth verification, password reset, login code, and notification mails). Theme
colours are extracted from the active theme's `theme.css` into `EMAIL_COLOR_*`
settings, exposed via the `email_settings` context processor, and inlined into
the email templates by a premailer template tag. The header shows a logo when
`EMAIL_LOGO_STATIC_PATH` is set, otherwise the site name.

This is **not a greenfield build**. It is a hardening + unification pass on the
existing system, in three parts:

1. **Harden theme-colour extraction** so it never silently renders a wrong
   colour and never leaks modern CSS colour syntax into email HTML.
2. **Drive fonts and the button radius from theme tokens** (email-safe subset).
3. **Unify the email logo and label with the existing web-header branding**, and
   fix a real bug where the logo URL is root-relative and never loads in email
   clients.

Theme remains **global** (one `FLS_THEME` env var). This work does **not**
introduce per-site theming, no new model, no migration, and no uploaded logos.

## Background: current state (verified)

Grounded in a codebase audit (`.sdd-work/` during authoring; key facts below).
This describes the **pre-work** state that the changes below replace:

- `parse_tailwind_colors` (`freedom_ls/accounts/email_utils.py`) uses
  `r"--color-([\w-]+):\s*(#[0-9A-Fa-f]{3,8});"` — it matches **only** hex values
  and silently skips any `--color-*` token written as `oklch()`, `color-mix()`,
  `var()`, `rgb()`, or a named colour. It raises `FileNotFoundError` on a missing
  file.
- Seven `EMAIL_COLOR_*` settings are produced in `config/settings_base.py` with
  hardcoded hex fallbacks (`.get("primary", "#2B6CB0")`, etc.). All base tokens in
  both shipped themes (`default`, `first_class`) are plain hex today, so the regex
  works **today** — but the same theme files already use `color-mix()`/`var()` for
  derived tokens, so a future base token in modern syntax would fail silently.
- `EMAIL_FONT_FAMILY` is hardcoded to `"Arial, Helvetica, sans-serif"` — not tied
  to the theme's `--fls-font-sans` (e.g. `first_class` uses `"DM Sans"`).
- `.email-button` in `base_email.html` hardcodes `border-radius: 6px` — not tied
  to `--fls-radius-md` (`0.375rem` default, `0.5rem` first_class).
- `emails/includes/header.html` builds the logo `src` with
  `{% static email_logo_static_path %}`, yielding a root-relative `/static/...`
  path that does **not load** in external email clients. The `<img>` has no
  `width`/`height` attributes. Alt text and the text fallback use
  `{{ current_site.name }}` directly, skipping `HEADER_TITLE`.
- `EMAIL_LOGO_STATIC_PATH` (default `None`) is entirely separate from the web
  header's `HEADER_LOGO_STATIC_PATH`; in dev the web header shows a logo while the
  email header shows text.
- `email_settings` context processor (`freedom_ls/accounts/context_processors.py`)
  injects `email_color_*`, `email_font_family`, `email_logo_static_path`. It has
  **no** access to `current_site`. The web header's `site_config` processor
  computes `header_title = settings.HEADER_TITLE or site_title`.
- `AccountAdapter.send_mail` (`freedom_ls/accounts/allauth_account_adapter.py`)
  builds `ctx` with `current_site = get_current_site(request)`; `request` (from
  `allauth.core.context`) **can be `None`** outside a request cycle.
  `ACCOUNT_DEFAULT_HTTP_PROTOCOL = "https"` is set only in `settings_prod.py`.
- Neither `coloraide` nor `tinycss2` is in `pyproject.toml` / `uv.lock`.
- `freedom_ls/accounts/checks.py` has an existing system-check pattern (deferred
  DB imports, catch `DatabaseError`, return `list[Warning]`, namespaced IDs).
- Tests: `test_email_utils.py` covers `parse_tailwind_colors`;
  `test_email_templates.py` uses `render_to_string`.

## Goals / Non-goals

### Goals

- Theme-colour extraction always resolves every email colour token to `#rrggbb`,
  converting modern CSS colour syntax in Python before it reaches email HTML.
- There are **no hardcoded colour/font/radius fallbacks**: the **default theme's
  `theme.css` is the single source of default values**, merged under the active
  theme. A token omitted by the active theme falls through to the default; an
  unresolvable required token **fails loud** (`EmailThemeError`/`ColorResolveError`)
  rather than silently rendering a wrong colour, and the same condition is reported
  at deploy/startup by a system check.
- The email font stack and button radius derive from theme tokens, reduced to an
  email-safe subset.
- The email logo and label are unified with the web-header settings, the logo URL
  is absolute (loads in real clients), and there is always something branded
  visible (logo image, else text label, else `Site.name`).

### Non-goals

- No per-site theme/branding model, DB fields, migration, or uploaded logos.
- No per-site theming — theme stays global via `FLS_THEME`.
- No VML/Outlook hacks for rounded buttons (Outlook ignoring `border-radius` is
  acceptable degradation).
- No dark-mode logo variants (a transparent PNG is sufficient).
- No change to which email types exist or to the premailer inlining mechanism.

## Decisions

| Decision | Choice | Why |
|---|---|---|
| Colour conversion | **Add `coloraide`** and do full conversion (hex/rgb/hsl/oklch/oklab/lch/lab/`color-mix()`/`var()`/named → hex) | User-selected. Pure-Python (~350KB), no runtime deps beyond Py≥3.10, covers every format that can appear in a Tailwind v4 theme. Robust against future themes, not just today's all-hex base tokens. |
| CSS parsing | **Hardened regex** capturing every `--<name>` token raw, classify in Python; not `tinycss2` | One known `@theme` block with a known property namespace; a widened regex (capture everything up to `;`, classify/resolve in Python) is simpler, independently testable, and adds no parser dependency. |
| Default values | **Merge the default theme under the active theme**; no hardcoded fallbacks | The default theme ships with FLS and defines every email token, so it is the single source of default values. `token_map = {**default, **active}` means active wins per-token and any token the active theme omits falls through to the default — no hardcoded hex to drift from the theme. |
| Failure mode | **Fail loud** — `EmailThemeError`/`ColorResolveError` on an unresolvable required token; surfaced early by a system **check** | Rendering a *wrong* colour silently is worse than failing. The default theme supplies every token, so the only ways to raise are a malformed value explicitly written into a theme, or a missing default `theme.css` (a broken install). The `check_email_colour_tokens` system check re-resolves the same merged map and reports an **Error** at deploy/startup, before any send can raise. (This reverses the earlier *warn + hardcoded fallback* decision.) |
| Resolution timing | **Lazy + cached** `get_email_theme()`, injected per-send in `send_mail` | Resolved the first time an email is rendered and cached for the process lifetime (`functools.cache`), so a bulk send resolves once. Built in `send_mail` (not a global context processor) so the work runs only when mail is actually sent, not on every web request. Tests overriding the theme call `get_email_theme.cache_clear()`. |
| Logo URL composition | **In `AccountAdapter.send_mail`** | User-selected. `current_site` (needed for the domain) is already at hand there. Keeps the template trivial (`{% if email_logo_url %}`). |
| Logo settings precedence | `EMAIL_LOGO_STATIC_PATH or HEADER_LOGO_STATIC_PATH` | Strictly additive: a site that sets a web-header logo automatically gets it in email, while the explicit email setting remains an override/escape-hatch. `None` default keeps existing installs unchanged. |
| Scheme when `request` is `None` | `ACCOUNT_DEFAULT_HTTP_PROTOCOL` (default `"https"`) | Already the single source of truth for email link scheme; works without a live request (management commands, background tasks). Dev mail goes to Mailpit, where scheme is irrelevant. |
| Logo format | **PNG, not SVG** | PNG is universally supported in email; embedded SVG ~40%, linked SVG ~93% but flaky in Gmail for non-Google accounts. |

## Requirements

### 1. Harden theme-colour extraction

1.1. Replace `parse_tailwind_colors` with `parse_tailwind_tokens(css_file_path)
-> dict[str, str]` that captures the **raw** value of **every** `--<name>` token
(everything up to the first `;`), keyed by the full name minus the leading `--`
(e.g. `{"color-primary": "#2B6CB0", "fls-radius-md": "0.375rem"}`). Internal
whitespace in a value (e.g. a multi-line font stack) is collapsed to single
spaces. The last declaration of a repeated token wins. Raises `FileNotFoundError`
on a missing file.

1.2. Add a pure helper `resolve_css_color(raw: str, token_map: dict[str, str])
-> str` that:
- Resolves `var(--x)` references against the token map by iterative substitution,
  detecting reference cycles and an excessive substitution depth and raising
  `ColorResolveError`. The same variable reused in sibling positions is **not** a
  cycle.
- Parses `color-mix(in <space>, A [pct], B [pct])` via a purpose-built parser and
  delegates the mix to `coloraide` (`Color(a).mix(b, space=..., weight=...)`),
  since coloraide does not parse `color-mix()` strings directly. Two percentages
  that do not sum to 100 are normalised; the spec's transparency-on-shortfall rule
  is intentionally simplified to full opacity (an opaque result is required
  anyway), and two zero percentages raise.
- Delegates every other format (hex 3/4/6/8-digit, `rgb()`/`rgba()`, `hsl()`,
  `oklch()`, `oklab()`, `lch()`/`lab()`, named colours) to `coloraide`.
- **Always returns opaque 6-digit `#rrggbb`.** An 8-digit hex has its alpha
  dropped to sRGB; a non-opaque result is rejected (email clients do not reliably
  support `#rrggbbaa`). Never returns `oklch()`/`color-mix()`/`var()`/`rgb()` —
  Gmail strips the entire inline style attribute on unsupported syntax.

1.3. Define `EMAIL_COLOR_ROLES` — the single source of truth mapping each email
colour role to the `EmailTheme` field it backs — and `resolve_color_token(
token_map, token) -> str`, which looks up `--color-<token>` and resolves it via
`resolve_css_color`. There is **no fallback**: a token absent from the map or with
an unresolvable value raises `ColorResolveError`. The nine roles are `primary`,
`on-primary`, `on-surface` (→ `color_foreground`), `muted`, `surface`,
`surface-2`, `border`, `header`, `on-header`. The header has its own role so a
theme can paint the header band independently of `primary`; the default theme
aliases `--color-header`/`--color-on-header` to `var(--color-primary)` /
`var(--color-on-primary)` (e.g. `first_class` paints a white header instead).

1.4. `get_email_theme() -> EmailTheme` (a frozen dataclass; `functools.cache`)
parses the **default theme's** `theme.css` as a baseline and layers the **active
theme's** tokens on top (`{**default, **active}`, active wins per-token), then
resolves all nine colour roles plus the font and radius. A missing **active**
`theme.css` is fine — `FileNotFoundError` there falls back to an empty active map
so the default baseline supplies every token. A missing **default** `theme.css`
(`FileNotFoundError`) propagates — a broken install must fail loud. Any required
token unresolvable even from the default raises `EmailThemeError` /
`ColorResolveError`. The default and active CSS paths are derived from settings by
`default_theme_css_path()` (via `FLS_THEMES_DIRS`, honouring a downstream shadow
of `default/`) and `active_theme_css_path()` (via `RESOLVED_THEME_DIR` +
`FLS_THEME`).

1.5. Remove the `EMAIL_COLOR_*` settings, the `EMAIL_FONT_FAMILY` setting, and the
`email_settings` context processor. Email theme values are injected per-send in
`AccountAdapter.send_mail` via `asdict(get_email_theme())`, so the template
context keys are the `EmailTheme` field names (`color_primary`, `color_foreground`,
`color_muted`, `color_surface`, `color_surface_2`, `color_on_primary`,
`color_border`, `color_header`, `color_on_header`, `font_family`,
`button_radius`).

### 2. Fonts & button radius from theme tokens

2.1. `parse_tailwind_tokens` already captures `--fls-font-sans` and
`--fls-radius-md` (it captures every `--<name>` token); radius/font values need no
colour conversion.

2.2. `extract_font_family(token_map) -> str` returns
`email_safe_font_stack(token_map["fls-font-sans"])`, and raises `EmailThemeError`
when `--fls-font-sans` is absent (the default theme always defines it).
`email_safe_font_stack` reduces a stack to an email-safe subset:
- Drop CSS keywords that are not real font names: `system-ui`, `ui-sans-serif`,
  `ui-monospace`, `-apple-system`, and any other `ui-*` keyword.
- Drop custom/web-font names not on a conservative web-safe allowlist (Arial,
  Helvetica, "Helvetica Neue", Verdana, Georgia, "Times New Roman", "Courier New",
  "Trebuchet MS", Tahoma); allowlist matching is case-insensitive and multi-word
  names are re-quoted.
- Keep (or append) the trailing generic family (`sans-serif`/`serif`/`monospace`).
- If stripping would leave only a generic family but a custom name was present,
  emit a `warnings.warn(UserWarning)` (a custom font alone won't render in most
  email clients).
- Examples: default theme → `"Helvetica Neue", Arial, sans-serif`; `first_class`
  (`"DM Sans", system-ui, sans-serif`) → `sans-serif` (+ warning).
- The resolved `font_family` is rendered with `|safe` in `base_email.html` so the
  quotes around multi-word names survive premailer's CSS parsing. This is only safe
  because every emitted name comes from the fixed allowlist or the fixed generic
  families — the output must stay allowlist-constrained.

2.3. `extract_button_radius(token_map) -> str` returns `--fls-radius-md` when it is
a **bare CSS length literal** (a `0`, or a number with a `px`/`rem`/`em`/`%` unit),
and raises `EmailThemeError` when the token is absent or its value is not a plain
length. The value is interpolated into the email `<style>` block, so constraining
it to a length keeps a malformed theme value from injecting CSS (mirroring the
`#rrggbb` colour constraint). The resolved radius is exposed in the template as
`button_radius` and used in `.email-button` (and each CTA button) in place of the
hardcoded `border-radius: 6px`. Outlook ignoring it is acceptable; no VML
workaround. Button background/text already come from theme tokens — unchanged.

### 3. Unify logo & label, fix the absolute-URL bug

3.1. **Logo path precedence**: `resolved_email_logo_path()` returns
`settings.EMAIL_LOGO_STATIC_PATH or settings.HEADER_LOGO_STATIC_PATH or None`.

3.2. **Absolute logo URL** (the real bug fix): `AccountAdapter.send_mail` composes
`email_logo_url` and injects it into the email context. `_resolve_email_logo_url`:
- Resolves the static URL via `static(logo_path)`; under
  `ManifestStaticFilesStorage` a missing asset raises `ValueError`, which is caught
  and degrades to `None` (the branded logo is best-effort and must not abort a
  transactional email).
- If the static URL is already absolute (`http://`/`https://`, e.g. a CDN), it is
  used verbatim (no double-prefixing).
- Otherwise, when a `request` is present, the request-based absolute URI is used
  (same builder as allauth's action links, so the logo resolves wherever the mail
  was triggered).
- With no request (mail sent outside a web request), it falls back to
  `f"{protocol}://{current_site.domain}{static_url}"` where `protocol =
  getattr(settings, "ACCOUNT_DEFAULT_HTTP_PROTOCOL", "https")`.
- The template uses `{% if email_logo_url %}<img src="{{ email_logo_url }}" ...>`,
  removing the broken `{% static email_logo_static_path %}` usage.

3.3. **Label precedence**: `email_label = settings.HEADER_TITLE or
current_site.name`, injected by `send_mail`. Used for the `<img alt>`, the `<h1>`
text fallback (HTML header), and the plain-text base template's site identity.
Templates reference it as `{{ email_label|default:current_site.name }}` so the
site name remains the final fallback when the label is absent from context (e.g.
templates rendered directly in tests).

3.4. **Logo dimensions**: `image_dimensions(path)` reads the intrinsic
`(width, height)` from a PNG/GIF/JPEG header using only the stdlib (returns `None`
for an unreadable/unsupported file). `email_logo_dimensions(logo_static_path)`
(`functools.cache`) locates the static file, reads its intrinsic size, and scales
it to a fixed display height of `EMAIL_LOGO_DISPLAY_HEIGHT` (48px), preserving
aspect ratio; it returns `None` when the file can't be located or measured.
`send_mail` injects `email_logo_width` / `email_logo_height`.

3.5. **Header markup** (`emails/includes/header.html`): the band background uses
`{{ color_header }}` and the text fallback uses `{{ color_on_header }}`. When
`email_logo_url` is set and dimensions are known, render `<img>` with the absolute
`src`, `alt="{{ email_label|default:current_site.name }}"`, explicit `width`/
`height` attributes and matching inline `style` (no stretch-prone `max-height`).
When dimensions are unknown, fall back to a height-only constraint
(`height="48"`, `width: auto`). When `email_logo_url` is absent, render the `<h1>`
text label.

3.6. **Plain-text** (`emails/base_email.txt`, sign-off, and any greeting using the
site name): use `{{ email_label|default:current_site.name }}` where the site label
is shown. No image references in plain text. `current_site.domain` stays as-is in
the footer line.

3.7. **Asset**: a PNG logo with a transparent background is used (rasterised from
SVG if needed). The web header may keep using SVG; only the email asset must be
PNG. No new upload mechanism — this is a static asset referenced by the existing
`*_STATIC_PATH` settings.

### 4. Dependency & checks

4.1. Add `coloraide` via `uv add coloraide`.

4.2. Add a Django system check `check_email_colour_tokens` in
`freedom_ls/accounts/checks.py`, registered under `Tags.compatibility`. It builds
the **same merged token map** `get_email_theme` uses (default baseline + active
overlay) and resolves every email token — the nine colour roles plus the font and
radius — reusing the `email_utils` helpers. Because there is no hardcoded fallback,
an unresolvable token is an **`Error`** (namespaced id `freedom_ls_accounts.E002`):
it surfaces at deploy/startup before any email send can raise. A token merely
absent from the active theme is fine (the default supplies it). The check stays
silent (returns `[]`) when the default `theme.css` cannot be read yet (e.g. a fresh
checkout) rather than crashing. **This check is the primary deploy-time guard for
the fail-loud behaviour — `manage.py check` must run in the deploy pipeline.**

## Testing

Follow project convention: assert on **string content / functionality**, not CSS
classes or inline-style strings. Use `render_to_string` as the existing email
template tests do.

### Token parsing / colour conversion (`test_email_utils.py`)

- `parse_tailwind_tokens` returns **all** `--<name>` tokens (not just `--color-*`)
  keyed by name-minus-`--`, captures non-hex colour values raw (`rgb`, `hsl`,
  `oklch`, `color-mix`, `var`, named), excludes standard (non-custom) properties,
  and collapses multi-line value whitespace. A missing file raises
  `FileNotFoundError`.
- `resolve_css_color` returns `#rrggbb` for hex (3/4/6/8-digit), named, rgb/rgba,
  hsl, oklch, oklab; resolves `var()` chains; raises `ColorResolveError` on an
  unknown var, a cycle (without hanging), and on a `color-mix` whose percentages
  sum to zero. The same var in sibling positions is not a cycle. `color-mix`
  resolves to a hex between its endpoints, including with `var()` operands, and
  normalises two non-100 percentages.
- `resolve_color_token` raises `ColorResolveError` (no silent fallback) on a
  missing token, an unparseable value, or a cyclic `var()`; returns the resolved
  hex for a valid token.
- Back-compat oracle: the real `default` and `first_class` `theme.css` files
  resolve the nine email colour roles to their known hex values; header roles
  resolve as documented (default aliases primary/on-primary; `first_class` header
  is white with a dark on-header).

### Font / radius (`test_email_utils.py`)

- `email_safe_font_stack` produces the expected email-safe stack across mixed /
  custom-only / keyword-only / generic-only inputs (allowlisted multi-word names
  re-quoted, case-insensitive matching), and emits a `UserWarning` when only a
  generic remains. The real `default` / `first_class` font tokens produce the
  documented stacks.
- `extract_font_family` raises `EmailThemeError` when `--fls-font-sans` is absent,
  else delegates to `email_safe_font_stack`.
- `extract_button_radius` returns bare length literals (stripped), raises
  `EmailThemeError` on a missing token and on non-length values (including CSS
  injection attempts like `0} body{display:none`), and yields the documented
  values for the real themes.

### Theme resolution (`test_email_utils.py`)

- `get_email_theme` resolves the active (default) theme; a missing **active**
  `theme.css` falls through to the default's values; a missing **default**
  `theme.css` raises `FileNotFoundError`. `image_dimensions` /
  `email_logo_dimensions` read the bundled PNG's real size, scale to the display
  height, and return `None` for non-images / missing files / unresolvable static
  paths.

### System check (`test_checks.py`)

- All-valid tokens → no errors. An unconvertible token value → one
  `freedom_ls_accounts.E002` Error naming the token; multiple bad tokens → multiple
  Errors. A token absent from the active theme is supplied by the default → no
  error. A missing active `theme.css` resolves entirely from the default → no
  error, no raise. The check is registered with Django.

### send_mail context (`test_allauth_account_adapter.py`, `test_context_processors.py`)

- `send_mail` injects `asdict(get_email_theme())` into the context. `email_logo_url`
  is built from the request host when a request is present, used verbatim when
  `STATIC_URL` is an absolute CDN URL, and degrades to `None` when the static
  lookup raises `ValueError`. `email_label` is `HEADER_TITLE` when set, else the
  site name. The `HEADER_LOGO_STATIC_PATH` fallback applies when
  `EMAIL_LOGO_STATIC_PATH` is unset.

### Logo / label / radius templates (`test_email_templates.py`)

- With `email_logo_url` set: HTML contains `<img>` whose `src` is the fully-
  qualified URL (never a bare `/static/` path), `alt` equals `email_label`, and
  computed `width`/`height` are used (no `max-height`); without dimensions it falls
  back to height-only. With `email_logo_url` `None`/absent: HTML shows the `<h1>`
  text label and no `<img>`. Plain text contains the label and no `/static/` URL.
  Back-compat: without `email_label`, output falls back to `current_site.name`.
- The rendered (premailer-inlined) email carries a `font-family` (regression for
  the autoescape bug where quoted family names were escaped and dropped). CTA
  buttons carry the resolved `button_radius`, never a hardcoded `6px`. Message
  bodies use `email_label` rather than the raw site name.

### Manual QA (Mailpit)

Trigger verification, password-reset, and login-code emails; in Mailpit confirm the
logo loads from the absolute URL (raw view), the alt text shows when images are
off, and colours/font/button radius reflect the active theme.

## Success criteria

1. No email colour token can silently produce the wrong colour: every email colour
   resolves to `#rrggbb` from the merged (default-under-active) token map, a token
   omitted by the active theme falls through to the default, and an unresolvable
   required token fails loud (`EmailThemeError`/`ColorResolveError`) and is reported
   as an `Error` by `check_email_colour_tokens` at deploy/startup. There are no
   hardcoded colour/font/radius fallbacks.
2. No email HTML ever contains `oklch()`, `oklab()`, `color-mix()`, `var()`, or
   `hsl()` in a colour position — every email colour is `#rrggbb`.
3. The email font stack and button radius reflect the active theme, reduced to an
   email-safe subset; `first_class` no longer mismatches web typography beyond the
   email-safe fallback.
4. The email logo loads in external clients (absolute URL), defaults to the
   web-header logo, and the explicit email setting still overrides it.
5. There is always visible branding: logo image if configured, else the text label
   (`HEADER_TITLE → Site.name`); alt text equals that label.
6. `coloraide` is added; the `EMAIL_COLOR_*`/`EMAIL_FONT_FAMILY` settings and the
   `email_settings` context processor are removed in favour of a cached
   `get_email_theme()` injected per-send; tests cover token parsing, colour
   conversion, fail-loud resolution, font stripping, radius validation, the system
   check, and the logo/label/radius template behaviour.
7. No per-site model, migration, or upload is introduced; theme stays global.
