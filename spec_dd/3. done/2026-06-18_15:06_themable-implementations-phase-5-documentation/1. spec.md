# Spec: Themable FLS — documentation (Phase 5 of 5)

## Why

This is **Phase 5** of the multi-phase Themable Freedom LS effort. Phases 1–4 ship the architecture (resolver, default theme, and the `first_class` canary spanning Tier-1 tokens and Tier-2 component-class overrides; Tier-3 template overrides are supported by the resolver but no shipped theme exercises them). This phase makes the architecture **usable by other humans**: install guide refresh, a new end-to-end theming guide, resource-doc updates, and minor housekeeping.

Without Phase 5, downstream consumers can technically theme FLS but have no documented path. The success criterion is **a fresh reader follows `docs/how tos/theme-fls.md` end-to-end and ships a working third theme** without needing to read any of Phases 1–4's commits.

For the holistic five-phase context, see `../themable-implementations-master-decomposed-into-phases/`.

## Scope of this phase

**In:**

- Update `docs/how tos/incorperate into another project.md` to reflect the new `tailwind.input.css` template and the `FLS_THEME` / `FLS_THEMES_DIRS` settings.
- Create `docs/how tos/theme-fls.md` — a complete end-to-end theming walkthrough.
- Update `fls-claude-plugin/resources/templates_and_cotton.md` with a short "Theme overrides" section.
- Update `fls-claude-plugin/resources/frontend_styling.md` to point token references at the default theme's `theme.css` and add the new role-token list.
- Update `fls-claude-plugin/resources/email_templates.md` so the email-colour stanza matches the code: colours are parsed from the active theme's `theme.css` (not `tailwind.components.css`), and the foreground value comes from the `on-surface` token (not the deleted `--color-foreground`).
- Add brand names introduced by the documentation (`DM Sans`, `Stratosphere`, `Cockpit`, `Altitude`, `Indigo`) to `cspell.json` (`Outfit` is already present).
- Fix the stale `theme.md` manifests. `freedom_ls/themes/first_class/theme.md` currently claims "Tier 1 only … (Tier 2 lands in Phase 3, Tier 3 in Phase 4)", but Tier-2 component-class overrides have shipped in its `theme.css`; correct it to reflect Tier-1 + Tier-2. Verify `freedom_ls/themes/default/theme.md` is accurate (it should remain Tier-1 only).
- Final repo grep to confirm no stale references to deleted/renamed tokens. Use the scoped utility-class patterns from the QA plan — `(text|bg|border|chip|btn)-(foreground|danger)` and `(text|bg)-(primary|success|danger)-bold` — **not** bare substrings, which would collide with the legitimate `font-bold` utility, iconify `-bold` icon variants, and the email parser's test fixtures.

**Out of this phase:**

- Any architectural change. This phase is documentation-only.
- Any new theme or template work — `default` and `first_class` are the canonical references.

**Phases 1–4 are hard prerequisites.**

## Goals & success criteria

1. **A reader can follow `docs/how tos/theme-fls.md` end-to-end and ship a working third theme.** This is verified in QA by actually doing it.
2. **The install guide reflects the new `tailwind.input.css` template + `FLS_THEME` settings.** A first-time consumer can install FLS, set `FLS_THEME=default`, and have a working site.
3. **Resource docs accurately describe where defaults live (main project) and how themes override (sparse).** `templates_and_cotton.md`, `frontend_styling.md`, and `email_templates.md` are aligned with the new architecture.
4. **`cspell.json` accepts the new brand names** introduced anywhere in the repo (theme manifests, docs).
5. **No remaining references to deleted or renamed tokens** in the repo. The scoped QA greps — `(text|bg|border|chip|btn)-(foreground|danger)` and `(text|bg)-(primary|success|danger)-bold` — return empty within `freedom_ls/`, `docs/`, and `fls-claude-plugin/`. (A naive `-bold`/`-foreground`/`-danger` substring grep is *not* the criterion: it matches the `font-bold` utility, iconify icon variants, and parser test fixtures, none of which are stale tokens.)
6. **`uv run pytest`, `uv run python manage.py check`, `npm run tailwind_build`** clean under both `default` and `first_class`.

## Documentation deliverables

### `docs/how tos/incorperate into another project.md` (updated)

- Drop "copy `tailwind.input.css` and edit it"; replace with "copy the documented downstream `tailwind.input.css` template" (block from master spec §"Tailwind build ownership", with `<theme_slug>` placeholder).
- Add the `FLS_THEME` / `FLS_THEMES_DIRS` settings stanza with a default-theme example.
- One sentence: "for theming beyond the default, see `docs/how tos/theme-fls.md`".
- Note the `@source` `.gitignore` pitfall (one-liner pointing to the new theming guide for the workaround details).

### `docs/how tos/theme-fls.md` (new)

Cover everything in master spec §"Install guide update + new theming guide". Suggested sections:

1. The three tiers (tokens → component classes → template overrides) and when to reach for each.
2. Theme directory shape (sparse — only `theme.css` is required) and the `theme.md` manifest.
3. `FLS_THEME` / `FLS_THEMES_DIRS` resolution rules; failure modes.
4. Token contract reference (full list with default values, `on-*` pairs, shape, type).
5. Tier-2: re-opening component classes in `@layer components`. Note that defaults live in `tailwind.components.css` in the FLS main project.
6. Tier-3: cotton / partial / `_base.html` overrides; the `<c-vars>` contract. Note that defaults live in their owning app, NOT in a default theme.
7. Build pitfalls: `@source` and `.gitignore` (with the symlink/copy workaround); new utility classes that need to be in the `@source` glob.
8. Worked examples:
   - Minimal Tier-1-only theme (just `theme.css` with token redeclares).
   - Pointer to `freedom_ls/themes/first_class/` as the Tier-1 + Tier-2 reference (token redeclares in `theme.css` plus an `@layer components` block re-opening the default-contract classes). No shipped theme uses Tier 3, so demonstrate Tier-3 template overrides with a self-contained inline snippet (e.g. shadowing a cotton component or `_base.html` block), not a pointer to first_class.

Adhere to `fls:brand-guidelines` for tone and naming.

### `fls-claude-plugin/resources/templates_and_cotton.md` (updated)

- Keep the existing convention: generic primitives in `freedom_ls/base/templates/cotton/`, app-owned primitives in their owning app's `templates/cotton/`.
- Add a short "Theme overrides" section: drop a same-named file in `themes/<slug>/templates/cotton/` to shadow either kind. Themes are sparse — they only override what they want to change.

### `fls-claude-plugin/resources/frontend_styling.md` (updated)

- The doc currently references only component classes (in `tailwind.components.css`) and has no token section. Add one: point token references at `freedom_ls/themes/default/static/themes/default/theme.css`. (If any token reference pointing at `tailwind.components.css` is present, repoint it to `theme.css`.)
- Keep component-class references pointing at `tailwind.components.css` — that's where they live.
- Add the role-token list (`primary`, `secondary`, `accent`, `success`, `warning`, `error`, `info`, `surface`, `surface-2`, `on-surface`, `on-*` pairs, `border`, `muted`, `focus-ring`) and the rule "use `text-on-X` whenever a coloured background is set".
- Note that hover behaviour is `hover:brightness-110` rather than `*-bold` tokens.

These edits are mechanical — keyword swaps and one new section. The skills under `fls-claude-plugin/skills/frontend-styling/SKILL.md` and `…/template/SKILL.md` are thin and reference these resource docs, so updating the resources fixes both surfaces.

### `fls-claude-plugin/resources/email_templates.md` (updated)

The email-colour stanza is stale after Phases 1–4. The code in `config/settings_base.py` now parses colours from the active theme's `theme.css` (`RESOLVED_THEME_DIR/static/themes/<slug>/theme.css`), and `EMAIL_COLOR_FOREGROUND` now derives from the `on-surface` token. Update the doc to match:

- Change "parsed from `tailwind.components.css`" to the active theme's `theme.css`.
- Replace the `--color-foreground` references with `on-surface` (the `EMAIL_COLOR_FOREGROUND` *Python* name is unchanged — only the looked-up token key changed).
- Keep `--color-primary`, `--color-muted`, `--color-surface`, etc. references but anchor them to `theme.css` rather than the components file.

This is a keyword/path swap only; the email contract itself is unchanged.
