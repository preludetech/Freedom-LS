# Frontend QA: Themable FLS — documentation (Phase 5)

The headline success criterion is: **a fresh reader follows
`docs/how tos/theme-fls.md` end-to-end and ships a working third theme** — and
the resource docs accurately describe the architecture. This QA validates that by
*actually doing it*: we follow the new guide to build a brand-new third theme,
confirm it renders in the browser, and confirm `default` and `first_class` still
work. Treat the guide as the source of truth — if a step here can't be performed
because the guide doesn't explain it, that's a documentation bug to report.

## Prerequisites & setup

All commands run from the repo root
(`…/themable-implementations-phase-5-documentation`).

1. **Pick a random port** (do not use the Django default):
   ```bash
   PORT=$(.claude/fls/scripts/find_available_port.sh)
   ```
   Base URL for everything below: `http://127.0.0.1:$PORT`.

2. **Test data:** the DemoDev demo dataset must exist. In dev,
   `config/settings_dev.py` sets `FORCE_SITE_NAME = "DemoDev"`, so the site is
   forced to DemoDev regardless of host/port — the random port is fine. If the
   demo data is missing, it is seeded by
   `freedom_ls/student_management/management/commands/create_demo_data.py`
   (the QA data helper will create it if needed).

3. **Login facts (passwordless magic-link via Mailpit):**
   - Login page: `http://127.0.0.1:$PORT/accounts/login/`
   - Demo student: `demodev_s1@email.com`
   - Dev mail inbox (Mailpit): `http://localhost:8025` — magic links land here.
   - Flow: submit the email on the login page → open the newest message in
     Mailpit → click the sign-in / confirm link → you are authenticated.

> Note on rebuilds: theme CSS only changes after a build. `npm run tailwind_build`
> runs two steps — it regenerates `tailwind.active_theme.css` from the current
> `FLS_THEME` (via `manage.py write_active_theme_css`) and then compiles
> `tailwind.input.css` → `static/vendor/tailwind.output.css`. **After every
> `FLS_THEME` change you must re-run `npm run tailwind_build`**, then hard-refresh
> the browser.

## Test 1 — Baseline renders under `default`

**Goal:** establish the "before" look so theme switches are visually obvious.

1. Build and run under the default theme:
   ```bash
   export FLS_THEME=default
   npm run tailwind_build
   uv run python manage.py runserver $PORT
   ```
2. Visit `http://127.0.0.1:$PORT/accounts/login/`.
   - **Expect:** the page renders with styling (header, a primary-coloured
     button, card/surface backgrounds, the default sans font). Default primary is
     a mid blue (`#2B6CB0`). No unstyled / flash-of-raw-HTML.
3. Log in as `demodev_s1@email.com` (magic link via Mailpit), then visit:
   - `http://127.0.0.1:$PORT/` (dashboard)
   - `http://127.0.0.1:$PORT/courses/` (course catalogue — shows course cards,
     chips, accent gradients)
   - a course detail page: `http://127.0.0.1:$PORT/courses/<course_slug>/detail/`
     (pick any course slug visible in the catalogue)
   - **Expect:** consistent default styling across all four pages — blue primary,
     white/light surfaces, status chips legible, header bar styled.

## Test 2 — Build a brand-new third theme by following the guide

**Goal:** the core success criterion. Following `docs/how tos/theme-fls.md` only,
create a **minimal Tier-1 theme** with deliberately loud, unmistakable colours so
the switch is impossible to miss. Use slug `qa_canary`.

1. Following the guide's "theme directory shape" + "minimal Tier-1 theme" worked
   example, create:
   ```
   freedom_ls/themes/qa_canary/theme.md
   freedom_ls/themes/qa_canary/static/themes/qa_canary/theme.css
   ```
   `theme.css` should redeclare a handful of role tokens with vivid values, e.g.:
   ```css
   :root {
     --color-primary: #C026D3;      /* loud magenta — unmistakable */
     --color-on-primary: #FFFFFF;
     --color-surface: #FFFBEB;      /* warm cream surface */
     --color-on-surface: #1A2332;
     --color-accent: #16A34A;       /* green accent */
     --color-on-accent: #FFFFFF;
   }
   ```
   (Follow the guide for the exact required tokens / file shape. The point of QA
   is that the guide tells you everything you need — if it doesn't, log the gap.)
   Give `theme.md` a short, accurate manifest matching the guide's example.

2. Switch to it and rebuild:
   ```bash
   export FLS_THEME=qa_canary
   npm run tailwind_build
   ```
   - **Expect:** the build succeeds with no error. `tailwind.active_theme.css`
     now `@import`s the `qa_canary` `theme.css`.

3. Restart the server (`uv run python manage.py runserver $PORT`) and hard-refresh
   `http://127.0.0.1:$PORT/accounts/login/`.
   - **Expect:** the primary button / accents now render **magenta**, surfaces
     are warm cream — clearly different from Test 1. The page is fully styled (no
     missing CSS), proving the new theme is wired end-to-end purely by following
     the guide.

4. Log in and re-check `/`, `/courses/`, and a course detail page.
   - **Expect:** the new primary/surface/accent tokens propagate everywhere —
     buttons, links, course cards, chips, header. Text on coloured backgrounds
     stays legible (the `text-on-X` rule holds).

5. **Failure-mode check (guide §resolution):** stop the server, set
   `export FLS_THEME=does_not_exist`, and run `uv run python manage.py check`.
   - **Expect:** an `ImproperlyConfigured` error naming the bad slug and the
     searched theme dirs — matching what the guide documents. Reset with
     `export FLS_THEME=qa_canary` (or `default`) afterwards.

## Test 3 — Tier-2 component override (as the guide documents)

**Goal:** confirm the guide's Tier-2 instructions actually work.

1. Following the guide's "Tier 2 — re-opening component classes" section, add an
   `@layer components` block to `qa_canary`'s `theme.css` that re-opens one
   class, e.g.:
   ```css
   @layer components {
     .btn { @apply rounded-pill px-8; }   /* pill-shaped, wider buttons */
   }
   ```
2. Rebuild (`npm run tailwind_build`) and hard-refresh the login page.
   - **Expect:** primary buttons are now pill-shaped and wider, proving a theme
     can override the `tailwind.components.css` defaults exactly as the guide says
     (cascade: default tokens → components → active theme wins).

## Test 5 — Email colours track the active theme (validates email_templates.md)

**Goal:** confirm the doc's corrected claim — email colours come from the active
theme's `theme.css` (`on-surface` for the foreground), not the old
`tailwind.components.css` / `--color-foreground`.

1. With `FLS_THEME=qa_canary` built and the server running, trigger a
   theme-coloured transactional email — e.g. start the magic-link login again for
   `demodev_s1@email.com`, or any flow that sends an FLS HTML email.
2. Open that message in Mailpit (`http://localhost:8025`).
   - **Expect:** the email's branded colours reflect the **active** theme
     (`qa_canary`'s magenta primary / cream surface), confirming the parser reads
     the resolved theme's `theme.css`. Switching `FLS_THEME` back to `default`,
     rebuilding, and re-sending should show the blue default palette instead.

## Cleanup

After QA, remove the throwaway theme and reset to default so the worktree is
clean:

```bash
rm -rf freedom_ls/themes/qa_canary
export FLS_THEME=default
npm run tailwind_build
```

The `qa_canary` theme is a QA scratch artifact only — it must **not** be
committed. (`default` and `first_class` remain the only shipped themes.)

## What counts as a pass

- Every page in Tests 1–4 is fully styled under each theme, with the expected
  palette, and switching `FLS_THEME` + rebuilding visibly changes the look.
- The third theme (`qa_canary`) was created using **only** the instructions in
  `docs/how tos/theme-fls.md` — any step that required guessing beyond the guide
  is a documentation bug to report.
- The unknown-slug failure mode matches the documented `ImproperlyConfigured`
  behaviour.
- (Test 5) email colours follow the active theme.
