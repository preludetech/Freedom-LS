# Spec: External requirements config — third-party services for concrete deployment

## Summary

Concrete (submodule-based) FLS production/staging deployments depend on three externally-hosted
services that need actual code and settings inside `freedom_ls`: **PostHog** (analytics),
**Sentry** (error tracking), and **Cloudflare R2** (media object storage). Today this wiring is
partial, crude, or subtly wrong for R2. This spec establishes a single, convention-following home
for that config so a downstream concrete project only supplies environment variables — it never
copy-pastes settings.

The work introduces a new **`freedom_ls/deployment`** app that owns the service config via a
`config.py` following the existing `app-settings` pattern, plus a Sentry init hook, an R2
correctness fix in `settings_prod.py`, a PostHog context-processor refactor, a staff-gated
Sentry verification endpoint, and a consolidated env-var checklist.

## Context and relationship to the parent effort

This spec supports the parent effort in `spec_dd/2. in progress/support-concrete-project-deployment/`,
which was decomposed into five numbered specs (see its `spec-order.md`). That effort establishes an
importable `freedom_ls/deployment/` home for deployment primitives (Spec 1's `settings_defaults.py`)
and catalogues every external account a deployment needs in `third-party-services.md`.

**This spec owns the in-repo *config wiring* for the three services that need code/settings inside
`freedom_ls`.** It does *not* touch the template-repo scaffolding (Spec 5) or ops-only credentials
(GHCR, VPS/SSH, off-box backups).

### App-scaffolding boundary (decided)

The parent's **Spec 1** was intended to create the `freedom_ls/deployment` app (with
`settings_defaults.py`), but Spec 1 has not shipped and the app directory does not exist yet
(confirmed: no `freedom_ls/deployment/` directory, no files). **This spec therefore creates the
`deployment` app scaffolding itself, idempotently**: `freedom_ls/deployment/__init__.py`,
`freedom_ls/deployment/apps.py`, and the `INSTALLED_APPS` registration in `config/settings_base.py`.
This keeps the spec independently implementable now. If Spec 1 later also creates the app, the
overlap is trivial (both only need the app to exist) and reconciles by keeping one copy of the
scaffolding. This spec does **not** create `settings_defaults.py` — that remains Spec 1's artifact.

## Goals

- A downstream deployment configures PostHog, Sentry, and R2 **entirely through environment
  variables**, with sensible in-repo defaults and no settings copy-paste.
- Each service's config lives in one authoritative place (`deployment/config.py`) that owns
  defaults, optionality, and a single resolution path — following the existing `AppSettings`
  pattern, not ad-hoc `os.environ` reads.
- R2 works correctly (no ACL landmines, checksum-compatible, `region_name="auto"`) and media is
  **private by default** so application-gated course files cannot leak.
- Sentry is a no-op until a DSN is supplied, initialises the FLS-recommended way (an
  `AppConfig.ready()` hook, not the settings file), and ships a safe, staff-only verification
  endpoint.
- One consolidated env-var / services checklist documents the contract, marking each var
  secret vs config with its default.

## Non-goals (out of scope)

- **Template-repo scaffolding** (Dockerfile/compose/Caddy/CI/Ansible/`.env.example` in the template
  repo) — that is the parent's **Spec 5**, targeting the separate `freedom-ls-concrete-template`
  repo.
- **Ops-only credentials** with no `freedom_ls` settings surface — GHCR tokens, VPS/SSH keys,
  off-box backup creds — catalogued in `third-party-services.md`, consumed by Spec 5.
- **`settings_defaults.py`** — Spec 1's artifact; this spec sits alongside it, not on top of it.
- **Sentry PII scrubbing** (`before_send` redaction). Deferred to a separate spec
  (`sentry-pii-scrubbing`, currently in `0. drafts`). This spec only exposes
  `SENTRY_SEND_DEFAULT_PII`; it builds no scrubbing.
- **Server-side `posthog-python` SDK, PostHog reverse proxy, Sentry profiling** — legitimate future
  work; do not build speculatively, but do not pick names that would collide.
- **Media view-proxy hard gate** — serving every media byte through a Django view that re-runs
  `can_access_content` per request. Strictly stronger than signed URLs but a new feature (view + URL,
  permission wiring, changes to all file cotton components, video range-request handling). Deferred
  to its own spec. This spec ships private-by-default signed URLs only.
- **Second (public) media bucket / per-file public flag.** There is no per-file public/private flag
  on the single `content_engine.File` model; routing media across two buckets is a separate, larger
  feature. One private bucket only.

## Design overview

### The `deployment` app and its `config.py`

Create `freedom_ls/deployment/` as a real Django app:

- `freedom_ls/deployment/__init__.py`
- `freedom_ls/deployment/apps.py` — the `AppConfig`, named `DeploymentAppConfig(AppConfig)`, with
  `name = "freedom_ls.deployment"`. `default_auto_field` is omitted (the app has no models of its
  own, matching `freedom_ls/icons`). Its `ready()` hook wires the Sentry init (see below).
- `freedom_ls/deployment/config.py` — the *settings* config, `DeploymentSettings(AppSettings)` (a
  distinct class from the `AppConfig`; kept differently named to avoid confusion). It declares the
  **PostHog and Sentry** settings via `declared_settings = {...}` with `Setting(default=...)`,
  exposes a module-level singleton `config = DeploymentSettings()`, and is **read-only** — it never
  mutates `STORAGES`, `INSTALLED_APPS`, or any Django setting. The R2 settings are **not** declared
  here (see the R2 section — they stay as direct `settings.X` reads in `settings_prod.py` per the
  `app-settings` skill).
- None of the declared settings are `required=True` (every service is optional — unset disables it),
  so this spec adds **no** `checks.py` / required-settings system check. If a future setting must be
  mandatory, wire the check via the shared `required_settings_errors(config, app_label)` helper from
  `base.app_settings`, per the `app-settings` skill.
- Register `"freedom_ls.deployment"` in `INSTALLED_APPS` (`config/settings_base.py`), in the
  `# CUSTOM APPS` block alongside the other `freedom_ls.*` apps.

**Why declare the PostHog and Sentry settings rather than read `settings.X` directly?** The
`app-settings` skill's general rule is that env-derived secrets and third-party settings stay as
direct `settings.X` reads, but it explicitly supports declaring a third-party setting *"only to
record ownership"* (its `COTTON_SNAKE_CASED_NAMES` example). PostHog and Sentry are read at runtime
(context-processor render / `AppConfig.ready()`), when `django.conf.settings` is fully configured, so
routing them through `config` is both feasible and useful: the `deployment` app owns the defaults
(PostHog host, Sentry sample rate), the optionality (which vars gate a feature on/off), and a single
resolution path. The declared defaults are the concrete value of "no copy-paste" — a downstream
project overrides only what it must. **R2 is the exception** (read at settings-load time, a Django
built-in storage setting) — it stays as direct `settings.X` reads, per the skill.

Note on the `AppSettings` resolution model (from `base/app_settings.py`): `config.NAME` reads
`getattr(django.conf.settings, NAME, None)` first (the project's Django-settings value wins), then
falls back to the declared default, then raises `ImproperlyConfigured` lazily if `required=True` and
still unset. So each declared setting must be surfaced onto `django.conf.settings` by the settings
module (via `os.getenv` in `settings_prod.py`/`settings_base.py`) for the project value to take
effect — `config.py` records ownership and defaults; the settings module is still where env vars are
read onto Django settings. Where a setting has a non-`None` declared default and the settings module
does not set it at all, `config.NAME` returns the declared default.

### PostHog — refactor + finish the wiring

Current state (confirmed): `freedom_ls/base/context_processors.py::posthog_config` reads
`os.environ.get("POSTHOG_API_KEY")` directly and returns only the key. The processor is **not
registered** in any `config/` `context_processors` list, and `_base.html` **hardcodes**
`api_host: 'https://eu.i.posthog.com'`. So today PostHog is effectively unwired in the reference
config, and the region/host is not configurable.

Changes:

- **Declare on `deployment.config`:** `POSTHOG_API_KEY` (public project token — safe in client-side
  HTML; `Setting()` optional, unset = disabled), `POSTHOG_API_HOST` (`Setting(default=
  "https://us.i.posthog.com")` — US default, overridable for EU), `POSTHOG_UI_HOST`
  (`Setting()` optional — only for a future reverse proxy).
- **Refactor the context processor** (it **stays in `base`**, next to `debug_branch_info` — see the
  `base → deployment` decision) to read through `deployment.config` instead of `os.environ`. It
  returns `posthog_api_key`, `posthog_api_host`, and `posthog_ui_host` (the last only when set). When
  the key is unset it renders nothing (disabled for local/dev).
- **Register the context processor** in the shared `TEMPLATES` `context_processors` list in
  `config/settings_base.py` so it is actually wired in prod (it is registered nowhere today).
- **Update `_base.html`** to read `api_host` from context (`{{ posthog_api_host }}`) instead of the
  hardcoded EU string, and to pass `ui_host` when present.
- **Per-environment:** staging and prod each get their own project token (separate PostHog
  projects). No shared default key is committed anywhere — `POSTHOG_API_KEY` has no default.

### Sentry — new integration

- **Add the dependency** `sentry-sdk[django]` (via `uv add`). `DjangoIntegration` auto-enables; no
  new app-level integration wiring needed beyond `sentry_sdk.init(...)`.
- **Initialise in `DeploymentAppConfig.ready()`**, reading `deployment.config` — *not* in a settings
  file. Sentry's own guidance flags settings-file init as an anti-pattern for layered settings, which
  is exactly FLS's `settings_base → settings_prod` shape. This is the first `ready()` hook in the
  codebase to do third-party SDK init (existing hooks only `import checks`); keep the init in a small
  helper (e.g. `deployment/sentry.py::init_sentry()`) that `ready()` calls, for testability.
- **No-op when `SENTRY_DSN` is unset** — `init_sentry()` returns early if the DSN is falsy, so
  dev/unconfigured deploys do nothing.
- **Config surface** (declared on `deployment.config`):
  - `SENTRY_DSN` — secret; `Setting()` optional; unset = Sentry off.
  - `SENTRY_ENVIRONMENT` — `Setting()`; passed explicitly to `init()`. The SDK silently defaults to
    `"production"` if unset, which is a footgun (staging events would look like prod), so FLS treats
    it as an explicit config value the deployment must supply per environment. It is not `required`
    at the `AppSettings` level (Sentry itself is optional), but when a DSN is set the environment
    should be set too; document this in the checklist.
  - `SENTRY_RELEASE` — `Setting()`; CI-injected git SHA. Auto-detection is unreliable in containers.
    Unset = "no release tag" (SDK default), which is acceptable.
  - `SENTRY_TRACES_SAMPLE_RATE` — `Setting(default=0.1)`; low default for cost control. Env vars are
    strings, so the init helper coerces the value read from `config` to `float` (the `0.1` default
    lives in `config`, not duplicated in the settings module).
  - `SENTRY_SEND_DEFAULT_PII` — `Setting(default=False)`; see the PII decision below. The init helper
    coerces the value read from `config` to `bool`.
- **Ship the official `sentry-debug/` verification endpoint** (`trigger_error` view → `1/0`),
  **gated staff-only**. This is the canonical way to confirm Sentry actually receives events from a
  running deployment. It stays wired in **every** environment (so production wiring can be verified
  where it matters) but is gated behind a staff/superuser check so it is never an unauthenticated
  guaranteed-500. Since no `staff_member_required`/`user_passes_test` convention exists in the
  codebase yet, use Django's stock `django.contrib.admin.views.decorators.staff_member_required` (no
  new dependency) on the view. Mount it via a `deployment/urls.py` (`app_name = "deployment"`,
  view name `trigger_error`, path `sentry-debug/`) `include()`d from `config/urls.py`, following the
  per-app URL convention.

### Cloudflare R2 — fix the existing S3 block for R2 correctness

Current state (confirmed, `config/settings_prod.py`): an S3 block gated on
`AWS_STORAGE_BUCKET_NAME` truthiness, falling back to `FileSystemStorage`. It passes
`default_acl=AWS_DEFAULT_ACL`, has a commented `ACL_OPTIONS` list, no checksum workaround, no
`region_name` default, and no custom-domain / querystring-auth settings. `django-storages[s3]`
(with transitive `boto3`) is already a dependency — no new package needed.

Fixes (the `STORAGES` assembly stays entirely in `settings_prod.py`, reading env vars directly with
inline defaults — R2 settings are **not** routed through `deployment.config`, because they are read
at settings-load time (before `config` can safely resolve) and because the `app-settings` skill
keeps Django built-in / storage / env-derived settings as direct `settings.X` reads):

- **Drop `AWS_DEFAULT_ACL` / `default_acl` entirely** — R2 does not implement S3 ACLs; setting one
  is a no-op at best, an upload error at worst. Remove the `AWS_DEFAULT_ACL` env read, the
  `default_acl` OPTION, and the commented `ACL_OPTIONS` guidance block.
- **Add the mandatory checksum workaround, unconditionally, on the R2/S3 branch.** Recent boto3
  (≥1.35.99) sends checksum headers R2 rejects. Set
  `client_config = Config(request_checksum_calculation="when_required",
  response_checksum_validation="when_required")` (from `botocore.config`) and pass it as the
  `client_config` OPTION. This is a compatibility fix, not a toggle.
- **Default `region_name` to `"auto"`** for R2 — resolve it so deployers do not need to know R2's
  region convention. `settings_prod.py` reads `AWS_S3_REGION_NAME` and falls back to `"auto"` inline
  (the default lives in the storage assembly, alongside the querystring-auth/expire defaults).
- **Private media by default:** set `AWS_QUERYSTRING_AUTH=True` (default) and no `custom_domain`, so
  `file.url` yields time-limited signed URLs. Add `AWS_QUERYSTRING_EXPIRE` (default `3600`s) for the
  signed-URL lifetime. Public serving (`AWS_S3_CUSTOM_DOMAIN` set + `AWS_QUERYSTRING_AUTH=False`,
  edge-cacheable) stays **available** for a deployment that deliberately opts in, but is **off by
  default**. See the R2 media-privacy decision below.
- Keep `signature_version="s3v4"` and the `FileSystemStorage` fallback branch unchanged.
- Do **not** remove the unrelated `# TODO @claude:` background-tasks comment that sits above
  `STORAGES` in the file.

## Decisions and rationale

### `SENTRY_SEND_DEFAULT_PII` — store PII for now, scrubbing deferred

FLS handles **learner PII**, and `send_default_pii=True` attaches user email/username plus full
request bodies to every event. For this spec we accept that trade-off to keep debugging signal
high, and expose `SENTRY_SEND_DEFAULT_PII` as a config var (declared default `False`, so the
deployment must consciously opt in to attaching PII). The `before_send` scrubbing work is deferred
to a separate spec (`sentry-pii-scrubbing`). **This spec only exposes the var; it builds no
scrubbing.**

### R2 media privacy — private bucket + signed URLs by default

FLS gates course *pages* at the view layer but **not the media bytes**: media is only ever rendered
as a direct `file.url` (never streamed through a permission-checked view), so a public bucket would
leak application-gated course files to anyone with the URL. Private bucket + signed URLs
(`AWS_QUERYSTRING_AUTH=True`, no `custom_domain`) gives storage-layer privacy with **no app-code
change** — `file.url` simply yields a time-limited URL. Public custom-domain serving stays available
as an explicit opt-in. The stronger per-request view-proxy gate is deferred (see Non-goals).
Rationale and evidence: `research_private_media_access_control.md`.

### `base → deployment` dependency — allowed (approved edge)

The PostHog context processor stays in `base` (next to `debug_branch_info`) and reads through
`deployment.config`, rather than relocating to `deployment`. Keeping it in `base` keeps all context
processors in one place and avoids rewiring the `TEMPLATES` `context_processors` path. This adds a
`base → deployment` app-graph edge; `deployment` already imports `base` (`AppSettings`/`Setting`
from `base.app_settings`), forming a reciprocal `base ↔ deployment` pair. **This is not a Python
import cycle** — `base`'s package never imports `deployment` at load; only
`base/context_processors.py` (a distinct module) does, lazily at template-render time. The edge is
accepted deliberately: the structure review should treat `base → deployment` as **approved**, not
flag it.

### `sentry-debug/` gating — staff-only, wired in every environment

The endpoint stays wired everywhere (so production Sentry wiring can be verified where it matters)
but is gated behind a staff/superuser check, so it is never an unauthenticated guaranteed-500 for
anonymous users. Chosen over a non-prod-only URL include (can't verify prod) and
manual-removal-after-verification (not repeatable, leaves nothing to re-verify after config changes).

### `SENTRY_ENVIRONMENT` / `SENTRY_RELEASE` source — explicit FLS config, no auto-detection

FLS exposes both as its own explicit `deployment.config` settings; it does not auto-detect. There is
no in-repo signal to reuse (`DJANGO_SETTINGS_MODULE` can't distinguish staging from prod — both use
`settings_prod` — and no git-SHA/release var exists anywhere in `config/` or `freedom_ls/`), and the
git SHA only exists in the CI pipeline (Spec 5 / out of scope). Populating these — mapping the
pipeline's `GITHUB_SHA` and env name onto them — is the deployment's (Spec 5's) job. FLS treats an
unset `SENTRY_RELEASE` as "no release tag" and enforces `SENTRY_ENVIRONMENT` explicitly because the
SDK's silent `"production"` default is a footgun.

## Consolidated env-var / services checklist (deliverable)

A single authoritative list of the env vars these three services introduce, marked **secret vs
config**, with defaults — extending and cross-referencing the parent's `third-party-services.md`
(which omits PostHog entirely and frames R2/Sentry as ops-account setup) so a downstream deployment
has one contract to satisfy. This checklist is a durable artifact of this spec (its home and exact
format — a section in this spec, a standalone doc, or an addition to `third-party-services.md` — is
decided during planning; the content below is authoritative).

| Var | Service | Secret? | Default | Notes |
|---|---|---|---|---|
| `POSTHOG_API_KEY` | PostHog | public token | *(none)* | per-env project token; unset = disabled |
| `POSTHOG_API_HOST` | PostHog | config | `https://us.i.posthog.com` | region host; override for EU |
| `POSTHOG_UI_HOST` | PostHog | config | *(none)* | optional; only for a reverse proxy |
| `SENTRY_DSN` | Sentry | **secret** | *(none)* | unset = Sentry off |
| `SENTRY_ENVIRONMENT` | Sentry | config | *(none)* | `staging`/`production`; set explicitly (SDK default `"production"` is a footgun) |
| `SENTRY_RELEASE` | Sentry | config | *(none)* | CI-injected git SHA; unset = no release tag |
| `SENTRY_TRACES_SAMPLE_RATE` | Sentry | config | `0.1` | low default for cost control |
| `SENTRY_SEND_DEFAULT_PII` | Sentry | config | `False` | attaches learner PII + request bodies when `True`; scrubbing deferred |
| `AWS_STORAGE_BUCKET_NAME` | R2 | config | *(none)* | also the on/off gate for object storage |
| `AWS_S3_ACCESS_KEY_ID` / `AWS_S3_SECRET_ACCESS_KEY` | R2 | **secret** | *(none)* | R2 API token |
| `AWS_S3_ENDPOINT_URL` | R2 | config | *(none)* | `https://<account-id>.r2.cloudflarestorage.com` |
| `AWS_S3_REGION_NAME` | R2 | config | `auto` | R2 region convention |
| `AWS_S3_CUSTOM_DOMAIN` | R2 | config | *(unset)* | only for opt-in public serving |
| `AWS_QUERYSTRING_AUTH` | R2 | config | `True` | private signed URLs; `False` opts into public serving |
| `AWS_QUERYSTRING_EXPIRE` | R2 | config | `3600` | signed-URL lifetime (seconds); raise for long media |
| ~~`AWS_DEFAULT_ACL`~~ | R2 | — | — | **removed** — R2 has no ACLs |

## Success criteria

- **PostHog:** with `POSTHOG_API_KEY` set, the client snippet renders with the configured
  `api_host` (US default, EU-overridable) drawn from context, not a hardcoded string; with the key
  unset, nothing renders. The context processor reads via `deployment.config`, not `os.environ`, and
  is registered in the shared `context_processors` list.
- **Sentry:** with `SENTRY_DSN` unset, `init_sentry()` is a no-op and no SDK network activity
  occurs; with a DSN set, `sentry_sdk.init` is called once from `ready()` with the configured
  environment, release, traces sample rate, and `send_default_pii`. `GET /sentry-debug/` raises a
  500 (→ Sentry) for staff/superusers and is blocked (redirect/403) for anonymous and non-staff
  users.
- **R2:** with `AWS_STORAGE_BUCKET_NAME` set, `STORAGES["default"]` uses `S3Storage` with the
  checksum `Config`, `region_name` defaulting to `"auto"`, **no** `default_acl`, and
  `AWS_QUERYSTRING_AUTH=True` so `file.url` produces a signed, time-limited URL by default; with it
  unset, `FileSystemStorage` is used. Opting into public serving
  (`AWS_S3_CUSTOM_DOMAIN` + `AWS_QUERYSTRING_AUTH=False`) yields unsigned public URLs.
- **App wiring:** `freedom_ls.deployment` is in `INSTALLED_APPS`; `deployment.config` follows the
  `AppSettings` pattern (Django settings value wins, then declared default, then lazy raise on
  `required`); Django system checks pass.
- **Config-as-ownership:** a downstream project configures all three services by supplying only the
  documented env vars — no settings copy-paste — and the consolidated checklist accurately lists
  every var with secret/config classification and default.
- Type hints on all new functions; no `Any`; tests cover the enabled/disabled branch of each service
  and the staff-gating of `sentry-debug/`. `uv run pytest` passes.
