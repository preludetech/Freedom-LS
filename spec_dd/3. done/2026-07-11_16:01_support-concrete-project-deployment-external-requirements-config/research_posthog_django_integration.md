# Research: PostHog integration for FLS (settings convention + staging/prod)

## Summary

FLS currently does client-side (JS snippet) PostHog integration only: a context processor
(`freedom_ls/base/context_processors.py::posthog_config`) reads `POSTHOG_API_KEY` straight from
`os.environ` and hands it to templates, which presumably load the PostHog JS snippet and call
`posthog.init(posthog_api_key)` with no other config. This is a reasonable *approach* (client-side
capture is PostHog's standard recommendation for a Django+HTMX server-rendered app), but the config
surface is incomplete: PostHog Cloud requires a **host** as well as a key (US and EU Cloud use
different hosts), there's no clean "disabled" story beyond "key is falsy", and there's no
per-environment (staging vs prod) separation built in. Moving this onto FLS's settings/`config.py`
machinery is what's needed, not a re-architecture of client- vs server-side.

No server-side SDK (`posthog-python`) usage was found in the codebase; if that's wanted later it's
a separate, additive piece of work (see §6/gotchas), not required to fix the current config gap.

## 1. Client-side (JS snippet) vs server-side (`posthog-python`)

- **Client-side (JS snippet)**: `posthog.init(<project_api_key>, { api_host, ... })` in a `<script>`
  tag, or the loaded via a small init snippet. Needs: the **project API key** (public token,
  `phc_...`) and an **`api_host`** (see §2). This is what FLS already does. It's PostHog's
  standard approach for capturing pageviews/clicks/session replay from a server-rendered site —
  no architectural change needed, just fill in the missing config values.
- **Server-side (`posthog-python`)**: `pip install posthog`; typically wired up in a Django
  `AppConfig.ready()` by setting `posthog.api_key` and `posthog.host`, then calling
  `posthog.capture(...)` from view/business logic. PostHog also ships
  `posthog.integrations.django.PosthogContextMiddleware` (added to `MIDDLEWARE`, placed after
  `AuthenticationMiddleware`) which auto-tags events with session/user/request metadata and can
  capture exceptions (`POSTHOG_MW_CAPTURE_EXCEPTIONS`, `POSTHOG_MW_EXTRA_TAGS`,
  `POSTHOG_MW_REQUEST_FILTER`, `POSTHOG_MW_TAG_MAP`).
- Not currently used in FLS. Not requested by this work — flagged only so config naming doesn't
  collide if it's added later (server-side would reuse the same project token, but is
  conventionally paired with a **personal API key** for read/management operations, which is a
  different secret — see §3).

## 2. `api_host` / region config

PostHog Cloud is regional; the project token alone does not tell the SDK where to send events.

- **US Cloud**: public/event ingestion host `https://us.i.posthog.com`; app/private host
  `https://us.posthog.com`.
- **EU Cloud**: public/event ingestion host `https://eu.i.posthog.com`; app/private host
  `https://eu.posthog.com`.
- `api_host` — where the client library sends events (and reads feature flags, etc). Default for
  most PostHog-provided snippets is `https://us.i.posthog.com` — **do not assume this default is
  correct**; it must match whichever region the project actually lives in.
- `ui_host` — optional, only matters when `api_host` points at a reverse proxy (see §6) rather
  than PostHog directly. It should be set to the real PostHog app host (e.g.
  `https://us.posthog.com`) so in-app links (toolbar, links generated by the SDK back to the
  PostHog UI) resolve correctly instead of pointing at the proxy.
- Practical takeaway: a robust config needs **both** an API key and a host value, not just the key.
  There is no safe universal default for the host — it depends on which PostHog Cloud region (or
  self-hosted instance) the project was created in — so it should be an explicit, documented
  setting with a stated default (e.g. US) that can be overridden per environment/self-host.

## 3. Public (project token) vs secret (personal API key)

- **Project API key / project token** (`phc_...`): used to *initialize* the client and *send*
  events / evaluate feature flags. Safe to expose in HTML/JS — this is what's already in the
  context processor. It cannot read private data.
- **Personal API key** (`phx_...`): tied to a user account, used for reading/managing PostHog data
  via its REST API (e.g. server-side feature-flag evaluation, exporting data, admin automation).
  **Must never be exposed client-side or committed to source** — treat as a secret exactly like a
  DB password, sourced from environment/secret manager only.
- **Project secret API key**: a newer, more scoped alternative to personal API keys for
  server-side use (e.g. `feature_flag:read` scope), hashed at rest, one can issue/rotate per
  deployment — preferable to a personal API key if/when server-side flag evaluation is added,
  since it's scoped and rotatable rather than tied to a human account.
- FLS currently only needs the **project token** (public). If server-side SDK usage is added
  later, that would introduce a genuinely secret credential requiring different handling (secret
  env var, never templated into HTML, never logged).

## 4. Making it optional / disabled

- Current behavior: context processor returns `posthog_api_key: None` when
  `POSTHOG_API_KEY` isn't set in the environment; presumably the template only emits the snippet
  `{% if posthog_api_key %}`. This "falsy key disables it" pattern is fine and standard — the gap
  is only that `api_host` isn't part of the equation at all yet, so even when a key *is* set, the
  snippet has no host to send events to (or silently relies on a hardcoded/wrong default).
- Best practice: treat "PostHog enabled" as "key is configured" — same as today — but the host
  must be resolved from config too (with a documented default) so a configured deployment actually
  sends events. Local/dev with no key configured should keep rendering nothing and doing nothing
  (no snippet, no network calls) — no change needed there, just don't require the host when the
  key is absent.
- PostHog JS also supports `opt_out_capturing_by_default` / `loaded` callback for finer-grained
  runtime opt-out (e.g. per-user consent), which is unrelated to this "is PostHog configured at
  all" gate and out of scope here.

## 5. Staging vs prod

- PostHog's own guidance: use **separate projects** (and therefore separate project tokens) per
  environment (dev/staging/prod), so test traffic doesn't pollute production analytics/session
  replay/feature-flag data. Feature flags can be copied across projects manually; dashboards/
  insights/experiments cannot be synced automatically — each project's config is independent.
  (PostHog has also added an "environments within a project" concept for shared
  dashboards/insights/flags across environment-scoped data, but the top-level recommendation for
  simple setups is still separate projects/tokens per environment.)
- Implication for FLS config: the project token (and potentially the host, if staging/prod ever
  live in different regions/instances) must be **settable per deployment environment**, i.e. an
  env var read per-environment (staging gets its own `POSTHOG_API_KEY`, prod gets its own), not a
  single value baked into shared settings.

## 6. Feature flags / reverse proxy (optional, brief)

- A reverse proxy for the PostHog snippet routes events through the app's own domain (e.g.
  `analytics.<yourdomain>` in Cloudflare) so ad-blockers (which blocklist known PostHog/analytics
  hostnames) don't drop the requests — reportedly recovers 10-30% of otherwise-blocked events.
  Since this deployment is already Cloudflare-fronted, a Cloudflare Worker proxy is a natural fit
  if this becomes a priority.
  Config impact if adopted: `api_host` would point at the proxy subdomain instead of PostHog
  directly, and `ui_host` would need to be set explicitly to the real PostHog app host (see §2) so
  in-app links still resolve. PostHog's managed Cloudflare proxy option is **not HIPAA-compliant**;
  a self-hosted Worker would be needed if that matters.
- This is flagged as optional/future — not required for the current "fix the config surface" work.

## Recommended PostHog config surface for FLS (staging + prod)

| Name | Public / Secret | Notes / default |
|---|---|---|
| `POSTHOG_API_KEY` (or `POSTHOG_PROJECT_API_KEY`) | Public (safe in HTML) | Per-environment project token (`phc_...`). Empty/unset = PostHog disabled entirely (no snippet rendered). No safe cross-environment default — one value per staging/prod deployment, sourced from env var per existing convention. |
| `POSTHOG_API_HOST` | Public | Where events are sent. Needs an explicit default (e.g. `https://us.i.posthog.com`) but must be overridable, since the correct value depends on which PostHog Cloud region (or self-hosted host) the project lives in. Only meaningful when a key is configured. |
| `POSTHOG_UI_HOST` | Public | Optional; only needed if/when a reverse proxy is used for `POSTHOG_API_HOST` (see §6). Default: unset/None until a proxy exists. |
| `POSTHOG_PERSONAL_API_KEY` (future, only if server-side SDK added) | **Secret** | Not currently used by FLS. Would gate any server-side `posthog-python` usage (e.g. `AppConfig.ready()` init, middleware). Must come from env var, never templated into a response, never logged. Consider a scoped "project secret API key" instead of a personal key if only feature-flag evaluation is needed. |

## Gotchas / decisions to make

- **Where config lives**: FLS's `app-settings` convention (`fls-claude-plugin/skills/app-settings/SKILL.md`)
  routes app-owned settings through a per-app `config.py` (`AppSettings`/`Setting`), but explicitly
  carves out "env-derived secrets" and "third-party settings" as staying direct `settings.X` reads.
  `POSTHOG_API_KEY` is exactly an env-derived value — decide whether it (and `POSTHOG_API_HOST`)
  should be: (a) read directly as `settings.POSTHOG_API_KEY`/`settings.POSTHOG_API_HOST` (set in
  `settings.py` from `os.environ`, consistent with the "env secrets bypass config.py" carve-out —
  the minimal fix, keeps `os.environ` reads centralized in `settings.py` instead of scattered in
  `context_processors.py`), or (b) declared via a `config.py` in whichever app owns the context
  processor (`freedom_ls.base`), with `settings.py` still doing the `os.environ` read and
  `config.py` supplying the default host / optionality. Given the convention explicitly says env
  secrets stay as direct settings reads, (a) looks like the better fit — but the *host* isn't a
  secret and has a sensible static default, which is more config.py-shaped. This is the one real
  design decision to make in idea/spec refinement.
- **Host has no universally-safe default** — whatever default is picked (US vs EU) will be wrong
  for some deployments; make sure the default is documented and that overriding it per environment
  is trivial (env var), not a code change.
- **Multiple projects across environments** means `POSTHOG_API_KEY` truly is per-environment data,
  not a single shared setting — make sure staging and prod deploy configs are expected to each set
  their own value, and that there's no "default project token" checked into any settings file.
  Local/dev should have no default key either (privacy) — PostHog stays fully off unless the key
  is deliberately set for that environment.
- **Don't conflate project token with personal/secret API keys** — nothing today needs a personal
  API key; if a future ticket adds server-side capture or flag evaluation, that introduces a real
  secret and should not reuse the `POSTHOG_API_KEY` name or its env-var-sourcing pattern implied
  for a public value.
- Reverse proxy (§6) and server-side SDK (§1) are both legitimate future work but are out of scope
  for "fix the config surface" — don't build them speculatively; just don't pick names that would
  collide with them later.
