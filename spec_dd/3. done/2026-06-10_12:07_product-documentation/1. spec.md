# Spec: Product Documentation for Freedom LS

_Last updated: 2026-06-09_

## 1. Overview

Produce a complete set of **product documentation** for Freedom LS under `docs/product/`, plus a **repeatable SDD mechanism** that keeps those docs current as features ship.

The docs describe **features and capabilities** — what the system can do — not implementation internals and not step-by-step user instructions. They serve two audiences:

1. **Compliance demonstration (primary).** A reviewer can read the docs and map Freedom LS's security and data-handling capabilities to their own framework (POPIA / ISO 27001 / GDPR / SOC 2). The docs state capabilities factually and framework-agnostically; they do **not** make claims against any specific framework's clauses.
2. **Sales support (secondary).** A `## Summary` block at the top of each doc gives sales-friendly capability bullets that can be lifted into decks or questionnaire responses.

All docs are **plain Markdown** — no cotton components, no custom widgets, no fancy rendering. Screenshots are referenced with standard `![](screenshots/…)` syntax.

### Why this matters

- The product has substantial, real security and multi-tenancy capabilities (Argon2, django-axes, SSRF-protected webhooks, site isolation, encrypted secrets, pre-commit security gates) that are currently undocumented and therefore invisible to a compliance reviewer or prospect.
- A single capability-oriented source of truth serves both compliance and sales without maintaining two artefacts (per `research_doc_structure.md` §5).
- Without a wired-in update mechanism, docs rot the moment the next feature ships. The repeatable SDD step keeps documentation honest over time.

## 2. Scope

### In scope

1. Create `docs/product/` with a README index and one Markdown file per product-feature category (§4).
2. A cross-cutting **Security & data handling** doc a reviewer can read end-to-end (§4, §6).
3. Screenshots of visible UI features, captured via Playwright MCP, stored in `docs/product/screenshots/` (§5).
4. A new SDD command `fls-claude-plugin/commands/sdd/update_product_docs.md` that refreshes the docs for a shipped feature (§7).
5. A new documentation step in the SDD todo template (`protected/setup_todo_list.md`), positioned **between QA and Pull request** (§7).
6. An update to `fls-claude-plugin/commands/sdd/README.md` describing the new step (§7).

### Out of scope

- A user manual / how-to-click-things guide.
- Framework-specific compliance claims or control mappings ("satisfies GDPR Art. 17").
- A separate "sales version" of any doc.
- New product features. This spec documents what exists and flags what does not; it does not build LMS functionality.
- Re-documenting `docs/deployment-security-checklist.md` — it exists and is **referenced**, not duplicated.
- The compliance-control file set recommended in `research_doc_structure.md` §2 (`incident-response.md`, `data-retention-and-deletion.md`, `data-subject-rights.md`, etc.) as **separate files**. See decision D1.

## 3. Source of truth and accuracy rules

These are non-negotiable and apply to every doc and every word.

- **Facts only.** Base every statement on code that exists, verified in `research_codebase_features.md` and the live repo. Do **not** guess, infer optimistically, or be creative.
- **State absence plainly.** Where a capability is absent, manual, operational, or half-built, say so explicitly. "There is no automated data-deletion tooling; deletion is a manual database/admin operation" is a factual capability statement, not a weakness to hide.
- **Built vs planned vs operational.** For each compliance-relevant area, distinguish what the **code ships**, what is **operational/manual** (a runbook step, not code), and what is **not yet built** (cross-reference the roadmap doc).
- **No duplication.** The canonical, detailed statement of a fact lives in exactly one doc's body; other docs link to it with relative links. The per-doc `## Summary` bullets and the README index are intentional excerpts/pointers, not violations of this rule.
- **Deployment source of truth** is `deployment-playbook.md` (this spec directory) for V1 architecture. The repo's older `docs/how tos/DOCKER_DEPLOY.md` (nginx) and `Caprover deploy.md` are **superseded** and must be flagged as such, not documented as current strategy.

### Corrections that MUST be honoured (from the idea + codebase research)

| Topic | Correct factual state |
|---|---|
| **2FA / MFA** | Does **not** exist in code. Document only under Future work / roadmap — never as a current feature. |
| **Content Security Policy** | Currently **report-only** (`SECURE_CSP_REPORT_ONLY`), not enforcing. State this limitation. |
| **AI-driven content** | A **workflow affordance** (authors may use AI to write markdown). No AI integration exists in code. Describe as such. |
| **RBAC roles** | Models + sync/validate commands exist but are minimally wired; many perms are `# FUTURE`. Educator access actually flows through **django-guardian**, not the role system. Roadmap, not a shipped capability. |
| **xAPI** | Placeholder only — model code commented out, app not in `INSTALLED_APPS`. Roadmap. |
| **`SiteGroup`** | Commented out. Roadmap. |
| **Educator interface limits** | Cohort membership, course registration, and deadline-setting are **admin-only** (not editable from the educator interface). No messaging. State honestly. |
| **Learner tracking limits** | No time-on-task duration; no score/grade export. State honestly. |
| **Encryption at rest** | Only webhook secrets are app-encrypted (Fernet). DB-level encryption is provider-dependent — say so, do not overstate. |
| **Backups / scale** | Strategy is defined in the playbook but parts are not-yet-automated and scale numbers are estimates, not load-tested. Label clearly. |

## 4. Documentation set (`docs/product/`)

Organised **by product feature**, not by compliance control. Compliance-relevant facts are surfaced *within* the relevant feature doc, plus one cross-cutting **Security & data handling** doc that a reviewer reads top-to-bottom. (Decision D1.)

### 4.1 File list

```
docs/product/
  README.md                       ← index / entry point
  content-editing-workflow.md
  authentication.md
  learner-experience.md
  learner-tracking.md
  educator-interface.md
  admin-interface.md
  webhooks.md
  multi-tenancy-and-isolation.md
  security-and-data-handling.md
  configuration-and-extension.md
  deployment.md
  roadmap.md
  screenshots/                    ← PNGs referenced from the docs
```

File names are kebab-case lowercase `.md`. The set maps to the idea's "Documents" list; `multi-tenancy-and-isolation.md` is the canonical home for the site-isolation fact, linked from both the security and configuration docs (Decision D2).

### 4.2 README index

`docs/product/README.md` is the sales-friendly entry point and the reviewer's map. It contains:

- A 3–5 sentence security-posture summary paragraph.
- A note that Freedom LS is not itself certified, but the **infrastructure provider (Vultr) is ISO 27001:2022 certified** with a South African (Johannesburg) point of presence (POPIA data-residency advantage; per `deployment-playbook.md`). State the shared-responsibility split exists and link to the security/deployment docs.
- One line per doc, grouped exactly as: **Product features** (content-editing-workflow, authentication, learner-experience, learner-tracking, educator-interface, admin-interface, webhooks), **Security & data** (multi-tenancy-and-isolation, security-and-data-handling), **Configuration** (configuration-and-extension), **Deployment** (deployment), **Roadmap** (roadmap).

### 4.3 Per-file structure

The README index has its own shape (§4.2) — no `## Summary` block. Every **other** doc in the set follows:

```markdown
# [Title]

_Last updated: YYYY-MM-DD_

## Summary
- 3–5 capability bullets (the sales-friendly excerpt)

## [Detail sections…]
```

Write the detail for the compliance reviewer first (precise, factual, caveated); the `## Summary` block is what sales lifts. Cross-link related facts with relative links (`[multi-tenancy](./multi-tenancy-and-isolation.md)`) and lowercase-hyphen anchors. Keep prose tight — list features with clear, basic descriptions; do not pad.

### 4.4 Required content per doc

Each doc must cover at least the following, all sourced from `research_codebase_features.md`. (Exhaustive field-level detail is the implementer's job; this is the capability checklist.)

- **content-editing-workflow.md** — Git-based markdown/YAML authoring; UUIDs in frontmatter for stable identity; Pydantic validation (`content_validate`, strict `extra="forbid"`); idempotent upsert (`content_save`); no GUI editor (by design); markdown pipeline (python-markdown → nh3 sanitiser → cotton compile → render); content types; file assets; Obsidian image syntax; AI authoring as a workflow affordance (not built-in). Compliance angle: version control gives timestamps/rollback/diff; legal docs read from git blob at HEAD with hash recorded.

- **authentication.md** — Email-based login (no usernames); mandatory email verification; `SiteSignupPolicy` per-site signup; additional/custom registration forms + post-registration completion step; profile editing; append-only `LegalConsent` (git hash + IP + timestamp + consent method). Security hardening: Argon2 hashing, django-axes (5 fails → 1h lockout), signup rate limiting, email-enumeration prevention, password strength rules (min 10 chars, common/numeric checks). Separate API-client (machine-to-machine) auth. **2FA → roadmap only.** Compliance angle: authentication factors, account lifecycle, consent audit trail.

- **learner-experience.md** — Dashboard (in-progress/recommended/completed); course listing with registration status; course detail (outcomes, difficulty, duration); self-registration; course player with sequential item unlock (BLOCKED/READY/IN_PROGRESS/COMPLETE/FAILED); course parts/chapters; multi-page forms; resume (`last_accessed_item`); quiz feedback (pass/fail, score, optional reveal of incorrect answers); course finish page; deadline surfacing (hard deadlines lock content, soft show overdue). **Screenshots required** (§5).

- **learner-tracking.md** — What the learner did and when. `TopicProgress`, `FormProgress` (multi-attempt, scores), `CourseProgress` (percentage, last-accessed for resume), `QuestionAnswer`. Auto-recalculation of percentage on completion; `recalculate_progress_percentages` command. Admin read visibility of all progress. **State limits:** no time-on-task duration, no score/grade export. Compliance angle: this is the closest thing to an activity log — describe what is recorded and who can read it.

- **educator-interface.md** — Single-page HTMX panel interface (cohorts / users / courses); cohort detail with course-progress matrix (students × items: completion, quiz scores, deadlines); object-level access via django-guardian (educators only see permissioned cohorts). **State limits honestly:** membership, course registration, and deadline-setting are admin-only; no messaging. **Screenshots required** (§5).

- **admin-interface.md** — Brief. Django admin enhanced with Unfold; django-guardian object permissions; configurable admin URL (`DJANGO_ADMIN_URL`); `LegalConsent` read-only; webhook admin test-send. Not a full model list. **Screenshots optional** (§5).

- **webhooks.md** — Outbound webhook system; events `user.registered`, `course.completed`, `course.registered`; HMAC signing or custom auth; per-site secrets encrypted at rest (Fernet); Jinja2 body/header templates; retry + circuit breaker; SSRF protection (HTTPS-only, no private/loopback IPs in production); admin test-send. Compliance angle: this is an outbound third-party data flow — describe what data leaves and the controls around it.

- **multi-tenancy-and-isolation.md** — One FLS install serves multiple sites (domains), each with isolated users, content, and settings. `SiteAwareModel` / `SiteAwareManager` auto-filter every query to the current site. This is the **canonical** isolation fact; the security and configuration docs link here. Compliance angle: tenant data-separation guarantee.

- **security-and-data-handling.md** — The cross-cutting reviewer doc (detailed structure in §6).

- **configuration-and-extension.md** — Branding (logo, favicon, header title, email branding); three-tier theming (CSS tokens → cotton slots + mergeable classes → whole-file template shadowing); two bundled themes (`default`, `first_class`); pluggable icon set (`FREEDOM_LS_ICON_SET`, currently heroicons); custom-app extension model (FLS installs into a host Django project; downstream apps/templates take priority; downstream cotton components registerable as markdown widgets); per-site signup policy / additional registration forms. Link to multi-tenancy doc.

- **deployment.md** — Source of truth is `deployment-playbook.md`. Target architecture: single **Vultr Johannesburg** VPS (ISO 27001:2022 provider, SA PoP); Docker Compose running **Caddy** (reverse proxy + automatic HTTPS) + Gunicorn + Django 6 + containerised PostgreSQL (named volume); Cloudflare free tier (CDN/WAF); **Ansible** provisioning (Terraform deferred); CI/CD GitHub Actions → GHCR → SSH pull; background tasks via Django 6 `django-tasks` (DatabaseBackend) — no Celery/Redis at launch. Backups: `pg_dump` cron + encrypted offsite sync (Backblaze B2) — strategy, with operational-vs-not-yet-automated flagged. Scale: Phase 1 ~50–200 concurrent / ~1,000 students (estimates, not load-tested); later phases triggered by monitoring. App-level facts the code ships: Whitenoise static, S3-compatible media, `/health/` endpoint, env-var config, required Tailwind build at image-build time. **Flag `DOCKER_DEPLOY.md` (nginx) and CapRover docs as superseded.**

- **roadmap.md** — Half-built / planned, stated honestly: 2FA (not built); RBAC (models + commands exist, minimally wired, many `# FUTURE` perms, educator access actually via django-guardian); xAPI (placeholder, commented out, app not installed); `SiteGroup` (commented out); educator-interface gaps (membership/registration/deadline editing, messaging).

## 5. Screenshots

- Captured via **Playwright MCP** against a locally running dev server, following the `do_qa.md` pattern: find a free port (`find_available_port.sh`), `uv run python manage.py runserver $PORT`, capture, compress (`compress_screenshots.py`, to stay under the 1024 KB pre-commit limit), kill the server (`kill_runserver.sh`).
- Stored in `docs/product/screenshots/`, referenced with plain `![](screenshots/…)`.
- Capture for features with **visible UI**: learner-experience (dashboard, course detail, course player, quiz feedback, finish page) and educator-interface (panel interface, cohort progress matrix) are **required**; admin-interface is optional. Skip pure backend/API features (webhooks, content CLI, auth internals).
- Use the **DemoDev** site / demo content for capture data where seed data is needed.

## 6. Security & data handling doc — required structure

A reviewer reads `security-and-data-handling.md` top to bottom. It must contain these sections, each labelling capabilities as built / operational / not-yet-built:

1. **Summary** — security posture in 3–5 bullets.
2. **Development-time controls** — pre-commit hooks (detect-secrets, detect-private-key, bandit `--ll`, ruff, mypy, shellcheck, large-file/merge-conflict/AST checks); GitHub security checklist (Dependabot, secret scanning, branch protection, CodeQL).
3. **Runtime application security** — CSRF; CSP (**note: report-only, not enforcing**); nh3 markdown sanitiser (allowlist); clickjacking (`X_FRAME_OPTIONS=SAMEORIGIN`); HSTS rollout guidance; django-axes; Argon2; Whitenoise; `SecurityMiddleware`; multi-site isolation (link to multi-tenancy doc); webhook secret encryption at rest + SSRF protection (link to webhooks doc).
4. **Data handling (current-state, honest)** — what personal data is collected and where it lives; encryption in transit (TLS/HSTS) and at rest (webhook secrets only; DB-level provider-dependent). Topics with little/no code today — **incident response, data retention/deletion automation, data-subject-rights tooling** — documented as current-state: what exists, what is manual/operational, what is not yet built. Cross-reference roadmap. Do not guess.
5. **Infrastructure & shared responsibility** — the **ISO 27001 shared-responsibility split** from `deployment-playbook.md`: Vultr's certification covers physical/infra/hypervisor; FLS owners own OS hardening, encryption (TLS via Caddy, encrypted backups, PostgreSQL SSL), access control, logging/monitoring, backup/DR, incident response, change management (git/PR), vulnerability management (Trivy/Dependabot), and ISMS docs — each stated as built/planned/operational. **POPIA data residency:** Vultr JNB keeps data in South Africa — a practical advantage, not a legal mandate.
6. **References** — link to `docs/deployment-security-checklist.md` (do not duplicate it) and to `deployment.md`.

## 7. Repeatable update mechanism

This is the first documentation pass; docs must be updated as features ship. Build a reusable mechanism wired into the SDD flow.

### 7.1 New command — `fls-claude-plugin/commands/sdd/update_product_docs.md`

- A **depth-0 orchestrator** following the established SDD fan-out recipe and conventions (`research_sdd_doc_workflow.md`, `claude-code-authoring` skill).
- Frontmatter `allowed-tools` includes `Read, Glob, Write, Edit, Bash, Agent, mcp__playwright*`.
- Behaviour: reads the spec/plan for the current feature to identify affected `docs/product/` files; fans out one `fls:sdd-worker` per affected doc section to draft updates; then **synthesises** by reading the worker output files (by path).
- For visual features: starts a dev server on a free port, captures screenshots via Playwright MCP, compresses them, kills the server (reuse `find_available_port.sh`, `compress_screenshots.py`, `kill_runserver.sh` per the `do_qa.md` pattern).
- Final step delegates the todo tick to `fls:sdd-mechanic`.
- Model tiering lives on the agent files (worker = Sonnet, mechanic = Haiku), **not** on the command file (inert when inlined by `/sdd:next`).

### 7.2 Todo template change — `protected/setup_todo_list.md`

Insert a new section **between QA (current §9) and Pull request**, then renumber the trailing sections:

```markdown
## 10. Product documentation

- [ ] (cmd) Run `/update_product_docs` to update docs/product/ for this feature
- [ ] (user) Review the updated documentation
```

Pull request becomes §11 and Cleanup becomes §12. This position is correct because the code is final and the UI is live for screenshots, and the doc changes ride along in the same PR.

### 7.3 README update — `fls-claude-plugin/commands/sdd/README.md`

Add the new documentation step to this README's workflow overview, placed after QA and before Ship. Note that this file uses its **own** step numbering (a "Step 1…8" sequence with an app-map-refresh step) — it does **not** share the `setup_todo_list.md` section numbers, so follow this file's existing scheme rather than mirroring the §7.2 renumbering.

## 8. Decisions

- **D1 — Feature-oriented structure, not a control-mapped file set.** The idea explicitly chooses organisation by product feature with compliance facts surfaced inline + one cross-cutting security doc. This overrides `research_doc_structure.md` §2's recommendation of separate `incident-response.md` / `data-retention-and-deletion.md` / `data-subject-rights.md` files. *Why:* it keeps each capability described once next to the feature it belongs to, avoids a parallel doc set that drifts, and a reviewer still gets a single end-to-end security read via `security-and-data-handling.md`. The compliance topics from the research are not dropped — they become labelled subsections of the security doc (§6.4).
- **D2 — Multi-tenancy is its own canonical doc.** The isolation fact lives in `multi-tenancy-and-isolation.md`; the security and configuration docs link to it rather than restating it (no-duplication rule).
- **D3 — Both deliverables ship in this one pass.** The product docs and the SDD update mechanism are built together, as the idea directs — the mechanism is what keeps the docs from rotting, so shipping docs without it would leave them stale on the next feature.
- **D4 — Superseded deployment docs are flagged, not deleted.** `DOCKER_DEPLOY.md` and CapRover docs are marked superseded by the playbook; deleting them is out of scope.

## 9. Success criteria

1. `docs/product/` exists with the README index (in the §4.2 shape) and all twelve feature docs (§4.1); each of the twelve has a `# Title`, `_Last updated:_` line, and `## Summary` block.
2. Every factual claim is backed by code that exists; every correction in §3 is honoured (no 2FA as current; CSP noted report-only; RBAC/xAPI/SiteGroup in roadmap; educator and tracking limits stated; encryption-at-rest scope accurate).
3. `security-and-data-handling.md` reads coherently top-to-bottom and covers all six sections in §6, labelling each capability built / operational / not-yet-built, and references (does not duplicate) `docs/deployment-security-checklist.md`.
4. The README's grouped index links to every doc; all cross-links resolve; no fact's canonical detailed statement is duplicated across doc bodies (Summary bullets and the index excerpt/point, per §3).
5. Required screenshots (learner-experience, educator-interface) are present in `docs/product/screenshots/`, compressed under the 1024 KB limit, and referenced with plain markdown.
6. `deployment.md` reflects the `deployment-playbook.md` V1 architecture and flags `DOCKER_DEPLOY.md` / CapRover as superseded.
7. `fls-claude-plugin/commands/sdd/update_product_docs.md` exists, runs at depth 0, fans out workers, captures screenshots for visual features, and ticks its own todo box via `fls:sdd-mechanic`.
8. `setup_todo_list.md` has the new documentation section between QA and Pull request with sections renumbered, and `commands/sdd/README.md` describes the new step.
9. Docs are plain Markdown only — no cotton components or custom widgets.
10. Documentation is not wordy: capabilities are listed with clear, basic descriptions.
