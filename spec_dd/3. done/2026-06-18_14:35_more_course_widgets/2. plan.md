# Implementation Plan: More Course Content Widgets

Implements `1. spec.md`. Delivers a shared icon resolver, the `c-admonition`,
`c-flashcard`, and `c-accordion` content widgets, moves `c-callout` to the `base`
app, and migrates all demo content off callouts.

We follow **TDD**: for each unit of behaviour, write a failing test first, then
make it pass. The plan does not enumerate every test — it names the behaviours to
cover and the files they live in. See the `testing` skill.

## Existing code this builds on (keep DRY)

- **Icon resolution** already lives in `freedom_ls/student_interface/course_icon.py`
  (`render_course_icon`, `_render_literal`, `_FALLBACK_RE`, `IconResolutionError`,
  `_active_set_name`). The 5-step resolution order is the contract to preserve.
  Backend/loader/mappings/semantic-names live in `freedom_ls/icons/` already, which
  has **no app-level dependencies** — the right home for the shared helper.
- **Semantic names** (`freedom_ls/icons/semantic_names.py`) already include every
  icon the default registry needs: `info`, `star`, `warning`, `error`, `notes`,
  `check`, `success`, `course`. No additions required.
- **Callout component** (`freedom_ls/content_engine/templates/cotton/callout.html`)
  is the styling/partialdef pattern to mirror for `c-admonition` (a `partialdef`
  box + `{% with %}` branch per colour role so every Tailwind class is a literal
  string).
- **Markdown tag** `{% markdown value %}` is in
  `freedom_ls/content_engine/templatetags/content_tags.py` — the same module the
  new `admonition_config` simple_tag goes into.
- **Alpine CSP registrations** for content widgets live in
  `freedom_ls/content_engine/static/content_engine/js/alpine-components.js`
  (currently `equation`, `contentLightbox`). This file is **already loaded** on
  course pages via `_course_base.html:18` and `_exam_runner_base.html:26` — the
  flashcard registration needs no new script include.
- **`@alpinejs/collapse`** is already loaded (`base/_base.html`); `x-collapse` is
  in use (`partials/course_minimal_toc.html`). Accordion reuses it for animation only.
- **Role tokens** `--color-{info,success,warning,error}-light` and
  `--color-on-{info,success,warning,error}-light` are defined in both themes
  (`themes/default/.../theme.css`, `themes/first_class/.../theme.css`), so
  `bg-{role}-light` / `text-on-{role}-light` resolve in every theme.
- **Hand-authored component CSS** lives in `tailwind.components.css` (root) — the
  home for static flashcard 3D-transform rules.
- **Demo-content file guard** test pattern:
  `freedom_ls/content_engine/tests/test_demo_content_picture_titles.py`.

## Phase 1 — Shared icon resolver (`freedom_ls/icons`)

Goal: a single source of truth for icon resolution that both `student_interface`
and `content_engine` can use **without `content_engine` importing
`student_interface`**.

> **Structure concern:** This refactor introduces two new runtime edges that are
> not in `docs/app_structure.md`: `student_interface --> icons` (Phase 1 step 2,
> `course_icon.py` imports `render_icon` from `freedom_ls/icons/render`) and
> `content_engine --> icons` (Phase 4b, `content_tags.py` `admonition_icon` imports
> `render_icon`). Both are deliberate — `freedom_ls/icons` has no app-level
> dependencies (spec §4.2), so neither edge creates a cycle, and depending on the
> shared resolver is the explicit alternative to `content_engine` importing
> `student_interface`. Options:
> 1. **Accept and document** (recommended) — these edges are the architectural goal
>    of the refactor; approve them, then regenerate `docs/app_structure.md` via
>    `/app_map` after implementation and commit the updated diagram.
> 2. **Restructure** — not advised here; there is no shared lower app to move the
>    resolver into other than `icons` itself, which is already the intended home.

1. **Create `freedom_ls/icons/render.py`** holding the generic resolver moved out
   of `course_icon.py`:
   - Move `IconResolutionError`, `_FALLBACK_RE`, `_active_set_name`, and
     `_render_literal` here verbatim (they have no `student_interface` ties).
   - Add a public `render_icon(...)` generalising `render_course_icon` by making the
     default-semantic icon a parameter:

     ```python
     def render_icon(
         icon: str,
         icon_fallback: str = "",
         *,
         default_semantic: str,
         variant: str = "outline",
         css_class: str = "size-5",
         aria_label: str = "",
     ) -> SafeString:
         icon = (icon or "").strip()
         icon_fallback = (icon_fallback or "").strip()
         backend = get_icon_backend()
         # 1. empty -> default semantic
         # 2. semantic name in SEMANTIC_ICON_NAMES -> backend.render(icon)
         # 3. literal glyph in active set -> _render_literal(active, icon, ...)
         # 4. <iconset>:<glyph> fallback -> _render_literal(set, glyph, ...)
         # 5. graceful default -> backend.render(default_semantic)
     ```

     Steps mirror the current `render_course_icon` body exactly, substituting
     `default_semantic` everywhere `"course"` is currently hard-coded (steps 1 and
     5, and the `aria_label or default_semantic` fallback in those branches).
2. **Rewrite `render_course_icon`** (`student_interface/course_icon.py`) to delegate:

   ```python
   def render_course_icon(icon, icon_fallback="", *, variant="outline",
                          css_class="size-5", aria_label="") -> SafeString:
       return render_icon(icon, icon_fallback, default_semantic="course",
                          variant=variant, css_class=css_class, aria_label=aria_label)
   ```

   Keep the public signature and module docstring; drop the now-moved internals and
   import them (or just `render_icon`) from `freedom_ls.icons.render`. **Re-export
   `IconResolutionError` from `course_icon.py` (unconditional)** —
   `freedom_ls/student_interface/tests/test_course_icon.py` imports it from there, so
   add `from freedom_ls.icons.render import IconResolutionError` to keep that import
   working.

**TDD / tests:**
- New `freedom_ls/icons/tests/test_render.py`: `render_icon` resolves (a) a semantic
  name, (b) a literal glyph in the active set, (c) an `<iconset>:<glyph>` fallback
  when the literal glyph is absent, (d) the configured `default_semantic` when
  nothing resolves — and never raises for an unknown icon. Parametrise
  `default_semantic` (`"course"`, `"info"`).
- The **existing** `render_course_icon` tests must pass unchanged — they now exercise
  the resolver through the delegating wrapper, proving the refactor preserves
  behaviour. Run them first to confirm green after the move.

## Phase 2 — Settings: allowlist + admonition registry (`config/settings_base.py`)

1. In `MARKDOWN_ALLOWED_TAGS` (line ~289):
   - **Remove** `"c-callout": {"level", "title"}`.
   - **Add** `"c-admonition": {"type", "title"}`, `"c-flashcard": set()`,
     `"c-accordion": {"title", "open"}`.
   - **Add `"c-slot": {"name"}`.** nh3 runs *before* cotton, so any tag not in the
     allowlist is stripped before the cotton compiler sees it. The flashcard is the
     **first content widget to author named slots in markdown**

     the code security review** (Phase 9 / the security-review step): `c-slot` only

     re-sanitised by `{% markdown front/back %}`, so the exposure is limited to
     letting authors name slots. The flashcard slot test (Phase 9) fails without
     this entry — that is the TDD signal it is required.
2. **Add `ADMONITION_TYPES`** exactly as in spec §4.2 (the 8 entries: `note`, `tip`,
   `important`, `warning`, `danger`, `key_takeaways`, `checklist`, `default`). The
   `"default"` entry is mandatory.

No custom/demo type goes here — that belongs in the dev settings override (Phase 7).

**TDD / tests:** covered via the `admonition_config` resolver tests (Phase 4) and
the markdown rendering tests (Phase 9), not as standalone settings assertions.

## Phase 4 — Admonition: resolver tag + component

### 4a. `admonition_config` simple_tag

Add to `content_engine/templatetags/content_tags.py`:

```python
from django.conf import settings

@register.simple_tag
def admonition_config(admonition_type: str) -> dict[str, str]:
    registry = settings.ADMONITION_TYPES
    return registry.get(admonition_type, registry["default"])
```

Keep this the **single source of truth** for type→entry lookup (a future DB layer
replaces only this tag). Resolution is tolerant: unknown `type` → `default` entry.

**TDD / tests** (`content_engine` tests, e.g.
`content_engine/tests/test_content_tags.py` — create if absent):
- known type returns its entry; unknown type returns the `default` entry;
- an overridden `ADMONITION_TYPES` (via `@override_settings`) is respected.

### 4b. `admonition.html` component

Create `content_engine/templates/cotton/admonition.html`, mirroring the
`callout.html` partialdef pattern:

- `<c-vars type="default" title="" />`.
- `{% load content_tags %}`; resolve `{% admonition_config type as cfg %}`.
- Visible label = `title` if provided else `cfg.label`; render it as **real DOM
  text** in an element with an `id` (e.g. `id="admonition-label-..."`), and set
  `aria-labelledby` to that id on the container.
- Container: `role="note"`, `aria-labelledby="<label-id>"`.
- **Colour:** a `{% with %}` branch per role (`info`/`success`/`warning`/`error`)
  emitting **literal** `bg-{role}-light` + `text-on-{role}-light` class strings —
  never `bg-{{ color }}-light`. Map `cfg.color` to the branch; a `cfg.color` outside
  the four roles falls through to the `info` branch (tolerant default).
- **Icon:** render server-side through the shared resolver, **not** a bare
  `<c-icon :name="cfg.icon" />` (that only handles semantic names). Use the icons
  resolver so `cfg.icon` may be a semantic name **or** a literal glyph backed by
  `cfg.icon_fallback`. Cotton templates can't call Python directly, so expose the
  rendered SVG via a thin simple_tag, e.g. add to `content_tags.py`:

  ```python
  from freedom_ls.icons.render import render_icon

  @register.simple_tag
  def admonition_icon(cfg: dict[str, str], css_class: str = "size-5") -> SafeString:
      return render_icon(cfg.get("icon", ""), cfg.get("icon_fallback", ""),
                         default_semantic="info", css_class=css_class)
  ```

  In the template: `{% admonition_icon cfg as icon_svg %}` then `{{ icon_svg }}`
  inside an `aria-hidden="true"` wrapper. (Decorative; the label carries meaning.)
- **Body:** `{% markdown slot %}` (markdown supported inside the box).

The label id must be unique per instance to keep `aria-labelledby` valid when
multiple admonitions appear on a page. **Use the existing
`{% unique_id "admonition-label" as label_id %}` tag from `base_tags`** — it already
backs this exact pattern in `base/templates/cotton/modal.html` and
`content_engine/templates/cotton/picture.html`. Load it in `admonition.html`
(`{% load base_tags %}` alongside `{% load content_tags %}`, as `picture.html` does),
then `id="{{ label_id }}"` on the label element and `aria-labelledby="{{ label_id }}"`
on the container. No new tag or module.

**TDD / tests** (markdown rendering — Phase 9): each default `type` renders its
label, the type's colour branch, and an icon; `title` overrides the label; markdown
in body (`**bold**`→`<strong>`); unknown `type` falls back to `default` (renders,
never errors/blank).

### 4c. key_takeaways & checklist

No new code — they are just default registry entries (Phase 2). `checklist` body is
a normal markdown task list rendered read-only by the existing pipeline. No
persistence, no interactivity (spec §10 — leave room, don't build).

## Phase 5 — Flashcard (`c-flashcard`)

### 5a. Component template

Create `content_engine/templates/cotton/flashcard.html`:
- Named slots `front` and `back`, each `{% markdown front %}` / `{% markdown back %}`.
- Root is `x-data="flashcard"` (CSP component, Phase 5b). The flip trigger is a real
  `<button type="button">` covering the whole card surface (≥44×44px), with
  `x-on:click="flip()"`, `x-bind:aria-pressed="flipped"`, and an accessible name
  (e.g. `aria-label` "Flip card" or visually-hidden text).
- **Fixed-size co-located faces:** both faces share one stacking container sized to
  the larger face (CSS-grid single-cell stack — both faces in the same grid area —
  is the suggested primitive; requirement is fixed size + co-located faces, not the
  primitive). Static 3D CSS (`perspective`, `transform-style: preserve-3d`,
  `backface-visibility: hidden`, grid-stack rules) goes in literal `style` attributes
  or hand-authored `.flashcard-*` rules in `tailwind.components.css` — **not**
  dynamically-assembled Tailwind classes.
- Dynamic rotation applied via `x-bind:style` returning a **style object**
  (`frontStyle()` / `backStyle()`), bypassing Tailwind purging.
- **A11y per face:** `x-bind:aria-hidden` toggled per face from `flipped`; the hidden
  face's interactive children get `tabindex="-1"` (bind from `flipped`) so focus
  can't land on an unseen link.

### 5b. Alpine component

Register `flashcard` in
`content_engine/static/content_engine/js/alpine-components.js` (CSP build — no inline
expressions), alongside `equation`/`contentLightbox`:

```js
Alpine.data("flashcard", () => ({
    flipped: false,
    flip() { this.flipped = !this.flipped; },
    frontStyle() { return { transform: this.flipped ? "rotateY(180deg)" : "rotateY(0deg)" }; },
    backStyle()  { return { transform: this.flipped ? "rotateY(0deg)" : "rotateY(180deg)" }; },
}));
```

- **Reduced motion:** keep the state change (flip), drop the animation — gate the
  CSS `transition` behind `@media (prefers-reduced-motion: no-preference)` in the
  `.flashcard-*` rules so the transform still applies instantly when motion is
  reduced.
- Update the file's top-of-file doc comment to list `flashcard`.

No new script include needed (already loaded on course pages — see "Existing code").

**TDD / tests** (markdown rendering — Phase 9): front and back slots both render
markdown; output contains a `<button>` with `aria-pressed`. Flip/keyboard/reduced-
motion behaviour is QA (Playwright), not unit tests.

## Phase 6 — Accordion (`c-accordion`)

Create `content_engine/templates/cotton/accordion.html`:
- `<c-vars title="" open="" />`.
- Native `<details>{% if open %} open{% endif %}` + `<summary>` carrying the
  `title`. Do **not** add `role="button"`/`aria-expanded` to `<summary>` — the
  browser sets these natively.
- Body: `{% markdown slot %}`.
- **Animation:** layer `x-collapse` (`@alpinejs/collapse`, already loaded) for smooth
  open/close only — the native `<details>` stays the source of truth and the no-JS
  fallback. **`x-collapse` cannot be dropped onto `<details>` naively:** it animates
  an element shown/hidden by Alpine's `x-show`, which is a *separate* system from the
  browser's native `<details open>` toggle. Copying the plain-`<div>` pattern from
  `course_minimal_toc.html` directly onto the `<details>` body makes Alpine's `x-show`
  fight the native toggle (no animation, or flicker). Two acceptable approaches —
  **pick one and implement it fully:**
  1. **Bridge (keeps x-collapse):** a tiny CSP `accordion` Alpine component holding an
     `open` boolean synced to the native toggle via the `<details>`'s `toggle` event
     (`x-on:toggle="open = $event.target.open"`), with the **body** wrapped in
     `<div x-show="open" x-collapse>`. The native `open` attribute remains the no-JS
     truth; Alpine only animates.
  2. **CSS-only (simpler, no Alpine):** animate the body with a CSS `grid-template-rows`
     / `max-height` transition driven by the `details[open]` selector, gated behind
     `@media (prefers-reduced-motion: no-preference)`. No Alpine on the accordion at
     all — most consistent with "native is the source of truth".

  CSS-only is the lighter option; the bridge satisfies the spec's literal "layer
  `x-collapse`" wording. Either is fine — do **not** leave it as "follow the existing
  usage", which doesn't transfer to `<details>`.
- **Chevron:** rotate on open — either two server-rendered `<c-icon>` toggled by
  `x-show` (the existing icon-toggle pattern, e.g. `cotton/dropdown-menu.html`) or a
  CSS rotation tied to the `[open]` attribute selector. Honour
  `prefers-reduced-motion` (no transition when motion reduced).
- Single independent disclosure — no one-open-at-a-time grouping, no nesting.
- Add a brief template comment recording the editorial rule (elective depth only,
  never hide required/assessed content) — documented, not code-enforced.

**TDD / tests** (markdown rendering — Phase 9): renders `<details>`/`<summary>` with
the title; `open` attribute maps to native open state; body markdown is rendered.

**Must-follow constraints (failure risk if ignored):**

- **Alpine CSP:** no inline expressions anywhere — `x-data="flashcard"` (registered
  name), all logic in `Alpine.data()`. No `x-data="{...}"`, no `@click="..."`
  ternaries. Breaks silently in the CSP build otherwise.
- **Tailwind v4 literal classes:** every colour class (`bg-{role}-light`,
  `text-on-{role}-light`) appears literally per-role; never `bg-{{ color }}-light`.
  Dynamically-assembled names are purged → invisible widget.
- **Decorative admonition icon:** wrap the rendered SVG in `aria-hidden="true"`. Do
  **not** rely on `aria_label=""` — the icon backend still emits a fallback
  `aria-label` (the glyph slug), which would double the visible label. (See the
  established `aria-hidden` icon convention.)
- **Allowlist before render:** register all three widgets (and their attrs) in
  `MARKDOWN_ALLOWED_TAGS` or nh3 strips them before cotton ever runs.
- **Body slots use `{% markdown slot %}`** (not `{{ slot }}`) so body markdown is
  processed — for all three new widgets. The **moved callout** is the exception:
