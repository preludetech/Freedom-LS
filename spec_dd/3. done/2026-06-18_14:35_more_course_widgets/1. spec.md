# Spec: More Course Content Widgets

## 1. Overview

Course content is authored in markdown and rendered through the pipeline in
`freedom_ls/markdown_rendering/markdown_utils.py`:

```
markdown parse → nh3 sanitize (allowlist: MARKDOWN_ALLOWED_TAGS) → CottonCompiler → Django render
```

nh3 runs **before** cotton, so it is the security boundary: any `c-*` tag (or attribute)
not present in `MARKDOWN_ALLOWED_TAGS` (`config/settings_base.py`) is stripped before the
template engine ever sees it. Content widgets are django-cotton components under
`freedom_ls/content_engine/templates/cotton/`, authored with the `c-` prefix
(`<c-admonition>`). Slot bodies are rendered as markdown via the `{% markdown slot %}`
template tag.

Interactivity uses **Alpine.js 3.15.8 (CSP build)** — no inline expressions, all logic
registered via `Alpine.data("name", () => ({...}))`. Styling uses **Tailwind v4** with theme
**role tokens** (`--color-info`, `--color-warning`, … plus `-light` tints and `on-*-light`
foregrounds), so a widget that references a role token adapts to every theme automatically.

This change delivers:

1. **Move `c-callout` to the `base` app** and drop it from the content allowlist — it is an
   application-level alerting component, not a content widget.
2. **`c-admonition`** — the content-layer replacement for callouts: a configurable, labelled,
   accessible box driven by a deploy-time settings registry (`ADMONITION_TYPES`). Includes
   built-in `key_takeaways` and `checklist` types (no separate widgets).
3. **`c-flashcard`** — a two-sided, click-to-flip card (front/back markdown).
4. **`c-accordion`** — a collapsible disclosure built on native `<details>`/`<summary>`.
5. **Demo content** updates: showcase every new widget and migrate all `c-callout` content uses.

## 2. Goals & success criteria

A change is complete when:

- `c-callout` no longer appears in `MARKDOWN_ALLOWED_TAGS`; its cotton template lives in
  `base`; it still renders where used directly in application templates.
- `c-admonition`, `c-flashcard`, and `c-accordion` are registered in `MARKDOWN_ALLOWED_TAGS`
  with the correct attribute allowlists, render correctly from topic markdown, and meet the
  accessibility requirements in this spec.
- The `ADMONITION_TYPES` settings registry exists, ships the agreed default types, is
  overridable per deploy, and an unknown `type` falls back to `default` (never an error/blank).
- A shared icon resolver lives in `freedom_ls/icons`; `render_course_icon` delegates to it (no
  behaviour change), and the admonition resolves its icon through the same helper — so a
  registry entry can use a semantic name **or** a literal glyph + `icon_fallback`, and
  `content_engine` does not import `student_interface`.
- No `demo_content/` file references `c-callout`; every content use is migrated to the right
  `c-admonition` type, and every new widget is showcased with real, readable content.
- `npm run tailwind_build` produces the classes the new widgets use (no purged dynamic classes).
- Tests cover the new widgets and the callout migration; the suite passes.

## 3. Move `c-callout` to `base` (application-level only)

The existing `c-callout` (`freedom_ls/content_engine/templates/cotton/callout.html`) is used
in exactly one application template:
`freedom_ls/student_interface/templates/student_interface/course_form_page.html:111`
(`<c-callout level="error">ERROR! UNHANDLED FORM TYPE …</c-callout>`), rendered directly in a
Django template, **not** through `render_markdown`.

Changes:

- **Move** `callout.html` from `freedom_ls/content_engine/templates/cotton/` to
  `freedom_ls/base/templates/cotton/callout.html`. The `<c-callout>` tag name and its
  `level`/`title` vars are unchanged. The `base` cotton directory already exists and is on the
  cotton loader path, so the application-template use keeps working with no further change.
- **Drop the content_engine dependency in the moved template.** The current callout renders its
  body with `{% markdown slot %}` (the `markdown` tag lives in
  `content_engine.templatetags.content_tags`). Once callout is an application-level component in
  `base`, it must **not** depend on `content_engine`: render the body as a plain slot
  (`{{ slot }}`) and remove the `{% load content_tags %}` line. The only current use
  (`course_form_page.html:111`) is plain text, so rendered output is unchanged, and `base` stays
  free of any `content_engine` reference. Markdown-bodied boxes are now the content layer's job
  (`c-admonition`).
- **Remove** `"c-callout": {"level", "title"}` from `MARKDOWN_ALLOWED_TAGS` in
  `config/settings_base.py`. After this, any `<c-callout>` left in markdown **content** is
  stripped by nh3 during sanitization rather than rendered — this is intentional and is the
  reason all demo content must be migrated (§7).
- **Remove** documentation/plugin references to `c-callout` as a *content* widget where they
  describe content authoring (`docs/product/content-editing-workflow.md` widget table,
  `fls-claude-plugin/resources/markdown_content.md` example). Replace with `c-admonition`.
  The "incorporate into another project" how-to example should reflect the new allowlist.

## 4. Admonition widget (`c-admonition`)

A styled, labelled box for supplementary content — the content-layer replacement for callouts.

### 4.1 Attributes

| Attribute | Required | Default | Meaning |
|-----------|----------|---------|---------|
| `type`    | no       | `default` registry entry | Selects the registry entry (icon, colour, label). |
| `title`   | no       | the type's registry `label` | Overrides the visible label text. |

Allowlist entry: `"c-admonition": {"type", "title"}`.

### 4.2 Configurable type registry (`ADMONITION_TYPES`)

A deploy-time settings dict in `config/settings_base.py`, shaped like `MARKDOWN_ALLOWED_TAGS`.
Each `type` maps to a `label`, a `color` (a theme **role token** name: one of `info`,
`success`, `warning`, `error`), and an `icon` — **resolved with the exact same mechanism as a
course icon** (`Course.icon` / `Course.icon_fallback`; see `render_course_icon` in
`freedom_ls/student_interface/course_icon.py`). The `icon` may be either a **semantic icon
name** (from `freedom_ls/icons/semantic_names.py`, e.g. `info`, `star`, `warning`) **or** a
**literal glyph name** from the active icon set (e.g. `lightbulb`). An entry may also carry an
optional `icon_fallback` of the form `<iconset>:<glyph>` (e.g. `phosphor:drone`), used only
when the literal `icon` does not resolve in the active set. This is what lets
downstream/domain types use **unusual icons** that have no semantic name.

**Default registry (ships in `settings_base.py`):**

```python
ADMONITION_TYPES = {
    "note":          {"label": "Note",          "icon": "info",    "color": "info"},
    "tip":           {"label": "Tip",           "icon": "star",    "color": "success"},
    "important":     {"label": "Important",      "icon": "warning", "color": "warning"},
    "warning":       {"label": "Warning",        "icon": "warning", "color": "warning"},
    "danger":        {"label": "Danger",         "icon": "error",   "color": "error"},
    "key_takeaways": {"label": "Key Takeaways",  "icon": "notes",   "color": "info"},
    "checklist":     {"label": "Checklist",      "icon": "check",   "color": "success"},
    "default":       {"label": "Note",           "icon": "info",    "color": "info"},
}
```

- Each entry may include an optional `icon_fallback` (`<iconset>:<glyph>`) alongside `icon`; the
  shipped defaults all use semantic-name icons, so none of them need a fallback.
- The `"default"` entry is **mandatory** and is used as the graceful fallback for any unknown
  `type`. Resolution: look up `type` in the registry; if absent, use `default`.
- `color` must be one of the four status role tokens (`info`, `success`, `warning`, `error`)
  because the template maps the role name to literal Tailwind class strings (§4.4). A registry
  entry whose `color` is outside this set is treated as the `default` colour (info) — keep the
  lookup tolerant, never raise at render time.
- `icon` is resolved with the **same mechanism as course icons**: semantic name → literal glyph
  in the active icon set → the entry's `icon_fallback` (`<iconset>:<glyph>`) → graceful default.
  An unresolved/unknown icon must **never break the render** — it falls back to the `default`
  type's icon (`info`).
- **Shared resolver (required, in scope):** extract the generic resolution that currently lives
  in `render_course_icon` (`freedom_ls/student_interface/course_icon.py`) into a reusable helper
  in `freedom_ls/icons` — e.g. `render_icon(icon, icon_fallback, *, default_semantic, variant,
  css_class, aria_label)` plus the shared `IconResolutionError` and the `_FALLBACK_RE` /
  `_render_literal` internals it depends on. `render_course_icon` is **rewritten to delegate** to
  it (`default_semantic="course"`), preserving its public signature and the existing 5-step order
  so course-icon behaviour and tests are unchanged. The admonition resolver calls the same helper
  with `default_semantic="info"`. This keeps the resolution logic a single source of truth and
  lets `content_engine` reuse it **without importing `student_interface`** (`freedom_ls/icons`
  has no app-level dependencies, so it is safe for both apps to depend on).
- **Downstream extension:** projects/themes add domain types (e.g. aviation `regulation`,
  parenting `try_this_with_your_child`) by overriding `ADMONITION_TYPES` in their own settings.
  No DB, no migration, no admin UI. Such domain types frequently need **unusual icons** with no
  semantic name — supported directly via a literal-glyph `icon` plus an optional `icon_fallback`,
  exactly as a course icon does.
- **Forward-compatibility:** the lookup is isolated in a single resolver (§4.3) so a future
  per-site DB layer can slot in with resolution order DB → settings → `default`. Out of scope
  to build now, but the template must not hardcode the registry inline in a way that precludes it.

### 4.3 Type resolution (server-side)

Tailwind purges runtime-assembled class names, and the icon must be resolved server-side
(`<c-icon>` is server-rendered). Therefore a **custom template tag** in `content_engine`
resolves a `type` to its registry entry. Add to
`freedom_ls/content_engine/templatetags/content_tags.py` (the existing tag module), e.g.:

```python
@register.simple_tag
def admonition_config(admonition_type: str) -> dict[str, str]:
    registry = settings.ADMONITION_TYPES
    return registry.get(admonition_type, registry["default"])
```

The component calls this tag to obtain `label`, `icon`, `icon_fallback` (optional), and `color`,
then renders. Keep the resolver the single source of truth for the lookup so a future DB layer
replaces only it. The `icon` itself is then rendered through the shared `freedom_ls/icons`
resolver (the same code path as `render_course_icon`), giving semantic-or-literal-glyph support
and the graceful fallback — `admonition_config` resolves the *type → entry*, the icons resolver
resolves the *entry's icon → SVG*.

### 4.4 Component template

`freedom_ls/content_engine/templates/cotton/admonition.html`. Follows the `callout.html`
pattern (a `partialdef` for the box, `{% with %}` branches per colour role so all Tailwind
classes appear as **literal strings** — required for Tailwind v4 source scanning; never build
`bg-{{ color }}-light` dynamically).

- Resolve `{% admonition_config type as cfg %}`; `cfg.label`, `cfg.icon`, `cfg.color`.
- Visible label text = `title` if provided, else `cfg.label`. The label is **always real DOM
  text** (never colour-alone signalling).
- Background/foreground use the `-light` token pairs for legible contrast: `bg-{role}-light`
  with `text-on-{role}-light`, enumerated per role (`info`/`success`/`warning`/`error`).
- Body is rendered with `{% markdown slot %}` (markdown supported inside the box, as callout
  does today).
- Icon: rendered via the shared `freedom_ls/icons` resolver (same path as `render_course_icon`)
  so `cfg.icon` may be a semantic name **or** a literal glyph backed by `cfg.icon_fallback`;
  decorative (`aria-hidden="true"`). Do **not** use a bare `<c-icon :name="cfg.icon" />` here —
  that only handles semantic names and would not support the unusual (literal-glyph) icons that
  domain types rely on.

### 4.5 Accessibility (non-negotiable)

- Container has `role="note"`.
- The visible label has an `id`; the container references it via `aria-labelledby`.
- Label is always real text (defaults to the registry `label`); colour is never the only signal.
- Icon is decorative: `aria-hidden="true"`.
- Use the `-light` background + `on-*-light` foreground pairs for contrast.

### 4.6 Key takeaways & checklist (built-in types, not separate widgets)

- **`key_takeaways`** — a "what was covered" summary box. Ships as a default registry type with
  its own icon (`notes`) and colour. No separate widget.
- **`checklist`** — body is a standard markdown task list (`- [ ]` / `- [x]`), rendered
  **read-only** via the normal markdown pipeline. Ships as a default registry type
  (`check` icon). No persistence, no per-learner state, no interactivity.

> Interactive / persisted checklists (learner ticks boxes, progress saved) are explicitly out
> of scope (§9) — leave room for it, do not build it.

### 4.7 Example

```html
<c-admonition type="regulation">Under SACAA's Part 101 framework, commercial work generally requires the pilot's **RPC** and the operation's **UASOC** as two separate things.</c-admonition>

<c-admonition type="key_takeaways">
- A drone operation is planning, the flight, and post-flight work.
- The real skill is judgement and planning.
</c-admonition>

<c-admonition type="checklist">
- [ ] Pre-flight inspection
- [x] Battery charged
</c-admonition>
```

(`regulation` only renders as styled if a deploy adds it to `ADMONITION_TYPES`; otherwise it
falls back to `default`. Demo content that shows a custom type **must** register it — §7.)

## 5. Flashcard widget (`c-flashcard`)

A two-sided flip card; the learner clicks/taps to flip. Front = prompt, back =
answer/explanation. Both faces are markdown.

### 5.1 Authoring

Named cotton slots `front` and `back`, each markdown-rendered (`{% markdown front %}` /
`{% markdown back %}`). No attributes. Allowlist entry: `"c-flashcard": set()` (registered so
the tag survives sanitization; no attributes permitted).

```html
<c-flashcard>
  <c-slot name="front">What is the capital of France?</c-slot>
  <c-slot name="back">**Paris** — capital since the 10th century.</c-slot>
</c-flashcard>
```

### 5.2 Interaction & implementation

- **Single click/tap** toggles flip (never hover — breaks on touch).
- Alpine.js component registered in
  `freedom_ls/content_engine/static/content_engine/js/alpine-components.js` (the existing file
  that already registers `equation` and `contentLightbox`). CSP build: register via
  `Alpine.data("flashcard", () => ({ flipped: false, flip() {...}, frontStyle() {...},
  backStyle() {...} }))` — no inline expressions.
- 3D flip via `transform-style: preserve-3d` + `backface-visibility: hidden` + `perspective`.
  Static transform CSS goes in literal `style` attributes or a hand-authored
  `tailwind.components.css` rule (`.flashcard-*`); the dynamic rotation is applied with
  `x-bind:style` returning a style object (bypasses Tailwind purging). Do **not** assemble
  arbitrary-value Tailwind classes dynamically.
- **Fixed card size across flip (non-negotiable):** flipping must **not** resize the card. The
  front and back faces occupy the **same box** and are the **same size** — the card's width and
  height never change when it flips. Both faces share one stacking container sized to the
  larger/taller face, so whichever face is shown always fills the same dimensions, avoiding the
  variable-height trap that makes naive flip cards jump in size. The stacking technique is left to
  the implementer (e.g. a CSS-grid single-cell stack that places both faces in the same grid area,
  which keeps them co-located for the 3D `backface-visibility` flip); the **requirement** is the
  fixed size and co-located faces, not any one layout primitive.
- The content_engine alpine-components.js must be loaded on course pages — confirm it is pulled
  in via the `{% block extra_alpine_components %}` chain used by `_course_base.html`; if not
  already loaded for content widgets, ensure it is.

### 5.3 Accessibility (non-negotiable)

- The flip trigger is a real `<button>` (free Enter/Space keyboard support); the whole card
  surface is the target and is ≥44×44px.
- `aria-pressed` reflects flip state; the button has an accessible name.
- The hidden face is removed from the a11y tree (`aria-hidden` toggled per face), and its
  interactive children get `tabindex="-1"` so focus can't land on an unseen link. Toggle these
  via Alpine bindings driven by `flipped`.
- Honour `prefers-reduced-motion`: keep the flip (state change), drop the animation.

## 6. Accordion widget (`c-accordion`)

A collapsible disclosure: a clickable title expands to reveal a markdown body. For elective
"optional depth" content the learner may open.

### 6.1 Attributes

| Attribute | Required | Default | Meaning |
|-----------|----------|---------|---------|
| `title`   | yes      | —       | The summary/clickable heading text. |
| `open`    | no       | closed  | Open by default when truthy. |

Allowlist entry: `"c-accordion": {"title", "open"}`.

### 6.2 Implementation

- Build on native `<details>`/`<summary>` — free keyboard support, no-JS fallback, and Ctrl+F /
  print / deep-link visibility. The `open` attribute maps to the native `<details open>`.
- Layer Alpine.js `x-collapse` (the `@alpinejs/collapse` plugin is already loaded) **only** for
  smooth animation. Do **not** re-add `role="button"` / `aria-expanded` to `<summary>` — the
  browser sets these natively.
- Body is `{% markdown slot %}`.
- Rotate a chevron on the open state (two server-rendered `<c-icon>` elements toggled by
  `x-show`, per the icon-toggle pattern, or a CSS rotation tied to `[open]`); honour
  `prefers-reduced-motion`.
- Single independent disclosure (not a one-open-at-a-time group). No nested accordions.
- Editorial rule (documented, not enforced in code): use only for elective depth — never hide
  required/assessed content behind it.

```html
<c-accordion title="Why two-stroke engines need oil mixed in" open>
Body markdown here…
</c-accordion>
```

## 10. Out of scope (noted, not built)

- Per-site **database-backed** admonition type configuration (settings dict is enough now; keep
  the resolver open to a DB layer later with order DB → settings → default).
- **Interactive / persisted checklists** with saved per-learner state (new model, progress
  tracking, HTMX — its own spec).
- **Collapsible admonitions** (the accordion widget covers the hide/reveal need).
- One-open-at-a-time accordion **groups** and nested accordions.

## 11. Key decisions & rationale

- **Default registry = content-semantic set** (`note`, `tip`, `important`, `warning`, `danger`,
  `key_takeaways`, `checklist`, `default`). Chosen over mirroring the old callout alert-levels
  because the content layer wants author-facing vocabulary distinct from app alerting, while
  still covering every callout migration (info→note, warning→warning, error→danger,
  success→tip) and shipping the two special types.
- **Settings dict, not DB** for types: zero-migration theming, overridable per deploy, and the
  `color` role token means each type looks right in every theme automatically. A single resolver
  keeps the door open to a DB layer later.
- **Graceful fallback** to `default` for unknown types: content authored against a downstream
  registry never hard-errors in a deploy that lacks that type.
- **Callout → base, not deleted**: it is still needed for application-level alerting (one
  template today); removing it from the content allowlist cleanly separates the two concerns.
- **Native `<details>` for the accordion**: free keyboard/no-JS/find-in-page/print behaviour;
  Alpine only sugars the animation.
- **Co-located stacked faces + real `<button>` for the flashcard**: a single fixed-size box
  holding both faces avoids the variable-height trap, and the real `<button>` gives keyboard
  support and correct a11y-tree behaviour for free.
- **Read-only checklist** as an admonition type: meets the "summary checklist" need without the
  much larger persisted-state feature.
```
