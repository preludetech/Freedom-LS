# Frontend QA: More Course Content Widgets

Manual browser QA for the three new content widgets (`c-admonition`, `c-flashcard`,
`c-accordion`), the `c-callout`→`c-admonition` demo migration, and the custom
`regulation` admonition type proving the settings override + literal-glyph icon
resolution. Functionality and accessibility are the focus — styling correctness is a
visual spot-check, not the point.

---

## 0. Start the dev server

```bash
PORT=$(.claude/fls/scripts/find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL: `http://127.0.0.1:$PORT`

If anything looks unstyled (e.g. admonition boxes have no background colour, the
flashcard has no 3D framing), build Tailwind first, then hard-reload:

```bash
npm run tailwind_build
```

## 2. Admonition widget

Navigate to the showcase topic. You should see admonition boxes for each type.

### 2a. Default types render correctly
- **Expect** one box per type: `note`, `tip`, `important`, `warning`, `danger`,
  `key_takeaways`, `checklist`.
- Each box shows: an **icon**, a **real text label** (e.g. "Note", "Tip", "Key
  Takeaways", "Checklist"), and a coloured **`-light` background** with legible
  foreground text. Colour roughly tracks meaning (info/blue-ish for note, green-ish
  for tip, amber for warning/important, red for danger) — but the **text label is
  always present**, never colour-alone.
- Body markdown renders: any `**bold**` / lists inside a box appear as real
  formatting, not literal asterisks.

### 2b. `title` override
- The box that sets `title="…"` shows the custom title instead of the registry
  label. **Expect:** your overriding text, not the default label.

### 2c. `checklist` is read-only
- The `checklist` box shows a markdown task list with checked/unchecked boxes.
- **Expect:** the boxes are **not interactive** — clicking a checkbox does nothing
  and nothing is persisted (read-only by design).

### 2d. Custom `regulation` type (override + unusual icon)
- **Expect:** a styled box labelled "Regulation" with warning colouring and the
  **unusual icon** (the literal glyph configured via `icon` + `icon_fallback` in
  `settings_dev.py`) — a real icon, not a broken/empty image and not the generic
  info icon. This proves the deploy-time override and the literal-glyph/fallback
  resolution.

### 2e. Unknown type falls back gracefully (optional spot-check)
- If convenient, temporarily author `<c-admonition type="does_not_exist">body</c-admonition>`
  in the topic and reload. **Expect:** it renders as the **default** "Note" box —
  never an error page, never a blank box.

### 2f. Accessibility (admonition)
- Inspect a box in dev tools. **Expect:** container has `role="note"` and
  `aria-labelledby` pointing at the id of the visible label element; the icon is
  inside an `aria-hidden="true"` wrapper (so a screen reader announces the label
  text once, not the icon slug).
- With multiple boxes on the page, each label `id` is **unique** (no duplicate ids —
  check the dev-tools console / accessibility tree).

## 3. Flashcard widget

Find the flashcard in the showcase topic.

### 3a. Flip on click/tap
- The card shows the **front** (prompt) markdown.
- **Click anywhere on the card.** **Expect:** it flips to show the **back**
  (answer/explanation) markdown. Click again → flips back to the front.
- Both faces render their markdown (e.g. `**Paris**` is bold on the back).

### 3b. Fixed size across flip (non-negotiable)
- Watch the card's footprint while flipping. **Expect:** the card's width and height
  **do not change** when it flips — no jump/resize, even if front and back have
  different amounts of text. Both faces occupy the same box.

### 3c. Keyboard & a11y
- `Tab` to the card. **Expect:** focus lands on a real `<button>` (visible focus
  ring), and the whole card surface is the target (large hit area, ≥44×44px).
- Press **Enter**, then **Space**. **Expect:** each toggles the flip.
- Inspect the button: `aria-pressed` reflects the flip state (`false` on front,
  `true` on back), and the button has an accessible name (e.g. "Flip card").
- Inspect the hidden face: it is `aria-hidden="true"` and any link/button inside it
  has `tabindex="-1"` — `Tab` from the visible face must **not** move focus into the
  hidden face's links.

### 3d. Reduced motion
- Enable "reduce motion" (OS setting, or dev tools → Rendering →
  "Emulate prefers-reduced-motion: reduce") and reload.
- **Expect:** clicking still flips the card (state change preserved) but **without
  the animated rotation** — it switches faces instantly.

## 4. Accordion widget

Find the accordions in the showcase topic (one default-closed, one `open`).

### 4a. Default closed + toggle
- The default accordion shows only its **title** (summary), body hidden.
- **Click the title.** **Expect:** the body expands (smooth animation) revealing the
  markdown body. Click again → collapses.

### 4b. `open` attribute
- The accordion authored with `open` is **already expanded** on page load.

### 4c. Native behaviour (keyboard / find-in-page / print)
- `Tab` to a closed accordion's summary, press **Enter** (or Space). **Expect:** it
  toggles — native `<details>` keyboard support, no custom JS needed.
- Use the browser's **Ctrl+F** to search for a word that is inside a **closed**
  accordion's body. **Expect:** modern browsers reveal/scroll to it (native
  `<details>` content is findable).
- Inspect the markup: it is `<details>`/`<summary>`; the `<summary>` does **not**
  have a manually-added `role="button"`/`aria-expanded` (the browser provides these).
- A chevron/indicator rotates between open and closed states.

### 4d. Reduced motion
- With "reduce motion" emulated, toggling still opens/closes but without the
  animated height transition.

## 5. Callout migration regression

### 5a. Migrated demo content renders as admonitions
- Visit the migrated topics (loaded in §1), e.g. the standard-markdown topic and the
  former callout-demo topic in `functionality_demo_end_with_topic`. **Expect:**
  where callouts used to be, you now see **admonition** boxes with the mapped type
  (info/none→note, warning→warning, error→danger, success→tip), titles preserved.
- **Expect:** no literal `<c-callout …>` text visible anywhere, and no stripped/empty
  gaps where a callout used to be.

### 5b. Application-level callout still works (moved to `base`)
- The `c-callout` move only affects the form-runner error path
  (`course_form_page.html`). It renders only on an **unhandled form type**, which is
  not normally reachable. No action needed unless you can force that path — its unit
  coverage in the form-runner tests is sufficient. Just confirm normal **form pages**
  in any demo course still render correctly (no regression from the template move).

## 6. Cross-cutting checks

- **Console:** open the browser dev-tools console on the showcase topic. **Expect:**
  no JavaScript errors (especially no Alpine CSP "inline expression" errors — those
  would mean the flashcard/accordion JS was authored with inline expressions).
- **Theme:** if practical, switch theme (DemoDev uses the `first_class` theme via
  `settings_dev`) — admonition `-light` backgrounds and foregrounds should remain
  legible in the active theme (role tokens adapt automatically).
