# Frontend QA - Pluggable Icon Backend

## Setup

```bash
PORT=$(./find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL: `http://127.0.0.1:$PORT`

Login credentials (DemoDev site):
- Student: Use the student account configured for the DemoDev site
- Educator: Use the educator account configured for the DemoDev site

## Test 1: Default icon set (Heroicons) - Visual regression check

**Goal:** Confirm that with default settings (`FREEDOM_LS_ICON_SET = "heroicons"`), the site looks identical to before.

1. Navigate to the student interface home page
2. Check that all icons are visible and render as SVG elements (inspect element should show `<svg>` tags)
3. Navigate to a course page with a table of contents
4. Verify icons appear for: reading, quiz, assessment, section, expand/collapse, complete, in_progress, locked, not_started
5. Click on a content item to open it, verify navigation icons (next, previous, home) render
6. Check the messages area -- trigger a success/error message and verify the status icons render
7. Check the header bar -- user icon, notifications bell, and dropdown chevron should all render
8. Inspect an icon in dev tools: verify the SVG path data matches what Heroicons would produce (same shapes as before the change)

**Expected:** All icons render as inline SVGs with correct sizing (size-5 default). No broken/missing icons. Visually identical to the previous heroicons rendering.

## Test 2: Switch to Lucide icon set

**Goal:** Confirm that changing `FREEDOM_LS_ICON_SET` swaps all icons.

1. Stop the server
2. In `config/settings_base.py`, add: `FREEDOM_LS_ICON_SET = "lucide"`
3. Restart the server
4. Navigate to the same pages as Test 1
5. Verify that icons are visibly different from the Heroicons set (Lucide has a distinct 2px stroke style)
6. Verify no broken/missing icons -- every place that had an icon in Test 1 should still have an icon

**Expected:** All icons render but with Lucide's visual style. No errors in the browser console.

**Cleanup:** Remove the `FREEDOM_LS_ICON_SET` setting after testing.

## Test 5: Icon overrides with default set

**Goal:** Confirm that `FREEDOM_LS_ICON_OVERRIDES` overrides individual icons.

1. Stop the server
2. In `config/settings_base.py`, add:
   ```python
   FREEDOM_LS_ICON_OVERRIDES = {
       "success": "star",
   }
   ```
3. Restart the server
4. Navigate to a page that shows the "success" icon (e.g., course completion or messages)
5. Verify the success icon now renders as a star instead of the default check-circle
6. Verify all other icons are unchanged

**Expected:** Only the overridden icon changes. All other icons use the default set's mapping.

**Cleanup:** Remove the override setting after testing.

## Test 11: Force parameter removed

**Goal:** Verify the `force` parameter is fully removed.

1. Search the codebase for any remaining references to `force` in icon-related files:
   - `cotton/icon.html` should not have a `force` prop
   - `icon_tags.py` should not have a `force` parameter
   - No template should use `force="true"` with `<c-icon>`
2. Try rendering `<c-icon name="arrow-right" force="true" />` in a test template -- it should either ignore the `force` param (cotton treats unknown props as slot content) or error

**Expected:** No functional `force` parameter exists anywhere in the icon system.
