# Spec: Public browsing for unauthenticated users

## Summary

Let visitors who are **not logged in** see what the platform offers and explore courses
before creating an account. Three surfaces become public — the home page (the existing
dashboard at `/`), the course catalogue, and course detail pages. Login is required only at
the **committing action** (enrol / apply), via a deferred-login flow that auto-completes the
intended action after the user logs in *or signs up*. Because the catalogue and course pages
become public, we also add basic SEO (per-page title/description, `schema.org` JSON-LD, a
dynamic per-site sitemap, and a robots.txt).

The backend is already largely anonymous-safe: the course-listing and course-access helpers
short-circuit for `AnonymousUser`, and site/multi-tenant filtering is request-driven and
applies identically to anonymous requests. The work is mostly **removing the gate on the
right views, fixing two anonymous-user bugs in the access backends, adapting templates, and
threading `next` through the auth flow** — not rebuilding logic.

This spec is global: anonymous browsing is on for all sites. Per-site "how much is public"
toggles are explicitly out of scope (see §9).

## Why this matters

Today anonymous visitors are redirected to login the instant they try to view anything
(`@login_required` on every `student_interface` view; `LOGIN_REDIRECT_URL = "/"`). That is a
"login wall before any value" — the visitor cannot self-qualify, so they bounce. Letting them
browse first, and only asking for an account when they commit, is the standard pattern across
comparable platforms (see the companion research files) and is expected to materially improve
conversion from visitor to registered learner.

## 1. Surface: Home page (dashboard at `/`, anonymous variant)

Reuse the existing dashboard view/template (`student_interface.views.dashboard`,
`student_interface/templates/student_interface/dashboard.html`) and branch on
`request.user.is_authenticated`. Do **not** build a separate marketing page.

**For anonymous users:**

- **Add an anonymous-only hero** at the top of the page: a short value-proposition headline
  and subtext introducing the platform, plus at most one body CTA into the catalogue
  ("Browse all courses"). The hero renders **only** when the user is anonymous — authenticated
  users never see it.
- **Hide the logged-in-only sections:** the "Welcome back" greeting, "In Progress"
  (`registered_courses`), and "Learning History" (`completed_courses`). These are meaningless
  for an anonymous visitor.
- **Keep discovery content:** the "Available courses" section (the existing
  `available-courses` partialdef, up to a few featured courses). "Recommended Courses" is
  inherently empty for anonymous users and is therefore not rendered.
- **Do not** render empty personalised sections with "sign in to see this" placeholders —
  that reads as a locked dead-end. Omit them entirely.
- The greeting block (currently hardcoded "Welcome back, …") must have an anonymous branch:
  show the hero instead, never a personalised greeting.

**For authenticated users:** unchanged — the dashboard renders exactly as it does today
(greeting, In Progress, Recommended, Available, Learning History, backend panels).

**Header:** add a "Log in" / "Sign up" affordance for anonymous visitors. Today
`header_bar.html` hides the entire nav behind `{% if user.is_authenticated %}`, so anonymous
visitors get no auth controls. Surface the existing `partials/login_prompt.html` (Login + Sign
Up buttons) in the header for anonymous users. See §4 for the `next` requirement on these links.

**Bug to fix (anonymous dashboard):** `dashboard` calls
`backend.get_dashboard_contributions(user=request.user)` unconditionally. The
`ApplicationCourseAccessBackend` implementation calls `get_active_applications(user)` →
`CourseApplication.objects.filter(user=user)`, which is not anonymous-safe. Guard so dashboard
contributions are only fetched for authenticated users (anonymous users get no backend panels).

## 2. Surface: Course catalogue (`all_courses`)

Make `student_interface.views.all_courses` (`courses/`) publicly viewable by removing
`@login_required`. The data path is already anonymous-safe: `get_course_listing` returns every
course with `NOT_REGISTERED` status at 0% progress for anonymous users, and `get_all_courses()`
is site-filtered via `SiteAwareManager`.

**Course cards must let a visitor self-qualify.** Today the not-registered card/row shows only
the title and a "Not registered" eyebrow. Add to each card:

- An **access-model badge**: "Free" vs "By application", rendered with the existing `<c-chip>`
  component. The badge is a dedicated, **config-only** backend signal: `get_access_badge(course=)`
  returns an `AccessBadge(label, variant)` (the backend owns both the copy and the chip variant),
  issuing **no per-user queries** so a listing can call it once per course without scaling
  registration lookups. It is a **separate** signal from the detail page's `enrolment_summary`,
  and the sibling of the config-only `is_accessible_for_free(course=)` (§5.2). Surface it into the
  catalogue listing — it rides on each `CourseListingEntry.access_badge` and the view stamps it
  onto the course via `stamp_course_access_badge` — so cards render `{{ course.access_badge.label }}`
  without per-card backend calls. Do **not** read `Course.access_config` directly in
  templates/views — go through the access backend.
- The "Not registered" status eyebrow is meaningless to a logged-out visitor and should be
  suppressed for anonymous users (the access badge replaces it as the at-a-glance signal).

Where the card already has the data, also surface short description / level / duration if it
can be done by reusing existing card structure; richer card redesign is not required by this
spec. Card links continue to point at the public course detail page.

## 3. Surface: Course detail (`course_detail`)

Make `student_interface.views.course_detail` (`courses/<slug>/detail/`) publicly viewable by
removing `@login_required`. Anonymous visitors see: title, subtitle, category, the hero stats
strip (lessons, level, duration, access model), "About this course"
(`course.rendered_content`), "What you'll learn" (`course.learning_outcomes`), and the
table-of-contents. With `can_access_content=False` (always the case for unregistered/anonymous
users) every ToC item already renders as `BLOCKED` with no URL — that is correct and expected.

**Access-model signal + acquisition copy near the CTA.** The access decision carries three
backend-owned copy fields so the template holds no access-type-specific marketing (a gated course
must not inherit the free funnel's copy). None-valued fields are omitted:

- `enrolment_summary` — the hero "Enrolment" stat-card value: free → **"Free · open"**, gated →
  **"By application"**. This is the at-a-glance signal near the CTA.
- `acquisition_heading` / `acquisition_subtext` — the sign-up panel heading + subtext: free →
  "Free · open to everyone" / "One click. No credit card."; gated → "Application required" /
  "Apply and we'll review your request."

**Action-forward CTA wording.** The button must not mention login; the auth requirement appears
only after the click.

**The acquisition CTA (for a not-registered visitor) is owned by the access backend, not
hardcoded in the view/template.** Additional access models will be added in future (e.g. paid,
invite-only, externally-gated), and each must be able to supply its own committing-action
destination without editing `course_detail` or the detail template. For a **not-registered**
visitor the view reads the backend-owned, fully resolved `cta_label` / `cta_url` off the
`CourseAccessDecision` and renders them generically — a new access model supplies its own CTA
simply by returning its own `cta_label` / `cta_url`, with no change to the detail page. The
access models that exist today resolve, for a not-registered visitor, to:

- Free course → **"Enrol for free"**, target
  `student_interface:initiate_course_access` (`courses/<slug>/access/`).
- Application-gated course, **no existing application** → **"Apply now"**, target
  `course_applications:apply` (`applications/apply/<slug>/`).
- Application-gated course, **existing application** → **"View my application"**, target
  `course_applications:status` (the applicant's status page). The gated backend detects a prior
  application and points a returning applicant at their status instead of re-applying.

These are the current backends' values, not an exhaustive `if free / elif application` branch in
the view — a new access model adds its CTA by returning its own `cta_label` / `cta_url`, with no
change to the detail page or the deferred-login routing (§4.1).

**Registered-learner CTA is computed in the view, not from the decision.** For an
already-registered learner the view uses two helpers in `student_interface.views` rather than the
backend decision: `_detail_cta_label` resolves the three-state progress-aware vocabulary
(**"Start course"** / **"Continue"** / **"Review course"**) from a single `CourseProgress` lookup,
and `_detail_start_url` resolves the target (into the first course item, or `course_home` /
`initiate_course_access` as appropriate). The registered path **deliberately does not** consume
`decision.cta_label` — the free backend returns "Continue" there, which would flatten the
three-state vocabulary. So `decision.cta_label` / `cta_url` govern only the not-registered
acquisition CTA; the registered CTA is view-owned and unchanged from before this feature.

**Bug to fix (anonymous detail on application-gated courses):**
`get_course_access_backend().get_access(user=request.user, course=course)` is called for every
visitor. For `ApplicationCourseAccessBackend` the anonymous path falls into the
`application_gated` branch and calls `get_application_for_course(user=user, course=course)` →
`CourseApplication.objects.filter(user=user, course=course)`, which is not anonymous-safe.
Guard the application lookup so an anonymous user is treated as "no existing application"
(decision = apply), without issuing the unsafe query. This bug also affects the catalogue badge
in §2 if the access decision is computed there, so the guard belongs in the backend/query
layer, not the view.

## 4. Deferred login (the committing action)

When an anonymous user clicks a CTA ("Enrol for free" / "Apply now" / "Log in" / "Sign up"),
they go through the **existing full-page allauth login/signup flow** via `?next=` — no modal.
After authenticating (login **or** signup), the intended action completes and they land on the
right place, with no extra dead-end click.

### 4.1 Routing into login with `next`

- Header Login/Sign-up links must carry `?next=<current-page-path>`. Header
  `login_prompt.html` currently passes no `next`; it must append the current page's path so a
  visitor who logs in from any public page returns there.
- Course-detail **acquisition** CTAs (the not-registered case, §3) render the backend-owned
  `cta_url` as the href — the committing-action target itself (`initiate_course_access` / `apply`),
  not a `login?next=` link. (A registered learner's CTA is view-owned, §3, and never reaches an
  anonymous visitor.) Because that
  target is still `@login_required`, Django redirects an anonymous visitor to `LOGIN_URL` with
  `?next=<that-url>` automatically. So the deferred-login `next` is server-constructed (it is the
  resolved committing-action URL, never client input) without the detail page building it
  explicitly, and a new access model gets correct deferred routing for free by returning its own
  `cta_url`. For an already-authenticated, not-registered user the same href enrols/applies
  directly with no login detour.

### 4.2 Intent auto-completes after auth

- **Free enrol:** `initiate_course_access` accepts GET and performs
  `UserCourseRegistration.objects.get_or_create(...)` idempotently, then redirects into the
  course (`course_home` → first item). So a logged-in user arriving via
  `next=/courses/<slug>/access/` is enrolled and dropped into the course with **no second
  click**. This already works once the gate is lifted and `next` survives the auth flow — keep
  it idempotent; do not introduce an intermediate "you need to enrol" page.
- **Apply (gated):** `apply` shows a GET confirmation page, and a POST creates the
  `CourseApplication`. Landing the authenticated user on this confirmation/apply page **is** the
  intended destination per the idea ("land on the application form") — we do **not** auto-POST.
  This is correct: applying is a deliberate action, and the form may collect input. Do not
  render the gated apply form to an anonymous user (their input would be lost across login);
  the `@login_required` gate on `apply` already enforces this. A learner who has **already
  applied** gets the "View my application" CTA (§3) instead and lands on their application status
  page, not the apply form.

### 4.3 `next` must survive the new-user signup path (critical)

`next` is most commonly lost on the **new-user signup** path. The flow must preserve it:

- The allauth **login** template threads `next` via the hidden `redirect_field`. ✓ (verify)
- The project **signup** template (`account/signup.html`) retains `redirect_field`. ✓ (verify)
- **Gap to close:** when a newly-signed-up user must complete extra registration forms,
  `RegistrationCompletionMiddleware` redirects to `accounts:complete_registration` **without
  carrying `next`**. The `complete_registration` view/template already read and re-emit `next`
  (hidden field; `_safe_post_completion_redirect`), but the value never reaches them because the
  middleware drops it on the forced redirect. Fix the middleware so the original `next`
  destination is preserved through the completion step and honoured by
  `_safe_post_completion_redirect` on completion. The new-user path (signup → complete
  registration → land on intended destination) must be covered by an explicit test.

### 4.4 Open-redirect safety

- Any `next` read from the request must be validated with
  `url_has_allowed_host_and_scheme` (same-host paths only) before redirecting. allauth's native
  `next` handling already does this for its own success redirects; the
  `complete_registration` path already uses `_safe_post_completion_redirect`
  (`accounts/views.py`). Ensure the middleware fix routes through the same validation and does
  not introduce an unvalidated redirect.
- Prefer server-constructed `next` targets (from the course slug) over client-supplied values
  wherever the destination is known.

## 5. Basic SEO / discoverability

Because the catalogue and course pages are now public, make them discoverable. Keep it
**basic** — no structured-data fields we have no data for.

### 5.1 Per-page title + meta description

- `_base.html` already defines `{% block head_title %}` and `{% block meta_description %}`
  (default "Learning management system"). Override `meta_description` on the catalogue and
  course-detail templates (currently neither does).
- Course detail: title `{{ course.title }}` (already set); description from
  `course.description`, falling back to `course.subtitle` (then a sensible site default) when
  `description` is blank — `description` is optional and often empty.
- Catalogue: a descriptive title (apply the existing `— {{ site_name }}` suffix pattern used by
  the course-player base) and a catalogue-level meta description.

### 5.2 JSON-LD structured data

Inject via the existing `{% block extra_head %}` slot in `_base.html`.

- **Course detail** → `schema.org/Course`. Populate only fields we can honestly source:
  `name` (`course.title`), `description` (`course.description` / `subtitle` fallback),
  `url` (absolute, via `request.build_absolute_uri(reverse(...))`),
  `educationalLevel` (`course.get_difficulty_display()` when set),
  `timeRequired` (ISO-8601 via `course.iso_estimated_duration()` — a serialiser added for this;
  the existing `display_estimated_duration()` is human-readable, not ISO-8601),
  `teaches` (`course.learning_outcomes` when non-empty),
  `isAccessibleForFree` (from the access backend decision, **not** `access_config`).
  Omit `provider`, `image`, `author`, `courseCode` — no model data exists for them.
- **Catalogue** → `schema.org/ItemList` of the visible `Course` items (each an item with its
  course-detail URL).

### 5.3 Sitemap + robots.txt (dynamic, per-site)

- Add `django.contrib.sitemaps` to `INSTALLED_APPS` and register a `sitemap.xml` URL in
  `config/urls.py`.
- Write the `Sitemap` classes in `config/sitemaps.py` (the composition root, which may depend on
  any app): `StaticViewSitemap` for the static catalogue URL and `CourseSitemap` for `Course`
  objects (the site-filtered queryset). Django's sitemap framework resolves the domain per-request
  via the Sites framework, so each tenant domain serves its own correct URLs automatically.
- Serve `robots.txt` via a small **dynamic view** (`robots_txt` in `config/views.py`, not a single
  static file) so it is correct per tenant domain: allow crawling of the public catalogue and
  course paths and reference that site's `sitemap.xml` (built with `request.build_absolute_uri`).
  Ensure nothing disallows `/courses/` or `/courses/<slug>/detail/`. Register the robots URL ahead
  of any static catch-all.

### 5.4 Absolute URLs in a multi-tenant context

Build absolute URLs from the request (`request.build_absolute_uri`) or the current `Site`
domain (`get_cached_site(request).domain`), never a hardcoded host, so sitemap `<loc>` and
JSON-LD `url` values are correct per tenant.

## 6. Cross-cutting: anonymous-safety guards

Two backend queries are not anonymous-safe and **must** be guarded (they will raise / produce
broken queries when the gate is removed):

1. `ApplicationCourseAccessBackend.get_dashboard_contributions` →
   `get_active_applications(user)` (used by `dashboard`). Guard: skip for anonymous users.
2. `ApplicationCourseAccessBackend` access path →
   `get_application_for_course(user, course)` (used by `course_detail` and the catalogue access
   badge). Guard: an anonymous user has no application; return the "apply" decision without
   querying.

Place the guards in the query/backend layer so every caller (view, catalogue badge) is
protected, not just one view. The already-safe helpers (`get_current_courses`,
`get_completed_courses`, `get_recommended_courses`, `get_course_listing`, `get_resume_index`,
`get_course_index`, `is_registered_for_course`) need no change.

## 7. Decisions made (and why)

- **Reuse the dashboard, branch on auth** rather than a separate marketing page — less
  duplication, single source of truth for course discovery, matches the idea.
- **Anonymous-only hero** at the top of the dashboard, hiding only the logged-in-only sections
  (In Progress, Learning History) and the greeting — gives anonymous visitors context without a
  bespoke landing page, while leaving the authenticated dashboard untouched. (User decision.)
- **Full-page allauth flow via `?next=`, not a modal** — lower-risk, reuses existing auth, more
  accessible.
- **Free enrol auto-completes; apply lands on the confirmation/apply page** — matches each
  action's intent (instant for free, deliberate for application) and avoids losing form input
  across login.
- **Dynamic per-site robots.txt + sitemaps framework** — correct for the multi-tenant Sites
  setup where each tenant has its own domain. (User decision.)
- **Server-constructed `next`** — the detail-page **acquisition** CTA (not-registered) links
  straight at the backend-owned committing-action URL and lets `@login_required` generate
  `?next=<that-url>`, so the value is always a resolved internal URL, never client input;
  `url_has_allowed_host_and_scheme` validation guards the remaining read points — prevents
  open-redirect/phishing. (The registered-learner CTA is view-owned, §3, and does not enter the
  deferred-login path.)
- **JSON-LD limited to honestly-sourced fields** — avoids fabricated structured data that could
  be flagged by search engines.

## 8. Success criteria

1. An anonymous visitor can load `/`, the catalogue, and any course detail page without being
   redirected to login. No `@login_required` redirect occurs on those three surfaces.
2. The anonymous home page shows the hero + discovery (Available) courses and a catalogue CTA;
   it shows **no** greeting, In Progress, Learning History, recommended, or "sign in to see
   this" placeholders. The authenticated dashboard is unchanged.
3. Catalogue cards show a Free vs By-application badge; a visitor can tell the access model
   before clicking.
4. Course detail renders for anonymous users on **both** free and application-gated courses
   (no crash on the application-gated path), with the access badge near the CTA and
   action-forward CTA wording ("Enrol for free" / "Apply now").
5. Header shows Log in / Sign up for anonymous users, carrying `next` to return to the current
   page after auth.
6. **Deferred login completes intent:**
   - Anonymous → "Enrol for free" → log in **or sign up** → lands in the course content,
     enrolled, with no extra click.
   - Anonymous → "Apply now" → log in **or sign up** → lands on the apply confirmation/form
     (authenticated), ready to submit.
   - The **new-user signup path that requires completing registration forms** preserves `next`
     and still lands on the intended destination (explicitly tested).
7. `next` is always validated with `url_has_allowed_host_and_scheme`; an off-host `next` is
   ignored and falls back to `LOGIN_REDIRECT_URL`.
8. Catalogue and course-detail pages emit a per-page `<title>` and `<meta name="description">`,
   valid `schema.org/Course` (detail) and `ItemList` (catalogue) JSON-LD, are listed in a
   per-site `sitemap.xml`, and are not disallowed by `robots.txt`.
9. Site/tenant isolation holds for anonymous users: a visitor on site A never sees site B's
   courses, sitemap entries, or JSON-LD URLs.
10. The full test suite passes, including new tests for: anonymous access to the three
    surfaces, the two anonymous-safety guards, free-enrol and apply deferred-login completion
    (login and signup), the new-user-with-registration-forms `next` survival, and `next`
    open-redirect rejection.

## 9. Out of scope (leave for future specs)

- Content previews / sample lessons for anonymous users (beyond the existing blocked ToC).
- Guest checkout / paid-course purchasing flows.
- Catalogue search and rich filtering beyond what already exists.
- Per-site configuration toggles for how much is public (this spec is global).
- Card redesign beyond surfacing the access badge (and existing fields).
- Ratings / structured-data fields with no backing model data
  (`provider`, `image`, `author`, …).
- New access models themselves (paid, invite-only, externally-gated, …). This spec only keeps
  the CTA (§3) backend-owned so a future access model can supply its own `cta_label` / `cta_url`;
  it does not add any new access model.

## 10. Anti-patterns to avoid (from research)

- Forcing signup before any value is shown (the thing we are fixing) — do not reintroduce it on
  sub-pages.
- Empty personalised sections shown to anonymous users as "log in to see this".
- Unclear gating — a visitor must be able to tell Free vs By application before clicking.
- Landing the user on a generic dashboard after login instead of completing their intent.
- Losing the new-user signup path's `next` (test this path explicitly).
