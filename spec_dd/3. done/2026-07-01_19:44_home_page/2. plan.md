**TDD** throughout (`fls:testing`): write the failing test first, then the
implementation. Do not write every test up front — write the test(s) for the unit

this plan; re-verify again at implementation time.

## Phase 0 — Anonymous-safety guards in the query/backend layer

These guards must land **first**: every later phase that removes a `@login_required`
gate depends on them, and putting them in the query layer protects every caller
(view + catalogue badge) at once (spec §6).

### 0.1 Guard `get_application_for_course` and `get_active_applications`

`freedom_ls/course_applications/queries.py` — both helpers run
`CourseApplication.objects.filter(user=user, ...)`, which is not anonymous-safe.

```python
def get_application_for_course(
    *, user: AbstractBaseUser, course: Course
) -> CourseApplication | None:
    if not user.is_authenticated:
        return None
    from freedom_ls.course_applications.models import CourseApplication
    return CourseApplication.objects.filter(user=user, course=course).first()

def get_active_applications(
    user: AbstractBaseUser,
) -> QuerySet[CourseApplication]:
    from freedom_ls.course_applications.models import CourseApplication
    if not user.is_authenticated:
        return CourseApplication.objects.none()
    return CourseApplication.objects.filter(user=user).select_related("course")
```

(Type hints are mandatory per CLAUDE.md — keep them on the real signatures; the
existing `TYPE_CHECKING` imports in `queries.py` already cover these types.)

Mirrors the existing anonymous-safe pattern in `student_interface/utils.py`
(`get_current_courses`, `get_recommended_courses`, …).

- `get_application_for_course` guard ⇒ `ApplicationCourseAccessBackend.get_access`
  falls through to the "Apply now" decision for an anonymous user with no DB query
  (fixes spec §3 / §6 bug 2).
- `get_active_applications` guard ⇒
  `ApplicationCourseAccessBackend.get_dashboard_contributions` returns `[]` for an
  anonymous user (fixes spec §1 / §6 bug 1) without the unsafe query.

### 0.2 The dashboard's contribution fetch

Even with 0.1, `dashboard` (`student_interface/views.py:165`) should not call the
backend for anonymous users at all (the anonymous dashboard has no panels — spec §1).
That **view-level** guard is implemented in Phase 2.1 (it is part of the dashboard
restructure), and its test (anonymous `GET /` issues no `get_dashboard_contributions`
call) lives in Phase 2.1's test list — not here. The query-layer guard in 0.1 is the
defence-in-depth that protects every caller (including the catalogue-badge path) and
is what Phase 0 lands and tests.

**Tests for Phase 0 (write first — query/backend layer only):**
- `get_application_for_course(user=AnonymousUser(), course=...)` returns `None`
  with **no** query (assert via `django_assert_num_queries(0)`).
- `get_active_applications(AnonymousUser())` returns an empty queryset, no query.
- `ApplicationCourseAccessBackend.get_access(user=AnonymousUser(), course=gated)`
  returns the `cta_label == "Apply now"` decision and issues no application query.
- `get_dashboard_contributions(user=AnonymousUser())` returns `[]`.

### 1.2 Honest "is accessible for free" signal for the catalogue badge + JSON-LD

The catalogue badge (Phase 3) and the JSON-LD `isAccessibleForFree` (Phase 6) both
need a free-vs-gated signal **without reading `Course.access_config`** (forbidden by
the model contract) and **without a per-card backend call** for the badge.

Add one field to `CourseAccessDecision` (`course_access/backends.py`):

```python
@dataclass(frozen=True)
class CourseAccessDecision:
    ...
    is_accessible_for_free: bool = True  # honest free/gated signal for badge + JSON-LD
```

- `FreeOnlyCourseAccessBackend` only ever handles free courses → leave the default
  `True` on both its decisions.
- `ApplicationCourseAccessBackend`: set `is_accessible_for_free=False` on the two
  `application_gated` decisions ("Apply now" and "View my application"). The
  registered / free branches delegate to `super()` and keep `True`.

This keeps the free/gated decision backend-owned and additive: a future paid backend
sets the flag itself. The human-facing badge **label** ("Free" / "By application")
is derived from this flag in the listing builder (Phase 3), not duplicated as copy in
the backend.

**Tests (write first):**
- free decision → `is_accessible_for_free is True`.
- gated "Apply now" and "View my application" decisions → `is_accessible_for_free is False`.

## Phase 2 — Home page (anonymous dashboard variant)

### 2.1 `dashboard` view (`student_interface/views.py:125`)

Remove `@login_required`. **Keep a single code path** — do not add a separate
anonymous early-return that re-implements the available-courses loop already in the
view (that loop, `views.py:150-160`, is the only part both auth states share). Guard
the personalised pieces with `if request.user.is_authenticated`:

```python
def dashboard(request: HttpRequest) -> HttpResponse:
    backend = get_course_access_backend()
    is_auth = request.user.is_authenticated

    registered_courses = get_current_courses(request.user)      # [] for anon (already guarded)
    completed_courses = get_completed_courses(request.user)     # [] for anon
    recommended_courses = get_recommended_courses(request.user) # none() for anon

    # Per-registered-course annotation + recommended cards: only meaningful when authed.
    if is_auth:
        for course in registered_courses:
            ...  # unchanged annotate-next-up loop
        for rec in recommended_courses:
            ...  # unchanged

    registered_ids = {c.id for c in get_course_registrations(request.user)} if is_auth else set()
    recommended_ids = {rec.collection_id for rec in recommended_courses}

    # Shared discovery loop — runs for both auth states (empty id-sets for anon).
    visible_courses = backend.filter_visible(user=request.user, courses=get_all_courses())
    available_courses = []
    for course in visible_courses:
        if course.id in registered_ids or course.id in recommended_ids:
            continue
        setattr(course, "is_registered", False)  # noqa: B010
        available_courses.append(course)
        if len(available_courses) == 3:
            break

    dashboard_panels = []
    if is_auth:
        contributions = backend.get_dashboard_contributions(user=request.user)
        dashboard_panels = [render_to_string(c.template_name, c.context, request=request)
                            for c in contributions]

    return render(request, "student_interface/dashboard.html", {
        "registered_courses": registered_courses,
        "completed_courses": completed_courses,
        "recommended_courses": recommended_courses,
        "available_courses": available_courses,
        "dashboard_panels": dashboard_panels,
    })
```

Notes:
- `get_current_courses`, `get_completed_courses`, `get_recommended_courses`,
  `get_course_registrations`, `filter_visible`, `get_all_courses` are all already
  anonymous-safe; the `is_auth` guards skip the *backend-panel* and per-course
  annotation work that is meaningless for anonymous users (and avoids the
  `get_dashboard_contributions` call — the view-level half of spec §6 guard 1).
- The authenticated result is identical to today's (the loop and context keys are
  unchanged); only the personalised work is now conditionally skipped.
- The badge attribute (Phase 3.2) renders only when present, so the anonymous
  discovery cards will show the access badge automatically once Phase 3 stamps
  `access_badge_label` onto courses going into `available_courses` (do the same
  stamping here as in the catalogue so the home discovery cards and catalogue agree).

## Phase 5 — Deferred login (intent survives auth)

The committing-action targets (`initiate_course_access`, `apply`) **stay
`@login_required`**. The detail-page CTA links straight at them; Django's
`@login_required` builds `?next=<that-url>` automatically for anonymous clicks
(spec §4.1, §4.2). So most of this phase is *verification + one middleware fix*.

### 5.1 Verify `next` threads through login + signup (no code expected)

- allauth login template emits `{{ redirect_field }}` — **confirmed present**
  (`.venv/.../allauth/templates/account/login.html:27`).
- Project signup template `account/signup.html` emits `{{ redirect_field }}` —
  **confirmed present**.
- `initiate_course_access` (`views.py:293`) already does idempotent
  `get_or_create` then redirects into `course_home` → first item. Keep it idempotent;
  do **not** add an intermediate "you need to enrol" page (spec §4.2).
- `apply` shows a GET confirmation page; landing the authenticated user there **is**
  the intended destination — do **not** auto-POST (spec §4.2). Its `@login_required`
  already prevents rendering the gated form to an anonymous user.

### 5.2 Fix `RegistrationCompletionMiddleware` to preserve `next` (the critical gap)

`freedom_ls/accounts/middleware.py:116` redirects to
`accounts:complete_registration` **dropping** the in-flight `next`. For the new-user
signup → forced-completion path this loses the intended destination
(spec §4.3). Fix:

```python
from urllib.parse import quote
from django.utils.http import url_has_allowed_host_and_scheme
...
        # Preserve the in-flight destination through the forced completion step.
        target = reverse("accounts:complete_registration")
        next_url = request.GET.get("next") or request.path
        if next_url and url_has_allowed_host_and_scheme(
            next_url,
            allowed_hosts={request.get_host()},
            require_https=request.is_secure(),
        ):
            target = f"{target}?next={quote(next_url)}"
        return redirect(target)
```

- Add `from urllib.parse import quote` to the middleware imports.
- The `complete_registration` view already reads `next` (GET/POST) and re-emits it,
  and `_safe_post_completion_redirect` re-validates with
  `url_has_allowed_host_and_scheme` (`accounts/views.py`). The
  `accounts/complete_registration.html` template **already** emits
  `<input type="hidden" name="next" value="{{ next }}">` inside its `<form>` (verified)
  — so once the middleware carries `next` into the GET, the rest of the chain
  (view → hidden field → POST → `_safe_post_completion_redirect`) already honours it.
  No template change is needed; just confirm the hidden field is still present.
- Route the middleware redirect through the **same** validation so no unvalidated
  redirect is introduced (spec §4.4). Use `request.GET.get("next")` when present
  (the value allauth forwarded after signup) and fall back to `request.path` (the
  intercepted destination, e.g. `/courses/<slug>/access/`). Use `request.path`, not
  `get_full_path()`, so an unrelated query string on the intercepted URL is not
  dragged along — the committing-action targets carry no meaningful query string.
- Consult `fls:registration` for the middleware's exempt-list semantics and its
  documented open-redirect caveat.

### 5.3 Open-redirect safety (spec §4.4)

Every `next` read point must validate with `url_has_allowed_host_and_scheme`
(same-host only): the middleware (5.2) and the already-correct
`_safe_post_completion_redirect`. Prefer server-constructed `next` (the resolved
committing-action URL from the course slug) over client input — which the
detail-CTA-via-`@login_required` flow already gives us.

**Tests (write first):**
- **Free deferred-login (login):** anonymous → GET `…/access/` → 302 to
  `login?next=…/access/`; log in → end state: registered + landed on first course item
  (no extra click).
- **Free deferred-login (signup):** same, via signup instead of login.
- **Apply deferred-login (login + signup):** anonymous → "Apply now" target →
  authenticate → lands on the apply confirmation page (authenticated, not POSTed).
- **New-user + registration-forms `next` survival (critical, explicit):** a site with
  an additional registration form; signup with `next=/courses/<slug>/access/` →
  middleware redirects to `complete_registration?next=/courses/<slug>/access/` →
  submit forms → lands on `…/access/` → enrolled + in course.
- **Open-redirect rejection:** `next=https://evil.example/` is ignored everywhere it
  is read; redirect falls back to `LOGIN_REDIRECT_URL` (`/`).

## Phase 6 — Basic SEO / discoverability

Keep it basic (spec §5) — only honestly-sourced fields.

### 6.1 Per-page `<title>` + `<meta description>`

`_base.html` already defines `{% block head_title %}` and `{% block meta_description %}`.

- **Course detail** (`course_detail.html`): title is already `{{ course.title }}`.
  Compute the description **once in the view** —
  `meta_description = course.description or course.subtitle or "<site default>"` — pass
  it as a context value, and render `{% block meta_description %}{{ meta_description }}{% endblock %}`.
  This single `meta_description` variable is **reused** when building the JSON-LD dict
  (§6.2) — do not re-derive the fallback there.
- **Catalogue** (`all_courses.html`): set a descriptive
  `{% block head_title %}` using the existing `— {{ site_name }}` suffix pattern, and a
  catalogue-level `{% block meta_description %}`.
- Copy/voice per `brand-guidelines`.

### 6.2 JSON-LD via `{% block extra_head %}`

- **Course detail → `schema.org/Course`.** Build the dict **in the view body**
  (`decision`, `course`, `request`, `meta_description` are all in scope there — no need
  to push a big dict through the template), pass it as a context value `json_ld`, and
  render it safely with Django's `json_script` filter inside `{% block extra_head %}`:
  `{{ json_ld|json_script:"course-jsonld" }}`. **Do not** use `{{ json_ld|safe }}` —
  that does not escape `</script>` in a course title/description and is an XSS vector;
  `json_script` escapes `<`, `>`, `&` and emits the wrapping `<script>` tag itself
  (it defaults to `type="application/json"`, which search engines accept; if a literal
  `type="application/ld+json"` is required, serialise with `django.utils.html.json_script`
  / `escapejs` equivalent in the view rather than `|safe`).
  Populate only:
  - `name` = `course.title`
  - `description` = the `meta_description` already computed in §6.1 (reused, not re-derived)
  - `url` = `request.build_absolute_uri(reverse("student_interface:course_detail",
    kwargs={"course_slug": course.slug}))`
  - `educationalLevel` = `course.get_difficulty_display()` **when set**
  - `timeRequired` = ISO-8601 from `course.estimated_duration` — via the new
    `Course.iso_estimated_duration()` method (§6.4); omit when duration unset
  - `teaches` = `course.learning_outcomes` when non-empty
  - `isAccessibleForFree` = `decision.is_accessible_for_free` (Phase 1.2 — **not**
    `access_config`)
  - **Omit** `provider`, `image`, `author`, `courseCode` (no model data).
- **Catalogue → `schema.org/ItemList`** of the visible courses, each item carrying its
  absolute course-detail URL. Build the dict in `all_courses` and render it the same
  way (`json_script`) in `extra_head`.

### 6.3 Sitemap + robots.txt (dynamic, per-site)

- Add `django.contrib.sitemaps` to `INSTALLED_APPS` (`config/settings_base.py`).
- New module `freedom_ls/base/sitemaps.py` (`base` is the natural home and already
  exists). It only `reverse()`s `student_interface` URL **names** at request time — it
  imports no `student_interface` Python module, so it creates **no new cross-app import
  dependency** (the URLconf is loaded at runtime; `reverse` by name is safe). Contains:
  - `StaticViewSitemap` → the catalogue URL (`student_interface:courses`).
  - `CourseSitemap` → `Course.objects.all()` (site-filtered automatically by
    `SiteAwareManager` per `fls:multi-tenant`), with a `location(self, obj)` method that
    `reverse(...)`s `student_interface:course_detail`. Do **not** add `get_absolute_url`
    to the `Course` model — the sitemap's `location()` is the seam.
- Register `path("sitemap.xml", sitemap, {"sitemaps": {...}}, name="sitemap")` in
  `config/urls.py`. Django's sitemap framework resolves the domain per-request via the
  Sites framework, so each tenant serves its own URLs.
- **robots.txt** as a small dynamic view living in **`freedom_ls/base/views.py`** (new,
  tiny — keeps SEO out of `student_interface`, which is not its concern, and out of an
  inline lambda in `config/urls.py` so it is testable). It returns
  `HttpResponse(content_type="text/plain")` allowing `/courses/` and
  `/courses/<slug>/detail/` and referencing this site's `sitemap.xml` via
  `request.build_absolute_uri(reverse("sitemap"))`. Register the robots URL in
  `config/urls.py` **ahead of** the `path("", include(student_interface))` include.
  Ensure nothing disallows `/courses/`.

### 6.4 ISO-8601 duration serialiser

`Course.display_estimated_duration()` is human-readable, not ISO-8601. Add a small
**model method** `Course.iso_estimated_duration() -> str` (no new file; sits next to
`display_estimated_duration` in `content_engine/models.py`) that converts the
`timedelta` to an ISO-8601 duration string (e.g. `PT1H30M`), returning `""` when unset.
Used only by the course-detail JSON-LD. Keep it tiny and well-tested.

### 6.5 Absolute URLs in multi-tenant context (spec §5.4)

Always build absolute URLs from `request.build_absolute_uri(...)` (views) or the
current `Site` domain — never a hardcoded host — for sitemap `<loc>` and JSON-LD `url`.
