# Spec: Remove the home-feature `next`-threading allauth customisations

## Summary

The `home` feature (`spec_dd/3. done/2026-07-01_19:44_home_page`) bolted custom
`next`-threading onto the allauth / registration machinery so a not-yet-logged-in visitor
clicking a course CTA would be carried back to their exact destination through the forced
signup + registration-completion flow. That machinery produced a spaghetti-mess and bugs
that are hard to fix without piling on more spaghetti.

This spec **reverts exactly those `next`-threading changes** so allauth works in standard,
vanilla ways. When a user hits a `@login_required` view, Django + allauth already build and
honour `?next=` on their own; the header still shows working (un-parameterised) Login /
Sign-up links for anonymous visitors. **No new functionality is introduced — this is a
surgical removal only.**

The home feature's *entire* auth/allauth footprint is **two files, one behavioural change
each**, confirmed by diffing the two owning commits against their parents (verified below)
and corroborated by the home spec's plan, its QA report, and the test suite.

## Why this matters

- The custom `next`-threading duplicates behaviour that vanilla Django/allauth already
  provide (`@login_required` → `?next=` generation, and allauth honouring that param on
  login). Re-implementing it by hand added a redirect branch and an open-redirect guard the
  framework already does for us.
- The custom path is the source of the bugs. Rather than layer more machinery on top to
  patch those bugs, removing the machinery returns the flow to a well-trodden, framework-
  supported path with far less surface area to break.
- The two "keep" surfaces below (the earlier "better registration" work and the anonymous-
  header affordance) worked cleanly and are relied upon by the vanilla behaviour we are
  restoring — so they stay.

## Hard constraints (do not violate)

1. **Only revert the allauth `next`-threading added by the `home` spec.** Introduce no new
   functionality — only remove.
2. **Keep the earlier "better authentication" / "better registration" changes** (commit
   `9077da89` era). Those worked great, added no spaghetti, and the vanilla behaviour we are
   restoring relies on them.
3. **Ignore the other in-progress bugfix worktree/spec entirely.** It tries to fix the bugs
   by layering more machinery on. Do not read it, base anything on it, or fix bugs it
   surfaced. Out of scope, full stop.

## In scope — the two reverts

### R1. `freedom_ls/accounts/middleware.py` — `RegistrationCompletionMiddleware.__call__`

The current code (commit `a47921bc`) reads and validates `request.GET.get("next")` with
`url_has_allowed_host_and_scheme`, falls back to `request.path`, and redirects to
`complete_registration?next=<quoted candidate>`.

**Revert:** replace that whole block with the single pre-home line:

```python
return redirect(reverse("accounts:complete_registration"))
```

Then drop the now-unused imports (both are used *only* inside the reverted block — grep
confirms `quote` appears once and `url_has_allowed_host_and_scheme` once, both in
`__call__`):

- `from urllib.parse import quote`
- `from django.utils.http import url_has_allowed_host_and_scheme`

The file docstring does not reference `next` and needs no change.

**Vanilla behaviour after the revert:** a user forced into `complete_registration` no longer
carries a `next`. The pre-existing `complete_registration_view` / `_safe_post_completion_redirect`
(kept — see below) simply receive no `next` from the middleware and gracefully fall back to
`LOGIN_REDIRECT_URL`. That is exactly the vanilla behaviour we want.

### R2. `freedom_ls/base/templates/partials/login_prompt.html`

Strip **only** `?next={{ request.path|urlencode }}` from both `<c-button>` `href`s, leaving:

```django
<c-button href="{{ login_url }}">Login</c-button>
...
<c-button href="{{ signup_url }}">Sign up</c-button>
```

**Do NOT** revert the rest of the file. Commit `9dc71a32` also restyled the wrapper
(`mt-8 flex justify-center gap-4` → `flex items-center gap-2`) and refactored to the
`{% url 'account_login' as login_url %}` variable form so the prompt fits the header bar.
That styling/refactor belongs to the anonymous-header affordance (kept — see below), not to
`next`-threading. Only the `?next=…` query param is removed.

## Explicitly OUT of scope — leave untouched

- **`header_bar.html`** rendering `login_prompt.html` for anonymous users — the "public
  browsing" header affordance (also from `9dc71a32`), not `next`-threading. Keep it and its
  styling. After R2, anonymous visitors still see working, un-parameterised Login / Sign-up
  links; allauth generates `?next=` on its own only when they hit a `@login_required` view.
- **`initiate_course_access` / `apply` views** — already plain `@login_required`; Django
  builds `?next=` automatically. Nothing custom to remove.
- **`_safe_post_completion_redirect`, `complete_registration_view`, its template's hidden
  `next` field, the allauth account adapter, all `account/*` template overrides, and all
  `ACCOUNT_*` / `LOGIN_*` settings and `config/urls.py` allauth wiring** — all pre-date the
  `home` feature ("better registration", commit `9077da89` era) or are stock allauth/Django.
  They survive the revert unchanged.

## Test cleanup (scope; per-test keep/delete/rewrite decided in the plan)

Reverting removes behaviour that a handful of tests assert on. All three test files exist and
their assertions match the map below. The **plan** makes the final keep-vs-delete-vs-rewrite
call per test; this spec fixes the scope and intent.

**Cleanup, not negation.** Where an assertion tied to the removed `next`-threading is dropped,
delete it outright — do **not** replace it with a negative-existence check like
`assert "next=" not in response.url`. Per the project testing norms, asserting the absence of a
value the code no longer produces proves nothing and is an anti-pattern. Keep the assertions
that describe surviving vanilla behaviour (the 302 status, the completion-URL prefix) and simply
remove the ones that described the deleted behaviour.

### `freedom_ls/accounts/tests/test_deferred_login.py` (whole-cloth, added by `a47921bc`)

- **Delete** the tests asserting the removed middleware `next`-forwarding:
  `test_middleware_preserves_next_from_get_param`,
  `test_middleware_falls_back_to_request_path_when_no_next_param`,
  `test_middleware_rejects_off_host_next_and_falls_back_to_path`.
- **Delete or rewrite** `test_next_survives_complete_registration_step` — as written it

  own POST-time validation),
  `test_anonymous_access_to_initiate_redirects_to_login_with_next` and

## Success criteria

1. `middleware.py`'s `RegistrationCompletionMiddleware.__call__` ends with the single
   `return redirect(reverse("accounts:complete_registration"))`, and the `quote` /
   `url_has_allowed_host_and_scheme` imports are gone.
2. `login_prompt.html`'s two buttons link to bare `{{ login_url }}` / `{{ signup_url }}` with
   no `?next=` query param; the wrapper restyle and `{% url … as … %}` refactor remain.
3. `header_bar.html`, the `initiate_course_access` / `apply` views, the "better registration"
   completion view/template/adapter, and all `ACCOUNT_*` / `LOGIN_*` settings are byte-for-byte
   unchanged.
4. Anonymous visitors clicking a course CTA are bounced through vanilla allauth login
   (`@login_required` → `?next=`), with no custom middleware/template `next`-threading
   involved. A visitor who logs in as an **already-registered** user is carried back to the
   CTA destination by allauth's stock `?next=` handling. A visitor who must additionally pass
   through the forced **registration-completion** step deliberately loses that destination and
   lands on `LOGIN_REDIRECT_URL` — this is the accepted vanilla trade-off of the revert (it is
   exactly the intent-preservation the `home` feature added and that we are removing), not a
   regression.
5. No test asserts the removed middleware `next`-forwarding or the removed `login_prompt`
   `?next=` markup; all kept tests (vanilla `@login_required`, pre-existing registration
   `next` handling, public-browsing) still pass. Stale docstrings/comments referencing the
   removed behaviour are cleaned up.
6. `uv run pytest` is green. No new functionality anywhere in the diff.

## Non-goals

- Fixing any bug surfaced by the other in-progress bugfix worktree.
- Changing, "improving", or re-adding any deferred-login / intent-preservation behaviour.
- Touching the "better registration" machinery or any stock allauth/Django config.
