# Frontend QA: Course page small fixes

Manual / Playwright-MCP verification that the two presentational fixes render correctly across viewport widths.

## Setup

1. Pick a free port and start the dev server:
   ```bash
   PORT=$(.claude/fls/scripts/find_available_port.sh)
   uv run python manage.py runserver $PORT
   ```
   Base URL for everything below: `http://127.0.0.1:$PORT`

2. Use the **DemoDev site** for test data (project convention).

3. Log in as a demo learner who has access to the `standard-markdown-demo-finance` course. If you do not already have such a learner, ask the `qa-data-helper` agent to create one — do not invent credentials. The login page is at `http://127.0.0.1:$PORT/accounts/login/`.

4. Use the **fls:use-playwright** skill to drive the browser. Capture screenshots with `mcp__plugin_fls_playwright__browser_take_screenshot` at each viewport size.

## Viewport sizes

For every page below, capture screenshots at all three widths using `mcp__plugin_fls_playwright__browser_resize`:

- Desktop — width 1280, height 800
- Tablet — width 768, height 1024
- Mobile — width 375, height 812

## Test 1 — Topic page, "not yet complete" state (Next is a submit button)

**Goal:** verify the "Next" submit button now shows the right-arrow on the *right* of the label, and the "Previous" link button shows the left-arrow on the *left*.

1. Navigate to `http://127.0.0.1:$PORT/courses/standard-markdown-demo-finance/2/` — topic 2 so that **Previous** is also visible. (Topic 1 has no Previous; topic 2 has both Previous and a not-yet-complete Next.)
   - If topic 2 is already complete for this learner, either pick a learner who hasn't completed it, or pick a higher topic number that is still incomplete and not the last item.
2. Confirm visually:
   - **Previous** button on the left side of the row, with a left-pointing arrow (`arrow-left`) **before** the word "Previous".
   - **Next** button on the right side of the row, with a right-pointing arrow (`arrow-right`) **after** the word "Next".
   - Spacing looks clean — no double-spacing, no missing gap between icon and label.
3. Resize and screenshot at 1280, 768, 375.
4. At 375 (mobile) confirm the buttons still fit on a single row or wrap cleanly without overlapping.

## Test 2 — Topic page, "Finish Course" state

**Goal:** verify the terminal `is_last_item` submit button still renders the `check` icon on the **left** with the label "Finish Course".

1. Navigate to the **last** topic of `standard-markdown-demo-finance` for a learner who has completed every prior item but not yet marked the last one complete. The exact URL depends on the seeded course length — inspect the course's topic list from `http://127.0.0.1:$PORT/courses/standard-markdown-demo-finance/` and pick the last topic.
2. Confirm visually:
   - The single submit button reads "Finish Course".
   - A check icon (`check` semantic name) sits to the **left** of "Finish Course".
   - No right-side arrow.
3. Screenshot at 1280, 768, 375.

## Test 3 — Topic page, "Next is a plain link" state (already-complete topic)

**Goal:** verify the link-style Next button (rendered when `next_url` is truthy and the topic is already complete) shows the arrow on the right.

1. As a learner who has already completed topic 2, navigate to `http://127.0.0.1:$PORT/courses/standard-markdown-demo-finance/2/`.
2. Confirm visually:
   - **Next** button rendered as a link (no surrounding form), with a right-pointing arrow **after** the word "Next".
   - Previous button still shows the arrow on the left.
3. Screenshot at 1280, 768, 375.

If reaching this state in demo content is not feasible (depends on seeded progress), it is acceptable to demonstrate it via the not-yet-complete state in Test 1 — both branches share the same icon-side rule and using `icon_right="next"`.

## Test 4 — Finish page

**Goal:** verify the "View Course" CTA is gone and only "Return to Home" remains.

1. Navigate to `http://127.0.0.1:$PORT/courses/standard-markdown-demo-finance/finish/` as a learner who has completed the course. (If the learner has not finished, complete the final topic first or pick a learner who has.)
2. Confirm visually:
   - The "Congratulations!" heading and course summary card render as before.
   - **Only one button** appears in the navigation row: **Return to Home**, with the `home` icon on its left.
   - The "View Course" button is **not present**.
   - The single button is horizontally centred in its row (the `flex justify-center` wrapper still works with one child).
3. Screenshot at 1280, 768, 375.

## Pass criteria

- All four tests render as described above.
- No layout breakage (no overflow, no orphaned spacing, no overlapping elements) at any of the three widths.
- Screenshots saved for each (page × viewport) combination so reviewers can compare.
- No console errors in the browser devtools (`mcp__plugin_fls_playwright__browser_console_messages`).
- No 4xx / 5xx requests in the network log (`mcp__plugin_fls_playwright__browser_network_requests`).

## Skills / MCPs

- **fls:use-playwright** — primary driver for browser interaction.
- **fls:icon-usage** — reference for the expected icon names (`next`, `previous`, `check`, `home`).
- **brand-guidelines** — sanity-check the visual outcome against the brand's button / spacing conventions.
- **mcp__plugin_fls_playwright__** browser tools — `browser_navigate`, `browser_resize`, `browser_take_screenshot`, `browser_snapshot`, `browser_console_messages`, `browser_network_requests`.

## If something fails

Hand the failure (with screenshot) to the implementer. Per project policy, bug fixes follow TDD: write a failing test first (or, for purely presentational regressions, document the visual reproduction), then fix.
