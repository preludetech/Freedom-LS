# Frontend QA: Course Access Types — Free & Application-Gated

Manual / Playwright-driven browser checks for this feature. Run after implementation.

## Setup

Start the dev server on a randomly-chosen free port (do **not** use the default runserver port):

```bash
PORT=$(.claude/fls/scripts/find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL: `http://127.0.0.1:$PORT`

All QA uses the **DemoDev** site. Required test data (create via the `fls:qa-data-helper` agent
if absent — never hand-build site-aware rows):

- A **free** course (no `access_config`, or `access_config: {access_type: free}`) on DemoDev.
- An **application-gated** course (`access_config: {access_type: application_gated}`) on DemoDev.
- A **learner** account (knows email + password) enrolled in nothing yet.

(No reviewer account is needed — review is deferred to the `application-review-ui` spec.)

Log in at `http://127.0.0.1:$PORT/accounts/login/` (email + password).

---

> **Note — `student_interface` was refactored after this doc was drafted.** The funnel surface is
> the **course detail** page (`/courses/<slug>/detail/`), **not** a "course preview" page or modal.
> Not-registered course **cards/rows are plain links to the detail page** — they carry **no** CTA
> button; the "Start"/"Apply now" affordance appears once, on the detail page. The bare course URL
> `/courses/<slug>/` (`course_home`) is now a **redirector**: an unregistered/unentitled learner is
> 302'd to the detail page, an entitled learner to their resume item. Item URLs are
> `/courses/<slug>/<n>/` (no `items/` segment).

## Test 1 — Free course is unchanged (regression)

1. Log in as the learner. Go to the dashboard (`/`) and the all-courses list (`/courses/`).
2. Find the **free** course card/row.
   - **Expect:** it is a plain link to the course **detail** page (no CTA button on the card
     itself).
3. Open the free course **detail** page (`/courses/<free-slug>/detail/`).
   - **Expect:** the CTA reads **"Start"** (an enabled link/button). NB: this is the access-backend
     label per the copy decision flagged in `2. plan.md` Task A.8 ("Start" vs the pre-feature
     "Enrol & start") — confirm it matches whatever was shipped.
4. Click the CTA.
   - **Expect:** you are registered and land on the first item — today's behaviour, no application
     flow.

## Test 1b — Free-course item content requires registration (deliberate hardening)

This verifies the content-access seam: with the default backend `can_access_content == is_registered`,
so item content gates on the backend, not just the TOC links. Use a learner **not yet registered**
for the free course (e.g. a fresh learner, before doing Test 1's Start click).

1. As the unregistered learner, open the bare free course URL (`/courses/<free-slug>/`).
   - **Expect:** you are **redirected to the detail page** (`/courses/<free-slug>/detail/`) —
     `course_home` no longer renders for an unregistered learner. The detail page (the funnel
     surface) renders the title/description, but every TOC item shows as **BLOCKED** (no link).
2. Attempt to hit a free-course **item URL directly** (e.g. `/courses/<free-slug>/1/`).
   - **Expect:** you are **redirected to the detail page** and do **not** see item content. This
     is the intentional change — previously the guessed item URL would render content even when
     unregistered (the view had no registration gate). After registering (Test 1), the same URL
     renders normally.

## Test 2 — Gated course shows "Apply now"

1. As the learner, find the **application-gated** course card/row on the dashboard and on the
   all-courses list.
   - **Expect:** the card/row is a plain link to the detail page (no CTA button on the card).
2. Open the gated course **detail** page (`/courses/<gated-slug>/detail/`).
   - **Expect:** the CTA reads **"Apply now"** (not "Start").
3. Confirm the gated course still shows its title/description (the detail page is the funnel surface
   and stays visible).

## Test 3 — Self-registration chokepoint is enforced server-side

1. As the learner, attempt to hit the registration URL for the gated course **directly**
   (`/courses/<gated-slug>/register/`).
   - **Expect:** you are **not** registered; you are redirected to the apply confirmation page
     (`/applications/apply/<gated-slug>/`) — the gate refuses self-registration.
2. Attempt to hit the gated course **content/home** URL directly
   (`/courses/<gated-slug>/` or a first-item URL `/courses/<gated-slug>/1/`).
   - **Expect:** course **content** is not shown; you are redirected to the detail page showing the
     "Apply now" affordance instead.

## Test 4 — Applying to a gated course

1. As the learner, click **"Apply now"** on the gated course.
   - **Expect:** an "Apply to <course>?" confirmation page (no form fields to fill in).
2. Confirm/submit.
   - **Expect:** redirect to the **applicant status page** showing a plain-language confirmation
     that the application has been received and is pending review.
3. Click **"Apply now"** for the same gated course a second time.
   - **Expect:** you are taken to the existing application's status page — **no duplicate**
     application is created.
4. Go to the learner **dashboard**.
   - **Expect:** the in-flight application is listed/linked and the link opens its status page.

> **Review is out of scope for this spec.** Reviewer actions in the admin, the applicant seeing a
> reviewer's message, withdraw, and the approval + enrol hand-off are QA'd by the
> `application-review-ui` spec (its idea.md carries that QA seed). Nothing here exercises a reviewer.

## Test 8 — Listings unchanged under the shipped backend

1. As the learner, compare the dashboard and all-courses lists before/after: the shipped
   applications backend inherits `filter_visible()` from the core default, which returns courses
   unchanged.
   - **Expect:** the same set of courses is listed as without the feature (no courses hidden).
