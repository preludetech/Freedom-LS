# Frontend QA: Phase 1 ŌĆö Theme resolver + default-theme tokens

## What we're verifying

1. The default theme renders the FLS dev server **identically to before** the migration. No visual regression on any major page.
2. Bad `FLS_THEME` values fail loudly at startup.
3. Email colour parser returns sensible values from the new default `theme.css`.
4. The default theme's static `theme.css` is resolvable via the static-files URL.

This phase ships **no second theme**, no Tier-2 component changes, and no Tier-3 template overrides. Those are validated in later phases.

We use the **DemoDev** site for all test data ŌĆö it has the demo content fixtures already loaded.

PORT=$(.claude/fls/scripts/find_available_port.sh)

## Test 1 ŌĆö Default theme: smoke + visual regression

### 1.1 Build and start

```bash
npm run tailwind_build
uv run python manage.py runserver $PORT
```

Expected: server starts cleanly with no `ImproperlyConfigured` traceback.

### 1.2 Login

- Navigate to `http://127.0.0.1:$PORT/`.
- Use the DemoDev student credentials documented in the project's QA fixture (the `qa-data-helper` agent will surface them if not in scratch memory).

Expected: login lands on the student home / dashboard.

### 1.3 Walk the major pages and check each renders as before

For each URL below, take a screenshot via Playwright MCP (`browser_take_screenshot`) and compare visually against the pre-migration look. The intent is "no visible change":

- `/` ŌĆö student home / course list.
- `/courses/` (or whichever path the all-courses view exposes; navigate via the "Courses" header link).
- A course home page ŌĆö click into a course from the list.
- A course topic page ŌĆö click any topic.
- A course form page ŌĆö find a topic that is a form / activity.
- Educator interface (if your test user has educator role) ŌĆö click the educator link in the user menu.
- Account pages ŌĆö click the user-menu avatar, then "Account" / "Settings".
- Login (sign out, then revisit `/`).

**Things to verify on each page:**

- Header bar: brand colour matches the previous default (`#2B6CB0` blue), header button labels render in the existing system font.
- Buttons (primary, outline, success, error): shape (radius), colour, hover behaviour. `btn-error` is the new name for the old `btn-danger`; functionality must be unchanged.
- Chips on dashboards / progress panels: pill shape, primary / success / warning / error colours.
- Surfaces / cards: white-ish background, subtle border, rounded corners.
- Form pages: inputs styled with the default border + focus ring; validation states (deliberately submit a form with a missing required field) show an error border in the new `error` red ŌĆö same hue as the old `danger`.
- Messages partial (e.g. after login) renders with correct colour.

Pay special attention to the templates whose tokens were renamed (anything formerly using `text-foreground`, `text-danger`, `text-on-primary` paired with success/error). They should look unchanged.

## Test 2 ŌĆö Bad `FLS_THEME` fails loudly

```bash
FLS_THEME=does_not_exist uv run python manage.py runserver $PORT
```

Expected: process exits with `ImproperlyConfigured` and a message that includes the slug (`does_not_exist`) and the searched dirs (both `BASE_DIR / "themes"` and `FREEDOM_LS_PACKAGE_DIR / "themes"`). Server does **not** start.

Also verify the same failure on `manage.py check`:

```bash
FLS_THEME=does_not_exist uv run python manage.py check
```

Expected: same `ImproperlyConfigured` traceback.

## Test 3 ŌĆö Email colour fallbacks

(Sanity check that the email-template colour parser still returns sensible values from the new theme.css.)

```bash
FLS_THEME=default uv run python manage.py shell -c "from django.conf import settings; print(settings.EMAIL_COLOR_PRIMARY, settings.EMAIL_COLOR_FOREGROUND)"
```

Expected: `#2B6CB0 #1A2332`.

If `EMAIL_COLOR_FOREGROUND` returns the hard-coded fallback `#1A2332` because the lookup key was wrong, the test still appears to pass ŌĆö so confirm by also checking a *new* role like `info`:

```bash
FLS_THEME=default uv run python manage.py shell -c "from django.conf import settings; print(settings.EMAIL_COLOR_PRIMARY)"
```

The intent: prove the parser is reading the real `theme.css` not just falling back to defaults. If you're unsure, temporarily edit the theme `--color-primary` value, restart, and confirm `EMAIL_COLOR_PRIMARY` follows the change.

## Test 4 ŌĆö Default theme static asset is resolvable

With the default theme active, request:

- `http://127.0.0.1:$PORT/static/themes/default/theme.css`

Expected: 200 OK with the theme.css contents (CSS, includes the `@theme` block with `--color-primary: #2B6CB0`).

The namespacing under `static/themes/<slug>/` must hold so future themes (Phase 2) don't collide.
