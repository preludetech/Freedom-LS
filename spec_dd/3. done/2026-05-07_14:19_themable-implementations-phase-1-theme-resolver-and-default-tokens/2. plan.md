# Implementation Plan: Theme resolver + finalised default-theme tokens (Phase 1 of 5)

This plan turns `1. spec.md` (in this folder) into shippable code. The phase delivers a complete vertical slice: theme resolver, default theme on disk, rewritten Tailwind input, settings wiring, and a one-pass token-class rename across templates. At the end of the phase the site looks visually identical to before.

For the holistic five-phase context, see `../themable-implementations-master-decomposed-into-phases/2. plan.md`.

---

## Architectural principles (driven from the spec)

1. **Themes are sparse.** A theme contains only what it overrides. The minimal theme is one `theme.css` file.
2. **Defaults stay in the main project.** `tailwind.components.css` (component classes) stays at the repo root. Generic primitives (`base/templates/cotton/*`, `base/templates/partials/*`, `_base.html`) stay where they are. Nothing is migrated wholesale into the default theme.
3. **This phase ends in a green build.** No half-implemented features.

These principles drive every task below.

---

## Pre-existing functionality to reuse (DRY)

- `freedom_ls/accounts/email_utils.py::parse_tailwind_colors` — already parses `--color-*` from a CSS file. The path it is called with changes (resolved theme's `theme.css`); the function itself does not.
- `STATICFILES_DIRS` and `TEMPLATES[0]["DIRS"]` — themes are surfaced via simple prepend. No new static finder, no new template loader.
- The existing role-token usages already in templates (`text-primary`, `bg-surface-2`, `border-border`, etc.) — only the *renamed* tokens need updating.

## Task 1 — Theme resolution module (`freedom_ls/base/theming.py`)

Build the resolver that the settings helper calls. Pure-Python, zero Django imports beyond `ImproperlyConfigured`, so it can be unit-tested without the test client.

**Public surface:**

```python
# freedom_ls/base/theming.py
from pathlib import Path
from django.core.exceptions import ImproperlyConfigured

FREEDOM_LS_PACKAGE_DIR: Path = Path(__file__).resolve().parent.parent  # freedom_ls/

def resolve_theme_dir(theme_slug: str, themes_dirs: list[Path]) -> Path:
    """Walk themes_dirs in order; return first <dir>/<theme_slug> that exists.

    Raises ImproperlyConfigured if nothing matches.
    """
    for parent in themes_dirs:
        candidate = Path(parent) / theme_slug
        if candidate.is_dir():
            return candidate
    raise ImproperlyConfigured(
        f"FLS theme {theme_slug!r} not found in any of: {themes_dirs!r}"
    )

def configure_theme(
    *,
    theme_slug: str,
    themes_dirs: list[Path],
    templates: list[dict],
    staticfiles_dirs: list,
) -> Path:
    """Resolve theme, prepend its templates/ dir to TEMPLATES[0]['DIRS']
    and its static/ dir to STATICFILES_DIRS. Also surface every inactive
    shipped theme's static/ dir (after the active one) so
    `/static/themes/<slug>/theme.css` resolves regardless of which theme is
    active. Returns the resolved theme dir.
    """
    resolved = resolve_theme_dir(theme_slug, themes_dirs)
    templates_dir = resolved / "templates"
    if templates_dir.is_dir():
        templates[0].setdefault("DIRS", [])
        templates[0]["DIRS"].insert(0, str(templates_dir))

    # Inactive themes' static dirs first (deduped by slug, respecting the
    # `themes_dirs` precedence used by `resolve_theme_dir`), then the active
    # theme's static dir on top.
    inactive: list[Path] = []
    seen = {theme_slug}
    for parent in themes_dirs:
        if not Path(parent).is_dir():
            continue
        for theme_dir in sorted(Path(parent).iterdir()):
            if not theme_dir.is_dir() or theme_dir.name in seen:
                continue
            static_dir = theme_dir / "static"
            if static_dir.is_dir():
                seen.add(theme_dir.name)
                inactive.append(static_dir)
    for static_dir in reversed(inactive):
        staticfiles_dirs.insert(0, static_dir)
    active_static = resolved / "static"
    if active_static.is_dir():
        staticfiles_dirs.insert(0, active_static)
    return resolved
```

**Tests** (TDD; `freedom_ls/base/tests/test_theming.py`):

- `resolve_theme_dir` returns the first directory that exists when walking the dirs list.
- Downstream dir takes precedence over the FLS package dir (covers the "downstream theme named `default` shadows FLS one" case).
- Raises `ImproperlyConfigured` when no candidate exists — message includes the slug and the dirs searched.
- `configure_theme` prepends to both `TEMPLATES[0]['DIRS']` and `STATICFILES_DIRS` (use `tmp_path` to fake themes on disk).
- `configure_theme` is a no-op for the missing static / templates subdirs (a Tier-1-only theme has no `templates/` and is still valid).
- `configure_theme` surfaces every inactive theme's `static/` dir (after the active one) so the inactive themes' `/static/themes/<slug>/theme.css` is still resolvable.
- Inactive theme without a `static/` subdir is silently skipped.
- Inactive theme whose slug is shadowed by an earlier `themes_dirs` entry is not added twice.
- Walking past a missing parent dir gracefully falls through to the next entry (covers `BASE_DIR / "themes"` not existing in the FLS dev repo).

Use `pytest` + `tmp_path` fixtures. Follow conventions in `fls:testing` (`pytestmark = pytest.mark.django_db` is **not** needed — pure filesystem logic).

## Task 2 — Create the default theme directory

Create the minimal default theme:

```
freedom_ls/themes/default/
    static/
        themes/default/
            theme.css
    theme.md
```

`theme.css` contains the finalised `@theme` block per spec "Default-theme values":

- All role tokens: `primary`, `secondary`, `accent`, `success`, `warning`, `error`, `info`, plus `on-*` pairs, `surface`, `surface-2`, `on-surface`, `border`, `muted`, `focus-ring`. Default values per spec table.
- Two FLS-namespace mix knobs (`--fls-hover-mix-color: white;` / `--fls-hover-mix-amount: 12%;`) plus seven `--color-*-hover` tokens (`primary`, `secondary`, `accent`, `success`, `warning`, `error`, `info`) auto-derived via `color-mix(in oklch, var(--color-<role>), var(--fls-hover-mix-color) var(--fls-hover-mix-amount))`. Sparse downstream themes get coherent hover states for free; explicit overrides remain possible.
- Shape tokens: `--fls-radius-sm/-md/-lg/-pill`, mapped via `@theme inline` to `--radius-*`.
- Type tokens: `--fls-font-sans`, `--fls-font-display`, mapped via `@theme inline` to `--font-sans` and `--font-display`.
- `--color-focus-ring: var(--color-primary);` (live alias, declared inside the `@theme inline` block).
- No `@layer base` and no `@layer components` blocks — those stay in `tailwind.components.css`.

`theme.md`: short manifest (name, version, what it overrides). One paragraph.

## Task 3 — Strip `@theme` from `tailwind.components.css`; rename tokens it references

Edit `tailwind.components.css`:

- Remove the entire `@theme { … }` block at the top.
- Update `@layer base` rules: `text-foreground` → `text-on-surface` (in `body`, `h1`–`h4`, `p`, `td`).
- Apply `font-display` to the `h1`–`h4` base-layer rules so themes that redeclare `--fls-font-display` actually drive the heading typeface (no-op under default; required by Phase 2 for `first_class` Outfit headings).
- Update `@layer components`:
  - `.btn-primary`: drop `bg-primary-bold`; hover uses the new `hover:bg-primary-hover` token (auto-derived in `default/theme.css` via `color-mix`).
  - `.btn-success`: use `text-on-success` (not `text-on-primary`); hover → `hover:bg-success-hover`.
  - `.btn-danger` → `.btn-error`: use `text-on-error`; hover → `hover:bg-error-hover`.
  - `.btn-outline` unchanged in semantics.
  - `.chip-warning`: `text-foreground` → `text-on-warning`.
  - `.chip-success`: `text-success-bold` → `text-success`.
  - `.chip-danger` → `.chip-error`: `text-danger-bold` → `text-error`.

## Pseudocode summary of new logic

```
# freedom_ls/base/theming.py

resolve_theme_dir(slug, dirs):
    for d in dirs:
        candidate = d / slug
        if candidate.is_dir(): return candidate
    raise ImproperlyConfigured(...)

configure_theme(slug, dirs, templates, staticfiles_dirs):
    resolved = resolve_theme_dir(slug, dirs)
    if (resolved / "templates").is_dir():
        templates[0]["DIRS"].insert(0, str(resolved / "templates"))
    # Inactive theme statics first (so the active one ends up on top).
    for parent in dirs:
        for theme_dir in parent.iterdir():
            if theme_dir.name == slug or already_seen(theme_dir.name): continue
            if (theme_dir / "static").is_dir():
                staticfiles_dirs.insert(0, theme_dir / "static")
    if (resolved / "static").is_dir():
        staticfiles_dirs.insert(0, resolved / "static")
    return resolved

# config/settings_base.py (additions)
FLS_THEME = os.environ.get("FLS_THEME", "default")
FLS_THEMES_DIRS = [BASE_DIR / "themes", FREEDOM_LS_PACKAGE_DIR / "themes"]
RESOLVED_THEME_DIR = configure_theme(
    theme_slug=FLS_THEME,
    themes_dirs=FLS_THEMES_DIRS,
    templates=TEMPLATES,
    staticfiles_dirs=STATICFILES_DIRS,
)
_theme_css = RESOLVED_THEME_DIR / "static" / "themes" / FLS_THEME / "theme.css"
_tw_colors = parse_tailwind_colors(str(_theme_css))
```

Everything else is mechanical CSS edits, class renames driven by grep, and a small set of new theme-directory files.
