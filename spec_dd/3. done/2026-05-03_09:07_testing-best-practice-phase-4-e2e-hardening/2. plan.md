# Implementation Plan — Phase 4: E2E hardening

This plan operationalises `1. spec.md`. The work is purely test-side hardening of the existing Playwright suite under `freedom_ls/**/tests/e2e/` plus extracting shared fixtures into a reusable module. There is one production-code carve-out: extracting the `coursePart_<slug>_<idx>` localStorage key into a single named constant.

We follow TDD where it makes sense (the new fixtures get exercised by the rewritten tests; new high-value E2E tests get written failing-first if we add any). Audit-driven rewrites of existing tests are not "TDD" in the textbook sense — but each rewritten test must still pass before and after, and any audit-surfaced bug becomes a *failing-test-first* fix in a separate spec (file-and-skip rule from the spec).

---

## Pre-flight: nail down the scope

The spec's audit scope is `freedom_ls/**/tests/e2e/`. Today that resolves to a single file:

- `freedom_ls/student_interface/tests/e2e/test_course_toc.py` — 2 tests, both `@pytest.mark.playwright`.

Other Playwright-marked tests live in `freedom_ls/student_interface/tests/form_ui_tests.py` (7 tests, named with the `*_tests.py` pattern, picked up by pytest's `python_files` glob in `pyproject.toml`). The spec's wording "every file under `freedom_ls/**/tests/e2e/`" excludes that file. We respect the spec's explicit scope: `form_ui_tests.py` is **out of scope** for the rewrite. Note this in the PR description and leave it as a follow-up so a future spec can consider relocating + cleaning it.

The shared-fixtures module, however, must work for *all* Playwright tests in the repo (the new login fixture replaces / wraps `logged_in_page` in `freedom_ls/conftest.py`, which `form_ui_tests.py` already consumes). Fixture compatibility is a hard requirement; rewriting `form_ui_tests.py` to *use* the new patterns is not.

## Step-by-step work

### Step 1 — Land the session-scoped `live_server` override

`pytest-django`'s `live_server` is function-scoped; the storage-state login fixture is session-scoped, so we hit `ScopeMismatch` at collection. Override once at the top-level `conftest.py` (or in `freedom_ls/conftest.py`, whichever is loaded first):

```python
# pseudocode
@pytest.fixture(scope="session")
def live_server(request, django_db_setup, django_db_blocker):
    from pytest_django.live_server_helper import LiveServer
    with django_db_blocker.unblock():
        server = LiveServer("localhost")
        try:
            yield server
        finally:
            server.stop()
```

Verify nothing else depends on per-test `live_server` reset — the existing `live_server_site` fixture mutates the `Site` domain, and that mutation must persist for the session under the new scoping. Bump `live_server_site` to session scope too (it composes with session-scoped `live_server` and session-scoped `site`).

Document the new fixture-graph contract in the `playwright_fixtures.py` module docstring: which fixtures are session-scoped, which are function-scoped, why.

### Step 2 — Build `playwright_fixtures.py`

New module: `freedom_ls/tests/playwright_fixtures.py`. Contents (pseudocode):

```python
"""Shared Playwright fixtures.

Available fixtures:
- ``reset_local_storage`` (autouse, function): clears localStorage and
  sessionStorage before each Playwright test that uses a `page`.
- ``authed_storage_state`` (session): logs a synthetic student in via the
  UI once per session and returns the resulting storage_state dict.
- ``logged_in_page`` (function): yields a `page` whose context was built
  from ``authed_storage_state`` — no UI login per test.
- ``fresh_login_page`` (function): yields a `page` with no auth state, for
  tests that genuinely need to drive the login or signup UI themselves.

Note on scoping: ``live_server`` is widened to session scope in the
top-level conftest so the session login can hit it. ``live_server_site``
follows.
"""

@pytest.fixture(autouse=True)
def reset_local_storage(page):
    # `page` is function-scoped from pytest-playwright; clear before the
    # test runs. Use add_init_script so the clear runs on every navigation
    # in the test, not just before the first goto.
    page.add_init_script("window.localStorage.clear(); window.sessionStorage.clear();")
    yield


@pytest.fixture(scope="session")
def authed_storage_state(live_server, browser, django_db_setup, django_db_blocker, ...):
    # Create a synthetic verified user, drive the login form once via
    # semantic locators, return context.storage_state().
    ...


@pytest.fixture
def logged_in_page(browser, authed_storage_state, live_server_site, mock_site_context):
    context = browser.new_context(storage_state=authed_storage_state)
    try:
        yield context.new_page()
    finally:
        context.close()


@pytest.fixture
def fresh_login_page(browser, live_server_site, mock_site_context):
    # No storage_state — caller drives login / signup themselves.
    context = browser.new_context()
    try:
        yield context.new_page()
    finally:
        context.close()
```

Implementation notes:

- `authed_storage_state` must not commit fixture credentials to the repo. It builds them at runtime from `UserFactory` and the synthetic email/password the existing `logged_in_page` already uses. The skill's security caveat applies: passwords stay in the fixture body, not in committed JSON.
- The login-flow code inside `authed_storage_state` switches from `page.fill('input[name="login"]', ...)` to `page.get_by_label("Email").fill(...)` (and similar for the password field and the submit button). Verify the actual labels on the rendered allauth login form before writing the locator — the form is rendered server-side by allauth + project templates.
- `reset_local_storage` is autouse; tests opt out by declaring `@pytest.mark.usefixtures` overrides or simply not using `page`. We don't need a global opt-out mechanism — the spec's success criterion is "remove inline localStorage.clear() calls", which is satisfied by autousing the fixture.
- For roles beyond student: the spec says "session-scoped `storage_state` for the common roles (student, educator, admin)", but the only existing Playwright tests use a student. Provide the **student** session fixture now; leave a comment + named-but-skipped `authed_storage_state_educator` / `authed_storage_state_admin` stubs *only if the audit surfaces a role need*. Otherwise YAGNI — adding speculative role fixtures with no consumer would clutter the module. The spec's "fixtures that appear in 2+ tests" gate keeps us honest.

### Step 3 — Rewrite `freedom_ls/conftest.py::logged_in_page`

The existing `logged_in_page` (lines 116–140 of `freedom_ls/conftest.py`) is consumed by both `test_course_toc.py` (in scope) and `form_ui_tests.py` (out of scope). To avoid breaking out-of-scope tests:

- **Preferred:** delete the old fixture body and have the *new* `logged_in_page` from `playwright_fixtures.py` take over the name. Both consumer files keep working unchanged. The CSS-selector login flow is gone.
- **Fallback:** if the storage-state-based fixture causes regressions in `form_ui_tests.py` (e.g. that suite mutates `mock_site_context` per test in a way that breaks under session-scoped login), restore a function-scoped UI-login `logged_in_page` *with* the locators upgraded to `get_by_label` / `get_by_role` per the skill, and offer the storage-state variant under a new name (`student_storage_page` or similar) for in-scope tests. Decide based on what the test run shows.

Either way: the inline `page.fill('input[name="login"]', ...)` and `page.click('button[type="submit"]')` calls in the current fixture get replaced with semantic locators.

### Step 4 — Extract the localStorage key constant

Two coordinated edits:

1. Create `freedom_ls/student_interface/templatetags/course_storage_keys.py` (or extend an existing tag library in the same app — check `freedom_ls/student_interface/templatetags/` first; reuse over create). Define a Python helper:

   ```python
   # pseudocode
   COURSE_PART_STORAGE_KEY_PREFIX = "coursePart"

   def course_part_storage_key(course_slug: str, index: int) -> str:
       return f"{COURSE_PART_STORAGE_KEY_PREFIX}_{course_slug}_{index}"
   ```

   Expose it as a template tag that renders the same string. Type-hint the function (project rule: "Type hints required on all functions").

2. Edit `freedom_ls/student_interface/templates/student_interface/partials/course_minimal_toc.html` line 109: replace the inline format string with `{% course_part_storage_key course.slug forloop.counter %}` (or equivalent).

3. In `test_course_toc.py`, import the helper and call it for the expected key — no more `f"coursePart_{course.slug}_2"` in the test.

If step 1 turns out non-trivial (e.g. the existing template tag library has incompatible registration patterns), file-and-skip: comment-tag the test with `# TODO(testing-phase-4): replace hardcoded storage key — see spec` and move on. Spec explicitly permits this carve-out failing.

### Step 5 — Rewrite `test_course_toc.py`

Per-test edits:

- Drop both `logged_in_page.evaluate("localStorage.clear()")` calls. The autouse `reset_local_storage` fixture handles it.
- Drop both `logged_in_page.wait_for_load_state("networkidle")` calls. They follow `page.goto(...)`, which already waits for `load`; the subsequent `expect(toggle_button).to_be_visible()` provides the auto-wait we actually want. `networkidle` is the footgun the spec calls out.
- Replace `f"coursePart_{course.slug}_2"` with `course_part_storage_key(course.slug, 2)` (importing the helper from step 4).
- Locators are already `get_by_role` / `get_by_text`; nothing to change there.
- The double-`goto` pattern (`goto → clear → goto`) becomes a single `goto`. Tighter, faster, fewer race surfaces.

Expected diff is small: ~10 lines of reduction across both tests, plus one import.

### Step 6 — Audit: what's missing?

Walk the existing E2E suite in scope (the single file) against the user-journeys map for the student interface. Concretely list 0–3 high-value gaps. Candidates worth checking:

- Signup → email-verify → first-lesson view. Currently no E2E coverage.
- Student completes a topic and the progress indicator updates without a hard refresh (HTMX swap territory).
- Login → logout → re-login (the current `logged_in_page` ends at "logged in"; logout is uncovered).

For each candidate, decide:

- *Already covered well-enough by an integration test in pytest?* → skip; not E2E-shaped.
- *Browser-required behaviour missing?* → write the test. Use `fresh_login_page` for signup/login flows; use `logged_in_page` (storage-state) for everything else.
- *Audit found nothing worth adding?* → fine. Spec explicitly says "If the audit produces no obvious gaps, *don't invent any*". Note the conclusion in the PR description.

Do not write 0–3 tests speculatively. Write only what the audit justifies. Use TDD on whatever you do add: red (test fails because the assertion is correct but the user journey hasn't been exercised end-to-end before), green (it passes because the production behaviour already exists), no refactor needed since we're not touching production code in this step.

### Step 7 — Run the suite end-to-end

- `uv run pytest -m playwright` must pass (success criterion 6).
- Force a sanity-check failure (e.g. inject a deliberately-wrong `expect(...)` in one test, run, confirm `test-results/<nodeid>/trace.zip` lands, revert). This validates Phase 2's trace config end-to-end as the spec asks.
- `uv run pytest` (full suite) must still pass — fixture-graph changes can break unit tests if `live_server` scope shifts surface a hidden dependency. If it does, fix the dependency, don't paper over by reverting the scope change.

### Step 8 — Document

Top of `playwright_fixtures.py` gets the docstring described in Step 2. No further documentation changes — the skill itself stays unedited (success criterion 7: no skill edits).
