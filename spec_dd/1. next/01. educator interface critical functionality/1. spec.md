# Educator Interface Critical Functionality

## Overview

The educator interface currently supports basic cohort CRUD and read-only views for courses and users. This specification extends it with full student management, bulk operations, cohort-course registration management, educator role assignment, and consistent permission enforcement.

## Why this matters

Educators need to manage their students and cohorts without relying on the Django admin. The admin is too technical, exposes too much, and doesn't enforce the role-based permission model. A purpose-built educator interface lets each role see exactly what they need — no more, no less.

---

## 1. Cohort Management Enhancements

### Current state

Cohort create, list, detail, edit name, and delete with confirmation are already implemented.

### What to add

#### 1.1 Search and filter on the list view

- Text search across cohort name
- Filter by active/inactive status (active by default)

#### 1.2 Key info at a glance

The cohort list view should show:
- Student count (number of active cohort members)
- Linked courses (names of courses registered via CohortCourseRegistration)
- Active/inactive status

#### 1.3 Deactivation and deletion

- Add an `is_active` field to the Cohort model (BooleanField, default=True)
- **Deactivation** is the default action. Deactivating a cohort does not affect student accounts or their individual course registrations — it just marks the cohort as inactive
- Deactivated cohorts are filtered out of list views by default but can be shown via a filter toggle
- Visual indicator (e.g., muted styling, "Inactive" badge) for deactivated cohorts when shown
- **Hard delete** is available as a separate, clearly distinct action for GDPR compliance. The delete confirmation must show what will be affected: number of memberships, course registrations, and deadline records that will be removed

#### 1.4 Managing students within a cohort

From the cohort detail view (students tab):
- Search and add existing students to the cohort
- Create new student accounts inline and add them to the cohort
- Remove students from the cohort (removes CohortMembership only — progress data is unaffected, but the student loses access to courses they were registered for via this cohort unless they have a direct UserCourseRegistration)
- Clearly communicate the access implications when removing a student

#### 1.5 Managing course registrations within a cohort

From the cohort detail view (courses tab):
- Register the cohort for a course (pick from available courses, confirm)
- Remove a course registration (deactivate CohortCourseRegistration)
- Show which courses are active vs inactive

**Why manage from the cohort side only:** The course detail view shows which cohorts are registered (read-only) with links back to the cohort detail view for management. This avoids duplicating management UI and keeps the source of truth in one place.

## 2. Student Management

### Current state

Users appear in the educator interface list view but with limited functionality.

### What to add

#### 2.1 Create student accounts

- Add individual students from the educator interface: email (required), first name, last name
- If a user with that email already exists on the site, surface this clearly and offer to link them rather than creating a duplicate
- New accounts are created with `is_active=True` and no special roles (they are students by default)

#### 2.2 Student list view

- Text search across email, first name, last name
- Filter by: active/inactive status (active by default), cohort membership
- Show at a glance: cohort memberships, course registrations (both direct and via cohort), active status

#### 2.3 Student detail view

- View and edit: first name, last name, email, active status
- View cohort memberships with ability to add to / remove from cohorts
- View course registrations (read-only — registrations are managed via cohorts or direct registration)

#### 2.4 Deactivation and deletion

- **Deactivation** sets `is_active=False` on the User model, preventing login. This is the default action. The UI must clearly label this as "Deactivate" (not "Delete")
- Deactivated students are filtered out of list views by default but can be shown via a filter toggle
- Visual indicator for deactivated students when shown
- **Reactivation** is available from the student detail view when viewing a deactivated student
- **Hard delete** is available as a separate, clearly distinct action for GDPR compliance. The delete confirmation must show what will be removed: cohort memberships, course registrations, progress data, deadline records

## 3. Bulk Operations

### 3.1 CSV import for students

Accessible from two places:
- **Cohort detail view**: Import students into a specific cohort. CSV columns: email (required), first_name, last_name. The cohort is implicit from context
- **Student list view**: Import students with optional cohort assignment. CSV columns: email (required), first_name, last_name, cohort (optional, matched by name)

**Import flow:**
1. Download a CSV template (with headers and example rows)
2. Upload a CSV file
3. Preview/confirmation step showing:
   - New accounts that will be created
   - Existing users that will be matched (by email)
   - Validation errors (missing email, unrecognized cohort names, malformed rows)
   - Row count summary
4. Confirm to commit the import
5. Show results: created count, matched count, error count with details

**Why a preview step:** Bulk operations are hard to undo. Showing what will happen before committing prevents mistakes and builds trust.

#### Matching logic

- Match existing users by email (case-insensitive)
- If a matched user is already in the target cohort, skip silently (idempotent)
- If a matched user is deactivated, flag this in the preview but still allow adding them

### 3.2 Multi-select with bulk actions

Checkboxes on the student list view to select multiple students, then apply bulk actions:
- Add to cohort (pick cohort from dropdown)
- Remove from cohort (pick cohort from dropdown)

Checkboxes on the cohort list view to select multiple cohorts, then apply bulk actions:
- Register for course (pick course from dropdown)
- Deactivate selected cohorts

**Why limited bulk actions:** Start with the most common operations. More can be added later without changing the UI pattern.

## 4. Educator Role Assignment

### Overview

Site admins need to be able to create educator accounts and assign them roles scoped to cohorts. This is not an invitation flow — it is direct account creation and role assignment.

### Flow

1. From the educator interface, navigate to a role management area
2. Enter an email address
3. If the user doesn't exist, create their account (email, first name, last name)
4. Select a role: Site Admin, Instructor, or TA
5. For Instructor and TA roles: select which cohort(s) to scope the role to
6. Confirm the assignment

**Why no invite flow:** Keep it lightweight. The educator creates the account directly. The new user can set their password via the standard password reset flow.

### Role display

- Show role descriptions in plain language so non-technical users understand what each role can do:
  - **Site Admin**: Full access to manage all cohorts, students, courses, and other educators on this site
  - **Instructor**: Can view and manage students in assigned cohorts
  - **TA**: Can view students and progress in assigned cohorts (read-only)
- On the user detail view, show current role assignments with the ability to add/remove roles
- Confirmation step before privilege changes (especially escalation or removal)

### Scoping

- Site Admin is site-scoped (applies to the entire site via SiteRoleAssignment)
- Instructor and TA are object-scoped to cohorts (via ObjectRoleAssignment)
- A user can hold multiple roles on different cohorts

**Decision:** Roles are scoped to cohorts only (not courses) for now. Course access follows from cohort access since courses are registered to cohorts.

## 5. Permission Enforcement

### Principles

These principles apply across the entire educator interface:

#### 5.1 Hide, don't disable

UI elements that are irrelevant to the user's role must be hidden entirely, not shown in a disabled state. Students should never see educator controls. Educators without delete permission should not see delete buttons.

**Why hide over disable:** A disabled button implies "you could do this, but not right now." Hiding communicates "this isn't part of your experience." Each role's view should feel like a complete, intentional interface — not a stripped-down version of the admin view.

#### 5.2 Server-side enforcement

Every action must be permission-checked on the backend. Frontend hiding is a UX convenience, not a security measure. If a user somehow submits a request they shouldn't (e.g., role changed while page was open), the server must reject it.

#### 5.3 Consistent strategy

Apply the same hide/show approach uniformly across all views. Do not mix hiding in some places and disabling in others.

#### 5.4 Meaningful errors

If a permission check fails server-side:
- Return HTTP 403 (not 404)
- Show a clear message explaining what happened, why, and what to do next (e.g., "You don't have permission to delete students. Contact your site administrator if you need this access.")
- For HTMX requests, return an appropriate error partial

#### 5.5 Filtering by access

List views must filter to show only objects the user has permission to see. Use `get_objects_for_user()` from django-guardian consistently. A TA scoped to Cohort A should only see Cohort A in their cohort list, not all cohorts with most actions disabled.

### Implementation approach

- Use the existing role-based permission system (SystemRoleAssignment, SiteRoleAssignment, ObjectRoleAssignment) with django-guardian for enforcement
- Permission checks should happen at the view level using guardian utilities
- Template-level hiding should be driven by permission context variables passed from views
- Create reusable permission-checking utilities/mixins for the panel framework to avoid repeating permission logic in every view

## 6. Course Detail View (Read-Only Enhancements)

The course detail view in the educator interface should show:
- Course information (title, description, content structure)
- Cohort registrations: which cohorts are registered for this course, with links to the cohort detail view for management
- Direct student registrations: which students are individually registered

All management actions (registering/unregistering cohorts, managing students) happen from the cohort detail view, not here.

## Success Criteria

### Cohort management
- Educators can search and filter cohorts
- Cohorts show student count, linked courses, and status at a glance
- Cohorts can be deactivated and reactivated
- Cohorts can be hard deleted with full impact disclosure
- Students can be added to and removed from cohorts with clear access implications

### Student management
- Educators can create student accounts from the educator interface
- Duplicate emails are detected and handled gracefully
- Students can be searched and filtered by name, email, cohort, and status
- Students can be deactivated/reactivated and hard deleted (GDPR)
- Student detail view shows cohort memberships and course registrations

### Bulk operations
- CSV import works from both cohort detail and student list views
- CSV template is downloadable
- Preview step shows exactly what will happen before committing
- Multi-select bulk actions work for common operations

### Educator roles
- Site admins can create educator accounts and assign roles scoped to cohorts
- Role descriptions are clear and non-technical
- Role changes require confirmation

### Permission enforcement
- UI elements are hidden (not disabled) based on the user's role
- Server-side permission checks on every action
- List views filtered to show only accessible objects
- Meaningful 403 errors with actionable messages
- Consistent approach across all views

### General
- All new functionality follows existing panel framework patterns
- All models remain site-aware (multi-tenant)
- No raw SQL or security vulnerabilities introduced
- Deactivated items filtered by default with toggle to show them
