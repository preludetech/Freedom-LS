## Problem

FLS-specific settings are read ad hoc from `django.conf.settings` scattered across
apps, with defaults and required-vs-optional status implicit. There is no single
place per app that declares "these are my settings, these are their defaults, and
these are required." The concrete consequence: a downstream that fails to set a
**required** setting (`COURSE_ACCESS_BACKEND`) gets no boot-time signal ‚Äî
`manage.py check` passes and the app 500s on the first real request (the First
Class `home_page` bug). Enforcing required settings cleanly needs a single-sourced
declaration to enforce *against*.

The sole existing instance of a per-app settings module is
`freedom_ls/student_management/config.py` (`DEADLINES_ACTIVE`) ‚Äî a pattern worth
generalising rather than a one-off.

## Goal

A codebase-wide convention: each FLS app that reads FLS-specific settings owns a
`freedom_ls/<app>/config.py` that declares its **defaults**, resolves **project
overrides**, and marks which settings are **required** ‚Äî and a required-empty
setting is surfaced by `manage.py check` (an `Error`), never a silent runtime 500.

### Success criteria

1. A new `freedom_ls/base/app_settings.py` provides `AppSettings`, `Setting`, and
   `required_settings_errors(...)`, importing only `django.conf`/`checks`/
   `exceptions` + stdlib (no models) so it is import-cycle-safe at any module top
   level.
2. `config.NAME` returns the project's Django-settings value when set, else the
   declared default; an unknown name raises `AttributeError`; a **required** unset
   setting raises `ImproperlyConfigured` **lazily on read**, never at import.
3. `config.missing_required()` enumerates missing required settings and **never
   raises**; `required_settings_errors(config, app_label)` returns
   `<app_label>.E001` Errors for each, and never raises.
4. Each app in the ownership map (see plan L0.3) reads its FLS-specific settings
   through its own `config.py`; `student_management/config.py` is migrated onto the
   shared base class; behaviour is unchanged (per-app regression guard reads the
   same value via `config` as before).
5. `COURSE_ACCESS_BACKEND` is declared `REQUIRED` in `course_access/config.py`, and
   `course_access/loader.py` reads it via `config`. When it is unset,
   `manage.py check` does **not** crash: `check_course_access_configs` returns `[]`
   early (RISK-1 guard, plan L0.4).

## Enforcement decision (D8-aligned)

A required-empty setting is reported by a **system check that returns an `Error`**
(exits `manage.py check` non-zero) ‚Äî *not* by raising at import/`ready()` time,
which would crash every command including `check` itself. As a **runtime
backstop**, the accessor raises `ImproperlyConfigured` **lazily, only when a
required-empty setting is actually read**. Any caller that must not raise (a system
check) wraps the read in `try/except ImproperlyConfigured`.

## Scope / non-goals

- **In scope:** `base/app_settings.py`, the per-app `config.py` modules and
  consumer reroutes in the ownership map, the `student_management` migration onto
  the base class, and the `course_access` loader reroute + RISK-1 guard.
- **Explicitly NOT routed through `config.py`** (stay direct `settings.X`): Django
  built-ins (`AUTH_USER_MODEL`, `MIDDLEWARE`, `INSTALLED_APPS`, `TEMPLATES`, ‚Ä¶),
  env-derived/third-party settings (`WEBHOOK_ENCRYPTION_SALT`,
  `ACCOUNT_DEFAULT_HTTP_PROTOCOL`), and theme-derived values (`RESOLVED_THEME_DIR`
  is computed by `configure_theme`, which mutates settings in place ‚Äî `config.py`
  may **read** `FLS_THEME`/`FLS_THEMES_DIRS` but must **never** call
  `configure_theme`). See plan L0.3.
- The `AUTH_USER_MODEL` pinned-value check (`freedom_ls_accounts.E003`) is **not**
  a `config.py` entry ‚Äî it lives in `test_portability_3_system_checks`.
