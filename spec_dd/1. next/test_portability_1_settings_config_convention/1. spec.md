## Problem

FLS ships as a package installed into concrete downstream projects (e.g. First
Class). "Did I wire FLS up correctly?" is the signal a downstream actually needs
from a test run, and today it **isn't a supported, dedicated thing at all**. The
canonical failure is the First Class integration of the `home_page` update:

- `student_interface` newly required `settings.COURSE_ACCESS_BACKEND`, which
  First Class had never set. FLS's loader (`get_course_access_backend()`,
  `freedom_ls/course_access/loader.py:23-39`) is `@functools.cache`d and does
  `import_string(settings.COURSE_ACCESS_BACKEND)` with **no fallback** ‚Äî an
  unset key raises only on the *first real request*, not at `manage.py check`
  time.
- First Class's root `urls.py` was missing the `applications/` include.

Both would 500 the catalogue/dashboard at runtime while `manage.py check` passed
clean. They were caught only as a side effect of FLS's own tests being vacuumed
up by the downstream's `pytest` ‚Äî the exact coupling Part 1 removed. Removing the
noise leaves this real signal absent. Part 2 supplies it.

This is not hypothetical drift: `fls-claude-plugin/resources/template_repo_manifest.md`'s
own `urls.py` checklist (lines ~210-219) still omits `applications/`,
`interest/`, and the sitemap/robots wiring that `config/urls.py:51-73` actually
ships. A downstream following that checklist reproduces the opening bug verbatim.

## Layer 0 ‚Äî settings convention (from parent plan ¬ßLayer 0)

A codebase-wide convention: each FLS app that reads FLS-specific settings owns a
`freedom_ls/<app>/config.py` declaring its defaults, resolving project overrides,
and marking which settings are **required**. This generalises the sole existing
instance (`freedom_ls/student_management/config.py`) and gives Layer 4 a
single-sourced required-settings declaration.

**Enforcement decision (matches spec D8 / "checks must never raise"):** a
required-empty setting is reported by a system check that **returns an `Error`**
(exits `manage.py check` non-zero, blocks `runserver`/`migrate`/`test`) ‚Äî *not*
by raising at import/`ready()` time, which would crash every command including
`check` itself before it can report. As a **runtime backstop**, the accessor
raises `ImproperlyConfigured` **lazily, only when a required-empty setting is
actually read** ‚Äî never at import.

## Key decisions

- **D8 ‚Äî required-setting presence/value is Layer 4 only, not a Layer-3 probe.**
  "Is the setting set / does it equal its pinned value" is a pure static
  question, and `manage.py check` is built to answer it (`base/checks.py` already
  reads `settings` this way; `E001` already does it for `COURSE_ACCESS_BACKEND`).
  So there is **no `test_required_setting_present`** in the conformance suite;
  the required-settings table is not a Layer-3 maintained surface. Each check is
  registered from the app that consumes the setting (`COURSE_ACCESS_BACKEND` ‚Üí
  `student_interface` E001/E002; `AUTH_USER_MODEL` pinned value ‚Üí `accounts`
  E003), inheriting the same install-gating for free. Layer 3 keeps only the
  *behavioural* seam ‚Äî `test_configured_backend_instantiates` executes the loader
  end-to-end, which a static check cannot. This avoids maintaining the same
  static assertion in two layers; it is deliberately not the `icons`
  "same-question-both-layers" echo, whose two sides ask materially different
  questions (see the Layer 3 vs Layer 4 division).
