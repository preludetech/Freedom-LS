## Layer 0 — per-app `config.py` settings convention (folded in during plan review)

A codebase-wide convention: each FLS app that reads FLS-specific settings owns a
`freedom_ls/<app>/config.py` declaring its defaults, resolving project overrides,
and marking which settings are **required**. This generalises the sole existing
instance (`freedom_ls/student_management/config.py`) and gives Layer 4 a
single-sourced required-settings declaration.

**Enforcement decision (matches spec D8 / "checks must never raise"):** a
required-empty setting is reported by a system check that **returns an `Error`**
(exits `manage.py check` non-zero, blocks `runserver`/`migrate`/`test`) — *not*
by raising at import/`ready()` time, which would crash every command including
`check` itself before it can report. As a **runtime backstop**, the accessor
raises `ImproperlyConfigured` **lazily, only when a required-empty setting is
actually read** — never at import.

### L0.1 — shared base `freedom_ls/base/app_settings.py` (new)

Imports **only** `django.conf.settings`, `django.core.exceptions`,
`django.core.checks` + stdlib (no models), so `from freedom_ls.<app>.config import
config` is safe at any module top-level (no import-cycle risk).

```python
from __future__ import annotations

from typing import NamedTuple

from django.conf import settings
from django.core.checks import Error
from django.core.exceptions import ImproperlyConfigured


class Setting(NamedTuple):
    """One declared setting: its fallback value and whether the project must set it."""

    default: object = None
    required: bool = False


class AppSettings:
    """Per-app settings: a project's Django settings override declared fallbacks.

    Each app subclass sets ``declared_settings`` to a ``{name: Setting(...)}`` map.
    ``config.NAME`` returns the project's value if set, else the declared default;
    a ``required`` setting the project has not supplied raises ImproperlyConfigured
    on read (lazily, never at import) as a runtime backstop. A caller that must not
    raise (e.g. a system check) wraps the read in ``try/except ImproperlyConfigured``.
    """

    declared_settings: dict[str, Setting] = {}

    def __getattr__(self, name: str) -> object:
        try:
            setting = self.declared_settings[name]
        except KeyError:
            raise AttributeError(name) from None
        value = getattr(settings, name, None)
        if isinstance(value, str):
            value = value.strip()
        if value not in (None, ""):
            return value
        if setting.required:
            raise ImproperlyConfigured(
                f"{name} is required but is not set. "
                f"Set {name} in your Django settings."
            )
        return setting.default

    def missing_required(self) -> list[str]:
        """Names of required settings the project has not supplied. Never raises."""
        missing: list[str] = []
        for name, setting in self.declared_settings.items():
            if not setting.required:
                continue
            try:
                getattr(self, name)
            except ImproperlyConfigured:
                missing.append(name)
        return missing


def required_settings_errors(config: AppSettings, app_label: str) -> list[Error]:
    """Build ``<app_label>.E001`` Errors for each missing required setting.

    The single reusable body every required-settings check calls, so no
    per-setting boilerplate is hand-written.
    """
    return [
        Error(
            f"{name} is required but is not set.",
            hint=f"Set {name} in your Django settings.",
            id=f"{app_label}.E001",
        )
        for name in config.missing_required()
    ]
```

**Typing (no `Any`, per CLAUDE.md):** each per-app subclass declares its settings
as **class-level annotations without assignment**, so consumers get real static
types while `__getattr__` supplies values at runtime (the django-stubs idiom).
Fall back to typed `@property` accessors only if the lint/type step rejects the
idiom — decide by running the step, not up front.

### L0.2 — per-app `config.py` (repeated pattern)

One module per app, ending in a module-level `config` singleton. Example
(`freedom_ls/course_access/config.py`):

```python
from __future__ import annotations

from freedom_ls.base.app_settings import AppSettings, Setting


class CourseAccessConfig(AppSettings):
    COURSE_ACCESS_BACKEND: str

    declared_settings = {
        "COURSE_ACCESS_BACKEND": Setting(required=True),
        # an optional setting reads: Setting(default="heroicons")
    }


config = CourseAccessConfig()
```

### L0.3 — settings ownership map (maintained surface)

`config.py` lives in the app that **reads** the setting. Dotted-string defaults do
**not** create an import edge. Most FLS settings have safe defaults; the only
genuinely REQUIRED one today is `COURSE_ACCESS_BACKEND`.

| App `config.py` | Settings (→ default) | Required? |
|---|---|---|
| `course_access` | `COURSE_ACCESS_BACKEND` → REQUIRED | ✅ |
| `content_engine` | `COURSE_ACCESS_CONFIG_VALIDATOR`, `MARKDOWN_ALLOWED_TAGS`, `ADMONITION_TYPES`, `MARKDOWN_TEMPLATE_RENDER_ON`, `COTTON_SNAKE_CASED_NAMES` | — |
| `icons` | `FREEDOM_LS_ICON_SET`→`"heroicons"`, `FREEDOM_LS_ICON_OVERRIDES`→`{}`, `FREEDOM_LS_ICON_BACKEND`→`None` | — |
| `accounts` | `TRUSTED_PROXY_IP_HEADER`→`None`, `REQUIRE_NAME`→`True`, `REQUIRE_TERMS_ACCEPTANCE`→`False`, `ADDITIONAL_REGISTRATION_FORMS`→`[]`, `ALLOW_SIGN_UPS`, `LEGAL_DOCS_MANIFEST_PATH`→`None` | — |
| `site_aware_models` | `FORCE_SITE_NAME`; **branding** (`HEADER_LOGO_STATIC_PATH`, `FAVICON_STATIC_PATH`, `HEADER_TITLE`, `HEADER_TITLE_STYLE`, `EMAIL_LOGO_STATIC_PATH`) → `None` — homed at this low layer so `accounts`/templates read *down*, no reverse edge | — |
| `webhooks` | `WEBHOOK_EVENT_TYPES` → `FLS_WEBHOOK_EVENT_TYPES` | — |
| `role_based_permissions` | `FREEDOMLS_PERMISSIONS_MODULES` → `{}` | — |
| `base` | `PREMAILER_OPTIONS`; theme values are **read-only** here (see below) | — |
| `student_management` | `DEADLINES_ACTIVE` → `True` (migrate existing `config.py` onto the base class) | — |

**Deliberately NOT routed through `config.py`** (stay direct `settings.X`):

- **Django built-ins** — `AUTH_USER_MODEL` (incl. every migration
  `swappable_dependency`), `MIDDLEWARE`, `INSTALLED_APPS`, `TEMPLATES`,
  `STATICFILES_DIRS`, `DATABASES`, `LOGIN_REDIRECT_URL`. The `AUTH_USER_MODEL`
  **pinned-value** check stays bespoke (`freedom_ls_accounts.E003`, T4.3), not a
  `config.py` entry.
- **Env-derived / third-party** — `WEBHOOK_ENCRYPTION_SALT`/`SALT_KEY` (derived
  dev fallback in `settings_base`), `ACCOUNT_DEFAULT_HTTP_PROTOCOL` (allauth).
- **Theme derived values** — `RESOLVED_THEME_DIR` is *computed* by
  `configure_theme(...)` (`config/settings_base.py:240`, mutates
  `TEMPLATES`/`STATICFILES_DIRS`). `config.py` may **read** `FLS_THEME` /
  `FLS_THEMES_DIRS` / `RESOLVED_THEME_DIR` but must **never** call
  `configure_theme`; no new theme check (the existing startup
  `ImproperlyConfigured` in `base/theming.py` already fails loud — D4).

### L0.4 — RISK-1 (must fix): loader-through-config vs `course_access` checks

Rerouting `course_access/loader.py:37` to read `config.COURSE_ACCESS_BACKEND`
makes `get_course_access_backend()` raise `ImproperlyConfigured` when the backend
is unset. `freedom_ls/course_access/checks.py::check_course_access_configs` calls
the loader and only catches DB errors, so an unset backend would crash
`manage.py check` with a traceback — the exact D8 failure. **Fix:** guard that
check to `return []` early when `config.missing_required()` is non-empty (the
Layer-4 `student_interface` E001 already reports the unset case cleanly).
