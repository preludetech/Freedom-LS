# Lightweight xAPI Event Tracking

We need to keep track of everything that users on the platform do.

We will base our approach on xAPI (Experience API) — using it as a **design guide**, not strict compliance. We use xAPI concepts (actor/verb/object/result) and standard ADL verb IRIs where they fit, but store data in a simple, queryable format in our own database. No full LRS.

## App structure

Rename the existing `xapi_learning_record_store` directory to `xapi` (the app config already uses `freedom_ls.xapi`). All tracking functionality lives in this app. The app must use `SiteAwareModel` for multi-tenant isolation.

## Database: PostgreSQL is sufficient

Research confirms PostgreSQL 17 handles this scale comfortably. At ~5K DAU with ~50 events/user/day = ~90M events/year, well within PostgreSQL's range.

Use a hybrid schema: typed columns for frequently-queried fields (actor, verb, object_type, object_id, timestamp) plus a JSONB column for the full xAPI-shaped statement data. This gives fast queries on common patterns while preserving flexibility.

Consider monthly table partitioning for the events table if/when data volume warrants it — but this can be deferred to a later phase. Start simple.

## Backend tracking: explicit helper functions

The primary tracking pattern is explicit helper function calls from views and services:

```python
from freedom_ls.xapi.tracking import track
from freedom_ls.xapi.verbs import COMPLETED, VIEWED

track(request.user, VIEWED, topic)
track(request.user, COMPLETED, form, result={"score": 85, "max_score": 100})
```

Key design principles:
- **Minimal required arguments**: `track(user, verb, object)` — infer the rest
- **Accept Django model instances**: the tracker extracts IDs, types, names from model instances
- **Verb constants**: defined as constants to prevent typos and enable IDE autocomplete
- **Optional result data**: scores, success, duration passed as keyword arguments
- **Return the created event**: useful for testing

We maintain our own controlled vocabulary of verbs and activity types instead of relying on external xAPI repositories. Start small (~10 verbs) covering: viewed, completed, attempted, answered, registered, progressed, submitted, interacted.

## Async event writing via Django Tasks

Use the Django Tasks framework for event persistence. Default to `ImmediateBackend` for now (synchronous writes). This gives us the abstraction layer to switch to async writes later without code changes when a production backend is configured.

## Frontend tracking

Include basic frontend tracking for client-side events that don't produce backend requests:

- **Time on page / engaged time**: track how long learners spend on content, distinguishing focused vs idle time using the Page Visibility API
- **Scroll depth**: use IntersectionObserver with sentinel elements to detect how far learners read content

Implementation approach:
- Use the **Beacon API** (`navigator.sendBeacon`) with `visibilitychange` events for reliable data delivery on page leave
- **Event batching**: collect events and flush periodically or on navigation, not per-event
- **HTMX integration**: hook into `htmx:pushedIntoHistory` and `htmx:afterSettle` to handle partial page swaps (reset timers, re-attach observers)
- Keep it lightweight — vanilla JS, no analytics libraries

## What to track (initial scope)

Backend events:
- Student views a topic → `experienced`
- Student completes a topic → `completed`
- Student starts a form → `attempted`
- Student answers a form question → `answered`
- Student submits/completes a form → `completed` (with scores in result)
- Student registers for a course → `registered`
- Course progress updates → `progressed`

Frontend events:
- Time on page (total and engaged time)
- Scroll depth on topic content pages

## What NOT to build

- No full LRS REST API
- No statement forwarding to external LRS
- No signed statements or sub-statements
- No OAuth endpoint for xAPI
- No voiding mechanism (yet)
- No keystroke/mouse tracking, no PII in frontend payloads

## Course content quality monitoring

We will implement course content quality monitoring later. The xapi data will support seeing where people typically struggle and how long people take to do different things. Keep this in mind when designing the data model.

## Privacy

- Actor identification uses internal user IDs (account-based IFI), not emails
- Frontend tracking sends no PII — the backend associates events with users via session auth
- Must support right-to-erasure (delete all tracking data for a user)
- Tracking data stays internal, no third-party sharing
- Define a retention policy for old event data
