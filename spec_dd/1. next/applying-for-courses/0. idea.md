In this spec, we describe a subscriptions backend system spec_dd/0. drafts/03. subscriptions

The first type of course subscription we really need is this:

1. Some courses are free, users can register whenever they want
2. Some courses require the learner to apply

# Applying for a course

Different courses might need different things. This needs to be configurable.

For example, a course application might require a learner to state their location, pick from some options, or upload a document (like an ID or similar)

An application process might have multiple stewps (we dont want to only be able to show users a wall of text type form)

In cases where they have filled in a form incorrectly (eg given us the wrong document, made a typo), they should be able to go back and edit it.

Administrators should be able to  mark an application as approved. This does not automatically enroll the student in a course, there might be extra steps that happen off the platform. Administrators can then add a learner to a cohort and register them for a course in the usual way.
