# Spec: FLS integration system checks (test-portability 3)

> **Layer 4**, split verbatim from the `fls-test-portability-part-2` spec.
> Everything below this note is copied unchanged from the parent spec; only this
> note and the selection of the relevant sections are new. Only success criterion
> `#3` and decisions `D3`/`D7`/`D8` pertain to this slice (parent numbering kept).

## Problem

FLS ships as a package installed into concrete downstream projects (e.g. First
Class). "Did I wire FLS up correctly?" is the signal a downstream actually needs
from a test run, and today it **isn't a supported, dedicated thing at all**. The
canonical failure is the First Class integration of the `home_page` update:

- `student_interface` newly required `settings.COURSE_ACCESS_BACKEND`, which
  First Class had never set. FLS's loader (`get_course_access_backend()`,
  `freedom_ls/course_access/loader.py:23-39`) is `@functools.cache`d and does
  `import_string(settings.COURSE_ACCESS_BACKEND)` with **no fallback** — an
  unset key raises only on the *first real request*, not at `manage.py check`
  time.
- First Class's root `urls.py` was missing the `applications/` include.

Both would 500 the catalogue/dashboard at runtime while `manage.py check` passed
clean. They were caught only as a side effect of FLS's own tests being vacuumed
up by the downstream's `pytest` — the exact coupling Part 1 removed. Removing the
noise leaves this real signal absent. Part 2 supplies it.

This is not hypothetical drift: `fls-claude-plugin/resources/template_repo_manifest.md`'s
own `urls.py` checklist (lines ~210-219) still omits `applications/`,
`interest/`, and the sitemap/robots wiring that `config/urls.py:51-73` actually
ships. A downstream following that checklist reproduces the opening bug verbatim.

### Success criteria

3. Deleting `COURSE_ACCESS_BACKEND` from settings makes `manage.py check` exit
   non-zero with `freedom_ls_student_interface.E001`; likewise overriding
   `AUTH_USER_MODEL` away from `freedom_ls_accounts.User` (with `accounts`
   installed) fails with `freedom_ls_accounts.E003`. Required-setting
   presence/value is enforced at `check` time, not by the conformance suite (D8).

## Layer 4 — Django system checks

Add `django.core.checks` so `manage.py check` **fails** on the static config
gaps, independent of any test run. This extends existing house style — FLS
already ships checks in `course_access/`, `icons/`, `accounts/`, `base/`; the
gap is that `student_interface/apps.py` is a bare `AppConfig` with no `ready()`.

Layer 4 is the **sole home for required-setting presence/value enforcement**
(D8): "is the setting set / does it equal its pinned value" is a pure static
question, exactly what `manage.py check` is for, and each check is registered
from the app that consumes the setting so the same override-gating that Part 1
established applies for free — a downstream that drops the owning app gets
nothing to enforce.

New `freedom_ls/student_interface/checks.py`, registered from a new
`StudentInterfaceConfig.ready()` (`from . import checks  # noqa: F401`). All
three are **conditional on `student_interface` being installed** — a downstream
that replaces or removes the app gets none of them (override-friendly by
construction; a check whose owning app is absent has nothing to enforce):

- **`freedom_ls_student_interface.E001`** — `student_interface` installed but
  `COURSE_ACCESS_BACKEND` unset → **Error**.
- **`freedom_ls_student_interface.E002`** — configured `COURSE_ACCESS_BACKEND`'s
  containing app not in `INSTALLED_APPS` → **Error** (resolve via
  `apps.get_containing_app_config()` / `apps.is_installed()`, mirroring Django's
  own `admin.check_dependencies`).
- **`freedom_ls_student_interface.W001`** — sitemaps wired but
  `django.contrib.sitemaps` absent → **Warning**.

Extend `freedom_ls/accounts/checks.py` (already registered via
`AccountsConfig.ready()`) with the pinned-value probe for its own required
setting — **conditional on `accounts` being installed**, gated identically:

- **`freedom_ls_accounts.E003`** — `accounts` installed but `AUTH_USER_MODEL !=
  "freedom_ls_accounts.User"` (unset or overridden away) → **Error**. (E001 is
  free but retired-looking; E002/W001 are taken — use a fresh id so a
  downstream's `SILENCED_SYSTEM_CHECKS` can target it unambiguously.)

### Conventions (from the existing check modules — follow, with one correction)

- **App-label-namespaced IDs** (`freedom_ls_student_interface.E001`), matching
  `course_access`/`base`/`accounts`. **Do not copy `icons/checks.py`'s flat
  `freedom_ls.E00N` scheme** — it is the one inconsistent precedent (see D3).
- Register at import time via `@register()`; `ready()` only imports the module.
- **No `Tags.database`, no `deploy=True`** — these are internal-consistency
  checks that must run on every `check`/`runserver`/`migrate`.
- **No DB access** — read `settings` + `apps` only (closest analog:
  `base/checks.py`).
- **Respect the `app_configs` contract:** when `app_configs is not None`,
  early-return `[]` unless the owning app's label (`student_interface`,
  `accounts`) is in it, so scoped `manage.py check <app>` runs stay correct.
- **Checks must never raise** — Django does not catch check exceptions; a raised
  exception breaks `check`/`runserver`/`migrate` with a raw traceback. Any
  dotted-path resolution (E002) must catch `ImportError`/`ModuleNotFoundError`/
  `AttributeError` and return an `Error`, never propagate.
- **Unique, stable IDs** so `SILENCED_SYSTEM_CHECKS` can target them precisely.

### Layer 3 vs Layer 4 division (both needed — the `home_page` bugs split along it)

Static config-shape questions ("is the setting set", "is its app installed")
belong in **checks** — they must fail even if the downstream never wrote a test.
Behavioural questions ("does the backend instantiate", "does this namespace
reverse") belong in the **conformance suite** — verifying them means *executing*
code. Checks would not have caught the missing URL include; the suite would not
have failed boot for the unset setting.

The division cuts cleanly for required settings: presence/value is Layer 4 only
(E001, E003), and its Layer-3 counterpart is the *behavioural*
`test_configured_backend_instantiates`, not a static re-assertion (D8). This is
deliberately **not** the `icons` "same question, both layers" pattern — there,
the Layer-4 `check_iconify_json_exists` (loads a JSON file) and the Layer-3
`test_active_icon_set_resolves` (renders an icon) ask genuinely *different*
questions that happen to rhyme; a static "is `AUTH_USER_MODEL` set" duplicated
across both layers would instead be the *same* assertion maintained twice.

## Key decisions

- **D3 — app-label-namespaced check IDs**, not the flat `freedom_ls.E00N` scheme
  `icons/checks.py` uses (the lone inconsistent precedent).
- **D7 — override-friendliness over completeness.** FLS is built to be extended
  and overridden; a downstream may discard whole apps. Probes and checks are
  gated on their app being installed and **skip/no-op when absent** — the suite
  never mandates using all of FLS. This constraint outranks D2's breadth: "probe
  all FLS app namespaces" means all *installed* ones. The sole enforced baseline
  is D1 (sitemap/robots), itself silenceable. Override-friendliness applies at
  **two granularities** — whole-app (gate on `INSTALLED_APPS`) and
  within-a-kept-app route (D9).
- **D8 — required-setting presence/value is Layer 4 only, not a Layer-3 probe.**
  "Is the setting set / does it equal its pinned value" is a pure static
  question, and `manage.py check` is built to answer it (`base/checks.py` already
  reads `settings` this way; `E001` already does it for `COURSE_ACCESS_BACKEND`).
  So there is **no `test_required_setting_present`** in the conformance suite;
  the required-settings table is not a Layer-3 maintained surface. Each check is
  registered from the app that consumes the setting (`COURSE_ACCESS_BACKEND` →
  `student_interface` E001/E002; `AUTH_USER_MODEL` pinned value → `accounts`
  E003), inheriting the same install-gating for free. Layer 3 keeps only the
  *behavioural* seam — `test_configured_backend_instantiates` executes the loader
  end-to-end, which a static check cannot. This avoids maintaining the same
  static assertion in two layers; it is deliberately not the `icons`
  "same-question-both-layers" echo, whose two sides ask materially different
  questions (see the Layer 3 vs Layer 4 division).

## Testing strategy

- **System checks:** mirror `course_access/tests/test_checks.py` — call each
  check directly with `app_configs=None`, drive config via `@override_settings`,
  assert on returned message IDs/levels. Cover: E001 fires when
  `COURSE_ACCESS_BACKEND` is unset; E003 fires when `AUTH_USER_MODEL` is
  overridden away from `freedom_ls_accounts.User`; the `app_configs`
  early-return; and the E002 exception-safety path (unimportable backend path →
  returned `Error`, not a raised exception).
