# Spec: FLS integration system checks (test-portability 3)

> Split out of `fls-test-portability-part-2`. **This is Layer 4** — boot-time
> `django.core.checks`. Design detail is in this folder's `2. plan.md` (§ "Layer
> 4", T4.1–T4.4). Assumes Part 1 (marker taxonomy, collection-safety, de-branding)
> is present.

## Problem

The First Class `home_page` integration 500'd the catalogue/dashboard at runtime
while `manage.py check` **passed clean**: `student_interface` newly required
`settings.COURSE_ACCESS_BACKEND`, which First Class had never set, and the loader
raises only on the first real request. `manage.py check` passing while the site was
broken is the strongest argument for this layer: convert the silent runtime 500
into a boot-time failure visible in CI/deploy, independent of any test run.

FLS already ships checks in `course_access/`, `icons/`, `accounts/`, `base/`; the
gap is that `student_interface/apps.py` is a bare `AppConfig` with no `ready()`.

## Goal

`manage.py check` **fails** on the static config gaps we hit, independent of any
test run.

### Success criteria

1. Deleting `COURSE_ACCESS_BACKEND` from settings makes `manage.py check` exit
   non-zero with `freedom_ls_student_interface.E001`.
2. Overriding `AUTH_USER_MODEL` away from `freedom_ls_accounts.User` (with
   `accounts` installed) makes `manage.py check` fail with
   `freedom_ls_accounts.E003`.
3. A configured `COURSE_ACCESS_BACKEND` whose containing app is not in
   `INSTALLED_APPS` fails with `freedom_ls_student_interface.E002` — and an
   unimportable dotted path returns that `Error` **without raising** (checks must
   never raise).
4. Sitemaps wired but `django.contrib.sitemaps` absent → warning
   `freedom_ls_student_interface.W001`.
5. All checks are **conditional on their owning app being installed** — a
   downstream that replaces/removes the app gets none of them; and scoped
   `manage.py check <app>` runs honour the `app_configs` contract.

## Layer 4 — Django system checks

Layer 4 is the **sole home for required-setting presence/value enforcement** (D8):
"is the setting set / does it equal its pinned value" is a pure static question,
exactly what `manage.py check` is for, and each check is registered from the app
that consumes the setting so the same override-gating Part 1 established applies for
free.

New `freedom_ls/student_interface/checks.py`, registered from a new
`StudentInterfaceConfig.ready()` (`from . import checks  # noqa: F401`). All three
are **conditional on `student_interface` being installed**:

- **`freedom_ls_student_interface.E001`** — `student_interface` installed but
  `COURSE_ACCESS_BACKEND` unset → **Error**.
- **`freedom_ls_student_interface.E002`** — configured `COURSE_ACCESS_BACKEND`'s
  containing app not in `INSTALLED_APPS` → **Error** (resolve via
  `apps.get_containing_app_config()` / `apps.is_installed()`, mirroring Django's own
  `admin.check_dependencies`).
- **`freedom_ls_student_interface.W001`** — sitemaps wired but
  `django.contrib.sitemaps` absent → **Warning**.

Extend `freedom_ls/accounts/checks.py` (already registered via
`AccountsConfig.ready()`) with the pinned-value probe for its own required setting —
**conditional on `accounts` being installed**:

- **`freedom_ls_accounts.E003`** — `accounts` installed but `AUTH_USER_MODEL !=
  "freedom_ls_accounts.User"` (unset or overridden away) → **Error**. (E001 is free
  but retired-looking; E002/W001 are taken — use a fresh id so a downstream's
  `SILENCED_SYSTEM_CHECKS` can target it unambiguously.)

### Conventions (from the existing check modules — follow, with one correction)

- **App-label-namespaced IDs** (`freedom_ls_student_interface.E001`), matching
  `course_access`/`base`/`accounts`. **Do not copy `icons/checks.py`'s flat
  `freedom_ls.E00N` scheme** — it is the one inconsistent precedent (D3).
- Register at import time via `@register()`; `ready()` only imports the module.
- **No `Tags.database`, no `deploy=True`** — internal-consistency checks that must
  run on every `check`/`runserver`/`migrate`.
- **No DB access** — read `settings` + `apps` only (closest analog: `base/checks.py`).
- **Respect the `app_configs` contract:** when `app_configs is not None`,
  early-return `[]` unless the owning app's label is in it.
- **Checks must never raise** — Django does not catch check exceptions. Any
  dotted-path resolution (E002) must catch `ImportError`/`ModuleNotFoundError`/
  `AttributeError` and return an `Error`, never propagate.
- **Unique, stable IDs** so `SILENCED_SYSTEM_CHECKS` can target them precisely.

### Layer 3 vs Layer 4 division (both needed — the `home_page` bugs split along it)

Static config-shape questions ("is the setting set", "is its app installed") belong
in **checks** — they must fail even if the downstream never wrote a test.
Behavioural questions ("does the backend instantiate", "does this namespace
reverse") belong in the **conformance suite** (`test_portability_2_conformance_suite`)
— verifying them means *executing* code. Checks would not have caught the missing
URL include; the suite would not have failed boot for the unset setting.

The division cuts cleanly for required settings: presence/value is Layer 4 only
(E001, E003), and its Layer-3 counterpart is the *behavioural*
`test_configured_backend_instantiates`, not a static re-assertion (D8). This is
deliberately **not** the `icons` "same question, both layers" pattern — there, the
Layer-4 `check_iconify_json_exists` (loads a JSON file) and the Layer-3
`test_active_icon_set_resolves` (renders an icon) ask genuinely *different*
questions; a static "is `AUTH_USER_MODEL` set" duplicated across both layers would
instead be the *same* assertion maintained twice.

## Key decisions (Layer-4 relevant)

- **D3 — app-label-namespaced check IDs**, not the flat `freedom_ls.E00N` scheme
  `icons/checks.py` uses (the lone inconsistent precedent).
- **D7 — override-friendliness.** Checks are gated on their app being installed and
  no-op when absent — a check whose owning app is absent has nothing to enforce.
- **D8 — required-setting presence/value is Layer 4 only, not a Layer-3 probe.**
  Each check is registered from the app that consumes the setting
  (`COURSE_ACCESS_BACKEND` → `student_interface` E001/E002; `AUTH_USER_MODEL` pinned
  value → `accounts` E003), inheriting the same install-gating for free.

## Testing strategy

Mirror `course_access/tests/test_checks.py` — call each check directly with
`app_configs=None`, drive config via `@override_settings`, assert on returned
message IDs/levels. Cover:

- E001 fires when `COURSE_ACCESS_BACKEND` is unset;
- E003 fires when `AUTH_USER_MODEL` is overridden away from `freedom_ls_accounts.User`,
  passes at the pinned value;
- the `app_configs` early-return;
- E002 exception-safety: `@override_settings(COURSE_ACCESS_BACKEND="nope.Nope")` →
  returned `Error` with id `...E002`, **not** a raised exception;
- App-absent gate: `@override_settings(INSTALLED_APPS=<without the app>)` → every
  check returns `[]`.
- Note: no prior art overrides `AUTH_USER_MODEL` in tests — the check only *reads*
  the string, so `@override_settings` is sufficient (no model swap / DB; keep it
  non-`django_db`).

## Relationship to the other test-portability slices

- **`test_portability_1_settings_config_convention` (Layer 0)** — the Part-2 *plan*
  single-sources E001 from Layer 0's `course_access.config` REQUIRED declaration via
  `required_settings_errors(...)`. This slice **can ship without Layer 0** by having
  E001 read the setting directly (the original spec framing). Decide when speccing:
  land Layer 0 first for the single-sourced version.
- **`test_portability_2_conformance_suite` (Layer 3)** — complementary (D8); the
  behavioural backend probe lives there.
- **`test_portability_4_upgrade_notes_and_docs` (Layers 5/6)** — the upgrade-notes
  hard-requirement guidance points downstreams at `manage.py check`.
