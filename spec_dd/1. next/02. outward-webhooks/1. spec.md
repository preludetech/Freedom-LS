# Outward Webhooks — Specification

## Overview

FLS needs to notify third-party services about platform events via outbound webhooks. For example, when a new user registers, a webhook can notify a transactional email provider to kick off an onboarding flow.

The webhook system lives in its own Django app (`webhooks`) and is site-aware: each site configures its own endpoints and subscriptions independently.

## Dependencies

This feature depends on the **Django 6.0 upgrade** completing first, as it uses Django's built-in Tasks framework (`django.tasks`) for asynchronous delivery.

## Architecture

### Triggering pattern

Other apps trigger webhook events by calling an explicit utility function:

```python
from freedom_ls.webhooks.events import fire_webhook_event

fire_webhook_event(event_type="user.registered", payload={...})
```

This matches the existing codebase convention of explicit function calls over Django signals. The function is called at the point where the event occurs (e.g., in a view or model method). Internally, `fire_webhook_event` uses `transaction.on_commit()` to ensure the event is only dispatched after the database transaction commits, then enqueues a background task for delivery.

**Why explicit calls over signals:** The codebase uses zero Django signals. Explicit calls are easier to trace, debug, and grep for. The trade-off is a direct import dependency on the webhooks app, but since webhooks is a core infrastructure app this is acceptable.

### Delivery pipeline

1. App code calls `fire_webhook_event(event_type, payload)`
2. Inside `transaction.on_commit()`, a background task is enqueued via `django.tasks`
3. The task looks up all active `WebhookEndpoint` records for the current site that subscribe to the given event type
4. For each matching endpoint, the task creates a `WebhookDelivery` record and sends an HTTP POST
5. On failure, the task schedules a retry with exponential backoff
6. After all retries are exhausted, the delivery is marked as dead-lettered

### Task framework

Uses Django 6.0's built-in Tasks framework (`django.tasks`). This provides a lightweight, database-backed task queue with no external dependencies (no Redis, no RabbitMQ). Tasks are processed by a worker that reads from the database.

**Why `django.tasks`:** It's built into the framework, requires zero additional infrastructure, and is sufficient for webhook delivery volumes in an LMS context.

## Models

All models extend `SiteAwareModel` for multi-tenant isolation.

### Event type constants

Event types are defined as a `TextChoices` enum in code — they are not stored in a database table:

```python
class EventType(models.TextChoices):
    USER_REGISTERED = "user.registered", "User registered"
    TOPIC_COMPLETED = "topic.completed", "Topic completed"
    FORM_COMPLETED = "form.completed", "Form completed"
    COURSE_COMPLETED = "course.completed", "Course completed"
    COURSE_REGISTERED = "course.registered", "Course registered"
    COHORT_MEMBER_ADDED = "cohort.member_added", "Cohort member added"
    COHORT_MEMBER_REMOVED = "cohort.member_removed", "Cohort member removed"
```

All event type fields (on endpoints and events) are validated against this enum. Adding a new event type means adding a constant to this enum and a schema migration for the new choice.

**Why hard-coded constants:** Event type strings must exactly match what's in the code that fires them. A database table would allow edits that break this contract. Constants are the source of truth, and the admin renders them as a selectable list — no typos possible.

### WebhookEndpoint

Represents a configured webhook destination for a site.

| Field | Type | Notes |
|---|---|---|
| `url` | URLField | The endpoint URL. HTTPS required in production |
| `description` | CharField | Human-readable label (e.g., "Mailchimp sync") |
| `secret` | CharField | HMAC-SHA256 signing secret, auto-generated |
| `event_types` | JSONField | List of subscribed `EventType` values, validated against the enum |
| `is_active` | BooleanField | Admin toggle to enable/disable |
| `failure_count` | PositiveIntegerField | Consecutive delivery failures (for circuit breaker) |
| `disabled_at` | DateTimeField, nullable | When the circuit breaker tripped |
| `created_at` | DateTimeField | Auto-set on creation |
| `updated_at` | DateTimeField | Auto-updated |

The `secret` is auto-generated on creation using `secrets.token_urlsafe(48)` (same pattern as `app_authentication.Client.api_key`).

The `event_types` JSONField stores a list of event type strings (e.g., `["user.registered", "course.completed"]`). Model validation ensures every value in the list is a valid `EventType` member. In the admin, this is rendered as a **checkbox widget** populated from the `EventType` enum — admins select from the known list rather than typing strings.

### WebhookEvent

Represents a single event that was fired. One event may produce multiple deliveries (one per matching endpoint).

| Field | Type | Notes |
|---|---|---|
| `event_type` | CharField(choices=EventType) | The type of event that occurred |
| `payload` | JSONField | The event data |
| `event_id` | UUIDField | Unique ID for idempotency (sent in headers and body) |
| `created_at` | DateTimeField | When the event was fired |

### WebhookDelivery

Tracks each delivery attempt to a specific endpoint for a specific event.

| Field | Type | Notes |
|---|---|---|
| `event` | ForeignKey(WebhookEvent) | The event being delivered |
| `endpoint` | ForeignKey(WebhookEndpoint) | The target endpoint |
| `status` | CharField(choices) | `pending`, `success`, `failed`, `dead_letter` |
| `attempt_count` | PositiveIntegerField | Number of attempts made |
| `next_retry_at` | DateTimeField, nullable | When the next retry is scheduled |
| `last_status_code` | PositiveIntegerField, nullable | HTTP status code from last attempt |
| `last_response_body` | TextField, nullable | Truncated response body (first 500 chars) |
| `last_attempt_at` | DateTimeField, nullable | When the last attempt was made |
| `last_latency_ms` | PositiveIntegerField, nullable | Response time of last attempt |
| `created_at` | DateTimeField | When the delivery was created |

**Why three models:** Separating events from deliveries allows one event to fan out to multiple endpoints, supports the dead letter queue, and keeps delivery history per-endpoint for debugging.

## Event Types and Payloads

Event types follow the `resource.action` dot-notation convention.

### Initial event types

#### `user.registered`

Fired when a new user account is created.

```json
{
  "user_id": "uuid",
  "email": "user@example.com",
  "date_joined": "2026-03-10T12:00:00Z"
}
```

#### `topic.completed`

Fired when a student completes a topic.

```json
{
  "user_id": "uuid",
  "topic_id": "uuid",
  "topic_title": "Introduction to Python",
  "completed_time": "2026-03-10T12:00:00Z"
}
```

#### `form.completed`

Fired when a student completes a form (quiz, survey, etc.).

```json
{
  "user_id": "uuid",
  "form_id": "uuid",
  "form_title": "Module 1 Quiz",
  "completed_time": "2026-03-10T12:00:00Z",
  "score_percentage": 85.0
}
```

`score_percentage` is `null` for non-quiz forms.

#### `course.completed`

Fired when a student completes a course (100% progress).

```json
{
  "user_id": "uuid",
  "course_id": "uuid",
  "course_title": "Python Fundamentals",
  "completed_time": "2026-03-10T12:00:00Z"
}
```

#### `course.registered`

Fired when a student is registered/enrolled in a course.

```json
{
  "user_id": "uuid",
  "course_id": "uuid",
  "course_title": "Python Fundamentals",
  "registered_at": "2026-03-10T12:00:00Z"
}
```

#### `cohort.member_added`

Fired when a student is added to a cohort.

```json
{
  "user_id": "uuid",
  "cohort_id": "uuid",
  "cohort_name": "Batch 2026-Q1"
}
```

#### `cohort.member_removed`

Fired when a student is removed from a cohort.

```json
{
  "user_id": "uuid",
  "cohort_id": "uuid",
  "cohort_name": "Batch 2026-Q1"
}
```

### Adding new event types

Adding a new event type requires two steps:

1. **Code:** Add a new constant to the `EventType` enum and generate a schema migration for the updated choices.
2. **Code:** Add a `fire_webhook_event()` call wherever the event occurs.

Endpoint admins can then subscribe to the new type via the checkbox widget in the admin.

### Payload envelope

Every webhook POST body follows this envelope format:

```json
{
  "id": "event-uuid",
  "type": "user.registered",
  "timestamp": "2026-03-10T12:00:00Z",
  "data": { ... }
}
```

Payloads are kept small — IDs and key fields only. Consumers fetch full resources via API if needed. Sensitive data (passwords, tokens) is never included.

## Standards and Security

### Standard Webhooks spec

Follows the [Standard Webhooks](https://www.standardwebhooks.com/) specification for interoperability with tools like Svix, Stripe, etc.

Each delivery includes these headers:

| Header | Value |
|---|---|
| `webhook-id` | The event UUID |
| `webhook-timestamp` | Unix timestamp (seconds) of the delivery attempt |
| `webhook-signature` | `v1,{base64-encoded-signature}` |

### Signing

The signature is computed as:

```
HMAC-SHA256(
    key=endpoint.secret,
    message="{webhook-id}.{webhook-timestamp}.{body}"
)
```

Where `body` is the raw JSON request body. This matches the Standard Webhooks signing scheme.

**Why per-endpoint secrets:** Each endpoint gets its own secret so compromising one doesn't affect others.

### HTTPS enforcement

In production, endpoint URLs must use HTTPS. The `WebhookEndpoint.clean()` method validates this when `settings.DEBUG` is `False`. In development, HTTP is permitted for local testing.

## Retry and Resilience

### Retry schedule

Failed deliveries are retried with exponential backoff and jitter:

| Attempt | Base delay |
|---|---|
| 1 | 1 minute |
| 2 | 5 minutes |
| 3 | 30 minutes |
| 4 | 2 hours |
| 5 | 12 hours |

Jitter of up to 20% is added to each delay to prevent thundering herd.

After 5 retries (6 total attempts including the initial), the delivery is marked `dead_letter`.

### Response classification

| Response | Action |
|---|---|
| 2xx | Success — delivery complete |
| 429 | Retry — honour `Retry-After` header if present |
| 4xx (except 429) | Permanent failure — no retry, mark `failed` |
| 5xx | Retry with backoff |
| Timeout (30s) | Retry with backoff |
| Connection error | Retry with backoff |

### Circuit breaker

Prevents wasting resources on consistently-failing endpoints.

- **Trip threshold:** 5 consecutive failures across any deliveries to the endpoint
- **Behaviour when tripped:** Endpoint is marked inactive (`disabled_at` is set), new deliveries are not attempted
- **Probe interval:** Every 1 hour, one pending delivery is attempted
- **Re-enable:** On first successful probe, `failure_count` resets to 0 and `disabled_at` is cleared

**Why a circuit breaker:** Without it, a single misconfigured or down endpoint generates retries indefinitely, consuming task queue capacity and potentially slowing down delivery to healthy endpoints.

### Dead letter queue

Deliveries that exhaust all retries are marked `status=dead_letter`. They remain in the database for manual inspection and can be replayed from the admin interface.

## Admin Interface

Managed through Django admin (Unfold) using `SiteAwareModelAdmin`. A dedicated UI may be added later.

### WebhookEndpoint admin

- List display: description, URL (truncated), event types, is_active, failure_count, created_at
- Filters: is_active, event_types, created_at
- Actions: enable/disable endpoints
- Detail view shows the secret (read-only after creation) and a "Send test ping" button

### WebhookDelivery admin

- List display: event type, endpoint (truncated URL), status, attempt_count, last_status_code, last_attempt_at
- Filters: status, endpoint, event type
- Actions: retry failed/dead-lettered deliveries
- Detail view shows full response body and delivery timeline

### WebhookEvent admin

- List display: event_type, event_id, created_at
- Filters: event_type, created_at
- Read-only — events are system-generated

### Test ping

Admins can send a test ping to any endpoint. This fires a `webhook.test` event with a static payload, creating a real delivery record that goes through the normal pipeline. This lets admins verify their endpoint configuration before subscribing to real events.

## Success Criteria

1. An admin can create a webhook endpoint, subscribe it to event types, and see it receive deliveries when those events occur
2. Failed deliveries are retried according to the backoff schedule
3. Endpoints that fail 5 times consecutively are auto-disabled and probed hourly
4. Dead-lettered deliveries are visible in admin and can be manually retried
5. All deliveries are signed per the Standard Webhooks spec and can be verified by consumers
6. The webhook system is fully site-aware — sites cannot see or affect each other's endpoints or deliveries
7. New event types can be added by extending the `EventType` enum and adding a `fire_webhook_event()` call
8. The test ping feature allows admins to verify endpoint configuration
