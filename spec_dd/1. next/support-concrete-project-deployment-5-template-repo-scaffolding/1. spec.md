# Spec 5 (TEMPLATE REPO): P3 reusable deployment scaffolding

> **⚠️ This spec targets the SEPARATE `freedom-ls-concrete-template` repo, not this repo.** It is
> kept in this directory for now but is meant to be **moved to the template repo and implemented
> there** (synced via `/fls:sdd:update_template_repo`). See `spec-order.md`.
>
> **Runs last.** It consumes the code primitives delivered by Spec 1 (`settings_defaults`), Spec 2
> (readiness endpoint), and Spec 3 (the durable `django-tasks-db` `DatabaseBackend` as the
> production default — so the `db_worker` worker container ships enabled by default). See
> `spec-order.md` for the full dependency chain and `third-party-services.md` for the external
> accounts these artifacts assume.

## Terminology

`ConcreteFlsImplementation` = the specific downstream production repo that raised this handoff
(installs `freedom_ls` as a read-only git submodule, owns its own `config/`, apps, and
`tailwind.input.css`). "Every concrete project" = concrete FLS projects in general. FLS is never
deployed standalone — only concretely, as a submodule of a downstream project.

## 1. Problem

A new concrete FLS project has to author its whole production deployment (single VPS, Docker
Compose, Caddy + Gunicorn + PostgreSQL) from scratch, working around the same gaps every time. The
reusable *artifacts* should live in the `freedom-ls-concrete-template` repo so a new concrete project
is "fill in the variables, run the bootstrap" — never an edit to shared logic.

## 2. Goal

Ship, into `freedom-ls-concrete-template`, a working Caddy-based, submodule-correct deployment
baseline. **Every site-specific value (domain, branding, secrets, DB names, R2 buckets, Sentry
project) must be a substitutable placeholder.**

## 3. Reuse boundary (why this is a separate repo)

The reuse boundary is **artifact-vs-code** (idea §7), split cleanly across two homes:

- **Reusable deploy *artifacts*** (Dockerfile, compose, Caddyfile, entrypoint, `.env.example`, CI,
  Ansible, backup scripts) live in **`freedom-ls-concrete-template`** and reach projects via
  `/fls:sdd:update_template_repo` — **this spec**.
- **Reusable *code* primitives** (health module, prod-settings defaults, the durable
  `DatabaseBackend` tasks primitive that is now the production default) live in **`freedom_ls`
  itself** so they are imported, not copy-pasted — Specs 1–3.

### 4.1 New Dockerfile bug beyond the idea's list

The `frontend-builder` stage copies only `tailwind.input.css` (`Dockerfile:14–17`) but that file
`@import`s three sibling root files (`tailwind.components.css`, `tailwind.base_interface.css`,
`tailwind.picture_spotlight.css`, none gitignored). The Tailwind CLI build would fail to resolve
these imports. The Dockerfile (§5.1) must `COPY` **every** `tailwind.*.css` partial the entry file
imports.

## 5. Requirements

### 5.1 — Submodule-aware Dockerfile

Multi-stage; `ARG FLS_THEME`; builds project-owned Tailwind; installs `freedom_ls` **from the
submodule**; uses `uv.lock` for reproducible builds. Concrete shape (validated against
cookiecutter-django + the astral uv Docker guide):

- **Pure-Node Tailwind stage** — the Tailwind build must **not** shell to `uv run …` inside a
  Node-only stage (the current break). Call the Tailwind CLI directly, or split a genuinely-needed
  Python step (`write_active_theme_css`) into its own earlier stage feeding the Node stage via
  `COPY --from=`.
- **`COPY` every *static* `tailwind.*.css` partial** the entry file `@import`s (the §4.1 bug), not
  just `tailwind.input.css`. The generated `tailwind.active_theme.css` is **not** COPY'd from the
  build context (it is gitignored and produced by the `write_active_theme_css` step above).
- **Two-phase `uv sync --frozen`** (deps-only `--no-install-project` before `COPY .`, full sync
  after; cache mounts; `--no-editable`), **non-root** runtime user, **`collectstatic` at build time**
  (matches WhiteNoise's local-disk model; keeps the SHA-tagged GHCR image self-contained).
- **Build-time `collectstatic` must satisfy the settings import.** Importing `config.settings_prod`
  hard-fails on missing `SECRET_KEY` (Spec 1 §5.1-F) and `HOST_DOMAIN` (already
  `os.environ[...]`), so the build step that runs `collectstatic` must supply **throwaway
  placeholder values** for these (build `ARG`/`ENV` scoped to the build) — never real secrets, and
  never baked into the runtime image or its environment. This keeps build-time `collectstatic`
  viable without weakening Spec 1's boot-time hard-fail.

Detail: `research_deployment_scaffolding_references.md`.

### 5.2 — Parameterised `docker-compose.yml`

- **Named volume, never a bind mount** for DB data.
- **`json-file` log driver with `max-size` *and* `max-file` caps per service** (required — makes Spec
  1 §5.1-B's stdout logging safe on a small VPS; `max-file` is a no-op without `max-size`), defined
  once via a **YAML anchor** (`x-logging: &default-logging`) applied to every service so a new
  service can't silently drop the cap.
- Healthchecks that speak to the Spec 2 **readiness** endpoint without tripping SSL redirect (the
  pre-wired `SECURE_REDIRECT_EXEMPT` handles this — no header gymnastics).
- `COMPOSE_PROJECT_NAME` + `--env-file` so the same file runs staging **and** prod on one host with
  isolated containers/networks/volumes.
- **No `migrate` in any entrypoint/command path** — migrations run as a one-off deploy step
  (validated against cookiecutter-django).
- The `db` service publishes **no `ports:`** to the host (the real DB-security control per Spec 1
  §5.1-D).
- A **`worker` container enabled by default** running `db_worker` against the durable
  `DatabaseBackend` (plus its `prune_db_task_results` retention job). Per Spec 3 the durable backend
  is now the **production default**, so the worker is a hard dependency of the shipped stack, not an
  opt-in profile — without it, enqueued tasks (webhook dispatch) persist in the DB and never
  execute. Never run it via the `DEBUG` autoreload path in prod.

### 5.3 — Caddyfile

One Caddy, automatic HTTPS, host-based routing; `reverse_proxy` sets
`X-Forwarded-Proto`/`-For`/`-Host` automatically (paired with Spec 1 §5.1-A) and ignores
client-supplied values unless `trusted_proxies` is set (scope to Cloudflare's ranges for
`CF-Connecting-IP`). WhiteNoise serves static and media is on S3, so Caddy **only proxies** — no
`file_server`/`handle_path` for `/static/*` or `/media/*`. Leave `auto_https` on. **Do not route
`/health/*`** (Spec 2). Domain via Caddy's native `{$HOST_DOMAIN}` env-substitution from the same
`.env` — reuse the **single** `HOST_DOMAIN` var the prod settings already read (`settings_prod.py`),
so there is exactly one domain value to fill in per deployment, not a separate Caddy `DOMAIN`.
(`third-party-services.md`, which currently names this `DOMAIN`, must be aligned to `HOST_DOMAIN`.)

### 5.4 — Supporting artifacts

`docker-entrypoint.sh`, `.env.example` (documents every var settings reads, including
`DB_SSLMODE=disable` with the same-host rationale, and a **single** `HOST_DOMAIN` consumed by both
the prod settings and the Caddyfile's `{$HOST_DOMAIN}`), `.dockerignore`.

### 5.5 — GHCR build-and-push CI

FLS ships no image-build/deploy workflow today; provide the SHA-tagged build+push template.

### 5.6 — Ansible / backups / Sentry

**Ansible** provisioning/hardening, **backups** (`pg_dump` + encrypted off-box sync), and **Sentry**
observability wiring — all as parameterised templates. External accounts and credentials these
assume are catalogued in `third-party-services.md`.

### 5.7 — Template `settings_prod.py` migration

Move the template's `settings_prod.py` from literals to **import-and-assign** against Spec 1's
`freedom_ls/deployment/settings_defaults.py` (flat constants + pure functions). This is the change
that collapses the propagation problem for **future** projects; `settings_prod.py` stays
project-owned and thin, supplying only genuinely project-specific values.

## 6. Success criteria

**Template repo is deployment-ready:** a new concrete project fills in placeholders and runs the
bootstrap to get a building, Caddy-fronted, submodule-correct stack:

- Dockerfile builds — Tailwind partials resolve, Node stage is pure-Node, `uv.lock` used,
  `collectstatic` at build time (with throwaway placeholder `SECRET_KEY`/`HOST_DOMAIN` for the build
  only, no real secrets baked in), non-root runtime.
- compose has a named volume, capped `json-file` logs (`max-size` + `max-file` via the shared
  anchor), no published `5432`, no entrypoint migrate, and a `worker` service **enabled by default**
  running `db_worker` + `prune_db_task_results`.
- Caddyfile fronts the app with auto-HTTPS, host routing via the shared `{$HOST_DOMAIN}`,
  `trusted_proxies` = Cloudflare ranges, no `/health/*` route, proxy-only.
- GHCR build+push CI present; Ansible/backups/Sentry templates present.
- `settings_prod.py` is import-and-assign against `freedom_ls.deployment.settings_defaults`.
- The `worker` service runs a real out-of-process `db_worker` by default (the durable backend is the
  prod default per Spec 3), alongside the `prune_db_task_results` retention job; deploy runs `migrate`
  as its one-off step so `django_tasks_db` tables exist before the worker starts.
