# Webhook Translation Layer

## Problem

The outbound webhooks spec (in `spec_dd/2. in progress/02. outward-webhooks/`) handles standard webhook delivery: HMAC-signed POST with a fixed JSON envelope. But many external platforms don't accept standard webhooks — they have their own API formats with different auth, headers, and payload structures.

For example, to trigger a Brevo email automation when `user.registered` fires, we need to call Brevo's custom event tracking API (`https://in-automate.brevo.com/api/v2/trackEvent`) with:
- A custom auth header (`ma-key: YOUR_MA_KEY`)
- A completely different payload shape (`{ "email": "...", "event": "...", "properties": {...} }`)
- No HMAC signing (Brevo uses its own API key auth)

Reference: https://developers.brevo.com/docs/track-custom-events-rest

## Solution: Per-endpoint transformation via Jinja2 templates

Extend the existing `WebhookEndpoint` model in the webhooks app to support optional transformation configuration. When a transformation is configured, the delivery pipeline uses it instead of the standard webhook format.

Each endpoint can optionally define:
- **URL** (can differ from standard webhook URL pattern)
- **HTTP method** (default POST, but configurable)
- **Headers template** — Jinja2-rendered, can reference secrets (e.g., `{"ma-key": "{{ secrets.brevo_api_key }}"}`)
- **Body template** — Jinja2 template that transforms the canonical event payload into the target API's format
- **Auth type** — None (custom headers handle it), or standard webhook signing

If no transformation is configured, the endpoint behaves exactly as the current webhook spec describes (Standard Webhooks format with HMAC signing).

## Secrets management

A separate `WebhookSecret` model stores named secrets (API keys, tokens) with encrypted values. Secrets are:
- Referenced by name in Jinja2 templates (e.g., `{{ secrets.brevo_api_key }}`)
- Never displayed in full in admin (masked after creation)
- Site-aware (each site has its own secrets)
- Resolved at delivery time, not stored in the template output

## Presets

Support pre-built integration presets (e.g., "Brevo") that pre-populate the transformation configuration. Presets:
- Populate all fields but don't constrain — admins can modify anything after selecting a preset
- Stored as data (fixtures or database) so new presets can be added without code deployment
- Include a "Reset to Preset Defaults" option

## Admin UX

Configured through Django admin (Unfold):
- Transformation fields appear on the existing `WebhookEndpoint` admin form
- Preset selector at the top populates fields
- Available template variables documented next to the body template field
- "Send Test" button renders the template with sample data and sends it, showing the response inline
- Validate templates at save time (check JSON validity, variable references)

## Jinja2 safety

Always use `jinja2.SandboxedEnvironment` to prevent template injection. Only trusted admin users should be able to edit templates.

## Key design decisions

1. **Part of the webhooks app** — not a separate app. Avoids duplicating the delivery pipeline.
2. **Jinja2 templates** — Python-native, admin-configurable, no JS sandbox needed. Proven by NetBox (a production Django app with the same pattern).
3. **Presets for common integrations** — reduces configuration effort and errors.
4. **Separate secrets model** — keeps credentials out of template text, supports rotation, and allows reuse across endpoints.
