# Implementation Plan: Educator Interface Better Navigation

## Overview

Four interconnected features to improve navigation in the educator panel framework:
1. HTMX-powered navigation (partial swaps instead of full page reloads)
2. Breadcrumbs
3. Sidebar active highlighting
4. Sidebar instance dropdown

All changes are centralised in the panel framework so any interface using `panel_framework_view` benefits automatically.

### 1a. Pass active state through `_build_menu_items`

**File:** `freedom_ls/panel_framework/views.py`

Modify `_build_menu_items` to accept the current `path_string` (or first path part) and mark the matching item as active:

```python
def _build_menu_items(
    config: dict[str, type[ListViewConfig]], url_name: str, active_section: str = ""
) -> list[dict[str, str | bool]]:
    return [
        {
            "label": conf.menu_label,
            "url": reverse(url_name, kwargs={"path_string": conf.url_name}),
            "active": conf.url_name == active_section,
        }

    ]
```

```python
active_section = parts[0] if parts else ""

}
```

## Task 2: Breadcrumbs

**Skills:** `fls:template`, `fls:htmx`, `fls:frontend-styling`, `fls:icon-usage`, `fls:testing`

### 2a. Generate breadcrumb data server-side

**File:** `freedom_ls/panel_framework/views.py`

Add a function `_build_breadcrumbs` that takes the resolved path info and returns a list of breadcrumb items:

```python
def _build_breadcrumbs(
    parts: list[str],
    config: dict[str, type[ListViewConfig]],
    url_name: str,
    current_instance: Model | None = None,
    root_label: str = "",
) -> list[dict[str, str]]:
    """Build hierarchy-based breadcrumbs.

    Returns list of dicts: [{"label": "...", "url": "..."}, ...]
    Last item has no "url" key (current page).
    """
```

Logic:
- Always start with root: `{"label": root_label, "url": reverse(url_name, kwargs={"path_string": ""})}`
- If parts[0] matches a config key, add section: `{"label": config.menu_label, "url": ...}`
- If an instance is resolved (parts has pk), add instance: `{"label": str(instance)}` (no url — current page)
- On a list page, the section item has no url (it's the current page)

Pass `root_label` from the template context. The educator interface passes `"Educator"` — derived from `header_bar_text` block or passed explicitly.

Update `panel_framework_view` to accept a `root_label` parameter and include breadcrumbs in context:

```python
def panel_framework_view(
    config, request, path_string, template_name, url_name, root_label: str = ""
) -> HttpResponse:
    ...
    context = {
        "menu_items": ...,
        "content": rendered_content,
        "heading": heading,
        "breadcrumbs": _build_breadcrumbs(parts, config, url_name, instance, root_label),
    }
```

**File:** `freedom_ls/educator_interface/views.py`

Update the `interface` view function to pass `root_label="Educator"` to `panel_framework_view`:

```python
return panel_framework_view(
    config=interface_config,
    request=request,
    path_string=path_string,
    template_name="educator_interface/interface.html",
    url_name="educator_interface:interface",
    root_label="Educator",
)
```

To resolve the instance for breadcrumbs: extract it during `_resolve_path` / `_dispatch_resolved`. The instance is available from `InstanceView.instance` when the resolved object is an `InstanceView` or `_ResolvedTab`. Track the section config and instance as the path resolves.

Write tests: verify breadcrumb output for each hierarchy level (root, section list, section instance).

### 2b. Create breadcrumb cotton component or template partial

**File:** `freedom_ls/panel_framework/templates/panel_framework/partials/breadcrumbs.html` (new file)

```html
<nav id="breadcrumbs" aria-label="Breadcrumb">
    <ol class="flex items-center text-sm text-muted breadcrumb-list">
        {% for crumb in breadcrumbs %}
            <li>
                {% if crumb.url %}
                    <a href="{{ crumb.url }}" class="hover:text-foreground hover:underline focus:outline-2 focus:outline-offset-2 focus:outline-primary">
                        {{ crumb.label }}
                    </a>
                {% else %}
                    <span aria-current="page" class="text-foreground font-medium">
                        {{ crumb.label }}
                    </span>
                {% endif %}
            </li>
        {% endfor %}
    </ol>
</nav>
```

Separators **must** be rendered via CSS `::before` pseudo-elements (not inline text), per the spec's accessibility requirements. This prevents screen readers from reading separator characters. Add a small CSS block:

```css
.breadcrumb-list li + li::before {
    content: "/";
    margin: 0 0.25rem;
    color: var(--color-muted);
    opacity: 0.5;
}
```

Breadcrumb links must have visible focus indicators (e.g. `focus:outline-2 focus:outline-offset-2`).

## Task 3: HTMX-Powered Navigation

**Skills:** `fls:htmx`, `fls:template`, `fls:alpine-js`, `fls:testing`, `fls:playwright-tests`

This is the most complex feature. It converts sidebar and data table links to HTMX partial swaps with OOB updates for breadcrumbs and sidebar.

### 3a. Define stable swap targets

**Files:** `educator_interface/templates/educator_interface/interface.html`, `_base_interface.html`

Add `id` attributes to the main content area and breadcrumb container:

- Breadcrumbs: already has `id="breadcrumbs"` from Task 2b
- Main content: wrap content block in `<div id="main-content">...</div>`
- Sidebar nav: already has `id="sidebar-nav"` from Task 1b

### 3b. Convert sidebar links to HTMX

**File:** `freedom_ls/educator_interface/templates/educator_interface/interface.html`

Change sidebar `<a>` tags:

```html
<a hx-get="{{ item.url }}"
   hx-target="#main-content"
   hx-push-url="true"
   hx-swap="outerHTML"
   {% if item.active %}aria-current="page"{% endif %}
   class="...">
    {{ item.label }}
</a>
```

Use `outerHTML` swap (per HTMX skill convention). The `#main-content` div must be included in the HTMX response so it replaces the whole container.

### 3c. Modify `panel_framework_view` to return OOB fragments for HTMX requests

**File:** `freedom_ls/panel_framework/views.py`

When `is_htmx` is True and the request is a top-level navigation (not a tab/panel sub-request), return the main content plus OOB fragments:

```python
if is_htmx and is_navigation_request:
    # Render main content
    main_html = rendered_content_with_heading
    # Render OOB breadcrumbs
    breadcrumb_html = render_to_string(
        "panel_framework/partials/breadcrumbs.html",
        {"breadcrumbs": breadcrumbs},
        request=request,
    )
    # Render OOB sidebar
    sidebar_html = render_to_string(
        "panel_framework/partials/sidebar_nav.html",
        {"menu_items": menu_items},
        request=request,
    )
    return HttpResponse(
        main_html + breadcrumb_html + sidebar_html
    )
```

The OOB fragments use `hx-swap-oob="true"` on their root elements (the `id="breadcrumbs"` nav and `id="sidebar-nav"` nav).

**Key distinction:** Differentiate top-level navigation HTMX requests (sidebar clicks, data table row clicks) from internal HTMX requests (tab loads, pagination, search). Use a custom header or check `HX-Target`:
- If `HX-Target` is `main-content`, it's a navigation request → return content + OOB fragments
- If `HX-Target` is something else (e.g. `data-table-container`, tab panel), it's an internal request → return fragment only (existing behaviour)

### 3d. Extract sidebar nav into its own partial template

**File:** `freedom_ls/panel_framework/templates/panel_framework/partials/sidebar_nav.html` (new file)

This template renders the sidebar `<nav>` with menu items. It's used both in the full-page render (via include) and as an OOB fragment. When rendered as OOB, it includes `hx-swap-oob="true"`:

```html
<nav id="sidebar-nav" class="sticky top-4 w-64" aria-label="Sections"
     {% if oob %}hx-swap-oob="true"{% endif %}>
    <ul class="list-none ml-0">
        {% for item in menu_items %}
            ...
        {% endfor %}
    </ul>
</nav>
```

Update the educator interface template to include this partial instead of inline HTML.

Similarly update `breadcrumbs.html` to support OOB mode.

### 3e. Convert data table row links to HTMX

**File:** `freedom_ls/base/templates/cotton/data-table-cells/link.html`

Add HTMX attributes to instance links so clicking a row item triggers an HTMX swap:

```html
<a href="{% resolve_url %}"
   {% if column.htmx_nav %}
   hx-get="{% resolve_url %}"
   hx-target="#main-content"
   hx-push-url="true"
   hx-swap="outerHTML"
   {% endif %}>
    {{ object|getattr_str:column.text_attr }}
</a>
```

The `href` remains for progressive enhancement (right-click, middle-click, JS disabled).

**Important:** The HTMX attributes are conditional via a `htmx_nav` flag on the column config. This prevents breaking data table links outside the panel framework (e.g. student interface or other contexts where `#main-content` doesn't exist). Panel framework `DataTable` subclasses set `htmx_nav: True` on their link columns.

### 3f. Prevent flash of unstyled/stale content (FOUC)

During HTMX navigation swaps, ensure there is no visible flash of stale content. Approaches:
- The `outerHTML` swap on `#main-content` is atomic — old content is replaced in one DOM operation, so there should be no flash
- If needed, add a brief CSS transition on `#main-content` (e.g. `opacity` fade) using HTMX's `htmx:beforeSwap` / `htmx:afterSwap` events or the `htmx-swapping` CSS class
- Test this during implementation; if no flash is observed with `outerHTML`, no extra work is needed

### 3g. Handle heading in HTMX content responses

When returning HTMX content, include the heading in the main content HTML so it appears correctly after swap. Currently `heading` is rendered by the template block — for HTMX responses, prepend the heading `<h1>` to the rendered content string in `panel_framework_view`.

### 3h. Ensure full page loads still work

No changes needed for full page loads — the existing code path renders the complete template. Verify that direct URL visits and browser refresh render correctly with breadcrumbs and active sidebar state.

### 3i. Write tests

- Unit tests: verify HTMX responses include OOB fragments when `HX-Request` and `HX-Target=main-content`
- Unit tests: verify non-HTMX responses render full page
- E2E tests (Playwright): verify navigation works, URL updates, back/forward works

## Task 4: Sidebar Instance Dropdown

**Skills:** `fls:alpine-js`, `fls:template`, `fls:htmx`, `fls:icon-usage`, `fls:frontend-styling`, `fls:testing`

### 4a. Enrich `_build_menu_items` with instance data

**File:** `freedom_ls/panel_framework/views.py`

Extend `_build_menu_items` (or create a new function) to include instance info when viewing an instance page:

```python
{
    "label": conf.menu_label,
    "url": reverse(...),
    "active": conf.url_name == active_section,
    "expanded": conf.url_name == active_section and instance is not None,
    "instance_label": str(instance) if instance and conf.url_name == active_section else "",
    "instance_url": request.path if instance and conf.url_name == active_section else "",
}
```

Pass the resolved instance to `_build_menu_items`.

### 4b. Update sidebar template with disclosure pattern

**File:** `freedom_ls/panel_framework/templates/panel_framework/partials/sidebar_nav.html`

Implement W3C disclosure pattern for each menu item. The toggle button is **always** rendered (not conditionally) with `aria-expanded` toggling between `"true"` and `"false"`. The nested `<ul>` is always in the DOM, shown/hidden via Alpine `x-show` + `x-collapse`:

```html
{% for item in menu_items %}
<li x-data="{ expanded: {{ item.expanded|yesno:'true,false' }} }">
    <div class="flex items-center justify-between">
        <a hx-get="{{ item.url }}" hx-target="#main-content" hx-push-url="true"
           hx-swap="outerHTML"
           {% if item.active and not item.expanded %}aria-current="page"{% endif %}
           class="...">
            {{ item.label }}
        </a>
        <button x-on:click="expanded = !expanded"
                x-on:keydown.escape="expanded = false"
                x-bind:aria-expanded="expanded"
                class="p-1 ..."
                aria-label="Toggle {{ item.label }} submenu">
            <c-icon name="expand" class="size-4" x-show="!expanded" />
            <c-icon name="collapse" class="size-4" x-show="expanded" />
        </button>
    </div>
    <ul x-show="expanded" x-collapse class="ml-4 mt-1">
        {% if item.instance_label %}
        <li>
            <a href="{{ item.instance_url }}"
               aria-current="page"
               class="... font-semibold">
                {{ item.instance_label }}
            </a>
        </li>
        {% endif %}
    </ul>
</li>
{% endfor %}
```

This ensures the disclosure pattern works correctly: the button always exists with a valid `aria-expanded` value, and keyboard users can toggle with Enter/Space (native `<button>` behaviour) or close with Escape.

### 4c. Alpine.js component for expand/collapse animation

**File:** `freedom_ls/panel_framework/static/panel_framework/js/alpine-components.js`

The expand/collapse is handled inline with Alpine `x-data` on each `<li>` (see Task 4b template). The server sets the initial `expanded` state. Alpine handles:
- `x-collapse` animation on the nested `<ul>`
- `x-on:keydown.escape` to close the expanded section
- `x-bind:aria-expanded` to keep the button's accessibility state in sync

No separate `sidebarNav` component registration is needed — the inline `x-data="{ expanded: true/false }"` is sufficient and CSP-safe since Alpine evaluates data attributes.

### 4d. Styling

- Parent section when expanded: subtle active indicator (e.g. left border only, no background fill)
- Instance item when active: full active treatment (background + left border + semibold)
- This creates visual hierarchy: parent section "owns" the area, instance is the specific active page

### 4e. Write tests

- Unit tests: verify `_build_menu_items` returns `expanded: True` and instance data when on an instance page
- Unit tests: verify `expanded: False` on list pages
- Unit tests: verify OOB sidebar fragment contains instance dropdown when navigating to instance via HTMX

## Task 5: Integration & Polish

**Skills:** `fls:testing`, `fls:playwright-tests`, `fls:request-code-review`

### 5a. Verify all features work together

- Navigate via sidebar → content swaps, breadcrumbs update, sidebar highlights update, instance dropdown collapses
- Click data table row → content swaps, breadcrumbs show instance, sidebar expands with instance name
- Click breadcrumb link → navigates via HTMX, all state updates

- Empty state (no path selected — root page): breadcrumbs show only root, no sidebar item active
