# Performance Measurements — Specification

## Goal

Build an automated performance profiling system that a developer can run with a single command to get a clear picture of which endpoints are slow, have too many queries, or have N+1 problems. No manual tooling, no external dependencies — just pytest, Django's test client, and `CaptureQueriesContext`.

## Why This Matters

Performance problems in Django apps are almost always caused by the ORM layer — N+1 queries, missing `select_related`/`prefetch_related`, duplicate queries. These issues are invisible during normal development (pages still load, tests still pass) but degrade as data grows. An automated profiling system catches these problems early, before they reach users.

## Scope

### In scope

- Server-side profiling of student-facing and accounts endpoints
- Response time, query count, duplicate query detection, slow query detection
- Terminal report, JSON output, and Markdown report
- pytest-based, isolated from the normal test suite via marker

### Out of scope

- Educator interface (SPA-style catch-all route — limited profiling value)
- Production monitoring or always-on middleware
- Client-side performance (Core Web Vitals, LCP, etc.)
- Adopting django-auto-prefetch or any auto-fix tools
- Adding external profiling dependencies (django-silk, nplusone, etc.)

## Architecture

### Approach: pytest + Django test client + CaptureQueriesContext

**Why this approach:**
- Zero external dependencies — uses only Django builtins and pytest (already in the project)
- Full control over report format and thresholds
- Runs in the same environment as the rest of the test suite
- `CaptureQueriesContext` captures raw SQL and timing for each query, which is everything we need

**How it works:**
1. Profiling tests create realistic test data using existing factory_boy factories
2. Each test authenticates via `client.force_login()` and hits an endpoint inside a `CaptureQueriesContext`
3. Captured data (response time, queries, duplicates) is collected by a pytest plugin/fixture
4. At the end of the test session, results are sorted by severity and output as a terminal table, a JSON file, and a Markdown report

### Test location

All profiling tests live in `tests/profiling/`. This keeps them in a dedicated, easy-to-find location, separate from functional tests. The `tests/` directory is already in `testpaths` in pyproject.toml.

```
tests/
  profiling/
    conftest.py          # Profiling fixtures, data setup, report generation
    test_student_views.py  # Student interface endpoint profiling
    test_account_views.py  # Account endpoint profiling
```

### pytest marker

All profiling tests are marked with `@pytest.mark.profiling`. This marker must be registered in `pyproject.toml` and excluded from default test runs.

**Why a marker (not just directory separation):** Markers integrate with pytest's `-m` flag, making it trivial to include/exclude profiling tests. Directory separation alone would require `--ignore` flags which are fragile.

**Running profiling tests:**
```bash
uv run pytest -m profiling
```

**Excluding from normal test runs:** Update `addopts` in pyproject.toml:
```toml
addopts = "--strict-markers -m 'not profiling'"
```

This ensures `uv run pytest` never accidentally runs profiling tests (they're slower and require specific data setup).

## Performance Targets

A single set of thresholds. The developer interprets results based on endpoint complexity.

| Metric | Good | Warning | Bad |
|--------|------|---------|-----|
| Server response time | < 200ms | 200–500ms | > 500ms |
| DB query count | 1–10 | 10–20 | > 20 |
| Individual query time | < 10ms | 10–50ms | > 50ms |
| Duplicate queries | 0 | 1–2 | > 2 |

**Why these thresholds:**
- Response time targets align with TTFB best practices (< 200ms = "feels instant")
- Query count thresholds are informed by django-query-counter defaults and community consensus
- Duplicate query threshold of 0 is the ideal — any duplicates are a signal worth investigating
- These are guidelines, not hard pass/fail gates. The report flags issues; the developer decides what to fix

## Test Data

### Scale

Profiling fixtures create data at a scale designed to reveal N+1 problems:

| Entity | Count | Rationale |
|--------|-------|-----------|
| Courses | 5–10 | Enough to populate list views |
| Topics per course | 10–20 | Enough to trigger N+1 on topic loops |
| Course parts per course | 3–5 | Realistic course structure |
| Students (users) | 20–50 | Enough to exercise student-related queries |
| Course registrations | Proportional | Each student registered to 1–3 courses |
| Progress records | Proportional | Topic and form progress for registered students |

### Factory reuse

**Decision:** All test data is created using the existing factory_boy factories defined in the app conftest files (e.g. `CourseFactory`, `TopicFactory`, `UserFactory`, `CourseProgressFactory`, etc.).

**Why:** Factories already handle site-aware context, required fields, and model relationships. Duplicating this logic would be fragile and a maintenance burden.

A shared profiling fixture (e.g. `profiling_data`) in `tests/profiling/conftest.py` composes these factories to create the full dataset. This fixture should be session-scoped or module-scoped to avoid recreating data for every test.

## Report Output

Three output formats, all generated at the end of the pytest session:

### 1. Terminal output

A table printed to stdout showing all profiled endpoints sorted by severity (worst first). Colour-coded using ANSI codes: green for good, yellow for warning, red for bad.

Example:
```
Performance Profiling Report
=============================

Endpoint                              Time(ms)  Queries  Duplicates  Slow Queries  Status
------------------------------------  --------  -------  ----------  ------------  ------
GET /courses/<slug>/                     487ms       34           8             1  !! BAD
GET /courses/                            312ms       18           3             0  ! WARN
GET /courses/<slug>/1/                   189ms        7           1             0  ! WARN
GET /                                     45ms        3           0             0  OK
GET /accounts/profile/                    38ms        2           0             0  OK

Duplicate Queries Detail:
  GET /courses/<slug>/ — 8 duplicates:
    SELECT "content_engine_topic".* FROM "content_engine_topic" WHERE ... (×8)
```

### 2. JSON file

Written to `tests/profiling/results/profiling_results.json`. Machine-readable for CI comparison or custom tooling.

Structure:
```json
{
  "timestamp": "2026-03-10T14:30:00Z",
  "results": [
    {
      "endpoint": "GET /courses/<slug>/",
      "view_name": "student_interface:course_detail",
      "response_time_ms": 487,
      "status_code": 200,
      "query_count": 34,
      "duplicate_query_count": 8,
      "slow_query_count": 1,
      "total_db_time_ms": 120.5,
      "queries": [
        {
          "sql": "SELECT ...",
          "time_ms": 3.2,
          "is_duplicate": false,
          "is_slow": false
        }
      ],
      "duplicate_queries": [
        {
          "sql_fingerprint": "SELECT ... FROM content_engine_topic WHERE ...",
          "count": 8
        }
      ]
    }
  ]
}
```

### 3. Markdown file

Written to `tests/profiling/results/profiling_report.md`. Human-readable, renders nicely on GitHub or in any editor.

Contains the same information as the terminal output in Markdown table format, plus a duplicate queries detail section and a summary of thresholds used.

**Why three formats:**
- Terminal: immediate developer feedback
- JSON: machine-readable for diffing, CI integration, or future tooling
- Markdown: shareable, reviewable, can be committed or attached to PRs

## Duplicate Query Detection

### Approach: SQL fingerprinting

To detect duplicates, normalize captured SQL by stripping parameter values and comparing the query structure. Two queries with the same structure but different parameter values (e.g. `WHERE id = 1` vs `WHERE id = 2`) are the same fingerprint — this is the classic N+1 signal.

**Implementation:** Strip quoted strings and numeric literals from SQL, then compare. This is a simple regex-based approach that handles the common cases without adding a dependency on a SQL parsing library.

**Why not exact string matching:** Exact matching only catches truly identical queries (same parameters). N+1 queries usually have different parameters (one per object), so fingerprinting is needed.

## Endpoints to Profile

All standard Django view endpoints in the student interface and accounts apps:

### Student interface

| Endpoint | Auth Required | Notes |
|----------|--------------|-------|
| `GET /` | No | Home page |
| `GET /courses/` | No | Course listing |
| `GET /partials/courses/` | No | HTMX partial for course listing |
| `GET /courses/<slug>/` | No | Course detail |
| `POST /courses/<slug>/register/` | Yes | Course registration |
| `GET /courses/<slug>/<index>/` | Yes | Course content page (topic/activity) |
| `GET /courses/<slug>/<index>/start_form` | Yes | Start a form |
| `GET /courses/<slug>/<index>/fill_form/<page>/` | Yes | Fill form page |
| `GET /courses/<slug>/finish/` | Yes | Course completion page |

### Accounts

| Endpoint | Auth Required | Notes |
|----------|--------------|-------|
| `GET /accounts/profile/` | Yes | User profile page |

Each endpoint gets its own test function. This makes it easy to identify which endpoint has issues and to update individual tests when endpoints change.

## Success Criteria

1. **Single command execution:** `uv run pytest -m profiling` runs all profiling tests and produces all three report outputs
2. **No interference with normal tests:** `uv run pytest` (without `-m profiling`) does not run profiling tests
3. **Zero external dependencies:** No new packages added — only Django builtins, pytest, and existing project dependencies
4. **Realistic data volumes:** Test data is sufficient to reveal N+1 query patterns (scaling with the counts specified above)
5. **Actionable output:** The report clearly identifies the worst-performing endpoints and shows which queries are duplicated, making it obvious where to apply `select_related`/`prefetch_related`
6. **Factory reuse:** All test data created via existing factory_boy factories — no manual `Model.objects.create()` calls
7. **Easy to maintain:** Adding a new endpoint to profile means adding a single test function — no framework code changes needed
