# Performance Measurements — Implementation Plan

## Overview

Build a pytest-based performance profiling system that measures response time, query count, duplicate queries, and slow queries for student-facing and accounts endpoints. Output results as terminal table, JSON, and Markdown.

**Skills to use:** `testing` skill for all test implementation (TDD approach), `request-code-review` after completing each batch.

## Batch 2: Core Profiling Infrastructure

All code in `tests/profiling/conftest.py`. Use the `testing` skill with TDD.

### Task 2.1: SQL Fingerprinting

Write a function to normalize SQL for duplicate detection:

```python
def fingerprint_sql(sql: str) -> str:
    """Strip parameter values from SQL to create a structural fingerprint.

    Replaces quoted strings and numeric literals with placeholders.
    e.g. WHERE id = 42 AND name = 'foo' → WHERE id = %s AND name = %s
    """
```

Use regex to:
1. Replace single-quoted strings (`'...'`) with `%s`
2. Replace numeric literals (standalone integers/floats) with `%s`

TDD: Write tests for fingerprinting in a separate test file `tests/profiling/test_fingerprinting.py` (this file does NOT need the profiling marker). Test cases:
- Simple WHERE clause with integer
- WHERE clause with string
- Multiple parameters
- IN clauses with multiple values
- Queries with no parameters (should return unchanged)

### Task 2.2: Profiling Result Data Structure

Create a dataclass to hold per-endpoint results:

```python
@dataclasses.dataclass
class EndpointResult:
    endpoint: str                    # e.g. "GET /courses/"
    view_name: str                   # e.g. "student_interface:courses"
    response_time_ms: float
    status_code: int
    query_count: int
    duplicate_query_count: int
    slow_query_count: int
    total_db_time_ms: float
    queries: list[QueryInfo]
    duplicate_queries: list[DuplicateInfo]

@dataclasses.dataclass
class QueryInfo:
    sql: str
    time_ms: float
    is_duplicate: bool
    is_slow: bool

@dataclasses.dataclass
class DuplicateInfo:
    sql_fingerprint: str
    count: int
```

### Task 2.3: Profiling Context Manager / Helper

Create a helper that wraps Django's `CaptureQueriesContext` and produces an `EndpointResult`:

```python
def profile_endpoint(
    client: django.test.Client,
    method: str,
    path: str,
    view_name: str,
    **request_kwargs,
) -> EndpointResult:
    """Hit an endpoint inside CaptureQueriesContext, return profiling results."""
```

This function should:
1. Get the DB connection
2. Start timing
3. Enter `CaptureQueriesContext`
4. Make the request via `client.get()` or `client.post()`
5. Stop timing
6. Analyze captured queries: count, fingerprint for duplicates, check for slow queries (>10ms)
7. Return an `EndpointResult`

Use thresholds from the spec:
- Slow query: > 10ms
- Duplicate detection: group by SQL fingerprint, count > 1

TDD: Test `profile_endpoint` with a simple view that does known queries. Verify it captures query count, detects duplicates, measures time.

### Task 2.4: Results Collector (pytest plugin)

Create a pytest plugin/fixture that collects `EndpointResult` objects across all profiling tests and generates reports at session end.

Use `pytest_configure` to register a plugin, or use a session-scoped fixture with `request.config` stash to accumulate results.

```python
# Session-scoped collector
@pytest.fixture(scope="session")
def profiling_collector(request):
    """Collects EndpointResults across all profiling tests."""
    collector = ProfilingCollector()
    request.config._profiling_collector = collector
    yield collector
    # Generate reports at end of session
    collector.generate_reports()
```

Each test will call `profiling_collector.add(result)`.

Implement `pytest_sessionfinish` hook to trigger report generation (this is more reliable than fixture teardown for session scope).

### Task 3.1: Status Classification

```python
def classify_status(result: EndpointResult) -> str:
    """Return 'OK', 'WARN', or 'BAD' based on thresholds."""
```

Thresholds from spec:
| Metric | Good | Warning | Bad |
|--------|------|---------|-----|
| Response time | < 200ms | 200–500ms | > 500ms |
| Query count | ≤ 10 | 11–20 | > 20 |
| Query time | < 10ms | 10–50ms | > 50ms |
| Duplicate queries | 0 | 1–2 | > 2 |

Overall status = worst status across all metrics.

TDD: Test classification with edge cases at boundaries.
