# Spec: conformance upgrade-notes & plugin-doc tie-ins (test-portability 4)

> Split out of `fls-test-portability-part-2`. Bundles **Layer 5** (the
> `upgrade_notes.md` tie-in) and the **Part-2 portion of Layer 6** (plugin-doc
> touches). Design detail is in this folder's `2. plan.md`. Part 1 covered the
> bulk of Layer 6 (testing/playwright skills, code-reviewer, `update_fls.md` marker
> selection); this slice adds only what's listed here.

## Problem

The `concrete-implementation-helpers` spec shipped structured per-spec
`upgrade_notes.md`, but there is no established convention for surfacing a **hard
config requirement** (as `home_page` did with `COURSE_ACCESS_BACKEND`) to a
downstream on upgrade — so a required-settings change can land silently. Separately,
`commands/concrete/update_fls.md` currently runs neither `manage.py check` nor any
conformance suite, and `template_repo_manifest.md`'s `urls.py` checklist is out of
sync with `config/urls.py` (omits `applications/`, `interest/`, sitemap/robots). The
new conformance suite and system checks only "stick" if the plugin docs point at
them.

## Goal

Make the new conventions stick: an upgrade-notes convention that surfaces hard
config requirements, and the plugin-doc touches that invoke the conformance suite
and `manage.py check` during an upgrade.

### Success criteria

1. `commands/sdd/update_upgrade_notes.md` gains guidance to recognise a
   **hard/required** settings change (vs optional/informational): set
   `requires_settings_change: true` with the specific keys in `changed_settings`,
   and, when the spec also adds a Layer-4 check enforcing it, say so in the notes.
   **No new schema flag** (D6).
2. `commands/concrete/update_fls.md` gains verification steps that (a) invoke the
   conformance suite as the positive signal and (b) run `uv run python manage.py
   check` so system-check failures surface on upgrade. Any runserver step uses the
   documented port pattern (no hardcoded 8000).

## Layer 5 — tie into `upgrade_notes.md`

`concrete-implementation-helpers` shipped structured per-spec `upgrade_notes.md`.
The authoritative schema is in `commands/sdd/update_upgrade_notes.md` (richer than
that spec's prose — it includes `changed_settings: []`, npm fields, etc.).

**Convention this slice establishes:** when a spec introduces a **hard config
requirement** (as `home_page` did with `COURSE_ACCESS_BACKEND`), its notes set
`requires_settings_change: true` with the specific keys in `changed_settings`, so
`update_fls` surfaces it — and ideally a Layer-4 check enforces the same requirement
at boot.

**Decision to record (D6):** `requires_settings_change: true` + `changed_settings`
is sufficient for the hard-requirement case (the field is free-text and already
carries the key list — Part 1's own notes prove it). No new flag is required.
`commands/sdd/update_upgrade_notes.md` gains guidance to recognise a *hard/required*
settings change (vs optional/informational) and, when the spec also adds a Layer-4
check enforcing it, to say so in the notes.

## Layer 6 — make the new conventions stick (Part-2 portion)

- **`commands/concrete/update_fls.md`:** it currently runs neither `manage.py check`
  nor any conformance suite. Add both to the verification steps — invoke the
  conformance suite (`test_portability_2_conformance_suite`) as the positive signal,
  and run `manage.py check` so Layer-4 failures
  (`test_portability_3_system_checks`) surface during an upgrade. (Part 1 already
  switched the bare `uv run pytest` call sites to the documented `-m "not playwright
  and not fls_internal and not ci_only"` selection.)
- **`commands/sdd/update_upgrade_notes.md`:** add the D6 guidance above.
- **`fls-claude-plugin/resources/template_repo_manifest.md`:** its `urls.py`
  checklist is out of sync with `config/urls.py`. The actual manifest edit is
  **`/update_template_repo`'s job (SDD step 12), NOT this slice** — but this spec
  **requires** that step to (a) fix the checklist and (b) ship the reference
  sitemap/robots wiring + a `tests/` dir importing the conformance suite, so new
  projects get the forced sitemap/robots existence (D1) and the positive signal by
  default. This spec only records the requirement; do not pre-empt step 12.

## Key decisions

- **D6 — Layer 5 needs no new upgrade-notes flag.** `requires_settings_change: true`
  + `changed_settings` (already a key list) suffices for the hard-requirement case.

## Scope / non-goals

- **In scope:** the guidance edit to `commands/sdd/update_upgrade_notes.md` and the
  verification-step edits to `commands/concrete/update_fls.md`.
- **Out of scope / deferred to `/update_template_repo` (SDD step 12):** the actual
  `template_repo_manifest.md` checklist edit and the reference sitemap/robots wiring
  + downstream `tests/` dir. Called out above so the two sides land together.

## Relationship to the other test-portability slices

- **`test_portability_2_conformance_suite` (Layer 3)** — the `update_fls.md` edit
  invokes that suite, so it should exist (or land together) for the instruction to
  be real.
- **`test_portability_3_system_checks` (Layer 4)** — the Layer-5 hard-requirement
  guidance points downstreams at `manage.py check`; it pairs with those checks.
- Independent of `test_portability_1_settings_config_convention` (Layer 0).
- Assumes Part 1 already switched the bare `uv run pytest` call sites to the
  documented marker selection.
