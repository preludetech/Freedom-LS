## Layer 5 — `update_upgrade_notes.md` guidance (D6)

Edit `fls-claude-plugin/commands/sdd/update_upgrade_notes.md`:

- Add guidance to recognise a **hard/required** settings change (vs
  optional/informational). When a spec introduces a hard config requirement, set
  `requires_settings_change: true` with the specific keys in `changed_settings`.
- When the spec also adds a Layer-4 check enforcing the requirement, say so in
  the notes (so `update_fls` can point the downstream at `manage.py check`).
- No new schema flag (D6): `requires_settings_change` + `changed_settings` is
  sufficient.

(Part 2's own `upgrade_notes.md` is authored later in the SDD flow, not here.
Part 2 requires no downstream settings change; it will note the new
`student_interface` checks and the availability of the conformance suite.)

## Layer 6 — plugin-doc touches (Part-2 portion)

- **`fls-claude-plugin/commands/concrete/update_fls.md`:** add to the
  verification steps (a) invoke the Layer-3 conformance suite as the positive
  signal, and (b) run `uv run python manage.py check` so Layer-4 failures surface
  during an upgrade. Use the documented port pattern for any runserver step (do
  not hardcode 8000). Part 1 already switched bare `pytest` call sites to the
  marker selection.
- **`fls-claude-plugin/resources/template_repo_manifest.md`:** the actual
  checklist edit (add `applications/`, `interest/`, sitemap/robots wiring, and a
  `tests/` dir importing the conformance suite) is `/update_template_repo`'s job
  (SDD step 12) — **not implemented here**. This plan records the requirement so
  step 12 lands it; do **not** pre-empt that step by editing the manifest now.
