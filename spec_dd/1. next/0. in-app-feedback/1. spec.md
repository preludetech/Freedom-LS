# In-App Feedback System — Specification

## Overview

FLS needs a configurable in-app feedback system that prompts users at natural pause points (e.g., after completing a course or topic). The system must be:

- **Site-aware**: Each site configures its own feedback forms
- **Extensible**: Downstream apps can add custom trigger points without modifying FLS code
- **Non-intrusive**: Anti-fatigue protections prevent survey fatigue

## Why This Matters

User feedback is essential for improving course content and the learning experience, but only if collected at the right moments without annoying users. A well-designed feedback system increases response rates while respecting the user's time.

## Models

### New App: `feedback`

All models are `SiteAwareModel` instances.

### `FeedbackForm`

Admin-configurable feedback form definition.

| Field | Type | Description |
|---|---|---|
| `name` | `CharField` | Internal name for admin reference |
| `trigger_point` | `CharField` | Named trigger (e.g., `"course_completed"`, `"topic_completed"`) |
| `is_active` | `BooleanField` | Whether this form is currently active (default `True`) |
| `rating_label` | `CharField` | Label for the rating question (e.g., "How would you rate this course?") |
| `text_prompt` | `CharField` | Prompt for the optional text field (e.g., "What could be improved?"), blank allowed |
| `thank_you_message` | `CharField` | Message shown after submission (e.g., "Thanks for your feedback!") |
| `min_occurrences` | `PositiveIntegerField` | Minimum trigger count before showing (default `1`). E.g., `3` means show only after the user's 3rd trigger of this type |
| `cooldown_days` | `PositiveIntegerField` | Minimum days between showing this form to the same user (default `0`) |

**Constraints:**
- A trigger point can have at most one active form per site at a time (enforced via unique constraint on `site` + `trigger_point` + `is_active` where `is_active=True`, or validated in admin)

**Why `trigger_point` is a free-text string:** This keeps the feedback app decoupled. FLS provides standard trigger points, but downstream apps can define their own without any code changes to the feedback app.

**Trigger point validation:** The `trigger_point` value must be validated against a registry of known trigger points. The admin interface should present the allowed trigger points as a dropdown/select rather than a free-text input. The registry should be extensible so downstream apps can register their own trigger points (e.g., via a simple setting or registry function). If a trigger point is not in the registry, form save should be rejected with a clear error message.

### `FeedbackTriggerLog`

Tracks how many times each trigger has fired for each user. Used for the `min_occurrences` threshold.

| Field | Type | Description |
|---|---|---|
| `user` | `ForeignKey(User)` | The user |
| `trigger_point` | `CharField` | The trigger point name |
| `count` | `PositiveIntegerField` | Number of times this trigger has fired for this user |

**Why a separate log:** This keeps the feedback app fully decoupled from progress models. The feedback app doesn't need to know about `TopicProgress` or `CourseProgress` — it maintains its own counters. This also works for custom trigger points from downstream apps.

**Constraints:**
- Unique together: `site` + `user` + `trigger_point`

### `FeedbackResponse`

Stores user feedback submissions.

| Field | Type | Description |
|---|---|---|
| `form` | `ForeignKey(FeedbackForm)` | The form that was responded to |
| `user` | `ForeignKey(User)` | The user who submitted |
| `content_type` | `ForeignKey(ContentType)` | Generic FK content type for context object |
| `object_id` | `PositiveIntegerField` | Generic FK object ID for context object |
| `content_object` | `GenericForeignKey` | The context object (e.g., a Course, Topic) |
| `rating` | `PositiveSmallIntegerField` | Star rating (1-5) |
| `comment` | `TextField` | Optional free-text comment (blank allowed) |
| `created_at` | `DateTimeField` | When the response was submitted |

**Why `GenericForeignKey` for context:** The feedback app shouldn't depend on specific models like `Course`. Using `GenericForeignKey` lets any model serve as the context object while still providing real object references, admin queryability, and proper foreign key semantics.

### `FeedbackDismissal`

Tracks when a user dismisses a feedback prompt.

| Field | Type | Description |
|---|---|---|
| `form` | `ForeignKey(FeedbackForm)` | The form that was dismissed |
| `user` | `ForeignKey(User)` | The user who dismissed |
| `content_type` | `ForeignKey(ContentType)` | Generic FK content type for context object |
| `object_id` | `PositiveIntegerField` | Generic FK object ID for context object |
| `content_object` | `GenericForeignKey` | The context object |
| `created_at` | `DateTimeField` | When the dismissal occurred |

## Trigger System

### Django Signal

A custom Django signal is the trigger mechanism.

```python
# freedom_ls/feedback/signals.py
import django.dispatch

feedback_trigger = django.dispatch.Signal()
# Sent with: sender=<trigger_point_string>, user=<User>, context_object=<model instance>
```

**Why Django signals (not the webhook pattern):** The feedback check must happen synchronously within the request — it needs to determine whether to show a modal and set session state or response headers. The existing webhook system is designed for async external delivery, which is a different concern. Signals are the standard Django mechanism for synchronous cross-cutting concerns.

### Standard Trigger Points

FLS fires the signal at these points:

| Trigger Point | Fired When | Context Object |
|---|---|---|
| `course_completed` | User completes a course | `Course` |
| `topic_completed` | User completes a topic | `Topic` |

### How Downstream Apps Add Custom Triggers

Downstream apps fire the same signal from their own views:

```python
from freedom_ls.feedback.signals import feedback_trigger

feedback_trigger.send(
    sender="custom_trigger_name",
    user=request.user,
    context_object=my_object,
)
```

No FLS code changes required.

### Signal Receiver Logic

When the signal fires, the receiver:

1. **Increment trigger log**: Update `FeedbackTriggerLog` count for this user + trigger point
2. **Find active form**: Look up `FeedbackForm` where `trigger_point` matches and `is_active=True`
3. **Check eligibility** (all must pass):
   - Trigger count >= `min_occurrences`
   - User has not already responded for this context object
   - User has not dismissed this form 3+ times
   - No feedback prompt shown in this session already
   - Cooldown period has elapsed (last response or dismissal for this form was `cooldown_days` ago or more)
4. **If eligible**: Store form ID and context object info in the session for the template to pick up

## UI & Frontend

### Modal Display

The feedback modal uses the existing cotton `<c-modal>` component. It appears after the page loads at a trigger point.

### Loading Mechanism

Two scenarios need handling:

**Full page loads** (e.g., user navigates to the course completion page):
- The signal receiver sets a session flag with the feedback form ID and context object info
- The template includes a feedback placeholder `<div>` that checks the session flag
- If a flag is present, HTMX lazy-loads the feedback form via `hx-get` with `hx-trigger="load"`

**HTMX partial responses** (e.g., marking a topic complete via HTMX):
- Middleware checks the session after each response
- If a feedback flag is present, add an `HX-Trigger` response header
- Client-side Alpine.js listens for the event and loads the feedback form

**Why this two-path approach:** FLS uses both full page navigations and HTMX partial updates. The feedback system must work seamlessly in both cases without requiring view authors to add feedback-specific code.

### Feedback Form UI

The modal contains:
- A **star rating** (1-5 stars) — universally understood, standard pattern
- An **optional text field** with the admin-configured prompt
- **Submit** and **Close** buttons
- The admin-configured rating label as the question text

### Form Submission

- Form submits via HTMX (`hx-post`)
- On success: the modal content swaps to show the thank-you message, then auto-closes after a brief delay
- On dismissal (close button, Escape, backdrop click): an HTMX request records the dismissal

### Session Management

- After showing a feedback prompt (whether responded to or dismissed), set a session flag to prevent showing another prompt in the same session
- Clear the "pending feedback" session data after the form is shown

## Anti-Fatigue Protections

These are critical — survey fatigue will undermine the entire system if not handled properly.

| Protection | Mechanism | Configurable? |
|---|---|---|
| **Response tracking** | Don't re-show a form to a user who has already responded for the same context object (GenericFK match) | No |
| **Dismissal tracking** | After 3 dismissals of the same form, stop showing it to that user permanently | No (fixed at 3) |
| **Session limit** | Never show more than one feedback prompt per session | No (hardcoded) |
| **Per-form cooldown** | Each form has a `cooldown_days` setting — minimum days between showing this form to the same user | Yes (per form) |

**Why fixed dismissal threshold (3):** Simplifies admin configuration. Three dismissals is a clear signal that the user isn't interested. Making this configurable adds complexity without meaningful benefit.

**Why session-based global limit:** Even if multiple triggers fire in one session, the user should only see one feedback prompt. This prevents stacking and keeps the experience unobtrusive.

## Admin Interface

### FeedbackForm Admin

- List display: name, trigger point, is_active, min_occurrences, cooldown_days
- Filters: is_active, trigger_point
- Uses `SiteAwareModelAdmin` (Unfold)

### FeedbackResponse Admin (read-only)

- List display: form name, user, rating, content object, created_at
- Filters: form, rating, created_at
- Search: user email
- Export capability (future consideration, not V1)

### FeedbackDismissal Admin (read-only)

- List display: form name, user, content object, created_at
- Minimal — mainly for debugging

## Demonstration

The system is demonstrated by triggering feedback after course completion:

1. Admin creates a `FeedbackForm` with `trigger_point="course_completed"`
2. The `course_finish` view fires the `feedback_trigger` signal
3. On the course completion page, the feedback modal loads via HTMX
4. User can rate and comment, or dismiss
5. Anti-fatigue rules apply on subsequent completions

## Success Criteria

- [ ] Feedback forms are configurable per-site via Django admin
- [ ] Modal appears after course completion (demonstration trigger)
- [ ] Star rating (1-5) and optional text field work correctly
- [ ] Modal can be dismissed via close button, Escape key, or backdrop click
- [ ] Thank-you message displays after successful submission
- [ ] Response tracking: form doesn't re-appear for the same user + context object
- [ ] Dismissal tracking: form stops appearing after 3 dismissals
- [ ] Session limit: only one feedback prompt per session
- [ ] Per-form cooldown respected
- [ ] Occurrence threshold (`min_occurrences`) works correctly
- [ ] Trigger log counts are maintained correctly
- [ ] Downstream apps can fire custom triggers via the signal
- [ ] Feedback system is fully decoupled from views (no view code changes needed beyond signal firing)
- [ ] Works for both full page loads and HTMX partial responses
- [ ] All models are site-aware
