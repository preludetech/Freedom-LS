# Privacy & Data Protection Compliance

FLS must comply with GDPR, POPIA (South Africa's Protection of Personal Information Act), and ISO 27001 data protection controls. This spec covers the compliance **features** that need to be built into the application - the code-level privacy capabilities that enable deployers and users to exercise their rights.

This is a companion to the security-audit spec, which covers hardening the existing codebase. This spec covers new functionality.

FLS serves minors (under 18 for POPIA, under 16 for GDPR), which triggers additional consent requirements.

## In scope

### Data subject rights (legally required)
- **Data export**: "Download my data" endpoint - exports all personal data as JSON (profile, course registrations, progress, form responses, cohort membership)
- **Account deletion**: Student-initiated deletion with 30-day grace period, then permanent deletion/anonymisation. Must cascade through all models correctly
- **Profile editing**: Students can view and correct their personal information
- **Processing restriction**: Flag to freeze a user's data from being used in reports/webhooks

### Consent management
- **Registration consent**: Explicit opt-in at registration, recording timestamp, policy version, IP
- **Consent withdrawal**: Settings page to withdraw consent, triggering account deactivation and deletion workflow
- **Consent records model**: Track what was consented to, when, which policy version
- **Parental/guardian consent**: Since FLS serves minors, implement a parental consent workflow (POPIA: under 18, GDPR: under 16)

### Webhook PII control
- **Configurable PII in webhooks**: `FLS_PRIVACY.WEBHOOK_INCLUDE_PII` setting, default False
- When False, webhooks send `user_id` only (no `user_email`)
- Deployers opt in to sending PII to external endpoints

### Privacy documentation
- **Privacy policy template**: Versioned, covers all POPIA Section 18 / GDPR Article 13 requirements
- **Data processing inventory**: Document all personal data fields, purposes, retention periods, legal bases
- **Data flow documentation**: How personal data moves through the system including webhook destinations

### Data retention
- **Retention management command**: `manage.py enforce_retention` that purges expired data
- **Configurable retention periods** via `FLS_PRIVACY` settings
- Support both hard deletion and anonymisation

### Audit logging
- Log educator/admin access to student data
- Log authentication events (login, logout, failed attempts)
- Log consent events (given, withdrawn)
- Log data export and deletion requests

### Cookie consent
- Cookie consent banner for non-essential cookies (analytics)
- Conditionally render analytics scripts based on consent
- Store consent in a strictly-necessary cookie

## Out of scope
- Breach notification tooling (document the process, don't build tooling yet)
- DPIA templates (static documentation, not application features)
- Encryption at rest for individual fields (evaluate separately)
- Educator interface access permissions (separate spec)
