# Content snapshots

Build a `content_snapshots` system that produces and stores immutable, point-in-time snapshots of content_engine objects. The system stands alone — it has no dependency on, or knowledge of, any particular consumer.

# Problem

A piece of content in this LMS is rarely "just a title". A `Form` is composed of one or more `FormPage`s, and each page interleaves `FormContent` blocks (markdown — instructional text, embedded diagrams, embedded video) with `FormQuestion`s. A `Topic`, `Course`, `CoursePart`, or `Activity` carries a full markdown body that may reference uploaded files via the `File` model.

Content changes over time. Educators edit forms, swap diagrams, rephrase questions, replace videos. Once content is edited, the previous version is gone — there is no record of what a learner actually saw on the day they interacted with it.

We need a way to capture the *full* state of a content object at a moment in time, including:

- The complete structural composition (e.g. all FormPages with their ordered children)
- The full markdown body of every content-bearing model (no truncation)
- The set of `File` references the markdown depends on, so the snapshot remains interpretable even if files are later renamed, replaced, or deleted

These snapshots must be **immutable** — they are a historical record, not a cache. Future tooling will need to ask "what did this Form look like on date X" and get a deterministic answer.

# Goal

Introduce a new `freedom_ls/content_snapshots/` app whose sole responsibility is producing and storing immutable snapshots of content_engine objects. The app exposes a small read/write API (e.g. "snapshot this object" / "fetch snapshot by id") and otherwise has no opinion on who calls it or why.

The app must not depend on any other freedom_ls app beyond `content_engine` (and `accounts`/`site_aware_models` where the base classes require it).

# Scope — content models covered

All content_engine content models that participate in a learner journey:

- `Course`
- `CoursePart`
- `Topic`
- `Activity`
- `Form` — including its child structure: `FormPage` → ordered `FormContent` + `FormQuestion`
- `FormContent` (markdown blocks within a page)
- `FormQuestion` (full body, not truncated; with options)

Embedded media references inside `MarkdownContent.content` (images, video, diagrams referenced via the `File` model) should be enumerated in the snapshot — at minimum the `File` IDs/paths the rendered markdown depends on, so the snapshot can be interpreted later even if files are renamed, replaced, or removed.

# Key decision — immutability and versioning

Snapshots are **immutable per content version**:

- A new snapshot row is created whenever the underlying content changes. Detection is by content hash over the relevant fields, or via a `post_save` signal that compares against the latest snapshot.
- A consumer references a specific `snapshot_id` that never changes meaning.
- The snapshot table is append-only. Old snapshots are retained for historical accuracy.

Main consequence: the snapshot table grows monotonically with content edits. The spec phase must address retention/garbage-collection policy — though the default position should be "never prune" until a concrete consumer needs otherwise.

# Public surface

The snapshot app should expose roughly:

- `take_snapshot(content_obj) -> Snapshot` — idempotent: if the current state already matches the latest snapshot for this object, return the existing one rather than creating a duplicate.
- `get_latest_snapshot(content_obj) -> Snapshot | None` — convenience for "what does this object look like right now".
- `get_snapshot(snapshot_id) -> Snapshot` — historical lookup.

Consumers of this API are out of scope for this spec. The app is built to be useful, not to satisfy any particular caller.

# Open questions for the spec phase

- **Storage shape**: one polymorphic table (generic FK + JSON body) vs. one table per content type vs. a JSON snapshot column on each content model. Trade-off is query ergonomics vs. schema sprawl.
- **Hashing strategy**: which fields participate in the content hash. A change to `FormPage.order` should produce a new Form snapshot. A change to `Form.updated_at` alone should not.
- **Trigger model**: snapshots created eagerly via signals on content save, vs. lazily when `take_snapshot` is called. Eager keeps snapshots aligned with reality but adds save-path overhead; lazy is cheaper but means the "latest" snapshot can lag.
- **Retention**: are old snapshots ever pruned, and on what trigger. Default to "never" unless there's a clear reason.
- **Backfill on first deploy**: snapshot all existing content eagerly on migration, or lazily on first call to `take_snapshot`.
- **Derived values**: do snapshots include computed values like a quiz's `max_score`, or only raw fields. Computed values are easier for consumers but couple snapshots to scoring logic.

# Success criteria for the eventual implementation

- A caller can invoke `take_snapshot(content_obj)` and receive an immutable snapshot reference.
- Form snapshots include all FormPages with their ordered FormContent blocks and FormQuestion bodies (full text, not truncated).
- Topic / Course / CoursePart / Activity snapshots include the full markdown body and a list of referenced `File` IDs.
- Editing content and calling `take_snapshot` again produces a new snapshot row; the prior snapshot is unaffected and still resolvable by id.
- Calling `take_snapshot` twice in a row without an intervening edit returns the same snapshot (idempotency).
- The `content_snapshots` app has no imports from apps other than `content_engine`, `accounts`, and `site_aware_models`.
