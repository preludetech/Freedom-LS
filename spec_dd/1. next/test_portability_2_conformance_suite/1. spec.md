# Spec: FLS conformance suite — `freedom_ls.contrib.conformance` (test-portability 2)

> Split out of `fls-test-portability-part-2`. **This is Layer 3** — the opt-in,
> importable conformance suite. Design detail is in this folder's `2. plan.md`
> (§ "Layer 3", T3.1–T3.8). Assumes Part 1 (marker taxonomy incl. `fls_internal`,
> `INSTALLED_APPS`-keyed collection-safety guards, de-branded assertions; **no
> `e2e` marker**) is already present.

## Problem

FLS ships as a package installed into concrete downstream projects (e.g. First
Class). "Did I wire FLS up correctly?" is the signal a downstream actually needs
from a test run, and today it **isn't a supported, dedicated thing at all**. The
canonical failure is the First Class integration of the `home_page` update:

- `student_interface` newly required `settings.COURSE_ACCESS_BACKEND`, which First
  Class had never set. FLS's loader (`get_course_access_backend()`,
  `freedom_ls/course_access/loader.py:23-39`) is `@functools.cache`d and does
  `import_string(settings.COURSE_ACCESS_BACKEND)` with **no fallback** — an unset
  key raises only on the *first real request*, not at `manage.py check` time.
- First Class's root `urls.py` was missing the `applications/` include.

Both would 500 the catalogue/dashboard at runtime while `manage.py check` passed
clean. They were caught only as a side effect of FLS's own tests being vacuumed up
by the downstream's `pytest` — the exact coupling Part 1 removed. Removing the
noise leaves this real signal absent. This slice supplies it.

This is not hypothetical drift:
`fls-claude-plugin/resources/template_repo_manifest.md`'s own `urls.py` checklist
(lines ~210-219) still omits `applications/`, `interest/`, and the sitemap/robots
wiring that `config/urls.py:51-73` actually ships. A downstream following that
checklist reproduces the opening bug verbatim.

## Goal

An importable pytest module the downstream runs, exercising the real integration
seams against *its own* settings, that catches the `COURSE_ACCESS_BACKEND` /
missing-include class of bug *before* runtime.

### Success criteria

1. **Under FLS's own `config.settings_dev`** (all apps installed, nothing pruned),
   `pytest --collect-only -q freedom_ls/contrib/conformance` lists at minimum the
   node IDs enumerated in "Acceptance node IDs" below. This is the full-install
   baseline the suite *offers*, not a per-downstream mandate: a downstream that
   removes an app or prunes an internal route legitimately collects fewer
   (criterion #7, D9).
2. The suite is **green under FLS's own `config.settings_dev`** (FLS is its own
   first consumer — `testpaths` already includes `freedom_ls`).
3. With `course_applications` installed, removing its `applications/` include makes
   `test_fls_namespace_reverses[course_applications:apply]` fail specifically (one
   named failure, not a collection abort).
4. A downstream missing sitemap/robots wiring fails
   `test_reference_url_reverses[sitemap]` / `[robots_txt]`.
5. The suite passes under the existing `--disable-socket --allow-hosts=127.0.0.1,::1`
   config with no DB required.
6. **Override-friendly at two granularities:** (a) a downstream that removes an FLS
   app from `INSTALLED_APPS` gets that app's probes **skipped**, not failed; (b) a
   downstream that *keeps* an app but customises an **internal**-tier route can
   `conformance.drop(...)` it and that probe **skips**, while **contract**-tier
   routes still hard-fail if they don't reverse (D9).

> Required-setting presence/value enforcement (unset `COURSE_ACCESS_BACKEND`,
> pinned `AUTH_USER_MODEL`) is deliberately **not** in this suite — it is a
> `manage.py check` job in `test_portability_3_system_checks` (D8). This suite
> keeps only the *behavioural* backend probe.

## The `freedom_ls.contrib.conformance` suite

A new, **opt-in importable module** at `freedom_ls/contrib/conformance/`. A
downstream references it from its own `tests/` dir. It is deliberately **not** a
`pytest11` entry-point plugin: an entry point on `sys.path` would auto-activate in
*every* downstream unconditionally, re-creating the coupling Part 1 removed. An
importable module makes opt-in explicit and greppable — matching how DRF
(`rest_framework.test`), django-cms (`CMSTestCase`), and Wagtail
(`wagtail.test.utils`) ship reusable test helpers.

### Override-friendliness (primary design constraint)

FLS is meant to be **extended and overridden**. A concrete project may replace or
discard whole FLS apps. The conformance suite must **verify only the seams a
downstream actually opted into, and never enforce rigidity on flexible surfaces.**

Mechanism, at **two granularities**:

- **Per app** — every probe is gated on the FLS app/feature it checks actually
  being installed, reusing Part 1's `INSTALLED_APPS`-keyed collection-safety
  pattern. When the relevant app is absent, its probes **skip** (not fail).
- **Per route, within a kept app** — namespace probes split into a **contract
  tier** (routes other FLS apps/backends reverse to — hard-required while the app
  is installed) and a prunable **internal tier** (app-only routes a downstream can
  `conformance.drop(...)` when it customises them). See D9 and the
  `FLS_NAMESPACE_PROBES` table in the plan.

The suite is a signal for *"the FLS pieces I chose to use are wired correctly"*,
never *"you must use all of FLS, unchanged."*

### Opt-in surface

The package `__init__.py` re-exports every conformance test function under an
explicit `__all__`. Downstreams opt in with either form (the second recommended
for collision safety):

```python
# tests/test_fls_conformance.py  (downstream) — simple form
from freedom_ls.contrib.conformance import *          # noqa: F401,F403

# ...or the collision-safe form (recommended in downstream docs)
from freedom_ls.contrib import conformance
test_fls_namespace_reverses = conformance.test_fls_namespace_reverses
```

`import *` works because pytest collects any module-level callable whose *name*
matches `python_functions` (default `test_*`), regardless of where it was defined;
parametrize marks travel with the function object.

### Hygiene rules (load-bearing — enforced in the module itself)

- **The package `__init__.py` defines `__all__`** listing only the intended test
  functions + `drop`, so `import *` is inert w.r.t. the module's own imports.
- **No non-test helper in the package is named `test_*`.** Private helpers use a
  `_`-prefix; if a public non-test callable is ever needed, set
  `helper.__test__ = False`.

### What it verifies

Reads the downstream's **real** config (zero `override_settings`) and calls FLS's
**own production resolution code** so it can't drift from runtime. All probes are
driven from small, explicitly-maintained data tables parametrized with
`pytest.mark.parametrize(..., ids=...)` so each seam surfaces as one
specifically-named failure, and "add a probe" is a documented step.

**Data tables (module-level, the maintained surface):**

- `FLS_NAMESPACE_PROBES` — per app namespace, split into two tiers (D9):
  - **`contract`** — routes FLS's *own* code reverses to from **another app** or a
    **pluggable backend** (established by grepping the codebase). **Hard-fail when
    their app is installed.** Current contract set:
    - `student_interface`: `course_home`, `initiate_course_access` (reversed by
      `course_access/backends.py`), `dashboard`, `course_detail` (reversed by
      `course_applications`);
    - `course_applications`: `apply`, `status`;
    - `course_interest`: `express_interest`;
    - `educator_interface`: `interface`;
    - `accounts`: `account_profile`, `legal_doc`.
  - **`internal`** — routes reversed **only** by their own app's templates
    (`student_interface:courses`, `form_start`, …; `course_interest:remove_interest`;
    `accounts:complete_registration`). Probed by default (D2) but
    `conformance.drop(...)`-able per route. Pruned entries **skip**, not fail.
- `REFERENCE_URL_NAMES` — the **required** downstream-replicated wiring:
  `"sitemap"`, `"robots_txt"` (unnamespaced project-root names — see D1).

**Probes** (each gated on its app being installed — skip when absent):

- `test_fls_namespace_reverses[<viewname>]` — each FLS route reverses, only for
  installed apps (and, for internal-tier entries, only those not pruned — D9).
  Contract-tier entries cannot be pruned.
- `test_reference_url_reverses[<name>]` — `sitemap` and `robots_txt` reverse.
  **Required** (D1): `NoReverseMatch` here is a hard conformance failure.
- `test_configured_backend_instantiates` — calls FLS's own
  `get_course_access_backend()` and asserts it returns a backend.
- `test_active_theme_resolves` — re-runs the **side-effect-free**
  `resolve_theme_dir(settings.FLS_THEME, settings.FLS_THEMES_DIRS)`. **Never call
  `configure_theme`** (D4).
- `test_active_icon_set_resolves` — `settings.FREEDOM_LS_ICON_SET in ICON_SETS`
  and a real `render_icon()` of a semantic name.
- `test_migration_state_consistent` — `call_command("makemigrations", check=True,
  dry_run=True)`. No DB connection, socket-safe (D5).

Tiny, fast, deterministic: stdlib + django + pytest only. No factories, demo
content, browser, or network.

### Packaging & placement

- `freedom_ls/contrib/conformance/` is auto-included by the existing
  `[tool.setuptools.packages.find] include = ["freedom_ls*"]` glob — **no
  `pyproject.toml` packaging change** needed.
- Because `testpaths` includes `freedom_ls`, FLS's own `uv run pytest` collects and
  runs these probes under `config.settings_dev` (criterion #2).
- **Probe-bearing submodules must be `test_*`-named** (e.g. `test_urls.py`,
  `test_settings.py`, `test_theme.py`) so they match FLS's `python_files`. A
  submodule named `urls.py` would **not** be collected and criterion #2 would pass
  vacuously.
- **Ruff:** add a per-file-ignore entry for `freedom_ls/contrib/conformance/**` (at
  minimum `S101`) — these ship as production code but are pytest-style test
  functions.

## Key decisions (Layer-3 relevant)

- **D1 — sitemap/robots are required downstream wiring, probed as a distinct
  category.** They live in project-root `config/` (excluded from the package),
  which only *demonstrates* the wiring. Per product decision a concrete project
  **must** have sitemaps and robots, so `test_reference_url_reverses[sitemap|
  robots_txt]` are required (hard failure if unwired), yet silenceable by a
  downstream that genuinely serves no public surface.
- **D2 — probe all FLS app namespaces**, bounded by D7/D9: "all" means all
  *installed* ones, contract routes required, internal routes prunable.
- **D4 — theme probe calls `resolve_theme_dir` (idempotent), never
  `configure_theme`** (which mutates settings in place).
- **D5 — migration probe uses `makemigrations --check --dry-run`** (no DB
  connection, socket-safe).
- **D7 — override-friendliness over completeness.** Probes gate on their app being
  installed and skip/no-op when absent. The sole enforced baseline is D1, itself
  silenceable. Two granularities: whole-app (`INSTALLED_APPS`) and within-a-kept-app
  route (D9).
- **D8 — required-setting presence/value is Layer 4 only, not a Layer-3 probe.**
  There is **no `test_required_setting_present`** here; Layer 3 keeps only the
  *behavioural* `test_configured_backend_instantiates`.
- **D9 — namespace probes split into contract vs internal tiers; internal routes
  are prunable.** The split is by **who depends on the route**: reversed by another
  FLS app/backend → contract (hard-fail while installed); reversed only by own-app
  templates → internal (prunable). Criterion #1's node IDs are the full-install
  baseline, never a per-downstream mandate.

## Acceptance node IDs

Full-install baseline under FLS's own `config.settings_dev` — the maximum the
suite offers, *not* a set every downstream must reproduce. `[C]` marks
contract-tier (hard-fail while their app is installed); the rest are internal-tier
(prunable):

```
test_fls_namespace_reverses[student_interface:dashboard]              # [C]
test_fls_namespace_reverses[student_interface:course_detail]          # [C]
test_fls_namespace_reverses[student_interface:course_home]            # [C]
test_fls_namespace_reverses[student_interface:initiate_course_access] # [C]
test_fls_namespace_reverses[student_interface:courses]
test_fls_namespace_reverses[course_applications:apply]                # [C]
test_fls_namespace_reverses[course_applications:status]               # [C]
test_fls_namespace_reverses[course_interest:express_interest]         # [C]
test_fls_namespace_reverses[educator_interface:interface]            # [C]
test_fls_namespace_reverses[accounts:account_profile]                 # [C]
test_fls_namespace_reverses[accounts:legal_doc]                       # [C]

```

## Testing strategy

- FLS's own suite runs the module under `config.settings_dev` (must be green). Add
  targeted negative meta-tests (style: `course_access/tests/test_checks.py`)
  proving each probe *fails* when its seam is broken — a bad URL table entry raises
  `NoReverseMatch`; `test_configured_backend_instantiates` raises when
  `COURSE_ACCESS_BACKEND` points at an unimportable path. These meta-tests are
  `fls_internal`-safe.
- **Override-friendliness (criterion #6):** two meta-tests. (a) With an FLS app
  removed from `INSTALLED_APPS`, its probes report `skipped` (not `failed`). (b)
  With the app kept but an internal-tier route `drop()`-ed, that probe reports
  `skipped` while a contract-tier route with a broken include still `failed`.

## Relationship to the other test-portability slices

- **`test_portability_3_system_checks` (Layer 4)** — complementary (D8): checks own
  static config-shape questions, this suite owns behavioural ones. Neither strictly
  requires the other.
- **`test_portability_4_upgrade_notes_and_docs` (Layers 5/6)** — the `update_fls.md`
  edit invokes this suite as the positive upgrade signal, so it assumes this slice
  has shipped.
- Independent of `test_portability_1_settings_config_convention` (Layer 0).
