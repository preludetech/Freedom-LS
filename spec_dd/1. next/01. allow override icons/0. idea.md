# Allow Override Icons

## Problem

Currently we have an icon component (`<c-icon />`) that fetches icons by name from a semantic registry (`base/icons.py`). It relies on Heroicons via `django-heroicons`.

Different applications that build on top of Freedom LS will have different visual styles and will therefore need different icons. Heroicons has a specific look and feel -- we need to support swapping to entirely different icon libraries (e.g., Lucide, Tabler, Phosphor, Material).

## Current Architecture (what we have right now)

- Templates use `<c-icon name="success" />` with semantic names
- `ICONS` dict in `base/icons.py` maps semantic names to Heroicon-specific names
- `cotton/icon.html` renders using `django-heroicons` template tags
- `force=True` escape hatch lets templates use raw Heroicon names directly

This semantic indirection layer is already the right architecture. The problem is that both the name mapping AND the rendering are hardcoded to Heroicons.

## Approach: Pluggable Icon Backend

Support swapping the entire icon rendering system, not just remapping names within Heroicons.

### How it works

1. **Ship pre-built mapping dicts** for a few popular icon sets (Heroicons as default, plus Lucide and Tabler). Each mapping translates our semantic names to that icon set's literal names.

2. **A Django setting** to select which icon set to use: `FREEDOM_LS_ICON_SET = "heroicons"` (default). Downstream developers change one setting to switch their entire icon style.

3. **A rendering layer** that knows how to turn an icon set + icon name into inline SVG. This needs to work without being locked to `django-heroicons` template tags. Options:
   - Use Iconify JSON data packages (`@iconify-json/*`) as the icon data source -- these contain SVG path data for 200+ icon sets in a standard format
   - Or install separate Django packages per supported icon set (e.g., `django-heroicons`, `django-lucide-icons`)

4. **Per-icon overrides** via `FREEDOM_LS_ICON_OVERRIDES = {"success": "some-other-icon"}` for fine-tuning individual icons without swapping the whole set.

5. **Full backend override** as an escape hatch: `FREEDOM_LS_ICON_BACKEND = "myproject.icons.MyBackend"` for downstream projects that want complete control over icon resolution and rendering.

### Key design decisions to make during spec

- Iconify JSON vs multiple Django packages for icon data?
- How to handle `force=True` usages when the icon set changes (these bypass the semantic layer and break on set swap)?
- Should we validate icon mappings at startup via Django system checks?
- How to handle icon variants (outline/solid/mini) across sets that have different variant systems?

## Target audience

Developers integrating FLS into their Django projects.
