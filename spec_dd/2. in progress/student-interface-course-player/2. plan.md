# Implementation Plan: Student Course Player

This plan turns `1. spec.md` into an ordered, TDD-friendly build. It is centred on
`student_interface` but also introduces a **shared `<dialog>` overlay foundation
in `base`** and migrates every existing sidebar onto it. Work proceeds in the
order below because later phases consume the shared mechanism built in the early
phases.

Follow project conventions throughout (type hints, `X | None`, ORM-only,
`select_related`/`prefetch_related`, `get_object_or_404`, snake_case URL names,
named brand classes — never raw hex, no new abstract base classes, no `Any`, no
`# type: ignore`, no new logging). We follow **TDD**: for each behavioural change
write a failing test first, then implement. Do **not** pre-write all tests here.

## Phase 1 — Shared `<dialog>` overlay foundation (in `base`)

> **As built — this shared shell was removed as dead code.** The `<c-overlay>`
> cotton shell (`base/templates/cotton/overlay.html`), the `overlayDialog` Alpine
> controller, and the `.overlay-*` CSS described in this phase were prototyped, but
> **nothing ended up consuming them**: the unified side panel (Phase 2) needs a
> dual desktop-docked / mobile-modal mode the always-`showModal()` shell can't
> model, so it drives its own `<dialog>` via the `sidePanel` controller; and the
> picture lightbox (Phase 3) stayed a bespoke `contentLightbox` overlay. The shell
> and its component/CSS were therefore **deleted** rather than left as dead code.
> The native-`<dialog>` accessibility this phase aimed for still ships — just
> realized directly in `sidePanel`. The 1.1/1.2 detail below is retained for
> historical context only.

**Goal (original):** one accessible overlay shell built on the native `<dialog>`
element, providing scrim/blur, background inertness, dismiss routes (scrim tap,
Escape, device Back), and focus management (focus in on open, return to trigger on
close) — intended for reuse by every consumer.

### 1.1 Overlay shell component (`base`)

Create a cotton shell, e.g. `base/templates/cotton/overlay.html`, that renders a
native `<dialog>` and a `::backdrop` scrim. The shell:

- Wraps a default slot for the consumer's foreground content.
- Accepts `c-vars` for: an `id`/label, a presentation variant
  (`bottom-sheet` | `side-drawer` | `lightbox`), and `aria-label`.
- Renders a visible close (×) `<button>` with an `aria-label`.
- Uses Tailwind transitions on `transform`/`opacity` (never `height`/`top`) for
  the open/close animation; per-variant slide direction (up for bottom-sheet,
  in-from-side for side-drawer, fade/scale-centre for lightbox) chosen by the
  variant class. `overscroll-behavior: contain` on the scrollable region.

### 1.2 Shared Alpine controller (`base/static/base/js/alpine-components.js`)

Register **one** new `Alpine.data("overlayDialog", …)` component (CSP build —
registered via `Alpine.data()`, no inline expressions). Responsibilities:

- Hold the `<dialog>` ref; `open()` calls `dialog.showModal()` (gives focus
  trap, inertness, Escape, `::backdrop` for free); `close()` calls
  `dialog.close()`.
- Wire `aria-expanded` on the external trigger.
- On `<dialog>`'s native `close` event (covers Escape **and** device Back, since
  `showModal()` participates in the close-watcher / history), restore focus to
  the trigger.
- Scrim tap → close (listen for clicks on the backdrop / `::backdrop` region).
- Optional swipe-down for the bottom-sheet variant is **polish only**; ×,
  Escape, scrim tap, and Back must work without it.

Pseudocode:

```js
Alpine.data("overlayDialog", () => ({
    triggerEl: null,
    init() {
        this.dialog = this.$refs.dialog;            // the <dialog>
        this.dialog.addEventListener("close", () => {
            this.$dispatch("overlay-closed");        // sync aria-expanded
            if (this.triggerEl) this.triggerEl.focus();
        });
        // scrim/backdrop click closes
        this.dialog.addEventListener("click", (e) => {
            if (e.target === this.dialog) this.close();  // click landed on backdrop
        });
    },
    open(triggerEl) { this.triggerEl = triggerEl; this.dialog.showModal(); },
    close() { this.dialog.close(); },
}));
```

> Note: a **modal** `<dialog>` (`showModal()`) inerts the page, which is correct
> for the mobile/overlay presentations but **wrong** for a desktop *docked*
> sidebar — see Phase 2 for how the docked desktop case avoids `showModal()`.

## Phase 2 — Unified side panel; retire `sidebarComponent`

**Goal:** replace the legacy Alpine `sidebarComponent` in `_base_interface.html`
with a single shared side-panel that all three host interfaces use. Desktop =
persistent docked column (NOT `showModal()`-ed). Below `lg` = a modal overlay
presented as a **bottom sheet for every interface** (the player TOC, educator nav,
and panel_framework nav all use the shared bottom-sheet default).

### 2.1 New side-panel controller

Add `Alpine.data("sidePanel", …)` to `base/static/base/js/alpine-components.js`.
It absorbs the responsibilities `sidebarComponent` had **plus** delegates the
mobile overlay to the Phase 1 dialog:

- Reads `data-storage-key` (preserve existing keys: `sidebar-course-toc`,
  `sidebar-educator`, `sidebar-test`) and persists desktop open/closed state to
  `localStorage` exactly as today.
- `matchMedia("(min-width: 1024px)")` to switch between **docked** (desktop) and
  **overlay** (mobile) modes, mirroring the current `_mq` logic.
- Desktop: docked column shown/hidden inline (no `<dialog>`), open-state
  persisted. Mobile: opening calls the embedded overlay dialog's `showModal()`
  (Phase 1) so scrim/inertness/focus/Escape/Back come for free.
- `aria-expanded` on the toggle; focus returns to the toggle on close (handled by
  Phase 1's `close` listener).

Reuse as much of the existing `sidebarComponent` body as possible (the
`--sidebar-top`, `matchMedia`, `localStorage` watch); do not write a parallel
implementation from scratch.

## Phase 4 — Resume redirect; retire the course start page

**Goal:** the bare course URL `/courses/{slug}/` (name `course_home`) becomes a
redirector, never a rendered start page.

### 4.0 Explicit `last_accessed_item` field on `CourseProgress`

Store the resume target **explicitly** on `CourseProgress` (the one row per
`(user, course)`) rather than deriving it from per-item progress timestamps.
Course items are polymorphic (`Topic | Form`), so use a `GenericForeignKey`,
mirroring `CohortDeadline` exactly (`student_management/models.py:134-141`):

```python
# CourseProgress, in student_progress/models.py
last_accessed_content_type = models.ForeignKey(
    DjangoContentType, on_delete=models.SET_NULL, null=True, blank=True,
    related_name="+",
)
last_accessed_object_id = models.UUIDField(null=True, blank=True)
last_accessed_item = GenericForeignKey(
    "last_accessed_content_type", "last_accessed_object_id"
)
```

- `null=True/blank=True`: existing rows (and freshly-registered, 0-progress
  learners) start with no last-accessed item → resume falls back to item 1.
- `on_delete=SET_NULL` on the content-type FK so deleting a Topic/Form model
  type can't cascade-delete progress rows; a dangling `object_id` simply resolves
  to `None` and falls back to item 1.
- `student_progress/models.py` already imports `DjangoContentType` (line 4); add
  `from django.contrib.contenttypes.fields import GenericForeignKey` (it does not
  import it yet — `student_management/models.py:5` shows the exact line).
- `last_accessed_object_id` is a `UUIDField` because every content model inherits
  `SiteAwareModel` whose PK is `id = models.UUIDField(primary_key=True, …)`
  (`site_aware_models/models.py:79`), exactly as `CohortDeadline.object_id`
  already assumes.
- Run `uv run manage.py makemigrations student_progress` then `migrate`. This
  **does** add one migration (no data migration needed — the field is nullable).

### 4.1 Resume-index lookup (`student_interface/utils.py`)

Add a single, testable helper `get_resume_index(user, course) -> int`:

- Returns the **1-based index in `course.viewable_items()`** to resume at.
- Read `CourseProgress.last_accessed_item` for `(user, course)`. If absent (no
  row, or the field is null), return `1` (first item / true start).
- Otherwise map the stored item to its index using a `{(type, pk): index}` map
  built once from `viewable_items()`. If the stored item is no longer viewable
  (deleted, unpublished, or fell out of the course), fall back to `1`.

```python
def get_resume_index(user, course) -> int:
    progress = (CourseProgress.objects
        .filter(user=user, course=course)
        .select_related("last_accessed_content_type")
        .first())
    if progress is None or progress.last_accessed_item is None:
        return 1
    item = progress.last_accessed_item
    index_by_key = {
        (type(i), i.pk): n
        for n, i in enumerate(course.viewable_items(), start=1)
    }
    return index_by_key.get((type(item), item.pk), 1)
```

One row + one FK resolve — no per-table scans, no Python `max()`.

### 4.2 Rewrite the `course_home` view as a redirector

Repurpose the existing `course_home` view (keep the URL name to avoid touching
every `reverse` — but it now **redirects**, never renders):

- Add `@login_required` (today it is unauthenticated). Anonymous users get the
  login flow as a side effect of `login_required`, satisfying "anonymous → login".
- `course = get_object_or_404(Course, slug=course_slug)`.
- If **not enrolled** (`get_is_registered` is False): `redirect(
  "student_interface:course_preview", course_slug=course_slug)` — never leaks
  per-user state, never enters the player.
- If enrolled: `index = get_resume_index(request.user, course)`;
  `redirect("student_interface:view_course_item", course_slug=..., index=index)`.
  302 to a **different** canonical URL (Back-button-safe; see 4.5).
- Delete the `course_home.html` template.

### 4.3 `register_for_course` → player

Change its final redirect from `course_home` to land **in the player**. Simplest
loop-safe choice: redirect to `course_home` is still fine *because* `course_home`
is now itself a resume redirector that lands on item 1 for a freshly registered
(0-progress) learner. Implement by leaving
`redirect("student_interface:course_home", …)` — it now resolves to the first
item. (Confirm with a test asserting the **final** landing URL is the item URL.)

### 4.4 `_preview_start_url` fallback

The `has_items=False` branch currently returns the `course_home` URL — now a
redirector, which is acceptable (it will bounce to preview for a course with no
items / unenrolled). Leave the registered+has_items branch pointing at item 1 as
today. No change required beyond confirming behaviour with a test; update the
docstring to reflect that `course_home` is now a redirector.

### 4.5 Deadline-locked redirect (loop fix)

In `view_course_item`, the deadline-locked branch currently
`redirect(... course_home ...)`. Because `course_home` now resumes back to the
same (locked) item, that would infinite-loop. Change it to
`redirect("student_interface:course_preview", course_slug=course_slug)` — the
existing loop-free preview page (per spec Migration/cleanup).

### 4.6 Record the last-accessed item at the single write point

`view_course_item` already fetches-and-saves `CourseProgress` on **every** item
view, for both topics and forms, *before* delegating to `view_topic`/`view_form`
(`views.py:278-282`). Extend that exact block to also stamp the GFK:

```python
if request.user.is_authenticated:
    course_progress, _ = CourseProgress.objects.get_or_create(
        user=request.user, course=course
    )
    course_progress.last_accessed_item = current_item
    course_progress.save()  # auto_now still bumps last_accessed_time
```

- This fires identically for Topic and Form items, so the old asymmetry
  (`view_topic` touched `TopicProgress` but `view_form` did not) **no longer
  matters for resume** — there is nothing to fix in `view_form`, and we never
  fabricate a spurious `FormProgress` on a bare GET.
- The deadline-locked branch returns at `views.py:273` **before** this block, so
  a locked item is never recorded as the resume target — correct, and it sidesteps
  the loop concern (see 4.5 for the redirect-target fix that still applies).
- `view_course_item` is the canonical player route; setting the field here makes
  it the single source of truth for "where the student was last".

### 4.7 Update inbound `course_home` template references

These currently point at the (now-removed) start page. Repoint per spec — none
may render a start page, and none may point the bare URL back at itself in a way
that loops:

- `course_topic.html` (Home / Finish buttons, 2 occurrences) → the breadcrumb
  course-root crumb already points at item 1; for the end-of-course "Home"/
  "Finish Course" buttons, point at **item 1** (`view_course_item` index=1) or
  the `course_finish` page as appropriate. (Finish-course button already submits
  the form; the "Home"/"Finish Course" anchor variants in the
  `is_course_complete` branch should go to item 1, never `course_home`.)
- `course_form_complete.html` ("Return to Course") → item 1

- `partials/course_card_registered.html` (registered-card link fallback when
  `next_up_url` is empty) → **kept at `course_home`**. As built, the dashboard

- `apis.py` line 78 is a comment only — leave it, but it is harmless.

> Per spec Section 1, the **player** must never link back to the bare course URL —

> card is exempt (above). Keeping the bare URL as a 302 to a *different* URL with

## Phase 5 — Item→parent-part helper + current-item threading

### 5.1 Item→parent-part helper (`student_interface/utils.py`)

Add `get_item_part(course, current_item) -> CoursePart | None`:

- Walks `course.children()` once; for each `CoursePart`, checks whether
  `current_item` is among its `children()`. Returns the containing `CoursePart`
  or `None`.
- Must reuse the already-loaded `children()` traversal and avoid N+1 (the
  caller — `view_course_item` — already has `course`; this is one in-memory walk
  over data the index build also uses). Used by the breadcrumb, the mobile

### 5.2 Thread a current-item marker through `get_course_index`

- Extend `get_course_index` / `create_child_dict_with_flattened_index` to accept

  current item) and set `child_dict["is_current"] = True` on the matching item
  dict (and expose the containing part so the template can auto-expand it).
- The flattened `global_index` already tracks each item's index, so marking is a
  cheap equality check during the existing walk — no extra queries.

### 5.3 Provide context from `view_course_item`

`view_course_item` (and the `view_topic`/`view_form` it delegates to) must put
into context: the course index with `current_index` marked, `current_part`
(from `get_item_part`), the `CourseProgress` (for the header progress bar/%),
the current item title, and the 1-based `index`. Use `select_related` where
fetching `CourseProgress`.

> `current_index`. Pass it as a query param on the `hx-get` URL (the player page

> `get_course_index`. Alternatively render the TOC inline server-side on the
> player page (preferred — avoids a second request and a race with the highlight;
> decide at implementation time, but inline render is simpler and keeps the
> highlight in the first paint).

### 6.3 Breadcrumbs (all breakpoints) — `{% block breadcrumbs %}`

`_base_interface.html` already exposes a `breadcrumbs` block. Add a player
breadcrumb partial (`partials/player_breadcrumbs.html`) rendered there. **As
built, the same breadcrumb renders at every breakpoint** (not desktop-only) — see
6.4 for the mobile augmentation:

- `<nav aria-label="Breadcrumb"><ol>…</ol></nav>`.
- Crumbs: course root → **first item** (`view_course_item` index=1, a concrete
  stable URL, never `course_home`); optional course-part → its first viewable
  item; current item as **plain text** with `aria-current="page"` (not a link).
- Drop the part crumb entirely when `current_part` is `None` (no dangling
  separator).
- `>` separators are **CSS** (`::before`/`::after` on `<li>`), not text/list
  items.
- Truncate long course/item titles with ellipsis (Tailwind `truncate`/
  `max-w-*`), keep full text for AT via `title` attribute.

### 6.4 Mobile header augmentation (unified, not a separate compact bar)

**As built**, the header chrome is unified across breakpoints rather than a
phone-only compact bar. The shared `_base_interface.html` content header renders
the same breadcrumb trail at every breakpoint, plus mobile-only additions:

- **No back arrow.** Backward navigation is covered by the course-root crumb, the
  per-item "Previous" button, and the browser Back button.
- The **same** breadcrumb trail as desktop (no middle-truncated condensed
  variant); long crumbs ellipsis-truncate at the ends via `max-width`.
- A **labelled** TOC toggle (icon **+** `aria-label` + `aria-expanded`) sits
  **beside the breadcrumbs**, `lg:hidden`, opening the bottom sheet.
- A thin progress bar with `%` (same `progress_percentage` source) rendered in the
  `_course_base.html` `header_extra` block **below the breadcrumbs, mobile-only**
  (`lg:hidden`).
- **No bookmark icon** (out of scope).

### 6.5 Bottom-sheet presentation for the player TOC

`_course_base.html` uses the base `bottom-sheet` default (Phase 2.2), so below
`lg` the TOC opens as a modal bottom sheet:

- Scrim + inert background (from `<dialog>.showModal()` in the `sidePanel`
  controller).
- Partial height; TOC list scrolls **inside** the sheet
  (`overscroll-behavior: contain`).
- `transform`/`opacity` slide-up animation.
- Dismiss via scrim tap, Escape, device Back; optional swipe-down polish.
  **No visible × control** (the panel carries no chrome of its own).
- Focus moves into the sheet on open, returns to the toggle on close.

### 6.6 Per-item browser title

- Switch player templates from the non-functional `{% block title %}` to the
  real **`head_title`** block.
- Title: `{% block head_title %}{{ item_title }} — {% if current_part %}{{ current_part.title }} — {% endif %}{{ course.title }} — {{ site_name }}{% endblock %}`.
- Most-specific first; part is between item and course and is the **first
  droppable** element on length concerns; en-dash `—` separator; `site_name`
  from the existing context processor.
- Applies to `course_topic.html`, `course_form.html`,
  `course_form_complete.html`, `course_form_page.html` — each supplies its own
  item title (topic/form title, "Form Complete", page title, etc.).
- Server-rendered full-page nav satisfies "updates on navigation" today; the
  requirement is documented for any future HTMX item swap.

### 6.7 Desktop layout

- Panel always open at `lg`+ (the side-panel desktop mode), **no close control**.
  As built, `data-desktop-lock` is set for **every** interface, so all three
  (player, educator, panels) are always-docked on desktop with no collapse/toggle
  — not just the player.
- Content area keeps a comfortable max-width (reuse `c-page` width handling
  already used in the templates).
- Focus on navigation: move focus to the new content `<h1>` on load (small Alpine
  `init` focusing the heading, `tabindex="-1"` on the `<h1>`).

conventions: test **behaviour, not CSS classes / styling**. Suggested coverage
