# Frontend QA: Student Course Player

Manual / Playwright-MCP browser checks for the course player rework. These verify
the feature end-to-end in a real browser — covering the resume redirect, breadcrumbs,
the desktop TOC, the mobile bottom sheet, the page title, and the unified-sidebar
migration across all three host interfaces.

## Setup

Use the **DemoDev** site for all test data (per project QA convention).

1. Find a free port and start the dev server:
   - `PORT=$(.claude/fls/scripts/find_available_port.sh)`
   - `uv run python manage.py runserver $PORT`
   - Base URL: `http://127.0.0.1:$PORT`
2. If demo data is missing, it will be created by the `qa-data-helper` agent
   during `/do_qa`. The relevant accounts (password = the email address):
   - Student: `demodev_s1@email.com`
   - (Demo courses live under `demo_content/`, e.g.
     `functionality-demo-course-parts` — a course **with** course parts — and
     `functionality-demo-end-with-topic` — a course that ends on a topic.)
3. Log in at `http://127.0.0.1:$PORT/accounts/login/` as `demodev_s1@email.com`.

Throughout: prefer a course that **has course parts** (e.g.
`functionality-demo-course-parts`) so breadcrumbs and the TOC disclosure are
exercised, and one **without** parts at the top level to confirm the part crumb
is dropped.

---

## Test 1 — Resume redirect on the bare course URL (enrolled, no progress)

1. As `demodev_s1`, ensure you are enrolled but have **no progress** on a course
   (use a fresh course, or reset by picking one you have not opened).
2. Visit the bare course URL: `http://127.0.0.1:$PORT/courses/{slug}/`.
3. **Expect:** a 302 redirect landing on `/courses/{slug}/1/` (the first item) —
   the player, not an old start page. The URL bar shows the per-item URL.
4. **Expect:** no "course home / start page" is shown at any point.

## Test 2 — Resume redirect (enrolled, with progress)

1. Open a course and navigate forward a few items (e.g. to item 3). The
   last-accessed item is recorded simply by **viewing** the item page (it is
   stamped on `CourseProgress.last_accessed_item` for both topics and forms — you
   do not need to start a form for it to count).
2. Navigate away (e.g. back to the dashboard).
3. Visit the bare course URL `http://127.0.0.1:$PORT/courses/{slug}/` again.
4. **Expect:** 302 to the **last-accessed** item (the one you were last on),
   not item 1. The TOC highlights that item.

## Test 3 — Unenrolled / anonymous visitor is kept out of the player

1. Log out. Visit a bare course URL `http://127.0.0.1:$PORT/courses/{slug}/`.
   **Expect:** redirected to login (never into the player).
2. Log in as a user **not enrolled** in some course, then visit that course's
   bare URL. **Expect:** redirected to the **preview** page
   `/courses/{slug}/preview/` (description / overview / register CTA). No
   per-user player state is shown.

## Test 4 — Back-button safety (no redirect loop)

1. As an enrolled learner with progress, visit the bare course URL and let it
   resolve to an item.
2. Click through to the next item (so history = item N → item N+1).
3. Press the browser **Back** button.
4. **Expect:** you land on the **previously viewed item**, not bounced forward
   again, and there is no redirect loop and no skipped history. Pressing Back
   again should continue sanely through item history (never re-triggering the
   bare-URL redirect).

## Test 5 — Desktop breadcrumbs

On a laptop-width viewport (≥ `lg`, e.g. 1280×800), open an item **inside a
course part**:

1. **Expect** at the top of the content area a breadcrumb trail:
   `Course > Part > Current item`.
2. The **course** and **part** crumbs are links; clicking the course crumb goes
   to item 1 (`/courses/{slug}/1/`), never a start page.
3. The **current item** crumb is plain text (not a link).
4. Open an item that has **no course part**. **Expect** only `Course > Item`
   (the part crumb is gone — no dangling `>` separator).
5. Open an item with a very long title. **Expect** it truncates with an ellipsis
   rather than wrapping or breaking the layout.

## Test 6 — Desktop TOC panel (always open)

At ≥ `lg`:

1. **Expect** the left TOC panel is always visible, with **no close control**.
2. **Header (pinned):** eyebrow `COURSE OUTLINE`, the course title, a progress
   bar, and `{X}% complete`. There is **no `N / M` counter**. Scroll the TOC
   body — the header stays pinned while the list scrolls.
3. Complete an item (mark a topic complete / pass a quiz), then reload. **Expect**
   the progress bar and the `%` both move together (same value).
4. The **current item** is visually highlighted (accent), and its containing
   course part is auto-expanded with the item scrolled into view.
5. Manually collapse a *different* part, then navigate to another item.
   **Expect** that manually-collapsed part stays collapsed (manual choice
   preserved); only the current item's part auto-expands.
6. The panel and the content area scroll **independently**.
7. With a screen-reader or the accessibility inspector, each status icon
   (complete / current / locked / retry) has accessible text (not colour-only),
   and the current item exposes `aria-current="page"`.

## Test 7 — Page title

1. Open a topic item. **Expect** the browser tab title reads
   `{Item} — {Course} — DemoDev` (en-dash separators, item first). If the item
   is inside a part, the part may appear between item and course.
2. Navigate to a different item. **Expect** the title updates to the new item.
3. Open a form / form-complete page. **Expect** a descriptive, item-specific
   title (no longer the old non-rendering title).

## Test 8 — Mobile header + bottom sheet

Resize to a phone viewport (e.g. 390×844) and open an item:

1. **Expect** the **same breadcrumb trail** as desktop (the header is unified
   across breakpoints — there is **no** separate compact bar, **no back arrow**,
   and **no** middle-truncated condensed trail). Long crumbs ellipsis-truncate at
   the ends. A clearly **labelled** TOC toggle sits **beside the breadcrumbs**,
   and a **thin progress bar with `%`** appears **below the breadcrumbs**. **No
   bookmark icon.**
2. The TOC panel is **collapsed by default**.
3. Tap the TOC toggle. **Expect** a **bottom sheet** that rises from the bottom
   (not a side drawer), with a scrim dimming the background and the background
   **inert** (you cannot scroll/tap the content behind it). It opens to a partial
   height; the TOC list scrolls **inside** the sheet without scrolling the page
   behind it.
4. Verify the dismiss routes close the sheet with the same outcome. There is
   **no visible × control** by design — the panel has no chrome of its own:
   - **tapping the scrim**,
   - the **Escape** key,
   - the device/browser **Back** button.
   (Swipe-down is optional polish; the three above must all work.)
5. **Focus:** on open, focus moves into the sheet; on close, focus returns to the
   TOC toggle.
6. Tap an item in the sheet. **Expect** navigation to that item and the sheet
   closes.

## Test 9 — Picture lightbox (shared overlay foundation)

1. Open a course item that contains an image/diagram (a `picture` component —
   `functionality-demo-standard-markdown` is a good candidate).
2. Click the **"Open image"** button. **Expect** it opens as a **centred
   lightbox** (not a bottom sheet) with a backdrop **blur**. Note: this lightbox
   is a focus-managed `x-show` overlay (`contentLightbox`), **not** the native
   `<dialog>` foundation, so the background is **not fully inert** — that is the
   intended current behaviour.
3. Dismiss via the visible **× close** button, by **clicking outside** the image,
   and via **Escape**. **Expect** each closes it and focus returns to the
   "Open image" trigger.

## Test 10 — Unified sidebar: no regressions in the other interfaces

The shared side-panel now backs all three interfaces with the **same** bottom-sheet
mechanism. Confirm:

1. **Educator interface** (`/educator/...` — log in as a user with educator
   access if needed): at desktop the nav sidebar is **docked and always open**
   (the unified panel is locked open on desktop — there is **no** desktop
   collapse/toggle control; this is the intentional unified-shell behaviour, a
   deliberate change from the old collapsible sidebar). At phone width the nav
   opens as a **bottom sheet** (rises from the bottom — same as the player, not a
   side drawer) with scrim, inert background, and dismiss via scrim tap, Escape,
   and Back (**no × control**).
2. **panel_framework test interface** (used in tests): same — docked/always-open
   above `lg`, bottom sheet below `lg`.
3. **Student course player**: confirms the same **bottom-sheet** presentation
   (Test 8).
4. Confirm focus returns to the relevant toggle on close in each interface.

---

## Pass criteria

All ten tests behave as described:
- bare URL resolves correctly for enrolled (first item / resume) and is kept off
  the player for unenrolled/anonymous;
- no Back-button loop;
- breadcrumbs correct (part dropped when absent, current item non-linked,
  truncation);
- desktop TOC always-open with pinned single-source-of-truth progress header,
  current-item highlight + auto-expand + preserved manual collapse, accessible
  status text, independent scroll;
- page title is item-first and updates on navigation;
- mobile unified header (same breadcrumb, toggle beside it, thin progress bar
  below — no back arrow, no bookmark) + bottom sheet with
  scrim/inert/partial-height/internal scroll and the three dismiss routes (scrim
  tap, Escape, Back — no ×) + focus management;
- picture lightbox is a focus-managed `contentLightbox` overlay (× / click-outside
  / Escape, focus restored; not the native-`<dialog>` foundation, so not fully
  inert);
- all three interfaces (player, educator, panel_framework) use the **same**
  bottom-sheet mechanism on mobile and are always-docked on desktop (no desktop
  collapse / no × control) — the unified-shell behaviour.
