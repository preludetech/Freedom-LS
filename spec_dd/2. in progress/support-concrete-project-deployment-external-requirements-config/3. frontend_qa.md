# Frontend QA: External requirements config (PostHog snippet + `sentry-debug/`)

Browser verification of the two user-visible surfaces this feature changes:
1. The PostHog client snippet in `_base.html` (renders only when a key is set; uses the configured
   `api_host`, not a hardcoded region).
2. The staff-gated `/sentry-debug/` endpoint (fires a 500 for staff, denied for everyone else).

The R2 media changes are **not** locally verifiable — dev uses `FileSystemStorage` (no bucket), so
signed-URL behaviour is a deployment-time check against a real R2 bucket, out of scope for this local
QA.

## Setup

Dev credentials (staff/superuser): `demodev@email.com` / `demodev@email.com`.

Pick a random port and note the base URL:

```
PORT=$(.claude/fls/scripts/find_available_port.sh)
echo "Base URL: http://127.0.0.1:$PORT"
```

Two server configurations are used below. Start/stop the server between them so the env change takes
effect.

- **Config A (PostHog enabled):**
  ```
  POSTHOG_API_KEY=phc_qa_test_key POSTHOG_API_HOST=https://us.i.posthog.com \
    uv run python manage.py runserver $PORT
  ```
- **Config B (PostHog disabled — no key):**
  ```
  uv run python manage.py runserver $PORT
  ```

Base URL for all steps: `http://127.0.0.1:$PORT`.

---

## Test 1 — PostHog snippet renders with the configured host (key set)

**Server:** Config A.

1. Visit `http://127.0.0.1:$PORT/` (any page rendered from `_base.html` — the login/landing page is
   fine; no auth needed).
2. Open **View Source** (or DevTools → Elements) and find the `posthog.init(...)` script in the
   `<head>`.

**Expect:**
- The `<script>` block **is present**.
- `posthog.init('phc_qa_test_key', { api_host: 'https://us.i.posthog.com', ... })` — the key matches
  `POSTHOG_API_KEY` and `api_host` matches `POSTHOG_API_HOST` (i.e. **not** the old hardcoded
  `https://eu.i.posthog.com`).
3. Restart Config A with `POSTHOG_API_HOST=https://eu.i.posthog.com`, reload, and confirm `api_host`
   now shows the **EU** host — proving the host is driven by config, not hardcoded.
4. (Optional, DevTools → Network) After the page loads, confirm a request goes out to a
   `*.i.posthog.com` host matching the configured region. With a fake key no real project receives
   data, but the beacon attempt to the configured host confirms wiring.

## Test 2 — PostHog snippet absent when the key is unset

**Server:** Config B (no `POSTHOG_API_KEY`).

1. Visit `http://127.0.0.1:$PORT/` and open **View Source**.

**Expect:**
- **No** `posthog.init(...)` script block anywhere in the page (the `{% if posthog_api_key %}` guard
  renders nothing). This is the dev/unconfigured default.

## Test 3 — `/sentry-debug/` fires for a staff user

**Server:** either config (the gate and the 500 don't depend on PostHog; Config B is fine).

1. Log in at `http://127.0.0.1:$PORT/accounts/login/` as `demodev@email.com` /
   `demodev@email.com` (superuser ⇒ passes the staff gate).
2. Visit `http://127.0.0.1:$PORT/sentry-debug/`.

**Expect:**
- The request raises `ZeroDivisionError` (`division by zero`). In dev (`DEBUG=True`) this shows
  Django's yellow debug traceback page with `ZeroDivisionError` at `trigger_error` in
  `freedom_ls/deployment/views.py`. This is the intended behaviour — it confirms the view executes
  for staff (in a real deployment this 500 is what Sentry captures).

---

## Test 4 — `/sentry-debug/` is denied for anonymous users

**Server:** either config.

1. Log out (visit `http://127.0.0.1:$PORT/accounts/logout/` or clear the session).
2. Visit `http://127.0.0.1:$PORT/sentry-debug/` while **not** logged in.

**Expect:**
- **No** `ZeroDivisionError` / 500. Instead a redirect to the admin login page
  (`staff_member_required` behaviour) — i.e. you land on a login form, not the error traceback.

---

## Test 5 — `/sentry-debug/` is denied for a logged-in non-staff user

**Server:** either config. Requires a **non-staff** account (e.g. a demo student). If none exists,
the `fls:qa-data-helper` can create a plain (non-staff, non-superuser) user on the DemoDev site.

1. Log in as the non-staff user.
2. Visit `http://127.0.0.1:$PORT/sentry-debug/`.

**Expect:**
- **No** 500 / `ZeroDivisionError`. `staff_member_required` redirects to the admin login page (the
  user is authenticated but not staff, so the endpoint stays gated) — confirming the endpoint is
  never an unauthenticated/non-staff guaranteed-500.
