# Frontend QA: Public browsing for unauthenticated users

Manual / browser QA for the feature. All checks are done in a browser against the dev
server. The dev settings pin `FORCE_SITE_NAME = "DemoDev"`, so the running site is
always **DemoDev** regardless of the port.

## Setup

### 1. Test data

Create the data on the **DemoDev** site (see `feedback_use_demodev_site`). The
`do_qa` flow auto-creates missing data via the `fls:qa-data-helper` agent; the commands
below are the canonical scenarios:

```bash
# Base demo site + a verified learner (demodev@email.com / demodev@email.com) + courses
uv run python manage.py create_demo_data

# A FREE course and an APPLICATION-GATED course on DemoDev, plus a learner
# enrolled in nothing with no existing application:
uv run python manage.py qa_create_course_access_types --site DemoDev
#   FREE course slug:  qa-free-course-access-types
#   GATED course slug: qa-application-gated-course-access-types
```

For the **new-user-with-registration-forms** test (Workflow 7) you also need DemoDev's
`SiteSignupPolicy` to require at least one `additional_registration_forms` entry, and a
fresh email address that has never signed up. Ask the `fls:qa-data-helper` to configure
an additional registration form for DemoDev if one is not already set.

Logins used below:
- **Existing learner:** `demodev@email.com` / `demodev@email.com` (verified, enrolled
  in nothing for the access-types courses).
- **New signups:** use a fresh, never-used email each time (e.g.
  `qa-newuser-1@email.com`); pick any password meeting the policy.

### 2. Run the dev server on a random port

```bash
PORT=$(.claude/fls/scripts/find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL for everything below: **`http://127.0.0.1:$PORT`** (substitute the actual
port). Build Tailwind first if styling looks wrong: `npm run tailwind_build`.

## Workflow 1 — Anonymous home page (`/`)

1. In a **logged-out** browser (use a private/incognito window, or log out first), go to
   `http://127.0.0.1:$PORT/`.
2. **Expect:** the page loads (HTTP 200) — you are **not** redirected to a login page.
3. **Expect to see:** an anonymous **hero** (value-prop headline + subtext) with a single
   "Browse all courses" CTA, and an **Available courses** section listing a few courses.
4. **Expect NOT to see:** a "Welcome back, …" greeting, an "In Progress" section, a
   "Learning History" section, a "Recommended Courses" section, or any "You haven't
   signed up… / sign in to see this" placeholder.
5. In the **header**, expect a **Login** button and (if signups are enabled) a **Sign up**
   button. Hover/inspect them: each `href` must carry `?next=/` (the current page path).
6. Click **Browse all courses** → you land on the catalogue (`/courses/`).

## Workflow 2 — Authenticated dashboard unchanged (regression)

1. Log in as `demodev@email.com` / `demodev@email.com` (go to the Login button → submit).
2. Go to `/`.
3. **Expect:** the normal dashboard — greeting ("Welcome back, …"), In Progress,
   Recommended (if any), Available, Learning History, and any backend panels — exactly as
   before. **Expect NOT to see** the anonymous hero.

## Workflow 3 — Anonymous catalogue + access badges (`/courses/`)

1. Logged out, go to `http://127.0.0.1:$PORT/courses/`.
2. **Expect:** the page loads (200, no login redirect) and lists every DemoDev course.
3. Find the **free** course (`qa-free-course-access-types`): its row shows a **"Free"**
   chip/badge.
4. Find the **gated** course (`qa-application-gated-course-access-types`): its row shows a
   **"By application"** chip/badge.
5. **Expect:** on not-registered rows the old **"Not registered"** eyebrow is **gone**
   for the anonymous viewer (the badge is the at-a-glance signal).
6. Click a course title → you land on that course's detail page.

## Workflow 4 — Anonymous course detail, free + gated (`/courses/<slug>/detail/`)

**Free course:**
1. Logged out, go to `http://127.0.0.1:$PORT/courses/qa-free-course-access-types/detail/`.
2. **Expect:** loads (200, no login redirect). Title, subtitle/category, the hero stats
   strip (Lessons / Level / Duration / Enrolment), "About this course", "What you'll
   learn", and the table of contents all render.
3. The stats strip shows the **Free** enrolment signal near the CTA.
4. The CTA button reads **"Enrol for free"**. Inspect its `href`: it points at
   `/courses/qa-free-course-access-types/access/` (not a `login?next=` link).
5. ToC items render as **blocked** (greyed / no clickable item links).

**Gated course (the §3 bug regression — must not crash):**
6. Logged out, go to
   `http://127.0.0.1:$PORT/courses/qa-application-gated-course-access-types/detail/`.
7. **Expect:** loads cleanly (200 — **no 500/error**). The CTA reads **"Apply now"** and
   its `href` points at the apply URL (`/applications/apply/qa-application-gated-course-access-types/`).
8. The enrolment signal near the CTA reads **"By application"**.

## Workflow 5 — Deferred login: free enrol completes intent

**Via existing-user login:**
1. Logged out, open the free course detail and click **"Enrol for free"**.
2. **Expect:** redirected to the login page with `?next=…/access/` in the URL.
3. Log in as `demodev@email.com` / `demodev@email.com`.
4. **Expect:** you land **inside the course** (first course item / course player),
   already enrolled — **no** intermediate "you need to enrol" page and no extra click.

**Via new signup:**
5. Repeat from a logged-out window, but this time choose **Sign up** on the auth page
   (the `next` must survive). Complete signup + email verification as the flow requires
   with a fresh email.
6. **Expect:** after authenticating you again land inside the course, enrolled.

## Workflow 6 — Deferred login: apply lands on the application form

**Via login and via signup:**
1. Logged out, open the gated course detail and click **"Apply now"**.
2. **Expect:** redirected to login with `?next=` pointing at the apply URL.
3. Authenticate (try once via **login** with `demodev@email.com`, once via **signup**
   with a fresh email).
4. **Expect:** you land on the **apply confirmation/form page** (authenticated), ready to
   submit — the application is **not** auto-submitted on your behalf.

## Workflow 7 — New-user signup that must complete registration forms (critical)

Precondition: DemoDev requires an additional registration form (see Setup §1).

1. Logged out, open the free course detail and click **"Enrol for free"** → you reach
   login with `?next=…/qa-free-course-access-types/access/`.
2. Choose **Sign up** and register with a brand-new email; verify the email if prompted.
3. **Expect:** you are redirected to the **Complete your registration** page (because the
   site requires extra forms). The URL should carry the original
   `next=…/qa-free-course-access-types/access/`.
4. Fill in and submit the registration form(s).
5. **Expect:** after submitting you land **inside the free course**, enrolled — the
   original intent survived the signup → complete-registration → destination chain. You
   are **not** dumped on the generic dashboard.

## Workflow 8 — Open-redirect rejection

1. Logged out, manually visit a login URL with an off-host `next`, e.g.
   `http://127.0.0.1:$PORT/accounts/login/?next=https://evil.example/`.
2. Log in as `demodev@email.com`.
3. **Expect:** you are **not** sent to `evil.example`. You land on the default
   post-login destination (`/`). The off-host `next` is ignored.

## Workflow 9 — SEO / discoverability

1. **Catalogue title + description:** view source of `/courses/` and confirm a
   descriptive `<title>` (with the `— DemoDev` suffix) and a `<meta name="description">`
   that is not the generic default.
2. **Detail title + description:** view source of the free course detail page; confirm
   `<title>` is the course title and `<meta name="description">` reflects the course
   description (or subtitle fallback).
3. **Course JSON-LD:** in the detail page source, find a JSON-LD script block; confirm it
   parses as JSON, has `"@type": "Course"`, and includes `name`, `url` (an absolute
   `http://127.0.0.1:$PORT/...` URL), and `isAccessibleForFree` (`true` for the free
   course, `false` for the gated one). Confirm `provider`/`image`/`author` are absent.
4. **Catalogue JSON-LD:** confirm the catalogue page source has a JSON-LD block with
   `"@type": "ItemList"` whose items carry the courses' absolute detail URLs.
5. **sitemap.xml:** go to `http://127.0.0.1:$PORT/sitemap.xml`. Expect 200, valid XML,
   containing the catalogue URL and a `<loc>` per course — all using the current host.
6. **robots.txt:** go to `http://127.0.0.1:$PORT/robots.txt`. Expect 200, `text/plain`,
   it does **not** `Disallow: /courses/`, and it references this host's `sitemap.xml`.

## Workflow 10 — Tenant isolation for anonymous visitors (spot check)

The dev server only serves DemoDev (`FORCE_SITE_NAME`), so a full cross-tenant check is
best left to the automated tests. As a spot check: confirm that the catalogue, the
`sitemap.xml`, and the JSON-LD URLs list **only** DemoDev courses (the slugs created in
Setup) and no courses belonging to any other site present in the database.
