# Implementation Plan: Course details polish + system-wide visibility/access overrides

This plan implements the two independent slices in `1. spec.md`. They share no
runtime dependency and can be built/reviewed separately, but land in adjacent
code so they share this worktree.

- **Problem 1** — a new `table_of_contents_in_development` course flag that hides
  the three TOC surfaces on the course detail page.
- **Problem 2** — two dev/staging-only global settings
  (`OVERRIDE_COURSE_VISIBILITY_TO_VISIBLE`, `OVERRIDE_COURSE_ACCESS_TO_FREE`)
  that let every course be previewed as fully visible and/or freely accessible.

Follow **TDD throughout**: for each task write the failing test first, then the
implementation. Do **not** pre-write every test up front — write them task by
task. Use the `fls:testing` skill for pytest conventions and the
`feedback_no_style_assertions` rule (test behaviour, never CSS classes).

# Problem 1 — `table_of_contents_in_development` course flag

### Task 1.1 — Schema field + validation rule (`content_engine/schema.py`)

Add the field to the pydantic `Course` model, alongside `visibility`
(follow the surrounding `Field(...)` style):

```python
table_of_contents_in_development: bool = Field(
    False,
    description="Hide the course's table-of-contents surfaces while it is being built.",
)
```

Add a new `@model_validator(mode="after")` sibling to `_validate_icon_fields`:

```python
@model_validator(mode="after")
def _validate_toc_in_development(self) -> "Course":
    # A published course must always show its contents.
    if self.table_of_contents_in_development and (
        self.visibility == CourseVisibility.PUBLISHED
    ):
        raise ValueError(
            f"table_of_contents_in_development must be false for a published "
            f"course (in {self.file_path})"
        )
    return self
```

Note: the model uses `use_enum_values=True`, so `self.visibility` is the string
value; `CourseVisibility` is a `StrEnum`, so `== CourseVisibility.PUBLISHED`
compares correctly. The default `visibility` is `PUBLISHED`, so a course that
omits `visibility` and sets the flag `true` correctly fails.

**TDD:** test that (a) a `published` course with the flag `true` raises
`ValueError` with the file path in the message, (b) a `coming_soon`/`hidden`
course with the flag `true` validates fine, (c) the field defaults to `False`.

### Task 1.2 — Model field + migration (`content_engine/models.py`)

Add to the Django `Course` model, next to `visibility`:

```python
table_of_contents_in_development = models.BooleanField(default=False)
```

Then `uv run manage.py makemigrations` (one `AddField`, mirroring
`0013_course_visibility.py`) and `uv run manage.py migrate`. **Never** edit an
existing migration.

**TDD:** a test asserting `content_save` succeeds for a valid (non-published)
course carrying the new field — i.e. schema⇄model reconciliation passes (no
`ValueError` from `save_with_uuid`).

Do **not** add a defence-in-depth re-check in `save_course()`. The spec marks it
optional and not required for correctness (no code path calls `save_course`
without schema validation for this pure intra-`Course` invariant); keep the
change minimal.

### Task 1.3 — Template: omit the three TOC surfaces (`course_detail.html`)

The template already exposes `course.table_of_contents_in_development`. Read it
directly (consistent with other `course.*` reads); no new view context needed.
Use the `fls:template`/`fls:frontend-styling` skills for markup conventions.

1. **Lessons stat card** (~line 92) — wrap in a guard, following the existing

   ```django
       {% with stat_icon="topic" stat_label="Lessons" stat_value=lesson_count_label %}
           {% partial stat-card %}

   ```

2. **"This course includes" panel** (~lines 149-166):
   - Wrap the whole `<div class="pt-2 border-t border-border">…</div>` panel so
     it renders only when it would have at least one item:
     `{% if not course.table_of_contents_in_development or includes_assessments %}`

     assessments the panel is empty → omit it entirely).
   - Inside, guard just the lesson-count `<li>` with

   - **Preserve the `@claude TODO` certificate comment** verbatim (CLAUDE.md).

3. **"Course content" section** (~lines 208-216) — wrap the entire

# Problem 2 — global visibility/access overrides (dev/staging only)

### Task 2.1 — Declare the two settings (`course_access/config.py`)

Add both as optional booleans (never `required=True`) with a dev/staging-only
inline comment. Use the `fls:app-settings` skill.

```python
class CourseAccessConfig(AppSettings):
    COURSE_ACCESS_BACKEND: str
    # DEV/STAGING ONLY — preview overrides. Never enable in production; a
    # DEBUG=False environment with either set True raises system check W001.
    OVERRIDE_COURSE_VISIBILITY_TO_VISIBLE: bool
    OVERRIDE_COURSE_ACCESS_TO_FREE: bool

    declared_settings = {
        "COURSE_ACCESS_BACKEND": Setting(required=True),
        "OVERRIDE_COURSE_VISIBILITY_TO_VISIBLE": Setting(default=False),
        "OVERRIDE_COURSE_ACCESS_TO_FREE": Setting(default=False),
    }
```

**TDD:** both default to `False` when the project sets neither; both read `True`
under `override_settings(...)`.

### Task 2.2 — Shared override-reading helpers (`course_access/overrides.py`, new)

Route every seam through one place so the ~half-dozen visibility sites cannot
drift (spec requirement). New small module:

```python
from __future__ import annotations

from typing import TYPE_CHECKING

from freedom_ls.course_access.config import config

if TYPE_CHECKING:
    from freedom_ls.content_engine.models import Course

def override_visibility_to_visible() -> bool:
    """True when every course should be treated as visible (dev/staging preview)."""
    return bool(config.OVERRIDE_COURSE_VISIBILITY_TO_VISIBLE)

def override_access_to_free() -> bool:
    """True when every course should be treated as freely accessible (dev/staging preview)."""
    return bool(config.OVERRIDE_COURSE_ACCESS_TO_FREE)
```

These are read **fresh on every call** (not baked into the cached backend), so
`get_course_access_backend`'s `@functools.cache` never needs invalidating and the
existing `cache_clear()` test contract holds.

For the cosmetic `coming_soon` reads, add one more helper so no view repeats the
`and not override_...` logic:

```python
def is_coming_soon_for_display(course: Course) -> bool:
    """Whether a course should present as coming-soon, honouring the visibility override."""
    from freedom_ls.content_engine.models import CourseVisibility
    return (
        course.visibility == CourseVisibility.COMING_SOON
        and not override_visibility_to_visible()
    )
```

**TDD:** each helper returns the expected boolean with the override off/on.

### Task 2.3 — Visibility override at the reachability seams

Wire the visibility override into every seam that special-cases
`coming_soon`/`hidden`. All reads go through `override_visibility_to_visible()`.

- **`backends.py::VisibilityEnforcingBackend.get_access`** — fold
  `and not override_visibility_to_visible()` into **both** the `COMING_SOON` and
  `HIDDEN` `if` conditions (matching `filter_visible`'s and Task 2.4's fold-into-
  the-condition style — do **not** add a separate early `return`, which would
  duplicate the existing fallthrough `return self._inner.get_access(...)` line).
  When the override is on, both intercepts are skipped and control falls through
  to the existing inner delegation (course behaves as published):
  ```python
  if (course.visibility == CourseVisibility.COMING_SOON
          and not is_registered_for_course(user, course)
          and not override_visibility_to_visible()):
      ...  # existing coming-soon decision
  if (course.visibility == CourseVisibility.HIDDEN
          and not is_registered_for_course(user, course)
          and not override_visibility_to_visible()):
      ...  # existing hidden decision
  # (access override, Task 2.5, slots in here) then:
  return self._inner.get_access(user=user, course=course)
  ```
- **`backends.py::VisibilityEnforcingBackend.filter_visible`** — when on, return
  `self._inner.filter_visible(...)` without the `HIDDEN` exclusion.
- **`visibility.py::raise_404_if_hidden_unregistered`** — when on, return early
  (do not raise `Http404`).
- **`student_interface/views.py::initiate_course_access`** — guard the
  `if course.visibility == CourseVisibility.COMING_SOON: return redirect(...)`
  short-circuit with `and not override_visibility_to_visible()` so a demoed
  `coming_soon` course flows through to the backend's free self-registration.

**TDD** (use `get_course_access_backend.cache_clear()` around
`override_settings`, per the loader caveat):
- `hidden` course detail returns 200 for an unregistered visitor when the
  override is on (404 when off).
- `hidden` courses appear in `filter_visible` output when on.
- `coming_soon` `get_access` returns the inner (free enrol) decision when on.
- `initiate_course_access` on a `coming_soon` course self-registers the learner
  when on (no bounce back to detail).

### Task 2.4 — Visibility override cosmetics ("look fully published")

Suppress the "Coming soon" badge / "I'm interested" CTA by routing every direct
`== CourseVisibility.COMING_SOON` cosmetic read through
`is_coming_soon_for_display(course)`:

- `student_interface/utils.py::get_course_listing` — both the anonymous branch
  (~line 732) and the authenticated branch (~line 755-756) `COMING_SOON`
  decisions.
- `student_interface/views.py::_annotate_recommendations` (~line 182).
- `student_interface/views.py::_available_courses` (~line 203-204).
- `student_interface/views.py::course_detail` — replace
  `is_coming_soon = course.visibility == CourseVisibility.COMING_SOON`
  (~line 353) with `is_coming_soon = is_coming_soon_for_display(course)`. This
  drives the template's `{% if is_coming_soon and not is_registered %}` branch,

**TDD:** with the override on, a `coming_soon` course shows no `COMING_SOON`
listing status / no express-interest affordance on detail, listings, dashboard
recommendations, and available-courses.

### Task 2.5 — Access override in the wrapper (`backends.py::VisibilityEnforcingBackend`)

Apply the access override in the **wrapper** (not `FreeOnlyCourseAccessBackend`),
so gated inner backends (e.g. `ApplicationCourseAccessBackend`, which overrides
these methods) are still forced to a free answer. Read via
`override_access_to_free()`:

- `is_accessible_for_free(course)` → `return True` when on (before delegating).
- `get_access_badge(course)` → when on, **call**
  `FreeOnlyCourseAccessBackend().get_access_badge(course=course)` (a real
  delegated call — safe, this method never touches `validate_course_config`).
  Do **not** inline `AccessBadge(label="Free")`: that literal already appears in
  `FreeOnlyCourseAccessBackend.get_access_badge` and
  `ApplicationCourseAccessBackend.get_access_badge`; a third copy is not DRY.
- `get_access(user, course)` → when on, return the canonical free decision.

  **Do NOT delegate `get_access` to a `FreeOnlyCourseAccessBackend` instance.**
  `FreeOnlyCourseAccessBackend.get_access` first calls
  `self.validate_course_config(course.access_config)`, and a bare
  `FreeOnlyCourseAccessBackend`'s `_ALLOWED_ACCESS_TYPES` is just `{"free"}`. For
  a gated course (`access_type: "application_gated"`) that raises `ValueError`
  and returns the safe *no-action* decision — the exact opposite of the override.

  Instead, **extract the free-decision branches** currently inside
  `FreeOnlyCourseAccessBackend.get_access` (registered → "Continue" /
  unregistered → "Enrol for free", built from the existing module constants
  `_FREE_ENROLMENT_SUMMARY` / `_FREE_ACQUISITION_HEADING` /
  `_FREE_ACQUISITION_SUBTEXT`) into a small shared helper in `backends.py`:

  ```python
  def _free_access_decision(*, user: RequestUser, course: Course) -> CourseAccessDecision:
      """Canonical free/open decision, independent of the course's real access_config."""
      if is_registered_for_course(user, course):
          return CourseAccessDecision(cta_label="Continue", ...)
      return CourseAccessDecision(cta_label="Enrol for free", ...)
  ```

  `FreeOnlyCourseAccessBackend.get_access` calls it **after** its own
  `validate_course_config` check (behaviour unchanged). The wrapper calls it
  **directly**, deliberately bypassing `validate_course_config` (the override is
  meant to ignore the course's real access-type). All reused symbols
  (`is_registered_for_course`, `reverse`, the `_FREE_*` constants) already live
  in `backends.py`, so no new imports.

Ordering inside `get_access` (both overrides layered): resolve visibility first
(Task 2.3 — possibly short-circuit coming_soon/hidden); then **if the access
override is on, `return _free_access_decision(user=user, course=course)`**; else
delegate to `self._inner.get_access(...)`. The free-decision helper **replaces**
the inner call when the access override is on — do not compute the inner decision
and then discard it. So both-on behaves as "visible + free".

**TDD:**
- With `OVERRIDE_COURSE_ACCESS_TO_FREE=True`, a gated (application) course reports
  `is_accessible_for_free=True`, a "Free" badge, and a `get_access` decision for
  an unregistered user with `can_self_register=True` / `is_accessible_for_free=True`.
- A learner already registered for a gated course gets `can_access_content=True`
  (the "Continue" branch of the free decision) with the override on.
- `Course.access_config` and `Course.visibility` are never written (assert the DB
  rows are unchanged after exercising the overrides — assert this for the
  visibility override in Task 2.3 as well).

### Task 2.6 — Production-safety system check (`course_access/checks.py`)

Add `W001` to the existing `checks.py`, modelled on
`accounts/checks.py::check_legal_docs_present_when_required`:

```python
@register(Tags.security)
def check_preview_overrides_disabled_in_production(
    app_configs: Sequence[AppConfig] | None, **kwargs: object
) -> list[Warning]:
    """W001: warn when a preview override is on while DEBUG is False."""
    from django.conf import settings
    from freedom_ls.course_access.config import config

    if settings.DEBUG:
        return []
    on = [
        name
        for name in ("OVERRIDE_COURSE_VISIBILITY_TO_VISIBLE", "OVERRIDE_COURSE_ACCESS_TO_FREE")
        if bool(getattr(config, name))
    ]
    if not on:
        return []
    return [
        Warning(
            f"Course preview override(s) enabled with DEBUG=False: {on}. "
            f"These are dev/staging only — unset them for this environment.",
            id="freedom_ls_course_access.W001",
        )
    ]
```

- Warning, not Error (must not block `check`/`migrate`/`test`/deploy).
- `DEBUG` read directly via `settings.DEBUG` (Django built-in; per the
  app-settings skill carve-out), never through `config.py`.
- Import `Tags`, `Warning`, `register` from `django.core.checks` (the module
  already imports `Error`, `register`).

**TDD:** fires (`W001`) when either override is `True` and `DEBUG=False`; silent
when `DEBUG=True`; silent when both overrides `False`. Use
`override_settings(DEBUG=False, OVERRIDE_...=True)` and call the check function.

### Task 2.7 — Document dev/staging-only in settings

Add only the inline dev/staging-only comment in `course_access/config.py`
(Task 2.1). Per spec, do **not** add either override to any shipped settings
module by default (`settings_base.py` / `settings_prod.py` must never set them;
`settings_dev.py` already keeps `DEBUG = True` but this plan adds nothing there).
Enabling an override is a per-environment choice made outside this change.
