# Implementation Plan: Themable Freedom LS

This plan turns `1. spec.md` into a TDD-friendly sequence. The work is largely a refactor (move templates, rename tokens) plus a small amount of new Python (theme resolution) and a new validation theme (`first_class`). Wherever possible, tasks below identify reusable existing functionality so we keep the code DRY.

---

## High-level ordering rationale

1. Build the new theme resolution layer **first**, with tests, but keep it inert (default theme not yet present). This isolates the new mechanics from the file moves.
2. Migrate the default theme into `freedom_ls/themes/default/` (templates, partials, `_base.html`, theme.css). Keep tokens **as they are today** during the move so we can prove migration in isolation from the rename.
3. Wire `tailwind.input.css` to the new layout and confirm a clean build.
4. Rename tokens in templates and component classes (the spec's biggest mechanical change). One PR-worthy commit per logical group.
5. Update `parse_tailwind_colors` callsite + email fallbacks to read from the resolved theme's `theme.css`.
6. Build the `first_class` canary theme (Tier 1 + Tier 2 + Tier 3).
7. Update resource docs (`templates_and_cotton.md`, `frontend_styling.md`) and write the new theming guide / install guide changes.

A canary theme that lights up every page proves the architecture before we cut docs and ship.

---

## Pre-existing functionality to reuse (DRY)

- `freedom_ls/accounts/email_utils.py::parse_tailwind_colors` — already parses `--color-*` from a CSS file. We will keep using it for email-template colour fallbacks; only the path it is called with changes (resolved theme's `theme.css` instead of the deleted `tailwind.components.css`). No need to write a new parser.
- Cotton's existing template loader chain (`django_cotton.cotton_loader.Loader` followed by `filesystem.Loader` and `app_directories.Loader`) — themes are surfaced purely by prepending a directory to `TEMPLATES[0]["DIRS"]`. No new loader logic, no cache-key plumbing.
- `STATICFILES_DIRS` — themes are surfaced via the same prepend pattern. No new static finder.
- The existing `find_available_port.sh` script for picking a runserver port when needed.
- `docs/how tos/incorperate into another project.md` — update in place, do not duplicate.

## Task list

### Task 1 — Theme resolution module (`freedom_ls/base/theming.py`)

**File to create:** `freedom_ls/base/theming.py`.

Build the resolver that the settings helper calls. Pure-Python, zero Django imports beyond `ImproperlyConfigured`, so it can be unit-tested without the test client.

**Public surface (suggested):**

```python
# freedom_ls/base/theming.py
from pathlib import Path
from django.core.exceptions import ImproperlyConfigured

FREEDOM_LS_PACKAGE_DIR: Path = Path(__file__).resolve().parent.parent  # freedom_ls/

def resolve_theme_dir(theme_slug: str, themes_dirs: list[Path]) -> Path:
    """Walk themes_dirs in order; return first <dir>/<theme_slug> that exists.

    Raises ImproperlyConfigured if nothing matches.
    """
    for parent in themes_dirs:
        candidate = Path(parent) / theme_slug
        if candidate.is_dir():
            return candidate
    raise ImproperlyConfigured(
        f"FLS theme {theme_slug!r} not found in any of: {themes_dirs!r}"
    )

def configure_theme(
    *,
    theme_slug: str,
    themes_dirs: list[Path],
    templates: list[dict],
    staticfiles_dirs: list,
) -> Path:
    """Resolve theme, prepend its templates/ dir to TEMPLATES[0]['DIRS']
    and its static/ dir to STATICFILES_DIRS. Returns the resolved theme dir.
    """
    resolved = resolve_theme_dir(theme_slug, themes_dirs)
    templates_dir = resolved / "templates"
    static_dir = resolved / "static"
    if templates_dir.is_dir():
        templates[0].setdefault("DIRS", [])
        templates[0]["DIRS"].insert(0, str(templates_dir))
    if static_dir.is_dir():
        staticfiles_dirs.insert(0, static_dir)
    return resolved
```

**Tests** (TDD; `freedom_ls/base/tests/test_theming.py`):

- `resolve_theme_dir` returns the first directory that exists when walking the dirs list.
- Downstream dir takes precedence over the FLS package dir (covers the "downstream theme named `default` shadows FLS one" case).
- Raises `ImproperlyConfigured` when no candidate exists — message includes the slug and the dirs searched.
- `configure_theme` prepends to both `TEMPLATES[0]['DIRS']` and `STATICFILES_DIRS` (use `tmp_path` to fake themes on disk).
- `configure_theme` is a no-op for the missing static / templates subdirs (theme that ships only templates is still valid).

Use `pytest` + `tmp_path` fixtures. Follow the conventions in `fls:testing` (`pytestmark = pytest.mark.django_db` is **not** needed here — pure filesystem logic).

Note: this task does not yet wire the resolver into `settings_base.py` — that happens in Task 4 once the default theme directory exists, so the dev server never loads with a missing theme.

### Task 3 — `tailwind.input.css` rewrite

Rewrite the FLS repo's own `tailwind.input.css` to the new shape — structurally the same as what we will document for downstream:

```css
@import "tailwindcss";

/* All FLS app templates (covers freedom_ls/themes/default/templates/). */
@source "./freedom_ls/**/templates/**/*.html";
/* Downstream theme's templates would normally be ./themes/<slug>/templates/.
   The FLS dev server uses the in-repo themes/. */

/* Default theme is always imported first; later themes / project overrides win
   per cascade order. */

    /* repo-level overrides go here, optional */
}
```

### Task 6 — Token rename + role contract finalisation

This is the biggest mechanical change. Do it in two commits to keep diffs reviewable:

**6a. Update `themes/default/static/themes/default/theme.css`** to the new role contract from spec §"Tier 1 — Design tokens":

- Wrap `--color-*` in `@theme inline { … }` (so utilities compile to `var(--color-…)` and follow the cascade — `research_tailwind_v4_theming.md` §2).
- Drop `-bold` tokens. Hover behaviour moves into `.btn-*` classes via `hover:brightness-110`.
- Rename `foreground` → `on-surface`, `danger` → `error`.
- Add `secondary`, `accent`, `info`, `on-secondary`, `on-accent`, `on-success`, `on-warning`, `on-error`, `on-info`, `muted` (kept as-is name), `focus-ring` (live alias of `--color-primary` via `var(--color-primary)`).
- Add the four shape tokens (`--fls-radius-sm/-md/-lg/-pill`) and the two type tokens (`--fls-font-sans`, `--fls-font-display`), both mapped via `@theme inline` so `rounded-md`, `font-sans`, `font-display` pick up theme values.
- Update Tier-2 component classes:
  - `.btn-primary`: drop `bg-primary-bold`, replace with `hover:brightness-110`.
  - `.btn-success`, `.btn-danger` → `.btn-success`, `.btn-error`. Use `text-on-success`, `text-on-error` instead of `text-on-primary`. Replace `*-bold` hovers with `hover:brightness-110`.
  - `.btn-outline` unchanged in semantics.
  - `.chip-danger` → `.chip-error`. `.chip-warning` uses `text-on-warning` (was `text-foreground`). `.chip-success` uses `text-success` (drop `-bold`).
  - Existing `@layer base` rules: `text-foreground` → `text-on-surface` everywhere.

**6b. Template-side rename pass.** Affects ~22 templates (grep result above). Mechanical:

| Old class | New class |
| --- | --- |
| `text-foreground` | `text-on-surface` |
| `bg-foreground` (if any) | `bg-on-surface` |
| `text-danger` | `text-error` |
| `bg-danger` | `bg-error` |
| `border-danger` | `border-error` |
| `chip-danger` | `chip-error` |
| `btn-danger` | `btn-error` |
| `text-danger-bold` | `text-error` |
| `text-success-bold` | `text-success` |
| `text-primary-bold` / `bg-primary-bold` | `text-primary` / `bg-primary` |
| `text-on-primary` (when paired with `bg-success` / `bg-danger`) | `text-on-success` / `text-on-error` accordingly |

A simple, easily-reviewable set of `sed`-style edits — but use `Edit` with `replace_all` so each file's diff is recorded individually. After the pass, grep again to verify no straggler references remain. Update `EMAIL_COLOR_FOREGROUND` → keep the variable name (it's an external email-template contract) but switch the lookup key:

```python

```

(Only the key changes; the public Python name stays so email templates keep rendering.)

**Tests** (TDD; piecemeal — tests for the rename are best expressed as targeted re-assertions of the existing email-utils tests):
