# Frontend QA: Themable Freedom LS

## What we're verifying

1. The default theme renders the FLS dev server **identically to before** the migration. No visual regression on any major page.
2. Switching `FLS_THEME=first_class` rebrands every page (Tier 1 tokens), changes button shape / chip flatness (Tier 2), and the overridden `header-button` component picks up the First Class display font (Tier 3).
3. Bad `FLS_THEME` values fail loudly at startup.

We use the **DemoDev** site for all test data — it has the demo content fixtures already loaded.

---

```bash
PORT=$(.claude/fls/scripts/find_available_port.sh)
```

Take note of `$PORT`; the base URL throughout this plan is `http://127.0.0.1:$PORT`.

## Test 1 — Default theme: smoke + visual regression

### 1.1 Build and start

```bash
npm run tailwind_build
uv run python manage.py runserver $PORT
```

Expected: server starts cleanly with no `ImproperlyConfigured` traceback.

### 1.2 Login

- Navigate to `http://127.0.0.1:$PORT/`.
- Use the DemoDev student credentials documented in the project's QA fixture (the `qa-data-helper` agent will surface them if not in scratch memory).

Expected: login lands on the student home / dashboard.

### 1.3 Walk the major pages and check each renders as before

For each URL below, take a screenshot via Playwright MCP (`browser_take_screenshot`) and compare visually against the pre-migration look. The intent is "no visible change":

- `/` — student home / course list.
- `/courses/` (or whichever path the all-courses view exposes; navigate via the "Courses" header link).
- A course home page — click into a course from the list.
- A course topic page — click any topic.
- A course form page — find a topic that is a form / activity.
- Educator interface (if your test user has educator role) — click the educator link in the user menu.
- Account pages — click the user-menu avatar, then "Account" / "Settings".
- Login (sign out, then revisit `/`).

**Things to verify on each page:**

- Header bar: brand colour matches the previous default (`#2B6CB0` blue), header button labels render in the existing system font.
- Buttons (primary, outline, success, error): shape (radius), colour, hover behaviour. `btn-error` is the new name for the old `btn-danger`; functionality must be unchanged.
- Chips on dashboards / progress panels: pill shape, primary / success / warning / error colours.
- Surfaces / cards: white-ish background, subtle border, rounded corners.
- Form pages: inputs styled with the default border + focus ring; validation states (deliberately submit a form with a missing required field) show an error border in the new `error` red — same hue as the old `danger`.
- Messages partial (e.g. after login) renders with correct colour.

Pay special attention to the templates whose tokens were renamed (anything formerly using `text-foreground`, `text-danger`, `text-on-primary` paired with success/error). They should look unchanged.

## Test 2 — `first_class` theme: end-to-end rebrand

### 2.1 Restart with the canary theme

Stop the server. Then:

```bash
FLS_THEME=first_class npm run tailwind_build
FLS_THEME=first_class uv run python manage.py runserver $PORT
```

Expected: server starts cleanly. Tailwind output regenerated with First Class tokens.

### 2.2 Walk the same pages

Visit the same URL list as Test 1.3 and check:

**Tier 1 (tokens) — should be visible everywhere:**

- Primary brand colour: deep indigo (`#283593`) instead of ocean blue. Header bar, primary CTAs, links — all indigo.
- Secondary colour: electric teal (`#00CEC9`) on any element using `bg-secondary` / `text-secondary` (likely `chip-*` variants if any wired to secondary).
- Accent: altitude orange (`#FF6B35`) — on whatever `bg-accent` / `text-accent` references exist.
- Surface: stratosphere off-white (`#F8F9FC`) — body background should look very faintly cooler than pure white.
- Body text: cockpit dark (`#1A1A2E`) — slightly bluer-black than the default.
- Border colour: lighter slate (`#E2E8F0`).
- Body font is **DM Sans** (visibly different from system sans). Headings (`h1`–`h4`) are **Outfit**.
- Radii are bumped one step — cards and inputs read slightly rounder.

**Tier 2 (component-class re-opens):**

- Primary buttons: rounder corners (`rounded-2xl` not `rounded-md`) and a heavier shadow than before. Confirm by visiting any page with a `btn-primary` (login, course CTAs).
- Chips: flat, no shadow. Compare to Test 1.3 chip rendering.

**Tier 3 (template override):**

- Header buttons (the cotton `header-button.html` override): label uses the Outfit display font. Visible in the top nav. Compare to Test 1.3 — under the default theme, header button labels use the body font.

**Things that must NOT have changed:**

- Layout / structure: same elements in the same places. We re-skinned, we did not restructure.
- All routes that worked under default still work (no template-resolution failure).
- Login / signup / form-submission flows still complete.

### 2.3 Quick interactive flow

Sign out, then sign back in, then visit a course and complete a form (any short activity in the demo content). The point is to exercise success / error / messages partials under the canary theme so we see the semantic-role rebrand applied to alerts.

```bash
FLS_THEME=does_not_exist uv run python manage.py runserver $PORT
```

Expected: process exits with `ImproperlyConfigured` and a message that includes the slug and the searched dirs. Server does **not** start.

## Test 4 — Email colour fallbacks

(Sanity check that the email-template colour parser still returns sensible values from the new theme.css.)

In a Django shell against the running default-theme server (or just inspect `settings.EMAIL_COLOR_*`):

```bash
FLS_THEME=default uv run python manage.py shell -c "from django.conf import settings; print(settings.EMAIL_COLOR_PRIMARY, settings.EMAIL_COLOR_FOREGROUND)"
FLS_THEME=first_class uv run python manage.py shell -c "from django.conf import settings; print(settings.EMAIL_COLOR_PRIMARY, settings.EMAIL_COLOR_FOREGROUND)"
```

Expected: default → `#2B6CB0` and `#1A2332`. first_class → `#283593` and `#1A1A2E`.

If a key returns the hard-coded fallback (e.g. `#2B6CB0` under `first_class`), that means `parse_tailwind_colors` did not find the new key in the resolved theme.css — investigate the path or token name.

## Test 5 — Static asset isolation

Confirm that brand assets do not collide between themes. With each theme active, request:

- `http://127.0.0.1:$PORT/static/themes/default/theme.css`
- `http://127.0.0.1:$PORT/static/themes/first_class/theme.css`

Expected: 200 OK on both regardless of which theme is active (because both `static/themes/<slug>/` paths are served via `STATICFILES_DIRS` once collected). When `FLS_THEME=default`, both files must still be retrievable individually — the namespacing prevents collision.
