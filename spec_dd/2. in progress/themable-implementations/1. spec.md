# Spec: Themable Freedom LS

## Why

Freedom LS is installed as a dependency in other Django projects (Bloom, FirstClass, etc.). Today, downstream projects that want a different look have to copy `tailwind.input.css`, hand-edit the `@theme` block, and — for anything beyond colour swaps — fork cotton component templates wholesale. There is no contract for what is themable, no opinion on how to package brand assets, and the install guide ("copy `tailwind.input.css` and edit it") locks downstream into maintaining a divergent file forever.

Other Django ecosystems have walked this road and offer instructive failure modes:

- **Bootstrap-style "just override variables"** breaks down the moment a downstream wants a different shape or a replaced shell.
- **Oscar-style "fork every template"** works but the maintenance cost is enormous and upgrades are painful.

The idea file (which all stakeholders have already aligned on) settles the architecture: a three-tier surface — design tokens, semantic component classes, template overrides — packaged as a self-contained theme directory selected by a single setting. This spec turns that direction into something implementable and pins down the decisions that the idea left open.

## Goals & success criteria

A downstream project must be able to:

1. **Cheap rebrand** (colours, radii, brand fonts) by writing one `theme.css` file and pointing `FLS_THEME` at their theme directory — no template edits, no Python.
2. **Component tweak** (rounder buttons, drop shadow on chips) by re-opening `.btn-primary` etc. in their own `@layer components` block — no template edits, no FLS fork.
3. **Structural override** (replace a cotton component or page shell) by dropping `templates/cotton/<name>.html` into their theme directory — without forking unrelated FLS templates.
4. Use the shipped `default` theme as a working reference: copy the directory, rename, edit, ship.

Success criteria:

- A second theme (`first_class`, used as the canary) can be built end-to-end in the FLS repo using only Tier-1 tokens, and visibly rebrands every page.
- Switching `FLS_THEME` between `default` and `first_class` requires only a settings change and a Tailwind rebuild — no template edits, no Python edits.
- All existing FLS pages render identically under `FLS_THEME=default` (no visual regression on the default theme).
- A new theming guide (`docs/how tos/theme-fls.md`) is written and a fresh consumer project can follow it to build a working custom theme. The existing install guide (`docs/how tos/incorperate into another project.md`) is updated only where install steps actually change (e.g. the new `tailwind.input.css` template, `FLS_THEME` setting); deep theming content lives in the new guide.

## Architecture: three tiers

Layered cheapest-first. Tier 1 covers the 80% case; Tier 2 covers shape/elevation tweaks; Tier 3 is the escape hatch.

### Tier 1 — Design tokens (CSS custom properties + `@theme inline`)

Token contract lives at `:root` (and later, optionally, `[data-theme="…"]`). Tailwind's `@theme inline` block maps the role-named tokens onto Tailwind colour utilities, so `bg-primary` etc. compile to `var(--color-primary)` and follow the cascade. This makes a static rebrand (downstream redeclares `:root`) and a future runtime swap (set `data-theme`) work the same way without further code changes (see `research_tailwind_v4_theming.md` §2).

#### Colour roles — the public token contract

The user has set the minimum role surface: **`primary`, `secondary`, `accent`, `success`, `warning`, `error`, `info`**. To make these usable in templates we also need text-pair (`on-*`) tokens for foreground-on-coloured-surface, plus structural tokens for surfaces, borders, and focus rings.

Final v1 contract:

| Role | Purpose | Pair (`on-…`) |
| --- | --- | --- |
| `primary` | Main brand colour. Primary CTAs, links, brand accents. | `on-primary` |
| `secondary` | Supporting brand colour. Secondary buttons, less-loud emphasis. | `on-secondary` |
| `accent` | Sparingly-used highlight. Decorative bursts, badges. | `on-accent` |
| `success` | Completion, positive state. | `on-success` |
| `warning` | Caution, important non-error notice. | `on-warning` |
| `error` | Destructive action, validation failure. | `on-error` |
| `info` | Informational notice (neutral, non-warning). | `on-info` |
| `surface` | Primary page background. | `on-surface` |
| `surface-2` | Recessed/raised surface (panels, table-row hover, header strip). | `on-surface` |
| `border` | Default border colour for surfaces, inputs, dividers. | — |
| `muted` | De-emphasised text (labels, helper text, secondary copy). | — |
| `focus-ring` | Focus outline (defaults to `primary` if downstream omits). | — |

Each role compiles to standard Tailwind utilities: `bg-primary`, `text-on-primary`, `border-border`, `ring-focus-ring`, etc.

##### Bold/hover variants (`-bold`, `-muted`)

The current contract has `primary-bold`, `success-bold`, `danger-bold` for hover states, and uses `*/15`, `*/30` opacity modifiers for chip backgrounds. We have two options:

- **Option A (recommended)**: drop the `-bold` tokens entirely. Hover states become `hover:brightness-110` (or similar) inside Tier-2 component classes, so the role contract stays narrow. Chip backgrounds stay as opacity modifiers (`bg-primary/15`).
- **Option B**: keep `-bold` and `-muted` variants on every role for explicit control. More tokens to define per theme.

Going with Option A in v1. Themes can override the hover behaviour at Tier 2 if needed. (User: flag if you want B.)

#### Shape tokens

```
--fls-radius-sm
--fls-radius-md
--fls-radius-lg
--fls-radius-pill
```

Mapped via `@theme inline` to `--radius-*` Tailwind keys so utilities like `rounded-md` pick up theme values. Defaults match current Tailwind defaults.

#### Type tokens

```
--fls-font-sans
--fls-font-display
```

Two knobs only. `--fls-font-sans` drives `font-sans`; `--fls-font-display` drives a new `font-display` Tailwind utility for `h1`–`h4`.

#### Out-of-scope token surfaces (deferred)

- Full 50–950 colour ramps. Tailwind v4 does not auto-derive (see `research_tailwind_v4_theming.md` §3). Skip until a component needs ramped colours.
- Density / spacing scale tokens.
- Per-site `data-theme` swaps. The token surface is *designed* to allow this later but no resolution code ships in v1.

### Tier 2 — Semantic component classes

FLS keeps `btn`, `btn-primary`, `chip`, `chip-success`, `surface`, etc. as the public visual contract. Class names become a **documented public API** with deprecation rules — same caution as a Python signature. A class rename is a breaking change.

A downstream theme extends these by re-opening them inside its own `theme.css` (the conventional home — Tier-2 tweaks travel with the theme directory, so the same theme can be moved between projects):

```css
@layer components {
  .btn-primary { @apply rounded-2xl shadow-sm; }
}
```

A downstream may also re-open the same classes in its top-level `tailwind.input.css` for project-level (non-theme) tweaks. Both rule sets survive; later declarations win per the cascade (see `research_tailwind_v4_theming.md` §5). Because the default theme's `theme.css` is `@import`-ed before the downstream theme's `theme.css`, downstream Tier-2 changes automatically win without needing `!important`.

### Tier 3 — Template overrides

For genuine restructure (different page shell, replaced header, custom card layout):

- Drop `templates/cotton/<name>.html` into the theme dir to override a cotton component.
- Drop `templates/partials/<name>.html` to override a page-shell partial.
- Drop `templates/_base.html` to replace the full page shell.

Cotton's `<c-vars …>` declaration plus `{{ attrs }}` and `{{ slot }}` forwarding form the override contract. A theme that overrides a cotton component must honour the full `<c-vars>` block — same caution as a Python function signature.

App-level templates (student dashboards, educator views, etc.) stay in their FLS apps. Themes contain *only* visual primitives (cotton components, page shells, brand assets, the token CSS).

#### Cotton components outside `base`

FLS today ships cotton components in three places:

- `freedom_ls/base/templates/cotton/` — the generic visual primitives (`button`, `chip`, `modal`, `dropdown-menu`, `pagination`, etc.).
- `freedom_ls/content_engine/templates/cotton/` — content-specific primitives (`callout`, `content-link`, `file-download`, `pdf-embed`, `picture`, `youtube`).
- `freedom_ls/icons/templates/cotton/` — `icon`.

Cotton's loader resolves on the bare `<name>.html` regardless of which app provided it, so a theme override at `themes/<theme_slug>/templates/cotton/<name>.html` already overrides any of these. The spec resolves the location question as follows:

- **`base/templates/cotton/`** — moves into `freedom_ls/themes/default/templates/cotton/`. These are the generic primitives the theming story is built around.
- **`content_engine/templates/cotton/` and `icons/templates/cotton/`** — stay where they are. They are domain primitives owned by their app, not theme primitives. Themes that want to restyle them still can (drop a same-named file in the theme dir), but the canonical home stays with the owning app. This matches the existing project convention (documented in `fls-claude-plugin/resources/templates_and_cotton.md`) for app-owned cotton components.

The convention doc will need an addendum: "generic visual primitives previously in `base/templates/cotton/` now live in `freedom_ls/themes/default/templates/cotton/`. App-specific cotton components continue to live in their owning app."

## Theme directory shape

```
themes/<theme_slug>/
    templates/
        cotton/                  # override any cotton component
        partials/                # override page-shell partials (header, footer, logo)
        _base.html               # optional: full page-shell override
    static/
        themes/<theme_slug>/
            theme.css            # token bundle + Tier-2 component tweaks
            logo.svg             # brand assets
            ...
    theme.md                     # human-readable manifest: name, version, what it overrides
```

The static path is intentionally namespaced as `static/themes/<theme_slug>/` so collected static files don't collide between themes.

## Selecting a theme

One setting, one env var:

```python
FLS_THEME = env("FLS_THEME", default="default")
FLS_THEMES_DIRS = [BASE_DIR / "themes", FREEDOM_LS_PACKAGE_DIR / "themes"]
```

Resolution: walk `FLS_THEMES_DIRS` in order, use the first directory whose name matches `FLS_THEME`. The downstream `themes/` dir comes first, so a downstream theme named `default` shadows the FLS one cleanly.

The resolution function (suggested location: `freedom_ls/base/theming.py`) returns the resolved theme dir and is called from a settings helper that:

- Prepends the resolved `templates/` dir to `TEMPLATES[0]["DIRS"]`.
- Prepends the resolved `static/` dir to `STATICFILES_DIRS`.

Suggested settings helper signature:

```python
def configure_theme(templates: list[dict], staticfiles_dirs: list[Path]) -> None: ...
```

Called once near the top of `settings_base.py`, after `BASE_DIR` is defined and before `TEMPLATES` is finalised.

**Failure modes:**

- `FLS_THEME` set to a slug that does not resolve in any `FLS_THEMES_DIRS` entry → raise `ImproperlyConfigured` at startup. Fail loud.
- `FLS_THEMES_DIRS` not set → default to `[FREEDOM_LS_PACKAGE_DIR / "themes"]`, log a warning if `FLS_THEME != "default"`.

## Default theme — first-class, not a fallback

Migrate the current cotton components, partials, and `tailwind.components.css` content into `freedom_ls/themes/default/`:

```
freedom_ls/themes/default/
    templates/
        cotton/                  # generic primitives moved from base/templates/cotton/
        partials/                # header_bar, messages, etc. moved from base/templates/partials/
        _base.html               # moved from base/templates/_base.html
        _base_interface.html     # moved from base/templates/_base_interface.html
    static/
        themes/default/
            theme.css            # tokens (in @theme inline), @layer base, @layer components — migrated from tailwind.components.css
    theme.md
```

After migration:
- The existing `freedom_ls/base/templates/cotton/` and `freedom_ls/base/templates/partials/` directories are deleted (their contents now live in the default theme).
- `tailwind.components.css` is deleted (its content is now in `themes/default/static/themes/default/theme.css`).
- The default theme is selected automatically when `FLS_THEME` is unset, so the FLS dev server keeps working unchanged.

The default theme is the **reference implementation**: structured and commented well enough that downstream theme authors copy the directory wholesale and start editing.

## Tailwind build ownership

Replace the "copy `tailwind.input.css` and edit it" step with a documented downstream input file:

```css
@import "tailwindcss";

/* Scan all FLS app templates (this also covers freedom_ls/themes/default/templates/). */
@source "<path-to-freedom_ls>/**/templates/**/*.html";
/* Scan the downstream theme's templates. */
@source "./themes/<theme_slug>/templates/**/*.html";

/* Always import the default theme's tokens + components first; the downstream
   theme.css may then re-open Tier-2 classes or re-declare tokens. */
@import "<path-to-freedom_ls>/themes/default/static/themes/default/theme.css";
@import "./themes/<theme_slug>/static/themes/<theme_slug>/theme.css";

@theme {
    /* downstream project-level overrides, optional */
}
```

The default theme's templates are scanned by the first `@source` glob (because `freedom_ls/themes/default/` lives under `freedom_ls/`). The default `theme.css` is always imported because cotton components the downstream *doesn't* override still come from default — and they reference Tier-2 classes that need to exist. A downstream `theme.css` that wants to *replace* (not extend) a Tier-2 class can re-declare it in its own `@layer components` block; cascade order means the later declaration wins.

`research_tailwind_v4_theming.md` §6 flags two pitfalls to call out in the install guide:
- `@source` honours `.gitignore`. If FLS is installed under a path excluded by an ancestor `.gitignore` (`.venv/`, `node_modules/`), the glob silently skips. Document the symlink / asset-pipeline-copy workaround.
- New cotton class names not present at build time are not generated. Themes that introduce new utility usages must be in the `@source` glob.

The FLS repo's own `tailwind.input.css` becomes a thin file structurally identical to the downstream template, with `<theme_slug>` resolved to whatever the dev environment selects (default `default`).

## Migration scope (breaking changes)

This is a **breaking change** for any downstream project currently consuming FLS.

### Token rename — affects all consumers

The new role contract renames tokens in templates. Concrete renames:

| Old class → | New class |
| --- | --- |
| `text-foreground`, `bg-foreground`, etc. | `text-on-surface`, `bg-on-surface` |
| `text-danger`, `bg-danger`, `border-danger` | `text-error`, `bg-error`, `border-error` |
| `text-danger-bold` | `text-error` (with optional `hover:brightness-110`) — see Option A above |
| `text-success-bold` | `text-success` |
| `text-primary-bold`, `bg-primary-bold` | `text-primary` (with optional `hover:brightness-110`) |
| `text-on-primary` (when on success/error/etc.) | `text-on-success`, `text-on-error`, etc. |

Token usage in FLS templates today (from grep of `freedom_ls/**/*.html`):

- 75 × `text-muted` (no rename)
- 25 × `border-border` (no rename)
- 20 × `text-primary` (no rename)
- 20 × `bg-surface-2` (no rename)
- 19 × `text-foreground` → `text-on-surface`
- 17 × `bg-surface` (no rename)
- 15 × `text-danger` → `text-error`
- 13 × `text-success` (no rename)
- 10 × `border-danger` → `border-error`
- 10 × `bg-danger` → `bg-error`
- 7 × `text-warning` (no rename)
- 7 × `text-danger-bold` → `text-error`
- … plus a long tail of `*-bold`, `border-warning`, etc.

Total: ~41 templates touched, ~290 individual token references to update. All within the FLS repo — no downstream template edits needed *for FLS templates*, because the rename happens at the source. Downstream **theme.css** files written against the old token names will need updating; that is the cost of the rename.

### Component class names — public API now

`btn`, `btn-primary`, `btn-outline`, `btn-success`, `btn-danger` (becomes `btn-error`), `chip`, `chip-primary`, `chip-warning`, `chip-success`, `chip-danger` (becomes `chip-error`), `chip-xs`, `surface` are now the documented public API. Renaming these in future is a breaking change requiring a migration note.

### Resource / convention docs

`fls-claude-plugin/resources/templates_and_cotton.md` documents `freedom_ls/<app_name>/templates/cotton/<component>.html` as the cotton-component location. After this work, that statement is partially superseded — generic primitives now live under `freedom_ls/themes/default/templates/cotton/`. The resource doc must be updated to:

- Note the theme directory as the home of generic primitives.
- Keep the app-owned cotton convention (e.g. `content_engine/templates/cotton/`) for app-specific primitives.
- Mention the override mechanism (drop a same-named file in a theme dir).

Similarly, `fls-claude-plugin/resources/frontend_styling.md` references `tailwind.components.css` as the home of component classes; that doc must be updated to point at the theme's `theme.css` instead.

### Install guide update + new theming guide

Split into two documents so the install path stays short and theming gets its own focused walkthrough.

`docs/how tos/incorperate into another project.md` (existing) is updated to:
- Drop the "copy tailwind.input.css and edit it" step.
- Document the new downstream `tailwind.input.css` template.
- Mention `FLS_THEME` / `FLS_THEMES_DIRS` settings with a default-theme example only.
- Link to the new theming guide for anything beyond "use the default theme".

`docs/how tos/theme-fls.md` (new) is written to cover theming end-to-end:
- The three-tier model (tokens → component classes → template overrides) and when to reach for each.
- Theme directory shape and the `theme.md` manifest.
- `FLS_THEME`, `FLS_THEMES_DIRS` resolution rules and failure modes.
- The full token contract (role names, `on-*` pairs, shape, type) with the `theme.css` template.
- Re-opening Tier-2 component classes in `@layer components`.
- Tier-3 cotton/partial/`_base.html` overrides and the `<c-vars>` contract.
- The `@source` `.gitignore` pitfall and the symlink/copy workaround.
- A worked example of a minimal Tier-1-only theme, plus a pointer to `first_class` as a worked all-three-tiers example.

## Validation: the `first_class` canary theme

To prove the architecture works end-to-end, ship a second theme `first_class` in `freedom_ls/themes/first_class/`, driven by the First Class "Modern Altitude" brand (see `first-class-theme.md`). It must exercise all three tiers:

- **Tier 1 — every token overridden** from First Class brand docs:
  - `primary` = Deep Indigo `#283593`, `on-primary` = `#FFFFFF`
  - `secondary` = Electric Teal `#00CEC9`, `on-secondary` = Cockpit Dark `#1A1A2E`
  - `accent` = Altitude Orange `#FF6B35`, `on-accent` = `#FFFFFF`
  - `success` = `#38A169`, `warning` = `#D69E2E`, `error` = `#E53E3E`, `info` = `#3182CE` (with `#FFFFFF` pairs)
  - `surface` = Stratosphere `#F8F9FC`, `surface-2` = `#EDF2F7`, `on-surface` = Cockpit Dark `#1A1A2E`
  - `border` = `#E2E8F0`, `muted` = `#718096`, `focus-ring` = alias of `primary`
  - `--fls-font-sans` = `"DM Sans", system-ui, sans-serif`
  - `--fls-font-display` = `"Outfit", system-ui, sans-serif`
  - Radii: bumped one step (e.g. `--fls-radius-md` ≈ `0.75rem`) to match the brand's softer card treatment.
- **Tier 2 — component-class tweaks** that demonstrate re-opening Tier-2 classes:
  - `.btn-primary` rounder + slightly stronger shadow.
  - `.chip` flatter (no shadow) to match the brand's data-forward look.
- **Tier 3 — at least one structural override** to prove the override mechanism: an overridden `cotton/header-button.html` (or equivalent small cotton primitive) that ships with First Class branding (e.g. uses the Outfit display font on its label).

`first_class` is a test fixture **and** a real downstream-style example. It lives in the repo so CI can rebuild Tailwind under both themes and Playwright can visit a few key pages under each. Brand-specific niceties beyond the token/component contract (IBM Plex Mono for code blocks, Success/Warning/Error *Light* tints, full type scale) are noted in `first-class-theme.md` and may be added if the canary needs them, but are not requirements for the architectural validation.

## Out of scope (deferred)

- **Per-site theming on a single deployment** (custom site-aware template loader, `cached.Loader` cache-key plumbing). Architecturally feasible but breaks for management-command/email rendering. Defer until a deployment actually needs to host two brands.
- **Per-site Tailwind-variable-only override** via `data-theme` on `<body>` keyed off `current_site`. Lighter than full per-site theming. Token contract is designed to allow this later (uses `@theme inline` per `research_tailwind_v4_theming.md` §2). Revisit when the demand arrives.
- **Full 50–950 colour ramps**, density tokens, fine-grained spacing scale.
- **Theme inheritance / composition** (a theme that extends another).
- **Hot-reload on theme.css changes.** Standard Tailwind watch handles this; no FLS-specific tooling.

## Decisions made (call out for user review)

1. **Drop `-bold` tokens, use `hover:brightness-110` in component classes** (Option A above). Smaller token surface; downstream can override at Tier 2 if needed.
2. **Rename `danger` → `error`** to match the user-specified role list.
3. **Rename `foreground` → `on-surface`** for consistency with the `on-*` pattern. Keep `muted` as-is (templates read better).
4. **`focus-ring` defaults to alias of `primary`** rather than a separate value.
5. **Keep `secondary` as a real third brand colour**, not a repurposing of the old `primary-bold` hover state. Suggested `#475569` slate.
6. **`info` is new** (suggested `#0EA5E9` sky); not currently in the codebase.
7. **`accent` is new** (suggested `#F59E0B` amber); not currently in the codebase.
8. **No backward-compatibility shim** for the old token names. Single breaking change, single migration.
9. **Default theme moves to `freedom_ls/themes/default/`**, current `cotton/` and `partials/` directories under `freedom_ls/base/templates/` are deleted (their contents move into the default theme). `base/templates/_base.html` and `_base_interface.html` move into the default theme. `account/` and `allauth/` directories stay where they are — they aren't visual primitives. **App-owned cotton components in `content_engine/templates/cotton/` and `icons/templates/cotton/` stay in their owning app** (see "Cotton components outside `base`" above).
10. **Fail-loud on bad `FLS_THEME`** at startup.
