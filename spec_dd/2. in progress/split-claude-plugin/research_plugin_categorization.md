# Research: Complete inventory and classification of `fls-claude-plugin/`

Every file under `fls-claude-plugin/` (105 files total), read in full, classified into one of:

- **django-stack** — portable Python 3.13/Django 6/Postgres/HTMX/Tailwind/Alpine/cotton best practice, no FreedomLS domain knowledge, usable in an unrelated Django project with the same stack.
- **fls** — tied to FreedomLS domain models/apps/product concepts (cohorts, content_engine, student_progress, multi-site accounts, registration, upgrade notes, template repo, fls-content, demo_content, etc.).
- **sdd** — part of the generic spec-driven-development workflow (idea → spec → plan → implement → QA → ship), not Django- or FLS-specific.
- **ambiguous/shared** — genuinely hard to place; tension explained, recommendation given.

FLS-coupling tokens searched for and cited where found: `freedom_ls`, `cohort`, `student`, `content_engine`, `AUTH_USER_MODEL`, `site_aware`/`SiteAware`, `accounts.User`, `docs/product`, `demo_content`, `qa_helpers`, `submodules/Freedom-LS`.

## Genuinely ambiguous items — consolidated list and recommendations

1. **`resources/testing.md` + `skills/testing/SKILL.md`** — Largest and highest-value split candidate. Recommendation: fork into a `django-stack` file with the philosophy/technique sections (tautology, mocking boundaries, AAA, parametrize, anti-pattern cheatsheet, HTMX-testing mechanics, Playwright-vs-pytest boundary) and an `fls` file with the FLS-only bits (`mock_site_context`, `fls_internal`/`playwright`/`ci_only` marker *rationale* as a downstream-distribution concern, `freedom_ls.course_applications` example, de-branding worked examples). The `django-stack` skill should cross-reference the `fls` skill's marker taxonomy as an *example* of how a package-that-ships-downstream might extend the base taxonomy, not assume every consumer needs it.

2. **`resources/playwright-testing.md` + `skills/playwright-tests/SKILL.md`** and **`skills/use-playwright/SKILL.md`** — Same split shape: generic Playwright/MCP mechanics → `django-stack` (or duplicated into `sdd` since `do_qa.md` uses it); the "why `playwright` marker exists for downstream exclusion" framing → `fls`.

3. **`resources/factory_boy.md`** — Split generic factory_boy technique (→ `django-stack`) from the mandatory `SiteAwareFactory` base class and FLS model examples (→ `fls`).

4. **`resources/templates_and_cotton.md` + `skills/template/SKILL.md`** — Strong `django-stack` candidates almost as-is (django-cotton/HTMX/Alpine conventions); only the theme-shadowing-by-path mechanism assumes FLS's `configure_theme()` multi-theme system and would need trimming/genericizing.

5. **`resources/frontend_styling.md` / `skills/frontend-styling/SKILL.md`, `skills/alpine-js/SKILL.md`, `resources/admin_interface.md` / `skills/admin-interface/SKILL.md`, `skills/app-settings/SKILL.md`** — Each has a clean generic-methodology core suited to `django-stack`, with a thin FLS-specific layer (concrete file paths, `SiteAwareModelAdmin`, `freedom_ls.base.app_settings`, FLS's Alpine component inventory) that belongs in `fls`. Recommend a "core convention doc in django-stack + a short FLS-addendum in fls that layers the FLS-specific requirement on top" pattern for all four, rather than trying to keep one shared file two plugins both `Read` (plugins are meant to be independently installable in unrelated projects, so django-stack cannot assume `fls` exists).

6. **`commands/sdd/README.md`, `commands/sdd/protected/setup_todo_list.md`** — The core SDD checklist/workflow is `sdd`, but both files currently name FLS-specific steps (`update_template_repo`, `update_claude_plugin_fls_content`, `update_upgrade_notes`) inline as if they were universal SDD steps. Recommendation: make the portable `sdd` README/checklist describe these as an **optional "downstream-distribution" phase** a consuming project can add its own steps into (with FLS's concrete versions documented in the `fls` plugin), rather than baking FLS-specific command names into the generic workflow doc.

7. **`commands/sdd/plan_security_review.md`, `commands/sdd/do_qa.md`, `commands/sdd/finish_worktree.md`** — All three are ~90% generic `sdd` workflow with one or two FLS-specific lines/calls (`fls:multi-tenant`/`SiteAwareModel` bullet; `fls:qa-data-helper` delegation + FLS credentials-config path; FLS's dev-DB-teardown script call). Recommendation: parameterize these single touchpoints (e.g. "delegate to a project-specific test-data-creation agent if one is configured" / "run the project's dev-DB-teardown script if one exists") so `sdd` stays installable standalone, while `fls` supplies the concrete agent/script.

8. **`.mcp.json` (Playwright MCP)** — Zero FLS content but used almost exclusively by FLS/SDD workflows in this repo. Recommendation: ship a copy in `django-stack` (since Playwright+Django is a stack-level pairing) and let `sdd`/`fls` rely on the user having it enabled, rather than duplicating server config three times without reason.

9. **`templates/settings.json`** — Split the generic permission allow/deny baseline (candidate for `django-stack` or `sdd`) from the FLS-only `SessionStart` `$FLS_PLUGIN` hook (`fls`).

10. **Wrapper-script generation set (`templates/wrapper_scripts/find_available_port.sh`, `db_clear.sh`, `kill_runserver.sh`, `fetch_pr_comments.sh`)** — These wrap fully generic scripts but exist only because of FLS's `fls:init` wrapper-generation mechanism (path substitution, `.claude/fls/scripts/` destination). Recommendation: keep as `fls` artifacts unless/until `django-stack` grows its own equivalent init command, at which point these four (but not the FLS-coupled ones: `dev_db_*`, `db_recreate`, `install_dev`) could be regenerated from django-stack's own generic scripts.

11. **`resources/agent_memory_guidelines.md`** — Generic Claude-Code agent-memory guidance referenced by two agents landing in two different target plugins (`fls`'s `qa-data-helper`, and a hypothetical `django-stack`/`sdd` `code-reviewer`). Recommendation: house it in `sdd` alongside `claude-code-authoring` (the other Claude-Code-meta-skill), and have `fls`'s agents reference it cross-plugin (Claude Code plugins can coexist and be enabled together, so a cross-plugin reference by path is acceptable as long as `fls` documents the dependency on `sdd` being installed — or duplicate the short file if plugins must be fully independent).

12. **`agents/code-reviewer.md`** — Recommend a generic reviewer body in `django-stack`, with two FLS-only checklist bullets (the brand-literal test-portability check, the optional-app collection-safety check) either dropped from the generic version or documented as something `fls` layers on via its own thin wrapper agent that extends the generic checklist.

13. **`commands/placeholder_page.md`** — Currently delegates to the FLS-flavoured `template` skill. Once `template`'s generic core moves to `django-stack` (see item 4), this command should move there too and reference the genericized skill.

## Notes on plugin independence

Several "ambiguous" items above resolve cleanly only if the three target plugins are designed to be **independently installable** (a `django-stack`-only consumer on an unrelated Django project must not require `fls` or `sdd` to be present). That constraint is what forces the "split into a generic core + a thin domain-specific addendum" pattern recommended repeatedly above, rather than leaving one file that silently assumes a sibling plugin exists. Where the split is cheap (most resources — they're markdown reference docs), do it. Where the split is expensive relative to value (e.g. `resources/agent_memory_guidelines.md`, a 34-line generic file), duplicating the small file across plugins is simpler than introducing a cross-plugin dependency.
