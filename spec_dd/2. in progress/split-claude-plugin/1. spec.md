# Spec: Split the monolithic plugin into focused, portable plugins under `claude_plugins/`

## 1. Summary

The single Claude Code plugin at `fls-claude-plugin/` today mixes three separable concerns:
portable Django-stack conventions, FreedomLS-product-specific conventions, and a generic
spec-driven-development (SDD) workflow. This spec defines a **refactor of the plugin authoring
artifacts** (commands, skills, agents, hooks, scripts, templates, configs) that splits the monolith
into three focused plugins and gathers all four plugins — including the existing
`fls-content-plugin/` — under a new top-level `claude_plugins/` directory.

> This changes **only** Claude Code plugin authoring artifacts. It touches no Django application
> code, no models, no migrations, no runtime behaviour of the FLS product itself.

The bar the idea sets for the SDD portion is **"only do it if it is quite straightforward."** The
research verdict is that ~60% of the SDD surface is a clean mechanical move and the rest is genuinely
tangled with Django/FLS. **We do the scoped move and explicitly defer full generalization.**

> **Naming note.** The FreedomLS-product-specific plugin is named **`fls-dev`** (manifest
> `name: "fls-dev"`, directory `fls-dev-claude-plugin/`), *not* `fls`. This disambiguates it from the
> sibling `fls-content` plugin and signals that it carries FreedomLS *developer* tooling. Throughout
> this spec, `fls-dev` refers to that **target** plugin; a bare `fls` (e.g. `fls:sdd-worker`,
> `Skill(fls:*)`, `fls-claude-plugin/`) refers to the **current** monolith being split, whose manifest
> name is `fls` today. The rename is part of the "update all references" workstream (§8).

## 2. Motivation — why this matters

- **Reuse.** The portable Django-stack skills (testing, tailwind, templates/cotton, htmx, alpine,
  playwright) and the generic SDD scaffolding carry zero FLS domain knowledge. Trapped inside a
  FreedomLS-branded plugin, they can't be reused on unrelated Django projects. Separating them makes
  that reuse *possible in principle* (actual cross-project distribution is deferred — see §12).
- **Clarity of ownership.** A developer extending FLS should be able to see, at the plugin boundary,
  what is a universal stack convention vs. what is specific to this product's multi-tenant/site-aware
  model, registration, content engine, and downstream-distribution machinery.
- **Single source of truth.** A queued idea (`extract-django-best-practices-claude-plugin`)
  independently proposed the same Django extraction. This spec absorbs and supersedes it so there is
  one plan of record.
- **Correctness is at stake.** Claude Code performs **no** cross-plugin static validation (verified
  against the 2.1.x docs). A missed rename fails *silently* at invoke time, often deep inside a
  fanned-out subagent. "Update all references" is therefore the load-bearing workstream, not an
  afterthought.

## 3. Scope

### In scope
- Create `claude_plugins/` and populate it with four plugins: `django-stack-claude-plugin` (`ds`),
  `fls-dev-claude-plugin` (`fls-dev`), `sdd-claude-plugin` (`sdd`), and the relocated
  `fls-content-plugin` (`fls-content`).
- Physically move/copy every artifact to its owning plugin; author three new plugin manifests and
  three per-plugin READMEs; rewrite the launcher, the per-plugin init commands, the `next.md`
  dispatcher, and every cross-plugin reference.
- Rename the FLS-product plugin `fls` → `fls-dev` (manifest name and every derived identifier).
- Rename the launch sentinel `$FLS_PLUGIN` → `$CLAUDE_PLUGINS_LOADED`.
- Delete the `request-code-review` skill; retire the superseded queued idea directory by moving it to
  `spec_dd/3. done/` with a superseded note.

### Out of scope (deferred — see §12)
- Making `ds`/`sdd` genuinely installable in *unrelated* repositories (marketplace, per-consumer
  launchers, distribution).
- Untangling the accepted `fls-dev` ↔ `sdd` bidirectional dependency.
- Fully generalizing the FLS/Django-woven SDD steps (`do_qa`, `plan_security_review`,
  `plan_structure_review`) into pluggable abstractions.
- Any change to the Django application.

## 4. Target architecture

```
claude_plugins/
  django-stack-claude-plugin/    # manifest name: "ds"          — portable Django/stack conventions
  fls-dev-claude-plugin/         # manifest name: "fls-dev"      — FreedomLS-product-specific dev tooling
  sdd-claude-plugin/             # manifest name: "sdd"          — portable SDD workflow (scoped)
  fls-content-plugin/            # manifest name: "fls-content"  — course-authoring (relocated as-is)
```

Each plugin keeps its commands and skills **flat at the plugin root** (`commands/*.md`,
`skills/<name>/SKILL.md`) — no extra grouping subdirectory. In particular the `sdd` plugin uses
`sdd-claude-plugin/commands/start.md` (invoked `/sdd:start`), **not** `commands/sdd/start.md`. The
only retained command subdirectory is `commands/protected/` for read-and-followed helper files.

**Verified rule (Claude Code 2.1.x):** the `commands/` scanner recurses into subdirectories and
registers **every** `.md` under `commands/`, using each subdirectory as a namespace segment. So
`commands/sdd/start.md` registers today as `/fls:sdd:start`, and
`commands/sdd/protected/update_todo.md` registers as `/fls:sdd:protected:update_todo` — nested command
files **are** invocable. (This very review was launched via `/fls:sdd:spec_review`, from
`commands/sdd/spec_review.md`, and the current registry lists `/fls:sdd:protected:*`.) There is **no**
way to make a command `.md` non-invocable by placement inside `commands/`. What flattening buys us is
the *namespace shape*: moving the SDD commands to the plugin root (`sdd-claude-plugin/commands/start.md`)
makes them `/sdd:start` — the `sdd:` now comes from the plugin `name` — whereas leaving them at
`commands/sdd/start.md` would yield the double-segment `/sdd:sdd:start`. A retained subdirectory still
adds a segment, so the `protected/` helpers register as `/sdd:protected:*`.

> **On the `protected/` helpers and cross-plugin use.** The four helper files
> (`update_todo`, `setup_todo_list`, `move_spec_to_in_progress`, `start_worktree`) are
> *read-and-followed by literal path* (via the Read tool), not invoked as slash commands, skills, or
> subagents. They **are** referenced across the plugin boundary — `fls-dev` commands and the
> `sdd-mechanic` agent read `sdd`'s `protected/` helpers — but because this is a raw filesystem
> path-read, a stale path fails **loudly** (file-not-found), *not* silently like the namespaced
> categories in §8; §8(g) covers keeping all 19 literals correct. Per the rule above, these files
> **also** register as invocable `/sdd:protected:*` commands — that is unavoidable for any `.md` under
> `commands/`, and is already the case today for `/fls:sdd:protected:*`. We simply never advertise or
> invoke them that way; the read-and-follow-by-path contract is the one that matters. Keeping them
> under `protected/` is for organization and namespacing, **not** because it makes them private — no
> file placement can do that.

### 4.1 Plugin responsibilities

- **`ds` (django-stack)** — Anything that helps *any* project on this stack (Python 3.13+, Django
  6.x, PostgreSQL, HTMX, Tailwind, optional Cotton/Alpine/django-tasks) with **zero** FLS domain
  knowledge. Owns the stack-generic skills, the generic dev/git commands, the `code-reviewer` agent,
  the three stack-generic hook scripts + `hooks.json`, `.mcp.json` (Playwright), `.lsp.json`
  (Pyright), and its own `/ds:init`.
- **`fls-dev`** — Everything tied to how *this* product works: multi-tenant/site-aware conventions,
  the custom user model, registration, markdown-content, admin, icons, brand, `qa-data-helper`, the
  FLS-specific SDD steps, the downstream `concrete/update_fls` flow, and `/fls-dev:init`. Where a
  portable skill carried an FLS-specific layer, `fls-dev` keeps a **thin overlay skill** that
  references the `ds` skill and adds only the FLS delta (no duplicated content).
- **`sdd`** — The portable SDD scaffolding: the `sdd-worker`/`sdd-mechanic` agents, the
  `claude-code-authoring` skill, and the generic workflow command files, plus the generic
  `git-worktree-setup` (core) and `update-claude-project-settings` skills, and `/sdd:init`.
- **`fls-content`** — Relocated unchanged; its only coupling is a one-way, file-copy sync.

## 5. Decisions

### 5.1 Carried over from the superseded idea
1. **Extract-and-remove with FLS overlays.** Portable skills leave `fls-dev` and live in `ds`;
   `fls-dev` keeps thin overlay skills for its site-aware layer. No duplicated content. `ds` must
   **never** reference `fls-dev` (so `ds` stays clean); overlays point one direction only
   (`fls-dev` → `ds`).
2. **Templates / cotton / partials are separate skills** under `ds`.
3. **Hook split 3-of-4:** `ruff_fix.sh`, `post-edit-bandit.sh`, `security-guard.sh` (+ `hooks.json`)
   → `ds`; the `uv … pytest` pre-commit runner stays FLS-only (it belongs in
   `.pre-commit-config.yaml` / the repo-root `.claude/settings.json`, not a plugin hook).
4. **Versioning:** rely on git SHA for now; add `version`/release tags later if `ds`/`sdd` gain
   external consumers.

### 5.2 New decisions in this spec

- **Name the FLS-product plugin `fls-dev`** (manifest name, directory `fls-dev-claude-plugin/`,
  namespace `/fls-dev:*`, `Skill(fls-dev:*)`, `enabledPlugins` key `fls-dev`, config dir
  `.claude/fls-dev/`, and every `subagent_type`/prose reference to its own agents/skills). Rationale:
  disambiguates from `fls-content` and marks it as developer tooling. Init must actively migrate an
  existing downstream `.claude/fls/` config dir to `.claude/fls-dev/` (§9.3).
- **General-dev command placement (per categorization research).** `commit`, `catchup`,
  `make_github_issue`, `rebase_main`, `tdd_implement`, `placeholder_page`, and the `periodic/`
  folder (`README` + `dependabot_prs`) → **`ds`**; `address_pr_review` → **`sdd`** (it is the SDD
  ship-stage step). `placeholder_page` references the genericized `ds:template` skill; `periodic/`
  is kept together in `ds` (the stronger signal — `dependabot_prs` is Django/uv-flavoured — wins
  over `periodic/README`'s weak lean toward `sdd`).
- **`code-reviewer` agent → `ds`, drop the two FLS-specific bullets.** The generic reviewer body
  moves to `ds`. The two FLS-only checklist bullets (brand-literal test-portability check; optional-
  app collection-safety for `freedom_ls.course_applications`) are **dropped**, not overlaid — this
  keeps `ds` free of FLS domain knowledge per Decision 5.1(1). No `fls-dev` reviewer overlay agent is
  created. (These checks can be re-added to an `fls-dev` overlay agent in a later pass if wanted;
  leaving them out now is the simpler, principle-consistent choice.)
- **Retire the superseded idea by relocation, not deletion.** `spec_dd/1. next/extract-django-best-practices-claude-plugin/`
  is **moved to `spec_dd/3. done/`** with a short superseded note added to its idea file pointing at
  this spec (its resolved decisions are already carried into this spec). This preserves the research
  trail while making this spec the single source of truth — nothing stays queued in `1. next/`.
- **`request-code-review` is deleted** — but it is **still referenced twice** by `implement_plan.md`
  today (the per-batch review step and the end-to-end review step), so deletion must include scrubbing
  **both** references (see §9.5). It is not yet dead code.
- **Per-plugin init.** Each plugin ships its own init (`/ds:init`, `/fls-dev:init`, `/sdd:init`, plus
  the existing `/fls-content:init`) rather than one looping init. Each stores config under its own
  `.claude/<plugin>/` directory (`.claude/ds/`, `.claude/fls-dev/`, `.claude/sdd/`) with the same
  internal structure `.claude/fls/` uses today (`config.md`, `config.local.md`, `scripts/`, …).
- **`ds` is the designated "primary" init owner** for the inherently shared, multi-plugin artifacts:
  the root `claude.sh` launcher, the `SessionStart` hook, and `.gitignore`. `fls-dev:init`/`sdd:init`
  detect-and-skip these. (Rationale: `ds` owns the wrapper-script templates and the launcher is a
  stack-generic artifact.)
- **Todo template stays whole.** The `setup_todo_list` helper moves to `sdd` and its generated
  `todo.md` keeps all 15 sections, including the FLS-only ones. We do **not** build a cross-plugin
  todo-composition seam now. Every command/skill the generated `todo.md` lists must be written
  **fully namespaced** (`/sdd:improve_idea`, `/fls-dev:do_qa`, `/ds:app_map`, `Skill(ds:testing)`).
- **`next.md` dispatch keeps the prefix.** Replace the "strip prefix, probe two dirs" heuristic with
  a deterministic `prefix → owning-plugin commands dir` map (see §9.4).
- **Sentinel rename.** `$FLS_PLUGIN` → `$CLAUDE_PLUGINS_LOADED` (one boolean set by the launcher,
  checked by the `SessionStart` hook). `FLS_PATH` / `__FLS_PATH__` is **kept** — it still accurately
  names the FLS repo/submodule checkout, under which `claude_plugins/` now lives.
- **No `dependencies` manifest field.** The `plugin.json` `dependencies` field resolves only within a
  marketplace (verified). This repo has no marketplace and loads via `--plugin-dir`, so a
  `dependencies` entry would be inert bookkeeping that misleadingly implies enforced ordering. The
  accepted `fls-dev` ↔ `sdd` coupling is enforced purely by `claude.sh`'s `--plugin-dir` flags and by
  documentation. Add real `dependencies` only if/when a marketplace is introduced (deferred).
- **`fls-content` is relocated but not added to the root launcher.** It is not loaded by this repo's
  `claude.sh` today (it is used independently by course-authoring). The split preserves that: only
  its physical location and the sync command's hardcoded paths change.

### 5.3 Accepted consequence — bidirectional coupling
The dependency is bidirectional and we accept it for this pass: `fls-dev` depends on `sdd` (its
FLS-specific SDD steps spawn `sdd:sdd-worker` / `sdd:sdd-mechanic`), and `sdd` depends on `fls-dev`
(the `setup_todo_list` template and `next.md` dispatch reference FLS-specific steps). So `sdd` is
**not yet** fully standalone-portable. Untangling this is deferred.

## 8. Reference-update workstream (load-bearing)

Claude Code does **no** cross-plugin static validation (verified: `claude plugin validate` checks a
single plugin's own manifest/frontmatter/hooks only). Every category below fails *silently* at invoke
time if missed. A grep checklist is the only mitigation. In the table, a bare `fls:` / `fls` names the
**current** monolith; the rewrite targets use `ds` / `fls-dev` / `sdd`.

| # | Category | Verified count | Rewrite rule |
|---|---|---|---|
| a | Launcher `--plugin-dir` | 2 files (root `claude.sh`, wrapper template) | one `--plugin-dir` per plugin (no parent-dir auto-discovery exists) — §9.1 |
| b | `Skill(fls:*)` permission glob | 1 live (`.claude/settings.json`) | → `Skill(ds:*)`, `Skill(fls-dev:*)`, `Skill(sdd:*)` |
| c | `subagent_type` agent strings | **40 occurrences of `fls:sdd-worker`/`fls:sdd-mechanic` across 16 files**; **12 of `fls:qa-data-helper`** | **Three distinct target prefixes — not one find-and-replace.** `fls:sdd-worker`/`fls:sdd-mechanic` → `sdd:sdd-worker`/`sdd:sdd-mechanic` in **all** files (even `fls-dev` commands now call the `sdd`-plugin agents); `fls:qa-data-helper` → `fls-dev:qa-data-helper` (the agent stays in the product plugin, which is renamed); `fls:code-reviewer` → `ds:code-reviewer`. Check each occurrence against which agent it names. |
| d | `mcp__plugin_fls_playwright__*` | 1 live | → `mcp__plugin_django-stack_playwright__*` (Playwright MCP moves to `ds`) |
| e | `enabledPlugins` keys | 1 live (`{"fls": true}`) | → `{"django-stack": true, "fls-dev": true, "sdd": true}`; each init merges only its own key. (Cosmetic for `--plugin-dir` plugins, but keep in sync.) |
| f | Slash-command prefixes | across the tree | resolved by construction once files move flat under each plugin (`/sdd:start`, `/ds:app_map`, `/fls-dev:do_qa`) |
| g | Hardcoded `fls-claude-plugin/commands/sdd/protected/*.md` self-refs | **19 occurrences across 17 files** | → literal `claude_plugins/sdd-claude-plugin/commands/protected/…` (these are read by **agents**, which don't get `${CLAUDE_PLUGIN_ROOT}` substituted into prose — so a literal repo-relative swap, in the same commit as the `git mv`) |

## 9. Infrastructure changes

### 9.1 Launcher (`claude.sh` + wrapper template)
Both the repo-root `claude.sh` and the generated-wrapper template must set the renamed sentinel and
pass **one `--plugin-dir` per plugin**:

```bash
CLAUDE_PLUGINS_LOADED=1 claude \
  --plugin-dir "$SCRIPT_DIR/claude_plugins/django-stack-claude-plugin" \
  --plugin-dir "$SCRIPT_DIR/claude_plugins/fls-dev-claude-plugin" \
  --plugin-dir "$SCRIPT_DIR/claude_plugins/sdd-claude-plugin" \
  "$@"
```

(The wrapper template uses the `"$SCRIPT_DIR/$FLS_PATH/claude_plugins/…"` form.) `fls-content` is
deliberately **not** added. Multiple `--plugin-dir` flags in one invocation is confirmed supported;
there is no parent-directory auto-discovery mode.

### 9.2 Sentinel rename — 4 sites + a new active-rewrite init step
Rename `$FLS_PLUGIN` → `$CLAUDE_PLUGINS_LOADED` at: (1) root `claude.sh`; (2) the wrapper template;
(3) `.claude/settings.json` `SessionStart` hook (also reword the user-facing "FLS PLUGIN NOT LOADED"
message to be plugin-neutral); (4) the shipped `templates/settings.json` `SessionStart` hook.

Because init is additive/skip-existing, a plain merge **cannot** rewrite the old name in a downstream
project that already has `claude.sh`/`settings.json` with `$FLS_PLUGIN` baked in. The primary init
(`ds`) therefore needs a **new, explicit step** that detects the old sentinel and actively rewrites
both occurrences — mirroring the existing "clean up legacy `CLAUDE.md` line" precedent.

### 9.3 Init split (8 steps → per-plugin)
Today's single 8-step init (`/fls:init`) assumes exactly one plugin. Split so each plugin's init
merges only its own `enabledPlugins` key + permissions, writes only its own `.claude/<plugin>/`
config, and validates its own plugin path. The genuinely shared, multi-plugin artifacts — the
`claude.sh` launcher (all three `--plugin-dir` flags), the `SessionStart` hook, and `.gitignore`'s
`settings.local.json` line — are owned by the **`ds` primary init**; `fls-dev:init`/`sdd:init`
detect-and-skip them. The "ask for FLS path" prompt is asked **once** by whichever init runs first and
persisted for the others (avoid three prompts for the same path). Steps 2–3 (the product config,
including the template-repo path used by `/update_template_repo`, which stays in `fls-dev`) become
cleanly `fls-dev`-only and write to `.claude/fls-dev/config*.md`. `ds`/`sdd` likely need little or no
per-project config beyond the shared bits.

**Config-dir migration.** Because the product plugin is renamed `fls` → `fls-dev`, `/fls-dev:init`
needs an **active-rewrite step** (like the sentinel migration in §9.2) that renames an existing
`.claude/fls/` config dir to `.claude/fls-dev/` (note: **this repo itself already has a `.claude/fls/`
dir**, so the migration runs here on first `/fls-dev:init`, not only in downstream projects) and updates any live
`Bash(.claude/fls/scripts/*.sh:*)` permission entries in `.claude/settings.json` — a plain additive
merge won't move the old directory.

### 9.4 `next.md` dispatch rewrite
Replace the current "strip `/sdd:` prefix, probe two hardcoded `fls-claude-plugin/commands/` dirs"
heuristic (lines 41–53) with a deterministic **keep-prefix, map-to-owning-plugin** lookup:

```
prefix, name = split_once(command, ":")     # "/sdd:do_qa" -> ("sdd", "do_qa")
dir = { "sdd":     "claude_plugins/sdd-claude-plugin/commands/",
        "fls-dev":  "claude_plugins/fls-dev-claude-plugin/commands/",
        "ds":      "claude_plugins/django-stack-claude-plugin/commands/" }[prefix]
target = dir + name + ".md"      # error clearly if it doesn't exist
```

This is deterministic (no probing, no collision risk) and depends on the fully-namespaced `todo.md`
lines (§5.2). It is a within-repo map (fine — cross-repo portability is deferred).

### 9.5 Deletions and reference scrubs

  references to it in `implement_plan.md` (the per-batch review step and the end-to-end review step),
  plus any SDD-README ship-step mention, in the same change — it is still referenced today.
- **Retire `spec_dd/1. next/extract-django-best-practices-claude-plugin/`**: move the whole directory
  to `spec_dd/3. done/` and add a one-line superseded note to its idea file linking to this spec. Do
  **not** delete it.

## 10. Migration / ordering guidance for the plan
- Do the physical `git mv` of a plugin's files and the reference rewrites for those files in the
  **same commit**, so no intermediate state has dangling paths.
- The launcher change (§9.1) is the single highest-risk edit — miss it and every cross-plugin
  reference breaks at once. Verify plugins load before touching anything else downstream.
- Category (c) subagent-type rewrites and category (g) protected-path rewrites are the two largest
  clusters; treat each as a dedicated fan-out unit with a grep-verified before/after count. Remember
  (c) has **three** target prefixes — never a blanket `fls:` → `sdd:`.

## 11. Known pre-existing issues to reconcile (surfaced by research; not caused by the split)
- **Two `.mcp.json` copies.** A root `.mcp.json` exists and differs from the plugin's. Make `ds`'s
  copy authoritative; align or delete the root leftover so the two don't drift/conflict.
- **Agent-memory dirs.** `code-reviewer`'s **live** memory dir is the unprefixed
  `.claude/agent-memory/code-reviewer/` (declared in the agent body, actively populated) — keep that
  path so the move to `ds` loses **no** memory. For `qa-data-helper` (moving into the renamed
  `fls-dev` plugin): its declared in-body path is a stale `qa-data-factory/` that doesn't exist, while
  its real populated content lives in `.claude/agent-memory/fls-qa-data-helper/` (19 files). The plan
  must (a) decide the correct target dir name for a `fls-dev`-scoped agent and (b) `git mv` the real
  content there and fix the agent's declared path to match — reconciling the stale path, the
  pre-existing `fls-`-prefixed dir, and the `fls` → `fls-dev` rename together. Also
  `.claude/agent-memory/fls-code-reviewer/` (5 files) is a pre-existing orphan not referenced by any
  current agent — decide keep-or-remove. *(Note: the idea's claim that memory dirs are simply
  `fls-`-prefixed and need a mechanical migration is incorrect against the current filesystem — verify
  the actual dir-naming derivation in the plan before relying on it.)*
- **Stale README counts.** `fls-claude-plugin/README.md` claims "13 skills / 16 commands"; disk has
  17 skills / 32 command files. Recount from disk when authoring the new per-plugin READMEs.

## 13. Risks & mitigations
- **Silent cross-plugin breakage** (no static validation) → grep checklist per category (§8) with
  before/after counts; post-split smoke test (§14).
- **Bulk find-and-replace error** on `fls:*` agent names → the per-occurrence, three-target-prefix
  rule in §8(c); never a blanket `fls:` → `sdd:` (which would wrongly rewrite `fls:qa-data-helper`).
- **Init ownership races** for shared artifacts → single designated `ds` primary owner + detect-skip
  in the others (§9.3).
- **Memory data loss** → keep `code-reviewer`'s existing unprefixed dir; reconcile `qa-data-helper`'s
  dir against the rename before any `git mv` (§11).
- **Config-dir/rename drift** → the `.claude/fls/` → `.claude/fls-dev/` migration is easy to forget
  because it spans the config dir, `.gitignore`, and `settings.json` permission globs at once (§8 rows
  k/l, §9.3).
- **`fls-content` sync silently writing to the old path** → fix all 6 literals; test the sync command
  once post-move.

## 14. Success criteria

1. **Everything loads.** `./claude.sh` starts a session with all three dev plugins (`ds`, `fls-dev`,
   `sdd`) loaded; `/plugin` (or `claude plugin list --plugin-dir …`) shows all three; `claude plugin
   validate` passes for each of the four plugins.
2. **No dangling references.** A grep of the **live plugin surface** (`claude_plugins/**`, `.claude/**`,
   and the repo-root launcher/config files) confirms **zero** remaining occurrences of:
   `fls:sdd-worker`/`fls:sdd-mechanic` (→ `sdd:…`), `fls:qa-data-helper` (→ `fls-dev:…`),
   `fls-claude-plugin/` (→ `claude_plugins/…`), `fls-claude-plugin/commands/sdd/protected/`,
   `$FLS_PLUGIN`, `Skill(fls:*)`, `mcp__plugin_fls_playwright__*`, and the old single `--plugin-dir`
   launcher line. **Exclude `spec_dd/**` and `docs/**`** from this grep — specs and product docs
   legitimately quote the old names as historical/example text (this spec itself does), so matches
   there are expected, not dangling references.
3. **Each plugin invokes.** A representative command from each plugin runs: `/ds:app_map`,
   `/fls-dev:do_qa` (dry), `/sdd:start`; spawning `sdd:sdd-worker` from an `fls-dev` command succeeds
   (proves the accepted `fls-dev` → `sdd` coupling works at runtime).
4. **SDD workflow runs end-to-end** on a trivial throwaway spec: `/sdd:start` → `/sdd:spec_from_idea`
   → … → generated `todo.md` lists **fully-namespaced** commands and `/sdd:next` dispatches each to
   the correct owning plugin via the new map.
5. **Init is correct on a clean target.** Running the three inits on a fresh checkout produces a
   working `claude.sh` (all `--plugin-dir` flags, `$CLAUDE_PLUGINS_LOADED`), per-plugin
   `.claude/<plugin>/` config dirs, correct `enabledPlugins` keys, and per-plugin `Skill(*)` /

7. **`fls-content` sync works** from its new location (writes to `claude_plugins/fls-content-plugin/…`).
