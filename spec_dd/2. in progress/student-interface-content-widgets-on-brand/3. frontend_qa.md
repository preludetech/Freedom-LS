# Frontend QA: On-brand course content widgets

Manual / Playwright QA for the eight content widgets, exercised through the
`functionality_demo_content_widgets` demo course. The demo course **is** the
fixture set — every widget and every option appears in it.

## Setup

> **Precondition:** the `functionality_demo_content_widgets` demo course must already
> exist in `demo_content/` (created during implementation — plan §5). It is the
> fixture set for every test below. The other demo courses do not exercise the new
> widgets.

1. **Load the demo content** into the QA site (DemoDev) — `content_save` loads the
   whole `demo_content/` directory, including the widgets course:
   ```
   uv run python manage.py content_save ./demo_content DemoDev
   ```
2. **Build CSS** (if widget classes changed):
   ```
   npm run tailwind_build
   ```
3. **Start the server on a random port** (never 8000):
   ```
   PORT=$(.claude/fls/scripts/find_available_port.sh)
   uv run python manage.py runserver $PORT
   ```
   Base URL: `http://127.0.0.1:$PORT`
4. **Student account on DemoDev**: log in as a learner registered for the demo
   course. If none exists, the missing test data (a DemoDev `User` + verified
   `EmailAddress` + `UserCourseRegistration` for the widgets course) is created via
   the `qa-data-helper` agent. Manual fallback if that agent is unavailable: create
   the student and registration with `uv run python manage.py shell` (or the Django
   admin) against the DemoDev site. Log in at
   `http://127.0.0.1:$PORT/accounts/login/`.
5. Navigate to the course: from the student dashboard open the
   **Content Widgets** demo course, then open each topic from the table of contents.
   Topic URLs follow `…/courses/<course-slug>/<topic-slug>/`.

## Theme switching (run every test in BOTH themes)

The theme is baked into the compiled CSS at **build time**: `npm run tailwind_build`
reads `FLS_THEME` from the environment, runs `write_active_theme_css`, and writes the
active-theme tokens into the Tailwind output. Restarting the server then serves the
freshly built CSS (Django does not re-read the theme at runtime). To verify
first_class:
```
# stop the server, then rebuild with FLS_THEME set on the build command:
FLS_THEME=first_class npm run tailwind_build
FLS_THEME=first_class uv run python manage.py runserver $PORT
```
Switch back with `FLS_THEME=default npm run tailwind_build` then restart (the
`FLS_THEME` prefix **must** be on the `tailwind_build` line — a bare rebuild reverts
to `default`). Every visual check below must pass in **both** themes — no broken
layout, readable contrast, brand look flows from tokens (no stray hardcoded colours).

## Topic 1 — Annotation & emphasis

### 1a. Pull quote
- **Go to** Topic 1. Find the pull-quote examples.
- **Expect**: a visually distinct quote block; the quote text; an attribution line
  ("— Name") below the quote; where a source work is given, its title rendered in
  italic `<cite>`; markdown inside the quote (e.g. **bold**) is rendered.
- **Inspect**: where a `cite` URL is given, the `<blockquote>` carries a `cite="…"`
  attribute (it is not navigable, just metadata).
- **first_class**: a decorative oversized quote mark / accent appears and is *not*
  read by screen readers (it is `aria-hidden`); it scales down on a narrow window.
- **Keyboard/SR**: the quote reads as a quotation + attribution; the decorative mark
  is skipped.

### 1b. Equation (KaTeX)
- **Expect**: real typeset math (not raw LaTeX) for the display equation; a labelled
  equation shows its number e.g. "(3)" beside it.
- **Long equation**: shows a horizontal scrollbar; **Tab** moves focus onto the
  equation container (visible focus ring) and arrow keys scroll it — content is never
  clipped.
- **Malformed equation** (intentional in the demo): the page does **not** break; the
  raw LaTeX source shows as a readable fallback.
- **Console**: no JS errors; KaTeX CSS/JS load (Network tab) only on content pages.
- **Currency check**: prose containing `$5` / `R1000` is **not** turned into math.

### 1c. (KaTeX scope) Visit a **non-content** page (e.g. dashboard) and confirm KaTeX
  assets are **not** requested there.

## Topic 2 — Media

### 2a. Video + caption + chapters
- **Expect**: responsive YouTube iframe (16:9, no overflow); a caption under it; a
  "Video chapters" list of timestamp links beneath.
- **Keyboard**: each chapter link is reachable by **Tab** with a visible focus ring;
  activating one opens the YouTube embed at that timestamp; tap targets feel ≥24px.
- iframe has a descriptive `title` (inspect element).

### 2b. Figure / photo + lightbox (regression-critical)
- **Expect**: a captioned figure; where numbered, the caption reads "Figure N".
- **Keyboard (the fix)**: **Tab** to the image — it is now a real **button** with a
  visible focus ring. Press **Enter/Space** → the lightbox opens. Focus moves into
  the dialog (close button). Press **Escape** → it closes **and focus returns to the
  thumbnail**. (Previously the `<img @click>` could not be opened by keyboard.)
- **Dialog semantics** (inspect): `role="dialog"`, `aria-modal="true"`, an accessible
  name.
- **Reduced motion**: with OS "reduce motion" on (or DevTools emulate
  `prefers-reduced-motion: reduce`), the backdrop blur/transition does not animate.

### 2c. Annotated diagram
- **Expect**: the annotated SVG/image inside a figure, plus a visible **text legend**
  (definition list) pairing each letter A/B/C with its meaning. Meaning is available
  without relying on colour/position alone.

### 2d. Image grid + stacked figures
- **Expect**: a static thumbnail **grid** (no carousel, no auto-advance). It reflows
  to a single column on a narrow window.
- **`columns` option**: the demo shows grids with `columns="2"`, `"3"`, `"4"` — each
  lays out the stated number of columns on a wide window. A grid authored with an
  invalid `columns` value falls back to the default 3-column layout (does not break).
- **Keyboard**: each thumbnail is a focusable button; Enter opens that image in the
  lightbox; Escape returns focus to the thumbnail.
- Below it, the **stacked captioned figures** pattern renders as a vertical list of
  figures.

## Topic 3 — Structured content

### 3a. Data table
- **Expect**: a semantic table with a visible **caption**; header row; brand styling.
- **Inspect (scope)**: the comparison table's header cells carry `scope="col"` and
  (for row-header tables) the first cell of each row is `<th scope="row">` — confirm
  the `scope` attributes survive the pipeline (not stripped by the sanitiser).
- **Narrow window / mobile**: the table sits in a horizontally **scrollable** wrapper
  — content is never clipped (no `overflow:hidden` cut-off), columns are **not**
  auto-stacked, comparison columns stay aligned.
- **Keyboard**: **Tab** focuses the scroll wrapper (visible focus ring); it scrolls
  horizontally. The wrapper has an accessible label.
- Confirm this is the **content** table, not the educator/panel `c-data-table` (no
  floating mobile row-label overlay behaviour from `scrollTableLabels`).

### 3b. Code / log block
- **Expect**: monospace `<pre><code>` with a filename/title chrome bar; optional
  language label shown; brand terminal look (especially in first_class). **No** copy
  button, **no** syntax-highlight colours.
- **Long line (default)**: horizontal scroll; **Tab** focuses the scroll container
  (visible ring); content not clipped.
- **`wrap` example**: long lines wrap instead of scrolling.
- Code containing escaped `&lt;`/`&gt;` renders the literal `<`/`>` characters
  correctly (not stripped, not interpreted as tags).

## Cross-cutting checks (all topics)

- **No console errors** on any topic page; Alpine initialises (no "component not
  found").
- **Keyboard sweep**: Tab through each topic — every interactive element (lightbox
  triggers, grid thumbnails, chapter links, scroll containers) is reachable, in a
  sensible order, with a visible focus ring; focus is never trapped or lost.
- **Both themes**: repeat the visual checks under `FLS_THEME=default` and
  `FLS_THEME=first_class`; layout and contrast hold in both.
- **No regressions**: open an existing demo course (e.g.
  `functionality_demo_standard_markdown`) and confirm `c-callout`, `c-picture`,
  `c-youtube`, tables, and code blocks still render correctly.
