# Spec: Theme resolver + finalised default-theme tokens (Phase 1 of 5)

## Why

This is **Phase 1** of the multi-phase Themable Freedom LS effort. It lands the foundational plumbing that every later phase consumes: the theme-resolution function, the on-disk default theme, the finalised role-token contract, the rewritten Tailwind build inputs, and a one-pass rename of legacy token classes throughout FLS templates.

After Phase 1, the site looks **visually identical to before** — no rebrand, no new theme. What changes is the *machinery underneath*: tokens live in a sparse `default` theme directory, classnames follow the new `on-*` and `error`/`on-surface` contract, and a single env var (`FLS_THEME`) selects the active theme.

For the holistic context (all five phases, the canary `first_class` theme, the long-term theming goals), see the master folder: `../themable-implementations-master-decomposed-into-phases/`.

## Scope of this phase

**In:**

- A pure-Python theme resolver (`freedom_ls/base/theming.py`) and its tests.
- A new sparse `freedom_ls/themes/default/` theme directory containing `theme.css` (finalised tokens) and `theme.md` (manifest).
- Strip the `@theme` block from `tailwind.components.css`; update its component classes to reference the renamed tokens.
- Rewrite `tailwind.input.css` to import the default theme tokens, then component classes, in the right order.
- Wire `FLS_THEME` resolution into `config/settings_base.py` and validate failure modes.
- Repoint `parse_tailwind_colors` callsite to the resolved theme's `theme.css`.
- One mechanical token-class rename pass across all FLS templates (~22 templates, ~290 references) per the rename table below.

**Out of this phase (covered later):**

- A second / canary theme (`first_class`) — Phase 2 (Tier-1 tokens), Phase 3 (Tier-2 components), Phase 4 (Tier-3 templates).
- Any theme-author documentation (install guide refresh, new theming guide, resource-doc updates) — Phase 5.
- Any `templates/` or component-class content inside the default theme — explicitly excluded by the "themes are sparse" principle: defaults stay in the main project.

## Goals & success criteria

1. **A reader of `tailwind.components.css` no longer sees an `@theme` block** — tokens live exclusively in `freedom_ls/themes/default/static/themes/default/theme.css`.
2. **`FLS_THEME=default` works** out of the box; an unknown slug raises `ImproperlyConfigured` at startup with a message naming the slug and the searched dirs.
3. **All FLS templates use the new token classnames** (`text-on-surface`, `text-error`, `text-on-success`, etc.). No occurrences of `*-foreground`, `*-danger`, or `*-bold` remain anywhere in `freedom_ls/`.
4. **`uv run pytest`, `uv run python manage.py check`, `npm run tailwind_build`** are all green.
5. **Visual smoke check** on the dev server (home, login, course list, course topic, educator dashboard, account pages) shows **no regression** vs. pre-phase appearance.
6. **Email colour parser** continues to return sensible values (`EMAIL_COLOR_PRIMARY`, `EMAIL_COLOR_FOREGROUND`) drawn from the new default `theme.css`.
7. **Base-layer headings (`h1`–`h4`) apply `font-display`** in `tailwind.components.css` so themes redeclaring `--fls-font-display` (e.g. First Class → Outfit in Phase 2) actually drive the heading typeface. Under the default theme, `--fls-font-display` is aliased to `--fls-font-sans`, so this is a visual no-op at Phase 1.

## Architecture established by this phase

### Token contract (the public role surface)

Role names ship in this phase and become the public token contract for every later phase:

| Role | Purpose | Pair (`on-…`) |
| --- | --- | --- |
| `primary` | Main brand colour. Primary CTAs, links, brand accents. | `on-primary` |
| `secondary` | Supporting brand colour. Secondary buttons, less-loud emphasis. | `on-secondary` |
| `accent` | Sparingly-used highlight. Decorative bursts, badges. | `on-accent` |
| `success` | Completion, positive state. | `on-success` |
| `warning` | Caution, important non-error notice. | `on-warning` |
| `error` | Destructive action, validation failure. | `on-error` |
| `info` | Informational notice (neutral, non-warning). | `on-info` |
| `surface` | Primary page background. | `on-surface` |
| `surface-2` | Recessed/raised surface (panels, table-row hover, header strip). | `on-surface` |
| `border` | Default border colour for surfaces, inputs, dividers. | — |
| `muted` | De-emphasised text (labels, helper text, secondary copy). | — |
| `focus-ring` | Focus outline (defaults to alias of `primary`). | — |

Plus seven auto-derived hover tokens (`primary-hover`, `secondary-hover`,
`accent-hover`, `success-hover`, `warning-hover`, `error-hover`,
`info-hover`) computed via `color-mix(in oklch, …)` against two FLS-namespace
mix knobs:

- `--fls-hover-mix-color` (default `white`) — the colour mixed in to brighten
  light-mode hovers. Dark themes can flip to `black` to invert direction
  without redeclaring every `*-hover` token.
- `--fls-hover-mix-amount` (default `12%`) — how strongly the mix is applied.

A theme that overrides only `--color-primary` automatically gets a coherent
`--color-primary-hover`. A theme that wants a non-mix hover (e.g. an explicit
brand hover swatch) overrides the `*-hover` token directly.

Plus shape tokens (`--fls-radius-sm/-md/-lg/-pill`, mapped via `@theme inline` to `--radius-*`) and type tokens (`--fls-font-sans`, `--fls-font-display`, mapped to `--font-sans` and `--font-display`).

### Bold → hover variants

The pre-existing `primary-bold`, `success-bold`, `danger-bold` tokens are
**dropped** as a token-shape (no more `*-bold` family). They are replaced by
the auto-derived `*-hover` tokens described in §Token contract above:
component classes consume `hover:bg-primary-hover` /
`hover:bg-success-hover` / `hover:bg-error-hover` etc., and the values
fall out of `color-mix` against the base role.

This keeps hovers theme-controlled (a sparse downstream theme that only
overrides `--color-primary` still gets a coherent `--color-primary-hover`)
while keeping the public class surface compatible with Tailwind utilities.
Chip backgrounds stay as opacity modifiers (`bg-primary/15`).

### Theme directory shape

A theme is a sparse directory. **Only `static/themes/<slug>/theme.css` is conventionally required** — everything else is optional:

```
themes/<theme_slug>/
    static/
        themes/<theme_slug>/
            theme.css            # required: token bundle
            logo.svg             # optional brand assets
    templates/                   # optional (Tier-3 overrides — out of scope this phase)
        cotton/                  # optional
        partials/                # optional
        _base.html               # optional
    theme.md                     # human-readable manifest
```

The `static/themes/<theme_slug>/` namespacing prevents asset collision between themes when collected.

The default theme that ships in this phase is **just `theme.css` + `theme.md`** — no templates, no component classes — to demonstrate that a Tier-1-only theme is valid.

### Theme selection

```python
FLS_THEME = env("FLS_THEME", default="default")
FLS_THEMES_DIRS = [BASE_DIR / "themes", FREEDOM_LS_PACKAGE_DIR / "themes"]
```

Resolution: walk `FLS_THEMES_DIRS` in order, use the first directory whose name matches `FLS_THEME`. Downstream `themes/` comes first so a downstream `default` shadows the FLS one cleanly.

The resolution function lives at `freedom_ls/base/theming.py` and is called from a settings helper that:

- Prepends the resolved `templates/` dir to `TEMPLATES[0]["DIRS"]` (only if it exists).
- Prepends the resolved `static/` dir to `STATICFILES_DIRS`.
- Also surfaces every *inactive* shipped theme's `static/` dir (after the active one) so `/static/themes/<slug>/theme.css` resolves regardless of which theme is currently active. Path namespacing under `static/themes/<slug>/` prevents collisions between themes; a parent that already contributed a slug (because of `FLS_THEMES_DIRS` shadowing) is not re-added.

Failure modes:

- `FLS_THEME` slug doesn't resolve in any `FLS_THEMES_DIRS` entry → raise `ImproperlyConfigured` at startup. Fail loud.
- `FLS_THEMES_DIRS` not set → default to `[FREEDOM_LS_PACKAGE_DIR / "themes"]`.

### Tailwind build ownership

This phase rewrites the FLS repo's own `tailwind.input.css` to the same shape downstream projects will eventually copy from the (Phase-5) install guide. The import order is the central architectural decision — it makes Phase 3's Tier-2 cascade work without `!important`:

```css
@import "tailwindcss";

@source "./freedom_ls/**/templates/**/*.html";
@source "./freedom_ls/themes/*/templates/**/*.html";

@import "./freedom_ls/themes/default/static/themes/default/theme.css";
@import "./tailwind.components.css";

/* Phase 2 will append the active-theme import here. */

@theme {
    /* repo-level overrides go here, optional */
}
```

`research_tailwind_v4_theming.md` §6 (in the master folder) flags two pitfalls to mention in the Phase-5 docs:

- `@source` honours `.gitignore`. If FLS is installed under a path excluded by an ancestor `.gitignore` (`.venv/`, `node_modules/`), the glob silently skips. Document the symlink / asset-pipeline-copy workaround.
- New cotton class names not present at build time are not generated. Themes that introduce new utility usages must be in the `@source` glob.

These pitfalls don't gate Phase 1 shipping — they are install-guide concerns for Phase 5.

## Migration scope (breaking changes this phase introduces)

### Token rename — affects all consumers

| Old class | New class |
| --- | --- |
| `text-foreground`, `bg-foreground`, etc. | `text-on-surface`, `bg-on-surface` |
| `text-danger`, `bg-danger`, `border-danger` | `text-error`, `bg-error`, `border-error` |
| `text-danger-bold` | `text-error` |
| `text-success-bold` | `text-success` |
| `text-primary-bold`, `bg-primary-bold` | `text-primary`, `bg-primary` |
| `text-on-primary` (when paired with `bg-success` / `bg-error`) | `text-on-success`, `text-on-error` |

Token usage in FLS templates today (from grep of `freedom_ls/**/*.html`):

- 75 × `text-muted` (no rename)
- 25 × `border-border` (no rename)
- 20 × `text-primary` (no rename)
- 20 × `bg-surface-2` (no rename)
- 19 × `text-foreground` → `text-on-surface`
- 17 × `bg-surface` (no rename)
- 15 × `text-danger` → `text-error`
- 13 × `text-success` (no rename)
- 10 × `border-danger` → `border-error`
- 10 × `bg-danger` → `bg-error`
- 7 × `text-warning` (no rename)
- 7 × `text-danger-bold` → `text-error`
- … plus a long tail of `*-bold`, `border-warning`, etc.

Total: ~22 templates touched, ~290 individual token references to update. All within the FLS repo. Downstream **theme.css** files written against the old token names will need updating; that is the cost of the rename.

### Component class names — public API now

`btn`, `btn-primary`, `btn-outline`, `btn-success`, `btn-danger` (becomes `btn-error`), `chip`, `chip-primary`, `chip-warning`, `chip-success`, `chip-danger` (becomes `chip-error`), `chip-xs`, `surface` are now the documented public API. The class definitions live in `tailwind.components.css` in the FLS repo root. Phase 5 documents them; this phase ships the renames.

## Decisions (locked in by this phase)

1. **Drop the `-bold` token family; replace with auto-derived `-hover` tokens via `color-mix`.** Hovers stay theme-controlled (override `--color-primary` and `--color-primary-hover` updates automatically) without requiring downstream Tier-2 component re-opens. Two mix knobs (`--fls-hover-mix-color`, `--fls-hover-mix-amount`) let dark themes invert direction without redeclaring every hover token.
2. **Rename `danger` → `error`** to match the user-specified role list.
3. **Rename `foreground` → `on-surface`** for consistency with the `on-*` pattern. Keep `muted` as-is.
4. **`focus-ring` defaults to alias of `primary`** rather than a separate value.
5. **Keep `secondary` as a real third brand colour**, not a repurposing of the old `primary-bold` hover state. Default `#475569` slate.
6. **`info` is new** (default `#0EA5E9` sky); not currently in the codebase.
7. **`accent` is new** (default `#F59E0B` amber); not currently in the codebase.
8. **No backward-compatibility shim** for the old token names. Single breaking change, single migration.
9. **Defaults stay in the main project; themes are sparse.** Component classes stay in `tailwind.components.css`; generic primitives stay in `base/templates/cotton/*` and `base/templates/partials/*`; `_base.html` stays in `base/templates/`. The default theme is just `theme.css` + `theme.md`.
10. **Fail-loud on bad `FLS_THEME`** at startup.

## Out of scope (deferred)

- A canary `first_class` theme (Tier-1 tokens, Tier-2 components, Tier-3 templates) — Phases 2–4.
- Any theme-author documentation — Phase 5.
- Per-site theming on a single deployment, per-site `data-theme` swaps, full 50–950 colour ramps, density/spacing scale tokens, theme inheritance — all deferred per master spec.
