# Pluggable Icon Backend

## Overview

Replace the hardcoded Heroicons rendering pipeline with a pluggable icon backend system. Downstream developers can swap the entire icon set by changing a single Django setting, override individual icons, or provide a fully custom icon backend.

## Why this matters

FLS is designed to be embedded into other Django projects. Different products have different visual identities, and icons are a core part of that identity. Currently, every FLS installation looks like it uses Heroicons. This feature lets downstream developers match their icon style without forking templates.

## Current architecture

- Templates use `<c-icon name="success" />` with semantic names
- `ICONS` dict in `base/icons.py` maps 47 semantic names to Heroicon-specific names
- `cotton/icon.html` renders via `django-heroicons` template tags (`{% heroicon_outline %}`, etc.)
- `force=True` bypasses the registry to use raw Heroicon names (unused in production templates)

The semantic indirection layer is already correct. The problem is that both the name mapping AND the rendering are hardcoded to Heroicons.

## Design

### Icon data source: Iconify JSON

Icon SVG data comes from Iconify JSON npm packages (`@iconify-json/*`). These packages contain SVG path data for icon sets in a standardised format.

**Why Iconify JSON:**
- Single standardised format covering 200+ icon sets
- Once the rendering code is written, adding a new icon set is just `npm install @iconify-json/whatever`
- No dependency on per-set Django packages
- The project already has an npm build step for Tailwind, so `node_modules/` is already present

**Packages to install:**
- `@iconify-json/heroicons` (default)
- `@iconify-json/lucide`
- `@iconify-json/tabler`
- `@iconify-json/ph` (Phosphor)

### Settings

Three new Django settings control the icon system:

#### `FREEDOM_LS_ICON_SET`

Selects which icon set to use. Default: `"heroicons"`.

```python
# settings.py
FREEDOM_LS_ICON_SET = "lucide"  # Switch entire icon style
```

Valid values: `"heroicons"`, `"lucide"`, `"tabler"`, `"phosphor"`, or any Iconify set prefix for which a mapping dict and npm package are available.

#### `FREEDOM_LS_ICON_OVERRIDES`

Dict of per-icon overrides. Merged on top of the active set's mapping. Allows fine-tuning individual icons without swapping the whole set.

```python
# settings.py
FREEDOM_LS_ICON_OVERRIDES = {
    "success": "party-popper",   # Override just this one icon
    "achievement": "sparkles",
}
```

#### `FREEDOM_LS_ICON_BACKEND`

Dotted path to a custom backend class. When set, this takes full control of icon resolution and rendering, bypassing the built-in Iconify pipeline entirely.

```python
# settings.py
FREEDOM_LS_ICON_BACKEND = "myproject.icons.MyIconBackend"
```

**Why three layers:** Most downstream developers just want to swap icon sets (layer 1). Some want to tweak a few icons (layer 2). A rare few need complete control (layer 3). Each layer is progressively more powerful without forcing complexity on simpler use cases.

### Pre-built mapping dicts

Ship mapping dicts for four icon sets. Each dict maps FLS's semantic names to that set's icon names.

**Sets:**
1. **Heroicons** (default) -- clean, minimal outline style
2. **Lucide** -- most popular open-source set, consistent 2px stroke
3. **Tabler** -- largest free outline set (5,900+ icons)
4. **Phosphor** -- soft, rounded, friendly aesthetic with multiple weights

Each mapping must cover all semantic names currently in `ICONS`:

| Category | Semantic names |
|---|---|
| Navigation | next, previous, home, expand, collapse, menu_open, menu_close, dropdown |
| Status | success, error, warning, info, in_progress, complete, locked, not_started, repeatable |
| Actions | check, close, retry, download, more_options, settings |
| Content types | reading, quiz, assessment, section |
| User/system | user, notifications, deadline, achievement, loading |
| Data display | sort_asc, sort_desc, sort_neutral, boolean_true, boolean_false |
| Educator | deadline_override |

Mappings are stored as Python dicts within the icon system module, not in external config files. This keeps them version-controlled and testable.

### Variant handling

**Decision:** Each icon set mapping declares which variants it supports. If a template requests an unsupported variant, a Django system check emits a warning.

**Why warnings (not silent fallback):** Silent fallback creates visual inconsistency that's hard to debug. Developers should know at startup that their chosen icon set doesn't support a variant they're using, so they can make an explicit decision about what to do.

Current variant usage in templates:
- `outline` (default, used everywhere)
- `solid` (used in a few places)
- `mini` and `micro` (rarely used)

Variant support by set:

| Set | Variants |
|---|---|
| Heroicons | outline, solid, mini, micro |
| Lucide | outline only |
| Tabler | outline, filled |
| Phosphor | thin, light, regular, bold, fill, duotone |

Each mapping declares a variant translation dict, e.g.:

```python
LUCIDE_VARIANTS = {
    "outline": None,  # default, no suffix needed
    # solid, mini, micro: not declared = system check warning
}

PHOSPHOR_VARIANTS = {
    "outline": "regular",
    "solid": "fill",
    # mini, micro: not declared = system check warning
}
```

### Rendering pipeline

Replace `django-heroicons` template tags with a custom rendering pipeline that reads Iconify JSON and outputs inline SVG.

**How it works:**

1. At startup (or first use), read the Iconify JSON file for the active icon set from `node_modules/@iconify-json/<set>/icons.json`
2. Parse the JSON to extract SVG path data, viewBox, and other attributes
3. Cache the parsed data in memory (these files are read once)
4. When `<c-icon name="success" variant="outline" />` is rendered:
   a. Resolve semantic name via mapping dict + overrides
   b. Look up the resolved name (with variant) in the cached Iconify data
   c. Output an inline `<svg>` element with the path data, classes, and ARIA attributes

**The `cotton/icon.html` template** will be updated to use the new rendering approach instead of `{% heroicon_* %}` tags. The template's public API (`name`, `variant`, `class`, `aria_label`) remains unchanged.

**Why inline SVG:** Same approach as `django-heroicons` uses today. No external requests, CSS-styleable (colour via `currentColor`), works with Tailwind size classes.

### Removal of `force` parameter

The `force` parameter is removed from `<c-icon />`.

**Why:** No production templates use it. It bypassed the semantic layer to use raw Heroicon names, which would break on icon set swap. The semantic registry + `FREEDOM_LS_ICON_OVERRIDES` covers the use case (need a specific icon? add it to overrides).

**Migration:** Remove the `force` parameter from `cotton/icon.html`, the `icon_from_name` template filter, and the documentation example. Update tests accordingly.

### Django system checks

Validate the icon configuration at startup:

1. **All semantic names resolve:** Every key in the active mapping (merged with overrides) maps to an icon name that exists in the Iconify JSON data for the active set
2. **Variant support:** Warn if any template uses a variant not supported by the active icon set's mapping
3. **Iconify data available:** Error if the required `@iconify-json/*` npm package is not installed (JSON file not found in `node_modules/`)
4. **Override validity:** Every icon name in `FREEDOM_LS_ICON_OVERRIDES` exists in the Iconify JSON data

**Why system checks:** Catches typos, missing icons, and missing npm packages at startup rather than at runtime when a user hits a broken page.

### Custom backend escape hatch

For downstream projects that want complete control, `FREEDOM_LS_ICON_BACKEND` points to a class that implements:

```python
class IconBackend:
    def resolve(self, semantic_name: str) -> str:
        """Map semantic name to icon identifier."""
        ...

    def render(self, icon_name: str, variant: str, css_class: str, aria_label: str) -> str:
        """Return HTML string (inline SVG or other markup)."""
        ...
```

When a custom backend is set, the built-in Iconify pipeline, mappings, and overrides are all bypassed. The backend has full responsibility.

**Why:** Some downstream projects may use a proprietary icon system, a CDN-based approach, or icon fonts. This escape hatch means FLS never blocks them.

### Dependency changes

**Add:**
- npm: `@iconify-json/heroicons`, `@iconify-json/lucide`, `@iconify-json/tabler`, `@iconify-json/ph`

**Remove:**
- Python: `heroicons[django]` (the `django-heroicons` package is no longer needed once rendering moves to Iconify JSON)

**Why remove django-heroicons:** Keeping it alongside the new system would mean two rendering paths. Since Heroicons data is available via `@iconify-json/heroicons`, there's no reason to keep the Django package.

## Success criteria

1. `<c-icon name="success" />` renders correctly with each of the four icon sets
2. Changing `FREEDOM_LS_ICON_SET` in settings swaps all icons across the entire application
3. `FREEDOM_LS_ICON_OVERRIDES` correctly overrides individual icons on top of any set
4. `FREEDOM_LS_ICON_BACKEND` allows a custom class to fully control icon rendering
5. Django system checks catch: missing Iconify data, unmapped semantic names, invalid overrides, unsupported variant usage
6. All existing tests pass (updated for new rendering)
7. No visual regression when using `FREEDOM_LS_ICON_SET = "heroicons"` (default should look identical to current behaviour)
8. The `force` parameter is fully removed with no remaining references
9. `django-heroicons` is fully removed as a dependency
