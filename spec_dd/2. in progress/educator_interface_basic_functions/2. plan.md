# Plan: Loading Indicators on Form Submissions

## Context

When forms are submitted via HTMX in the educator interface (Create Cohort, Edit Cohort, Delete Cohort), there is no visual feedback that the submission is in progress. This plan adds loading indicators to submit buttons so users know their action is being processed.

## Approach

HTMX has built-in support for loading indicators via the `htmx-request` class. When an HTMX request is in flight, HTMX adds the `htmx-request` class to the element that triggered the request (or a specified indicator element). We will use this to show a spinner and disable the button during submission.

The existing `<c-loading-indicator>` cotton component uses `htmx-indicator` (which requires `hx-indicator` attribute targeting). For buttons, a simpler approach is better: use the `htmx-request` class that HTMX automatically adds to the requesting element, combined with CSS to swap button text for a spinner.

## Tasks

### Task 1: Add a `loading` prop to the `<c-button>` component

**File:** `freedom_ls/base/templates/cotton/button.html`

**Skills:** `template`, `frontend-styling`, `icon-usage`

Add two new props to `<c-button>`:
- `loading` (boolean): When set, the button renders with two states controlled by CSS using the `htmx-request` class
- `loading_text` (string, default "Saving..."): The text shown during loading state. Use "Deleting..." for delete buttons.

Behavior when `loading="true"`:
- **Default state:** Normal button content (visible), spinner (hidden)
- **During request:** Normal button content (hidden), spinner + loading_text (visible)
- The button should also be disabled during the request to prevent double-submission

**Important:** The `htmx-request` class is added by HTMX to the element that triggers the request. Since forms use `hx-post` on the `<form>` element, the `htmx-request` class will be on the form. The CSS selectors use ancestor-based matching (`.htmx-request .htmx-hide-on-request`) so this works correctly - the form is the ancestor of the button's inner spans.

Pseudocode for the template changes:

```html
<c-vars ... loading="" loading_text="Saving..." />

{% if loading %}
  {# Wrap the button content to toggle visibility #}
  <span class="htmx-hide-on-request">
    {% if icon_left %}...{% endif %}{{ slot }}{% if icon_right %}...{% endif %}
  </span>
  <span class="htmx-show-on-request">
    <c-icon name="loading" class="size-5 animate-spin" /> {{ loading_text }}
  </span>
{% else %}
  {# Normal button rendering - unchanged #}
  {% if icon_left %}...{% endif %}{{ slot }}{% if icon_right %}...{% endif %}
{% endif %}
```

The `loading` prop is opt-in so existing buttons are unaffected.

**TDD:** Write a test that renders the `<c-button loading="true">` component and verifies both the default content span and the loading span are present in the output.

### Task 2: Add CSS utilities for HTMX request state

**File:** `tailwind.components.css`

**Skills:** `frontend-styling`

Add utility classes that leverage the `htmx-request` class HTMX automatically adds to requesting elements:

```css
/* Inside @layer components */

/* Hide element when an ancestor (the form) has htmx-request class */
.htmx-request .htmx-hide-on-request {
    display: none;
}

/* Show element only when an ancestor has htmx-request class (hidden by default) */
.htmx-show-on-request {
    display: none;
}
.htmx-request .htmx-show-on-request {
    display: inline-flex;
    align-items: center;
    gap: 0.5rem;
}
```

Note: Double-submission prevention is handled by HTMX's `hx-disabled-elt` attribute on the form element, not by CSS. This is more reliable than CSS-only approaches.

**TDD:** No unit test needed for CSS. This will be verified via the frontend QA plan.

After adding CSS, run `npm run tailwind_build`.

### Task 3: Apply loading indicators when building modal form templates

**Skills:** `template`

**Note:** This task is executed as part of the main spec's template creation tasks (FormPanelAction modal, Delete confirmation modal, Edit modal). It is not a standalone task — it defines requirements that MUST be followed when those templates are built.

When creating `modal_form.html` in `panel_framework/templates/panel_framework/partials/`, all submit buttons must use the `loading` prop:

```html
{# Save button #}
<c-button type="submit" variant="primary" loading="true">Save</c-button>

{# Save and add another button #}
<c-button type="submit" variant="outline" loading="true" name="action" value="save_and_add">Save and add another</c-button>

{# Delete button (in delete confirmation modal) #}
<c-button type="submit" variant="danger" loading="true" loading_text="Deleting...">Delete</c-button>
```

The form element must include `hx-disabled-elt="find button[type='submit']"` to prevent double-submission at the HTMX level.

**TDD:** When creating the modal form template, write a test that renders it and verifies:
- Submit buttons contain the `htmx-hide-on-request` and `htmx-show-on-request` spans (confirming loading is active)
- The form element includes `hx-disabled-elt`
