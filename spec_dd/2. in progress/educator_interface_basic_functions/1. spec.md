# Specification: Educator Interface Basic Functions

## Overview

This spec covers four interconnected features for the educator interface, all built on a reusable framework extracted into a new `panel_framework` app:

1. **Refactor** - Extract reusable panel/view infrastructure from `educator_interface/views.py` into a new `panel_framework` app
2. **PanelActions** - A reusable action system for panels, demonstrated with "Create Cohort"
3. **Tabs** - Tab containers for organizing panels within instance views, demonstrated on CohortInstanceView
4. **Edit & Delete Instance** - Reusable edit/delete functionality on instance detail panels, demonstrated on Cohort

---

## 1. Refactoring: New `panel_framework` App

### Why

The `educator_interface/views.py` file is ~1200 lines and contains both reusable framework code (Panel, DataTable, DataTablePanel, InstanceDetailsPanel, InstanceView, ListViewConfig, etc.) and educator-specific configuration (CohortDataTable, StudentDetailsPanel, etc.). Future interfaces (e.g., TA interface) will need the same framework classes without depending on the educator_interface app.

### What

Create a new Django app `freedom_ls/panel_framework/` containing the reusable base classes. The educator_interface app will import from panel_framework and only contain its own configuration.

### Structure

```
freedom_ls/panel_framework/
    __init__.py
    apps.py
    panels.py          # Panel, DataTablePanel, InstanceDetailsPanel
    tables.py           # DataTable
    views.py            # InstanceView, ListViewConfig, PanelGetter, interface_view
    actions.py          # PanelAction, FormPanelAction (new, see section 2)
    tabs.py             # Tab, TabbedInstanceView (new, see section 3)
    templates/panel_framework/
        partials/
            panel_container.html
            instance_details_panel.html
            list_view.html
            tab_container.html      (new)
            modal_form.html         (new)
```

The educator_interface app retains:
- Its own views.py with educator-specific DataTables, Panels, InstanceViews, and ListViewConfigs
- Its own templates (course_progress_panel.html, data-table-cells/*)
- Its own urls.py
- Its own forms.py (new)

### Success Criteria

- All existing educator interface functionality works identically after refactoring
- No code duplication between panel_framework and educator_interface
- panel_framework has no imports from educator_interface
- Existing tests pass without modification

## 2. PanelActions

### Why

Panels currently display data but offer no way to trigger actions. Educators need to create, edit, and manage records directly from the interface. A reusable action system means we define the pattern once and apply it to any panel across any interface.

### Design

#### PanelAction (base class)

```python
class PanelAction:
    label: str          # Button text
    variant: str        # Button variant (e.g., "primary", "outline", "danger")

    def render(self, request, panel, base_url) -> str:
        """Render the action button HTML"""
```

- Each Panel gets a `get_actions()` method returning a list of `PanelAction` instances (empty by default)
- Actions render as buttons at the bottom of the panel container
- The `panel_container.html` template is updated to render action buttons if any exist

#### FormPanelAction (subclass)

```python
class FormPanelAction(PanelAction):
    form_class: type[forms.Form]        # The Django form class
    form_title: str                      # Modal title
    success_url: str | None              # Where to redirect on success (None = stay)

    def get_form(self, request, instance) -> forms.Form
    def get_form_url(self, base_url) -> str
    def handle_submit(self, request, instance) -> HttpResponse
    def render(self, request, panel, base_url) -> str:
        """Render button + modal with form"""
```

- Clicking the button opens a modal containing the form
- The modal uses the existing `<c-modal>` cotton component
- The form renders using the existing `partials/form.html` partial
- Form submission is via HTMX (`hx-post`)
- On validation error: return HTTP 422 with re-rendered form (modal stays open, errors displayed)
- On success: behavior depends on the action configuration (redirect or refresh)

### Create Cohort Implementation

A `CreateCohortAction` (FormPanelAction subclass) is added to the cohorts list view panel.

#### Form

A `CohortForm` (ModelForm) in `educator_interface/forms.py`:
- Model: `Cohort`
- Fields: `["name"]`
- Site is automatically set by `SiteAwareModel.save()`

#### Two submit buttons

The form has two submit options:

1. **"Save and add another"** - Submitted via HTMX
   - On success: the datatable panel reloads (HTMX swap), the modal stays open, the form clears
   - The submit button sends a specific value (e.g., `name="action" value="save_and_add"`) so the view knows which behavior to use
   - Response: returns the re-rendered empty form (modal stays open) + triggers an HTMX event to reload the datatable

2. **"Save"** - Submitted via HTMX
   - On success: full page redirect to the new cohort's detail page
   - Response: HX-Redirect header pointing to the new cohort's URL

#### Error handling

- Form validation errors (e.g., duplicate cohort name within site) render inline in the modal
- HTTP 422 status on validation failure so HTMX processes the response

### Success Criteria

- PanelAction and FormPanelAction are generic and reusable - no cohort-specific logic in the base classes
- Create Cohort modal opens, validates, and submits correctly
- "Save and add another" keeps modal open, clears form, refreshes datatable
- "Save" redirects to new cohort detail page
- Duplicate cohort names within the same site show a validation error
- Form errors display correctly in the modal

## 3. Tabs

### Why

Instance views can have many panels. Grouping them into tabs reduces visual clutter and lets users focus on what's relevant. The CohortInstanceView has 4 panels and the course progress panel is complex enough to deserve its own tab.

### Design

#### Tab class

```python
class Tab:
    label: str                              # Tab display name
    panels: dict[str, type[Panel]]          # Panel registry for this tab
```

#### TabbedInstanceView

InstanceView is extended to support tabs:

- An InstanceView can define `tabs: list[Tab]` instead of (or alongside) `panels`
- If tabs are defined, the view renders a tab bar followed by the active tab's panels
- Tab navigation is URL-driven (query parameter or path segment) so tabs are bookmarkable

#### Tab switching behavior (Hybrid)

- The first tab's panels render immediately on page load
- Other tabs lazy-load their panels via HTMX on first click
- Once a tab's content has been loaded, it remains in the DOM (hidden via Alpine.js) so switching back is instant
- This combines good initial performance with smooth subsequent navigation

#### Tab rendering

- A new `tab_container.html` template renders the tab bar and content areas
- Tab bar uses standard button/link styling consistent with the rest of the interface
- Active tab is visually distinguished
- Each tab's content area has a unique ID for HTMX targeting

### CohortInstanceView Tabs

Update CohortInstanceView to use two tabs:

| Tab | Label | Panels |
|-----|-------|--------|
| 1 | "Course Progress" | `course_progress` |
| 2 | "Details" | `details`, `courses`, `students` |

The first tab ("Course Progress") loads immediately. The "Details" tab lazy-loads on first click.

### Success Criteria

- Tab framework is generic and reusable - any InstanceView can define tabs
- CohortInstanceView displays two tabs with correct panel grouping
- First tab loads immediately, second tab lazy-loads on click
- Tab state persists when switching back (no re-fetch)
- Individual panels within tabs can still be independently reloaded via HTMX (e.g., datatable sorting/pagination)
- Tabs work correctly with the URL routing system (panel URLs still resolve correctly within tabs)

## 4. Edit & Delete Instance

### Why

Educators need to modify and remove records. Building these as reusable patterns on InstanceDetailsPanel means any detail view can gain edit/delete capabilities with minimal configuration.

### 4a. Edit Instance

#### Design

- The `InstanceDetailsPanel` gains an optional edit capability
- Configured by providing a `form_class` on the panel subclass and setting `editable = True`
- When editable, an "Edit" button appears in the panel header (or as a panel action)
- Clicking "Edit" opens a modal with a pre-filled form (using the existing `<c-modal>` component)
- Form submission is via HTMX (`hx-post`)
- On success: modal closes, the details panel reloads via HTMX to show updated data
- On validation error: HTTP 422, form re-renders with errors in the modal

#### Edit Cohort

- `CohortDetailsPanel` sets `editable = True` and `form_class = CohortForm`
- The same `CohortForm` used for creation works for editing (standard ModelForm behavior)

#### Success Criteria

- Edit button appears on CohortDetailsPanel
- Modal opens with current cohort data pre-filled
- Successful edit updates the details panel without full page reload
- Validation errors display correctly in the modal
- Edit pattern is reusable for any InstanceDetailsPanel subclass

### 4b. Delete Instance

#### Design

- A reusable `DeleteAction` (PanelAction subclass) that can be added to any instance view
- Renders a "Delete" button with `danger` variant
- Clicking opens a confirmation modal showing:
  - A warning message: "Are you sure you want to delete {instance}?"
  - A summary of related records that will be deleted (e.g., "This will also delete: 3 student memberships, 2 course registrations")
  - "Cancel" and "Delete" buttons
- The related records summary is generated automatically by inspecting Django's cascade deletion collector
- Deletion is submitted via HTMX (`hx-delete`)
- On success: full page redirect to the list view (e.g., back to the cohorts list)

#### Delete Cohort

- Add `DeleteAction` to `CohortInstanceView`
- On confirmation, deletes the cohort and all related records (CohortMemberships, CohortCourseRegistrations, CohortDeadlines, StudentCohortDeadlineOverrides)
- Redirects to the cohorts list page after deletion

#### Success Criteria

- Delete button appears on cohort detail page
- Confirmation modal shows what will be deleted (cascade summary)
- Successful deletion redirects to cohorts list
- Delete pattern is reusable for any instance view
- Appropriate permission checks are enforced

## Cross-cutting Concerns

### Permissions

- All actions (create, edit, delete) must respect django-guardian object-level permissions
- The existing `get_objects_for_user()` pattern in DataTable querysets handles read permissions
- Write/delete permissions should be checked before rendering action buttons and before processing submissions

### Site Awareness

- All model operations go through SiteAwareModel, which automatically handles site assignment
- Forms don't need to include the site field
- Querysets are automatically filtered by site via SiteAwareManager

### HTMX Conventions

- Form validation errors return HTTP 422
- CSRF handled globally via `<body hx-headers>`
- Panel reloads use the existing `__panels/{panel_name}` URL pattern
- Full page redirects use the `HX-Redirect` response header

### Template Conventions

- Use existing cotton components (`<c-modal>`, `<c-button>`, `<c-icon>`, `<c-data-table>`)
- Use the existing `partials/form.html` for form rendering
- New templates go in `panel_framework/templates/` for reusable pieces
- Educator-specific templates stay in `educator_interface/templates/`
