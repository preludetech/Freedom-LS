# Plan: P0 production settings defaults + importable `settings_defaults` module

Implements `1. spec.md`. This is the **foundational** spec of the
`support-concrete-project-deployment` effort — it delivers the importable primitives every later
spec consumes and migrates this repo's reference `config/settings_prod.py` to the import-and-assign
form.

## Overview

Two changes, tightly coupled:

1. **New importable module** `freedom_ls/deployment/settings_defaults.py` — flat constants + small
   pure functions exposing the P0 deployment defaults, added to the **existing** `freedom_ls/deployment/`
   app. It follows the app's own precedent `freedom_ls/deployment/storage.py`
   (`build_s3_media_storage`) — a pure, Django-free, unit-tested helper already imported at
   settings-load time by `config/settings_prod.py:173` — and the older
   `freedom_ls/base/theming.py` (`configure_theme`) / `freedom_ls/base/webhook_event_types.py`
   (`FLS_WEBHOOK_EVENT_TYPES`) precedent (`config/settings_base.py:23-25` already imports and
   assigns/calls those).
2. **Migrate `config/settings_prod.py`** from hard-coded literals to *import-and-assign* consumers
   of those primitives, applying the six P0 fixes (§5.1 A–F of the spec).

Plus three supporting doc/contract edits (checklist, `.env.example`, template manifest) and pytest
unit tests written TDD-style.

**No frontend, view, model, migration, template, or URL is touched.** This is pure
settings/primitive code plus docs and tests.

### Key sequencing constraint (item B — logging)

The spec, `spec-order.md`, and success criterion 2 all state: **do not ship stdout-only logging
ahead of Spec 5's Docker `json-file` log caps.** Moving to stdout relocates the disk-fill risk onto
uncapped Docker logs, which only Spec 5 caps.

Resolution in this plan: we **deliver** the `build_logging_config()` primitive (stdout-only by
default), but in this repo's reference `config/settings_prod.py` we call it **with `log_dir`
supplied**, which keeps the current rotating-file behaviour and the existing `logs/` bind mount
unchanged. The stdout-only flip (dropping the `log_dir` argument and the bind mount) is a one-line
change deferred to Spec 5, once the caps land. This satisfies "reference config is import-and-assign"
without shipping the disk-fill risk early. `docker-compose.yml` is **not** modified by this spec (its
`logs/` bind mount stays).

## Existing code to reuse (keep it DRY)

- **Closest precedent — `freedom_ls/deployment/storage.py`.** A pure, Django-free
  (`from __future__ import annotations`, no Django import at module top), unit-tested helper
  (`build_s3_media_storage`) that already lives in the deployment app and is imported at
  settings-load time by `config/settings_prod.py:173`, then assigned into `STORAGES`. It is **not**
  routed through the app's `config.py`. `settings_defaults.py` mirrors it exactly. This is a stronger

- **Import-and-assign precedent (older)** — `config/settings_base.py:23-25` imports

  then assigns/calls them (`RESOLVED_THEME_DIR = configure_theme(...)` at :243,
  `WEBHOOK_EVENT_TYPES = FLS_WEBHOOK_EVENT_TYPES` at :385). Do **not** invent a new mechanism (no
  settings framework, no `globals()`-mutating `apply_defaults(globals())` — see spec §6 and

- **Env-parsing helpers already exist** — `freedom_ls/base/env.py` (`env_bool`, `env_int`,
  `env_float`, raising `ImproperlyConfigured` on malformed input) is already imported at
  `config/settings_prod.py:3`. Use `env_int` for the numeric env-guard in item E rather than a raw

- **`ImmediateBackend` default already inherited** — `config/settings_base.py:406-409` already
  declares `TASKS = {"default": {"BACKEND": "...ImmediateBackend"}}` with the plain comment
  (`:404-405`) "A production backend … is needed before using tasks in production". Item C's removal

- **Test layout** — the deployment app already has `freedom_ls/deployment/tests/` (e.g.

## Detailed changes

### 1. New module in the existing `freedom_ls/deployment/` app

The `freedom_ls/deployment/` app **already exists** and is registered in `INSTALLED_APPS`
(`config/settings_base.py:100`) — do **not** create the package or its `__init__.py` files. Add a

New files only:

- `freedom_ls/deployment/settings_defaults.py` — the primitives (below).
- `freedom_ls/deployment/tests/test_settings_defaults.py` — unit tests (written TDD-style,
  see Testing).

Existing (do **not** recreate): `freedom_ls/deployment/__init__.py`,
`freedom_ls/deployment/tests/__init__.py`.

#### `settings_defaults.py` contents

Module docstring: explain it is a thin, pure module of deployment-settings primitives consumed by
`config/settings_prod.py` — mirror the shape of the app's existing `storage.py` docstring (thin /

**Flat constants:**

```python
# kill the redirect loop behind a TLS-terminating proxy.
# (enumerate the five trust preconditions here as a plain comment — see "Item A comment" below.)
SECURE_PROXY_SSL_HEADER: tuple[str, str] = ("HTTP_X_FORWARDED_PROTO", "https")

# persistent DB connections. Recommended 60-300s; never None/unlimited.
CONN_MAX_AGE: int = 60
CONN_HEALTH_CHECKS: bool = True
```

**Pure functions:**

```python
def require_secret_key() -> str:
    """Return SECRET_KEY from the environment, raising KeyError at import time if
    unset/empty/whitespace-only.

    Fails fast during Gunicorn boot (visible crash-loop) instead of lazily on the first
    request that signs a cookie. Strips first so a whitespace-only value (e.g. " ") — which
    is truthy in Python but a functionally-broken key — is rejected too, matching the
    whitespace convention in freedom_ls/base/env.py's _raw().
    """
    key = os.environ["SECRET_KEY"].strip()
    if not key:
        raise KeyError("SECRET_KEY")
    return key


def database_ssl_options(sslmode: str) -> dict[str, str]:
    """Return the DATABASES OPTIONS dict for a libpq sslmode, e.g. {"sslmode": "prefer"}.

    Kept as a function (not a bare literal in settings_prod) so future sslmode-adjacent
    options (e.g. sslrootcert) can be added in one place and reach every project on a bump.
    """
    return {"sslmode": sslmode}


def build_logging_config(*, log_dir: Path | None = None) -> dict:
    """Return a Django LOGGING dict.

    Default (log_dir=None): stdout/stderr only — container-friendly, one StreamHandler.
    When log_dir is supplied: additionally write rotating file handlers under log_dir
    (opt back into on-disk logs). This preserves the current reference-config behaviour
    until Spec 5's Docker log caps land, at which point callers drop log_dir.

    WARNING: do not call this without log_dir in a Docker deployment until the container's
    log driver caps max-size/max-file — omitting log_dir relocates the disk-fill risk onto
    uncapped container json-file logs (see the item-B sequencing constraint).
    """
    ...
```

`build_logging_config` design notes:
- Always includes a `console` StreamHandler (stdout) and wires the `django`, `django.request`,
  `django.security`, `freedom_ls`, and root loggers to it — matching the loggers currently in
  `config/settings_prod.py:116-147`.
- **Preserve the `django.db.backends` logger at `level="WARNING"`, `propagate=False`** (currently
  `config/settings_prod.py:132-136`). Dropping it would let `django.db.backends` fall through to the
  root logger at INFO and flood logs with every SQL query — a behaviour regression from "reproduces
  today's on-disk behaviour". When `log_dir` is `None`, point it at the console handler; when
  `log_dir` is given, point it at the file handler (as today).
- When `log_dir` is given, add `RotatingFileHandler`s (10 MB, `backupCount=5`) under `log_dir` and
  attach them to the loggers, reproducing today's on-disk behaviour **exactly** — the same
  "reproduce today's behaviour" bar applied to `django.db.backends` above. That means preserving the
  dedicated `security_file` handler (`security.log`) for the `django.security` logger, not folding it
  into the general `file` handler: security events (`SuspiciousOperation`/`DisallowedHost`/CSRF
  failures) stay isolated for audit/alerting, matching `config/settings_prod.py:107-114,127-131`
  today. So `log_dir` mode wires three file handlers (`file`, `error_file`, `security_file`), one per
  the current split. In stdout mode (`log_dir=None`) all these loggers point at the single `console`
  handler instead.
- `os` / `Path` imported at module top.

**Item A comment (five preconditions).** Enumerate, as a plain code comment beside
`SECURE_PROXY_SSL_HEADER` in `settings_defaults.py` (the durable single source), the five trust
preconditions from spec §5.1-A:
1. origin firewalled to Cloudflare's published IP ranges (no direct client path to the proxy);
2. Cloudflare SSL/TLS mode = **Full (strict)**, never Flexible;
3. `trusted_proxies` scoped exactly to Cloudflare's ranges, never `0.0.0.0/0`;
4. the app container publishes no port except through the proxy;
5. no custom `header_up` overriding the proxy's default `X-Forwarded-Proto`.

Follow the `code-comments` skill: explain *why the header is trustworthy*; do **not** cite the spec,
section numbers, or "item A" in the comment.

### 2. Migrate `config/settings_prod.py`

Add near the existing imports:

```python
from freedom_ls.deployment import settings_defaults as fls_defaults
```

Then apply the six fixes:

- **A. `SECURE_PROXY_SSL_HEADER`** — add
  `SECURE_PROXY_SSL_HEADER = fls_defaults.SECURE_PROXY_SSL_HEADER` (hard default, not env-guarded).
  A short comment at the assignment may point the reader to the primitive for the trust
  preconditions (no spec citation).

  **Reviewed & resolved (keep as the secure default).** A plan-security-review flagged that the
  repo's *currently-shipped* reference artifacts (`nginx.conf` on `80` only, no TLS, forwarding
  `X-Forwarded-Proto: http`; `docker-compose.yml` publishing only `80:80`) don't satisfy item A's
  "behind a TLS-terminating proxy" premise. **Decision:** those nginx/compose artifacts are
  **legacy** — the target topology is Caddy + Cloudflare **Full (strict)** (the parent
  `support-concrete-project-deployment` spec is the source of truth; the legacy nginx path is being
  removed, not maintained). So item A ships **now as the secure hard default** (best-practice, and it
  does not worsen the current state — `SECURE_SSL_REDIRECT = True` already assumes TLS). No
  nginx-specific doc warning is added (that path is not the source of truth). **Document the reason
  at the assignment**: a plain comment noting the default is safe because production terminates TLS
  at a trusted proxy that forwards `X-Forwarded-Proto: https`, with the five trust preconditions
  living beside the primitive in `settings_defaults.py` (per the "Item A comment" section). No spec
  citation in the comment (`code-comments` skill).
- **B. Logging** — replace the ~80-line literal `LOGGING` dict (`:66-148`) with
  `LOGGING = fls_defaults.build_logging_config(log_dir=BASE_DIR / "logs")`. Behaviour is unchanged
  (still writes to `logs/`); the `logs/` bind mount in `docker-compose.yml` stays. The stdout-only
  flip is deferred to Spec 5 (see the sequencing note above). Add a short comment at the call site
  (no spec citation) noting the `log_dir` argument is temporary — drop it once container-level log
  size/rotation caps exist — so whoever lands that later change doesn't have to rediscover why.
- **C. Tasks** — delete the redundant `TASKS` block **and** its `# TODO @claude` at `:192-199`.
  Prod already inherits `ImmediateBackend` from `settings_base.py:406-409`. This removal is
  explicitly authorised by spec §5.1-C (the decision the TODO flagged is now deliberately made, and
  the caveat survives as the plain comment in `settings_base.py:404-405`), so it does **not** violate
  CLAUDE.md's "never delete a TODO" rule. Remove **only** this TODO; touch no other. Because this
  exercises a CLAUDE.md exception, **surface the deletion explicitly**: call it out in the commit
  message / PR description (with the §5.1-C authorisation and the pointer to the surviving caveat at
  `settings_base.py:404-405`) and flag it during the `fls:request-code-review` pass, so the
  exception has a visible paper trail rather than being exercised silently.
- **D. DB SSL** — add an `OPTIONS` key to `DATABASES["default"]` sourced from `DB_SSLMODE`:
  ```python
  DATABASES["default"]["OPTIONS"] = fls_defaults.database_ssl_options(
      os.getenv("DB_SSLMODE", "prefer")
  )
  ```
  `DB_SSLMODE` is a plain string with no coercion, so `os.getenv` is used deliberately — the
  `freedom_ls.base.env` helpers only cover bool/int/float. Default `prefer` (never `require`) so
  first boot against same-host `postgres:17` (no TLS) still connects.

  **Reviewed & resolved (no validation — deliberate pass-through).** A plan-security-review asked
  whether to validate `DB_SSLMODE` against libpq's known modes at import time. **Decision: skip it.**
  An unknown mode fails loudly at first DB connect (libpq rejects it — no silent TLS downgrade to
  `disable`), so this is robustness, not a security gap, and the spec doesn't require it.
- **E. `CONN_MAX_AGE` + `CONN_HEALTH_CHECKS`** — env-guarded, defaulting to the primitives. Use the
  `env_int` helper (already imported at `settings_prod.py:3`), not a raw `int(os.getenv(...))`, so a
  malformed value raises `ImproperlyConfigured` at import rather than a bare `ValueError`:
  ```python
  DATABASES["default"]["CONN_MAX_AGE"] = env_int(
      "DB_CONN_MAX_AGE", fls_defaults.CONN_MAX_AGE
  )
  DATABASES["default"]["CONN_HEALTH_CHECKS"] = fls_defaults.CONN_HEALTH_CHECKS
  ```
  (Assigning after the `DATABASES` dict literal, or splicing into it — keep it greppable.)
- **F. `SECRET_KEY`** — replace `SECRET_KEY = os.getenv("SECRET_KEY", "")` (`:48`) with
  `SECRET_KEY = fls_defaults.require_secret_key()`. Raises `KeyError` at settings-import (Gunicorn
  boot) if the env var is missing or empty.

Keep `settings_prod.py` thin and greppable — explicit named assignments, no `globals()` mutation.
Project-specific values (`HOST_DOMAIN`, S3 vars, `DB_SSLMODE` source-of-truth) stay in the file.

## Testing (TDD)

Follow the `fls:testing` skill and TDD: write a failing test first, then implement. Do **not**
pre-write the full suite here — the list below is the coverage target, added incrementally.

New file `freedom_ls/deployment/tests/test_settings_defaults.py` (pure unit tests, no DB, no Django
test client needed — these are plain functions/constants):

- `require_secret_key()`:
  - raises `KeyError` when `SECRET_KEY` is unset (monkeypatch `os.environ`);
  - raises `KeyError` when `SECRET_KEY` is empty string;
  - returns the value when set.
- `database_ssl_options("prefer")` returns `{"sslmode": "prefer"}`; likewise for `disable`,
  `require`.
- `build_logging_config()` (no `log_dir`): returns a dict with a `console`/StreamHandler and **no**
  file handlers; expected loggers present.
- `build_logging_config(log_dir=tmp_path)`: includes rotating file handler(s) whose filenames sit
  under `log_dir`; console handler still present.
- Constants: `SECURE_PROXY_SSL_HEADER == ("HTTP_X_FORWARDED_PROTO", "https")`;
  `CONN_HEALTH_CHECKS is True`; `CONN_MAX_AGE` is a positive int in the 60–300 range.
- Proxy-header effect (closes success-criterion 1's "verify in the same pass" for the redirect
  loop, as an automated check rather than manual verification): with
  `@override_settings(SECURE_PROXY_SSL_HEADER=settings_defaults.SECURE_PROXY_SSL_HEADER)`, a
  `RequestFactory` request carrying `HTTP_X_FORWARDED_PROTO="https"` has `request.is_secure()` is
  `True`, and one without it is `False`. This guards that the tuple's header name/value are correct
  (a typo'd header would silently reintroduce the loop). Full HSTS-header/proxy behaviour behind the
  live Caddy proxy is a deployment-time concern outside this repo (Spec 5).

Do not assert on styling/CSS (none here). Run `uv run pytest freedom_ls/deployment/` to verify.
