# Spec 3 (this repo): production defaults to a durable database-backed task backend

> **Part of the `support-concrete-project-deployment` effort.** See `spec-order.md` for ordering and
> cross-spec dependencies. Can run in **parallel with Specs 2 and 4** once Spec 1 has landed.
> **Production must default to a production-ready, durable, database-backed task backend**
> (`django-tasks-db`'s `DatabaseBackend`, ORM on PostgreSQL вЂ” no Celery/Redis) instead of the
> synchronous `ImmediateBackend`. FLS's backend-agnostic core `django.tasks` code already supports
> this, so the producer needs no edit вЂ” but the switch is a real settings + dependency + migration
> change, not docs-only.
>
> **Target repo:** this repo (`freedom_ls`). Concrete changes: a durable-backend primitive in
> `freedom_ls/deployment/settings_defaults.py`, its assignment in `config/settings_prod.py`, the
> `django_tasks_db` app + `django-tasks-db==0.12.0` dependency, a `WebhookDelivery` idempotency
> guard (so the at-least-once switch cannot double-send), and the `docs/` rewrite. The **worker
> container** that executes the tasks lives in **Spec 5** (template repo); because the durable
> backend is now the shipped production default, that worker must ship **enabled by default** вЂ”
> this spec coordinates that change into Spec 5.

## Terminology

`ConcreteFlsImplementation` = the specific downstream production repo that raised this handoff
(installs `freedom_ls` as a read-only git submodule). FLS is never deployed standalone вЂ” only
concretely, as a submodule.

## 1. Problem

Production currently runs `ImmediateBackend` (inherited from `settings_base.py:403вЂ“410`;
`settings_prod.py` does not override `TASKS`). That means every background task вЂ” today, webhook
dispatch and its outbound HTTP POST вЂ” runs **synchronously inside the request/response cycle**. A
slow or unreachable webhook endpoint blocks the user's request; a burst of events serializes onto
the web worker. That is not a production-ready posture: "in production" should default to a durable,
out-of-process backend so producing a task is a fast enqueue and the actual work happens on a
dedicated worker.

The capability already exists вЂ” FLS's producer enqueues through backend-agnostic core
`django.tasks` (`webhooks/events.py:2,35`), so the backend is chosen entirely by the `TASKS`
setting. What is missing is (a) making the durable backend the **production default** (settings +
`django-tasks-db` dependency + `django_tasks_db` app + migrate), (b) a **`WebhookDelivery`
idempotency guard** so the switch from once-inline to at-least-once execution cannot double-send,
(c) the **docs** describing the new default and its required worker, and (d) coordinating **Spec
5** so its `db_worker` container ships enabled by default.

## 3. Requirements

- **Make the durable backend the production default.** Add a `DATABASE_TASKS` primitive to
  `freedom_ls/deployment/settings_defaults.py` (`{"default": {"BACKEND": "django_tasks_db.DatabaseBackend"}}`)
  and assign `TASKS = fls_defaults.DATABASE_TASKS` in `config/settings_prod.py`. Keep on-commit
  enqueue (Django default) so the worker reads the committed `WebhookEvent` row.
- **Only production flips.** `config/settings_base.py` keeps `ImmediateBackend`, so dev and the full
  test suite keep executing tasks inline with no worker (the `transaction=True` webhook task tests
  depend on this).
- **Add the package + app.** `django-tasks-db==0.12.0` (exact pin вЂ” single PyPI release, package
  `django-tasks-db`, **not** `django-tasks`, which since `0.12.0` ships only Immediate/Dummy) to
  `pyproject.toml`; `django_tasks_db` to `INSTALLED_APPS`; run `migrate` (it ships its own tables).
  Tasks then become visible in the Django admin.
- **Guard against duplicate deliveries.** Because the durable backend is at-least-once, add a
  `UniqueConstraint` on `WebhookDelivery(event, endpoint)` and make `dispatch_event` use
  `get_or_create`, sending only when the delivery row is newly created (TDD вЂ” failing test first).
- **Rewrite the docs.** `docs/product/deployment.md` and `docs/product/README.md` describe the
  durable `django-tasks-db` `DatabaseBackend` as the **production default** with a **required**
  `db_worker` process, the `prune_db_task_results` retention job, no `DEBUG` autoreload worker in
  prod, the `==0.12.0` pin, admin visibility, and idempotent redelivery. *(Spec 4 also rewrites
  `deployment.md`; land this before/with it вЂ” see `spec-order.md`.)*
- **Coordinate Spec 5.** Because the worker is now a hard dependency of the shipped production
  default, Spec 5's `db_worker` container + `prune_db_task_results` job must ship **enabled by
  default** (not disabled via Compose `profiles:`).

Detail: `research_django_tasks_durable_backend.md`.

## 4. Success criteria

- **Production defaults to the durable backend:** loading `config.settings_prod` yields
  `TASKS["default"]["BACKEND"] == "django_tasks_db.DatabaseBackend"`; `config.settings_base` and
  dev/tests still yield `ImmediateBackend`. `django-tasks-db==0.12.0` is a pinned dependency,
  `django_tasks_db` is in `INSTALLED_APPS`, `manage.py db_worker --help` succeeds, and `migrate`
  applies cleanly.
- **No duplicate sends:** `dispatch_event` is idempotent per `(event, endpoint)` вЂ” the new test
  passes and the full webhook suite still passes.
- **Docs match behaviour:** `deployment.md` / `README.md` describe the durable backend as the
  production default with a **required** `db_worker`, its `prune_db_task_results` job, and the
  `==0.12.0` pin вЂ” no claim that prod runs tasks inline.
- **Spec 5 coordinated:** its spec now ships the `db_worker` worker + prune job **enabled by
  default**.
