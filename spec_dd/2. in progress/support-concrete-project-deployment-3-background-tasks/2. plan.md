# Plan: production defaults to a durable database-backed task backend (Spec 3)

## Scope and shape of this change

This spec now makes **production default to a durable, database-backed task backend**
(`django-tasks-db`'s `DatabaseBackend`, ORM on PostgreSQL — no Celery/Redis) instead of the
synchronous `ImmediateBackend`. "Production-ready by default while in production" is the goal.

This is **no longer docs-only**. It touches:

- **Settings** — a new importable primitive in `freedom_ls/deployment/settings_defaults.py`,
  assigned in `config/settings_prod.py`; the `django_tasks_db` app added to `INSTALLED_APPS`.
- **Dependency** — `django-tasks-db==0.12.0` added to `pyproject.toml` (+ `uv.lock`).
- **A small model change + migration** — a uniqueness guard on `WebhookDelivery` so the switch
  to at-least-once execution cannot double-send webhooks (TDD).
- **Docs** — `docs/product/deployment.md` and `docs/product/README.md` rewritten to describe the
  durable backend as the production default with a required worker.
- **Cross-spec coordination** — Spec 5's worker container + `prune_db_task_results` job move from
  *disabled-by-default* to *enabled by default*, because the worker is now a hard dependency of the
  shipped production default.

**Principle — prod flips, base/dev/tests stay `ImmediateBackend`.** Only the production settings
layer flips. `config/settings_base.py` keeps `ImmediateBackend`, so dev and the entire test suite
keep executing tasks inline with no worker. This is load-bearing: the webhook task tests
(`freedom_ls/webhooks/tests/test_events.py`, marked `@pytest.mark.django_db(transaction=True)`)
depend on inline on-commit execution; a durable backend with no in-process worker would silently
stop running the enqueued task under test.

## Why this is safe (the capability already exists)

FLS's only task producer enqueues through Django 6 core's **backend-agnostic** task API:

- `freedom_ls/webhooks/events.py:2` — `from django.tasks import default_task_backend, task`
- `freedom_ls/webhooks/events.py:35` — `default_task_backend.enqueue(_dispatch_event_task, ...)`

Because the producer talks to `django.tasks` (core) and never to a concrete backend class, the
backend is selected purely by the `TASKS` setting. `django-tasks-db`'s `DatabaseBackend` is built
against Django 6 core's `django.tasks`, so flipping the production `TASKS` setting switches
backends with **no `freedom_ls` producer edit**. The task body already captures request-scoped
data (`site_id`) in the producer and passes only serializable args, so it is safe to run
out-of-process (verified: `events.py:22-28`, `events.py:37`, `events.py:64-72`).

## Two risks the flip activates (both handled here)

1. **Duplicate deliveries (at-least-once).** `dispatch_event` currently `create()`s a
   `WebhookDelivery` and POSTs unconditionally, with **no** unique constraint on `(event, endpoint)`
   (`webhooks/models.py:394-395`, `webhooks/events.py:78`). Under a durable backend a
   redelivered task double-sends. Task 4 adds a uniqueness guard + create-only send.
2. **Worker coupling.** `DatabaseBackend` executes nothing unless `python manage.py db_worker`
   runs. That worker is a Spec 5 (template repo) container, currently disabled-by-default. Task 6
   coordinates Spec 5 to ship it **enabled**, so the shipped production default is genuinely
   runnable end-to-end.

## Task 1 — Add the durable backend as an importable primitive

**File:** `freedom_ls/deployment/settings_defaults.py` (flat-constants module already imported by
`config/settings_prod.py` as `fls_defaults`).

Add a constant next to `SECURE_PROXY_SSL_HEADER` / `CONN_MAX_AGE`:

```python
# Durable, database-backed task backend for production (django-tasks-db, ORM/Postgres —
# no Celery/Redis). HARD operational dependency: an out-of-process `python manage.py db_worker`
# must be running, or enqueued tasks persist in the DB and never execute. Enqueue stays
# on-commit (Django default) so the worker sees the committed WebhookEvent row.
DATABASE_TASKS: dict[str, dict[str, str]] = {
    "default": {"BACKEND": "django_tasks_db.DatabaseBackend"},
}
```

**Verify** the exact dotted path against the installed `django-tasks-db==0.12.0` (app label
`django_tasks_db`, backend `django_tasks_db.DatabaseBackend` per
`research_django_tasks_durable_backend.md`) before finalizing. Add a matching assertion to
`freedom_ls/deployment/tests/test_settings_defaults.py`.

## Task 2 — Flip the production settings layer

**File:** `config/settings_prod.py`. Assign the primitive alongside the other `fls_defaults`
assignments (SECRET_KEY / DATABASES / LOGGING):

```python
TASKS = fls_defaults.DATABASE_TASKS
```

Do **not** set `ENQUEUE_ON_COMMIT=False` — the on-commit default is required so the worker reads
the committed `WebhookEvent` row (otherwise `dispatch_event` hits its `DoesNotExist` early return
and drops the event).

## Task 4 — Idempotency guard for at-least-once execution (TDD)

**Write the failing test first.** In `freedom_ls/webhooks/tests/test_events.py`, add
`test_dispatch_event_is_idempotent`: call `dispatch_event(event_id, site_id)` twice (with
`attempt_delivery` patched) and assert exactly one `WebhookDelivery` exists for `(event, endpoint)`
and `attempt_delivery` was called once. This fails on current code.

Then implement:

- `freedom_ls/webhooks/models.py` — add to `WebhookDelivery.Meta`:
  ```python
  constraints = [
      models.UniqueConstraint(fields=["event", "endpoint"], name="uniq_delivery_event_endpoint"),
  ]
  ```
  (`event` already implies `site`, so the key needs no `site_id`.)
  Then `uv run manage.py makemigrations webhooks` + `uv run manage.py migrate`.
- `freedom_ls/webhooks/events.py:78` — replace `WebhookDelivery.objects.create(...)` with
  `get_or_create(event=event, endpoint=endpoint, defaults={"status": "pending", "site_id": site_id})`
  and call `attempt_delivery(delivery)` **only when `created` is True** (a redelivered task must not
  re-send). Manual retries and the "send test event" flow reuse/own their rows and are unaffected.

**Migration caveat (flag for the downstream migration):** the unique constraint fails to apply if
existing rows already duplicate `(event, endpoint)`. Fine for fresh/dev DBs; a live deployment with
historical duplicates needs a one-off dedupe before the constraint migration.

## Success criteria

- Loading `config.settings_prod` yields
  `TASKS["default"]["BACKEND"] == "django_tasks_db.DatabaseBackend"`; `config.settings_base`
  (and dev/tests) still yield `ImmediateBackend`.
- `django-tasks-db==0.12.0` is a pinned dependency; `django_tasks_db` is in `INSTALLED_APPS`;
  `uv run manage.py db_worker --help` succeeds; `migrate` applies cleanly.
- `dispatch_event` is idempotent per `(event, endpoint)` — the new test passes and the full webhook
  suite still passes.
- `deployment.md` / `README.md` describe the durable backend as the production default with a
  required `db_worker`; Spec 5's spec ships that worker enabled by default.
