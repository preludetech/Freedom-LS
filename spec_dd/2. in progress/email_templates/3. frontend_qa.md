# Frontend QA: Professional Branded Email Templates

## Prerequisites

- Dev server running: `uv run python manage.py runserver`
- Dev settings use file-based email backend — emails are saved to `gitignore/emails/`
- Check that directory exists: `ls gitignore/emails/` (create if needed)
- Verify `ACCOUNT_EMAIL_NOTIFICATIONS = True` is in settings (required for notification emails to be sent)
- You'll need to read the saved email files to inspect the HTML output
- Use Playwright MCP (`use-playwright` skill) to interact with the site and trigger email flows

## Test 1: Signup Confirmation Email

**Goal**: Verify new user signup sends a branded HTML confirmation email.

**Steps**:
1. Navigate to `http://localhost:8000/accounts/signup/`
2. Fill in the signup form with a test email (e.g., `testuser@example.com`) and password
3. Submit the form
4. Check `gitignore/emails/` for the most recent file
5. Open/read the email file

**Expected**:
- Email is multipart (contains both HTML and text sections, separated by MIME boundary)
- HTML section contains:
  - Table-based layout (look for `<table` tags with `role="presentation"`)
  - Header with site name (or logo if configured)
  - Header background color matches `#2B6CB0` (primary)
  - Greeting: "Hi" (possibly with first name if provided)
  - Body text about confirming email address
  - A "Confirm Email Address" button (bulletproof button pattern — `<table>` with background color and `<a>` tag)
  - Button color matches `#2B6CB0`
  - Sign-off: "Cheers, The [Site Name] Team"
  - Footer with site name and copyright year
  - Font family includes Arial
  - Text color matches `#1A2332` (foreground)
  - Inline styles on elements (premailer has inlined the CSS)
- Text section contains:
  - Greeting
  - Confirmation URL as plain text
  - Sign-off
  - Footer line
- Subject line file present and contains branded subject

2. Navigate to `http://localhost:8000/accounts/password/reset/`

4. Submit the form

## Test 6: Logo Fallback

**Goal**: Verify header falls back to text when no logo is configured.

**Steps**:
1. Ensure `EMAIL_LOGO_STATIC_PATH` is not set (default `None`) in settings
2. Trigger any email (e.g., signup)
3. Check the HTML header section

**Expected**:
- Header shows the site name as styled text (no `<img>` tag)
- Header has primary color background (`#2B6CB0`)

## Test 7: Logo Display

**Goal**: Verify header shows logo when configured.

**Steps**:
1. Set `EMAIL_LOGO_STATIC_PATH = "images/test-logo.png"` in `config/settings_dev.py`
2. Place a test image at `static/images/test-logo.png`
3. Trigger any email
4. Check the HTML header section

**Expected**:
- Header contains an `<img>` tag pointing to the static path for the logo
- Site name may still appear as alt text

## Test 8: Mobile Responsiveness

## Test 13: Multipart Format Verification

**Goal**: Verify all emails are truly multipart.

**Steps**:
1. Open any raw email file from `gitignore/emails/`
2. Look at the MIME structure

**Expected**:
- Content-Type header shows `multipart/alternative`
- Both `text/plain` and `text/html` parts are present
- Text part is readable and contains the same essential information as the HTML part

## Test 14: Cross-Client Rendering Check

**Goal**: Verify HTML emails render correctly across email clients.

**Steps**:
1. Extract the HTML portion from a generated email file
2. Open it directly in a browser — verify the table layout renders correctly
3. For production release: send test emails to actual Gmail, Outlook, and Apple Mail accounts to verify rendering
4. Alternatively, use an email preview service (e.g., Litmus, Email on Acid) to check cross-client rendering

**Expected**:
- Table-based layout renders correctly without broken formatting
- No missing styles (premailer has inlined everything)
- Colors, fonts, and spacing are consistent
- Bulletproof buttons display correctly

## Notes on Untested Email Types

The following email types are difficult to trigger via the UI but are covered by automated tests:

- **Password set**: Typically triggered when a user who signed up via social auth sets a password for the first time. Verified via automated tests.
- **Email confirmed notification**: This template exists for completeness — allauth does not currently send it. Verified via automated tests.
- **Login code**: Only testable if the login code authentication flow is enabled in allauth settings. If not enabled, verified via automated tests.
