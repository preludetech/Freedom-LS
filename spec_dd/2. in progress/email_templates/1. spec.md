# Spec: Professional Branded Email Templates

## Overview

Replace allauth's default plain-text emails with professional, on-brand HTML emails that render well across all major email clients. The system should be easy to customise for any project built on FreedomLS.

## Why This Matters

- Plain text emails look unprofessional and reduce trust
- Branded emails reinforce the platform identity and improve user experience
- A well-structured, overridable template system makes FreedomLS easy to customise for downstream projects
- Multi-part emails (HTML + text) improve deliverability (spam filters prefer them) and accessibility

## Requirements

### 1. Base Email Templates

Create two base email templates, mirroring allauth's own structure:

1. **Base message template** — Used by regular emails (confirmation, password reset, login code, unknown account, account already exists)
2. **Base notification template** — Extends the base message template; used by security notification emails (password changed, password set, email changed, email confirmed, email deleted). Adds a security information section showing the IP address, browser, and timestamp of the change.

Both HTML and text versions are needed for each base template.

**Layout:**
- Single-column, table-based layout (no flexbox/grid — these don't work in email clients)
- Maximum width: 600px (industry standard)
- Total email size must stay under 100KB to avoid Gmail clipping
- Mobile-responsive using `max-width: 600px; width: 100%` pattern

**Header:**
- Support an optional logo image via a configurable URL (Django setting)
- Fall back to a styled text header showing the site name when no logo is configured
- Header background color from `EMAIL_COLOR_PRIMARY` (parsed from Tailwind)

**Content area:**
- Clean white background
- Font family from `EMAIL_FONT_FAMILY` setting (defaults to `Arial, Helvetica, sans-serif` — system-safe fonts since custom fonts don't work reliably in email clients)
- Text colors from `EMAIL_COLOR_FOREGROUND` and `EMAIL_COLOR_MUTED` (parsed from Tailwind)
- Consistent padding (20px)

**Footer:**
- Site name and copyright
- Configurable footer links via an includable template file (e.g., privacy policy, terms of service, unsubscribe)
- Muted text styling, smaller font size

**Buttons (CTAs):**
- "Bulletproof button" pattern using a table cell with background color and a styled `<a>` tag
- Button color from `EMAIL_COLOR_PRIMARY` (parsed from Tailwind)
- Consistent padding and border-radius

**Why table-based layout:** Email clients (especially Outlook's Word rendering engine and Gmail's aggressive CSS sanitization) don't support modern CSS layout. Tables with `role="presentation"` are the only reliable cross-client layout method.

### 2. CSS Inlining with django-premailer

Use `django-premailer` to automatically convert CSS in `<style>` blocks into inline `style=""` attributes for all outgoing HTML emails.

**Why:** Gmail strips `<style>` tags, so inline styles are required. A CSS inlining library lets us write maintainable CSS during development while producing email-client-compatible output. This is more maintainable than hand-writing inline styles, especially as the template grows.

**How it works:** `django-premailer` provides a `{% premailer %}` template tag. Wrapping HTML content in `{% premailer %}...{% endpremailer %}` converts any CSS in `<style>` blocks into inline `style=""` attributes at render time. By using this tag in the base email template, all emails that extend it automatically get CSS inlining.

**Configuration:**
- Install `django-premailer` (`uv add django-premailer`)
- Add `'django_premailer'` to `INSTALLED_APPS`
- Wrap the base email template's HTML in `{% load premailer %}{% premailer %}...{% endpremailer %}`
- The `<style>` block in the base template should use the configurable color/font settings (injected via template context)

### 3. Overridable Template Includes

The base email template uses `{% include %}` for sections that downstream projects may want to customise. Each includable section is a separate template file. To override, a downstream project simply places a file at the same template path in their own app (Django's template resolution handles the rest).

**Required include files:**
- `emails/includes/greeting.html` — The opening greeting (e.g., "Hi {{ user.first_name }},"). Projects can override to match their voice (formal, casual, etc.)
- `emails/includes/sign_off.html` — The closing sign-off (e.g., "Cheers, The {{ current_site.name }} Team"). Projects can override for their brand voice
- `emails/includes/footer_links.html` — Optional footer links (privacy policy, terms, unsubscribe). Empty by default — projects add links relevant to their deployment
- `emails/includes/header.html` — The email header section. Projects can replace the entire header (e.g., to add a logo or change styling)

Corresponding `.txt` versions for text emails where applicable (greeting, sign-off).

### 4. Text Email Fallback

Every HTML email must have a corresponding `.txt` version.

- Override allauth's default `.txt` templates to ensure consistent messaging between HTML and text versions
- Text templates should extend a base text email template with the same greeting/sign-off pattern
- Use `{% autoescape off %}` in text templates to avoid HTML entities

**Why:** Some users prefer text emails, some corporate firewalls strip HTML, and multi-part emails score better with spam filters.

### 5. Allauth Email Template Overrides

Create both `.html` and `.txt` templates for all allauth email types.

**Allauth file naming convention:** Allauth's `render_mail()` appends `_message.{ext}` and `_subject.txt` to the template prefix. For example, prefix `account/email/email_confirmation` produces:
- `account/email/email_confirmation_message.html`
- `account/email/email_confirmation_message.txt`
- `account/email/email_confirmation_subject.txt`

| Email Type | Template Prefix | Context Variables |
|---|---|---|
| Email confirmation (signup) | `account/email/email_confirmation_signup` | Same as general confirmation (see note below) |
| Email confirmation (general) | `account/email/email_confirmation` | `user`, `activate_url`, `current_site`, `key` |
| Password reset | `account/email/password_reset_key` | `user`, `password_reset_url`, `current_site` |
| Password changed | `account/email/password_changed` | `user`, `current_site`, `ip`, `user_agent`, `timestamp` |
| Password set | `account/email/password_set` | `user`, `current_site`, `ip`, `user_agent`, `timestamp` |
| Email changed | `account/email/email_changed` | `user`, `current_site`, `from_email`, `to_email`, `ip`, `user_agent`, `timestamp` |
| Unknown account | `account/email/unknown_account` | `email`, `signup_url`, `current_site` |
| Login code | `account/email/login_code` | `code`, `email`, `current_site` |
| Account already exists | `account/email/account_already_exists` | `email`, `password_reset_url`, `signup_url`, `current_site` |
| Email confirmed | `account/email/email_confirm` | `user`, `current_site`, `ip`, `user_agent`, `timestamp` (see note below) |
| Email deleted | `account/email/email_deleted` | `user`, `current_site`, `deleted_email`, `ip`, `user_agent`, `timestamp` |

**Note on email_confirmation_signup:** This template should include/extend `email_confirmation` (matching allauth's default pattern). They share the same content by default, but downstream projects can override just the signup version if they want different copy for new signups vs. adding a secondary email.

**Note:** Password changed, password set, email changed, email confirmed, and email deleted are *notification* emails — they extend the `base_notification` template and include security context (IP address, browser, timestamp) to help users detect unauthorized changes.

**Note on email_confirm:** The `email_confirm` notification template exists in allauth's template directory but is not currently sent by allauth's code. We create it for completeness — allauth may use it in a future version, or we can send it ourselves via the adapter.

**Note on notification context:** Allauth's `send_notification_mail` does not include `user` in the template context by default — it only provides `ip`, `user_agent`, `timestamp`, and any extra context (e.g., `from_email`/`to_email` for email_changed). The `user` variable listed above will need to be injected by overriding `send_notification_mail` in the custom `AccountAdapter`. Alternatively, `{{ request.user }}` is available since the request object is in context.

Each email should:
- Extend the appropriate base template (base message or base notification)
- Include a clear call-to-action button where appropriate (confirmation, password reset)
- Use concise, friendly copy consistent with the brand voice
- Include the text fallback version

Subject line templates (`_subject.txt`) should also be overridden for consistent branding.

### 6. Documentation

Create `.claude/docs/email_templates.md` documenting:

- The base email template structure and how to extend it
- Available template includes and how to override them
- Context variables available in each email type (table from section 5)
- How to add new email types using the base template
- How CSS inlining works with django-premailer
- How to configure the optional logo static path, brand colors, and fonts
- How to preview emails during development (mention the file-based email backend in dev settings)
- Example of creating a custom email template

## Template Location

All email templates should be placed in:
```
freedom_ls/accounts/templates/account/email/
```

The base email templates and includes should be at:
```
freedom_ls/accounts/templates/emails/base_email.html
freedom_ls/accounts/templates/emails/base_email.txt
freedom_ls/accounts/templates/emails/base_notification_email.html  (extends base_email.html)
freedom_ls/accounts/templates/emails/base_notification_email.txt   (extends base_email.txt)
freedom_ls/accounts/templates/emails/includes/greeting.html
freedom_ls/accounts/templates/emails/includes/greeting.txt
freedom_ls/accounts/templates/emails/includes/sign_off.html
freedom_ls/accounts/templates/emails/includes/sign_off.txt
freedom_ls/accounts/templates/emails/includes/footer_links.html
freedom_ls/accounts/templates/emails/includes/header.html
```

This follows the existing pattern where the `accounts` app owns authentication-related templates, and keeps allauth overrides in the standard `account/email/` path that allauth expects.

## Settings / Configuration

**Why email templates can't use Tailwind classes directly:** Email clients don't support CSS custom properties (`var()`), `@layer`, flexbox, grid, or `<style>` tags (Gmail strips them). Email CSS must use literal hex values in inline styles. However, we keep `tailwind.components.css` as the single source of truth for brand colors — see below.

**Brand colors — single source of truth:** A small utility parses the `--color-*` custom property definitions from `tailwind.components.css` at Django startup. This feeds the literal hex values into Django settings automatically, so there is zero duplication of color values. Change a color in `tailwind.components.css` and emails pick it up.

| Setting | Source | Description |
|---|---|---|
| `EMAIL_LOGO_STATIC_PATH` | Django setting (default `None`) | Path to logo image relative to static files (e.g., `images/logo.png`). Resolved via `{% static %}` in templates. Falls back to styled text header with site name when not set |
| `EMAIL_COLOR_PRIMARY` | Parsed from `--color-primary` in `tailwind.components.css` | Header background, button background, link color |
| `EMAIL_COLOR_FOREGROUND` | Parsed from `--color-foreground` in `tailwind.components.css` | Primary body text color |
| `EMAIL_COLOR_MUTED` | Parsed from `--color-muted` in `tailwind.components.css` | Secondary/footer text color |
| `EMAIL_FONT_FAMILY` | Django setting (default `Arial, Helvetica, sans-serif`) | Font stack for all email text |
| `ACCOUNT_EMAIL_NOTIFICATIONS` | Django setting (set to `True`) | Required for allauth to send notification emails (password changed, password set, email changed, email confirmed, email deleted). Defaults to `False` in allauth |

These settings should be made available to email templates via a context processor or by injecting them in the email rendering pipeline (e.g., in the custom `AccountAdapter`).

## Success Criteria

1. All allauth emails are sent as multipart (HTML + text)
2. HTML emails render correctly in Gmail, Outlook (desktop and web), Apple Mail, and major mobile email apps
3. Emails use configurable brand colors and fonts (defaulting to FreedomLS brand)
4. The greeting, sign-off, header, and footer links are overridable via separate template include files
5. A downstream project can customise the email appearance without forking the base templates
6. Documentation clearly explains the template system, available context variables, and how to extend/override
7. Email total size stays under 100KB
8. Logo is optional and falls back gracefully to text
9. All existing allauth email flows continue to work (signup confirmation, password reset, etc.)
