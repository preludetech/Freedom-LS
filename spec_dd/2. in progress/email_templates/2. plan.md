# Implementation Plan: Professional Branded Email Templates

## Overview

Replace allauth's default plain-text emails with professional, branded HTML+text multipart emails. This involves creating base templates, a color-parsing utility, django-premailer integration, overridable includes, all allauth email overrides, and documentation.

## Prerequisites

- Read the spec: `spec_dd/2. in progress/email_templates/1. spec.md`
- Read brand guidelines skill before writing any template copy or styling

## Existing Code to Reuse

- **AccountAdapter**: `freedom_ls/accounts/allauth_account_adapter.py` — already exists with `is_open_for_signup`. We'll extend it with `send_notification_mail` to inject `user` into context.
- **Tailwind colors**: `tailwind.components.css` — contains `--color-primary: #2B6CB0`, `--color-foreground: #1A2332`, `--color-muted: #4A5568` in the `@theme` block. The color parser will read these.
- **Context processors**: `freedom_ls/accounts/context_processors.py` and `freedom_ls/site_aware_models/context_processors.py` (provides `site_name`) — existing patterns to follow for a new email context processor.
- **Settings**: `config/settings_base.py` — add new email settings here.
- **Dev email backend**: `config/settings_dev.py` — already configured with `filebased.EmailBackend` writing to `gitignore/emails`. Useful for previewing emails during development.

## Task 2: Create the Tailwind color parser utility

**Skills**: `testing` (TDD approach)

**Description**: Create a utility that parses `--color-*` custom property definitions from `tailwind.components.css` and exposes them as Django settings (`EMAIL_COLOR_PRIMARY`, `EMAIL_COLOR_FOREGROUND`, `EMAIL_COLOR_MUTED`).

**Steps**:
1. Create `freedom_ls/accounts/email_utils.py` with a function:

```python
def parse_tailwind_colors(css_file_path: str) -> dict[str, str]:
    """Parse --color-* custom properties from a CSS file.
    Returns a dict like {"primary": "#2B6CB0", "foreground": "#1A2332", ...}
    """
```

2. The function should:
   - Read the CSS file
   - Use regex to find `--color-<name>: <value>;` patterns
   - Return a dict mapping color names to hex values

3. Create a Django `AppConfig.ready()` hook or import in settings to call this at startup and set `EMAIL_COLOR_PRIMARY`, `EMAIL_COLOR_FOREGROUND`, `EMAIL_COLOR_MUTED` on `django.conf.settings` if they aren't already explicitly set.

4. Add the color parsing call to `config/settings_base.py` after the existing settings:

```python
from freedom_ls.accounts.email_utils import parse_tailwind_colors
_tw_colors = parse_tailwind_colors(str(BASE_DIR / "tailwind.components.css"))
EMAIL_COLOR_PRIMARY = _tw_colors.get("primary", "#2B6CB0")
EMAIL_COLOR_FOREGROUND = _tw_colors.get("foreground", "#1A2332")
EMAIL_COLOR_MUTED = _tw_colors.get("muted", "#4A5568")
```

**Files to create**:
- `freedom_ls/accounts/email_utils.py`
- `freedom_ls/accounts/tests/test_email_utils.py`

**Files to edit**:
- `config/settings_base.py`

## Task 3: Create the email context processor

**Skills**: `testing` (TDD approach)

**Description**: Create a context processor that injects email settings into template context. This is needed so templates can access the brand colors, font family, and logo path.

**Steps**:
1. Add an `email_settings` context processor to `freedom_ls/accounts/context_processors.py`:

```python
def email_settings(request: HttpRequest) -> dict[str, str | None]:
    return {
        "email_color_primary": settings.EMAIL_COLOR_PRIMARY,
        "email_color_foreground": settings.EMAIL_COLOR_FOREGROUND,
        "email_color_muted": settings.EMAIL_COLOR_MUTED,
        "email_font_family": settings.EMAIL_FONT_FAMILY,
        "email_logo_static_path": settings.EMAIL_LOGO_STATIC_PATH,
    }
```

2. Register it in `config/settings_base.py` TEMPLATES context_processors list.

**Note**: Allauth's `render_mail` uses `get_template().render(context)` which goes through the template engine, so context processors registered in the TEMPLATES setting **are** applied if a `request` is available in the context. Allauth does pass `request` through its email rendering pipeline. If during testing we find context processors aren't being called, the fallback is to inject these values in the AccountAdapter's `send_mail` method instead.

**Files to edit**:
- `freedom_ls/accounts/context_processors.py`
- `config/settings_base.py`

**Files to create**:
- `freedom_ls/accounts/tests/test_context_processors.py` (or add to existing test file)

## Task 5: Override AccountAdapter.send_notification_mail

**Skills**: `testing` (TDD approach)

**Description**: Override `send_notification_mail` in the existing `AccountAdapter` to inject `user` into the template context, since allauth's default implementation doesn't include it.

**File to edit**: `freedom_ls/accounts/allauth_account_adapter.py`

```python
def send_notification_mail(self, template_prefix: str, user, context: dict | None = None) -> None:
    context = context or {}
    context["user"] = user
    super().send_notification_mail(template_prefix, user, context)
```

**Test file**: `freedom_ls/accounts/tests/test_allauth_account_adapter.py`

**Skills**: `testing` (TDD — tests should be written alongside each task above, but this task covers integration tests)

**Test file**: `freedom_ls/accounts/tests/test_email_templates.py`

**Tests to write**:
1. **Base template rendering**: Render `emails/base_email.html` with mock context and verify:

   - Header shows site name when no logo configured

   - Text version is present and readable
   - Context variables are rendered (e.g., activation URL, reset URL, login code)
   - Subject line is correct

4. **AccountAdapter.send_notification_mail**: Verify `user` is injected into context

5. **Color parser**: Verify `parse_tailwind_colors` correctly extracts colors from CSS (covered in Task 2)

6. **Overridability**: Verify a downstream project can override include files by placing templates at the same path

## TDD Approach

Each task should follow the RED-GREEN-REFACTOR cycle as described in `docs/testing.md`:
1. Write a failing test for the specific behavior
2. Implement the minimum code to make it pass
3. Refactor while keeping tests green
4. Repeat

Tests for Tasks 2, 3, and 5 should be written **before** the implementation code. Template tests (Tasks 4, 6, 7) can be written alongside the templates since template rendering tests require the template files to exist.

## Notes

- All templates use table-based layout with `role="presentation"` — no flexbox/grid
- The `{% premailer %}` tag in base_email.html means all child templates automatically get CSS inlining
- The `email_confirmation_signup` template should include/extend `email_confirmation` to avoid duplication
- Context processors only work when `request` is in the email context — verify this with allauth's rendering pipeline early; fallback is to use adapter's `send_mail` method
- Consult the `brand-guidelines` skill for all copy and styling decisions
