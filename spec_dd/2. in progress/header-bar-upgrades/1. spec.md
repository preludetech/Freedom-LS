# Specification: Header bar upgrades

## Context

The site header (`freedom_ls/base/templates/partials/header_bar.html` plus its
user-menu partial) is the single most-visible chrome surface in FLS. Three
shortcomings have surfaced:

1. The user-menu trigger shows the raw `display_name` (or, on small
   viewports, a generic user icon). This is visually noisy, doesn't scale,
   and gives every theme the same look.
2. Themes that want a different header colourway must currently rebrand the
   global role colours (`--color-primary` / `--color-on-primary`), which
   ripples through buttons, links, focus rings, chip tints, and so on.
3. The header scrolls away with the page. As learners move through long
   topic content, navigation and the user menu disappear when they're most
   useful.

This spec defines three coordinated upgrades to fix all three, scoped to the
header partial and a small amount of supporting work on the `User` model and
theme CSS.

The decisions below override the corresponding sections of `idea.md` and the
three research files (`research_avatar.md`, `research_header_tokens.md`,
`research_scroll_header.md`) wherever they differ. Research files remain in
place as evidence and rationale, not as the authoritative design.

---

## 1. Initials avatar in place of the name/email trigger

### Why

Replacing `{{ user.display_name }}` text (and the small-screen generic user
icon) with a circular initials avatar:

- Compacts the trigger to a fixed footprint (one circle), so the header
  layout no longer flexes with name length.
- Unifies desktop and small-screen treatment — one widget at every breakpoint
  rather than two divergent ones.
- Gives `first_class` (and any future light-header theme) a recognisable
  branded "profile pill", which the current text trigger cannot express.
- Builds the foundation for re-using the user's initials in other surfaces
  later (educator student lists, comment threads, cohort views) without
  re-implementing the derivation. We are not migrating those surfaces in
  this spec.

### Initials derivation

A `User.initials` property on the custom user model returns either a string
of one or two characters or `None`. Cascade — first match wins:

1. `first_name` and `last_name` both present, after `.strip()` →
   first letter of each.
2. Only `first_name` present → if it splits on whitespace into ≥2 tokens,
   take the first letter of the first two; otherwise take the first two
   letters of `first_name`.
3. Neither name is present → take the first two alphabetic characters of
   the email local-part (the part before `@`), skipping any leading
   non-alphabetic characters (digits, punctuation, emoji).
4. Nothing usable (e.g. `123@x.com`) → return `None`. The template renders
   a `c-icon name="user"` fallback.

Rules:

- Output is uppercased via Python's Unicode-aware `.upper()`.
- Diacritics are preserved (no ASCII folding). `Élise Önen` → `ÉÖ`.
- For non-Latin scripts (CJK, Arabic, Devanagari, etc.) the implementation
  returns a single grapheme — the first character of the chosen source —
  rather than attempting two. This matches the simpler cascade described in
  `idea.md` and is sufficient for the current FLS user base. The full
  grapheme-cluster handling sketched in `research_avatar.md` is out of scope.

This is simpler than the multi-branch algorithm in research because we don't
need email-local-part splitting on `.`/`_`/`-`/`+`, and we're not building a
deterministic colour palette. The simpler cascade has fewer edge cases to
test and is enough for what the avatar communicates.

### Decision: per-user colour hashing is out of scope

The research file recommends a deterministic email-hash → palette mapping so
two users with the same initials still look distinct. We are deliberately
**not** adopting that pattern in this spec. The avatar uses one themable
colour (see §2). Reasons:

- The header avatar represents *the current user only* — there is no
  collision problem in this surface.
- Adding a curated WCAG-validated palette is real design work that would
  need to be done per-theme (`default`, `first_class`, and any future
  theme).
- If/when the avatar is reused in surfaces with multiple users (educator
  student lists, comments), the per-user palette can be added then. The
  derivation in `User.initials` is independent of colour, so nothing here
  blocks that future work.

This is recorded as out-of-scope in the existing idea and is reaffirmed
here.

### Visual

- Circle, 36–40px (final value chosen during implementation planning),
  always `rounded-full`.
- Background: `bg-header-action` (see §2). In the default theme this
  resolves to the same hue as the header itself, so the avatar visually
  merges into the action region. In `first_class` it resolves to the
  primary colour, producing the indigo profile pill the brand wants.
- Foreground: `text-on-header-action`.
- Font weight semibold, font-size ≈ 40% of diameter, `line-height: 1`,
  `select-none`.
- The avatar replaces both the desktop `display_name` text and the
  small-screen generic user icon — one consistent display at every
  breakpoint.

### Decision: dropdown caret is removed

The current trigger has a `c-icon name="dropdown"` next to the user-menu
text, signalling "this opens a menu". With the avatar-only trigger we
follow Gmail / Atlassian / Microsoft 365 convention and **drop the caret
entirely**. Reasons:

- The avatar in a header position is a near-universal "click for user
  menu" signifier; the redundant caret adds chrome without adding meaning.
- The trigger keeps its `<button>` element with an explicit `aria-label`
  and `aria-haspopup="menu"`, so assistive tech still announces the
  affordance correctly. (`aria-haspopup` and `aria-expanded` are not
  currently emitted by the `c-dropdown-menu` cotton component or the
  user-menu partial — wiring them in is part of this work, not a
  preserved baseline.)

### Accessibility

- The dropdown trigger remains a `<button type="button">` with an
  `aria-label="Open user menu for {{ user.display_name }}"`.
  `aria-haspopup="menu"` is set on the trigger and `aria-expanded` is
  bound to the dropdown's open state (Alpine `x-bind:aria-expanded`),
  matching the pattern already used for the sidebar toggle in
  `_base_interface.html`.
- The visible initials are wrapped in `<span aria-hidden="true">` so
  screen readers announce the accessible name once.
- When `User.initials` returns `None`, the fallback `c-icon name="user"`
  is rendered with `aria-hidden="true"` for the same reason.
- Focus ring uses the existing `--color-focus-ring` token; no new focus
  styling is introduced.
- Contrast between `--color-header-action` and `--color-on-header-action`
  must hit WCAG AA (4.5:1 for the small initial glyphs) in every theme
  that ships these tokens. This is verified at QA time per theme.

## 2. Header-scoped theme tokens

### Why

The current header consumes `bg-primary` and `text-on-primary` directly. To
let `first_class` flip the header to white-with-an-indigo-pill, we'd have to
rebrand `--color-primary` itself, which would also affect every button,
link, focus ring, badge, and any other surface keyed off the brand role.
That's the wrong override surface.

The fix is a small component-tier token group, defaulted to the brand role
colours, that themes can override in isolation. This pattern is what
Material 3, IBM Carbon, and Adobe Spectrum all do for their top-app-bar
surfaces (see `research_header_tokens.md` §1).

### Tokens

Four tokens, no more:

| Token | Used for |
| --- | --- |
| `--color-header` | header background |
| `--color-on-header` | foreground on the header (site title, generic header text/icons) |
| `--color-header-action` | avatar (and any future header action chip) background |
| `--color-on-header-action` | foreground on the action (avatar initials, action icons) |

Tailwind v4 auto-generates `bg-header`, `text-on-header`,
`bg-header-action`, `text-on-header-action`, and the rest of the colour
utility family from these (`research_header_tokens.md` §4).

### Defaulting

Defaults are declared via `@theme inline` so themes that override
`--color-primary` propagate automatically into the header tokens, while
themes that explicitly set the new tokens win:

```css
@theme inline {
    --color-header: var(--color-primary);
    --color-on-header: var(--color-on-primary);
    --color-header-action: var(--color-primary);
    --color-on-header-action: var(--color-on-primary);
}
```

### Decision: `--color-header-action` defaults to `--color-primary`, not transparent

The research file recommends defaulting `--color-header-action` to
`transparent` so the default theme renders no visible pill (matching today's
look). The idea overrides this to default to `--color-primary`.

We follow the idea. Reasons:

- The avatar is the *only* consumer of `--color-header-action` in this
  spec. With `transparent` the avatar would render as bare initials on the
  header, which is uglier than the current text trigger and would force
  the default theme to special-case the avatar.
- With the avatar background equal to the header background, the avatar
  reads as a focused glyph inside the header in the default theme, and as
  a coloured pill in light-header themes (`first_class`). One token
  expresses both intents.
- If a future theme wants a literally transparent action background, it
  can override `--color-header-action: transparent;` explicitly. The
  default remaining solid is the better starting point.

### `first_class` overrides

`first_class/theme.css` redeclares all four tokens:

```css
@theme inline {
    --color-header: var(--color-surface);              /* white */
    --color-on-header: var(--color-on-surface);        /* dark text */
    --color-header-action: var(--color-primary);       /* indigo pill */
    --color-on-header-action: var(--color-on-primary); /* white initials */
}
```

### Deliberately not added

- `--color-header-hover` — existing `color-mix` hover machinery covers it.
- `--color-header-focus-ring` — `--color-focus-ring` already exists.
- `--color-header-border` — current header uses `shadow-md`, not a
  border. Adding a border token now is speculative.
- `--color-header-scrolled` — see §3; opacity-on-the-base-token via
  `bg-header/85` is sufficient.
- Separate icon vs text foreground tokens — icons in the current header
  inherit `currentColor` from the text utility.

These can be added later if a brand requires them.

## 3. Sticky header with theme-divergent on-scroll treatment

### Why

Learners scroll through long topic content. Hiding the header below the
fold means navigation, the user menu, and any future header-mounted action
all disappear when they're useful. Per NN/g's research summarised in
`research_scroll_header.md` §1, persistent headers measurably reduce
navigation time and are preferred even when users can't articulate why.

### Sticky, always visible

The header becomes `position: sticky; top: 0; z-30`, in flow, in both
themes.

### Decision: no auto-hide-on-scroll-down

`research_scroll_header.md` §1 considers and rejects auto-hide for an LMS:
adds JS scroll-direction state, complicates HTMX swap interactions, has
keyboard-focus edge cases, and the LMS context favours steady access to
nav. We affirm that decision here and do not implement auto-hide.

### On-scroll treatment per theme

Per-theme implementations of the visual change once the page has scrolled:

- **Default theme (coloured header):** stays opaque. A subtle shadow /
  border-bottom appears once `scrollY > 0`. **No `backdrop-blur`** —
  research confirmed it has no visual effect against an opaque background
  and just costs GPU. This matches NN/g's recommendation that sticky
  headers should be opaque and high-contrast.
- **`first_class` theme (white header):** at `scrollY == 0` the header is
  fully opaque and flush with the page; once scrolled it becomes
  `bg-header/85` (or similar alpha — final value at implementation time)
  with `backdrop-blur-md saturate-150`. This produces the iOS / Linear /
  Vercel "frosted glass" feel that a white-header theme can actually
  exploit, since real content shows through behind it. Contrast against
  worst-case page content is verified at QA per `first_class`.

The "scrolled" state is exposed to CSS via a class or attribute on the
header (e.g. `data-scrolled` or an Alpine-managed `scrolled` class). The
exact mechanism — Alpine + `scroll.window.passive` listener, vs the
CSS-only `@container scroll-state(stuck: top)` query — is decided during
implementation planning. Both paths satisfy this spec.

### Interaction with the existing dropdown-menu close-on-scroll

The shared `c-dropdown-menu` cotton component currently closes the menu
on `window` scroll (`x-on:scroll.window="close"`). With a sticky
header, the menu and its trigger remain on screen when the user scrolls,
so the auto-close becomes a more visible behaviour: the user could
scroll a few pixels and the menu vanishes underneath them. We accept
this for now — the close-on-scroll rule is shared by every dropdown in
the app and changing it is out of scope for this spec. If QA finds it
jarring on a sticky header specifically, a follow-up can scope the
close-on-scroll rule to non-sticky triggers.

### Layout / a11y guardrails

- `scroll-padding-top` (or `scroll-margin-top` on anchor targets) is set
  to the header height so deep links and keyboard focus aren't hidden
  behind the header. The exact placement (on `html`, on the scroll
  container, or per-target) is decided at implementation time.
- Ancestors of the header in `base.html` are audited for `overflow:
  hidden` / `transform` / `filter` — these silently break
  `position: sticky`.
- The colour/blur transition is wrapped in `motion-reduce:transition-none`
  so users with `prefers-reduced-motion` get an instant change rather than
  a fade. The static stuck state itself is not motion and is fine.
- Text contrast at the translucent setting in `first_class` is verified
  at QA time against typical and worst-case page content.

## Out of scope

These are not part of this spec. The reasons are recorded so a future
spec can build on them deliberately.

- **Per-user hash-coloured avatars.** Single themable colour for the
  current user only. Revisit if the avatar is reused in surfaces with
  multiple users.
- **Avatar uploads / profile pictures.** Initials only. The
  `<AvatarImage>`-with-`<AvatarFallback>` pattern from shadcn/ui (cited
  in research) is a possible future direction but introduces storage,
  upload, validation, and per-tenant policy concerns that don't belong
  here.
- **Auto-hide-on-scroll-down.** Deferred until a concrete mobile
  real-estate complaint exists.
- **`--color-header-scrolled` token.** Opacity on the base header token
  via `bg-header/85` is sufficient. Add only if a brand wants a different
  *hue* when scrolled.
- **`--color-header-hover`, `--color-header-focus-ring`,
  `--color-header-border`.** No current consumer; add later if a theme
  needs them.
- **Generalising the avatar to other surfaces.** `User.initials` is the
  reusable building block; no other call sites are migrated as part of
  this work.

## Decisions deferred to the implementation plan

These are real choices that don't affect the spec's contract and can be
made during planning:

- `User.initials` as `@property` vs `@cached_property` (request-lifetime
  caching may be valuable if the avatar appears multiple times on a page,
  but is irrelevant for the header alone).
- Final exact avatar diameter within the 36–40px range and the
  corresponding font-size.
- Mechanism for exposing the scrolled state to CSS — Alpine + scroll
  listener vs CSS `@container scroll-state(stuck: top)` (with or without
  a JS fallback).
- Where to apply `scroll-padding-top` (`html` element vs scroll container
  vs per-target `scroll-margin-top`).
- Final translucent alpha for `first_class` once scrolled (e.g. 80% vs
  85% vs 90%) — driven by QA contrast checks.

## Success criteria

The spec is satisfied when, with no manual configuration beyond switching
themes:

1. `User.initials` returns the correct one or two characters for each
   case in the cascade and `None` for the unrecoverable case, with unit
   tests covering: both names; first-name-only single token;
   first-name-only multi-token; email-only with an alphabetic
   local-part start; email-only with non-alphabetic local-part start;
   diacritics preserved; non-Latin script returns single grapheme;
   all-whitespace name strings.
2. The header in the default theme renders the avatar with brand-coloured
   background and white initials at every breakpoint, with no caret next
   to it, and with `display_name` text no longer present in the trigger.
3. The header in `first_class` renders the avatar as a primary-coloured
   indigo pill with white initials on a white header bar.
4. When `User.initials` is `None`, the trigger falls back to a
   `c-icon name="user"` glyph at the same size and shape.
5. The header stays at the top of the viewport in both themes when the
   page is scrolled, with `z-30` keeping it above content but below
   modal overlays.
6. In the default theme, scrolling reveals an additional shadow /
   border-bottom and no other visual change. No `backdrop-blur` is
   applied.
7. In `first_class`, scrolling switches the header to a translucent
   blurred state. At `scrollY == 0` the header is fully opaque and
   visually flush with the page top.
8. With `prefers-reduced-motion: reduce`, the on-scroll transition
   resolves instantly with no animated fade.
9. Anchor links and keyboard `:focus-visible` scrolls land below the
   sticky header (verified by tabbing through the topic page on both
   themes).
10. Header text and avatar contrast meet WCAG AA in both themes,
    including the translucent state in `first_class` against typical
    page content.
11. No existing button / link / chip / focus-ring colour changes as a
    side-effect of the header tokens being introduced. Themes that don't
    redeclare the new tokens get their existing look unchanged.
