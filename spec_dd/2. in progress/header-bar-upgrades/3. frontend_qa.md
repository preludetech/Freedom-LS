# Frontend QA: Header bar upgrades

Manual / Playwright-driven verification of the three coordinated upgrades:

1. `User.initials`-driven avatar trigger replaces the old name/email text trigger.
2. Header colour comes from new component-tier tokens (`--color-header*`).
3. Header is sticky with theme-divergent on-scroll treatment.

Use the `fls:use-playwright` skill (Playwright MCP) to drive the browser. Use the `fls:request-code-review` skill if any UI tweaks are required during QA.

---

## Setup

Always run against the **DemoDev** site (per `feedback_use_demodev_site.md`).

1. Pick a free port:

   ```bash
   PORT=$(.claude/fls/scripts/find_available_port.sh)
   ```

2. Build Tailwind so the new `bg-header*` / `text-on-header*` utilities are present:

   ```bash
   npm run tailwind_build
   ```

3. Boot the dev server (default theme first):

   ```bash
   FLS_THEME=default uv run python manage.py runserver $PORT
   ```

4. Base URL: `http://127.0.0.1:$PORT`. (Visit via 127.0.0.1, not `localhost`, so the cookie domain matches DemoDev.)

5. Test data — needs at least these users on DemoDev (create via the qa-data-helper agent if missing):

   - `mary.jane@demodev.example.com` — `first_name="Mary"`, `last_name="Jane"` (expected initials `MJ`).
   - `single.first@demodev.example.com` — `first_name="Mary"`, `last_name=""` (expected `MA`).
   - `multi.token@demodev.example.com` — `first_name="Mary Jane"`, `last_name=""` (expected `MJ`).
   - `noname@demodev.example.com` — `first_name=""`, `last_name=""` (expected `NO`).
   - `123@demodev.example.com` — `first_name=""`, `last_name=""` (expected fallback icon — local-part has no leading alpha).
   - `elise@demodev.example.com` — `first_name="Élise"`, `last_name="Önen"` (expected `ÉÖ`).

   Passwords for QA users follow the standard DemoDev convention (`UserFactory` defaults to email-as-password; if the demo seed differs use the documented DemoDev password).

6. There must be at least one **long topic page** on DemoDev — long enough that the page genuinely scrolls (≥ 1.5× viewport height). The default DemoDev course content satisfies this; if not, ask the qa-data-helper to add a long markdown topic.

## Test 1 — Default theme: avatar appearance

**Goal:** the avatar replaces text/icon trigger in the default theme; brand-coloured background; no caret.

1. With `FLS_THEME=default`, navigate to `http://127.0.0.1:$PORT/`.
2. Log in as `mary.jane@demodev.example.com`.
3. Inspect the top-right user-menu region.

Expected:

- A circular widget (`rounded-full`, ~40px diameter) with the text `MJ` inside, semibold, centered.
- Background colour is the brand primary (default theme primary `#2B6CB0`-ish) — the avatar visually merges with the header.
- Foreground (the `MJ` glyphs) is white-ish.
- **No** dropdown caret/chevron next to the avatar.
- **No** `Mary Jane` text or email shown in the trigger.

Resize the browser to mobile width (≤ 640px). The avatar still renders identically — no separate small-screen "user icon" treatment.

Take a screenshot for visual reference: `default-mary.png`.

## Test 3 — Default theme: dropdown opens, accessibility

1. Logged in as `mary.jane@demodev.example.com`.
2. Click the avatar.

Expected:

- The user-menu dropdown opens (Profile, optionally Educator Interface / Admin Panel for staff, Logout).
- The avatar button's `aria-label` reads `"Open user menu for Mary Jane"` (inspect via Playwright `accessibility.snapshot()` or DevTools).
- `aria-haspopup="menu"` is set on the button.
- `aria-expanded` flips between `"false"` (closed) and `"true"` (open) as the menu toggles.
- Tab focus from the address bar through the page lands on the avatar with a visible focus ring (the `--color-focus-ring` ring with offset).

Press Tab again — focus moves into the menu items in order. Press Escape — menu closes.

## Test 4 — Default theme: header sticky + scrolled state

1. Navigate to a long topic page on DemoDev (browse to any course → topic with long markdown content).
2. Confirm the page scrolls (page height > viewport height).
3. Scroll down ~ 200px.

Expected:

- The header stays at the top of the viewport (`position: sticky`).
- The header's box-shadow is visibly **stronger** than at `scrollY === 0` (`shadow-lg` instead of `shadow-md`).
- No backdrop-blur or translucency — the header remains fully opaque, brand-coloured.
- The avatar and site title remain readable; nothing slides under the header text.

Scroll back to the top. The shadow returns to its baseline (`shadow-md`).

## Test 5 — Default theme: anchor / focus scroll-padding

1. On the long topic page, find any in-page anchor link or use the keyboard to Tab through interactive elements far down the page.
2. When an element receives focus that would otherwise sit just below the header, confirm the browser scrolls so the focused element is **below** the header, not hidden behind it.

Expected: ~5rem (80px) of padding between the top of the focused element and the header bottom.

If anchor links exist on the topic page (e.g. table-of-contents jump links), click one and confirm the destination heading lands below the header.

## Test 6 — Default theme: reduced motion

1. Open browser DevTools → Rendering → "Emulate CSS media feature `prefers-reduced-motion: reduce`".
2. Scroll up and down on the long topic page.

Expected: the shadow change between baseline and scrolled is **instant** — no fade/transition. The static stuck state is fine (it's not motion).

## Test 7 — first_class theme: avatar appearance

Restart the dev server with `first_class`:

```bash
# Stop the previous runserver (Ctrl-C or kill the process).
.claude/fls/scripts/kill_runserver.sh
PORT=$(.claude/fls/scripts/find_available_port.sh)
FLS_THEME=first_class npm run tailwind_build && \
  FLS_THEME=first_class uv run python manage.py runserver $PORT
```

Log in as `mary.jane@demodev.example.com`.

Expected:

- The header background is **white-ish** (`--color-surface` resolves to `#F8F9FC`).
- The site title text is **dark** (`--color-on-surface`).
- The avatar reads as a **deep indigo** circle (`#283593` from first_class `--color-primary`) with **white** initials `MJ` — the indigo profile pill the brand wants.
- No caret, no name text, no separate small-screen icon treatment.

Take a screenshot for visual reference: `first_class-mary.png`.

## Test 8 — first_class theme: contrast

Use the Playwright MCP (or browser DevTools Accessibility panel) to check contrast ratios:

- White avatar text on the indigo `bg-header-action` → must be ≥ **4.5:1**.
- Dark site title (`text-on-header`) on the white `bg-header` → must be ≥ **4.5:1**.

Per `feedback_no_style_assertions.md` we don't bake these into automated tests, but the manual / Playwright accessibility audit must pass. If a colour combination fails, raise a follow-up to retune the token values rather than special-casing utilities.

## Test 9 — first_class theme: scrolled state (translucent + blur)

1. Navigate to the long topic page.
2. At `scrollY === 0`, observe the header.

Expected at the top:

- Header is **fully opaque** white-ish.
- Header sits **flush with the page top** — no visible seam between header and the page surface (which is also `bg-surface` in first_class).

3. Scroll down ~ 200px.

Expected once scrolled:

- The header transitions to a **translucent + frosted-glass** state.
- The page content underneath is partially visible through the header but **softened** by the backdrop blur.
- Header text and avatar remain readable.

Take a screenshot: `first_class-scrolled.png`.

4. Scroll back to the top. The header returns to fully opaque, flush state.

## Test 10 — first_class: contrast under translucency

While in the scrolled state on the long topic page:

1. Take a screenshot mid-scroll where there's a chunk of body text directly behind the header.
2. Sample the rendered colour at the avatar-initial pixels and the site-title pixels.
3. Confirm both still meet WCAG AA (4.5:1) against the text colour.

If contrast fails, the alpha is too low. Tune `bg-header/85` → `bg-header/90` (or a higher value) in `first_class/theme.css` and re-test.

## Test 11 — first_class: reduced motion

Same as Test 6 but in first_class. Confirm the colour-and-blur transition is **instant** when `prefers-reduced-motion: reduce` is emulated. The transient blur should not animate in.

## Test 12 — first_class: anchor / focus scroll-padding

Same as Test 5 but in first_class. Confirm focused elements and anchor targets land below the (now-translucent-when-scrolled) header.

## Test 13 — Both themes: dropdown close-on-scroll

The shared dropdown component closes on scroll (current behaviour, in scope to **leave alone** per the spec).

1. Open the user-menu dropdown.
2. Scroll the page slightly.

Expected: the dropdown closes. (Spec Section 3 acknowledges this may feel jarring on a sticky header and explicitly defers any change.) Note in the QA report whether the close-on-scroll feels acceptable on the now-sticky header — this informs the deferred follow-up but is not a blocker.

## Test 14 — Both themes: regression on existing surfaces

Spot-check that the new tokens did not accidentally repaint other surfaces (success criterion 11):

1. In default, navigate to the educator interface → some page with brand-coloured buttons / chips / focus rings. Confirm they look identical to pre-change (use git history to compare colours / take a "before" screenshot if uncertain).
2. In first_class, do the same. Buttons, chips, alerts, focus rings should be unaffected by header-token additions.

If anything visibly shifted, the most likely cause is that the new `--color-header*` declarations leaked outside the `header` selector. Re-check the `@theme inline` blocks.

## Test 15 — Both themes: sidebar / mobile sidebar still works

The `_base_interface.html` shell sits below the header. Quick check that the sticky header doesn't break the sidebar overlay on mobile:

1. Resize to mobile width.
2. Open a course / topic page that uses the sidebar shell.
3. Tap the sidebar toggle.

Expected: sidebar slides in, backdrop appears, sidebar closes when the backdrop is tapped. The sticky header sits above the backdrop's `z-30` (the header is also `z-30`; modals are `z-50`) — both are visible together.

If the sidebar backdrop covers the header in a way that blocks the avatar, that's a regression — file a fix to either lower the backdrop's `z-index` below the header or adjust the header's `z-index`.

## Done criteria

QA passes when:

- Every user in the test matrix shows the correct initials or fallback icon.
- Default theme stays opaque + brand-coloured + caret-less + sticky with shadow-step on scroll.
- first_class shows the indigo avatar pill on a white header, transitions to translucent+blur on scroll, opaque + flush at the top.
- Reduced-motion turns off transitions on both themes.
- Anchor / focus scrolls land below the header on both themes.
- WCAG AA contrast holds in both themes, including the translucent first_class state against typical content.
- No existing buttons / chips / focus rings / alerts changed colour.
