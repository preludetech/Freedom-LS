# Implementation Plan: Theme & site branding in emails (hardening + unification)

Derived from `1. spec.md`. This is a **hardening + unification** pass on the
existing branded-email system, not a greenfield build. Three areas of work:
colour-extraction hardening, theme-driven fonts/radius, and logo/label
unification + the absolute-URL bug fix. No new model, migration, per-site
theming, or upload mechanism.

Work proceeds **test-first (TDD)**: for each behaviour, write a failing test,
then implement the smallest change to make it pass. Do not pre-write the whole
test suite — write tests per task as you reach it.

## Area 1 — Harden theme-colour extraction (`email_utils.py`, `settings_base.py`)

### Task 1.1 — Widen raw-value capture; introduce a full-token parser

`fls:testing` first. Update the **existing** tests in `test_email_utils.py`
that assert non-hex tokens are skipped — that behaviour is intentionally
changing (spec Testing note). Then implement.

**Token-map key scheme (decided here, used everywhere).** To make `var()`
resolution unambiguous, the parser keys tokens by their **full custom-property
name minus the leading `--`** (e.g. `color-primary`, `fls-font-sans`,
`fls-radius-md`). A `var(--color-primary)` reference then resolves by stripping
the leading `--` and looking up `color-primary` directly — no special-casing of
the `color-` segment. (This also serves font/radius extraction in Area 2, so the
generalisation the spec mentions in 2.1 is introduced once, here.)

- Add `parse_tailwind_tokens(css_file_path) -> dict[str, str]`: regex
  `--([\w-]+)\s*:\s*([^;]+);`, returning `{full-name: raw-value.strip()}` for
  every `--<name>` token (e.g. `{"color-primary": "#2B6CB0", "fls-radius-md":
  "0.375rem", ...}`). Raises `FileNotFoundError` on a missing file (spec 1.4).
- Keep `parse_tailwind_colors(css_file_path)` as a thin back-compat filter over
  it: return `{name: raw for "color-<name>" in tokens}` (strips the `color-`
  prefix) so existing call sites and the de-prefixed `{"primary": ...}` shape
  are preserved. Other tests/call sites depend on this signature.

Tests to (re)write:
- A `--color-*` token written as `rgb(...)`, `hsl(...)`, `oklch(...)`,
  `color-mix(in oklch, ...)`, `var(--other)`, or a named colour is **captured**
  (raw string) by `parse_tailwind_colors`, not skipped.
- Hyphenated names still captured (keep existing test).
- `parse_tailwind_tokens` captures non-colour tokens too (`fls-radius-md`,
  `fls-font-sans`); `parse_tailwind_colors` still returns only `color-*` (so
  `--font-size-base`/`--spacing-lg` excluded from the colour dict).
- Missing-file `FileNotFoundError` preserved (keep existing test).

### Task 1.2 — Add `resolve_css_color(raw, token_map) -> str`

Pure helper in `email_utils.py`. `fls:testing` first (parametrized). Always
returns 6-digit `#rrggbb`; never returns modern syntax.

Pseudocode:

```python
from coloraide import Color

_VAR_RE = re.compile(r"var\(\s*(--[\w-]+)\s*\)")

class ColorResolveError(Exception):
    """Raised internally when a raw CSS colour cannot be resolved to hex."""

def resolve_css_color(raw: str, token_map: dict[str, str]) -> str:
    # token_map is the FULL map from parse_tailwind_tokens, keyed by
    # full-name-minus-"--" (e.g. "color-primary", "fls-hover-mix-color").
    resolved = _expand_vars(raw, token_map).strip()  # substitute var(--x)
    if resolved.startswith("color-mix("):
        return _resolve_color_mix(resolved, token_map)
    # hex / rgb / hsl / oklch / oklab / lch / lab / named all handled here:
    return Color(resolved).convert("srgb").to_string(hex=True)  # -> #rrggbb
```

- `resolve_css_color` takes the **full** token map (from
  `parse_tailwind_tokens`), not the de-prefixed colour dict — it needs to
  resolve `var()` references that may point at any token (e.g.
  `var(--fls-hover-mix-color)`).
- `_expand_vars(raw, token_map)`: repeatedly replace each `var(--x)` with
  `token_map[x]` where `x` is the matched name **with the leading `--`
  stripped** (so `var(--color-primary)` → `token_map["color-primary"]`). This
  matches the key scheme from Task 1.1 exactly — no `color-` special-casing.
  Loop until no `var(` remains, guarding with a `seen`/max-depth check; on a
  reference cycle or an unknown var name, raise `ColorResolveError` (the caller
  turns it into a warning + fallback). (The token map is already fully collected
  by the parser, so this is the substitution pass over those raw strings.)
- `_resolve_color_mix(value, token_map)`: purpose-built parser for
  `color-mix(in <space>, A [p1%], B [p2%])`. coloraide cannot parse the
  `color-mix()` *string*, so:
  - split top-level commas (respecting nested parens),
  - parse the space (e.g. `oklch`, `oklab`, `srgb`),
  - for each of the two colour args, recursively `resolve`/normalise the
    colour part and read the optional trailing percentage,
  - compute `weight` from the percentages (CSS rule: if only one pct `p` is
    given it belongs to A and B gets `100-p`; if neither, 50/50; coloraide's
    `mix(b, weight=...)` takes B's weight),
  - `Color(a).mix(b, space=space, weight=b_weight).convert("srgb").to_string(hex=True)`.
- Any parse/convert failure raises `ColorResolveError`.

Tests:
- Each format (hex 3/4/6/8, rgb/rgba, hsl, oklch, oklab, lch/lab, named) →
  valid `#rrggbb`.
- `var()` two-pass resolution returns the referenced colour.
- Cyclic `var()` reference → `ColorResolveError` (no infinite loop).
- `color-mix(in oklch, A 30%, B)` → a hex plausibly between A and B
  (assert it's a valid `#rrggbb` and not equal to either endpoint).

### Task 1.3 — Resolve the seven `EMAIL_COLOR_*` through the new helper, with warn-on-failure

Put the warn-on-failure resolution in **`email_utils.py`** (not in settings) so
the system check (Task 4) reuses the exact same logic — DRY, settings stays
thin. Add:

```python
def resolve_color_token(
    token_map: dict[str, str], token: str, fallback: str
) -> str:
    """Resolve full token `color-<token>` to #rrggbb; warn + fallback on failure."""
    raw = token_map.get(f"color-{token}")
    if raw is None:
        warnings.warn(
            f"Email colour token --color-{token} not found; using fallback "
            f"{fallback}.", UserWarning, stacklevel=2)
        return fallback
    try:
        return resolve_css_color(raw, token_map)
    except (ColorResolveError, ValueError) as exc:
        warnings.warn(
            f"Email colour token --color-{token}={raw!r} could not be resolved "
            f"({exc}); using fallback {fallback}.", UserWarning, stacklevel=2)
        return fallback
```

Then in `config/settings_base.py`, replace the seven `.get(...)` lines
(token→fallback pairs unchanged):

```python
from freedom_ls.accounts.email_utils import (
    parse_tailwind_tokens, resolve_color_token,
)

_tw_tokens = parse_tailwind_tokens(str(_theme_css))   # full raw-value map

EMAIL_COLOR_PRIMARY    = resolve_color_token(_tw_tokens, "primary",    "#2B6CB0")
EMAIL_COLOR_FOREGROUND = resolve_color_token(_tw_tokens, "on-surface", "#1A2332")
EMAIL_COLOR_MUTED      = resolve_color_token(_tw_tokens, "muted",      "#4A5568")
EMAIL_COLOR_SURFACE    = resolve_color_token(_tw_tokens, "surface",    "#FFFFFF")
EMAIL_COLOR_SURFACE_2  = resolve_color_token(_tw_tokens, "surface-2",  "#F3F4F6")
EMAIL_COLOR_ON_PRIMARY = resolve_color_token(_tw_tokens, "on-primary", "#FFFFFF")
EMAIL_COLOR_BORDER     = resolve_color_token(_tw_tokens, "border",     "#D1D5DB")
```

- `resolve_color_token` catches only the specific exceptions our helper/coloraide
  can raise (`ColorResolveError`, `ValueError`); **no** bare-`except`. It never
  raises (startup must not be blocked). `FileNotFoundError` from
  `parse_tailwind_tokens` still propagates (spec 1.4).
- Type hints on all functions, `X | None` syntax, no `Any` (per `CLAUDE.md`).

Tests (`test_email_utils.py`, pure — drive the helper directly with crafted
token maps, not real settings):
- Missing token → `UserWarning` (use `pytest.warns`) + returns the fallback.
- Unparseable value → `UserWarning` + returns the fallback.
- A token whose value is a **cyclic `var()`** reference → `UserWarning` +
  fallback (the `ColorResolveError` path, not an infinite loop).
- Real `default` and `first_class` token maps resolve the seven values to
  exactly today's hex (back-compat oracle — hard-code expected hex).

### Task 2.2 — Derive `EMAIL_FONT_FAMILY` (email-safe subset)

`fls:testing` first. Add a pure helper, e.g.
`email_safe_font_stack(raw_font_sans: str) -> str` in `email_utils.py`.

Rules (spec 2.2):
- Split the stack on top-level commas; trim and de-quote each entry.
- Drop CSS keywords that aren't real font names: `system-ui`, `ui-sans-serif`,
  `ui-monospace`, `-apple-system` (spec 2.2; treat other `ui-*` keywords the
  same way if encountered).
- Drop names **not** on the web-safe allowlist: `Arial`, `Helvetica`,
  `Helvetica Neue`, `Verdana`, `Georgia`, `Times New Roman`, `Courier New`,
  `Trebuchet MS`, `Tahoma` (case-insensitive compare).
- Always keep a trailing generic family (`sans-serif`/`serif`/`monospace`); if
  none present, append `sans-serif`.
- Re-quote multi-word names that need it (e.g. `"Helvetica Neue"`).
- If stripping leaves only a generic family **but** a custom (non-allowlisted,
  non-keyword) name was present, emit `warnings.warn(..., UserWarning)` (a
  custom font alone won't render in most clients).
- Examples to assert:
  `default` → `"Helvetica Neue", Arial, sans-serif`;
  `first_class` (`"DM Sans", system-ui, sans-serif`) → `sans-serif` (+ warning).

In `settings_base.py`, set `EMAIL_FONT_FAMILY` from
`email_safe_font_stack(_tw_tokens["fls-font-sans"])`. If `fls-font-sans` is
**missing** from the map, warn and fall back to the current hardcoded
`"Arial, Helvetica, sans-serif"` (e.g. guard with `if "fls-font-sans" in
_tw_tokens` / `.get(...)` before calling the helper). Setting name,
context-processor key (`email_font_family`), and template usage are unchanged.

Tests:
- Mixed custom+keyword stack → allowlisted names + generic only.
- Custom-only stack (`"DM Sans", system-ui, sans-serif`) → generic + warning.
- A **missing** `--fls-font-sans` token → fallback `"Arial, Helvetica,
  sans-serif"` + warning.
- Both shipped themes produce the documented stacks.

## Area 3 — Unify logo & label, fix the absolute-URL bug

### Task 3.1 — Logo path precedence (single helper)

- Add a single helper, e.g. `resolved_email_logo_path() -> str | None` returning

  reuse it — one source of truth for the precedence.
- **Remove** `email_logo_static_path` from the `email_settings` context
  processor (and drop it from the test contexts that set it). `send_mail` is the
  only place that needs the path, and it consumes the helper directly. This
  honours the spec's precedence intent while avoiding dead code.

### Task 3.2 — Absolute logo URL + label in `AccountAdapter.send_mail`

This is the real bug fix. `fls:testing` first (write a failing test asserting
the rendered `<img src>` is fully-qualified). Edit
`allauth_account_adapter.py` `send_mail` (keep it in sync with the parent as the
existing comment warns).

Pseudocode (extend the existing `ctx`):

```python
from django.templatetags.static import static

request = allauth_context.request
current_site = get_current_site(request)

logo_path = resolved_email_logo_path()   # helper from Task 3.1 (single source)
protocol = getattr(settings, "ACCOUNT_DEFAULT_HTTP_PROTOCOL", "https")

    f"{protocol}://{current_site.domain}{static(logo_path)}" if logo_path else None
)
email_label = settings.HEADER_TITLE or current_site.name

ctx = {
    "request": request,
    "email": email,
    "current_site": current_site,
    "email_logo_url": email_logo_url,
    "email_label": email_label,
}
ctx.update(context)
```

  `protocol://domain` makes it absolute. Works without a live request (the
  scheme comes from settings, not the request) — fine for management commands /
  background tasks; dev mail goes to Mailpit where scheme is irrelevant.
- `current_site.name` is the final label fallback (Django `Site` always has a
  name).
