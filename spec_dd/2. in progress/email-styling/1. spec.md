# Spec: Theme & site branding in emails (hardening + unification)

## Summary

The Freedom Learning System already ships a branded transactional-email system
(allauth verification, password reset, login code, and notification mails). Theme
colours are extracted from the active theme's `theme.css` into `EMAIL_COLOR_*`
settings, exposed via the `email_settings` context processor, and inlined into
the email templates by a premailer template tag. The header shows a logo when
`EMAIL_LOGO_STATIC_PATH` is set, otherwise the site name.

This is **not a greenfield build**. It is a hardening + unification pass on the
existing system, in three parts:

1. **Harden theme-colour extraction** so it never silently falls back to a wrong
   colour and never leaks modern CSS colour syntax into email HTML.
2. **Drive fonts and the button radius from theme tokens** (email-safe subset).
3. **Unify the email logo and label with the existing web-header branding**, and
   fix a real bug where the logo URL is root-relative and never loads in email
   clients.

Theme remains **global** (one `FLS_THEME` env var). This work does **not**
introduce per-site theming, no new model, no migration, and no uploaded logos.

## Background: current state (verified)

Grounded in a codebase audit (`.sdd-work/` during authoring; key facts below):

- `parse_tailwind_colors` (`freedom_ls/accounts/email_utils.py`) uses
  `r"--color-([\w-]+):\s*(#[0-9A-Fa-f]{3,8});"` — it matches **only** hex values
  and silently skips any `--color-*` token written as `oklch()`, `color-mix()`,
  `var()`, `rgb()`, or a named colour. It raises `FileNotFoundError` on a missing
  file.
- Seven `EMAIL_COLOR_*` settings are produced in `config/settings_base.py` with
  hardcoded hex fallbacks (`.get("primary", "#2B6CB0")`, etc.). All base tokens in
  both shipped themes (`default`, `first_class`) are plain hex today, so the regex
  works **today** — but the same theme files already use `color-mix()`/`var()` for
  derived tokens, so a future base token in modern syntax would fail silently.
- `EMAIL_FONT_FAMILY` is hardcoded to `"Arial, Helvetica, sans-serif"` — not tied
  to the theme's `--fls-font-sans` (e.g. `first_class` uses `"DM Sans"`).
- `.email-button` in `base_email.html` hardcodes `border-radius: 6px` — not tied
  to `--fls-radius-md` (`0.375rem` default, `0.5rem` first_class).
- `emails/includes/header.html` builds the logo `src` with
  `{% static email_logo_static_path %}`, yielding a root-relative `/static/...`
  path that does **not load** in external email clients. The `<img>` has no
  `width`/`height` attributes. Alt text and the text fallback use
  `{{ current_site.name }}` directly, skipping `HEADER_TITLE`.
- `EMAIL_LOGO_STATIC_PATH` (default `None`) is entirely separate from the web
  header's `HEADER_LOGO_STATIC_PATH`; in dev the web header shows a logo while the
  email header shows text.
- `email_settings` context processor (`freedom_ls/accounts/context_processors.py`)
  injects `email_color_*`, `email_font_family`, `email_logo_static_path`. It has
  **no** access to `current_site`. The web header's `site_config` processor
  computes `header_title = settings.HEADER_TITLE or site_title`.
- `AccountAdapter.send_mail` (`freedom_ls/accounts/allauth_account_adapter.py`)
  builds `ctx` with `current_site = get_current_site(request)`; `request` (from
  `allauth.core.context`) **can be `None`** outside a request cycle.
  `ACCOUNT_DEFAULT_HTTP_PROTOCOL = "https"` is set only in `settings_prod.py`.
- Neither `coloraide` nor `tinycss2` is in `pyproject.toml` / `uv.lock`.
- `freedom_ls/accounts/checks.py` has an existing system-check pattern (deferred
  DB imports, catch `DatabaseError`, return `list[Warning]`, namespaced IDs).
- Tests: `test_email_utils.py` covers `parse_tailwind_colors`;
  `test_email_templates.py` uses `render_to_string`.

## Goals / Non-goals

### Goals

- Theme-colour extraction always resolves every base email token to `#rrggbb`,
  converting modern CSS colour syntax in Python before it reaches email HTML.
- Unparseable/missing tokens fall back to the existing hardcoded default
  **visibly** (a `warnings.warn`), never silently and never blocking startup.
- The email font stack and button radius derive from theme tokens, reduced to an
  email-safe subset.
- The email logo and label are unified with the web-header settings, the logo URL
  is absolute (loads in real clients), and there is always something branded
  visible (logo image, else text label, else `Site.name`).

### Non-goals

- No per-site theme/branding model, DB fields, migration, or uploaded logos.
- No per-site theming — theme stays global via `FLS_THEME`.
- No VML/Outlook hacks for rounded buttons (Outlook ignoring `border-radius` is
  acceptable degradation).
- No dark-mode logo variants (a transparent PNG is sufficient).
- No change to which email types exist or to the premailer inlining mechanism.

## Decisions

| Decision | Choice | Why |
|---|---|---|
| Colour conversion | **Add `coloraide`** and do full conversion (hex/rgb/hsl/oklch/oklab/`color-mix()`/`var()` → hex) | User-selected. Pure-Python (~350KB), no runtime deps beyond Py≥3.10, covers every format that can appear in a Tailwind v4 theme. Robust against future themes, not just today's all-hex base tokens. |
| CSS parsing | **Hardened two-pass regex**, not `tinycss2` | One known `@theme` block with a known property namespace; a widened regex (capture everything up to `;`, classify in Python) is simpler, independently testable, and adds no parser dependency. |
| Logo URL composition | **In `AccountAdapter.send_mail`** | User-selected. `current_site` (needed for the domain) is already at hand there; the `email_settings` context processor has no `current_site`. Keeps the template trivial (`{% if email_logo_url %}`). |
| Logo settings precedence | `EMAIL_LOGO_STATIC_PATH or HEADER_LOGO_STATIC_PATH` | Strictly additive: a site that sets a web-header logo automatically gets it in email, while the explicit email setting remains an override/escape-hatch. `None` default keeps existing installs unchanged. |
| Scheme when `request` is `None` | `ACCOUNT_DEFAULT_HTTP_PROTOCOL` (default `"https"`) | Already the single source of truth for email link scheme; works without a live request (management commands, background tasks). Dev mail goes to Mailpit, where scheme is irrelevant. |
| Failure mode | `warnings.warn(UserWarning)` at settings-load + hardcoded fallback | Surfaces in the process log at import time without a DB/registry; does not block startup. `ImproperlyConfigured` rejected (a bad token should not stop the server — the fallback is a safe degraded state). |
| Logo format | **PNG, not SVG** | PNG is universally supported in email; embedded SVG ~40%, linked SVG ~93% but flaky in Gmail for non-Google accounts. |

## Requirements

### 1. Harden theme-colour extraction

1.1. Widen the parser to capture the **raw** value of each `--color-*` token
(everything up to the first `;`): `--color-([\w-]+)\s*:\s*([^;]+);`. Return a dict
of `{token-name: raw-value-string}` for `--color-*` tokens.

1.2. Add a pure helper, e.g. `resolve_css_color(raw: str, token_map: dict[str, str])
-> str`, that:
- Resolves `var(--x)` references against the full token map in a **two-pass**
  manner (collect raw strings first, then substitute), detecting cycles and
  treating them as parse failures.
- Parses `color-mix(in <space>, A [pct], B [pct])` via a purpose-built parser and
  delegates the mix to `coloraide` (`Color(a).mix(b, space=..., weight=...)`),
  since coloraide does not parse `color-mix()` strings directly.
- Delegates every other format (hex 3/4/6/8-digit, `rgb()`/`rgba()`, `hsl()`,
  `oklch()`, `oklab()`, `lch()`/`lab()`, named colours) to
  `coloraide.Color(raw).convert("srgb").to_string(hex=True)`.
- **Always returns 6-digit `#rrggbb`.** Never returns `oklch()`/`color-mix()`/
  `var()`/`rgb()` — Gmail strips the entire inline style attribute on unsupported
  syntax.

1.3. For each of the seven `EMAIL_COLOR_*` settings, resolve the corresponding
token through `resolve_css_color`. On a **missing token** or a **conversion
failure**, emit `warnings.warn(..., UserWarning, stacklevel=...)` naming the token,
the offending value (if any), and the fallback, then use the existing hardcoded hex
default. Startup must not be blocked. The seven tokens and their current fallbacks
are unchanged (`primary→#2B6CB0`, `on-surface→#1A2332`, `muted→#4A5568`,
`surface→#FFFFFF`, `surface-2→#F3F4F6`, `on-primary→#FFFFFF`, `border→#D1D5DB`).

1.4. Preserve the existing `FileNotFoundError` behaviour for a missing CSS file
(theme misconfiguration is a hard error today; out of scope to change).

### 2. Fonts & button radius from theme tokens

2.1. Extend extraction to read `--fls-font-sans` and `--fls-radius-md` from the
same `theme.css` (the widened raw-value capture generalises to any `--<name>`
token; radius/font values need no colour conversion).

2.2. **`EMAIL_FONT_FAMILY`** is derived from `--fls-font-sans` but reduced to an
email-safe subset:
- Drop CSS keywords that are not real font names: `system-ui`, `ui-sans-serif`,
  `ui-monospace`, `-apple-system`.
- Drop quoted custom/web-font names not on a conservative web-safe allowlist
  (Arial, Helvetica, "Helvetica Neue", Verdana, Georgia, "Times New Roman",
  "Courier New", "Trebuchet MS", Tahoma).
- Keep the trailing generic family (`sans-serif`/`serif`/`monospace`).
- If stripping would leave only a generic family but a custom name was present,
  emit a `warnings.warn` (a custom font alone won't render in most email clients).
- Examples: default theme → `"Helvetica Neue", Arial, sans-serif`; `first_class`
  (`"DM Sans", system-ui, sans-serif`) → `sans-serif`.
- The setting name stays `EMAIL_FONT_FAMILY`; the context-processor key and the
  template (`{{ email_font_family }}`) do not change. On extraction failure, fall
  back to the current hardcoded `"Arial, Helvetica, sans-serif"` with a warning.

2.3. **Button radius**: add a new setting (e.g. `EMAIL_BUTTON_RADIUS`, default
`"6px"`) resolved from `--fls-radius-md` (a plain length literal — passed through
as-is, no conversion). Expose it via the `email_settings` context processor as
`email_button_radius` and use it in `.email-button` in `base_email.html` in place
of the hardcoded `border-radius: 6px`. Outlook ignoring it is acceptable; no VML
workaround. Button background/text already come from theme tokens — unchanged.

### 3. Unify logo & label, fix the absolute-URL bug

3.1. **Logo path precedence**: in the `email_settings` context processor, set the
logo path to `settings.EMAIL_LOGO_STATIC_PATH or settings.HEADER_LOGO_STATIC_PATH`
(currently it passes `EMAIL_LOGO_STATIC_PATH` straight through).

3.2. **Absolute logo URL** (the real bug fix): compose the fully-qualified URL in
`AccountAdapter.send_mail`, where `current_site` is available, and inject it into
the email context as `email_logo_url`:
- `protocol = getattr(settings, "ACCOUNT_DEFAULT_HTTP_PROTOCOL", "https")`
- `email_logo_url = f"{protocol}://{current_site.domain}{static(logo_path)}"` when
  a logo path resolves, else `None`.
- The template uses `{% if email_logo_url %}<img src="{{ email_logo_url }}" ...>`,
  removing the broken `{% static email_logo_static_path %}` usage.

3.3. **Label precedence**: compute the resolved label
`email_label = settings.HEADER_TITLE or current_site.name` in `send_mail` and inject
it. Use it for the `<img alt>`, the `<h1>` text fallback (HTML header), and the
plain-text base template's site identity. Today these use `current_site.name` and
skip `HEADER_TITLE`. `current_site.name` remains the final fallback (Django `Site`
always has a name).

3.4. **Header markup** (`emails/includes/header.html`): when `email_logo_url` is
set, render `<img>` with the absolute `src`, `alt="{{ email_label }}"`, an explicit
HTML `width` attribute (for Outlook), `style` keeping `height:auto; display:block;
border:0;`, and a `max-height` constraint. Otherwise render the `<h1>` text label.

3.5. **Plain-text** (`emails/base_email.txt` and any sign-off/greeting using the
site name): use the resolved `email_label` where the site label is shown. No image
references in plain text. `current_site.domain` stays as-is in the footer line.

3.6. **Asset**: ensure a PNG logo with a transparent background is used (rasterise
from SVG at ~2x if needed). The web header may keep using SVG; only the email asset
must be PNG. No new upload mechanism — this is a static asset referenced by the
existing `*_STATIC_PATH` settings.

### 4. Dependency & checks

4.1. Add `coloraide` via `uv add coloraide`.

4.2. (Optional, recommended) Add a Django system check in
`freedom_ls/accounts/checks.py` following the existing pattern that re-resolves the
theme colour tokens and surfaces a `Warning` (namespaced ID, e.g.
`freedom_ls_accounts.W00N`) for any token that failed to parse/convert — giving CI
visibility for the same condition the load-time `warnings.warn` reports. This is
additive and must not raise.

## Testing

Follow project convention: assert on **string content / functionality**, not CSS
classes or inline-style strings. Use `render_to_string` as the existing email
template tests do.

### Colour extraction / conversion (`test_email_utils.py` + new)

- Raw-value capture: a `--color-*` token written as `rgb(...)`, `hsl(...)`,
  `oklch(...)`, `color-mix(in oklch, ...)`, `var(--other)`, or a named colour is
  captured (not skipped) and converted to a valid `#rrggbb`.
- `var()` two-pass resolution works; a cyclic `var()` reference is treated as a
  parse failure (warning + fallback), not an infinite loop.
- `color-mix(in oklch, A pct, B)` resolves to a plausible hex between the inputs.
- A missing token and an unparseable value each emit a `UserWarning` and return the
  hardcoded fallback; settings still load (no exception).
- Existing hex behaviour for both shipped themes is unchanged (the seven
  `EMAIL_COLOR_*` resolve to the same values as today).
- Note: the existing test asserting `rgb()`/named colours are *not* captured must
  be updated — that behaviour is intentionally changing.

### Font / radius

- `EMAIL_FONT_FAMILY` derived from a `--fls-font-sans` containing custom + keyword
  entries yields an email-safe stack (custom names and CSS keywords stripped,
  generic family retained); a custom-only stack triggers a warning.
- `EMAIL_BUTTON_RADIUS` reflects `--fls-radius-md`; rendered button HTML contains
  the resolved radius value rather than a stale hardcoded one.

### Logo / label (`test_email_templates.py`)

- With `HEADER_LOGO_STATIC_PATH` set and `EMAIL_LOGO_STATIC_PATH` `None`: rendered
  HTML contains `<img` whose `src` is a fully-qualified URL (`http://`/`https://`),
  not a bare `/static/` path.
- `<img alt>` equals the resolved label: `HEADER_TITLE` when set, else `Site.name`.
- With neither logo setting set: HTML contains the `<h1>` text label and no `<img`.
- Plain-text output contains the resolved label and no `/static/` URL.
- Back-compat: with neither `HEADER_TITLE` nor any logo configured, output matches
  current behaviour (site name as text).

### Manual QA (Mailpit)

Trigger verification, password-reset, and login-code emails; in Mailpit confirm the
logo loads from the absolute URL (raw view), the alt text shows when images are
off, and colours/font/button radius reflect the active theme.

## Success criteria

1. No base email colour token can silently produce the wrong colour: any
   missing/unparseable token emits a `UserWarning` and uses the documented hex
   fallback; startup is never blocked.
2. No email HTML ever contains `oklch()`, `oklab()`, `color-mix()`, `var()`, or
   `hsl()` in a colour position — every email colour is `#rrggbb`.
3. `EMAIL_FONT_FAMILY` and the button radius reflect the active theme, reduced to
   an email-safe subset; `first_class` no longer mismatches web typography beyond
   the email-safe fallback.
4. The email logo loads in external clients (absolute URL), defaults to the
   web-header logo, and the explicit email setting still overrides it.
5. There is always visible branding: logo image if configured, else the text label
   (`HEADER_TITLE → Site.name`); alt text equals that label.
6. `coloraide` is added; existing email tests pass with updated expectations; new
   tests cover conversion, fallback-with-warning, font stripping, radius, and the
   logo/label precedence.
7. No per-site model, migration, or upload is introduced; theme stays global.
