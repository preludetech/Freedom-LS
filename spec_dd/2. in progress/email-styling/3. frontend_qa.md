# Frontend QA: Theme & site branding in emails

Manual browser QA for the email branding work. Emails are rendered HTML, so the
"frontend" under test is what arrives in the **Mailpit** inbox. We trigger real
emails through the dev server and inspect them in Mailpit.

## Prerequisites

- Dev DB and Mailpit running (the dev `docker-compose` in `dev_db/` provides
  Mailpit; SMTP on `localhost:1025`, web UI on **http://localhost:8025**).
- Dev settings active. Relevant dev config (already set in
  `config/settings_dev.py`):
  - `FORCE_SITE_NAME = "DemoDev"` → `current_site.name` is `DemoDev`.
  - `HEADER_LOGO_STATIC_PATH = "images/first_class_logo.png"`,
    `EMAIL_LOGO_STATIC_PATH` unset (None) → email logo resolves via the
    header-logo precedence to `first_class_logo.png`.
  - `HEADER_TITLE = "FirstClass"` → the resolved **email label** is `FirstClass`
    (overrides the site name for alt text / text fallback).

## Start the server

Use a random port (not the default), per project convention:

```bash
PORT=$(.claude/fls/scripts/find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL: `http://127.0.0.1:$PORT`. Mailpit UI: `http://localhost:8025`.

Before each test, you may clear the Mailpit inbox (Delete all) so the newest
message is unambiguous.

## Test data

Use the **DemoDev** site. Create a QA student if you need a known account:

```bash
uv run python manage.py qa_create_course_player_student DemoDev
```

This creates/uses `demodev_s1@email.com` with password equal to the email
(`demodev_s1@email.com`).

## Test 1 — Email verification email (signup flow)

Verification is mandatory (`ACCOUNT_EMAIL_VERIFICATION = "mandatory"`), so signup
sends a confirmation email.

1. Go to `http://127.0.0.1:$PORT/accounts/signup/`.
2. Register a new account (use a fresh email, e.g. `qa_verify_1@email.com`, and
   accept terms if prompted).
3. Open Mailpit at `http://localhost:8025` and open the newest message
   ("Confirm your email address").

**Expect:**
- The **header logo image loads** — you see the FirstClass logo at the top, not
  a broken-image icon. (This is the core bug fix: the `src` must be an absolute
  `http://.../static/images/first_class_logo.png`, not a bare `/static/...`.)
- In Mailpit, open the **Source / raw** view (or the "HTML" source tab) and
  confirm the logo `<img src=...>` is a **fully-qualified URL** beginning with
  `http://` (the domain comes from the current site), not `/static/...`.
- Turn images off (or read the `alt`): the logo's **alt text reads `FirstClass`**
  (the resolved label = `HEADER_TITLE`), not `DemoDev`.
- Colours match the active theme: header band, button, and links use the theme
  primary colour; body text uses the theme foreground.
- **No raw modern CSS colour syntax** appears anywhere — searching the HTML
  source for `oklch`, `oklab`, `color-mix`, `var(`, or `hsl(` in a colour
  position returns nothing; every colour is a `#rrggbb` hex.
- The CTA button has rounded corners reflecting the theme radius
  (`first_class` → `0.5rem`; default theme → `0.375rem`). (Outlook may show
  square corners — acceptable.)
- The font in the body is a readable sans-serif (email-safe stack), not a
  missing web font.

## Test 2 — Password reset email

1. Go to `http://127.0.0.1:$PORT/accounts/password/reset/`.
2. Enter the QA student email (`demodev_s1@email.com`) and submit.
3. In Mailpit, open the newest message ("Reset your password").

**Expect:**
- Same branding as Test 1: logo image loads from an absolute URL, alt text is
  `FirstClass`, theme colours/font/button-radius applied, no modern colour
  syntax in the source.
- The "Reset Password" button/link is present and styled with the theme primary
  colour.
- The sign-off reads **"The FirstClass Team"** (resolved label), and the footer
  shows the site domain.

## Test 3 — Login-code email (if login-by-code is enabled)

Only if the request-login-code page is available
(`http://127.0.0.1:$PORT/accounts/login/code/`). If it 404s, skip this test.

1. Go to `http://127.0.0.1:$PORT/accounts/login/code/`.
2. Enter `demodev_s1@email.com` and request a code.
3. In Mailpit, open the newest message ("Your login code").

**Expect:**
- The 6-digit code is clearly displayed.
- Same branding checks as Tests 1–2 (logo from absolute URL, `FirstClass` label,
  theme colours/font/radius, no modern colour syntax).

---

## Test 4 — Plain-text part

For any of the above emails, in Mailpit switch to the **Text** tab.

**Expect:**
- The site identity / sign-off uses the label **`FirstClass`** (e.g. "The
  FirstClass Team").
- There are **no image references** and **no `/static/...` URLs** in the text
  part.
- The footer line still shows the site domain.

---

## Test 5 — Text-label fallback (no logo configured)

Verifies branding still shows when no logo is set.

1. Temporarily unset both logo settings for the dev run, e.g. start the server
   with them blanked (edit `config/settings_dev.py` to set
   `HEADER_LOGO_STATIC_PATH = None` and leave `EMAIL_LOGO_STATIC_PATH` unset, or
   override via env), then restart `runserver`.
2. Trigger any email (repeat Test 2 is easiest).
3. Open the newest message in Mailpit.

**Expect:**
- No `<img>` logo; instead a **text heading** showing the label `FirstClass`
  (or `DemoDev` if you also clear `HEADER_TITLE`) rendered in the header band.
- Everything else (colours, font, button) still themed.
4. Restore `config/settings_dev.py` afterwards.

---

## Test 6 — Email logo override precedence (optional)

1. In `config/settings_dev.py`, set `EMAIL_LOGO_STATIC_PATH` to a *different*
   existing static image path and restart `runserver`.
2. Trigger an email and view it in Mailpit.

**Expect:**
- The email shows the **`EMAIL_LOGO_STATIC_PATH`** image (the explicit email
  setting wins over `HEADER_LOGO_STATIC_PATH`).
3. Restore `config/settings_dev.py` afterwards.
