# Implementation Plan: Student interface ŌĆö course test/exam UI

This plan implements `1. spec.md`. It is a **presentation re-skin of the existing Form (test/exam)
flow** plus one small optional model field (`Form.submit_on_exit`) and a server-side safety net for
submit-on-exit. Read the spec and the two research files
(`research_theming_and_components.md`, `research_runner_interactions.md`) alongside this plan.

We follow **TDD**: for each behavioural change write a failing test first, then implement. Tests assert
**functionality, not CSS classes**.

---

## 0. Key decisions baked into this plan

- **Start & results screens keep their current chrome** ŌĆö they continue to extend
  `student_interface/_course_base.html` (global header + course TOC sidebar). This is "normal app
  chrome" per spec ┬¦1/┬¦3 and success criterion 2, which contrasts them with the *sidebar-less* runner.
  We only restyle their content; we do **not** add new base templates for them. (The research file
  ┬¦4 floated extending `_base.html` directly for these two; we deliberately keep the existing
  `_course_base.html` to stay minimal and low-risk ŌĆö "re-skin of an existing flow".)
- **The runner gets a new sidebar-less base** (`_exam_runner_base.html`) that extends `_base.html`,
  suppresses the global header, and owns the full viewport.
- **We keep the existing view ŌåÆ template wiring and template filenames** (`course_form.html`,
  `course_form_page.html`, `course_form_complete.html`). We change `course_form_page.html`'s
  `{% extends %}` to the new runner base and restructure the three templates' bodies. This avoids
  churn in `views.py`'s `render(...)` calls and URL names.
- **Selected-state styling uses native inputs + Tailwind `has-[:checked]:` variant utilities** (which
  resolve through `@theme` tokens), not a hand-written CSS rule and not Alpine selection state. This
  keeps everything token-driven and template-only.
- **One-off exam UI blocks are partials, not cotton components.** Per the `fls:template` skill, blocks
  used only by the exam screens (meta grid, runner bar, page dots, score ring, results stats, previous
  attempts) are `{% partialdef %}` / `partials/<name>.html`. Cotton components are reserved for
  genuinely reusable/parameterised pieces; we reuse existing ones (`c-button`, `c-icon`, `c-page`,
  `c-markdown-container`, `c-modal`) but add no new cotton components.
- **No new icons.** All semantic icon names the design needs already exist in
  `freedom_ls/icons/semantic_names.py` (`next`, `previous`, `success`, `error`, `warning`, `close`,
  `retry`, `achievement`, `course`, `info`, `home`, `check`). The `flag` icon mentioned in research is
  **not** added because flagging is out of scope (spec ┬¦7).

## 2. Pydantic schema: allow `submit_on_exit` in frontmatter

**File:** `freedom_ls/content_engine/schema.py` (the `Form` model at line 163).

- Add a field, **not** gated on strategy and **not** referenced in `validate_quiz_fields`:

  ```python
  submit_on_exit: bool = Field(
      False,
      description="If True, leaving the test mid-attempt finalises and scores it. Default False.",
  )
  ```

- Rationale: the content schema uses `model_config = ConfigDict(extra="forbid")`, so the key must be
  declared for `submit_on_exit:` frontmatter to load at all. `content_save.py` does
  `item.model_dump(exclude_none=True)` then validates each key against real model field names
  (`content_save.py:253ŌĆō280`) and `update_or_create(defaults=fields)`. A literal `False` is not
  `None`, so it is passed through and matches the model field ŌĆö no special handling needed. When the
  key is absent the schema default `False` flows through (also fine; equals the model default).

**Test (TDD):** schema/loader test ŌĆö a `form.md` with `submit_on_exit: true` validates and produces a
`Form` with `submit_on_exit=True`; a form without the key produces `False`. (Mirror the existing
content-schema/loader tests in `content_engine/tests/`.)

## 3. Model: idempotent `complete()` + helper for stale-attempt safety net

**File:** `freedom_ls/student_progress/models.py` (`FormProgress`).

### 3a. Idempotent `complete()`

`complete()` (line 263) currently sets `completed_time` unconditionally. Add an early return so a
double submit / double exit cannot re-score:

```python
def complete(self):
    """Mark the form as completed and calculate the final score (idempotent)."""
    if self.completed_time:
        return
    self.completed_time = timezone.now()
    self.score()
    self.save()
```

**Test (TDD):** calling `complete()` twice does not change `completed_time` and does not re-run
scoring (e.g. assert `completed_time` unchanged after a second call).

### 3b. Stale-attempt completion for submit-on-exit forms

Add a small classmethod/helper used when a learner **returns** to a submit-on-exit form that still has
an incomplete attempt ŌĆö it finalises that stale attempt before a fresh one is created:

```python
@classmethod
def finalise_stale_incomplete(cls, user, form):
    """
    For submit-on-exit forms: if the user has an incomplete attempt, complete it.
    Safe for save-on-exit forms (no-op) and idempotent via complete().
    Returns the finalised FormProgress or None.
    """
    if not form.submit_on_exit:
        return None
    incomplete = cls.get_latest_incomplete(user, form)
    if incomplete is None:
        return None
    incomplete.complete()
    return incomplete
```

**Tests (TDD):**
- submit-on-exit form with an incomplete attempt ŌåÆ `finalise_stale_incomplete` completes it (scores
  set, `completed_time` set).
- save-on-exit (default) form ŌåÆ returns `None`, incomplete attempt untouched.
- no incomplete attempt ŌåÆ returns `None`.

## 4. Views & URLs (`student_interface`)

### 4a. `form_start` & `view_form` ŌĆö apply the stale-attempt safety net

**File:** `freedom_ls/student_interface/views.py`.

The reliable safety net (spec ┬¦4) runs when the learner next **returns** to the form. Both `view_form`
(the start screen) and `form_start` (the resume/begin redirect) are entry points, so call the helper at
the top of each, before reading/creating progress:

- In `view_form` (line 363) ŌĆö before `FormProgress.get_latest_incomplete(...)`:
  `FormProgress.finalise_stale_incomplete(request.user, form)`. After this, for a submit-on-exit form
  there is no longer an incomplete attempt, so the start screen will correctly show **Try again /
  results** rather than **Continue**.
- In `form_start` (line 403) ŌĆö before `get_or_create_incomplete(...)`, call
  `FormProgress.finalise_stale_incomplete(request.user, form)` so a submit-on-exit form starts a fresh
  attempt instead of resuming the (now finalised) one.

For **save-on-exit** (default) forms the helper is a no-op and the existing resume behaviour is
unchanged.

### 4b. `view_form` ŌĆö context for the restyled start screen

Add to the context (spec ┬¦1 ŌĆö **only truthful, derivable** facts):

- `question_count` ŌĆö total `FORM_QUESTION` children across all pages.
- `page_count` ŌĆö `form.pages.count()`.
- Previous-attempts summary: reuse the existing `completed_form_progress` queryset; in the template
  show best / most-recent score where applicable (QUIZ). **No "View all" link, no stub.**

Compute `question_count` with a small helper to avoid repetition (it is also needed by the runner and
the submit dialog ŌĆö see 4c). Suggested helper in `student_interface/utils.py`:

```python
def count_form_questions(form: Form) -> int:
    return FormQuestion.objects.filter(form_page__form=form).count()
```

This is a single `COUNT` query that traverses the FK chain `FormQuestion.form_page` ŌåÆ `FormPage.form`
(the FK field is `form_page`, not `page`). It avoids the N+1 of iterating `form.pages` and calling
`page.children()` (which itself queries `text_items` + `questions` per page) and avoids loading every
child into Python just to filter on the `content_type` property.

Only prefetch `pages__questions` / `pages__text_items` where the **template** actually iterates
pages/children to render them ŌĆö it is no longer needed for the count.

### 4c. `form_fill_page` ŌĆö runner context + no-store + honest answered count

**File:** `views.py` (line 427).

### 4d. New view: `form_submit_and_exit` (POST-only) + URL

For the exit dialog's "Leave and submit" action on **submit-on-exit** forms (research ┬¦2 Path A) ŌĆö a
dedicated POST endpoint that finalises and redirects to results. Pseudocode:

```python
@login_required
@require_POST
def form_submit_and_exit(request, course_slug, index):
    course = get_object_or_404(Course, slug=course_slug)
    viewable_items = course.viewable_items()
    # validate index (same guard as the other form views)
    form = viewable_items[index - 1]
    form_progress = FormProgress.get_latest_incomplete(user=request.user, form=form)
    if form_progress:
        form_progress.complete()  # idempotent (3a)
    return redirect(
        "student_interface:course_form_complete", course_slug=course_slug, index=index
    )
```

- `@require_POST` comes from `django.views.decorators.http` ŌĆö add the import.
- The index-validation guard (`if index < 1 or index > len(viewable_items): raise Http404`) is
  repeated across the form views; extract it into a small helper
  (`get_form_for_index(course, index) -> Form`) in `views.py`/`utils.py` and reuse it in `view_form`,
  `form_start`, `form_fill_page`, `course_form_complete`, and `form_submit_and_exit` to keep DRY.
- **URL:** add to `student_interface/urls.py` (kebab path, snake_case name):

  ```python
  path(
      "courses/<slug:course_slug>/<int:index>/submit-and-exit",
      views.form_submit_and_exit,
      name="form_submit_and_exit",
  ),
  ```

**Tests (TDD):**
- POST to `form_submit_and_exit` with an incomplete attempt ŌåÆ attempt completed, redirect to results.
- A GET is rejected (405) via `@require_POST`.
- `answered_count` reflects only persisted answers (answer some questions on page 1, advance, assert
  count; type into page 2 without submitting, assert count unchanged).
- Returning to a submit-on-exit form mid-attempt finalises the stale attempt (integration through
  `form_start` / `view_form`).
