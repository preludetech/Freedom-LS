# Spec: Student interface — course test/exam UI

## Summary

Re-skin and restructure the three student-facing screens of the existing Form (test/exam)
flow — **start screen**, **questions runner**, and **results page** — to match the designs at
`/home/sheena/workspace/lms/design/computed/exam` (`desktop/`, `tablet/`, `mobile/`). The designs
target the **first-class** theme; everything must work in the **default theme first**, with
first-class styling layered purely through `@theme` tokens.

This is a **presentation re-skin of an existing flow**. It does **not** add new scoring strategies,
question types, or model fields **except** one small, optional `Form` field for exit behaviour
(see §5). The goal is to **copy the design, not implement every design element** — several mockup
features have no model backing and are explicitly omitted (§7).

The current flow stays intact: `view_form` (info) → `form_start` (creates/gets `FormProgress`) →
`form_fill_page` (GET render / POST save→redirect, the PRG pattern) → `course_form_complete`
(results). We restyle these and add a dedicated sidebar-less base layout for the runner.

## Why this matters

- The exam experience is a high-stakes moment for learners; the current screens reuse the generic
  course-player chrome and don't present tests as a focused, trustworthy activity.
- The new runner gives the test its own full-viewport interface (own bar, progress strip, footer
  nav), reducing distraction and accidental navigation away mid-test.
- Doing it token-first means the same templates serve every theme with zero template edits — a core
  FLS principle ("easy to theme, style and change").

## Background — what already exists (reuse it)

Confirmed by codebase research (`.sdd-work/spec_research_codebase.md`):

- **Models** (`content_engine/models.py`): `Form` (`strategy` ∈ {`QUIZ`, `CATEGORY_VALUE_SUM`};
  `quiz_show_incorrect`, `quiz_pass_percentage`; inherits `TitledContent` → `title`, `subtitle`,
  `slug`; `MarkdownContent` → intro body), `FormPage` (ordered), `FormQuestion` (4 types:
  `multiple_choice`, `checkboxes`, `short_text`, `long_text`), `QuestionOption` (`text`, `value`,
  `correct`).
- **Progress** (`student_progress/models.py`): `FormProgress` (one per attempt, full history per
  user+form; `start_time`, `completed_time`, `scores`) and `QuestionAnswer` (one per question per
  attempt). Methods: `save_answers`, `complete`, `score`, `quiz_percentage`, `passed`,
  `get_incorrect_quiz_answers`, `get_current_page_number`, `get_or_create_incomplete`.
- **Views / flow** (`student_interface`): `view_form` → `form_start` → `form_fill_page` →
  `course_form_complete`. URL names `form_start`, `form_fill_page`, `course_form_complete`. Answers
  persist on page advance via `save_answers`; pages advance with POST→save→redirect.
- **Templates**: `course_form*.html` extend `student_interface/_course_base.html` →
  `_base_interface.html` (the full sidebar + course-TOC chrome). Question types render as inline
  `{% partialdef %}` blocks in `course_form_page.html`; field name convention `question_{id}`.
- **Theming**: Tailwind v4 `@theme` tokens with `default` and `first_class` themes; icons via
  `c-icon` (inline SVG, Heroicons default + Phosphor mapping). Cotton components in use:
  `c-page`, `c-button`, `c-button-group`, `c-icon`, `c-markdown-container`, `c-form-page-link`.
- **Content authoring**: Forms are defined in content files (`form.md` frontmatter, `content_type:
  FORM`) and loaded via the `content_save` management command, validated by a Pydantic schema
  (`content_engine/schema.py:163`). Frontmatter keys must map to real Django model fields.

Full design→token mapping and component reuse map live in the durable research files
`research_theming_and_components.md` and `research_runner_interactions.md`; design element catalogue
in `.sdd-work/spec_research_design.md`.

## Confirmed decisions (from the idea + clarifications)

1. **Exit-behaviour field is content-authored.** The optional `Form.exit_behaviour` field is set in
   the form content file's frontmatter and loaded via `content_save` — **not** configured through
   Django admin or the educator interface. This means: add the model field + migration, add the
   field to the Pydantic `Form` schema so frontmatter validates, and (lightly) demo it. No new
   educator UI.
2. **Demo content: reuse / lightly extend.** Use existing demo Forms; add only what's missing so
   all four question types, both scoring strategies, and both exit modes can be demoed and QA'd. No
   comprehensive new demo set.
3. **Responsive single templates.** One set of templates using Tailwind responsive utilities adapts
   across desktop / tablet / mobile. The distinct mobile mockups (`ckm-*` namespace) are
   **approximated** with responsive utilities, not reproduced pixel-for-pixel as separate templates.

## Requirements

### 1. Test start screen

Re-skin the existing form overview (keeps **normal app chrome** — global header, no runner layout):

- Title, eyebrow/subtitle (`Form.subtitle`), and intro/markdown content (existing `MarkdownContent`
  body via `c-markdown-container`).
- A **meta grid** showing only **truthful, derivable** facts: number of questions and number of
  pages (both derived server-side). **Drop** the design's estimated-time and "unlimited tries" cells
  — no model backing.
- A compact **previous attempts** summary from `FormProgress` history (best / most-recent scores
  where applicable). **Omit the "View all" link entirely** — the history page is a future spec; show
  no link, stub, or disabled placeholder.
- A clear **Start / Continue / Try again** CTA, reusing existing button + progress logic.
  "Continue" appears when an incomplete `FormProgress` exists (save-on-exit forms).

### 2. Test questions page (the "runner")

- Displays **one `FormPage`'s questions at a time**.
- **Dedicated full-screen, sidebar-less base layout** — does **not** render the course-player
  sidebar / TOC or standard course chrome. The runner owns the viewport: own top bar, progress
  strip, scrollable question body, footer nav. Add a new base template that skips the
  `_base_interface.html` sidebar grid (start and results screens keep normal chrome).
  See `research_theming_and_components.md` §4.
- **Progress bar / strip** at the top: page X of Y, derived server-side.
- Render all four existing question types as **real, accessible form controls**: native `<input>`
  wrapped in styled `<label>` tiles (not click-handler divs), grouped in `<fieldset>`/`<legend>`.
  Use the `:has(input:checked)` CSS pseudo-class for selected-state styling (no Alpine selection
  state). `short_text` → `<input type="text">`, `long_text` → `<textarea>`.
  See `research_runner_interactions.md` §5.
- **Navigation (keep the existing PRG pattern):**
  - **Next** saves the current page (POST → `save_answers`) then redirects to the next page.
  - **Previous** is a plain **GET** link — does *not* save in-progress edits — and re-populates from
    saved answers.
  - **Page dots** reflect accessible/visited pages; locked/inaccessible pages render as non-link
    `<span>`, accessible ones as `<a>`.
  - Set `Cache-Control: no-store` on runner responses.
- An **honest answered count** ("5 of 8 answered") reflecting **persisted** answers only — computed
  server-side from `QuestionAnswer` records. **No live per-keystroke autosave** and no misleading
  "saved" claim while a page is mid-edit.
- **Final page → review-or-submit dialog** (Alpine, accessible): on the final page, Next opens a
  dialog:
  - Title ~ "Ready to submit?"; explains submitting scores the attempt and answers can't be changed.
  - Shows only truthful counts (answered / unanswered — **no** "flagged").
  - **Go back and review** (dismiss, stay) vs **Submit** (POST → `complete()` → results). Submit
    button disables on click to prevent double-submit.

### 3. Results page

Redirect here after submit. Keeps **normal app chrome**.

- If the test has **automated marking** (`QUIZ` strategy): show score, pass/fail state, and — where
  `quiz_show_incorrect` is set — the "worth another look" review of incorrect answers via
  `get_incorrect_quiz_answers`. Match the design's banner / SVG score-ring / stats treatment, but
  **only with data we actually have**.
- Otherwise (no automated score, e.g. `CATEGORY_VALUE_SUM`): clearly state **marking is in
  progress**. (Preserve the existing `course_form_complete.html` TODO for CATEGORY_VALUE_SUM results
  display — do not delete it.)
- Allow the learner to **navigate back to the course player** (and retry where appropriate).
- **Drop** the per-topic breakdown ("How you did by topic") and per-question "Here's the idea"
  explanations — out of scope.

### 4. Navigation & exit behaviour

A learner can launch a form, move back/forth, fill answers, and retry — reusing the existing flow.

Mid-test exit is **configurable via the optional `Form.exit_behaviour` field** (§5), with two modes:

- **Save on exit (default)** — leaving keeps an **incomplete** `FormProgress`; the learner resumes
  from where they stopped on next visit (existing behaviour via `get_or_create_incomplete` /
  `get_current_page_number`, surfaced as the start screen's "Continue" CTA). The exit confirmation
  dialog warns that progress is saved and they can resume.
- **Submit on exit** — leaving the runner mid-test **finalises and scores** the attempt. The exit
  confirmation dialog warns that leaving will submit their answers.

In both modes:

- Intercept the in-runner exit control (the "X") with an **Alpine confirmation dialog** → on
  confirm, take the mode-appropriate action (submit → `complete()` → results, or just leave and
  return to the course player).
- Use **`beforeunload`** as a best-effort courtesy warning for raw browser navigation, accepting its
  limits (generic text only, unreliable on mobile, doesn't fire for in-app swaps). Never call Django
  endpoints from `beforeunload`. See `research_runner_interactions.md` §2.
- For **submit-on-exit**, the **reliable safety net is server-side**: when a learner with an
  incomplete `FormProgress` for a submit-on-exit form next returns, **complete the stale attempt
  before starting a new one**. Add an **idempotency guard** so `complete()` is not applied twice
  (e.g. early-return when `completed_time` is already set). In **save-on-exit** mode the incomplete
  attempt is simply reopened.

### 5. The `Form.exit_behaviour` field

- Add `exit_behaviour` to the `Form` model (`content_engine/models.py`) as a `TextChoices` field
  with values `SAVE_ON_EXIT` (default) and `SUBMIT_ON_EXIT`. Create a **new migration** (do not edit
  existing ones); the migration backfills existing rows to the default.
- Add the corresponding field to the Pydantic `Form` schema (`content_engine/schema.py`) so the
  frontmatter key `exit_behaviour:` validates and loads via `content_save`. Default applies when the
  key is absent. (Mirror the existing optional-field pattern used for the quiz fields.)
- Do **not** add educator-interface UI for this field. Default Django admin display is acceptable
  but not required.

### 6. Theming & accessibility (cross-cutting)

- **No hardcoded hex / fonts / radii in templates.** Every colour, font, and radius flows through
  `@theme` tokens / Tailwind utilities (`bg-primary`, `bg-secondary`, `bg-success-light`,
  `rounded-md/lg/pill`, `font-display/sans/mono`, etc.) so a theme override needs zero template
  changes. Map the design's first-class palette to FLS tokens per
  `research_theming_and_components.md` §1 & §5. Do **not** add Phosphor CSS/CDN font links — icons go
  through `c-icon`.
- The SVG **score ring** strokes via `stroke="currentColor"` driven by a token-backed text colour
  wrapper (e.g. `text-primary` / `text-success`).
- **Reuse existing cotton components** where they fit (`c-button`, `c-button-group`, `c-icon`,
  `c-page`, `c-markdown-container`, `.surface`, and `c-chip` / `c-modal` if present). Add small new
  partials for: the meta grid, score ring, runner top bar, progress strip, page dots, and results
  stats. See `research_theming_and_components.md` §2.
- **Accessibility baseline:** real inputs + labels + `fieldset`/`legend`; focus management
  (`tabindex="-1"` + focus the page heading on page change) and an `sr-only`, `aria-live="polite"`
  progress announcement on page change; dialogs with `role="dialog"`, `aria-modal="true"`,
  `aria-labelledby`, focus trap, Escape to close, and focus return to the trigger — or a native
  `<dialog>` with `.showModal()`.

### 7. Out of scope — omit entirely (leave related TODO/@claude comments intact)

Per the idea and design catalogue (`.sdd-work/spec_research_design.md`):

- **History page + trend chart** and the **"View all" link** (deferred to a future spec alongside
  retry limits). The lightweight previous-attempts summary on the start screen stays, but with no
  link/placeholder.
- **Per-question "Flag for review"** (button + flagged count in the submit dialog).
- **Per-question "Here's the idea" explanations.**
- **Any timer / countdown / "relaxed pace" pill.**
- **Estimated completion time** (e.g. "~10 min" meta cell, "~8 min" stats cell).
- **"Unlimited tries" meta cell.**
- **Per-topic results breakdown** ("How you did by topic").
- **No new question types.** Demonstrate only the four existing field types with standard widgets.
  Do **not** special-case True/False — a two-option `multiple_choice` renders like any other
  multiple-choice question (no bespoke True/False styling, no hint subtext).
- Mobile-only mockup extras with no backing (e.g. a results "share" button) are omitted.

If a feature isn't implemented, the learner must not see a control for it (no stubs, no disabled
placeholders).

### 8. Preserve existing TODO / @claude comments

Do **not** delete any TODO or `@claude` comment. Specifically noted by research:
`student_progress/models.py` (~lines 32, 112–113), `content_engine/.../utils.py` (~line 126), and

## Success criteria

1. **Start, runner, and results** screens visually match the first-class designs (desktop / tablet /
   mobile, responsive) while rendering correctly in the **default theme** with no template changes.
2. The runner uses a **dedicated sidebar-less base layout**; start and results keep normal app
   chrome.
3. All **four question types** render as native, accessible controls (inputs + labels +
   fieldset/legend), with keyboard and screen-reader support (focus management + sr-only progress
   announcement).
4. **Navigation** preserves PRG: Next saves then advances; Previous is a GET that re-populates from
   saved answers; page dots reflect accessible pages; answered count reflects only persisted
   answers.
5. The **final-page submit dialog** and **exit-confirmation dialog** behave per spec, are
   accessible, and prevent double-submit.
6. **`Form.exit_behaviour`** exists as an optional, content-authored field (model + migration +
   Pydantic schema), defaulting to save-on-exit; submit-on-exit finalises on exit with a
   **server-side stale-attempt safety net** and an **idempotent `complete()`**.
7. **No hardcoded colours / fonts / radii** in the new/changed templates; a theme override needs
   zero template edits.
8. **No out-of-scope element** (§7) appears in the UI; all existing TODO/@claude comments remain.
9. Demo content lets QA exercise: all four question types, both scoring strategies (QUIZ score/pass
   + CATEGORY_VALUE_SUM "marking in progress"), and both exit modes.
10. `uv run pytest` passes; new behaviour (exit modes, stale-attempt completion, answered-count,
    schema loading of `exit_behaviour`) is covered by tests. Tests assert **functionality, not CSS
    classes**.
