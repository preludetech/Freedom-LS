# Frontend QA: Student interface — course test/exam UI

Manual / Playwright-MCP QA plan for the re-skinned test/exam flow (start → runner → results), the new
`submit_on_exit` behaviour, and accessibility. Verify in the **default theme** (must work) and, if a
first-class build is available, spot-check first-class styling.

---

## 0. Setup

### Dev server (random port — never 8000)

```bash
PORT=$(.claude/fls/scripts/find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL: `http://127.0.0.1:$PORT`

The dev settings force the **DemoDev** site (`FORCE_SITE_NAME = "DemoDev"`), so any port serves DemoDev.

### Demo data

Ensure demo data + content are loaded (re-run after any demo-content edits):

```bash
uv run python manage.py create_demo_data
uv run python manage.py content_save ./demo_content DemoDev
```

If the new question-type / `submit_on_exit` demo content from the plan (§7) is missing, the data is
incomplete — load it before running the question-type and exit-mode tests below. (The `qa-data-helper`
agent can create any missing learner/registration/progress data on the DemoDev site.)

## 1. Start screen (normal chrome)

URL: course item index for a form, e.g. `/courses/<course-slug>/<index>` (the form overview).

Check:
- [ ] Global header + course chrome present (this screen keeps normal app chrome — **not** the
  sidebar-less runner).
- [ ] Title, eyebrow/subtitle, and intro markdown render.
- [ ] **Meta grid** shows exactly two truthful cells: **number of questions** and **number of pages**.
  No "estimated time", no "unlimited tries" cell.
- [ ] If the learner has previous completed attempts: a **previous-attempts summary** shows
  (best / most-recent score for a quiz). **No "View all" link** and no disabled placeholder anywhere.
- [ ] CTA button is correct for state: **Start** (no progress), **Continue** (incomplete save-on-exit
  attempt exists), or **Try again** (failed quiz). Clicking it enters the runner.

---

## 2. Runner — layout & accessibility

Click Start/Continue to enter the runner (`form_fill_page`).

Check:
- [ ] **No course sidebar / TOC and no global site header** — the runner owns the full viewport with
  its own top bar, progress strip, scrollable body, and footer nav.
- [ ] **Top bar:** exit "X" control (left), form title (centre), honest answered count
  "**N of M answered**" (right). The count must **not** say "saved".
- [ ] **Progress strip:** "Page X of Y" and a fill bar whose width matches the page fraction.
- [ ] **Page dots:** visited/accessible pages are links; locked (not-yet-reached) pages are
  non-clickable.
- [ ] **Question controls are native and accessible:** each question is a `<fieldset>` with a
  `<legend>`. Multiple-choice = radios, checkboxes = checkboxes (each a label tile). Selecting a tile
  visibly highlights it (driven by the checked input, not script). Keyboard: Tab into a radio group,
  arrow keys move between options, Space toggles checkboxes. (Use `browser_snapshot` to confirm roles.)
- [ ] A two-option multiple-choice renders like any other multiple-choice (no bespoke True/False UI).
- [ ] No console errors (check `browser_console_messages`) — in particular no Alpine CSP
  "blocked inline expression" errors.

### Answered count honesty

- [ ] On page 1, answer some questions and click **Next**. On page 2 the top-bar count reflects the
  **persisted** page-1 answers.
- [ ] Type into page-2 fields **without** advancing — the count does **not** increase (only persisted
  answers count).

---

## 3. Navigation (PRG)

- [ ] **Next** saves the current page then advances (answers persist — go Back and confirm they're
  pre-filled).
- [ ] **Previous** is a plain link (GET) and does **not** save in-progress edits: change an answer,
  click Previous, return via Next — the previously **saved** answer is shown (the unsaved edit is
  discarded). This is expected.
- [ ] Page dots navigate to accessible pages; locked pages can't be clicked to skip ahead.
- [ ] Browser **Back** shows previously saved answers, not a stale cached page (responses send
  `Cache-Control: no-store` — verify via `browser_network_requests` on a runner GET).

---

## 4. Final-page submit dialog

On the last page, click **Next**.

- [ ] A dialog "**Ready to submit?**" opens (it does **not** submit immediately).
- [ ] Body explains the attempt will be scored and answers can't change.
- [ ] Shows only **answered / unanswered** counts — **no "flagged"** count.
- [ ] **Go back and review** dismisses the dialog and keeps the learner on the page.
- [ ] **Submit** finalises and lands on the results page.
- [ ] **Double-submit guard:** the Submit button disables on first click (can't fire twice).
- [ ] Dialog accessibility: `role="dialog"`, focus moves into the dialog, **Escape** closes it, focus
  returns to the Next button, Tab is trapped inside while open.

---

## 5. Exit behaviour (the X control)

### 5a. Save-on-exit form (`submit_on_exit = False`, default)

- [ ] Click the top-bar **X**: an exit dialog warns that **progress is saved and can be resumed**.
- [ ] Confirm exit → returns to the course player / form start; the attempt stays **incomplete**.
- [ ] Re-enter the form: the start screen offers **Continue** and the runner resumes at the right page.

### 5b. Submit-on-exit form (`submit_on_exit = True`)

(Use the demo form that has `submit_on_exit: true` from plan §7.)

- [ ] Click the **X**: the exit dialog warns that **leaving will submit and score** the attempt.
- [ ] Confirm → the attempt is finalised and the **results** page shows.
- [ ] **Server-side safety net:** start a fresh attempt on this form, answer one page, then navigate
  away via the browser (close-tab/back) **without** using the X. Return to the form's start screen —
  the stale attempt is **completed** (no lingering "Continue"; previous-attempts summary shows the
  finalised attempt). Starting again creates a new attempt.

### 5c. beforeunload courtesy warning

- [ ] While in the runner, attempt a raw browser navigation (reload / address bar). A generic browser
  "leave site?" prompt appears (text is browser-controlled). Confirm it does **not** call any Django
  endpoint (check `browser_network_requests`).

## 6. Results screen (normal chrome)

### 6a. QUIZ result

- [ ] Normal chrome (header) present — not the runner layout.
- [ ] Pass/fail banner, **score ring** (SVG) rendering the percentage, and a stats treatment, using
  only real data (score, percentage, pass threshold).
- [ ] Score ring colour comes from a token (`text-primary`/`text-success`) — visible in default theme.
- [ ] If `quiz_show_incorrect` is set, the incorrect-answers review lists the learner's wrong answers
  and the correct options.
- [ ] **No** per-topic breakdown and **no** per-question "Here's the idea" explanations.
- [ ] Navigation back to the course player (and retry where appropriate) works.

### 6b. CATEGORY_VALUE_SUM result

- [ ] Completing the survey shows a clear **"marking is in progress"** state (no fabricated score).
- [ ] Existing category-display/TODO block is preserved (not removed).

## 8. Question-type coverage

Using the demo form(s) extended per plan §7, confirm all four types render and submit correctly:
- [ ] `multiple_choice` (single select)
- [ ] `checkboxes` (multi-select)
- [ ] `short_text` (`<input type="text">`)
- [ ] `long_text` (`<textarea>`)

All four must be keyboard-operable, correctly labelled, and persist their answers through Next/Back and
into scoring/results.
