We need to track an ordered log of form completions when a form is a part of a course. This will be for quizzes and exams and such like. but really any form completion can be logged.

The log must be a separate immutable data structure. We can't delete it. Even if the form itself that was created, that was filled out, gets deleted, the log data must not get deleted.

The log entry should include at least the following.
- exact questions (text, options, etc)
- exact answer given
- time of answer (page submission time)
- student information: full name, email, etc
- course / form / page context (titles + identifiers)
- site context (which tenant the attempt belongs to)
- the score that was awarded and the scoring strategy used
- a content hash of the captured question + options, for tamper detection

## High-level approach

Today (see `freedom_ls/content_engine/models.py` `FormQuestion` / `QuestionOption` and `freedom_ls/student_progress/models.py` `FormProgress` / `QuestionAnswer`):

- Questions live in markdown (git) and are synced into the DB at content-save time. They mutate in place — there is no version field and no historical record of what a question used to say.
- `QuestionAnswer` references `FormQuestion` by FK. If the markdown changes (or the form/question is deleted), the historical attempt loses the ability to reconstruct what the student actually saw.
- `FormProgress` and `QuestionAnswer` are deletable via CASCADE.

We need to add an **append-only attempt-log layer** that sits alongside the existing progress models, not on top of them.

### What to build

1. **`ExamAttempt`** — one row per form submission. Created at the moment the student submits a page/quiz. Holds:
   - Denormalized student identity (user id + full name + email captured at submit time, not via FK lookup at read time).
   - Denormalized form/course/page context (titles + uuids + site).
   - Submission timestamp, start timestamp, scoring strategy, awarded score, pass/fail.
   - Optional FK to the live `FormProgress` for convenience (nullable, `SET_NULL` on delete) — the audit row must remain valid without it.

2. **`ExamAttemptQuestion`** — one row per question in that attempt. Snapshots the exact question content at submission time:
   - Question text (rendered or raw markdown — likely both, so the audit reader doesn't need to re-render against a future template).
   - Question type, category, ordering on the page.
   - Full list of options as a JSON blob: text, value, `correct` flag, ordering.
   - The student's answer: selected option values + free-text answer, captured by value not by FK.
   - A SHA-256 hash of the canonicalised question+options blob, so tampering is detectable.

   No FK to `FormQuestion` — the snapshot is self-contained. (A nullable reference is fine for forensics, but the audit must read correctly with it null.)

3. **Immutability enforcement**
   - Override `save()` to forbid updates after creation, and `delete()` to raise.
   - Do not register the models in admin with delete permission.
   - Add a DB-level safeguard: a Postgres trigger (or restricted role) that blocks `UPDATE` and `DELETE` on these two tables. Code-level guards alone are not enough for a SACAA audit posture.
   - No CASCADE inbound: every FK pointing into these tables must be `SET_NULL` or `PROTECT`, never `CASCADE`.

4. **Write path**
   - Hook into the existing form-submission flow (where `FormProgress.completed_time` and quiz scoring are set, around `student_progress/models.py:400`).
   - The write must be transactional with the scoring write — either both succeed or neither does — so the audit log can never be missing an attempt that the student saw a score for.

5. **Read path**
   - An educator/moderator-only view that lists attempts for a student or a form, and shows the snapshot exactly as the student saw it.
   - A `get_question_snapshot()` helper returning the JSON blob, used both by the moderator UI and by future moderation/remediation specs.

### What this gives us

- Clause 1(c) (tamper-proof exam records) is satisfied: even deleting the underlying `Form`, `FormQuestion`, or `User` leaves the audit row intact and readable.
- Clause 1(e) (old question version retained) is satisfied as a side-effect: every attempt is a frozen snapshot, so historical question text is always recoverable per attempt without needing a separate question-versioning table. A future question-versioning spec can build on this rather than duplicating it.
- The audit reader does not depend on git, on deployment archaeology, or on the markdown still existing. Git remains the source of truth for editing; the audit log is the source of truth for what happened.

### Out of scope for this spec

- Active remediation loop (clause 1(v)) — separate spec, but will read from `ExamAttemptQuestion`.
- Internal moderation / mark override workflow (clause 1(d)) — separate spec; will append moderation rows that reference (but cannot mutate) the original attempt row.
- Question versioning UI — not needed; the snapshot model removes the need for a separate version table on `FormQuestion`.
