# Spec: Make the course details page reachable for registered learners

## 1. Overview

Once a learner registers for a course, the app drops them into the course
player and gives them no obvious way back to the **course details page** — the
overview surface carrying the description, "what you'll learn", and the content
table-of-contents. That page is valuable for two jobs: revisiting the overview,
and sharing the course (its stable, public URL is the shareable link).

The details page (`student_interface:course_detail`) **already exists and is
already reachable** by registered learners. This feature adds nothing to the
page itself and adds no access control, no new URL, no new model, and no bespoke
"share" feature. It only adds **consistent, secondary "Details" entry points**
on the three surfaces where a registered learner naturally is:

1. Dashboard course cards
2. Course index / list rows (the all-courses page)
3. Course player breadcrumbs

The details page *is* the shareable URL — landing on it and copying the address
bar is enough, matching how Udemy, Coursera, and edX handle sharing. No "share"
mechanism is built here.

## 2. Why this matters

- **Findability.** A registered learner currently has no in-app route back to
  the overview once they are inside the player. The card/row is a stretched link
  straight into the player; the player's first breadcrumb sends them to lesson
  one ("start over"), never to the overview.
- **Sharing without new machinery.** Every major platform surveyed treats the
  public overview page as the shareable artifact and lets learners copy its URL;
  none of the mainstream flows (Udemy, Coursera, edX, Moodle, Canvas) ship a
  bespoke in-player share button. Making the existing page reachable satisfies
  "share this with a friend" for free.
- **Consistency and least surprise.** Linking a course title (breadcrumb) to the
  course overview is the standard Moodle/Canvas behaviour. Keeping one primary
  action ("Continue / Start") per card with a low-emphasis secondary "Details"
  link matches mainstream button-hierarchy guidance.

Full supporting research: `research_course_details_entry_points.md` (reference
implementations + best practices) and `research_kebab_overflow_menu_ux.md`
(affordance/overflow-menu UX). Codebase grounding: `.sdd-work/` was used during
authoring and is deleted on completion; exact file references are inlined below.

## 3. Decisions made (and why)

These resolve the idea's open questions and correct two assumptions the codebase
research disproved.

### 3.1 A visible "Details" text link, not a kebab/overflow menu

The originally-floated single-item kebab menu is rejected. Every design authority
surveyed (NN/g, Material, Carbon, Apple HIG) reserves overflow menus for hiding
**two or more** secondary actions; wrapping one link in a menu only adds a click
and roughly halves discoverability (~27% vs ~48% for visible controls). A visible
affordance is simpler and more discoverable. *If per-surface actions grow to 2+
later, revisit a menu then.*

**Chosen presentation (all surfaces): a plain, low-emphasis "Details" text
link.** It reuses the codebase's existing "Browse all courses" affordance pattern
(`course_list.html:92-98`): `text-sm font-medium no-underline
hover:text-primary-hover`, with a trailing `next` chevron icon wrapped in an
`aria-hidden="true"` span. Icon-only was rejected (lowest discoverability, and no
dedicated "details" icon exists — only `info`/`more_options`/`next`).

**Label wording is fixed as "Details" across all three surfaces.** Consistency
here means identical *meaning, wording, and destination* — not pixel-identical
presentation. Presentation may differ per surface (see §5).

### 3.2 The "reserved seam" assumption is false — correct handling required

The idea assumed the card/row shells "already reserve a z-index seam" for a
control above the stretched link. The codebase does **not**:

- The stretched click target is a `::before` pseudo-element on the **title
  anchor** (`before:absolute before:inset-0` in `card_title_link.html` and
  inlined in the not-registered/coming-soon leaves) — **not** a full absolutely
  positioned sibling `<a>` overlaying the whole card.
- There is **no `z-index` rule wired up anywhere** in the card shell. The row
  shell only carries a *comment* conceding "z-index … (none currently required)"
  (`course-row-shell.html:30-32`). Nothing is actually carved out.

Because the overlay is a `::before` on the title anchor (not a full-card sibling
overlay), a **new sibling `<a>` placed after the title anchor, with
`position: relative` and a `z-index` above the pseudo-element**, is a small,
isolated, correct addition. The architecture permits it; the seam is not
pre-built. The implementation must therefore *establish* the stacking (add
`relative z-10` to the new link), not assume it exists.

### 3.3 All learner-facing states get the affordance

Every course card/row **state** gets the "Details" link:

| State | Card leaf / Row leaf | Treatment |
|---|---|---|
| Registered / in-progress | `course_card_registered.html` / `course_row_registered.html` | Add "Details" link |
| Complete (history) | `course_card_complete.html` / `course_row_complete.html` | Add "Details" link |
| Not registered | `course_card_not_registered.html` / `course_row_not_registered.html` | Add "Details" link (in addition to the existing whole-card stretched link, which already points at the details page) |
| Coming soon | `course_card_coming_soon.html` / `course_row_coming_soon.html` | Add "Details" link |

Rationale: a signed-in registered learner can see all four states on one
dashboard at once (their in-progress/completed courses plus not-registered and
coming-soon courses in Recommended/Available). A uniform affordance is the least
surprising outcome. On the **not-registered** card the explicit link is
technically redundant with the whole-card stretched link (both go to the details
page) but the redundancy is harmless and preserves grid-wide consistency; both
must resolve to the identical URL. The details page is reachable for coming-soon
courses too (only `HIDDEN`-visibility courses 404 for the unregistered, per
`course_access/visibility.py:20-33`).

### 3.4 Card placement: content footer, bottom-right, own line

The hero band fills the entire card top (no free top-right corner); the card
bottom edge is occupied by the progress bar / completed badge. The "Details" link
therefore sits **inside the text-content block below the hero, right-aligned, on
its own line, clear of the progress bar / badge** — not overlaid on the hero and
not colliding with existing chrome. It is a sibling of the title anchor raised
above the `::before` overlay (§3.2).

## 4. The destination (no work required, stated for completeness)

- URL: `student_interface:course_detail`, `courses/<slug:course_slug>/detail/`
  (`freedom_ls/student_interface/urls.py:10-14`).
- Invocation everywhere: `{% url 'student_interface:course_detail'
  course_slug=course.slug %}`.
- View `views.py:315-417` has no `@login_required` and no backend gate; only
  `HIDDEN`-visibility courses 404 for unregistered users. Registered learners
  already have full access.
- **No change** to the page content, its URL, the view, or who may access it.

## 5. Requirements per surface

### 5.1 Dashboard course cards

- Add a "Details" link to all four card-state leaf templates (§3.3), placed per
  §3.4 (content footer, bottom-right, own line).
- The link is a real `<a href="{% url 'student_interface:course_detail'
  course_slug=course.slug %}">` that is a **sibling** of the stretched title
  anchor, given `position: relative` + a `z-index` above the `::before` overlay
  so it is independently clickable.
- Presentation: the "Browse all courses" text-link pattern — `text-sm
  font-medium no-underline hover:text-primary-hover`, trailing `next` icon in an
  `aria-hidden="true"` span.
- The stretched link keeps its current whole-card behaviour into the player
  (except the not-registered card, whose whole-card link already targets the
  details page — unchanged).

### 5.2 Course index / list rows (all-courses page)

- Add the same "Details" link, identical label and destination, to all four
  row-state leaf templates.
- Rows have materially more horizontal room (`flex-1` details column) than the
  fixed-size card tiles, so the link renders **inline** within the row's details
  column rather than on a dedicated footer line. The coming-soon row already
  demonstrates an inline, right-aligned element beside the stretched title
  (`course_row_coming_soon.html:29-37`) — follow that precedent for placement.
- Same stretched-link sibling + `relative z-10` handling as the card.
- The all-courses page is public; the link must render correctly for anonymous,
  registered, and authenticated-but-unregistered visitors (destination is public
  for all non-hidden courses).

### 5.3 Course player breadcrumbs

- Repoint the **first breadcrumb** (the course title) from the first course item
  to the course details page. This is a one-line edit in
  `partials/player_breadcrumbs.html:16` (and the corresponding `href` at line
  21 / variable name): replace
  `{% url 'student_interface:view_course_item' course_slug=course.slug index=1 %}`
  with `{% url 'student_interface:course_detail' course_slug=course.slug %}`.
- The current behaviour (linking to lesson index 1) reads as "start over", which
  is surprising. Linking the course title to the overview mirrors Moodle/Canvas
  and doubles as the shareable-URL entry point from inside the player.
- **Also update the now-stale comment** at `player_breadcrumbs.html:6`
  ("Ancestor crumbs link to concrete item URLs (never the bare course URL)"),
  which becomes inaccurate for the first crumb once it points at the details
  page. Do not leave the comment contradicting the code.

## 6. Accessibility

- Each "Details" link is a real anchor with the visible text "Details" as its
  accessible name; the trailing chevron icon is decorative (`aria-hidden="true"`
  span), per the codebase's established icon convention.
- Adequate touch target (aim ≥44×44px via padding), consistent with WCAG 2.2
  target-size guidance and the app's existing icon-button padding pattern
  (`p-2` around a `size-4`/`size-5` icon).
- The link must be keyboard-focusable and clickable independent of the stretched
  card link (guaranteed by the `relative` + `z-index` sibling treatment in §3.2);
  it must not be nested inside the title anchor.
- Placement is the **same corner/position on every card**, and the same inline
  position on every row, so it becomes a learned, predictable location.

## 7. Reuse notes (no new models / URLs / views)

- **Destination:** existing `student_interface:course_detail` — no change.
- **Text-link pattern:** copy `course_list.html:92-98` ("Browse all courses").
- **Icon:** existing `next` semantic icon via `<c-icon name="next">` in an
  `aria-hidden` span. No new icon needs to be registered.
- **Shared markup:** if the exact link markup is identical across the leaves of a
  surface, extract a small Django template partial alongside its siblings at
  `templates/student_interface/partials/course_details_link.html` (included as
  `"student_interface/partials/course_details_link.html"`) and `{% include %}` it, rather than
  duplicating markup across the four card leaves and four row leaves. Do **not**
  create a one-off Cotton component (per `templates_and_cotton.md`: partials for
  one-off reuse). Keep the card vs row presentation difference (footer-line vs
  inline) at the call site.
- **Styling:** Tailwind utilities only; no custom CSS. `<c-button>` is explicitly
  *not* used here — its full button chrome is too heavy for a low-emphasis
  secondary link.

Files this feature touches (all under `freedom_ls/student_interface/`):
`templates/student_interface/partials/course_card_registered.html`,
`course_card_complete.html`, `course_card_not_registered.html`,

## 8. Testing

Follow project testing conventions (pytest; **no CSS-class/style assertions** —
assert behaviour and rendered `href`s only).

- **Card affordance** — extend `tests/test_course_cards.py`: for each of the four
  card states, assert the rendered dashboard markup contains a "Details" link
  whose `href` resolves to `course_detail` for that course.
- **Row affordance** — extend `tests/test_all_courses_rows.py`: same assertion
  per row state on the all-courses page, including for anonymous and
  authenticated-unregistered users.
- **Breadcrumb** — new test (a plain Django `Client` test asserting the first
  breadcrumb anchor's `href` on a `view_course_item` response resolves to
  `course_detail`, not to item index 1). No breadcrumb test file exists today.
- **Optional E2E** — a Playwright test (in `tests/e2e/`, following
  `test_course_toc.py` conventions) that clicks a card/row "Details" link and
  asserts navigation to the details page, and clicks the player's first
  breadcrumb and asserts the same. Include only if the plan deems the added
  coverage worthwhile.

## 9. Success criteria

1. From the dashboard, a registered learner can reach a course's details page via
   a visible, low-emphasis "Details" link on the course card, for every card
   state — without that link competing with the primary Continue/Start action.
2. From the all-courses list, the same "Details" link is present on every row
   state, with identical label and destination.
3. Inside the player, clicking the first breadcrumb (course title) lands on the
   course details page, not on lesson one.
4. The "Details" affordance is keyboard-accessible, has an accessible name, meets
   touch-target guidance, and is independently clickable over the stretched link.
5. No change to the details page content, its access rules, or any model/URL.
6. Tests cover the card link (all states), the row link (all states), and the
   breadcrumb href change; existing suite stays green.

## 10. Out of scope

- Any bespoke "share" feature (social buttons, copy-link, Web Share API). The
  details-page URL already satisfies the sharing need. A "Copy link" action could
  be a fast-follow if demand appears — **not** part of this work.
- Any change to the details page content, its layout, or who may access it.
- A kebab/overflow menu. Reserved for the future case of ≥2 secondary actions per
  surface; the groundwork (visible affordance, `relative`+`z-index` seam) makes
  that easy to revisit later.
