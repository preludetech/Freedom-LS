# Implementation Plan: Product Documentation for Freedom LS

_Last updated: 2026-06-09_

Derived from `1. spec.md`. Read the spec first — this plan does not restate the accuracy rules (§3) or the per-doc content checklist (§4.4); it sequences the work and points at the exact files to use.

## 0. Nature of this work

This feature produces **documentation and one new SDD command file** — there is **no application/runtime code** in it. Consequences for how we execute:

- There is no Django code to test, so **TDD does not apply** and no `pytest` tests are written. Verification is link-resolution + a success-criteria pass (Phase 6), not a test suite.
- No models, migrations, views, templates, or settings change. Nothing in `freedom_ls/` is edited.
- The only non-doc artefacts are three SDD-tooling files under `fls-claude-plugin/` (Phase 5).
- Every factual sentence in the docs must trace to code. The **source of truth for facts** is `research_codebase_features.md` (this spec directory), cross-checked against the live repo and the skills listed in §2. Do **not** write from memory of how LMSs "usually" work.

### Phase 3 — Capture screenshots

Follow the `do_qa.md` server lifecycle. **Required**: learner-experience (dashboard, course detail, course player, quiz feedback, finish page) and educator-interface (panel interface, cohort progress matrix). **Optional**: admin-interface. **Skip** pure backend/API features (webhooks, content CLI, auth internals).

1. `PORT=$(.claude/fls/scripts/find_available_port.sh)`
2. `uv run python manage.py runserver $PORT` (read `.claude/fls/config.md` for admin creds; base URL `http://127.0.0.1:$PORT`).
3. Confirm the `debug-branch-badge` names this branch (port-collision guard, per `do_qa.md`).
4. Use **DemoDev** site / demo content for seed data. If required data is missing, delegate to the `fls:qa-data-helper` agent — do not hand-create data.
5. Capture via Playwright MCP into `docs/product/screenshots/` with descriptive names (e.g. `learner_dashboard.png`, `educator_cohort_progress_matrix.png`).
6. Compress: `uv run --with pillow python ${CLAUDE_PLUGIN_ROOT}/scripts/compress_screenshots.py` — must land **under the 1024 KB** pre-commit large-file limit.
7. Kill: `.claude/fls/scripts/kill_runserver.sh $PORT`.
8. Reference from the docs with plain `![](screenshots/<file>.png)` — no cotton, no custom widgets.
9. Once real PNGs are committed, delete the `docs/product/screenshots/.gitkeep` placeholder from Phase 1 — it has served its purpose and should not linger.

The `kill_runserver.sh` teardown (step 7) must run even if a capture step fails, so no dev server is left orphaned on a free port.

### Phase 5 — Build the repeatable update mechanism

Three artefacts under `fls-claude-plugin/`. Author against the `claude-code-authoring` skill.

**5a. New command `fls-claude-plugin/commands/sdd/update_product_docs.md`** (spec §7.1)

- Frontmatter: `description:` + `allowed-tools: Read, Glob, Write, Edit, Bash, Agent, mcp__playwright*`. **No `model:`** on the command file — it is inert when `/sdd:next` inlines it; tiering lives on the agent files.
- Depth-0 orchestrator following the shared fan-out recipe (mirror `improve_idea.md` structure; mirror `do_qa.md` for the Playwright lifecycle).
- Behaviour:
  1. Read the current feature's spec/plan to identify which `docs/product/` files are affected.
  2. Fan out **one `fls:sdd-worker` per affected doc section** to draft updates, each writing to a `.sdd-work/` scratch file with a `status:` footer (resume = skip `status: ok`).
  3. **Synthesise** by reading the worker output files **by path** (never dump contents into the prompt) and applying edits to the real docs.
  4. For visual features: start a dev server on a free port, capture screenshots via Playwright MCP, compress, kill the server — reuse `find_available_port.sh`, `compress_screenshots.py`, `kill_runserver.sh`.
  5. Delete `.sdd-work/` on success.
  6. Final step delegates the todo tick to **`fls:sdd-mechanic`** (calling `update_todo.md`). The mechanic edits `todo.md` directly — it does not depend on `.sdd-work/`, so running it after the step-5 cleanup is correct. (The spec mandates the mechanic hop here per §7.1; this is consistent with the SDD tiering rule that todo-ticking is mechanical work.)

Pseudo-outline of the command body:

```
frontmatter (allowed-tools incl. mcp__playwright*, no model:)
Fan-out recipe block (copied conventions, same as improve_idea.md)
Step 1: Identify affected docs/product/ files from the feature's spec/plan
Step 2: Fan out one fls:sdd-worker per affected doc section → .sdd-work/<doc>.md
Step 3: Synthesise — read worker files by path, edit the real docs
Step 4: If a visual feature changed → screenshot lifecycle (do_qa.md pattern)
Step 5: Delete .sdd-work/
Step 6: Delegate todo tick to fls:sdd-mechanic via update_todo.md
```

**5b. Todo template `fls-claude-plugin/commands/sdd/protected/setup_todo_list.md`** (spec §7.2)

Insert a new section **between QA (current §9) and Pull request**, then renumber trailing sections (Pull request → §11, Cleanup → §12):

## 10. Product documentation

Edit the canonical checklist block inside that file (lines ~104–121). This positioning is correct: code is final, UI is live for screenshots, and the doc change rides the same PR.

**5c. SDD README `fls-claude-plugin/commands/sdd/README.md`** (spec §7.3)

Add the documentation step to the "workflow at a glance" and as a numbered step **after QA, before Ship**. This file uses its **own** Step 1…8 numbering (with an app-map-refresh step) — follow that scheme; do **not** mirror the §7.2 section renumbering.

- The three Phase-5 artefacts exist and read correctly (criteria 7, 8).
- Screenshots present, compressed under 1024 KB, referenced with plain markdown (criterion 5).
