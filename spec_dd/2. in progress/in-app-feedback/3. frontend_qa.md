# In-App Feedback System — Frontend QA Plan

## Prerequisites

- Dev server running: `PORT=$(./find_available_port.sh) && uv run python manage.py runserver $PORT`
- Base URL: `http://127.0.0.1:$PORT`
- A superuser account exists (for admin access)
- At least one course with topics exists and a student user is registered for it
- Use the DemoDev site for all test data

---

## Test 1: Admin — Create a Feedback Form

**Goal:** Verify that feedback forms can be created and configured via the admin.

1. Navigate to `http://127.0.0.1:$PORT/admin/`
2. Log in as a superuser
3. Click on **Feedback Forms** in the admin sidebar
4. Click **Add Feedback Form**
5. **Verify:** The "Trigger point" field is a dropdown/select (not a free text input)
6. **Verify:** The dropdown contains at least `course_completed` and `topic_completed`
7. Fill in:
   - Name: "Course Feedback"
   - Trigger point: `course_completed`
   - Is active: checked
   - Rating label: "How would you rate this course?"
   - Text prompt: "Any suggestions for improvement?"
   - Thank you message: "Thanks for your feedback!"
   - Min occurrences: 1
   - Cooldown days: 0
8. Click **Save**
9. **Verify:** The form is created successfully and appears in the list view
10. **Verify:** List shows columns: name, trigger point, is_active, min_occurrences, cooldown_days

---

## Test 2: Admin — Unique Active Constraint

**Goal:** Verify only one active form per trigger point per site.

1. In the admin, try to create a second FeedbackForm with:
   - Name: "Another Course Feedback"
   - Trigger point: `course_completed`
   - Is active: checked
2. Click **Save**
3. **Verify:** An error is shown indicating that an active form for this trigger point already exists

---

## Test 3: Admin — Invalid Trigger Point Rejected

**Goal:** Since the trigger point is a select dropdown, this should be naturally prevented. But if someone bypasses the UI:

1. **Verify:** The dropdown only shows registered trigger points — no way to type an arbitrary value

---

## Test 4: Course Completion — Feedback Modal Appears

**Goal:** Verify the feedback modal appears after completing a course.

1. Log in as a student user
2. Navigate to a course the student is registered for
3. Work through the course content (or mark topics complete) until you reach the last topic
4. Mark the last topic as complete — this should redirect to the course finish page
5. **Verify:** The course completion page loads with the congratulations message
6. **Verify:** A feedback modal appears on the page (may appear after a brief moment as it lazy-loads)
7. **Verify:** The modal title/header is visible
8. **Verify:** The modal shows "How would you rate this course?" (or whatever rating_label was set)
9. **Verify:** 5 star icons are displayed for rating
10. **Verify:** A text area is shown with the prompt "Any suggestions for improvement?"
11. **Verify:** Submit and Close buttons are present

---

## Test 5: Star Rating Interaction

**Goal:** Verify the star rating works correctly.

1. With the feedback modal open from Test 4:
2. Hover over the 3rd star
3. **Verify:** Stars 1-3 highlight (change color) on hover
4. Click the 4th star
5. **Verify:** Stars 1-4 remain highlighted after click
6. Hover away
7. **Verify:** Stars 1-4 stay highlighted (the selected rating persists)

---

## Test 6: Submit Feedback

**Goal:** Verify feedback submission works.

1. With the feedback modal open and a rating selected:
2. Select a 4-star rating
3. Type "Great course, learned a lot!" in the text field
4. Click **Submit**
5. **Verify:** The modal content changes to show the thank-you message: "Thanks for your feedback!"
6. **Verify:** The modal auto-closes after a brief delay (~2 seconds)
7. Go to the admin → **Feedback Responses**
8. **Verify:** A new response exists with:
   - The correct form name
   - The student's email
   - Rating: 4
   - Comment: "Great course, learned a lot!"
   - The correct course as the content object

---

## Test 7: Dismiss Feedback — Close Button

**Goal:** Verify dismissal via the close button is recorded.

1. Set up a fresh scenario: create a new FeedbackForm for `topic_completed` (or use a new session/user)
2. Trigger the feedback modal to appear
3. Click the **X (close) button** on the modal
4. **Verify:** The modal closes
5. Go to the admin → **Feedback Dismissals**
6. **Verify:** A dismissal record was created for this user and form

---

## Test 8: Dismiss Feedback — Escape Key

**Goal:** Verify dismissal via Escape key.

1. Trigger the feedback modal to appear
2. Press the **Escape** key
3. **Verify:** The modal closes
4. **Verify:** A new dismissal record appears in admin

---

## Test 9: Dismiss Feedback — Backdrop Click

**Goal:** Verify dismissal via clicking outside the modal.

1. Trigger the feedback modal to appear
2. Click on the **backdrop** (dark area outside the modal)
3. **Verify:** The modal closes
4. **Verify:** A new dismissal record appears in admin

## Test 10: Anti-Fatigue — No Re-show for Same Context

**Goal:** Verify the form doesn't appear again for the same user + course after responding.

1. Complete a course and submit feedback (done in Test 6)
2. Log out and log back in (new session)
3. Navigate to the same course's finish page (e.g., via direct URL)
4. **Verify:** No feedback modal appears (the user already responded for this course)

---

## Test 11: Anti-Fatigue — Session Limit

**Goal:** Verify only one feedback prompt per session.

1. Create FeedbackForm entries for both `course_completed` and `topic_completed`
2. Log in as a student
3. Complete a topic (triggers `topic_completed`)
4. **Verify:** Feedback modal appears (or would appear on the next page)
5. Submit or dismiss the feedback
6. Complete another topic or a course in the SAME session
7. **Verify:** No second feedback modal appears in the same session

---

## Test 12: Anti-Fatigue — 3 Dismissals Stop Showing

**Goal:** Verify the form stops after 3 dismissals.

1. Create a fresh user or clear existing dismissal records
2. Set up a FeedbackForm for `topic_completed`
3. Complete a topic → feedback modal appears → dismiss it. Log out and log back in.
4. Complete another topic → feedback modal appears → dismiss it. Log out and log back in.
5. Complete another topic → feedback modal appears → dismiss it (3rd time). Log out and log back in.
6. Complete yet another topic
7. **Verify:** No feedback modal appears — the user has dismissed 3 times

---

## Test 13: Anti-Fatigue — Cooldown Period

**Goal:** Verify the cooldown_days setting is respected.

1. Create a FeedbackForm with `cooldown_days = 30`
2. Complete a course and submit feedback
3. Log out and log back in (new session)
4. Complete a different course
5. **Verify:** No feedback modal appears (within the 30-day cooldown window)

*Note: To fully test this, you may need to adjust the `created_at` timestamp of the existing FeedbackResponse in the database to be > 30 days ago, then verify the modal appears again.*

---

## Test 14: Min Occurrences Threshold

**Goal:** Verify the min_occurrences setting delays showing the form.

1. Create a FeedbackForm for `topic_completed` with `min_occurrences = 3`
2. Log in as a fresh user
3. Complete a topic
4. **Verify:** No feedback modal (only 1st trigger)
5. Complete a second topic
6. **Verify:** No feedback modal (only 2nd trigger)
7. Complete a third topic
8. **Verify:** Feedback modal appears (3rd trigger meets the threshold)

---

## Test 15: Admin — Read-Only Response and Dismissal Views

**Goal:** Verify response and dismissal admin views are read-only.

1. Go to admin → **Feedback Responses**
2. **Verify:** There is no "Add" button
3. Click on an existing response
4. **Verify:** All fields are read-only (no editable inputs)
5. Go to admin → **Feedback Dismissals**
6. **Verify:** There is no "Add" button
7. **Verify:** Records are displayed in a list view

---

## Test 16: HTMX Partial Response — Modal Loads via HX-Trigger

**Goal:** Verify feedback works when triggered during an HTMX partial update (not a full page load).

1. Set up a FeedbackForm for `topic_completed`
2. If topic completion uses a form POST that results in a redirect (full page load), this test may not apply directly. Check whether any topic completion flow uses HTMX partial updates.
3. If applicable: trigger a completion via HTMX
4. **Verify:** The feedback modal appears even though the page didn't fully reload — it was triggered by the HX-Trigger response header

*Note: The current `view_topic` uses a redirect for topic completion, so the full-page-load path is the primary flow. The HTMX path is primarily for future/downstream use. The middleware should still be verified with a manual test or integration test.*
