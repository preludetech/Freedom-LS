# Frontend QA: Coming Soon & Hidden Courses

Manual / browser QA for the visibility feature. Verifies the student-facing surfaces
(discovery, badges, detail page, express-interest HTMX flow, hidden 404) and the educator
surfaces (visibility column, interest count, drill-down, read-only visibility).

## Setup

1. Start the dev server on a random free port (not 8000):
   - `PORT=$(.claude/fls/scripts/find_available_port.sh)`
   - `uv run python manage.py runserver $PORT`
   - Base URL: `http://127.0.0.1:$PORT`
2. Use the **DemoDev** site for all test data.
3. Test data needed (create via the `fls:qa-data-helper` agent if missing):
   - A **student** account and an **educator** account on the DemoDev site.
   - At least four courses on DemoDev:
     - `published-free` — visibility `published`, free access.
     - `coming-soon` — visibility `coming_soon` (any access type).
     - `hidden-course` — visibility `hidden`, with **no** registration for the test student.
     - `hidden-registered` — visibility `hidden`, **with** the test student registered for it.
   - (The demo content from Task 6.1 already includes a `coming_soon` course; the others may
     need to be created.)

Throughout: assert on **behaviour and visible text/labels**, not on CSS classes or colours.

---

## Test 1 — Coming-soon course is discoverable and badged

1. Log in as the **student**.
2. Go to the all-courses page: `http://127.0.0.1:$PORT/` → "All courses" (or
   `/courses/` per the `student_interface:courses` route).
3. **Expect:** the `coming-soon` course is listed, shows a **"Coming soon"** badge/eyebrow,
   and shows an **"I'm interested"** control. It must **not** show an "Enrol" / "Start" /
   "Apply now" button.
4. Switch to the card/grid view (dashboard) and confirm the same coming-soon presentation on
   the card.

## Test 2 — Express interest (HTMX swap) and idempotency

1. As the **student**, on the `coming-soon` course (listing or detail page), click
   **"I'm interested"**.
2. **Expect:** the CTA area changes **immediately** (no page reload) to an **"Interested"**
   state with a confirmation message and a quiet **"Remove interest"** secondary link. The
   confirmation must **not** promise an email / notification.
3. Click "I'm interested" again is not possible (it is now the interested state) — reload the
   page and confirm it still shows **"Interested"** (state persisted, exactly one interest
   recorded).
4. Click **"Remove interest"**.
5. **Expect:** the CTA swaps back to **"I'm interested"** immediately. Reload — still
   not-interested.

(Record this flow as a GIF if a shareable artefact is wanted.)

## Test 3 — Coming-soon detail page

1. As the **student**, open the `coming-soon` course detail page directly
   (`/courses/<slug>/` → `student_interface:course_detail`).
2. **Expect:** the page is **accessible** (not 404), reads as "coming soon" (coming-soon funnel
   copy, not the free "One click. No credit card." or gated "Application required" copy), and
   the primary affordance is the **express-interest control** reflecting the current interest
   state — **not** a plain enrol link.
3. Confirm there is no content/player access (clicking through does not open lessons).

## Test 4 — Hidden course is invisible to unregistered students

1. As the **student** (not registered for `hidden-course`), browse all-courses and the dashboard.
2. **Expect:** `hidden-course` does **not** appear anywhere in discovery.
3. Navigate directly to its detail URL `/courses/<hidden-course-slug>/`.
4. **Expect:** **HTTP 404** (not a redirect, not a login prompt that confirms existence).

## Test 5 — Hidden course stays available to a registered student

1. As the **student** registered for `hidden-registered`, go to the dashboard.
2. **Expect:** `hidden-registered` **appears** in the registered-courses list and is openable.
3. Navigate directly to its detail URL and to its content.
4. **Expect:** detail page loads (200) and content is accessible — mid-course access is
   preserved despite the hidden state.

## Test 6 — Published courses unchanged

1. As the **student**, open `published-free` in discovery and on its detail page.
2. **Expect:** unchanged behaviour — normal "Start"/"Continue" CTA, no coming-soon badge, no
   express-interest control, content accessible per access type.

## Test 7 — Educator demand view + visibility management

1. Log out; log in as the **educator**.
2. Go to the educator course-management list.
3. **Expect:** every course shows its **visibility** state, and a **interest count** column;
   the `coming-soon` course shows a count matching the number of students who expressed interest
   (≥1 after Test 2 if you re-expressed interest).
4. Open the `coming-soon` course's detail/instance view.
5. **Expect:** a panel listing **interested students** (name + when they expressed interest).
6. On the course details panel, look for any way to **change** the visibility.
7. **Expect:** visibility is **read-only** — there is no Edit affordance for it in the educator
   interface (and likewise the Django admin shows it greyed-out / non-editable). Visibility is
   set only via course content front-matter.
8. To flip the course `coming_soon → published`, edit the course `course.md` front-matter
   `visibility:` value (or use the `qa_create_course_visibility` management command) and
   re-import. Then log back in as the **student**, revisit discovery, and confirm the
   (formerly coming-soon) course is now **enrollable** (normal CTA, no express-interest
   control) — and that no launch notification was sent or promised anywhere.

## Test 8 — No scarcity / no notification promises

1. Across all coming-soon surfaces (cards, rows, detail, confirmation swap), confirm there is
   **no** queue position, **no** "X people ahead", **no** countdown, **no** interest count shown
   to students, and **no** copy promising an email or in-app notification on launch.
