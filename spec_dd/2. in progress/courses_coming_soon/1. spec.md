# Spec: Coming Soon & Hidden Courses

## 1. Overview

Today every course in FLS is always discoverable; access is decided entirely by the
active **course-access backend** (`settings.COURSE_ACCESS_BACKEND`) reading each course's
opaque `access_config` (its *access type*: free vs application-gated).

This feature adds a **course visibility** dimension that is *orthogonal* to access type. A
course is independently *free-or-gated* (its `access_config`) **and**
*published / coming-soon / hidden* (its `visibility`). The two compose freely: an
application-gated course can be "coming soon", a free course can be "hidden", and so on.

Three visibility states ship:

- **Published** (default) — current behaviour: discoverable everywhere; access decided by
  the active backend.
- **Coming soon** — discoverable and clearly badged, but not yet enrollable. Students can
  **express interest** (a lightweight, capacity-free waitlist), giving educators a demand
  signal for what to launch next.
- **Hidden** — removed from student discovery: excluded from all browse/discovery surfaces,
  and the detail page returns 404 for students who aren't already registered.

Visibility is enforced **once, in shared code that wraps every backend**, so free,
application-gated, and any future backend honour coming-soon and hidden with no per-backend
code and no way to bypass or accidentally override it.

### Why this matters

- **Coming soon** lets educators advertise upcoming courses and measure demand *before*
  investing in launch — turning a guess into a signal (which course to prioritise).
- **Hidden** supports courses that should exist (for in-progress learners, deep links, or
  staged rollouts) without cluttering discovery or leaking unfinished material.
- **Orthogonality** keeps the lifecycle concept (published/coming-soon/hidden) cleanly
  separated from the backend-private access mechanism (free/gated), so each evolves
  independently and neither validation pipeline contaminates the other.

## 2. Goals and non-goals

### Goals

- A first-class `visibility` field on `Course` with three states, defaulting to `published`
  (existing courses stay visible after migration, no data migration needed).
- Coming-soon and hidden enforced uniformly across **all** access backends via a single,
  un-bypassable wrapper.
- An express-interest waitlist for coming-soon courses with immediate visible feedback,
  idempotency, and the ability to leave.
- An educator demand signal: interest count + drill-down to interested students.
- Visibility set from course front-matter at import, defaulting to `published`.

### Out of scope (deferred)

- Email / in-app notifications on launch (FLS has no notification system yet).
- Auto-enrolling interested students on launch.
- A separate `draft` state (educator-only, invisible even by direct link) — a one-line enum
  addition later if needed.
- Capacity-limited waitlists, queue positions, "X people ahead of you", or any scarcity
  mechanic. FLS courses have no capacity limit; implying one would mislead.

## 3. Key design decisions

### 3.1 Single enum field, not boolean flags

`visibility` is one `CharField` backed by a `CourseVisibility` `TextChoices` enum, mirroring
the existing `DifficultyLevel` pattern (`content_engine/models.py:33-39`). Rationale:

- States are mutually exclusive by construction — no `clean()` guards against impossible
  combinations like `is_hidden=True, is_coming_soon=True`.
- Student-facing filtering is a one-line predicate.
- Trivially extensible (`draft`, `archived`) as a future one-line enum + migration addition.
- Consistent with the codebase's established `TextChoices` convention.

`visibility` is deliberately **not** part of `access_config`. `access_config` is opaque and
backend-private (only the active backend may interpret it); `visibility` is a universal
lifecycle concept the platform, templates, and the import layer all read directly. The two
must not be conflated — see §9 (content import) and §5.4 (validation boundary).

### 3.2 Visibility enforced by a loader-level wrapper

The single hardest requirement is: *coming-soon and hidden must be honoured by every
backend, with no per-backend configuration and no way to bypass or accidentally override
it.* A check placed inside `FreeOnlyCourseAccessBackend.get_access` would be **silently
bypassed**, because `ApplicationCourseAccessBackend.get_access` overrides it and only calls
`super().get_access()` for the free/registered cases — its `application_gated`-and-
unregistered branch returns its own "Apply now" decision without ever delegating up
(`course_applications/backends.py:61-122`, confirmed). A base-class concrete `get_access`
would still be overridable by a future subclass; a "shared helper you must call first" can
be forgotten.

**Decision:** enforce visibility in a `VisibilityEnforcingBackend` wrapper applied by the
backend loader. `get_course_access_backend()` (`course_access/loader.py`) returns the
configured backend wrapped in this decorator. The wrapper:

- implements the full `CourseAccessBackend` interface,
- applies visibility in `get_access()` and `filter_visible()` **before/around** delegating
  to the inner backend,
- forwards `validate_course_config()` and `get_dashboard_contributions()` unchanged.

Because the loader **only ever hands out the wrapper**, nothing — not a forgetful caller,
not a future subclass — can bypass visibility, and **no existing call sites or backend
subclasses change**. Backends still implement `get_access` / `filter_visible` as their
seams, which is exactly what they should override; the visibility layer lives outside what
subclasses touch, so it cannot be clobbered by mistake.

```python
# course_access/loader.py
@functools.cache
def get_course_access_backend() -> CourseAccessBackend:
    inner_class = import_string(settings.COURSE_ACCESS_BACKEND)
    return VisibilityEnforcingBackend(inner_class())
```

> Note: the loader is `@functools.cache`d. Tests overriding `COURSE_ACCESS_BACKEND` must
> call `get_course_access_backend.cache_clear()`, as today. The wrapper is transparent to
> this.

### 3.3 New `course_interest` app

The `CourseInterest` model, the express-interest views/URLs, and the educator drill-down
data live in a new app `course_interest` (app label `freedom_ls_course_interest`), mirroring
the `course_applications` structure. This keeps the waitlist concern self-contained and
avoids growing `student_management`. One entry added to `INSTALLED_APPS`.

## 4. Data model

### 4.1 `CourseVisibility` + `visibility` field (on `Course`, `content_engine`)

```python
class CourseVisibility(models.TextChoices):
    PUBLISHED = "published", _("Published")
    COMING_SOON = "coming_soon", _("Coming soon")
    HIDDEN = "hidden", _("Hidden")

# on Course:
visibility = models.CharField(
    max_length=20,
    choices=CourseVisibility.choices,
    default=CourseVisibility.PUBLISHED,
    db_index=True,
)
```

- `default=PUBLISHED` means the `makemigrations`/`migrate` backfills every existing row to
  `published` — no separate data migration.
- `db_index=True` because the field is filtered on every discovery query.
- New migration in `content_engine` (latest is `0012_course_access_config.py`). **Never edit
  existing migrations.**

### 4.2 `CourseInterest` (new `course_interest` app)

```python
class CourseInterest(SiteAwareModel):
    user = models.ForeignKey(
        settings.AUTH_USER_MODEL, on_delete=models.CASCADE,
        related_name="course_interests",
    )
    course = models.ForeignKey(
        "freedom_ls_content_engine.Course", on_delete=models.CASCADE,
        related_name="interests",
    )
    created_at = models.DateTimeField(auto_now_add=True)

    class Meta:
        constraints = [
            models.UniqueConstraint(
                fields=["user", "course"], name="unique_course_interest"
            )
        ]
```

- Inherits `SiteAwareModel` (UUID pk + `site` FK + site-aware manager) for automatic site
  isolation — follow the `fls:multi-tenant` skill for setting `site` on create.
- FK name `course` (not `collection`) — follows the `CourseApplication` precedent rather
  than the older generic-FK `collection` naming in `UserCourseRegistration`.
- No `notified_at` field in v1 — building an unused column now is out of scope. Notify-on-
  launch adds it via its own migration when that feature is implemented. See §10.
- Unique on `(user, course)`; created via `get_or_create` so repeated clicks are idempotent.

## 5. Backend enforcement (`VisibilityEnforcingBackend`)

Lives in `course_access` (alongside the backend hierarchy). Implements
`CourseAccessBackend`, holds the inner backend, and applies visibility.

### 5.1 `get_access(*, user, course)` — visibility resolved first

```python
def get_access(self, *, user, course):
    if course.visibility == CourseVisibility.COMING_SOON and not is_registered_for_course(user, course):
        return self._coming_soon_decision(course)
    if course.visibility == CourseVisibility.HIDDEN and not is_registered_for_course(user, course):
        return self._no_access_decision()
    return self._inner.get_access(user=user, course=course)
```

Both `coming_soon` and `hidden` **exempt users already registered for the course**: a
registered learner delegates to the inner backend and keeps full access, so a visibility
change never disrupts a mid-course learner.

- `coming_soon` + unregistered → an **express-interest** decision: `can_self_register=False`,
  `can_access_content=False`, `cta_label="I'm interested"`,
  `cta_url=reverse("course_interest:express_interest", kwargs={"course_slug": course.slug})`.
  The decision **also sets/blanks** the detail-page funnel copy fields
  (`enrolment_summary`, `acquisition_heading`, `acquisition_subtext`) so the page reads as
  "coming soon" rather than inheriting free ("One click. No credit card.") or gated
  ("Application required") copy.
  - **`cta_url` is an HTMX POST target, not a plain link.** `express_interest` is
    POST-only/HTMX (§7.1), so the detail page must **not** render this CTA as the generic
    `<c-button href="…">` anchor used for free/gated courses (`course_detail.html:132-134`)
    — that would issue a GET to a POST-only endpoint and never reflect existing interest.
    The coming-soon detail-page affordance is an HTMX express-interest control that reflects
    the user's interest state on load. It is the **only** surface that renders this control —
    coming-soon listings are plain detail links (§8). See §6.3.
- `hidden` + unregistered → a safe **no-access** decision: `can_self_register=False`,
  `can_access_content=False`, `cta_label=None`, `cta_url=None`, funnel copy blanked.
- `published` (and `coming_soon`- or `hidden`-but-registered) → delegate to
  `self._inner.get_access(...)`, preserving every existing access-type behaviour (free →
  "Start", gated → "Apply now", registered → "Continue", etc.).

`is_registered_for_course` is the shared `student_management.utils` implementation already
used by both backends (`student_management/utils.py:67-91`) — reuse it; do not reimplement.

> **Cross-app reference (flag for `/plan_structure_review`):** the coming-soon decision's
> `cta_url` is built with `reverse("course_interest:express_interest", ...)`. This is a
> runtime URL-name lookup (no Python import), a loose coupling from `course_access` →
> `course_interest` introduced deliberately because the idea requires the CTA URL to come
> from the `CourseAccessDecision` so every backend presents it consistently. The wrapper
> does **not** import the `CourseInterest` model or the `course_interest` app.

### 5.2 `filter_visible(*, user, courses)` — hidden dropped after delegation

```python
def filter_visible(self, *, user, courses):
    courses = self._inner.filter_visible(user=user, courses=courses)
    if not user.is_authenticated:
        return courses.exclude(visibility=CourseVisibility.HIDDEN)
    registered = Exists(  # mirrors is_registered_for_course at the queryset level
        UserCourseRegistration.objects.filter(
            collection=OuterRef("pk"), user=user, is_active=True
        )
    ) | Exists(
        CohortCourseRegistration.objects.filter(
            collection=OuterRef("pk"),
            cohort__cohortmembership__user=user, is_active=True,
        )
    )
    return courses.annotate(_is_registered=registered).exclude(
        Q(visibility=CourseVisibility.HIDDEN) & Q(_is_registered=False)
    )
```

- Delegates to the inner `filter_visible` first (which currently returns everything), then
  **drops `hidden` courses except those the user is registered for** — so registered
  learners keep access mid-course.
- `coming_soon` courses are **not** dropped: students see Published + Coming soon in
  discovery (`§6`). A `coming_soon` course the learner is registered for is surfaced through
  the registered-course lists (which bypass `filter_visible`) and shows its normal
  registered status, not the express-interest affordance.
- Uses `Exists()` subqueries (not joins) to avoid row duplication. The exact registration
  predicate mirrors `is_registered_for_course`; the implementation may extract a shared
  queryset helper to keep the two in lockstep.

### 5.3 Discovery surfaces are already wired

`filter_visible()` is already the discovery choke-point, called by `dashboard`
(`student_interface/views.py:150-152`) and `all_courses` (`:185-189`) over
`get_all_courses()`. No new filtering is added anywhere else. Registered- and
completed-course lists on the dashboard are fetched **independently of** `filter_visible`
(direct registration/progress queries), which is exactly what keeps a hidden-but-registered
course on the learner's dashboard.

### 5.4 Validation boundary is untouched

`visibility` is **not** subject to `validate_course_config()` nor the
`freedom_ls_course_access.E001` system check — both validate only `access_config`
(`course_access/checks.py:22-59`, confirmed). The wrapper forwards `validate_course_config`
unchanged. Keep the two pipelines distinct.

## 6. Visibility rules (summary)

Both `coming_soon` and `hidden` exempt already-registered learners — a registered learner
sees the course exactly as `published` (normal detail page, content open, registered CTA).
The columns below marked "(unregistered)" describe the non-registered experience.

| Surface | Published | Coming soon (unregistered) | Hidden (unregistered) | Coming soon / Hidden (registered) |
|---|---|---|---|---|
| Browse / discovery (`filter_visible`) | shown | shown, badged, plain detail link (no express-interest CTA — §8) | filtered out | shown (via registered-course lists, bypasses filter) |
| Detail page | normal | accessible, HTMX express-interest CTA reflecting interest state (§6.3) | **404** | accessible, normal registered CTA |
| Content player (`can_access_content`) | per access type | closed (`False`) | closed (`False`) | open |
| Educator / admin querysets | all states shown, badged | all states shown | all states shown | all states shown |

### 6.1 Hidden detail page → 404 derivation

`get_access()` carries no 404 signal (confirmed — it returns a `CourseAccessDecision`,
never raises `Http404`). `course_detail` (`student_interface/views.py:203-261`) currently
does a bare `get_object_or_404(Course, slug=...)` with **no** visibility check. Add, right
after the registration lookup it already performs:

```python
course = get_object_or_404(Course, slug=course_slug)
raise_404_if_hidden_unregistered(request.user, course)
```

The hidden-404 rule is centralised in `course_access.visibility.raise_404_if_hidden_unregistered`
(a shared helper: `HIDDEN and not is_registered_for_course` → `Http404`) so every view
chokepoint enforces it identically and cannot drift from the wrapper's `filter_visible`
rule. It is reused by `course_detail`, the `course_applications` `apply` endpoint, and the
`course_interest` `express_interest` endpoint (§13) — hidden courses never confirm their
existence through any of those surfaces. `coming_soon` and `published` detail pages remain
accessible.

### 6.2 Content gate needs no new wiring

`course_home` (`:264-289`) and `view_course_item` (`:334-424`) already gate purely on
`decision.can_access_content`; coming-soon-unregistered and hidden-unregistered both yield
`False`, so the player closes (redirect to detail) with no extra code. A learner registered
for a coming-soon or hidden course delegates to the inner backend and gets
`can_access_content=True`, so their content player stays open.

### 6.3 Coming-soon detail-page CTA reflects interest state

The detail page currently renders the not-registered CTA as a GET anchor
(`course_detail.html:132-134`, `<c-button href="{{ start_url }}">`). For a `coming_soon`
course this is wrong on two counts: `express_interest` is POST-only/HTMX (§7.1), and a plain
link cannot show whether the viewer has **already** expressed interest. So for `coming_soon`
courses the viewer is **not** registered for, `course_detail` must render the
**express-interest control** instead of the generic href button (a registered learner falls
through to the normal registered CTA):

- The view stamps `is_interested` on the course (a single per-user lookup for the one
  coming-soon course being viewed) so the page renders in the correct state on load —
  **"I'm interested"** (HTMX POST to `decision.cta_url`) when not interested,
  **"Interested" + quiet "Remove interest"** when already interested (§7.2).
- `decision.cta_url` remains the source of the action URL (the loose `course_access →
  course_interest` coupling of §5.1 is preserved), but it is used as the HTMX target, not an
  `<a href>`.
- `published`, and any course the viewer is registered for (`coming_soon`- or
  `hidden`-registered), keep the existing generic CTA unchanged.

The detail page is the **only** surface that renders the express-interest control; coming-soon
listings are plain detail links with no CTA (§8).

## 7. Express interest (waitlist) for coming-soon courses

### 7.1 Endpoints (`course_interest` app, `app_name = "course_interest"`)

- `express_interest` — **POST** (HTMX), login required. `get_or_create(CourseInterest,
  user=request.user, course=course)`; idempotent. Guard: the course must be `coming_soon`
  (otherwise return 422 / no-op). Returns the **swapped CTA partial** in the "interested"
  state. URL path `courses/<slug:course_slug>/express-interest/`.
- `remove_interest` — **POST** (HTMX), login required. Deletes the user's `CourseInterest`
  for the course (no error if absent). Returns the CTA partial in the "not interested"
  state. URL path `courses/<slug:course_slug>/remove-interest/`.

Conventions: URL names snake_case, paths kebab-case; HTMX validation errors return **422**;
CSRF via the global `hx-headers` body attribute. Anonymous users hitting the affordance are
sent through login and returned to the course page (standard `login_required` redirect).

### 7.2 Feedback and state (UX requirements)

- After clicking, the CTA area **must visibly change immediately** (HTMX swap) — no change
  reads as failure and provokes repeat-clicks. The swap shows a confirmation acknowledging
  the action and setting the expectation that the learner will be told when the course is
  ready (e.g. "we'll let you know when the course is ready"). v1 records interest only and
  sends **no** automated notification — the copy sets a soft expectation that notify-on-launch
  fulfils later; see §10.
- The "interested" state shows a muted **"Interested"** label plus a quiet, low-prominence
  **"Remove interest"** secondary link — not a prominent toggle competing with the primary
  CTA.
- No queue position, no count shown to students, no countdown — nothing implying scarcity.

## 8. Templates: cards, rows, badges

- **Badge:** a "Coming soon" badge reads `course.visibility` **directly** (it is a normal
  field, not `access_config`, so the backend-private rule does not apply).
- **Card/row dispatch:** add a `coming_soon` branch to the existing dispatch seams —
  `dispatch-card` in `course_list.html` and `dispatch-row` in `course_row_list.html` —
  keying on `course.visibility == coming_soon`, reusing the existing `course-card-shell` /
  `course-row-shell` components. The coming-soon leaf renders a **single** plain,
  stretched-link card/row pointing at `student_interface:course_detail` (the same pattern as
  the `course_..._not_registered.html` leaves), distinguished only by the "Coming soon"
  chip/eyebrow. There is **no enrol/apply control and no express-interest CTA** in listings —
  the express-interest affordance lives only on the course detail page (§6.3).
- **Per-user interest state:** listings do **not** show the interested-vs-not state, so they
  need no `CourseInterest` lookup — no `is_interested` stamp, no batching. Only `course_detail`
  stamps `is_interested`, for the single coming-soon course being viewed, so the detail-page
  express-interest control renders in the right state on load (§6.3). The express-interest
  control (not-interested / interested states) is a shared partial used only by the detail
  page.
- `course_status_eyebrow.html` gains a `coming_soon` status string.
- Follow the `fls:frontend-styling`, `fls:template`, `fls:icon-usage`, `fls:htmx`, and
  `brand-guidelines` skills for all markup, icons, HTMX, and copy. No styling assertions in
  tests.

## 9. Educator demand view and visibility management

- **Visibility badge column** in the educator course-management list (`CourseDataTable`,
  `educator_interface/views.py:776-858`) — educators see every course's state at a glance.
  Educator querysets already use `Course.objects.all()` (no visibility filter), so educators
  and admins see all states unchanged.
- **Interest count** per course: annotate `CourseDataTable.get_queryset` with a count of
  `CourseInterest` records (meaningful for coming-soon courses) and show it as a column.
- **Drill-down** to the list of interested students (name + `created_at`) for a course —
  surfaced as a panel on `CourseInstanceView` (or an equivalent educator view) reading
  `CourseInterest`. This is what makes the waitlist actionable (which course to launch next)
  rather than data that goes nowhere.
- **Setting visibility (content-file-only):** `visibility` is **not** editable in the educator
  interface or the Django admin. It is configured solely in the course content front-matter
  (§11) and loaded into the DB at import. The educator `CourseDetailsPanel` and the admin
  display `visibility` **read-only**; flipping a course `coming_soon → published` is done by
  editing the course `course.md` front-matter and re-importing. So the educator
  `CourseDetailsPanel` stays read-only (`fields = ["title", "category"]`, not editable) and the
  admin lists `visibility` in `readonly_fields`.

`educator_interface` reading the `CourseInterest` model is a new read-only cross-app
reference (educator_interface → course_interest); `educator_interface` already reads across
apps (Course, registrations). Flag for `/plan_structure_review`.

## 10. Launch transition and the notification dependency

**v1 records interest only.** When an educator flips `visibility` from `coming_soon` to
`published`, the course simply becomes enrollable (or apply-gated, per its access type);
interested students see it as available on their next visit. FLS has **no** email/in-app
notification system yet, so v1 sends **no automated notification**. The coming-soon copy may
set a soft expectation that the learner will be told when the course is ready (e.g. "we'll
let you know when the course is ready"); that expectation is fulfilled by notify-on-launch
when it ships — it must not, however, imply scarcity (queue position, count, countdown; §7.2).

Notify-on-launch (or auto-enrol) is a **deferred dependency** on a future notification system
— flagged here and excluded from scope (§2). The `notified_at` stamp it needs is **not** built
now; that feature adds the field via its own migration when it is implemented.

## 13. Security and correctness considerations

- **Hidden means hidden:** unregistered students get a 404 on the detail page (not a
  redirect that confirms existence) and the course never appears in discovery. The 404
  derivation (§6.1) and the `filter_visible` drop (§5.2) must agree. Every view chokepoint
  that resolves a course by slug enforces this via the shared
  `raise_404_if_hidden_unregistered` helper (§6.1): `course_detail`, the `apply` endpoint,
  and `express_interest` all 404 for a hidden course rather than leaking its existence (e.g.
  via a distinguishable 422). The `apply` endpoint additionally redirects `coming_soon`
  courses to the detail page rather than creating an application (they are not enrollable).
- **Authn required** for express/remove interest; **idempotent** create; users can only
  create/remove **their own** interest record.
- **Site isolation:** `CourseInterest` is a `SiteAwareModel`; interest counts and
  drill-downs are automatically site-scoped via the site-aware manager.
- **No bypass:** because enforcement is in the loader wrapper, a future custom backend
  configured via `COURSE_ACCESS_BACKEND` inherits coming-soon/hidden automatically.

## 14. Success criteria

1. A course can be set to `published`, `coming_soon`, or `hidden` via course content
   front-matter import only (the educator panel and Django admin display it read-only and
   cannot change it); existing courses are `published` after migration with no data
   migration.
2. With **either** the free or the application-gated backend active (and any future
   backend), a `coming_soon` course shows the "I'm interested" affordance and blocks
   content for unregistered students, and a `hidden` course is filtered from discovery and
   404s for unregistered students — verified without per-backend code.
3. A `coming_soon` **or** `hidden` course that a student is **already registered for**
   remains on their dashboard, accessible by direct URL, with content open and the normal
   registered CTA — a visibility change never disrupts a mid-course learner.
4. On the course detail page, clicking "I'm interested" creates exactly one `CourseInterest`
   (idempotent on repeat clicks), immediately swaps the CTA to an "Interested" state with
   confirmation, and the student can remove their interest via a quiet secondary action.
5. Coming-soon cards/rows are clearly badged and visually distinct from enrollable ones,
   with no enrol/apply button and no express-interest CTA — they link to the course detail
   page, where interest is expressed.
6. The educator course-management list shows each course's visibility and, for coming-soon
   courses, an interest count with a drill-down to interested students.
7. No UI implies scarcity (queue position, count shown to students, countdown); coming-soon
   copy may set a soft "we'll let you know when it's ready" expectation, but v1 sends no
   automated notification.
8. All new behaviour is covered by tests (functionality, not styling); `uv run pytest`
   passes.
