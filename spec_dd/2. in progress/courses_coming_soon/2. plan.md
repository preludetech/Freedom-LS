# Implementation Plan: Coming Soon & Hidden Courses

This plan implements the spec (`1. spec.md`). It is sequenced so each phase builds on a
working, tested predecessor. Follow **TDD** throughout (red → green → refactor): for each
task write the failing test(s) named below *first*, then the implementation. Do **not**
pre-write the whole test suite up front.

Skills are the source of truth for *how* to write each kind of code — consult the named
skill before starting a task, not after. Key skills per area: `fls:multi-tenant`,
`fls:admin-interface`, `fls:htmx`, `fls:template`, `fls:frontend-styling`, `fls:icon-usage`,
`fls:markdown-content`, `fls:testing`, `brand-guidelines`, `code-comments`. (Full per-area
mapping below; `fls:registration` and `fls:claude-code-authoring` are **not** relevant.)

> Conventions reminder (CLAUDE.md): type hints on all functions, no `Any`, modern syntax
> (`X | None`, `list[str]`), ORM only, `select_related`/`prefetch_related` for related
> queries, 422 for HTMX validation errors, `get_object_or_404` for view lookups, every app
> defines `app_name`, URL names snake_case, paths kebab-case. Never edit existing migrations.

## Phase 2 — Visibility enforcement wrapper (the un-bypassable seam)

This is the single hardest requirement and the highest-value test target. It must be in place
before any surface relies on coming-soon/hidden behaviour.

### Task 2.1 — shared registration-`Exists` queryset helper

**File:** `freedom_ls/student_management/utils.py` (alongside `is_registered_for_course`,
`:67-91`).
**Skills:** `fls:testing`.

`filter_visible` (§5.2) needs a *queryset-level* version of the same predicate
`is_registered_for_course` computes per-row, and `course_detail` (§6.1) needs the per-row
check. Extract a small helper so the two cannot drift:

```python
def registered_course_exists(user: RequestUser) -> Exists:
    """Exists() subquery: courses (OuterRef pk) this user is registered for.

    Queryset-level mirror of is_registered_for_course, so the wrapper's
    filter_visible and the per-row check stay in lockstep.
    """
    # Lazy import inside the body — mirrors is_registered_for_course (utils.py:76-79),
    # which imports these models locally to avoid a module-load import cycle.
    from freedom_ls.student_management.models import (
        CohortCourseRegistration,
        UserCourseRegistration,
    )

    return Exists(
        UserCourseRegistration.objects.filter(
            collection=OuterRef("pk"), user=user, is_active=True
        )
    ) | Exists(
        CohortCourseRegistration.objects.filter(
            collection=OuterRef("pk"),
            cohort__cohortmembership__user=user, is_active=True,
        )
    )
```

`Exists` / `OuterRef` come from `django.db.models` (module-level import is fine; only the FLS
models must be imported lazily). Keep `is_registered_for_course` as the per-row source of
truth; this helper is its set-level twin. (The exact registration filter mirrors
`is_registered_for_course`: `collection=course`, `is_active=True`, direct **or** cohort.)

**Tests (write first):** a queryset test annotating a 2-course set for a user registered for
one (directly) and one via cohort, asserting both annotate `True` and an unrelated course
annotates `False`.

### Task 2.2 — `VisibilityEnforcingBackend` wrapper

**Files:** `freedom_ls/course_access/backends.py` (add the wrapper class alongside the
hierarchy) and `freedom_ls/course_access/loader.py` (wrap at the loader).
**Skills:** `fls:testing` (MUST — TDD the wrapper, the loader, and `cache_clear`).

Implement the full `CourseAccessBackend` interface, holding the inner backend:

```python
class VisibilityEnforcingBackend(CourseAccessBackend):
    """Applies Course.visibility around any inner backend. Applied by the loader so
    no backend (present or future) can bypass coming-soon / hidden."""

    def __init__(self, inner: CourseAccessBackend) -> None:
        self._inner = inner

    def get_access(self, *, user, course):
        if course.visibility == CourseVisibility.COMING_SOON:
            # Build the express-interest decision INLINE (see below) — match the
            # existing backends, which construct their decisions inline.
            return CourseAccessDecision(...)  # coming-soon decision, fields below
        if course.visibility == CourseVisibility.HIDDEN and not is_registered_for_course(user, course):
            return CourseAccessDecision(  # safe closed decision
                cta_label=None, cta_url=None,
                can_self_register=False, can_access_content=False,
            )
        return self._inner.get_access(user=user, course=course)

    def filter_visible(self, *, user, courses):
        courses = self._inner.filter_visible(user=user, courses=courses)
        if not user.is_authenticated:
            return courses.exclude(visibility=CourseVisibility.HIDDEN)
        return courses.annotate(_is_registered=registered_course_exists(user)).exclude(
            Q(visibility=CourseVisibility.HIDDEN) & Q(_is_registered=False)
        )

    def validate_course_config(self, raw, *, file_path=""):
        return self._inner.validate_course_config(raw, file_path=file_path)

    def get_dashboard_contributions(self, *, user):
        return self._inner.get_dashboard_contributions(user=user)
```

- The **coming-soon decision** (built inline in `get_access`, no private helper — single call
  site, matching `FreeOnlyCourseAccessBackend.get_access` and
  `ApplicationCourseAccessBackend.get_access`, which construct their decisions inline) is a
  `CourseAccessDecision` with `can_self_register=False`, `can_access_content=False`,
  `cta_label="I'm interested"`,
  `cta_url=reverse("course_interest:express_interest", kwargs={"course_slug": course.slug})`,
  and **explicitly set funnel copy** (`enrolment_summary`, `acquisition_heading`,
  `acquisition_subtext`) so the detail page does not inherit free/gated copy. Pick copy per
  `brand-guidelines` that does **not** promise a launch notification (spec §7.2, §10).
  Suggested: `enrolment_summary="Coming soon"`, heading `"Coming soon"`, subtext like
  `"Register your interest and it'll show up here when it opens."` (no "we'll email you").
  - `cta_url` is built via `reverse()` on the URL name only — **no import** of
    `course_interest`. This is the deliberate loose `course_access → course_interest`
    coupling (spec §5.1, flagged for `/plan_structure_review`).
- The **hidden-unregistered decision** (also inline) is the safe closed decision: all
  `False`/`None`, funnel copy blanked.
- `is_registered_for_course` is the existing shared helper (`student_management/utils.py:67-91`)
  — reuse, do not reimplement.
- `filter_visible` **drops hidden** (except registered) and **keeps coming_soon** in
  discovery. Uses `Exists()` (not joins) to avoid row duplication.

Then wrap at the loader (`loader.py:20-33`):

```python
@functools.cache
def get_course_access_backend() -> CourseAccessBackend:
    inner_class = import_string(settings.COURSE_ACCESS_BACKEND)
    return VisibilityEnforcingBackend(inner_class())
```

- The loader **only ever** returns the wrapper, so nothing can bypass it and **no existing
  call site or backend subclass changes**. `@functools.cache` is unchanged; tests overriding
  `COURSE_ACCESS_BACKEND` still call `get_course_access_backend.cache_clear()`.
- `validate_course_access_config` (`loader.py:36-50`) is unaffected — it already calls
  `get_course_access_backend().validate_course_config(...)`, which the wrapper forwards
  unchanged. The `visibility` field is **not** validated by `validate_course_config` nor the
  `freedom_ls_course_access.E001` system check (spec §5.4) — keep the two pipelines distinct.

**Tests (write first):** parametrised over **both** `FreeOnlyCourseAccessBackend` and
`ApplicationCourseAccessBackend` (override `COURSE_ACCESS_BACKEND`, `cache_clear` before/after):
- `coming_soon` → decision has `can_access_content=False`, `can_self_register=False`,
  `cta_label == "I'm interested"`, `cta_url` resolves to the express-interest URL, funnel copy
  is the coming-soon copy (not free/gated copy).
- `hidden` + unregistered → fully-closed decision; `hidden` + registered → delegates (Continue).
- `published` → delegates unchanged (free → Start, gated-unregistered → Apply now).
- `filter_visible`: hidden dropped for anon and unregistered authed users; hidden **kept** for
  a registered user; coming_soon kept for everyone; no duplicate rows when registered via cohort.
- loader: `get_course_access_backend()` returns a `VisibilityEnforcingBackend` instance.
- (Optional, closes criterion 2's "any future backend" clause mechanically) wrap a **minimal
  stub** `CourseAccessBackend` and assert the wrapper still enforces coming-soon/hidden over it
  — proving enforcement is structural, not per-backend.

## Phase 3 — Express-interest endpoints + shared CTA partial

### Task 3.1 — `express_interest` / `remove_interest` views + URLs

**Files:** `freedom_ls/course_interest/views.py`, `freedom_ls/course_interest/urls.py`,
`freedom_ls/course_interest/queries.py`. Wire `course_interest.urls` into the root URLconf
(`config/urls.py`) the same way `course_applications` is wired. The app's own patterns already
carry the `courses/<slug>/…` prefix, so include it at the **root** with an empty prefix —
`path("", include("freedom_ls.course_interest.urls"))` — giving final URLs
`/courses/<slug>/express-interest/` and `/courses/<slug>/remove-interest/`. `reverse(
"course_interest:express_interest", …)` (used by the wrapper) then resolves to that exact path.
**Skills:** `fls:htmx` (MUST — POST-only, 422, partial responses, HTMX detection),
`fls:multi-tenant` (site-scoped create), `fls:testing`.

- **View naming:** both views return an HTMX **partial** (the CTA in its two states), so per
  `fls:htmx` / `fls:template` the Python functions take the `partial_` prefix:
  `partial_express_interest` / `partial_remove_interest`. The **URL `name=` values stay**
  `express_interest` / `remove_interest` (the spec §7.1 and the wrapper's `reverse()` depend on
  them).
- `urls.py`: `app_name = "course_interest"`; two patterns:
  - `path("courses/<slug:course_slug>/express-interest/", views.partial_express_interest, name="express_interest")`
  - `path("courses/<slug:course_slug>/remove-interest/", views.partial_remove_interest, name="remove_interest")`
- `partial_express_interest` — `@login_required`, **POST/HTMX only**:
  1. `course = get_object_or_404(Course, slug=course_slug)`.
  2. **Guard:** if `course.visibility != CourseVisibility.COMING_SOON` → return **422**
     (no-op). This is the validation-error path.
  3. `CourseInterest.objects.get_or_create(user=request.user, course=course)` — idempotent,
     site auto-set by `SiteAwareModel`.
  4. Return the **CTA partial** rendered in the **interested** state (HTMX `outerHTML` swap).
- `partial_remove_interest` — `@login_required`, **POST/HTMX only**:
  1. `get_object_or_404(Course, …)`; delete the user's `CourseInterest` for it if present
     (no error if absent).
  2. Return the CTA partial in the **not-interested** state.
- Users can only create/remove **their own** interest (always filter on `request.user`).
- `queries.py`: add a batched lookup helper used by listings + detail (Task 4.x):

  ```python
  # Type the params via a TYPE_CHECKING guard (User, QuerySet[Course] | Sequence[Course]),
  # mirroring course_applications/queries.py. The pk type is UUID (SiteAwareModel).
  def get_interested_course_ids(
      user: RequestUser, courses: QuerySet[Course] | Sequence[Course]
  ) -> set[UUID]:
      """Course ids (from `courses`) the user has expressed interest in. One query."""
      if not user.is_authenticated:
          return set()
      return set(
          CourseInterest.objects.filter(
              user=user, course__in=courses
          ).values_list("course_id", flat=True)
      )
  ```

**Tests (write first):** with `client.force_login()` and `HTTP_HX_REQUEST="true"`:
- POST `express_interest` on a coming-soon course creates exactly one row; a second POST is a
  no-op (still one row); response renders the interested state.
- POST `express_interest` on a **published** course returns **422** and creates no row.
- POST `remove_interest` deletes the row; calling it when none exists is a no-op (not an error).
- GET on either endpoint is rejected (POST-only).
- Anonymous user is redirected through login.
