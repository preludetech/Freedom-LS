# Toast Notifications — Frontend QA Plan

These tests verify the bottom-anchored toast region behaves as the spec
describes — position, severity routing, auto-dismiss timing, manual dismiss,
HTMX delivery, stacking, animation, and pointer-events. Use the Playwright
MCP for the browser-driven tests; capture before/after screenshots at three
viewport widths.

## Prerequisites

- Dev server running:
  - `PORT=$(.claude/fls/scripts/find_available_port.sh)`
  - `uv run python manage.py runserver $PORT`
- Base URL: `http://127.0.0.1:$PORT`
- Use the **DemoDev** site for all test data
- A superuser account and a regular student account on DemoDev (use the
  `qa-data-helper` agent to create them if missing)
- A page that triggers a Django messages success on full-page load
  (e.g. profile save: `freedom_ls/accounts/views.py` already calls
  `messages.add_message(request, messages.SUCCESS, "Profile saved")`)
- A page that fires `messages.error(...)` from an HTMX endpoint. If none
  exists in the codebase, create a temporary view via the QA helper that
  calls `messages.error("Test error")` and returns an HTMX partial.

For each test below, note whether the toast appears in
`#toast-region-polite` (success / info / warning / debug) or in
`#toast-region-assertive` (error). Verify by inspecting the DOM in DevTools
or by Playwright `browser_snapshot`.

---

## Test 1: Layout — desktop position

**Goal:** Toasts render bottom-right on desktop; do not overlap any heading
or sticky chrome.

1. Resize the browser to 1440x900.
2. Navigate to `http://127.0.0.1:$PORT/accounts/login/` and log in as the
   student user.
3. Navigate to a page where you can save a profile (or any view that calls
   `messages.success`).
4. Submit the form to fire a success message and a full-page reload.
5. **Verify:** A toast appears in the bottom-right corner of the viewport.
6. **Verify:** The toast does not overlap the page heading, header bar, or
   sidebar.
7. **Verify:** The toast disappears within ~5 seconds.
8. Capture a screenshot and save it as
   `qa-artifacts/toast-desktop-1440.png`.

## Test 2: Layout — tablet position

1. Resize to 834x1194 (iPad portrait).
2. Repeat steps 3–4 from Test 1.
3. **Verify:** The toast appears at the bottom-right (still md+ breakpoint).
4. Capture screenshot `qa-artifacts/toast-tablet-834.png`.

## Test 3: Layout — mobile position

1. Resize to 390x844 (iPhone-class).
2. Repeat steps 3–4 from Test 1.
3. **Verify:** The toast spans the full viewport width minus a small margin,
   anchored to the bottom edge.
4. **Verify:** No horizontal scroll appears.
5. Capture screenshot `qa-artifacts/toast-mobile-390.png`.

## Test 4: Live regions present at first paint

1. Open the page **before** triggering any message.
2. Inspect the DOM.
3. **Verify:** `#toast-region-polite` exists and has `role="status"`,
   `aria-live="polite"`, no `aria-atomic` attribute on the region.
4. **Verify:** `#toast-region-assertive` exists and has `role="alert"`,
   `aria-live="assertive"`, no `aria-atomic` on the region.

## Test 5: Severity routing

1. Trigger a `messages.success(...)` toast.
2. **Verify:** The toast appears as a child of `#toast-region-polite`.
3. Trigger a `messages.error(...)` toast.
4. **Verify:** The toast appears as a child of `#toast-region-assertive`.
5. **Verify:** Each toast element carries `aria-atomic="true"`.

## Test 6: Auto-dismiss timing — success

1. Fire a success toast.
2. Start a stopwatch when it appears.
3. **Verify:** It dismisses ~5 seconds (4.5s–5.5s).

## Test 7: Auto-dismiss timing — warning

1. Fire a warning toast.
2. **Verify:** It dismisses ~7 seconds.

## Test 8: Auto-dismiss timing — error is persistent

1. Fire an error toast.
2. Wait 15 seconds.
3. **Verify:** The error toast is still visible.

## Test 9: Pause on hover

1. Fire a success toast (5s lifetime).
2. Hover the toast immediately.
3. Wait 10 seconds while hovering.
4. **Verify:** The toast is still visible.
5. Move the pointer away.
6. **Verify:** The toast dismisses ~5 seconds after the hover ended (it had
   the full duration remaining when hover started, so resumes from there).

## Test 10: Pause on focus

1. Fire a success toast.
2. Tab into the close button immediately.
3. Wait 10 seconds with focus on the close button.
4. **Verify:** Toast still visible.
5. Tab away.
6. **Verify:** Toast dismisses after the remaining time.

## Test 11: Pause on window blur

1. Fire a success toast.
2. Switch to another browser tab or app immediately.
3. Wait 10 seconds.
4. Return focus to the original tab.
5. **Verify:** Toast still present and resumes its timer.

## Test 12: Manual dismiss — close button

1. Fire any toast.
2. Click the close button.
3. **Verify:** Toast dismisses immediately and the timer is cleared (no
   ghost re-appearance).
4. **Verify:** The close button has `aria-label="Dismiss notification"`.

## Test 13: Manual dismiss — Esc key

1. Fire any toast.
2. Tab into the close button.
3. Press **Esc**.
4. **Verify:** That toast dismisses; other toasts (if any) remain.

## Test 14: HTMX delivery — success on partial response

1. Trigger an HTMX form submission that emits `messages.success(...)` and
   returns an HTMX partial response (no full-page reload).
2. **Verify:** No full page reload occurred (URL unchanged, no flash).
3. **Verify:** A toast appeared in `#toast-region-polite`.
4. **Verify:** The toast was announced once (only one OOB fragment in the
   response — inspect DevTools network tab).

## Test 15: HTMX delivery — error on 4xx/5xx response

1. Trigger an HTMX request that returns 4xx or 5xx with
   `messages.error(...)`.
2. **Verify:** A toast appears in `#toast-region-assertive`.
3. **Verify:** No duplicate toast announcements.

## Test 16: Full-page delivery — login redirect

1. Log out, then log in via a flow that emits a success message.
2. **Verify:** The toast renders on the next page after redirect.

## Test 17: Stacking — five concurrent

1. Trigger five concurrent success toasts (e.g. five rapid form submissions
   or a single view that emits five).
2. **Verify:** All five render and stack vertically with a gap between them.
3. **Verify:** No overlap.

## Test 18: Stacking — sixth non-error evicts the oldest

1. With five visible non-error toasts, trigger a sixth non-error toast.
2. **Verify:** The oldest non-error toast is dismissed; the new one appears.

## Test 19: Stacking — errors are persistent

1. Trigger five concurrent error toasts.
2. Trigger a sixth non-error toast.
3. **Verify:** The non-error toast is dropped (or evicted immediately) — all
   five errors remain visible.

## Test 20: Mid-stack dismiss animation

1. With three visible toasts, dismiss the middle one via its close button.
2. **Verify:** The toast below slides up smoothly to fill the gap (transform
   animation, no width/height jump).

## Test 21: Reduced motion

1. In OS settings, enable "Reduce motion" (or Chrome DevTools → Rendering →
   Emulate CSS media feature → `prefers-reduced-motion: reduce`).
2. Fire a toast.
3. **Verify:** Toast fades in/out without translate motion.

## Test 22: Pointer-events pass-through

1. Position the page so a clickable element (link or button) sits in a
   visual gap between two stacked toasts.
2. Click through the gap onto the underlying element.
3. **Verify:** The click reaches the underlying element. Only direct clicks
   on a toast are captured by the toast.

## Test 23: iOS safe-area (manual; optional)

1. Open the dev server on a notched iOS device (or via the iOS Simulator
   with `viewport-fit=cover` enabled in the meta tag).
2. Fire a toast.
3. **Verify:** The toast does not sit underneath the home indicator.

## Visual regression artifacts

Screenshots from Tests 1–3 (desktop, tablet, mobile) form the before/after
visual artifacts referenced in spec criterion 11. Capture them once before
any code changes (label `before-`) and once after (label `after-`). Attach
both sets to the QA report.

## Reporting

For each test, record one of:

- **PASS** — observed behaviour matches expectation.
- **FAIL** — describe what was observed and a screenshot or DOM snippet.
- **N/A** — could not be performed; explain.
