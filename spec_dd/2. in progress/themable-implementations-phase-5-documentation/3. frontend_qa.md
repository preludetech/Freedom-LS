# Frontend QA: Phase 5 — Documentation

## What we're verifying

1. A first-time consumer can follow the updated install guide (`docs/how tos/incorperate into another project.md`) and end up with a runnable downstream FLS install on the default theme.
2. A first-time theme author can follow the new theming guide (`docs/how tos/theme-fls.md`) end-to-end and ship a working **third theme** that is not `default` or `first_class` — exercising all three tiers — without referring to any other documentation.
3. Resource docs (`templates_and_cotton.md`, `frontend_styling.md`) accurately describe the new architecture.
4. Cspell passes on theme manifests and docs.
5. No stale token references remain in the repo.
6. Phases 1–4 are still green under both `default` and `first_class`.

This phase has no Playwright surface in the strict sense — it's a docs walkthrough. The "test" is **doing the thing the docs describe**.

We use the **DemoDev** site for the throwaway theme's verification, same as earlier phases.

## Test 1 — Theming guide walkthrough (the headline test)

Following **only** `docs/how tos/theme-fls.md`, build a third throwaway theme called `qa_check` that exercises all three tiers:

- **Tier 1**: token redeclares — pick distinct, garish-but-distinct values (e.g. lime green primary, hot pink secondary) so any leakage is immediately visible.
- **Tier 2**: at least one component-class re-open in `@layer components` (e.g. `.btn-primary { @apply rounded-none; }`).
- **Tier 3**: at least one cotton override (e.g. `templates/cotton/button.html` with a marker string like `<!-- qa_check override -->`).

Place the theme at `freedom_ls/themes/qa_check/` (or wherever the guide tells you).

Then:

```bash
FLS_THEME=qa_check npm run tailwind_build
FLS_THEME=qa_check uv run python manage.py runserver $PORT
```

Walk: `/`, login, course list, one course topic. Confirm:

- Tier-1 colours apply (lime/pink visible).
- Tier-2 component-class change applies (e.g. square `btn-primary`).
- Tier-3 cotton override applies (marker string visible in rendered HTML — view source or grep the response).

If any of these doesn't apply, **the bug is in the docs, not the architecture** (Phases 1–4 are already known green from pre-flight). Update the docs.

When done, **delete the throwaway theme**:

```bash
rm -rf freedom_ls/themes/qa_check
```

## Test 4 — Cspell + housekeeping

```bash
# Cspell — should pass without errors on all .md files touched.
npx cspell --no-progress --config cspell.json "docs/**/*.md" "freedom_ls/themes/**/*.md" "fls-claude-plugin/resources/**/*.md"

# Stale-token grep — should return no matches in any committed source/docs.
grep -rn -E "(text|bg|border|chip|btn)-(foreground|danger)\b" freedom_ls/ docs/ fls-claude-plugin/
grep -rn -E "(text|bg)-(primary|success|danger)-bold\b" freedom_ls/ docs/ fls-claude-plugin/
```

Expected:

- Cspell exit 0.
- Both greps return no matches. Any stragglers indicate Phase 1's rename didn't catch everything; fix in place.
