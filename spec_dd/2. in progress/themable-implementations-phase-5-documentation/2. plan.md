# Implementation Plan: Themable FLS — documentation (Phase 5 of 5)

This plan turns `1. spec.md` (in this folder) into shippable docs. The phase is documentation-only: install guide refresh, a new theming walkthrough, resource-doc updates, cspell additions, and a final housekeeping grep.

For the holistic five-phase context, see `../themable-implementations-master-decomposed-into-phases/2. plan.md`.

**Phases 1–4 are hard prerequisites.**

---

## Architectural principles inherited

1. **Themes are sparse.** Documentation must reflect this — the minimal theme is one `theme.css` file.
2. **Defaults stay in the main project.** Documentation must point readers at `tailwind.components.css` for component classes and at the owning app for templates — not at the default theme.
3. **This phase ends in a green build.** Cspell, tests, manage.py check, and Tailwind build all clean under both themes.

---

## Task 2 — Write new theming guide

**Create:** `docs/how tos/theme-fls.md`. Cover everything in spec §"Install guide update + new theming guide". Suggested sections:

1. The three tiers (tokens → component classes → template overrides) and when to reach for each.
2. Theme directory shape (sparse — only `theme.css` is required) and the `theme.md` manifest.
3. `FLS_THEME` / `FLS_THEMES_DIRS` resolution rules; failure modes.
4. Token contract reference (full list with default values, `on-*` pairs, shape, type).
5. Tier-2: re-opening component classes in `@layer components`. Note that defaults live in `tailwind.components.css` in the FLS main project.
6. Tier-3: cotton / partial / `_base.html` overrides; the `<c-vars>` contract. Note that defaults live in their owning app, NOT in a default theme.
7. Build pitfalls: `@source` and `.gitignore` (with the symlink/copy workaround); new utility classes that need to be in the `@source` glob.
8. Worked examples:
   - Minimal Tier-1-only theme (just `theme.css` with token redeclares).
   - Pointer to `freedom_ls/themes/first_class/` as the all-three-tiers reference.

Adhere to `fls:brand-guidelines` for tone and naming.
