# Plan: Themable FLS — documentation (Phase 5 of 5)

## Overview

This is a **documentation-only** phase. There are no Python, model, or template
*code* changes. The work is prose + markdown + a JSON word-list edit, plus two
stale `theme.md` manifest fixes, all validated against the architecture that
Phases 1–4 already shipped. Every claim a doc makes must match the code as it
exists today — the verification gates below exist to keep the docs honest.

### Ground truth confirmed in the current tree (do not re-derive — but do re-read before editing)

- **Theme resolver:** `freedom_ls/base/theming.py`.
  - `FLS_THEME` env var, default `"default"` (`config/settings_base.py`).
  - `FLS_THEMES_DIRS = [BASE_DIR / "themes", FREEDOM_LS_PACKAGE_DIR / "themes"]`
    — first match wins, so a downstream `themes/<slug>/` at `BASE_DIR` shadows
    the FLS package copy.
  - `resolve_theme_dir()` raises `ImproperlyConfigured` (naming the slug and the
    searched dirs) when the slug is absent — this is the documented failure mode.
  - `configure_theme()` prepends `<resolved>/templates/` to `TEMPLATES[0]["DIRS"]`
    (Tier 3) and `<resolved>/static/` to `STATICFILES_DIRS`.
- **Default theme:** `freedom_ls/themes/default/static/themes/default/theme.css`
  holds the full role-token contract. `--color-foreground` is **gone**;
  `--color-on-surface` exists. Manifest: `freedom_ls/themes/default/theme.md`
  (Tier-1 only — accurate today, verify in Task F).
- **first_class theme:** `freedom_ls/themes/first_class/static/themes/first_class/theme.css`
  ships **Tier 1 (tokens) + Tier 2 (`@layer components` overrides re-opening the
  default-contract classes)**. There is **no `templates/` dir → no Tier 3**.
  Its manifest `freedom_ls/themes/first_class/theme.md` is **stale** (claims
  "Tier 1 only … Tier 2 lands in Phase 3, Tier 3 in Phase 4") — fixed in Task F.
- **Component-class defaults:** `tailwind.components.css` (repo root) — the only
  home for `.btn`, `.chip`, `.alert`, `.course-card`, `.header`, `.surface`, etc.
- **Email colours:** `config/settings_base.py` parses the active theme's
  `theme.css` (`RESOLVED_THEME_DIR / "static" / "themes" / FLS_THEME / "theme.css"`)
  via `parse_tailwind_colors()` in `freedom_ls/accounts/email_utils.py`.
  `EMAIL_COLOR_FOREGROUND` now reads the `on-surface` token (the *Python* name is
  unchanged; only the looked-up key moved off the deleted `--color-foreground`).
- **Tailwind input:** `tailwind.input.css` (repo root) is FLS's own build file;
  the **downstream** template (with `<theme_slug>` placeholder) lives in master
  spec §"Tailwind build ownership". `@source` honours `.gitignore` — the pitfall.
- **Build pipeline / active-theme mechanism:** `npm run tailwind_build` is **two
  steps** (see `package.json`): `npm run _write_active_theme`
  (→ `uv run manage.py write_active_theme_css`) regenerates
  `tailwind.active_theme.css` — a one-line `@import` pointing at the
  `FLS_THEME`-resolved theme's `theme.css` — then `@tailwindcss/cli` compiles
  `tailwind.input.css` → `static/vendor/tailwind.output.css`. So inside the FLS
  repo, **switching the active theme = set `FLS_THEME`, then `npm run
  tailwind_build`** (the management command picks up the new slug and rewrites
  the generated import; the repo's `tailwind.input.css` imports that generated
  file last so the active theme wins the cascade). This is distinct from the
  *downstream* template, which hardcodes the active-theme `@import` for its own
  `<theme_slug>` instead of generating it. The docs must not conflate the two.
- **Stale-token grep is already clean** in the current tree (Phases 1–4 cleaned
  up). Task G re-runs it as a regression gate, not a clean-up.

### Task A — Refresh the install guide

**File:** `docs/how tos/incorperate into another project.md`

edit it" and never mentions the theme settings. Update it:

   the documented downstream `tailwind.input.css` template". Reproduce the
   template block from master spec §"Tailwind build ownership" verbatim, keeping

   ```css
   @import "tailwindcss";

   @source "<path-to-freedom_ls>/**/templates/**/*.html";
   @source "./themes/<theme_slug>/templates/**/*.html";

   @import "<path-to-freedom_ls>/themes/default/static/themes/default/theme.css";
   @import "<path-to-freedom_ls>/tailwind.components.css";
   @import "./themes/<theme_slug>/static/themes/<theme_slug>/theme.css";

   @theme { /* downstream project-level overrides, optional */ }
   ```

   Note the cascade order (default tokens → components → active theme so the
   active theme wins) and that `<path-to-freedom_ls>` must match the actual
   submodule/package location.

2. **Add** an `FLS_THEME` / `FLS_THEMES_DIRS` settings stanza with a

3. **Add** one sentence: "for theming beyond the default, see

4. **Add** a one-liner noting the `@source` / `.gitignore` pitfall and pointing

Keep the rest of the guide (uv/git-submodule/startproject steps, `npm run

### Task B — New end-to-end theming guide

**File (new):** `docs/how tos/theme-fls.md`

Write a complete walkthrough whose acceptance bar is: *a fresh reader follows it
and ships a working third theme without reading any Phase 1–4 commits.* Use the
`brand-guidelines` skill for tone/voice. Suggested section order (from spec §
"theme-fls.md"):

1. **The three tiers** (tokens → component classes → template overrides) and when
   to reach for each.
2. **Theme directory shape** — sparse; only `theme.css` is required. Show the
   canonical layout:

   ```
   themes/<slug>/
     theme.md                                  # manifest (recommended)
     static/themes/<slug>/theme.css            # Tier 1 — required
     static/themes/<slug>/...                   # optional Tier-2 @layer components
     templates/cotton/<component>.html          # optional Tier 3 overrides
     templates/<app>/...                        # optional Tier 3 overrides
   ```

   Document the `theme.md` manifest (name, version, which tiers it overrides,
   brand reference) — model it on `freedom_ls/themes/default/theme.md`.
3. **`FLS_THEME` / `FLS_THEMES_DIRS` resolution rules + failure modes.** Pull the
   facts from `freedom_ls/base/theming.py`: search order (`BASE_DIR/themes` then
   the FLS package `themes`, first match wins), the `ImproperlyConfigured` error
   on an unknown slug, and how `templates/` and `static/` get wired in. Also
   explain the **build-time half**: in the FLS repo, `FLS_THEME` drives a
   generated `tailwind.active_theme.css` via `npm run _write_active_theme`
   (`manage.py write_active_theme_css`), so switching themes is `set FLS_THEME` +
   `npm run tailwind_build`. Contrast with the downstream template (Task A), where
   the consumer hardcodes the active-theme `@import` for their own slug. Be
   explicit about which mechanism applies where so a reader doesn't mix them.
4. **Token contract reference** — the full role-token list with default values
   (source of truth: `freedom_ls/themes/default/static/themes/default/theme.css`).
   Include the `on-*` pairs and the rule that any coloured background needs its
   `text-on-X`. The contract (verify against the file before publishing):

   | Role | Token | Default | `on-` pair |
   |---|---|---|---|
   | primary | `--color-primary` | `#2B6CB0` | `--color-on-primary` `#FFFFFF` |
   | secondary | `--color-secondary` | `#475569` | `--color-on-secondary` `#FFFFFF` |
   | accent | `--color-accent` | `#F59E0B` | `--color-on-accent` `#1A2332` |
   | success | `--color-success` | `#38A169` | `--color-on-success` `#FFFFFF` |
   | warning | `--color-warning` | `#F6E05E` | `--color-on-warning` `#1A2332` |
   | error | `--color-error` | `#E8553D` | `--color-on-error` `#FFFFFF` |
   | info | `--color-info` | `#0EA5E9` | `--color-on-info` `#FFFFFF` |
   | surface | `--color-surface` | `#FFFFFF` | `--color-on-surface` `#1A2332` |
   | surface-2 | `--color-surface-2` | `#F3F4F6` | (uses `on-surface`) |
   | border | `--color-border` | `#D1D5DB` | — |
   | muted | `--color-muted` | `#4A5568` | — |
   | focus-ring | `--color-focus-ring` | `var(--color-primary)` | — |

   Also mention the status-`-light` tints (`--color-success-light` + its
   `on-` pair, and the warning/error/info equivalents), the hover-mix knobs
   (`--fls-hover-mix-color`, `--fls-hover-mix-amount`), shape tokens
   (`--fls-radius-*`, `--fls-card-*`), type tokens (`--fls-font-sans`,
   `--fls-font-display`, `--fls-font-mono`), the course-card accent palette
   (`--fls-course-accent-*` slots), and the header/sidepanel component-tier tokens
   (`--color-header`, `--color-on-header`, `--color-header-action`,
   `--color-on-header-action`, `--color-sidepanel`).

   **This table is a scaffold, not the contract.** Treat it as non-exhaustive:
   the complete contract is *everything* declared inside the `@theme {}` and
   `@theme inline {}` blocks of `theme.css` — read them at write time and
   reproduce them, rather than trusting this list for either values or
   completeness. Note in particular that `--color-focus-ring` is an `@theme
   inline` **alias** (to `--color-primary`), not a standalone token — call that
   out, since it changes how a theme overrides it.
5. **Tier 2 — re-opening component classes** in `@layer components`. State
   clearly that the *defaults* live in `tailwind.components.css` in the FLS main
   project, and a theme overrides by re-opening the same class so the cascade
   (set by `tailwind.input.css`: default tokens → components → active theme) lets
   it win. `first_class`'s `theme.css` `@layer components` block is the reference.
6. **Tier 3 — template overrides** (cotton / partial / `_base.html`); the
   `<c-vars>` contract. State that the *defaults* live in their **owning app**
   (e.g. `freedom_ls/base/templates/`), **not** in a default theme. Because no
   shipped theme uses Tier 3, demonstrate it with a **self-contained inline
   snippet** — e.g. shadowing a cotton component by dropping a same-named file in
   `themes/<slug>/templates/cotton/<name>.html`, or overriding a `_base.html`
   block. Do **not** point at `first_class` for Tier 3 (it has none).
7. **Build pitfalls:**
   - `@source` honours `.gitignore` — if FLS sits under an ignored path
     (`.venv/`, `node_modules/`), the glob silently skips its templates. Give the
     symlink/copy (or `.gitignore` adjustment) workaround.
   - New utility classes must fall inside an `@source` glob at build time or
     Tailwind won't generate them.
8. **Worked examples:**
   - A minimal **Tier-1-only** theme: just `theme.css` redeclaring a few tokens +
     a `theme.md`, point `FLS_THEME` at it, rebuild.
   - Pointer to `freedom_ls/themes/first_class/` as the **Tier-1 + Tier-2**
     reference (token redeclares + the `@layer components` block).

1. **Cspell additions.** Add the brand names to `cspell.json`'s word list so

2. **Stale-token regression greps** — must return **no matches** across
   `freedom_ls/`, `docs/`, and `fls-claude-plugin/`:
   - `(text|bg|border|chip|btn)-(foreground|danger)`
   - `(text|bg)-(primary|success|danger)-bold`

   Use these scoped patterns, **not** bare `-bold` / `-foreground` / `-danger`
   substrings — those legitimately match the `font-bold` utility, iconify
   `-bold` icon variants, and the email parser's test fixtures.
3. **Build / check / test — clean under BOTH themes.** For each of
   `FLS_THEME=first_class` and `FLS_THEME=default`:
   - `uv run pytest` (per `fls:testing`)
   - `uv run python manage.py check`
   - `npm run tailwind_build`

   Run `FLS_THEME=first_class` **first** and `FLS_THEME=default` **last**, so the
   generated `tailwind.active_theme.css` is left pointing at `default` and the
   working tree ends on the default theme. (Both build outputs —
   `tailwind.active_theme.css` and `static/vendor/tailwind.output.css` — are
   gitignored, so this is developer ergonomics, not a commit-safety issue.)
