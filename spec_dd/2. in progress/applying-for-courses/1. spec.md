## 1. Summary

Introduce an explicit, configurable **per-course access config** interpreted by a
**pluggable course-access backend** (modelled on Django's `EMAIL_BACKEND`). Core stores only
an opaque `access_config` blob and interprets nothing; the backend owns all interpretation.
The default backend recognises two values â€” `free` and `application_gated`. `free` is
today's behaviour made explicit. `application_gated` routes a learner into an **application**
that an educator reviews and approves; approval is decoupled from enrolment. In this work
the application is **form-less**: an **Apply** button creates a reviewable submission. The
authored multi-step **application form** (questions, file uploads, content-loaded config) is
a separate follow-up spec â€” `spec_dd/0. drafts/application-forms` â€” that slots in front of
this same submission with no rearchitecture.

Two new low-level concerns ship:

1. **`freedom_ls/course_access/`** â€” the backend interface, a settings-selected loader,
   and exactly one concrete `DefaultCourseAccessBackend` (the only reader of `access_config`).
   The seam is the deliverable: it must be broad enough that future subscription/purchase
   backends slot in as a new class plus a settings change â€” bringing their own config keys,
   vocabulary, and validation â€” never a rearchitecture of the student interface or a core
   migration.
2. **`freedom_ls/course_applications/`** â€” applicant **submissions** (DB-only, PII-bearing,
   state-machine-governed), created by a bare **Apply** action and reviewed/actioned through
   the **Django admin** in v1 (approve / reject / request-changes, with the full audit trail).
   The authored application **form** (config models, content-loading, per-question answers,
   file uploads, multi-step UX) is deferred to the `application-forms` follow-up spec, and the
   bespoke educator-facing review UI (global inbox + single-application screen in
   `educator_interface`) is deferred to the `application-review-ui` follow-up spec
   (`spec_dd/0. drafts/application-review-ui`); every model and seam both need is left in
   place here.

### Why both access types matter now

Free courses are the funnel that drives learners toward application-gated ("paid")
courses. The cross-promotion CTA itself is **future work**, but both access types must be
configurable now so that funnel can be built on top later. Building the backend seam now
(rather than a single `if access_type == ...`) is a deliberate, explicitly-requested
exception to the project's "don't build unrequested abstractions" convention, because the
mechanism is known to need to grow to subscriptions, per-course purchase, and feature
unlocks.

## 2. Goals & success criteria

A change is done when all of the following hold:

1. **Access config is a property of a course.** `Course.access_config` exists (opaque JSON,
   default `{}`), is set via content-loading, and existing courses behave exactly as today
   after migration (back-filled to `{}` â†’ default backend reads as `free`, no data migration
   needed).
2. **Every access question routes through the backend.** No view, template, or utility reads
   or branches on `course.access_config`; all access decisions and CTAs come from
   `get_access(*, user, course)`. Only the active backend interprets `access_config`; core
   branches on nothing.
3. **The gate is enforced server-side.** The single self-registration chokepoint
   (`student_interface.register_for_course`, `views.py:234`) refuses to self-register a
   learner for an `application_gated` course and routes them to the application flow. Admin
   and cohort enrolment deliberately bypass the gate.
4. **Course listings route through `filter_visible()`.** `get_all_courses()`
   (`utils.py:443`) and its dashboard/all-courses call sites pass through the backend; the
   default backend returns the queryset unchanged.
5. **A learner can apply to a gated course.** The gated-course CTA "Apply now" routes to an
   application-entry view that creates a `CourseApplication` for that learner+course and
   submits it for review (no form to fill in this work). Applying again is governed by the
   active-application uniqueness rule.
6. **A reviewer can review submitted applications** from the Django admin â€” a filterable,
   searchable changelist of applications, each showing its applicant and full history â€” and
   request changes (with a required applicant-facing message), approve, or reject; every
   transition is recorded for audit. (The bespoke educator-facing inbox / review screen is the
   `application-review-ui` follow-up spec.)
7. **Approval is not enrolment.** Approving exposes a convenience "enrol this learner"
   hand-off into the existing cohort/registration flow but never auto-enrols.
8. **Multi-tenancy holds.** All new models inherit `SiteAwareModel`; reviewers only ever
   see their own site's applications and configs. Reviewer access is object-level via
   `django-guardian`.
9. **A future backend is a class + settings change.** Adding a subscription,
   per-course-purchase, prerequisite, or intake-window backend requires no change to
   templates, the chokepoint, or the listing call sites â€” only a new backend class + the
   `COURSE_ACCESS_BACKEND` setting.

## 3. Scope

### In scope

- `Course.access_config` opaque JSONField (default `{}`), loaded via content-loading; core
  interprets nothing.
- `freedom_ls/course_access/`: `CourseAccessBackend` base class (incl.
  `validate_course_config()`), `CourseAccessDecision` + `CourseAccessCTA` (the open
  display+behavior contract â€” no fixed status enum), the default backend's private
  `CourseAccessType`, `get_course_access_backend()` loader, `COURSE_ACCESS_BACKEND` setting,
  a Django system check for config validity, and one `DefaultCourseAccessBackend`.

- `django-fsm-2` added as a dependency.

### Deferred to the `application-review-ui` follow-up spec (`spec_dd/0. drafts/application-review-ui`)

### Out of scope (deliberately deferred â€” do not architect away)

On-platform payment; the freeâ†’application funnel CTA; further concrete access backends

`CourseApplication` + its state machine are the seam the `application-forms` spec hangs
answers, files, and config off; every transition fires a signal so an email backend can be
added later; `RecommendedCourse` remains the funnel CTA's future home.

## 4. Part A â€” Course access type & pluggable backend

### 4.1 `Course.access_config` â€” opaque, backend-owned

Core stores **opaque per-course access configuration and interprets none of it**. Store the
per-course access configuration as a single JSON blob (there is no typed `access_type` field
on `Course` today â€” this is the first access-config field); the active backend owns the
schema, vocabulary, and validation for its own slice.

Add to `content_engine/models.py` (after `category`, line 154):

```python
access_config = models.JSONField(
    default=dict,
    blank=True,
    help_text=_(
        "Opaque per-course access configuration. Interpreted ONLY by the active "
        "course-access backend (settings.COURSE_ACCESS_BACKEND); core never reads or "
        "branches on its contents. The default backend stores {'access_type': ...}."
    ),
)
```

- `CourseAccessType` is **not a core enum** â€” it lives only in the default backend
  (`course_access`, Â§4.3) as that backend's private vocabulary. `content_engine` does not
  import or know it.
- `JSONField(default=dict)` â†’ Django back-fills existing rows to `{}`; **no `RunPython` data
  migration**. A backend reads missing keys as its own default (the default backend treats
  absent `access_type` as `free`), so existing courses behave exactly as today.
- Because `Course` is site-aware, config is implicitly per-site: the same slug can carry
  different `access_config` on different sites.
- **Rule:** `access_config` is backend-private. No view, template, or utility may read or
  branch on its contents â€” only the active backend, via `get_access()` / `filter_visible()`.
  A future subscription backend stores entirely different keys (e.g. `{'plan': 'pro',
  'price_id': ...}`) with **no change to this field, the loader, or any migration**. Enforced
  by review, not the type system â€” call it out explicitly in code comments at the field and
  in the backend.

### 4.2 Setting & validating `access_config` via content-loading

Per the idea and the locked decision, access config is authored through the content-loading
pipeline only (no admin/educator toggle in v1).

- Add `access_config: dict[str, Any] | None = Field(None, ...)` to the `Course` Pydantic
  model in `content_engine/schema.py` (line 96). The schema layer treats it as an **opaque
  dict** â€” it does not know backend-specific keys. `save_with_uuid`
  (`content_engine/management/commands/content_save.py:224`) maps it to the model field
  automatically via the field-existence gate (lines 267â€“279); the
  Pydantic field and model field share the name `access_config`, so no other change is needed.
- Frontmatter:

  ```yaml
  access_config:
    access_type: application_gated
  ```

  Absent â†’ `{}` â†’ default backend behaves as `free`.
- **Backend-owned validation, implemented once, called at both ends** (per the locked
  decision): the backend exposes a single
  `validate_course_config(raw: dict, *, file_path: str = "") -> dict` that validates and
  normalizes its slice, raising `ValueError` on unknown keys/values (with `file_path`
  context, consistent with existing content validation). It is called from:
  - **load time** â€” `save_course()` (`content_engine/management/commands/content_save.py:342`)
    calls
    `get_course_access_backend().validate_course_config(item.access_config or {}, file_path=...)`
    and stores the normalized result â†’ bad config is rejected at author time (fail loud).
  - **read / ops time** â€” a Django **system check** (`@register` in `course_access`) runs the
    same validator over every `Course.access_config` for the active backend, so swapping
    `COURSE_ACCESS_BACKEND` surfaces now-invalid existing config via `manage.py check` rather
    than at request time. (DB-querying system checks are an established pattern here â€” see
    `accounts/checks.py`, which queries `Site`/`SiteSignupPolicy`; follow it and tolerate the
    table not yet existing on a fresh DB.)
  - `get_access()` also runs the validator defensively and, on failure, returns a safe
    no-action decision (`cta=None`, both enforcement gates `False`; never 500s a learner's
    page); the load step and system check are the proactive surfaces.

  One validation implementation, reused everywhere.

### 4.3 `freedom_ls/course_access/` â€” the backend

New app, `label = "freedom_ls_course_access"`, added to `INSTALLED_APPS`. Follows the
existing pluggable-backend precedent in `freedom_ls/icons/backend.py` (`import_string` +
`@functools.cache` + base class + `DefaultXxxBackend`).

**Default backend's private vocabulary** â€” `CourseAccessType` lives here, in `course_access`,
**not** in `content_engine` (core knows nothing of it):

```python
class CourseAccessType(models.TextChoices):
    FREE = "free", _("Free")
    APPLICATION_GATED = "application_gated", _("Application gated")
```

**Call-to-action + decision objects** â€” the backend fully specifies what the learner sees
(`label`) and what it does (`url`); there is **no enumerated set of access situations**. New
cases (prerequisites unmet, intake window not open, purchase/subscription required, â€¦) are
just a different CTA the backend returns â€” no shared enum, no template change.

```python
@dataclass(frozen=True)
class CourseAccessCTA:
    label: str        # "Start", "Enroll", "Apply now", "Continue", "Subscribe", â€¦
    url: str | None   # where the affordance points / what it triggers; None = not actionable

@dataclass(frozen=True)
class CourseAccessDecision:
    cta: CourseAccessCTA | None   # the affordance to show; None = show nothing
    can_self_register: bool       # enforcement: may the self-register chokepoint register now?
    can_access_content: bool      # enforcement: may the user view course content now?
```

- `cta` is the **only** display/behavior contract; templates render it through the existing
  `<c-button>` (already does `<a href>` vs disabled `<button>`). The two booleans are
  enforcement gates, not display, and collapse every situation into the only two yes/no
  questions the system asks. Richer display fields (variant, help_text, icon) are added to
  `CourseAccessCTA` only when a concrete backend needs them â€” not pre-assumed now.

**Base class** (`course_access/backends.py`):

```python
class CourseAccessBackend:
    def get_access(self, *, user, course) -> CourseAccessDecision: ...
    def filter_visible(self, *, user, courses): ...   # -> QuerySet[Course]
    def validate_course_config(self, raw: dict, *, file_path: str = "") -> dict:
        """Validate & normalize this backend's slice of Course.access_config.
        Single implementation; called at content-load (fail loud), by a Django
        system check (after a backend swap), and defensively in get_access().
        Raise ValueError on invalid config."""
        ...
    # has_feature(*, user, feature) -> bool is intentionally NOT defined in v1;
    # it is the designated additive home for feature-unlock backends. Documented, not built.
```

**Loader** (`course_access/__init__.py` or `loader.py`):

```python
@functools.cache
def get_course_access_backend() -> CourseAccessBackend:
    path = settings.COURSE_ACCESS_BACKEND
    return import_string(path)()
```

**Settings** (`config/settings_base.py`, near `TASKS`):

```python
COURSE_ACCESS_BACKEND = "freedom_ls.course_access.backends.DefaultCourseAccessBackend"
```

**`DefaultCourseAccessBackend`** â€” the only concrete backend in v1, the only reader of
`access_config`:

- `validate_course_config`: accept only the key `access_type`; value âˆˆ `CourseAccessType`;
  default to `free` when absent; reject unknown keys/values with `file_path` context; return
  `{"access_type": <value>}`.
- `get_access`: resolve
  `access_type = self.validate_course_config(course.access_config)["access_type"]` (on
  `ValueError` â†’ safe no-action decision: `cta=None`, `can_self_register=False`,
  `can_access_content=False`). Then:
  - already registered (direct `UserCourseRegistration` or via `CohortCourseRegistration`,
    both in `student_management.models`) â†’ `cta=CTA("Continue", course_home url)`,
    `can_self_register=False`, `can_access_content=True`. Query those registration models
    **directly** (the same trivial check `student_interface.get_is_registered()`,
    `utils.py:149`, performs) â€” do **not** import that helper: `student_interface` already
    depends on `course_access` (Â§4.4/Â§8), so importing it back would create a cycle. If the
    shared check is worth extracting, it belongs in `student_management`, not
    `student_interface`.
  - else `access_type == FREE` â†’ `cta=CTA("Start", register_for_course url)`,
    `can_self_register=True`, `can_access_content=False`.
  - else `access_type == APPLICATION_GATED` â†’ `cta=CTA("Apply now", <application confirmation
    page url>)` â€” that url is a **GET** the chokepoint can redirect to (Â§5.3) â€”
    `can_self_register=False`, `can_access_content=False`.
  - **Never inspects `course_applications` records** â€” it distinguishes the gated CTA from the
    registered one purely from registration state. This keeps `course_access` from depending
    on `course_applications` (no dependency cycle; the application app stays a leaf).
- `filter_visible`: returns `courses` unchanged (today's behaviour).

### 4.4 Wiring the backend into the student interface

- **Chokepoint gate** â€” `student_interface/views.py:234` `register_for_course`: before
  `UserCourseRegistration.objects.get_or_create(...)` (`views.py:240`), call

  gated courses; fall back to the course preview when `cta`/`cta.url` is `None`) instead of
  registering. This is the **single** server-side enforcement point; admin/cohort
  registration paths are untouched and continue to bypass the gate.
- **CTA branching** â€” `_preview_start_url()` (`views.py:70`) and `_annotate_preview_context()`
  (`views.py:94`) attach the whole `decision.cta` to the course instead of always emitting the
  `register_for_course` URL + a hardcoded "Start". Templates
  `partials/course_preview_content.html` (the `<c-button>` Start affordance, ~line 37) and
  `partials/course_card_not_registered.html` (the `course.preview_start_url` CTA â€” note there

  1 vs course_home" refinement for already-registered learners stays in `student_interface`

- **Direct course-URL access** â€” `course_home` / `course_preview` may additionally consult

  registration state or config; a subscription backend can grant content access without
  enrolment. (Preview remains visible because it is the funnel surface.)

## 5. Part B â€” `course_applications`

New app, `label = "freedom_ls_course_applications"`, added to `INSTALLED_APPS`. It is a
**leaf** app: it may read `content_engine.Course` and `student_management` enrolment
primitives, but `course_access` must **not** depend on it.

In this work the app is **submissions only**: a learner clicks **Apply**, a
`CourseApplication` is created and submitted for review. There is **no form** â€” no authored
config, no per-question answers, no uploads. That authored **application form** is the
`application-forms` follow-up spec (`spec_dd/0. drafts/application-forms`); the models and
seams it slots into are all defined here.

### 5.1 Submission models (DB-only, PII-bearing)

```
CourseApplication              # one applicant attempt for one course
  user        FK User
  course      FK Course
  state       FSMField(default="draft", protected=True)   # django-fsm-2
  created_at, updated_at, submitted_at, decided_at
  decided_by  FK User | None
  # Partial unique index: at most one ACTIVE application per (site, user, course),
  # where active = state in (draft, submitted, under_review, changes_requested).
  # Rejected/withdrawn/approved do not block re-application.
  # NOTE (forms spec): gains a `config FK ApplicationConfig` pinning the form version
  # at apply time, plus `answers` / file children. Left off here â€” there is no form yet.

ApplicationNote                # staff-only internal note (structurally distinct from applicant message)
  application FK, author FK User, body Text, created_at

ApplicationStateTransition     # audit log; one row per transition
  application FK (related_name="transitions")
  from_state, to_state, actor FK User | None
  message       TextField   # applicant-visible message (e.g. the request_changes reason)
  created_at
```

All inherit `SiteAwareModel`. The answer rows (`ApplicationAnswer`), the document row
(`ApplicationFile`, private storage + `scan_status` seam), and the `config` FK are added by
the `application-forms` spec â€” they are the form's data, and there is no form here. Keeping
`CourseApplication` standalone now means that follow-up is purely additive.

### 5.2 State machine (`django-fsm-2`)

Add the dependency: `uv add django-fsm-2` (confirmed absent today). `state` is an
`FSMField(protected=True)`. The full transition set ships now (it is the review workflow,
independent of whether a form exists), per `research_review_workflow.md Â§1.3`:

| Transition | Actor | source â†’ target | Notes |
|---|---|---|---|
| `submit` | applicant | draft â†’ submitted | no answer validation in this work; the `application-forms` spec adds required-answer validation here |
| `withdraw` | applicant | [draft, submitted, changes_requested] â†’ withdrawn | |
| `pick_up` | reviewer | submitted â†’ under_review | |
| `request_changes` | reviewer | under_review â†’ changes_requested | **requires** an applicant-visible message |
| `resubmit` | applicant | changes_requested â†’ submitted | re-submits (gains answer re-validation with the form) |
| `approve` | reviewer | under_review â†’ approved | terminal; optional internal note |
| `reject` | reviewer | under_review â†’ rejected | terminal; optional internal note |

- Reviewer transitions use `permission=` callables delegating to
  `user.has_perm("change_application", instance)` (django-guardian, object-level).
- Each transition writes an `ApplicationStateTransition` row and fires an
  `application_state_changed` Django signal (the seam for a future email backend; no
  notifications in v1).
- Templates drive buttons off `get_available_user_state_transitions(user)` â€” never hardcode
  stateâ†’button maps.
- `expired`/`waitlisted` and multi-reviewer are out of v1 (do not architect away â€” the
  audit row + signal already accommodate them).
- `request_changes` / `resubmit` ship now as part of the workflow even though, with no form,
  there is nothing for the applicant to edit between them; they become fully meaningful once
  the `application-forms` spec lands.

### 5.3 Apply entry (form-less)

- **Entry:** the gated-course CTA "Apply now" (from the access backend, Â§4.3) points at a
  **GET** confirmation page at `/applications/apply/<course_slug>/` â€” a redirect target the
  chokepoint (Â§4.4) can send the learner to. Confirming **POSTs** to the same route, which
  creates the learner's active `CourseApplication` for that course (or loads an existing active
  one) and calls `submit` so it lands in the reviewer inbox immediately. There is no form to
  render.
- **Confirmation:** the confirmation page ("Apply to <course>?") makes applying a deliberate
  action; on a successful POST it redirects to the applicant status page (Â§5.4).
- **Re-apply:** the active-application partial unique index (Â§5.1) means a learner with an
  in-flight application is taken to its status page instead of creating a duplicate; a
  previously rejected/withdrawn learner can apply again.
- **Seam:** the `application-forms` spec replaces this view's body â€” instead of
  create-and-submit, it resolves the course's config, creates a `draft`, and redirects into
  the multi-step form â€” without changing the CTA, the chokepoint, or the models below it.

### 5.4 Applicant status page

Plain-language current state; when `changes_requested`, show the reviewer's message (the
multi-step edit/resubmit affordance arrives with the `application-forms` spec); when
`approved`/`rejected`, show the decision and any applicant-visible message. A `withdraw`
action is available while the application is active. In-flight applications are reachable
from the learner's dashboard.

## 6. Part C â€” Application review in the Django admin (v1)

In v1, reviewers action applications through the **Django admin**, to keep the first cut
simple. The bespoke educator-facing review UI (global inbox + full-screen single-application
screen in `educator_interface`) is deferred to the `application-review-ui` follow-up spec
(`spec_dd/0. drafts/application-review-ui`); everything below is UI-agnostic and that spec
reuses this machinery unchanged.

**Tradeoff (explicit):** reviewers must therefore be Django **staff** users in v1, not just
educators. Access is still constrained per-object (`django-guardian`) and per-site
(`SiteAwareModel`), so a staff reviewer only sees applications they hold `view_application` on,
on their own site. Lifting review out of the admin so non-staff educators / programme managers
can review is the entire point of the `application-review-ui` spec.

Admin classes live in `course_applications/admin.py` â€” the app owns its own admin, so review
introduces **no** `educator_interface â†’ course_applications` dependency (Â§8). Per the

## 9. Key decisions & rationale

| Decision | Choice | Why |
|---|---|---|
| Per-course access config | opaque `access_config` JSONField on `Course`; backend owns schema/vocabulary/validation | core interprets nothing; a new backend stores its own keys with no core migration; satisfies "class + settings change" |
| Build the backend now? | yes â€” interface + loader + one backend | explicitly requested; known to need to grow to subscriptions/purchase; seam is the deliverable |
| Who reads `access_config` | only the active backend | default backend reads `access_type`; a subscription backend ignores it entirely; keeps it backend-private config |
| Config validation | one `validate_course_config()`, called at load + system check + read | implement once, call as needed; a backend swap can't leave silently-invalid config |
| Display + behavior contract | backend-supplied `CourseAccessCTA` (label + url) + two enforcement booleans; no status enum | the backend fully specifies what the learner sees and what it does; new situations are new CTAs, needing no enumerated cases or caller changes |
| Separate `course_applications` app | yes | applications are PII submissions (and, later, admin-shaped form config); reusing `Form` tables would muddy content invariants, scoring, and PII isolation |
| Form **in front of** apply | deferred to `application-forms` spec | **user decision** â€” ship the access backend + bare Apply/review flow first; add the authored form later with no rearchitecture |
| State machine | `django-fsm-2`, `protected=True` | matches the subscriptions spec; greenfield; transition perms integrate with guardian; full review workflow is independent of whether a form exists |
| Audit | custom `ApplicationStateTransition` | lighter than `django-simple-history`; sufficient for who/when/what |
| Review UI home (v1) | **Django admin** (`course_applications/admin.py`) | **user decision** â€” keep v1 simple; the FSM, guardian scoping, notes, and audit log are all UI-agnostic, so the admin gives a filterable inbox + per-application actions for free, and keeps `course_applications` a leaf (no `educator_interface` edge). Cost: reviewers must be `is_staff`. The bespoke educator UI (correct audience, panel framework, non-staff reviewers) is deferred to `application-review-ui` |
| Approval vs enrolment | decoupled, with convenience hand-off | off-platform steps may intervene; matches serious admissions systems |

## 10. Risks & mitigations

- **Direct-comparison drift** (someone reads/branches on `course.access_config`, or hardcodes
  per-situation CTA logic outside the backend) â€” mitigate with explicit comments at the field
  + backend and a review check; callers use `decision.can_self_register` /
  `decision.can_access_content` / `decision.cta`, never the raw config. It is the one rule
  that keeps the future refactor cheap. The Django system check additionally catches
  `access_config` invalidated by a backend swap.
- **Loader cross-app edge** â€” the remaining edge in this spec is load-time `access_config`
  validation (`content_engine â†’ course_access`); see Â§8 for its fallback.
- **PII handling** â€” in this work an application carries only who-applied-to-what; document
  uploads, private storage, and permission-checked serving arrive with the `application-forms`
  spec, where the threat model (`/threat-model`) and security reviews should scrutinise the
  serve view and upload validation specifically.
- **Concurrent reviewer actions** â€” `FSMField` guards serialise transitions cleanly
  (second action fails on a now-invalid source state).
