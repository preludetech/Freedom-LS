# Spec: Course Access Types — Free & Application-Gated

Status: draft for review
Source idea: `0. idea.md`
Research: `research_course_access_types.md`, `research_form_schema.md`,
`research_multistep_ux.md`, `research_review_workflow.md`,
`research_reference_implementations.md`, `.sdd-work/spec_research_codebase.md`

---

## 1. Summary

Introduce an explicit, configurable **per-course access type** with two first-class
values — `free` and `application_gated` — behind a **pluggable course-access backend**
(modelled on Django's `EMAIL_BACKEND`). `free` is today's behaviour made explicit.
`application_gated` routes a learner into a **per-course, multi-step application** that an
educator reviews and approves; approval is decoupled from enrolment.

Two new low-level concerns ship:

1. **`freedom_ls/course_access/`** — the backend interface, a settings-selected loader,
   and exactly one concrete `DefaultCourseAccessBackend`. The seam is the deliverable: it
   must be broad enough that future subscription/purchase backends slot in as a new class
   plus a settings change, never a rearchitecture of the student interface.
2. **`freedom_ls/course_applications/`** — per-course application **config** (authored as
   content, loaded from yaml/markdown like every other content type) and applicant
   **submissions** (DB-only, PII-bearing, state-machine-governed), plus a review UI surfaced
   through `educator_interface`.

### Why both access types matter now

Free courses are the funnel that drives learners toward application-gated ("paid")
courses. The cross-promotion CTA itself is **future work**, but both access types must be
configurable now so that funnel can be built on top later. Building the backend seam now
(rather than a single `if access_type == ...`) is a deliberate, explicitly-requested
exception to the project's "don't build unrequested abstractions" convention, because the
mechanism is known to need to grow to subscriptions, per-course purchase, and feature
unlocks.

---

## 2. Goals & success criteria

A change is done when all of the following hold:

1. **Access type is a property of a course.** `Course.access_type` exists, defaults to
   `free`, is set via content-loading, and existing courses behave exactly as today after
   migration (all back-filled to `free`, no data migration needed).
2. **Every access question routes through the backend.** No view, template, or utility
   compares `course.access_type` directly; all access decisions and CTAs come from
   `get_access(*, user, course)`. Only `DefaultCourseAccessBackend` reads `access_type`.
3. **The gate is enforced server-side.** The single self-registration chokepoint
   (`student_interface.register_for_course`, `views.py:246`) refuses to self-register a
   learner for an `application_gated` course and routes them to the application flow. Admin
   and cohort enrolment deliberately bypass the gate.
4. **Course listings route through `filter_visible()`.** `get_all_courses()`
   (`utils.py:427`) and its dashboard/all-courses call sites pass through the backend; the
   default backend returns the queryset unchanged.
5. **A learner can complete a per-course multi-step application**, save a draft that
   survives logout and device switch, move back and forth to correct answers, upload
   documents, review a GOV.UK-style "check your answers" summary, and submit.
6. **An educator can review submitted applications** from a global, filterable inbox, see
   all answers and documents together, request changes (with a required applicant-facing
   message), approve, or reject — every transition recorded for audit.
7. **Approval is not enrolment.** Approving exposes a convenience "enrol this learner"
   hand-off into the existing cohort/registration flow but never auto-enrols.
8. **Multi-tenancy holds.** All new models inherit `SiteAwareModel`; reviewers only ever
   see their own site's applications and configs. Reviewer access is object-level via
   `django-guardian`.
9. **A future backend is a class + settings change.** Adding a `REQUIRES_PURCHASE` or
   `REQUIRES_SUBSCRIPTION` backend requires no change to templates, the chokepoint, or the
   listing call sites.

---

## 3. Scope

### In scope

- `Course.access_type` field + `CourseAccessType` choices, loaded via content-loading.
- `freedom_ls/course_access/`: `CourseAccessBackend` base class, `CourseAccessDecision`,
  `CourseAccessStatus` enum (full set), `get_course_access_backend()` loader,
  `COURSE_ACCESS_BACKEND` setting, one `DefaultCourseAccessBackend`.
- `freedom_ls/course_applications/`: content-loaded config models; DB-only submission
  models with a `django-fsm-2` state machine; per-question file uploads with private
  storage + permission-checked serving; multi-step applicant UX; applicant status page.
- Review UI in `educator_interface`: global inbox + single-application review screen +
  internal notes + transition log + "enrol approved learner" hand-off.
- Wiring: chokepoint gate, CTA branching, `filter_visible()` at listing call sites.
- `django-fsm-2` added as a dependency.

### Out of scope (deliberately deferred — do not architect away)

On-platform payment; the free→application funnel CTA; further concrete access backends
(subscriptions, per-course purchase, feature unlocks); `has_feature()` call sites;
waitlists; application/intake windows and auto-expiry; multi-reviewer assignment and
rubric/score review; bulk approve/reject; conditional/branching questions; email
notifications; in-app identity verification and virus scanning; an in-UI toggle to flip a
course's `access_type` (model supports it; only the UI is deferred).

Each deferred item must leave its seam: `CourseAccessStatus` defines all foreseeable
statuses now; the backend base class is the designated home for `has_feature()`;
`ApplicationFile` carries a `scan_status`; every transition fires a signal so an email
backend can be added later; `RecommendedCourse` remains the funnel CTA's future home.

## 4. Part A — Course access type & pluggable backend

### 4.1 `Course.access_type`

Add to `content_engine/models.py` (after `category`, line 154), mirroring the existing
`QuestionType`/`FormStrategy` `TextChoices` pattern:

```python
class CourseAccessType(models.TextChoices):
    FREE = "free", _("Free")
    APPLICATION_GATED = "application_gated", _("Application gated")
```

```python
access_type = models.CharField(
    max_length=30,
    choices=CourseAccessType.choices,
    default=CourseAccessType.FREE,
    help_text=_("Private config for the default course-access backend."),
)
```

- A non-nullable `CharField` with a hardcoded default → Django back-fills existing rows to
  `"free"` automatically; **no `RunPython` data migration**.
- Because `Course` is site-aware, the field is implicitly per-site: the same slug can be
  `free` on one site and `application_gated` on another.
- **Rule:** `access_type` is the default backend's private config, **not core truth**. Only
  `DefaultCourseAccessBackend` may read it. This is enforced by review, not the type system
  — call it out explicitly in code comments at the field and in the backend.

### 4.2 Setting `access_type` via content-loading

Per the idea and the locked decision, `access_type` is configured through the
content-loading pipeline only (no admin/educator toggle in v1).

- Add `access_type: str | None = Field(None, ...)` to the `Course` Pydantic model in
  `content_engine/schema.py` (line 96). `save_with_uuid` (`content_save.py:246`) maps it to
  the model field automatically once it exists on both sides; its field-existence gate
  (lines 267–279) means no other change is needed.
- Frontmatter key: `access_type: application_gated`. Absent → model default (`free`).
- Validate the value against `CourseAccessType` during load; reject unknown values with a
  clear error (consistent with existing content validation).

### 4.3 `freedom_ls/course_access/` — the backend

New app, `label = "freedom_ls_course_access"`, added to `INSTALLED_APPS`. Follows the
existing pluggable-backend precedent in `freedom_ls/icons/backend.py` (`import_string` +
`@functools.cache` + base class + `DefaultXxxBackend`).

**Status enum (full set defined now; default backend returns only the first three):**

```python
class CourseAccessStatus(models.TextChoices):
    REGISTERED            = "registered"
    OPEN                  = "open"               # may self-register
    REQUIRES_APPLICATION  = "requires_application"
    REQUIRES_PURCHASE     = "requires_purchase"
    REQUIRES_SUBSCRIPTION = "requires_subscription"
    LOCKED                = "locked"
```

**Decision object** — carries the status and a derived CTA so templates never branch on
type:

```python
@dataclass(frozen=True)
class CourseAccessDecision:
    status: CourseAccessStatus
    can_self_register: bool
    cta_label: str        # e.g. "Start", "Apply", "Continue"
    cta_url: str          # resolved URL the CTA points at
```

**Base class** (`course_access/backends.py`):

```python
class CourseAccessBackend:
    def get_access(self, *, user, course) -> CourseAccessDecision: ...
    def filter_visible(self, *, user, courses): ...   # -> QuerySet[Course]
    # has_feature(*, user, feature) -> bool is intentionally NOT defined in v1;
    # it is the designated additive home for feature-unlock backends. Documented, not built.
```

**Loader** (`course_access/__init__.py` or `loader.py`):

```python
@functools.cache
def get_course_access_backend() -> CourseAccessBackend:
    path = settings.COURSE_ACCESS_BACKEND
    return import_string(path)()
```

**Settings** (`config/settings_base.py`, near `TASKS`):

```python
COURSE_ACCESS_BACKEND = "freedom_ls.course_access.backends.DefaultCourseAccessBackend"
```

**`DefaultCourseAccessBackend`** — the only concrete backend in v1:

- `get_access`:
  - already registered (direct `UserCourseRegistration` or via `CohortCourseRegistration`,
    reusing `get_is_registered()` logic, `utils.py:133`) → `REGISTERED`, CTA "Continue" →
    course home / next item.
  - else `access_type == FREE` → `OPEN`, `can_self_register=True`, CTA "Start" →
    `register_for_course`.
  - else `access_type == APPLICATION_GATED` → `REQUIRES_APPLICATION`,
    `can_self_register=False`, CTA "Apply" → the application entry URL.
  - **Never inspects `course_applications` records** — it distinguishes
    `REQUIRES_APPLICATION` from `REGISTERED` purely from registration state. This keeps
    `course_access` from depending on `course_applications` (no dependency cycle; the
    application app stays a leaf).
- `filter_visible`: returns `courses` unchanged (today's behaviour).

### 4.4 Wiring the backend into the student interface

- **Chokepoint gate** — `student_interface/views.py:246` `register_for_course`: before
  `UserCourseRegistration.objects.get_or_create(...)`, call
  `get_course_access_backend().get_access(user=request.user, course=course)`. If
  `not decision.can_self_register`, redirect to `decision.cta_url` (the application flow for
  gated courses) instead of registering. This is the **single** server-side enforcement
  point; admin/cohort registration paths are untouched and continue to bypass the gate.
- **CTA branching** — `_preview_start_url()` (`views.py:68`) and `_annotate_preview_context()`
  (`views.py:92`) consume `decision.cta_label` + `decision.cta_url` instead of always
  emitting the `register_for_course` URL + "Start". Templates
  `course_preview_content.html:34` and `course_card.html:78` render `cta_label` / `cta_url`.
- **Listing visibility** — route the two listing call sites through `filter_visible()`:
  `dashboard` (`views.py:134`) and `all_courses` (`views.py:155`), both of which iterate
  `get_all_courses()` (`utils.py:427`). Wrap as
  `backend.filter_visible(user=user, courses=get_all_courses())`. Default behaviour
  unchanged.
- **Direct course-URL access** — `course_home` / `course_preview` may additionally consult
  `get_access()` so a learner cannot reach gated course *content* by guessing the slug; the
  preview page (title/description + "Apply" CTA) stays visible, but content access requires
  `REGISTERED`. (Preview remains visible because it is the funnel surface.)

## 5. Part B — `course_applications`

New app, `label = "freedom_ls_course_applications"`, added to `INSTALLED_APPS`. It is a
**leaf** app: it may read `content_engine.Course` and `student_management` enrolment
primitives, but `course_access` must **not** depend on it.

Two clearly separated halves:

- **Config** = authored content, loaded from yaml/markdown (read-only at runtime).
- **Submissions** = applicant-created user data, DB-only, PII-bearing.

### 5.1 Config models (content-loaded, mirror the `Form` family)

These mirror `Form → FormPage → FormQuestion → QuestionOption`
(`content_engine/models.py:302–451`) in shape and are loaded by the **same content_save
pipeline** (new `content_type`s + Pydantic schema entries), exactly as the user directed —
keeping a single authoring paradigm for all content-shaped data. All inherit
`SiteAwareModel`; all carry the `BaseContent` `file_path` so they are file-backed like
other content.

```
ApplicationConfig          # one per Course (per site). content_type: APPLICATION_CONFIG
  course           FK Course        (the gated course this config governs)
  title, intro     (markdown)
ApplicationStep            # ordered steps within a config
  config           FK ApplicationConfig (related_name="steps")
  title, intro
  order            PositiveSmallInteger
ApplicationQuestion        # mirrors FormQuestion + a file type
  step             FK ApplicationStep (related_name="questions")
  prompt, help_text
  type             ApplicationQuestionType  (see below)
  required         bool
  order            PositiveSmallInteger
  # file-upload config (null unless type == FILE_UPLOAD):
  file_max_bytes   PositiveInteger | None
  file_mime_allowlist  JSON list[str] | None
  file_max_count   PositiveSmallInteger | None
ApplicationQuestionOption  # mirrors QuestionOption (no `correct` — applications aren't scored)
  question         FK ApplicationQuestion (related_name="options")
  text, value, order
```

**`ApplicationQuestionType`** (`TextChoices`) = the existing four plus one new value:
`SHORT_TEXT`, `LONG_TEXT`, `MULTIPLE_CHOICE`, `CHECKBOXES`, `FILE_UPLOAD`. The file type is
the one genuinely new question type (the existing `QuestionType` has no file option).

Content-loading detail:

- Add an `APPLICATION_CONFIG` content type and matching Pydantic schema classes in
  `content_engine/schema.py`, plus a `save_application_config()` analogous to
  `save_course()` (`content_save.py:342`). Because the loader lives in `content_engine`,
  the config models either (a) live in `content_engine`, or (b) the loader is extended to
  register the `course_applications` config models. **Decision:** the config models live in
  `course_applications`, and `content_save` is extended to dispatch the new content type to
  them. This keeps all application code in one app. The structure review must confirm this
  `content_engine → course_applications` *loader-only* edge is acceptable (it mirrors how
  the loader already knows about every content model); if not, fall back to (a). Flag for
  `/plan_structure_review`.
- Example content file:

  ```yaml
  ---
  content_type: APPLICATION_CONFIG
  course: data-science-bootcamp        # course slug on this site
  title: Apply to the Data Science Bootcamp
  steps:
    - title: About you
      questions:
        - {type: short_text, prompt: Full name, required: true}
        - {type: long_text,  prompt: Why this course?, required: true}
    - title: Documents
      questions:
        - type: file_upload
          prompt: Proof of ID
          required: true
          file_mime_allowlist: [application/pdf, image/png, image/jpeg]
          file_max_bytes: 5242880
          file_max_count: 1
  ---
  ```

### 5.2 Submission models (DB-only, relational — consistent with `QuestionAnswer`)

Per the locked decision, applicant answers are stored **relationally, mirroring
`student_progress.QuestionAnswer`** (`models.py:503`) — one answer row per question, no
JSON answer blob. The only new element is file storage, added as a dedicated PII row.

```
CourseApplication              # ≈ FormProgress: one applicant attempt
  user        FK User
  course      FK Course
  config      FK ApplicationConfig
  state       FSMField(default="draft", protected=True)   # django-fsm-2
  created_at, updated_at, submitted_at, decided_at
  decided_by  FK User | None
  # Partial unique index: at most one ACTIVE application per (site, user, course),
  # where active = state in (draft, submitted, under_review, changes_requested).
  # Rejected/withdrawn/approved do not block re-application.

ApplicationAnswer              # ≈ QuestionAnswer
  application      FK CourseApplication (related_name="answers")
  question         FK ApplicationQuestion
  selected_options M2M ApplicationQuestionOption   # choice/checkbox
  text_answer      TextField (blank, default="")    # text
  last_updated_time
  # unique_together = [application, question]   (same as QuestionAnswer)

ApplicationFile                # the new piece: PII document, private storage
  answer        FK ApplicationAnswer (related_name="files")
  file          FileField(storage=<private>, upload_to=<non-guessable, pk-based>)
  original_name CharField        # for display only, never used in the storage path
  mime_type     CharField        # validated by magic bytes, not the browser
  size_bytes    PositiveInteger
  scan_status   CharField(choices: pending|clean|infected, default=pending)  # seam; no scanner in v1
  superseded    bool(default=False)  # replaced files kept for audit, not hard-deleted
  uploaded_at

ApplicationNote                # staff-only internal note (structurally distinct from applicant message)
  application FK, author FK User, body Text, created_at

ApplicationStateTransition     # audit log; one row per transition
  application FK (related_name="transitions")
  from_state, to_state, actor FK User | None
  message       TextField   # applicant-visible message (e.g. the request_changes reason)
  created_at
```

Notes:

- `ApplicationAnswer` is deliberately a 1:1 copy of the `QuestionAnswer` shape
  (`selected_options` M2M + `text_answer` TextField + `unique_together`) so the two systems
  read the same way. File answers carry no value on the answer row itself; the
  `ApplicationFile` rows hang off it.
- Keeping documents in their own table (not a column on the answer) gives the PII the
  isolation the idea requires: private storage, replace-history (`superseded`), targeted
  retention/deletion, and a future `scan_status` workflow — all without touching the
  textual answers.

### 5.3 State machine (`django-fsm-2`)

Add the dependency: `uv add django-fsm-2` (confirmed absent today). `state` is an
`FSMField(protected=True)`. Transitions (per `research_review_workflow.md §1.3`):

| Transition | Actor | source → target | Notes |
|---|---|---|---|
| `submit` | applicant | draft → submitted | validates required answers across all steps |
| `withdraw` | applicant | [draft, submitted, changes_requested] → withdrawn | |
| `pick_up` | reviewer | submitted → under_review | |
| `request_changes` | reviewer | under_review → changes_requested | **requires** an applicant-visible message |
| `resubmit` | applicant | changes_requested → submitted | re-runs validation |
| `approve` | reviewer | under_review → approved | terminal; optional internal note |
| `reject` | reviewer | under_review → rejected | terminal; optional internal note |

- Reviewer transitions use `permission=` callables delegating to
  `user.has_perm("change_application", instance)` (django-guardian, object-level).
- Each transition writes an `ApplicationStateTransition` row and fires an
  `application_state_changed` Django signal (the seam for a future email backend; no
  notifications in v1).
- Templates drive buttons off `get_available_user_state_transitions(user)` — never hardcode
  state→button maps.
- `expired`/`waitlisted` and multi-reviewer are out of v1 (do not architect away — the
  audit row + signal already accommodate them).

### 5.4 Multi-step applicant UX

Per `research_multistep_ux.md` (DB-backed draft, one URL per step, GOV.UK check-answers):

- **Entry:** the gated-course CTA "Apply" creates/loads the learner's active
  `CourseApplication` (draft) for that course and redirects to step 1.
- **One real URL per step:** `GET/POST /applications/<id>/step/<order>/`. GET renders the
  step pre-populated from saved answers; POST validates only that step's fields, upserts the
  relevant `ApplicationAnswer` rows (mirroring `FormProgress.save_answers`,
  `models.py:226`), then 303-redirects to the next step or the review page. Browser back,
  refresh, bookmark all work; draft survives logout/device switch (it is on the user's
  account in the DB, not session/cookie).
- **HTMX inside a step only** (per project conventions): inline 422 validation, file upload,
  conditional reveal. Whole-step swaps are not used.
- **Progress indicator:** all steps visible, numbered + labelled, current highlighted,
  completed ticked; collapses to "Step N of M: <label>" on mobile.
- **Check your answers (final step):** GOV.UK summary cards per step; per-section "Change"
  link → `GET .../step/<order>/?return=review`, whose POST honours `?return=review` and
  redirects back to review without walking later steps. Optional unanswered questions show
  "Not provided". Submit button labelled "Submit application for <course>". Submit runs full
  cross-step required-answer validation; on failure, redirect to the first incomplete step
  with errors.
- **Lock on submit:** after `submitted`, the applicant view is read-only; the only re-edit
  path is a reviewer `request_changes` → `changes_requested`, which re-enables Change links
  and surfaces the reviewer's required message + a per-step indication of what to fix.

### 5.5 File uploads

- **A question type** (`FILE_UPLOAD`), configured per question: `file_max_bytes`,
  `file_mime_allowlist`, `file_max_count`.
- **Upload endpoint** (HTMX): native `<input type="file">` → `hx-post` to an upload view
  that creates an `ApplicationFile`, returns a partial showing name/size + Replace/Remove.
  Validation: size (per-question + Django's `FILE_UPLOAD_MAX_MEMORY_SIZE`), MIME **by magic
  bytes** (not the browser `Content-Type`), extension allowlist, count. On failure return
  HTTP 422 with a re-rendered field partial.
- **Private storage + permission-checked serving.** No private media path exists today
  (`spec_research_codebase.md §11`); build it: a non-public storage location and a
  login-required serve view that checks the requester is the owning applicant **or** a
  reviewer with `view_application` on that application. Never serve documents from public
  `MEDIA_URL`. Storage path is pk/uuid-based; the original filename is display-only.
- **Replace** keeps the old row (`superseded=True`) for audit; hard-deletion happens only on
  retention/erasure.
- **Out of v1 (seams kept):** virus scanning (`scan_status` field present, defaults
  `pending`, no scanner wired) and EXIF stripping.

### 5.6 Applicant status page

Plain-language current state; when `changes_requested`, show the reviewer's message and
which steps need attention with an Edit/Resubmit action; when `approved`/`rejected`, show
the decision and any applicant-visible message. A `withdraw` action is available while the
application is active. Drafts and in-flight applications are reachable from the learner's
dashboard.

## 6. Part C — Review UI in `educator_interface`

Reviewers are educators / programme managers, not Django superusers — so the review UI
lives in `educator_interface`, which already houses cohort/student/registration UI and
already scopes access with `django-guardian`
(`get_objects_for_user(request.user, "view_cohort", klass=Cohort)`,
`educator_interface/views.py:84`).

- **Global applications inbox** (not per-course): a filterable table across the reviewer's
  site — filter by course, status, date; search by applicant name/email; default sort
  oldest-submitted-first. Built on the existing panel/list-config framework
  (`CohortConfig`/`CohortDataTable` are the model, `views.py:972`). Queryset is scoped via
  `get_objects_for_user(request.user, "view_application", klass=CourseApplication)` and
  inherits site isolation from `SiteAwareModel`.
- **Single-application review screen** (row click → full screen, not a modal): main area =
  all answers + inline document previews (signed/permission-checked serve on click); sticky
  sidebar = state-aware decision actions (rendered from
  `get_available_user_state_transitions`), internal notes (`ApplicationNote`), and the
  transition log (`ApplicationStateTransition`). `request_changes` and `reject` require a
  confirmation step capturing their message/reason. On resubmit, prominently mark
  "resubmitted at <time>" and link the prior reviewer message in the log.
- **Internal notes vs applicant-facing message are separate fields**, distinctly labelled,
  never merged (avoids leaking internal commentary).
- **Approval ≠ enrolment.** The approved-application screen exposes an "Enrol this learner"
  deep-link into the existing cohort + `UserCourseRegistration` flow with the user
  pre-filled. Approving never auto-creates a registration. (Because admin/cohort enrolment
  bypasses the access backend, the approved learner gets in cleanly.)
- **No bulk approve/reject in v1.** Django/Unfold admin may expose the models read-mostly
  for support, with state transitions hidden (`FSM_ADMIN_FORCE_PERMIT = True` if
  `django-fsm-admin` is used).

### Permissions

- Model permissions `view_application`, `change_application` on `CourseApplication`, granted
  object-level via `django-guardian` (`assign_perm`), mirroring the cohort pattern
  (`role_based_permissions/utils.py:179`). State-changing transitions delegate to
  `user.has_perm("change_application", instance)`.
- Site isolation is automatic via `SiteAwareModel` — Site A reviewers never see Site B
  applications or configs.

## 8. Dependency direction (for the structure review)

- `course_access` → depends on `content_engine` (reads `Course`) and `student_management`
  (reads registration state). Must **not** depend on `course_applications`.
- `student_interface` → depends on `course_access`.
- `course_applications` → may read `content_engine.Course` and `student_management`
  enrolment primitives; is otherwise a leaf.
- `educator_interface` → depends on `course_applications` (review UI) and reuses its
  existing cohort/registration flow for the post-approval hand-off.
- **Open edge to confirm:** the content-loader (`content_engine`) dispatching the
  `APPLICATION_CONFIG` content type to `course_applications` models creates a loader-only
  `content_engine → course_applications` reference. This mirrors how the loader already
  knows every content model. If the structure review rejects it, move the config models into
  `content_engine` instead (the submission models stay in `course_applications`
  regardless). Flagged explicitly for `/plan_structure_review`.

## 9. Key decisions & rationale

| Decision | Choice | Why |
|---|---|---|
| Access type representation | one `CharField` on `Course`, default `free` | one type at a time; minimal; implicitly per-site; no back-fill migration |
| Build the backend now? | yes — interface + loader + one backend | explicitly requested; known to need to grow to subscriptions/purchase; seam is the deliverable |
| Who reads `access_type` | only `DefaultCourseAccessBackend` | a future subscription backend ignores it entirely; keeps it private config |
| Status enum scope | full set now, default returns 3 | future backends + templates need no change |
| Separate `course_applications` app | yes | applications are admin-shaped config + unscored PII submissions; reusing `Form` tables would muddy content invariants, scoring, and PII isolation |
| Form **config** authoring | content-loading (yaml/markdown) | **user decision** — consistent with the entire content system; version-controlled; one authoring paradigm; far less code than an admin/builder UI |
| Answer **storage** | relational, mirroring `QuestionAnswer`; files as `ApplicationFile` rows | **user decision** — be consistent with the current implementation; strong PII query/redaction/deletion; FK integrity; relational needs no migration to change a course's questions (a question is config data, not a column) |
| State machine | `django-fsm-2`, `protected=True` | matches the subscriptions spec; greenfield; transition perms integrate with guardian |
| Audit | custom `ApplicationStateTransition` | lighter than `django-simple-history`; sufficient for who/when/what |
| Review UI home | `educator_interface`, global inbox | correct audience; reuses guardian scoping + panel framework; global inbox is a recurring unmet need elsewhere |
| Approval vs enrolment | decoupled, with convenience hand-off | off-platform steps may intervene; matches serious admissions systems |
| Multi-step storage | DB-backed draft, one URL per step | survives logout/device switch; browser back/refresh/bookmark work; session/cookie wizards fail these |

## 10. Risks & mitigations

- **Direct-comparison drift** (someone writes `course.access_type == ...` outside the
  backend) — mitigate with explicit comments at the field + backend and a review check; it
  is the one rule that keeps the future refactor cheap.
- **Loader cross-app edge** — see §8; fallback is config models in `content_engine`.
- **PII handling** — private storage + permission-checked serving are in scope; magic-byte
  validation prevents content-type spoofing; replace-history + `scan_status` + targeted
  deletion are built/seamed now. The threat model (`/threat-model`) and security reviews
  should scrutinise the serve view and upload validation specifically.
- **Concurrent reviewer actions** — `FSMField` guards serialise transitions cleanly
  (second action fails on a now-invalid source state).
