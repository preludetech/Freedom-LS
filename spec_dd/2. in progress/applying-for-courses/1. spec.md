# Spec: Course Access Types — Free & Application-Gated

Status: draft for review
Source idea: `0. idea.md`
Research: `research_course_access_types.md`, `research_form_schema.md`,
`research_multistep_ux.md`, `research_review_workflow.md`,
`research_reference_implementations.md`, `.sdd-work/spec_research_codebase.md`

## 1. Summary

Introduce an explicit, configurable **per-course access config** interpreted by a
**pluggable course-access backend** (modelled on Django's `EMAIL_BACKEND`). Core stores only
an opaque `access_config` blob and interprets nothing; the backend owns all interpretation.
The **core** default backend (`DefaultCourseAccessBackend`, in `course_access`) recognises one
value — `free` — today's behaviour made explicit; core knows nothing of applications. The
`application_gated` value, the Apply flow, and the in-flight-application dashboard panel are all
owned by an **applications backend** (`ApplicationCourseAccessBackend`) that ships in
`course_applications` and subclasses the default. `application_gated` routes a learner into an
**application** that an educator reviews and approves; approval is decoupled from enrolment. The
active backend plugin is the single seam that contributes affordances to course cards, course
detail/preview pages, **and** the learner dashboard — no core surface ever names a concrete
backend. In this work
the application is **form-less**: an **Apply** button creates a reviewable submission. The
authored multi-step **application form** (questions, file uploads, content-loaded config) is
a separate follow-up spec — `spec_dd/0. drafts/application-forms` — that slots in front of
this same submission with no rearchitecture.

Two new low-level concerns ship:

1. **`freedom_ls/course_access/`** — the backend interface, a settings-selected loader,
   and a `free`-only `DefaultCourseAccessBackend` (core's only reader of `access_config`, which
   it interprets solely as `free`/registered). The seam — `get_access` + `filter_visible` +
   `validate_course_config` + `get_dashboard_contributions` — is the deliverable: it must be
   broad enough that the applications backend, and future subscription/purchase backends, slot in
   as a new class plus a settings change — bringing their own config keys, vocabulary, validation,
   and dashboard/CTA contributions — never a rearchitecture of the student interface or a core
   migration. `course_access` never imports `course_applications`.
2. **`freedom_ls/course_applications/`** — applicant **submissions** (DB-only, PII-bearing,
   state-machine-governed), created by a bare **Apply** action and reviewed/actioned through
   the **Django admin** in v1 (approve / reject / request-changes, with the full audit trail).
   The authored application **form** (config models, content-loading, per-question answers,
   file uploads, multi-step UX) is deferred to the `application-forms` follow-up spec, and the
   bespoke educator-facing review UI (global inbox + single-application screen in
   `educator_interface`) is deferred to the `application-review-ui` follow-up spec
   (`spec_dd/0. drafts/application-review-ui`); every model and seam both need is left in
   place here.

### Why both access types matter now

Free courses are the funnel that drives learners toward application-gated ("paid")
courses. The cross-promotion CTA itself is **future work**, but both access types must be
configurable now so that funnel can be built on top later. Building the backend seam now
(rather than a single `if access_type == ...`) is a deliberate, explicitly-requested
exception to the project's "don't build unrequested abstractions" convention, because the
mechanism is known to need to grow to subscriptions, per-course purchase, and feature
unlocks.

## 2. Goals & success criteria

A change is done when all of the following hold:

1. **Access config is a property of a course.** `Course.access_config` exists (opaque JSON,
   default `{}`), is set via content-loading, and existing courses behave exactly as today
   after migration (back-filled to `{}` → default backend reads as `free`, no data migration
   needed).
2. **Every access question routes through the backend.** No view, template, or utility reads
   or branches on `course.access_config`; all access decisions and CTAs come from
   `get_access(*, user, course)`, and any backend-specific dashboard content comes from
   `get_dashboard_contributions(*, user)`. Only the active backend interprets `access_config`;
   core branches on nothing. **No core surface (dashboard, course cards, course detail/preview)
   names a concrete backend or imports `course_applications`** — the active backend plugin is the
   only thing that contributes affordances to those surfaces.
3. **The gate is enforced server-side.** The single self-registration chokepoint
   (`student_interface.register_for_course`, `views.py:234`) refuses to self-register a
   learner for an `application_gated` course and routes them to the application flow. Admin
   and cohort enrolment deliberately bypass the gate.
4. **Course listings route through `filter_visible()`.** `get_all_courses()`
   (`utils.py:443`) and its dashboard/all-courses call sites pass through the backend; the
   default backend returns the queryset unchanged.
5. **A learner can apply to a gated course.** The gated-course CTA "Apply now" routes to an
   application-entry view that creates a `CourseApplication` for that learner+course and
   submits it for review (no form to fill in this work). Applying again is governed by the
   active-application uniqueness rule.
6. **A reviewer can review submitted applications** from the Django admin — a filterable,
   searchable changelist of applications, each showing its applicant and full history — and
   request changes (with a required applicant-facing message), approve, or reject; every
   transition is recorded for audit. (The bespoke educator-facing inbox / review screen is the
   `application-review-ui` follow-up spec.)
7. **Approval is not enrolment.** Approving exposes a convenience "enrol this learner"
   hand-off into the existing cohort/registration flow but never auto-enrols.
8. **Multi-tenancy holds.** All new models inherit `SiteAwareModel`; reviewers only ever
   see their own site's applications and configs. Reviewer access is object-level via
   `django-guardian`.
9. **A future backend is a class + settings change.** Adding a subscription,
   per-course-purchase, prerequisite, or intake-window backend requires no change to
   templates, the chokepoint, or the listing call sites — only a new backend class + the
   `COURSE_ACCESS_BACKEND` setting.

## 3. Scope

### In scope

- `Course.access_config` opaque JSONField (default `{}`), loaded via content-loading; core
  interprets nothing.
- `freedom_ls/course_access/`: `CourseAccessBackend` base class (incl.
  `validate_course_config()` and `get_dashboard_contributions()`), `CourseAccessDecision` +
  `CourseAccessCTA` + `DashboardContribution` (the open display+behavior contract — no fixed
  status enum), the default backend's private `CourseAccessType` (`free` only),
  `get_course_access_backend()` loader, `COURSE_ACCESS_BACKEND` setting, a Django system check for
  config validity, and the `free`-only `DefaultCourseAccessBackend`. Core knows nothing of
  applications.
- `freedom_ls/course_applications/`: DB-only submission models (`CourseApplication`,
  `ApplicationNote`, `ApplicationStateTransition`) with a `django-fsm-2` state machine; a
  form-less **Apply** entry view that creates + submits an application; applicant status page; an
  `ApplicationCourseAccessBackend` (subclasses `DefaultCourseAccessBackend`) that owns the
  `application_gated` access type, the Apply CTA, and the in-flight-application dashboard
  contribution; a `get_active_applications(user)` query helper.
- Application review in the **Django admin** (`course_applications/admin.py`): a filterable /
  searchable changelist; per-application state transitions (approve / reject / request-changes,
  the latter two capturing their required message via an intermediate action page); internal
  notes + transition log; an "enrol approved learner" link into the existing cohort /
  registration flow. Object-level access via `django-guardian`, site isolation preserved.
- Wiring: chokepoint gate, CTA branching, `filter_visible()` at listing call sites, and
  generic rendering of `get_dashboard_contributions()` on the learner dashboard.
- `django-fsm-2` added as a dependency.

### Deferred to the `application-forms` follow-up spec (`spec_dd/0. drafts/application-forms`)

The **authored application form** is split into its own spec, leaving this one to ship the
access backend + bare apply/review flow. Deferred there (every model/seam it needs is left
in place here — do not architect it away): content-loaded config models (`ApplicationConfig`,
`ConfiguredCourse`, `ApplicationStep`, `ApplicationQuestion`, `ApplicationQuestionOption`);
the `APPLICATION_CONFIG` content type + Pydantic schema + `save_application_config()` loader;
per-question answer rows (`ApplicationAnswer`) and the `config` FK on `CourseApplication`;
file-upload questions + `ApplicationFile` with private storage + permission-checked serving;
multi-step applicant UX (one-URL-per-step draft, GOV.UK check-answers); the
`get_application_config(*, course)` helper.

### Deferred to the `application-review-ui` follow-up spec (`spec_dd/0. drafts/application-review-ui`)

The bespoke **educator-facing review UI** is split into its own spec so v1 can ship review on
the Django admin (§6). Deferred there: the global, filterable **applications inbox** in
`educator_interface` (built on the existing panel/list-config framework, guardian-scoped); the
full-screen **single-application review screen** (applicant identity + metadata main area,
sticky state-aware decision sidebar, inline notes, transition log, "resubmitted at" markers);
the in-context "enrol this learner" hand-off; and serving review to educators / programme
managers who are **not** Django staff. That spec also introduces the
`educator_interface → course_applications` edge this one deliberately avoids. The admin review
built here keeps every seam it needs: the FSM transitions,
`get_available_user_state_transitions`, `ApplicationNote`, and `ApplicationStateTransition`
are all UI-agnostic.

### Out of scope (deliberately deferred — do not architect away)

On-platform payment; the free→application funnel CTA; further concrete access backends
(subscriptions, per-course purchase, feature unlocks); `has_feature()` call sites;
waitlists; application/intake windows and auto-expiry; multi-reviewer assignment and
rubric/score review; bulk approve/reject; conditional/branching questions; email
notifications; in-app identity verification and virus scanning; an in-UI toggle to flip a
course's `access_type` (model supports it; only the UI is deferred).

Each deferred item must leave its seam: the `CourseAccessDecision` / `CourseAccessCTA` object
is the open contract — new access situations are new CTAs, needing no enum or template
change; the backend base class is the designated home for `has_feature()`; the form-less
`CourseApplication` + its state machine are the seam the `application-forms` spec hangs
answers, files, and config off; every transition fires a signal so an email backend can be
added later; `RecommendedCourse` remains the funnel CTA's future home.

## 4. Part A — Course access type & pluggable backend

### 4.1 `Course.access_config` — opaque, backend-owned

Core stores **opaque per-course access configuration and interprets none of it**. Store the
per-course access configuration as a single JSON blob (there is no typed `access_type` field
on `Course` today — this is the first access-config field); the active backend owns the
schema, vocabulary, and validation for its own slice.

Add to `content_engine/models.py` (after `category`, line 154):

```python
access_config = models.JSONField(
    default=dict,
    blank=True,
    help_text=_(
        "Opaque per-course access configuration. Interpreted ONLY by the active "
        "course-access backend (settings.COURSE_ACCESS_BACKEND); core never reads or "
        "branches on its contents. The default backend stores {'access_type': ...}."
    ),
)
```

- `CourseAccessType` is **not a core enum** — it lives only in the default backend
  (`course_access`, §4.3) as that backend's private vocabulary. `content_engine` does not
  import or know it.
- `JSONField(default=dict)` → Django back-fills existing rows to `{}`; **no `RunPython` data
  migration**. A backend reads missing keys as its own default (the default backend treats
  absent `access_type` as `free`), so existing courses behave exactly as today.
- Because `Course` is site-aware, config is implicitly per-site: the same slug can carry
  different `access_config` on different sites.
- **Rule:** `access_config` is backend-private. No view, template, or utility may read or
  branch on its contents — only the active backend, via `get_access()` / `filter_visible()`.
  A future subscription backend stores entirely different keys (e.g. `{'plan': 'pro',
  'price_id': ...}`) with **no change to this field, the loader, or any migration**. Enforced
  by review, not the type system — call it out explicitly in code comments at the field and
  in the backend.

### 4.2 Setting & validating `access_config` via content-loading

Per the idea and the locked decision, access config is authored through the content-loading
pipeline only (no admin/educator toggle in v1).

- Add `access_config: dict[str, Any] | None = Field(None, ...)` to the `Course` Pydantic
  model in `content_engine/schema.py` (line 96). The schema layer treats it as an **opaque
  dict** — it does not know backend-specific keys. `save_with_uuid`
  (`content_engine/management/commands/content_save.py:224`) maps it to the model field
  automatically via the field-existence gate (lines 267–279); the
  Pydantic field and model field share the name `access_config`, so no other change is needed.
- Frontmatter:

  ```yaml
  access_config:
    access_type: application_gated
  ```

  Absent → `{}` → default backend behaves as `free`.
- **Backend-owned validation, implemented once, called at both ends** (per the locked
  decision): the backend exposes a single
  `validate_course_config(raw: dict, *, file_path: str = "") -> dict` that validates and
  normalizes its slice, raising `ValueError` on unknown keys/values (with `file_path`
  context, consistent with existing content validation). It is called from:
  - **load time** — `save_course()` (`content_engine/management/commands/content_save.py:342`)
    calls
    `get_course_access_backend().validate_course_config(item.access_config or {}, file_path=...)`
    and stores the normalized result → bad config is rejected at author time (fail loud).
  - **read / ops time** — a Django **system check** (`@register` in `course_access`) runs the
    same validator over every `Course.access_config` for the active backend, so swapping
    `COURSE_ACCESS_BACKEND` surfaces now-invalid existing config via `manage.py check` rather
    than at request time. (DB-querying system checks are an established pattern here — see
    `accounts/checks.py`, which queries `Site`/`SiteSignupPolicy`; follow it and tolerate the
    table not yet existing on a fresh DB.)
  - `get_access()` also runs the validator defensively and, on failure, returns a safe
    no-action decision (`cta=None`, both enforcement gates `False`; never 500s a learner's
    page); the load step and system check are the proactive surfaces.

  One validation implementation, reused everywhere.

### 4.3 `freedom_ls/course_access/` — the backend

New app, `label = "freedom_ls_course_access"`, added to `INSTALLED_APPS`. Follows the
existing pluggable-backend precedent in `freedom_ls/icons/backend.py` (`import_string` +
`@functools.cache` + base class + `DefaultXxxBackend`).

**Default backend's private vocabulary** — `CourseAccessType` lives here, in `course_access`,
**not** in `content_engine` (core knows nothing of it). Core recognises only `free`;
`application_gated` is **not** a core value — it is added by the applications backend (§5.5),
which subclasses the default and extends this vocabulary. Core never references applications:

```python
class CourseAccessType(models.TextChoices):
    FREE = "free", _("Free")
    # application_gated is NOT here — it is the applications backend's vocabulary (§5.5).
```

**Call-to-action + decision objects** — the backend fully specifies what the learner sees
(`label`) and what it does (`url`); there is **no enumerated set of access situations**. New
cases (prerequisites unmet, intake window not open, purchase/subscription required, …) are
just a different CTA the backend returns — no shared enum, no template change.

```python
@dataclass(frozen=True)
class CourseAccessCTA:
    label: str        # "Start", "Enroll", "Apply now", "Continue", "Subscribe", …
    url: str | None   # where the affordance points / what it triggers; None = not actionable

@dataclass(frozen=True)
class CourseAccessDecision:
    cta: CourseAccessCTA | None   # the affordance to show; None = show nothing
    can_self_register: bool       # enforcement: may the self-register chokepoint register now?
    can_access_content: bool      # enforcement: may the user view course content now?

@dataclass(frozen=True)
class DashboardContribution:
    template_name: str   # a partial the contributing backend owns (e.g. an applications panel)
    context: dict        # render-ready context for that partial
```

**Dashboard contributions** — a second open seam, the dashboard counterpart of the CTA. The
backend returns a list of `DashboardContribution`s; the dashboard renders each generically
(`render_to_string(template_name, context, request=request)`) and never inspects `context` keys,
so it stays decoupled from any backend's data shape. The default backend returns `[]` (today's
dashboard, unchanged). This is how a backend "adds things to the dashboard" — the same way `cta`
is how it adds things to cards and detail pages — without any core surface naming the backend.

- `cta` is the **only** display/behavior contract; templates render it through the existing
  `<c-button>` (already does `<a href>` vs disabled `<button>`). The two booleans are
  enforcement gates, not display, and collapse every situation into the only two yes/no
  questions the system asks. Richer display fields (variant, help_text, icon) are added to
  `CourseAccessCTA` only when a concrete backend needs them — not pre-assumed now.

**Base class** (`course_access/backends.py`):

```python
class CourseAccessBackend:
    def get_access(self, *, user, course) -> CourseAccessDecision: ...
    def filter_visible(self, *, user, courses): ...   # -> QuerySet[Course]
    def validate_course_config(self, raw: dict, *, file_path: str = "") -> dict:
        """Validate & normalize this backend's slice of Course.access_config.
        Single implementation; called at content-load (fail loud), by a Django
        system check (after a backend swap), and defensively in get_access().
        Raise ValueError on invalid config."""
        ...
    def get_dashboard_contributions(self, *, user) -> list[DashboardContribution]:
        """Backend-owned panels for the learner dashboard. Default: nothing.
        The dashboard renders each contribution generically and never names this
        backend — this is the seam that replaces a hard-coded dashboard import."""
        return []
    # has_feature(*, user, feature) -> bool is intentionally NOT defined in v1;
    # it is the designated additive home for feature-unlock backends. Documented, not built.
```

**Loader** (`course_access/__init__.py` or `loader.py`):

```python
@functools.cache
def get_course_access_backend() -> CourseAccessBackend:
    path = settings.COURSE_ACCESS_BACKEND
    return import_string(path)()
```

**Settings** (`config/settings_base.py`, near `TASKS`):

```python
# Ships with applications enabled so the feature works out of the box. A deployment that does
# not want applications sets this to the free-only core default backend:
#   "freedom_ls.course_access.backends.DefaultCourseAccessBackend"
COURSE_ACCESS_BACKEND = "freedom_ls.course_applications.backends.ApplicationCourseAccessBackend"
```

This default lives in `config` (the project), not in `course_access`, and is a settings string —
not a code import — so `course_access` still never depends on `course_applications`.

**`DefaultCourseAccessBackend`** — core's `free`/registered-only backend, the core reader of
`access_config`. It knows **nothing** about applications; the applications backend (§5.5)
subclasses it to add `application_gated`:

- `validate_course_config`: accept only the key `access_type`; value ∈ `CourseAccessType`
  (i.e. only `free`); default to `free` when absent; reject unknown keys/values — **including
  `application_gated`, which is not core vocabulary** — with `file_path` context; return
  `{"access_type": <value>}`. (A deployment running the core default backend therefore cannot
  author `application_gated` courses — that requires the applications backend. The system check,
  §4.2, flags any pre-existing `application_gated` config after a swap back to the default.)
- `get_access`: resolve
  `access_type = self.validate_course_config(course.access_config)["access_type"]` (on
  `ValueError` → safe no-action decision: `cta=None`, `can_self_register=False`,
  `can_access_content=False`). Then:
  - already registered (direct `UserCourseRegistration` or via `CohortCourseRegistration`,
    both in `student_management.models`) → `cta=CTA("Continue", course_home url)`,
    `can_self_register=False`, `can_access_content=True`. Use the extracted
    `student_management.is_registered_for_course()` helper (§5.5/§8); do **not** import
    `student_interface` (that would create a cycle).
  - else `access_type == FREE` → `cta=CTA("Start", register_for_course url)`,
    `can_self_register=True`, `can_access_content=False`.
  - No application branch — the default backend never references `course_applications` (no URL,
    no records), keeping `course_access` independent of it.
- `filter_visible`: returns `courses` unchanged (today's behaviour).
- `get_dashboard_contributions`: returns `[]` (today's dashboard, unchanged).

### 4.4 Wiring the backend into the student interface

- **Chokepoint gate** — `student_interface/views.py:234` `register_for_course`: before
  `UserCourseRegistration.objects.get_or_create(...)` (`views.py:240`), call
  `get_course_access_backend().get_access(user=request.user, course=course)`. If
  `not decision.can_self_register`, redirect to `decision.cta.url` (the application flow for
  gated courses; fall back to the course preview when `cta`/`cta.url` is `None`) instead of
  registering. This is the **single** server-side enforcement point; admin/cohort
  registration paths are untouched and continue to bypass the gate.
- **CTA branching** — `_preview_start_url()` (`views.py:70`) and `_annotate_preview_context()`
  (`views.py:94`) attach the whole `decision.cta` to the course instead of always emitting the
  `register_for_course` URL + a hardcoded "Start". Templates
  `partials/course_preview_content.html` (the `<c-button>` Start affordance, ~line 37) and
  `partials/course_card_not_registered.html` (the `course.preview_start_url` CTA — note there
  is no single `course_card.html`; the card is split into `course_card_not_registered.html` /
  `course_card_registered.html` / `course_card_complete.html`) render `cta.label` + `cta.url`
  via `<c-button>` (omit or disable the affordance when `cta` or `cta.url` is `None`). Note: the progress-aware "resume at item
  1 vs course_home" refinement for already-registered learners stays in `student_interface`
  presentation — the access backend owns the headline affordance, not which item to land on.
- **Listing visibility** — route the two listing call sites through `filter_visible()`:
  `dashboard` (`views.py:117`) and `all_courses` (`views.py:155`), both of which iterate
  `get_all_courses()` (`utils.py:443`). Wrap as
  `backend.filter_visible(user=user, courses=get_all_courses())`. Default behaviour
  unchanged.
- **Direct course-URL access** — `course_home` / `course_preview` may additionally consult
  `get_access()` so a learner cannot reach gated course *content* by guessing the slug; the
  preview page (title/description + "Apply now" CTA) stays visible, but content access gates on
  `decision.can_access_content` — a single enforcement boolean, never an inspection of
  registration state or config; a subscription backend can grant content access without
  enrolment. (Preview remains visible because it is the funnel surface.)
- **Dashboard contributions** — the `dashboard` view (`views.py:117`) calls
  `backend.get_dashboard_contributions(user=request.user)` and renders each generically:
  `render_to_string(contribution.template_name, contribution.context, request=request)`, passing
  the resulting HTML fragments to the dashboard template (loop + `|safe`). The view never reads
  `contribution.context` keys and never imports `course_applications`; the contributing backend
  owns the partial and its data. The default backend returns `[]`, so the dashboard is unchanged
  when applications are not in use. This is the seam that **replaces** the previously-considered
  `student_interface → course_applications` dashboard import (§8).

## 5. Part B — `course_applications`

New app, `label = "freedom_ls_course_applications"`, added to `INSTALLED_APPS`. It is a
**leaf** app: it may read `content_engine.Course` and `student_management` enrolment
primitives, but `course_access` must **not** depend on it.

In this work the app is **submissions only**: a learner clicks **Apply**, a
`CourseApplication` is created and submitted for review. There is **no form** — no authored
config, no per-question answers, no uploads. That authored **application form** is the
`application-forms` follow-up spec (`spec_dd/0. drafts/application-forms`); the models and
seams it slots into are all defined here.

### 5.1 Submission models (DB-only, PII-bearing)

```
CourseApplication              # one applicant attempt for one course
  user        FK User
  course      FK Course
  state       FSMField(default="draft", protected=True)   # django-fsm-2
  created_at, updated_at, submitted_at, decided_at
  decided_by  FK User | None
  # Partial unique index: at most one ACTIVE application per (site, user, course),
  # where active = state in (draft, submitted, under_review, changes_requested).
  # Rejected/withdrawn/approved do not block re-application.
  # NOTE (forms spec): gains a `config FK ApplicationConfig` pinning the form version
  # at apply time, plus `answers` / file children. Left off here — there is no form yet.

ApplicationNote                # staff-only internal note (structurally distinct from applicant message)
  application FK, author FK User, body Text, created_at

ApplicationStateTransition     # audit log; one row per transition
  application FK (related_name="transitions")
  from_state, to_state, actor FK User | None
  message       TextField   # applicant-visible message (e.g. the request_changes reason)
  created_at
```

All inherit `SiteAwareModel`. The answer rows (`ApplicationAnswer`), the document row
(`ApplicationFile`, private storage + `scan_status` seam), and the `config` FK are added by
the `application-forms` spec — they are the form's data, and there is no form here. Keeping
`CourseApplication` standalone now means that follow-up is purely additive.

### 5.2 State machine (`django-fsm-2`)

Add the dependency: `uv add django-fsm-2` (confirmed absent today). `state` is an
`FSMField(protected=True)`. The full transition set ships now (it is the review workflow,
independent of whether a form exists), per `research_review_workflow.md §1.3`:

| Transition | Actor | source → target | Notes |
|---|---|---|---|
| `submit` | applicant | draft → submitted | no answer validation in this work; the `application-forms` spec adds required-answer validation here |
| `withdraw` | applicant | [draft, submitted, changes_requested] → withdrawn | |
| `pick_up` | reviewer | submitted → under_review | |
| `request_changes` | reviewer | under_review → changes_requested | **requires** an applicant-visible message |
| `resubmit` | applicant | changes_requested → submitted | re-submits (gains answer re-validation with the form) |
| `approve` | reviewer | under_review → approved | terminal; optional internal note |
| `reject` | reviewer | under_review → rejected | terminal; optional internal note |

- Reviewer transitions use `permission=` callables delegating to
  `user.has_perm("change_application", instance)` (django-guardian, object-level).
- Each transition writes an `ApplicationStateTransition` row and fires an
  `application_state_changed` Django signal (the seam for a future email backend; no
  notifications in v1).
- Templates drive buttons off `get_available_user_state_transitions(user)` — never hardcode
  state→button maps.
- `expired`/`waitlisted` and multi-reviewer are out of v1 (do not architect away — the
  audit row + signal already accommodate them).
- `request_changes` / `resubmit` ship now as part of the workflow even though, with no form,
  there is nothing for the applicant to edit between them; they become fully meaningful once
  the `application-forms` spec lands.

### 5.3 Apply entry (form-less)

- **Entry:** the gated-course CTA "Apply now" (from the access backend, §4.3) points at a
  **GET** confirmation page at `/applications/apply/<course_slug>/` — a redirect target the
  chokepoint (§4.4) can send the learner to. Confirming **POSTs** to the same route, which
  creates the learner's active `CourseApplication` for that course (or loads an existing active
  one) and calls `submit` so it lands in the reviewer inbox immediately. There is no form to
  render.
- **Confirmation:** the confirmation page ("Apply to <course>?") makes applying a deliberate
  action; on a successful POST it redirects to the applicant status page (§5.4).
- **Re-apply:** the active-application partial unique index (§5.1) means a learner with an
  in-flight application is taken to its status page instead of creating a duplicate; a
  previously rejected/withdrawn learner can apply again.
- **Seam:** the `application-forms` spec replaces this view's body — instead of
  create-and-submit, it resolves the course's config, creates a `draft`, and redirects into
  the multi-step form — without changing the CTA, the chokepoint, or the models below it.

### 5.4 Applicant status page

Plain-language current state; when `changes_requested`, show the reviewer's message (the
multi-step edit/resubmit affordance arrives with the `application-forms` spec); when
`approved`/`rejected`, show the decision and any applicant-visible message. A `withdraw`
action is available while the application is active. In-flight applications are reachable
from the learner's dashboard — surfaced there by the applications backend's
`get_dashboard_contributions` (§5.5), **not** by any `student_interface` import of this app.

### 5.5 `ApplicationCourseAccessBackend` — the applications plugin

`course_applications` ships the access backend that teaches the system about applications. It
**subclasses** `course_access.backends.DefaultCourseAccessBackend` (so `course_applications` →
`course_access` is the dependency direction — a leaf consuming its dependency; `course_access`
still never imports `course_applications`, §8). It is the default `COURSE_ACCESS_BACKEND` (§7),
so applications work out of the box.

- **`CourseAccessType`** — extends core's with the app's own value:
  `APPLICATION_GATED = "application_gated"`. This vocabulary lives here, not in core.
- **`validate_course_config`**: accept `access_type` ∈ {`free`, `application_gated`}; absent →
  `free`; reject anything else with `file_path` context; return the normalized dict. (Reuse the
  parent for the `free` case; this method's job is to additionally admit `application_gated`.)
- **`get_access`**: inherit the registered → "Continue" and `free` → "Start" branches from the
  parent; add the `application_gated` branch → `CTA("Apply now", apply url)`,
  `can_self_register=False`, `can_access_content=False`. Re-clicking "Apply now" with an in-flight
  application lands on the status page via the apply view's re-apply redirect (§5.3) — so a single
  "Apply now" label covers both first-time and returning applicants and matches the QA flow. (This
  backend *may* inspect application records for a richer label later; not required in v1.)
- **`get_dashboard_contributions(*, user)`**: query `get_active_applications(user)`; when
  non-empty, return one `DashboardContribution(template_name="course_applications/partials/
  dashboard_applications.html", context={"applications": <qs>})`; else `[]`. The partial — owned
  by this app — renders the in-flight-applications panel, each row linking to its status page.
- **`get_active_applications(user)`** lives in `course_applications/queries.py` (filter by `user`,
  active states; site isolation automatic). It is consumed by this backend, **not** by
  `student_interface` (the whole point of the seam).

A deployment that does not want applications simply leaves `COURSE_ACCESS_BACKEND` on the core
default backend; this class, its CTA, and its dashboard panel all vanish with it — nothing in a
core surface to remove.

## 6. Part C — Application review in the Django admin (v1)

In v1, reviewers action applications through the **Django admin**, to keep the first cut
simple. The bespoke educator-facing review UI (global inbox + full-screen single-application
screen in `educator_interface`) is deferred to the `application-review-ui` follow-up spec
(`spec_dd/0. drafts/application-review-ui`); everything below is UI-agnostic and that spec
reuses this machinery unchanged.

**Tradeoff (explicit):** reviewers must therefore be Django **staff** users in v1, not just
educators. Access is still constrained per-object (`django-guardian`) and per-site
(`SiteAwareModel`), so a staff reviewer only sees applications they hold `view_application` on,
on their own site. Lifting review out of the admin so non-staff educators / programme managers
can review is the entire point of the `application-review-ui` spec.

Admin classes live in `course_applications/admin.py` — the app owns its own admin, so review
introduces **no** `educator_interface → course_applications` dependency (§8). Per the
`fls:admin-interface` skill: Unfold `ModelAdmin`, object permissions via
`guardian.admin.GuardedModelAdmin` (mirroring `CohortAdmin`), `exclude = ["site"]`, with the
changelist queryset scoped and site-filtered.

- **Applications changelist = the v1 inbox.** `list_display` (applicant, course, state,
  submitted_at, decided_at), `list_filter` (state, course, submitted_at), `search_fields`
  (applicant name/email, course title), default ordering oldest-submitted-first. The changelist
  queryset is scoped via
  `get_objects_for_user(request.user, "view_application", klass=CourseApplication)` and inherits
  site isolation from `SiteAwareModel`.
- **State transitions in the admin.** Transitions are exposed as admin actions/buttons driven by
  `instance.get_available_user_state_transitions(request.user)` (never a hardcoded
  state→button map), each gated on `user.has_perm("change_application", instance)`.
  `request_changes` and `reject` use Django's **intermediate action page** to capture their
  required applicant-facing message / reason before the transition is applied; `approve` takes
  an optional internal note. Each transition writes its `ApplicationStateTransition` row and
  fires the `application_state_changed` signal exactly as in §5.2 — the admin is just another
  caller of the state machine. (`django-fsm-admin` with `FSM_ADMIN_FORCE_PERMIT = True` MAY
  back the buttons; if it doesn't compose cleanly with django-fsm-2 + Unfold, fall back to
  plain Unfold admin actions — the transition methods and permission callables are identical
  either way.)
- **Internal notes vs applicant-facing message stay separate.** `ApplicationNote` is a
  distinct, staff-only inline; the applicant-facing message lives on the
  `ApplicationStateTransition` (`request_changes`). They are never merged — no internal
  commentary leaks to applicants.
- **Transition log** is a read-only inline (`ApplicationStateTransition`, newest-first):
  who/when/from→to plus the applicant-visible message — the full audit trail on the change page.
- **Approval ≠ enrolment.** Approving never auto-creates a registration. The approved
  application surfaces an "Enrol this learner" link into the existing cohort +
  `UserCourseRegistration` flow with the user pre-filled (a URL hand-off, not a code
  dependency). Because admin/cohort enrolment bypasses the access backend (§4.4), the approved
  learner enrols cleanly.
- **No bulk approve/reject in v1.**

### Permissions

- Model permissions `view_application`, `change_application` on `CourseApplication`, granted
  object-level via `django-guardian` (`assign_perm`), mirroring the cohort pattern
  (`role_based_permissions/utils.py:179`). State-changing transitions delegate to
  `user.has_perm("change_application", instance)`; the admin changelist scopes to
  `get_objects_for_user(...)`. Reviewers must additionally be `is_staff` to reach the admin in
  v1 (removed by the `application-review-ui` spec).
- Site isolation is automatic via `SiteAwareModel` — Site A reviewers never see Site B
  applications.

## 8. Dependency direction (for the structure review)

- `course_access` → depends on `content_engine` (reads `Course`) and `student_management`
  (reads registration state). Must **not** depend on `course_applications` (no import, no URL
  reverse, no record inspection) — core has zero knowledge of applications.
- `student_interface` → depends on `course_access` **only** (for `get_course_access_backend()`,
  the CTA, `filter_visible`, and `get_dashboard_contributions`). It must **not** import
  `course_applications`. It renders the applications dashboard partial by *template name* via
  `render_to_string` of a `DashboardContribution` returned by the backend; Django template
  resolution is app-agnostic, so naming the partial is **not** an import edge.
- `course_applications` → depends on `course_access` (its `ApplicationCourseAccessBackend`
  subclasses `DefaultCourseAccessBackend` and uses its contract objects), and may read
  `content_engine.Course` and `student_management` enrolment primitives. It is otherwise a leaf;
  nothing else imports it. This `course_applications → course_access` arrow is the only new
  inter-app edge introduced by the access work, and it is acyclic (`course_access` does not import
  back).
- **Dashboard edge — avoided, not accepted.** An earlier draft had the `student_interface`
  dashboard import `course_applications.get_active_applications`. That `student_interface →
  course_applications` edge is **eliminated**: the dashboard panel is contributed by the active
  backend via `get_dashboard_contributions` (§4.3/§4.4/§5.5), and the application-aware code
  (`get_active_applications`, the panel partial) lives in `course_applications`, consumed by its
  own backend. Core surfaces name no concrete backend (goal §2.2).
- `course_applications` review lives in its **own** `admin.py` in v1, so it introduces **no**
  `educator_interface → course_applications` edge. The post-approval "enrol this learner"
  hand-off is a URL link into the existing cohort / registration flow, not a code dependency.
  (The deferred `application-review-ui` spec is what would add the
  `educator_interface → course_applications` edge — out of scope here.)
- **Load-time `access_config` validation — resolved via a settings hook, no edge.** An earlier
  draft had `save_course` (`content_engine`) import `get_course_access_backend()` directly. Since
  `course_access` already depends on `content_engine`, that would have closed a **dependency
  cycle** (`content_engine → course_access → content_engine`), not merely added an edge — so it is
  *not* the same shape as the leaf `icons` backend. Resolution: `content_engine` reads a
  `COURSE_ACCESS_CONFIG_VALIDATOR` setting (a dotted path, resolved via `import_string`, defaulting
  to `course_access.loader.validate_course_access_config`) and calls it at load time. A settings
  string is not an import edge, so `content_engine` gains **no** dependency on `course_access` and
  no cycle is introduced; `course_access → content_engine` stays the only arrow. The validator
  delegates to the backend's single `validate_course_config`, preserving author-time fail-loud
  validation. A future backend registers its own validator by swapping the setting — no
  `content_engine` change (goal §2.9). If the setting is unset, `access_config` is stored verbatim
  and the system check + `get_access` defensive validation remain the backstops.
- (The `application-forms` spec introduces one further loader-only edge —
  `content_engine → course_applications` for dispatching the `APPLICATION_CONFIG` content
  type — to confirm at that spec's structure review, not here.)

## 9. Key decisions & rationale

| Decision | Choice | Why |
|---|---|---|
| Per-course access config | opaque `access_config` JSONField on `Course`; backend owns schema/vocabulary/validation | core interprets nothing; a new backend stores its own keys with no core migration; satisfies "class + settings change" |
| Build the backend now? | yes — interface + loader + two backends (`free`-only core default + applications) | explicitly requested; known to need to grow to subscriptions/purchase; seam is the deliverable |
| Where application logic lives | the applications backend in `course_applications` (subclasses the core default), **not** core | **user decision** — core (`course_access`, `student_interface`) knows nothing of applications; `application_gated`, the Apply CTA, content gating, and the dashboard panel all move into the plugin; a deployment without applications drops them by switching `COURSE_ACCESS_BACKEND` |
| Who reads `access_config` | only the active backend | core default reads `access_type=free`; the applications backend additionally reads `application_gated`; a subscription backend ignores it entirely; keeps it backend-private config |
| Config validation | one `validate_course_config()`, called at load + system check + read | implement once, call as needed; a backend swap can't leave silently-invalid config |
| Display + behavior contract | backend-supplied `CourseAccessCTA` (label + url) + two enforcement booleans; no status enum | the backend fully specifies what the learner sees and what it does; new situations are new CTAs, needing no enumerated cases or caller changes |
| Dashboard contributions | backend-supplied `get_dashboard_contributions()` → `DashboardContribution`s rendered generically; default `[]` | **user decision** — the dashboard is a plugin surface like cards/detail; the active backend adds panels without any core surface naming or importing a concrete backend; eliminates the `student_interface → course_applications` edge |
| Separate `course_applications` app | yes | applications are PII submissions (and, later, admin-shaped form config); reusing `Form` tables would muddy content invariants, scoring, and PII isolation |
| Form **in front of** apply | deferred to `application-forms` spec | **user decision** — ship the access backend + bare Apply/review flow first; add the authored form later with no rearchitecture |
| State machine | `django-fsm-2`, `protected=True` | matches the subscriptions spec; greenfield; transition perms integrate with guardian; full review workflow is independent of whether a form exists |
| Audit | custom `ApplicationStateTransition` | lighter than `django-simple-history`; sufficient for who/when/what |
| Review UI home (v1) | **Django admin** (`course_applications/admin.py`) | **user decision** — keep v1 simple; the FSM, guardian scoping, notes, and audit log are all UI-agnostic, so the admin gives a filterable inbox + per-application actions for free, and keeps `course_applications` a leaf (no `educator_interface` edge). Cost: reviewers must be `is_staff`. The bespoke educator UI (correct audience, panel framework, non-staff reviewers) is deferred to `application-review-ui` |
| Approval vs enrolment | decoupled, with convenience hand-off | off-platform steps may intervene; matches serious admissions systems |

## 10. Risks & mitigations

- **Direct-comparison drift** (someone reads/branches on `course.access_config`, or hardcodes
  per-situation CTA logic outside the backend) — mitigate with explicit comments at the field
  + backend and a review check; callers use `decision.can_self_register` /
  `decision.can_access_content` / `decision.cta`, never the raw config. It is the one rule
  that keeps the future refactor cheap. The Django system check additionally catches
  `access_config` invalidated by a backend swap.
- **Loader cross-app edge** — load-time `access_config` validation does **not** add a
  `content_engine → course_access` edge: `content_engine` resolves the validator through the
  `COURSE_ACCESS_CONFIG_VALIDATOR` setting (`import_string`), so the only arrow stays
  `course_access → content_engine` (no cycle). See §8.
- **PII handling** — in this work an application carries only who-applied-to-what; document
  uploads, private storage, and permission-checked serving arrive with the `application-forms`
  spec, where the threat model (`/threat-model`) and security reviews should scrutinise the
  serve view and upload validation specifically.
- **Concurrent reviewer actions** — `FSMField` guards serialise transitions cleanly
  (second action fails on a now-invalid source state).

## 11. Out of scope — explicit deferral list (do not architect away)

conditional/branching questions · email notifications (seam: `application_state_changed`
signal) · in-app identity verification · virus scanning (seam: `ApplicationFile.scan_status`)
· EXIF stripping · in-UI `access_type` toggle.
