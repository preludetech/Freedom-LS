# Frontend QA: Course Access Types — Free & Application-Gated

Manual / Playwright-driven browser checks for this feature. Run after implementation.

## Setup

Start the dev server on a randomly-chosen free port (do **not** use the default runserver port):

```bash
PORT=$(.claude/fls/scripts/find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL: `http://127.0.0.1:$PORT`

All QA uses the **DemoDev** site. Required test data (create via the `fls:qa-data-helper` agent
if absent — never hand-build site-aware rows):

- A **free** course (no `access_config`, or `access_config: {access_type: free}`) on DemoDev.
- An **application-gated** course (`access_config: {access_type: application_gated}`) on DemoDev.
- A **learner** account (knows email + password) enrolled in nothing yet.
- A **staff reviewer** account: `is_staff=True`, holding object-level `view_application` +
  `change_application` on the gated course's applications on DemoDev (so the admin changelist is
  populated and transitions are permitted).

Log in at `http://127.0.0.1:$PORT/accounts/login/` (email + password). Admin is at
`http://127.0.0.1:$PORT/admin/`.

## Test 1 — Free course is unchanged (regression)

1. Log in as the learner. Go to the dashboard (`/`) and the all-courses list
   (`/courses/` — confirm the actual path from `student_interface/urls.py`).
2. Find the **free** course card.
   - **Expect:** the CTA reads **"Start"** (an enabled link/button), exactly as before this
     feature.
3. Click **Start**.
   - **Expect:** you are registered and land on the course home / first item — today's behaviour,
     no application flow.

## Test 2 — Gated course shows "Apply now"

1. As the learner, find the **application-gated** course card on the dashboard and on the
   all-courses list, and open its preview page.
   - **Expect:** the CTA reads **"Apply now"** (not "Start"), on the card, the preview modal, and
     the standalone preview page.
2. Confirm the gated course still shows its title/description (the preview is the funnel surface
   and stays visible).

## Test 3 — Self-registration chokepoint is enforced server-side

1. As the learner, attempt to hit the registration URL for the gated course **directly**
   (e.g. `/courses/<gated-slug>/register/` — confirm the real path from
   `student_interface/urls.py`).
   - **Expect:** you are **not** registered; you are redirected to the apply confirmation page
     (`/applications/apply/<gated-slug>/`) — the gate refuses self-registration.
2. Attempt to hit the gated course **content/home** URL directly
   (`/courses/<gated-slug>/` or the first-item URL).
   - **Expect:** course **content** is not shown; you see the preview / "Apply now" affordance
     instead.

## Test 4 — Applying to a gated course

1. As the learner, click **"Apply now"** on the gated course.
   - **Expect:** an "Apply to <course>?" confirmation page (no form fields to fill in).
2. Confirm/submit.
   - **Expect:** redirect to the **applicant status page** showing the application as
     **submitted** (plain-language state, brand-coloured status indicator).
3. Click **"Apply now"** for the same gated course a second time.
   - **Expect:** you are taken to the existing application's status page — **no duplicate**
     application is created.
4. Go to the learner **dashboard**.
   - **Expect:** the in-flight application is listed/linked and the link opens its status page.

## Test 5 — Reviewer actions in the Django admin

1. Log in as the **staff reviewer**. Go to `/admin/` → Course applications.
   - **Expect:** a changelist (the v1 inbox) showing the learner's application: applicant, course,
     state, submitted/decided columns. Confirm filtering by **state** and **course** works and
     search by applicant email works. Confirm only this reviewer's own-site applications appear.
2. Open the application. Use **pick up** (submitted → under review).
   - **Expect:** state becomes **under review**; a new row appears in the read-only transition-log
     inline (who/when/from→to).
3. Use **request changes** without entering a message.
   - **Expect:** the intermediate page re-renders with an error; **no** transition happens.
4. Use **request changes** with a message (e.g. "Please clarify your eligibility.").
   - **Expect:** state becomes **changes requested**; the transition log records the message;
     the message is stored as applicant-visible (not as an internal note).
5. Add an **internal note** via the notes inline and save.
   - **Expect:** the note is saved with the reviewer as author; it is **not** shown to the
     applicant (Test 6 confirms separation).

## Test 6 — Applicant sees the reviewer's message (not internal notes)

1. Log in as the **learner**, open the application status page.
   - **Expect:** state shows **changes requested**; the reviewer's applicant-facing message
     ("Please clarify your eligibility.") is shown; the internal note from Test 5 is **not**
     visible.
2. (If a withdraw affordance is offered while active) click **Withdraw**.
   - **Expect:** state becomes **withdrawn**; the withdraw action then disappears.

## Test 7 — Approval and the enrol hand-off (approval ≠ enrolment)

1. (Reset: as the learner, apply again — a withdrawn application does not block re-applying.)
   Then as the reviewer: **pick up** → **approve** the submitted application.
   - **Expect:** state becomes **approved**; transition logged with the reviewer as actor.
2. Confirm the learner is **not** auto-enrolled: as the learner, the gated course should still
   not show as registered until an explicit enrolment happens.
3. On the approved application's admin change page, find the **"Enrol this learner"** link.
   - **Expect:** it opens the existing cohort / registration flow with the **learner pre-filled**.
   Completing that enrolment registers the learner (admin/cohort enrolment bypasses the access
   gate), and the learner can then access the course content.

## Test 8 — Listings unchanged under the shipped backend

1. As the learner, compare the dashboard and all-courses lists before/after: the shipped
   applications backend inherits `filter_visible()` from the core default, which returns courses
   unchanged.
   - **Expect:** the same set of courses is listed as without the feature (no courses hidden).
