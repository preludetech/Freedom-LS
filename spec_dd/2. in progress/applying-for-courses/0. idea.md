# Course Access Types: Free & Application-Gated

This is the first concrete piece of the subscriptions / course-access backend
described in `spec_dd/0. drafts/03. subscriptions`.

We introduce an explicit, configurable **per-course access type**. Two types are
first-class from the start:

1. **Free** â€” a learner self-registers whenever they want. This is today's
   behaviour, made explicit.
2. **Application-gated** â€” a learner must apply, be reviewed, and be approved before
   an administrator enrols them. Any fee is handled **off-platform** (there is no
   on-platform payment in this work); "approval" is the on-platform mechanism, not a
   purchase.

The reason both types matter now: **free courses are the funnel that gets learners
to apply to application-gated ("paid") courses.** The cross-promotion CTA itself is
future work (see below), but free and application-gated must both be configurable
now so that funnel can be built on top later.

This idea stays high-level; the schema, URLs, templates and state-machine code are
worked out in the spec and plan that follow.

## Access type: making it configurable

- **One field on the course.** A course has exactly one access type at a time
  (`free` | `application_gated`). It is a property of the course, not a separate
  policy object â€” a single field is the right, minimal expression. Because courses
  are already site-aware, this is implicitly per-site: the same course content can
  be `free` on one site and `application_gated` on another.
- **`free` is the default**, so every existing course keeps behaving exactly as it
  does today and nothing needs back-filling beyond the field default.
- **The student-facing CTA branches on the type.** A `free` course shows
  "Start"/Register and self-enrols as today. An `application_gated` course shows
  "Apply" and routes the learner into the application flow instead of registering
  them.
- **The gate is enforced server-side, not just by hiding a button.** Self-registration
  funnels through a single entry point today; that one place must refuse to register a
  learner for an application-gated course and send them to the application flow
  instead. That single check does **not** compare `access_type` directly â€” it asks the
  course-access backend (see *Course-access backend architecture* below), so the same
  chokepoint keeps working unchanged when subscription/purchase backends arrive. Admin
  enrolment (educator interface, cohort registration) intentionally bypasses the gate â€”
  that is how an approved applicant ultimately gets in.
- **Decision: access type is set via content loading only** (for now). A course's
  `access_type` is configured through the content-loading pipeline â€” no
  educator-interface toggle and no Django-admin editing UI are built in this work. The
  model supports an in-UI flip later; that is deliberately deferred and only the (unbuilt)
  UI work differs.

## What an application needs to do

The rest of this idea is the application-gated path.

- **Per-course configuration.** Different application-gated courses ask for different
  things. A site admin configures, per course, which questions an applicant must
  answer â€” free text, a choice from options, a document upload (ID, proof of address,
  etc.). No engineer or code deploy should be required to change a course's questions.
- **Multi-step.** An application can be split across several steps so applicants are
  never shown a wall-of-text form. A single-step application is just a one-page
  application â€” the same model, configured with one step.
- **Drafts that survive.** Answers are saved as the applicant advances, and a draft
  survives logout and device switching, so an applicant can leave and come back days
  later. (This rules out session/cookie-based form wizards.)
- **Edit and correct.** Before submitting, an applicant can move back and forth between
  steps and fix mistakes (a typo, the wrong document) without losing data. After
  submitting, the application is locked â€” see the reviewer round-trip below.
- **Review and decision.** An administrator / educator reviews a submitted application,
  sees the answers and any uploaded documents together, and records a decision.
- **Approval is not enrolment.** Approving an application does **not** auto-enrol the
  learner â€” there may be off-platform steps first. After approval the educator enrols
  the learner separately, adding them to a cohort and registering them for the course
  in the usual way. The two actions stay fully decoupled.

## Application lifecycle

The research strongly converges on a small, well-tested state set. v1 uses:

```
draft â†’ submitted â†’ under_review
                 â†’ changes_requested â†” submitted
                 â†’ approved | rejected | withdrawn
```

- **draft** â€” the applicant is filling it in; the only mutable state for the applicant.
- **submitted** â€” locked from applicant edits, awaiting review.
- **under_review** â€” a reviewer has picked it up.
- **changes_requested** â€” a reviewer sent it back with a *required* applicant-facing
  message; the applicant fixes the flagged items and resubmits. This avoids the
  "reject and re-apply" pattern that loses context.
- **approved / rejected / withdrawn** â€” terminal. `withdrawn` lets an applicant
  abandon an application without leaving it pending.

State changes should be modelled with a proper state machine (the subscriptions spec
already adopts `django-fsm-2`; use the same library) and every transition should be
recorded for audit (who, when, what message). Note: the state machine governs the
*application lifecycle* only â€” the course **access type** is a static property, not a
state machine, and the two are independent concerns.

## Design direction (from research)

These are the convergent recommendations across the `research_*.md` files. They are
direction for the spec, not yet final:

- **A new app** (e.g. `course_applications`) owns the application models, rather than
  extending `content_engine.Form`. The existing `Form` is file-backed, scored, and
  counts toward course progress â€” none of which fits admin-built, unscored, PII-bearing
  applications. Reuse the *patterns* (page/question/option shape, the file-upload
  handler) without reusing the tables. The `access_type` field itself sits on `Course`
  in `content_engine`, since it is a property of the course.
- **Review UI lives in `educator_interface`.** Reviewers are educators / programme
  managers, not Django superusers. That app already houses the cohort, student and
  registration UI and already uses per-object `django-guardian` permissions. Provide a
  **global applications inbox** (filterable by course / status / date), not a per-course
  list â€” the lack of one is a recurring complaint in comparable systems.
- **File uploads are a question type**, configured per question (max size, MIME
  allowlist, count). Documents are PII: store them on a private path behind
  permission-checked access (not public media), validate by magic bytes rather than
  trusting the browser, and role-gate who can download them. In-app virus scanning and
  identity verification are out of scope for v1 but should not be architected out (e.g.
  leave room for a `scan_status`).
- **Multi-step UX**: one real URL per step (browser back / refresh / bookmark work), a
  labelled, all-steps-visible progress indicator, and a GOV.UK-style "check your
  answers" final step with per-section "change" links. HTMX is used *inside* a step
  (inline 422 validation, file upload, conditional reveal), not to swap whole steps.
- **Multi-tenancy & permissions**: all new models inherit `SiteAwareModel` so reviewers
  only ever see their own site's applications. Reviewer access is granted with
  object-level `django-guardian` permissions (`view_application`, `change_application`),
  mirroring the existing cohort scoping.
- **Notifications are out of scope for v1**, but each transition fires a signal so an
  email backend can be added later without touching the state machine.

## Course-access backend architecture

**Decision (changed from the original draft): we design and build the pluggable
course-access backend *now*.** Earlier this idea proposed deferring the abstraction and
shipping only "one field plus one `can_self_register()` function," with the backend
left as future work. We are reversing that. The two access types are the first two
cases of a mechanism that must later grow to cover subscriptions, per-course purchase,
and feature unlocks (see *Future work*), and we do not want a major rearchitecture of
the student interface when that happens. So **the seam is the architecture**: we build
the interface, the settings-based selection, the loader, and exactly one concrete
backend. We do **not** build any of the other backends now.

This is a deliberate, explicitly-requested exception to the usual conventions ("don't
build unrequested functionality," "don't add abstract base classes unless asked"). The
spec should treat the interface below as in scope and get it right.

### Shape â€” mirrors Django's email/task backends

- **Settings key `COURSE_ACCESS_BACKEND`** holds a dotted path to the active backend
  class, defaulting to the bundled default backend so out-of-the-box behaviour is
  exactly today's. A deployment swaps mechanisms by changing this one setting.
- **A loader/accessor** (e.g. `get_course_access_backend()`) resolves and instantiates
  the configured class via `import_string`, cached per process. Callers never import a
  concrete backend.
- **A base class `CourseAccessBackend`** defines the interface; new deployments
  subclass it. It and the loader live in a new low-level module/app
  `freedom_ls/course_access/`.
- **One concrete backend now â€” `DefaultCourseAccessBackend`** â€” implements exactly the
  two cases this spec ships (`free` and `application_gated`) by reading
  `Course.access_type`. It is the only backend built in v1.

### Interface â€” broad enough for the known futures

The interface must already answer the questions the listed future backends will ask, so
it is not retrofitted later. Illustrative signatures (final shape is the spec's job):

- **Per-(user, course) access decision** â€” the authoritative answer that both the
  student CTA and the server-side gate consume:
  `get_access(*, user, course) -> CourseAccessDecision`. The decision carries a
  `status` and a derived CTA (label + URL). The `status` enum is defined now with all
  the foreseeable cases â€” `REGISTERED | OPEN | REQUIRES_APPLICATION | REQUIRES_PURCHASE
  | REQUIRES_SUBSCRIPTION | LOCKED` â€” even though `DefaultCourseAccessBackend` only ever
  returns the first three. Defining the full enum now means a future backend and the
  templates that branch on `status` need no change.
- **Course visibility** â€” `filter_visible(*, user, courses) -> queryset`, so a future
  "subscription unlocks certain courses" backend can hide what a learner can't reach.
  The default backend returns the queryset unchanged (today's behaviour). The listing
  call sites (`get_all_courses()`, dashboard) route through this now, so the seam is
  real rather than notional.

Feature unlocks ("subscription unlocks live support chat") are **not** built now and
get no call site, but the backend object is their designated home: adding a
`has_feature(*, user, feature)` method later is a purely additive change to the base
class and default backend, not a rearchitecture. We note it here only to confirm it is
not architected out.

### Two rules that keep the future refactor cheap

- **`access_type` is the default backend's private config, not core truth.** The
  CharField stays on `Course`, but **only `DefaultCourseAccessBackend` may read it.** No
  view, template, or utility may compare `course.access_type == ...` directly; they all
  go through `get_access()`. A subscription backend is then free to ignore `access_type`
  entirely and use its own plan/price tables.
- **Every access question routes through the backend.** The single self-registration
  chokepoint asks the backend before creating a `UserCourseRegistration`; the CTA label
  and URL come from the same decision object. Admin and cohort enrolment deliberately
  bypass the backend, exactly as they bypass the gate today.

### Dependency direction (for the structure review)

`course_access` depends downward on `content_engine` (reads `Course`) and
`student_management` (reads registration state); `student_interface` depends on
`course_access`. `course_access` must **not** depend on `course_applications`: the
default backend distinguishes `REQUIRES_APPLICATION` from `REGISTERED` purely from
registration state, never by inspecting application records. This keeps the application
app a leaf and avoids a dependency cycle.

## Open question for the spec

**How applicant answers are stored is deliberately left open** and should be resolved
in the spec, because it depends on how per-course application forms are specified
(different courses have different requirements):

- *Relational answer tables* (one row per answer, FKs to question / options / file).
  Best for querying, redacting and deleting PII, FK integrity, and matching the existing
  `content_engine.Form` shape â€” but adding/removing a course's questions is a data change
  against a fixed schema.
- *JSON answers* (`{question_id: value}` on the application). No migrations when a course
  changes its question set, but weaker integrity and harder PII query/redaction.

The spec should pick one (or a hybrid â€” e.g. relational for documents/PII, JSON for
simple answers) once the per-course form-definition model is nailed down.

## Out of scope for v1 (don't architect away)

Flag these as deliberately deferred: on-platform payment; the freeâ†’application funnel
CTA (see future work); waitlists; application/intake windows and auto-expiry;
multi-reviewer assignment and rubric/score review; bulk approve/reject;
conditional/branching questions; email notifications; in-app identity verification and
virus scanning.

## Future work

### The free â†’ application funnel

The motivating use case for having both access types now is to drive learners from free
courses into application-gated ones ("you've finished this free course â€” apply to the
paid programme it leads into"). The seam already exists and should be kept clean:
`student_management.RecommendedCourse` is the natural home for such a CTA, and the
dashboard already surfaces recommended-course cards. The course-completion surface is a
secondary trigger point. **We do not build this CTA now** â€” only the two access types it
will sit on top of.

### Further course-access backends

The pluggable backend **architecture** is in scope for this spec (see *Course-access
backend architecture* above). What stays as future work is the set of additional
*concrete backends* that the architecture is being designed to admit without a
rearchitecture:

- subscriptions that unlock certain courses
- subscriptions that unlock all courses
- ways to purchase individual courses
- subscriptions that unlock certain functionality (e.g. live support chat)
- things not yet thought of

Each of these is a future `CourseAccessBackend` subclass selected via
`COURSE_ACCESS_BACKEND`, plus whatever config tables it needs (plans, prices). None are
built now. The job of this spec is only to ship `DefaultCourseAccessBackend` (`free` +
`application_gated`) behind an interface broad enough that the above slot in as new
classes and a settings change â€” never a refactor of the student interface.
