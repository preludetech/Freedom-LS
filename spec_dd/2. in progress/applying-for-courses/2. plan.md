# Implementation Plan: Course Access Types — Free & Application-Gated

Spec: `1. spec.md`. Read it first — this plan executes that spec and does not re-justify
its decisions.

This plan is **TDD-first**: for each behavioural unit, write a failing test, then implement.
Do not pre-write the whole test suite. Use `factory_boy` factories (never `.objects.create()`
on site-aware models in tests) and the `mock_site_context` fixture per the `fls:testing` and
`fls:multi-tenant` skills.

## Skills to consult (per phase)

- **`fls:multi-tenant`** — every new model extends `SiteAwareModel`; factories extend
  `SiteAwareFactory`; never set `site_id` manually; `CourseApplication`'s `(site, user, course)`
  unique constraint includes the **site column explicitly**.
- **`fls:testing`** — tests under `freedom_ls/<app>/tests/test_<module>.py`,
  `@pytest.mark.django_db`, `mock_site_context`, `client.force_login`,
  `reverse()` for URLs, no control flow in test bodies, order-independent.
- **`fls:template`** — pages extend `_base.html`; render the CTA through the existing
  `<c-button>`; `{% url %}` not hardcoded; `{% comment %}` not `{# #}`; run
  `npm run tailwind_build` after new Tailwind classes.
- **`fls:htmx`** — HTTP 422 on HTMX validation errors; detect via `HX-Request` header; never add
  per-request CSRF tokens.
- **`fls:frontend-styling`**, **`fls:icon-usage`**, **`brand-guidelines`** — Tailwind only;
  `<c-icon name="…">` not raw SVG; "learners" not "students"; plain-language error/state copy;
  brand colours for state badges (Forest=approved, Signal=rejected, Sand=changes-requested,
  Ocean=in-progress/submitted).

### Task A.2 — Pydantic schema field

  access_config: dict[str, Any] | None = Field(

  **Deliberate `Any` exception:** `access_config` is a genuinely opaque JSON blob, so

### Task A.3 — Backend contract objects + base class (`course_access/backends.py`)
Follow the `freedom_ls/icons/backend.py` precedent (`import_string` + `@functools.cache` +
base class + `DefaultXxxBackend`).

- Dataclasses (frozen):
  ```python
  @dataclass(frozen=True)
  class CourseAccessDecision:
      cta_label: str | None   # "Start", "Apply now", "Continue", …; None = no affordance
      cta_url: str | None     # None = not actionable
      can_self_register: bool
      can_access_content: bool

  @dataclass(frozen=True)
  class DashboardContribution:
      template_name: str   # a partial the contributing backend owns
      context: dict        # render-ready context for that partial; the dashboard never reads keys
  ```
- Base class:
  ```python
  class CourseAccessBackend:
      def get_access(self, *, user: "User", course: "Course") -> CourseAccessDecision: ...
      def filter_visible(self, *, user: "User", courses: "QuerySet[Course]") -> "QuerySet[Course]": ...
      def validate_course_config(self, raw: dict[str, Any], *, file_path: str = "") -> dict[str, Any]:
          """Validate & normalize this backend's slice of Course.access_config.
          Single implementation; called at content-load (fail loud), by a Django
          system check (after a backend swap), and defensively in get_access().
          Raise ValueError on invalid config."""
          raise NotImplementedError
      def get_dashboard_contributions(self, *, user: "User") -> list[DashboardContribution]:
          """Backend-owned panels for the learner dashboard. Default: nothing.
          The dashboard renders each generically and never names the backend — this is
          the seam that replaces a hard-coded student_interface -> course_applications import."""
          return []
  ```
  - `has_feature(*, user, feature)` is **intentionally not defined** — leave a documenting
    comment that it is the additive home for feature-unlock backends. Do not implement (spec §4.3,
    §11). Do not delete this comment.
  - `get_dashboard_contributions` ships now (the dashboard seam) with a concrete `[]` default on
    the base class, unlike `validate_course_config` (which `raise`s) — so a backend that adds no
    dashboard panels needs no override. Tests: base/default backend returns `[]`.
  - Type hints required (project convention). Use `TYPE_CHECKING` imports for `User`, `Course`,
    `QuerySet` to avoid import-time cycles.

### Task A.4 — Loader (`course_access/loader.py`)
```python
@functools.cache
def get_course_access_backend() -> CourseAccessBackend:
    return import_string(settings.COURSE_ACCESS_BACKEND)()
```
- Re-export `get_course_access_backend` from `course_access/__init__.py` for ergonomic imports,
  **but** guard against AppRegistryNotReady — keep the import inside the function if `__init__`
  import ordering bites (mirror how `icons` exposes its loader). Document the test caveat:
  callers using `override_settings(COURSE_ACCESS_BACKEND=…)` must call
  `get_course_access_backend.cache_clear()` (same note as `get_icon_backend`).
- **Module-level validator hook target** — the function `content_engine` resolves via
  `settings.COURSE_ACCESS_CONFIG_VALIDATOR` (Task 0.3) so that `content_engine` validates
  `access_config` at load time **without importing `course_access`** (Task A.7):
  ```python
  def validate_course_access_config(raw: dict[str, Any], *, file_path: str = "") -> dict[str, Any]:
      """Hook target for content_engine's COURSE_ACCESS_CONFIG_VALIDATOR.
      Delegates to the active backend's single validate_course_config() — no
      duplicate validation logic lives here."""
      return get_course_access_backend().validate_course_config(raw, file_path=file_path)
  ```
  This is a thin delegator: the one validation implementation stays on the backend (A.5), reused
  by load time (A.7 via this hook), the system check (A.6), and `get_access` (A.5). Same
  documented `Any`-on-opaque-blob exception as A.2/A.3.

### Task A.5 — `DefaultCourseAccessBackend` + `CourseAccessType` (core: `free`/registered only)
Core's `DefaultCourseAccessBackend` knows **nothing** about applications. `application_gated`,
the Apply CTA, and the dashboard panel all belong to the applications backend (Task B.6), which
subclasses this one. In `course_access/backends.py` (core's reader of `access_config`):
```python
class CourseAccessType(models.TextChoices):
    FREE = "free", _("Free")
    # application_gated is NOT a core value — the applications backend extends this.
```
- `validate_course_config(raw, *, file_path="")`:
  - accept only the key `access_type`; reject unknown keys with `ValueError` carrying
    `file_path` context (consistent with existing content-validation messages).
  - value must be in `CourseAccessType.values` (i.e. only `free`); absent → default `free`.
    **`application_gated` is rejected by the core default backend** — authoring it requires the
    applications backend. (The system check, A.6, catches `application_gated` config left over
    after a swap back to the default.)
  - return `{"access_type": <value>}` (normalized).
- `get_access(*, user, course)` — pseudocode:
  ```
  try:
      access_type = self.validate_course_config(course.access_config)["access_type"]
  except ValueError:
      return CourseAccessDecision(cta_label=None, cta_url=None, can_self_register=False, can_access_content=False)

  if is_registered_for_course(user, course):   # direct OR cohort, see below
      return CourseAccessDecision(
          cta_label="Continue",
          cta_url=reverse("student_interface:course_home", kwargs={"course_slug": course.slug}),
          can_self_register=False, can_access_content=True)

  # access_type == CourseAccessType.FREE (the only remaining core value)
  return CourseAccessDecision(
      cta_label="Start",
      cta_url=reverse("student_interface:register_for_course", kwargs={"course_slug": course.slug}),
      can_self_register=True, can_access_content=False)
  ```
  (Note: `reverse()` takes `kwargs=`, **not** a bare positional slug — the second positional arg
  is `urlconf`. Do not copy `reverse(name, slug)` shorthand literally.)
  - **No `application_gated` branch and no `reverse("course_applications:apply")`** — the core
    default backend never references `course_applications` (no URL, no records, no import), keeping
    `course_access` fully independent of it. The applications branch is added by the subclass (B.6).
  - **`is_registered_for_course(user, course)` — extract, don't duplicate.** Today the
    direct-or-cohort registration check lives only in `student_interface.utils.get_is_registered`
    (`utils.py:149`). `course_access` cannot import it (cycle: `student_interface` → `course_access`).
    Per the spec's own note ("if the shared check is worth extracting, it belongs in
    `student_management`"), extract it to **`student_management/utils.py`** as
    `is_registered_for_course(user, course) -> bool` (querying `UserCourseRegistration` +
    `CohortCourseRegistration`, guarding `user.is_authenticated`). Then **update**
    `student_interface.get_is_registered` to delegate to it, and call it from the backend. This is
    additive, breaks no cycle (`student_management` depends on neither), and satisfies CLAUDE.md
    "avoid repeating code". Do **not** inline a second copy in `course_access`.
    - **Cross-ref to A.8:** `is_registered_for_course` is for the **backend** (the default backend
      uses it to derive `can_access_content`). It is **not** what `student_interface` content
      surfaces consult — those consult the seam (`decision.can_access_content`, A.8). Keep the two
      distinct: extraction feeds the backend; the seam is what callers read. `get_course_index` no
      longer calls `get_is_registered` for content gating (A.8) — it receives `can_access_content`.
- `filter_visible(*, user, courses)`: return `courses` unchanged.
- `get_dashboard_contributions(*, user)`: inherits the base `[]` (no override needed) — today's
  dashboard, unchanged.
- Tests (TDD): unknown key → `ValueError`; absent `access_type` → `free`; bad value (incl.
  `application_gated`) → `ValueError`. `get_access` for: anonymous on free → Start/self-register;
  registered → Continue/content; invalid config → safe no-action decision.
  `get_dashboard_contributions` → `[]`. Use `mock_site_context` + factories. (The
  `application_gated` CTA + dashboard panel are tested in B.6 against the applications backend.)

### Task A.7 — Load-time validation in `save_course` (via the settings hook, no `course_access` import)
- In `content_engine/management/commands/content_save.py`, `save_course` (~line 342), before
  `save_with_uuid(...)`, resolve the validator from settings and call it — **do not** import
  `course_access`:
  ```python
  validator_path = getattr(settings, "COURSE_ACCESS_CONFIG_VALIDATOR", None)
  if validator_path:
      validate_access_config = import_string(validator_path)
      item.access_config = validate_access_config(
          item.access_config or {}, file_path=str(item.file_path)
      )
  ```
  (`from django.conf import settings` and `from django.utils.module_loading import import_string`
  — match however `content_save.py` already imports them; `import_string` is the same idiom the
  `icons` backend uses.) The hook target is `course_access.loader.validate_course_access_config`
  (Task A.4), which delegates to the active backend's single `validate_course_config`. So the
  normalized dict is what gets stored, and bad config is rejected at author time (fail loud).
- **Why the hook, not a direct import:** `course_access` already depends on `content_engine`
  (it reads `Course`, spec §8). A direct `content_engine → course_access` import here would close
  a dependency **cycle** (`content_engine → course_access → content_engine`) — not merely a new
  edge. (It would only avoid an `ImportError` because both sides defer their imports; the cycle is
  still real in the dependency graph and is exactly what `/plan_structure_review` rejects.) The
  `icons` precedent does **not** cover this: `icons` is a leaf, so `content_engine → icons` is
  acyclic. Reading a settings-configured dotted path inverts the dependency: `content_engine`
  never imports `course_access`; the settings string is the only seam. A settings reference is not
  an import edge, so **no new `content_engine` arrow is added to `docs/app_structure.md`** and no
  cycle is introduced. This also advances spec goal §2.7 — a future backend brings its own config
  keys and registers its own validator by swapping `COURSE_ACCESS_CONFIG_VALIDATOR`, with zero
  `content_engine` change.
- **Degrades safely:** if `COURSE_ACCESS_CONFIG_VALIDATOR` (or `COURSE_ACCESS_BACKEND`) is unset,
  `save_course` stores `access_config` verbatim (still opaque); the system check (A.6) +
  defensive validation in `get_access` (A.5) remain the backstops.
- Test: with the default settings, loading a course with
  `access_config: {access_type: application_gated}` persists `{"access_type": "application_gated"}`;
  loading with an unknown key raises `ValueError` with the file path in the message. Add a test
  that with `COURSE_ACCESS_CONFIG_VALIDATOR` unset (`override_settings`), `access_config` is stored
  verbatim and no validation runs.

> **Structure concern — RESOLVED (settings hook, was open for `/plan_structure_review`).** The
> earlier draft of this task imported `get_course_access_backend` directly into `save_course`,
> introducing `content_engine --> course_access`. Because `course_access` already depends on

> `COURSE_ACCESS_CONFIG_VALIDATOR` setting (Task 0.3) resolved via `import_string`.

- **Content-access gate (item view + TOC + landing)** — content reachability gates on the single
  boolean `decision.can_access_content` at **every** content surface, not just `course_home`.
  Compute `decision = get_course_access_backend().get_access(user=request.user, course=course)`
  **once per request in the view layer** and pass `decision.can_access_content` down — keep the
  backend call out of `utils.py` so `get_access` runs once per request.
  **Rule:** content *access enforcement* (may this user view item content / see unlocked TOC items)
  → `decision.can_access_content`. The headline *affordance* (what to click to gain access) →
  `decision.cta_label`/`decision.cta_url`. `is_registered` is retained **only** for registration-relationship display
  (resume position, progress %, the "Register for course" form state) — never to decide whether
  content is reachable. Do **not** add a separate `get_is_registered()` / "and not registered"
  check — the backend folds registration into `can_access_content` (spec §4.4), and a content-
  granting backend (subscription/feature-unlock) could grant access without enrolment.
  - **`view_course_item` (`views.py:254`) — the hard redirect gate.** After resolving `course`
    (before rendering the item), `if not decision.can_access_content:` `redirect(
    "student_interface:course_preview", course_slug=course_slug)`. This view is `@login_required`.
    - **Deliberate behaviour change to flag:** today an unregistered learner can view a *free*
      course's item content by guessing the URL (the TOC hides the links as BLOCKED, but the URL
      works). With the default backend `can_access_content == is_registered`, so item URLs now
      require registration — closing the hole and aligning the URL guard with the existing TOC

## Phase B — `course_applications/`

### Task B.1 — Submission model (`course_applications/models.py`)
Extends `SiteAwareModel` (UUID pk + `site` FK). **No `state` field, no `django-fsm-2`, no audit/
notes models** — those are added by the `application-review-ui` spec.
```python
class CourseApplication(SiteAwareModel):
    user        = models.ForeignKey(settings.AUTH_USER_MODEL, on_delete=models.CASCADE, related_name="course_applications")
    course      = models.ForeignKey("freedom_ls_content_engine.Course", on_delete=models.CASCADE, related_name="applications")
    created_at  = models.DateTimeField(auto_now_add=True)
    updated_at  = models.DateTimeField(auto_now=True)

    # NOTE (review spec — application-review-ui): gains `state = FSMField(protected=True)`, the
    #   submit/withdraw/pick_up/request_changes/resubmit/approve/reject transitions,
    #   submitted_at/decided_at/decided_by, the view_application/change_application permissions,
    #   ApplicationNote + ApplicationStateTransition, the application_state_changed signal, and the
    #   active-state PARTIAL unique index that REPLACES the plain constraint below.
    # NOTE (forms spec — application-forms): gains `config FK ApplicationConfig` + answer/file children.
    # Do not architect these away — leave this model standalone and additive.

    class Meta:
        constraints = [
            models.UniqueConstraint(
                fields=["site", "user", "course"],
                name="unique_application_per_site_user_course",
            )
        ]
```
- **No custom permissions** in this spec (the reviewer `view_application`/`change_application`
  codenames are declared by the review spec). Django's auto-created defaults are sufficient.
- **Plain unique constraint** (one application per `(site, user, course)`, site column explicit per
  `fls:multi-tenant`). The review spec drops this and adds the active-state partial index once
  withdraw/approve/reject exist.
- Test: a second `CourseApplication` for the same `(site, user, course)` → `IntegrityError`
  (use `mock_site_context` + factories).

### Task B.4 — Apply entry view (form-less) + URLs (`course_applications/views.py`, `urls.py`)
