# Implementation Plan: Course Access Types ŌĆö Free & Application-Gated

Spec: `1. spec.md`. Read it first ŌĆö this plan executes that spec and does not re-justify
its decisions.

This plan is **TDD-first**: for each behavioural unit, write a failing test, then implement.
Do not pre-write the whole test suite. Use `factory_boy` factories (never `.objects.create()`
on site-aware models in tests) and the `mock_site_context` fixture per the `fls:testing` and
`fls:multi-tenant` skills.

## Skills to consult (per phase)

- **`fls:multi-tenant`** ŌĆö every new model extends `SiteAwareModel`; factories extend
  `SiteAwareFactory`; never set `site_id` manually; the active-application uniqueness must be a
  **partial DB index that includes the site column explicitly**.
- **`fls:admin-interface`** ŌĆö `course_applications/admin.py`: `GuardedModelAdmin` does **not**
  inherit `SiteAwareModelAdmin`, so set `exclude = ["site"]` manually; Unfold `TabularInline` /
  `StackedInline` (never Django's built-ins); `autocomplete_fields` for `user`/`course`;
  `readonly_fields` for timestamps + transition fields.
- **`fls:testing`** ŌĆö tests under `freedom_ls/<app>/tests/test_<module>.py`,
  `@pytest.mark.django_db`, `mock_site_context`, factory traits per state, `client.force_login`,
  `reverse()` for URLs, no control flow in test bodies, order-independent.
- **`fls:template`** ŌĆö pages extend `_base.html`; render the CTA through the existing
  `<c-button>`; `{% url %}` not hardcoded; `{% comment %}` not `{# #}`; run
  `npm run tailwind_build` after new Tailwind classes.
- **`fls:htmx`** ŌĆö HTTP 422 on HTMX validation errors; detect via `HX-Request` header; never add
  per-request CSRF tokens.
- **`fls:frontend-styling`**, **`fls:icon-usage`**, **`brand-guidelines`** ŌĆö Tailwind only;
  `<c-icon name="ŌĆ”">` not raw SVG; "learners" not "students"; plain-language error/state copy;
  brand colours for state badges (Forest=approved, Signal=rejected, Sand=changes-requested,
  Ocean=in-progress/submitted).

### Task A.3 ŌĆö Backend contract objects + base class (`course_access/backends.py`)
Follow the `freedom_ls/icons/backend.py` precedent (`import_string` + `@functools.cache` +
base class + `DefaultXxxBackend`).

- Dataclasses (frozen):
  ```python
  @dataclass(frozen=True)
  class CourseAccessCTA:
      label: str        # "Start", "Apply now", "Continue", ŌĆ”
      url: str | None   # None = not actionable

  @dataclass(frozen=True)
  class CourseAccessDecision:
      cta: CourseAccessCTA | None
      can_self_register: bool
      can_access_content: bool

  @dataclass(frozen=True)
  class DashboardContribution:
      template_name: str   # a partial the contributing backend owns
      context: dict        # render-ready context for that partial; the dashboard never reads keys
  ```
- Base class:
  ```python
  class CourseAccessBackend:
      def get_access(self, *, user: "User", course: "Course") -> CourseAccessDecision: ...
      def filter_visible(self, *, user: "User", courses: "QuerySet[Course]") -> "QuerySet[Course]": ...
      def validate_course_config(self, raw: dict[str, Any], *, file_path: str = "") -> dict[str, Any]:
          """Validate & normalize this backend's slice of Course.access_config.
          Single implementation; called at content-load (fail loud), by a Django
          system check (after a backend swap), and defensively in get_access().
          Raise ValueError on invalid config."""
          raise NotImplementedError
      def get_dashboard_contributions(self, *, user: "User") -> list[DashboardContribution]:
          """Backend-owned panels for the learner dashboard. Default: nothing.
          The dashboard renders each generically and never names the backend ŌĆö this is
          the seam that replaces a hard-coded student_interface -> course_applications import."""
          return []
  ```

    comment that it is the additive home for feature-unlock backends. Do not implement (spec ┬¦4.3,
    ┬¦11). Do not delete this comment.
  - `get_dashboard_contributions` ships now (the dashboard seam) with a concrete `[]` default on
    the base class, unlike `validate_course_config` (which `raise`s) ŌĆö so a backend that adds no
    dashboard panels needs no override. Tests: base/default backend returns `[]`.
  - Type hints required (project convention). Use `TYPE_CHECKING` imports for `User`, `Course`,

### Task A.5 ŌĆö `DefaultCourseAccessBackend` + `CourseAccessType` (core: `free`/registered only)
Core's `DefaultCourseAccessBackend` knows **nothing** about applications. `application_gated`,
the Apply CTA, and the dashboard panel all belong to the applications backend (Task B.6), which
subclasses this one. In `course_access/backends.py` (core's reader of `access_config`):
```python
class CourseAccessType(models.TextChoices):
    FREE = "free", _("Free")
    # application_gated is NOT a core value ŌĆö the applications backend extends this (Task B.6).
```
- `validate_course_config(raw, *, file_path="")`:
  - accept only the key `access_type`; reject unknown keys with `ValueError` carrying
    `file_path` context (consistent with existing content-validation messages).
  - value must be in `CourseAccessType.values` (i.e. only `free`); absent ŌåÆ default `free`.
    **`application_gated` is rejected by the core default backend** ŌĆö authoring it requires the
    applications backend. (The system check, A.6, catches `application_gated` config left over
    after a swap back to the default.)
  - return `{"access_type": <value>}` (normalized).
- `get_access(*, user, course)` ŌĆö pseudocode:
  ```
  try:
      access_type = self.validate_course_config(course.access_config)["access_type"]
  except ValueError:
      return CourseAccessDecision(cta=None, can_self_register=False, can_access_content=False)

  if is_registered_for_course(user, course):   # direct OR cohort, see below
      return CourseAccessDecision(
          cta=CourseAccessCTA("Continue", reverse("student_interface:course_home", kwargs={"course_slug": course.slug})),
          can_self_register=False, can_access_content=True)

  # access_type == CourseAccessType.FREE (the only remaining core value)
  return CourseAccessDecision(
      cta=CourseAccessCTA("Start", reverse("student_interface:register_for_course", kwargs={"course_slug": course.slug})),
      can_self_register=True, can_access_content=False)
  ```
  (Note: `reverse()` takes `kwargs=`, **not** a bare positional slug ŌĆö the second positional arg
  is `urlconf`. Do not copy `reverse(name, slug)` shorthand literally.)
  - **No `application_gated` branch and no `reverse("course_applications:apply")`** ŌĆö the core
    default backend never references `course_applications` (no URL, no records, no import), keeping
    `course_access` fully independent of it. The applications branch is added by the subclass (B.6).
  - **`is_registered_for_course(user, course)` ŌĆö extract, don't duplicate.** Today the
    direct-or-cohort registration check lives only in `student_interface.utils.get_is_registered`
    (`utils.py:149`). `course_access` cannot import it (cycle: `student_interface` ŌåÆ `course_access`).
    Per the spec's own note ("if the shared check is worth extracting, it belongs in
    `student_management`"), extract it to **`student_management/utils.py`** as
    `is_registered_for_course(user, course) -> bool` (querying `UserCourseRegistration` +
    `CohortCourseRegistration`, guarding `user.is_authenticated`). Then **update**
    `student_interface.get_is_registered` to delegate to it, and call it from the backend. This is
    additive, breaks no cycle (`student_management` depends on neither), and satisfies CLAUDE.md
    "avoid repeating code". Do **not** inline a second copy in `course_access`.
- `filter_visible(*, user, courses)`: return `courses` unchanged.
- `get_dashboard_contributions(*, user)`: inherits the base `[]` (no override needed) ŌĆö today's
  dashboard, unchanged.
- Tests (TDD): unknown key ŌåÆ `ValueError`; absent `access_type` ŌåÆ `free`; bad value (incl.
  `application_gated`) ŌåÆ `ValueError`. `get_access` for: anonymous on free ŌåÆ Start/self-register;
  registered ŌåÆ Continue/content; invalid config ŌåÆ safe no-action decision.
  `get_dashboard_contributions` ŌåÆ `[]`. Use `mock_site_context` + factories. (The
  `application_gated` CTA + dashboard panel are tested in B.6 against the applications backend.)

### Task A.7 ŌĆö Load-time validation in `save_course` (via the settings hook, no `course_access` import)
- In `content_engine/management/commands/content_save.py`, `save_course` (~line 342), before
  `save_with_uuid(...)`, resolve the validator from settings and call it ŌĆö **do not** import
  `course_access`:
  ```python
  validator_path = getattr(settings, "COURSE_ACCESS_CONFIG_VALIDATOR", None)
  if validator_path:
      validate_access_config = import_string(validator_path)
      item.access_config = validate_access_config(
          item.access_config or {}, file_path=str(item.file_path)
      )
  ```
  (`from django.conf import settings` and `from django.utils.module_loading import import_string`
  ŌĆö match however `content_save.py` already imports them; `import_string` is the same idiom the
  `icons` backend uses.) The hook target is `course_access.loader.validate_course_access_config`
  (Task A.4), which delegates to the active backend's single `validate_course_config`. So the
  normalized dict is what gets stored, and bad config is rejected at author time (fail loud).
- **Why the hook, not a direct import:** `course_access` already depends on `content_engine`
  (it reads `Course`, spec ┬¦8). A direct `content_engine ŌåÆ course_access` import here would close
  a dependency **cycle** (`content_engine ŌåÆ course_access ŌåÆ content_engine`) ŌĆö not merely a new
  edge. (It would only avoid an `ImportError` because both sides defer their imports; the cycle is
  still real in the dependency graph and is exactly what `/plan_structure_review` rejects.) The
  `icons` precedent does **not** cover this: `icons` is a leaf, so `content_engine ŌåÆ icons` is
  acyclic. Reading a settings-configured dotted path inverts the dependency: `content_engine`
  never imports `course_access`; the settings string is the only seam. A settings reference is not
  an import edge, so **no new `content_engine` arrow is added to `docs/app_structure.md`** and no
  cycle is introduced. This also advances spec goal ┬¦2.9 ŌĆö a future backend brings its own config
  keys and registers its own validator by swapping `COURSE_ACCESS_CONFIG_VALIDATOR`, with zero
  `content_engine` change.
- **Degrades safely:** if `COURSE_ACCESS_CONFIG_VALIDATOR` (or `COURSE_ACCESS_BACKEND`) is unset,
  `save_course` stores `access_config` verbatim (still opaque); the system check (A.6) +
  defensive validation in `get_access` (A.5) remain the backstops.
- Test: with the default settings, loading a course with
  `access_config: {access_type: application_gated}` persists `{"access_type": "application_gated"}`;
  loading with an unknown key raises `ValueError` with the file path in the message. Add a test
  that with `COURSE_ACCESS_CONFIG_VALIDATOR` unset (`override_settings`), `access_config` is stored
  verbatim and no validation runs.

### Task B.2 ŌĆö State machine + signal (`course_applications/models.py` + `signals.py`)
`state` is `FSMField(protected=True)`. Define an `application_state_changed` Django signal in
`course_applications/signals.py`. Transitions (spec ┬¦5.2):

| method | source ŌåÆ target | actor / notes |
|---|---|---|
| `submit` | draft ŌåÆ submitted | applicant; set `submitted_at` |
| `withdraw` | [draft, submitted, changes_requested] ŌåÆ withdrawn | applicant |
| `pick_up` | submitted ŌåÆ under_review | reviewer (`change_application`) |
| `request_changes` | under_review ŌåÆ changes_requested | reviewer; **requires** applicant message |
| `resubmit` | changes_requested ŌåÆ submitted | applicant |
| `approve` | under_review ŌåÆ approved | reviewer; terminal; set `decided_at`/`decided_by` |
| `reject` | under_review ŌåÆ rejected | reviewer; terminal; set `decided_at`/`decided_by` |

- Use `@transition(field=state, source=ŌĆ”, target=ŌĆ”, permission=<callable>)`. Reviewer
  transitions: `permission=lambda instance, user: user.has_perm("change_application", instance)`.
- **One DRY audit helper, with the `from_state` captured *before* the transition fires.** By the
  time a `@transition`-decorated method body runs, `django-fsm-2` has already flipped
  `self.state`, so the source state is gone. Therefore the **call site** (view/admin action)
  captures it first:
  ```python
  with transaction.atomic():
      from_state = application.state          # capture BEFORE
      application.submit(actor=request.user)  # FSM flips state, sets submitted_at, etc.
      application.save()
      application._record_transition(from_state=from_state, to_state=application.state,
                                     actor=request.user, message=message)
  ```
  The single helper:
  ```python
  def _record_transition(self, *, from_state: str, to_state: str,
                         actor: "User | None", message: str = "") -> None:
      # NOTE: site-aware ŌĆö ApplicationStateTransition.save() reads the site from the
      # active request thread-local, so this must run inside a request/admin action
      # (or a mock_site_context in tests). Documented dependency, not a bug.
      ApplicationStateTransition.objects.create(
          application=self, from_state=from_state, to_state=to_state,
          actor=actor, message=message)
      application_state_changed.send(sender=type(self), application=self,
                                     from_state=from_state, to_state=to_state, actor=actor)
  ```
  All seven transitions log + signal identically through this one helper; no per-transition
  copies. The transition methods themselves only mutate FSM-adjacent fields (`submitted_at`,
  `decided_at`/`decided_by`). Wrapping flip + save + record in `transaction.atomic()` at the call
  site keeps the state change and its audit row atomic.
- **Wire the signal:** `course_applications/apps.py` `ready()` imports `signals` (Task 0.2) so
  `application_state_changed` is registered for future subscribers even though v1 has no listener.
- `request_changes(self, *, actor, message)` and `reject(self, *, actor, message=ŌĆ”)` carry the
  message into the audit row. `approve` takes an optional internal note ŌåÆ an `ApplicationNote`,
  not a transition message.
- Use `instance.get_available_user_state_transitions(user)` (django-fsm-2 API) to drive buttons ŌĆö
  never hardcode stateŌåÆbutton maps (used by both the status page and the admin).
- Tests (TDD, one per transition): legal transition flips state + writes exactly one audit row +
  fires the signal once; illegal source raises `TransitionNotAllowed`; reviewer transition without
  `change_application` perm is refused; `request_changes` with empty message raises (validation).
  Use `time_machine` for `submitted_at`/`decided_at` assertions per `fls:testing`.

### Task B.3 ŌĆö Guardian permissions (mirror the cohort role pattern)
The "cohort pattern" the spec points at (`role_based_permissions/utils.py`) is **not** raw
`assign_perm`; it is `assign_object_role(user, target, role)` ŌåÆ creates an `ObjectRoleAssignment`
ŌåÆ `sync_user_object_permissions` translates the role's configured permission set (filtered to the
target's content type) into guardian grants. **Use that machinery, not ad-hoc `assign_perm`**, so
applications participate in the same role/permission model as cohorts:

- Add a reviewer role to the role config whose permission set includes
  `view_application` + `change_application`. `_filter_perms_for_content_type` already restricts
  grants to the object's content type, so once the role lists those codenames,
  `assign_object_role(reviewer, application, role="<reviewer_role>")` grants exactly the two
  object-level perms on that `CourseApplication`.
- **Grant trigger (one approach, chosen):** assign the role on the `submit`/`resubmit` transition
  (in the same atomic block as the audit row), to the users who already hold the course's
  reviewer/educator role on that site.
- **Genuinely open ŌĆö surface for `/plan_structure_review` + the user, do not invent:** there is no
  existing model edge linking a *course* to its set of reviewers, so "who are this course's
  reviewers" is not pinned by the spec. Flag this explicitly; if no answer is available at
  implementation time, the minimal correct fallback is to grant on submit to site **staff**
  holding the reviewer role (matching the v1 "reviewers must be `is_staff`" constraint, ┬¦6) and
  leave a `# TODO` for per-course reviewer scoping. Do not silently pick a courseŌåöreviewer mapping.
- Test: a reviewer with `change_application` on an application can `pick_up`/`approve`; a reviewer
  without it cannot; a Site-B reviewer never sees a Site-A application
