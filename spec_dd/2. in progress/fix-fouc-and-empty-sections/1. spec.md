# Spec: Fix FOUC on Alpine modals and hide empty dashboard sections

Two related bugs that share the theme "things rendering when they shouldn't." Background and root-cause analysis already live in `research_fouc_and_empty_states.md` next to this spec; this spec captures decisions and success criteria.

## Why this matters

- The `c-picture` zoom modal flashes fullscreen on every page load that includes a picture. It is a visible, jarring regression of perceived quality on what is otherwise a polished course-content page.
- Empty `Recommended Courses` and `Learning History` sections render as a heading plus a faint "no … yet" line on every dashboard load for users without recommendations or completed courses. The heading-with-nothing-under-it is noise for sighted users and disorienting for screen-reader users (the heading is announced, then the user is dropped into nothing meaningful).

Both fix the same underlying class of problem: the server is sending HTML that says "render this," when the correct answer is "don't render this at all (or render it hidden)."

## Scope

### Part A — FOUC fix for Alpine `x-cloak`

1. Add a global CSS rule
   ```css
   @layer base {
     [x-cloak] { display: none !important; }
   }
   ```
   to `tailwind.input.css` (or to `tailwind.components.css`, which `tailwind.input.css` already imports — pick whichever fits the project's existing layer conventions). The rule must land in the built `static/vendor/tailwind.output.css`.
2. Run `npm run tailwind_build` and verify the rule survives the build.
3. Audit every `x-show` usage in the project's templates. For each one, confirm that either:
   - the same element carries `x-cloak`, or
   - the element is otherwise guaranteed to be `display: none` at HTML parse time (e.g. an inline `style="display: none"`), or
   - the element is intentionally visible at parse time (e.g. it starts in the open state) — in which case explicitly call this out in the spec/PR description.

   The current audit (from the research file plus a fresh grep) shows:
   - `freedom_ls/content_engine/templates/cotton/picture.html` — outer `x-show="open"` on the modal overlay has `x-cloak`. The fix is purely the missing CSS rule.
   - `freedom_ls/base/templates/_base_interface.html` — `x-show` elements (`backdropVisible`, `sidebarOpen`) carry `x-cloak`.
   - `freedom_ls/base/templates/cotton/modal.html` — the outer wrapper carries `x-cloak`. There are nested `x-show="open"` elements (the overlay container, the backdrop, and the panel) that do **not** individually carry `x-cloak`. This is acceptable because the cloaked outer wrapper hides the entire subtree at parse time, so the nested elements never render before Alpine boots. No change needed.
   - `freedom_ls/student_interface/templates/student_interface/partials/course_minimal_toc.html` — the expand/collapse icon `x-show` (line ~85) carries `x-cloak`. There is also a nested `x-show="expanded" x-collapse` on a `<div>` (line ~127) that does **not** currently carry `x-cloak`. Add `x-cloak` to that element so the collapsed-by-default child list does not flash open on first paint.
   - `freedom_ls/panel_framework/templates/panel_framework/partials/sidebar_nav.html` — all `x-show` usages carry `x-cloak`.
   - `freedom_ls/base/templates/cotton/dropdown-menu.html` — uses `x-show="open"` without `x-cloak`, but has an inline `style="display: none"` fallback. Add `x-cloak` for consistency with the rest of the codebase; the inline style can stay or go (it is now redundant once the global rule lands).
   - `freedom_ls/base/templates/partials/messages.html` — uses `x-show="show"` without `x-cloak`. This element starts in the open state (`show: true` initially), so it currently does not flash. Leave as-is, but document the choice in the audit notes / PR description so future contributors understand. If `x-cloak` is added here without a paired transition strategy, the toast enter animation would not fire on first paint.
   - `freedom_ls/base/templates/_base.html` (debug branch badge, line ~92) — uses `<span x-show="expanded" x-transition.opacity>` without `x-cloak`. This is rendered only when `DEBUG` is on, and the element is a small inline span with no layout impact when briefly visible. Leave as-is; this is dev-only chrome and adding `x-cloak` would suppress the `x-transition.opacity` enter animation. Document the choice in the audit notes.

4. After the fix, the picture-zoom modal must not flash open on `http://127.0.0.1:8000/courses/functionality-demo-show-end-with-quiz/1/`. Verify with Playwright MCP at desktop, tablet, and mobile widths, capturing before/after screenshots. The same verification applies to other audited Alpine components.

### Part B — Hide empty dashboard sections entirely

The dashboard partials live in `freedom_ls/student_interface/templates/student_interface/partials/course_list.html`. Two of the partials (`recommended-courses` and `learning-history`) render a `<div class="surface">` containing an `<h2>` and either a grid of cards or a muted "No … yet" line.

Decision: **render nothing at all when the corresponding queryset is empty.** No placeholder, no CTA, no muted line. The `<div class="surface">`, the `<h2>`, and the inner empty-state paragraph all disappear from the DOM.

Specifically:
- `recommended-courses` partial: the entire `<div class="surface">…</div>` block must be wrapped in `{% if recommended_courses %} … {% endif %}` so nothing reaches the DOM when the iterable is empty.
- `learning-history` partial: same treatment guarded by `{% if completed_courses %}`.
- The `current-courses` partial is **out of scope** for hide-when-empty. The "Your Courses" section is the user's own collection and the existing "You haven't signed up for any courses yet" message there is the intended onboarding affordance for new users. Leaving it as-is is deliberate.
- The "All Courses" CTA button group at the bottom of the wrapper (`partial all-courses` + the centered button-group) is out of scope and continues to render unconditionally.
- The fallback case in `partial all-courses` (`No courses available yet.` when `all_courses` is empty *and* the user has no registrations) is out of scope and stays as-is — it is the dashboard-wide empty state, not a section-level empty state.

#### Why this design

- The research file (Part B) recommends: hide recommendations when empty; show an onboarding card for first-run learning history; show a subtle line for returning users. The idea file explicitly overrides this and asks for "pure hide" of both sections. The reason: keeping the rule simple and uniform avoids the need for a "has the user ever had data here" signal and avoids two new placeholder designs. The `current-courses` partial already provides an onboarding affordance for brand-new users via "You haven't signed up for any courses yet." A new user with no enrolments will see that message — the absence of recommendations and history alongside it is not surprising.
- Hiding via Django template `{% if %}` (rather than `display:none` or `aria-hidden`) ensures the heading is removed from both the visual layout *and* the accessibility tree. This is the option the research file recommends for accessibility.

## Out of scope

- Onboarding placeholders / illustrated empty states for any dashboard section.
- "Has this user ever had data here" signal on the dashboard view — not needed because the design uniformly hides.
- Refactoring `c-picture` (or any other modal) to use `<template x-if>` instead of `x-show` + `x-cloak`. The research file mentions this as an option; it is a larger change and not required to fix the flash.
- Changes to other dashboard sections (cohorts/classmates, notifications) — the idea is scoped to recommendations + history.
- Changes to `messages.html`'s `x-show` (kept as-is; see audit note).
- Changes to the debug branch badge in `_base.html` (kept as-is; see audit note).
- Adding `x-cloak` to the nested `x-show="open"` elements inside `cotton/modal.html` — the cloaked outer wrapper already hides them at parse time.

## Success criteria

1. Loading `http://127.0.0.1:8000/courses/functionality-demo-show-end-with-quiz/1/` with a hard refresh and slow network throttling does not show the `c-picture` zoom modal at any point before the user clicks the thumbnail. Verified with Playwright MCP at desktop, tablet, and mobile widths.
2. Loading `http://127.0.0.1:8000/` as an authenticated user with no recommendations and no completed courses renders no `Recommended Courses` heading and no `Learning History` heading anywhere in the rendered HTML. Verified with Playwright MCP and by inspecting the response body.
3. Loading `http://127.0.0.1:8000/` as an authenticated user with one or more recommendations and one or more completed courses renders both sections normally. Verified with Playwright MCP.
4. The audit table in Part A is reproduced in the PR description, with each Alpine `x-show` usage either confirmed `x-cloak`-protected or justified.
5. `static/vendor/tailwind.output.css` contains the `[x-cloak] { display: none !important; }` rule after `npm run tailwind_build`.
6. No regressions in any existing tests (`uv run pytest`).
7. Before/after screenshots at desktop, tablet, and mobile widths are captured for:
   - the picture-zoom flash (page load),
   - the dashboard with empty sections,
   - the dashboard with populated sections.

## Notes on testing

- The Tailwind change is CSS-only and has no Python tests. Verification is via the built CSS output and the visual smoke test.
- The dashboard hide-when-empty change is template-only. A pytest test that asserts the rendered HTML for an authenticated user with empty `recommended_courses` and `completed_courses` does **not** contain the strings `"Recommended Courses"` and `"Learning History"`, and **does** contain them when the contexts are populated, is appropriate. (Per project convention: test functionality, not styling.)
- Use the DemoDev site for QA test data (per project convention).
