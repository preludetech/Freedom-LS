# Frontend QA: P1 importable health module

Browser-observable behaviour for the new `freedom_ls.health` endpoints and the registration-middleware
exemption. There is no new UI, but the endpoints are reachable in a browser and the middleware
exemption is observable, so verify them there.

## Setup

Pick a random free port and run the dev server (do **not** use the default runserver port):

```
PORT=$(.claude/fls/scripts/find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL: `http://127.0.0.1:$PORT`

Use the **DemoDev** site for any test data / logins. If a learner account is needed and none exists,
create one via the `fls:qa-data-helper` agent (a plain learner on the DemoDev site is enough).

Note on redirects: in dev, `DEBUG=True` so `SECURE_SSL_REDIRECT` is **off** — the endpoints serve
plain HTTP directly. The `SECURE_REDIRECT_EXEMPT` behaviour is a production-settings concern and is
covered by the settings unit test, not by this dev-server QA.

## Test 1 — Liveness endpoint (no dependencies)

1. Visit `http://127.0.0.1:$PORT/health/liveness/`.
2. **Expect:** HTTP 200 and a JSON body `{"status": "alive"}`.
3. This must work while logged **out** — liveness touches no auth, no DB, no dependency.

## Test 2 — Readiness endpoint (DB check)

1. Visit `http://127.0.0.1:$PORT/health/readiness/`.
2. **Expect:** HTTP 200. In a browser you'll see `django-health-check`'s status dashboard listing the
   **database** backend as working/OK. (Check the response status code via the browser dev-tools
   Network tab, or `curl -i http://127.0.0.1:$PORT/health/readiness/` — it should be `200`, and
   `500` only if the DB is unreachable.)
3. **Expect:** the dashboard reports **only** a database check — no cache/storage checks, since those
   are opt-in and not registered by default.

## Test 3 — Old static path is gone (regression)

1. Visit `http://127.0.0.1:$PORT/health/` (the old static endpoint's path, with no sub-path).
2. **Expect:** HTTP 404 — the old `{"status":"healthy"}` static view has been removed and there is no
   longer a view mounted at exactly `health/`. (Only `/health/liveness/` and `/health/readiness/`
   resolve now.)

## Test 4 — Registration-completion middleware exempts the health paths

The `RegistrationCompletionMiddleware` redirects logged-in learners with an **incomplete** registration
to the completion form for normal pages, but must let the health paths through. This is the
`EXEMPT_URL_NAMES` change in action.

Precondition: a learner account on the DemoDev site whose registration is **incomplete** (the site's
`SiteSignupPolicy` has an additional registration form the learner has not filled in). If unavailable,
ask `fls:qa-data-helper` to set one up.

1. Log in as that learner at `http://127.0.0.1:$PORT/accounts/login/`.
2. Visit any normal learner page (e.g. the dashboard / a course page). **Expect:** you are redirected
   to the complete-registration form — confirming the middleware is actively gating this account.
3. Now visit `http://127.0.0.1:$PORT/health/liveness/`. **Expect:** HTTP 200 JSON `{"status":
   "alive"}` — **not** a redirect to the completion form.
4. Visit `http://127.0.0.1:$PORT/health/readiness/`. **Expect:** HTTP 200 dashboard — **not** a
   redirect to the completion form.

If step 3 or 4 redirects you to the registration form instead of serving the health response, the
`EXEMPT_URL_NAMES` entries do not match the resolved view names (`health:liveness` /
`health:readiness`) — a bug to fix.

## Test 5 — Superuser is unaffected (sanity)

1. Log in as a superuser and visit `http://127.0.0.1:$PORT/health/liveness/` and
   `/health/readiness/`. **Expect:** both return their normal 200 responses (superusers already bypass
   the registration middleware; this just confirms no interaction regression).
