# Spec 2 (this repo): P1 importable health module

> **Part of the `support-concrete-project-deployment` effort.** See `spec-order.md` for ordering and
> cross-spec dependencies. **Depends on Spec 1's §5.1-A** (the proxy-header change it pairs
> `SECURE_REDIRECT_EXEMPT` with). Can run in **parallel with Specs 3 and 4** once Spec 1 has landed.
>
> **Target repo:** this repo (`freedom_ls`) — new `freedom_ls/health/` package.

## Terminology

`ConcreteFlsImplementation` = the specific downstream production repo that raised this handoff
(installs `freedom_ls` as a read-only git submodule). "Every concrete project" = concrete FLS
projects in general. FLS is never deployed standalone — only concretely, as a submodule.

## 1. Problem

The current health view returns a static `{"status":"healthy"}` with **no DB check**
(`config/urls.py:35–37,52`); there is no `freedom_ls.health` module and `django-health-check` is not
a dependency. Every concrete project needs real liveness/readiness probes for its compose
healthchecks and load balancer, and today has to build them from scratch.

## 2. Goal

Ship an **importable** FLS health package so a downstream gets both probes with **zero
`HEALTH_CHECK` config** and a clean upgrade path — imported, not copy-pasted (per the
artifact-vs-code home split, see `spec-order.md`).

## 4. Requirements

Ship an **importable** FLS health package (new `freedom_ls/health/` — does not exist today) exposing
a `freedom_ls.health.urls` that projects `include(...)`, with **two endpoints**:

- **liveness** — shallow "process is up", touches **no dependencies**, always 200 if the process can
  serve. Never checks the DB (a DB blip must not trigger restart-loops).
- **readiness** — **DB-connectivity by default**; cache and storage checks are **opt-in** (via
  `django-health-check`'s shipped backends), not on by default. Returns non-200 when the checked
  dependency is down. Applied-migrations is **not** offered as a readiness check — it is a
  deploy-smoke-test concern (see below).

Requirements:

- **Engine:** depend on [`django-health-check`](https://github.com/revsys/django-health-check)
  (Django-6 compatible; add to `freedom_ls` deps — versioned with the submodule SHA, nothing for
  downstream to install). Expose an opinionated thin wrapper: fixed `liveness`/`readiness`
  endpoints and an FLS-chosen default check set (liveness = none, readiness = DB) — so a downstream
  gets both probes with **zero `HEALTH_CHECK` config** and an upgrade path (add S3/cache checks in
  their own settings). `SECURE_REDIRECT_EXEMPT` is shipped separately as an importable settings
  default the project assigns (see the next bullet), not baked into the URLconf.
- **`SECURE_SSL_REDIRECT` gotcha (depends on Spec 1 §5.1-A):** with the proxy-header change, an
  internal plain-HTTP probe to `/health/…` gets a `301 → https`, which a naive `healthcheck:` reads
  as unhealthy. Ship `SECURE_REDIRECT_EXEMPT = [r"^health/"]` as an **importable default the
  project's `settings_prod.py` imports and assigns** — the same named-symbol pattern as Spec 1 §6's
  `settings_defaults` (`SECURE_REDIRECT_EXEMPT` is a Django *setting* read by `SecurityMiddleware`, so
  a `include()`d URLconf cannot set it; assigning an importable constant keeps it greppable and fails
  loudly on drift, rather than mutating it invisibly from an `AppConfig`). More robust than making
  every prober forge `X-Forwarded-Proto`. Document header-forging as the fallback for teams whose
  policy forbids a redirect-exempt path.
- **Applied-migrations belongs in the deploy smoke test, not polled readiness** — use
  `manage.py migrate --check` (exit-code, no HTTP) as a pre-cutover gate, not a per-poll
  `django_migrations` scan. (`django-health-check` ships no migrations backend, so it is not offered
  as a readiness check by default.)
- **Keep health paths off the public Caddy vhost** — readiness reveals dependency state (recon);
  every legitimate consumer reaches the app container-to-container/over localhost. The Spec 5
  Caddyfile must **not route `/health/*`** (stated there; noted here so the contract is visible).

Detail: `research_health_liveness_readiness_endpoints.md`.

## 5. Success criteria

- **Health probes:** importable `freedom_ls.health.urls` gives liveness (no deps) + readiness (DB)
  with zero `HEALTH_CHECK` config; internal HTTP probes are not 301'd (the importable
  `SECURE_REDIRECT_EXEMPT` default the project assigns handles it); `/health/*` is not routed on the
  public Caddy vhost (Spec 5).
