# Webhook Translation Layer — Frontend QA Plan

- Dev server running: `uv run python manage.py runserver`
- Login: Navigate to `http://127.0.0.1:8000/admin/` and log in as `demodev@email.com` / `demodev@email.com`

## Test 1: WebhookSecret CRUD

### Create a secret
1. Navigate to `http://127.0.0.1:8000/admin/webhooks/webhooksecret/add/`
2. Fill in:
   - Name: `brevo_api_key`
   - Description: `Brevo marketing API key`
   - Encrypted value: `xkeysib-test-api-key-12345678`
3. Click Save
4. **Expected:** Redirected to list view. The list displays columns: name, description, masked value, created_at, updated_at. The `brevo_api_key` row shows masked value `••••••••5678`

### Verify secret masking on edit
1. Click on the `brevo_api_key` secret in the list
2. **Expected:**
   - The encrypted value field is a password input with placeholder text
   - The full value is NOT displayed anywhere
   - A masked preview shows `••••••••5678` as a read-only field
3. Click Save without changing anything
4. **Expected:** Secret value is preserved (not overwritten with blank)

### Verify name validation
1. Navigate to `http://127.0.0.1:8000/admin/webhooks/webhooksecret/add/`
2. Enter name: `123-invalid`
3. Click Save
4. **Expected:** Validation error on the name field

### Search secrets
1. Navigate to `http://127.0.0.1:8000/admin/webhooks/webhooksecret/`
2. Type `brevo` in the search box and press Enter
3. **Expected:** The `brevo_api_key` secret appears in results
4. Search for `marketing` (part of the description)
5. **Expected:** The `brevo_api_key` secret appears (search covers description field)

## Test 2: WebhookEndpoint transformation fields

### View the transformation fieldset
1. Navigate to `http://127.0.0.1:8000/admin/webhooks/webhookendpoint/add/`
2. **Expected:** The form shows:
   - Standard fields: description, URL, event types, is_active
   - Transformation fieldset: preset selector, HTTP method, content type, auth type, headers template, body template
   - Template fields use a code editor (Ace) with syntax highlighting
   - Help text below template fields lists available variables

### Create endpoint without transformation
1. Fill in only standard fields:
   - Description: `Standard test endpoint`
   - URL: `https://httpbin.org/post`
   - Event types: check `user.registered`
2. Leave all transformation fields empty
3. Click Save
4. **Expected:** Endpoint saved successfully. No transformation configured.

### Create endpoint with transformation
1. Navigate to add endpoint
2. Fill in:
   - Description: `Transformed test endpoint`
   - URL: `https://httpbin.org/post`
   - Event types: check `user.registered`
   - HTTP method: `POST`
   - Content type: `application/json`
   - Auth type: `None`
   - Headers template: `{"Authorization": "Bearer {{ secrets.brevo_api_key }}"}`
   - Body template: `{"event": "{{ event.type }}", "email": "{{ event.data.user_email }}"}`
3. Click Save
4. **Expected:** Endpoint saved successfully with transformation configured.

## Test 3: Preset selector

### Select a preset
1. Navigate to `http://127.0.0.1:8000/admin/webhooks/webhookendpoint/add/`
2. In the Transformation fieldset, select "Brevo — Track Event" from the preset dropdown
3. **Expected:** The following fields are auto-populated:
   - URL: `https://api.brevo.com/v3/events`
   - HTTP method: `POST`
   - Content type: `application/json`
   - Headers template: populated with Brevo headers including `{{ secrets.brevo_api_key }}`
   - Body template: populated with Brevo body template
4. The fields should all be editable after preset selection

### Modify preset values before saving
1. After selecting the Brevo preset, change the body template slightly (e.g., add a new field)
2. Fill in description and event types
3. Click Save
4. **Expected:** Endpoint saved with the modified template, `preset_slug` set to `brevo-track-event`

### Reset to Preset Defaults
1. Open the endpoint saved above (which has `preset_slug` set to `brevo-track-event`)
2. Modify the body template (e.g., delete a line)
3. Look for a "Reset to Preset Defaults" option
4. Click it
5. **Expected:** All transformation fields are re-populated with the original Brevo preset values (URL, method, content type, headers, body)

## Test 4: Template validation

### Invalid Jinja2 syntax
1. Navigate to add endpoint
2. Fill in standard fields (description, URL, event types)
3. Set body template to: `{{ event.type`  (missing closing braces)
4. Click Save
5. **Expected:** Validation error mentioning template syntax error

### Invalid JSON output
1. Set content type to `application/json`
2. Set body template to: `not valid json {{ event.type }}`
3. Click Save
4. **Expected:** Validation error mentioning invalid JSON output

### Transformation fields without body_template
1. Set HTTP method to `PUT` but leave body template empty
2. Click Save
3. **Expected:** Validation error stating that body_template is required when other transformation fields are set

### Unknown template variables warning
1. Navigate to add endpoint
2. Fill in standard fields (description, URL, event types)
3. Set body template to: `{{ foobar }}`
4. Click Save
5. **Expected:** A warning is shown about unknown top-level variable `foobar` (not in `event`, `secrets`)

### Headers template must render to JSON object
1. Navigate to add endpoint
2. Fill in standard fields (description, URL, event types)
3. Set body template to: `{"valid": true}`
4. Set headers template to: `["not", "an", "object"]`
5. Click Save
6. **Expected:** Validation error stating headers template must render to a JSON object

## Test 6: Auth type signing with transformation

### HMAC signing combined with custom body
1. Navigate to add endpoint
2. Fill in:
   - Description: `Signed transformed endpoint`
   - URL: `https://httpbin.org/post`
   - Event types: check `user.registered`
   - Auth type: `HMAC Signing (Standard)` (the default)
   - Body template: `{"custom": "{{ event.type }}"}`
   - Content type: `application/json`
   - HTTP method: `POST`
3. Click Save
4. Use Send Test on this endpoint, select `user.registered`, and click "Preview & Send"
5. **Expected:** The response body from httpbin shows:
   - The custom body (not the standard webhook envelope)
   - The `webhook-signature`, `webhook-id`, and `webhook-timestamp` headers are present (HMAC signing was applied)
