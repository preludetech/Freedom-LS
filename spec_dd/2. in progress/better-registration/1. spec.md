# Better Registration — Specification

## Context

FLS currently collects only email and password at signup (with a redundant email confirmation field). This is insufficient for an LMS where:

- Certificates require the learner's name
- Educators identify students by name, not email
- Legal compliance (GDPR, POPIA) requires explicit, recorded consent to Terms & Conditions and Privacy Policy
- Different sites need different registration requirements

This spec covers improving the signup flow to collect names, obtain legal consent, and support per-site additional registration fields.

## 1. Name Fields at Signup

**What:** Add first name (required) and last name (optional) to the signup form.

**Why:** Names are needed for certificates, educator-facing student lists, and personalised communication. First name is required because it's essential for these use cases. Last name is optional to respect cultural flexibility (not all naming conventions include a surname).

**Configurability:** A `require_name` boolean on `SiteSignupPolicy` (default `True`) controls whether first name is required. When `False`, first name becomes optional. Last name is always optional.

## 2. Drop Email Confirmation Field

**What:** Remove the `email2` field from signup.

**Why:** Users copy-paste between email fields, defeating the purpose. FLS already has mandatory email verification which reliably catches typos. Removing it reduces friction — each extra field reduces signup conversion by ~5-10%.

## 3. T&C and Privacy Policy Acceptance

### 3.1 Clickwrap Checkboxes

**What:** Two separate unchecked checkboxes at signup — one for Terms & Conditions, one for Privacy Policy.

**Why:**
- GDPR requires consent to be "freely given, specific, informed and unambiguous"
- Pre-ticked boxes are NOT valid consent (GDPR Art. 7)
- T&C acceptance and privacy policy consent MUST be separate (GDPR requires unbundled, specific consent)
- Clickwrap (unchecked checkbox user must check) is the most legally defensible pattern

**Configurability:** A `require_terms_acceptance` boolean on `SiteSignupPolicy` (default `False`) controls whether these checkboxes appear. When enabled, both checkboxes are required to complete signup (assuming documents exist for the site).

Each checkbox links to a page displaying the full document text.

### 3.2 Legal Document Storage

**What:** T&C and Privacy Policy content stored as markdown files in the git repo.

**Why:** Git provides a built-in audit trail — every change is committed with a timestamp and author. Storing in the repo (not the database) means legal documents go through the same review process as code.

**Structure:**
- Location: `legal_docs/<site_domain>/terms.md` and `legal_docs/<site_domain>/privacy.md`
- Fallback: `legal_docs/_default/terms.md` and `legal_docs/_default/privacy.md` used when no site-specific document exists
- Frontmatter contains metadata:

```yaml
---
version: "1.0"
title: "Terms and Conditions"
type: "terms"
effective_date: "2026-04-01"
---

Markdown content here...
```

- The `version` field is a human-readable string used for business logic (e.g. "has user accepted v2 or later?")
- At acceptance time, the git hash of the file is also recorded — this serves as cryptographic proof of exactly what the user saw

**Source of truth for content and hash (integrity):**

- At request time, document content is read from the git blob at `HEAD` (e.g. `git show HEAD:legal_docs/<site_domain>/terms.md`), NOT from the working filesystem. The recorded `git_hash` is the blob SHA returned by `git rev-parse HEAD:<path>`. This means a tampered working tree cannot change what users see or what is recorded.
- For deployments without a `.git` directory (e.g. Docker images built without the checkout): the build step embeds a manifest mapping `<path> → blob_sha` plus the `HEAD` commit SHA, and the runtime reads from that manifest + a snapshot of file content captured at build time. Either mechanism MUST yield the same blob SHA the file would have had via `git rev-parse`.
- The `<site_domain>` directory segment is derived from `Site.domain` (DB-controlled). Before using it as a path component, validate it against a strict regex (`^[A-Za-z0-9.\-]+$` or equivalent) and resolve the final path to confirm it stays inside the `legal_docs/` root. Reject path-traversal attempts.
- Markdown is rendered via `freedom_ls.markdown_rendering.markdown_utils.render_markdown` (which sanitizes with `nh3`). This is a new leaf app extracted from `content_engine` so that `accounts` can depend on it without taking a runtime edge on `content_engine` — see plan Task 0.3. Never use `markdown.markdown(...)|safe`.
- Frontmatter is parsed with `yaml.safe_load` only.
- If neither the site-specific document nor the `_default/` fallback exists for a doc type while `require_terms_acceptance=True`, the signup form refuses to render the corresponding checkbox and surfaces a clear admin-visible warning (startup check + per-request log) instead of producing a broken link or 500.
- The default `legal_docs/_default/privacy.md` MUST disclose that IP addresses are captured for consent records and state a retention period (the period itself is project-specific but must be present).

### 3.3 Consent Tracking

**What:** A `LegalConsent` model (SiteAwareModel) recording every consent event.

**Fields:**
- `user` — FK to User
- `document_type` — "terms" or "privacy"
- `document_version` — from frontmatter (e.g. "1.0")
- `git_hash` — git hash of the file at time of acceptance
- `timestamp` — UTC, auto-set
- `ip_address` — extracted from request
- `consent_method` — how consent was obtained (e.g. "signup_checkbox")

**Why each field matters:**
- `document_version`: Enables business queries like "which users haven't accepted v2?"
- `git_hash`: Cryptographic proof of exact content accepted — essential if disputed
- `ip_address`: Additional evidence for legal compliance (recommended by GDPR Art. 7 guidance)
- `consent_method`: Distinguishes signup consent from future re-consent, important for audit trails

**Integrity and access controls:**

- `LegalConsent` is `SiteAwareModel`; all reads/writes go through the site-aware default manager. No view, helper, or admin inline may bypass site scoping. The `site` field is set automatically from the user's site on creation.
- Records are append-only. The model's `save()` raises if the row already has a primary key (insert-only), and the admin registers `LegalConsent` with `has_add_permission=False`, `has_change_permission=False`, `has_delete_permission=False` (i.e. read-only inline on `User`). No `update()`, `bulk_update()`, or admin "give consent" action exists.
- Only the signup adapter (`custom_signup`) and the future re-consent flow may create `LegalConsent` rows. These code paths always use `request.user`; no view accepts a user-id parameter when recording consent.
- `ip_address` is extracted via a single helper `freedom_ls.accounts.utils.get_client_ip(request)` that respects a configurable trusted-proxy header. Direct reads of `request.META["REMOTE_ADDR"]` are prohibited for this purpose.
- Email rebind does not invalidate existing consent — consent is bound to the user, not the email address. This is documented behavior, not a bug.

### 3.4 Legal Document View

A public view at `/accounts/legal/<doc_type>/` renders the markdown content for the current site's T&C or Privacy Policy. This is what the checkbox labels link to.

## 4. Additional Registration Fields (Post-Verification)

### 4.1 Motivation

Some sites need more information than email, password, and name — e.g. ID number, phone number, employer. But collecting everything on the initial signup page would hurt conversion. The solution: collect essentials at signup, then require additional fields after email verification.

### 4.2 Design: Django Form Classes

**What:** `SiteSignupPolicy` stores a list of dotted paths to Django Form classes (in a JSONField). Each form class is defined in code and implements a protocol:

- `applies_to(user) -> bool` (classmethod) — whether this form is required for the given user. Defaults to `True`. Use this to opt specific users out (e.g. superusers, staff, users without a `Student` profile).
- `is_complete(user) -> bool` (classmethod) — checks if this user has already provided the required data
- `save(user) -> None` — persists the form data for the user

**Why Django Forms:** This is Django's native form framework. Each form handles its own fields, validation, and persistence logic. New fields are added by writing a new Form class and adding its dotted path to the site's config — no schema changes, no dynamic form builder complexity.

**Why `applies_to`:** Not every form applies to every authenticated user. Superusers created via `createsuperuser` and staff/support accounts shouldn't be blocked from the platform by learner-only fields. Each form decides its own audience rather than the middleware hardcoding a global bypass.

**Where the data lives:** Each form is responsible for persisting its own fields somewhere appropriate. Some forms will set fields directly on the `User` model (e.g. a phone number added via a downstream project's user extension); others will write to a related profile model defined by the consuming project. FLS itself has no dedicated learner profile model today, so the form classes shipped with FLS will be limited and any project-specific fields are added by the deploying project.

**Example (illustrative, using a hypothetical `phone_number` field added by a downstream project to its custom User):**
```python
class PhoneNumberForm(forms.Form):
    phone_number = forms.CharField(max_length=20, label="Phone Number")

    @classmethod
    def applies_to(cls, user) -> bool:
        # Don't pester staff/superusers for learner-only data
        if user.is_superuser or user.is_staff:
            return False
        return True

    @classmethod
    def is_complete(cls, user) -> bool:
        return bool(getattr(user, "phone_number", "") or "")

    def save(self, user) -> None:
        user.phone_number = self.cleaned_data["phone_number"]
        user.save(update_fields=["phone_number"])
```

The `get_incomplete_forms(user)` loader filters out forms where `applies_to(user)` is `False` before checking `is_complete`. As a baseline safety net, the loader also short-circuits to an empty list for `is_superuser` users, so a misconfigured form can never lock out a superuser.

**Loader hardening:**

- After resolving a dotted path via `import_string`, the loader verifies the target is a `forms.Form` subclass and exposes the protocol methods (`applies_to`, `is_complete`, `save`). Anything else — non-callable, wrong type, missing method, `ImportError`, `AttributeError` — is logged at WARNING and skipped. The loader MUST NOT raise into a request: a misconfigured `additional_registration_forms` entry degrades gracefully rather than 500-ing every authenticated request.
- Forms MUST NOT define a field that identifies the user (e.g. `user_id`, `user`, `email`). The completion view always passes `request.user` to `save()`. This is part of the form protocol contract.

### 4.3 Completion View

A view at `/accounts/complete-registration/` renders all incomplete forms on a single page. All forms are submitted together. Each form uses a class-name prefix to avoid field name collisions.

The view always calls `form.save(request.user)` — it never resolves a target user from form data, query string, or session payload. Any post-submit redirect (e.g. honoring a `next` parameter) MUST go through Django's `url_has_allowed_host_and_scheme` against `request.get_host()`; unsafe values fall back to a known-safe internal URL (e.g. `LOGIN_REDIRECT_URL`).

### 4.4 Middleware Enforcement

**What:** `RegistrationCompletionMiddleware` redirects authenticated users to the completion view if they have incomplete registration forms.

**Why middleware:** Users might verify their email, get redirected, then close their browser. When they return and log in, the middleware ensures they complete registration before accessing the platform. This is the same pattern used by many LMS platforms for onboarding.

**Exempt URLs:** Matched as an explicit allow-list of resolved URL names or path prefixes (no substring/`in` matching). The list MUST include, at minimum:

- `account_login`, `account_logout`, `account_signup`
- `account_email_verification_sent`, `account_confirm_email`, `account_email`
- `account_reset_password`, `account_reset_password_done`, `account_reset_password_from_key`, `account_reset_password_from_key_done`
- The legal-doc views (`/accounts/legal/terms/`, `/accounts/legal/privacy/`)
- The completion view itself (`/accounts/complete-registration/`)
- `/static/`, `/media/`, `/health/`

Password-reset URLs MUST be exempt — otherwise a logged-in user with incomplete registration who follows a reset link would be bounced into the completion view and unable to reset.

**Exempt users:** Superusers are always exempt. Other users are exempted on a per-form basis via each form's `applies_to(user)` classmethod (see 4.2) — if no forms apply to a user, the middleware lets them through. `is_staff` users are NOT blanket-exempted: each form decides via its own `applies_to`.

**Middleware placement:** Registered in `MIDDLEWARE` after `django.contrib.auth.middleware.AuthenticationMiddleware` and `allauth.account.middleware.AccountMiddleware`, before view dispatch. Wrong placement either leaves `request.user` undefined or causes the middleware never to fire.

**Performance and cache invalidation:** The middleware caches completion status in the session to avoid hitting the database on every request. The cache key MUST incorporate a hash of the configured dotted-path list from `SiteSignupPolicy.additional_registration_forms` (so a site config change invalidates the cache automatically). The cache is additionally cleared explicitly when the user submits the completion view.

**Admin-edit invalidation is intentionally best-effort.** If an admin wipes profile data that one of the registration forms persists, the affected user MAY remain "complete" for the rest of their current session and only be redirected back to the completion view on a future visit (next login / new session). This is an explicit trade-off in favour of simplicity over a strict same-request invariant — the cost of a one-session delay is acceptable for this feature.

## 5. SiteSignupPolicy Extensions

The existing `SiteSignupPolicy` model gains three new fields:

| Field | Type | Default | Purpose |
|-------|------|---------|---------|
| `require_name` | BooleanField | `True` | Whether first name is required at signup |
| `require_terms_acceptance` | BooleanField | `False` | Whether T&C and Privacy checkboxes appear |
| `additional_registration_forms` | JSONField | `[]` | Dotted paths to Form classes for post-verification |

The existing `allow_signups` field is unchanged.

## 6. Security and Abuse Mitigations

This section captures controls surfaced by the threat model that aren't naturally placed in the feature sections above. These are spec-level requirements; implementation specifics live in the plan.

- **Signup rate limiting.** Removing `email2` marginally lowers the friction of bulk signup. django-axes covers login but NOT signup. The implementation MUST configure allauth's `ACCOUNT_RATE_LIMITS` for `signup` (e.g. a per-IP and per-email rate) or document an existing global limit that already covers the signup endpoint. A honeypot field is a recommended cheap additional control.
- **`ACCOUNT_PREVENT_ENUMERATION` enabled.** Confirm this allauth setting is on in `config/settings_base.py` so the signup response is consistent for known vs. unknown emails.
- **Trusted IP extraction.** A single helper `freedom_ls.accounts.utils.get_client_ip(request)` is the only sanctioned way to derive the client IP for `LegalConsent`. It honors a configurable trusted-proxy header (defaulting to `REMOTE_ADDR` when no proxy is configured).
- **Open-redirect protection.** Any `next`-style redirect on the completion view passes through `url_has_allowed_host_and_scheme`.
- **Misconfiguration alerts.** A startup check warns (loud log) when a site has `require_terms_acceptance=True` but neither site-specific nor `_default/` doc files resolve. The signup form refuses to render the corresponding checkbox in that case rather than rendering a broken link.
- **Default for `require_terms_acceptance`.** Defaults to `False` to preserve existing behavior on upgrade. From a pure compliance posture, `True` would be safer for new sites; flagged here as a deliberate trade-off rather than an oversight.

## 7. Implementation Approach

### Custom Signup Form

A `SiteAwareSignupForm` subclasses allauth's `SignupForm`. In `__init__`, it reads the current site's `SiteSignupPolicy` and dynamically:
- Adjusts whether `first_name` is required
- Adds `accept_terms` and `accept_privacy` checkbox fields if terms acceptance is required

The `custom_signup()` method (allauth hook called after user creation) records consent in `LegalConsent`.

### Settings Changes

- `ACCOUNT_SIGNUP_FIELDS`: Remove `email2`, add `first_name` and `last_name`
- `ACCOUNT_FORMS`: Point to `SiteAwareSignupForm`
- `ACCOUNT_RATE_LIMITS`: Configure `signup` rate limits
- `ACCOUNT_PREVENT_ENUMERATION`: Confirm enabled
- `MIDDLEWARE`: Add `RegistrationCompletionMiddleware` after `AuthenticationMiddleware` and `AccountMiddleware`

### New Files

- `freedom_ls/markdown_rendering/` — new leaf app holding the `render_markdown` helper (extracted from `content_engine`). See plan Task 0.3.
- `freedom_ls/accounts/legal_docs.py` — reading markdown docs from git blob, hash retrieval, consent recording helpers; markdown rendered via `freedom_ls.markdown_rendering.markdown_utils.render_markdown`; frontmatter parsed with `yaml.safe_load`
- `freedom_ls/accounts/registration_forms.py` — form protocol, loader utility (with subclass/protocol validation and graceful failure), `get_incomplete_forms`
- `freedom_ls/accounts/middleware.py` — `RegistrationCompletionMiddleware`
- `freedom_ls/accounts/utils.py` — `get_client_ip(request)` trusted-proxy helper
- `legal_docs/_default/terms.md` and `privacy.md` — sample documents; default privacy doc discloses IP capture and retention

### Modified Files

- `freedom_ls/accounts/models.py` — extend `SiteSignupPolicy`, add `LegalConsent` (with `save()` guard against updates)
- `freedom_ls/accounts/forms.py` — add `SiteAwareSignupForm`
- `freedom_ls/accounts/views.py` — add legal doc view and completion view
- `freedom_ls/accounts/urls.py` — new URL patterns
- `freedom_ls/accounts/admin.py` — register `SiteSignupPolicy`, add `LegalConsent` as fully read-only inline on User (no add/change/delete permissions)
- `config/settings_base.py` — update allauth settings, add middleware, register new `freedom_ls.markdown_rendering` app
- `freedom_ls/content_engine/models.py`, `freedom_ls/content_engine/templatetags/content_tags.py` — repoint imports from `freedom_ls.content_engine.markdown_utils` to `freedom_ls.markdown_rendering.markdown_utils` (Task 0.3)

## 8. Out of Scope

- Re-consent flow when T&Cs are updated (see `re-consent-idea.md`)
- Social login / passwordless auth
- Admin UI for editing legal document content (managed via git)
- Marketing consent checkbox (separate from T&C/privacy)

## 9. Success Criteria

- [ ] Signup form collects first name (required by default), last name (optional), email, password — no email confirmation field
- [ ] Sites with `require_terms_acceptance=True` show separate T&C and Privacy Policy checkboxes that must be checked
- [ ] Each consent is recorded with user, document type, version, git hash, timestamp, IP, and method
- [ ] `LegalConsent` records are append-only at the model layer and read-only in admin
- [ ] Legal documents are readable at `/accounts/legal/terms/` and `/accounts/legal/privacy/`, rendered via the nh3-sanitized markdown renderer, with content sourced from the git blob at `HEAD`
- [ ] Sites with `require_name=False` make first name optional
- [ ] Sites with additional registration forms redirect users to complete them after email verification
- [ ] Users cannot access the platform until all required registration forms are complete
- [ ] Password-reset, email-verification, and other allow-listed flows are reachable for users with incomplete registration
- [ ] Superusers are never blocked by the registration completion check
- [ ] Forms can opt specific users out via `applies_to(user)` (e.g. staff, users to whom a given form does not apply)
- [ ] A misconfigured `additional_registration_forms` entry (bad dotted path, wrong type) logs and is skipped — never raises into a request
- [ ] Signup endpoint is rate-limited; `ACCOUNT_PREVENT_ENUMERATION` is enabled
- [ ] All configuration is per-site via `SiteSignupPolicy`
- [ ] Existing signup functionality (site-level allow/deny, webhook on registration) continues to work
