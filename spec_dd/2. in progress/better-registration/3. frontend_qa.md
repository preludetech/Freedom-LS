# Better Registration — Frontend QA Plan

This is the manual / Playwright QA pass. It exists to verify behaviour in a real browser, end to end, that the unit tests don't cover (visual rendering, real allauth flow, real session cookies, real middleware redirect). All test data should live on the **DemoDev** site.

## Setup

```bash
PORT=$(.claude/fls/scripts/find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL: `http://127.0.0.1:$PORT`

You will need (create via the `qa-data-helper` agent if missing):

- A `Site` named `DemoDev` (the project default).
- A `SiteSignupPolicy` for that site. Tests below toggle the policy fields between scenarios.
- Default legal docs at `legal_docs/_default/terms.md` and `legal_docs/_default/privacy.md`.
- A superuser, at least one regular learner with verified email, and a fresh email address you can sign up with.

For scenarios that exercise additional registration forms, register a sample form class (e.g. one that adds a `phone_number` field to a related profile model) and add its dotted path to `SiteSignupPolicy.additional_registration_forms`. If the project does not yet ship a real registration form, create a minimal one in a test-only app or use a temporary form in `freedom_ls.accounts.tests.fixtures`.

---

## Test 1 — Signup form: defaults (`require_name=True`, `require_terms_acceptance=False`)

1. Set `SiteSignupPolicy(require_name=True, require_terms_acceptance=False)` for DemoDev.
2. Navigate to `http://127.0.0.1:$PORT/accounts/signup/`.
3. Expect to see fields: **Email**, **Password**, **Password (again)**, **First name**, **Last name**.
4. Expect there to be **no** "confirm email" field (this used to be there).
5. Expect there to be **no** Terms or Privacy checkboxes.
6. Submit with a blank first name. Expect a validation error on first name.
7. Submit with a valid first name (last name blank). Expect signup success and redirect to the email-verification-sent page.
8. Check the user's email is created in the DB and `first_name` is populated, `last_name` is empty.

## Test 2 — Signup form: `require_name=False`

1. Update DemoDev policy: `require_name=False`.
2. Navigate to `/accounts/signup/`.
3. Expect first name field to be **not required** (no asterisk / can submit empty).
4. Submit with first name blank. Expect signup to succeed.

## Test 3 — Signup form: `require_terms_acceptance=True` (docs present)

1. Ensure `legal_docs/_default/terms.md` and `legal_docs/_default/privacy.md` exist with valid frontmatter.
2. Set DemoDev policy: `require_terms_acceptance=True`, `require_name=True`.
3. Navigate to `/accounts/signup/`.
4. Expect two checkboxes labelled "I accept the Terms and Conditions" and "I accept the Privacy Policy".
5. Each checkbox label should contain a link. Click the **Terms** link.
6. Expect a new page at `/accounts/legal/terms/` showing the rendered Terms content with title and version visible.
7. Go back. Click the **Privacy** link.
8. Expect `/accounts/legal/privacy/` rendered similarly. Confirm the privacy doc explicitly mentions IP address capture and a retention period.
9. Go back to signup. Try to submit with the boxes unchecked. Expect a validation error on each unchecked box.
10. Tick both boxes, fill in a fresh email + password + first name, and submit. Expect signup success.
11. In the Django admin, navigate to the new user's detail page. Expect a **Legal consents** inline showing two rows: one for `terms`, one for `privacy`, each with a non-empty `git_hash`, a timestamp, an IP address, and `consent_method=signup_checkbox`.
12. Try to edit one of the inline rows. Confirm there are no edit/save buttons (read-only).
13. Try to add a new row from the inline. Confirm the "Add another Legal consent" link is absent.

## Test 4 — Signup form: `require_terms_acceptance=True` but docs missing

1. Temporarily rename `legal_docs/_default/terms.md` to `terms.md.disabled` (or work in a branch with no default docs).
2. Restart the dev server. Confirm the Django startup output includes a system-check warning about missing terms doc for any site requiring acceptance.
3. Navigate to `/accounts/signup/`. Expect the Terms checkbox to be absent (form still rendered, no broken link). Privacy checkbox should still appear if its doc exists.
4. Restore the file.

## Test 5 — Path-traversal attempt on legal-doc view

1. Try `http://127.0.0.1:$PORT/accounts/legal/../../etc/passwd/` (manually craft the URL).
2. Expect a 404 (URL routing rejects it via Django's `<str:doc_type>` constraint and the view's allow-list check).
3. Try `/accounts/legal/something/` — expect 404.
4. Confirm no Django error page leaks the filesystem.

## Test 6 — Completion view: triggered by middleware

1. Configure DemoDev policy `additional_registration_forms` to include the dotted path of a sample form (e.g. one requiring a phone number).
2. Sign up a fresh user (Test 1 or Test 3 flow). Verify their email by clicking the link in MailHog / the dev mail backend.
3. After verification, you should land on a page that is NOT the homepage. Expect to be redirected to `/accounts/complete-registration/`.
4. Expect the form fields from the configured form classes to be visible.
5. Try to navigate to the homepage (`/`). Expect to be redirected back to the completion view.
6. Try to navigate to `/accounts/legal/terms/`. Expect that page to load (it's exempt).
7. Try `/accounts/logout/`. Expect logout to work (also exempt).
8. Log back in. Expect another immediate redirect to the completion view.
9. Fill in the form and submit. Expect to land at `LOGIN_REDIRECT_URL` (typically `/`).
10. Navigate around the platform freely — no more redirects to completion.

## Test 7 — Completion view: superuser bypass

1. Log in as a superuser (regardless of whether `additional_registration_forms` is configured).
2. Navigate to the homepage. Expect no redirect to the completion view.
3. Manually navigate to `/accounts/complete-registration/`. Expect it to render but say "no incomplete forms" or redirect to `LOGIN_REDIRECT_URL`. (Either is acceptable; the unit tests pin the exact behaviour.)

## Test 8 — Open-redirect on completion view

1. As a user with incomplete forms, log in.
2. Manually navigate to `http://127.0.0.1:$PORT/accounts/complete-registration/?next=https://evil.example.com/`.
3. Fill in and submit the form.
4. Expect the redirect after submit to go to `LOGIN_REDIRECT_URL` — **not** to `evil.example.com`.
5. Repeat with `?next=/some-internal-path/`. Expect the redirect to go to `/some-internal-path/`.

## Test 9 — Password reset is not blocked by middleware

1. Configure incomplete forms as in Test 6.
2. Log in as a user with incomplete registration.
3. Navigate to `/accounts/password/reset/`. Expect the password-reset form to render, NOT a redirect to the completion view.
4. Submit a reset request. Expect the email-sent confirmation page.
5. Click the reset link in the email. Expect the password-reset-from-key page to load.
6. Set a new password. Confirm flow completes.

## Test 10 — Visual / responsive

For each of the above pages (signup, legal-doc view, completion view):

1. View at desktop width (≥ 1280px).
2. View at tablet width (~768px).
3. View at mobile width (~375px).
4. Confirm: no horizontal scroll, labels readable, checkboxes hit-target ≥ 24×24, form errors visible.
5. Confirm legal-doc page is readable (line length sensible, headings styled).

Take screenshots of:
- Signup page with checkboxes (Test 3, step 4) — desktop + mobile.
- Legal-doc page (Test 3, step 6) — desktop + mobile.
- Completion view (Test 6, step 4) — desktop + mobile.

## Test 11 — Webhook regression

1. With the webhook system configured (existing functionality), sign up a fresh user.
2. Confirm the `user.registered` webhook fires with `user_id`, `user_email`, `first_name`, `last_name` payload (already wired in `AccountAdapter.save_user`).
3. Confirm the new consent flow does NOT cause duplicate webhook firings.

## Test 12 — Existing site allow/deny still works

1. Set DemoDev policy: `allow_signups=False`.
2. Navigate to `/accounts/signup/`. Expect the existing "signups closed" behaviour (whatever the project currently shows).
3. Set back to `allow_signups=True`. Expect signup to be reachable again.

## Pass criteria

All 12 tests above pass. Screenshots from Test 10 are captured and look reasonable against `brand-guidelines`. No console errors in the browser dev tools at any step.
