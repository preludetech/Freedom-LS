# Implementation plan: Make QA more efficient

## Nature of this work

This spec changes the **SDD / QA tooling itself**, not the FLS Django application. The
deliverables are markdown command/agent files, JSON config, an allow-list, and a README edit —
plus one **new spawnable agent** (`fls:qa-bugfixer`). There is **no FLS frontend change**, so
there is no browser-driven QA plan for this work; the success criteria are validated by reading
the changed files and (optionally) by running `/do_qa` once against a representative diff.

Because the artifacts are prompt/command/config files rather than Python, classic
red-green pytest TDD does **not** apply to most tasks here. The one place real TDD lives is
*inside* the new `fls:qa-bugfixer` agent's runtime behaviour (it writes a failing test first),
not in this plan's own construction. Where a change is verifiable by a shell check (JSON
validity, allow-list match semantics) the task says so.

All edits stay inside the plugin and project-config surface:

| # | File | Kind |
|---|---|---|
| A | `.mcp.json` (repo root) | edit — the server `/do_qa` actually binds to |
| B | `fls-claude-plugin/.mcp.json` | edit/remove — reconcile the duplicate |
| C | `fls-claude-plugin/playwright-mcp-config.json` *(new)* | new — pinned MCP defaults |
| D | `.claude/settings.json` | edit — allow-list entries |
| E | `fls-claude-plugin/commands/sdd/do_qa.md` | edit — gates, batching rules, tiering, bug-fix loop |

> **How `.claude/fls/scripts/` works (verified):** those scripts are **not** independent copies —
> they are thin wrappers **generated by `/init`** from `fls-claude-plugin/templates/wrapper_scripts/`,
> each delegating to the canonical `$PLUGIN_DIR/scripts/<name>.sh` (see `find_available_port.sh`'s
> wrapper). So to make a QA script callable at a stable `.claude/fls/scripts/<name>.sh` path you add
> **two** files: the canonical script under `fls-claude-plugin/scripts/` (if new) **and** a wrapper
> template under `fls-claude-plugin/templates/wrapper_scripts/`. The deployed `.claude/fls/scripts/`
> copy then has to be generated — re-run `/init` (it skips existing files, never overwrites) or
> create the wrapper by hand in this worktree so the path exists for the QA run.

> The skills/MCP-scan fold-in (added after the first draft of this plan) records the exact
> `@playwright/mcp` flag/config semantics and which skills each step should lean on. See the
> **Skills & MCP** section near the end.

## Step 0 — Resolve the two-server question (do this first; it gates Phase 1.1)

There are two Playwright MCP definitions today:

- repo-root `./.mcp.json` → server name `playwright` → tools namespaced `mcp__playwright__*`.
- `fls-claude-plugin/.mcp.json` → plugin server → tools namespaced `mcp__plugin_fls_playwright__*`.

`do_qa.md` frontmatter grants `mcp__playwright*`. That wildcard matches `mcp__playwright__…`
but **not** `mcp__plugin_fls_playwright__…` (the latter starts with `mcp__plugin`). So **`/do_qa`
binds to the repo-root server**; the plugin server's tools are present in the session but are not
what the command is told to use.

**Decision (carry into Phase 1.1):** treat the **repo-root `./.mcp.json` as authoritative** and
tune it. Reconcile the plugin one (task B) so the tuning is not silently shadowed by a second,
un-tuned browser. Confirm at implementation time that, after the change, `/do_qa` still has a
working `mcp__playwright__*` toolset (the frontmatter grant is unchanged, so it should).

> If reconciliation removes the plugin server, the `mcp__plugin_fls_playwright__*` allow entry in
> `.claude/settings.json` becomes dead but harmless. Leave a one-line note rather than churning it,
> unless task D is already touching that file (it is) — in which case drop the dead entry there.

## Phase 1 — Cheap wins (config + scoping + resilience)

### Task A — Tune the repo-root `./.mcp.json`

Rewrite the root server `args` to load the new defaults config (task C) and set the
per-run-authoritative inline flags. Target shape (exact flag spellings confirmed in **Skills &
MCP**):

```json
{
  "mcpServers": {
    "playwright": {
      "type": "stdio",
      "command": "npx",
      "args": ["@playwright/mcp@latest",
        "--config", "${CLAUDE_PROJECT_DIR}/fls-claude-plugin/playwright-mcp-config.json",
        "--output-dir", "${CLAUDE_PROJECT_DIR}/qa-screenshots",
        "--headless", "--isolated",
        "--image-responses", "omit",
        "--caps", "core,testing"],
      "env": {}
    }
  }
}
```

- Keep `@playwright/mcp@latest` (root file already pins `@latest`; the plugin file used the
  unpinned `@playwright/mcp` — another reason to converge on root).
- **Use `${CLAUDE_PROJECT_DIR}`, not a relative path, for both `--config` and `--output-dir`.** The
  scan confirmed Claude Code expands `${CLAUDE_PROJECT_DIR}` in `.mcp.json` `args`, but the stdio
  MCP server's launch **CWD is not guaranteed to be the repo root** (clients often spawn it at `/`),
  so a relative `--config`/`outputDir` risks file-not-found / wrong-dir. (Refs: Claude Code MCP
  docs; playwright-mcp issues #1509, #1372.)
- **`outputDir` is set via the inline `--output-dir` arg, not in the config file.** The config file
  is plain JSON read by `@playwright/mcp`, which does **not** expand `${CLAUDE_PROJECT_DIR}` — so an
  `outputDir` written inside task C's file would be a literal unexpanded string. Pin it here, in the
  args, where Claude Code does the expansion. CLI `--output-dir` is the documented alias for config
  `outputDir`.
- **`--headless` stays a CLI flag** (the scan found `headless` is *not* a top-level config key; it
  is a launch option). `--isolated` works as a flag too (mirrors config `browser.isolated`).
- `--caps core,testing` **drops the `vision` capability** (pixel/coordinate-click tools). Confirm no
  `do_qa.md` step relies on vision tools — current steps use `browser_navigate`, `browser_snapshot`,
  `browser_click` (by ref/role — core), `browser_type`, `browser_take_screenshot` (core), all of
  which survive. Screenshot **capture** is core and unaffected (spec §1.1).
- Inline args override the config file for the same keys — intended (config = defaults, inline =
  authoritative).

**Verify:** `python -c "import json; json.load(open('.mcp.json'))"` (or `jq . .mcp.json`) parses.

### Task E1 — `do_qa.md`: "Batching safety rules" CRITICAL block

Add a CRITICAL block near the top of `do_qa.md` (alongside the existing two CRITICAL rules),
encoding the anti-churn rules from spec §1.4:

- Never issue a `Bash` call in the **same turn** as a Playwright MCP call unless **both** are
  pre-approved in `.claude/settings.json`. If either could prompt or trip a hook, issue the Bash
  call **solo** first, wait, then proceed with Playwright. (Why: cascade-cancel — a single denied
  sibling cancels the whole batch.)
- Never run a raw recursive force-delete directly — only committed wrapper scripts
  (`qa_cleanup.sh`, and a new `qa_clear_screenshots.sh` only if a mid-run clear is ever needed).
  Describe the dangerous delete in prose; never paste the literal flag string (the `security-guard`
  Write hook matches the substring and would block this very file — see spec Constraints).
- Never `cd` before a command — use absolute or CWD-relative paths (a `cd X && git …` compound
  always prompts and would collapse a batch).
- The dev-server start (`runserver`) is **always a solo call**.
- Any `Agent` spawn (`fls:qa-data-helper`, `fls:sdd-worker`, `fls:sdd-mechanic`,
  `fls:qa-bugfixer`) is **always a solo call**.

### Task E2 — `do_qa.md`: Diff-scoping gate (new step, before the server starts)

Insert a new step (after Step 1 cleanup, before/around Step 2) that classifies the diff and shrinks
the run. Pseudocode:

```
changed = `git diff main...HEAD --name-only`     # solo Bash call; git diff is allow-listed
if any path matches templates/ or *.html or *.css or *.js or static/ → CLASS = FULL
elif every changed path is under an admin.py / admin/ surface only        → CLASS = ADMIN_ONLY
elif changed paths are only *.py (views/models/logic), no template/css/js  → CLASS = BACKEND_ONLY
else → CLASS = FULL          # safe default
```

- `FULL` → desktop + mobile + tablet (current Steps 5–7).
- `ADMIN_ONLY` → desktop only (admin is already exempt from mobile/tablet per Steps 6–7).
- `BACKEND_ONLY` → single-viewport smoke pass only.

The gate MUST be **explicit and reported** in `qa_report.md`: which classification fired and what
was therefore **not** run (no silent scoping — SC#2). Record the classification into the scratch
list (task E5) so the report renderer states it.

> Keep the classifier rules in prose + a short table in `do_qa.md`; do not over-fit. The `else →
> FULL` fallback guarantees we never *under*-test by misclassifying.

### Task E3 — `do_qa.md`: Smoke gate (new step, before the full matrix)

Add a step that, after login (Step 4) and before the matrix (Step 5), loads **home + the primary

## Phase 2 — Model tiering of the cheap tail

> `"${CLAUDE_PLUGIN_ROOT}/scripts/hooks/security-guard.sh"`. If `${CLAUDE_PLUGIN_ROOT}` is unset in

> if the resolved script path/var is empty, `exit 0`) rather than weakening any pattern. This is a
> **pre-flight check** — run it early (right after Step 0): it gates not only E6/E7/F but also E5's
> scratch-list `Write` and E4a's collect step, all of which go through `Write`/`Bash` that the hook

## Phase 3 — Automatic TDD bug-fix + re-verify loop

### Task F — New spawnable agent `fls:qa-bugfixer.md`

Create `fls-claude-plugin/agents/qa-bugfixer.md`. Frontmatter:

```
name: qa-bugfixer
description: <one bug, TDD: failing test → minimal fix → green → full suite → commit. Non-interactive.>
tools: Bash, Read, Edit, Write, Glob, Grep, Skill
model: sonnet
```

- **Include `Skill` in `tools:`** (the skill-contradiction review flagged this) so the fixer can
  explicitly invoke the `fls:testing` skill at runtime, mirroring `sdd-mechanic.md` which keeps
  `Skill`. Belt-and-braces: also rely on description-matching auto-load. Do **not** add `Agent`
  (no nested spawns) or `AskUserQuestion` (non-interactive).

Body (mirror the `sdd-worker`/`sdd-mechanic` conventions — non-interactive, one unit, no nested
spawns, structured footer):

- **Runtime TDD sequence (this is where real TDD lives):** write a **failing** test that reproduces
  the bug → run it, confirm **RED** → make the **minimal** fix → run it, confirm **GREEN** → run the
  full `uv run pytest` → `uv run git commit` (the pre-commit PreToolUse hook fires
  `ruff check && mypy && pytest` as the gate; `--no-verify` is denied, so a green commit *is* the
  regression proof).
- Follow `CLAUDE.md`: commit via `uv run git commit`; type hints required; never delete TODO/@claude
  comments; use the `fls:testing` skill for test conventions.
- **Reports** to a structured `.sdd-work/bugfix_<slug>.md` (`status` / root-cause / fix / commit hash
  / suite-result). The **file MUST end with the same `status:` footer** (`status: ok|failed|blocked ·
  reason: <short>`) — this is the completeness contract that lets the orchestrator skip/resume, exactly
  as `sdd-worker.md` requires. **Separately**, return a one-line structured summary to the orchestrator.
  (Two contracts: the file footer AND the return line — match the `sdd-worker` shape.)
- **Records the files it created vs. modified** in `bugfix_<slug>.md`, so the loop guard (task E10)
  can revert precisely: `git checkout -- <modified-tracked-files>` for edits **and** `git clean -f
  <new-file>` for any new test file it wrote (a `git checkout --` alone silently ignores untracked
  new files — junk-files review finding 6).
- One bug per spawn; cannot spawn further subagents.

### Task E8 — `do_qa.md`: Triage at depth 0 (before spawning the fixer)
