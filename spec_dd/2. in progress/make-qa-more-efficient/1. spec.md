# Spec: Make QA more efficient

## Summary

The `/do_qa` SDD step is slow and token-hungry even for trivial changes, and it gets
stuck on tool-batch cascade-cancels and security-hook trips. This spec makes `/do_qa`
**cheaper, faster, more resilient, and self-healing** through three sequenced phases:

1. **Phase 1 — Cheap wins:** tune the Playwright MCP server (file-based screenshots,
   incremental snapshots, trimmed capabilities, headless/isolated), add a **diff-scoping
   gate** and a **smoke gate** before the full matrix, and bake in **anti-churn batching
   rules** plus the missing **allow-list entries**.
2. **Phase 2 — Model tiering:** move the non-judgement tail (screenshot compression, report
   rendering) off the strong session model onto spawned agents; keep browser-driving at
   depth 0 on the session model.
3. **Phase 3 — Automatic TDD bug-fix loop:** add a spawnable `fls:qa-bugfixer` agent and a
   depth-0 triage step so `/do_qa` can fix clear, fixable bugs (failing test → fix → green →
   re-verify) in the same run, closing the README's "Command for reacting to QA-reported
   bugs" TODO.

**Phase 4 (viewport parallelism) is explicitly out of scope for this spec** — see
[Out of scope](#out-of-scope--explicitly-rejected). It is deferred to a future spec, gated
behind the `agent-browser` go/no-go spike, and only worth building after Phases 1–3 land
(the diff-scoping gate already removes the viewport matrix for most runs).

## Problem & motivation

`/do_qa` (`fls-claude-plugin/commands/sdd/do_qa.md`) is an 11-step command that drives
Playwright MCP through a full **desktop (1920×1080) → mobile (375×812) → tablet (768×1024)**
matrix on every run, regardless of what changed, then compresses screenshots, assembles a
report, and ticks the todo — all on the strong session model at depth 0.

Two distinct failure modes make this expensive and fragile:

### 1. Cost & latency

The dominant token sinks, confirmed by research and grounded in the current config:

- **Inline screenshots.** `browser_take_screenshot` base64-encodes the image straight into
  context. A single full-page screenshot has been measured at **232,000 tokens** — more than
  a 200k context window. `/do_qa` takes screenshots at every step across 3 viewports, so this
  is the single biggest sink. The current `fls-claude-plugin/.mcp.json` is bare
  (`npx @playwright/mcp` with no flags), so images stream inline by default.
- **Full accessibility snapshots re-sent every turn.** Each interaction auto-attaches the
  page's accessibility tree (3,800–50,000 tokens for content-rich pages); without
  `incremental` mode these accumulate and are reprocessed on every subsequent turn.
- **Tool-schema overhead.** ~4,200 tokens of tool definitions load at session start; trimming
  capabilities to `core,testing` drops the vision/pdf/devtools/network tool schemas.
- **The whole report-assembly tail runs on the session model** at depth 0. A simple one-view
  change costs as much as a full feature because nothing is scoped or tiered.

Measured baseline (independent 2026 benchmarks): MCP default ≈ **114k tokens** per 8-step
authenticated workflow; up to **232k+** with screenshots; file-based CLI ≈ 27k. Target
saving from the config changes alone: **~50–70% of tokens per run.**

### 2. Churn / getting stuck

- **Cascade-cancel (Claude Code issue #22264).** When Claude issues several tool calls in one
  turn and any one is rejected — a denied permission prompt, a PreToolUse hook exit 2, or a
  runtime error — all sibling calls in that batch are cancelled ("Sibling tool call errored").
  The work must be re-read and retried individually: extra round-trips, tokens, latency.
- **Hook-block stall (issue #24327).** When `security-guard.sh` exits 2, Claude often pauses
  and waits for a nudge rather than autonomously adapting — treating the block like a user
  denial.
- **`cd` compound-command prompts.** Claude Code evaluates each sub-command of a compound
  (`cd X && …`) independently; `cd`+`git` always prompts. A prompt batched with Playwright
  calls collapses the whole batch.
- **Missing allow-list entries.** `runserver`, `find_available_port.sh`, `qa_cleanup.sh`, and
  `compress_screenshots.py` are **not** in `.claude/settings.json` today, so each prompts on
  every QA run — every prompt is a cascade-cancel risk.

### 3. Bugs are filed, not fixed

When `/do_qa` finds a clear bug today it records a todo for a human (Step 11 adds
`Fix QA bug: …` items). The SDD README carries an explicit TODO — *"Command for reacting to
QA-reported bugs"* — that has never been built. For clear, unit-testable regressions this is
wasted latency: the fix is mechanical and could happen in the same run.

## Goals

- **Cut tokens and wall-clock for the common case** (small changes) without losing the
  human-like exploratory quality that makes `/do_qa` valuable.
- **Stop the cascade-cancel / hook churn** so QA runs don't get stuck.
- **Auto-fix clear, fixable bugs TDD-style and re-verify** inside `/do_qa`, instead of only
  filing a human todo.

## Non-goals (this spec)

- Viewport parallelism / multi-session browser harness (**Phase 4** — deferred to a future
  spec, gated behind the `agent-browser` spike).
- A reusable Playwright test-runner fast-path for re-verification (attractive future idea;
  pytest already covers the cheap deterministic regression layer for the bug-fix loop).
- Replacing Playwright MCP wholesale.

## Constraints (these shape every decision)

These are hard properties of the FLS / Claude Code environment, verified against the actual
files:

- **Fan-out is depth-0 only.** Subagents cannot nest or spawn further subagents
  (`fls:sdd-worker` and `fls:sdd-mechanic` both state "You cannot spawn further subagents").
- **Playwright MCP is granted to `/do_qa` (depth 0) only.** `do_qa.md` frontmatter has
  `allowed-tools: …, mcp__playwright*`; `sdd-worker` (Read/Glob/Grep/WebFetch/WebSearch/Write),
  `sdd-mechanic` (Bash/Read/Edit/Write/Glob/Skill), and `qa-data-helper` (Glob/Grep/Read/
  WebFetch/WebSearch/Bash) have **no** Playwright. One MCP server = one browser, so the three
  viewports are serial within a single session. (This is *why* Phase 4 needs an external
  orchestrator and is out of scope here.)
- **Subagents can't call `AskUserQuestion`;** they fail-fast with `status: blocked`. All
  user interaction stays at depth 0.
- **Model tiering only works on spawnable agent files** (`model:` frontmatter). A
  command/helper followed *inline* keeps the caller's (session) model — this is the documented
  `/sdd:next` behaviour. So anything we want on Haiku/Sonnet must be a spawned agent.
- **`security-guard.sh` cannot be overridden by an allow-list.** It is a PreToolUse hook on
  `Bash|Write|Edit`; an exit-2 block takes precedence over allow rules. It blocks the literal
  string for recursive force-delete in Bash, plus `.env`/`id_rsa`/`.pem`/`.key` access in
  Bash, and a set of dangerous code substrings in Write/Edit content — the raw-SQL queryset
  escape hatches, unsafe-HTML marking, CSRF-exempt decorators, dynamic code-evaluation calls,
  insecure deserialization (pickle/yaml load), and request-dict-splat patterns — plus edits to
  `.env*` and existing migration files. **Prefer script-wrapping over weakening it** (the
  existing `qa_cleanup.sh` already wraps its delete so the hook never sees the literal). Note
  for authors: because the Write branch matches on substrings, documentation that quotes those
  exact tokens is itself blocked — describe them, don't paste them.
- **Pre-commit gate fires on `uv run git commit`.** The plugin's PreToolUse hook on
  `Bash(git commit:*)` runs `uv run ruff check . && uv run mypy . && uv run pytest --tb=short
  -q`. The `fls:qa-bugfixer` commit therefore runs the full suite + lint + types as a gate —
  this is the regression safety net for the bug-fix loop, and `git commit --no-verify` is
  denied in settings.

## Phase 1 — Cheap wins (config + scoping + resilience)

Mostly config/wording changes with the biggest payoff per unit of effort.

### 1.1 Tune the Playwright MCP server

Reconcile to a **single authoritative** Playwright MCP definition and add efficiency flags. The
config file and inline `args` work **together**, not as either/or alternatives:

- A committed **`--config` file holds the sensible defaults** that don't vary run-to-run:
  `browser.headless/isolated`, `imageResponses: "omit"`, `snapshot.mode: "incremental"`,
  `capabilities: ["core","testing"]`, and — **critically** — an `outputDir` pinned to the
  report's `screenshots/` directory.
- **Inline `args` are authoritative** for anything we need to vary per run, so we keep
  flexibility only where it's actually needed; inline args override the config file.

Inline args (pointing at the defaults config):

```json
{
  "mcpServers": {
    "playwright": {
      "type": "stdio",
      "command": "npx",
      "args": ["@playwright/mcp@latest",
        "--config", "<path-to-defaults-config>",
        "--headless", "--isolated",
        "--image-responses", "omit",
        "--caps", "core,testing"],
      "env": {}
    }
  }
}
```

**Non-negotiable:** however the knobs are split between the config file and inline args, the
screenshot **output directory MUST be pinned to the report's `screenshots/` directory**.
`--image-responses omit` only keeps the image *bytes* out of context — it does **not** choose
where the screenshot file lands. Without a pinned output dir the report's `![](screenshots/…)`
references won't resolve, breaking Success Criterion #1.

**Why each knob:**
- `--image-responses omit` — **highest-leverage single change.** It keeps screenshot image
  bytes out of context (est. 50–70% token cut). Screenshots are still *taken* and saved to the
  pinned `outputDir`, so the report can reference them as `![](screenshots/…)`.
- `snapshot.mode: incremental` — send only the diff from the previous snapshot; prevents
  full-tree resends accumulating across a multi-step run. (Already the recent default; set it
  explicitly in the config defaults to future-proof.)
- `--caps core,testing` — drops ~4.2k tokens of vision/pdf/devtools/network tool schema and
  discourages screenshot-based *interaction* tools.
- `--headless` — no visible window; saves CPU on unattended runs.
- `--isolated` — fresh in-memory profile per session; no state leakage between runs.

**Screenshot-based judgement stays available** (per the idea's decision). Mind the distinction:
the `vision` *capability* dropped by `--caps core,testing` is the screenshot-*coordinate
interaction* toolset (clicking by pixel position) — **not** the ability to capture or read
screenshots. Capturing stays available via core `browser_take_screenshot`, and the agent may
deliberately read a saved screenshot file back into context when it genuinely needs to judge a
visual layout (mobile/tablet overflow/overlap). `--image-responses omit` keeps those images
*out of context by default*; it does not remove the ability to read a saved screenshot file
when judgement requires it. Do **not** block CSS at the network level — the page must render
correctly for visual QA.

### 1.2 Diff-scoping gate (before the server starts)

Classify `git diff main...HEAD --name-only` and shrink the run accordingly:

- **`FULL`** — templates / frontend changed → all 3 viewports (current behaviour).
- **`ADMIN_ONLY`** — only Django admin touched → desktop only, no mobile/tablet (admin already
  exempt from mobile/tablet per Steps 6–7).
- **`BACKEND_ONLY`** — only Python view/model/logic, no templates/CSS/JS → single viewport
  smoke pass.

This is the **biggest wall-clock win for the common case** and largely removes the *need* for
viewport parallelism — which is precisely why Phase 4 can be deferred. The gate must be
explicit and reported in `qa_report.md` (which classification fired, and therefore what was
*not* run) so scoping is never silent.

### 1.3 Smoke gate (before the full matrix)

Load home + the primary changed page first. If there's a 500/404/missing critical element,
**abort early** and write a report saying so, instead of walking 20 tests after a crash-on-load.

### 1.4 Resilience / anti-churn

Add a **"Batching safety rules"** CRITICAL block near the top of `do_qa.md`:

- Never issue a Bash call in the same turn as a Playwright MCP call unless **both** are
  pre-approved in `.claude/settings.json`. If either could prompt or trip the hook, issue the
  Bash call solo first, wait, then proceed with Playwright.
- Never run a raw recursive force-delete directly — only committed wrapper scripts.
- Never `cd` before a command — use absolute or CWD-relative paths.
- The dev-server start (`runserver`) is always a solo call.
- `fls:qa-data-helper` (and any other) `Agent` spawn is always a solo call.

Add the **missing allow-list entries** to `.claude/settings.json` so the QA flow is prompt-free
(literal pattern strings; `$CLAUDE_PLUGIN_ROOT` is **not** expanded at match time, so use
resolved paths):

```json
"Bash(uv run python manage.py runserver *)",
"Bash(.claude/fls/scripts/find_available_port.sh*)",
"Bash(fls-claude-plugin/scripts/qa_cleanup.sh)",
"Bash(uv run --with pillow python * compress_screenshots.py *)"
```

Consolidating QA scripts under `.claude/fls/scripts/` would let one wildcard
(`Bash(.claude/fls/scripts/* *)`) cover them — **only if** the wildcard's space/`*` semantics
actually match the call forms; verify before relying on it. If a mid-run screenshot clear is
ever needed, add a committed `qa_clear_screenshots.sh` rather than a raw recursive delete.

## Phase 2 — Model tiering of the cheap tail

Tiering only works on spawned agent files, so move the **non-judgement** steps off the session
model:

- **Screenshot compression → `fls:sdd-mechanic` (Haiku).** Step 8's
  `compress_screenshots.py` invocation becomes a solo `fls:sdd-mechanic` spawn.
- **Report assembly → `fls:sdd-worker` (Sonnet).** Depth-0 maintains a structured scratch list
  as it goes — `{test_id, viewport, status, screenshot_path, notes}` — written to
  `.sdd-work/`. After testing, a single `fls:sdd-worker` renders `qa_report.md` **from the
  file** (path passed, contents never dumped into the prompt), keeping the long synthesis off
  the saturated depth-0 context.
- **Confirm the `todo.md` update is tiered.** Step 11 currently invokes `update_todo.md`
  inline (keeping the session model). Move it to an `fls:sdd-mechanic` spawn that follows the
  protected helper file, consistent with the README's tiering guidance.
- **Discipline:** pass **file paths, not snapshot/screenshot contents** between steps; never
  replay raw Playwright output back into depth-0 context.
- **Browser-driving stays at depth 0 on the session model** — that exploratory visual
  judgement is the core value and must **not** be tiered down.

> Risk noted during research: the `security-guard.sh` PreToolUse hook resolves
> `${CLAUDE_PLUGIN_ROOT}`; if that variable is unset in a spawned agent's environment, the
> agent's `Write` is blocked. The report-rendering and todo-tick fan-out in Phase 2 depend on
> spawned-agent `Write`/`Edit` working. The plan must verify spawned agents inherit a working
> hook environment (or adjust the hook to no-op when the var is unset) before relying on tiered
> file authoring.

## Phase 3 — Automatic TDD bug-fix + re-verify loop

Close the README's "Command for reacting to QA-reported bugs" TODO, **in-line inside
`/do_qa`** (triage → fix → re-verify in one run; not a separate human-gated command).

### 3.1 New spawnable agent `fls:qa-bugfixer`

- **Frontmatter:** `model: sonnet`; `tools: Bash, Read, Edit, Write, Glob, Grep`. Lives at
  `fls-claude-plugin/agents/qa-bugfixer.md`. Non-interactive, depth-1, one bug per spawn.
- **TDD sequence:** write a **failing** test reproducing the bug → confirm **RED** → make the
  **minimal** fix → confirm **GREEN** → run full `uv run pytest` → `uv run git commit` (the
  pre-commit hook fires ruff + mypy + pytest as the gate).
- **Reports** via a structured `.sdd-work/bugfix_<slug>.md`
  (`status` / root-cause / fix / commit / suite-result) and a one-line structured return.

### 3.2 Triage at depth 0 (before spawning)

`/do_qa` decides what is safe to auto-fix. **Green lane (auto-fix)** only when *all* hold:

- clear functional regression in the feature under test, **and**
- unit-testable without a browser, **and**
- root cause in one app, **and**
- no product/UX decision required, **and**
- no schema migration required, **and**
- not security-adjacent.

Otherwise **red lane** → record **UNRESOLVED** + a human todo (the existing Step 11
`Fix QA bug: …` mechanism).

### 3.3 Re-verify after a successful fix

- Trust the fixer's pytest pass for the regression layer (don't re-drive what pytest covers).
- Depth-0 re-drives **only the specific Playwright flow that failed**, plus 2–3 adjacent
  spot-checks **if** the fix touched shared code. **Not** the whole plan.

### 3.4 Loop guard

- **Max 1 fix attempt per bug per run.**
- On fixer failure / broken suite / re-check failure → **safe revert**
  (`git checkout -- <files>`; never `reset --hard` without explanation) and escalate to a
  human todo.
- `qa_report.md` ends with each bug marked **FIXED** (with commit hash) or **UNRESOLVED**.

### 3.5 README

Remove the "Command for reacting to QA-reported bugs" TODO and the "*(A dedicated bugfix
command is still TODO.)*" note; document that the loop now lives inside `/do_qa`.

## Out of scope / explicitly rejected

- **Phase 4 — viewport parallelism.** Genuinely parallel desktop/mobile/tablet needs an
  external orchestrator launching separate top-level Claude sessions, each with its own
  Playwright MCP server on its own runserver port, then a merge step — it cannot be done from
  inside one session or via subagents (no MCP at depth 1). Deferred to a future spec, and
  gated behind the **`agent-browser` go/no-go spike** (`vercel-labs/agent-browser` is a
  Bash-driven CLI that *could* be driven by depth-1 subagents, turning parallel QA into an
  ordinary depth-0 fan-out — but it's pre-1.0 with weekly breaking-change risk, true isolation
  needs separate Chrome processes anyway, and adopting it forfeits the Playwright
  codegen/Healer ecosystem). The Phase-1 diff-scoping gate already eliminates the viewport
  matrix for most runs, so this is the last lever to pull and only pays off when the full
  matrix is genuinely the bottleneck. See `research_agent_browser_alternative.md`.
- **Weakening `security-guard.sh`** to permit recursive force-delete — wrap deletions in
  committed scripts.
- **Tiering the browser-driving steps below the session model** — visual judgement is the point.
- **Replacing Playwright MCP wholesale** (Chrome DevTools MCP, browser-use) — MCP is the right
  tool for the exploratory role; we tune it, we don't replace it.
- **A reusable Playwright test-runner fast-path** for re-verification — attractive future idea
  (near-zero-token re-checks) but not part of this work; the pytest suite already covers the
  cheap deterministic regression layer for the bug-fix loop.

## Success criteria

1. **Token cut.** A small single-view change runs `/do_qa` at materially lower token cost than
   today — target ~50–70% reduction, driven primarily by `--image-responses omit` and
   `incremental` snapshots. Screenshots still appear in `qa_report.md` as file references.
2. **Scoped runs.** `ADMIN_ONLY` and `BACKEND_ONLY` diffs skip/shrink the mobile/tablet matrix;
   the report states which classification fired and what was therefore not run (no silent
   scoping).
3. **Smoke gate.** A crash-on-load (500/404/missing critical element) aborts early with a clear
   report instead of walking the full plan.
4. **No churn.** A full `/do_qa` run on a representative change completes without a
   permission-prompt-induced cascade-cancel or a hook-block stall — every QA-flow Bash command
   is either allow-listed or issued solo, and no raw recursive-delete/`cd`-compound appears.
5. **Tiering.** Screenshot compression runs on `fls:sdd-mechanic` (Haiku); `qa_report.md` is
   rendered by `fls:sdd-worker` (Sonnet) from a file; the todo tick runs on
   `fls:sdd-mechanic`. Browser-driving remains on the session model. No raw snapshot/screenshot
   contents are replayed into depth-0 context.
6. **Self-healing.** A planted clear, unit-testable regression in the feature under test is
   auto-fixed within one `/do_qa` run: a failing test is written first (RED), the fix lands
   (GREEN), the full suite passes via the commit gate, the failed Playwright flow is
   re-verified, and `qa_report.md` marks the bug **FIXED** with a commit hash.
7. **Triage holds the line.** A bug that fails any green-lane criterion (needs a migration, a
   UX decision, spans apps, or is security-adjacent) is left **UNRESOLVED** with a human todo —
   not auto-fixed.
8. **Loop guard holds.** At most one fix attempt per bug; a failed fix/suite/re-check triggers a
   safe `git checkout -- <files>` revert (never silent `reset --hard`) and a human todo.
9. **README TODO closed.** The "Command for reacting to QA-reported bugs" TODO is removed and
   the in-`/do_qa` loop is documented.
