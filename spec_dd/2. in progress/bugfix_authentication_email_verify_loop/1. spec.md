# Spec: Bugfix — authentication email-verification loop

## 1. Summary

A demo of the "enrol for free" journey exposed a chain of distinct auth defects. The headline
symptom — a user who resets their password but can never finish logging in ("the loop") — is
one of several defects on the unauthenticated → "enrol for free" → sign-up/login/verify/reset
pathways. This spec makes **all** of these pathways reliably land the user in the correct
authenticated state, while **preserving allauth's account-enumeration prevention absolutely**.

The underlying loop is an allauth-layer gap (the relevant settings pre-date recent work). It was
made *reachable for ordinary users* by the `2026-07-01_19:44_home_page` spec, which opened public
browsing and routed the anonymous "enrol for free" CTA into sign-up. The
`2026-07-03_07:52_courses_coming_soon` spec is ruled out — for a published/free course its
changes are no-ops on this path (see `research_suspect_specs.md`).

Full technical grounding (with `file:line` and allauth source citations) lives alongside this
spec in `research_auth_flow_map.md`, `research_allauth_enumeration_pitfalls.md`, and
`research_suspect_specs.md`. This spec is the decision record; the plan will carry the mechanics.

## 2. Root cause

With this project's allauth config (`config/settings_base.py:326-350`):
`ACCOUNT_EMAIL_VERIFICATION="mandatory"`, `ACCOUNT_LOGIN_ON_PASSWORD_RESET=True`,
`ACCOUNT_LOGIN_ON_EMAIL_CONFIRMATION=True`, `ACCOUNT_PREVENT_ENUMERATION=True`:

1. **Existing-email sign-up is intentionally silent.** Signing up with an email that already has
   an account does not error (enumeration prevention). Allauth sends an "account already exists"
   email pointing at the generic "forgot password" *request* page, and shows the caller the same
   "check your email" screen as a real signup (`research_auth_flow_map.md` §2, §2a).
2. **Link-based password reset does not verify the email.** Completing a reset via the emailed
   keyed link changes the password but never sets `EmailAddress.verified` — allauth's link token
   encodes only the user id, not which email is being reset
   (`research_allauth_enumeration_pitfalls.md` §3).
3. **Reset-triggered auto-login re-enters mandatory verification.** Because
   `ACCOUNT_LOGIN_ON_PASSWORD_RESET=True`, a completed reset attempts an immediate login that
   runs the full `LoginStageController` pipeline, including `EmailVerificationStage`. For an
   account whose email was never verified, that stage blocks the login and fires *another*
   verification email instead of logging the user in. Every subsequent login/reset attempt hits
   the same gate → the "loop". A 180s resend cooldown means some attempts send no email at all
   while still showing "check your email" (`research_auth_flow_map.md` §3, §S3, §S2).

## 3. Goals & success criteria

**Goal:** every sign-in / sign-up / verify / reset pathway — direct and via the course CTA, new
user and existing email, verified and unverified pre-existing account — reliably lands the user
in the correct authenticated state, with **no way for one user to learn whether another user's
email is registered**.

Success criteria — SC1–SC3, SC5 and SC6 are verified by automated tests (TDD — see §7).
SC4 is verified by a test only if the investigation confirms the defect is reachable; if it is
not reachable, it is discharged by a documented finding instead. SC7 is a process/guidance change,
verified by review rather than by a test:

- **SC1 (loop closed):** A user with an unverified pre-existing account who completes a password
  reset ends up **logged in**, not bounced back into mandatory email verification. Completing the
  reset marks their email verified.
- **SC2 (intent preserved):** A user who arrives via the course "enrol for free" CTA
  (`?next=<…/access/>`) and hits the existing-email branch is returned to the course they were
  enrolling in once they recover their account — not dropped at `/`.
- **SC3 (honest, non-leaking UX):** The "verification sent" screen and any resend/recovery entry
  point are truthful **and enumeration-safe**: they present identically whether or not the email
  is registered, and give a stuck user a reachable, unauthenticated way to get a fresh
  verification email.
- **SC4 (site-scoped reset resolves):** If confirmed real, a keyed password-reset link resolves
  the correct user regardless of the current request's resolved Site; a valid reset link never
  dead-ends as "invalid" purely due to site scoping. If investigation shows it is not reachable,
  that finding is documented and no change is made.
- **SC5 (no enumeration regression):** All fixes preserve `ACCOUNT_PREVENT_ENUMERATION`
  behaviour. No new response, timing, redirect, status code, template, or message reveals to the
  *requester* whether an email already has an account.
- **SC6 (no mandatory-verification bypass):** Fixes must not let an unverified account log in by
  any path *other than* one that genuinely proves inbox control (completing a reset, or clicking
  a real verification link). "Mandatory" still means mandatory for ordinary login/signup.
- **SC7 (process retro):** The SDD QA-planning guidance is updated to prompt side-effect /
  failure-mode reasoning, so branches like "existing account signs up again" are exercised.

## 4. Hard constraints (privacy / security)

These are non-negotiable and gate every design choice below:

- **C1 — Enumeration prevention is absolute.** The frontend must **never** tell a caller that an
  email is (or is not) already registered. Do not add a "this account already exists" message,
  a different redirect, a different status code, a distinguishable error, or a materially
  different response time on the existing-email branch. Any "account already exists" information
  only ever goes to the **real inbox owner** via email — never to the person driving the browser.
- **C2 — Verification stays meaningful.** Only an action that proves control of the inbox may
  mark an email verified: clicking a genuine verification link, or completing a keyed
  password-reset link delivered to that inbox. Nothing else.
- **C3 — Keep link-based reset.** Do not switch the product to code-based reset
  (`ACCOUNT_PASSWORD_RESET_BY_CODE_ENABLED`) as the fix — it is a broader UX change with its own
  template/flow review. (Recorded as a considered-and-rejected alternative, §6.)

## 5. Scope — the defects to fix

### 5.1 Defect 1 — the verification loop (primary)

**Requirement:** A completed keyed password reset must mark the account's email address
**verified**, so that the existing `ACCOUNT_LOGIN_ON_PASSWORD_RESET=True` auto-login passes the
mandatory `EmailVerificationStage` and logs the user straight in.

**Chosen approach & why:** Mark the reset email verified as part of finalising the password
reset. This is the smallest change that closes the loop, keeps link-based reset (C3), and
preserves enumeration prevention (C1). Completing a keyed reset proves control of the inbox, so
treating it as verification of that address is sound for this project's single-email-per-user
model (`User.email` is globally unique — `research_auth_flow_map.md` §6). This satisfies C2:
verification is granted only for proven inbox control.

**Notes for the plan (mechanics deferred):** the natural seam is the project's existing
`AccountAdapter` / the password-reset finalisation flow
(`allauth/account/internal/flows/password_reset.py::finalize_password_reset` →
`perform_password_reset_login`; see `research_auth_flow_map.md` §3). The email that was reset must
be the one marked verified. Guard the edges: an account with **no** `EmailAddress` row, an
account whose reset email differs from its primary address, and the ordering so the
`EmailVerificationStage` sees `verified=True` when the reset-triggered login runs.

### 5.2 Defect 2 — lost enrolment intent on the existing-email branch

**Requirement:** When a user reaches sign-up via the course CTA (carrying
`?next=<initiate_course_access URL>`) and hits the existing-email/enumeration branch, the
enrolment intent must survive so that, once they recover their account, they land back on the
course they were enrolling in.

**Why it matters:** Today `SignupView.form_valid()` returns the `prevent_enumeration()` response
*before* `get_success_url()` is reached, so the `?next=` is silently dropped
(`research_auth_flow_map.md` §S1, §4). Even a fully successful eventual login drops the learner at
`/` instead of the course.

**Constraint interaction (C1):** Preserving `next` must not become an enumeration oracle. The
mechanism that carries intent forward must behave **identically** for a brand-new email and an
existing email as far as the *requester's* browser can observe. Preferred direction: carry the
intent through channels the requester already controls and that are the same in both branches
(e.g. the reset link the real owner clicks resuming a stored destination, or `next` re-derivation
that does not depend on whether the account exists). The plan must show, explicitly, that its
chosen mechanism does not differ observably between the two branches.

### 5.3 Defect 3 — misleading, and potentially leaky, verification UX

**Requirement:** The "verification sent" experience must be **honest and enumeration-safe**, and
a stuck user must have a **reachable, unauthenticated way to get a fresh verification email**.

**Why it matters:** Today the "verification sent" screen shows identical copy whether a real
verify link, an "account already exists" reset email, or **no** email at all (cooldown) was sent
— and it actively tells the user to look for a "verification link" that may not be in their inbox
(`research_auth_flow_map.md` §S2, §S5). Users get stuck with no way forward.

**Chosen approach & why (honours the C1 override):**

- **On-screen copy is generic and identical in all branches.** Reword the verification-sent
  screen so it is *truthful without leaking*: it must not claim a specific "verification link"
  was sent, and must not imply anything about whether an account exists. Copy along the lines of
  "If we could set up or find your account, we've emailed you a link to continue — check your
  inbox (and spam). Didn't get anything? You can request another below." The exact same screen
  and wording render for a new email, an existing email, and a cooldown-suppressed send. **No
  branch may differ in what the browser shows.**
- **Provide an enumeration-safe resend/recovery entry point.** A stuck, unauthenticated user must
  be able to request another email from that screen. This entry point must itself be
  enumeration-safe: it always shows the same confirmation ("if an account for that address needs
  a link, we've sent one"), never confirms existence, and respects the existing resend cooldown.
  Prefer reusing allauth's existing resend / password-reset-request affordances over building new
  surfaces; the plan decides the concrete mechanism.
- **Email-body content (to the real inbox only) may be improved.** Because the "account already
  exists" email is delivered only to the genuine account owner — never shown to the requester —
  its copy may be made clearer/more helpful (e.g. guiding an unverified pre-existing account
  toward recovering access). This does not violate C1 because the requester never sees it. This
  is optional polish, not required for the loop fix.

**Explicitly forbidden (C1):** branching the *on-screen* response on whether the account exists
or is verified; any frontend text, redirect, or timing that lets the requester infer another
user's registration status. The research's "full adapter branching on the visible response"
option is **rejected** on these grounds.

### 5.4 Defect 4 — site-scoped password-reset dead-end (investigate, fix if confirmed)

**Requirement:** Investigate whether a keyed password-reset link can dead-end as
"invalid_password_reset" in production because the site-scoped `UserManager` makes
`UserTokenForm._get_user()` → `User.objects.get(pk=…)` raise `DoesNotExist` when the request's
resolved Site differs from the user's `site` FK (`research_auth_flow_map.md` §S4, §6). This is
masked in dev by `FORCE_SITE_NAME="DemoDev"`.

**Decision (user-confirmed):** *Investigate; fix within this bugfix if confirmed real.* If a
reproduction confirms it, ensure reset-key (and any related email-based) user resolution finds
the correct user regardless of the current request's Site — a valid reset link must never fail
purely due to site scoping (SC4). If investigation shows the path is not actually reachable
(e.g. another layer already constrains it), document that finding in the plan/QA and make no
change. Any fix must remain site-safe elsewhere — do not broadly unscope `User.objects` in a way
that leaks cross-site data; scope the change to the reset-key resolution seam.

### 5.5 SDD process retro (in scope for this spec)

**Requirement:** Update the SDD QA-planning guidance/skill so QA scoping explicitly reasons about
**side-effects and failure modes**, not just the golden path.

**Why it matters:** Every QA workflow in the recent specs used a **fresh, never-used email** for
sign-up, so the "existing account tries to sign up again" branch — the one this bug travels
through — was never exercised, even though `better-registration`'s own threat model had already
flagged enumeration prevention as a known concern (`research_suspect_specs.md` §4).

**Chosen approach & why:** The fix is **not** "check authentication on every spec" (most changes
don't touch auth). It is to make QA scoping more deliberate: when a change touches some
functionality, explicitly reason about *what else is affected and what could reasonably break* —
including unintended side-effects and failure/adversarial branches — and go check those. Keep it
**proportionate**: encourage side-effect/failure-mode thinking, don't mandate exhaustive
matrices. The plan identifies the specific QA-planning skill/guidance file(s) to amend and the
wording to add.

## 6. Non-goals & rejected alternatives

- **Not** switching to code-based password reset (`ACCOUNT_PASSWORD_RESET_BY_CODE_ENABLED`) — C3.
  It is the maintainer-endorsed built-in that verifies email on reset, but it changes reset UX
  from "click a link" to "enter a code" and needs its own template/UX review
  (`research_allauth_enumeration_pitfalls.md` §6.1). Recorded for a future decision if desired.
- **Not** loosening `ACCOUNT_EMAIL_VERIFICATION` to `"optional"` — that weakens the security
  posture and enumeration efficacy; the loop is fixable without it
  (`research_allauth_enumeration_pitfalls.md` §6.4).
- **Not** relaxing or removing `ACCOUNT_PREVENT_ENUMERATION` — it is working as intended; the
  defect is the mandatory-verification vs. link-reset gap, independent of enumeration prevention
  (`research_allauth_enumeration_pitfalls.md` §6.6). C1 keeps it in place.
- **Not** re-architecting the site-aware `UserManager` broadly (Defect 4 is scoped to the
  reset-key resolution seam only).
- **Not** building new auth surfaces where an existing allauth affordance can be reused.

## 7. How the work is done — TDD

Per the idea and project convention, fix this with **TDD**, exercising **all** sign-in/sign-up
pathways, not just the golden path:

- Direct sign-up page (not via CTA): new email; existing email.
- Sign-up via the course CTA: new email; existing email.
- For existing-email cases, both a **verified** and an **unverified** pre-existing account.
- The password-reset-then-login completion, asserting the user ends **logged in** (SC1) and, for
  the CTA path, **on the intended course** (SC2).
- Enumeration-safety assertions: the requester-visible response (screen, redirect, status) is
  **identical** for new-vs-existing email across every entry point (SC5).
- No mandatory-verification bypass for ordinary unverified login/signup (SC6).

Reproduce the end-to-end journeys manually with Playwright and Mailpit (dev,
`http://localhost:8025`) to confirm which email is actually sent; the automated tests assert on
the sent mail programmatically via the test mail backend (`mail.outbox` / the rendered message),
not via Mailpit (which only receives mail under the dev SMTP backend). Write failing tests that
expose each defect (RED), make them pass (GREEN), then refactor. Follow the `fls:testing` and
`fls:playwright-tests` skills; do not assert on CSS/styling (functionality only).
