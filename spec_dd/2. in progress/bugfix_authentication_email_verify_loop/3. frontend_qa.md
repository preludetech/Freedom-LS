# Frontend QA — authentication email-verification loop bugfix

Manual, browser-driven verification of the auth fixes. Automated tests cover the assertions
programmatically; this plan confirms the **real** end-to-end journeys a user takes, and — crucially
— confirms **which email actually lands in the inbox** (the bug hinges on that), using Mailpit.

## Setup

1. **Server** — pick a random free port and run the dev server (do **not** use the default port):
   - `PORT=$(.claude/fls/scripts/find_available_port.sh)`
   - `uv run python manage.py runserver $PORT`
   - Base URL: `http://127.0.0.1:$PORT`
2. **Site** — all testing uses the **DemoDev** site (the dev default; `FORCE_SITE_NAME="DemoDev"`).
3. **Mailpit** — the dev SMTP inbox is at `http://localhost:8025`. Every email the app "sends" lands
   here. Keep it open in a second tab; **clear it before each workflow** so you know which message a
   step produced.
4. **Admin** (for inspecting/creating accounts, if needed): `demodev@email.com` /
   `demodev@email.com`, Django admin at `http://127.0.0.1:$PORT/admin/`.
5. **A free, published course** — from `http://127.0.0.1:$PORT/courses/`, pick any course whose
   card shows an **"Enrol for free"** CTA (not "I'm interested" / not application-gated). Note its
   detail URL. If none exists, ask `fls:qa-data-helper` to create a published free course.
6. **Test emails** — this bug lives on the *existing-account* branch, so you will deliberately reuse
   emails. Use three throwaway addresses on `@email.com`:
   - `qa-newuser@email.com` — never used before (control).
   - `qa-unverified@email.com` — an account that exists but was **never verified** (the bug's
     victim). Create it by signing up once and **not** clicking any verification link (Workflow 1
     sets this up), or ask `fls:qa-data-helper` for an unverified account.
   - `qa-verified@email.com` — an account that exists **and** is verified (regression control). Ask
     `fls:qa-data-helper`, or sign up + click the verification link once.

Throughout: you are checking **behaviour and content**, never CSS/styling. When comparing the
"new email" vs "existing email" screens for enumeration safety, compare the **visible text,
redirect URL, and page** — nothing else.

## Workflow 1 — Create an unverified pre-existing account (fixture + baseline)

**Goal:** produce the exact account the bug strands, and observe today's honest-UX screen.

1. Clear Mailpit. Go to `http://127.0.0.1:$PORT/accounts/signup/`.
2. Sign up as `qa-unverified@email.com` with a password and name. Submit.
3. **Expect:** you land on the "check your email" verification-sent screen.
   - **Verify the new copy (Defect 3):** the page must **not** say "finalize the signup process" or
     tell you to look for a specific "verification link". It should read generically (e.g. "If we
     could set up or find your account, we've emailed you a link to continue… check your inbox and
     spam. Didn't get anything? You can request another link below.") and must show a **"Request
     another link"** entry point.
4. In Mailpit: confirm a genuine **verification** email arrived. **Do NOT click its link** — we want
   this account to stay unverified.
5. Screenshot the verification-sent screen (`desktop_wf1_verification_sent.png`).

## Workflow 2 — The loop is closed: unverified account resets password and ends up logged in (SC1)

**Goal:** the headline fix. A password reset must both verify the email **and** log the user in.

1. Clear Mailpit. Go to `http://127.0.0.1:$PORT/accounts/password/reset/`.
2. Enter `qa-unverified@email.com`. Submit. **Expect** the generic "we've sent a link if needed"
   confirmation.
3. In Mailpit, open the **password-reset** email and click its keyed link. Set a new password and
   submit.
4. **Expect (the fix):** you are **logged in** immediately — you land on an authenticated page
   (dashboard/course), **not** bounced back to the "verify your email" screen. Before the fix, this
   step looped back to verification.
5. Confirm you are authenticated (your avatar/nav shows a logged-in state; visiting `/` does not
   redirect you to login).
6. **Confirm verification stuck:** in admin (`/admin/`), the `qa-unverified@email.com` EmailAddress
   now shows **verified = true**. Screenshot the logged-in landing (`desktop_wf2_logged_in.png`).

## Workflow 3 — Enrolment intent survives the existing-email branch, via the course CTA (SC2)

**Goal:** a user who tries to "enrol for free" using an email that already exists still lands on the
intended course once recovered — in the **same browser session**.

1. Clear Mailpit. Make sure you are **logged out**. Create a fresh unverified account to victimise:
   sign up as `qa-loop2@email.com` (as in Workflow 1) but do not verify. (Or reuse an existing
   unverified account and skip to step 2 in the same browser.)
2. Log out. In the **same browser**, open the free course detail page and click **"Enrol for
   free"**. Because you are anonymous, you are routed to login/sign-up carrying
   `?next=…/access/`.
3. Follow the "Sign up" link and sign up **again with the same existing email** (`qa-loop2@email.com`).
   **Expect:** the same generic verification-sent screen (no "this email is taken" error — that is
   correct, enumeration-safe behaviour).
4. Clear Mailpit is **not** needed here — instead, recover the account: go to
   `http://127.0.0.1:$PORT/accounts/password/reset/`, request a reset for `qa-loop2@email.com`,
   open the reset email in Mailpit, click the link, set a new password — **all in the same browser
   session**.
5. **Expect (the fix):** after the reset completes and you are logged in, you land on the **course**
   you were enrolling in (the course home/first item), **not** the generic dashboard `/`. You should
   be enrolled in that course.
6. Screenshot the landing (`desktop_wf3_course_landing.png`).

> Note: if the reset link is opened in a *different* browser/device (fresh session), landing on the
> course is best-effort and may fall back to `/` — that is expected and matches the new-user path's
> own session behaviour. Do this workflow in one session.

## Workflow 4 — Enumeration safety: new vs existing email look identical (SC5, C1)

**Goal:** prove the browser cannot tell whether an email is registered.

1. Clear Mailpit. Log out. From the free course, click "Enrol for free" → Sign up with a **brand-new**
   email `qa-enum-new@email.com`. Record: the exact screen text, the URL you end on, and the page.
2. Clear Mailpit. Log out. Repeat the identical steps but sign up with the **existing**
   `qa-verified@email.com` (or `qa-unverified@email.com`).
3. **Expect:** the two runs are **indistinguishable** in the browser — same verification-sent page,
   same wording, same redirect URL, same status. (The *emails* differ in Mailpit — a real
   verification link vs an "account already exists" notice — but that difference is only ever
   visible in the inbox, never in the browser.)
4. Screenshot both screens side by side (`desktop_wf4_new.png`, `desktop_wf4_existing.png`) to show
   they match.

## Workflow 5 — Resend/recovery entry point is reachable and enumeration-safe (SC3)

**Goal:** a stuck, logged-out user can get a fresh link, and that entry point never confirms
existence.

1. From the verification-sent screen (reach it via any signup), click **"Request another link"**.
2. **Expect:** you land on the password-reset **request** page (`/accounts/password/reset/`),
   reachable while logged out.
3. Submit a **registered** email, then (clear Mailpit, repeat) an **unregistered** email
   (`qa-nobody@email.com`). **Expect:** the on-screen confirmation is **identical** in both cases and
   never says whether the account exists. In Mailpit, a message arrives only for the registered one.

---

## Workflow 6 — Regression: mandatory verification still blocks ordinary login (SC6)

**Goal:** the fix must not let unverified accounts in through the front door.

1. Clear Mailpit. Log out. Create a fresh unverified account `qa-block@email.com` (sign up, do not
   verify).
2. Go to `http://127.0.0.1:$PORT/accounts/login/` and log in with those credentials **without**
   having reset the password or clicked any verification link.
3. **Expect:** you are **not** logged in — you are sent to the verification-sent screen (mandatory
   verification still enforced). Only a completed reset (Workflow 2) or a clicked verification link
   may verify an account.

## Workflow 7 — (Investigation, conditional) site-scoped reset link — Defect 4

This defect is masked in dev by `FORCE_SITE_NAME="DemoDev"`, so it **cannot** be reproduced through
the normal DemoDev browser flow. It is investigated by an automated multi-site test in the plan.
No manual browser step is expected here — note in the report that Defect 4 is covered by the
investigation test, and record its outcome (fixed with a regression test, or documented as
not-reachable).

## Mobile / tablet

The only visual surface this bugfix changes is the **verification-sent screen** copy and its
"Request another link" control. On a 375×812 (mobile) and 768×1024 (tablet) viewport, confirm that
screen and the password-reset request page render readably and the "Request another link" control is
tappable. No other responsive checks are needed (the rest is stock allauth pages / existing course
pages).

## Reporting

For each workflow, record pass/fail with a screenshot. For any failure, state the workflow, the
expected vs actual behaviour, and attach the relevant Mailpit screenshot (which email actually
arrived is central to this bug). Clean up: kill the server with
`.claude/fls/scripts/kill_runserver.sh $PORT`.
