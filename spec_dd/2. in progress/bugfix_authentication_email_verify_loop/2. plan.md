## 1. Defect 1 — completed keyed password reset must verify the email (primary, SC1)

### Mechanism (confirmed from vendored source)

`PasswordResetFromKeyView.form_valid()` (`allauth/account/views.py:657-664`) runs:

```python
def form_valid(self, form):
    form.save()                                              # ResetPasswordKeyForm.save() → sets new password
    resp = flows.password_reset.finalize_password_reset(     # → perform_password_reset_login → LoginStageController
        self.request, self.reset_user)
    if resp:
        return resp
    return super().form_valid(form)
```

`finalize_password_reset` (`allauth/account/internal/flows/password_reset.py:38-61`) never touches
`EmailAddress.verified`; with `ACCOUNT_LOGIN_ON_PASSWORD_RESET=True` it calls
`perform_password_reset_login`, which runs the full login pipeline including the mandatory
`EmailVerificationStage`. For an unverified account that stage blocks the login → the loop.

**`form.save()` runs before `finalize_password_reset`.** So if a custom
`ResetPasswordKeyForm` marks the reset user's email verified inside its `save()`, the subsequent
reset-triggered login sees `verified=True` and logs the user straight in. Ordering is guaranteed by
the view — no signal/adapter timing games needed.

### Change

New form in `freedom_ls/accounts/forms.py`, subclassing allauth's `ResetPasswordKeyForm`:

```python
from allauth.account.forms import ResetPasswordKeyForm
from allauth.account.models import EmailAddress

class SiteAwareResetPasswordKeyForm(ResetPasswordKeyForm):
    """On a completed keyed reset, mark the reset address verified.

    Completing a keyed reset proves control of the inbox, so the address it
    was delivered to is verified. This lets the ACCOUNT_LOGIN_ON_PASSWORD_RESET
    auto-login pass the mandatory EmailVerificationStage instead of looping.
    """
    def save(self) -> None:
        super().save()                      # resets the password (unchanged)
        user = self.user
        if user is None:
            return
        # Single-email-per-user model: User.email is globally unique. Mark the
        # address that equals user.email verified — this is the reset target.
        address, _created = EmailAddress.objects.get_or_create(
            user=user,
            email__iexact=user.email,
            defaults={"email": user.email, "verified": True},
        )
        if not address.verified:
            address.set_verified(commit=True)   # allauth EmailAddress.set_verified()
```

**Do not** put `"primary": True` in `defaults`. It is not needed to close the loop —
`EmailVerificationStage` calls `has_verified_email(user, login.email)` with `login.email=None` on the
reset-triggered login, which only checks whether *any* of the user's addresses is verified
(`allauth/account/utils.py`), not the primary one. Worse, forcing `primary=True` on the create path
can violate allauth's `unique_primary_email` constraint when the user already has a different primary
row, raising an unhandled `IntegrityError` (a 500 in the exact flow this fix is meant to rescue).
This mirrors allauth's own `sync_email_address`, which deliberately does not force `primary`.

Register in `config/settings_base.py`:

```python
ACCOUNT_FORMS = {
    "signup": "freedom_ls.accounts.forms.SiteAwareSignupForm",
    "reset_password_from_key": "freedom_ls.accounts.forms.SiteAwareResetPasswordKeyForm",  # pragma: allowlist secret
}
```

**Edge guards (spec §5.1):**
- *No `EmailAddress` row* → `get_or_create` creates one, verified (not forced primary — the
  `defaults` deliberately omit `"primary"`, matching the code above and its `unique_primary_email`
  warning).
- *Reset email differs from primary* → in this project `user.email` is the single canonical
  address; we verify the address that equals `user.email`. Use `email__iexact` to match allauth's
  case handling; the `defaults` carry the canonical-cased `user.email`.
- *Ordering* → handled by the view calling `form.save()` before `finalize_password_reset` (above).
- Do **not** flip any *other* user's address, and do not set `verified` from any path other than a
  genuinely completed reset (preserves C2 / SC6).

### Test data factory (add first)

The Defect 1 and Defect 4 tests need `allauth.account.models.EmailAddress` rows in various states
(verified/unverified, present/absent). `freedom_ls/accounts/factories.py` has **no** `EmailAddress`
factory today, and `fls:testing` forbids inline `.objects.create()` in tests. Add an
`EmailAddressFactory` (a plain `factory.django.DjangoModelFactory` — `EmailAddress` is **not** a
site-aware model, so do not subclass `SiteAwareFactory`) with a `verified` param, and build all
email-address fixtures through it.

### Tests (RED first)

In `freedom_ls/accounts/tests/` (new `test_password_reset_verifies_email.py`, or extend an
existing reset test module if one exists):

- Unverified pre-existing account completes a keyed reset → assert its `EmailAddress.verified` is
  `True` **and** the response leaves the user authenticated (`response.wsgi_request.user.is_authenticated`,
  or a follow-up request to a `@login_required` page returns 200). **SC1.**
- Account with **no** `EmailAddress` row completes a reset → a verified primary row now exists and
  the user is logged in.
- Already-verified account completes a reset → still logged in, still verified (no regression).
- **SC6 guard:** an ordinary unverified account that *logs in normally* (no reset) is still blocked
  by `EmailVerificationStage` — resetting is the only new way an unverified account gets verified.
- Drive the reset end-to-end through allauth's views/URLs with the Django test `Client` (request
  the reset key via `EmailAddress`/allauth helpers, POST the new password to
  `account_reset_password_from_key`), asserting on `mail.outbox` where mail is involved — **not**
  Mailpit (tests use the locmem backend).

## 2. Defect 2 — enrolment intent survives the existing-email branch (SC2, must stay enumeration-safe)

### Why intent is lost today

On the existing-email branch, `SignupForm.try_save()` calls
`flows.signup.prevent_enumeration(request, email=email)` which builds `Login(user=None, email=…)`
**with no `redirect_url`** (`allauth/account/internal/flows/signup.py:65-69`) and returns before
`SignupView.form_valid()` reaches `get_success_url()` (`allauth/account/views.py:143-146`). The
`?next=<…/access/>` is dropped. The genuine-new-user branch, by contrast, threads `next` onto
`Login.redirect_url` (stashed in the **session** by the stage machinery). So the new-user path is
**already session-bound** — using the session for the existing-email path is symmetric with existing
behaviour, not a new fragility.

### Chosen mechanism (enumeration-safe by construction)

Stash the validated enrolment `next` in the **session** *before* the account-exists branch is
decided, then consume it when the user next authenticates if allauth supplied no explicit
redirect. Because the stash happens on the common code path (identical write for new-vs-existing
email) and lives only in server-side session state, **the requester's browser observes nothing
different between the two branches** (C1 / SC5).

**Shared helper first (DRY).** The open-redirect guard already exists inline in
`_safe_post_completion_redirect` (`freedom_ls/accounts/views.py:60-68`). The stash and consume steps
below need the *same* validation, so **extract it once** rather than copy it a third time (CLAUDE.md:
"favor extracting it into a new function/class"). Add to `freedom_ls/accounts/utils.py` (already
imported by `forms.py`, no circular-import risk):

```python
def is_safe_next_url(request: HttpRequest, candidate: str | None) -> str | None:
    """Return `candidate` if it is a safe same-host redirect target, else None."""
    if candidate and url_has_allowed_host_and_scheme(
        candidate,
        allowed_hosts={request.get_host()},
        require_https=request.is_secure(),
    ):
        return candidate
    return None
```

Refactor `_safe_post_completion_redirect` to call it too, so all three sites share one guard. Also
define the session key once — put `SIGNUP_NEXT_SESSION_KEY = "signup_enrolment_next"` in
`forms.py` and import it in the adapter (no circular import: `forms.py` does not reference the
adapter module).

**Stash** — override `try_save` in `SiteAwareSignupForm` (`freedom_ls/accounts/forms.py`). `try_save`
is allauth's common entry for *both* branches (`allauth/account/forms.py:411-428`), so the write is
branch-independent. Signature/return type match allauth's own `BaseSignupForm.try_save`:

```python
def try_save(self, request: HttpRequest) -> tuple[User | None, HttpResponse | None]:
    # Stash a safe in-flight destination BEFORE allauth decides new-vs-existing,
    # so both branches write identically (no enumeration signal). Consumed by the
    # adapter's login-redirect when allauth itself supplies no redirect_url.
    candidate = is_safe_next_url(request, request.POST.get("next") or request.GET.get("next"))
    if candidate:
        request.session[SIGNUP_NEXT_SESSION_KEY] = candidate
    else:
        # A later, next-less signup in the same session must not inherit an older,
        # abandoned CTA stash — clear it so an unrelated login can't be redirected to
        # a course the user is no longer enrolling in. (pop-on-consume alone doesn't
        # cover the "earlier signup abandoned, unrelated login later" case.)
        request.session.pop(SIGNUP_NEXT_SESSION_KEY, None)
    return super().try_save(request)
```

**Consume** — override `get_login_redirect_url` in `AccountAdapter`
(`freedom_ls/accounts/allauth_account_adapter.py`). allauth only calls this when the `Login` object
carried no explicit `redirect_url` (the genuine-new-user path keeps its own `redirect_url`, so this
does not disturb it):

```python
def get_login_redirect_url(self, request: HttpRequest) -> str:
    stashed = is_safe_next_url(request, request.session.pop(SIGNUP_NEXT_SESSION_KEY, None))
    if stashed:
        return stashed
    return super().get_login_redirect_url(request)
```

Notes:
- **Re-validate on consume**, not just on stash — session contents are attacker-influenceable in
  principle; the open-redirect guard must hold at the point of use (hence `is_safe_next_url` is
  called on both stash and consume).
- `pop` on consume so a stash cannot be reused across two logins; **and** clear-on-empty-stash in
  `try_save` (above) so an older, abandoned CTA stash cannot be inherited by an unrelated,
  next-less signup/login later in the same session. Both are needed — `pop` alone leaves the
  earlier-abandoned-then-unrelated-login case open.
- `RegistrationCompletionMiddleware` may still insert one `complete_registration?next=…` hop before
  the course (`research_auth_flow_map.md` §5); it already forwards `next`, so the destination
  survives that hop. No middleware change needed.

### Enumeration-safety argument (must be shown explicitly, spec §5.2)

The only state written differently between "new email" and "existing email" is entirely
server-side (the same session key, written on the same common line before the branch). No response
body, status code, redirect, template, or timing differs for the requester. The reset link the real
owner later clicks carries no course info and is not needed to be — the destination is recovered
from *that user's own session* if still present, else falls back to `LOGIN_REDIRECT_URL` (`/`),
exactly as the new-user path degrades when its session is lost.

### Tests (RED first) — extend `freedom_ls/accounts/tests/test_deferred_login.py`

- Existing-email signup arriving with `?next=<course access url>` → after the account is recovered
  and logged in **in the same session**, the user lands on the course access URL. **SC2.**
- **SC5 enumeration parity — two cases (spec §7 bullet 1 requires both):**
  1. **Direct sign-up page (no CTA, no `next`):** POST signup with a *new* email vs an *existing*
     email — assert the requester-visible response (status code, redirect `Location`, rendered
     template/content) is **identical**. This is the baseline branch that currently has **no**
     coverage (spec §8) — the exact gap that let this bug ship.
  2. **Via the CTA (both carrying the same `?next=`):** same identical-response assertion.
  Both are headline anti-enumeration tests and also cover Defect 3's screen (below). Include an
  existing-account variant for **both** a verified and an unverified pre-existing account (spec §7).
- Open-redirect: an off-host `next` is ignored on both stash and consume (falls back to `/`).
- **Stale-stash isolation:** a CTA signup stashes `next=<course>`, is then abandoned; a later
  next-less signup/login in the **same session** does **not** land on the stale course URL (the
  `else`-clear in `try_save` wiped it) — it goes to the default `LOGIN_REDIRECT_URL`.

## 3. Defect 3 — honest, non-leaking verification UX (SC3, SC5)

### Change A — override the verification-sent screen copy

The project does **not** currently override `verification_sent.html` (only
`allauth/templates/account/verification_sent.html` exists; the project already overrides
`account/signup.html` under `freedom_ls/base/templates/account/`). Add a project override:

`freedom_ls/base/templates/account/verification_sent.html` — extend the same allauth base the stock
template uses (`account/base_entrance.html`) so layout/branding is inherited. Rewrite the copy to be
truthful without leaking (spec §5.3): it must **not** claim a specific "verification link" was sent
and must **not** imply anything about whether an account exists. Example copy:

> **Check your email**
> If we could set up or find your account, we've emailed you a link to continue. Check your inbox
> (and your spam folder). Didn't get anything? You can request another link below.

The **identical** screen renders for a new email, an existing email, and a cooldown-suppressed send
— because allauth already routes all three to `respond_email_verification_sent`
(`adapter.py:694-695` → `account_email_verification_sent`); we only change the copy, not the
branching, so no branch can differ.

### Change B — enumeration-safe resend / recovery entry point

From the verification-sent screen, link to the **existing** allauth password-reset **request** page
`account_reset_password` (reverse `"account_reset_password"`), labelled generically ("Request
another link"). Rationale:
- It works **unauthenticated** and is already **enumeration-safe** (`ResetPasswordForm` shows the
  same "we've sent one if needed" confirmation regardless of existence,
  `research_allauth_enumeration_pitfalls.md` §1) and respects allauth rate limits.
- With Defect 1 in place, completing that reset both sets a password **and** verifies the email, so
  it genuinely unsticks an unverified pre-existing account — coherent with the generic copy.
- Reuses an existing surface — satisfies the non-goal "not building new auth surfaces" (spec §6).

Do **not** add a bespoke "resend verification" view. Do **not** branch the screen on account
state.

### Change C (optional polish) — "account already exists" email body

`freedom_ls/accounts/templates/account/email/account_already_exists_message.html` (and its `.txt`
twin) go **only to the real inbox owner**, never to the requester, so their copy may be made more
helpful (e.g. explicitly guiding an unverified pre-existing account to use the reset link to regain
access). This does not touch C1 (requester never sees it). Mark clearly as optional — it is not
required to close the loop. Keep both the `.html` and `.txt` variants in sync.

Follow `brand-guidelines` for tone/voice and `fls:template` for the template mechanics; do not
assert on styling in tests.

### Tests (RED first)

- GET `account_email_verification_sent` → the rendered page does **not** contain the misleading
  "finalize the signup process" / "verification link" wording, **and** contains a link to
  `reverse("account_reset_password")`. (Assert on functional content/links, not CSS.)
- Covered by the **SC5 parity test** in §2: the screen is byte-identical for new-vs-existing email.

## 4. Defect 4 — site-scoped password-reset dead-end (INVESTIGATE, fix only if confirmed) (SC4)

### Investigation first (this is a gate, not a given)

`UserTokenForm._get_user()` resolves the reset link's user via `User.objects.get(pk=pk)`
(`allauth/account/forms.py:668-674`), which goes through the project's **site-scoped** `UserManager`
(`freedom_ls/accounts/models.py:52-60`). Hypothesis (`research_auth_flow_map.md` §S4): if the
request's resolved Site differs from the user's `site` FK, the lookup raises `DoesNotExist` and the
reset form reports `invalid_password_reset`.

**Reproduction test (write this first).** The test settings pin `FORCE_SITE_NAME="DemoDev"`
(`config/settings_dev.py:107`, used as the test settings module — `pyproject.toml:278`), which masks
this. So the test must **override** `FORCE_SITE_NAME` (via the `settings` fixture / `override_settings`)
and drive host-based site resolution:

1. Create Site A and Site B; create a user whose `site` FK is A.
2. Request a reset for that user; obtain the keyed link.
3. Submit the keyed link with the request resolving to Site **B** (Host header / forced site) and
   assert whether it dead-ends as `invalid_password_reset`.
4. **Assert the session persists, not just the first hop.** After the reset POST, make a
   **follow-up** request to a `@login_required` page **still resolving to Site B** and assert the
   user is still authenticated. This is required because the reset-key `_get_user` bypass only fixes
   the *first* lookup: on every subsequent request Django re-resolves `request.user` via
   `Backend.get_user()` → `User._default_manager.get(pk=…)`, and `User._default_manager` **is** the
   site-scoped `UserManager` (`User` declares only `objects = UserManager()`, so `_default_manager`
   is not the unscoped `_base_manager`). If Site B still mismatches `user.site`, that re-resolution
   raises `DoesNotExist`, the backend returns `None`, and the browser is silently anonymous on the
   next page — a quieter version of the same dead-end. A test that stops at step 3 would miss this.

**Decision rule (spec §5.4):**
- **If step 3 dead-ends but step 4 passes** (the reset-key bypass alone gets the user through *and*
  the follow-up request stays authenticated) → implement the fix below and keep the test (steps 1–4)
  as a regression guard. This is the clean "confirmed and fully fixed" outcome.
- **If step 3 dead-ends AND step 4 also fails** (session silently drops to `AnonymousUser` on the
  next request because `_default_manager` re-resolution still mismatches) → **do not widen the fix**
  to touch project-wide per-request user re-resolution (user decision, 2026-07-11: that change is
  too tenant-isolation-sensitive for this bugfix). Instead: keep the narrow reset-key `_get_user`
  fix (it's still a strict improvement — it gets the user through the reset screen), and **document
  the residual limit** — a genuinely mismatched-Site reset link is not fully recoverable within this
  bugfix — in this §4 and in the test (as an explanatory assertion/xfail-with-reason on step 4).
  **Re-scope SC4's claim** to "resolves the reset key correctly" rather than "guarantees a persisted
  logged-in session across a Site mismatch", and note the follow-up work (per-request re-resolution
  or ensuring reset links always resolve to the home Site) as a separate future item. Do not silently
  ship a fix that looks complete but leaves step 4 red.
- **If it does not reproduce at all** (e.g. the reset link's own domain always re-resolves to the
  user's home Site, so a valid link never lands on a mismatched Site in practice) → **make no code
  change**; record the finding (why it is unreachable) in a short note in this plan's §4 and in the
  test as an explanatory docstring/xfail-with-reason. Do not add speculative code.

> **Security concern:** The `SiteUnscopedUserTokenForm._get_user` bypass only unscopes the
> **reset-key lookup**. It does **not** fix per-request user re-resolution: `django.contrib.auth` (and
> the allauth/axes backends that subclass `ModelBackend` without overriding `get_user`) re-resolve
> `request.user` on every request via `User._default_manager.get(pk=…)`, which is the site-scoped
> `UserManager`. So even with this fix, a genuinely mismatched-Site reset gets the user *through* the
> reset screen but the session may silently drop to `AnonymousUser` on the very next request (step 4
> above catches this). Widening the fix to cover per-request re-resolution would change how **every**
> authenticated request re-identifies the user project-wide — materially bigger and more
> tenant-isolation-sensitive than "scope the bypass to reset-key resolution". **Resolved (user
> decision 2026-07-11):** do **not** widen the fix — keep the narrow reset-key bypass, and if step 4
> reproduces the drop-out, document the residual limit and re-scope SC4 per the Decision rule below,
> rather than changing project-wide user re-resolution.

### Fix (only if confirmed)

Register a custom `user_token` form scoped **only** to reset-key resolution — do **not** broadly
unscope `User.objects` (would leak cross-site data, spec §6):

```python
from allauth.account.forms import UserTokenForm
from allauth.account.utils import url_str_to_user_pk

class SiteUnscopedUserTokenForm(UserTokenForm):
    def _get_user(self, uidb36: str) -> User | None:
        try:
            pk = url_str_to_user_pk(uidb36)
            # _base_manager is the vanilla, un-site-scoped manager; the reset
            # token is a proof of inbox control, so resolving the user must not
            # depend on the current request's Site. Scope of this bypass is
            # exactly reset-key resolution — nothing else is unscoped.
            return User._base_manager.get(pk=pk)
        except (ValueError, User.DoesNotExist):
            return None
```

```python
ACCOUNT_FORMS = {
    ...,
    "user_token": "freedom_ls.accounts.forms.SiteUnscopedUserTokenForm",
}
```

Confirm `User._base_manager` bypasses the site filter (Django's default `_base_manager` is a plain
`Manager`, not `UserManager`, so it has no `get_queryset` site filter). Add a test asserting the
resolved user is correct **and** that this seam does not widen any other query's Site scope.

### Finding (batch 4, 2026-07-11)

Investigated and **confirmed reachable** — reproduction in
`freedom_ls/accounts/tests/test_site_scoped_reset.py`. Verified `User._base_manager` is Django's
plain, unscoped `Manager` and `User._default_manager` is the site-scoped `UserManager` (`User`
declares only `objects = UserManager()`), matching this section's assumption exactly.

Ran the full 4-step reproduction (RED first, without the fix): step 3 dead-ended as
`invalid_password_reset` (200 token-fail render instead of the 302 key-in-session redirect) when
the request resolved to a Site other than the user's `site` FK. Implemented the
`SiteUnscopedUserTokenForm` fix exactly as specified above and registered it as the `user_token`
`ACCOUNT_FORMS` key. Re-ran: step 3 now passes (the reset key resolves and the reset completes,
logging the user in for that response). Step 4 — a follow-up request, still resolving to the
mismatched Site, to a `@login_required` page — **fails**: the session drops to `AnonymousUser`
(redirected to login), because `django.contrib.auth`'s per-request `get_user()` re-resolves via
`User._default_manager` (the same site-scoped `UserManager`) on every request, exactly as the
boxed Security concern above predicted.

This is **Decision rule branch 2** (step 3 dead-ends and step 4 also fails). Per the branch 2
disposition: the narrow reset-key fix is kept (it is still a strict improvement — it resolves the
reset key correctly and lets a mismatched-Site reset complete instead of dead-ending outright), no
widening to per-request re-resolution was made (user decision 2026-07-11 stands), and the residual
limit is documented in the test module's docstring and in a dedicated regression test
(`test_reset_key_resolves_but_session_does_not_survive_a_site_mismatch`, step 4 assertion) rather
than left silently unaddressed.

**SC4 is re-scoped** accordingly: it now reads as "a keyed password-reset link resolves the
correct user regardless of the current request's resolved Site" (confirmed, fixed) rather than
"guarantees a persisted logged-in session across a Site mismatch" (not achieved, and out of scope
for this bugfix). Ensuring reset links always resolve to the user's home Site, or a project-wide
change to per-request user re-resolution, is left as separate future work.

## 5. Process retro — SDD QA-planning side-effect reasoning (SC7, §5.5)

Not a code/runtime change and **not** verified by a test — verified by review. Amend the QA-planning
guidance so QA scoping reasons about side-effects and failure/adversarial branches, not just the
golden path.

**Primary file:** `fls-claude-plugin/commands/sdd/plan_from_spec.md`, the "Step 5" section that
authors `3. frontend_qa.md`. Add proportionate wording, e.g.:

> When scoping QA, don't only walk the golden path. For each area of functionality the change
> touches, explicitly reason about what *else* is affected and what could reasonably break —
> including unintended side-effects and failure/adversarial branches (e.g. "an existing account
> signs up again", enumeration/permission branches, invalid input, repeat submissions) — and add QA
