# Implementation Plan: All-courses list page & course-card simplification

Derived from `1. spec.md`. Two complementary work streams:

- **Part A** ã redesign the `all_courses` page as a flat list of compact
  `[icon][details]` rows; make the four statuses correct for every course; derive
  status + progress from a constant set of batched queries (kill the N+1).
- **Part B** ã clean, themeable card/row templates: tokenise card shape, extract
  duplicated markup into one place, add named slots + a mergeable `class`, split leaf
  templates so one state = one file, and document the override ladder.

**Guardrails carried from the spec (apply to every task):**

- **No new features**; no behaviour change to navigation/registration/preview.
- **Dashboard must stay pixel-identical.** Every dashboard-touching refactor must emit
  byte-identical (or visually identical) markup and is gated by the existing card tests
  *and* QA.
- **Do not** reintroduce "Next up:" on the all-courses rows.
- **Do not** wrap a whole row in one `<a>`; use the stretched-link pattern.
- **Do not** rely on colour alone for status (WCAG 1.4.1).
- **Keep the `--fls-course-accent-*` palette + `color-mix` derivation unchanged.**
- Follow TDD: write a failing test first, then implement.

## Part A ã view, helper, and batched status derivation

### Task A1 ã New status-derivation helper in `student_interface/utils.py`

Single, unit-testable helper that returns each course's status + percentage from a
**constant** number of batched queries, with **zero per-course queries**.

**New definitions in `utils.py`:**

```python
from enum import StrEnum
from dataclasses import dataclass

class CourseListingStatus(StrEnum):
    NOT_REGISTERED = "not_registered"
    REGISTERED = "registered"        # registered, 0%, not complete
    IN_PROGRESS = "in_progress"      # registered, >0%, completed_time is None
    COMPLETE = "complete"            # registered, completed_time is not None

@dataclass(frozen=True)
class CourseListingEntry:
    course: Course
    status: CourseListingStatus
    progress_percentage: int
```

> Note: the existing module-level constants `IN_PROGRESS = "IN_PROGRESS"` /
> `COMPLETE = "COMPLETE"` are **content-item** statuses (different domain). The new
> course-listing states live on the `CourseListingStatus` enum so there is **no
> collision** and the two concepts stay clearly separated.

**Helper (pure-Python classification over batched reads):**

```python
def get_course_listing(user: RequestUser) -> list[CourseListingEntry]:
    """Status + progress for every course, from a constant set of batched queries.

    Query budget is a small CONSTANT, independent of course count:
      1) get_all_courses()              -> 1 query (the listing loop)
      2) get_course_registrations(user) -> the registered set (incl. completed)
      3) one CourseProgress .values()   -> 1 query
    No per-course queries. The exact constant is locked by the regression
    test in Task A3 (determined empirically ã do not hard-code a guess here).
    """
    if not user.is_authenticated:
        return [
            CourseListingEntry(course, CourseListingStatus.NOT_REGISTERED, 0)
            for course in get_all_courses()
        ]
    courses = get_all_courses()
    registered_ids = {c.id for c in get_course_registrations(user)}
    progress_rows = {
        row["course_id"]: row
        for row in CourseProgress.objects.filter(
            user=user, course__in=registered_ids
        ).values("course_id", "progress_percentage", "completed_time")
    }

    entries: list[CourseListingEntry] = []
    for course in courses:
        if course.id not in registered_ids:
            entries.append(CourseListingEntry(course, CourseListingStatus.NOT_REGISTERED, 0))
            continue
        row = progress_rows.get(course.id)          # may be missing -> treat as 0%
        pct = row["progress_percentage"] if row else 0
        if row and row["completed_time"] is not None:
            status = CourseListingStatus.COMPLETE
        elif pct > 0:
            status = CourseListingStatus.IN_PROGRESS
        else:
            status = CourseListingStatus.REGISTERED
        entries.append(CourseListingEntry(course, status, pct))
    return entries
```

Notes:
- Uses `get_course_registrations` per spec A3 (it **includes completed** courses, unlike
  `get_current_courses`). This is the deliberate correctness fix: completed courses now
  classify as **Complete** instead of falling through to "not registered".
- A registered course with **no** `CourseProgress` row ã `pct = 0` ã **Registered**.
- `all_courses` is `@login_required`, so `user` is always authenticated here.

> **Security concern (RESOLVED):** The helper's `CourseProgress.objects.filter(user=user,
> course__in=registered_ids)` (and the `get_all_courses()` / `get_course_registrations`
> reads) rely entirely on the `SiteAwareManager` thread-local for multi-tenant isolation ã
> there is no explicit `site=` constraint. In production this is correct (middleware sets
> the thread-local before any view runs) and is **consistent with the rest of the
> codebase** (`get_current_courses` relies on the same mechanism). Adding a one-off
> explicit `site=` filter here as defence-in-depth would *break* that consistency.
> **Decision:** Rely on the `SiteAwareManager` ã no explicit `site=` filter. This matches
> existing code, and cross-site isolation is enforced by the test added in Task A1's TDD
> note (every test branch runs inside `mock_site_context`, with one branch asserting a
> different-site course is excluded).

**TDD (helper unit tests, new file `tests/test_course_listing.py`):** one test per
branch ã not-registered, registered-0%, registered-missing-row (still 0% ã Registered),
in-progress (>0%), complete (`completed_time` set). Assert `entry.status` and
`entry.progress_percentage` real values. Use factory_boy + `mock_site_context`.
`get_course_listing` is a plain function, so the `SiteAwareManager`'s automatic
`site=` filtering only applies when thread-local request context is set ã **every**
test branch (not just some) must run inside `mock_site_context` so site isolation is
actually exercised. Include at least one branch that asserts a course belonging to a
*different* site is excluded from the helper's output, proving cross-site data does not
leak through `get_all_courses()` / `get_course_registrations` / the `CourseProgress`
read.

*Skills:* **`fls:testing`** (pytest/factory conventions, no styling assertions).

### Task A2 ã Rewrite the `all_courses` view

Replace the N+1 loop with a thin view over the helper.

**`views.py` ã new `all_courses`:**

```python
@login_required
def all_courses(request):
    """Flat list of all courses with correct, cheaply-derived statuses."""
    entries = get_course_listing(request.user)
    for entry in entries:
        course = entry.course
        setattr(course, "listing_status", entry.status)       # noqa: B010
        setattr(course, "progress_percentage", entry.progress_percentage)  # noqa: B010
        if entry.status == CourseListingStatus.NOT_REGISTERED:
            # Pre-existing preview behaviour for unregistered rows (out of scope
            # to change). Registered/in-progress/complete rows need no preview
            # context and add no per-course query.
            _annotate_preview_context(course, is_registered=False)
    return render(
        request,
        "student_interface/all_courses.html",
        {"all_courses": [e.course for e in entries]},
    )
```

Key points satisfying the spec:
- **`_annotate_next_up` is no longer called** here (criterion 5). The function stays for
  the dashboard.
- Registered / in-progress rows link straight to the **first item**
  (`view_course_item`, `index=1`) ã no `has_items` check, no `children()`/
  `get_course_index` walk (A5). Every course is guaranteed ãË1 item.
- Complete rows link to `student_interface:course_finish`.
- Not-registered rows keep the existing desktop-modal / mobile-deep-link preview (A5);
  `_annotate_preview_context` remains for those rows only (pre-existing, out of scope).
- Context key stays `all_courses` (a list of status-annotated `Course`s) so existing
  context-based tests keep working; the new `course.listing_status` attribute drives the
  template dispatch.

**Imports to update in `views.py`:** add `get_course_listing`, `CourseListingStatus`;
drop the now-unused `_annotate_next_up`/`get_course_index`/`get_current_courses` imports
**only if** nothing else in the module references them (dashboard still uses
`_annotate_next_up`, `get_current_courses`, `get_completed_courses`,
`get_recommended_courses`, `get_course_registrations`, `get_all_courses` ã keep those).
`IN_PROGRESS`/`READY` imports remain used by `_annotate_next_up`.

**TDD (view behaviour + routing, in `tests/test_course_list_views.py`):** one test per
status asserting the rendered row routes to the correct URL (semantics, not CSS):
- **Not registered** ã preview affordance present (modal/deep-link to
  `student_interface:course_preview`), no progress bar.
- **Registered (0%)** ã links to `view_course_item` index 1; `aria-valuenow="0"` bar.
- **In progress** (force `progress_percentage` > 0 via a `CourseProgressFactory`) ã
  links to `view_course_item` index 1; bar `aria-valuenow` > 0.
- **Complete** (set `completed_time`) ã links to `student_interface:course_finish`; **no**
  progress bar. (This is the bug fixed: completed courses are mis-rendered today.)

Verify the existing tests still pass:
`test_all_courses_started_course_has_progress_percentage`,
`test_all_courses_annotates_accent_slot_key`,
`test_all_courses_title_tag_says_all_courses` (update only if a rename forces it ã the
plan keeps the `all_courses` context key precisely to avoid churn).

*Skills:* **`fls:testing`**.

### Task A3 ã Query-count regression test (criterion 5)

Add `django_assert_num_queries` tests around `all_courses` that lock the **constant**
budget for the status/progress path and prove it does **not** grow with the number of
registered courses.

**Test design (isolating the batched path):** register **all** created courses (so there
are **no** unregistered rows, whose pre-existing preview `children()` walk is out of
scope and would otherwise add per-row queries):

  of registered courses, assert the **same** count.

Determine the exact number empirically during implementation (write the assertion, run,
lock the observed constant). Document in a test comment that the all-registered setup
isolates the batched status/progress derivation; the unregistered preview walk is
pre-existing and out of scope (spec A3 note).
