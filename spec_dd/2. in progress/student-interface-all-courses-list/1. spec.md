# Spec: All-courses list page & course-card simplification

## Overview

Two related, complementary pieces of work on the student interface:

- **Part A — All-courses list page.** Redesign `student_interface:courses` (view
  `all_courses`) as a simpler, denser, scannable list of compact horizontal
  `[icon][details]` rows. Make the four course statuses **correct everywhere** and derive
  them **cheaply** (remove the current N+1), keeping a progress bar where it carries
  information.
- **Part B — Card components & partials.** Make course-card templates clean, clear and
  maintainable so a downstream FLS implementation can restyle the **look** of cards
  without touching their **behaviour**. Introduce the new row templates following clean
  conventions, tokenise card shape, kill duplication, and write down the override ladder.

This is a **simplification + correctness** pass. **No new features.** The student
**dashboard must stay pixel-identical** throughout.

The work is informed by three research docs in this directory:
`research_ux_patterns.md`, `research_component_theming.md`, `research_query_costs.md`.

---

## Why this matters

- **Correctness.** Today `all_courses` derives the registered set from
  `get_current_courses` (`utils.py:449-478`), which **excludes completed courses**. A
  completed course therefore falls through to the "not registered" branch and is rendered
  as a *preview/registration* card — wrong. The redesign makes all four statuses correct
  for every course.
- **Performance.** `all_courses` is **strongly N+1** today
  (`research_query_costs.md`). The cost is dominated by `_annotate_next_up` →
  `get_course_index` (re-checked registration + recursive `course.children()` GFK walks +
  per-item `get_content_status` lookups), ~20–30 queries **per registered course**, plus
  `_annotate_preview_context` → `course.children()` (~`1+k`) per unregistered course.
- **Scannability.** For a homogeneous, mostly-text, mixed-status list, a dense horizontal
  list is the right pattern; tall vertical browse-cards are not (NN/g; `research_ux_patterns.md`).
- **Maintainability / theming.** The current card templates entangle structure, behaviour
  and skin, duplicate the title/eyebrow/link markup across files, and hard-code shape
  utilities — so a downstream that only wants a different *look* must fork whole files
  including logic it does not want (`research_component_theming.md`).

---

## Decisions settled before this spec

These were confirmed with the product owner and drive the spec:

1. **Progress-bar policy (resolves an idea ↔ research conflict).** Show the bar on
   **Registered (0%)** and **In-progress** rows. **Complete** rows show the check badge
   only, **no bar** (a 100% bar is redundant). This matches the idea text and the existing
   dashboard / completed-card behaviour, which is convenient since the dashboard must stay
   identical anyway. (The UX research preferred "in-progress only"; the owner chose
   dashboard consistency.)
2. **Separate row templates.** The all-courses page gets its **own new compact-row leaf
   templates + a row shell**. The dashboard keeps its existing tall-card shell and leaf
   partials, guaranteeing it stays pixel-identical. The new row shell follows the same
   token/slot conventions defined in Part B. (The two layouts are genuinely different;
   forcing one shell to branch on layout would add risk and complexity for no gain.)
3. **Scope discipline.** The end result must be clean, clear and maintainable, with **no
   scope creep**. Refactor where it genuinely improves the code; do not add speculative
   structure or features.

# Part A — All-courses list page

## A1. Layout

- Replace the tall vertical grid cards (accent hero + centred icon) with a compact
  **horizontal row**: `[icon][details]` — icon area on the **left**, details on the
  **right**.
- The details area leads with the **course title** and a **status indicator**.
- A single **flat list** of all courses — **no grouping, no special ordering** (the
  dashboard already owns the focused "continue learning" view).
- The whole row is the click target via the **stretched-link** pattern — the `<a>` lives
  on the **title** and its hit area is expanded with a `::before`/`::after` overlay
  (`inset:0`) over a `position:relative` row. **Do not** wrap the whole row in one `<a>`
  (breaks screen-reader naming and forbids nested interactive elements —
  `research_ux_patterns.md` §3).
- Rows must collapse cleanly to a comfortable full-width touch target on mobile.
- Any secondary interactive control sits **above** the stretched link
  (`position:relative; z-index`) so it stays independently operable. (None is required by
  this spec — see A5 — but the row shell must not preclude it.)

## A2. The four statuses (must be correct for every course)

Every course shows exactly one status:

| Status | Definition |
|---|---|
| **Not registered** | course id is not in the user's registered set |
| **Registered** | registered, `progress_percentage == 0`, not complete |
| **In progress** | registered, `progress_percentage > 0`, `completed_time is None` |
| **Complete** | registered, `completed_time is not None` |

`completed_time` and `progress_percentage` are fields on `CourseProgress`
(`student_progress/models.py:565-572`); `progress_percentage` is a **stored, pre-computed**
column (recalculated on item completion), so reading it is just a column read.

A course can be registered with **no** `CourseProgress` row; treat a missing row as 0% →
**Registered** (`research_query_costs.md` §3).

## A3. Cheap, batched status derivation (remove the N+1)

Statuses **and** progress for the whole page must come from a **constant** set of batched
queries, with **zero per-course queries**:

1. **All courses** — `get_all_courses()` (`Course.objects.all()`) → 1 query.
2. **Registered course-id set** — from `get_course_registrations(user)`
   (`utils.py:148-156`). This must include **completed** courses (unlike
   `get_current_courses`, which excludes them) so completed courses classify correctly.
3. **One `CourseProgress` read** over the registered set selecting the three deciding
   fields:
   ```python
   CourseProgress.objects
       .filter(user=user, course__in=registered_ids)
       .values("course_id", "progress_percentage", "completed_time")
   ```
   → 1 query, mapped by `course_id`.

Then classify each course and read its percentage in **pure Python**, per the table in A2.

- **Remove `_annotate_next_up` from `all_courses`** (and stop computing "Next up" for this
  page). This deletes the `get_course_index` walk that is the page's dominant N+1.
- This derivation logic should live in a **single reusable helper** (e.g. in
  `student_interface/utils.py`) that returns each course's status + percentage, so the view
  stays thin and the behaviour is unit-testable. Exact name/shape settled in the plan.

**Query budget:** the page issues a small **constant** number of queries (~4–5) for the
status/progress of the registered set, independent of how many courses exist — **no
per-course status/progress query**.

> Note (unregistered preview): clicking a Not-registered row still opens the preview
> (A5), and the preview content (`course_preview_content.html`) lists preview children, so
> `_annotate_preview_context` → `course.children()` (~`1+k` per unregistered course)
> remains for rows whose preview is rendered. Removing "Next up" already eliminates the
> dominant N+1; the preview walk is pre-existing behaviour and **out of scope** to change
> here (would alter the preview — a behaviour change).

## A4. Status display & progress bar

- Convey status with **text + icon**, colour as **reinforcement only** — never colour
  alone (WCAG 1.4.1). Follow the existing completed-card pattern (eyebrow label +
  check badge): it already does this well (`course_card_complete.html`).
- The status reads clearly in **greyscale**.
- **Progress bar:** show on **Registered (0%)** and **In-progress** rows (a 0% bar mirrors
  the dashboard). **Do not** show a bar on **Not registered** or **Complete** rows. Reuse
  the existing accessible `<progress>` markup and `.course-progress-{accent}` styling.
- Final status label wording follows the existing UI / brand voice (consult the
  `brand-guidelines` skill at implementation); the four *states* are fixed by A2.

## A5. Click behaviour (status-aware routing)

Routing matches today's registered/unregistered behaviour, with completed routed to the
finish page:

- **Not registered** → open the **preview**: modal on desktop (≥ md), preview page
  (`student_interface:course_preview`) on mobile (< md) — exactly as today.
- **Registered / In progress** → go straight into the course. Because "Next up" is
  removed, the row links to the course entry point — the first item
  (`view_course_item`, `index=1`) when the course has items, else `course_home`. (Reuse the
  existing start-URL helper logic; do not reintroduce the per-course index walk.)
- **Complete** → `student_interface:course_finish`.

## A6. Regression test

- Add a `django_assert_num_queries` test around `all_courses` locking in the constant
  query count (no test guards it today — `research_query_costs.md` §5). The test asserts a
  fixed budget that does **not** grow with the number of registered courses.
- Add functional tests asserting each of the four statuses renders correctly (especially
  the completed-course case that is mis-rendered today) and routes to the correct URL.
  Per house rules, tests assert **behaviour/markup semantics, not CSS classes or styling**.

# Part B — Course-card components & partials

Goal: clean, maintainable card templates where downstream implementations override the
**look** (never the **behaviour**) through a clear, documented ladder. Keep the dashboard
pixel-identical.

## B1. Separate, clean row templates for all-courses (Decision 2)

- Build the new compact-row templates for the all-courses page as **thin, single-purpose
  leaf templates** following the conventions below (named slots, no duplication,
  token-driven shape). The all-courses dispatch selects the leaf by **status**, not by a
  re-branch inside the leaf.
- **One state = one file.** Because click behaviour groups by status, the leaf set is:
  - **not-registered row** — opens preview (keeps the desktop-modal vs mobile-deep-link
    branch internally; that is *behaviour*, not look),
  - **registered row** — progress bar + link into the course (serves both Registered-0%
    and In-progress; the only difference is the bar's value),
  - **complete row** — check badge + link to finish page, no bar.

## B2. Tokenise card shape (component-tier tokens)

- Promote the hard-coded shape constants in the shell (radius `rounded-2xl`, hero height
  `h-28`, padding `p-4`/`p-0` — `cotton/course-card-shell.html`) to **component-tier theme
  tokens**, e.g. `--fls-card-radius`, `--fls-card-hero-height`, `--fls-card-padding`,
  applied via a `.course-card` component class in `tailwind.components.css` (the way
  `.surface` / `.btn` already work). Default them to **today's values** so the dashboard is
  unchanged.
- Both the dashboard shell and the new row shell consume these tokens, so a theme can
  reshape cards by overriding token **values** — matching the existing colour-token story.
- **Keep the `--fls-course-accent-*` palette exactly as-is** (`color-mix` derivation of
  `-gradient`/`-soft`). It is the model the rest of the card follows; do not regress it.

## B3. Kill duplication; add named slots

- The stretched-link **title block**, the **eyebrow** `<p>`, and the repeated link class
  string (`text-inherit no-underline before:absolute before:inset-0 …`) are re-typed across
  `course_card.html` (×3) and `course_card_complete.html`. Hoist the shared title-link and
  eyebrow into the **shared shell as named slots** (e.g. `eyebrow`, `footer`) and/or a small
  partial, so "the card title link" has **one** definition.
- Add a mergeable **`class`** on the shell so an implementation can recompose layout without
  forking behaviour. Resist a `<c-vars>` parameter explosion — prefer **slots + tokens** over
  many boolean/string props (`research_component_theming.md` §3 Pattern D).

## B4. Make leaf templates single-purpose at the seam

- Lift the `is_registered` branch out of `course_card.html` up to the **dispatch seam**
  (`partials/course_list.html`), so each leaf template is thin and single-purpose, where
  doing so makes the code genuinely cleaner. Keep the **modal-vs-deep-link** branch as the
  only internal branch in the not-registered leaf (that is behaviour, not look).
- Apply this to the dashboard templates **only insofar as it improves clarity while keeping
  the dashboard pixel-identical** — verified by QA. Do not rewrite live dashboard markup for
  its own sake (Decision 3: no scope creep).

## B5. Override ladder (look, never behaviour)

Document the three override tiers in `docs/` (e.g. the "incorporate into another project"
how-to), ordered cheapest-first:

1. **Theme tokens** — colour (`--fls-course-accent-*`) and shape
   (`--fls-card-radius/-hero-height/-padding`) via CSS custom properties.
2. **Cotton slots / vars / mergeable `class`** — recompose content and layout.
3. **Whole-file template shadowing** — replace a leaf template as a last resort (already
   supported by Django's loader order + the `themes/*/templates` glob).

---

## Out of scope / guardrails

- **No new features**; no behaviour changes to navigation or registration.
- **Dashboard cards stay pixel-identical** (visual QA must confirm).
- **Do not** reintroduce "Next up: …" on the all-courses rows.
- **Do not** change the preview modal / preview page content or the registration flow.
- **Do not** rely on colour alone for status; **do not** wrap the whole row in a single
  `<a>`.
- Keep the accent palette and `color-mix` derivation unchanged.
- Exact token names, label wording, and markup are finalised in the plan/implementation
  stages.

---

## Success criteria

1. The all-courses page renders a **flat list of compact `[icon][details]` rows**, each
   showing the **correct** one of four statuses (Not registered / Registered / In progress /
   Complete) — including **completed courses**, which are mis-rendered today.
2. Status is conveyed by **text + icon** and survives **greyscale**; colour is reinforcement
   only (WCAG 1.4.1).
3. A **progress bar** shows on Registered (0%) and In-progress rows; **not** on Not-registered
   or Complete rows.
4. The whole row is clickable via the **stretched-link** pattern (title `<a>` + `::before/after`
   overlay), and routes correctly: preview (not registered) / into the course (registered &
   in progress) / finish page (complete).
5. The page issues a **constant** number of queries for status + progress regardless of how
   many courses/registrations exist; a `django_assert_num_queries` regression test guards it.
   `_annotate_next_up` is no longer called by `all_courses`.
6. Card shape (radius, hero height, padding) is driven by **component-tier tokens**; the
   `--fls-course-accent-*` palette is unchanged.
7. The shared title-link/eyebrow markup exists in **one** place; new row leaf templates are
   thin and single-purpose; a mergeable `class` and named slots exist on the shell.
8. The **override ladder** (token → slot/class → template shadow) is documented.
9. The **dashboard is pixel-identical** to before.
10. Code is clean, clear and maintainable, with no scope creep.
