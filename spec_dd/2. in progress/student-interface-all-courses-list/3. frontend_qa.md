# Frontend QA: All-courses list page & card simplification

Manual browser QA for the redesigned `all_courses` page and the (must-stay-identical)
dashboard. Covers the four statuses, click routing, progress-bar policy, accessibility
(text+icon / greyscale), and dashboard pixel-identity.

---

## 0. Start the dev server

```bash
PORT=$(.claude/fls/scripts/find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL: `http://127.0.0.1:$PORT`

If the styles look unstyled, build Tailwind first: `npm run tailwind_build`.

## 2. All-courses page — the four statuses

Go to `http://127.0.0.1:$PORT/courses/`.

**Expect:**

- A single **flat list** of compact horizontal **`[icon][details]` rows** — icon on the
  left, title + status on the right. **No** tall accent-hero browse cards, **no** grouping
  or section headings, **no** "Next up: …" text anywhere.
- Every course appears exactly once with **one** clear status.

Check each row:

| Row | Status indicator | Progress bar? | Notes |
|---|---|---|---|
| **NR** (not registered) | "Not registered"-style label + icon | **No bar** | — |
| **R0** (registered 0%) | "Registered"/"Not started" label + icon | **Yes, 0%** | bar reads empty |
| **IP** (in progress) | "In progress" label + icon | **Yes, > 0%** | bar partially filled |
| **C** (complete) | "Completed" label + **check badge** | **No bar** | check icon present |

Confirm the **completed** course (**C**) shows the **Completed** row — **not** a
preview/registration card. (This is the bug being fixed.)

---

## 3. Click routing (whole-row clickable)

For each row, the **whole row** should be clickable (stretched-link). Verify the target:

1. **NR** → opens the **preview**:
   - **Desktop (window ≥ md / ~≥768px):** clicking opens the **preview modal** (course
     title, contents list, Start button, Close). Close dismisses it.
   - **Mobile (window < md):** resize the browser narrow, reload, click the row → it
     navigates to `…/courses/<slug>/preview/` (a full page, not a modal).
2. **R0** → navigates straight into the course **first item**:
   `…/courses/<slug>/1/`.
3. **IP** → navigates into the course **first item**: `…/courses/<slug>/1/`.
4. **C** → navigates to the **finish** page: `…/courses/<slug>/finish/`.

Keyboard check: `Tab` to a row's title link and press `Enter` — it activates the same
target (focus ring visible on the row).

**Stretched-link structure (anti-pattern check, criterion 4).** A whole-row click must be
implemented as a **stretched link**, not by wrapping the row in one giant `<a>`. Open
devtools → Inspect a registered/complete row and confirm:
- There is **one** `<a>` per row, on the **title** (its accessible name is just the course
  title, e.g. "Intro to Python" — not the whole row's text).
- The row container is `position: relative` and the title `<a>` has a `::before`/`::after`
  overlay with `inset: 0` (the stretched hit area).
- The row is **not** a single `<a>` wrapping everything, and there are no interactive
  elements nested inside the title link.

## 4. Accessibility — status survives greyscale (WCAG 1.4.1)

- Each status must be distinguishable by **text + icon**, not colour alone.
- Simulate greyscale (browser devtools → Rendering → "Emulate vision deficiencies →
  Achromatopsia", or a greyscale OS filter) and reload `/courses/`.
- **Expect:** you can still tell Not-registered / Registered / In-progress / Complete
  apart by **label text and icon shape** (e.g. the check badge on Complete) with no colour.

---

## 5. Mobile layout

- Narrow the window to a phone width (~375px) and reload `/courses/`.
- **Expect:** rows go full-width, `[icon]`+details stack/flow cleanly, each row is a
  comfortable touch target (no cramped/overlapping text, no horizontal scroll).

## 6. Dashboard must be pixel-identical (regression)

Go to `http://127.0.0.1:$PORT/` (dashboard).

**Expect — unchanged from before this work:**

- "In Progress", "Recommended Courses", "Available courses" (+ "Browse all courses"
  link), and "Learning History" sections render as tall **accent-hero grid cards** (icon
  band on top), exactly as before.
- In-progress cards still show the progress bar + **"Next up: …"** line (the dashboard
  keeps "Next up"; only the all-courses page drops it).
- Not-started, completed, and recommended/available cards look identical to before.
- Registering / opening a preview / continuing a course all behave as before.

If anything on the dashboard looks different, that is a **regression** — the card-shape
tokens must default to today's values and the leaf-template split must emit identical
markup.

## 7. Theming smoke check (optional, override ladder)

Switch the active theme to `first_class` and rebuild, to confirm cards/rows still resolve
through tokens (colour + shape) with no broken markup:

`FLS_THEME` must be set for **both** the build *and* the runserver (the build bakes the
theme tokens into the CSS; the runserver resolves the theme at runtime):

```bash
FLS_THEME=first_class npm run tailwind_build
FLS_THEME=first_class uv run python manage.py runserver $PORT
```

Visit `/courses/` and `/` — rows and cards should restyle via tokens (accent colours /
radius) with no layout breakage. Rebuild with the default theme afterwards
(`npm run tailwind_build`).

      each (incl. the completed course rendering as **Completed**, not a preview card).

      Not-registered and Complete.

      R0 & IP → first course item, C → finish page.
- [ ] Status is readable in greyscale (text + icon, not colour alone).
- [ ] Rows collapse cleanly to a comfortable full-width touch target on mobile.
- [ ] Dashboard is visually identical to before (cards, "Next up", all sections/links).
