# Spec: FLS test-suite portability — Part 2 (positive signal)

> **Part 2 of a two-part effort.** Part 1 ("quiet the noise", already merged —
> `spec_dd/3. done/2026-07-09_09:37_fls-test-portability-part1/`) shipped the
> `fls_internal` marker taxonomy, `INSTALLED_APPS`-keyed collection-safety
> guards, and de-branded assertions. **This spec assumes all of that is already
> in place.** Part 2 adds the *positive* "am I wired correctly?" signal: an
> importable conformance suite (Layer 3), boot-time system checks (Layer 4), the
> upgrade-notes tie-in (Layer 5), and the plugin-doc touches that make them
> stick (Layer 6, Part-2 portion).

## Problem

FLS ships as a package installed into concrete downstream projects (e.g. First
Class). "Did I wire FLS up correctly?" is the signal a downstream actually needs
from a test run, and today it **isn't a supported, dedicated thing at all**. The
canonical failure is the First Class integration of the `home_page` update:

- `student_interface` newly required `settings.COURSE_ACCESS_BACKEND`, which
  First Class had never set. FLS's loader (`get_course_access_backend()`,
  `freedom_ls/course_access/loader.py:23-39`) is `@functools.cache`d and does
  `import_string(settings.COURSE_ACCESS_BACKEND)` with **no fallback** — an
  unset key raises only on the *first real request*, not at `manage.py check`
  time.
- First Class's root `urls.py` was missing the `applications/` include.

Both would 500 the catalogue/dashboard at runtime while `manage.py check` passed
clean. They were caught only as a side effect of FLS's own tests being vacuumed
up by the downstream's `pytest` — the exact coupling Part 1 removed. Removing the
noise leaves this real signal absent. Part 2 supplies it.

This is not hypothetical drift: `fls-claude-plugin/resources/template_repo_manifest.md`'s
own `urls.py` checklist (lines ~210-219) still omits `applications/`,
`interest/`, and the sitemap/robots wiring that `config/urls.py:51-73` actually
ships. A downstream following that checklist reproduces the opening bug verbatim.

## Goal

Give a concrete project a positive, executable **"am I wired correctly?"**
signal that catches the `COURSE_ACCESS_BACKEND` / missing-include class of bug
*before* runtime, at two altitudes:

- **Layer 3 — conformance suite:** an importable pytest module the downstream
  runs, exercising the real integration seams against *its own* settings.
- **Layer 4 — system checks:** `manage.py check` fails on the static config
  gaps, independent of any test run.

Plus the Layer 5 upgrade-notes tie-in and Layer 6 doc touches that keep the
convention alive.

### Success criteria

1. `pytest --collect-only -q freedom_ls/contrib/conformance` lists at minimum
   the node IDs enumerated in "Acceptance node IDs" below.
2. The conformance suite is **green under FLS's own `config.settings_dev`** (FLS
   is its own first consumer — `testpaths` already includes `freedom_ls`).
3. Deleting `COURSE_ACCESS_BACKEND` from settings makes `manage.py check` exit
   non-zero with `freedom_ls_student_interface.E001`.
4. With `course_applications` installed, removing its `applications/` include
   makes `test_fls_namespace_reverses[course_applications:apply]` fail
   specifically (one named failure, not a collection abort).
5. A downstream missing sitemap/robots wiring fails
   `test_reference_url_reverses[sitemap]` / `[robots_txt]`.
6. The suite passes under the existing `--disable-socket
   --allow-hosts=127.0.0.1,::1` config with no DB required.
7. **Override-friendly:** a downstream that removes an FLS app from
   `INSTALLED_APPS` (e.g. replaces `student_interface`) gets that app's probes
   **skipped**, not failed, and Layer-4 checks for that app do not fire.

## Layer 3 — the `freedom_ls.contrib.conformance` suite

A new, **opt-in importable module** at `freedom_ls/contrib/conformance/`. A
downstream references it from its own `tests/` dir. It is deliberately **not** a
`pytest11` entry-point plugin: an entry point on `sys.path` would auto-activate
in *every* downstream unconditionally, re-creating the coupling Part 1 removed.
An importable module makes opt-in explicit and greppable — matching how DRF
(`rest_framework.test`), django-cms (`CMSTestCase`), and Wagtail
(`wagtail.test.utils`) ship reusable test helpers.

### Override-friendliness (primary design constraint)

FLS is meant to be **extended and overridden**. A concrete project may replace
or discard whole FLS apps — e.g. throw away `student_interface` entirely and
ship its own. The conformance suite must **verify only the seams a downstream
actually opted into, and never enforce rigidity on flexible surfaces.**

Mechanism: **every probe is gated on the FLS app/feature it checks actually
being installed**, reusing Part 1's `INSTALLED_APPS`-keyed collection-safety
pattern. When the relevant app is absent from `settings.INSTALLED_APPS`, its
probes **skip** (not fail). A downstream that replaced `student_interface`
simply doesn't run the `student_interface:*` probes — it gets a clean skip, not
a wall of red. The suite is therefore a signal for *"the FLS pieces I chose to
use are wired correctly"*, never *"you must use all of FLS."*

Consequences baked into the design:

- Data tables are organised **per app** and importable in slices
  (`from freedom_ls.contrib.conformance.urls import *`), so a downstream can also
  opt into a subset explicitly rather than gate-and-skip.
- Required-setting probes are likewise gated: `COURSE_ACCESS_BACKEND` is only
  required when an app that consumes it (`student_interface` /
  `course_applications`) is installed; `AUTH_USER_MODEL` is checked only when
  `accounts` is installed.
- The **only** deliberately-enforced baseline is sitemap/robots existence
  (Decision D1, a product requirement) — and even that is silenceable by a
  downstream that genuinely serves no public surface. Everything app-specific is
  skip-when-absent.
- Layer 4's checks are already app-conditional (E001/E002 fire only when
  `student_interface` is installed — see Layer 4), so the same
  override-friendliness holds at boot time.

### Opt-in surface

The package `__init__.py` re-exports every conformance test function under an
explicit `__all__`. Downstreams opt in with either form; the spec documents
both and recommends the second for collision safety:

```python
# tests/test_fls_conformance.py  (downstream) — simple form
from freedom_ls.contrib.conformance import *          # noqa: F401,F403

# ...or the collision-safe form (recommended in downstream docs)
from freedom_ls.contrib import conformance
test_fls_namespace_reverses = conformance.test_fls_namespace_reverses
# (etc.)
```

`import *` works because pytest collects any module-level callable whose *name*
matches `python_functions` (default `test_*`), regardless of where it was
defined; parametrize marks travel with the function object. The wildcard form is
zero-effort but silently shadows any same-named test the downstream already has,
and can leak the module's own imports into the downstream namespace — hence the
hard hygiene rules below.

### Hygiene rules (load-bearing — enforced in the module itself)

- **Every conformance submodule defines `__all__`** listing only its intended
  test functions, so `import *` is inert w.r.t. the module's own imports
  (`reverse`, `settings`, `apps`, …).
- **No non-test helper in the package is named `test_*`.** Private helpers use a
  `_`-prefix (`_resolve_backend_app()`); if a public non-test callable is ever
  needed, set `helper.__test__ = False`.

### What it verifies

Reads the downstream's **real** config (zero `override_settings` — it must
reflect actual runtime state) and calls FLS's **own production resolution code**
so it can't drift from runtime behaviour. All probes are driven from small,
explicitly-maintained data tables parametrized with
`pytest.mark.parametrize(..., ids=...)` so each seam surfaces as one
specifically-named failure, and "add a probe" is a documented step when a new
FLS app/setting lands.

**Data tables (module-level, the maintained surface):**

- `REQUIRED_SETTINGS` — `("COURSE_ACCESS_BACKEND", "AUTH_USER_MODEL", …)` plus,
  where applicable, the expected value (`AUTH_USER_MODEL ==
  "freedom_ls_accounts.User"`).
- `FLS_NAMESPACE_PROBES` — one entry per **all FLS app namespaces**, each a
  concrete `viewname` + any minimal kwargs needed to reverse:
  - `student_interface:dashboard`, `student_interface:courses`,
    `student_interface:course_detail`, `student_interface:course_home`, … (the
    student journey);
  - `course_applications:apply`, `course_applications:status`;
  - `course_interest:express_interest`, `course_interest:remove_interest`;
  - `educator_interface:interface`;
  - `accounts:*` (the shipped account routes).
- `REFERENCE_URL_NAMES` — the **required** downstream-replicated wiring:
  `"sitemap"`, `"robots_txt"` (unnamespaced project-root names — see Decision D1).

**Probes** (each gated on its app being installed — skip when absent, per
"Override-friendliness"):

- `test_required_setting_present[<name>]` — each required setting present,
  importable, and (where pinned) equal to its expected value. Gated on the app
  that consumes the setting.
- `test_fls_namespace_reverses[<viewname>]` — each FLS namespace reverses,
  **only for apps the downstream has installed**. Would have caught the missing
  `applications/` include; skips cleanly for an app the downstream replaced.
- `test_reference_url_reverses[<name>]` — `sitemap` and `robots_txt` reverse.
  These are **required** (see D1): a downstream must replicate FLS's reference
  sitemap/robots wiring; a `NoReverseMatch` here is a hard conformance failure.
- `test_configured_backend_instantiates` — calls FLS's own
  `get_course_access_backend()` and asserts it returns a backend (exercises
  `import_string` + instantiation end-to-end).
- `test_active_theme_resolves` — asserts `settings.RESOLVED_THEME_DIR` is an
  existing directory, or re-runs the **side-effect-free** `resolve_theme_dir(
  settings.FLS_THEME, settings.FLS_THEMES_DIRS)`. **Never call `configure_theme`**
  (it mutates `TEMPLATES`/`STATICFILES_DIRS` in place — see D4).
- `test_active_icon_set_resolves` — `settings.FREEDOM_LS_ICON_SET in ICON_SETS`
  and/or a real `render_icon()` of a semantic name (exercises
  `get_icon_backend()` + `build_svg()`).
- `test_migration_state_consistent` — `call_command("makemigrations",
  check=True, dry_run=True)`. This diffs models against migration files and
  needs **no DB connection**, so it stays green under `--disable-socket` with no
  `django_db` marker (see D5).

Tiny, fast, deterministic: stdlib + django + pytest only. No factories, demo
content, browser, or network.

### Packaging & placement

- `freedom_ls/contrib/conformance/` is auto-included by the existing
  `[tool.setuptools.packages.find] include = ["freedom_ls*"]` glob — **no
  `pyproject.toml` packaging change** needed to ship it.
- Because `testpaths` includes `freedom_ls`, FLS's own `uv run pytest` collects
  and runs these probes under `config.settings_dev`. This is desired (FLS
  asserts it conforms to its own suite) and is success criterion #2.
- **Ruff:** the existing `[tool.ruff.lint.per-file-ignores]` only relaxes paths
  containing a literal `tests/` segment, so these shipped-but-pytest-style files
  would otherwise trip `S101` (assert) etc. Add a per-file-ignore entry for
  `freedom_ls/contrib/conformance/**` (at minimum `S101`). They are pytest test
  functions even though they ship as production code.

## Layer 4 — Django system checks

Add `django.core.checks` so `manage.py check` **fails** on the static config
gaps, independent of any test run. This extends existing house style — FLS
already ships checks in `course_access/`, `icons/`, `accounts/`, `base/`; the
gap is that `student_interface/apps.py` is a bare `AppConfig` with no `ready()`.

New `freedom_ls/student_interface/checks.py`, registered from a new
`StudentInterfaceConfig.ready()` (`from . import checks  # noqa: F401`):

All three are **conditional on `student_interface` being installed** — a
downstream that replaces or removes the app gets none of them (override-friendly
by construction; a check whose owning app is absent has nothing to enforce):

- **`freedom_ls_student_interface.E001`** — `student_interface` installed but
  `COURSE_ACCESS_BACKEND` unset → **Error**.
- **`freedom_ls_student_interface.E002`** — configured `COURSE_ACCESS_BACKEND`'s
  containing app not in `INSTALLED_APPS` → **Error** (resolve via
  `apps.get_containing_app_config()` / `apps.is_installed()`, mirroring Django's
  own `admin.check_dependencies`).
- **`freedom_ls_student_interface.W001`** — sitemaps wired but
  `django.contrib.sitemaps` absent → **Warning**.

### Conventions (from the existing check modules — follow, with one correction)

- **App-label-namespaced IDs** (`freedom_ls_student_interface.E001`), matching
  `course_access`/`base`/`accounts`. **Do not copy `icons/checks.py`'s flat
  `freedom_ls.E00N` scheme** — it is the one inconsistent precedent (see D3).
- Register at import time via `@register()`; `ready()` only imports the module.
- **No `Tags.database`, no `deploy=True`** — these are internal-consistency
  checks that must run on every `check`/`runserver`/`migrate`.
- **No DB access** — read `settings` + `apps` only (closest analog:
  `base/checks.py`).
- **Respect the `app_configs` contract:** when `app_configs is not None`,
  early-return `[]` unless `student_interface`'s label is in it, so scoped
  `manage.py check <app>` runs stay correct.
- **Checks must never raise** — Django does not catch check exceptions; a raised
  exception breaks `check`/`runserver`/`migrate` with a raw traceback. Any
  dotted-path resolution (E002) must catch `ImportError`/`ModuleNotFoundError`/
  `AttributeError` and return an `Error`, never propagate.
- **Unique, stable IDs** so `SILENCED_SYSTEM_CHECKS` can target them precisely.

### Layer 3 vs Layer 4 division (both needed — the `home_page` bugs split along it)

Static config-shape questions ("is the setting set", "is its app installed")
belong in **checks** — they must fail even if the downstream never wrote a test.
Behavioural questions ("does the backend instantiate", "does this namespace
reverse") belong in the **conformance suite** — verifying them means *executing*
code. Checks would not have caught the missing URL include; the suite would not
have failed boot for the unset setting. (`icons` already demonstrates the
"same question, both layers" pattern: its `check_iconify_json_exists` is the
Layer-4 echo of the Layer-3 `test_active_icon_set_resolves` probe.)

## Layer 5 — tie into `upgrade_notes.md`

`concrete-implementation-helpers` shipped structured per-spec `upgrade_notes.md`.
The authoritative schema is in `commands/sdd/update_upgrade_notes.md` (richer
than that spec's prose — it includes `changed_settings: []`, npm fields, etc.).

**Convention Part 2 establishes:** when a spec introduces a **hard config
requirement** (as `home_page` did with `COURSE_ACCESS_BACKEND`), its notes set
`requires_settings_change: true` with the specific keys in `changed_settings`, so
`update_fls` surfaces it — and ideally a Layer-4 check enforces the same
requirement at boot.

**Decision to record (D6):** `requires_settings_change: true` +
`changed_settings` is sufficient for the hard-requirement case (the field is
free-text and already carries the key list — Part 1's own notes prove it). No
new flag is required. `commands/sdd/update_upgrade_notes.md` gains guidance to
recognise a *hard/required* settings change (vs optional/informational) and,
when the spec also adds a Layer-4 check enforcing it, to say so in the notes.

Part 2's **own** `upgrade_notes.md` (authored later in the SDD flow): Part 2
requires no downstream settings change, but downstreams should be told the new
`student_interface` checks exist and that the conformance suite is now available.

## Layer 6 — make the new conventions stick (Part-2 portion)

Part 1 covered the bulk of Layer 6 (testing/playwright skills, code-reviewer,
`update_fls.md` marker selection). Part 2 adds:

- **`commands/concrete/update_fls.md`:** it currently runs neither `manage.py
  check` nor any conformance suite. Add both to the verification steps — invoke
  the Layer-3 conformance suite as the positive signal, and run `manage.py
  check` so Layer-4 failures surface during an upgrade. (Part 1 already switched
  the bare `uv run pytest` call sites to the documented `-m "not playwright and
  not fls_internal and not ci_only"` selection.)
- **`commands/sdd/update_upgrade_notes.md`:** add the D6 guidance above.
- **`fls-claude-plugin/resources/template_repo_manifest.md`:** its `urls.py`
  checklist is out of sync with `config/urls.py` (omits `applications/`,
  `interest/`, sitemap/robots). The actual manifest edit is
  `/update_template_repo`'s job (SDD step 12), but this spec **requires** that
  step to (a) fix the checklist and (b) ship the reference sitemap/robots wiring
  + a `tests/` dir importing the conformance suite, so new projects get the
  forced sitemap/robots existence (D1) and the positive signal by default.

## Key decisions

- **D1 — sitemap/robots are required downstream wiring, probed as a distinct
  category.** `sitemap`/`robots_txt` are *not* FLS-shipped namespaces; they live
  in the project-root `config/` (excluded from the package), which only
  *demonstrates* the wiring. Per product decision, a concrete project **must**
  have sitemaps and robots. So: the conformance suite includes **required**
  `test_reference_url_reverses[sitemap|robots_txt]` probes (hard failure if
  unwired), and the template-repo step ships the reference wiring by default.
  These probe unnamespaced names and are documented as "did you replicate FLS's
  reference wiring", distinct from the FLS-namespace probes.
- **D2 — probe all FLS app namespaces**, not just the idea's named subset:
  broadest wiring signal and matches the "add a probe when you add an app"
  discipline.
- **D3 — app-label-namespaced check IDs**, not the flat `freedom_ls.E00N` scheme
  `icons/checks.py` uses (the lone inconsistent precedent).
- **D4 — theme probe calls `resolve_theme_dir` (idempotent), never
  `configure_theme`** (which mutates settings in place and would double-insert
  theme dirs).
- **D5 — migration probe uses `makemigrations --check --dry-run`** (no DB
  connection, socket-safe) rather than any form needing a live connection.
- **D6 — Layer 5 needs no new upgrade-notes flag** (see Layer 5).
- **D7 — override-friendliness over completeness.** FLS is built to be extended
  and overridden; a downstream may discard whole apps. Probes and checks are
  gated on their app being installed and **skip/no-op when absent** — the suite
  never mandates using all of FLS. This constraint outranks D2's breadth: "probe
  all FLS app namespaces" means all *installed* ones. The sole enforced baseline
  is D1 (sitemap/robots), itself silenceable.

## Scope / non-goals

**In scope:** the `freedom_ls.contrib.conformance` module (Layer 3), the new
`student_interface` system checks (Layer 4), the upgrade-notes tie-in (Layer 5),
and the Part-2 plugin-doc touches (Layer 6).

**Depends on Part 1 (assumed present):** the `fls_internal` marker taxonomy,
`INSTALLED_APPS`-keyed collection-safety guards, and de-branded assertions. Note:
Part 1 shipped **no `e2e` marker** — `playwright` is the browser-test marker;
Part 2 must not assume `e2e`.

**Out of scope / deferred to `/update_template_repo` (SDD step 12):** the
concrete project's own pytest config (`testpaths`, `--ignore=submodules`, its
`tests/` dir) and the actual template-manifest edits. Called out in Layer 6 so
the two sides land together — Layer 3 is what makes a `tests/` dir *worth*
shipping downstream, and D1's "forced existence" depends on the template
shipping the wiring.

## Testing strategy

- **Conformance module:** FLS's own suite runs it under `config.settings_dev`
  (must be green). Add targeted negative tests (style: existing
  `course_access/tests/test_checks.py`) proving each probe *fails* when its seam
  is broken — e.g. an `@override_settings`-driven test asserting
  `test_required_setting_present` raises when `COURSE_ACCESS_BACKEND` is unset,
  and that a bad URL table entry raises `NoReverseMatch`. These meta-tests are
  themselves `fls_internal`-safe (they use FLS settings) and prove the probes
  have teeth without a downstream.
- **System checks:** mirror `course_access/tests/test_checks.py` — call each
  check directly with `app_configs=None`, drive config via `@override_settings`,
  assert on returned message IDs/levels. Cover the `app_configs` early-return and
  the E002 exception-safety path (unimportable backend path → returned `Error`,
  not a raised exception).
- **Override-friendliness (criterion #7):** a meta-test proving that with an FLS
  app removed from `INSTALLED_APPS`, its probes report `skipped` (not `failed`)
  and its Layer-4 checks return `[]` — the executable proof the suite doesn't
  enforce rigidity on overridable surfaces.
