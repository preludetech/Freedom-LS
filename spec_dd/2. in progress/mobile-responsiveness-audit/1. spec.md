# Mobile Responsiveness Audit & Bug Fixes - Specification

## Overview

Audit and fix all mobile responsiveness issues across the user-facing application. The result is an app that works well on mobile devices (360px+) across all student, educator, and authentication interfaces.

### Scope

**In scope:**
- Student interface (course browsing, content viewing, quizzes, progress tracking)
- Educator interface (cohort management, progress grids, student lists)
- Auth pages (login, signup, password change/reset, profile)
- Shared components (header, navigation, modals, buttons, data tables, cotton components)

**Out of scope:**
- Django admin interface
- Non-mobile bugs (tracked separately in `spec_dd/1. next/non-mobile-bugs/`)

### Why This Matters

Mobile users currently encounter overlapping sidebars, unusable progress grids, cramped tables, and undersized touch targets. These issues make the educator and student interfaces partially or fully unusable on phones and small tablets. Fixing these issues broadens accessibility and makes the platform viable for users who primarily use mobile devices.

---

## Issues

### 1. Educator sidebar overlaps content on mobile

**Problem:** On mobile viewports (375px), the educator sidebar is expanded by default (`sidebarOpen: true`). It uses `w-64` (256px) which pushes main content off-screen. There is no mobile detection — unlike the student course sidebar which uses `matchMedia`.

**Fix:** Build on the matchMedia/localStorage pattern used in the student course sidebar (`_course_base.html`), and improve it:
- Use `window.matchMedia('(min-width: 1024px)').matches` to determine default state
- Default to collapsed on mobile (< 1024px), expanded on desktop
- Store state in localStorage so it persists across navigations
- On mobile when expanded, use an overlay pattern (sidebar over content with a backdrop) rather than pushing content — this is a new improvement over the current student sidebar, which pushes content off-screen on mobile

**Why this approach:** The student sidebar already uses matchMedia and localStorage successfully. Reusing and improving that pattern ensures consistency and reduces maintenance burden. The overlay pattern prevents content from being pushed off-screen on small viewports.

**Success criteria:**
- Sidebar is collapsed by default on viewports < 1024px
- Sidebar is expanded by default on viewports >= 1024px
- Toggle button is always visible and accessible
- When sidebar is open on mobile, main content remains accessible (close by tapping backdrop)
- The student and Educator interface both make use of the same basic component, this can be a cotton component

### 2. Sidebar toggle state persists via localStorage

**Problem:** Collapsing the educator sidebar resets on full page navigation. The current implementation has no persistence.

**Fix:** Use localStorage to persist the sidebar state, reading it on page load with a fallback to the `matchMedia` default. This is the same mechanism the student course TOC already uses.

**Why localStorage:** It works client-side without server round-trips, is already proven in the student sidebar, and avoids adding cookies to every request.

**Success criteria:**
- Sidebar state persists across full page navigations
- On first visit (no localStorage value), defaults based on viewport width
- State is scoped to the educator interface (separate key from student TOC)
- The student and Educator interface both make use of the same basic component, this can be a cotton component

### 3. Progress grid unusable on mobile

**Problem:** The course progress panel table has many columns with `min-w-[100px] max-w-[140px]` and a sticky first column with `min-w-[180px]`. On mobile, only the student name column is visible with no scroll hint.

**Fix:** On mobile (below `md:` breakpoint),unfreeze the first column so the whole thing can scroll. When the user scrolls and the first column is out of sight, display the value in the first column slightly above the related row so that the user can see what row they are looking at.

**Decision rationale:** This retains the full table functionality that we want, and allows the user to scroll without losing track of where they are.

**Success criteria:**
- Below `md:` breakpoint, first column (student name) is not frozen
- Entire table scrolls horizontally
- When scrolled past the first column, the student name for each visible row appears above that row so the user knows which student they are looking at
- Grid remains fully functional (no data hidden or removed)

### 4. Course dropdown text truncated on mobile

**Problem:** The course selection dropdown in the progress panel shows truncated course names on mobile. The dropdown currently uses `w-full max-w-lg`, which should give full width on mobile — the truncation may be caused by the container being constrained by the sidebar or other layout elements.

**Fix:** Verify that the dropdown's parent container allows full width on mobile. If the truncation is caused by the sidebar layout, this may be resolved by issue #1 (sidebar overlay). If not, ensure the dropdown container itself has sufficient width on mobile.

**Success criteria:**
- Course names are fully visible in the dropdown on mobile
- Dropdown does not overflow the viewport

### 5. Educator data tables cramped on mobile

**Problem:** The `c-data-table` cotton component (used for cohort lists, student lists, etc.) has columns that clip on mobile. The horizontal scroll works (via `overflow-x-auto`) but the experience is poor — the user loses track of which row they're looking at when scrolling.

**Fix:** Apply the same pattern as the progress grid to the `c-data-table` component itself. Below `md:` breakpoint, unfreeze the first column but make the first column value visible as we scroll. Since this is a shared cotton component, the fix applies to all data tables automatically.

**Success criteria:**
- Below `md:` breakpoint, first column (identifier) is not frozen
- Entire table scrolls horizontally
- When scrolled past the first column, the identifier for each visible row appears above that row
- Table remains fully functional (no data hidden or removed)

### 6. YouTube embed has hardcoded 500px height

**Problem:** `content_engine/cotton/youtube.html` uses `style="height: 500px;"` which is unresponsive and causes excessive scrolling on mobile.

**Fix:** Replace the fixed height with Tailwind's `aspect-video` class (`aspect-ratio: 16/9`) and `w-full`. Remove the inline style.

**Why `aspect-video`:** It maintains the correct aspect ratio at any width, is a single Tailwind class, and is the standard approach for responsive video embeds.

**Success criteria:**
- YouTube embeds maintain 16:9 aspect ratio at all viewport sizes
- No inline style for height
- Video is fully visible without horizontal scroll on 360px viewport

### 7. Dropdown menu can render off-screen

**Problem:** `cotton/dropdown-menu.html` uses a fixed `w-40` width and positions with `rect.right - 160` which can place the menu off-screen on mobile. No viewport boundary detection exists.

**Fix:** Add viewport boundary detection in the Alpine.js positioning logic. If the menu would extend beyond the right edge of the viewport, anchor it to the right edge instead. If it would extend below the viewport, open upward.

**Success criteria:**
- Dropdown menu is fully visible on 360px viewport regardless of trigger position
- Menu repositions automatically when near viewport edges
- Existing desktop behavior is not affected

### 8. Form long-text input has `ml-4` indentation

**Problem:** The `ml-4` margin on long-text inputs reduces available width on mobile, causing layout issues.

**Fix:** Make the margin responsive: remove `ml-4` on mobile, keep it on `sm:+` (`sm:ml-4`).

**Success criteria:**
- Long-text inputs use full available width on mobile
- Indentation is preserved on tablet and desktop

### 9. Form page navigation buttons awkwardly spaced

**Problem:** `justify-between` on form navigation buttons causes wide gaps on mobile, pushing buttons to far edges of the screen.

**Fix:** Stack buttons vertically on mobile (`flex-col`) and use horizontal layout on larger screens (`sm:flex-row sm:justify-between`). Primary action (next/submit) should appear first in visual order on mobile.

**Success criteria:**
- Navigation buttons stack vertically on mobile with appropriate spacing
- Buttons are full-width on mobile for easy touch targeting
- Horizontal layout with `justify-between` is preserved on `sm:+`

### 10. Pagination touch targets too small

**Problem:** Pagination links in `c-form-page-link` and the progress panel (`course_progress_panel.html`) use `px-3 py-1` which results in touch targets below the 44px minimum recommended by Apple HIG and WCAG 2.2.

**Fix:** Increase padding to `px-4 py-3` on mobile to meet the 44px minimum height. Can use responsive padding: `px-4 py-3 sm:px-3 sm:py-1` if the compact desktop style is preferred.

**Success criteria:**
- All pagination links have a minimum touch target of 44x44px on mobile
- Minimum 8px spacing between adjacent pagination links
- Desktop styling may remain compact if desired

### 11. Instance details panel has no responsive styling

**Problem:** `partials/instance_details_panel.html` uses a plain `<table>` with no overflow handling or mobile layout.

**Fix:** Wrap in `overflow-x-auto` as a minimum. For better mobile UX, convert to a stacked key-value layout on mobile (label above value) instead of a two-column table.

**Success criteria:**
- No horizontal overflow on 360px viewport
- Data is readable without horizontal scrolling on mobile
- Desktop table layout preserved on `md:+`

### 12. Entrance/auth layout missing mobile side padding

**Problem:** `allauth/layouts/entrance.html` uses `max-w-2xl mx-auto` but has no explicit side padding for very small screens, causing content to touch screen edges.

**Fix:** Add `px-4` padding to the container for mobile viewports.

**Success criteria:**
- Auth pages have visible side padding on 360px viewport
- Content does not touch screen edges
- Centered layout preserved on larger screens

## Verification Requirement

Every issue must be confirmed to still exist before any fix is attempted:

- **Frontend/UI issues** (all 12 issues above): Use Playwright MCP to reproduce the issue at the relevant viewport size (360px or 375px width). If test data is needed, use the `qa-data-helper` agent to set up appropriate data in the development database.

This ensures no effort is wasted fixing issues that may have been resolved by other changes.

## Mobile Table/Grid Strategy

A consistent pattern should be used across all data tables and grids on mobile:

1. Below `md:` breakpoint, first column is no longer frozen
2. The first column value (identifier — student name, cohort name) remains visible as we scroll, it appears above the row values


This pattern applies to: progress grid (#3), educator data tables (#5).

**Why this pattern over cards:**
We need to maintain functionality, a grid is superior for search, filtering and quick comparisons.

## Success Criteria (Overall)

1. All 12 issues verified and fixed
2. No horizontal scroll on any page at 360px viewport width
3. All interactive elements have minimum 44x44px touch targets
4. Sidebar defaults to collapsed on mobile, persists state via localStorage
5. All data tables use the mobile table/grid strategy (unfrozen first column with floating row identifier)
6. YouTube embeds are responsive
7. Dropdown menus stay within viewport bounds
8. Auth pages have proper mobile padding
9. All fixes verified via Playwright at 360px and 375px viewports
10. No regressions on desktop (1280px) layout


## A note on implementation

- Favor making changes to base templates rather than every inheriting template.  This will make it easier to implement new functionality in a way that does not require tweaking and polishing in future. Reusability and DRY code matters!
