# Plan: more deploy preparation

Implementation plan for [`1. spec.md`](./1.%20spec.md). Two threads, both deploy-readiness:
(A) a Django system check that makes a blank Sentry **release** visible on every boot, and
(B) closing the `.env.example` gaps the infra `app_env` role depends on — the load-bearing one
being a production fail-fast when `WEBHOOK_ENCRYPTION_SALT` is unset.

**Framing (from the spec): smallest change that removes a real footgun, no scope creep.** The
only behavioural change to running code is the new webhook-salt fail-fast in production.
Everything else is a system check, a `.env`/doc parity fix, or a recorded template-repo handoff.

Follow **TDD**: for each unit write the failing test(s) first, then the implementation. Do not
write all tests up front.

## Existing code this builds on (keep DRY)

- **Check pattern** — `freedom_ls/course_access/checks.py` `check_preview_overrides_disabled_in_production`
  (W001): a conditional `Warning` with an early-return guard, registered via a decorator, and the
  app's `apps.py` importing the checks module in `ready()`
  (`freedom_ls/course_access/apps.py`, which sets `label = "freedom_ls_course_access"`). The new
  check follows this early-return-guard-then-`Warning` shape, but registers **untagged**
  (`@register()`, not `@register(Tags.security)`) and adds a `hint=` (idiomatic in the repo's other
  checks — `course_access.E001`, `icons/checks.py` — for a check that tells the operator what to do).
- **Fail-fast primitive** — `require_secret_key()` in
  `freedom_ls/deployment/settings_defaults.py:54-71` (read env → `.strip()` → raise
  `ImproperlyConfigured` when blank). `require_webhook_encryption_salt()` is a direct sibling;
  reuse the same shape and message style. Its tests mirror
  `test_require_secret_key_*` in `freedom_ls/deployment/tests/test_settings_defaults.py`.
- **Settings config accessor** — `freedom_ls/deployment/config.py` `DeploymentSettings` already
  exposes `SENTRY_DSN` / `SENTRY_RELEASE`, and `AppSettings.__getattr__`
  (`freedom_ls/base/app_settings.py:30-47`) already strips whitespace and normalises `""`/unset
  to the declared default (`None`). The check reads `config.SENTRY_DSN` / `config.SENTRY_RELEASE`
  and tests plain falsiness — **no** blank-string handling of its own.
- **`settings_prod` reassignment site** — `config/settings_prod.py:55`
  (`SECRET_KEY = fls_defaults.require_secret_key()`), after `from .settings_base import *`. The
  new `SALT_KEY` reassignment goes in the same region, same style.
- **Dev/test salt fallback** — `config/settings_base.py:434-450` derives a deterministic dev
  `SALT_KEY` from a hardcoded string. Leave this intact; the prod reassignment overrides it only
  in `settings_prod`.
- **CI deploy check** — `.github/workflows/security.yml` `django-check-deploy` job runs

## Thread A — Sentry release warning (Django system check)

### A1. `DeploymentAppConfig` label + checks wiring

**File:** `freedom_ls/deployment/apps.py`

- Add `label = "freedom_ls_deployment"` to `DeploymentAppConfig` (the app has no models/migrations
  and nothing references it by label — `INSTALLED_APPS` uses the full name `freedom_ls.deployment`
  — so this is safe). The check `id=` string is free-form and independent of the label; the reason
  to set the label is so app-scoped `manage.py check freedom_ls_deployment` and
  `SILENCED_SYSTEM_CHECKS` entries stay self-consistent with the `freedom_ls_deployment.W001`
  prefix used in code, matching the `freedom_ls_course_access` precedent (vs Django's bare default
  `deployment`).
- In `ready()`, alongside the existing `init_sentry()` call, import the checks module so its
  `@register()`ed check is discovered:

  ```python
  def ready(self) -> None:
      from freedom_ls.deployment import checks  # noqa: F401
      from freedom_ls.deployment.sentry import init_sentry

      init_sentry()
  ```

### A2. The check itself

**File (new):** `freedom_ls/deployment/checks.py`

Model the module docstring and check on `freedom_ls/course_access/checks.py`. Local imports inside
the function (avoid touching settings/registry at import time). **Type hints are required**
(CLAUDE.md) — type the signature exactly as `course_access/checks.py` does, with the matching
top-level imports (`from collections.abc import Sequence`, `from django.apps import AppConfig`,
`from django.core.checks import CheckMessage, Warning, register`).

```python
@register()
def check_sentry_release_set_when_dsn_set(
    app_configs: Sequence[AppConfig] | None, **kwargs: object
) -> list[Warning]:
    """W001: warn when SENTRY_DSN is set but SENTRY_RELEASE is blank/unset.

    A blank release only degrades Sentry's release-based features; it never
    breaks the running app — hence a Warning, not an Error. Silenceable via
    SILENCED_SYSTEM_CHECKS ("freedom_ls_deployment.W001").
    """
    from freedom_ls.deployment.config import config

    if not config.SENTRY_DSN:
        return []
    if config.SENTRY_RELEASE:
        return []
    return [
        Warning(
            "SENTRY_DSN is set but SENTRY_RELEASE is blank — Sentry events will "
            "be untagged, so regressions cannot be tied to a deploy.",
            hint=(
                "Set SENTRY_RELEASE (e.g. the git SHA) in this environment, or "
                "silence freedom_ls_deployment.W001 if release tracking is "
                "intentionally disabled."
            ),
            id="freedom_ls_deployment.W001",
        )
    ]
```

- **Registration:** plain `@register()` (untagged). Not `deploy=True` (would exclude it from the
  default `manage.py check`), not `Tags.security` (a missing telemetry tag is not a security
  boundary). Untagged means it fires on `check` / `migrate` / `runserver` / `test` and under
  `check --deploy` — so it surfaces during the deploy pipeline's one-off `manage.py migrate` and
  the CI `django-check-deploy` job for free.
- **Severity:** `Warning` — consistent with `course_access.W001`, never blocks `migrate`/`runserver`.
- **Scope guard:** cover **only** the `DSN × RELEASE` pair. Do **not** grow it to cover
  `SENTRY_ENVIRONMENT` (explicit spec non-goal).

### A3. Tests for the check

**File (new):** `freedom_ls/deployment/tests/test_checks.py`

Use `@override_settings(...)` to drive `config.SENTRY_DSN` / `config.SENTRY_RELEASE` (the config
accessor reads from Django settings, same mechanism `test_sentry.py` already uses). Call the check
function directly (`check_sentry_release_set_when_dsn_set(None)`) and assert on the returned list.

Cover all four truthiness combinations plus the blank-normalises-to-falsy case:

| `SENTRY_DSN` | `SENTRY_RELEASE` | Expected |
|---|---|---|
| set | blank/unset (`None`) | one `Warning`, `id == "freedom_ls_deployment.W001"` |
| set | `""` (empty string) | one `Warning` (normalises to falsy) |
| set | set | `[]` |
| unset (`None`) | unset | `[]` |
| unset (`None`) | set | `[]` |

Optionally assert the check is discoverable through `django.core.checks.registry` (i.e. `ready()`
wired it), but the direct-call tests are the core coverage.

## Thread B — `.env.example` gaps + webhook-salt fail-fast

### B1. `require_webhook_encryption_salt()` primitive (TDD)

**File:** `freedom_ls/deployment/settings_defaults.py`

Add a sibling to `require_secret_key()`:

```python
def require_webhook_encryption_salt() -> str:
    """Return WEBHOOK_ENCRYPTION_SALT from the environment, raising
    ImproperlyConfigured if unset/empty/whitespace-only.

    In production the dev fallback (a hardcoded deterministic salt in
    settings_base.py) silently weakens webhook-secret Fernet encryption, so this
    fails fast at settings-import time instead of shipping the insecure salt.
    """
    salt = os.environ.get("WEBHOOK_ENCRYPTION_SALT", "").strip()
    if not salt:
        raise ImproperlyConfigured(
            "WEBHOOK_ENCRYPTION_SALT must be set to a non-empty value in production."
        )
    return salt
```

**Tests (write first)** in `test_settings_defaults.py`, mirroring the four `require_secret_key`
tests: missing env → raises; empty string → raises; whitespace-only → raises; set → returns the
stripped value.

### B2. Reassign `SALT_KEY` in `settings_prod`

**File:** `config/settings_prod.py`

After `from .settings_base import *` (near the existing `SECRET_KEY = fls_defaults.require_secret_key()`
at line 55), add:

```python
SALT_KEY = fls_defaults.require_webhook_encryption_salt()
```

This overrides `settings_base`'s dev fallback only in production, closing the "insecure dev salt
silently reaches prod" path. Dev/test keep the deterministic fallback in `settings_base.py`.

### B3. Give the two existing `settings_prod` loaders a dummy salt

Both currently load `config.settings_prod` with no `WEBHOOK_ENCRYPTION_SALT`; without this the new
fail-fast raises `ImproperlyConfigured` at import, before either does anything useful. Each already
supplies a dummy `SECRET_KEY` — add the salt the same way.

- **CI:** `.github/workflows/security.yml`, `django-check-deploy` job `env:` block — add
  `WEBHOOK_ENCRYPTION_SALT: ci-dummy-webhook-salt  # pragma: allowlist secret` alongside the
  existing dummy `SECRET_KEY`.
- **Test:** `freedom_ls/deployment/tests/test_settings_defaults.py`
  `test_prod_settings_uses_database_task_backend` — it `monkeypatch.setenv`s `HOST_DOMAIN` and
  `SECRET_KEY` before `importlib.reload(config.settings_prod)`; add
  `monkeypatch.setenv("WEBHOOK_ENCRYPTION_SALT", "test-webhook-salt")`.
