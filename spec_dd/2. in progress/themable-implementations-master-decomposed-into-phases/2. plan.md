## Architectural principles (driven from the spec)

1. **Themes are sparse.** A theme contains only what it overrides. The minimal theme is one `theme.css` file.
2. **Defaults stay in the main project.** `tailwind.components.css` (component classes) stays at the repo root. Generic primitives (`base/templates/cotton/*`, `base/templates/partials/*`, `_base.html`) stay where they are. Nothing is migrated wholesale into the default theme.
3. **Each phase ends in a green build.** No half-implemented features.

These principles drive every task below.

## Pre-existing functionality to reuse (DRY)

- `freedom_ls/accounts/email_utils.py::parse_tailwind_colors` â€” already parses `--color-*` from a CSS file. The path it is called with changes (resolved theme's `theme.css`); the function itself does not.
- Cotton's existing template loader chain (`django_cotton.cotton_loader.Loader` followed by `filesystem.Loader` and `app_directories.Loader`) â€” themes are surfaced purely by prepending a directory to `TEMPLATES[0]["DIRS"]`. No new loader logic, no cache-key plumbing.
- `STATICFILES_DIRS` â€” themes are surfaced via the same prepend pattern. No new static finder.
- The existing role-token usages already in templates (`text-primary`, `bg-surface-2`, `border-border`, etc.) â€” only the *renamed* tokens need updating in Phase 1.

### Task 1.1 â€” Theme resolution module (`freedom_ls/base/theming.py`)

Build the resolver that the settings helper calls. Pure-Python, zero Django imports beyond `ImproperlyConfigured`, so it can be unit-tested without the test client.

**Public surface:**

```python
# freedom_ls/base/theming.py
from pathlib import Path
from django.core.exceptions import ImproperlyConfigured

FREEDOM_LS_PACKAGE_DIR: Path = Path(__file__).resolve().parent.parent  # freedom_ls/

def resolve_theme_dir(theme_slug: str, themes_dirs: list[Path]) -> Path:
    """Walk themes_dirs in order; return first <dir>/<theme_slug> that exists.

    Raises ImproperlyConfigured if nothing matches.
    """
    for parent in themes_dirs:
        candidate = Path(parent) / theme_slug
        if candidate.is_dir():
            return candidate
    raise ImproperlyConfigured(
        f"FLS theme {theme_slug!r} not found in any of: {themes_dirs!r}"
    )

def configure_theme(
    *,
    theme_slug: str,
    themes_dirs: list[Path],
    templates: list[dict],
    staticfiles_dirs: list,
) -> Path:
    """Resolve theme, prepend its templates/ dir to TEMPLATES[0]['DIRS']
    and its static/ dir to STATICFILES_DIRS. Returns the resolved theme dir.
    """
    resolved = resolve_theme_dir(theme_slug, themes_dirs)
    templates_dir = resolved / "templates"
    static_dir = resolved / "static"
    if templates_dir.is_dir():
        templates[0].setdefault("DIRS", [])
        templates[0]["DIRS"].insert(0, str(templates_dir))
    if static_dir.is_dir():
        staticfiles_dirs.insert(0, static_dir)
    return resolved
```

**Tests** (TDD; `freedom_ls/base/tests/test_theming.py`):

- `resolve_theme_dir` returns the first directory that exists when walking the dirs list.
- Downstream dir takes precedence over the FLS package dir (covers the "downstream theme named `default` shadows FLS one" case).
- Raises `ImproperlyConfigured` when no candidate exists â€” message includes the slug and the dirs searched.
- `configure_theme` prepends to both `TEMPLATES[0]['DIRS']` and `STATICFILES_DIRS` (use `tmp_path` to fake themes on disk).
- `configure_theme` is a no-op for the missing static / templates subdirs (a Tier-1-only theme has no `templates/` and is still valid).
- Walking past a missing parent dir gracefully falls through to the next entry (covers `BASE_DIR / "themes"` not existing in the FLS dev repo).

Use `pytest` + `tmp_path` fixtures. Follow conventions in `fls:testing` (`pytestmark = pytest.mark.django_db` is **not** needed â€” pure filesystem logic).

### Task 2.2 â€” Wire active-theme `theme.css` import into `tailwind.input.css`

The dev `tailwind.input.css` must import the active theme's `theme.css` after `tailwind.components.css` so future Tier-2 overrides cascade correctly. Two acceptable mechanisms:

```css
@import "./freedom_ls/themes/${FLS_THEME}/static/themes/${FLS_THEME}/theme.css";
```

If `${FLS_THEME}` interpolation isn't trivial in CSS-build tooling, fall back to A and document that downstreams should use the same template with `<theme_slug>` substituted.

```
resolve_theme_dir(slug, dirs):

        candidate = d / slug
```
