### Step 3 — flag, never edit

Reviewers emit findings reports per the schema in Task 3. They do not edit `2. plan.md`. The orchestrator parses each report.

**Validation gate (closes G-03).** Before applying any edits, the orchestrator validates each finding has all required fields (`bucket`, `confidence`, `plan section`, `problem`, `proposed fix`, `rationale`) and that `bucket ∈ {must-address, should-consider, fyi}` and `confidence ∈ {high, low}`. Any out-of-shape finding is itself recorded as a `must-address` item under `## Reviewer findings`, with text like:

### Step 4 — orchestrator applies edits

For each in-shape finding, in the order they came in:

1. Decide whether the finding's fix is **mechanical and unambiguous**. The definition is narrow and bounded by two layers (closes **G-04**):

   **Hard floor (always enforced).** A fix is mechanical only if it is one of:
   - Typo correction (spelling, obvious grammar).
   - **Addition** of `select_related` or `prefetch_related` to a query that accesses related objects.
   - **Addition** of a named factory to a test plan.
   - **Addition** of a test case.

   Any fix that *removes* an element, or that *renames* a function, model, URL, or field, is **not** mechanical regardless of bucket or confidence. Document this list inline in the command — do not just refer to "the spec" because future maintainers might miss it.

   **Soft gate (confidence-based).** A `low`-confidence finding is surfaced as an inline callout even when its proposed edit otherwise falls inside the allowlist. Confidence can only *downgrade* an auto-fix to a callout — never upgrade a disallowed edit to an auto-fix.

2. **If mechanical and `confidence: high`**: apply in place. Log under "Auto-applied" in the review log entry for this pass.

3. **If `bucket: must-address`** (and not auto-applied): insert an inline `> **<reviewer> concern:**` blockquote at the named plan section. Use the existing standalone-mode shape so the user sees a familiar pattern. The blockquote contents are the finding's `problem` and `proposed fix`, with the orchestrator stamping the reviewer name. Reviewer-emitted text is **escaped** before being spliced (closes **C-04** / V3c): any line in the reviewer's `problem`/`proposed fix` that starts with one of the following is escaped (prefixed with a non-breaking literal so it cannot break out of the blockquote nesting or open a new structural element):
   - `#` — would open a new heading
   - `> ` — would change blockquote nesting depth
   - `[!` — would open a GitHub-alert callout
   - ` ``` ` or `~~~` — would open or close a code fence
   - `---` (at start of line, followed only by `-`/whitespace) — would open YAML frontmatter or a horizontal rule that some renderers treat as a section break
   - `<!--` — would open an HTML comment some renderers strip silently

   Document this in the command file as a small "Splicing safety" subsection with the literal list above. Future maintainers must not shorten the list without re-running the threat model.

4. **If `bucket: should-consider` or `bucket: fyi`** (and not auto-applied): append a formatted bullet under `## Reviewer findings` at the bottom of the plan. Format: `- **<reviewer> § <plan section> (<bucket>):** <problem>. *Proposed:* <fix>.`

5. GitHub-alert callouts (`[!IMPORTANT]` / `[!WARNING]` / `[!NOTE]`) are reserved for *decisions* the orchestrator records — never for reviewer concerns. When a reviewer's finding leads to a new decision, the orchestrator authors and stamps the callout (`*plan-dev orchestrator, <today>*`); the reviewer name appears in the rationale, not the byline (closes **G-08**).

6. After processing all findings from a pass, append one section to `plan_review_log.md` per the spec's "Review log" format. The orchestrator stamps the date and pass number. Reviewers never write to `plan_review_log.md` directly. Append-only — never edit or remove past entries.

### Step 5 — mid-flow user input only for blockers

A "blocker" is narrowly defined:

- **Any `must-address` finding from the security reviewer is a blocker by definition** (closes **G-05**, control **C-06**). Stop and ask the user.
- A finding that would invalidate the work the next reviewer is about to do (e.g. structure: "this whole approach is wrong; replan from scratch").
- A contradiction between spec, plan, and QA plan that the orchestrator cannot resolve.

Everything else batches to the end-of-flow summary.

Document the blocker definition inline in the command — this is the load-bearing rule and must not drift.

### Step 6 — iteration cap

Each reviewer runs at most twice. **Pass 2 trigger**: pass 2 runs when pass 1 produced *any* edit to `2. plan.md` (auto-applied fix, inline callout, or `## Reviewer findings` append). If pass 1 returned no findings, pass 2 is skipped.

**Pass 2 outcome**: still-flagged `must-address` items become inline callouts; the reviewer is not re-run a third time. *Except* for security:

**Security exception (closes G-06, control C-07).** The security reviewer caps at 3 passes instead of 2. If the security reviewer's second pass produces *any* `must-address` finding, the orchestrator addresses it (per Step 4) and re-invokes security for a third and final pass. The exception applies *only* to security; testing and structure cap at 2 unconditionally. Don't try to distinguish "new" vs "re-flagged" findings across passes — any pass-2 `must-address` from security earns pass 3.
