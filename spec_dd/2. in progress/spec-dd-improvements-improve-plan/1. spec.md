# Spec: Less-interactive plan phase

## Summary

The current SDD plan phase requires three slash-command invocations (`/plan_from_spec`, `/plan_security_review`, `/plan_structure_review`) with a user gate between each. That is too much friction for the value the gates deliver. The structural/security/testing reviews don't need a human between them; they need their findings landing somewhere actionable.

This spec replaces the three-command plan phase with **two** commands:

1. **`/plan_qa`** — generates `3. frontend_qa.md` from the spec, fronted with declarative behaviour rules. Runs first so misunderstandings surface before mechanics get committed to.
2. **`/plan_dev`** — takes the spec (and the QA plan, if there is one) and produces `2. plan.md`. Internally orchestrates the existing reviewers as subagents, with a single end-of-flow report instead of three separate user gates.

The goals are: fewer context switches, behaviour caught before mechanics, reviewer findings batched and actionable, and the existing standalone review commands remain runnable on their own.

## Why this matters

- **Three context-switches per spec is too many.** Each gate forces the user to load the plan file fresh, re-read it, decide if anything needs editing, and re-launch the next command. The cost of the gate has to be justified by the chance it catches something — and most of the time it doesn't, because the gates sit *after* internal reviews that already auto-fix mechanical issues.
- **Mechanics-first planning misses misunderstandings cheaply.** Today's `/plan_from_spec` produces `2. plan.md` first and `3. frontend_qa.md` as an afterthought. The behaviour view (what the user sees, what done looks like) is the cheapest place to catch wrong assumptions; doing it second means the implementation plan has already committed to a shape.
- **Reviewers undoing each other ("Logic Lock") is a real failure mode.** Multi-agent review research (`research_multi_agent_review.md`) names this explicitly. Running reviewers sequentially in a fixed order, with one writer (the orchestrator), avoids it.
- **Lightweight BDD as a thinking tool, not a deliverable.** 71% of teams that say they "do BDD" never automate the Gherkin (`research_bdd_and_test_first.md`). The leverage comes from writing the rules, not running them. Plain markdown given/when/then captures that leverage at zero tooling tax.

## Scope

### In scope

- New command `/plan_qa` (writes `3. frontend_qa.md`).
- New command `/plan_dev` (writes `2. plan.md`, orchestrates reviewers as subagents).
- New standalone command `/plan_testing_review` (also called as a subagent by `/plan_dev`).
- Decisions and concerns surfacing scheme inside `2. plan.md` (inline GitHub alerts for *decisions*, inline `> **<reviewer> concern:**` blockquotes for `must-address` *concerns*, three concern buckets, end-of-flow terminal summary).
- Updates to `setup_todo_list.md` so the generated `todo.md` reflects the new two-command shape.
- Existing standalone commands (`/plan_security_review`, `/plan_structure_review`) keep their current behaviour when run directly by the user, but gain a "subagent mode" entrypoint that the orchestrator invokes. In subagent mode the reviewer emits a structured findings report instead of editing `2. plan.md` (see "Step 3 — flag, never edit (reviewer rule)" below). In standalone mode the reviewer continues to edit `2. plan.md` as today.

### Out of scope

- **Replacing the human review checkpoint between `/plan_qa` and `/plan_dev`.** The QA-plan review *is* the conversation that lightweight BDD depends on. Protect it.
- **Replacing `/plan_security_review` and `/plan_structure_review` as standalone commands.** They stay first-class so a user can re-run them after manually editing the plan.
- **Living documentation / executable specs.** No Gherkin runners (Cucumber, pytest-bdd, behave). Plain markdown only.
- **Renaming or renumbering existing artifacts.** `1. spec.md`, `2. plan.md`, `3. frontend_qa.md` keep their names. The order they're produced in changes (`3.` before `2.`) but the numbers reflect the position of each artifact in the workflow's *output set*, not the order of creation, so renumbering would create churn for nil benefit.
- **Modifying existing commands beyond what is required to add a subagent-mode entrypoint to `/plan_security_review` and `/plan_structure_review`, delete `/plan_from_spec`, and update the helper that writes the todo template.** Standalone-mode behaviour of the reviewer commands stays unchanged.
- **Changing the implementation phase or anything downstream** (`/implement_plan`, `/security-review`, `/do_qa`, etc).

## Functional requirements

### `/plan_qa`

1. **Input.** Reads `1. spec.md` from the same directory. Reads any `research*.md` files for context. Reads any threat-model artefact that exists.
2. **Skippable.** If the feature has no frontend AND no frontend-triggered side effects worth checking manually, write nothing and tick the `/plan_qa` todo item. Do not add a "No QA needed" todo entry — todo boxes represent work that still needs doing, not a log of decisions. The skip reason is reported in the command's terminal output, not in `todo.md`. (This is a deliberate departure from the prior `/plan_from_spec` behaviour, which appended a `No QA needed` entry under the downstream `## 9. QA` section.)
3. **Output structure** (`3. frontend_qa.md`): one `### BR-NN — <short title>` heading per behaviour rule. Under each heading, in order:
   - The rule itself, as plain-markdown given/when/then bullets, declarative ("the learner sees their progress update", not "click Submit").
   - The Playwright walkthrough that exercises it, written as concrete steps for the QA agent.

   The number of rules is whatever the feature actually has — sometimes one, sometimes twenty. Do not pad the list to hit a target, and do not collapse genuinely distinct behaviours to fit under a cap. IDs are stable and **append-only** (`BR-01`, `BR-02`, …): when a new rule is added later it gets the next free number; existing IDs never renumber, even if a rule is removed (its ID is retired, not reused). This preserves references from `2. plan.md`. Gaps in the sequence are acceptable.

   If a single walkthrough exercises more than one rule, place it under the primary rule and add `Also exercises: BR-NN, BR-NN` at the top of that walkthrough rather than duplicating the steps.
4. **Invisible side effects.** If the frontend isn't visibly affected but a frontend interaction triggers backend behaviour the user can't see (e.g. a webhook fires, a snapshot gets recorded), the walkthrough under the relevant rule verifies the side effect via dev tools or admin (whatever path is cheapest).
5. **Behaviour rules are the contract.** They're sourced from the spec's user-facing requirements. The Playwright walkthrough is a *derivable* artefact — if the rules change, the walkthrough is regenerated, so it isn't double-maintenance.
6. **No Gherkin tooling.** Plain markdown only. No Cucumber, no pytest-bdd, no behave.
7. **Todo update.**
   - On success (a QA plan was written): tick the "Run `/plan_qa`" item, and append a `(user) Review the QA plan — especially the behaviour rules — and edit where needed` item under `## 4. QA plan`.
   - On skip (no QA plan was written): tick the "Run `/plan_qa`" item only. No `add:` entries.

   The `(user)` review item is added by `/plan_qa` rather than living in the `setup_todo_list` template, because there is nothing to review when QA is skipped and todo boxes only represent outstanding work.

### `/plan_dev`

1. **Input.** Reads `1. spec.md`, `3. frontend_qa.md` (if it exists), any `research*.md`, and any threat-model artefact.
2. **Step 1 — first-pass plan.** The orchestrator drafts `2. plan.md` using essentially the same logic as today's `/plan_from_spec`, except: tasks reference behaviour-rule IDs (`BR-N`) where applicable, and the plan file gains a `## Decisions index` section at the top (initially empty) and a `## Reviewer findings` section at the bottom (initially empty).
3. **Step 2 — sequential reviewer subagents.** Spawn three subagents in this order, each as a separate Task invocation:
   1. `/plan_testing_review` (new) — testing-practice review.
   2. `/plan_structure_review` (existing) — cross-app boundary review.
   3. `/plan_security_review` (existing) — security review.
   The order is fixed because testing changes the *shape* of tasks, structure looks at *where* code lives across those tasks, and security audits the *final* shape. Running them in parallel, or in a different order, causes Logic Lock or wasted work (`research_multi_agent_review.md`).

   Slash commands cannot be invoked as agents directly — agents are defined under `fls-claude-plugin/agents/` and the Task tool dispatches by agent name. The plan-phase implementation will need to either (a) create one wrapper agent per reviewer that loads the corresponding slash-command instructions, or (b) inline the reviewer instructions into a generic Task prompt. Pick at planning time; this spec stays neutral on the mechanism.
4. **Step 3 — flag, never edit (reviewer rule).** Reviewers do not edit `2. plan.md`. They emit a structured findings report back to the orchestrator. Each finding has: a bucket (`must-address` / `should-consider` / `fyi`), a one-sentence problem statement, a concrete proposed fix, and the section of the plan it applies to.

   This is the architect/editor split (Aider, obra/superpowers). Two writers on the same file is what causes Logic Lock; one writer keeps it stable.
5. **Step 4 — orchestrator applies edits.** The orchestrator processes findings in the order they came in. For each finding:
   - If the fix is mechanical and unambiguous (typo, missing `select_related`, missing factory in test plan, demonstrably-redundant migration step), apply it in place and log it under "What I changed already" in the end-of-flow summary.
   - If the fix involves a real trade-off and the bucket is `must-address`, insert an inline `> **<reviewer> concern:**` blockquote (see the "Concerns" section below for the exact shape) at the relevant section of the plan.
   - If the bucket is `should-consider` or `fyi`, append the finding under `## Reviewer findings` at the bottom of the plan as a formatted bullet (reviewer name, plan section, problem, proposed fix). The reviewer's emitted text is rendered into that bullet — it is not placed inline anywhere else in the plan.

   GitHub-alert callouts (`[!IMPORTANT]` / `[!WARNING]` / `[!NOTE]`) are reserved for *decisions* the orchestrator records — see the "Decisions" section below. Reviewer concerns are surfaced as the `> **<reviewer> concern:**` blockquote pattern, not as alert callouts.

   At the end of each reviewer pass, the orchestrator appends one section to `plan_review_log.md` listing every auto-applied edit and every surfaced concern from that pass, each with its *why* (the reviewer's stated rationale, or the orchestrator's reason for surfacing rather than auto-applying). See the "Review log" subsection below for the file format. Reviewers never write to the review log directly.
6. **Step 5 — mid-flow user input only for blockers.** A "blocker" is narrowly defined: a finding that would invalidate the work the next reviewer is about to do (e.g. structure reviewer: "this whole approach is wrong; replan from scratch") or a contradiction in the spec the orchestrator can't resolve. Everything else batches to the end-of-flow summary. This rule is what makes the command "less-interactive" — it's the load-bearing decision.
7. **Step 6 — iteration cap.** Each reviewer runs at most twice (with one narrow exception for security, below).

   **Pass 2 trigger.** Pass 2 runs when pass 1 produced *any* edit to `2. plan.md` — auto-applied mechanical fix, inline `must-address` callout, or a `## Reviewer findings` append. If pass 1 returns no findings at all, pass 2 is skipped and the reviewer is considered done.

   **Pass 2 outcome.** If pass 2 still flags `must-address` items, those become inline `> **<reviewer> concern:**` blockquotes in the plan; the reviewer is not re-run a third time. This guards against Logic Lock without infinite re-runs.

   **Security-reviewer exception.** The security reviewer is capped at 3 passes instead of 2. If the security reviewer's second pass produces *any* `must-address` finding, the orchestrator addresses it (per Step 4 rules) and re-invokes the security reviewer for a third and final pass. If the third pass still flags `must-address` items, those become inline `> **Security concern:**` blockquotes; there is no fourth pass. The exception applies *only* to the security reviewer — testing and structure cap at 2 unconditionally. Rationale: a security regression introduced while fixing an earlier security finding is exactly the case where stopping at 2 ships a real vulnerability, and the cost of one more focused pass is small compared to the cost of missing it. We do not try to distinguish "new" vs "re-flagged" findings across passes — any pass-2 `must-address` from security earns pass 3.
8. **Step 7 — commit per reviewer.** Once a reviewer has finished its passes, the orchestrator commits the changes to `2. plan.md` and `plan_review_log.md` (and any other touched files) as a single git commit before moving on to the next reviewer. Commit message format: `plan: <reviewer> review pass` (e.g. `plan: testing review pass`). This makes each reviewer's contribution visible in `git log` and `git diff`, so the user can see exactly what each reviewer changed without having to read the whole plan from scratch. Use `uv run git commit` per project convention.

   **No-change reviewers.** If a reviewer made zero changes (no auto-applied fixes, no callouts, no `## Reviewer findings` appends, and pass 2 was skipped per Step 6), there is nothing to commit. The orchestrator skips the commit for that reviewer and notes the no-op in the end-of-flow summary and the review log. `git commit` is never invoked on a clean working tree.
9. **Step 8 — end-of-flow terminal summary.** After all three reviewers finish, the orchestrator prints a single summary:
   ```
   ## Plan complete — 3 reviewers ran (testing, structure, security)

   **Auto-applied (N):** Fixed in the plan, no action needed. Full list with rationale in `plan_review_log.md`.
   - testing: <one-liner>
   - structure: <one-liner>
   - security: <one-liner>
   - … (testing/structure entries may collapse if >5; security entries never collapse here)

   **Must-address concerns (M):** See inline `> **<reviewer> concern:**` blockquotes in plan.md.
   1. **<reviewer> § <section>**: <problem>. Proposed: <answer>. OK?
   2. …

   **Open decisions (D):** See inline `[!NOTE]` callouts in plan.md (if any).
   - <one-liner>

   **Worth considering (K):** See `## Reviewer findings` at bottom of plan.
   - <reviewer>: <one-liner>
   - …

   **FYI (L):** Listed in `## Reviewer findings`. No response required.
   ```
   The user reads ~10 lines and knows: what's done, what blocks them, what's optional.
10. **Todo update.** On success, ticks the new "Run `/plan_dev`" todo item, and appends one todo entry per inline `must-address` callout that still needs the user's decision under the `Implementation plan` section (so the user can't silently miss them). Marker is `user`; text is `"Resolve <reviewer> concern: <short label>"` (matching the existing per-callout pattern from the standalone reviewer commands).

### `/plan_testing_review` (new standalone command)

A new reviewer command parallel to `/plan_security_review` and `/plan_structure_review`. Same shape:

- Reads `1. spec.md`, `2. plan.md`, `3. frontend_qa.md` (if present).
- Walks through the plan applying the conventions documented in the `fls:testing` skill (and `fls:playwright-tests` where E2E coverage is in scope). The reviewer must not introduce its own checklist that diverges from those skills — its job is to enforce the skill, not to redefine what good testing looks like. The one plan-phase-specific check that isn't in the skills: tasks should reference behaviour-rule IDs (`BR-N`) from `3. frontend_qa.md` where applicable, so the implementer can tick rules off as they implement.
- In standalone mode: edits the plan directly for mechanical fixes; emits `> **Testing concern:**` callouts otherwise (same shape as the other standalone review commands).
- In subagent mode (invoked by `/plan_dev`): emits a structured findings report instead of editing the plan. The orchestrator applies edits per the rules in `/plan_dev` Step 4.
- Same `update_todo` contract as the other reviewers when run standalone (ticks its own item, adds per-callout user todos). In subagent mode, todo updates are deferred to the orchestrator.

### Decisions and concerns surfacing scheme

This is the in-document UX inside `2. plan.md`. It applies to whatever the orchestrator and reviewers produce; the user uses it to navigate.

#### Decisions

A "decision" is something the planning agent (or a reviewer) decided on the user's behalf. Three kinds, distinguished by GitHub-native alert blockquotes:

- `[!IMPORTANT]` — **user-confirmed.** The user answered an explicit question. Reviewer can skim past.
- `[!WARNING]` — **auto-decided.** The agent made a judgement call without asking. Reviewer must eyeball.
- `[!NOTE]` — **open question.** The agent flagged but did not decide. Must be answered before plan freeze.

Each callout uses a Y-statement-derived MADR-bare-minimal block:
```markdown
> [!WARNING]
> **Decision D-007 (auto)** — *plan-reviewer subagent, 2026-04-28*
> **Context:** <one sentence>
> **Decision:** <what was chosen>
> **Rationale:** <why>
> **Alternatives considered:** <optional>
> **Revisit if:** <optional>
```

A `## Decisions index` section sits at the top of the plan with one bullet per decision linking to its anchor. The index is auto-regenerated, never hand-maintained. There is no separate `decisions.md`. (See `research_decision_logging.md` for why inline + index beats both alternatives.)

#### Concerns

A "concern" is a reviewer finding. Three buckets, named in human language (no P0/P1/P2 — numeric severity inflates):

- **Must address** — the plan can't proceed safely without a decision. Lives inline in `2. plan.md` as a `> **<reviewer> concern:**` blockquote at the relevant section. (Compatible with the existing `/plan_security_review` callout shape, just generalised across reviewers.)
- **Should consider** — a real trade-off, but the plan still works as drafted. Lives in `## Reviewer findings` at the bottom of the plan.
- **FYI** — context, observation, optional. Lives in `## Reviewer findings`.

The end-of-flow summary tells the user where to find each bucket.

#### Review log

A separate `plan_review_log.md` (sibling to `2. plan.md`, no number prefix because it is a record of actions rather than a workflow output) records the operational history of every reviewer pass: every auto-applied edit and every surfaced concern, each with its *why*. This is distinct from the inline decision callouts above — those capture architectural choices in/about the plan content; the review log captures the *process* of how those choices and edits got made. The "no separate `decisions.md`" rule above still holds: the review log is not a decisions store.

Created by `/plan_dev` in Step 1 with a header but no entries. Appended to once per reviewer pass by the orchestrator (Step 4). Reviewers never write to the review log directly. Append-only across the whole `/plan_dev` run — past entries are never edited or removed, even when a later reviewer overrides an earlier auto-fix (the override gets its own entry).

Format per pass:

```markdown
## <Reviewer> review pass — <YYYY-MM-DD> (pass <N>)

### Auto-applied
- **<plan section> — <one-line what>.**
  *Why:* <reviewer's stated rationale>. *Source:* <reviewer>, confidence <high|low>.

### Surfaced as inline callouts (must-address)
- **<plan section> — <one-line concern>.**
  *Why surfaced:* <orchestrator's reason for not auto-applying>. *Source:* <reviewer>.

### Surfaced under Reviewer findings (should-consider / fyi)
- **<plan section> — <one-line>.** *Source:* <reviewer>, bucket <should-consider|fyi>.
```

This is the untruncated source of truth that the Step 8 end-of-flow summary points at. It also closes the audit-trail gap that the terminal summary's collapse rule would otherwise create for high-volume auto-fix passes (see threat model `G-10`).

### Updates to `setup_todo_list.md` and the todo template

The generated `todo.md` currently has section "4. Plan" with a single `(cmd) Run /plan_from_spec`, then sections "5. Plan security review" and "6. Plan structure review", each with their own command + user-review pair. This becomes:

```markdown
## 4. QA plan

- [ ] (cmd) Run `/plan_qa` to generate the QA plan

## 5. Implementation plan

- [ ] (cmd) Run `/plan_dev` to generate the implementation plan (runs testing, structure, and security reviews internally)
- [ ] (user) Address any decisions surfaced in the plan
- [ ] (user) Review reviewer findings at the bottom of the plan
```

The `(user) Review the QA plan` item is **not** in the template. Todo boxes represent outstanding work, and there is no QA plan to review until `/plan_qa` has run successfully. `/plan_qa` appends that user-review item itself when it produces a QA plan; on skip it appends nothing.

The standalone `/plan_security_review`, `/plan_structure_review`, and `/plan_testing_review` are *not* listed as separate top-level steps in the new template — they run inside `/plan_dev`. They remain available as escape hatches the user can invoke manually after editing the plan, but they aren't on the default happy path.

The artifact-detection rules in `setup_todo_list.md` step 3 also need updating: tick `## 4. QA plan`'s command item if `3. frontend_qa.md` exists, tick `## 5. Implementation plan`'s command item if `2. plan.md` exists. When `3. frontend_qa.md` is detected, also append the `(user) Review the QA plan…` item under `## 4. QA plan` (so a user who manually wrote the QA plan still gets the review item) — `setup_todo_list` mirrors what `/plan_qa` would have done on success.

### `/plan_from_spec` removal

The current `/plan_from_spec` command file is **deleted**. Its responsibilities are split between `/plan_qa` and `/plan_dev`. Anything in the existing command that needs to survive (the spec-context-reading logic, the plan-document writing logic, the subagent-skills-review pass) is moved verbatim into the appropriate new command.

> [!WARNING]
> **Decision D-001 (auto)** — *spec author, 2026-04-28*
> **Context:** The user's idea introduces `/plan_qa` and `/plan_dev` but doesn't explicitly say what happens to `/plan_from_spec`.
> **Decision:** Delete `/plan_from_spec`. Keep nothing under that name.
> **Rationale:** Two commands replacing one is the whole point. Leaving a third "old" command around invites confusion about which is canonical. The two new commands fully cover its responsibilities.
> **Alternatives considered:** Keep `/plan_from_spec` as a deprecated alias for `/plan_dev` (rejected: zero benefit, ongoing maintenance), keep it as the no-QA path (rejected: `/plan_qa` is already skippable, so `/plan_dev` works fine alone).
> **Revisit if:** users are observed running `/plan_from_spec` from muscle memory and getting confused.

## Non-functional requirements

- **Reviewer commands stay individually runnable.** A user who has manually edited `2. plan.md` after `/plan_dev` ran must be able to do `/plan_security_review` (or `/plan_structure_review`, `/plan_testing_review`) from the CLI and get the same behaviour they get today. This is the escape hatch.
- **No silent overrides of architectural intent.** Reviewers may auto-apply mechanical fixes; they may not silently change the plan's overall approach. Anything bigger than a "mechanical and unambiguous" fix becomes a callout.
- **Audit trail.** Every reviewer pass logs to the end-of-flow summary what it changed and what it flagged. The user is never left wondering what happened.
- **Reproducibility.** The orchestrator's edits are deterministic with respect to reviewer input — re-running `/plan_dev` on a clean spec produces a comparable plan (same structure, same callouts where the spec was ambiguous).
- **Fail-loud on contradictions.** If the spec contradicts itself or the QA plan, `/plan_dev` halts and asks the user to fix the spec before continuing. Don't try to plan around a broken spec.

## Edge cases

- **No frontend and no side effects.** `/plan_qa` writes nothing, ticks the `/plan_qa` todo item, prints the skip reason in its terminal output (no `add:` entry — see "QA plan" Step 2/Step 7), and exits cleanly. `/plan_dev` runs without a `3. frontend_qa.md` input, generates a plan whose tasks reference no behaviour rules, and the testing reviewer skips the "tasks reference BR-N" check.
- **QA plan exists but has no behaviour rules.** Equivalent to "no frontend" for the dev plan's purposes — testing reviewer skips the behaviour-rule cross-reference. Possible if the feature only has Playwright walkthroughs that aren't generalisable into rules (rare but possible for one-off admin flows).
- **A reviewer crashes mid-run.** The orchestrator records the crash in the end-of-flow summary (`reviewer X failed: <one-line reason>`), continues with the remaining reviewers, and surfaces the failure as a `must-address` item. Don't silently swallow.
- **Two reviewers disagree on the same section.** Each reviewer gets one finding per section per pass. If reviewer A flags it as `should-consider` and reviewer B flags it as `must-address`, the higher bucket wins and the section gets an inline callout citing both reviewers.
- **User runs `/plan_dev` when `2. plan.md` already exists.** Ask before overwriting. Don't silently regenerate. (Same rule as `setup_todo_list.md` step 2.)
- **User runs `/plan_qa` when `3. frontend_qa.md` already exists.** Same ask-before-overwriting rule.

## Success criteria

- A user can run the full plan phase with **two** slash-command invocations (`/plan_qa`, then `/plan_dev`) where they previously needed three.
- Behaviour rules are present in `3. frontend_qa.md` for any feature with frontend behaviour, in declarative given/when/then form, with stable IDs.
- `2. plan.md` produced by `/plan_dev` contains: a `## Decisions index` at the top, inline GitHub-alert callouts (`[!IMPORTANT]` / `[!WARNING]` / `[!NOTE]`) for decisions, inline `> **<reviewer> concern:**` blockquotes for `must-address` concerns, a `## Reviewer findings` section at the bottom for `should-consider` and `fyi` items, and tasks that reference behaviour-rule IDs where applicable.
- The end-of-flow summary printed by `/plan_dev` lets the user know in ~10 lines exactly: what was auto-fixed, what blocks them (with proposed answers), and what's optional.
- `plan_review_log.md` exists alongside `2. plan.md` after `/plan_dev` runs, with one append-only section per reviewer pass listing every auto-applied edit and every surfaced concern, each with its *why*.
- The standalone commands (`/plan_security_review`, `/plan_structure_review`, `/plan_testing_review`) all still run from the CLI on their own and produce their existing-shape output.
- `setup_todo_list.md` writes a todo whose plan-phase items match the new two-command shape.
- No new third-party dependencies (no Cucumber, pytest-bdd, behave, or similar).
- No changes to anything downstream of the plan phase (`/implement_plan`, `/security-review`, `/do_qa`, etc).
