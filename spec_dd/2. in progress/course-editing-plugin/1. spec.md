# Specification: Course Editing Plugin (`fls-content`)

A self-contained Claude Code plugin that helps people author FLS course content
correctly — and converts messy Markdown into valid FLS content — **without needing
access to the FLS source code**.

> Status note: derived from `idea.md` and the four `research_*.md` files in this
> directory (authoritative codebase reference, plugin-authoring patterns, markdown
> conversion best practices, and SDD-sync integration). Those files are the
> source of truth for the literal facts this spec refers to; this spec defines
> *what to build and why*, not the full content of every reference table.

---

## 1. Problem and audience

People authoring FLS courses write Markdown and YAML by hand against a specific
set of conventions: content types, file/folder layout, `NN. name` numbering,
frontmatter schemas, `c-*` cotton widgets, and UUID rules. Getting any of this
wrong means content fails `content_validate`/`content_save` or renders
incorrectly.

The authors who hit this hardest work in **downstream projects that do not have
the FLS source code**. They cannot read `schema.py`, `settings_base.py`, or the
cotton component templates to discover the rules. Today their only reference is
the running FLS host (if they have one) and trial-and-error against
`content_save`.

This plugin is the reference and tooling those authors rely on **instead of** the
unavailable codebase. Its job is **integration** — what tool to use where, the
exact syntax, and the correct structure — **not pedagogy or course design**.

---

## 2. Goals

1. Explain the FLS content **file types** (course, course part, topic, activity,
   form/quiz/survey, form page, form question/content) and how to author each.
2. Explain how to **organise** files — directory layout and the `NN. name`
   numbering convention.
3. Explain the **widgets** (`c-*` cotton components): purpose, syntax, attributes,
   and simple best practices, including the complete allowlist so authors know
   they cannot invent widget names.
4. Make the **UUID rules** unmissable: `content_save` writes UUIDs; authors never
   create, edit, or duplicate them.
5. **Convert messy Markdown into FLS content** — a single `.md` file or a
   directory of them — structurally and losslessly.
6. Let authors **validate offline**, without the FLS source, via a bundled
   trimmed copy of the FLS schema validator.
7. **Stay in sync with FLS** automatically: when FLS course-authoring code
   changes, an SDD workflow step updates the plugin (and the bundled validator).

### Why these matter

Goals 1–4 remove the most common authoring failures (wrong frontmatter key,
unknown widget, hand-edited UUID) for an audience that has no other reference.
Goal 5 is the highest-leverage action — turning existing content into valid FLS
structure is the main reason an author reaches for this plugin. Goal 6 closes the
feedback loop offline so authors are not blocked on an FLS host. Goal 7 is what
keeps the whole thing trustworthy over time: a reference that drifts from the code
is worse than no reference.

---

## 3. Scope

In scope (all four deliverables):

- **D1 — Reference skills** (passive, always-available): `content-types`,
  `widget-reference`, `conventions`, `markdown-conversion`.
- **D2 — Conversion command + agent**: `/fls-content:format-content`, backed by
  the `markdown-conversion` skill, fanning out one converter agent per source
  file, plus a `_conversion_review.md` written only when something needs the author.
- **D3 — Bundled standalone validator**: a trimmed, Django-free copy of FLS's
  `schema.py` + `validate.py` shipped inside the plugin, plus a
  `/fls-content:validate-content` command that runs it for non-technical authors
  (no Python or `uv` knowledge needed).
- **D4 — SDD sync wiring**: an `/update_claude_plugin_fls_content` command plus a
  `todo.md` template item that keeps the plugin and its bundled validator in sync
  when FLS authoring code changes.

Plus a small **`init` command** (`/fls-content:init`) that scaffolds the
deployment config file (`.fls-content.yaml`) in a content repo and is
non-destructive on re-run (see §5).

Out of scope:

- **Distribution to other projects.** The plugin lives in this repo for now; how
  it ships to source-less downstream projects is decided later.
- **Teaching pedagogy or course design.** The plugin is about format and
  structure only.
- **Rewriting, summarising, or generating course prose.** Conversion is a
  structural transform; see §9.
- **Replacing `content_save`.** The authoritative pass (UUID assignment,
  cross-reference resolution, asset upload) still happens on an FLS host.

## 4. Plugin shape and naming

- A new directory at the repo root, sibling to `fls-claude-plugin/`:
  **`fls-content-plugin/`**.
- Plugin name: **`fls-content`** (in `.claude-plugin/plugin.json`). This gives the
  namespace `fls-content:` for skills/agents and `/fls-content:format-content`
  for the command. It must **not** reuse the developer plugin's `fls:` namespace —
  that would collide.
- The plugin must be **self-contained and literal**: every authoring fact (valid
  `content_type` values, FORM `strategy` values, the full widget allowlist with
  attributes, escaping rules, numbering, UUID rules) is stated in the plugin, not
  referenced from FLS source the author cannot see.

Layout (per `research_claude_plugin_structure.md`):

```
fls-content-plugin/
├── .claude-plugin/plugin.json          # name: "fls-content"
├── skills/
│   ├── content-types/                  # SKILL.md + resources/ (per-file-type frontmatter, layout, numbering)
│   ├── widget-reference/               # SKILL.md + resources/ (every c-* widget, attributes, examples)
│   ├── conventions/                    # SKILL.md (numbering, UUID, file naming, escaping) — self-contained
│   └── markdown-conversion/            # SKILL.md + resources/ (conversion patterns)
├── commands/
│   ├── format-content.md               # /fls-content:format-content
│   ├── validate-content.md             # /fls-content:validate-content (runs the bundled validator; no Python/uv needed)
│   └── init.md                         # /fls-content:init (scaffolds .fls-content.yaml; non-destructive)
├── agents/
│   └── content-formatter.md           # per-file conversion worker (Sonnet-tier)
└── validate/                           # bundled trimmed schema.py + validate.py (D3)
```

## 5. Project configuration: deployment-specific values

Some authoring facts are **not fixed across FLS deployments** — the clearest case
is **admonition types**. The base set (`note`, `tip`, `important`, `warning`,
`danger`, `key_takeaways`, `checklist`, plus the `default` fallback) is defined in
`config/settings_base.py`, but downstream projects extend `ADMONITION_TYPES` in
their own settings (e.g. `settings_dev.py` adds a `regulation` type). An author in
deployment A may have `regulation` available; an author in deployment B may not. A
hardcoded list in the plugin would therefore be wrong for some deployments — and
an unknown `c-admonition type="..."` value **fails silently**: the cotton template
falls back to the `default` style with no error, so the author never learns the
type was invalid.

### Solution: the content repo declares its types; the plugin reads them

The **content repository** (where the author's course content lives) carries a
small config file declaring the deployment-specific values, and the plugin reads
it instead of relying on a hardcoded list.

- **File:** `.fls-content.yaml` at the content root (course root). The leading dot
  means `content_save`'s scanner skips it (files starting with `.` are excluded),
  so it is never mistaken for content.
- **Discovery:** when run against a single file or a subdirectory, the plugin
  searches upward from the target to find the nearest `.fls-content.yaml`.
- **Shape** — `admonition_types` is the only key required for now; the file is a
  deliberate home for future deployment-specific values:

  ```yaml
  # .fls-content.yaml — deployment-specific FLS authoring config
  admonition_types:
    - note
    - tip
    - important
    - warning
    - danger
    - key_takeaways
    - checklist
    - regulation   # custom type added in this deployment
  ```

- **Who populates it:** whoever sets up the content repo for a deployment copies
  that deployment's `ADMONITION_TYPES` keys into this file. The plugin documents
  how and ships the base set as a starting template.
- **Fallback:** if no `.fls-content.yaml` is found, the plugin uses the documented
  **base set** as a safe default and notes (in conversion output, or when asked)
  that deployment-specific types cannot be confirmed without the config.

### Initialising a content repo: `/fls-content:init`

`/fls-content:init [target]` scaffolds the config so an author never has to write
it from memory. It:

- **Creates `.fls-content.yaml`** at the content root (default: the current
  directory) when absent, pre-populated with the **base admonition set** and a
  commented example showing how to add deployment-specific types.
- **Is non-destructive and safe to re-run.** If `.fls-content.yaml` already
  exists it is **left exactly as-is** — never overwritten, reordered, merged, or
  deleted. The command reports that the file is already present (and, as a
  convenience, which expected keys it does and does not contain) and exits without
  writing.
- **Never touches existing content.** It does not read, modify, rename, or delete
  any course files, images, or UUIDs; its only ever write is *creating* the single
  config file when absent.

This is the only command that writes the config file; every other part of the
plugin only reads it.

### What consumes it

- **`widget-reference` skill** — when documenting or recommending a
  `c-admonition type`, it treats the config's `admonition_types` as the valid set
  for the author's project, with the base set as the documented default.
- **Conversion (D2)** — when proposing a blockquote/callout → `c-admonition`, it
  only proposes a `type` present in the config (or the base set on fallback), and
  flags any source whose intended type is not in the list rather than emitting a
  silently-wrong one.

The **bundled validator (D3) is not involved**: an admonition `type` lives in the
markdown body, not frontmatter, so the Pydantic schema never checks it. This is an
authoring/conversion-time lint, not a schema concern.

D4 (§10) keeps the plugin's documented **base set** current with
`settings_base.py`; per-deployment additions live only in each content repo's
`.fls-content.yaml` and are not the plugin's to maintain.

## 6. D1 — Reference skills

Four skills. Each `SKILL.md` is the fast-load summary whose `description`
frontmatter is the trigger; deep reference material lives in `resources/` files
loaded only when needed. Keep `SKILL.md` bodies lean (≈ ≤ 80 lines); push
exhaustive tables and multi-example walkthroughs to `resources/`.

| Skill | Purpose | Trigger keywords (author vocabulary) |
|---|---|---|
| `content-types` | Per-type frontmatter, file format, directory structure, numbering | topic, form, quiz, survey, course, course part, chapter, module, content file |
| `widget-reference` | Every `c-*` widget: purpose, attributes, examples, the full allowlist | widget names (admonition, flashcard, accordion, youtube, picture, table, code block, card, pull quote, equation), "widget", "component" |
| `conventions` | Numbering, file naming, UUID rules, HTML-escaping rules | uuid, numbering, ordering, file name, folder, directory |
| `markdown-conversion` | When/how to map Markdown onto FLS structure (backs D2) | convert, messy markdown, restructure, import, paste content |

Content requirements (the literal facts, drawn from `research_authoring_format.md`):

- **All eight content types** with their file format, required/optional fields,
  and at least one copy-pasteable frontmatter example each. Note the non-obvious
  ones explicitly: `part.yaml` has **no closing `---`**; `FORM_QUESTION` and
  `FORM_CONTENT` inherit a smaller base (no `title`/`subtitle`/`description`);
  the parser picks `FORM_CONTENT` vs `FORM_QUESTION` by whether the YAML section
  has a `content` key or a `question` key.
- **FORM rules**: `strategy` is `QUIZ` or `CATEGORY_VALUE_SUM`; `quiz_show_incorrect`
  and `quiz_pass_percentage` are required for `QUIZ` and must be absent otherwise.
- **The complete `MARKDOWN_ALLOWED_TAGS` allowlist** with each widget's permitted
  attributes — any attribute outside the set is silently stripped. **Include
  `c-card`**, which is registered in code but missing from the current product
  docs and the developer `markdown_content.md` resource (code is the source of
  truth).
- **Admonition types are deployment-specific**: `widget-reference` documents the
  base `c-admonition` types but states that the valid set for *this* project comes
  from `.fls-content.yaml` (see §5), and that unknown types fall back silently to
  `default`. It never presents a hardcoded list as exhaustive.
- **HTML-escaping rules**: `c-code-block` and `c-equation` bodies are **not**
  markdown-processed; authors must escape `<`, `>`, `&` (and `"` for code blocks).
- **Authoring quirks** worth one line each: `c-image-grid` requires the closed
  `<c-picture ...></c-picture>` form with blank lines between children; flashcard
  slots need blank lines so markdown block parsing works; `c-accordion`'s bare
  `open` attribute.
- **Heading hierarchy in topic bodies**: the topic's `title` lives in frontmatter
  and is rendered as the page H1 — **do not repeat it as a heading in the body**.
  Body headings start at `#` (H1), which `mdx_headdown` renders as H2 beneath the
  title; nest sub-sections with `##`, `###`, …, without skipping levels.
- **Numbering**: `NN. name` prefix, alphabetical sort, zero-pad past 9 items;
  `course.md`/`form.md`/`part.yaml` are unnumbered single files that identify
  their directory's role; files/dirs starting with `_` or `.`, and `README.md`/
  `CLAUDE.md`, are skipped by the scanner.
- **UUID rules** (also in `conventions`): omit `uuid` on new content; never edit,
  duplicate, or hand-create one; `content_save` writes them back on first run.

Skills reference each other by namespaced prose mention (e.g. "See the
`fls-content:widget-reference` skill"). Each `resources/` file carries a top
comment recording the FLS source file(s) it was distilled from, so D4 updates are
mechanical.

**Depth discipline (load-bearing).** The audience is course authors, not FLS
developers. Skills state the syntax/attributes an author types, when to use a
widget, and what to avoid — **not** rendering-pipeline internals, schema class
hierarchy, or settings plumbing. Prefer one good example over a reference essay.
Match the existing plugin's depth and voice.

## 7. D2 — Conversion command and agent

`/fls-content:format-content <file-or-directory>` converts messy Markdown into
FLS content structure. It is a command (not a skill) because it is an explicit,
user-invoked, multi-step action that takes an argument and benefits from fan-out.

### Orchestration

1. Enumerate the input source files: a single `.md`, or all eligible content
   files under a directory — both Markdown (`.md`) **and pure-YAML files**
   (`.yaml`/`.yml`, e.g. an existing `part.yaml`). The same skip rules as the FLS
   scanner apply (ignore names starting with `_` or `.`, plus `README.md`/
   `CLAUDE.md`), so `.fls-content.yaml` and a prior `_conversion_review.md` are
   never treated as input. Pure-YAML role files are enumerated so the converter
   sees the full existing structure (COURSE_PARTs, role detection, naming). They
   are not re-converted as prose, but they are still checked for correctness — their
   YAML fields, naming, and placement — and fixed where wrong (see *Idempotent*
   below).
2. **Fan out one `content-formatter` agent per source file** — each agent loads
   the `markdown-conversion` skill and converts exactly one file in its own
   focused context, so a directory converts in parallel.
3. Collect only the items that need the author's attention into a single
   `_conversion_review.md` written alongside the output — and write nothing if there
   are none (see below). Routine, lossless changes are left to the git diff.

### Conversion behaviour

- **Conservative by default.** Auto-apply only unambiguous, lossless transforms:
  - local image `![alt](path)` → `<c-picture src alt title>` (derive `title` from
    surrounding caption; never invent);
  - Obsidian `![[image]]` / `![[image | title]]` → `<c-picture>` — **expanded
    during conversion, not left for `content_save`**, because `content_save`'s
    `![[...]]` translation **drops alt text**; expanding here lets us set `alt`
    from any caption/title, or flag it when none exists;
  - YouTube watch URL → `<c-youtube video_id>`.
- **Everything semantic is proposed, never applied silently**: blockquote →
  admonition/pull-quote, Q&A → flashcard, "optional reading" → accordion, math →
  equation, image sequence → image-grid. When in doubt, leave it as plain
  Markdown. (See `research_markdown_conversion.md` §2 for the full
  auto/propose/never table.)
- **Alt text**: when no caption/title is available, emit `alt=""` and **flag it in
  the review file** — never fabricate alt text and never emit a misleading
  placeholder.
- **Admonition types from config**: any proposed blockquote/callout →
  `c-admonition` uses only a `type` declared in `.fls-content.yaml` (or the base
  set on fallback; see §5). An intended type not in that set is flagged in the
  review file, never emitted as a silently-wrong (`default`-rendering) value.
- **Splitting**: split a source document at H1 (or H2 if it has no H1). Derive
  `title`/`subtitle`/`description` strictly from existing heading and intro-
  paragraph text — never paraphrase. Single file → one or more TOPICs; directory →
  a COURSE, with subdirectories becoming COURSE_PARTs.
- **Naming and layout (load-bearing).** Output directories and files must follow
  the FLS conventions exactly so `content_save` scans them in the right order and
  with the right roles: the `NN. name` numbered prefix (alphabetical sort, zero-
  padded once a directory passes 9 ordered items) for ordered content, and the
  unnumbered role-identifying files `course.md`, `form.md`, and `part.yaml` for
  their directories. When converting an existing directory, any file or folder
  that is mis-named for its role — a wrong or absent number prefix, or a role file
  under the wrong name — is renamed to the correct form (the rename shows up in the
  git diff; it is not separately logged). Never silently leave a mis-named file that
  `content_save` would mis-scan or skip. (See the `conventions` skill / §6 for the
  numbering and role-file rules.)
- **Heading handling (load-bearing).** The heading text is structure, not prose —
  re-levelling and de-duplicating headings is a structural transform, never a
  content change.
  - **Never duplicate the title in the body.** The heading promoted to the
    frontmatter `title` (and any heading used for `subtitle`) is **removed** from
    the body. The page template renders the frontmatter `title` as the visible
    H1, so repeating it in the body would produce two titles.
  - **Re-base the body heading hierarchy to start at the right level.** FLS
    applies `mdx_headdown` at render, shifting every body heading down one level
    (body `#` → rendered H2, body `##` → rendered H3, …) so it nests beneath the
    page-title H1. After the title heading is lifted out, **promote the remaining
    body headings so the topmost section is `#` (H1)**, which renders as H2 under
    the title — preserving relative nesting. Only the `#` markers change; heading
    **text is never altered**.
  - **Warn, don't rewrite, on skipped levels.** If the source skips a heading
    level (e.g. H1 → H3 with no H2), flag it in `_conversion_review.md` with the
    source line; do not silently insert or renumber headings, as that risks
    changing meaning.
  - Normalise setext headings (`===`/`---` underlines) to ATX (`#`) before any of
    the above — purely structural.
- **Existing widgets (load-bearing).** Source content may already contain `c-*`
  widgets — hand-written, pasted from another deployment, or produced by an older
  tool — and they are frequently subtly wrong. Treat each existing widget as
  structure to validate against the `widget-reference` allowlist and its documented
  attributes/forms, not prose to leave alone:
  - A widget that already matches the correct name, attribute set, and required
    form is **left untouched**.
  - A widget whose form is unambiguously and losslessly fixable is **normalised to
    the correct form** (the fix shows up in the git diff; it is not separately
    logged): an attribute outside
    the permitted set (which the sanitiser would silently strip) corrected or
    dropped, a `c-image-grid` child rewritten to the required closed
    `<c-picture ...></c-picture>` form, missing blank lines inserted so block
    parsing works, the bare `c-accordion open` attribute fixed, and `c-code-block`/
    `c-equation` bodies escaped per the HTML-escaping rules.
  - A widget that is **not in the allowlist** (an invented or unknown `c-*` name the
    sanitiser would strip wholesale) is **flagged**, never emitted as-is. If it maps
    unambiguously to a real widget the mapping is proposed in the review file; if
    not, it is flagged for the author with its source line.
  - A `c-admonition type` outside the active set is handled exactly as a proposed
    admonition (above): flagged, never emitted as a silently-wrong `default`-
    rendering value.
  - **Only widget syntax, attributes, and structure change — never the prose inside
    a widget.** Normalising an existing widget is a structural transform, subject to
    the same prose-preservation check as every other change.
- **Idempotent — check everything, fix only what is broken.** Nothing is skipped
  wholesale. The presence of `content_type:` frontmatter (or a pure-YAML role file
  like `course.md`/`form.md`/`part.yaml`) does **not** mean the file is correct, so
  every file — Markdown and pure-YAML alike — has its frontmatter, widgets, naming,
  and placement checked against the conventions and the bundled schema. Anything
  already correct needs no edit and gets none; anything wrong (a bad/missing
  frontmatter field, a malformed or out-of-set widget, a mis-named or mis-placed
  file) is corrected in place — the git diff is the record — and only flagged in
  `_conversion_review.md` if it needs an author decision. This is
  naturally idempotent: re-running on already-correct content makes no changes.
  Two things are never touched: an existing `uuid:`, and prose that is already
  valid — do not re-split, re-paraphrase, or rewrite already-structured prose; only
  broken structure and syntax are fixed. If it isn't broken, don't fix it. Safe to
  re-run on partially converted content.
- **Frontmatter safety**: quote all generated string values; merge (don't clobber)
  any pre-existing YAML frontmatter; normalise setext headings to ATX before
  splitting (structural only). (See `research_markdown_conversion.md` §5.)

### Why in-place, no dry-run

Content lives in **git**, so the author already has a full before/after diff and
one-command rollback. The conversion therefore writes in place and does **not**
add a dry-run default or contrive its own backup/versioning. (This deliberately
overrides the generic "dry-run by default" CMS-migration pattern from research,
because git already provides the safety that pattern is reaching for.)

### `_conversion_review.md` (only when something needs the author)

This is **not** a changelog. Git is the record of what changed — auto-applied,
lossless changes (image syntax, renames, widget normalisation, heading re-levelling)
show up in the diff and are **not** listed anywhere. The review file exists for one
purpose: to set the human up with what needs **their** attention — decisions to
make, ambiguities, and anything the converter could not safely resolve. **If a
conversion has nothing to flag, no review file is written.**

When written, it lists only attention-worthy items, with source-line references:

- proposed-but-not-applied semantic conversions awaiting a yes/no (blockquote →
  admonition, Q&A → flashcard, "optional reading" → accordion, math → equation,
  image sequence → image-grid);
- things the converter could not resolve: missing alt text, an unknown widget or an
  out-of-set `c-admonition type`, a skipped heading level, unresolved remote URLs,
  links whose target is outside the converted set;
- any concern that substantive prose may have been lost or added — a clean
  conversion needs no note; **only a possible discrepancy is reported.**

The author resolves the flagged items, reviews the full change set via the git
diff, then **deletes the file** before running `content_save`.

## 8. D3 — Bundled standalone validator

Authors without the FLS source still need the strict structural check that
`content_validate` runs. Ship a trimmed copy of the FLS validator inside the
plugin (`fls-content-plugin/validate/`).

- It is a **copy of FLS `schema.py` + `validate.py`**, which at module level
  already depend only on `frontmatter`, `yaml`, and `pydantic` — no Django ORM, no
  database.
- The only Django-dependent piece, `Course._validate_icon_fields` (which calls
  `freedom_ls.content_engine.icon_validation`), is **stubbed/skipped** in the
  bundled copy. (It is a deferred import *inside* that method in the source, so the
  rest of the module imports cleanly without Django.)
- **A thin standalone wrapper is required — it is not a pure verbatim copy.** The
  source `validate.py` cannot be run as a script as-is: it has no
  `if __name__ == "__main__"` entry point (FLS only ever invokes it via the
  `content_validate` management command — its docstring claiming
  `python validate.py …` is stale), and it imports the schema with a
  package-relative `from .schema import SCHEMAS`, which fails when run directly.
  The bundled copy therefore adds a small standalone CLI shim — a `__main__` entry
  point that takes a path argument, prints the same human-readable failures, and
  returns a non-zero exit code on error — and resolves the schema import so it runs
  outside the FLS package (e.g. a same-directory import, or invocation as a module).
  This wrapper is plugin code, not FLS source, and must be **re-applied on every D4
  re-sync** (§10).
- Technical authors can run it via `uv` (never `python`/`python3` directly), e.g.
  `uv run --with pydantic --with pyyaml --with python-frontmatter python fls-content-plugin/validate/validate.py <path>`,
  and get the same required-field, unknown-field (`extra="forbid"`), and
  type checks `content_validate` performs — offline.
- **A thin slash command wraps it for non-technical authors.**
  `/fls-content:validate-content [target]` runs the bundled validator for the author —
  they never need to know Python, `uv`, or the validator's path. It defaults the target to
  the current directory, runs the script through `uv` (which supplies the deps ephemerally —
  never a bare `python`/`python3`), and reports
  the result in plain, actionable language (which file, what is wrong) rather than a raw
  traceback, restating that this is a structural pre-flight and `content_save` remains
  authoritative. It adds **no** validation logic of its own — it only invokes the D3 script,
  so D4 re-sync never has to touch it.
- It is a **structural pre-flight check only.** The authoritative pass is still
  `content_save` on an FLS host (UUID assignment, icon resolution, cross-reference
  resolution, asset upload). The plugin documents this boundary clearly.

Because it is copied FLS code, it **will drift** unless maintained — which is
exactly what D4 keeps current.

## 9. Hard constraints (apply to all deliverables)

1. **Never change the substantive content of a course.** Conversion is a
   structural/format transform only — no rewriting, summarising, or inventing
   prose. Frontmatter values are derived from existing heading/paragraph text,
   never fabricated. The converter checks source-vs-output prose completeness and
   flags any possible discrepancy for the author rather than letting loss go unseen.
2. **Never invent or edit UUIDs.** New content omits the `uuid` field entirely;
   existing UUIDs are never altered or duplicated. `content_save` owns UUIDs.
3. **Self-contained and literal.** Every authoring fact lives in the plugin, not
   by reference to FLS source.
4. **Right altitude.** Author-facing, task-oriented, shallow. Resist dumping
   exhaustive developer documentation — both when first building and when D4
   updates the plugin.

## 10. D4 — Keeping the plugin in sync with FLS

When FLS course-authoring functionality changes (new/changed widgets,
frontmatter, content types, conventions, or the validator's source), the plugin
must be updated to match. This is wired into the **SDD workflow itself** as a
lightweight built-in check, not a separate pipeline or external manual checklist.

### Detection heuristic (cheap, zero-token)

A single `git diff main --name-only | grep -qE …` over authoring-relevant paths:

```
freedom_ls/content_engine/schema.py
freedom_ls/content_engine/validate.py
freedom_ls/content_engine/templates/cotton/*.html
config/settings_base.py                 # MARKDOWN_ALLOWED_TAGS, ADMONITION_TYPES
freedom_ls/content_engine/management/commands/content_save.py
freedom_ls/content_engine/management/commands/content_validate.py
demo_content/**
```

No match → tick the box and exit (no LLM, no fan-out). Match → proceed.

### Update action (on a match)

Fan out **one worker** to read only the changed authoring-relevant files, identify
exactly what changed, and draft scoped edits; a depth-0 synthesis step applies
them. It covers **both** targets:

- the author-facing **reference skills** (kept shallow), and
- the **bundled validator** (D3) — when `schema.py`/`validate.py` change, re-sync
  the trimmed copy, re-applying both the Django stub *and* the standalone CLI
  wrapper/import fix (§8), so it never drifts from the real schema and stays
  runnable as a script.

It must touch **only** the affected sections — not rewrite the whole plugin, and
not add implementation detail beyond what the source files express.

### Wiring

- New command file: `fls-claude-plugin/commands/sdd/update_claude_plugin_fls_content.md`.
  The name is explicit about which plugin it syncs, so it stays unambiguous if
  more author-facing plugins are added later.
- New `todo.md` template item near the "product documentation" stage in
  `protected/setup_todo_list.md`, e.g.:
  `- [ ] (cmd) Run `/update_claude_plugin_fls_content` to sync the course-author plugin if authoring functionality changed`
- A matching entry in `fls-claude-plugin/commands/sdd/README.md`.

The source of truth for these updates is the FLS code (schema, cotton templates,
settings), with `demo_content/` as living convention examples — but the output is
the author-facing *distillation*, never a copy of their detail.

## 11. Key decisions and rationale

| Decision | Why |
|---|---|
| Reference material as **skills**, conversion as a **command** | Skills are passive, always-available, and reusable inside the converter agent (subagents can't invoke slash commands); the command is an explicit, argument-taking, fan-out action. |
| Separate `content-types` and `widget-reference` skills | They're queried for different tasks; keeping them separate keeps each trigger precise and each body lean. |
| **Expand** `![[...]]` during conversion | `content_save`'s built-in translation drops alt text; expanding here lets us set or flag `alt`. |
| **No dry-run; convert in place** | Content is in git — the diff and rollback already make the change safe to inspect and revert. |
| **Bundle** a trimmed validator rather than document-only | Authors have no FLS source; offline structural validation is the only way to close the loop before `content_save`. Drift risk is accepted and mitigated by D4. |
| Sync via an **SDD checklist item + detection heuristic**, not a separate doc pipeline | Parity with every other workflow step: self-ticking, skippable, visible; fast-skip keeps non-authoring runs nearly free. |
| New plugin namespace **`fls-content`** | Avoids collision with the existing developer `fls:` plugin. |
| **`c-card` documented** despite being absent from current docs | It is registered in `MARKDOWN_ALLOWED_TAGS` with a template; code is the source of truth. |
| **Admonition types read from a content-repo `.fls-content.yaml`**, not hardcoded | `ADMONITION_TYPES` is extended per deployment, and unknown types fail silently; the plugin can only be correct per-project by reading the project's declared set, with the base set as fallback. |
| **`init` never overwrites an existing config** | Re-initialising must be safe; an author's customised `.fls-content.yaml` (and all content) must survive a re-run untouched, so `init` only creates the file when absent. |

## 12. Success criteria

1. **Self-containment**: a Claude session with only `fls-content-plugin/` present
   (no FLS source) can answer "what frontmatter does a TOPIC/FORM/quiz need?",
   "how do I write an accordion / flashcard / code block?", and "what are the
   UUID rules?" correctly and completely.
2. **Allowlist completeness**: the documented widget set exactly matches
   `MARKDOWN_ALLOWED_TAGS` (including `c-card`); no widget or attribute is
   documented that the sanitiser would strip, and none registered is omitted.
3. **Conversion correctness**: converting a representative messy `.md` file and a
   messy directory produces a structure that passes the bundled validator and
   then `content_save` on an FLS host, with UUIDs assigned only by `content_save`.
4. **Content preservation**: a conversion that only re-wraps structure produces no
   substantive prose loss or addition and no `_conversion_review.md` prose note; any
   possible discrepancy is flagged for the author rather than passing unseen.
5. **Conservative behaviour**: semantic widget conversions are proposed in
   `_conversion_review.md`, not silently applied; ambiguous constructs are left as
   plain Markdown.
5b. **Heading handling**: converted topic bodies do not repeat the frontmatter
   `title` as a heading; the body heading hierarchy is re-based so the top section
   is `#` (rendering as H2 under the title) with relative nesting preserved and no
   heading text altered; skipped source heading levels are flagged, not rewritten.
5c. **Existing-widget handling**: converting content that already contains `c-*`
   widgets normalises malformed-but-fixable ones (stripped attributes, the
   `c-image-grid` closed-`c-picture` form, missing blank lines, unescaped
   code/equation bodies) to correct FLS syntax in place (the git diff is the
   record); widgets with an unknown name or an out-of-set `c-admonition type` are
   flagged in `_conversion_review.md`, never emitted as-is; widget prose is never
   altered.
6. **Idempotency**: re-running conversion on already-*correct* content (Markdown
   and pure-YAML alike) makes no changes and never touches existing `uuid:` values;
   no file is skipped unchecked merely for having `content_type:` frontmatter, and a
   file that is wrong is fixed even on a re-run.
7. **Offline validation parity**: the bundled validator flags the same
   required-field, unknown-field, and type errors as FLS `content_validate` for a
   set of deliberately broken samples, without Django installed.
8. **Sync fast-skip**: on an SDD run that touches no authoring-relevant files,
   `/update_claude_plugin_fls_content` detects this in one `git diff`, ticks its
   box, and makes no edits. On a run that does, it updates only the affected
   skill/validator sections.
9. **Right altitude**: skill bodies stay shallow and task-oriented (no
   pipeline/schema-internals deep-dives) and match the existing plugin's voice.
10. **Deployment-specific admonition types**: given a content repo with a
   `.fls-content.yaml` listing a custom type (e.g. `regulation`), the
   `widget-reference` skill and conversion treat that type as valid; given no
   config file, they fall back to the base set and say so; in neither case is a
   hardcoded list presented as exhaustive, and a proposed type outside the active
   set is flagged rather than silently emitted.
11. **Idempotent init**: `/fls-content:init` creates `.fls-content.yaml` with the
   base set when absent; run a second time (or against a repo with a customised
   config), it leaves the existing file and all content files byte-for-byte
   unchanged and reports rather than writing.
