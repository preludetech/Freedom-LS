# Implementation Plan: Make the SDD workflow commit and push often

Slug: `spec-dd-commit-often`

This plan implements the spec at `1. spec.md`. The work is almost entirely **plugin
tooling** — markdown slash-command files, one bash hook, one settings template, a new
subagent file, and READMEs. There is **no Django application code** and **no frontend**.
Therefore `uv run pytest` does not exercise most of this work; the load-bearing
verification is a dedicated bash test harness for the `security-guard.sh` git-safety scan
(see Batch 7) plus per-artifact manual-equivalent checks expressed as re-runnable scripts.

All file paths below are relative to the repo root
(`/home/sheena/workspace/lms/freedom-ls-worktrees/spec-dd-commit-often`).

## Shared command shape (the fixed template)

Per the spec, every non-exempt SDD command gains this shape. Steps 1, 4, 5, 6 are
mechanical and identical; only the `stage_prefix` differs.

```text
1. Pre-stage assertion   (clean tree, not on main/master)
2. Do the actual stage work (existing logic)
3. Update todo           (existing update_todo call)
4. Stage commit + push   (delegate to commit-helper subagent)
5. Verify the commit landed (git log -1, check SHA changed + prefix)
6. Post-stage assertion  (clean tree)
```

### Extract the two assertions into one shared snippet (DRY)

Steps 1 and 6 are pure bash and would otherwise be copy-pasted ~11 times. The spec
explicitly says: *"Steps 1 and 6 should be extracted into a shared helper so the bash logic
isn't duplicated."*

**Decision: ship one shell script in the plugin and have each command call it.**

Create `fls-claude-plugin/scripts/sdd_clean_tree_guard.sh`:

```bash
#!/bin/bash
# Usage: sdd_clean_tree_guard.sh pre   -> pre-stage gate (clean tree + not on main/master)
#        sdd_clean_tree_guard.sh post  -> post-stage gate (clean tree only)
set -euo pipefail

mode="${1:-pre}"

if [[ "$mode" == "pre" ]]; then
  branch="$(git branch --show-current)"
  if [[ "$branch" == "main" || "$branch" == "master" ]]; then
    echo "ABORT: refusing to run SDD command on $branch."
    exit 1
  fi
fi

if [[ -n "$(git status --porcelain)" ]]; then
  if [[ "$mode" == "pre" ]]; then
    echo "ABORT: working tree is dirty. The previous stage did not close cleanly."
    echo "Either commit (/fls:sdd:commit), stash, or revert the pending changes before running this command."
    # Surface any labelled mid-stage stash so the agent can offer to pop it.
    git stash list | grep -E 'wip:' || true
  else
    echo "ABORT: working tree is still dirty after the stage commit. Stage did not close cleanly."
  fi
  exit 1
fi
```

- `chmod +x` it (Batch 1).
- It must pass shellcheck (it is covered by the repo's shellcheck pre-commit hook).
- Each command's pre-stage step becomes: *"Run
  `${CLAUDE_PLUGIN_ROOT}/scripts/sdd_clean_tree_guard.sh pre`. If it exits non-zero, STOP —
  do not proceed. If it reported a `wip:` stash, surface it to the user and ask whether to
  pop it (mid-stage interruption protocol)."*
- Each command's post-stage step becomes: *"Run
  `${CLAUDE_PLUGIN_ROOT}/scripts/sdd_clean_tree_guard.sh post`. It must exit zero."*

> Using a script (not inline bash pasted into each `.md`) is what actually makes steps 1/6
> DRY and keeps them shellcheck-clean in one place. The command files reference the script
> by `${CLAUDE_PLUGIN_ROOT}` path, exactly like the existing hook scripts and
> `compress_screenshots.py` are referenced.

### Step 4/5/6: one shared `commit_stage` helper, invoked by a minimal snippet (DRY)

The commit/verify/post-guard wording must **not** be copy-pasted into every command. The
verbose orchestration lives in **one** file and each command invokes it with a three-line
snippet — exactly the way `protected/update_todo.md` is already invoked from every SDD
command. Only `stage_prefix` and the description differ between sites.

**New file:** `fls-claude-plugin/commands/sdd/protected/commit_stage.md` — a *protected*
helper command (a skill the SDD commands call; never run directly by the user, same
category as `update_todo`/`setup_todo_list`). It owns the full Step 4/5/6 contract:

```markdown
## Inputs
- `stage_prefix` (required), `short_description` (required).
- `slug` is inferred from the in-progress spec dir under `spec_dd/2. in progress/`
  (same self-location update_todo already does — not passed by the caller).

## Step A: Commit and push this stage
Delegate to the `commit-helper` subagent (via the Task/Agent tool), passing
`stage_prefix`, `short_description`, and the inferred `slug`. The subagent stages the
right files, composes the Conventional-Commits message, runs `uv run git commit`, handles
auto-fix hook rewrites, pushes with a normal fast-forward push, and prints
`git log -1 --format='%H %s'`.

## Step B: Verify the commit landed
Run `git log -1 --format=%H%n%s`. Confirm the SHA differs from the pre-stage SHA (a new
commit landed) AND the subject starts with `<stage_prefix>:`. If either check fails, STOP
and surface the `commit-helper` output — do not silently continue.

## Step C: Post-stage clean-tree assertion
Run `${CLAUDE_PLUGIN_ROOT}/scripts/sdd_clean_tree_guard.sh post`. It must exit zero.
```

**The only thing that repeats in each command** — a three-line snippet pasted after the
existing `update_todo` step:

```markdown
## Step N: Commit, verify, and close this stage
Invoke `/fls:sdd:protected:commit_stage` with:
- `stage_prefix`: `<PREFIX-FOR-THIS-COMMAND>`   e.g. `spec`
- `short_description`: one imperative line describing what this stage produced
```

This makes **three** things single-source — nothing else repeats:

| Concern | Single source | Per-command cost |
|---|---|---|
| pre/post clean-tree bash | `scripts/sdd_clean_tree_guard.sh` | one-line `pre` call (Step 1) |
| git mechanics (stage/message/commit/push) | `agents/commit-helper.md` | none (reached via commit_stage) |
| commit → verify → post-guard orchestration | `commands/sdd/protected/commit_stage.md` | three-line invoke snippet |

The pre-stage guard call goes in as the command's **Step 1** (before any existing step).
The `commit_stage` snippet goes in **after** the existing `update_todo` step, because the
`todo.md` tick must be part of the same diff the stage commit captures.

> `/fls:sdd:commit` (the user-facing wrapper) reuses the **same** `commit_stage` helper for
> its Step 4/5 instead of inlining the delegate+verify+post-guard wording — see that
> section.

## Agent topology / nesting constraint (read before wiring delegation)

The whole commit pipeline depends on a Claude Code rule that is invisible until it fails at
runtime. Verified against the official docs (May 2026):

- [sub-agents](https://code.claude.com/docs/en/sub-agents): *"Subagents cannot spawn other
  subagents. If your workflow requires nested delegation, use Skills or chain subagents from
  the main conversation."* The mechanism is enforced, not advisory — the same page lists the
  tools that are **"not available to subagents, even when listed in the `tools` field:
  `Agent`, `AskUserQuestion`, `EnterPlanMode`, `ExitPlanMode`, `ScheduleWakeup`,
  `WaitForMcpServers`"**. So a subagent cannot reach the `Agent` (Task) tool at all.
- [skills](https://code.claude.com/docs/en/skills): *"Custom commands have been merged into
  skills … both create `/deploy` and work the same way."* A skill/command runs **inline in
  the invoking loop by default**, but one with **`context: fork` runs as its own subagent**.

This dictates three hard rules for this work. **Our design — main loop chains down to a leaf
`commit-helper` — is exactly the "chain subagents from the main conversation" pattern the
docs recommend.**

1. **`commit-helper` must always be reached from the main loop, never from inside another
   subagent.** The legal topology is `main loop → commit-helper` (one level). The SDD stage
   commands (`/spec_from_spec`, `/plan_from_spec`, `/implement_plan`, …) and the
   `commit_stage` / `/fls:sdd:commit` wrappers are **slash commands**; run inline they
   execute in the main loop, so delegating to `commit-helper` is a legal `main → leaf` spawn.

   The trap: if any stage command is ever refactored to run its work *inside* a spawned
   worker subagent (e.g. `/implement_plan` batches executed by an `Agent(...)` call) and that
   worker tries to delegate to `commit-helper`, the inner spawn **fails** — the worker has no
   `Agent` tool (it is hard-excluded per the doc quote above). If a stage ever must run in a
   subagent, that subagent hands control back to the main loop to commit; it cannot spawn
   `commit-helper` itself.

2. **The commit-delegating commands must NOT set `context: fork`.** Because a forked
   skill/command *is* a subagent, marking `commit_stage` (or any SDD stage command that
   reaches `commit-helper`) as `context: fork` would turn it into a subagent and strip its
   `Agent` tool — the `commit-helper` delegation would then fail. These commands must run
   inline in the main loop. (This is the merged-commands-into-skills gotcha; the per-command
   delegation only works because the command body runs in the main conversation.)

3. **`commit-helper` stays self-contained — `tools: Bash, Read`.** It cannot use the `Agent`
   tool (hard-excluded), so it can never delegate onward. Whether a subagent can invoke an
   *inline* skill is **not documented** — so do not design `commit-helper` to depend on
   `/fls:sdd:protected:commit_stage`, `update_todo`, or any skill. Keeping it pure bash+Read
   sidesteps the undocumented behaviour entirely and is the conservative, verifiable choice.

The same applies to `qa-data-helper` and any other leaf subagent: they are leaves by design.
Keep delegation flat — orchestration lives in the slash-command layer (main loop, run
inline), mechanical work lives in leaf subagents.

## The `commit-helper` subagent

**File:** `fls-claude-plugin/agents/commit-helper.md`

Mirror the frontmatter shape of `agents/qa-data-helper.md`. Frontmatter:

```yaml
---
name: commit-helper
description: >-
  Mechanical commit-and-push helper for the SDD workflow. Stages files, composes a
  Conventional-Commits message, runs `uv run git commit`, handles auto-fixable
  pre-commit-hook rewrites, and pushes with a normal fast-forward push. Invoked by SDD
  slash commands and by /fls:sdd:commit. Does no SDD-stage logic and never force-pushes.
tools: Bash, Read
model: haiku
color: green
---
```

> **Model:** `haiku` — the spec calls for "the smallest model the harness supports for
> subagents". The existing agents use `model: opus`; the value is a free-form harness model
> alias, so `haiku` is the correct minimal selection here. If the harness rejects `haiku`,
> fall back to the smallest alias it accepts and note it; do not silently use `opus`.

**Inputs** (named arguments passed in the invocation prompt): `stage_prefix`,
`short_description`, `slug`.

**Body — the documented contract (pseudocode the agent follows):**

```text
1. Inspect:  git status --porcelain   and   git diff --stat
   - If there is nothing staged AND nothing unstaged -> return "NOTHING TO COMMIT" to caller. Do not create an empty commit.

2. Stage explicitly:  git add <each changed path>   (NEVER `git add .` / `git add -A`).
   - Enumerate paths from `git status --porcelain` and add them one by one.
   - Skip anything that looks like a secret/credential file (.env*, *.pem, *.key) and report it.

3. Compose the message:
   subject = "<stage_prefix>: <short_description>"   (imperative, <=72 chars)
   body:
     - one short paragraph of WHY (not what), wrapped at 72 cols
     - blank line
     - "Refs: spec_dd/2. in progress/<slug>/1. spec.md"
     - blank line
     - the Co-Authored-By trailer per the harness commit guidance
   (The trailer text follows whatever the harness commit guidance specifies at commit
    time. Do NOT hardcode a model version into the agent file — read it from guidance.)

4. Commit with the HEREDOC pattern (so multi-line body + quotes are safe):
     uv run git commit -m "$(cat <<'EOF'
     <subject>

     <body>
     EOF
     )"

5. HOOK-FAILURE PROTOCOL (commit exited non-zero => NOTHING was committed):
   a. Read the hook output.
   b. AUTO-FIXABLE REWRITE case (ruff-check --fix changed files, ruff-format reformatted,
      trailing-whitespace / end-of-file-fixer rewrote files): `git status` now shows
      modified files. Re-stage exactly those paths and retry the commit ONCE.
      This is the ONLY case the subagent retries on its own.
   c. REAL FAILURE case (mypy error, bandit/detect-secrets/shellcheck failure, ruff rule
      with no autofix, failing test): STOP. Do NOT `--amend`. Return the structured hook
      output to the caller (main agent) so it can fix the source and make a NEW commit.
   d. LOOP LIMIT: if 3 consecutive `uv run git commit` attempts fail with the same root
      cause, STOP and surface the failure. Never keep retrying.
   e. NEVER `--no-verify` / `-n` / `--no-gpg-sign`. (Also blocked at the harness level.)

6. On a SUCCESSFUL commit, push (normal fast-forward, NEVER force):
     git push -u origin "$(git branch --show-current)"
   The `-u` is idempotent once upstream tracking is set. Per-stage commits are append-only
   relative to the worktree branch, so this always fast-forwards.

7. Print:  git log -1 --format='%H %s'   so the caller can verify.
```

**What it must NOT do (explicit in the agent body):**

- No `git rebase`, `git reset --hard`, or any destructive op.
- No force-push of any kind — not `--force`, not `--force-with-lease`.
- No knowledge of `todo.md` or the SDD workflow. It is a pure commit-and-push helper.
- No fixing of non-trivial hook failures — it returns those to the caller.

## `security-guard.sh` git-safety scan (LOAD-BEARING — highest risk)

Extend the **Bash branch** of `fls-claude-plugin/scripts/hooks/security-guard.sh` (the
`if [[ "$TOOL_NAME" == "Bash" ]]; then … fi` block) with a git-safety scan. **A prototype
of this exact logic was validated at plan time against all 19 rows of the spec's test
matrix (10 blocks, 9 allows) — see "Verified prototype" below. Implement that logic.**

Behaviour:

1. Strip single- and double-quoted substrings from the command first (so a forbidden flag
   *named inside* a commit message body is not a false positive).
2. If the (stripped) command contains a `git … commit` invocation, block when it contains:
   - `--no-verify`
   - `--no-gpg-sign`
   - `-c commit.gpgsign=false`
   - a short-flag cluster containing `n` (e.g. `-n`, `-nm`, `-an`) — but NOT `--no-edit`
     etc. (double-dash long flags must not trip the short-flag rule).
3. If the (stripped) command contains a `git … push` invocation:
   - If it contains `--force-with-lease` or `--force-if-includes` → ALLOW.
   - Else if it contains a standalone `--force` or `-f` → BLOCK.
4. `--force` / `-f` outside a `git push` (e.g. `git checkout --force`, `rm -f`) → ALLOW.
5. On block: `echo "BLOCKED: …"; exit 2` (the hook uses exit code 2 to deny, matching the
   existing `.env` / `rm -rf` blocks in the same file).

**Verified prototype** (drop this into the Bash branch, adapting the `BLOCKED`/`exit 2`
style of the surrounding code):

```bash
    # --- git safety: block hook-bypass and history-rewrite flags ---
    # Strip quoted strings so a flag NAMED inside a commit message isn't matched.
    STRIPPED=$(printf '%s' "$COMMAND" | sed -E "s/'[^']*'//g; s/\"[^\"]*\"//g")

    if printf '%s' "$STRIPPED" | grep -qE '(^|[[:space:]])git[[:space:]]+(.*[[:space:]])?commit([[:space:]]|$)'; then
        if printf '%s' "$STRIPPED" | grep -qE -- '--no-verify|--no-gpg-sign'; then
            echo "BLOCKED: git commit hook-bypass flags (--no-verify/--no-gpg-sign) are not allowed."
            exit 2
        fi
        if printf '%s' "$STRIPPED" | grep -qE -- '-c[[:space:]]+commit\.gpgsign=false'; then
            echo "BLOCKED: disabling commit signing via -c commit.gpgsign=false is not allowed."
            exit 2
        fi
        if printf '%s' "$STRIPPED" | grep -qE -- '(^|[[:space:]])-[a-zA-Z]*n[a-zA-Z]*([[:space:]]|$)'; then
            echo "BLOCKED: 'git commit -n' (skip hooks) is not allowed."
            exit 2
        fi
    fi

    if printf '%s' "$STRIPPED" | grep -qE '(^|[[:space:]])git[[:space:]]+push([[:space:]]|$)'; then
        if ! printf '%s' "$STRIPPED" | grep -qE -- '--force-with-lease|--force-if-includes'; then
            if printf '%s' "$STRIPPED" | grep -qE -- '(^|[[:space:]])--force([[:space:]]|$)|(^|[[:space:]])-f([[:space:]]|$)'; then
                echo "BLOCKED: plain 'git push --force/-f' is not allowed. Use --force-with-lease --force-if-includes."
                exit 2
            fi
        fi
    fi
```

> This block goes **inside** the existing `if [[ "$TOOL_NAME" == "Bash" ]]` branch,
> alongside the existing `rm -rf` / `.env` / `id_rsa` checks. The file already runs under
> `set -euo pipefail`; the `grep -qE … || true` idiom is not needed here because these are
> standalone `if` tests, not pipeline-fatal commands. The file is covered by shellcheck
> (pre-commit) — keep it shellcheck-clean.

Pseudocode for the squash-by-prefix step (Step 3):

```bash
base="$(git merge-base HEAD main)"
branch="$(git branch --show-current)"
git branch "${branch}-pre-squash"          # recovery ref

# keeping commit order and collapsing same-prefix runs/groups into one commit each.
```
