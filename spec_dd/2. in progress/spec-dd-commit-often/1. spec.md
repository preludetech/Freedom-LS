# Spec: Make the SDD workflow commit and push often

## Goal

Treat each SDD stage as an atomic, reviewable unit of work. One stage = one diff = one commit, pushed to the remote before the next stage starts.

This replaces today's pattern (3 commits across an entire SDD run, no pushes) with a pattern where every slash command leaves the working tree clean and the remote up-to-date.

## Why this matters

Today the SDD flow only commits at:

1. `protected/start_worktree.md` (when the worktree is created)
2. End of `implement_plan.md` (after every batch is done)
3. Inside `finish_worktree.md`

Nothing pushes. The consequences are real:

- A long session produces a huge un-bisectable diff before the first checkpoint.
- Work between checkpoints disappears if the worktree machine dies.
- `git revert` and `git reset --hard HEAD` can't be used to undo a single drift.
- A stage's edits can leak silently into the next stage with no clear boundary.

The research notes accompanying the idea (see `research_commit_best_practices.md`, `research_pitfalls_and_ux.md`, `research_reference_implementations.md`) confirm the converged industry pattern: per-stage commits, push as backup, conventional-commits prefixes, hard tooling-level prevention of `--no-verify`.

## Scope

### In scope

1. **Add a stage commit + push step to every SDD slash command**, using a conventional-commits stage prefix.
2. **Add pre- and post-stage clean-tree assertions** to every SDD slash command.
3. **Define and enforce the hook-failure protocol** when `uv run git commit` exits non-zero.
4. **Add harness-level deny rules** in `.claude/settings.json` to block `--no-verify`, `-n`, `--no-gpg-sign`, and command-chaining variants on `git commit`.
5. **Create a `commit-helper` subagent** that runs the mechanical commit/push steps, keeping diff output out of the main agent's context.
6. **Upgrade `finish_worktree.md`** to squash by stage prefix, enable `git rerere`, and use `git push --force-with-lease --force-if-includes` (never plain `--force`).
7. **Document the mid-stage interruption protocol**: agents stash, they do not commit partial work.
8. **Document recommended server-side backstops** (branch protection on `main`) without taking on the work of configuring them.
9. **Ship the deny rules and the flag-scan enforcement inside the plugin** — `templates/settings.json` (declarative deny list) and `scripts/hooks/security-guard.sh` (load-bearing scan) — and update `/fls:init` so a freshly-initialised project is protected with no manual settings editing.
10. **Add a user-facing `/fls:sdd:commit` command** that wraps the `commit-helper` subagent for manual fix-up commits, and **remove the existing generic `/fls:commit` command**.

### Out of scope

- Server-side branch protection on `main` (a repo configuration concern; called out as a recommended backstop only).
- Secret-scanning hooks (gitleaks, detect-secrets, etc.) — separate hardening initiative.
- Opening intermediate / draft PRs as the worktree progresses. None of the surveyed tools do this; PR creation stays at the `ship` step.
- Per-edit auto-commit (the Aider model). Surveyed users complain about this loudly; we explicitly reject it.
- Sub-commits within a single implementation batch. Defer until real batches show the pain. The hook-failure protocol must still cope if a batch produces an unwieldy diff, but no automatic split is required at this spec.

## Stage → commit map

Every SDD slash command must end with a stage commit + push. Conventional-commits prefixes make squash groups identifiable later.

| Slash command | Stage prefix | Commit subject pattern |
|---|---|---|
| `protected/start_worktree.md` | (bootstrap; see below) | (bootstrap; see below) |
| `improve_idea.md` | `docs(sdd):` | `docs(sdd): refine idea for <slug>` |
| `spec_from_idea.md` | `spec:` | `spec: write initial spec for <slug>` |
| `spec_review.md` | `spec-review:` | `spec-review: address review for <slug>` |
| `/threat-model` (external) | `spec:` | (out of our control; recommend `spec: threat-model <slug>` if user runs it) |
| `plan_from_spec.md` | `plan:` | `plan: write implementation plan for <slug>` |
| `plan_security_review.md` | `plan-review:` | `plan-review: apply plan security review` |
| `plan_structure_review.md` | `plan-review:` | `plan-review: apply plan structure review` |
| `implement_plan.md` (per batch) | `impl(<N>):` | `impl(2): <what the batch did>` |
| `/security-review` (external) | (no commit; runs against existing diff) | n/a |
| User fix-ups after a review cmd | (inherit surrounding stage; see below) | (inherit surrounding stage; see below) |
| `do_qa.md` | `qa:` | `qa: run QA plan for <slug>` |
| QA bug fixes (TDD) | `qa:` | `qa: <what the fix did>` |
| `/app_map` (when re-run after structural change) | `chore(sdd):` | `chore(sdd): refresh app_structure.md` |
| `address_pr_review.md` (per review round) | `fix(review):` | `fix(review): address PR feedback for <slug>` |
| `finish_worktree.md` | `chore(sdd):` | `chore(sdd): finalize <slug>` (after squash) |

Subjects are imperative, ≤72 chars (Conventional Commits + Tim Pope 50/72). Bodies wrap at 72, explain *why*, reference `Refs: spec_dd/2. in progress/<slug>/...`. The project's existing `Co-Authored-By: Claude … <noreply@anthropic.com>` trailer (as configured for the model in use — the repo history currently shows `Claude Opus 4.7 (1M context)`) is preserved. The spec does not pin a model version; the trailer follows whatever the harness commit guidance specifies at the time.

### User fix-up commits inherit the surrounding stage's prefix

Several SDD steps end with a user task to "address issues raised" by the preceding command. The commits the user produces during these tasks reuse the prefix of the surrounding stage so that the squash-by-prefix step in `finish_worktree.md` keeps each stage's work in one group:

- Spec edits after `/threat-model` → `spec:`
- Plan edits after `/plan_security_review` (when made *outside* the command's own commit) → `plan-review:`
- Plan edits after `/plan_structure_review` → `plan-review:`
- Code edits after `/security-review` → `impl(<next>):` (continuing implementation; pick the next batch number)
- QA bug fixes after `/do_qa` → `qa:` (already in the table; recorded here for symmetry)

The slash-command commits themselves are written by the `commit-helper` subagent. The user fix-up commits are made the same way — via the `/fls:sdd:commit` command (see below), which is a thin wrapper over the same `commit-helper` subagent — so they follow the same Conventional-Commits subject shape (`<prefix>: <imperative>`), the same `Refs:` body convention, and the same pre/post clean-tree gating as any other stage commit. The user never composes the message shape by hand.

### How a user commits their own changes: `/fls:sdd:commit`

User fix-ups must be as easy to commit as the automated stages — if committing by hand is fiddly, the "clean tree between stages" invariant rots and people reach for `git commit` directly (bypassing the prefix discipline and the gating). So there is **one** answer to "how do I commit my changes": run

```
/fls:sdd:commit
```

This is a **new** SDD-namespaced command (`fls-claude-plugin/commands/sdd/commit.md`). It is the single user-facing entry point to the same commit pipeline the automated SDD slash commands use. It must:

1. Run the relevant **pre-stage assertions**: refuse on `main`/`master`, and refuse if there is nothing to commit.
2. **Infer the stage prefix** for the current point in the workflow from `todo.md` (which stage is in progress / was last completed) and the inheritance rules in the table above (e.g. plan-review fix-ups → `plan-review:`, post-`/security-review` code → `impl(<next>):`). If the prefix cannot be inferred unambiguously, it asks the user to confirm or supply one rather than guessing.
3. Accept a one-line description as an inline argument (or prompt for it), then delegate to the **`commit-helper` subagent** with `stage_prefix`, `short_description`, and `slug` — exactly the contract the automated stages use. The helper owns the Conventional-Commits subject, the `Refs:` body, the `Co-Authored-By` trailer, the HEREDOC pattern, and `uv run git commit`.
4. **Verify** the commit landed (`git log -1`) and run the **post-stage clean-tree assertion**.

Power users can skip the prompt by passing the description inline: `/fls:sdd:commit address review feedback on the plan`.

The result is a commit indistinguishable from an automated stage commit — same prefix discipline, same gating, same deny-rule enforcement — so `finish_worktree.md`'s squash-by-prefix step treats user fix-ups and automated stages identically.

**The existing generic `/fls:commit` command (`fls-claude-plugin/commands/commit.md`) is removed.** Its ad-hoc staging + free-form message logic predates this workflow and would let a user produce un-prefixed, un-gated commits that break the squash-by-prefix invariant — exactly the drift this spec closes. Its useful safety checks (not on `main`, no secrets, individual `git add`) are not lost: the `commit-helper` already enforces them, and `/fls:sdd:commit` inherits them through it. Any documentation or README references to `/fls:commit` must be repointed to `/fls:sdd:commit`.

### `address_pr_review.md` commits after the squash

`address_pr_review.md` currently makes code changes but has **no commit step** — under this spec it gains one (the standard pre-stage / `commit-helper` / post-stage shape, prefix `fix(review):`). Because the workflow reorder runs `/finish_worktree` (and its squash) *before* the PR is opened, any `address_pr_review` commits land **after** the squash. They are therefore **not** part of the squash-by-prefix grouping; they stand as the PR's incremental review-response history. This is intended: a reviewer can see exactly what changed in response to each review round. The squash already produced the clean per-stage narrative; review responses are a legitimately separate phase on top of it.

### Why these prefixes

The squash-by-stage step in `finish_worktree.md` (see below) groups commits by prefix. A reviewer reading `git log` on `main` after merge sees a clear stage narrative: spec → plan → impl(1)…impl(N) → qa. Implementation batches keep their own commits because each batch is genuinely a separate logical unit; reviewers benefit from per-batch granularity even after squash.

### `protected/start_worktree.md` is exempt

`protected/start_worktree.md` is a **bootstrap** command: it runs *before* the worktree exists, on the user's current branch (typically `main`). It cannot honour the pre-stage branch assertion (there is no worktree branch yet) and its "move spec to in progress" commit necessarily lands on the user's starting branch, not on the worktree branch.

Therefore `start_worktree.md` is exempt from:

- the pre-stage branch assertion ("not on main/master"),
- the per-stage commit + push step (it keeps its current `git commit` semantics),
- the post-stage clean-tree assertion that other commands run after the stage commit.

It still uses `uv run git commit` and is still bound by the harness deny rules (no `--no-verify`, no plain `git push --force`). Every *other* SDD slash command — including the ones invoked from inside the worktree — obeys the full pre-stage / commit / post-stage shape.

## Clean-tree gating

Every SDD slash command runs the same two assertions, in pure bash, not the subagent:

**Pre-stage assertion** (first step of the command, before any work):

```bash
if [[ -n "$(git status --porcelain)" ]]; then
  echo "ABORT: working tree is dirty. The previous stage did not close cleanly."
  echo "Either commit, stash, or revert the pending changes before running this command."
  exit 1
fi
```

**Post-stage assertion** (after the stage commit + push):

```bash
if [[ -n "$(git status --porcelain)" ]]; then
  echo "ABORT: working tree is still dirty after the stage commit. Stage did not close cleanly."
  exit 1
fi
```

The pre-stage assertion is what makes "next stage's diff = `HEAD~1..HEAD`" hold for the whole workflow. The post-stage assertion catches the "agent silently moved on after a hook failure" failure mode documented in the research.

**Branch assertion** is added to the pre-stage block: refuse to run if the current branch is `main` or `master`.

```bash
branch="$(git branch --show-current)"
if [[ "$branch" == "main" || "$branch" == "master" ]]; then
  echo "ABORT: refusing to run SDD command on $branch."
  exit 1
fi
```

## Mid-stage interruption protocol

When the agent is interrupted mid-stage (user takes over, context switch, machine sleep), partial work is **stashed**, not committed:

```bash
git stash push -m "wip: <stage> <short description>"
```

The next time the relevant slash command runs, the agent's first move (after the pre-stage clean-tree assertion fails) is to surface the stash and let the user decide whether to pop it.

Rationale: the whole point of "commit at each stage" is that each commit is a logical unit. Mid-stage commits would defeat that. Stash is the right tool for "not ready to commit, must switch context".

## Hook-failure protocol

`uv run git commit` is mandatory; pre-commit hooks must run. Hook failures are common. The project's pre-commit (see `.pre-commit-config.yaml`) runs ruff-check `--fix`, ruff-format, the trailing-whitespace / end-of-file-fixer file fixers, mypy, bandit, detect-secrets, and shellcheck. The protocol is:

1. **`uv run git commit` exits non-zero ⇒ the commit did not happen.** Never `--amend` after a hook failure — there's nothing to amend the new work into. The previous commit is unrelated.
2. **Read the hook output** to identify the failure class:
   - **Auto-fixable rewrites** (ruff-check `--fix` applied a fix, ruff-format reformatted, or a file fixer like trailing-whitespace / end-of-file-fixer rewrote a file). The hook rewrote files; `git status` will show them as modified. Re-stage and retry — this is the only retry case the subagent is allowed to handle on its own.
   - **Real failure** (a mypy error, a bandit / detect-secrets / shellcheck failure, a failing test if the commit hook runs the suite, or a ruff rule with no auto-fix). The subagent stops and returns control to the main agent with the structured failure output. The main agent has the planning context to fix the source, re-stage, and create a **new** commit.
3. **`--no-verify` and equivalents are forbidden.** Tooling-level enforcement (see "Harness deny rules" below) makes this a hard barrier, not a guideline.
4. **Loop limit.** If three consecutive `uv run git commit` attempts fail with the same root cause, the subagent stops and surfaces the failure to the user. The agent does not keep retrying.
5. **Verification.** After every successful commit, the caller (slash command body) runs `git log -1 --format=%H%n%s` and confirms (a) the SHA changed, and (b) the subject starts with the expected stage prefix. This catches "subagent reported success but commit didn't actually land" silently.

## Harness deny rules

Add the following to `.claude/settings.json` `permissions.deny` (the existing entries `Bash(git commit --no-verify:*)` and `Bash(git push --force:*)` stay; we extend them):

- `Bash(git commit*--no-verify*)` — covers any position of the flag
- `Bash(git commit*-n*)` — covers the short form (must be a careful pattern; see Open Questions)
- `Bash(git commit*--no-gpg-sign*)`
- `Bash(git commit*-c commit.gpgsign=false*)`
- `Bash(git push --force *)` and `Bash(git push -f *)` — plain force-push to any branch is forbidden; only `--force-with-lease --force-if-includes` is allowed
- `Bash(git push --force)` (no trailing args, exact match)

These deny rules apply to **all** tool calls in the harness — main agent and subagents alike. The spec asserts this rather than relying on per-agent CLAUDE.md rules.

### Packaging and init propagation

The deny rules only protect a project if they actually land in that project's `.claude/settings.json`. FLS is a plugin **installed into other Django projects**, so the enforcement must travel with the plugin and be applied at init time, never hand-added per project. Two plugin assets carry it, and both must be updated by this work:

1. **`fls-claude-plugin/templates/settings.json`** — holds the declarative `permissions.deny` list. `/fls:init` (Step 1) already merges this template's `deny` entries into the target project's `.claude/settings.json`, adding any missing rules without duplicating existing ones. Adding the new rules here means every newly-initialised *and* re-initialised project picks them up automatically.

2. **`fls-claude-plugin/scripts/hooks/security-guard.sh`** — the `PreToolUse` (`Bash|Write|Edit`) hook already shipped by the plugin and registered in `fls-claude-plugin/hooks/hooks.json`. Because Claude Code's `Bash(...)` deny matching is prefix-style and cannot reliably catch a forbidden flag that appears *mid-command*, the load-bearing enforcement is a command-string scan added to this hook. The hook is owned by the plugin, runs for **every** agent and subagent, and cannot be edited away per project — it is the mechanism that actually satisfies Success Criterion 4. The declarative deny rules remain as a fast, visible permissions-layer block and as documentation of intent; the hook is the guarantee.

The hook scan must: strip quoted strings before matching (so a flag *named* inside a commit-message body is not a false positive); block `--no-verify`, `--no-gpg-sign`, and `-c commit.gpgsign=false`; block the `git commit -n` short form (including combined clusters such as `-nm`); and block plain `git push --force` / `-f` while allowing `--force-with-lease --force-if-includes`. Exact patterns are the plan's job.

`/fls:init` must be updated so that **Step 1** carries the new deny rules over and **Step 8 (validation)** confirms both that the deny rules are present in the target `.claude/settings.json` and that the `security-guard.sh` git-safety scan is active. The plugin's `templates/settings.json` is the single source of truth for the deny list; the FLS repo's own `.claude/settings.json` is regenerated from it like any other consumer (no hand-maintained drift).

> **Spec note:** the exact glob/regex form of these deny patterns must be tested at implementation time. Claude Code's deny matching is prefix-style for `Bash(...)`. Critically, this project commits via **`uv run git commit`** (a CLAUDE.md rule), so a pattern anchored on `git commit …` will not even match the actual command string — it starts with `uv run`. The `git commit` deny rules are therefore expected to be largely **decorative** for this project, and the `security-guard.sh` command-string scan — which matches on substrings regardless of the `uv run` prefix — is the mandatory, load-bearing enforcement, not an optional fallback. (Plain `git push --force` is not wrapped in `uv run`, so its deny rule can still fire; the hook covers it too for defence in depth.) The plan must validate matching behaviour before relying on any deny rule.

## The `commit-helper` subagent

A new subagent named `commit-helper` runs the mechanical commit pipeline so diff output stays out of the main agent's context.

**Lives at:** `fls-claude-plugin/agents/commit-helper.md`

**Inputs (from the calling slash command):**

- `stage_prefix` — e.g. `spec`, `plan`, `impl(2)`, `qa`, `chore(sdd)`
- `short_description` — one line, used to compose the subject
- `slug` — the spec directory slug, used for the body's `Refs:` line

**What it does:**

1. `git status` and `git diff --stat` to see what's about to be committed.
2. Stage the right files (`git add <paths>` — never `git add .` or `git add -A`).
3. Compose a subject (`<stage_prefix>: <short_description>`) and a body (why-not-what, ≤72 chars wrapping, `Refs:` line, `Co-Authored-By` trailer).
4. Run `uv run git commit -m "$(cat <<'EOF' ... EOF)"` (HEREDOC pattern from the project's commit-safety guidance).
5. On failure, follow the hook-failure protocol — auto-fix rewrite case is handled in-subagent, anything else returns structured "hook failed, here is the output" to the main agent.
6. On success, push to the worktree branch with a normal (non-force) push: `git push -u origin "$(git branch --show-current)"` (the `-u` is idempotent — a no-op for the upstream flag once tracking is set). The helper **never force-pushes**. Per-stage commits are append-only relative to the worktree branch, so this push always fast-forwards. The single place a force-push is required — `finish_worktree.md`, after the squash/rebase — does its own push directly and does not go through this helper.
7. Print `git log -1 --format='%H %s'` so the caller can verify.

**What it does not do:**

- It does not run `git rebase`, `git reset --hard`, or any destructive op.
- It does not force-push in any form — neither `--force` nor `--force-with-lease`. It only ever does a normal fast-forward push.
- It does not run any SDD-stage-specific logic. It does not know about `todo.md` or the workflow at all. It is a pure commit-and-push helper.
- It does not fix non-trivial hook failures.

**Model:** Haiku-class. The job is template-shaped and well within Haiku's capability. The model field in the agent file should select the smallest model the harness supports for subagents.

**Why a subagent at all (vs. inline bash in the slash command):** the diff output is the load-bearing reason. Even a moderate stage produces tens of KB of diff that the main agent doesn't need. Keeping that out of the main context buys real headroom on long workflows. The dollar cost is secondary. Tradeoff: the subagent adds a few seconds per stage and one extra failure mode (subagent reports success when commit didn't land). The post-call `git log -1` verification mitigates that.

## `finish_worktree.md` upgrade

`/finish_worktree` runs **before** the pull request is opened. The squash-by-stage-prefix step produces a clean per-stage history that becomes the PR's review surface. The SDD README and the generated `todo.md` template must be updated so that the "Open a pull request" step comes *after* `/finish_worktree`, not before. This is a deliberate departure from the current ordering.

> **Note on ordering change:** today the `todo.md` template has PR creation in step 10 and `/finish_worktree` in step 11. After this spec lands, finish-worktree comes first (squash + force-push-with-lease to set up the review history), then PR creation, then merge. Implementation must update both the README and the `setup_todo_list.md` template accordingly.

The current command does `git rebase main` then a final commit. After the upgrade it must:

1. **Pre-stage assertion** (clean tree, not on main).
2. **`git config rerere.enabled true`** in the worktree, if not already set. Cheap safety net for repeated rebases.
3. **Squash by stage prefix.** Walk the commit log on the worktree branch (the range `$(git merge-base HEAD main)..HEAD`) and group commits by their conventional-commits prefix. Within each group, squash to one commit, preserving the first commit's subject. Use `git rebase -i --autosquash` with `--fixup` markers, or an interactive rebase with a generated todo list. Keep a `<branch>-pre-squash` ref locally before the squash so the agent can recover if anything goes wrong.
4. **Rebase onto main.** Resolve any conflicts; `rerere` reuses prior resolutions if `main` moved during the worktree's life.
5. **Run unit tests** (existing step).
6. **Frontend QA confirmation** (existing step).
7. **`dev_db_delete.sh`** (existing step).
8. **Update todo, move spec to done, tidy project settings** (existing steps).
9. **Final stage commit** with prefix `chore(sdd):` covering any post-rebase tidy-ups.
10. **Push** with `git push --force-with-lease --force-if-includes`. Plain `--force` is denied at the harness level, so this is enforced.
11. **Post-stage assertion** (working tree clean).
12. **Optional remote branch cleanup** (out of scope for this spec — the existing finish_worktree leaves remote branch deletion to the user / GitHub auto-delete-after-merge setting).

The squash step preserves the per-stage narrative on `main` (one commit per stage, not one per checkpoint) while still leaving the worktree branch's full history available locally and on the remote until cleanup.

## Slash-command structure (template)

Every SDD slash command (except `protected/update_todo.md`) gains a fixed shape:

```text
1. Pre-stage assertion (clean tree, not on main, branch is the worktree branch)
2. Do the actual stage work (existing logic of the command)
3. Update todo (existing step)
4. Stage commit + push via `commit-helper` subagent
5. Verify the commit landed (git log -1)
6. Post-stage assertion (clean tree)
```

Steps 1, 4, 5, 6 are mechanical and identical across commands. Step 4 needs the right `stage_prefix` for that command (table above). Steps 1 and 6 should be extracted into a shared helper so the bash logic isn't duplicated 12 times — concrete shape is the plan's job, not the spec's.

Exceptions:

- `protected/update_todo.md` is invoked from inside other commands and runs as part of the same stage. It does not commit. Its caller commits.
- `protected/start_worktree.md` is a bootstrap command (see "`protected/start_worktree.md` is exempt" above) and follows its own simpler shape.
- `implement_plan.md` commits **once per batch** (it loops steps 4–5 of the shape over each batch), producing `impl(1)…impl(N)` rather than a single stage commit. The pre-stage assertion runs once at the start; the post-stage clean-tree assertion runs after the final batch.
- `finish_worktree.md` makes its own final commit **and** push directly (not via `commit-helper`), because the post-squash/post-rebase push requires `git push --force-with-lease --force-if-includes`, which `commit-helper` never performs (see "`finish_worktree.md` upgrade" and the `commit-helper` contract).

## Recommended backstops (out of scope but called out)

The spec calls these out so the project can configure them outside the SDD workflow:

- **Server-side branch protection on `main`.** Direct push blocked, PR + review required. The agent assertion on "not on main" is the local backstop; branch protection is the server-side one. Configure in GitHub repo settings.
- **`pre-commit` framework's `no-commit-to-branch` hook** as a final local backstop against accidental `main` commits. **Caveat:** `protected/start_worktree.md` deliberately commits the "move spec to in progress" change on the user's starting branch (typically `main`) *before* the worktree branch exists (verified: its Step 3 commit precedes Step 4's `git worktree add`). A blanket local `no-commit-to-branch` on `main` would block that bootstrap commit. If this backstop is adopted, it must exempt the bootstrap commit, or `start_worktree.md` must be restructured to create the branch before committing. (Server-side branch protection on `main` is unaffected — the bootstrap commit is local and never pushed directly to `main`.)
- **Secret-scanning** (gitleaks etc.) as a separate hardening initiative.

## Correctness and verification

The final result must ship with **no bugs**. Most of this work is *not* Python and is therefore invisible to `uv run pytest`: bash gating blocks, the `security-guard.sh` flag-scan, the `.claude/settings.json` deny patterns, the `commit-helper` subagent, and the squash-by-prefix logic. "It looked right" is not sufficient for any of them. Every artifact this spec touches must be verified before the work is considered done, and the verification itself must be a re-runnable artifact (a script or test), not a one-off manual check.

### The flag-scan test matrix (highest-risk; mandatory)

The `security-guard.sh` git-safety scan is the single most bug-prone piece: a regex that is too loose blocks legitimate commits (false positive) and a regex that is too tight lets a forbidden flag through (false negative — a silent security hole). It must be covered by an automated test that feeds each command below to the hook (as the real `PreToolUse` JSON payload) and asserts the exit code. The list is a **minimum**; the plan may add more.

**MUST be blocked (hook exits non-zero / denies):**

| Command | Why |
|---|---|
| `git commit --no-verify` | bypasses hooks |
| `uv run git commit -m "x" --no-verify` | flag after the message, real project prefix |
| `git commit -n` | short form |
| `git commit -nm "msg"` | combined short cluster |
| `git commit --no-gpg-sign` | disables signing |
| `git -c commit.gpgsign=false commit` | disables signing via config override |
| `git push --force` | plain force |
| `git push -f origin feature` | short force |
| `git push origin feature --force` | force flag in trailing position |
| `git status && git push --force` | chained after another command |

**MUST be allowed (hook exits zero):**

| Command | Why it must not trip the scan |
|---|---|
| `uv run git commit -m "fix: handle the -n flag in the parser"` | forbidden flag named *inside* the message body |
| `git commit -m "msg"` | ordinary commit |
| `git commit --amend` | allowed at the hook layer (the no-`--amend`-after-hook-failure rule is protocol, not a hook block) |
| `git commit --no-edit` | harmless flag containing no forbidden token |
| `git commit --dry-run` | harmless flag |
| `git push --force-with-lease --force-if-includes` | the one sanctioned force form |
| `git push -u origin feature` | first push, sets upstream |
| `git checkout --force somefile` | `--force` on a non-`push` subcommand |
| `rm -f scratch.txt` | `-f` outside `git push` |

A change to the scan regex that breaks any row of this matrix is a bug and blocks completion.

### Verification of the rest

- **Modified slash commands** — each is exercised on a throwaway worktree to confirm the fixed shape holds: pre-stage assertion aborts on a dirty tree and on `main`/`master`; the stage commit lands with the expected prefix; the post-stage assertion leaves the tree clean. A command that "ran" but left the tree dirty or committed with the wrong prefix is a bug.
- **`commit-helper` subagent** — both branches are verified: the auto-fixable-rewrite retry path (e.g. ruff reformats, helper re-stages and retries once) *and* the structured-failure return on a non-trivial hook error (helper stops, does not `--amend`, surfaces output). The three-consecutive-failure halt is verified.
- **`/fls:sdd:commit`** — verified to infer the correct prefix from `todo.md`, to refuse on `main` and on an empty tree, and to produce a commit indistinguishable from an automated stage commit.
- **`finish_worktree.md` squash** — verified on a branch with a realistic multi-stage history (spec → plan → impl(1) → impl(2) → qa): the squash collapses each prefix group to one commit, preserves the per-stage narrative, keeps the `<branch>-pre-squash` recovery ref, and the rebase+`--force-with-lease --force-if-includes` push succeeds. No commit is lost or reordered across stage boundaries.
- **`/fls:init` propagation** — verified by running init into a scratch project and confirming the new deny rules land in its `.claude/settings.json` and the `security-guard.sh` scan actually blocks a sample `--no-verify` there.
- **Existing suite stays green** — `uv run ruff check .`, `uv run mypy .`, and `uv run pytest` all pass; no `--no-verify` was used to get there (the whole point).

### Bug-free bar

The work is **not done** while any of the following is true: a matrix row fails; a slash command can finish on a dirty tree; the `commit-helper` can report success when no new commit landed (the post-call `git log -1` check is precisely the guard against this); `/fls:init` can leave a project without the deny rules or the active scan; or the pre-commit suite does not pass cleanly. Each of these is treated as a release-blocking defect, not a polish item.

## Success criteria

The spec is met when all of the following hold:

1. Every SDD slash command, when run on a clean worktree, ends with at least one new commit on the worktree branch and a push to the remote. All commands produce exactly one commit per run **except** `implement_plan.md`, which produces one commit per batch (`impl(1)…impl(N)`).
2. Every SDD slash command refuses to start if the working tree is dirty or the current branch is `main`/`master`.
3. Every per-stage commit message starts with the conventional-commits prefix listed in the stage-commit map and follows the 50/72 / Conventional Commits / `Co-Authored-By` shape.
4. `--no-verify`, `-n` (on `git commit`), `--no-gpg-sign`, `-c commit.gpgsign=false`, and plain `git push --force` cannot be invoked from any agent or subagent in the harness — they are blocked by `.claude/settings.json` deny rules (or a `PreToolUse` hook if pattern matching forces that).
5. The `commit-helper` subagent exists, is invoked from each per-stage SDD slash command (every command except `finish_worktree.md`, which makes its own final commit + force-push because of the post-rebase force requirement), and follows the documented contract (no destructive ops, no `--no-verify`, no force-push, returns structured failure on non-trivial hook errors).
6. After a hook failure, the agent never `--amend`s. Three consecutive failures with the same root cause halt the workflow with a surfaced error.
7. `finish_worktree.md` enables `rerere`, squashes per-stage commits into one commit per stage prefix before rebasing onto main, and uses `git push --force-with-lease --force-if-includes`.
8. Mid-stage interruption is handled by `git stash` (with a labelled message), not by a partial commit. The next slash command surfaces the stash if the pre-stage assertion fails.
9. The `git log` on the worktree branch reads as a clear stage narrative (`spec → plan → impl(1) → impl(2) → qa`) before the squash, and the same narrative shows up on `main` (one commit per stage prefix) after the squash.
10. A reviewer reading `git log main..<branch>` mid-workflow can see exactly which stages have completed without needing to read any other artefact.
11. The deny rules and the `security-guard.sh` git-safety scan ship **inside the plugin** (`templates/settings.json` + `scripts/hooks/security-guard.sh`), and a project freshly set up with `/fls:init` is protected by them with no manual `.claude/settings.json` editing. `/fls:init`'s validation step confirms both are present.
12. A user with manual changes can commit them as a correctly-prefixed, gated stage commit by running `/fls:sdd:commit` (optionally with an inline description), without hand-writing the Conventional-Commits shape or calling `uv run git commit` directly. The old generic `/fls:commit` command is removed and no documentation references it.
13. The result is bug-free per the "Correctness and verification" section: every row of the flag-scan test matrix passes as an automated test, every modified slash command / subagent / squash / init path is verified by a re-runnable artifact, and `ruff`/`mypy`/`pytest` all pass without any verification bypass. No item on the "bug-free bar" list holds.

## Open questions resolved at spec time

The idea listed three open questions; the spec resolves them as follows:

1. **Sub-commits within a single implementation batch.** Out of scope. The hook-failure protocol's loop limit catches truly enormous batches indirectly (a batch big enough to fail hooks repeatedly stops the workflow and surfaces). If real batches start producing reviewable-but-too-big commits, address it in a follow-up spec.
2. **Mid-stage interruption.** Stash with a labelled message, never commit partial work. Captured in the "Mid-stage interruption protocol" section above.
3. **Conflict resolution during `finish_worktree`'s rebase.** Enable `git rerere` in the worktree as part of `finish_worktree.md`. Squash-by-stage-prefix before rebase reduces the number of conflict rounds.

## Spec-time decisions worth recording

- **Subagent over inline bash.** The diff-out-of-context win is real on long workflows. The cost (latency + one extra failure mode) is bounded by the post-call `git log -1` verification.
- **Per-stage commits, not per-edit.** Per-edit (Aider model) is rejected on the basis of documented user pain.
- **Push every stage commit.** Worktrees are single-author and short-lived; force-push hazard is contained. Backup and reviewability win.
- **Conventional Commits prefixes are load-bearing.** The squash-by-prefix step in `finish_worktree.md` depends on them.
- **Harness-level enforcement of `--no-verify` denial, not memory rules.** The research documents at least one incident (Claude Code issue #40117, six broken commits with 104 failing tests) where the agent bypassed memory rules. Tooling enforcement is the only reliable answer.
- **`finish_worktree.md` is the only command that uses force-push.** Any per-stage commit that needs to overwrite remote history is a bug — per-stage commits are append-only relative to the worktree branch. Only the post-rebase, post-squash push needs `--force-with-lease`.
