# Spec: Replace Font Awesome with Heroicons

## Overview

Replace Font Awesome (loaded via CDN) with Heroicons (by Tailwind Labs) using Adam Johnson's `heroicons` Python package, with a custom semantic icon template tag providing a central registry for all icon usage.

### Why this matters

- **Privacy & reliability**: Removes third-party CDN dependency (cdnjs.cloudflare.com)
- **Performance**: Eliminates ~100KB CSS download for only 28 icons used
- **Consistency**: Unifies three different icon systems (Font Awesome `<i>` tags, inline SVGs, Unicode characters) into one
- **Maintainability**: Central registry means changing an icon's meaning requires editing one line, not hunting through templates

---

## Architecture

### Icon package: `heroicons` by Adam Johnson

- **PyPI**: `heroicons[django]`
- **Why**: Made by Tailwind Labs (visual harmony with TailwindCSS stack), MIT license, excellent Django support via Adam Johnson's package (supports Django up to 6.0, Python up to 3.14), renders inline SVGs styled with Tailwind classes
- **Why not cotton-icons**: The cotton-icons package has an open issue (#4) where it fails when django-cotton's template settings are customized. Adam Johnson's package is more mature and maintained by a Django core contributor.

### Custom `{% icon %}` template tag

A custom template tag that provides semantic icon lookup and renders heroicons:

```html
{% load icons %}

{% icon "next" class="size-5 text-blue-500" %}
{% icon "success" variant="solid" class="size-6" %}
```

**How it works internally:**

1. Receives a name (e.g., `"next"`)
2. Looks up the name in the central `ICONS` registry dict
3. If found, uses the mapped heroicon name (e.g., `"arrow-right"`)
4. If not found, raises a `KeyError` â€” all icons must be defined in the registry to prevent typos and ensure consistency
5. If `force=True` is passed, skips the registry lookup and uses the name as a literal heroicon name (escape hatch for one-off cases: `{% icon "arrow-right" force=True %}`)
6. Renders the SVG using the `heroicons` package's rendering functions

**Parameters:**

| Parameter | Default | Description |
|-----------|---------|-------------|
| `name` | (required) | Semantic name from registry (or direct heroicon name when `force=True`) |
| `variant` | `"outline"` | Heroicon variant: `outline`, `solid`, `mini`, `micro` |
| `class` | `"size-5"` | CSS classes (Tailwind sizing, color, animation) |
| `force` | `False` | If `True`, skip registry lookup and use name as a literal heroicon name |
| `aria_label` | `None` | If set, renders `role="img"` + `<title>` instead of `aria-hidden="true"` |

**Why a template tag instead of a Cotton component**: Cotton components cannot dynamically dispatch to other components by variable name. A template tag provides full dynamic lookup capability while keeping the API clean.

### Variant strategy

- **Default variant**: `outline` (stroke-based, 24px design)
- Templates override per-use with `variant="solid"` etc. when needed
- The registry maps names to icon names only, not variants
- **Why**: Keeps the registry simple. Variant choice is a presentation concern that varies by context (e.g., a status icon might be `outline` in a list but `solid` in a badge).

## Accessibility

All icons default to **decorative** (`aria-hidden="true"`) because most icons in FLS accompany visible text.

For **informative/standalone icons** (icon is the only content conveying meaning), the `{% icon %}` tag accepts an `aria_label` parameter:

```html
<!-- Decorative (default) â€” icon accompanies text -->
<button>{% icon "check" class="size-5 mr-2" %} Mark Complete</button>
<!-- Renders: <svg aria-hidden="true" ...> -->

<!-- Informative â€” icon-only button -->
<button aria-label="Close dialog">{% icon "close" class="size-5" %}</button>
<!-- Icon is still aria-hidden; button's aria-label provides the name -->

<!-- Standalone informative â€” icon conveys status without text or interactive parent -->
{% icon "success" aria_label="Completed" class="size-5" %}
<!-- Renders: <svg role="img" aria-labelledby="..."><title>Completed</title>...</svg> -->
```

**Why this approach**: Following the Smashing Magazine accessible SVG patterns research â€” `role="img"` with `<title>` and `aria-labelledby` has the broadest screen reader compatibility.

## CI regression test

Write a test that greps the entire template directory for leftover Font Awesome patterns to prevent regression:

**Patterns to check for:**
- `fa-` (FA icon class prefix)
- `fa ` (old FA class prefix)
- `fas ` / `far ` / `fab ` / `fal ` (FA style prefixes)
- `fa-solid` / `fa-regular` / `fa-brands` (FA 6 style prefixes)
- `font-awesome` (CDN URL or references)
- `cdnjs.cloudflare.com` (CDN URL)

Also check for leftover inline SVG icons:
- `<svg` elements in templates that contain icon paths (e.g., `viewBox="0 0 24 24"` with `<path` children) â€” all standard icons should use `{% icon %}` instead

**Why**: The most common icon migration failure is missed replacements. An automated grep catches any regressions when new templates are added.

## Testing

### Unit tests for the icon system

1. **Registry validation**: Test that every value in `ICONS` dict corresponds to a valid heroicon name
2. **Template tag rendering**: Test that `{% icon "next" %}` renders an `<svg>` element
3. **Semantic lookup**: Test that `{% icon "next" %}` renders the same SVG as `{% icon "arrow-right" force=True %}`
4. **Variant override**: Test that `variant="solid"` renders solid variant
5. **Accessibility defaults**: Test that rendered SVG includes `aria-hidden="true"` by default
6. **Accessibility label**: Test that `aria_label="Close"` renders `role="img"` and `<title>`
7. **Unknown icon**: Test that an unregistered icon name raises `KeyError`
8. **Force passthrough**: Test that `{% icon "arrow-right" force=True %}` renders the heroicon directly without registry lookup

### Regression test

9. **No leftover FA patterns**: Grep all `.html` templates for Font Awesome class patterns â€” test fails if any are found
10. **No leftover inline SVG icons**: Grep all `.html` templates for hand-coded `<svg` elements that represent icons (exclude legitimate non-icon SVGs if any). All standard icons should use `{% icon %}` instead of inline SVG paths
