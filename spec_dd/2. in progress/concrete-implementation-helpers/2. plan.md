# Plan: Concrete Implementation Helpers

This plan implements `1. spec.md`. The work is **almost entirely documentation and
Claude-plugin command authoring**, plus a small file-deletion cleanup. There is **no Django
application code, no models, no migrations, and no frontend templates** in scope — the external
template repo (which *does* contain a Django `config/`) is specified here but **not created** here.

Because there is no application code, this plan adds **no pytest tests**. The only test obligation
is success criterion 7: run the existing suite after the Caprover cleanup to confirm nothing breaks.

## Skills to use

- **`fls:claude-code-authoring`** — PRIMARY. Consult before authoring the new
  `/update_upgrade_notes` command (Task D1) and before editing `init.md` (Task B1), `update_fls.md`
  (Task E), `setup_todo_list.md` (Task D2) and the SDD `README.md` (Task D3). It governs command
  frontmatter, the depth-0 fan-out recipe, model tiering, the `status:` footer contract, and the
  "subagents cannot spawn subagents / cannot invoke slash commands" constraint that any new command
  must respect.
- **`code-comments`** — when editing markdown command bodies, do not leave section-citation
  breadcrumbs (`§4.2`, `per spec`) in the shipped files.
- **`brand-guidelines`** — light touch for the voice of the new/edited how-to docs (Tasks A, C, F).
- **`fls:update-claude-project-settings`** — only if the new `/update_upgrade_notes` command needs a
  tool permission not already granted in `.claude/settings.json` (it should not — it uses
  Read/Write/Glob/Edit/Bash/Agent, same toolset as existing SDD commands; `Bash` is for `git diff`/
  `git log` to inspect the actual diff).

Not relevant (no app code / no frontend): `frontend-styling`, `template`, `htmx`, `alpine-js`,
`testing`, `playwright-tests`, `multi-tenant`, `admin-interface`, `icon-usage`, `markdown-content`,
`registration`, `use-playwright`. Playwright MCP is not used (no UI changes).

## Task A — Template-repo manifest doc (spec §3, §4.3; success criteria 1, 3, 8)

Deliver the in-repo specification of the external GitHub Template Repository so the external repo can
be created and kept in sync against a known reference. **This document does not create the external

**A1. Write `docs/how tos/template repo manifest.md`** (new file). It must specify, as a complete
manifest:

- **Scaffolding mechanism**: GitHub Template Repository ("Use this template" includes the submodule);
  why this over cookiecutter / clone+script (cite the reasoning briefly, not the research filenames).
- **Repo file tree** the template must ship (spec §3.1):
  - `submodules/Freedom-LS` — initialised submodule pinned to a recent FLS commit; FLS plugin
    available at `submodules/Freedom-LS/fls-claude-plugin`.
  - `pyproject.toml` with `freedom_ls = { path = "submodules/Freedom-LS", editable = true }` + a
    committed `uv.lock`.
  - A fresh Django `config/` with the **three-module split** `settings_base.py` / `settings_dev.py` /
    `settings_prod.py` (dev & prod each `from .settings_base import *`).
  - Tailwind: `package.json` (downstream-appropriate `tailwind_build`), `tailwind.input.css`,
    `themes/` with the placeholder theme (Task A2).
  - Wrapper-script stubs + `.claude/` setup (finished by `fls:init`).
  - Project-level `.claude/settings.json` with `Write(submodules/**)` / `Edit(submodules/**)` **deny**
    rules + a `CLAUDE.md` "do not edit the FLS submodule" instruction (belt-and-suspenders read-only).
  - A README documenting post-generation steps (`git clone --recurse-submodules`, `uv sync`,
    `npm i && npm run tailwind_build`, `./claude.sh /fls:init`).
  - An auto-update workflow (Dependabot `gitsubmodule` or a scheduled GitHub Action) keeping the FLS
    pointer current.
- **`config/` content contract** — for each of the three settings modules, enumerate exactly what to
  mirror from FLS and what to **exclude**. Rather than copy-pasting the full lists/values (which would
  re-introduce the drift risk the spec warns about), the manifest must state plainly that **the live
  FLS `config/` is the authority** and that the key lists below are a **completeness checklist**, not
  a substitute for the real file. Concretely:
  - State that `settings_base.py` **mirrors the real FLS `config/settings_base.py`**: full
    `INSTALLED_APPS`, full `MIDDLEWARE`, `AUTH_USER_MODEL = "freedom_ls_accounts.User"` (call out the
    `freedom_ls_accounts` label explicitly — not `accounts`), Sites framework (no hardcoded
    `SITE_ID`; `CurrentSiteMiddleware` resolves per request), the `TEMPLATES` block (cotton cached
    loader, FLS context processors + cotton builtins), and the FLS settings block (list the keys:
    `FLS_THEME`, `FLS_THEMES_DIRS`, `configure_theme(...)`, `FREEDOM_LS_ICON_SET`, `MARKDOWN_*`,
    `ADMONITION_TYPES`, `MARKDOWN_TEMPLATE_RENDER_ON`, `COTTON_SNAKE_CASED_NAMES`, `AXES_*`,
    `AUTHENTICATION_BACKENDS`, allauth account settings, `WEBHOOK_EVENT_TYPES`, `TASKS`,
    `SALT_KEY`/`WEBHOOK_ENCRYPTION_SALT`, `LEGAL_DOCS_MANIFEST_PATH`, `TRUSTED_PROXY_IP_HEADER`,
    static/media).
  - State the `config/urls.py` contract: `health/`, env-var admin (`DJANGO_ADMIN_URL`), `educator/`,
    **both** `accounts/` includes (`allauth.urls` + `freedom_ls.accounts.urls`), the
    student-interface catch-all, and DEBUG-only standard dev includes (`debug_toolbar`,
    `django_browser_reload`) — **excluding** the FLS-internal `freedom_ls.qa_helpers.urls`.
  - State the `settings_dev.py` contract: `DEBUG = True`, insecure dev `SECRET_KEY`, relaxed
    `AUTH_PASSWORD_VALIDATORS`, `INTERNAL_IPS`, dev tooling (`debug_toolbar` +
    `django_browser_reload` with middleware), local Postgres `DATABASES`, Mailpit-style email
    backend — **excluding** FLS-internal bits (`freedom_ls.qa_helpers`, branch→db-name logic,
    `FORCE_SITE_NAME = "DemoDev"`, FirstClass branding, `FREEDOMLS_PERMISSIONS_MODULES`,
    `debug_branch_info`).
  - State the `settings_prod.py` contract: `DEBUG = False`; env-driven `SECRET_KEY` / `ALLOWED_HOSTS`
    / `CSRF_TRUSTED_ORIGINS` / `DATABASES`; the HTTPS+HSTS+secure-cookie+security-header block;
    logging; env-driven email; the S3/filesystem `STORAGES` switch with WhiteNoise manifest static
    storage; a production `TASKS` backend (note the base `ImmediateBackend` is dev-only — and that the
    template should carry the same `TODO @claude` note FLS does).
  - **Design principle to record** (spec §3.2): keep the template thin; never fork FLS

the template ships:

- A single placeholder theme, slug `custom`, at
  `themes/custom/static/themes/custom/theme.css` — **largely empty**: a header comment, commented-out

Create `fls-claude-plugin/commands/sdd/update_upgrade_notes.md`. **Consult `fls:claude-code-authoring`
first.** Mirror the shape of an existing SDD command (e.g. `update_product_docs.md`):

- **Frontmatter**: `description: Author the structured upgrade_notes.md for downstream FLS projects`;
  `allowed-tools: Read, Write, Glob, Edit, Bash, Agent`.
- **Behaviour**: read the spec + plan + the actual diff (`git diff`/`git log` via Bash) for the
  feature, then write `upgrade_notes.md` into the spec directory. It runs at depth 0; depth-0 fan-out
  is permitted (e.g. to inspect a large diff) but optional — the command produces a single artifact,
  so keep it lean and do not over-build it into a busy multi-phase command.
- **`upgrade_notes.md` format** (planning decision — the spec leaves the exact format to planning;
  choose **YAML frontmatter for the machine-readable flags + a short markdown body for prose**, since
  frontmatter is reliably parseable and the file stays short):

  ```markdown
  ---
  requires_migrations: false
  requires_template_review: false
  changed_template_paths: []          # populated when requires_template_review is true
  requires_settings_change: false
  changed_settings: []                # keys/settings when requires_settings_change is true
  requires_package_upgrade: false
  changed_packages: []                # package==version entries when true
  requires_tailwind_rebuild: false
  ---

  # Upgrade notes: <spec-name>

  ## Breaking changes
  <prose, or "None">

  ## Manual steps
  <prose, or "None">
  ```

  Document this schema inside the command file so every run produces the same shape, and so
  `update_fls` (Task E) can parse the frontmatter flags reliably.
- **Tick its own todo box**: at the end, delegate to `fls:sdd-mechanic` to run the

## Task E — `update_fls` improvements (spec §5.3; success criterion 6)

Edit `fls-claude-plugin/commands/concrete/update_fls.md`. Keep the existing correct core (advance
spec-by-spec, one commit per spec `Update FLS: <spec-name>`, test gate). Add:

- **Read `upgrade_notes.md`**: for each completed spec, read
  `submodules/Freedom-LS/spec_dd/3. done/<spec-dir>/upgrade_notes.md` and use its frontmatter flags
  to decide which integration steps to run. **Backward compatibility**: if absent, **warn** and fall
  back to the current prose-inference behaviour.
- **Dry-run / preview step**: before any pointer moves, list all pending completed specs and, per
  spec, its `upgrade_notes.md` flags (or "no notes — will infer"), so the operator sees the full plan
  first.
- **Migration safety**: run `uv run python manage.py migrate --check` **before** applying each spec
  and `uv run python manage.py makemigrations --check` **after**, to catch migration-state issues and
  silent conflicts.
- **Template-drift detection**: for specs whose notes flag `requires_template_review`, diff the listed
  changed FLS template paths against the concrete project's overrides and **flag** (do not auto-merge)
  any override whose upstream source changed.
- **Always `uv sync`** after a pointer move (catches new FLS deps). Note that concrete-project CI uses
  `uv sync --locked`.
- **Documented rollback procedure**: how to reset the submodule pointer and discard the in-progress
  integration commit, returning to the last good state, when a spec fails mid-integration.

Pseudocode for the per-spec loop (illustrative, to fold into the command prose):

```
for spec in pending_completed_specs (chronological):
    notes = read frontmatter of submodules/Freedom-LS/spec_dd/3. done/<spec>/upgrade_notes.md
    if notes missing: warn("no upgrade_notes.md — falling back to current prose-inference"); use existing prose-inference path (no new inference code)
    migrate --check          # fail early on dirty migration state
    move submodule pointer to the commit that completed <spec>
    uv sync                  # pick up new deps
    if notes.requires_migrations: makemigrations && migrate
    if notes.requires_template_review: flag drift for notes.changed_template_paths (no auto-merge)
    if notes.requires_tailwind_rebuild: npm run tailwind_build
    makemigrations --check   # catch conflicts introduced by this spec
    uv run pytest            # test gate
    commit "Update FLS: <spec>"
# on failure mid-spec: follow documented rollback (reset pointer, discard integration commit)
```
