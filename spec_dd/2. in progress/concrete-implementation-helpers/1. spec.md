# Spec: Concrete Implementation Helpers

## 1. Overview

FLS (Freedom Learning System) is designed to be embedded into other Django projects. A **concrete
implementation** is a fresh Django project that:

- includes FLS as a **git submodule** at `submodules/Freedom-LS` (read-only reference — Claude and
  developers must **never** edit it),
- installs it **editably** via uv (`freedom_ls = { path = "submodules/Freedom-LS", editable = true }`),
- wires up the FLS Claude plugin (via `./claude.sh` and `/fls:init`),
- supplies its own theme, config, password rules, settings, and — optionally — its own Django apps.

Today, standing up and maintaining such a project is a manual, error-prone exercise: the wiring
instructions in `docs/how tos/incorperate into another project.md` are half-finished (the `urls.py`
section is a bare `TODO`; the settings section is missing ~15 required settings), and keeping a
downstream project current with FLS relies on `concrete/update_fls` *inferring* integration steps
from spec/plan prose.

This spec makes creating and maintaining concrete implementations **easy and safe** by delivering
four things:

1. A specification for a **separate GitHub Template Repository** (structure + contents).
2. **FLS-side helpers** the template and downstream projects depend on — chiefly clarified
   `fls:init` behaviour, a completed/authoritative wiring story, and the how-to docs.
3. A **safe, structured FLS-update flow** built around a new per-spec `upgrade_notes.md` artifact
   authored as part of FLS's own spec-completion workflow.
4. **Caprover / dead-deployment cleanup**.

### Why this matters

- **Correctness.** A working `config/` out of the box eliminates the most common failure mode:
  a downstream that boots with a half-wired settings module and silently misbehaves (missing
  middleware, missing apps, wrong `AUTH_USER_MODEL` label).
- **Safety.** The submodule must stay read-only; structured upgrade notes replace fragile prose
  inference so upgrades are predictable and auditable.
- **Maintainability.** Keeping the template **thin** and FLS **fat** means downstream sync stays
  cheap — the lesson from django-oscar's migration-divergence trap (see
  `research_comparable_frameworks.md`).

## 2. Scope

### In scope

1. **Define the separate template repo** — its structure and contents. The repo itself lives
   outside this repository; this spec specifies what goes in it and FLS ships a manifest/doc
   describing it.
2. **FLS-side helpers**: clarified `fls:init`, an authoritative downstream-wiring story, the new
   `upgrade_notes.md` command, and updated how-to docs.
3. **A safe, structured FLS-update flow**: a new per-spec `upgrade_notes.md` artifact + a new SDD
   step that authors it, plus improvements to `concrete/update_fls` (dry-run, migration safety,
   template-drift detection, documented rollback, `uv sync`).
4. **Caprover / dead-deployment cleanup** (inventory in §6).

### Out of scope (explicit future work — do **not** build now)

- **Deployment / infrastructure helpers** (CI/CD, Caddy, Ansible, media storage, etc.). The
  template repo should leave room for an `env/stage` and `env/prod` branch story later, but we do
  not build it here. See the `deployment-phase-1` spec for where this is heading.
- **Do not** research, reference, or build on the current Caprover deployment setup — it is not in
  use and never will be. We only remove its dead artifacts (§6).
- Generic Docker assets (`Dockerfile`, `docker-compose.yml`, `nginx.conf`, `docker-entrypoint.sh`,
  `.dockerignore`, `docs/how tos/DOCKER_DEPLOY.md`) are **not** Caprover-specific — leave them
  alone; they belong to the future deployment-helpers work.
- **Actually creating** the external template repo. This spec specifies its contents; creating and
  publishing the GitHub repo is a follow-up action outside this codebase.

## 3. The template repo (separate repo)

**Scaffolding mechanism: GitHub Template Repository.** "Use this template" generates a new repo with
the submodule included (GitHub has supported submodules in template generation since 2020). No
external tooling is required; `fls:init` handles post-generation customisation. The chief
downside — the generated repo starts pinned to whatever FLS commit the template last pinned — is
mitigated by an auto-update workflow on the template repo (below).

Rejected alternatives (from `research_submodule_uv_mechanics.md` / `research_comparable_frameworks.md`):
**cookiecutter** (incomplete submodule support; adds a tool dependency for a one-time op) and
**clone + script** (more manual, no upstream-update story).

### 3.1 Template repo contents

The template repo ships:

- **`submodules/Freedom-LS`** — an initialised submodule pinned to a recent FLS commit, with the FLS
  Claude plugin available from inside it at `submodules/Freedom-LS/fls-claude-plugin`.
- **`pyproject.toml`** with `freedom_ls = { path = "submodules/Freedom-LS", editable = true }` and a
  committed **`uv.lock`**.
- **A fresh Django `config/`** pre-wired with FLS settings — this is the **authoritative** wiring
  (see §4.2). It ships **three settings modules mirroring FLS's own split** —
  `settings_base.py` + `settings_dev.py` + `settings_prod.py`, where `dev` and `prod` each start
  with `from .settings_base import *` — so the downstream inherits the same environment story.
  `settings_base.py` must mirror the real FLS `config/settings_base.py`:
  - the full `INSTALLED_APPS` (all 39 entries from the real `settings_base.py`, incl. `whitenoise.runserver_nostatic`, `django_cotton`,
    `django.contrib.sites`, `django.contrib.postgres`, `django.tasks`, the `unfold.*` set,
    `guardian`, all `freedom_ls.*` apps, `encrypted_fields`, `django_ace`, `allauth`/`allauth.account`,
    `axes`),
  - the full `MIDDLEWARE` (14 entries incl. `HtmxMessagesMiddleware`, `CurrentSiteMiddleware`,
    `RegistrationCompletionMiddleware`, `AxesMiddleware`, CSP, WhiteNoise, allauth `AccountMiddleware`),
  - `AUTH_USER_MODEL = "freedom_ls_accounts.User"` (note the `freedom_ls_accounts` label, **not**
    `accounts`),
  - the Sites framework wiring (`django.contrib.sites`; site resolved per-request by
    `CurrentSiteMiddleware` — no hardcoded `SITE_ID`),
  - `TEMPLATES` with the cotton cached loader, FLS context processors and cotton builtins,
  - the FLS-specific settings block (`FLS_THEME`, `FLS_THEMES_DIRS`, `configure_theme(...)`,
    `FREEDOM_LS_ICON_SET`, `MARKDOWN_ALLOWED_TAGS`, `ADMONITION_TYPES`,
    `MARKDOWN_TEMPLATE_RENDER_ON`, `COTTON_SNAKE_CASED_NAMES`, `AXES_*`,
    `AUTHENTICATION_BACKENDS`, allauth account settings, `WEBHOOK_EVENT_TYPES`, `TASKS`,
    `SALT_KEY`/`WEBHOOK_ENCRYPTION_SALT`, `LEGAL_DOCS_MANIFEST_PATH`, `TRUSTED_PROXY_IP_HEADER`,
    static/media settings),
  - a root **`config/urls.py`** including `health/`, the env-var-configurable admin
    (`DJANGO_ADMIN_URL`), `educator/`, both `accounts/` includes (`allauth.urls` and
    `freedom_ls.accounts.urls`), the student-interface catch-all, and DEBUG-only dev includes
    (standard dev tooling such as `debug_toolbar` and `django_browser_reload` — **not** the
    FLS-internal QA-TEMP routes like `freedom_ls.qa_helpers.urls`, which are FLS-only).
  - **`settings_dev.py`** carries the local-development story only: `DEBUG = True`, an insecure
    dev `SECRET_KEY`, relaxed `AUTH_PASSWORD_VALIDATORS`, `INTERNAL_IPS`, the dev tooling
    (`debug_toolbar` + `django_browser_reload`, with their middleware), a local Postgres
    `DATABASES`, and a Mailpit-style local email backend. It must **not** copy FLS's
    *internal* dev bits — `freedom_ls.qa_helpers`, the branch→db-name worktree logic,
    `FORCE_SITE_NAME = "DemoDev"`, the `FirstClass` branding overrides, or
    `FREEDOMLS_PERMISSIONS_MODULES` — those are FLS-only (mirrors the `qa_helpers.urls`
    carve-out in the urls bullet above).
  - **`settings_prod.py`** mirrors FLS's production hardening: `DEBUG = False`, env-driven
    `SECRET_KEY` / `ALLOWED_HOSTS` / `CSRF_TRUSTED_ORIGINS` / `DATABASES`, the HTTPS+HSTS+secure-cookie
    +security-header block, logging config, env-driven email, the S3/filesystem `STORAGES` switch
    with WhiteNoise manifest static storage, and a production `TASKS` backend (the base
    `ImmediateBackend` is dev-only; leave the existing `TODO @claude` note in place).
- **Tailwind setup**: `package.json` (with a downstream-appropriate `tailwind_build`), a
  `tailwind.input.css` template, and a `themes/` directory holding a **placeholder theme** (§3.3).
  The downstream's `tailwind.input.css` hardcodes its theme imports directly — the FLS `default`
  baseline tokens **and** the placeholder theme's `theme.css` — and does **not** use FLS's
  `write_active_theme_css` / `tailwind.active_theme.css` mechanism (that is FLS-internal). The
  template must document the `.gitignore`/`@source` glob pitfall: Tailwind's `@source` honours
  `.gitignore`, so a submodule under a gitignored path will be silently skipped.
- **Wrapper-script stubs and `.claude/` setup** — finished by `fls:init` post-generation.
- **A project-level `.claude/settings.json`** with `Write(submodules/**)` / `Edit(submodules/**)`
  **deny** rules, plus a `CLAUDE.md` "do not edit the FLS submodule" instruction —
  belt-and-suspenders so the submodule is read-only even before `fls:init` runs.
- **A README** documenting the post-generation steps:
  `git clone --recurse-submodules`, `uv sync`, `npm i && npm run tailwind_build`, then
  `./claude.sh /fls:init`.
- **An auto-update workflow** (e.g. Dependabot `gitsubmodule` or a scheduled GitHub Action) that
  keeps the template's FLS pointer current so new projects don't start badly stale.

### 3.2 Design principles (from research)

- **Keep the template thin.** Anything reusable belongs in FLS proper, not the template — the more
  logic lives in the template, the harder downstream sync becomes.
- **Never fork FLS models/migrations.** Concrete implementations must never need to fork FLS
  models or migrations. Custom models live in the **concrete project's own apps**, so FLS owns its
  migrations entirely and the downstream owns theirs. This is the migration-divergence trap that
  breaks django-oscar downstreams (`research_comparable_frameworks.md`).

### 3.3 Explicit theming for downstream projects

A concrete implementation "supplies its own theme" (§1). FLS already has a capable theming cascade —
`FLS_THEME` selects an active theme; `FLS_THEMES_DIRS` searches `BASE_DIR / "themes"` **before** the
in-package `freedom_ls/themes/`, so a downstream theme shadows the FLS defaults; and the FLS `default`
theme is an always-on token baseline that any other theme inherits and only overrides what it wants.
The gap is that this is invisible to someone scaffolding a new project — there is no obvious "edit
this to rebrand" file, and the settings give no hint of how to switch. This section makes theming
**explicit and discoverable** in the template.

- **Ship a placeholder theme.** The template's `themes/` directory carries a single placeholder
  theme (working slug `custom`) at `themes/custom/static/themes/custom/theme.css`. It is
  **largely empty** — a header comment plus commented-out examples of the most common role tokens
  (`--color-primary`, `--color-surface`, `--color-accent`, a radius, a font) and a pointer to the
  FLS `theme-fls.md` how-to and the canonical FLS `default` `theme.css` for the full token contract.
  Because it overrides nothing, the FLS `default` baseline (imported first in `tailwind.input.css`)
  supplies every token, so a freshly generated project looks exactly like stock FLS until a developer
  edits this file. The placeholder is the **obvious, documented place to rebrand**: uncomment a token,
  rebuild Tailwind, done — no template edits, no Python.
- **`FLS_THEME` must default to `custom`.** The template's `config/settings_base.py` sets
  `FLS_THEME = os.environ.get("FLS_THEME", "custom")` — i.e. the default is the placeholder theme,
  **not** `"default"`. This is a hard requirement: the placeholder is wired in and ready to edit from
  the first boot, while (being empty) it still renders the FLS default look.
- **Make switching obvious in settings.** `FLS_THEME` in `settings_base.py` carries a comment that
  enumerates the **valid values**: `"custom"` (this project's placeholder theme under `themes/`),
  `"default"` (the FLS baseline shipped in the submodule), `"first_class"` (the FLS demo brand
  shipped in the submodule), or any other theme slug the project adds under `themes/<slug>/`.
  Switching themes is then a one-line settings change (or an `FLS_THEME` env var override).
- **Document the `tailwind.input.css` contract.** Per §3.1 the downstream's `tailwind.input.css`
  imports the FLS `default` baseline and then the active theme's `theme.css` directly. The template
  documents that **the theme imported here must match `FLS_THEME`** — they are kept in sync by hand
  in a downstream (the FLS-internal `write_active_theme_css` autogeneration is not used), so adding a
  new theme means both pointing `FLS_THEME` at it and updating the import.
- **How-to coverage.** The README/wiring overview (§4.2) gains a short "to rebrand, edit
  `themes/custom/static/themes/custom/theme.css`" note and points at the FLS `theme-fls.md` how-to
  for the three-tier model (tokens / component classes / template overrides). The how-to itself lives
  in FLS (thin-template principle, §3.2); the template only points at it.

## 4. FLS-side helpers

### 4.1 `fls:init` (clarify, do not expand)

`fls:init` (`fls-claude-plugin/commands/init.md`) stays a **plugin-bootstrap** command, not a
full-project scaffolder. The template repo supplies project structure (`config/`, `pyproject.toml`,
Tailwind, `.claude/settings.json`, `CLAUDE.md`); `fls:init` finishes the plugin wiring on the
generated repo.

**Non-destructive / idempotent.** `fls:init` must **never overwrite existing configuration
wholesale**. Where config is already in place it makes only the necessary changes and leaves the
rest untouched; running it again is safe. Concretely:

- **`.claude/settings.json`** — merge into the existing file (add missing `allow`/`deny` rules,
  `enabledPlugins`, the `SessionStart` hook); never replace the whole file or clobber the user's
  existing entries.
- **`.claude/fls/config.md` + `config.local.md`** — create only when absent; if they already exist,
  **skip** them (do not regenerate over user values).
- **`.gitignore`** — append the missing entries only; never rewrite the file.
- **Wrapper scripts** (`claude.sh` at root, others under `.claude/fls/scripts/`) — copy with the
  `__FLS_PATH__` placeholder replaced by `submodules/Freedom-LS` only when the script is absent;
  **skip** any that already exist.
- then validates the result.

This matches the current `init.md` behaviour; the spec records it as a hard requirement so future
edits don't regress it.

No new scaffolding behaviour is added to `fls:init` in this spec. (If the docs anywhere imply
`fls:init` scaffolds Django structure, correct them — the template repo does that.)

### 4.2 Authoritative downstream-wiring story

**Decision:** the template repo's working `config/` is the **single source of truth** for downstream
wiring. To avoid maintaining two copies of the settings/urls lists that will inevitably drift:

- The half-finished `docs/how tos/incorperate into another project.md` is **trimmed to a conceptual
  overview** that explains *why* each piece exists (submodule + editable uv install, Sites
  framework, theming cascade, Tailwind `@source` pitfall, read-only submodule rule) and **points at
  the template repo** for the canonical, copy-pasteable config. Its bare `urls.py` `TODO` and the
  incomplete `settings.py` `TODO` sections are resolved by this redirection — the doc no longer
  tries to enumerate every setting inline.
- The misspelled filename (`incorperate`) is corrected to `incorporate into another project.md`,
  and inbound links updated. *(Confirm during planning whether any docs link to the old name.)*
- The doc's incorrect/stale specifics are fixed where they remain (e.g. `AUTH_USER_MODEL` shown as
  `"accounts.User"` should be `"freedom_ls_accounts.User"`; `HEADLESS_CLIENTS` is commented out in
  real settings).

### 4.3 Template-repo manifest

FLS ships a doc (e.g. under `docs/how tos/`) that specifies the template repo's structure and
contents (§3.1) so the external repo can be created and kept in sync against a known reference. This
is the in-repo deliverable for scope item 1; it does not create the external repo.

## 5. Keeping concrete implementations up to date

The existing `concrete/update_fls` core design is **correct and stays**: advance the submodule
**spec-by-spec**, read each completed spec to understand what changed, run a test gate, and make
**one commit per spec** (`Update FLS: <spec-name>`). The improvements below make it structured and
safe rather than prose-inferred.

### 5.1 Structured per-spec `upgrade_notes.md` (new artifact)

Going forward, completing an FLS spec authors a small **structured** `upgrade_notes.md` in the
spec directory. It carries machine-readable flags plus prose for the human/agent doing the upgrade:

- `requires_migrations` (bool)
- `requires_template_review` (bool) — and, if true, the list of changed FLS template paths
- `requires_settings_change` (bool) — and which settings/keys
- `requires_package_upgrade` (bool) — and which packages/versions
- `requires_tailwind_rebuild` (bool)
- free-text **breaking-change notes** and any manual steps

`update_fls` reads this file to decide which integration steps to run, instead of inferring from
prose. **Backward compatibility:** older completed specs have no `upgrade_notes.md`; `update_fls`
**warns** and **falls back to prose inference** (its current behaviour) for those.

The exact file format (YAML frontmatter vs a fenced block vs a small fixed markdown schema) is a
planning-stage decision; the requirement is that the flags are reliably machine-parseable and the
file is short.

### 5.2 New SDD step that authors `upgrade_notes.md`

**Decision:** a **new dedicated SDD command/step** authors the upgrade notes (not folded into
`/update_product_docs` or `/finish_worktree`). Rationale: clean separation of concerns
(downstream-upgrade notes vs FLS product docs), and a focused command is less likely to rush the
notes than the busy worktree-finish step.

- A new command (working name `/update_upgrade_notes`) reads the spec + plan + actual diff and
  writes the structured `upgrade_notes.md` into the spec directory, then ticks its own todo item.
- A new line is added to the SDD `todo.md` template (`setup_todo_list.md` helper) between
  **Product documentation** (step 10) and **Pull request** (step 11).
- The SDD `README.md` is updated; its existing `## TODO` line ("Command for producing an upgrade
  guide when a change affects downstream projects that extend FLS") is resolved by this work.

This closes the acknowledged gap that `update_fls` currently works around.

### 5.3 `update_fls` improvements

- **Dry-run / preview step**: before any pointer moves, list all pending completed specs and their
  `upgrade_notes.md` flags (or "no notes — will infer") so the operator sees the whole plan first.
- **Migration safety**: run `manage.py migrate --check` **before** applying each spec and
  `makemigrations --check` **after**, to catch concrete-project migration-state issues and avoid
  silent conflicts.
- **Template-drift detection**: for specs whose notes flag `requires_template_review`, diff the
  changed FLS template paths against the concrete project's overrides and **flag** any override
  whose upstream source changed (a lightweight take on Symfony Flex's diff-based patching). Flag,
  do not auto-merge.
- **Always `uv sync`** after a pointer move (catches new FLS dependencies). Concrete-project CI uses
  `uv sync --locked`.
- **Documented rollback procedure** for when a spec fails mid-integration (how to reset the
  submodule pointer and discard the in-progress integration commit, returning to the last good
  state).

## 7. Success criteria

1. **Template-repo spec exists**: an in-repo manifest/doc fully specifies the template repo's
   structure and contents (§3.1), enough to create the external GitHub Template Repository from it.
2. **A generated-from-template project boots**: following the template README
   (`clone --recurse-submodules` → `uv sync` → `npm i && npm run tailwind_build` →
   `./claude.sh /fls:init`) yields a project whose `config/` mirrors FLS's real settings/urls and
   that runs `manage.py check` / the dev server without missing-app/middleware errors. *(The boot
   is validated against the spec'd config; full end-to-end validation of the external repo is a
   follow-up.)*
3. **Read-only submodule enforced**: the template ships `.claude/settings.json` deny rules for
   `submodules/**` and a `CLAUDE.md` instruction; FLS docs state the rule clearly.
4. **Wiring has one source of truth**: the how-to doc is a conceptual overview pointing at the
   template repo; no duplicated full settings/urls list remains to drift. Stale specifics
   (`AUTH_USER_MODEL` label, `HEADLESS_CLIENTS`, etc.) are corrected. Filename misspelling fixed.
5. **Structured upgrade notes are part of the FLS workflow**: a new SDD command authors
   `upgrade_notes.md`, a new todo.md line exists, the SDD README is updated, and its TODO line is
   resolved.
6. **`update_fls` is structured and safe**: it reads `upgrade_notes.md` (warns + falls back to
   inference when absent), offers a dry-run preview, runs `migrate --check`/`makemigrations --check`
   around each spec, detects template drift for flagged specs, always `uv sync`s after a pointer
   move, and documents rollback.
7. **Caprover cleanup complete**: `captain-definition` and `Caprover deploy.md` deleted, README line
   edited, `deployment.md` link fixed; no broken links; generic Docker assets untouched; full test
   suite green.
8. **Theming is explicit and discoverable**: the template ships a largely-empty placeholder theme
   that is the obvious place to rebrand; `FLS_THEME` defaults to it and its `settings_base.py`
   comment enumerates the valid values; with the placeholder unedited the generated project renders
   the FLS default look; and the wiring overview documents how to rebrand and switch themes.

## 8. Decisions log

- **GitHub Template Repository** over cookiecutter (submodule-support gaps, extra tool dependency)
  and clone+script (more manual, no upstream-update story).
- **Thin template / fat FLS** — reusable logic lives in FLS; the template carries only config and
  structure, never forked FLS models/migrations.
- **Upgrade notes get a new dedicated SDD command/step** (user decision) rather than being folded
  into `/update_product_docs` or `/finish_worktree` — keeps concerns separated and the notes
  unrushed.
- **Template repo is the authoritative source of downstream wiring** (user decision); the how-to
  doc is trimmed to a conceptual overview that points at it, eliminating the drift risk of two
  full settings/urls lists.
- **`fls:init` stays plugin-bootstrap-only**; the template repo (not `fls:init`) supplies Django
  project structure.
- **Backward-compatible upgrade flow**: specs without `upgrade_notes.md` fall back to prose
  inference with a warning, so `update_fls` keeps working on the existing history.
- **Caprover artifacts removed, generic Docker assets retained** for the future
  deployment-helpers work.
- **Theming made explicit via a placeholder theme** rather than expecting downstreams to discover the
  cascade themselves: the template ships a largely-empty `custom` theme, defaults `FLS_THEME` to it,
  and documents the valid `FLS_THEME` values in `settings_base.py` — the existing FLS theming
  machinery is reused as-is (thin-template principle), not extended.
