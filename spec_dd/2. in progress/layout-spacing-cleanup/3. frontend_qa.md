# Frontend QA: Layout Spacing Cleanup

This QA plan verifies the spec's success criteria using the **fls:use-playwright**
skill (Playwright MCP). The goal is to confirm:

1. Site-header → first-heading gap is consistent across all learner pages.
2. TOC sidebar has visible horizontal space from main content at desktop and
   visible vertical space when stacked (mobile only if it ever stacks).
3. Every learner page has visible bottom-of-page breathing room beneath
   the last visible element / action button.
4. Anchor links land target headings with breathing room above them.
5. The Alpine sidebar still toggles, persists state, and handles the mobile
   overlay correctly.
6. Educator interface, allauth pages, legal docs, and admin still render
   correctly.

## Setup

### Start the dev server on a random port

```
PORT=$(.claude/fls/scripts/find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL: `http://127.0.0.1:$PORT`

### Test data

Use the **DemoDev** site (per user memory: always use the DemoDev site for
QA test data). The course `standard-markdown-demo-finance` and equivalent
DemoDev courses already include topics, forms, and quiz pages.

Login credentials and exact course slugs come from the DemoDev seed data
already in the project — verify with the seed script / admin if a slug is
unknown, do not invent one.

### Viewport sizes

All visual checks are run at three widths (per spec D6):

- **Desktop**: `1280 x 800`
- **Tablet**: `768 x 1024`
- **Mobile**: `375 x 812`

For each test below, take a Playwright screenshot at each width.

## Test 1 — Site header to first heading gap (the core regression)

**Goal:** functional req 1 — every learner-facing page has a consistent gap
between the site header bar and the first page heading. The gap comes from
the `c-page` wrapper, not from per-page `mt-*`.

**Steps**, repeat at desktop / tablet / mobile:

1. Navigate to `http://127.0.0.1:$PORT/` — the user home page.
2. Screenshot. Visually confirm there is a comfortable gap between the
   bottom of the blue site header bar and the top of "Welcome" / "Welcome
   back" `h1`.
3. Navigate to the all-courses page (link from home) — confirm the same
   gap above the "All Courses" `h1`.
4. Navigate to a course home, e.g.
   `/courses/standard-markdown-demo-finance/` (or the equivalent DemoDev
   course slug). Confirm the same gap above the course title `h1`.
5. Navigate to a topic page inside that course. Confirm the same gap
   between `_base_interface.html`'s sidebar header bar and the `h1`.
   (Note: on `_base_interface.html` pages, the gap is between the
   sidebar's header bar — which sits below the site header bar — and the
   first heading inside the right-hand content column. This is a
   different "header" from the `_base.html` case, but the breathing-room
   should still feel comfortable and consistent.)
6. Navigate to a form page. Confirm the same gap above the form `h1`.

**Pass criteria:**

- The gap between header and `h1` looks visually similar (within a
  few px) across all five pages at the same width.
- The gap is comfortable, not flush, not excessive.
- No page has its `h1` jammed under the header.

## Test 2 — Page horizontal padding from a single source

**Goal:** functional req 2 — content does not touch the screen edge at
mobile width, content sits in a centered max-width well at desktop.

**Steps**, repeat at all three widths:

1. From each page in Test 1, screenshot at full viewport (not just the
   header area).
2. At mobile (375px): visually confirm there is at least ~16px of
   horizontal space between the screen edge and the content. The "Welcome"
   text and other content should not be pressed against the left/right
   edges.
3. At desktop (1280px): visually confirm content sits inside a centered
   well (some empty space on left and right of the well).

**Pass criteria:**

- No content touches the screen edge at any width.
- Centered well at desktop is visibly centered (whitespace on both sides).

## Test 3 — TOC sidebar / main content gap at desktop

**Goal:** functional req 3 — when the TOC sidebar is open at `lg:` width
or wider, there is visible horizontal space between sidebar and main
content. The space comes from grid `gap-*`, not margin.

**Steps**, at desktop width (1280px):

1. Navigate to a course topic page, e.g.
   `/courses/standard-markdown-demo-finance/topics/<slug>/` (or whatever
   the DemoDev seed exposes).
2. Confirm the sidebar is open by default (Alpine restores it from
   `localStorage` for the `sidebar-course-toc` key — clear local storage
   first if you want a fresh open-by-default behaviour, or click the
   open-sidebar button if it's collapsed).
3. Screenshot. Visually confirm there is a clear gap (at least ~24px)
   between the right edge of the sidebar panel and the left edge of the
   main content column. Lessons / content should not feel pressed
   against the TOC.
4. Repeat on a course form page (any form content in DemoDev).

**Pass criteria:**

- Visible horizontal gap between sidebar panel and main content.
- The gap looks the same on the topic page and the form page (single
  source of truth, not per-page).

## Test 4 — TOC sidebar mobile / tablet behaviour

**Goal:** functional req 4 — on small screens the sidebar collapses to an
overlay. The Alpine toggle, persistence, and backdrop must all still work.

**Steps**, at mobile (375px) and tablet (768px):

1. Navigate to a course topic page. The sidebar should be collapsed.
2. Click the open-sidebar button in the sidebar header bar.
3. Confirm the sidebar slides in as an overlay (not in flow), and the
   semi-transparent black backdrop is visible behind it.
4. Click the backdrop. Confirm the sidebar closes.
5. Open it again. Click a TOC link inside the sidebar. Confirm the sidebar
   closes (`handleSidebarClick` behaviour) and navigation happens.
6. Reload the page. Confirm sidebar state is restored from `localStorage`
   correctly (closed on mobile by default; whatever the last toggle state
   was).
7. Resize from mobile to desktop without reloading. Confirm the sidebar
   adapts (the alpine `_mqHandler` listens to the `(min-width: 1024px)`
   media query).

**Pass criteria:**

- Sidebar overlay opens/closes from the toggle button.
- Backdrop click closes the sidebar.
- Clicking a sidebar link closes the sidebar (mobile only).
- Sidebar state persists across reloads.
- Sidebar adapts smoothly when resizing across the `lg:` breakpoint.

## Test 5 — Bottom-of-page breathing room

**Goal:** functional req 5 — every learner-facing page has visible space
below the last element before the viewport ends.

**Steps**, at all three widths:

1. Navigate to a course topic page. Scroll to the bottom (or land on a
   short topic where the bottom action bar is visible without
   scrolling).
2. Screenshot. Confirm visible space (at least ~32px at desktop, ~24px
   at mobile) below the bottom action bar (Previous / Next buttons).
3. Repeat on a course form page (last page with the Finish button).
4. Repeat on `course_form_complete.html` (after submitting a quiz). The
   "Continue" / "Return to Course" button should not sit flush against
   the bottom of the viewport.
5. Repeat on `course_finish.html` (after completing a course). The
   "Return to Home" / "View Course" buttons should not sit flush against
   the bottom.

**Pass criteria:**

- Every page has visible space beneath the last visible element.
- The space looks consistent across all four pages (single source of
  truth from `c-page`).

## Test 6 — Anchor link target offset

**Goal:** functional req 6 — when an anchor link (`#some-id`) is followed,
the target heading lands with a comfortable buffer above it.

**Steps**, at desktop width:

1. Navigate to a markdown-heavy topic page that has multiple `h2` /
   `h3` headings. The DemoDev `standard-markdown-demo-finance` course
   has topics with markdown content that auto-generates `id` attributes
   on headings.
2. Inspect the page (Playwright `browser_evaluate`) to grab a heading
   `id`. Or look at one of the TOC anchors if the TOC links point to
   headings.
3. Navigate directly to `…/topic-url/#that-heading-id`.
4. Confirm the heading is visible **with breathing room above it** —
   not flush against the top of the viewport.
5. Click another anchor link (within the same page if available, or
   navigate to another `…#id` URL). Confirm same behaviour.

**Pass criteria:**

- Anchor-targeted headings land with at least ~24px–96px clear above them.
- The behaviour comes from `scroll-margin-top` on heading elements (not
  `scroll-padding-top` on `<html>`). Verify with `browser_evaluate`:
  `getComputedStyle(document.documentElement).scrollPaddingTop` should be
  `'auto'` or `'0px'` (i.e. unset) and an `h2[id]` should have a non-zero
  `scrollMarginTop`.

## Test 7 — No regression on educator interface

**Goal:** functional req 8 — educator interface still renders correctly
after the layout changes.

**Steps**, at desktop and mobile:

1. Log in as an educator account (DemoDev seed includes one).
2. Navigate to `/educator/` (or whatever the educator landing URL is).
3. Screenshot. Visually confirm:
   - The sidebar header bar isn't pushed to the screen edge.
   - The breadcrumbs and main content have horizontal padding.
   - The page has top and bottom breathing room.
4. Click into a cohort detail / course progress page. Same checks.

**Pass criteria:**

- No visible regression vs the spec's "before" expectation. Pages are
  not visually broken. If something is broken, the minimum fix is to
  wrap the right-hand content column's inner content in `<c-page>` —
  larger refactor is out of scope.

## Test 8 — No regression on allauth & legal docs

**Goal:** functional req 8 again — auth and legal pages still render.

**Steps**, at desktop and mobile:

1. Log out. Navigate to the login page. Screenshot.
2. Confirm the login form sits in a centered, comfortably-padded well —
   not flush to top, not flush to edges.
3. Navigate to a legal doc URL (e.g.
   `/accounts/legal/<doc-slug>/` or whatever the project routes legal
   docs at). Screenshot.
4. Confirm the document renders with reading-width content, top/bottom
   padding, and horizontal padding.

**Pass criteria:**

- Login / signup / email / password pages render normally.
- Legal docs render with comfortable reading width and breathing room.

## Test 9 — No regression on admin

**Goal:** Unfold admin doesn't extend `_base.html`, so it should be
unaffected. Confirm.

**Steps**, at desktop:

1. Log in as an admin account. Navigate to `/admin/`.
2. Confirm the admin loads normally (sidebar, content, no broken layout).

**Pass criteria:**

- Admin renders identically to before.

## Test 10 — Existing student flows are not broken

**Goal:** functional req 7 — full regression-style smoke test.

**Steps**, at desktop:

1. Log in as a student. Browse to a course.
2. Register for a course (if not already registered).
3. Open a topic. Mark it complete. Move to the next item.
4. Open a form / quiz. Fill in answers. Submit. View results.
5. Click "Continue" / "Next" through to the end of the course. Land on
   `course_finish.html`.
6. Confirm at every step the page renders, the buttons work, and the
   content flows correctly.

**Pass criteria:**

- All existing student flows complete end-to-end without errors.

## Reporting

For each test, record:

- The viewport width(s) tested.
- Pass / fail.
- A screenshot saved to a temporary path (Playwright MCP supports
  screenshotting).
- For any fail: the page URL, the viewport, and a one-line description
  of the issue.

A single failure on a regression test (Tests 7, 8, 9, 10) is a blocker.
A failure on Tests 1–6 means the wrapper / grid / scroll-margin values
need tuning — re-tune in the implementation, not in the QA, and re-run
the affected test.
