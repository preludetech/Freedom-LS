# Spec: Layout Spacing Cleanup

## Overview

Three related layout/spacing problems in the learner experience are being fixed
together because they share a single root cause: spacing is currently being
applied at the wrong layer (per-page templates) instead of being owned by the
shared layout. Fixing #1 (page-wrapper) is what unblocks #3 (bottom breathing
room), and the TOC sidebar in #2 lives inside the same layout system.

The work is **layout / CSS only**. No model, view, URL, permission, or data
changes. No new user-visible features. Existing functionality must continue to
work unchanged.

Scope: the learner-facing pages reachable through `_base.html` and
`_base_interface.html` — primarily the student interface (course home, topic,
form, form page, form complete, course finish) and the home page. Other apps
that extend `_base.html` (educator interface, account pages, legal docs) inherit
the layout fix and must be visually verified, not refactored beyond what's
needed to remove now-redundant per-page padding.

Reference research: `research_spacing.md` (in this directory). The spec follows
the conventions documented in §9 of that document.

## Why this matters

**For learners:** the current pages feel inconsistent — the gap between the
site header and the page heading varies; on the course topic pages the TOC
sidebar sits flush against the main content on large screens; and pages with
bottom action buttons end abruptly with no breathing room beneath them. None of
these are bugs, but cumulatively they make the product look unfinished.

**For developers:** spacing is currently a per-template concern. Pages opt
back in to consistency by adding `mt-8`, `mb-12`, `px-4` etc. wherever they
remember to. New pages start inconsistent by default. Fixing the layer where
spacing lives means new pages get correct spacing automatically and template
authors can stop thinking about it.

## Goals

1. The site header bar and the first heading on every page have consistent,
   sufficient vertical breathing room — set once, in the shared layout.
2. The TOC sidebar and main content on course topic / form pages have
   consistent horizontal separation on large screens, and consistent vertical
   separation when the sidebar is collapsed on smaller screens. Spacing comes
   from a `gap-*` on a parent grid, not from margins on children.
3. Every page has consistent bottom-of-page breathing room beneath any final
   content (including action buttons), set once in the shared layout.
4. Page templates do not need to add their own outer padding/margin to look
   correct.
5. The conventions in `research_spacing.md` §9 are adopted in code, so
   future pages start consistent.

## Non-goals

- Making the site header sticky. The header is currently non-sticky and stays
  that way for this spec.
- Redesigning anything visual beyond the spacing/layout layer (no colour, type,
  component, or copy changes).
- Refactoring the educator interface or admin UI. Those pages must be
  visually verified after the layout change but should not be reworked here.
- Replacing the existing `_base_interface.html` sidebar component (the Alpine
  sidebar). The TOC sidebar fix happens *inside* that component's content
  area, not by replacing it.
- Introducing new spacing tokens or arbitrary values. Stick to Tailwind's
  default spacing scale (4, 6, 8, 12, 16, 24…).
- Bottom action bar that floats over content. Out of scope.

## Decisions

### D1: Anchor-link offset uses per-heading `scroll-mt-*`, not global `scroll-padding-top`

The idea file proposed adding `scroll-padding-top` on `<html>` so that anchor
links land below a sticky header. Two problems with that:

- The site header is **not currently sticky** (see
  `partials/header_bar.html` — it's a plain block element). Reserving space for
  a sticky header that doesn't exist would push every anchor target down for
  no reason.
- `research_spacing.md` §5 specifically recommends per-heading
  `scroll-mt-*` (or an equivalent base-CSS rule keyed on `id`-bearing
  headings) over global `scroll-padding-top`, because the global rule
  affects all anchor scrolls including focus-management cases where it can
  surprise users.

**Decision:** keep the site header non-sticky for this spec (out of scope to
change). Apply `scroll-mt-*` to heading targets so anchor links land correctly
even if the header becomes sticky later. Sized to a small comfortable buffer
(matching the page wrapper's top breathing room is fine — a Tailwind `scroll-mt-8`
to `scroll-mt-24` range covers both the current non-sticky case and a future
sticky case at the current header height).

The cleanest way to apply this without touching every heading by hand is to
add a base CSS rule (in the project's compiled CSS, e.g. via a small
`@layer base` rule in the Tailwind input) that gives `scroll-margin-top` to
heading elements that have an `id` (or to all headings inside the markdown
content area). This way author-written headings in markdown content get it
automatically. If a CSS-rule approach is awkward, applying `scroll-mt-*`
classes inside `markdown-container` and adding it to the page-wrapper's
heading slot is acceptable — the goal is "no per-template fiddling".

### D2: Single shared page wrapper, not a hand-rolled `<div>` per template

Adopt the page-wrapper pattern from `research_spacing.md` §4. Express it as a
**cotton component** (e.g. `c-page`) consistent with the rest of this
codebase's frontend conventions, and use it for the content well of every
learner-facing page. As a layout-level component shared across apps, the
natural home is `freedom_ls/base/templates/cotton/page.html` — implementation
may relocate if there's a stronger reason.

The wrapper owns:

- `max-width` and centering (`mx-auto max-w-7xl` — match the current
  `_base.html` width).
- Horizontal page padding. Indicative starting values: `px-4 sm:px-6 lg:px-8`.
  These are tunable during implementation; the existing `_base.html` uses
  `px-2 sm:px-6 lg:px-8` and the bump from `px-2` to `px-4` at mobile is
  intentional (current mobile padding is unusually tight) but the final
  number should be agreed during the brand-guidelines check.
- Top **and bottom** vertical padding. Indicative starting values:
  `py-8 lg:py-12`. The values can be tuned during implementation as long as
  the ramp stays small (see non-functional req 2).

The wrapper does **not** own inter-section spacing — sibling sections use a
single `space-y-*` on the parent inside the wrapper.

`_base.html`'s `<main>` keeps its responsibility for the page chrome (header,
messages bar, debug badge) but gives up its current `max-w-7xl mx-auto px-2 sm:px-6 lg:px-8`
content padding. That moves into the page wrapper. The existing inner
`<div class="py-6 sm:py-12 space-y-6 sm:space-y-12">` that currently wraps
`{% block content %}` in `_base.html` is also retired — its `py-*` moves into
the page wrapper, and inter-section `space-y-*` becomes the responsibility of
content inside the wrapper (per §9 rule: parent owns inter-child rhythm). This
is what stops every page wrestling with double padding when it tries to add
its own top/bottom breathing room.

**Implication for `_base_interface.html`:** that template currently overrides
`{% block body %}` and therefore already bypasses the inner `_base.html`
content-well wrapper. After this change `<main>` no longer owns horizontal
padding either, so the sidebar/header-bar/content layout inside
`_base_interface.html` must take responsibility for its own outer horizontal
padding (so the sidebar header bar and overlay backdrop don't push to the
screen edge). The page wrapper is then used inside `{% block content %}` of
pages that extend `_base_interface.html` — it wraps only the right-hand
content column, which is the correct scope for the wrapper's max-width and
top/bottom padding.

### D3: TOC sidebar layout uses CSS `grid` with `gap-*`, not flex with margins

Per `research_spacing.md` §3. The current `_base_interface.html` uses
`flex` with no `gap-*` between the sidebar column and the main content column.
On large screens the columns are flush.

Switch the inner sidebar+content container from flex to grid at the `lg:`
breakpoint:

- Sidebar column gets a fixed track when open, content column gets
  `minmax(0, 1fr)` so long words / wide code blocks don't blow out the grid.
- `lg:gap-12` (or comparable) on the parent gives a single source of truth
  for the sidebar/content distance.
- On smaller screens the sidebar is overlay/drawer (existing behaviour) and
  vertical separation is not needed because they don't occupy the same flow.
  When the sidebar collapses to a stacked layout (rather than overlay), use
  `gap-8` on the same parent so vertical separation comes from the same source.
- `min-w-0` stays on the main content column.

The Alpine logic that toggles the sidebar must keep working. The change is
to the container's display mode and to the spacing source — not to the
toggle behaviour, the storage key, the keyboard handling, or the mobile
backdrop.

### D4: Per-template padding/margin is removed where it duplicates the wrapper

Once the page wrapper provides the top/bottom and horizontal padding, every
learner-facing page template gets reviewed for now-redundant outer
padding/margin and that padding/margin gets removed. Inter-section spacing
inside the wrapper stays where it is (or is consolidated to a single
`space-y-*` on the parent if it isn't already).

Specific known offenders to clean up (non-exhaustive — implementation will
audit):

- `course_topic.html`: outer `mt-8 pt-4` on the bottom action bar — should
  come from `space-y-*` on the parent and the wrapper's bottom padding.
- `course_form_complete.html`: `<div class="max-w-4xl mx-auto py-12">` —
  duplicates the page wrapper's job. The `mx-auto py-12` is redundant once
  the wrapper provides them, so it goes. The narrower `max-w-4xl` is an
  intentional readability constraint for the results page and must be
  preserved. Either (a) keep an inner `<div class="mx-auto max-w-4xl">` for
  this single page inside the wrapper's slot, or (b) add a `width` variant to
  `c-page` (e.g. `<c-page width="narrow">`) per the variant approach in
  `research_spacing.md` §4. Implementation can pick whichever has lower
  blast radius; if no other learner page needs a narrower well, option (a) is
  simpler.
- `course_home.html`: cascading `mt-6 sm:mt-8`, `mt-8 sm:mt-12`, `mt-4 sm:mt-6`
  on sequential sections — should consolidate to a single `space-y-*` on
  the parent.
- `home.html`: `mt-8 sm:mt-12` on the all_courses block — same treatment.

The audit must follow `research_spacing.md` §9 rule "only one ancestor per
chain owns horizontal padding; only one ancestor per chain owns vertical
rhythm".

### D5: Scope of pages refactored

In scope (must look right after the change, must use the new wrapper):

- All `student_interface` pages (`home.html`, `all_courses.html`,
  `course_home.html`, `course_topic.html`, `course_form.html`,
  `course_form_page.html`, `course_form_complete.html`, `course_finish.html`,
  partials they include). Note: `home.html` and `course_home.html` extend
  `_base.html` directly; the topic/form/finish/form-complete pages extend
  `student_interface/_course_base.html`, which extends `_base_interface.html`.

In scope for **visual verification** only (no template refactor unless the
page is visibly broken by the layout change):

- `educator_interface` pages.
- `accounts/legal_doc.html` and other allauth-layout pages.
- Admin pages reached via Unfold (these don't extend `_base.html`, so should
  be unaffected — but check).

If a non-student-interface page is visibly broken by the change, the minimum
fix is to wrap its content in the new page wrapper. Larger refactor of those
apps' templates is out of scope.

### D6: Visual verification at three widths

Per the idea file, every visible change is verified with playwright MCP
screenshots before/after, at desktop, tablet, and mobile widths. The fls
playwright skill (`fls:use-playwright`) is the right tool. The QA plan
(produced later by `/plan_from_spec`) will list specific URLs and viewport
sizes; for the spec it's enough to require:

- Desktop ≈ 1280px width
- Tablet ≈ 768px width
- Mobile ≈ 375px width

…and to require that the demo content site (DemoDev — see user memory
guidance) is the source of test data.

## Functional requirements

1. **Site header to first-heading gap.** On every learner-facing page, the
   distance between the bottom of the site header bar and the top of the
   first page heading is consistent, comfortable, and set by the layout —
   not by per-page padding. The url called out in the idea file
   (`/courses/standard-markdown-demo-finance/`) and equivalents
   (course_home for any other course, the user home page, a topic page, a
   form page) all show the same gap.

2. **Page horizontal padding.** Every learner-facing page has horizontal
   padding from a single source (the page wrapper). At mobile width, content
   does not touch the screen edge. At desktop width, content sits inside a
   max-width well that's centered in the viewport.

3. **TOC sidebar / main content gap (large screens).** On the course topic
   and course form pages, when the TOC sidebar is open at `lg:` width or
   wider, there is visible horizontal space between the right edge of the
   sidebar panel and the left edge of the main content. The space comes
   from `gap-*` on a grid parent, not from margins on children.

4. **TOC sidebar / main content gap (small screens).** On smaller viewports,
   if the sidebar collapses to a stacked layout (not an overlay), there is
   visible vertical space between the sidebar block and the main content
   below it. If the sidebar is overlay-only on small screens (current
   behaviour), no extra spacing is needed because they aren't both in flow
   at once.

5. **Bottom-of-page breathing room.** On every learner-facing page,
   including pages with action buttons at the bottom (`course_topic.html`,
   `course_form.html`, `course_form_complete.html`, `course_finish.html`),
   there is consistent space below the last visible element before the
   viewport / footer ends. Pages do not end flush against the bottom of the
   visible area. This breathing room comes from the page wrapper, not from
   per-template `mb-*`.

6. **Anchor-link targets.** When an anchor link (`#some-id`) is followed,
   the target heading lands at a comfortable position below the top of the
   viewport — not flush against it. The mechanism is `scroll-mt-*` on the
   heading element (or a base CSS rule that applies it to headings with
   `id`s), not `scroll-padding-top` on `<html>`.

7. **No regression on existing functionality.** All existing student
   interface flows (browsing courses, registering, completing topics,
   filling forms, viewing scores, finishing courses) work exactly as
   before. The Alpine sidebar toggle still works (open/close, persistence
   to local storage, mobile overlay, backdrop click-to-close).

8. **No regression on other apps.** Educator interface, allauth pages,
   legal docs, and admin still render correctly.

## Non-functional requirements

1. **Convention adoption.** The conventions listed in
   `research_spacing.md` §9 are followed by the new wrapper, the sidebar
   layout, and the per-template cleanup.

2. **Responsive ramp.** Layout padding changes at most at two breakpoints,
   chosen from `sm:` and `lg:`. A given padding axis may use either or both
   of those breakpoints (e.g. `px-4 sm:px-6 lg:px-8` uses both;
   `py-8 lg:py-12` uses one). No `md:` or `xl:` overrides for layout-level
   padding.

3. **Spacing scale.** All layout spacing uses Tailwind's default scale
   (e.g. `4, 6, 8, 12, 16, 24`). No arbitrary values like `px-[14px]` for
   layout work.

4. **Component reuse.** The page wrapper is one cotton component, used by
   all learner-facing pages. New pages added later use it by default.

5. **Tests.** Per project conventions, tests must not assert on CSS classes
   or styling (see user memory `feedback_no_style_assertions.md`). Existing
   tests must continue to pass. The verification mechanism for the visual
   changes is playwright screenshots + manual review against the success
   criteria, not unit tests asserting on classes.

6. **Brand alignment.** Consult the `brand-guidelines` skill during
   implementation to confirm the chosen padding/gap values match the
   FreedomLS visual system.

## Success criteria

The spec is satisfied when **all** of the following hold:

1. Visiting `http://127.0.0.1:8000/courses/standard-markdown-demo-finance/`,
   `…/1/`, and `…/2/` (or equivalent on the DemoDev site) at desktop,
   tablet, and mobile widths shows:
   - Comfortable, consistent gap between the site header and the page
     heading.
   - On the topic/form pages: visible space between the open TOC sidebar
     and the main content at desktop width.
   - On the bottom-of-page pages: visible space beneath the action buttons.
2. The same visual comparison on at least one page from each of: home,
   educator interface, allauth (login), legal doc — shows nothing is
   visually broken by the change.
3. Searching the learner-facing templates (`student_interface/templates/`,
   `accounts/templates/`, `base/templates/`) shows that the page wrapper
   is the source of horizontal page padding and top/bottom page padding —
   per-template `mt-*`, `mb-*`, `px-*`, `py-*` for outer padding has been
   removed.
4. Searching `_base.html` shows it no longer carries horizontal page
   padding or a `max-w-*`/`py-*` content well on `<main>`. Equivalently, in
   any ancestor chain reaching a learner page's content, exactly one
   ancestor owns horizontal padding and exactly one owns top/bottom
   padding — and that ancestor is the page wrapper (with the single
   exception of `_base_interface.html`, which owns horizontal padding for
   its own sidebar/header-bar shell because the page wrapper sits inside
   its right-hand content column rather than around it).
5. The Alpine sidebar still toggles, persists open/closed state, and
   handles the mobile overlay correctly. Existing pytest/playwright tests
   for the student interface still pass.
6. Anchor-link navigation lands target headings with breathing room above
   them (manual check on a markdown page that has heading anchors).
7. No `scroll-padding-top` is added to `<html>` (verifies D1).
8. No new arbitrary spacing values appear in the diff — all spacing uses
   Tailwind's default scale (verifies non-functional req 3).

## Out of scope (explicit reminders)

- Sticky site header. **Not in this spec.** Future work; D1 leaves the door
  open by using `scroll-mt-*`.
- Sticky bottom action bar. **Not in this spec.**
- Footer top padding (research §6). The current site has no global footer
  in `_base.html` — only a debug badge — so this convention is unused for
  now. **Not in this spec.** If a footer is added later, it should own its
  own top breathing room.
- New visual design (colours, type scale, components). **Not in this spec.**

## Open questions

None. The earlier ambiguity about anchor-link offset (`scroll-padding-top` vs
`scroll-mt-*`) is resolved in D1.
