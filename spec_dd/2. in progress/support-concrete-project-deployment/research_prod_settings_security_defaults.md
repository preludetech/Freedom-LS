# Research: prod-settings security defaults (P0 items 1, 4 + SECRET_KEY + CONN_MAX_AGE)

Scope: validate the idea's P0 claims (`idea.md` items 1 and 4, plus the two extra
P0-class landmines — empty `SECRET_KEY`, missing `CONN_MAX_AGE`) against Django's
official deployment checklist, `manage.py check --deploy` system checks, Django
security docs, libpq `sslmode` docs, and Caddy's own header-trust docs. Grounded
against this worktree's `config/settings_prod.py` and `config/settings_base.py`.

## Summary verdict (see full table at the end)

All five claims are **directionally correct**; none needs reversing. Four earn a
**nuance-added** verdict because the primary sources sharpen or partially correct the
idea's framing:

- **Claim 1** (`SECURE_PROXY_SSL_HEADER`): confirmed as the right fix, but the "Caddy
  always overwrites the header" assumption needs a caveat once Caddy's `trusted_proxies`
  is configured to recover `CF-Connecting-IP` (required for real client IPs) — that
  setting changes header-trust behavior for **all** `X-Forwarded-*` headers, not just
  `X-Forwarded-For`. Should be a **hard default in `settings_prod.py`**, not env-guarded.
- **Claim 2** (`sslmode=prefer`, `.env.example` ships `disable`): the libpq semantics
  are confirmed exactly as stated, and `prefer`'s "silent plaintext fallback" caveat is
  real — but the more useful finding is that for **same-host containerised Postgres**,
  `sslmode` is close to security-theatre either way (no network segment for a MITM to
  sit on); the control that actually matters is **not publishing Postgres's port to the
  host**. `.env.example` should say so plainly, and the checklist item should be scoped
  to external/managed DBs.
- **Claim 3** (hard-fail on empty `SECRET_KEY`): Django **already** raises
  `ImproperlyConfigured` on an empty `SECRET_KEY` — but lazily, on first
  `settings.SECRET_KEY` access, not at process boot. That's a real gap: Gunicorn can
  report "up" and a naive liveness probe can pass while every real request 500s. The
  idea's proposed fix (`os.environ["SECRET_KEY"]`, `KeyError` at settings-**import**
  time) is confirmed as strictly better because it moves the failure earlier —refine
  the rationale accordingly.
- **Claim 4** (`CONN_MAX_AGE`): confirmed, with a concrete recommended range
  (60–300s) and a hard recommendation to ship `CONN_HEALTH_CHECKS = True` alongside
  any non-zero value, not as an afterthought.
- **Claim 5** (scan for other `check --deploy` gaps): most of the obvious items
  (`ALLOWED_HOSTS`, `CSRF_TRUSTED_ORIGINS`, `SESSION_COOKIE_SECURE`,
  `CSRF_COOKIE_SECURE`, `SECURE_HSTS_*`, `SECURE_SSL_REDIRECT`, nosniff/referrer/COOP/
  X-Frame-Options) are **already set correctly** in `settings_prod.py` — not gaps. One
  genuinely missing control: nothing in the repo runs `check --deploy` itself, so a
  **present-but-weak** `SECRET_KEY` (short, low-entropy, or the `django-insecure-`
  autogenerated prefix) is never caught — the code fix in claim 3 only catches *empty*,
  not *weak*. Recommend wiring `check --deploy` into the entrypoint or CI as a release
  gate.

## Claim 1 — `SECURE_PROXY_SSL_HEADER` behind Caddy + Cloudflare (Full strict)

**Django's own warning** (docs.djangoproject.com, `SECURE_PROXY_SSL_HEADER` reference):

> Modifying this setting can compromise your site's security. Ensure you fully
> understand your setup before changing it.
>
> You should only set this setting if you control your proxy or have some other
> guarantee that it sets/strips this header appropriately. If you set this to a header
> that your proxy allows through from the request unmodified (i.e. a header that can be
> spoofed), you are allowing an attacker to pretend that any request is secure, even if
> it is not.

Django's security topic guide restates the exact conditions under which it's safe to
set — paraphrased, all three must hold:

1. Your Django app is behind a proxy you control or fully trust.
2. **The proxy strips the header from incoming (client-supplied) requests** — an
   end-user cannot inject `X-Forwarded-Proto` and have it survive to Django.
3. **The proxy sets the header itself, and only when the original request came in over
   HTTPS.**

This confirms the idea's framing precisely: `SECURE_PROXY_SSL_HEADER =
("HTTP_X_FORWARDED_PROTO", "https")` is the correct fix for the "redirect-loop
landmine," and the safety condition is exactly the trust model the idea already
names. Given `SECURE_SSL_REDIRECT = True` is already set (`settings_prod.py:16`) with
no `SECURE_PROXY_SSL_HEADER`, Django's `is_secure()` currently always returns `False`
behind Caddy, so `SecurityMiddleware` 301-redirects every request to `https://` —
which the proxy again forwards as plain HTTP — an infinite loop. This matches item
1's description exactly; no correction needed there.

### What must be true for the Caddy + Cloudflare (Full strict) setup specifically

Caddy's own docs (`reverse_proxy` directive reference) confirm the mechanism:

> Caddy sets or augments `X-Forwarded-For`... sets `X-Forwarded-Proto`... [and] sets
> `X-Forwarded-Host`. By default, [it] ignores [these headers'] values from incoming
> requests, to prevent spoofing.

That default behavior — Caddy freshly computes and overwrites these headers based on
its own view of the connection it terminated, discarding whatever a client sent — is
exactly condition (2)/(3) above, satisfied out of the box.

**The nuance the idea should add:** that default changes once `trusted_proxies` is
configured. The FLS/downstream plan explicitly wants Caddy's `trusted_proxies` set to
Cloudflare's IP ranges so it can recover the real client IP from `CF-Connecting-IP`
(per the downstream Cloudflare research the idea references). Caddy's own docs say:

> If trusted, the `reverse_proxy` handler will proxy and augment the sensitive
> `X-Forwarded-*` request headers.

That phrasing — "augment" rather than "overwrite" — is ambiguous about whether, once a
source is trusted, `X-Forwarded-Proto` is still freshly set by Caddy from its own
TLS-terminated connection state, or whether an upstream-supplied value (Cloudflare's)
is passed through unmodified. Both readings are actually **safe here**, because:

- Cloudflare's edge already sets `X-Forwarded-Proto` correctly (reflecting the
  original browser-to-edge scheme) — Cloudflare's edge is itself a trusted TLS
  terminator in **Full (strict)** mode.
- Caddy's own hop to Cloudflare is TLS-terminated by Caddy (Full strict requires a
  valid cert at the origin), so even if Caddy freshly computes the header itself
  rather than passing Cloudflare's through, it will still correctly say `https`.

So the practical outcome is safe under either interpretation — but it is **not
something to assume**; it should be verified empirically against the actual Caddyfile
(e.g. `curl` the origin with a forged `X-Forwarded-Proto: http` through a path that
bypasses Cloudflare, if such a path is reachable — see next bullet — and confirm
Django still sees it as insecure/redirects).

**Concrete preconditions to state explicitly** (the idea should enumerate these,
not just gesture at "the trust model"):

1. **Origin firewalled to Cloudflare's published IP ranges** on 80/443, so no client
   can reach Caddy directly and self-supply `X-Forwarded-Proto: https` while actually
   connecting over plain HTTP (or, worse, feed a value Caddy passes through unmodified
   under a broader `trusted_proxies` CIDR than intended).
2. **Cloudflare SSL/TLS mode = Full (strict)**, never Flexible or Full — Flexible means
   the Cloudflare→origin hop can be plain HTTP even though the edge told Cloudflare's
   client it was HTTPS, and (per the idea) combined with `SECURE_SSL_REDIRECT=True`
   this *is itself* a redirect loop from the opposite direction. Full (non-strict)
   accepts any cert (including self-signed/expired) at the origin — accept TLS but
   without real certificate validation, which is weaker than Full (strict) but not
   part of this specific header-trust argument.
3. **`trusted_proxies` scoped exactly to Cloudflare's published ranges**, not `0.0.0.0/0`
   or an over-broad private-network CIDR — otherwise anything inside that CIDR can
   supply trusted `X-Forwarded-*` headers.
4. **The Django/Gunicorn container is not reachable on any published port other than
   through Caddy** — no `ports:` mapping exposing the app container directly to the
   host network, so Caddy is the only path in.
5. **Caddy's own Caddyfile actually issues `header_up X-Forwarded-Proto {?}` correctly
   or relies on `reverse_proxy`'s default** — verify this isn't accidentally overridden
   by a custom `header_up` directive elsewhere in the Caddyfile.

### Hard default vs. env-guarded

The idea's "env-guardable if FLS wants it optional" framing should be resolved in
favor of a **hard default**, not an env toggle, given the idea's own architectural
premise (item 8): FLS is *never* deployed standalone, only behind a TLS-terminating
proxy, and `settings_prod.py` is exclusively the production settings module (dev uses
`settings_dev.py`). There is no legitimate deployment shape reachable through
`settings_prod.py` where the app is *not* behind such a proxy. Making this an env
toggle reproduces exactly the "silent default that bites every project" failure mode
this P0 section exists to eliminate — someone has to remember to flip it. Hard-code it
in `settings_prod.py` with a comment documenting the five preconditions above (so a
future operator changing the proxy topology has to consciously edit code, not silently
inherit a stale env default).

## Claim 2 — DB `sslmode` default (`prefer`, not `require`; `.env.example` ships `disable`)

**libpq's own semantics** (PostgreSQL docs, `libpq` SSL support, Table "SSL Mode
Descriptions"), confirmed verbatim:

| `sslmode` | Eavesdropping protection | MITM protection | Intent |
|---|---|---|---|
| `disable` | No | No | Don't want the overhead of encryption. |
| `allow` | Maybe | No | Don't care, but will pay for encryption if the server insists. |
| `prefer` | Maybe | No | Don't care about encryption, but wish to pay for it if the server supports it. |
| `require` | Yes | No | Want encryption; trust the network to route me to the right server. |
| `verify-ca` | Yes | Depends on CA policy | Want encryption and to trust the server's CA. |
| `verify-full` | Yes | Yes | Want encryption and to be sure it's the exact server I specify. |

This confirms the idea's characterisation exactly:

- **`prefer`'s "Maybe" eavesdropping protection is real and worth stating loudly.**
  `prefer` negotiates TLS opportunistically: if the server doesn't offer TLS (or
  anything strips/interferes with the negotiation), the client silently falls back to
  an unencrypted connection **with no error and no application-visible signal**. A
  project reading `sslmode=prefer` in a diff might reasonably believe "we're now
  encrypting the DB connection" when, against `postgres:17` with the default
  `ssl = off`, nothing changed.
- **`require`/`verify-full` is correctly scoped to external/managed DB.** `require`
  alone only guarantees encryption-in-transit against a passive eavesdropper — it does
  **not** verify server identity ("trust the network will make sure I always connect
  to the server I want," per the libpq docs), so on the public internet, `require`
  without `verify-ca`/`verify-full` gives no MITM protection. Managed Postgres
  providers (RDS, DigitalOcean, Neon, etc.) universally ship a signed cert, so
  `verify-full` is achievable and is the correct target for that topology —
  confirming the idea's split.
- **The specific first-boot failure mode is confirmed.** `postgres:17`'s stock image
  does not enable TLS (`ssl = off` in `postgresql.conf` by default; enabling it
  requires generating/mounting a cert+key and flipping `ssl = on`). If the *client*
  demands `sslmode=require` and the *server* has no TLS listener at all, the server
  simply cannot satisfy the negotiation and the connection is refused outright — this
  is not a graceful degrade, it's a hard connection failure. Shipping `require` as a
  reference default against a same-host containerised Postgres that nothing in the
  same PR enables TLS on is exactly the "silent breaker on first boot" the idea
  describes.

### Is `prefer` or `disable` the better *default*? (the tension, discussed honestly)

The idea's split — `prefer` as the **code** default (`config/settings_prod.py`'s
`sslmode` handling), `disable` as the **`.env.example`** shipped value — is defensible
but worth stating precisely, because `prefer` and `disable` produce an **identical
security outcome** against today's actual topology (same-host `postgres:17`, TLS
never enabled): both connect in plaintext. The only difference is:

- `prefer` attempts (and fails) an `SSLRequest` negotiation first, costing one extra
  round-trip, and — the real point of shipping it as the *code* default — if a future
  operator *does* enable TLS on the Postgres container (mounts a cert, flips
  `ssl = on`) without touching the app's `DB_SSLMODE` env var, the connection silently
  upgrades to encrypted with zero code change. That's a genuine, if narrow, forward-
  compatibility win for `prefer` over `disable` as the *code*-level default.
- `disable` in `.env.example` is the more **honest** value for what the shipped
  reference stack actually does today — it doesn't claim a security property
  (encryption) that isn't present, and it documents the true state of the connection
  rather than inviting a false sense of "we set sslmode so we're encrypted" during a
  security review.

**The honest tension:** `docs/deployment-security-checklist.md:22` (per the idea)
requires encrypted DB connections, and neither `prefer` nor `disable` *delivers* that
against the shipped same-host Postgres topology — `prefer` only delivers it if a
separate, currently-nonexistent piece of work (enabling TLS inside the `postgres:17`
container: generating a cert, mounting it, setting `ssl = on`) is *also* done. Shipping
`prefer`/`disable` without also either (a) doing that TLS-enablement work, or (b)
narrowing the checklist item's scope to "encrypted DB connections **when the DB is
external to the app's own trusted network boundary**," leaves the checklist item
formally unsatisfied for the default topology. The idea's fix is the right *pragmatic*
move (don't ship a first-boot breaker to chase a checklist box), but the checklist doc
itself should be corrected in the same pass to say so explicitly, rather than left
silently unmet.

**Also worth stating as the control that actually matters here:** for a same-host
containerised Postgres reached over the Docker Compose internal network, there is no
network segment external to the host for a MITM to sit on — the traffic between app
and DB containers never leaves the Docker host's own network stack (bridge or overlay,
depending on Compose networking mode). The security control doing real work in this
topology is **not publishing Postgres's `5432` to the host's public interface at all**
(no `ports:` mapping on the `db` service beyond an internal Compose network alias) —
that closes off the only realistic attacker vantage point, independent of `sslmode`.
This is worth a line in `.env.example` alongside the `DB_SSLMODE` documentation, since
it's the mitigation that's actually load-bearing for the shipped topology, and the
idea's proposed `.env.example` prose should say so rather than implying `sslmode`
alone is the relevant control.

## Claim 3 — Hard-fail on empty/missing `SECRET_KEY`

**Confirmed with an important correction to the "silent boot" framing.**

Django's `SECRET_KEY` reference doc states: "Django will refuse to start if
`SECRET_KEY` is not set." Checking where that's enforced (`django/conf/__init__.py`,
`LazySettings.__getattr__`):

```python
elif name == "SECRET_KEY" and not val:
    raise ImproperlyConfigured("The SECRET_KEY setting must not be empty.")
```

This check already exists in Django itself and already fires for `SECRET_KEY = ""`
(exactly what `os.getenv("SECRET_KEY", "")` produces when the env var is unset). **So
Django does not, in fact, silently accept an empty key forever** — but the check is
**lazy**, wired into attribute access on the settings object, not into settings
*loading*. It only fires the first time some code actually reads
`settings.SECRET_KEY`.

**Why that timing matters in practice:** nothing in Django's own startup path (app
registry population, URL conf loading, WSGI app construction, `manage.py check`
without `--deploy`) necessarily touches `settings.SECRET_KEY` before the first real
request. Session and CSRF machinery only read it lazily, when a value actually needs
signing (creating a session cookie, issuing a CSRF token). Practical consequence:

- Gunicorn boots, all workers report ready.
- A shallow liveness probe (or any health endpoint that doesn't touch
  sessions/CSRF/signing — see the idea's own P1(5) liveness/readiness split) returns
  200 and looks perfectly healthy.
- The **first real page load that sets a session cookie or renders a CSRF token**
  throws `ImproperlyConfigured` as an unhandled 500 — which, depending on
  `ADMINS`/error-reporting wiring, may or may not be loudly surfaced to an operator.

That is a real, if narrower, version of the "silent misconfigured prod" risk the idea
names — not silent forever, but silently *green* at the process/orchestration level
while broken for actual users, which is arguably worse for on-call diagnosis than a
crash loop.

**The idea's proposed fix (`os.environ["SECRET_KEY"]`) is confirmed as strictly
better, with a sharper rationale than "fail loud instead of silent":** `os.environ[...]`
raises `KeyError` at the moment `config/settings_prod.py` is *imported* — which happens
during Gunicorn's own boot (loading the WSGI application), before any worker reports
ready. That moves the failure from "lazy, first-request, possibly-invisible-to-health-
checks 500" to "process fails to start, container crash-loops visibly in `docker
compose ps`/`docker logs`." Recommend stating the fix in exactly these terms in the
idea (fail-fast-at-import vs. fail-lazy-at-first-signing-use), since it's a stronger
argument than "empty default is bad practice."

**`check --deploy` and weak-but-present keys:** the `security.W009` check (SECRET_KEY
reference/checks doc) is:

> Your `SECRET_KEY` has less than 50 characters, less than 5 unique characters, or it's
> prefixed with `'django-insecure-'` indicating that it was generated automatically by
> Django.

This only runs (and only *can* run) against a non-empty key — an empty key never
reaches W009 because `LazySettings.__getattr__`'s `ImproperlyConfigured` fires first,
during the check framework's own attempt to read `settings.SECRET_KEY`. So the code-
level fix in this item (hard-fail on *empty*) and `check --deploy`'s W009 (catches
*weak-but-present*, e.g. a short key, low-entropy key, or an accidentally-committed
`django-insecure-...` autogenerated default) are complementary, not overlapping —
fixing the empty-default gap does **not** give you W009's protection. See Claim 5 for
the concrete recommendation (wire `check --deploy` into the entrypoint/CI).

**Multi-env/CI nuance:** `os.environ["SECRET_KEY"]` means *every* process that imports
`config.settings_prod` — prod, staging (if staging reuses `settings_prod` with a
different `HOST_DOMAIN`/DB per the idea's `COMPOSE_PROJECT_NAME` multi-env-on-one-host
plan), and any CI job that runs `manage.py check --deploy` or migrations against prod
settings as a pipeline gate — must have its own `SECRET_KEY` env var set. Recommend:
staging and prod each get their **own**, independently generated key (never share
prod's actual key with staging, since a staging compromise would then also compromise
prod sessions/tokens); CI pipelines that need to import `settings_prod` (e.g. to run
`check --deploy` as a release gate) should set a throwaway pipeline-secret key that
still satisfies W009's entropy/length bar, rather than a trivial placeholder — a CI-
only weak key would make `check --deploy` itself report a false-clean W009 result if
the pipeline's dummy key happens to also be short/low-entropy, defeating the point of
running the check at all.

## Claim 4 — `CONN_MAX_AGE` for persistent DB connections

**Confirmed, with a concrete recommended range.**

Django's `CONN_MAX_AGE`/persistent-connections docs confirm:

- Default `0`: connection closed at the end of every request (Django's historical
  behavior) — exactly what `settings_prod.py`'s `DATABASES` block does today, since
  `CONN_MAX_AGE` is absent entirely (not merely defaulted — absent, same effective
  result).
- `None`: unlimited persistent connections.
- Positive integer: connection lifetime in seconds; Django closes and reopens once a
  connection exceeds that age.
- **Explicit warning on connection-count exhaustion:** "your database must support at
  least as many simultaneous connections as you have worker threads" — i.e. this
  scales with the deployment's actual concurrency model, not just request rate.
- **`CONN_HEALTH_CHECKS`** (default `False`): "existing persistent database
  connections will be health checked before they are reused in each request... If the
  health check fails, the connection will be reestablished without failing the
  request" — explicitly the mitigation for a DB restart (e.g. `docker compose restart
  db`, or a redeploy) leaving stale sockets behind that would otherwise surface as a
  broken-pipe error on the next request that reuses them.

**Interaction with Gunicorn `gthread` workers, confirmed:** Django's DB connections
are per-thread, not shared across an internal pool — each `gthread` worker thread that
touches the DB holds its own connection. With `CONN_MAX_AGE = 0` (today's absent
setting, effective default), that connection is opened and torn down on **every
single request**, paying full TCP-connect + Postgres auth-handshake cost each time —
exactly the "fresh DB connection per request" cost the idea names. With a positive
`CONN_MAX_AGE`, each thread's connection persists across multiple requests, amortising
that cost — but the total number of simultaneous open connections is bounded by
`gunicorn workers × threads per worker`, all of which count against Postgres's
`max_connections` (default 100) for as long as `CONN_MAX_AGE` keeps them alive, *even
when idle between requests*. Too-high a value (or `None`) on a small VPS risks
accumulating idle connections that starve headroom needed for `psql`/`pg_dump`
backups/migrations/admin connections running alongside the app.

**No PgBouncer is part of this architecture** (same-host containerised Postgres, no
pooler mentioned in the idea or the settings files) — so the "risk of a too-high value"
is bounded purely by `gunicorn_workers × threads × CONN_MAX_AGE` idle-connection
accumulation, not by pooler-mode incompatibilities. If connection-count pressure ever
does become real at larger scale, Django's own docs flag that PgBouncer in
**transaction-pooling mode requires `DISABLE_SERVER_SIDE_CURSORS = True`** in the
`DATABASES` `OPTIONS`, since server-side cursors are connection-local and transaction
pooling can hand different transactions different underlying connections — worth a
one-line forward-reference in the fix's comment, not something to build now.

**Concrete recommended default:** `CONN_MAX_AGE` in the **60–300 second** range
(e.g. `os.getenv("DB_CONN_MAX_AGE", "60")`), **always paired with
`CONN_HEALTH_CHECKS = True`** the moment `CONN_MAX_AGE` is non-zero — the health-check
cost is one cheap validation query only when a persistent connection is actually being
reused, and it directly guards against the DB-restart/redeploy scenario that's
otherwise a real footgun on a single-VPS stack where `docker compose up -d db` restarts
happen during routine maintenance. Avoid `None` (unlimited) as the shipped default —
reserve it for a deliberate opt-in once the deployment has grown enough to justify a
pooler, at which point PgBouncer (not raising `CONN_MAX_AGE` further) is the standard
next step, not this setting.

## Claim 5 — Other high-value `check --deploy` items for this topology

Cross-checked `settings_prod.py` against the full `security.*` check list
(`django.core.checks`, `--deploy` tag) and Django's deployment checklist. Most of the
obvious candidates are **already handled correctly** in this repo's reference config —
worth stating so the idea doesn't imply they're gaps:

| Setting | Status in `settings_prod.py` | Check code |
|---|---|---|
| `ALLOWED_HOSTS` | Set (`:10`) — non-empty | W020 pre-empted |
| `CSRF_TRUSTED_ORIGINS` | Set, scheme-qualified `https://{HOST_DOMAIN}` (`:13`) | n/a (not a `check --deploy` item, but required since Django 4 for cross-origin unsafe-method POSTs — correct given Cloudflare/Caddy sit in front) |
| `SESSION_COOKIE_SECURE` | `True` (`:26`) | W010/W011/W012 pre-empted |
| `CSRF_COOKIE_SECURE` | `True` (`:29`) | W016 pre-empted |
| `SESSION_COOKIE_HTTPONLY` | `True` (`:27`) | W013/W014/W015 pre-empted |
| `SECURE_SSL_REDIRECT` | `True` (`:16`) | W008 pre-empted (but see Claim 1 — currently loops without the proxy-header fix) |
| `SECURE_HSTS_SECONDS`/`INCLUDE_SUBDOMAINS`/`PRELOAD` | Set, env-configurable, conservative defaults (1hr, both `False`) (`:19-23`) | W004/W005/W021 pre-empted; the conservative rollout (short duration first, ramp later) matches Django's own HSTS rollout guidance |
| `SECURE_CONTENT_TYPE_NOSNIFF` | `True` (`:34`) | W006 pre-empted |
| `SECURE_REFERRER_POLICY` | Set (`:35`) | W022/E023 pre-empted |
| `SECURE_CROSS_ORIGIN_OPENER_POLICY` | Set (`:36`) | E024 pre-empted |
| `X_FRAME_OPTIONS` | `"DENY"` in prod (`:37`, overriding base's `SAMEORIGIN`) | W019 pre-empted |
| `DEBUG` | `False` (`:8`) | W018 pre-empted |
| `DEFAULT_HASHING_ALGORITHM` | unset → Django default (`sha256`) | E100 pre-empted |

**One dependency worth flagging explicitly:** `SECURE_HSTS_SECONDS` is only
*meaningful* once Claim 1's fix lands — `SecurityMiddleware` only emits the
`Strict-Transport-Security` header on responses it considers secure, and today (no
`SECURE_PROXY_SSL_HEADER`) Django never considers any request secure behind Caddy, so
the already-correct HSTS settings are currently inert in practice, not just the
redirect. Not a new gap — just worth sequencing HSTS verification *after* Claim 1
lands, in the same PR/test pass.

### Genuinely missing items

1. **Nothing runs `check --deploy` anywhere in the repo** (no entrypoint step, no CI
   job, no pre-deploy hook found). This is the concrete gap Claim 3's analysis
   surfaces: hard-failing on an *empty* `SECRET_KEY` does not catch a *present but
   weak* one (short, low-entropy, or a leftover `django-insecure-` autogenerated
   value pasted into `.env` by mistake) — only W009 catches that, and W009 only runs
   under `--deploy`. **Recommend:** add `python manage.py check --deploy
   --fail-level WARNING` (or equivalent) as a release gate — either in the Docker
   `docker-entrypoint.sh` before Gunicorn starts, or as a CI step against the prod
   settings module with a pipeline-secret `SECRET_KEY` — so a weak-but-present key,
   or any other `security.*` regression, blocks a deploy instead of shipping quietly.
2. **`ADMINS`/`SERVER_EMAIL` are not set** anywhere in `settings_base.py` or
   `settings_prod.py`. Not a `check --deploy` item and not urgent (Sentry is already
   planned in P3 and supersedes this), but worth a one-line footnote: until Sentry
   lands, Django's built-in `AdminEmailHandler` (wired to the `django.request` logger
   at `ERROR` level) is a no-op without `ADMINS` configured, so unhandled 500s are
   currently visible only via the LOGGING fix in P0(2) (stdout), not via any
   push-notification path. Low priority — flagging so it isn't assumed to already
   exist.
3. **Cloudflare's own edge-level HSTS ("Always Use HTTPS"/HSTS toggles on the free
   tier) can double up with Django's `Strict-Transport-Security` header.** Not a
   Django `check --deploy` item, but a cross-layer footgun worth deciding
   intentionally: if both Cloudflare and Django inject `Strict-Transport-Security`
   with different `max-age` values, browsers take the longer of the two (harmless in
   practice) but it's worth a one-line note in the deployment doc so it's a decision,
   not an accident, when HSTS is eventually ramped up toward preload eligibility.
4. **`USE_X_FORWARDED_HOST`** is unset (defaults `False`). For the current
   single-`HOST_DOMAIN` Caddyfile this is almost certainly fine as-is — Caddy forwards
   the original client `Host` unmodified for host-based routing to a single backend —
   but if the P3 template's multi-env-on-one-host plan ever routes multiple
   `HOST_DOMAIN`s through a shared front-end that rewrites `Host`, this would need
   revisiting. Flagging as a "verify, don't change" item, not a gap to fix now.
5. **`DB_HOST` default of `"localhost"`** (`settings_prod.py:54`,
   `os.getenv("DB_HOST", "localhost")`) is the same silent-default *pattern* as the
   old `SECRET_KEY` default, just lower severity — a misconfigured/missing env var
   fails loud immediately on first query (connection refused) rather than serving
   broken pages, so it's not P0-class, but it's the same class of "guessable string
   default" this section is otherwise trying to eliminate. Noting for completeness,
   not proposing to relitigate every `os.getenv(..., default)` in this file.

## Open questions for the human

1. **Claim 1:** should the settings-level comment documenting the five trust
   preconditions live in `settings_prod.py` itself, or in the P3 Caddyfile/deployment
   doc (or both)? The code fix (hard default) and the infra precondition (firewalling,
   `trusted_proxies` scope, no direct port exposure) live in different repos/artifacts
   per the idea's own artifact-vs-code split — worth deciding where the authoritative
   explanation of *why* the header is trustworthy lives, so it isn't lost between them.
2. **Claim 2:** does the idea want `docs/deployment-security-checklist.md:22`
   ("encrypted DB connections") explicitly rescoped to "external/managed DB
   connections only" in the same PR, given `prefer`/`disable` don't satisfy it for the
   shipped same-host topology? Left as an explicit recommendation above but not
   resolved — this is a doc-honesty decision, not a settings one.
3. **Claim 2:** should `.env.example`/the compose template also enforce "don't publish
   Postgres's port to the host" as a hard rule (e.g. no `ports:` on the `db` service in
   the P3 template), given that's the control doing the real security work in this
   topology? That's a P3 artifact decision, out of this research's scope, but flagged
   here since it directly bears on how much weight the `sslmode` default should carry.
4. **Claim 3:** confirm whether `check --deploy` should be a hard release gate
   (blocking deploy on any `security.*` warning) or advisory-only (logged but
   non-blocking) — the recommendation above assumes blocking, but that's a process
   decision for whoever owns the P3 CI/entrypoint work, not something this research
   can settle unilaterally.
5. **Claim 4:** confirm the target Gunicorn worker/thread count for the "small VPS"
   sizing referenced in the idea, so the 60–300s `CONN_MAX_AGE` recommendation and
   Postgres `max_connections` headroom can be sanity-checked against a real number
   rather than a generic range.
