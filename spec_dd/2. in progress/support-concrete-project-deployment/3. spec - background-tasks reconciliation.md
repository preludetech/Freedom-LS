## 1. Problem

The background-tasks story is inconsistent across the stack: `settings_prod.py:208–215` re-declares
`ImmediateBackend` with a live `# TODO @claude` (duplicating `settings_base.py:400–407`), while
`deployment.md:52` promises a `db_worker` container that **does not exist**. The **shipped default**,
the **opt-in primitive**, the **worker container**, and the **docs** must all agree.

## 3. Requirements

- **Shipped default stays `ImmediateBackend`, no worker container** (per the downstream V1 decision —
  must not be contradicted). Spec 1 §5.1-C removes the misleading TODO that made this look
  accidental.
- **Ship the durable backend as an importable, opt-in primitive:** Postgres as broker, **no
  Redis/Celery**. Use the **`django-tasks-db`** package (`django_tasks_db` in `INSTALLED_APPS` +
  `TASKS = {"default": {"BACKEND": "django_tasks_db.DatabaseBackend"}}`). A project turns it on with
  a settings flip, not a code port against the read-only submodule. Bonus: tasks visible in the
  Django admin.
  - **Name `django-tasks-db`, not `django-tasks`.** The durable backend was split out into
    `django-tasks-db` at `0.12.0`; `django-tasks` now ships only Immediate/Dummy backends.
    `django-tasks-db` works against Django 6 core's `django.tasks` (which
    `freedom_ls/webhooks/events.py:35` already imports via `default_task_backend.enqueue`).
  - **Operational specifics (baked into Spec 5's worker container):** the worker command is
    **`db_worker`**; enabling it is `INSTALLED_APPS` **plus its own `migrate`** (it ships tables),
    not just the `TASKS` flip; ship the **`prune_db_task_results`** retention job alongside the
    worker (un-pruned results table = unbounded growth on a small VPS); never run the worker via the
    `DEBUG` autoreload path in prod; **pin an exact version** (single PyPI release so far, `0.12.0` —
    itself why this is an opt-in primitive, not a shipped default).
- **Fix the docs** (`docs/product/deployment.md` / any playbook): describe this as an **upgrade**
  ("install `django-tasks-db`, flip `TASKS`, enable the worker service"), not a `db_worker` the
  shipped stack runs. Killing the doc-vs-reality lie is the real ask. *(Spec 4 also rewrites
  `deployment.md`; coordinate — land this doc fix before or together with Spec 4's rewrite. See
  `spec-order.md`.)*

Detail: `research_django_tasks_durable_backend.md`.

## 4. Success criteria

- **No tasks lie:** `ImmediateBackend` is the deliberate default; docs describe the `django-tasks-db`
  **upgrade** path; and a project can enable a real out-of-process `db_worker` with a settings flip +
  Spec 5's `profiles:` flag + migrate. (The `settings_prod.py` TODO removal itself is verified in
  Spec 1; the worker container is verified in Spec 5.)
