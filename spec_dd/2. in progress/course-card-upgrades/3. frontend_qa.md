# Frontend QA — Course-card upgrades & dashboard cleanup

This QA plan exercises every visible behaviour of the spec. Use Playwright MCP to walk through it. The goal is a hands-on verification that complements the unit/integration tests.

## Test environment setup

1. Pick a random port and start the dev server:
   ```
   PORT=$(.claude/fls/scripts/find_available_port.sh)
   uv run python manage.py runserver $PORT
   ```
   Base URL: `http://127.0.0.1:$PORT`

2. Build CSS once (only required if it has not been built since the theme.css change):
   ```
   npm run tailwind_build
   ```

3. Migrate and load demo content for the **DemoDev** site:
   ```
   uv run manage.py migrate
   uv run manage.py content_save demo_content DemoDev
   ```
   (DemoDev is the QA test data site. Always use it.)

4. Have a learner account that is registered for at least:
   - one course in the *Standard Markdown* demo (so it gets the `notes` icon),
   - one course where the user has zero progress (so the *Not started* card shows),
   - one course where the user has partial progress (so the *In progress* card shows),
   - one completed course (so the *Complete* card shows).

   The DemoDev fixtures already cover three of these via `functionality_demo_standard_markdown`, `functionality_demo_course_parts`, `functionality_demo_end_with_quiz`, and `functionality_demo_end_with_topic`. To complete a course quickly, register and step through every topic / form. To produce partial progress, step through one or two topics and stop.

5. Log in as the learner: visit `http://127.0.0.1:$PORT/accounts/login/`, sign in, and confirm you land at `/`.

1. In a fresh incognito window (or after logout), visit `http://127.0.0.1:$PORT/`.
2. **Expected**: you are redirected to the login page (URL contains `/accounts/login/` or similar).
3. Log in.
4. **Expected**: you are redirected back to `/` and see the authenticated dashboard.

## Workflow D — Three card states are visually distinct

On the dashboard, find the three card variants:

1. **In-progress card**:
   - Has the eyebrow text `In progress`.
   - Shows `Next up: <some item title>` (assuming the course has a ready/in-progress item).
   - Shows a numeric percentage label (e.g. `28%`) and a progress bar.
   - The progress bar's `<progress>` element has `role="progressbar"`, `aria-valuemin`, `aria-valuemax`, and `aria-valuenow` set.
   - Clicking the card navigates straight to the next ready/in-progress item (the same destination the legacy `Continue` button used).

2. **Not-started card**:
   - Has the eyebrow text `Not started`.
   - Shows the course title and nothing else (no progress bar, no description, no deadline).
   - On a desktop viewport (≥ 768px wide): clicking the card opens a modal with the course title, subtitle, description, table of contents, and a `Start` button. Closing the modal returns you to the dashboard with no navigation.
   - On a mobile viewport (< 768px wide): clicking the card navigates to `/courses/<slug>/preview/`. The page renders the same content as the modal (title, subtitle, description, ToC, `Start` button). The browser back button returns to the dashboard.

3. **Complete card**:
   - Has the eyebrow text `Completed`.
   - Shows the course title and a check-shape badge plus the word `Completed`. The state is conveyed by **text + shape**, not colour alone.
   - There is **no** progress bar (no 100%-filled bar).
   - Clicking the card navigates to `/courses/<slug>/finish/`.

## Workflow E — Per-course icon + accent stability

1. On the dashboard, locate the *Standard Markdown* course card (`functionality_demo_standard_markdown`).
2. **Expected**: the icon tile shows the `notes`-themed icon (a document-text glyph in the active icon set).
3. Locate any of the other demo courses (which have no `icon` set in frontmatter).
4. **Expected**: those cards show the default `course` icon (e.g. `academic-cap` on heroicons, `graduation-cap` on lucide/phosphor, `school` on tabler) — the actual glyph depends on `FREEDOM_LS_ICON_SET`.
5. Reload the page **3 times** and confirm: each course card renders the *same accent colour tile* (per-course identity is stable). The tile colour is one of the five role colours (primary / secondary / accent / info / success).
6. Visit `http://127.0.0.1:$PORT/courses/` (the all-courses page). **Expected**: every course card uses the same icon and accent colour as on the dashboard.

## Workflow F — Stretched link, focus ring, text selection

1. On any card variant, click anywhere inside the card body (not just the title).
2. **Expected**: the click activates the primary action (in-progress → continue page, not-started → preview, complete → finish page).
3. Tab into the card with the keyboard.
4. **Expected**: the focus ring wraps the **whole card**, not just the title link.
5. Try to highlight some text inside the card with click-and-drag.
6. **Expected**: text selection still works; selecting text does **not** trigger the card click.

## Workflow H — Modal-vs-page parity

1. Desktop viewport. On the dashboard, click a *Not started* card.
2. Note the modal contents — title, subtitle, description, table of contents, the labelled topics/forms, and the `Start` button.
3. Resize the viewport to mobile (< 768px). Visit `/courses/<the-same-slug>/preview/` directly.
4. **Expected**: the page contents match the modal contents item-for-item (same title, subtitle, description, same ToC item titles in the same order, same `Start` button label).

## Workflow I — Course-icon authoring sanity

1. Pick another demo course markdown file (e.g. `demo_content/functionality_demo_course_parts/course.md`).
2. Add `icon: drone` (a literal glyph not in `SEMANTIC_ICON_NAMES`).
3. Run `uv run manage.py content_save demo_content DemoDev`.
4. If the active icon set has `drone`, the card renders that glyph. If not, add `icon_fallback: phosphor:drone` and re-run `content_save`.
5. **Expected**: the card now renders the phosphor `drone` glyph regardless of the active icon set.
6. Add a typo: `icon: dron`.
7. Re-run `uv run manage.py content_save demo_content DemoDev`.
8. **Expected**: the command fails with a `ValidationError` whose message lists what `icon` should look like and what `icon_fallback` should look like.
9. Restore the file: revert to `icon: drone` (or remove the field) and re-run `content_save` to leave the demo content tidy.

4. **Expected**: the "Return to Home" button now reads **Return to Dashboard**, and clicking it lands at `/`.

## Done criteria

All workflows above pass. No console errors related to icons, missing CSS variables, or missing template files. `manage.py check` passes.
