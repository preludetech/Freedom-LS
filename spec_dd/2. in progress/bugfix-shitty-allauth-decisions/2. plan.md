# Implementation Plan: Remove the home-feature `next`-threading allauth customisations

This plan implements `1. spec.md` — a **surgical revert** of the custom `next`-threading the
`home` feature bolted onto the allauth / registration machinery. It is a *removal only*: no new
functionality, no new files, no migrations. The whole change is two source edits plus targeted
pytest cleanup across three test files.

## Ground truth (verified against the working tree)

- **`freedom_ls/accounts/middleware.py`** — `RegistrationCompletionMiddleware.__call__` currently
  ends (lines ~118–134) with a `next`-preservation block; imports `quote` (line 23) and
  `url_has_allowed_host_and_scheme` (line 30) are used **only** inside that block.
- **`freedom_ls/base/templates/partials/login_prompt.html`** — both `<c-button>` `href`s carry
  `?next={{ request.path|urlencode }}` (lines 3 and 8).
- **`accounts:complete_registration` is in `EXEMPT_URL_NAMES`** (middleware.py line 46) — the
  middleware never intercepts the completion view. This matters for a test decision below.
- **`login_prompt.html` is included only by `header_bar.html`** (line 22) — no other template
  or view references it; no e2e/playwright test exercises it.
- **The pre-existing "better registration" machinery stays**: `_safe_post_completion_redirect`
  (`accounts/views.py:60-68`, reads a POSTed/GET `next`, validates it, falls back to
  `LOGIN_REDIRECT_URL`) and `complete_registration_view` (`accounts/views.py:77-113`, re-emits
  `next`). The revert simply stops *feeding* these a `next` from the middleware; they degrade
  gracefully to `LOGIN_REDIRECT_URL`.

## Skills to apply

  `login_prompt.html` edit keeps the existing `{% url 'account_login' as login_url %}` +
  `<c-button href="…">` style and Django-style comments; nothing new is introduced.
- **`code-comments`** — use in **Task 1** (and as a final check on every code edit): when deleting
  the middleware's `# Preserve the in-flight destination…` comment block, do **not** leave behind a

- **`fls:testing`** — governs pytest conventions (`mock_site_context`, factories, `reverse()`,
  one-behaviour-per-test, and the "cleanup, not negation" refactor rule). Use throughout **Task 3**.

## Reuse / DRY note

Nothing new is introduced, so there is no new code to keep DRY. The point of the revert is to
**stop duplicating framework behaviour**: Django's `@login_required` already builds `?next=` for
anonymous hits on `initiate_course_access` / `apply`, and allauth already honours that `next` on
login. The hand-rolled middleware branch and the template `?next=` param duplicated that and are
being deleted. The surviving `_safe_post_completion_redirect` / `complete_registration_view` are
reused as-is (unchanged).

## Task 1 — Revert the middleware `next`-preservation block (spec R1)

**File:** `freedom_ls/accounts/middleware.py`

**1a. Replace the `next`-preservation block** at the end of `__call__`. Current tail:

```python
        # Preserve the in-flight destination through the forced completion step.
        # Prefer a validated ?next= that allauth already forwarded; otherwise
        # fall back to the intercepted path itself, which is always same-host so
        # the user never silently loses their destination. `request.path` (not
        # `get_full_path()`) avoids dragging unrelated query params along.
        candidate = request.GET.get("next")
        if not (
            candidate
            and url_has_allowed_host_and_scheme(
                candidate,
                allowed_hosts={request.get_host()},
                require_https=request.is_secure(),
            )
        ):
            candidate = request.path
        target = f"{reverse('accounts:complete_registration')}?next={quote(candidate)}"
        return redirect(target)
```

Replace the whole block (comment included) with the single vanilla line:

```python
        return redirect(reverse("accounts:complete_registration"))
```

**1b. Drop the now-unused imports** (grep-confirmed used only in the deleted block):

- Remove `from urllib.parse import quote` (line 23).
- Remove `from django.utils.http import url_has_allowed_host_and_scheme` (line 30).

Leave `from django.shortcuts import redirect` and `from django.urls import ... reverse` — both
still used by the surviving one-liner and elsewhere.

**1c. Leave the module docstring (lines 1–15) untouched** — it does not mention `next`.

**Do not touch** `_should_check`, `_is_exempt`, the caching helpers, or `EXEMPT_URL_NAMES`.

## Task 3 — Test cleanup

This is a revert, so the sequence is: make the Task 1 / Task 2 source edits, then bring the
suite back to green by removing/updating exactly the tests that asserted the deleted behaviour.
**Cleanup, not negation** — where an assertion tied to the removed `next`-threading is dropped,
delete the line outright; never replace it with a negative-existence check like
`assert "next=" not in response.url` (per the project testing norms, asserting the absence of a
value the code no longer produces proves nothing). Keep the assertions that describe surviving
vanilla behaviour (302 status, completion-URL prefix, `@login_required` → login redirect).

### 3a. `freedom_ls/accounts/tests/test_deferred_login.py`

**Delete these four tests:**

Three assert the removed middleware `next`-forwarding directly:

- `test_middleware_preserves_next_from_get_param` (lines ~41–66)
- `test_middleware_falls_back_to_request_path_when_no_next_param` (lines ~69–85)
- `test_middleware_rejects_off_host_next_and_falls_back_to_path` (lines ~88–114)

Plus `test_next_survives_complete_registration_step` (lines ~122–155). Decision rationale (this is
the spec's explicit "delete **or** rewrite" call, resolved here as **delete**): it POSTs *directly*

46), so it never exercised the middleware code at all — yet its docstring and inline comment
(lines ~143–144) narrate "the middleware forwards `next`", i.e. exactly the removed behaviour.

`freedom_ls/accounts/tests/test_complete_registration_view.py::test_next_param_honored_when_safe`
(lines 91–105), so deleting it loses no coverage and removes the stale narration. (An earlier
draft proposed keeping+reframing this test on the false premise that it was the *only* coverage of

survival").

## Success-criteria traceability (from spec §Success criteria)

1. Middleware ends with the single `return redirect(reverse("accounts:complete_registration"))`,
   `quote` / `url_has_allowed_host_and_scheme` imports gone → **Task 1**.
2. `login_prompt.html` buttons link to bare `{{ login_url }}` / `{{ signup_url }}`; wrapper &
   `{% url … as … %}` refactor retained → **Task 2**.
3. `header_bar.html`, `initiate_course_access` / `apply`, the completion view/template/adapter,
   and all `ACCOUNT_*` / `LOGIN_*` settings unchanged → no task edits them, and the **Task 4
   `git diff --stat` blast-radius guard** actively confirms only the five intended files changed.
4. Anonymous CTA click → vanilla allauth login; already-registered user carried back via stock
   `?next=`; registration-completion user lands on `LOGIN_REDIRECT_URL` (accepted trade-off) →
   guaranteed by removing the custom code (Task 1) + relying on surviving machinery. This
   post-revert behaviour is already regression-covered by existing tests (no new test needed):
   `test_registration_completion_middleware.py::test_user_with_incomplete_forms_redirected_to_completion`
   (after Task 3b, asserts the middleware redirects to the completion view **without** a `next`), and
   `test_complete_registration_view.py::test_posting_valid_data_persists_and_redirects` (completion
   POST with no `next` falls back to `LOGIN_REDIRECT_URL`).
5. No test asserts the removed behaviour; kept tests pass; stale docstrings/comments cleaned →
   **Task 3** (the stale middleware-narrating comment is removed by deleting
   `test_next_survives_complete_registration_step`, not by editing it).
6. `uv run pytest` green; no new functionality in the diff → **Task 4**.
