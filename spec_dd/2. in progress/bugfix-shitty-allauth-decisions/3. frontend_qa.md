# Frontend QA: vanilla allauth login after removing `next`-threading

This QA verifies the behaviour that survives the revert of the home-feature `next`-threading:

- The anonymous header shows **un-parameterised** Login / Sign-up links (no `?next=`).
- Clicking a login-required course CTA while anonymous still bounces through **vanilla** allauth
  login (Django-generated `?next=`) and, for an already-registered user, lands back on the CTA.
- A user forced through the registration-completion step lands on the site home
  (`LOGIN_REDIRECT_URL`), **not** their original destination — the accepted trade-off of the revert.

Use the **DemoDev** site for all test data. If any required user/course does not exist, create it
first via the `fls:qa-data-helper` agent (do not hand-write `.objects.create` calls).

## Start the server

```
PORT=$(.claude/fls/scripts/find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL: `http://127.0.0.1:$PORT`

All steps below are performed against the **DemoDev** site.

---

## Test 1 — Anonymous header Login/Sign-up links have NO `?next=` (the reverted markup)

This directly checks the `login_prompt.html` revert.

1. In a fresh browser context (not logged in), visit `http://127.0.0.1:$PORT/` (the home page).
2. Locate the **Login** link in the top header bar.
3. **Inspect its `href`.** Expected: exactly `/accounts/login/` — with **no** `?next=...` query
   string appended.
4. If the DemoDev site allows signups, locate the **Sign up** link in the header.
   **Inspect its `href`.** Expected: exactly `/accounts/signup/` — again with **no** `?next=`.

**Pass:** neither header link carries a `?next=` parameter.
**Fail:** either link's `href` contains `?next=` — the template revert is incomplete.

### 1b — Adversarial: deeper page, still no `?next=`

1. Still anonymous, visit `http://127.0.0.1:$PORT/courses/` (the catalogue).
2. Inspect the header **Login** link `href` again. Expected: still bare `/accounts/login/`, with
   **no** `?next=/courses/`. (The old behaviour appended the current path here — confirm it is gone.)

### 1c — Adversarial: signups disabled hides the Sign-up link

1. Ensure (or ask `fls:qa-data-helper` to configure) a site whose signup policy has
   `allow_signups=False`.
2. Visit `/` anonymously on that site.
3. Expected: the header shows the **Login** link but **no Sign up** link.

---

## Test 2 — Vanilla `@login_required` still carries an existing user back to the CTA

This checks that removing the custom middleware/template threading did not break the normal flow —
Django's `@login_required` builds `?next=` on its own and allauth honours it.

Preconditions (create via `fls:qa-data-helper` on DemoDev if missing):
- A **free-access** course with a known slug (referred to below as `<free-slug>`).
- A **student user** with a **complete** registration (email verified, no outstanding additional
  registration forms) and known credentials.

Steps:

1. In a fresh anonymous browser context, visit the course CTA directly:
   `http://127.0.0.1:$PORT/courses/<free-slug>/access/`.
2. Expected: you are redirected to the login page and the **URL now carries a Django-generated
   `next`** — i.e. `http://127.0.0.1:$PORT/accounts/login/?next=/courses/<free-slug>/access/`.
   (This `?next=` comes from `@login_required`, not from our template — that is the point.)
3. Log in with the existing student's credentials.
4. Expected: after login you are taken to the course access flow for `<free-slug>` (enrolled and
   dropped into the course), i.e. back to your original destination.

**Pass:** login redirect URL carries the correct `?next=`, and post-login you land on the course.
**Fail:** post-login you land on the home page instead of the course, or the login URL had no
`?next=`.

### 2b — Application-gated course variant

1. Repeat Test 2 with an **application-gated** course slug against the `apply` CTA:
   `http://127.0.0.1:$PORT/apply/<gated-slug>/`.
2. Expected while anonymous: redirect to `/accounts/login/?next=/apply/<gated-slug>/`.
3. After logging in as the existing student: you land on the **apply confirmation page** (a GET
   page, HTTP 200) — it must NOT auto-submit an application.

---

## Test 3 — Registration-completion trade-off (the deliberately-dropped destination)

This confirms the *accepted* consequence of the revert: a user who must pass through the forced
registration-completion step loses their original destination and lands on the site home.

Preconditions (create/configure via `fls:qa-data-helper` on DemoDev if missing):
- A site (or the DemoDev site) whose signup policy defines at least one **additional registration
  form** (so a freshly-signed-up user is "incomplete" and gets intercepted by
  `RegistrationCompletionMiddleware`).
- A free-access course slug `<free-slug>`.

Steps:

1. In a fresh anonymous browser context, visit `http://127.0.0.1:$PORT/courses/<free-slug>/access/`.
2. Expected: redirect to `/accounts/login/?next=/courses/<free-slug>/access/`.
3. From the login page, follow the **Sign up** link and register a brand-new account; complete
   email verification as required by the DemoDev flow.
4. After verification/login, expected: because the new account has an incomplete additional
   registration form, the middleware sends you to the **complete-registration** page.
   Confirm the URL is the completion page and that it does **not** carry a `?next=` for the course.
5. Submit the completion form with valid data.
6. Expected: you land on the **site home** (`/`, i.e. `LOGIN_REDIRECT_URL`) — **not** back on
   `/courses/<free-slug>/access/`.

**Pass:** after completing registration you land on the home page, not the course CTA.
**Fail:** you are carried back to the course CTA — that would mean the reverted `next`-threading is
still active.

> Note: if the DemoDev site has **no** additional registration forms configured, a new user is
> "complete" immediately and this interception never fires — in that case Test 3 is not applicable;
> record it as N/A rather than a failure.
