# Frontend QA: vanilla allauth login after removing `next`-threading

This QA verifies the behaviour that survives the revert of the home-feature `next`-threading:

- The anonymous header shows **un-parameterised** Login / Sign-up links (no `?next=`).
- Clicking a login-required course CTA while anonymous still bounces through **vanilla** allauth
  login (Django-generated `?next=`) and, for an already-registered user, lands back on the CTA.
- A user forced through the registration-completion step lands on the site home
  (`LOGIN_REDIRECT_URL`), **not** their original destination — the accepted trade-off of the revert.
- The earlier **"better registration"** feature — which the revert must **keep** intact — still
  works: name capture at signup, T&C/Privacy clickwrap consent recorded to `LegalConsent`, the
  dropped email-confirmation field, and the post-verification additional-registration form. The
  revert touches the signup → completion machinery, so this QA confirms it did not damage the
  kept feature.

Use the **DemoDev** site for all test data. If any required user/course does not exist, create it
first via the `fls:qa-data-helper` agent (do not hand-write `.objects.create` calls).

## Start the server

```
PORT=$(.claude/fls/scripts/find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL: `http://127.0.0.1:$PORT`

All steps below are performed against the **DemoDev** site.

---

## Test 1 — Anonymous header Login/Sign-up links have NO `?next=` (the reverted markup)

This directly checks the `login_prompt.html` revert.

1. In a fresh browser context (not logged in), visit `http://127.0.0.1:$PORT/` (the home page).
2. Locate the **Login** link in the top header bar.
3. **Inspect its `href`.** Expected: exactly `/accounts/login/` — with **no** `?next=...` query
   string appended.
4. If the DemoDev site allows signups, locate the **Sign up** link in the header.
   **Inspect its `href`.** Expected: exactly `/accounts/signup/` — again with **no** `?next=`.

**Pass:** neither header link carries a `?next=` parameter.
**Fail:** either link's `href` contains `?next=` — the template revert is incomplete.

### 1b — Adversarial: deeper page, still no `?next=`

1. Still anonymous, visit `http://127.0.0.1:$PORT/courses/` (the catalogue).
2. Inspect the header **Login** link `href` again. Expected: still bare `/accounts/login/`, with
   **no** `?next=/courses/`. (The old behaviour appended the current path here — confirm it is gone.)

### 1c — Adversarial: signups disabled hides the Sign-up link

1. Ensure (or ask `fls:qa-data-helper` to configure) a site whose signup policy has
   `allow_signups=False`.
2. Visit `/` anonymously on that site.
3. Expected: the header shows the **Login** link but **no Sign up** link.

---

## Test 2 — Vanilla `@login_required` still carries an existing user back to the CTA

This checks that removing the custom middleware/template threading did not break the normal flow —
Django's `@login_required` builds `?next=` on its own and allauth honours it.

Preconditions (create via `fls:qa-data-helper` on DemoDev if missing):
- A **free-access** course with a known slug (referred to below as `<free-slug>`).
- A **student user** with a **complete** registration (email verified, no outstanding additional
  registration forms) and known credentials.

Steps:

1. In a fresh anonymous browser context, visit the course CTA directly:
   `http://127.0.0.1:$PORT/courses/<free-slug>/access/`.
2. Expected: you are redirected to the login page and the **URL now carries a Django-generated
   `next`** — i.e. `http://127.0.0.1:$PORT/accounts/login/?next=/courses/<free-slug>/access/`.
   (This `?next=` comes from `@login_required`, not from our template — that is the point.)
3. Log in with the existing student's credentials.
4. Expected: after login you are taken to the course access flow for `<free-slug>` (enrolled and
   dropped into the course), i.e. back to your original destination.

**Pass:** login redirect URL carries the correct `?next=`, and post-login you land on the course.
**Fail:** post-login you land on the home page instead of the course, or the login URL had no
`?next=`.

### 2b — Application-gated course variant

1. Repeat Test 2 with an **application-gated** course slug against the `apply` CTA:
   `http://127.0.0.1:$PORT/apply/<gated-slug>/`.
2. Expected while anonymous: redirect to `/accounts/login/?next=/apply/<gated-slug>/`.
3. After logging in as the existing student: you land on the **apply confirmation page** (a GET
   page, HTTP 200) — it must NOT auto-submit an application.

## Test 3 — Registration-completion trade-off (the deliberately-dropped destination)

This confirms the *accepted* consequence of the revert: a user who must pass through the forced
registration-completion step loses their original destination and lands on the site home.

Preconditions (create/configure via `fls:qa-data-helper` on DemoDev if missing):
- A DemoDev `SiteSignupPolicy` with `require_terms_acceptance=True` and at least one **additional
  registration form** in `additional_registration_forms` (so a freshly-signed-up user is
  "incomplete" and gets intercepted by `RegistrationCompletionMiddleware`). Reuse the existing
  `freedom_ls.accounts.tests._completion_view_fixtures.PhoneNumberForm` fixture for the additional
  form unless the helper suggests otherwise.
- The `legal_docs/_default/terms.md` and `legal_docs/_default/privacy.md` documents must be
  **committed** to the worktree branch — the loader reads the git blob at `HEAD`, so uncommitted
  docs are invisible and the T&C checkboxes are suppressed. Commit them first if the signup form
  shows no consent checkboxes.
- A free-access course slug `<free-slug>`.

Steps:

1. In a fresh anonymous browser context, visit `http://127.0.0.1:$PORT/courses/<free-slug>/access/`.
2. Expected: redirect to `/accounts/login/?next=/courses/<free-slug>/access/`.
3. From the login page, follow the **Sign up** link and register a brand-new account; complete
   email verification as required by the DemoDev flow.
4. After verification/login, expected: because the new account has an incomplete additional
   registration form, the middleware sends you to the **complete-registration** page.
   Confirm the URL is the completion page and that it does **not** carry a `?next=` for the course.
5. Submit the completion form with valid data.
6. Expected: you land on the **site home** (`/`, i.e. `LOGIN_REDIRECT_URL`) — **not** back on
   `/courses/<free-slug>/access/`.

**Pass:** after completing registration you land on the home page, not the course CTA.
**Fail:** you are carried back to the course CTA — that would mean the reverted `next`-threading is
still active.

> Note: if the DemoDev site has **no** additional registration forms configured, a new user is
> "complete" immediately and this interception never fires — in that case Test 3 is not applicable;
> record it as N/A rather than a failure.

## Test 4 — "Better registration" surfaces still work after the revert (the KEPT feature)

The revert edits `RegistrationCompletionMiddleware` and the signup-adjacent `login_prompt.html`.
Those surfaces sit right next to the "better registration" feature the spec must **keep**. This
test re-confirms that feature is intact — it is not being changed, but we verify the revert did no
collateral damage. It reuses the **same signup + completion journey** you already walk in Test 3
(same DemoDev preconditions), adding content assertions at each step.

### 4a — Signup form still renders the better-registration fields

1. Anonymous, visit `http://127.0.0.1:$PORT/accounts/signup/` on DemoDev.
2. Inspect the rendered form. Expected fields: **email**, **first name** (required), **last name**
   (optional), **password**, **password confirmation**.
3. Expected **absent**: any email-confirmation / "confirm email" (`email2`) field — the revert must
   not have resurrected allauth's default signup form.

**Pass:** name fields present (first required, last optional), no email-confirmation field.
**Fail:** an email-confirmation field is shown, or the name fields are missing (would indicate the
`ACCOUNT_FORMS` signup form wiring regressed to the allauth default).

### 4b — T&C + Privacy clickwrap consent is captured

1. On the same signup page (with `require_terms_acceptance=True` and both legal docs committed),
   confirm **two separate, initially-unchecked** checkboxes are shown: **Accept Terms** and
   **Accept Privacy Policy**, each linking to `/accounts/legal/terms/` and
   `/accounts/legal/privacy/` respectively.
2. Attempt to submit with the checkboxes **unchecked**. Expected: signup is blocked (the form
   re-renders with a validation error; the account is not created).
3. Complete a real signup (this is the Test 3 signup — do it once and reuse it): fill valid data,
   check both boxes, submit, and complete email verification.
4. Verify consent was recorded. Ask `fls:qa-data-helper` (or inspect via the Django admin at
   `Accounts → Legal consents`) that **two** `LegalConsent` rows exist for the new user — one
   `document_type="terms"`, one `document_type="privacy"` — each with a non-empty `git_hash`,
   `document_version`, `timestamp`, and `consent_method="signup_checkbox"`.

**Pass:** both checkboxes shown and required; two audit-complete `LegalConsent` rows written.
**Fail:** checkboxes missing, submit succeeds with them unchecked, or consent rows are missing /
have empty `git_hash`.

### 4c — Additional registration form gates and persists (completion page content)

This looks at the **content** of the completion page that Test 3 only checked for its redirect.

1. Continue as the newly-verified user from 4b (the middleware has sent you to
   `/accounts/complete-registration/`).
2. Expected: the completion page renders the field(s) declared by the configured additional
   registration form (e.g. the **Phone Number** field for `PhoneNumberForm`).
3. Submit the completion form with valid data.
4. Expected: the submitted data is persisted (the user is now "complete") — re-visiting an
   internal page such as `/` no longer bounces you back to `/accounts/complete-registration/`.

**Pass:** the additional form's fields render, submit persists, and the completion gate no longer
fires afterwards.
**Fail:** the completion page is blank / errors, or the gate keeps re-firing after a valid submit.

### 4d — Adversarial: `require_name=False` makes first name optional

1. Ask `fls:qa-data-helper` to set the DemoDev `SiteSignupPolicy` `require_name=False` (or use a
   second site configured that way).
2. Visit `/accounts/signup/`. Expected: the **first name** field is now labelled optional and a
   signup submitted with an empty first name **succeeds**.
3. Restore `require_name=True` afterwards so later runs use the default.

**Pass:** with `require_name=False`, first name is optional and signup succeeds without it.
**Fail:** first name is still enforced as required.

> Note: 4b/4c depend on the DemoDev preconditions from Test 3 (terms acceptance on, legal docs
> committed, an additional registration form configured). If those cannot be configured, record the
> affected sub-tests as N/A rather than a failure — but 4a (form fields) and 4d (name toggle) should
> always be exercisable.

| Area touched by the change | Golden path | Side-effect / adversarial branch |
|---|---|---|
| `login_prompt.html` header links | Test 1 (no `?next=`) | 1b (deeper page), 1c (signups disabled) |
| Vanilla `@login_required` → `?next=` | Test 2 (free course) | 2b (application-gated apply page, no auto-POST) |
| Middleware forced completion | — | Test 3 (destination deliberately dropped → home) |
| "Better registration" kept intact | Test 4a/4b (signup fields, T&C/Privacy consent recorded), 4c (completion form gates/persists) | 4d (`require_name=False` toggle) |
