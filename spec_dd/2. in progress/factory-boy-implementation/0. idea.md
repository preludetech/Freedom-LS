# Factory Boy implementation

Currently, a lot of our tests make use of complicated fixtures for creating objects from the ORM. This is not DRY.

It also means that knowledge of what a valid object hierarchy looks like changes then the tests might not pick up on how those changes create bugs. 

## requirement

Make use of factory boy and faker in the tests:



### Pytest Fixtures
- Primary Use	Environment setup, resource management, and state isolation.
- Invocation	Injected into test functions automatically by name (dependency injection).

### Factory Boy

- Primary Use	Declarative and dynamic generation of complex test data/model instances.
- Invocation	Explicitly called within a test function (e.g., UserFactory(name="Test")).


## QA helpers 

Update the QA helper scripts so that they also make use of factoryboy factories.

## Keep it DRY and consistent 

Factories should be created in a consistent way. Factories used to create instances of models in a specific app should be defined within that app.

Always check what factories exist before making new ones.

## Docs

Update the documentation and the relevant claude skills, commands and agents so that factory_boy is used going forward.
