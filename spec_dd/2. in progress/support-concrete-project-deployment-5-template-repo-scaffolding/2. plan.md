## Approach & sequencing

These are **infra artifacts**, so classic red/green TDD does not apply file-by-file. Instead each
task has an **automated acceptance gate** (a command that must pass): `docker compose config`,
`docker build`, a Python-based container healthcheck probe, `caddy validate`/`caddy fmt`, and
`uv lock --check`. Author the artifact, then run its gate. There is **no pytest surface** for these
files (the template's pytest suite targets Django integration, not Dockerfiles).

Recommended order: **Task 1 → 2 → (3, 4 in parallel) → 5 → 6 → 7 → 8 → 9 → 10**. Task 2 (Dockerfile)
is the linchpin — compose (Task 3) and CI (Task 7) both reference the image it builds.

## Task 2 — Submodule-aware multi-stage `Dockerfile`

**File:** `Dockerfile` (new, repo root)

Three stages. The Docker **build context is the template repo root** with the submodule checked out
(`git submodule update --init`), which is what the CI checkout and a local build both provide.

### Stage A — `tailwind-builder` (pure Node)

```dockerfile
FROM node:22-slim AS tailwind-builder
WORKDIR /app
COPY package.json package-lock.json ./
RUN npm ci
# Bring in everything Tailwind's @source globs + @import theme.css files need to
# resolve. .dockerignore keeps .venv/node_modules/.git/staticfiles out, so this is
# bounded. The @imports pull:
#   - submodules/Freedom-LS/freedom_ls/themes/default/.../theme.css  (baseline)
#   - themes/custom/static/themes/custom/theme.css                   (active theme)
# and @source scans submodule templates/*.py + local templates/*.py.
COPY . .
RUN npm run tailwind_build          # → static/vendor/tailwind.output.css (pure npx CLI, no uv)
```

- **No `ARG FLS_THEME` needed in *this* stage** — the template selects its Tailwind theme via the
  `@import` in `tailwind.input.css`, not via an env var (unlike the FLS repo's
  `write_active_theme_css` flow). `FLS_THEME` matters at `collectstatic` time (Stage B), where it
  selects which theme's *static dir* is collected. Add a code comment noting this divergence so a
  future maintainer doesn't wire an unused build arg here.
- If a leaner cache is wanted later, narrow the second `COPY` to `tailwind.input.css themes/
  templates/ config/ apps/ manage.py submodules/`. Ship the simple `COPY . .` first; optimise only
  if build times demand it.

### Stage B — `python-builder` (uv + Django)

```dockerfile
FROM ghcr.io/astral-sh/uv:python3.13-bookworm-slim AS python-builder
ENV UV_COMPILE_BYTECODE=1 UV_LINK_MODE=copy
WORKDIR /app

# Phase 1 — third-party deps only (the big, slow, cache-friendly layer). Skip the
# freedom_ls path dependency here: its source (the submodule) isn't COPY'd yet, and
# installing it now would bust this layer on every app-code change.
RUN --mount=type=cache,target=/root/.cache/uv \
    --mount=type=bind,source=uv.lock,target=uv.lock \
    --mount=type=bind,source=pyproject.toml,target=pyproject.toml \
    uv sync --frozen --no-install-project --no-install-package freedom_ls

# Phase 2 — full source (incl. the submodule), then install freedom_ls + the project.
COPY . /app
COPY --from=tailwind-builder /app/static/vendor/tailwind.output.css /app/static/vendor/tailwind.output.css
RUN --mount=type=cache,target=/root/.cache/uv \
    uv sync --frozen

# Build-time collectstatic. WhiteNoise writes to local disk, so no cloud secrets are
# needed — but importing config.settings_prod hard-fails without SECRET_KEY and
# HOST_DOMAIN. Supply THROWAWAY build-scoped values; they never enter the runtime
# image or its env (this RUN's env does not persist).
ARG FLS_THEME=custom
# Throwaway, build-scoped placeholder — NOT a real secret; only satisfies the
# settings import so collectstatic can run. Never persisted to the runtime image.
ARG BUILD_ONLY_SECRET_KEY=throwaway-build-value  # pragma: allowlist secret
RUN SECRET_KEY="${BUILD_ONLY_SECRET_KEY}" \
    HOST_DOMAIN="build.invalid" \
    FLS_THEME="${FLS_THEME}" \
    DJANGO_SETTINGS_MODULE=config.settings_prod \
    AWS_STORAGE_BUCKET_NAME="" \
    uv run python manage.py collectstatic --noinput
```

Notes / things to **validate empirically** with a real `docker build` (call these out — they are the
two genuine unknowns):
1. **The two-phase `uv sync` with the editable submodule path dependency.** `--no-install-package
   freedom_ls` on Phase 1 is the mechanism that keeps the third-party layer cacheable without the
   submodule present. If uv rejects that flag combination under `--frozen`, fall back to a
   single-phase `COPY . /app && uv sync --frozen` (correct, just less cache-optimal) — do **not**
   block the build on cache perfection.
2. **`collectstatic` succeeds under `settings_prod`.** `AWS_STORAGE_BUCKET_NAME=""` forces local
   `FileSystemStorage` (no R2 creds at build). `CompressedManifestStaticFilesStorage` requires every
   referenced asset to exist — that is why Stage A's CSS is `COPY`'d in *before* this step.

**Reproducibility nit:** pin the uv base image to a minor version (e.g.
`ghcr.io/astral-sh/uv:0.9-python3.13-bookworm-slim`) rather than the floating
`:python3.13-bookworm-slim` — `uv.lock` pins the package graph but not the `uv` binary, so an
unpinned tag lets the builder itself drift over time. Cheap, no added complexity.

### Stage C — runtime (non-root)

> **⚠ Documented, signed-off deviation from spec §5.1 `--no-editable`.** §5.1's `--no-editable` +
> `.venv`-only slim copy assumes `freedom_ls`'s wheel carries its templates/static. It does **not**:
> `freedom_ls`'s `pyproject.toml` uses setuptools `[tool.setuptools.packages.find]` with **no
> `MANIFEST.in`, no `package-data`, no `include-package-data`**, so a non-editable build drops every
> non-`.py` file (templates, static) and the app fails to render/collect at runtime. Fixing that
> means editing `freedom_ls`'s packaging, which lives in the **read-only submodule** and is out of
> this spec's scope. Decision (confirmed): ship the **editable install + submodule-source-in-image**
> variant below — the two-phase `uv sync --frozen` (Stage B) stays **without** `--no-editable`, and
> Stage C copies the whole `/app` so `freedom_ls`'s editable `.pth` resolves against the present
> submodule source. Spec §5.1 has been softened to match. Revisit `--no-editable` only after
> `freedom_ls` packaging includes its template/static data.

```dockerfile
FROM python:3.13-slim-bookworm AS runtime
RUN useradd --system --create-home --uid 10001 appuser
WORKDIR /app
# Copy the whole built tree: .venv + collected staticfiles + config + submodule source.
# The submodule source must stay in the image so freedom_ls's editable install (.pth
# → submodules/Freedom-LS/) can resolve its templates and app modules at runtime.
COPY --from=python-builder --chown=appuser:appuser /app /app
ENV PATH="/app/.venv/bin:$PATH" \
    DJANGO_SETTINGS_MODULE=config.settings_prod \
    PYTHONUNBUFFERED=1
USER appuser
ENTRYPOINT ["/app/docker-entrypoint.sh"]
CMD ["gunicorn", "config.wsgi:application", "--bind", "0.0.0.0:8000", "--workers", "3"]
```

- **Non-root** runtime user (`appuser`, fixed uid). `--chown` on the copy.
- The runtime image needs **no `curl`/`wget`** — the compose healthcheck uses a Python one-liner
  (Task 3), keeping the image slim.
- `docker-entrypoint.sh` is created in Task 6 and copied in via the `COPY . /app` in Stage B (it
  lives at repo root); ensure it is `chmod +x` (Task 6 sets the mode, and `.dockerignore` must not
  exclude it).

**Gate:** `docker build -t fls-template:test .` completes; `docker run --rm fls-template:test ls
/app/staticfiles/vendor/tailwind.output.css` shows the collected CSS.

## Task 3 — `docker-compose.yml`

**File:** `docker-compose.yml` (new, repo root)

Services: `web`, `worker`, `task-prune`, `db`, `caddy`. Key requirements (spec §5.2):

```yaml
x-logging: &default-logging          # required per-service json-file cap; max-file is a no-op without max-size
  driver: json-file
  options:
    max-size: "10m"
    max-file: "3"

x-app-env: &app-env                  # DRY: web/worker/task-prune share the same env_file + image
  image: ${IMAGE:-ghcr.io/OWNER/REPO}:${TAG:-latest}
  env_file: [.env]
  depends_on:
    db:
      condition: service_healthy
  restart: unless-stopped
  logging: *default-logging

services:
  web:
    <<: *app-env
    command: gunicorn config.wsgi:application --bind 0.0.0.0:8000 --workers 3
    # No host ports — Caddy reaches it over the compose network. (Do NOT publish 8000.)
    healthcheck:
      # Hits readiness over plain HTTP on localhost. SECURE_REDIRECT_EXEMPT (=[r"^health/"])
      # keeps it from being 301'd to https, so no header gymnastics are needed.
      test: ["CMD", "python", "-c",
             "import urllib.request,sys; sys.exit(0 if urllib.request.urlopen('http://localhost:8000/health/readiness/').status==200 else 1)"]
      interval: 30s
      timeout: 5s
      retries: 3
      start_period: 20s

  worker:                            # enabled by default — the durable task backend is the prod default
    <<: *app-env
    command: python manage.py db_worker
    # A plain long-lived process — never the DEBUG autoreload path.

  task-prune:                        # retention job for the django_tasks_db results table
    <<: *app-env
    # Lightweight container-native "cron": run, then sleep a day, repeat.
    command: sh -c 'while true; do python manage.py prune_db_task_results; sleep 86400; done'

  db:
    image: postgres:17
    environment:
      POSTGRES_USER: ${DB_USER}
      POSTGRES_PASSWORD: ${DB_PASSWORD}
      POSTGRES_DB: ${DB_NAME}
    volumes:
      - pg_data:/var/lib/postgresql/data      # NAMED volume, never a host bind mount
    # NO ports: — 5432 is never published to the host (this is the real DB-security control)
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U ${DB_USER} -d ${DB_NAME}"]
      interval: 10s
      timeout: 5s
      retries: 5
    restart: unless-stopped
    logging: *default-logging

  caddy:
    image: caddy:2-alpine
    ports: ["80:80", "443:443"]
    environment:
      HOST_DOMAIN: ${HOST_DOMAIN}    # native {$HOST_DOMAIN} substitution in the Caddyfile
    volumes:
      - ./Caddyfile:/etc/caddy/Caddyfile:ro
      - caddy_data:/data             # cert persistence — losing this re-triggers ACME
      - caddy_config:/config
    depends_on:
      web:
        condition: service_healthy   # matches the service_healthy pattern used elsewhere
    restart: unless-stopped
    logging: *default-logging

volumes:
  pg_data:
  caddy_data:
  caddy_config:
```

- **`x-logging` anchor applied to every service** (via `<<: *app-env` for the three app services and
  explicitly on `db`/`caddy`) so a newly added service can't silently ship uncapped logs.
- **No `migrate`** in any `command:` or entrypoint — Task 6's entrypoint only waits for the DB.
- **`COMPOSE_PROJECT_NAME` + `--env-file`**: documented, not hard-coded. Prod and staging run on one
  host with `docker compose --env-file .env.prod -p fls_prod up -d` and `--env-file .env.staging -p
  fls_staging up -d`; distinct project names give isolated container/network/volume namespaces.
  Document this in the `.env.example` header and the deploy notes (Task 10). `COMPOSE_PROJECT_NAME`
  may also be set inside each env file.
- Replace `ghcr.io/OWNER/REPO` with a placeholder the bootstrap doc tells implementors to fill (kept
  as `${IMAGE}` so it is one env value, not an edit to committed YAML).

**Gate:** `docker compose config` (with a filled `.env`) resolves with no interpolation errors and
shows the log driver on every service.

## Task 4 — `Caddyfile`

**File:** `Caddyfile` (new, repo root)

```caddyfile
{
	# Global trusted-proxy config. Cloudflare fronts the origin (SSL Full (strict)),
	# so trust ONLY Cloudflare's published ranges — then Caddy honours X-Forwarded-*
	# / CF-Connecting-IP from CF and ignores them from anyone else. NEVER 0.0.0.0/0.
	# Refresh the ranges from https://www.cloudflare.com/ips/ when Cloudflare updates.
	# NB: the `static` keyword is valid HERE (global `servers > trusted_proxies`), not
	# inside a `reverse_proxy { }` block — putting it in the block is a parse error.
	servers {
		trusted_proxies static <CF_IPV4_RANGES...> <CF_IPV6_RANGES...>
	}
}

{$HOST_DOMAIN} {
	encode zstd gzip

	# Keep /health/* off the public edge: readiness reveals dependency state. The
	# compose healthcheck probes web:8000/health/readiness/ directly over the internal
	# network, so blocking it at the edge costs nothing. Caddy's fixed directive-order
	# priority places `respond` ahead of `reverse_proxy`, so this intercepts /health/*
	# before the catch-all proxy sees it — regardless of textual order in the file.
	@health path /health/*
	respond @health 404

	reverse_proxy web:8000
}
```

- `reverse_proxy` **only** — no `file_server`/`handle_path` for `/static/*` or `/media/*`
  (WhiteNoise serves static from the app; media is on R2). Caddy sets `X-Forwarded-Proto/-For/-Host`
  automatically; do not hand-set headers.
- **Leave `auto_https` on** (default) — automatic Let's Encrypt is the reason to use Caddy here.
- Domain via native `{$HOST_DOMAIN}` — the **single** domain var also read by `settings_prod.py`
  (`os.environ["HOST_DOMAIN"]`). One value per deployment, no separate Caddy `DOMAIN`.
- **Access log:** leave Caddy at its default stdout logging (container-native, matches the app's
  stdout logging). Because it is not logging to a file, no `Authorization`/cookie-scrubbing block is
  needed (resolves the research's open question in favour of stdout).
- Paste the **actual current Cloudflare IPv4 + IPv6 ranges** in place of the placeholders (from
  `https://www.cloudflare.com/ips/`, fetched at implementation time), with the refresh comment
  above — §6 requires real ranges in the artifact, not a `<...>` placeholder or a TODO.
- **`trusted_proxies` syntax gotcha:** the `static <ranges>` form belongs in the **global**
  `servers { trusted_proxies ... }` block (as above), **not** inside a `reverse_proxy { }` block —
  inside `reverse_proxy`, `trusted_proxies` takes bare CIDRs with no `static` keyword. The global
  placement is Caddy's recommended form and also enables client-IP parsing for logging/matchers.

**Gate:** `docker run --rm -v "$PWD/Caddyfile:/etc/caddy/Caddyfile:ro" -e HOST_DOMAIN=example.com
caddy:2-alpine caddy validate --config /etc/caddy/Caddyfile` passes. Optionally `caddy fmt`.

## Task 6 — `docker-entrypoint.sh` + `.env.example` + `.dockerignore`

**File:** `docker-entrypoint.sh` (new, repo root, `chmod 755`)

```bash
#!/bin/sh
set -eu
# Wait for Postgres to accept connections before starting the app/worker. This does
# NOT run migrations — migrations run as a one-off deploy step, never in an entrypoint,
# which keeps them concurrency-safe, failure-isolated, and rollback-compatible.
: "${DB_HOST:=db}" "${DB_PORT:=5432}"
until python -c "import socket,sys; s=socket.socket(); s.settimeout(2); s.connect(('${DB_HOST}', ${DB_PORT}))" 2>/dev/null; do
  echo "waiting for postgres at ${DB_HOST}:${DB_PORT}..." >&2
  sleep 1
done
exec "$@"
```

- Pure-`sh` (works in slim images), waits for DB TCP, then `exec "$@"` (runs the service `command:`).
- Set executable bit before commit so the `COPY` into the image preserves it.

**File:** `.env.example` (new, repo root) — document **every** var the settings + compose + Caddy
read. Model it on this FLS worktree's `.env.example` but make it the authoritative superset:

  `settings_prod.py` **and** the Caddyfile's `{$HOST_DOMAIN}`; one value only), `DJANGO_ADMIN_URL`
  (config; obscures the admin path), `FLS_THEME` (config, default `custom`),
  `WEBHOOK_ENCRYPTION_SALT` (secret; prod must set it — dev fallback is insecure).

## Task 7 — GHCR build-and-push CI

**File:** `.github/workflows/build-and-push.yml` (new)

Standard `docker/*-action` shape (research §"GHCR SHA-tagged build-and-push CI"). **Pin each action
to its current major at implementation time** — verify against each action's GitHub releases page;
do not copy the `@vN` numbers below verbatim, they drift (e.g. `actions/checkout`,
`docker/build-push-action`, `docker/metadata-action`, `docker/login-action` have all had major bumps).

```yaml
name: Build and push image
on:
  push:
    branches: [main]
  workflow_dispatch:
permissions:
  contents: read
  packages: write
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4          # pin to current major at implement time
        with:
          submodules: recursive          # REQUIRED — freedom_ls path dep + Tailwind @source
      - uses: docker/setup-buildx-action@v3
      - uses: astral-sh/setup-uv@v5        # so the drift check below has uv available
      - run: uv lock --check              # fail loudly on a stale lockfile (the --frozen build won't)
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - id: meta
        uses: docker/metadata-action@v5
        with:
          images: ghcr.io/${{ github.repository }}
          tags: |
            type=sha                       # immutable per-commit deploy/rollback unit
            type=ref,event=branch
      - uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
          # build-args: FLS_THEME=<slug>   # affects ONLY collectstatic's theme (Stage B). A real
          #   theme switch ALSO needs the @import in tailwind.input.css updated (Stage A compiles
          #   the theme's CSS from that hardcoded import, not from this arg) — see Task 2's note.
```

- `submodules: recursive` on checkout is **load-bearing** — without it the build context lacks the
  FLS submodule and every stage fails.
- `type=sha` is the deploy/rollback unit; the deploy references the image by SHA, not `latest`.
- The `uv lock --check` step closes a real gap: the Dockerfile's `--frozen` install trusts the
  lockfile as-is and would happily build+push an image against a `pyproject.toml`/`uv.lock` drift.
  Caveat: in this submodule/workspace layout uv may not fully verify lock freshness — if the check
  false-positives in practice, relax it to a warning rather than dropping the intent.
- Optionally add a `docker` ecosystem entry to `.github/dependabot.yml` to keep base images current
  (nice-to-have; mention, don't require).

**Gate:** `actionlint .github/workflows/build-and-push.yml` if available; otherwise a YAML-parse
check. (A live push to GHCR is out of scope for local validation.)

## Task 10 — Deploy notes (bootstrap + one-off migrate + staging/prod isolation)

**File:** extend `README.md` (a short "## Production deployment" section) — or a dedicated
`docs/deployment.md` if the reviewer prefers; keep it in the template repo.

Document the fill-in-and-run flow so the template is genuinely "deployment-ready":

1. `cp .env.example .env` and fill placeholders (domain, secrets, DB, R2, Sentry, `IMAGE`/`TAG`,

2. Build/pull the image (CI pushes a SHA tag to GHCR; set `IMAGE`/`TAG` accordingly).
3. **One-off migrate before starting services** (never in an entrypoint):
   `docker compose --env-file .env -p <project> run --rm web python manage.py migrate` — this
   creates the `django_tasks_db` tables so the `worker` has something to poll.
4. Bootstrap initial data: `... run --rm web python manage.py setup_initial_data` (the template's
   own first-run command).
5. `docker compose --env-file .env -p <project> up -d`.
6. **Staging + prod on one host:** repeat with a second env file and `-p` project name
   (`fls_prod` vs `fls_staging`) for isolated containers/networks/volumes.
7. Post-deploy checks (Sentry `sentry-debug/`, readiness via the internal network).

- **Which steps repeat vs. run once.** Steps 2, 3, 5, 7 repeat on **every** redeploy (pull the new
  SHA image → one-off migrate → `up -d` → check); steps 1 and 4 are **one-time** setup. Call this
  out so a routine redeploy doesn't skip the migrate step.
