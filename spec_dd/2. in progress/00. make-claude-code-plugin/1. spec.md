# Spec: Extract `.claude/` Configuration into an FLS Claude Code Plugin

## Overview

Turn the current `.claude/` directory contents into a reusable Claude Code plugin called `fls`. The plugin lives inside the FreedomLS repository. Concrete FreedomLS implementations install the plugin and get all FLS development conventions, skills, commands, agents, hooks, and scripts out of the box.

## Why This Matters

The `.claude/` directory contains 53 files (14 skills, 13 commands, 2 agents, 9 docs, 3 hooks, shell scripts, MCP config) that define FreedomLS development conventions. These are generic to any FreedomLS implementation but currently only exist in this repo. Extracting them into a plugin makes them:

- **Portable**: Any FreedomLS implementation gets the full development experience by enabling one plugin
- **Maintainable**: Conventions are updated in one place (the plugin) rather than copied across implementations
- **Clean**: The project's `.claude/` directory becomes minimal — just project-specific config and plugin enablement

## Key Design Decisions

### 1. No `@` Path References Inside the Plugin

**Decision**: Plugin files must not use `@` syntax for referencing other plugin files.

**Why**: In plugins, `@` resolves to the **project root**, not the plugin root. Different FreedomLS implementations have different project roots, so `@` references to plugin-internal files would break.

**How it works**: All internal references use `${CLAUDE_PLUGIN_ROOT}/path/to/file`. For example, the testing skill references docs via `${CLAUDE_PLUGIN_ROOT}/docs/factory_boy.md` instead of `@.claude/docs/factory_boy.md`.

### 2. `@` Paths for Project Root Scripts

**Decision**: Plugin commands/skills that invoke developer-facing scripts use `@/script_name.sh` to reference the project root wrapper scripts.

**Why**: Wrapper scripts at the project root contain project-specific customizations. The plugin must call the wrappers (not its own base scripts directly) so that all custom code is executed. `@` correctly resolves to the project root in this context.

### 3. No Hardcoded Absolute Paths

**Decision**: All paths within the plugin use `${CLAUDE_PLUGIN_ROOT}` or relative paths. No `/home/sheena/...` or similar.

**Why**: The plugin must work on any machine, in any FreedomLS implementation.

**Files requiring changes**:
- `git-worktree-setup/SKILL.md` — currently hardcodes `/home/sheena/workspace/lms/freedom-ls-worktrees/`. Generalize to describe the bare-repo + worktree pattern without specific paths.
- `qa-data-helper.md` — currently hardcodes absolute paths for agent memory search and session transcript paths. Use relative paths (`.claude/agent-memory/qa-data-factory/`) and remove session transcript references (those are user-specific).

### 4. Permissions Stay Project-Local (with Template)

**Decision**: The plugin does NOT ship permissions. The project's `.claude/settings.json` carries all allow/deny rules.

**Why**: Claude Code plugin `settings.json` currently only supports the `agent` key — not `permissions`. This is a platform limitation.

**Mitigation**: The plugin includes a `templates/settings.json` with recommended permissions. The `init` command merges these into the project's settings.json (adding to existing rules, never overwriting).

### 5. Hook Ownership Moves to Plugin

**Decision**: All hook definitions (PostToolUse, PreToolUse) move to the plugin's `hooks/hooks.json`. The project's `settings.json` removes its `hooks` section.

**Why**: If both define hooks, they run twice. The plugin is the single owner of FLS development hooks.

**hooks/hooks.json contents**:
```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/scripts/hooks/ruff_fix.sh"
          }
        ]
      },
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/scripts/hooks/post-edit-bandit.sh"
          }
        ]
      }
    ],
    "PreToolUse": [
      {
        "matcher": "Bash(git commit:*)",
        "hooks": [
          {
            "type": "command",
            "command": "uv run ruff check . && uv run mypy . && uv run pytest --tb=short -q"
          }
        ]
      },
      {
        "matcher": "Bash|Write|Edit",
        "hooks": [
          {
            "type": "command",
            "command": "${CLAUDE_PLUGIN_ROOT}/scripts/hooks/security-guard.sh"
          }
        ]
      }
    ]
  }
}
```

### 6. Credential Configuration via `.claude/fls.md`

**Decision**: Dev credentials (e.g., `demodev@email.com`) are configured in `.claude/fls.md`, a project-level file committed to git. Machine-specific overrides go in `.claude/fls.local.md` (gitignored).

**Why**: Dev credentials are project-scoped — all dev machines share the same test users set up by shared dev utilities. Unlike `userConfig` in plugin.json (which is per-user and prompted at enable time), `.claude/fls.md` is shared via version control.

**How skills access it**: Skills/commands that need credentials include an instruction to `Read .claude/fls.md` before proceeding. This adds one tool call per invocation but is explicit and reliable.

**Template `.claude/fls.md`**:
```markdown
# FLS Plugin Configuration

## Dev Credentials
- Admin email: demodev@email.com
- Admin password: demodev@email.com

## Project Settings
- Base URL: http://127.0.0.1:8000
```

### 7. Extensible Script Architecture

**Decision**: The plugin ships base scripts in `scripts/`. The `init` command generates wrapper scripts at the project root. Plugin commands call the project root wrappers (via `@/script_name.sh`), not the plugin's base scripts directly.

**Why**: Different FreedomLS implementations need to customize developer scripts. For example:
- `install_dev.sh` might have extra setup steps for a specific implementation
- Demo data loading might differ entirely between implementations
- Some scripts may not be needed at all

**How it works**:

1. **Plugin base scripts** (`fls-claude-plugin/scripts/`) contain the shared FLS logic (DB creation, migrations, base setup)
2. **`init` command** generates wrapper scripts at the project root from templates (`fls-claude-plugin/templates/wrapper_scripts/`)
3. **Wrapper scripts** call the plugin's base script and have a clearly marked section for project-specific additions:

```bash
#!/bin/bash
# install_dev.sh — Generated by fls plugin init
# Customize the "Project-specific setup" section below.

PLUGIN_DIR="$(dirname "$(readlink -f "$0")")/fls-claude-plugin"

# === FLS base setup ===
"$PLUGIN_DIR/scripts/install_dev.sh"

# === Project-specific setup ===
# Add your implementation-specific setup below:

```

4. **Plugin commands** reference `@/install_dev.sh` (the project root wrapper), ensuring customizations are always included
5. **`init` command** never overwrites existing wrapper scripts — if a project root script already exists, it's left as-is

## The `init` Command

A new command (`commands/init.md`) that bootstraps the plugin for a concrete FreedomLS implementation.

### What It Does

1. **Merges recommended permissions into `.claude/settings.json`**
   - Reads the plugin's `templates/settings.json` for recommended allow/deny rules
   - If `.claude/settings.json` exists, adds missing permissions without overwriting existing ones
   - If it doesn't exist, creates it from the template
   - Adds `"fls": true` to `enabledPlugins`
   - Removes any `hooks` section (plugin now owns hooks)

2. **Creates `.claude/fls.md`**
   - Asks the user for dev credentials (email, password) and base URL
   - Generates the file from the plugin's template
   - If the file already exists, skips creation

3. **Creates `.claude/fls.local.md`**
   - Creates an empty template file for machine-specific overrides
   - If it already exists, skips

4. **Adds `.claude/fls.local.md` to `.gitignore`**
   - Checks if the entry already exists before adding

5. **Generates wrapper scripts at project root**
   - Copies wrapper script templates from `templates/wrapper_scripts/`
   - Never overwrites existing scripts
   - Makes all scripts executable

6. **Validates the setup**
   - Confirms plugin is enabled in `settings.json`
   - Confirms `.claude/fls.md` exists and has required fields
   - Confirms wrapper scripts exist and are executable
   - Confirms hook scripts in the plugin are executable
   - Reports any issues found

### Interactive Flow

```
$ /fls:init

Initializing FLS plugin...

> Dev admin email [demodev@email.com]:
> Dev admin password [demodev@email.com]:
> Base URL [http://127.0.0.1:8000]:

✓ Created .claude/fls.md
✓ Created .claude/fls.local.md

```

## Success Criteria

1. **Plugin loads correctly**: Running `claude --plugin-dir ./fls-claude-plugin` loads all skills, commands, agents, hooks, and MCP config without errors
2. **No broken references**: All `${CLAUDE_PLUGIN_ROOT}` paths resolve correctly. No `@` references to plugin-internal files. No hardcoded absolute paths.
3. **`init` command works end-to-end**: Running `/fls:init` on a fresh project sets up settings.json, fls.md, fls.local.md, .gitignore, and wrapper scripts correctly
4. **`init` is idempotent**: Running it twice doesn't duplicate permissions, overwrite existing files, or create duplicate .gitignore entries
5. **Hooks run once**: With hooks in the plugin and removed from project settings.json, each hook fires exactly once per event
6. **Credentials are configurable**: Skills that need dev credentials read from `.claude/fls.md` and work with project-specific values
7. **Scripts are extensible**: Wrapper scripts at the project root correctly call plugin base scripts and allow project-specific additions
8. **Existing project still works**: After extraction, the current FreedomLS repo works identically — the plugin is installed and enabled, all functionality preserved
9. **`/reload-plugins`** picks up changes during development without restarting Claude Code
10. **Validation passes**: The `init` command's validation step reports no issues on a correctly configured project
