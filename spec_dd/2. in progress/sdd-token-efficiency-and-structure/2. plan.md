## 1. Two new tiering agents (do this first ‚Äî later tasks depend on them)

Create two dedicated subagent definitions in `fls-claude-plugin/agents/`. They are the **only** place tier
`model:` frontmatter lives (per D2 ‚Äî command/helper files run *inline* and their `model:` is inert, so tiering
must sit on spawnable agent files). Model values use **aliases** (`haiku`/`sonnet`), per the D2 sub-decision

### 1a. `fls-claude-plugin/agents/sdd-mechanic.md` ‚Äî Haiku, mechanical chores

```markdown
---
name: sdd-mechanic
description: |-
  Mechanical SDD chores that need no design judgement: running the test suite, making git commits,
  moving/renaming files, ticking/appending the SDD todo.md (by following the protected helper files),
  and worktree/todo setup. Spawn this agent from a depth-0 SDD command to push cheap work onto a faster,
  cheaper model. Non-interactive: it never asks the user ‚Äî if it lacks an input it returns
  `status: blocked`.
tools: Bash, Read, Edit, Write, Glob, Skill
model: haiku
effort: low      # optional; remove if the alias proves too weak for the chore
---

<system prompt ‚Äî see body spec below>
```

Body must instruct the agent to:
- Do exactly the chore described in its spawn prompt; make no design decisions.
- When asked to tick/append a todo, **read the named `protected/update_todo.md` (or
  `setup_todo_list.md` / `start_worktree.md`) file and follow it literally** ‚Äî these helper files remain the
  single source of truth for that logic (a subagent can read & follow a file; it just cannot type the slash
  command).
- Be **non-interactive**: never call `AskUserQuestion`. If a required input is missing or a step is
  ambiguous, stop and return `status: blocked` with `reason:` and `needs: [...]`.
- End every run with a one-line structured footer: `status: ok|failed|blocked` ¬∑ `reason: <short>` (and
  `needs: [...]` when blocked).

### 1b. `fls-claude-plugin/agents/sdd-worker.md` ‚Äî Sonnet, non-interactive fan-out unit

```markdown
---
name: sdd-worker
description: |-
  A single non-interactive unit of SDD fan-out work: one research topic, one review dimension, or one
  skills/MCP scan. Spawn one per unit from a depth-0 SDD command. Writes its findings to a file path given
  in its prompt and returns a structured status. Never asks the user.
tools: Read, Glob, Grep, WebFetch, WebSearch, Write
model: sonnet
---

<system prompt ‚Äî see body spec below>
```

Body must instruct the agent to:
- Perform **only** the one unit described in its prompt; write its output to the **exact path** given.
- Write the output in a **single `Write`** to the exact path (a `Write` puts the whole file down in one
  operation, so a crash leaves either the complete file or nothing ‚Äî never a half-file). The worker has no
  `Bash`/rename tool and needs none: the `status:` footer below, not a temp-then-rename dance, is the
  completeness signal, and the resume scan re-runs any file that lacks it.
- Finish the output file with a footer: `status: ok|failed|blocked` ¬∑ `reason:` (+ `needs:` if blocked). Cite
  reference URLs when findings are web-sourced.
- Be **non-interactive**: never call `AskUserQuestion`. Missing input ‚Üí write what it can, set
  `status: blocked`, list `needs:`, and return.
- Return a one-line summary to the orchestrator: `status=<‚Ä¶> path=<‚Ä¶> reason=<‚Ä¶>`.

> Both agents inherit permissions from the session and may auto-invoke skills. Neither gets the `Agent`/`Task`
> tool (subagents can't nest anyway) ‚Äî so neither can fan out further, which is correct.

> **`subagent_type` naming ‚Äî settle one form.** The repo's existing precedent uses **bare** names:
> `do_qa.md` spawns `qa-data-helper` and `next.md` spawns `general-purpose` (no `fls:` prefix). The harness
> also exposes plugin agents under the `fls:` namespace (`fls:code-reviewer`, `fls:qa-data-helper`). Before
> wiring spawns, the implementer must **verify which form actually resolves** for these plugin agents (spawn a
> trivial test, or check current docs), then **use that one form uniformly** across all SDD spawn sites ‚Äî
> including updating `do_qa.md`'s existing bare `qa-data-helper` reference to match. This plan writes
> `fls:sdd-mechanic` / `fls:sdd-worker` as placeholders; swap to bare `sdd-mechanic` / `sdd-worker` if that is
> the form that resolves. Do not leave the two forms mixed.

## 2. The shared Fan-out recipe (¬ß3 + ¬ß4 of the spec)

Write this compact recipe into every command that fans out (improve_idea, spec_from_idea, plan_from_spec,
implement_plan, and the two plan reviews). It combines batch resilience and input contracts.

A depth-0 command orchestrating units `U1‚Ä¶Un`:

1. **Declare inputs up front.** List the user input the phase needs. Gather it now via `AskUserQuestion`
   (legal at depth 0; subagents can't). Bake the answers into each worker prompt.
2. **Assign a deterministic output path per unit.** Durable artifacts keep their real names (e.g.
   `research_<topic>.md`). Intermediate outputs go in a scratch dir `.sdd-work/` inside the spec directory,
   named `<phase>_<unit-id>.md`.
3. **Resume scan (before spawning).** For each unit, if its output file exists **and** ends with
   `status: ok`, skip it. Only spawn units that are missing or not-ok.
4. **Spawn one worker per remaining unit**, in parallel, via the `Agent` tool with
   `subagent_type: "fls:sdd-worker"` (or `fls:sdd-mechanic` for mechanical units). Pass the exact output path
   and the baked-in inputs. Never one worker looping over the whole batch.
5. **Collect structured returns** and act per status:
   - `ok` ‚Üí done.
   - `failed` ‚Üí retry the **same** unit, up to **2 attempts**, including the prior error in the retry brief.
   - `blocked` ‚Üí orchestrator gathers the listed `needs` via `AskUserQuestion`, then **re-spawns a fresh
     worker** with the original brief + answers (pointing it at any partial file it wrote).
6. **Synthesis is a separate step.** A later step reads the output **files** (pass paths, never dump file
   contents into the orchestrator prompt) and produces the artifact. Because synthesis is separate, it can be
   retried without re-running workers.
7. **Clean up on success.** When the phase's artifact is finalised, delete the `.sdd-work/` scratch dir so no
   junk is left. Durable artifacts (e.g. `research_*.md`) are **not** deleted. An abandoned `.sdd-work/` left
   by an interrupted run is intentional ‚Äî it is what makes resume cheap.

**Housekeeping task:** add `.sdd-work/` to the repo root `.gitignore` so scratch files can never be
accidentally committed. Insert the line **directly after the existing `*.sarif` / `bandit.sarif`
scan-artifacts lines** (the block is mid-file, not at EOF ‚Äî don't append at the very end).

   - **Step 3 `(cmd)` branch:** do **not** wrap the command in a fresh subagent. Resolve the command file
     (see resolver fix below), **read it and follow its steps inline at depth 0**. The command ticks its own
     box. Running the command inline at depth 0 ‚Äî rather than one level down in a wrapper subagent ‚Äî is what
     keeps the single fan-out level free, so the command's own fan-out (research/review), spawned via
     `next.md`'s `Agent` tool, is legal again.

   Pseudocode:
   ```
   name = strip_leading("/sdd:", "/", extracted_command_name)   # e.g. "threat-model", "do_qa"
   candidates = [
     "fls-claude-plugin/commands/sdd/{name}.md",
     "fls-claude-plugin/commands/{name}.md",
   ```

4. **Step 5 report wording:** replace "Ran `<command>` in a fresh agent" with "Ran `<command>` on the main

5. **Leave `(user + cmd)` handling as-is** (next.md still refuses combined markers and asks the user to split).
   This is intentional friction for the QA-bugfix items `do_qa` appends; it is **not** a resolution failure.

> **Inline-execution note (load-bearing).** When `next.md` runs a `(cmd)` file by reading-and-following it,
> that file is **not** invoked as a slash command ‚Äî so its frontmatter (`model:`, `allowed-tools:`) is

> invocation paths are supported, so keep both. (Worker/mechanic **agent** files are unaffected: they are
> *spawned*, not inlined, so their `model:`/`tools:` frontmatter is always live ‚Äî which is why tiering lives
> there.)
