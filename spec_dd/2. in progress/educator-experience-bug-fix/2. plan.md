# Implementation Plan: Educator Experience Bug Fix

This plan covers two related frontend fixes in the educator interface:

1. **Bug 1** — `DataTablePanel` always wraps HTMX responses in the panel container, causing recursive nesting when sorting/filtering/searching.
2. **Bug 2** — Pagination markup is inconsistent: numbered buttons in `data-table.html`, minimal prev/next in `course_progress_panel.html`. Consolidate into a reusable `<c-pagination>` cotton component.

Both bugs are template / panel-layer; no model, view, URL, or business-logic changes are required.

We are following TDD: every behaviour change starts with a failing test that captures the desired behaviour, then the implementation makes it pass. Use the `fls:testing` skill for pytest patterns and the `fls:htmx` skill for HTMX conventions throughout.

### 1.2 Implement the fix in `DataTablePanel.render()`

Pseudocode:

```python
def render(self, request: HttpRequest, base_url: str = "", panel_name: str = "") -> str:
    if request.headers.get("HX-Request") == "true":
        return self.get_content(request, base_url=base_url, panel_name=panel_name)
    return super().render(request, base_url=base_url, panel_name=panel_name)
```

**Pseudocode:**

```python

import pytest
from playwright.sync_api import Page, expect

from freedom_ls.panel_framework.tests.conftest import _make_stub


@pytest.mark.playwright
@pytest.mark.django_db(transaction=True)
def test_data_table_pagination_numbered_link_navigates_and_preserves_sort(
    live_server, page: Page,
):
    # Create enough rows that page 2 exists. Pad names so lexical sort is stable.
    page_size = 10  # confirm via DataTable default in panel_framework/tables.py
    for i in range(page_size + 2):
        _make_stub(name=f"row-{i:02d}")

    page.goto(f"{live_server.url}/test-panel/framework/stubs/")

    # Sort ascending so row-00 is on page 1 and row-{page_size+1} is on page 2.
    page.get_by_role("button", name="Name").click()
    expect(page).to_have_url(re.compile(r"sort=name"))
    expect(page.get_by_text("row-00")).to_be_visible()

    page.get_by_role("link", name="2").click()
    expect(page).to_have_url(re.compile(r"page=2"))
    expect(page).to_have_url(re.compile(r"sort=name"))  # sort survived the click
    expect(page.get_by_text(f"row-{page_size + 1:02d}")).to_be_visible()
    expect(page.get_by_text("row-00")).not_to_be_visible()


@pytest.mark.playwright
@pytest.mark.django_db(transaction=True)
def test_data_table_pagination_prev_next_links(live_server, page: Page):
    page_size = 10
    for i in range(page_size + 2):
        _make_stub(name=f"row-{i:02d}")

    page.get_by_role("link", name="Previous").click()

```

Note the parameter mapping: `<c-pagination>` always emits `?page=N`, so for the **student** paginator we keep that natural mapping and pass `col_page` via `extra_params`. For the **column** paginator we cannot use `?page=` for both (the view treats `page` as the student page, per `views.py` lines 303–304 and 338–339). Two options — pick whichever is cleaner during implementation:

- **Option A (preferred)**: leave the column paginator's prev/next as-is (small inline block) since its query parameter name is `col_page`, not `page`. Then the new component is only used for the student paginator. Document this in a comment.
- **Option B**: extend `<c-pagination>` to accept an optional `page_param_name` input (default `"page"`). The course-progress column paginator would pass `page_param_name="col_page"`. Slightly more complexity but keeps both paginators on the component.

The implementing agent should pick the option that minimises new template logic. Default to Option B if it's a one-line addition to the component (it is — just substitute the param name in the suffix builder); fall back to Option A otherwise.

Whichever is chosen, the success criterion is the same: the course progress panel uses the numbered-button visual style for at least the student paginator, paginators remain independent, and clicking page 2 of one paginator does not reset the other.

### 2.6 Add a regression test for paginator independence

In `freedom_ls/educator_interface/tests/test_cohort_course_progress_panel.py` (existing file), add:

- `test_column_pagination_links_preserve_student_page` — render the panel with both paginators having multiple pages and student page set to 2, assert the column pagination links contain `page=2`.
- `test_student_pagination_links_preserve_column_page` — symmetrical.

These tests close the spec's success criterion: "clicking page 2 of course items leaves student pagination on its current page, and vice versa."
