# Spec: Upgrade Django 5.2 to Django 6.0

## Overview

Upgrade the Freedom Learning System from Django 5.2.9 to Django 6.0.x (currently 6.0.3). This is a compatibility-focused upgrade that adopts Django 6's native template partials, enables CSP in report-only mode, sets up the background tasks framework for future use, and verifies all third-party dependencies work correctly.

## Why upgrade

- Django 5.2 LTS support ends April 2028, but Django 6.0 brings native features we currently get from third-party packages (template partials)
- Built-in CSP middleware improves security posture
- Background tasks framework provides a standard interface for future async work
- Staying current reduces the cost of future upgrades

## Prerequisites

- Python 3.13+ (already met)
- `DEFAULT_AUTO_FIELD = "django.db.models.BigAutoField"` (already set)

## Phase 1: Verify compatibility before upgrading

### 1.1 Fix deprecation warnings on Django 5.2

**Why:** Deprecated features in 5.x are removed in 6.0. Fixing warnings first ensures a smooth transition.

Run the test suite with deprecation warnings enabled:
```
python -Wd -m pytest
```

Fix all Django deprecation warnings before proceeding.

### 1.2 Verify django-cotton compatibility (HARD BLOCKER)

**Why:** django-cotton is deeply integrated into the template system (custom loader, tag library). If it doesn't work with Django 6, we cannot upgrade.

**Decision:** If django-cotton is not compatible with Django 6.0, **stop the upgrade entirely** and wait for cotton to add support.

**Verification steps:**
1. Update Django to `>=6.0,<6.1` (all work happens in a worktree)
2. Run the full test suite
3. Use Playwright to manually verify template rendering across key pages (student interface, educator interface, course content)
4. Specifically check cotton component rendering (callouts, UI components)

### 1.3 Verify django-premailer compatibility

**Why:** Unmaintained since 2016 (officially supports Django 1.6-1.9), but it's a thin wrapper and likely still works.

**Decision:** Test it and only replace if it actually breaks.

**Verification steps:**
1. During the django-cotton verification (same temporary branch), also test email rendering
2. Trigger email flows (registration, password reset) and verify HTML emails render with inlined CSS
3. If broken: evaluate replacement options (`premailer` library directly, `css-inline`, or manual inlining)

**Why:** django-stubs doesn't have a Django 6.0 release yet. Full type coverage is not available.

**Decision:** Accept partial type coverage. Temporary `# type: ignore` comments are allowed **only** for django-stubs gaps (new Django 6 APIs that lack stubs). Each must include a comment explaining it's stubs-related:

```python
result = some_django6_api()  # type: ignore[no-untyped-def]  # django-stubs lacks Django 6 support
```

**Success criterion:** All such comments are documented in a tracking list so they can be removed when django-stubs releases Django 6 support. Add a comment at the top of any file that needs these ignores referencing this spec.

## Phase 3: Migrate django-template-partials to built-in

### Why

Django 6.0 includes template partials natively (same `partialdef`/`partial` syntax, created by the same author). Removing the third-party package reduces dependencies and uses the supported core implementation.

### Important: cotton and partials interaction

django-template-partials and django-cotton interfere with each other (both hook into the template loader). When migrating to Django 6's built-in partials, verify that **both** partials and cotton components still render correctly together.

### Migration steps

1. **Remove from INSTALLED_APPS:** Delete `"template_partials.apps.SimpleAppConfig"` from `config/settings_base.py`
2. **Remove template loader:** Remove `"template_partials.loader.Loader"` from the template loaders configuration in `config/settings_base.py`
3. **Remove from builtins:** Remove `"template_partials.templatetags.partials"` from template builtins in `config/settings_base.py`
4. **Remove package dependency:** Delete `django-template-partials` from `pyproject.toml`
5. **No template changes needed:** Templates don't use `{% load partials %}` (it was configured as a builtin), so the 5 template files using partials should work without modification
6. **Verify cotton + partials together:** Run tests and use Playwright to check that cotton components render correctly alongside built-in partials

### Fallback

If the built-in Django 6 template partials don't work correctly (especially in combination with django-cotton), **keep the external `django-template-partials` dependency** rather than breaking the template system. The external package should continue to work on Django 6.

### Files using partials (verify all render correctly)

1. `freedom_ls/content_engine/templates/cotton/callout.html`
2. `freedom_ls/student_interface/templates/student_interface/course_form_page.html`
3. `freedom_ls/student_interface/templates/student_interface/partials/course_list.html`
4. `freedom_ls/student_interface/templates/student_interface/partials/course_minimal_toc.html`
5. `freedom_ls/student_interface/templates/student_interface/partials/form_progress_scores.html`

## Phase 4: Handle removed/changed Django features

### 4.1 Model.save() positional arguments

**Why:** Django 6 no longer accepts positional arguments to `Model.save()`.

**Current state:** Codebase analysis shows all 3 custom `save()` overrides use `super().save(*args, **kwargs)` which passes through whatever the caller sends. This is safe as long as no callers pass positional args.

**Action:** Search for any `.save(` calls with positional arguments and convert to keyword arguments. The existing `*args, **kwargs` pattern in overrides is compatible.

Files with custom save():
- `freedom_ls/app_authentication/models.py` (Client.save)
- `freedom_ls/site_aware_models/models.py` (SiteAwareModelBase.save)
- `freedom_ls/student_progress/models.py` (CourseItemProgress.save)

### 4.2 format_html() without args

**Current state:** No uses of `format_html()` found in the codebase. No action needed.

### 4.3 send_mail() / EmailMessage() positional args

**Current state:** No direct uses found. Email is handled by django-allauth. No action needed.

### 4.4 forms.URLField default scheme

**Current state:** No uses of `forms.URLField` found. No action needed.

## Phase 5: Adopt new Django 6 features

### 5.1 Content Security Policy (report-only mode)

**Why:** CSP hardens security by controlling which resources the browser is allowed to load. Starting in report-only mode lets us gather data on what would be blocked without breaking anything.

**Implementation:**
1. Add `"django.middleware.csp.ContentSecurityPolicyMiddleware"` to `MIDDLEWARE`
2. Add `"django.template.context_processors.csp"` to template context processors
3. Configure `SECURE_CSP_REPORT_ONLY` in settings with a baseline policy:
   ```python
   from django.utils.csp import CSP

   SECURE_CSP_REPORT_ONLY = {
       "default-src": [CSP.SELF],
       "script-src": [CSP.SELF, CSP.UNSAFE_INLINE],  # HTMX needs inline
       "style-src": [CSP.SELF, CSP.UNSAFE_INLINE],    # Tailwind/inline styles
       "img-src": [CSP.SELF, "data:"],
       "connect-src": [CSP.SELF],                      # HTMX fetch requests
       "frame-src": [CSP.NONE],
   }
   ```
4. A separate follow-up spec (`spec_dd/1. next/CSP-rollout/`) covers tightening the policy with nonces and enforcing mode

### 5.2 Background Tasks Framework (dev backend only)

**Why:** Django 6 includes a built-in task framework, but it ships without a production backend. Setting it up now with the dev backend establishes the pattern for future use.

**Decision:** Set up with the development backend only. This prepares the codebase to use `@task` decorators and `enqueue()` when a production backend becomes available.

**Implementation:**
1. Add `"django.tasks"` to `INSTALLED_APPS` (if required)
2. Configure the development backend in settings:
   ```python
   TASKS = {
       "default": {
           "BACKEND": "django.tasks.backends.immediate.ImmediateBackend",
       },
   }
   ```
3. Do **not** convert any existing functionality to use tasks yet — this is infrastructure-only
4. Add a comment noting that a production backend (e.g., database-backed) is needed before using in production
