# Spec: Fix premailer SyntaxWarning and remove unmaintained django-premailer

## Problem

Production logs show a `SyntaxWarning` from the premailer library (v3.0.0):

```
SyntaxWarning: invalid escape sequence '\s'
at premailer/premailer.py:105
```

This is because premailer 3.0.0 (from 2016) uses `re.compile('\s*!important')` instead of a raw string. Python 3.12+ flags this as a SyntaxWarning, and future Python versions will make it a SyntaxError — meaning emails would break entirely.

Additionally, we depend on `django-premailer` (v0.2.0, last released 2016), an unmaintained 38-line wrapper that provides a template tag around the premailer library. Keeping an unmaintained dependency is unnecessary risk.

## Solution

Two changes:

### 1. Upgrade premailer to 3.10.0

Upgrade the `premailer` library from 3.0.0 to 3.10.0 (latest maintained version). This fixes the `\s` escape sequence warning since newer versions use raw strings for regex patterns.

**Why this is safe:** django-premailer (and our replacement template tag) uses the stable public API: `Premailer(html, **kwargs).transform()`. This API has not changed between 3.0.0 and 3.10.0. The only new transitive dependency is `cachetools`.

### 2. Replace django-premailer with a custom template tag

Vendor the template tag functionality (currently 38 lines in django-premailer) into the `accounts` app, then remove the `django-premailer` package dependency.

**Why:** django-premailer is unmaintained (last release 2016, officially supports Django 1.6-1.9). It's a trivial wrapper — owning it directly means we control upgrades and can adapt if the premailer API ever changes.

**What the template tag does:**
- Provides `{% premailer %}...{% endpremailer %}` block tag
- Renders the enclosed template content
- Passes the rendered HTML through `Premailer(html, **kwargs).transform()` to inline CSS from `<style>` tags into element `style` attributes
- Reads optional `PREMAILER_OPTIONS` from Django settings

**Where to put it:** Create `freedom_ls/base/templatetags/premailer.py` with the same tag name and behavior so existing templates (`base_email.html`, `base_notification_email.html`) continue to work with `{% load premailer %}`. The `base` app is the right home since email inlining is a shared utility not specific to any one app.

## Changes Required

1. **Create `freedom_ls/base/templatetags/premailer.py`** — custom template tag that wraps `premailer.Premailer`
2. **Update `pyproject.toml`** — replace `django-premailer>=0.2.0` with `premailer>=3.10.0`
3. **Update `config/settings_base.py`** — remove `django_premailer` from `INSTALLED_APPS`
4. **Run `uv lock` and `uv sync`** — update the lock file and install
5. **No template changes needed** — the tag name stays `premailer` and the `{% load premailer %}` directive will resolve to the new local templatetags module

## Success Criteria

- No `SyntaxWarning` about invalid escape sequences in logs
- Email HTML still has inline CSS styles (premailer transform is working)
- `django-premailer` is no longer in dependencies
- `premailer>=3.10.0` is a direct dependency
- All existing email template tests pass without modification
- The custom template tag supports `PREMAILER_OPTIONS` from settings (matching existing behavior)
