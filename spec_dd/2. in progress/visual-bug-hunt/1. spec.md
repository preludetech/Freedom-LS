# Spec: Visual Bug Hunt

Fix three reported frontend/UI defects in the Freedom Learning System (FLS). All
three are presentational or interaction bugs in the student course player and the
educator cohorts interface. None change data models or business logic.

The bugs were reported by a user against the running dev site. This spec records
the confirmed root cause of each, the chosen fix approach, and the success
criteria. Implementation must follow TDD where a behavioural test is meaningful
(bug 3); the presentational bugs (1 and 2) are verified by visual QA rather than
by asserting CSS classes in unit tests.

---

## Bug 1 — Course player content renders in the wrong column, then jumps

### Reported behaviour
URL: `/courses/<course-slug>/<index>/`
(e.g. `/courses/functionality-demo-show-end-with-topic/2/`)

On first (cold) load the main content briefly appears inside the left-hand
panel (the course-outline / table-of-contents column) while the content area is
empty. Then the main content jumps to where it belongs and the table of contents
populates. Things should render in their final positions without jumping.

### Why it matters
A visible layout shift on every cold load of a course makes the product feel
broken and unpolished. It is the first thing a student sees when opening a
course. It also hurts perceived performance and Cumulative Layout Shift.

### Root cause
The layout is a CSS grid (`side-panel-grid` in
`freedom_ls/base/templates/_base_interface.html`). The grid's column count is
keyed on the `data-panel-open` attribute of the `panelGrid` div:

- `data-panel-open="true"` (or absent) → two columns (`--sidebar-width` + `1fr`)
- `data-panel-open="false"` → single column

The static HTML ships with `data-panel-open="true"` (line ~30), so the browser
applies a **two-column** template immediately. But the sidebar is a native
`<dialog>` element, which is `display:none` until Alpine's `sidePanel`
component calls `dialog.show()` during `init()`. A `display:none` grid item does
not participate in the grid, so on first paint `#interface-main` (the content)
is the only participating child and gets assigned to **track 1 — the narrow
sidebar column**. When Alpine opens the dialog, the content is pushed to track 2
and visibly jumps.

### Chosen fix
Change the initial static value to `data-panel-open="false"` on the `panelGrid`
div in `_base_interface.html`. With a single-column template on first paint, the
content renders full-width in its correct horizontal position. Alpine's
`sidePanel._syncGrid()` already writes the correct `data-panel-open` value during
`init()`, so when the dialog opens, the second column appears with the sidebar
already in it — no content reflow into the sidebar track.

This is a one-attribute change to the HTML default; it does not alter the Alpine
logic, which is the source of truth for the panel's open/closed state after init.

### Out of scope
The HTMX-boosted Previous/Next navigation (`c-player-nav`, using
`hx-select-oob="#course-toc-region"`) already swaps content and outline together
in a single response and is **not** the cause — it correctly avoids a second
request. Do not change it.

### Success criteria
- On a cold load of a course topic/form page, the main content never appears
  inside the sidebar column and does not horizontally jump after Alpine
  initialises.
- The sidebar/outline still opens and renders correctly once Alpine is ready
  (desktop and mobile).
- HTMX Previous/Next navigation continues to work unchanged.

## Bug 2 — Excess whitespace under the course-player header

### Reported behaviour
Same URL as bug 1. There is a bottom border under the header area (breadcrumb +
title, e.g. "Callouts") followed by an oversized gap before the body content
begins. The gap should be smaller.

### Why it matters
The large dead space pushes content below the fold and makes each course screen
look unfinished. It is visible on every topic and form in the player.

### Root cause
The gap is **double-spacing** — two stacked sources of vertical space between the
header's bottom border and the first line of content:

1. The shared header in `_base_interface.html` (line ~80) has `pb-4` (16px)
   inside its `lg:border-b`.
2. The content wrapper `<c-page>` (`freedom_ls/base/templates/cotton/page.html`,
   line ~10) adds `py-8 lg:py-12` (32px mobile / 48px desktop) of top padding.

Combined gap below the border: ~48px mobile, ~64px desktop. The codebase
convention (confirmed against the modal and panel-section headers) is that a
bordered header owns its own `pb-*`/`py-*` and the content below should not add a
large additional `pt-*` on top of it.

### Chosen fix (scope: course player only)
Reduce the **top** padding of the `c-page` content wrapper in the course-player
templates only, via the component's pass-through `class` attribute (e.g.
`<c-page flush="true" class="pt-... lg:pt-...">`). `cotton/page.html` already
renders `{{ class }}` on the content wrapper, so this is the supported mechanism
and keeps the change scoped to the player. This mirrors the existing precedent
in `course_detail.html`, which already trims the same `py-8 lg:py-12` default
with a plain `class="pt-2 lg:pt-2"` (no `!important` needed — Tailwind v4 orders
`pt-*` after the `py-*` shorthand; if an override does prove necessary, the v4
important syntax is the suffix form `pt-4!`, not the v3 prefix `!pt-4`). The
shared header in `_base_interface.html` and the educator interface are left
untouched.

This must be applied consistently to every course-player template that uses
`<c-page flush="true">` under `_base_interface.html`, namely (verify the full
set during implementation): `course_topic.html`, `course_form.html`,
`course_form_complete.html`, `course_finish.html`. Note `course_form_page.html`
is **not** in scope — it extends the sidebar-less `_exam_runner_base.html` and
does not use `<c-page>`, so it has neither the shared bordered header nor the
double-spacing.

The exact reduced value is a presentation detail to settle visually during QA;
the target is a tight, balanced gap below the border (roughly the header's own
padding, not double it) while keeping the layout breathing room on desktop.

### Success criteria
- The gap between the header's bottom border and the first line of body content
  in the course player is visibly reduced and looks intentional on both mobile
  and desktop.
- The change is confined to the course player; the educator interface and other
  screens sharing the header keep their current spacing.
- Header border, breadcrumb, and title remain correctly positioned.

## Bug 3 — "Save and add another" does not refresh the cohorts table

### Reported behaviour
URL: `/educator/cohorts`

The user clicks Create Cohort, fills the modal, and clicks **"Save and add
another"**. After they finish and close the modal, the newly created cohort is
**not** in the cohorts table until the page is manually refreshed.

### Why it matters
The educator gets no confirmation that their cohorts were created — the table
silently lies until a reload. "Save and add another" is exactly the path a user
takes when bulk-creating cohorts, so it is the most affected workflow.

### Root cause
The "Save and add another" path
(`CreateInstanceAction.form_valid` in
`freedom_ls/panel_framework/actions.py`, ~lines 162–168) saves the instance,
re-renders an empty modal form (200), and sets the response header
`HX-Trigger: <created-event-name>` (for cohorts, `cohortCreated`, from
`get_created_event_name()`). **The event fires on the document body, but nothing
in the DOM listens for it**, so the cohorts table is never re-fetched. (The plain
"Save" path returns 204 + `HX-Redirect` and navigates away, so it never needed a
table refresh — which is why only "Save and add another" exhibits the bug.)

The codebase already has the matching half of this pattern working elsewhere:
`tabContainer`'s `panelChanged` listener
(`freedom_ls/panel_framework/static/panel_framework/js/alpine-components.js`)
listens on `document.body` for a server-sent `HX-Trigger` event and calls
`htmx.ajax` to re-fetch content. The table markup is also refreshable in
principle: `cotton/data-table.html` wraps everything in `id="{{ table_id }}"`.

There are **two** missing pieces, not one:

1. **No client-side listener** for the created-event — nothing re-fetches the
   table when `cohortCreated` fires (the bug as reported).
2. **The list-view render path returns more than the table.** The cohorts list
   is a `ListViewConfig` (`CohortConfig`), and its content is rendered by
   `_render_list_view_content` in `freedom_ls/panel_framework/views.py`, which
   **always prepends the create-action button + modal** (`actions_html`) above
   the table and has **no `HX-Target` short-circuit**. The
   `DataTablePanel.render` short-circuit (`panels.py`, returns just the table
   when `HX-Target == table_id`) applies only to **instance-view** panels (the
   tabs inside a single cohort), not to list views. So a refresh that targets
   `#<table_id>` with `swap:outerHTML` against `base_url` would swap in
   `actions_html + table_html` and **duplicate the create button/modal on every
   refresh**. (This has never surfaced because `CohortDataTable` has no search,
   and the only list with search — Users — has no create action.)

### Chosen fix (scope: generic panel framework; mechanism: JS event listener)
Wire a generic, event-driven table refresh into the panel framework so that
**any** list view with a create-in-modal action refreshes its table when its
created-event fires — not just cohorts. This both fixes cohorts and prevents the
same defect recurring for every other list configuration.

Approach (mirroring the existing `panelChanged` convention) — both pieces:

1. **Server-side short-circuit on the list-view render path.** Make the
   list-view dispatch return **only** the table HTML (not `actions_html`) when
   the request targets the table, mirroring the existing
   `DataTablePanel.render` short-circuit on `HX-Target == table_id`. This keeps
   the targeted refresh from duplicating the create button/modal, and also
   fixes the latent case of search on a list that has a create action.
2. **Client-side listener.** The list view already knows its create action(s)
   and therefore the created-event name(s) from `get_created_event_name()`, plus
   its `table_id` and `base_url`. Pass these to the list template/JS, and add a
   listener on `document.body` for the created-event that calls
   `htmx.ajax("GET", base_url, { target: #<table_id>, swap: "outerHTML" })` to
   re-fetch the table. This reuses the already-fired `HX-Trigger` event and the
   targeted refresh enabled by piece 1.

Decisions, and why:
- **Generic over cohort-only** — the trigger event name is produced by the
  generic `CreateInstanceAction` base class and the data table is a generic
  framework component, so fixing it at the framework level is the natural seam
  and avoids per-list duplication.
- **JS event listener over OOB swap** — it matches the project's established
  `panelChanged`/`tabContainer` convention, keeps the action's save path free of
  table-rendering concerns, and the server already emits the event. No new
  template convention (`hx-trigger="… from:body"`) is introduced, consistent with
  how the codebase does body-level event handling today.

### Success criteria
- After "Save and add another" creates a cohort and the user closes the modal,
  the new cohort appears in the cohorts table **without a manual page refresh**.
- Creating several cohorts in a row via "Save and add another" results in all of
  them being present in the table.
- The plain "Save" path still redirects to the new cohort's detail page,
  unchanged.
- The refresh mechanism is generic: any panel-framework list view whose create
  action fires a created-event refreshes its table the same way.
- The targeted refresh swaps in only the table — the create-action button/modal
  is not duplicated, no matter how many times the table is refreshed.

## Testing strategy

Per the project convention and the idea's instruction, fix bugs using TDD, but
**do not assert CSS classes or styling in tests** — presentational bugs are
verified visually.

- **Bug 1 (presentational):** No unit test asserting CSS/grid classes. Verified
  via Playwright QA on a cold course-player load — content does not appear in or
  jump out of the sidebar column. The change is a static-default flip, so a
  template snapshot/regression check of the default attribute value is acceptable
  if it adds value, but the real verification is visual.
- **Bug 2 (presentational):** No CSS-class assertions. Verified visually via
  Playwright QA — the gap under the header is reduced in the player and unchanged
  in the educator interface.
- **Bug 3 (behavioural):** TDD. Write a failing test first that exercises the
  create-in-modal flow and asserts the table reflects the new instance without a
  full page reload (e.g. a Playwright E2E test of the "Save and add another" →
  close → table-updated flow, and/or a server-side test asserting the list view
  emits the listener wiring / the created-event). Also add a server-side test for
  the short-circuit: a targeted refresh (`HX-Target == table_id`) of a list view
  that has a create action returns **only** the table HTML and does not duplicate
  the create-action button/modal. Then implement the fix.

## Overall success criteria
- All three reported behaviours are gone when reproduced on the dev site at the
  given URLs.
- No regression to HTMX player navigation, the educator interface spacing, the
  plain "Save" cohort flow, or any other panel-framework list view.
- Changes follow existing codebase conventions (server-side render + OOB for the
  player; `panelChanged`-style body event listeners for table refresh; cotton
  `class` pass-through for scoped spacing).
