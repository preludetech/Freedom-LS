# Frontend QA: Visual Bug Hunt

Browser verification for the three fixes. All three are frontend, so all three
need a visual/interaction check. Bugs 1 and 2 are verified by eye (no unit
tests); Bug 3 also has automated tests but the end-to-end browser flow is the
real confirmation.

Use the `fls:use-playwright` skill conventions. If required test data is missing
(a registered student with a course, an educator who can see cohorts), delegate
its creation to the `fls:qa-data-helper` agent — do **not** create it by hand.
Use the **DemoDev** site for all test data.

## Setup

Start the dev server on a random port (never 8000):

```bash
PORT=$(.claude/fls/scripts/find_available_port.sh)
uv run python manage.py runserver $PORT
```

Base URL: `http://127.0.0.1:$PORT`

Credentials (dev): `demodev@email.com` / `demodev@email.com`.

Login: navigate to `http://127.0.0.1:$PORT/` and sign in with the credentials
above. For the student-player checks you need an account enrolled in a demo
course (e.g. the course at slug `functionality-demo-show-end-with-topic`); for
the educator check you need an account that can reach `/educator/cohorts`. If the
logged-in account can't reach one of these, ask `fls:qa-data-helper` for the
right account/enrolment on the DemoDev site.

## Test 1 — Bug 1: no layout shift on cold course-player load

Goal: the main content must render in its correct (full-width) horizontal
position on first paint and must **not** appear inside, or jump out of, the
left-hand outline column.

1. Open a course topic page **cold** (force a fresh load — disable cache / hard
   reload), e.g. `http://127.0.0.1:$PORT/courses/functionality-demo-show-end-with-topic/2/`.
2. Watch the very first paint as the page loads:
   - **Expected:** the main content appears full-width in the content area
     immediately; it does not briefly sit in the narrow left column and then jump
     right. The outline/sidebar appears in the second column once Alpine
     initialises.
   - **Fail:** content flashes inside the left outline column, then jumps to the
     right.
3. Once loaded, confirm the sidebar/outline opens and renders correctly:
   - Desktop (resize to ~1920×1080): outline is docked in the left column; toggle
     it closed and open — content reflows smoothly, no jump.
   - Mobile (resize to ~375×812): tap the table-of-contents toggle in the header;
     the outline opens as an overlay/sheet and closes cleanly.
4. Click **Next** / **Previous** in the player nav and confirm content + outline
   still swap together correctly (no regression to HTMX navigation).

Capture a screenshot of the cold-loaded page at desktop and mobile widths for the
QA report.

## Test 2 — Bug 2: reduced whitespace under the player header

Goal: the gap between the header's bottom border and the first line of body
content in the course player is visibly smaller and looks intentional — on mobile
and desktop — while the educator interface is unchanged.

1. On the same player page (`/courses/.../2/`), look at the area directly under
   the header (breadcrumb + title, e.g. "Callouts") above the first body content:
   - **Expected:** a tight, balanced gap below the border — roughly the header's
     own padding, not double it. Content sits higher up the page.
   - **Fail:** a large dead-space gap pushing content well down the page.
2. Check at both widths (desktop ~1920×1080 and mobile ~375×812) — the gap should
   look intentional at both, with desktop keeping a little more breathing room.
3. Spot-check the other player screens use the same reduced gap: a form page, the
   form-complete screen, and the course-finish screen.
4. **Control / no-regression:** open the educator interface
   (`http://127.0.0.1:$PORT/educator/cohorts`) and confirm its header-to-content
   spacing is **unchanged** (the fix must be confined to the player).
5. Confirm the header border, breadcrumb, and title are still correctly
   positioned.

Capture before/after-style screenshots of the player header area (desktop +
mobile) and one of the educator cohorts header for the report.

## Test 3 — Bug 3: "Save and add another" refreshes the cohorts table

Goal: after creating cohorts via "Save and add another", the new rows appear in
the cohorts table without a manual page refresh, and the create button/modal is
not duplicated.

1. Navigate to `http://127.0.0.1:$PORT/educator/cohorts`. Note the current rows in
   the cohorts table.
2. Click **Create Cohort**. The create modal opens.
3. Fill in the form and click **"Save and add another"**.
   - **Expected:** the modal stays open with an empty form, and the cohorts table
     behind it now shows the just-created cohort — **without** reloading the page.
   - **Fail:** the new cohort is absent from the table until a manual refresh.
4. Repeat step 3 two more times with different cohort names.
   - **Expected:** every newly created cohort appears in the table; the table
     updates each time.
5. Close the modal.
   - **Expected:** all created cohorts are present in the table.
   - Confirm there is exactly **one** "Create Cohort" button/modal trigger above
     the table — it is **not** duplicated after the refreshes.
6. **No-regression — plain Save:** click **Create Cohort** again, fill the form,
   and click **"Save"** (not "Save and add another").
   - **Expected:** you are redirected to the new cohort's detail page, as before.

Capture a screenshot of the cohorts table after several "Save and add another"
creations (modal closed) showing all new rows and a single create button.
