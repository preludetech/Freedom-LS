# Frontend QA: Phase 4 — First Class Tier-3 template overrides

## What we're verifying

1. Under `FLS_THEME=first_class`, the three overridden cotton primitives (`header-button`, `chip`, `button`) render the override variants visibly different from the default.
2. Under `FLS_THEME=default`, the same primitives render unchanged from Phase-1 baseline — overrides do not leak.
3. The new integration test (`test_theme_template_override.py`) passes — proves resolver wiring picks up theme `templates/` dirs.
4. Phases 1–3 deliverables remain green.

We use the **DemoDev** site for all test data.

## Test 2 — Tier-3 overrides under first_class

(Continuing with `FLS_THEME=first_class`.)

For each of the three overridden cotton primitives, find a page that renders it and take a screenshot.

- **`header-button.html`** — visible in the top nav. Compare to Test 1.1 default screenshot. Expected difference: label uses the Outfit display font with a brand-coloured underline accent.
- **`chip.html`** — visible on dashboards / progress panels / course list. Expected difference: pill-shaped (fully rounded) with a left-border colour stripe instead of a background tint.
- **`button.html`** — visible wherever a generic `<c-button>` is used (CTAs that aren't `btn-primary` etc.). Expected difference: uppercase tracked label with a small accent dot.

**Things to verify on each:**

- The override is functional — clicking still works, links still resolve, slot content still renders, attributes (e.g. `disabled`, `type=submit`) still apply. The override honours the original `<c-vars>` declaration plus `{{ attrs }}` and `{{ slot }}` forwarding.
- Layout of containing pages is otherwise unchanged.

---

## Test 3 — Default theme: overrides must NOT bleed across

Stop the server. Then:

```bash
FLS_THEME=default npm run tailwind_build
FLS_THEME=default uv run python manage.py runserver $PORT
```

Re-visit the same pages from Test 2 and confirm:

- `header-button` is back to the default font / no underline accent.
- `chip` is back to the default background tint, not the override stripe.
- `button` is back to default casing / no accent dot.

If any override visual leaks under `default`, the resolver is incorrectly attaching the first_class `templates/` dir even when first_class isn't active — investigate the Phase 1 resolver code.

## Test 4 — Static asset and route smoke

Quickly confirm no template-resolution failures by hitting routes that render each primitive:

- `/` — header buttons in nav.
- A course-list page — chips on course cards.
- A page with generic `<c-button>` usage — find via grep for `<c-button` in templates.

All routes should return 200 with sensible markup. Any 500 error during render indicates a broken `<c-vars>` block in the override.
