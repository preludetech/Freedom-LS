# Spec: First Class theme — Tier 3 template overrides (Phase 4 of 5)

## Why

This is **Phase 4** of the multi-phase Themable Freedom LS effort. It validates the Tier-3 architecture (template overrides) by demonstrating that a theme can drop in single overridden cotton templates **without forking the rest of the FLS template tree**. Generic primitives in the main project remain the default; the theme selectively shadows them.

After Phase 4, the `first_class` canary theme overrides three cotton primitives — `header-button.html`, `chip.html`, `button.html` — each with a small but visibly different rendering. Switching back to `FLS_THEME=default` restores the FLS defaults.

This phase is the canary that proves Tier 3 of the master spec: **a downstream project can replace a cotton component or page shell by dropping `templates/cotton/<name>.html` into their theme directory, without forking unrelated FLS templates.**

For the holistic five-phase context, see `../themable-implementations-master-decomposed-into-phases/`.

## Scope of this phase

**In:**

- `freedom_ls/themes/first_class/templates/cotton/header-button.html` — visibly different from default.
- `freedom_ls/themes/first_class/templates/cotton/chip.html` — visibly different from default.
- `freedom_ls/themes/first_class/templates/cotton/button.html` — visibly different from default.
- An integration test that asserts the resolver actually picks up theme `templates/` dirs (the codepath wired in Phase 1 was previously untested in practice because no theme had a `templates/` folder).
- `cspell.json` additions for any brand names introduced by the overrides (`Outfit`, `DM Sans`).

**Out of this phase (covered later):**

- Documentation explaining Tier-3 to downstream theme authors — Phase 5.
- Override of `_base.html` or any `partials/*` template — not required for the canary; the master spec lists those as supported but unused by the demo.

**Phases 1–3 are hard prerequisites.**

## Goals & success criteria

1. **All three override files exist** at `freedom_ls/themes/first_class/templates/cotton/{header-button,chip,button}.html`.
2. **Each override clearly looks different from the default** (different shape, brand-coloured stripe, layout shift) and **honours the original's full `<c-vars>` declaration plus `{{ attrs }}` and `{{ slot }}` forwarding**.
3. **Pages that render those cotton primitives use the overridden versions** when `FLS_THEME=first_class`.
4. **Pages render unchanged when `FLS_THEME=default`** — no leakage of overrides.
5. **A new integration test in `freedom_ls/base/tests/test_theme_template_override.py` passes** — proves the resolver wiring picks up theme template dirs.
6. **Phases 1–3 deliverables remain green.**

## Architecture used by this phase

### Tier 3 — template overrides (recap)

For genuine restructure (different page shell, replaced header, custom card layout):

- Drop `templates/cotton/<name>.html` into the theme dir to override a cotton component.
- Drop `templates/partials/<name>.html` to override a page-shell partial.
- Drop `templates/_base.html` to replace the full page shell.

Cotton's `<c-vars …>` declaration plus `{{ attrs }}` and `{{ slot }}` forwarding form the override contract. **A theme that overrides a cotton component must honour the full `<c-vars>` block — same caution as a Python function signature.**

**Generic primitives (`base/templates/cotton/*`, `base/templates/partials/*`, `_base.html`) and app-owned cotton (`content_engine/templates/cotton/*`, `icons/templates/cotton/*`) stay in their current locations.** A theme is *not* required to own templates; if it doesn't, every template still resolves from the existing FLS source tree. A theme that wants to override one drops a same-named file into `themes/<slug>/templates/cotton/<name>.html`.

App-level templates (student dashboards, educator views, etc.) stay in their FLS apps. Themes contain *only* visual primitives.

### Resolver wiring (already in place from Phase 1)

The Phase-1 resolver prepends the active theme's `templates/` dir to `TEMPLATES[0]["DIRS"]`. Until Phase 4 the active-theme dir has had no `templates/`, so this codepath has been untested in practice. Phase 4 exercises it for the first time and adds an integration test to lock the behaviour in.

### First Class Tier-3 demo overrides

Each override is a small, deliberately-different copy of the FLS default. Read the original first to capture its full `<c-vars>` block, then ship a version that:

- **`templates/cotton/header-button.html`** — labels in the `font-display` (Outfit) family with a brand-coloured underline accent.
- **`templates/cotton/chip.html`** — pill-shaped (`rounded-full`) with a left-border colour stripe instead of a background tint.
- **`templates/cotton/button.html`** — uppercase tracked label with a small accent dot.

These are demos, not polished design. The bar is "obviously different from default; still functional".

## Out of scope (deferred)

- `_base.html` or `partials/*` overrides — not required for the canary.
- Theming-guide content explaining Tier 3 to downstream authors — Phase 5.
- Polished First Class production templates — these are demos.
