# Implementation Plan: Student dashboard (course index) visual refresh

A **visual / layout refresh** of the student dashboard plus one genuinely new,
data-only section ("Available courses"). No new models, migrations, or faked
content. All data comes from existing helpers.

This plan is written for TDD: write the failing test(s) for each behavioural
task first, then implement. Do **not** pre-write every test up front.

## Source of truth / skills

Follow these skills while implementing (see "Skills & MCPs" at the end for the
per-task mapping):

- `fls:alpine-js` — the date eyebrow (CSP-safe Alpine component, registered in
  `alpine-components.js`, no inline expressions).
- `fls:frontend-styling` + `brand-guidelines` — Tailwind conventions, theme role
  tokens, eyebrow/heading scale. No hardcoded hex.
- `fls:template` — cotton components, `partialdef`/`partial`, template conventions.
- `fls:testing` — pytest conventions for the view tests.
- `fls:multi-tenant` — site-aware querysets (the existing helpers already handle
  this; no new model work).

### Two registration sets are intentionally different (do NOT merge)

`all_courses` excludes only **in-progress** registrations (`registered_ids` from
`get_current_courses`) so completed courses appear as re-previewable cards. The
dashboard's Available section must exclude **all** registrations (via
`get_course_registrations`) because completed courses already render under
Learning History on the same page. Keep both derivations; do not refactor into a
single shared helper.

## Task 1 — View: compute `available_courses`

**File:** `freedom_ls/student_interface/views.py` (`dashboard` view).

Add `get_course_registrations` to the existing `from .utils import (...)` block
(`get_all_courses` is already imported).

After the existing `recommended_courses` annotation loop, build the capped,
filtered Available list. Pseudocode:

```python
registered_ids = {c.id for c in get_course_registrations(request.user)}
recommended_ids = {rec.collection_id for rec in recommended_courses}

available_courses: list[Course] = []
for course in get_all_courses():
    if course.id in registered_ids or course.id in recommended_ids:
        continue
    setattr(course, "is_registered", False)  # noqa: B010
    _annotate_preview_context(course, is_registered=False)
    available_courses.append(course)
    if len(available_courses) == 3:
        break

context = {
    "registered_courses": registered_courses,
    "completed_courses": completed_courses,
    "recommended_courses": recommended_courses,
    "available_courses": available_courses,
}
```

Notes:
- Exclusion happens **before** the cap (filter, then take first 3).
- `recommended_courses` is a `QuerySet`; it is already evaluated by the preceding
  annotation loop, so `rec.collection_id` reads from cache (no extra query). Use
  `collection_id` (FK id) — no need to touch `.collection`.
- Ordering is whatever `get_all_courses()` yields (same as all-courses page); do
  **not** add ordering.
- No `today`/date passed to the template — the date is client-side only.

**Tests (write first)** in `test_course_list_views.py`:
1. `available_courses` excludes every registered course — both in-progress and
   **completed** (register one course, complete another; neither appears in
   `response.context["available_courses"]`).
2. Excludes courses present in `recommended_courses` (recommend course X → X not
   in available).
3. Capped at 3 (create ≥4 eligible courses → `len(available_courses) == 3`).
4. Eligible courses with no registration/recommendation **do** appear.
5. Each available course has preview annotations (`preview_is_registered is
   False`, has `preview_start_url`).

## Task 3 — JS: `dateEyebrow` Alpine component

**File:** `freedom_ls/student_interface/static/student_interface/js/alpine-components.js`.

Register a new component alongside `coursePart`. It formats `new Date()` in the
browser's own locale + timezone and refreshes when the tab regains visibility
(no per-minute tick). Pseudocode:

```javascript
Alpine.data("dateEyebrow", () => ({
    formatted: "",
    _onVisibility: null,
    init() {
        this._format();
        this._onVisibility = () => {
            if (!document.hidden) this._format();
        };
        document.addEventListener("visibilitychange", this._onVisibility);
    },
    destroy() {
        if (this._onVisibility) {
            document.removeEventListener("visibilitychange", this._onVisibility);
        }
    },
    _format() {
        this.formatted = new Date().toLocaleDateString(undefined, {
            weekday: "long",
            day: "numeric",
            month: "long",
            year: "numeric",
        });
    },
}));
```

- `undefined` locale → browser locale; no timezone arg → browser timezone.
- Spelled-out result (e.g. `Friday, 30 May 2026` / `Friday, May 30, 2026`) —
  exact word order follows the browser, never numeric.
- `destroy()` removes the listener (per `fls:alpine-js` rule 9).
- Update the skill's "Existing Components" table is **not** required, but keep the
  component minimal and clearly commented (skill's IMPORTANT note).

This component cannot be exercised by Django unit tests — its behaviour is
verified in the browser.

## Task 5 — Template: new "Available courses" section + header link

**File:** `partials/course_list.html`.

5a. **Add an `available-courses` partialdef** modelled on `recommended-courses`
but **without `.surface`**, with a header row containing the heading *and* a
right-aligned "Browse all courses" link, rendering the unregistered preview card
via the existing `dispatch-card` seam (so `course_card.html`'s not-registered
branch is reused — desktop modal, mobile preview link). Pseudocode:

```html
{% partialdef available-courses %}
    {% if available_courses %}
        <div id="available-courses" class="space-y-4">
            <div class="flex items-center justify-between gap-4">
                <h2 class="<section-heading tokens>">Available courses</h2>
                <a href="{% url 'student_interface:courses' %}"
                   class="<link tokens>">
                    Browse all courses
                    <c-icon name="..." aria-hidden="true" />{# decorative arrow only #}
                </a>
            </div>
            <div class="grid gap-4 md:grid-cols-2 lg:grid-cols-3">
                {% for course in available_courses %}
                    {% partial dispatch-card %}
                {% endfor %}
            </div>
        </div>
    {% endif %}
{% endpartialdef %}
```

- The `{% if available_courses %}` guard makes the **whole section (heading +
  link) hide gracefully** when empty — no empty box, no orphan link (success
  criterion 5).
- The link is a real, keyboard-focusable `<a>` with self-describing visible text
  **"Browse all courses"**; the arrow is decorative (`aria-hidden`). Use the
  `fls:icon-usage` skill for the arrow glyph (it's optional/decorative).
- `dispatch-card` uses `course` (not `course_status`), so the loop variable
  `course` drives the unregistered preview card — matching how
  `recommended-courses` aliases `course=recommendation.collection`.

5b. **Page order.** Update the bottom of `course_list.html` so partials render:
`current-courses` → `recommended-courses` → `available-courses` →
`learning-history`.

5c. **Remove the bottom "All Courses" button.** Delete the
`<c-button-group variant="centered"> … <c-button>All Courses</c-button> …` block
at the foot of `course_list.html`. The header-right "Browse all courses" link
replaces it.

5d. **Keep** the empty-state "Browse courses" `<c-button>` inside
`current-courses` (shown when the learner has no registered courses).

**Tests (write first)** in `test_course_list_views.py` (render assertions on
`response.content`, **no CSS/style assertions**):
1. When eligible courses exist, the dashboard HTML contains an `<a ... href="{%
   url 'student_interface:courses' %}">` with text "Browse all courses".
2. When `available_courses` is empty (every course registered or recommended),
   the response does **not** contain "Available courses" heading text nor the
   "Browse all courses" link.
3. The old bottom "All Courses" button text is gone from the dashboard.
4. The empty-state "Browse courses" button still renders when the user has no
   registered courses.
5. A **completed** course appears under Learning History but **not** in the
