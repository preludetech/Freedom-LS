# Spec: Course-player image spotlight modal

## 1. Summary

Redesign the course-player image lightbox into a good-looking **spotlight**: a
clean image card centred on a dimmed, blurred backdrop, with the figure
number/title and a description arranged as chrome around it.

This is a **redesign of existing functionality**, not a new feature. The inline
thumbnail card and its "Open image" trigger stay; only the expanded overlay is
rewritten. The rewrite swaps the hand-rolled Alpine `x-show` overlay for a native
`<dialog>` opened with `showModal()`, and ships a cross-cutting refactor that
unifies the three inconsistent modal backdrops in the codebase behind one shared
utility.

Source design: `image.png` (drawn in the `first_class` theme). The `default`
theme inherits sensible results for free via semantic tokens.

Research backing this spec:

- `research_lightbox_ux.md` — lightbox/spotlight UX & accessibility best practice
  (native `<dialog>`, scrim opacity, the backdrop-click pitfall, caption
  placement, reduced motion, mobile).
- `research_codebase_conventions.md` — current lightbox, standard modal, theme
  tokens, brand guidelines, icons (file paths + line numbers).

## 2. Why this matters

The current overlay (`freedom_ls/content_engine/templates/cotton/picture.html`
lines 44–77, backed by the `contentLightbox` Alpine component) looks unfinished
and is harder to maintain than it needs to be:

- **No dark scrim.** It only blurs (`backdrop-blur-lg`); the page bleeds through
  and the modal feels ungrounded. Light-content images appear to float with no
  separation from the page.
- **Close button is a corner hack.** `absolute top-0 right-0` with `opacity-50`
  and an oversized `size-12` icon — no padding, no rest-state affordance.
- **Caption floats in dead space.** A `text-4xl` headline pinned `absolute top-0`,
  intrusive and structurally inconsistent (absent when there is no caption).
- **No spotlight structure.** The image sits in a plain `bg-surface rounded-sm`
  box with no header row, no description area, no shadow, no defined border.
- **Re-implements browser behaviour.** The Alpine component hand-rolls a focus
  trap (`onTab`), Escape handling (`onEscape`), and manual trigger
  capture/restore (`_trigger`) — all of which `<dialog>.showModal()` provides
  natively and correctly.

Fixing the look and moving to `<dialog>` removes code *and* improves
accessibility, because the browser's modal semantics replace the hand-written
ones.

## 3. Goals

1. A clean spotlight: image card on a dimmed + blurred backdrop, with figure
   chrome (number/title top-left, close top-right, title + description
   bottom-left) faithful to `image.png`.
2. Rebuild the lightbox as a native modal `<dialog>` (`showModal()`), mirroring
   the existing `sidePanel` precedent but much smaller. Delete the now-redundant
   focus-trap / Escape / trigger-restore code.
3. Hard-rename the `c-picture` `caption` attribute to `title` and add a new
   `description` attribute, updating the full blast radius in this PR.
4. Extract the modal backdrop look (dark scrim + blur) into one shared utility
   and apply it consistently across all three modal surfaces.
5. Keep all three standard close mechanisms (button, Escape, click-outside) and
   the existing inline thumbnail card untouched in behaviour.

## 4. Out of scope (explicitly NOT building)

The mockup includes a toolbar and arrows that are **not** implemented:

- no zoom controls;
- no download button;
- no pagination / next–previous arrows;
- no thumbnail strip.

`c-modal` is **not** converted to a `<dialog>`. It has many consumers plus
form-reset and HTMX-204 auto-close logic in its Alpine component; it stays the
overlay-`<div>` consumer of the shared backdrop utility. Only the lightbox —
which is being rewritten here anyway — becomes a dialog.

## 5. Component attribute changes

### 5.1 New attribute set for `c-picture`

| Attribute     | Status                  | Use                                                   |
|---------------|-------------------------|-------------------------------------------------------|
| `src`         | unchanged (required)    | file path, resolved via `get_file_by_path`            |
| `alt`         | unchanged               | screen-reader alt text                                |
| `title`       | **renamed** from `caption` | visible title (top-left + bottom; figcaption on thumbnail) |
| `description` | **new** (optional)      | longer description shown at the bottom of the spotlight |
| `number`      | unchanged               | figure number → "Figure N" prefix                     |

Authoring rules carry over: `alt` describes the image for screen-reader users and
must not duplicate the visible `title`; decorative images use `alt=""`.
`description` is optional and only meaningful in the expanded spotlight.

### 5.2 Rename strategy: hard rename (clean break)

**Decision (confirmed with the user):** `caption` is renamed to `title` with no
backward-compatible alias. `caption` stops being a recognised `c-picture`
attribute. Rationale: a single clean attribute set is simpler to document and
reason about than carrying two names; the cost is updating every current
producer/consumer of `caption`, all of which live in this repo and are updated in
this PR.

> Trade-off noted: FLS is an installable, extensible library, so downstream
> projects that authored content with `caption=` will need to migrate to `title=`.
> This is an accepted breaking change for the in-repo redesign; it should be
> called out in the PR description / changelog so downstream installs know to
> update their content. (The `![[file | …]]` shorthand still works — see 5.3 —
> so content using the shorthand needs no change.)

### 5.3 Full blast radius of the rename (all in this PR)

Every place that produces or consumes the `caption` attribute must move to
`title`:

1. **Component definition** — `freedom_ls/content_engine/templates/cotton/picture.html`:
   `<c-vars>` (`caption=""` → `title="" description=""`), the figcaption/thumbnail
   markup, and the lightbox markup. Update the authoring-note comment.
2. **Sanitiser allow-list** — `config/settings_base.py` `MARKDOWN_ALLOWED_TAGS`
   line 291: `"c-picture": {"src", "alt", "caption", "number"}` →
   `{"src", "alt", "title", "description", "number"}`. (Tags not in this set are
   stripped by the markdown sanitiser, so `title`/`description` must be added or
   they will never render.)
3. **Markdown shorthand rewriter** — `content_save.py` `markdown_translate`
   (line ~316): the `![[file | caption]]` pattern must emit `title="…"` instead of
   `caption="…"`. The shorthand's single text slot maps to `title` (there is no
   shorthand slot for `description`; `description` is authored only via the
   explicit `<c-picture description="…">` form). Update the explanatory comments
   that say "caption".
4. **Tests that assert on the attribute** —
   `freedom_ls/content_engine/tests/test_content_save_save_with_uuid.py` and
   `freedom_ls/markdown_rendering/tests/test_markdown_utils.py` reference
   `caption="…"` in `c-picture` fixtures/expectations; update them to `title`.
   (The `c-youtube`/`c-table` `caption` usages in those files are unrelated and
   stay as-is — the rename is scoped to `c-picture` only.)
5. **Demo content** — every `<c-picture … caption="…">` in `demo_content/`
   (markdown topics and quiz `*.yaml` pages, including grid and stacked-gallery
   examples) → `title="…"`. The authoring docs in
   `demo_content/functionality_demo_content_widgets/2. media.md` and
   `…/functionality_demo_end_with_topic/5. topic.md` that describe the `caption`
   attribute must be rewritten to describe `title` (+ mention `description`).
6. **Component docs/comments** — `image-grid.html`'s comment refers to the
   `![[file|caption]]` shorthand; reword to match the new behaviour ("each child
   keeps its own title and lightbox").

> Note: the `caption` attribute on **other** components (`c-youtube`, `c-table`,
> `c-pdf-embed`) is out of scope and unchanged. Only `c-picture` is renamed.

## 6. The spotlight `<dialog>`

### 6.1 Native dialog, not an Alpine `x-show` div

**Decision:** build the spotlight as a native modal `<dialog>` opened with
`showModal()`, following the established `sidePanel` precedent
(`freedom_ls/base/static/base/js/alpine-components.js`, the `sidePanel`
controller, and `_base_interface.html` line 45). `showModal()` provides for free:

- focus trap (browser-managed Tab/Shift+Tab wrapping);
- Escape-to-close;
- inert background + correct `role="dialog"` / modal ARIA semantics;
- focus restore to the trigger on close;
- a `::backdrop` pseudo-element for the scrim.

This lets us **delete** the bespoke `contentLightbox` logic that re-implements
these: the `onTab` focus trap, the `onEscape` handler, and the manual `_trigger`
capture/restore all go away.

### 6.2 Replacement Alpine controller

A new, much smaller controller replaces `contentLightbox` in
`freedom_ls/content_engine/static/content_engine/js/alpine-components.js`
(CSP build — register via `Alpine.data()`, no inline expressions). It mirrors
`sidePanel` but minimal:

- `open()` — call `dialog.showModal()` on the dialog `$ref`.
- a backdrop/click-outside handler — close only when the click is outside the
  spotlight card (see 6.4).
- focus restore is handled natively by `showModal()`; no manual `_trigger` is
  needed. (If the design wants focus to land on the close button on open, use the
  `autofocus` attribute on the close button per the research recommendation,
  rather than JS.)

The equation component in the same file is unrelated and stays.

### 6.3 Layout (follow `image.png`)

Text sits **on the dark backdrop chrome** (light text on the dimmed scrim, so
contrast is guaranteed), arranged around a centred image card:

- **Top-left:** figure number + title (e.g. "Figure 3 — Propeller inspection
  points"). Marked up as the dialog's heading and used as `aria-labelledby`.
- **Top-right:** close (X) button.
- **Centre:** the spotlight card holding the image — rounded, soft shadow, image
  scaled to fit with `object-contain`.
- **Bottom-left:** title + description.

The title intentionally appears in **both** the top-left and the bottom — this is
faithful to the mockup and confirmed as intended. The figure-number prefix
("Figure N") appears with the top-left title; the bottom shows the title +
description without the number prefix.

When `title` is empty, the chrome must degrade cleanly (no empty heading box);
when `description` is empty, the bottom block omits it. The dialog still needs an
accessible name in that case (fall back to `aria-label` from `alt`).

### 6.4 Backdrop & close behaviour

- **Backdrop:** dark scrim + blur via the shared utility (section 7). Because the
  spotlight is a native `<dialog>`, the dialog box itself stays `bg-transparent`
  and fills the viewport; the dim + blur is carried by the `::backdrop` pseudo
  (`.modal-backdrop-host::backdrop`). The scrim sits outside the theme tokens so
  it stays consistent across themes.
- **Close mechanisms (all three, standard):**
  - close button (top-right);
  - `Escape` (native to `showModal()`);
  - click-outside — clicking the surrounding scrim closes; clicking the white
    card or the image must **not** close.
- **Click-outside guard.** A modal `<dialog>` reports `event.target` as the inner
  element when the card/image is clicked, and the `::backdrop` pseudo cannot
  receive events by default. Use the documented robust pattern: the full-viewport
  dialog means backdrop clicks land on the dialog element, so guard on
  `event.target === dialog` to close (mirroring the `sidePanel` mobile handler at
  `alpine-components.js` lines 365–371). The card/image are descendants, so their
  clicks are correctly ignored. (The research's alternative `pointer-events: none`
  + `document`-level `.closest('dialog')` approach is available if the
  full-viewport-dialog target check proves unreliable in practice.)

## 7. Shared `.modal-backdrop` utility (cross-cutting refactor)

### 7.1 Problem

The three modal backdrops are inconsistent — only the side panel dims:

| Place | Markup | Scrim | Blur |
|-------|--------|-------|------|
| `_base_interface.html` side panel | native `<dialog>` `::backdrop` | `bg-black/50` | `backdrop-blur-sm` |
| `cotton/modal.html` | Alpine overlay `<div>` | none | `backdrop-blur-sm` |
| `cotton/picture.html` lightbox | Alpine overlay `<div>` | none | `backdrop-blur-lg` |

### 7.2 Single source of truth

Add one utility to `tailwind.components.css` (the documented home for reusable
primitives — `@layer components`). It needs two hooks because the scrim is
rendered two different ways — a real overlay `<div>` (Alpine) vs the `::backdrop`
pseudo of a native `<dialog>`, whose own box must stay `bg-transparent`:

```css
/* @layer components */
.modal-backdrop,                 /* overlay <div>: Alpine modals + lightbox */
.modal-backdrop-host::backdrop { /* native <dialog>: styles its ::backdrop  */
    @apply bg-black/50 backdrop-blur-sm;
}
```

The class carries **only** the scrim look (colour + blur). Positioning,
`z-index`, and transitions stay as inline utilities on each element. Blur strength
is standardised at `backdrop-blur-sm`; if the spotlight wants a stronger blur it
layers one inline rather than forking the utility.

### 7.3 Apply in all three places

**Decision (confirmed with the user):** standardise on dim + blur everywhere.

- **Side panel** (`_base_interface.html` line 47): add `modal-backdrop-host`,
  drop the inline `backdrop:bg-black/50 backdrop:backdrop-blur-sm` (same look, now
  shared).
- **`cotton/modal.html`** (backdrop div, line 38): add `modal-backdrop`. **This is
  a deliberate, visible change** — standard modals (course preview, delete
  dialogs, etc.) gain the dark scrim they currently lack. QA should eyeball the
  main `c-modal` consumers to confirm the new scrim reads well.
- **New spotlight lightbox** (native `<dialog>`): add `modal-backdrop-host`
  (the `::backdrop` hook), alongside the side panel.

Hook split: `.modal-backdrop` → `c-modal` (overlay div);
`.modal-backdrop-host::backdrop` → side panel + spotlight (native dialogs).

This touches `base` templates + `content_engine` + `tailwind.components.css`; it
is in scope and ships in the same PR.

## 8. Theming

- Use semantic tokens so the card adapts automatically: `bg-surface`,
  `text-on-surface`, `text-muted`, `border-border`, `rounded-lg`, focus ring on
  `primary` (`--color-focus-ring` maps to primary), theme font stack.
- Light chrome text on the scrim uses a fixed light colour (the scrim is
  theme-neutral, so the text on it must be too — do not use `text-on-surface`
  there).
- Hard-code only the theme-neutral bits: the dark scrim opacity (`bg-black/50`),
  `shadow-xl`, transition timings. No `z-index` is needed — a modal `<dialog>`
  renders in the top layer.
- `first_class` is the source design; `default` gets sensible defaults for free
  via the tokens.

## 9. Accessibility

Most of this is free from `<dialog>`:

- Focus trap, Escape-to-close, inert background, and focus-restore-to-trigger all
  come from `showModal()` — no manual `onTab`/`onEscape`/`_trigger` code.
- Label the dialog via the title heading (`aria-labelledby`). `role="dialog"` and
  modal semantics are implicit for a modal `<dialog>`. When there is no `title`,
  fall back to `aria-label` derived from `alt`.
- Heading level: use an `<h2>` for the figure title per the dialog pattern (it is
  the labelled name of the dialog).
- **Close button:** always visible (no opacity hack), accessible label
  ("Close image"), comfortable touch target (~44×44 px), a standard `size-6`
  `c-icon name="close"` inside a properly padded button styled like the `c-modal`
  close button (`text-muted hover:text-on-surface focus:ring-2 focus:ring-primary
  rounded-md`) — not a bare `size-12` icon.
- **Image alt:** the spotlight `<img>` keeps a meaningful `alt`. If `description`
  meaningfully describes the image, associate it via `aria-describedby` on the
  `<img>`.

### 9.1 Transitions / reduced motion

- Native `<dialog>` enter/leave needs `@starting-style` +
  `transition-behavior: allow-discrete` (plus `overlay`/`display` discrete
  transitions). Copy the side-panel pattern already in
  `tailwind.base_interface.css` (the `.side-panel-dialog` rules). Because that
  file is documented as single-use for the one side-panel dialog, the spotlight's
  dialog/transition CSS lives in `tailwind.components.css` alongside the shared
  `.modal-backdrop` utility (reusable home), not in `tailwind.base_interface.css`.
- Respect `prefers-reduced-motion: reduce`: fade only, drop the scale animation.
  Do not animate the backdrop blur level on open/close.

### 9.2 Mobile / small screens

- The dialog fills the viewport and centres the card; the full-viewport dialog is
  also what makes click-outside land on the dialog element (see 6.4).
- The card adapts to small screens: `max-height: ~90dvh`, with scroll inside the
  card for long descriptions (only the card interior scrolls, never the page
  behind). Image scales to fit.
- Card width: narrower than the viewport on desktop (visible scrim margin is the
  "this is a modal layer" cue); on mobile it may take most of the width with
  inner padding. Use `max-width` + `width`/viewport-relative sizing, not fixed
  pixel dimensions.

## 10. Implementation notes / to verify (not spec decisions)

- **`fullscreen` icon name — verified valid, keep as-is.** The "Open image"
  trigger uses `<c-icon name="fullscreen">` (`picture.html` line 37). This was
  checked against the icon registry: `fullscreen` is a registered semantic name
  in `freedom_ls/icons/semantic_names.py` and is mapped in
  `freedom_ls/icons/mappings.py` (→ `arrows-pointing-out` / `maximize`). The
  `icon-usage` skill's examples happen to show `expand` but do not enumerate the
  full set; `fullscreen` is correct. No change to the trigger icon is needed.
- The thumbnail card, its "Open image" button, and the `image-grid` layout
  wrapper keep their current behaviour — only the attribute name and the expanded
  overlay change.

## 11. Success criteria

1. Opening an image in the course player shows a centred image card on a **dark,
   blurred** scrim (no page bleed-through), matching `image.png` in `first_class`
   and rendering sensibly in `default`.
2. The figure number/title appears top-left, the close button top-right, and the
   title + description bottom-left, per the mockup. Empty `title`/`description`
   degrade cleanly.
3. The spotlight is a native modal `<dialog>` opened with `showModal()`; the old
   `contentLightbox` `onTab`/`onEscape`/`_trigger` code is gone.
4. All three close mechanisms work: close button, Escape, and click-outside —
   and clicking the card or the image does **not** close the dialog.
5. Keyboard + screen-reader behaviour is correct: focus is trapped in the dialog,
   Escape closes, focus returns to the "Open image" trigger on close, the dialog
   is labelled by its heading, and the close button has an accessible name and a
   ~44 px target.
6. `prefers-reduced-motion: reduce` removes the scale animation (fade only).
7. On small screens the card fits within the viewport (`~90dvh`), long
   descriptions scroll inside the card, and the page behind does not scroll.
8. The `caption` → `title` rename is complete and consistent: `c-vars`,
   `MARKDOWN_ALLOWED_TAGS`, the `![[file | …]]` rewriter, all affected tests, all
   `demo_content` `c-picture` usages, and the authoring docs all use `title`
   (with the new optional `description`). No `c-picture caption=` remains in the
   repo. `c-youtube`/`c-table`/`c-pdf-embed` `caption` usages are untouched.
9. The shared `.modal-backdrop` / `.modal-backdrop-host::backdrop` utility exists
   in `tailwind.components.css` and is applied to all three surfaces; the side
   panel looks unchanged, and `c-modal` now shows a dark scrim.
10. `uv run pytest` passes (including the updated rename tests) and
    `npm run tailwind_build` succeeds.
