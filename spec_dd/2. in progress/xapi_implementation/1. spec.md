# Lightweight xAPI Event Tracking — Specification

## Overview

FLS needs an immutable, queryable record of what users do on the platform. We want this to be useful for content-quality analytics ("where do people get stuck", "how long does this take"), for educator-facing progress views, for compliance/audit, and for future engagement-tracking work — without dragging in a full Learning Record Store (LRS) or external dependency.

We use **xAPI as a design guide, not a compliance target**. The actor / verb / object / result / context shape gives us a coherent vocabulary; the standard ADL verb IRIs let us avoid arguing about what to call things; but persistence is plain PostgreSQL with typed hot columns plus a JSONB payload, queried through the Django ORM. No LRS REST API, no statement forwarding, no signed statements, no sub-statements.

The spec covers the storage model, the `track()` helper API, the schema catalogue and validation layer, the immutability guarantees, multi-tenant isolation, the privacy/erasure path, and the Claude skills that future contributors will use to extend it. Frontend tracking is out of scope (separate spec — see `spec_dd/0. drafts/frontend_engagement_tracking/`); the storage model is JSONB-flexible so it can be added later without retrofitting.

**Why this matters:** Tracking that's bolted on after the fact ends up half-implemented, inconsistently called, full of gaps, and impossible to audit. By committing to a single helper API, an enforced schema catalogue, and a documented "how to add an event type" workflow up-front, the cost of recording the 50th kind of event stays roughly the same as recording the 5th.

## Scope

In scope:

- Replace the empty `xapi_learning_record_store` app with a new `experience_api` app
- Single immutable storage model for xAPI-shaped events, multi-tenant via `SiteAwareModel`
- `track()` helper that accepts Django model instances and infers/derives context where possible
- Verb constants module (`verbs.py`) backed by ADL IRIs
- Pydantic-based schema catalogue with required/optional/snapshot fields per `(verb, object_type)` pair, validated at write time
- Async-capable write path via the Django Tasks framework, defaulting to `ImmediateBackend`
- Right-to-erasure support: a separate, audited operation that anonymises actor snapshots
- Initial event coverage: topic viewed/completed, form attempted/answered/completed, course registered/progressed
- Claude skill (`experience-api`) and resource doc that codify the "add an event type" and "emit an event" workflows
- Tests at every layer (validation, end-to-end, deletion-survival)

Out of scope:

- Full LRS REST API
- Statement forwarding to external LRSs
- Signed statements, sub-statements, or the xAPI Profile API
- OAuth endpoints
- Voiding mechanism (the model leaves room for a `VOIDED` verb later but no UX is built)
- Frontend / browser-side event capture (deferred to a separate spec)
- Analytics dashboards (deferred — the data model is designed to support them)
- Monthly partitioning of the events table (deferred until volume warrants it)
- TimescaleDB or ClickHouse (revisit at >100GB raw or >500M total rows)

## Dependencies

- Existing Django Sites framework + `SiteAwareModel` (already in place)
- Existing `_thread_locals.request` pattern from `freedom_ls/site_aware_models/middleware.py` (the tracker uses this to derive site, session, IP, user-agent without forcing every call site to pass a `request`)
- New package: `pydantic` (for schema validation; v2.x)
- Django 6's built-in Tasks framework (`django.tasks`) — no extra package needed. Configured with `ImmediateBackend` initially.

Use `uv` to install `pydantic`. No other new infrastructure (same database, same Django, same Postgres).

## App structure

### Renaming `xapi_learning_record_store` to `experience_api`

The existing `freedom_ls/xapi_learning_record_store/` directory contains only commented-out scaffolding — no real models, no migrations, no admin classes, no API endpoints with handlers. Its `apps.py` even references a non-existent `freedom_ls.xapi` module. The "rename" is therefore a delete-and-replace, not a true Django app rename:

1. Delete `freedom_ls/xapi_learning_record_store/` in full
2. Create a new `freedom_ls/experience_api/` directory with the standard FLS app layout
3. `apps.py` → `name = "freedom_ls.experience_api"`, `label = "experience_api"`
4. Add to `INSTALLED_APPS` (and remove the old entry, if present)

## Storage model

Single Django model: `Event`. Stored in one table. Every row is one xAPI-shaped event.

### Fields

The model uses a **hybrid schema**: typed columns for fields we query frequently, plus a JSONB column for the full xAPI-shaped payload.

| Field | Type | Notes |
|---|---|---|
| `id` | UUIDField, primary key | Tracker-generated UUID. Referenced by `PROGRESSED` events' `trigger_event_id`. |
| `site` | FK → Site, `null=False, on_delete=PROTECT` | Multi-tenant pointer; inherited unchanged from `SiteAwareModelBase`. Deleting a site that still has events is blocked by `PROTECT`; we do not expect to delete a running site without first archiving its event history through a separate audited path. Auto-filled from `_thread_locals.request` on save. |
| `site_domain` | CharField, required | Snapshot of the site domain at event time. |
| `actor_user` | FK → `accounts.User`, `null=True, on_delete=SET_NULL` | Nullable pointer; may dangle. |
| `actor_email` | EmailField, required | Snapshot. Cleared by the erasure operation. |
| `actor_display_name` | CharField, required | Snapshot. Cleared by the erasure operation. |
| `actor_role` | CharField, required | `student` / `educator` / `admin` / `system`. |
| `actor_ifi` | CharField, required | The xAPI Inverse-Functional Identifier emitted in the JSONB statement. Account-based: the string `"<site_homepage>|<str(user.id)>"` (where `user.id` is the user's internal PK, a bigint). **Not** the email. (See "Privacy" below.) The field stores the serialised IFI so the audit record is self-contained if the underlying user is later deleted. |
| `verb` | CharField, indexed, required | The ADL IRI (e.g. `http://adlnet.gov/expapi/verbs/completed`). |
| `verb_display` | CharField, required | Human-readable label snapshot (e.g. `completed`). |
| `object_type` | CharField, indexed, required | The activity-type IRI suffix or our internal type name (e.g. `Topic`, `Form`, `Question`). |
| `object_id` | UUIDField, `null=True`, indexed | Nullable pointer to the underlying record (Topic, Form, etc.). May dangle. |
| `object_definition` | JSONField | The xAPI `object` block — full snapshot (slug, title, type, etc.). |
| `result` | JSONField, nullable | The xAPI `result` block (score, completion, success, duration, response). |
| `context` | JSONField | The xAPI `context` block — includes `contextActivities`, `extensions`, etc. All FK-shaped fields inside this are paired with their snapshots. |
| `statement` | JSONField | The full xAPI-shaped statement (denormalised composition of all of the above). This is what we'd hand to a future LRS export. |
| `timestamp` | DateTimeField, indexed, required | UTC, microsecond precision. Server-set on write. Never overwritten. |
| `stored` | DateTimeField, auto_now_add | Receipt time. Equal to `timestamp` in the synchronous-write path; differs once async writes are enabled. |
| `session_id_hash` | CharField, indexed, nullable | SHA-256 of the Django session key. Groups events from one browser session. Null for system / management-command events. |
| `user_agent` | CharField, nullable | Snapshot from request headers. |
| `ip_address` | GenericIPAddressField, nullable | Captured only if site privacy policy permits. |
| `platform` | CharField, required | `backend` or `frontend`. Always `backend` for the initial scope. |

Inheritance: `Event(SiteAwareModel)` to get site-scoped queryset filtering, `PROTECT` semantics on the `site` FK, and the auto-fill of `site_id` from `_thread_locals.request` on save. No fields are redeclared. Deleting a site is intentionally blocked while events exist for it; the operational expectation is that site teardown goes through a separate archive-then-delete process, not a casual `Site.delete()`.

### Indexes

- `(site, verb, timestamp DESC)` — the workhorse query: events for a site, by type, most recent first
- `(site, actor_user, timestamp DESC)` — per-user history
- `(site, object_type, object_id, timestamp DESC)` — per-object history
- BRIN on `timestamp` — lightweight, append-only column
- GIN on `context` with `jsonb_path_ops` — for `extensions` queries (e.g. by `course_id`, `cohort_id`)
- GIN on `result` with `jsonb_path_ops` — for score / completion queries

Why `jsonb_path_ops` over the default GIN ops: 16% storage overhead vs ~79% for the default, sufficient for `@>` containment queries which are the common pattern.

### Why a single table, not one-table-per-event-type

- Querying across event types ("everything this user did today") is the most common analytical pattern
- The schema catalogue + Pydantic validation enforces consistency without needing the database to enforce it
- Type-per-table is a refactor we can do later if a single event type becomes pathologically large; in practice, for the scales discussed in `research_database_strategy.md`, one table is fine for years

### Immutability — at the storage layer

The schema doc spells out the rules; the model enforces them:

1. `Event.save()` raises `EventImmutableError` if called on an existing row (i.e. `self.pk` already in the DB). New rows save normally.
2. `Event.delete()` raises `EventImmutableError`. There is no path to delete events from application code. (The retention-driven purge — when we add it — runs at the database level, behind a separate audited operation.)
3. The Django manager exposes no `update_or_create`. `bulk_update` is similarly blocked by the `save()` override only firing on the row level; we add a class-level guard on the manager that raises if `update()` or `delete()` is called on a queryset that includes already-persisted rows. (Practically: nothing in the app should ever call them on this model.)
4. All FK fields on the Event model are declared `null=True, on_delete=SET_NULL` — except `site`, which is inherited from `SiteAwareModelBase` and stays `null=False, on_delete=PROTECT`. None use `CASCADE`.
5. **Database-level enforcement** (defence in depth): a `pre_save` signal mirrors the model-level guard; a migration revokes `UPDATE` and `DELETE` on the events table from the app's database role. The erasure operation uses a separately-granted role.
6. **Erasure exception** — see "Privacy" below. Anonymisation of actor snapshots is the *one* documented mutation, performed by a dedicated management command, audited separately.

### Partition maintenance

Partition *activation* (converting the `events` table to a monthly range-partitioned table) is deferred per `research_database_strategy.md` — see "Open questions". The maintenance tooling, however, ships in the initial scope so activation is a single operator step later.

A management command `maintain_event_partitions` handles partition lifecycle:

```
uv run manage.py maintain_event_partitions [--lookahead-months N] [--retention-months M] [--drop-old] [--dry-run]
```

- **Idempotent and cron-friendly.** Safe to run on any schedule (daily is typical). No-op if the table is not (yet) partitioned — logs a warning and exits 0.
- **Creates upcoming partitions.** Ensures partitions exist for the next `--lookahead-months` (default: `3`). Partition naming: `events_YYYY_MM`.
- **Rotates old partitions.** If `--drop-old` is passed, `DETACH`es partitions older than `--retention-months` (default: `24`) and drops them. Without `--drop-old`, it only reports what would be rotated.
- **Backend.** DDL is issued via `django-postgres-extra`'s partitioning helpers (already needed for the `Event` model's composite-PK requirement once partitioned) or raw `RunSQL` — the choice lands with the activation migration. The command itself is a thin wrapper so the mechanism can evolve without a CLI change.
- **Audit.** Each partition-create / detach / drop action is logged (structured log line; no DB mutation to the `events` table).
- **Scheduling.** The command is invoked by whatever scheduler the deployment uses (cron, systemd timer, Kubernetes CronJob). FLS does not ship a scheduler.

Until activation, the command exists as documented scaffolding and is covered by a smoke test that asserts it exits 0 with the expected "not partitioned — nothing to do" warning.

### Object types

`freedom_ls/experience_api/object_types.py` defines the object-type names used in the `object_type` column:

- `Topic` — maps to `freedom_ls.content_engine.models.Topic`
- `Form` — maps to `freedom_ls.content_engine.models.Form`
- `Question` — maps to `freedom_ls.content_engine.models.FormQuestion` (the codebase's model class is `FormQuestion`; the xAPI `object_type` label is the shorter, user-facing `Question`)
- `Course` — maps to `freedom_ls.content_engine.models.Course`

Internal names rather than IRIs. The full IRI form is in the JSONB `object_definition` block per the schemas. The registry in `snapshots.py` keys on the Django model class, so passing a `FormQuestion` instance to `track()` resolves to `object_type="Question"`.

## Schemas (`event_schemas.md` is the source of truth)

The full schema catalogue lives in `event_schemas.md` next to this spec. It is the **authoritative description** of every `(verb, object_type)` pair we emit, including:

- Required vs optional fields
- The nullable-FK-plus-required-snapshot convention (every ID is nullable; every snapshot — slug, title, name, email — is required at write time)
- Global context fields (site, session, user-agent, IP, timestamp, platform) injected automatically by the tracker
- Actor snapshot fields (email, display name, role) injected automatically by the tracker
- The immutability rules and the validation behaviour

`schemas.py` mirrors the catalogue in code as Pydantic v2 models, one per `(verb, object_type)` pair. `schema_registry.py` provides:

```python
SCHEMA_REGISTRY: dict[tuple[Verb, ObjectType], type[BaseModel]] = {
    (VIEWED, Topic):        ViewedTopicSchema,
    (COMPLETED, Topic):     CompletedTopicSchema,
    (ATTEMPTED, Form):      AttemptedFormSchema,
    (ANSWERED, Question):   AnsweredQuestionSchema,
    (COMPLETED, Form):      CompletedFormSchema,
    (REGISTERED, Course):   RegisteredCourseSchema,
    (PROGRESSED, Course):   ProgressedCourseSchema,
}
```

Every Pydantic model produces validated dicts for the three xAPI sub-blocks (`object_definition`, `result`, `context.extensions`); the tracker composes the final statement.

## The `track()` helper

The public API has two layers:

1. A single generic `track(actor, verb, target, ...)` function — the one code path that validates, composes, and persists. All events flow through it.
2. Per-event-type wrapper functions (`track_topic_viewed`, `track_topic_completed`, `track_form_attempted`, `track_question_answered`, `track_form_completed`, `track_course_registered`, `track_course_progressed`) — thin convenience shims that hardcode the `(verb, object_type)` pair and type-hint the target. **Call sites use the wrappers.** The generic `track()` stays exported for tests, ad-hoc scripts, and the handful of places where the verb is chosen dynamically.

Both layers live in `tracking.py`. The wrappers are one-liners; the interesting code is all in `track()`.

### Public API — wrappers (the call-site idiom)

```python
from freedom_ls.experience_api.tracking import (
    track_topic_viewed,
    track_topic_completed,
    track_form_attempted,
    track_question_answered,
    track_form_completed,
    track_course_registered,
    track_course_progressed,
)

# Topic viewed — minimal
track_topic_viewed(request.user, topic, referrer=request.META.get("HTTP_REFERER"))

# Topic completed — caller passes the things the tracker can't derive
track_topic_completed(
    request.user, topic,
    result={"completion": True, "duration": "PT5M30S"},
    view_count=3,
    total_time_on_topic="PT12M00S",
    completion_source="manual",
)

# Form question answered — high-volume call site
track_question_answered(
    request.user, question,
    result={"response": answer_text, "duration": "PT45S",
            "score": {"raw": 1, "max": 1}, "success": True},
    form_attempt_id=form_progress.id,
    attempt_number=1,
    question_position=3,
    changed_answer=False,
)

# Course registered
track_course_registered(
    request.user, course,
    result={"success": True},
    registered_by="self",
    cohort=cohort,
)
```

### Public API — generic `track()` (escape hatch)

```python
from freedom_ls.experience_api.tracking import track
from freedom_ls.experience_api.verbs import VIEWED, COMPLETED

# Equivalent to track_topic_viewed(...)
track(request.user, VIEWED, topic, referrer=request.META.get("HTTP_REFERER"))

# Dynamic-verb cases (rare): loop over a batch of mixed events
for item in batch:
    track(item.actor, item.verb, item.target, **item.extensions)
```

### Signature

```python
def track(
    actor: User,
    verb: Verb,
    target: models.Model,
    *,
    result: dict | None = None,
    request: HttpRequest | None = None,
    strict: bool | None = None,
    **context_extensions: object,
) -> Event: ...
```

- `actor`, `verb`, `target` — required positional. `target` is the Django model instance the action was performed against (the xAPI `object`); it's named `target` here to avoid shadowing the Python builtin `object` and to keep the type hint `models.Model`. The tracker extracts its type from the registry mapping and pulls the snapshot fields (slug, title, etc.) via the appropriate `snapshots.py` helper.
- `result` — optional dict matching the `result` Pydantic schema for the `(verb, object_type)` pair. The tracker validates and persists.
- `request` — optional explicit request. If not passed, the tracker falls back to `_thread_locals.request` (already in place via `site_aware_models/middleware.py`). If neither is available, the tracker sets `platform="backend"`, `session_id_hash=None`, `user_agent=None`, `ip_address=None` — these are all nullable / system-event-friendly. The `actor_role` defaults to `"system"` if no request is available.
- `strict` — optional per-call override of `EXPERIENCE_API_STRICT_VALIDATION`. `None` (default) means "use the setting". Named as a reserved keyword in the signature so it doesn't collide with schema extension fields; `strict` is therefore **never** a valid schema extension key.
- `**context_extensions` — schema-defined `context.extensions` fields. The tracker validates them against the registered Pydantic model for the `(verb, object_type)` pair. Unknown keys are persisted but logged at warning level. Missing required keys raise `TrackingSchemaError`. Note the value annotation uses `object` (not `Any`) per the project no-`Any` rule; Pydantic handles per-field type coercion internally.

### Wrapper signatures

Each wrapper hardcodes its `(verb, object_type)` pair, narrows the `target` type hint to the concrete Django model, and delegates to `track()`. One wrapper per entry in `SCHEMA_REGISTRY`; adding an 8th event type adds an 8th wrapper.

```python
def track_topic_viewed(
    actor: User,
    topic: Topic,
    *,
    result: dict | None = None,
    request: HttpRequest | None = None,
    strict: bool | None = None,
    **context_extensions: object,
) -> Event:
    return track(actor, VIEWED, topic,
                 result=result, request=request, strict=strict,
                 **context_extensions)

def track_topic_completed(actor: User, topic: Topic, *, ...): ...
def track_form_attempted(actor: User, form: Form, *, ...): ...
def track_question_answered(actor: User, question: FormQuestion, *, ...): ...
def track_form_completed(actor: User, form: Form, *, ...): ...
def track_course_registered(actor: User, course: Course, *, ...): ...
def track_course_progressed(actor: User, course: Course, *, ...): ...
```

Rules for wrappers:

- The wrapper body is a single `return track(...)`. No extra logic, no branching, no per-event default values baked in — all of that belongs in the schema / in `track()`.
- The target parameter is named for its type (`topic`, `form`, `question`, `course`) rather than `target`, for readability.
- The `result`, `request`, `strict`, `**context_extensions` kwargs pass through untouched.
- A wrapper whose schema declares no `result` block (e.g. `(VIEWED, Topic)`) still accepts `result=None` — it's forwarded, and Pydantic validation handles the rest. Don't special-case that in the wrapper.

### What the tracker derives automatically

Per the schemas' "global context" and "actor snapshot" sections, the tracker injects on every call:

- `site`, `site_domain` — from `_thread_locals.request` or current `Site`
- `session_id_hash` — SHA-256 of `settings.SECRET_KEY + request.session.session_key`, lower-case hex; null if no request or no session key. Salting with `SECRET_KEY` makes hashes non-correlatable across installs and adds a defence-in-depth layer if the events DB is exposed without the salt. Rotating `SECRET_KEY` orphans historical session groupings, which is acceptable since session keys are short-lived anyway.
- `user_agent`, `ip_address` — from request headers; `ip_address` is `None` when `EXPERIENCE_API_CAPTURE_IP` is `False` (see "Privacy" and "Configuration")
- `timestamp` — `timezone.now()` at write time (UTC, microsecond precision)
- `actor_user`, `actor_email`, `actor_display_name`, `actor_role`, `actor_ifi` — from the `actor` argument; `actor_ifi` is the user's account-based IFI (`account.homePage = current site URL`, `account.name = str(user.id)`)
- `platform` — `"backend"` always for this scope

### What the tracker derives via relationship-walking

Per-object-type helpers in `snapshots.py` walk known relationships to pre-fill schema fields the caller didn't pass. Because FLS courses contain child content via `ContentCollectionItem` (a generic-FK through-model, see `freedom_ls/content_engine/models.py`) rather than via a direct FK, the walks below are reverse generic relationships and may return zero, one, or multiple parents. The tracker prefers exactly-one matches and falls back to requiring the caller to pass the field explicitly:

- `Topic` → reverse-walk via `ContentCollectionItem` to find the parent `Course` (directly, or via an intervening `CoursePart`) → `(course_id, course_slug, course_title)`. If the topic appears in exactly one course, auto-filled; if zero or multiple, the caller must pass `course` (a `Course` instance) or `course_id`/`course_slug`/`course_title` explicitly.
- `Form` → reverse-walk via `ContentCollectionItem` to find a parent `Topic` (when embedded) → `(topic_id, topic_slug, topic_title)`. Same zero/one/multiple semantics as above.
- `Form` → reverse-walk via `ContentCollectionItem` (through a `Topic` or `CoursePart` or directly) to find the parent `Course` → `(course_id, course_slug, course_title)`. Same zero/one/multiple semantics.
- `User + Course` → `UserCourseRegistration` lookup (filtering to active registrations for this user/course) → `(registration_id,)` — auto-filled when a single active registration exists; if none or multiple, the caller must pass `registration` explicitly.
- `User` → `CohortMembership` lookup → `(cohort_id, cohort_name)` — auto-filled when the user has exactly one cohort membership for the current site; otherwise the caller must pass `cohort` explicitly. **Cohort is not derivable from course registration**: `UserCourseRegistration` has no cohort field, and `CohortCourseRegistration` is separate. Cohort is attached via `CohortMembership`, which is per-user-per-site.

When relationship-walking fails to find a value for a *required* schema field, the tracker raises `TrackingSchemaError` with a clear message that names the walk that was attempted and the argument the caller should pass instead, e.g. "I tried to walk `Topic` → `ContentCollectionItem` → `Course` and got 2 candidate courses — pass `course=...` explicitly".

### Validation behaviour

Validation happens before any DB write:

1. The `(verb, object_type)` pair is looked up in `SCHEMA_REGISTRY`. Unknown pair → `TrackingSchemaError`.
2. The composed `object_definition`, `result`, and `context.extensions` dicts are passed to the Pydantic model. Missing required field → `TrackingSchemaError`. Type mismatch on an ID-shaped field → coerced if safe (int↔str), error otherwise. Unknown extra fields → log warning, persist anyway.
3. **Dangling-pointer guard**: if any schema-recognised `_id` field is populated but its paired `_slug`/`_title`/`_name` snapshot is missing, raise `TrackingSchemaError`. We never persist a pointer without its snapshot.

#### Strict-vs-permissive mode

A new setting `EXPERIENCE_API_STRICT_VALIDATION` controls failure mode:

- `True` (default in `settings_dev.py` and during tests): validation errors raise `TrackingSchemaError` and propagate. Useful for catching gaps during development.
- `False` (default in `settings_prod.py`): validation errors are logged at error level and the event is dropped — tracking failures must not break the user's primary action.

Per-call override: pass `strict=True` or `strict=False` to `track(...)` (the kwarg is reserved in the signature — see above — and is therefore never confused with a schema extension field). Tests that need to invert the default use this directly.

### Async write path

The actual DB write is wrapped in a Django-Tasks task:

```python
@task()
def write_event(payload: dict) -> None:
    Event.objects.create(**payload)
```

The `track()` helper validates and composes, then enqueues the task. With `ImmediateBackend` configured (the default for now) the task runs synchronously inside the calling request — same behaviour as `Event.objects.create(**payload)` would give us, but with the abstraction in place. Switching to a real backend (e.g. `DatabaseBackend` with a worker process) is a settings change with no application-code impact.

Because request-bound context (`_thread_locals.request`) is **not** available inside a worker process, `track()` must resolve and include every context-dependent field (`site_id`, `site_domain`, `session_id_hash`, `user_agent`, `ip_address`, etc.) in the payload **before** enqueuing the task. The task itself must not rely on `_thread_locals` or on the `SiteAwareModelBase.save()` auto-fill for `site_id`. Tests must cover this explicitly (one test fixture that enqueues the task without a request in scope and asserts the row is still correct).

### Return value

`track()` returns the persisted `Event` instance. Useful for tests; usually ignored at call sites.

## Initial event coverage

The seven `(verb, object_type)` pairs already specified in `event_schemas.md`:

| Verb | Object | Where it's emitted from |
|---|---|---|
| `VIEWED` | `Topic` | `student_interface` topic detail view |
| `COMPLETED` | `Topic` | `student_progress` topic-completion service / signal |
| `ATTEMPTED` | `Form` | `student_interface` form view, on first interaction |
| `ANSWERED` | `Question` | `student_interface` form-answer endpoint |
| `COMPLETED` | `Form` | `student_interface` form-submit endpoint |
| `REGISTERED` | `Course` | `student_management` registration service |
| `PROGRESSED` | `Course` | `student_progress` course-progress recompute service; the `trigger_event_id` field references the topic/form completion event that caused the progress update |

The `PROGRESSED` event chains off another event — the implementation must capture the upstream event's `id` and pass it through. The chain is one level deep (no recursive progression).

## Privacy

### Actor identification (IFI)

The xAPI `actor.account` IFI is `(homePage, name)` where:

- `homePage` = the current site URL (e.g. `https://lms.example.com`)
- `name` = `str(user.id)` — the opaque internal user ID

This is what gets serialised into the JSONB `statement.actor` block and what would be exported to a future LRS. **Email is never used as an IFI.** Email is captured separately as `actor_email` for in-database audit/debugging convenience and is the field that gets cleared by erasure.

### Right to erasure

A management command `erase_actor` performs a documented anonymisation of all events for one user:

```
uv run manage.py erase_actor --user-id <int> --confirm
```

`--user-id` takes the `accounts.User` primary key, which is a `BigAutoField` (integer) — not a UUID. (`SiteAwareModel` uses UUID primary keys, but `User` inherits from `SiteAwareModelBase` with `default_auto_field = BigAutoField`.)

Behaviour:

1. For every `Event` row where `actor_user_id == <id>` OR the user-identifying component of `actor_ifi` matches `str(<id>)` (IDs may have been nulled by deletion, but `actor_ifi` is always populated and contains `account.name = str(user.id)`):
   - Replace `actor_email` with `erased-<random>@example.invalid`
   - Replace `actor_display_name` with `Erased actor <random>`
   - Set `actor_user` to NULL (idempotent if already NULL)
   - Replace the `actor` block inside the JSONB `statement` with the same erasure tombstone
   - Leave `actor_ifi`, `verb`, `object`, `result`, `context`, `timestamp`, snapshots-of-things-other-than-the-user untouched
2. Write a separate `ActorErasure` audit row recording (acting admin, target user UUID, erased-actor token, count of events touched, timestamp). This row lives outside the `Event` table to avoid having to mutate it from a different code path.
3. The command refuses to run without `--confirm`. It also refuses if the target user still has any non-erasure-related history that would be broken (e.g. active `UserCourseRegistration` rows) — the user's own deletion path must run before erasure.

This is the **only** path that mutates already-persisted Event rows. The DB-level `REVOKE` on the application role means the regular app cannot perform the mutation; the erasure command runs as a separately-granted role.

### Other privacy controls

- IP capture is opt-in. Initial implementation uses a single global `EXPERIENCE_API_CAPTURE_IP` Django setting (defaults True in dev, False in prod). "Per-site" IP capture (so a multi-tenant installation can opt sites in independently) is deferred — see "Open questions / explicit deferrals". Until then, all sites hosted on a given deployment share the same setting.
- Tracking data stays internal. No third-party export in this scope.
- Retention policy: events are retained indefinitely in the initial scope. Documented as a TODO — the partitioning + retention design from `research_database_strategy.md` is the next iteration. Per-site retention overrides will be needed once we add this.

## Multi-tenant isolation

`Event(SiteAwareModel)` gives us:

- Automatic site filtering on `Event.objects.all()` (only sees events for the current site)
- Site-set on save (the model manager fills `site_id` from `_thread_locals.request.site`)
- The admin uses `SiteAwareModelAdmin`

Sites are populated from the request's site context at write time. The `site_domain` snapshot column survives subsequent site renames or deletions.

## Admin

`SiteAwareModelAdmin` subclass for read-only browsing:

- List display: `timestamp`, `actor_email`, `verb_display`, `object_type`, `object_definition` summary
- List filter: `verb`, `object_type`, date hierarchy on `timestamp`
- Search: `actor_email`, `actor_display_name`. (Searching over JSONB columns in the Django admin requires custom `get_search_results` overrides or typed-column copies of the searchable fields; for the initial scope we keep admin search to the typed columns only. A richer search is a follow-up.)
- Change form: read-only — every field, including the JSONB blocks, displayed but not editable. (The model's `save()` override would block edits anyway, but read-only admin makes the intent explicit and avoids confusing 500 errors.)
- No `add` permission, no `delete` permission. The admin is for inspection only.

## Future analytics — design implications already baked in

Course content quality monitoring is a future scope, but the data model already captures everything needed:

| Analytical question | Schema field that answers it |
|---|---|
| Which topics do learners struggle with? | `(COMPLETED, Topic)` `result.duration` distribution; `(VIEWED, Topic)` count before completion |
| Which questions trip people up? | `(ANSWERED, Question)` `result.success`, `result.duration`, `changed_answer`, `correct_answer` |
| Where do learners drop off in a course? | `(VIEWED, Topic)` ordered by `timestamp` per actor, terminated by absence of next-topic events |
| How long does this course take? | `(PROGRESSED, Course)` chain from `progress.scaled=0` to `1.0`, per actor |
| Which forms have a low pass rate? | `(COMPLETED, Form)` `result.success` aggregated by `form_id` |
| Which cohorts are succeeding? | All of the above grouped by `context.extensions.cohort_id` |

The cost of recording these fields *now* (durations, attempt counts, change counts, position-in-course, view counts, cohort/course pointers) is bytes. The cost of backfilling them later is impossible — the source data isn't reproducible.

### Resource: `${CLAUDE_PLUGIN_ROOT}/resources/experience_api.md`

- **Emit-an-event recipe** — how to pick the right `track_<event>` wrapper (or `track()` for dynamic-verb cases), which arguments are inferred vs must be passed, how to read a schema entry in `event_schemas.md` to know what `result` and `context.extensions` to pass, how to handle the relationship-walking failures
- **Add-a-new-event-type recipe** — the six-step workflow:
  1. Define the verb constant in `verbs.py` if it doesn't exist (reuse ADL IRIs where possible; document why if not)
  2. Add the `(verb, object_type)` schema entry to `event_schemas.md`, following the required/optional/snapshot conventions
  3. Add the matching Pydantic model to `schemas.py` and register it in `SCHEMA_REGISTRY`
  4. Add a one-line `track_<object>_<verb>` wrapper to `tracking.py` that delegates to `track()` (see "Wrapper signatures")
  5. Wire the wrapper call into the originating view/service; if the object's snapshot helpers don't exist yet, add them to `snapshots.py`
  6. Write tests: a validation test (required field missing → raises), an end-to-end test (call site → row in DB with expected snapshots), and a deletion-survival test (delete the referenced object, assert the event still reads cleanly)
- **Querying events** — typed-column lookups vs JSONB containment queries; when to reach for which; example queries for the analytical questions table above
- **Anti-patterns** — don't `Event.objects.create()`; don't add `_id` fields without snapshots; don't use `CASCADE` on any FK into the events table; don't mutate existing events (express corrections as new events); don't use email as an actor IFI

The resource doc uses worked examples from this codebase — real `track_<event>` wrapper calls from `student_interface`, real schema entries — not invented ones. We populate it with the seven initial event types as worked examples.

## Tests

Required test coverage for this spec:

- **Per `(verb, object_type)` end-to-end test** — one file per pair under `tests/test_per_event_type/`. Each test calls the matching `track_<event>` wrapper from a realistic call-site context, then asserts:
  - A row exists with expected typed columns
  - The JSONB `statement` matches the xAPI shape
  - All required snapshot fields are populated
  - The pointer FKs are populated when the source records still exist
- **Per-schema validation tests** — for each Pydantic model, one test per required field that omits it and asserts `TrackingSchemaError`
- **Deletion-survival tests** — for each event type, delete the object after the event, then re-fetch the event and assert all snapshot fields still read correctly and FK columns are NULL
- **Immutability tests** — assert `Event.save()` on an existing instance raises; assert `Event.delete()` raises; assert `Event.objects.filter(...).delete()` raises; assert the DB-level `REVOKE` is in place (probe via a raw connection that holds the app role)
- **Erasure tests** — assert the management command anonymises actor snapshots, leaves everything else untouched, writes the audit row; assert it refuses without `--confirm`; assert the `_thread_locals` are correctly threaded so the command runs as the erasure role
- **Multi-tenant tests** — assert that `Event.objects.all()` from site A does not see events from site B; assert that the `site_domain` snapshot survives site deletion
- **Strict / permissive validation mode** — assert `STRICT=True` raises, `STRICT=False` logs and drops; assert per-call override works
- **Async-task wiring** — assert that with `ImmediateBackend`, `track()` writes synchronously; assert that switching to a queued backend (in a test fixture) defers the write
- **Wrapper delegation** — for each `track_<event>` wrapper, assert that it calls `track()` with the correct hardcoded verb and forwards all other arguments unchanged (one parametrised test, not seven separate ones)
- **Skill files** — assert their existence and that frontmatter parses (existing pattern in the repo)

## Success criteria

1. The `xapi_learning_record_store` directory is gone; `freedom_ls/experience_api/` exists with the layout above and is in `INSTALLED_APPS`.
2. `track(user, verb, target, ...)` can be called from views, services, and management commands; it derives site, session, IP, user-agent from `_thread_locals.request` (or accepts an explicit `request=`). The seven per-event-type wrappers (`track_topic_viewed`, `track_topic_completed`, `track_form_attempted`, `track_question_answered`, `track_form_completed`, `track_course_registered`, `track_course_progressed`) exist in `tracking.py` and each is a one-line delegation to `track()`.
3. All seven initial `(verb, object_type)` pairs are wired into the originating call sites **via their wrappers** and have passing end-to-end tests.
4. Calling `track()` with a known `(verb, object_type)` and missing a required field raises `TrackingSchemaError` with a clear message that names the missing field.
5. Calling `Event.save()` on a previously-persisted row raises `EventImmutableError`. Calling `Event.delete()` raises. The DB role used by the application has `UPDATE` and `DELETE` revoked on the events table.
6. Deleting any object referenced by an event (user, course, cohort, form, question) leaves the event row intact and the snapshot fields still readable; only the FK column goes NULL. Deleting a `Site` that still has events is blocked by `on_delete=PROTECT` (intentional: site teardown is not a casual operation).
7. The `erase_actor` management command anonymises all events for a given user, writes an audit row, and is the only mutation path that succeeds against an existing row.
8. Events are site-scoped: a user logged into site A sees no events from site B in the admin, in querysets, or via `track()` writes.
9. The Django Tasks framework is configured with `ImmediateBackend`; events write synchronously today; switching to a queued backend is a settings change.
10. The `experience-api` skill exists in `fls-claude-plugin/skills/experience-api/` with `SKILL.md` and the linked resource file, and follows the conventions of the existing skills in the repo. A new contributor (human or AI) can use the skill to add an 8th event type without re-deriving the workflow.
11. All new code passes the project's existing lint, type-check, and pre-commit hooks. Type hints are required everywhere; no `Any`; modern union syntax (`X | None`, `list[str]`).
12. The `maintain_event_partitions` management command exists, is idempotent, exits 0 with a "not partitioned — nothing to do" warning when the `events` table is not partitioned, and accepts `--lookahead-months`, `--retention-months`, `--drop-old`, and `--dry-run` flags. A smoke test covers the pre-activation no-op path.

## Open questions / explicit deferrals

These are decisions the spec makes a default for, but flags for review:

- **Helper API shape** — *resolved: hybrid.* A single `track(actor, verb, target, ...)` function carries all the logic (validation, relationship-walking, composition, persistence), and per-event-type wrappers (`track_topic_viewed`, `track_topic_completed`, `track_form_attempted`, `track_question_answered`, `track_form_completed`, `track_course_registered`, `track_course_progressed`) provide one-line typed call-site shims. Call sites use the wrappers; `track()` remains the escape hatch for dynamic-verb and test code. This combines the research-doc recommendation with the idea text's central-pipe preference.
- **Salting `session_id_hash`** — *resolved: salt with `SECRET_KEY`.* Hashes are non-correlatable across installs and require both the DB and `SECRET_KEY` to link events back to raw session keys. Rotating `SECRET_KEY` orphans historical session groupings, which is acceptable.
- **Verbs `SUBMITTED`, `INTERACTED`, `VOIDED`** — defined as constants but no schema yet. They wait for their first call site / use case.
- **Partitioning** — table *activation* deferred, *tooling* in scope. The conversion to monthly range partitioning (per `research_database_strategy.md`) waits until storage exceeds ~10GB or query latency degrades. The `maintain_event_partitions` management command lands now (see "Partition maintenance") and is a safe no-op until activation; this means turning partitioning on is a single operator step plus a cron entry, not a scope discussion.
- **Retention policy** — explicitly deferred. Events are kept forever for now; per-site retention windows will need a follow-up spec.
- **Voiding mechanism** — explicitly out of scope. The `VOIDED` verb constant exists, the storage model doesn't prevent emitting `(VOIDED, Statement)` events, but no UX is built. Corrections in this scope are simply new events that supersede earlier ones in analytical queries.
- **Frontend tracking** — separate spec. The JSONB-flexible storage means it can land without retrofitting.
- **Skill split** — `xapi-schema-authoring` is not split out initially; revisit if `experience-api` grows past ~50 lines of `SKILL.md` content.
- **Form-attempt identity for `(ANSWERED, Question)`** — *resolved: reuse `FormProgress.id`.* FLS already has `student_progress.FormProgress` (one row per user-per-form-per-incomplete-attempt) with a UUID primary key inherited from `SiteAwareModel`. Call sites pass `form_attempt_id=form_progress.id`. No new schema; the snapshot in `event_schemas.md` continues to survive deletion of the `FormProgress` row because the UUID is copied into the event at write time.
- **Per-site IP-capture toggle** — the initial implementation uses a single global `EXPERIENCE_API_CAPTURE_IP` setting (see "Privacy"). Supporting per-site overrides (a new small `SiteSetting`-style model, or reuse of an existing one if any) is deferred; should it be in scope? User input needed.
- **Admin search over JSONB** — the admin is typed-column search only for this scope. If richer search across `object_definition` / `statement` is desirable sooner, a custom `get_search_results` is a small follow-up.
- **`Event` `site` FK semantics** — *resolved: inherit `PROTECT` unchanged.* Earlier drafts considered relaxing the `site` FK to `SET_NULL` so a site deletion wouldn't cascade-block the event history. Decision: keep `PROTECT` (the `SiteAwareModel` default). A running site is unlikely to be deleted without first archiving its event data, and `PROTECT` is the right safety rail for that. If a site teardown ever needs to happen, it goes through a separate audited archive-then-delete path, not a `Site.delete()`.
