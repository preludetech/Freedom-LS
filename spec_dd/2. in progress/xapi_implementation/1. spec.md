# Lightweight xAPI Event Tracking — Specification

## Overview

FLS needs an immutable, queryable record of what users do on the platform. We want this to be useful for content-quality analytics ("where do people get stuck", "how long does this take"), for educator-facing progress views, for compliance/audit, and for future engagement-tracking work — without dragging in a full Learning Record Store (LRS) or external dependency.

We use **xAPI as a design guide, not a compliance target**. The actor / verb / object / result / context shape gives us a coherent vocabulary; the standard ADL verb IRIs let us avoid arguing about what to call things; but persistence is plain PostgreSQL with typed hot columns plus a JSONB payload, queried through the Django ORM. No LRS REST API, no statement forwarding, no signed statements, no sub-statements.

The spec covers the storage model, the generic `track()` helper API, the schema-registration mechanism, the immutability guarantees, multi-tenant isolation, the privacy/erasure path, and the Claude skills that future contributors will use to extend it. Frontend tracking is out of scope (separate spec — see `spec_dd/0. drafts/frontend_engagement_tracking/`); the storage model is JSONB-flexible so it can be added later without retrofitting.

**Why this matters:** Tracking that's bolted on after the fact ends up half-implemented, inconsistently called, full of gaps, and impossible to audit. By committing to a single helper API, an enforced schema registry, and a documented "how to add an event type" workflow up-front, the cost of recording the 50th kind of event stays roughly the same as recording the 5th.

## Design principle — portable, domain-agnostic `experience_api`

The `experience_api` app is **generic event-logging infrastructure**. It knows how to persist an immutable xAPI-shaped row, validate against a registered Pydantic schema, isolate events by site, and erase actor PII on request. It does **not** know about topics, forms, questions, courses, cohorts, or registrations.

Concretely:

- `experience_api` depends only on `accounts` (for `User`) and `site_aware_models` (for `SiteAwareModel` and the `_thread_locals` plumbing).
- Domain apps (`student_interface`, `student_progress`, and any future consumer) **own their event types**: they define the Pydantic schemas, the snapshot helpers, and the per-event-type `track_*` wrappers in a module named `xapi_events.py` inside the owning app.
- Each domain app registers its event types with `experience_api`'s schema registry at `apps.py.ready()` time.

This split means `experience_api` can be copied into any Django project that has a `SiteAwareModel`-style base and a user model — no FLS-specific domain baggage travels with it.

## Scope

In scope for `experience_api`:

- Replace the empty `xapi_learning_record_store` app with a new `experience_api` app
- Single immutable storage model (`Event`) for xAPI-shaped events, multi-tenant via `SiteAwareModel`
- Append-only audit model (`ActorErasure`)
- Generic `track()` helper — takes a verb, an object-type string, pre-built snapshot dicts, and a caller-supplied result/context — validates against the registered Pydantic schema, composes the xAPI statement, and persists
- Verb constants module (`verbs.py`) backed by ADL IRIs
- A `BaseEventSchema` Pydantic base class and a small registry API: `register_event_type(verb, object_type, schema_cls)` / `get_schema(verb, object_type)`
- Async-capable write path via the Django Tasks framework, defaulting to `ImmediateBackend`
- Right-to-erasure support: a separate, audited operation that anonymises actor snapshots
- A `maintain_event_partitions` management command that is a no-op until partitioning is activated
- Claude skill (`experience-api`) and resource doc that codify the "add an event type in a consumer app" and "emit an event" workflows
- Tests at every layer (validation, immutability, erasure, multi-tenant isolation)

In scope for the domain apps (`student_interface`, `student_progress`):

- A `xapi_events.py` module in each app defining the Pydantic schemas, snapshot helpers, and `track_*` wrappers for the event types that app emits
- `apps.py.ready()` in each app imports `xapi_events` so the schemas register
- Call-site wiring from views/services to the local `track_*` wrappers
- Per-event-type tests (end-to-end, validation, deletion-survival)

- Full LRS REST API

### `student_interface/xapi_events.py` — topic/form/question/course emission

Defines Pydantic schemas, snapshot helpers, and wrappers for every event this app emits:

- Schemas: `ViewedTopicSchema`, `AttemptedFormSchema`, `AnsweredQuestionSchema`, `CompletedFormSchema`, `RegisteredCourseSchema`
- Snapshot helpers: `snapshot_topic`, `snapshot_form`, `snapshot_question`, `snapshot_course`, `walk_topic_to_course`, `walk_form_to_topic`, `walk_form_to_course`, `resolve_user_course_registration`, `resolve_cohort_for_user`

`student_interface/apps.py.ready()` imports `xapi_events` so that `register_event_type(...)` calls register the schemas at app load. This is the same pattern used elsewhere for signal registration.

### `student_progress/xapi_events.py` — topic-completion and course-progression emission

- Schemas: `CompletedTopicSchema`, `ProgressedCourseSchema`
- Snapshot helpers: reuse the topic/course snapshot helpers from `student_interface.xapi_events` by importing them (edge already exists: `student_progress --> student_interface`? **No** — `student_progress` does not currently depend on `student_interface`; instead, each app defines its own minimal snapshot helpers for the models it operates on, and `content_engine`-based helpers can live in `content_engine/xapi_snapshots.py` if duplication becomes painful. See "Snapshot-helper placement" below.)
- Wrappers: `track_topic_completed`, `track_course_progressed`
- `apps.py.ready()` imports `xapi_events` to register schemas.

### Snapshot-helper placement

Snapshot helpers walk model instances to build the `object_definition` / `context.extensions` dicts. They belong with the models they walk:

- `content_engine` snapshots (`snapshot_topic`, `snapshot_form`, `snapshot_question`, `snapshot_course`, `walk_topic_to_course`, `walk_form_to_topic`, `walk_form_to_course`) → `freedom_ls/content_engine/xapi_snapshots.py`. Both `student_interface` and `student_progress` already depend on `content_engine`, so importing these is a no-new-edge operation.
- `student_management` snapshots (`resolve_user_course_registration`, `resolve_cohort_for_user`) → `freedom_ls/student_management/xapi_snapshots.py`. Both `student_interface` and `student_progress` already depend on `student_management`.

The `xapi_snapshots.py` modules are **pure read helpers** — no writes, no registration calls. They do not import from `experience_api`. They do not constitute a new dependency edge; they are plain functions next to the models they operate on.

## Storage model

Single Django model: `Event`. Stored in one table. Every row is one xAPI-shaped event.

### Fields

The model uses a **hybrid schema**: typed columns for fields we query frequently, plus a JSONB column for the full xAPI-shaped payload.

| Field | Type | Notes |
|---|---|---|
| `id` | UUIDField, primary key | Tracker-generated UUID. Referenced by downstream events (e.g. `PROGRESSED` via `trigger_event_id`). |
| `site` | FK → Site, `null=False, on_delete=PROTECT` | Multi-tenant pointer; inherited unchanged from `SiteAwareModelBase`. Deleting a site with events is blocked. Auto-filled from `_thread_locals.request` on save. |
| `site_domain` | CharField, required | Snapshot of the site domain at event time. |
| `actor_user` | FK → `accounts.User`, `null=True, on_delete=SET_NULL` | Nullable pointer; may dangle. |
| `actor_email` | EmailField, required | Snapshot. Cleared by the erasure operation. |
| `actor_display_name` | CharField, required | Snapshot. Cleared by the erasure operation. |
| `actor_ifi` | CharField, required | The xAPI Inverse-Functional Identifier: `"<site_homepage>|<str(user.id)>"`. Never email. |
| `verb` | CharField, indexed, required | The ADL IRI. |
| `verb_display` | CharField, required | Human-readable label snapshot. |
| `object_type` | CharField, indexed, required | Free-form string supplied by the caller (e.g. `Topic`, `Form`, `Question`, `Course`). `experience_api` does not enumerate valid object types — domain apps register whatever they emit. |
| `object_id` | UUIDField, `null=True`, indexed | Nullable pointer to the underlying record. |
| `object_definition` | JSONField | The xAPI `object` block — full snapshot. |
| `result` | JSONField, nullable | The xAPI `result` block. |
| `context` | JSONField | The xAPI `context` block — `contextActivities`, `extensions`, etc. |
| `statement` | JSONField | The full xAPI-shaped statement (denormalised composition of all of the above). |
| `timestamp` | DateTimeField, indexed, required | UTC, microsecond precision. Server-set on write. Never overwritten. |
| `stored` | DateTimeField, auto_now_add | Receipt time. |
| `session_id_hash` | CharField, indexed, nullable | SHA-256 of (`SECRET_KEY` + session key), lower-case hex. Null for system events. |
| `user_agent` | CharField, nullable | Snapshot from request headers. |
| `ip_address` | GenericIPAddressField, nullable | Captured only if `EXPERIENCE_API_CAPTURE_IP` is `True`. |
| `platform` | CharField, required | `backend` or `frontend`. Always `backend` for the initial scope. |

Inheritance: `Event(SiteAwareModel)`. No fields are redeclared.

### Indexes

- `(site, verb, timestamp DESC)` — events for a site, by type, most recent first
- `(site, actor_user, timestamp DESC)` — per-user history
- `(site, object_type, object_id, timestamp DESC)` — per-object history
- BRIN on `timestamp` — lightweight, append-only column
- GIN on `context` with `jsonb_path_ops`
- GIN on `result` with `jsonb_path_ops`

### Why a single table, not one-table-per-event-type

- Querying across event types ("everything this user did today") is the most common analytical pattern
- The schema registry + Pydantic validation enforces consistency without needing the database to enforce it
- Type-per-table is a refactor we can do later if one event type becomes pathologically large

### Immutability — at the storage layer

1. `Event.save()` raises `EventImmutableError` when `self._state.adding` is False (row already in DB).
2. `Event.save()` on a *new* row raises `EventImmutableError` unless `_tracker_authorised=True` is set on the instance immediately before `save()`. `track()` / `write_event` set and consume this flag. The attribute is never persisted.
3. `Event.delete()` raises `EventImmutableError`.
4. Queryset `delete()` and `update()` on `Event` raise `EventImmutableError` via a custom manager/queryset extending `SiteAwareManager`.
5. All FK fields on `Event` are `null=True, on_delete=SET_NULL` except `site` (inherited `PROTECT`). No `CASCADE`.
6. **Database-level enforcement** (defence in depth): a `pre_save` signal mirrors rules 1 and 2; a migration revokes `UPDATE` and `DELETE` on the events table from the app's DB role. The erasure operation uses a separately-granted role.
7. **Erasure exception** — anonymisation of actor snapshots is the *one* documented mutation, performed by a dedicated management command, audited separately. See "Privacy".

### Partition maintenance

Partition *activation* is deferred (see `research_database_strategy.md`). The `maintain_event_partitions` management command ships in the initial scope as scaffolding so activation is a single operator step later.

```
uv run manage.py maintain_event_partitions [--lookahead-months N] [--retention-months M] [--drop-old] [--dry-run]
```

- Idempotent and cron-friendly. No-op (exit 0, warn log) when the table is not partitioned.
- Creates upcoming partitions (default: 3 months ahead).
- With `--drop-old`: detaches and drops partitions older than `--retention-months` (default 24).
- Every action is structured-log-audited.

Until activation, the command exists as documented scaffolding and is covered by a smoke test.

## Object types

`object_type` is a **free-form string**. `experience_api` does not enumerate valid object types. Domain apps choose strings when they register schemas (`"Topic"`, `"Form"`, `"Question"`, `"Course"`, etc.). The only constraint is uniqueness of the `(verb, object_type)` registry key.

There is no `object_types.py` module in `experience_api` — that was a convention of the earlier, content-engine-aware design.

## Schema registration

### `BaseEventSchema`

`freedom_ls/experience_api/schema_base.py` defines a Pydantic v2 base class that domain-app schemas inherit:

```python
from pydantic import BaseModel, ConfigDict, Field

class BaseEventSchema(BaseModel):
    """Base for every (verb, object_type) schema.

    Subclasses declare nested `ObjectDefinition`, `Result`, and
    `ContextExtensions` models, and reference them as fields here.
    """
    model_config = ConfigDict(extra="forbid")
    object_definition: BaseModel
    result: BaseModel | None = None
    context_extensions: BaseModel
```

`schema_base.py` also exposes shared size-cap constants and convenience `Field(...)` helpers:

- `STRING_EXTENSION_MAX_LENGTH = 2048`
- `RESULT_RESPONSE_MAX_LENGTH = 20480`
- `SNAPSHOT_STRING_MAX_LENGTH = 512`
- `LIST_EXTENSION_MAX_ITEMS = 100`
- `HARD_CEILING_BYTES = 65536` — enforced by the tracker before Pydantic runs, as a DoS ceiling.

### The registry

`freedom_ls/experience_api/registry.py`:

```python
REGISTRY: dict[tuple[Verb, str], type[BaseEventSchema]] = {}

def register_event_type(verb: Verb, object_type: str, schema_cls: type[BaseEventSchema]) -> None:
    key = (verb, object_type)
    if key in REGISTRY and REGISTRY[key] is not schema_cls:
        raise RuntimeError(f"Event type {key} already registered with {REGISTRY[key]}")
    REGISTRY[key] = schema_cls

def get_schema(verb: Verb, object_type: str) -> type[BaseEventSchema]:
    try:
        return REGISTRY[(verb, object_type)]
    except KeyError:
        raise TrackingSchemaError(
            f"No schema registered for (verb={verb.display}, object_type={object_type!r})"
        )
```

Idempotent re-registration of the same schema is allowed (Django's `ready()` can run twice under some test configurations); registering a *different* schema for an existing key raises.

### Domain-app schema example

Inside `freedom_ls/student_interface/xapi_events.py`:

```python
from pydantic import BaseModel, Field
from uuid import UUID
from freedom_ls.experience_api.schema_base import (
    BaseEventSchema, SNAPSHOT_STRING_MAX_LENGTH, STRING_EXTENSION_MAX_LENGTH,
)
from freedom_ls.experience_api.registry import register_event_type
from freedom_ls.experience_api.verbs import VIEWED

class _ViewedTopicObject(BaseModel):
    topic_id: UUID | None
    topic_slug: str = Field(max_length=SNAPSHOT_STRING_MAX_LENGTH)
    topic_title: str = Field(max_length=SNAPSHOT_STRING_MAX_LENGTH)
    topic_type: str = Field(max_length=SNAPSHOT_STRING_MAX_LENGTH)

class _ViewedTopicContext(BaseModel):
    course_id: UUID | None
    course_slug: str = Field(max_length=SNAPSHOT_STRING_MAX_LENGTH)
    course_title: str = Field(max_length=SNAPSHOT_STRING_MAX_LENGTH)
    cohort_id: UUID | None = None
    cohort_name: str | None = Field(default=None, max_length=SNAPSHOT_STRING_MAX_LENGTH)
    registration_id: UUID | None = None
    referrer: str | None = Field(default=None, max_length=STRING_EXTENSION_MAX_LENGTH)
    position_in_course: int | None = None

class ViewedTopicSchema(BaseEventSchema):
    object_definition: _ViewedTopicObject
    result: None = None
    context_extensions: _ViewedTopicContext

register_event_type(VIEWED, "Topic", ViewedTopicSchema)
```

The shape — nullable `_id`, required paired snapshot, `cohort_name` conditional on `cohort_id` — is a **convention** the domain apps follow, not a mechanism `experience_api` enforces. The `experience-api` skill documents the convention and it is enforced in code review / via per-schema tests.

### Conventions for domain-app schemas

These rules apply to every schema registered by any consumer app. They exist because the `Event` table is an immutable audit record: once a statement is written it is never updated or deleted by application code, so the snapshot must contain enough information to reconstruct what happened without joining to live data.

1. **All foreign keys and UUID references are nullable pointers, not required links.** A `course_id`, `form_id`, `user_id`, etc. may point at a record that no longer exists. The pointer is a convenience for joining to live data when it's still there; it is not the source of truth for the audit record. **Exception: `site_id`** is `null=False, on_delete=PROTECT` on the storage model itself.
2. **Denormalised snapshots are the source of truth.** Every event must carry enough self-contained information — titles, slugs, email addresses, display names, scores, thresholds, question text, option labels — that an auditor reading the event row alone can reconstruct what happened. When a Pydantic field marked required is a snapshot (slug / title / name / email), it is required *at write time*; the paired ID may become dangling later, but the snapshot must always be populated.
3. **Pair every `_id` with a snapshot.** Whenever a field ending in `_id` appears, the accompanying `_slug` / `_title` / `_name` field is the authoritative snapshot.
4. **Always include course / cohort context where it applies.** `course_id` + `course_slug` + `course_title` and `cohort_id` + `cohort_name` drive the bulk of analytical queries and must survive deletion of the referenced records.
5. **Snapshot all mutable context at event time** — titles, question text, thresholds, pass marks, cohort dates, role labels. Source records mutate; events must not silently change.
6. **Err toward required fields.** It's easier to relax than to tighten.
7. **Record fields that will obviously be useful for struggle-point analysis** — durations, attempt counts, change counts, `changed_answer`, `correct_answer` snapshots.
8. **No event type may define an `UPDATE`, `DELETE`, `CORRECT`, or `AMEND` verb over an existing event.** Corrections are expressed as new events (e.g. `(VOIDED, Statement)` per the xAPI spec) that point at the earlier event's immutable id.
9. **Size caps.** Use `Field(max_length=...)` / `Field(max_items=...)` on every string and list field. Defaults: strings in `context.extensions` ≤ 2 KiB; `result.response` ≤ 20 KiB; snapshot strings (slug / title / name) ≤ 512 chars; list extensions ≤ 100 items. Override with care, document why. The tracker enforces a hard 64 KiB per-string ceiling before Pydantic runs, regardless of schema.

The `experience-api` skill resource doc (`${CLAUDE_PLUGIN_ROOT}/resources/experience_api.md`) restates these conventions for future contributors.

## Initial event schemas (field detail)

This section is the authoritative description of the fields each of the seven initial `(verb, object_type)` pairs must capture. Each section corresponds to one Pydantic schema that lives in the owning app's `xapi_events.py`.

### How to read the detail blocks

Each schema has three named sub-models:

- **`ObjectDefinition`** — the xAPI `object` block. Describes *what* the event was about.
- **`Result`** — the xAPI `result` block. Describes the outcome. Some events have no result block (e.g. `(VIEWED, Topic)`) — the schema declares `result: None = None`.
- **`ContextExtensions`** — the contents of `context.extensions`. Describes the circumstances (cohort, registration, attempt, trigger).

Field notation below: `name  type, required|optional  — description`.

**Convention reminder** (also listed in "Conventions for domain-app schemas"):

- Every field ending in `_id` is a **nullable pointer** (`UUID | None`). The paired `_slug` / `_title` / `_name` field is the **authoritative snapshot** and is required at write time.
- Snapshots are captured by the wrapper from the model instance, using helpers in `content_engine/xapi_snapshots.py` and `student_management/xapi_snapshots.py`. When the helper prefers model-derived values, that's the "caller-supplied-vs-model-derived" rule (see "What domain-app wrappers do").
- Global context fields (site, session_id_hash, user_agent, ip_address, timestamp, platform) and actor snapshot fields (`actor_user`, `actor_email`, `actor_display_name`, `actor_ifi`) are **not** declared per-schema — `experience_api.tracking.track()` fills them automatically on every call.

```
  None
```

### `(COMPLETED, Topic)` — `student_progress`

Emitted when a student marks a topic complete (or the system does on their behalf).

```
ObjectDefinition:
  topic_id              UUID | None,   optional (nullable pointer)
  topic_slug            str,           required  — snapshot
  topic_title           str,           required  — snapshot
  topic_type            str,           required  — snapshot
Result:
  completion            bool,          required  — always true
  duration              ISO 8601 duration, required — time from first VIEWED to COMPLETED in this session
```

## The `track()` helper

`experience_api.tracking.track()` is the single generic entry point. It does not know about Django model instances — callers pass pre-built snapshot dicts. Domain apps wrap it in per-event-type helpers that do know about model instances.

### Signature

```python
def track(
    *,
    actor: User | None,
    verb: Verb,
    object_type: str,
    object_id: UUID | None,
    object_definition: dict,
    result: dict | None = None,
    context_extensions: dict | None = None,
    request: HttpRequest | None = None,
    strict: bool | None = None,
) -> Event: ...
```

- `actor`, `verb`, `object_type` — required. `verb` is a `Verb` instance from `verbs.py`. `object_type` is an opaque string; it must be registered in the schema registry paired with `verb`.
- `object_id` — nullable pointer (typed as `UUID | None`). Stored in the `object_id` column for indexed queries.
- `object_definition` — a dict matching the registered schema's `ObjectDefinition` sub-model. Required.
- `result` — optional dict matching the registered schema's `Result` sub-model. `None` is valid when the schema declares no result block.
- `context_extensions` — optional dict matching the registered schema's `ContextExtensions` sub-model. Required when the schema declares required context-extension fields; pass `None` or `{}` otherwise.
- `request` — optional explicit request. Falls back to `_thread_locals.request`. If neither is available, `session_id_hash`, `user_agent`, `ip_address` default to `None` and `platform` defaults to `"backend"`.
- `strict` — optional per-call override of `EXPERIENCE_API_STRICT_VALIDATION`. Reserved keyword; never a valid extension field.

All arguments are keyword-only. There is no `**kwargs` pass-through — callers must put extension fields into the `context_extensions` dict explicitly. This keeps the generic helper free of domain-specific argument names.

### What the tracker derives automatically

Per the conventions above, the tracker injects on every call:

- `site`, `site_domain` — from `_thread_locals.request` or current `Site`.
- `session_id_hash` — `hashlib.sha256((settings.SECRET_KEY + request.session.session_key).encode("utf-8")).hexdigest()`. Lower-case hex, 64 chars. Null if no request or no session key. Salting with `SECRET_KEY` makes hashes non-correlatable across installs.
- `user_agent`, `ip_address` — from request headers; `ip_address` is `None` when `EXPERIENCE_API_CAPTURE_IP` is `False`.
- `timestamp` — `timezone.now()` at write time (UTC, microsecond precision).
- `actor_user`, `actor_email`, `actor_display_name`, `actor_ifi` — from the `actor` argument; `actor_ifi` is `"<site_homepage>|<user.id>"`.
- `platform` — `"backend"` always for this scope.

### What domain-app wrappers do

A wrapper in `student_interface/xapi_events.py` looks like:

```python
def track_topic_viewed(
    actor: User,
    topic: Topic,
    *,
    request: HttpRequest | None = None,
    strict: bool | None = None,
    referrer: str | None = None,
    position_in_course: int | None = None,
) -> Event:
    course = walk_topic_to_course(topic)   # may be None
    registration = (
        resolve_user_course_registration(actor, course) if (actor and course) else None
    )
    cohort = resolve_cohort_for_user(actor, site=_current_site(request))
    return track(
        actor=actor,
        verb=VIEWED,
        object_type="Topic",
        object_id=topic.id,
        object_definition={
            "topic_id": topic.id,
            "topic_slug": topic.slug,
            "topic_title": topic.title,
            "topic_type": topic.topic_type,
        },
        context_extensions={
            "course_id": course.id if course else None,
            "course_slug": course.slug if course else "",
            "course_title": course.title if course else "",
            "cohort_id": cohort.id if cohort else None,
            "cohort_name": cohort.name if cohort else None,
            "registration_id": registration.id if registration else None,
            "referrer": referrer,
            "position_in_course": position_in_course,
        },
        request=request,
        strict=strict,
    )
```

Key points:

- The wrapper does **not** pass a Django model instance to `track()` — it builds the snapshot dict itself.
- Relationship-walking happens inside the wrapper, using helpers from the owning app (or `content_engine/xapi_snapshots.py` / `student_management/xapi_snapshots.py`).
- When relationship-walking is ambiguous (zero or ≥2 candidates), the wrapper decides what to do — typically: require the caller to pass the missing context as a kwarg, and raise `TrackingSchemaError` with a clear message if they didn't. The generic `track()` only sees a filled or `None` snapshot.
- The **caller-supplied-vs-model-derived** preference rule (persist model-derived values when the record still exists, fall back to caller-supplied strings only when the record is gone) is the wrapper's responsibility. The wrapper performs the lookup before building the snapshot. `experience_api` does not implement this rule — it has no access to the models being walked.

### Wrapper conventions

- One wrapper per `(verb, object_type)` pair the owning app emits.
- Wrapper name: `track_<object>_<verb>` (e.g. `track_topic_viewed`, `track_form_attempted`, `track_question_answered`).
- Target parameter named for its type (`topic`, `form`, `question`, `course`) — not `target`.
- `request` and `strict` passed through to `track()`.
- No domain-specific logic beyond snapshot assembly. Validation, composition, persistence all land in `track()`.

### Validation behaviour

Validation happens before any DB write:

1. The `(verb, object_type)` pair is looked up in the registry. Unknown → `TrackingSchemaError`.
2. The composed `object_definition` / `result` / `context_extensions` dicts are passed to the registered Pydantic model. Missing required field → `TrackingSchemaError`. Type mismatch on an ID-shaped field → coerced if safe (int↔str), error otherwise. Unknown extra fields → `TrackingSchemaError` in strict mode; in permissive mode the tracker logs the field **name** (never the value) at warning level, drops the key, and persists the rest.
3. **Dangling-pointer guard**: if any schema-recognised `_id` field is populated but its paired `_slug` / `_title` / `_name` snapshot is empty, raise `TrackingSchemaError`. Implementation: the generic tracker pairs `_id` with any field whose name is the same prefix followed by `_slug`, `_title`, or `_name`; wrappers are responsible for supplying both halves.

#### Size caps

Per-field `max_length` / `max_items` on schemas (see "Conventions for domain-app schemas"). Tracker enforces a hard 64 KiB per-string ceiling before Pydantic runs, regardless of schema.

#### Log payload hygiene

All log lines emitted by the tracker — validation warnings, `TrackingSchemaError` messages, the unknown-key warning, the oversize-value error — include **field names only**, never values. Examples:

- **OK:** `experience_api: missing required field 'course_title' for (COMPLETED, Topic)`
- **OK:** `experience_api: unknown extension key 'extra_field' on (ANSWERED, Question) — dropped`
- **NOT OK:** `... value was 'sheena.oconnell@example.com'`

Applies to log lines, error messages raised by `TrackingSchemaError`, and any structured-log fields.

#### Strict-vs-permissive mode

`EXPERIENCE_API_STRICT_VALIDATION` controls failure mode:

- `True` (dev/test default): validation errors — including unknown extension keys — raise `TrackingSchemaError` and propagate.
- `False` (prod default): validation errors are logged at error level (field name only) and the event is dropped — tracking failures must not break the user's primary action. Unknown extension keys in this mode are dropped from the payload but the rest of the event still persists.

Per-call override via `strict=True|False` kwarg on `track()`.

### Async write path

The DB write is wrapped in a Django-Tasks task:

```python
@task()
def write_event(payload: dict) -> None:
    instance = Event(**payload)
    instance._tracker_authorised = True
    instance.save()
```

The `track()` helper validates and composes, then enqueues the task. With `ImmediateBackend` (the default) the task runs synchronously. Switching to a queued backend is a settings change.

Because request-bound context is **not** available inside a worker, `track()` resolves every context-dependent field in the payload **before** enqueuing. The task must not rely on `_thread_locals`. Tests cover this explicitly.

#### Preconditions for switching off `ImmediateBackend`

Moving to a queued backend in production is a settings flip *plus*:

- Explicit retry policy on `write_event`: 3 retries with exponential backoff for transient errors; no retries for `TrackingSchemaError`.
- A dead-letter sink: tasks that exhaust retries write a redacted row (field names + types only, never values) to a separate `EventWriteFailure` table with the same append-only guarantees as `Event`.
- Monitoring alert on `EventWriteFailure` row-count growth.

Until those land, `ImmediateBackend` is the only supported configuration for production. This is enforced by a settings check at `apps.py` `ready()` that raises `ImproperlyConfigured` when `TASKS` is non-`ImmediateBackend` without a corresponding `EXPERIENCE_API_QUEUED_BACKEND_OBSERVABILITY_OK = True` flag.

### Return value

`track()` returns the persisted `Event` instance. Callers that chain events (e.g. `PROGRESSED, Course` carrying `trigger_event_id`) use the returned `Event.id`.

## Privacy

### Actor identification (IFI)

The xAPI `actor.account` IFI is `(homePage, name)` where:

- `homePage` = the current site URL (e.g. `https://lms.example.com`)
- `name` = `str(user.id)` — the opaque internal user ID

**Email is never used as an IFI.** Email is captured separately as `actor_email` for audit/debugging convenience and is cleared by erasure.

### Right to erasure

A management command `erase_actor` anonymises all events for one user:

```
uv run manage.py erase_actor --user-id <int> --confirm [--admin-user-id <int>]
```

`--user-id` is `accounts.User` PK (`BigAutoField`, integer).

Behaviour:

1. For every `Event` where `actor_user_id == <id>` OR `actor_ifi` ends with `|<id>`:
   - Replace `actor_email` with `erased-<random>@example.invalid`
   - Replace `actor_display_name` with `Erased actor <random>`
   - Set `actor_user` to NULL
   - Replace the `actor` block inside the JSONB `statement` with the same tombstone
   - Leave `actor_ifi`, `verb`, `object`, `result`, `context`, `timestamp`, and other snapshots untouched
2. Write a separate `ActorErasure` row capturing: `target_user_id`, erased token, event count, timestamp, `invoking_os_user` (`getpass.getuser()`), `invoking_hostname` (`socket.gethostname()`), `invoking_admin_user_id` (the `--admin-user-id` value, required when `STRICT=True`).
3. The command refuses without `--confirm`. Refuses if the target user still has non-erasure-related history (e.g. active `UserCourseRegistration` rows).

This is the **only** path that mutates already-persisted `Event` rows. The DB-level `REVOKE` on the application role blocks it from performing the mutation; the erasure command connects via a separately-granted role (see `erasure.md`).

#### `ActorErasure` is itself append-only

Mirrors `Event`:

- Migration revokes `UPDATE`/`DELETE` on `actor_erasure` from **both** the app role *and* the erasure role.
- `ActorErasure.save()` on an existing row raises. `delete()` raises. Queryset `delete` / `update` raise.
- Erasure command opens the erasure connection, performs the `Event` updates and the `ActorErasure` insert in a single transaction, closes the connection.
- Tests assert that even the erasure role cannot `UPDATE actor_erasure`.

### Other privacy controls

- IP capture is opt-in via `EXPERIENCE_API_CAPTURE_IP` (global). Per-site override deferred.
- Tracking data stays internal. No third-party export in this scope.
- Retention: events retained indefinitely for now. Documented as a TODO.

## Multi-tenant isolation

`Event(SiteAwareModel)` gives us site-scoped querysets, site-set on save from `_thread_locals.request.site`, and `SiteAwareModelAdmin`. The `site_domain` snapshot survives subsequent site renames.

## Success criteria

1. `freedom_ls/xapi_learning_record_store/` is gone. `freedom_ls/experience_api/` exists with the layout above, is in `INSTALLED_APPS`, and imports only from `accounts` and `site_aware_models`.
2. `experience_api.tracking.track()` is the single generic entry point; it takes pre-built snapshot dicts, validates against the registered schema, composes the xAPI statement, and persists via the async-task path.
3. Each domain app that emits events has a `xapi_events.py` module that defines its schemas, registers them via `register_event_type`, and exposes one `track_<object>_<verb>` wrapper per `(verb, object_type)` it owns. `student_interface` owns `(VIEWED, Topic)`, `(ATTEMPTED, Form)`, `(ANSWERED, Question)`, `(COMPLETED, Form)`, `(REGISTERED, Course)`. `student_progress` owns `(COMPLETED, Topic)` and `(PROGRESSED, Course)`.
4. All seven initial `(verb, object_type)` pairs are wired into their originating call sites via their wrappers and have passing end-to-end tests.
5. Calling `track()` with an unregistered `(verb, object_type)` raises `TrackingSchemaError`.
6. Calling a domain-app wrapper with a required field missing raises `TrackingSchemaError` with a clear message naming the field.
7. Calling `Event.save()` on a previously-persisted row raises. Calling `Event.delete()` raises. The DB role used by the app has `UPDATE`/`DELETE` revoked on the events table.
8. Deleting any object referenced by an event (user, course, cohort, form, question) leaves the event intact; only the FK column goes NULL. Deleting a `Site` with events is blocked by `PROTECT`.
9. The `erase_actor` management command anonymises all events for a user, writes an audit row, and is the only mutation path that succeeds against an existing row. `ActorErasure` itself is append-only.
10. Events are site-scoped: a user logged into site A sees no events from site B.
11. Django Tasks is configured with `ImmediateBackend`; events write synchronously.
12. The `experience-api` skill exists in `fls-claude-plugin/skills/experience-api/` with `SKILL.md` and its resource file. A new contributor can use the skill to add a new event type to a domain app without re-deriving the workflow.
13. The `maintain_event_partitions` command exits 0 with a "not partitioned — nothing to do" warning before partition activation, and accepts `--lookahead-months`, `--retention-months`, `--drop-old`, `--dry-run`.
14. All new code passes the project's lint, type-check, and pre-commit hooks. Type hints everywhere; no `Any`; modern union syntax.
15. `experience_api` has no imports from `content_engine`, `role_based_permissions`, `student_management`, `student_progress`, or `student_interface`. Verified by a one-line grep test in the test suite.

- **Helper API shape** — *resolved: generic `track()` in `experience_api` + per-event-type wrappers in each owning app's `xapi_events.py`.* The generic helper is domain-agnostic (takes dicts, not model instances); the wrappers do the model-walking.
- **Salting `session_id_hash`** — *resolved: salt with `SECRET_KEY`.*
- **Verbs `SUBMITTED`, `INTERACTED`, `VOIDED`** — defined as constants but no schema yet.
- **Partitioning** — activation deferred; tooling in scope.
- **Retention policy** — deferred; events kept forever for now.
- **Voiding mechanism** — out of scope.

- **Skill split** — single `experience-api` skill initially.
