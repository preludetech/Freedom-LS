# Lightweight xAPI Event Tracking

We need to keep track of everything that users on the platform do.

We will base our approach on xAPI (Experience API) — using it as a **design guide**, not strict compliance. We use xAPI concepts (actor/verb/object/result) and standard ADL verb IRIs where they fit, but store data in a simple, queryable format in our own database. No full LRS.

## App structure

Rename the existing `xapi_learning_record_store` directory to `experience_api` (and update the app config to use `freedom_ls.experience_api`). All tracking functionality lives in this app. The app must use `SiteAwareModel` for multi-tenant isolation.

## Database: PostgreSQL is sufficient

Research confirms PostgreSQL 17 handles this scale comfortably. At ~5K DAU with ~50 events/user/day = ~90M events/year, well within PostgreSQL's range.

Use a hybrid schema: typed columns for frequently-queried fields (actor, verb, object_type, object_id, timestamp) plus a JSONB column for the full xAPI-shaped statement data. This gives fast queries on common patterns while preserving flexibility.

Consider monthly table partitioning for the events table if/when data volume warrants it — but this can be deferred to a later phase. Start simple.

## Backend tracking: explicit helper functions

The primary tracking pattern is explicit helper function calls from views and services:

```python
from freedom_ls.experience_api.tracking import track
from freedom_ls.experience_api.verbs import COMPLETED, VIEWED

track(request.user, VIEWED, topic)
track(request.user, COMPLETED, form, result={"score": 85, "max_score": 100})
```

Key design principles:
- **Minimal required arguments**: `track(user, verb, object)` — infer the rest
- **Accept Django model instances**: the tracker extracts IDs, types, names from model instances
- **Verb constants**: defined as constants to prevent typos and enable IDE autocomplete
- **Optional result data**: scores, success, duration passed as keyword arguments
- **Return the created event**: useful for testing

We maintain our own controlled vocabulary of verbs and activity types instead of relying on external xAPI repositories. Start small (~10 verbs) covering: viewed, completed, attempted, answered, registered, progressed, submitted, interacted.

## Event schemas

Every `(verb, object_type)` pair has a defined schema that specifies required/optional fields for `object`, `result`, and `context.extensions`. The guiding principle is **capture too much rather than too little** — fields like `cohort_id`, `course_id`, `attempt_number`, and duration snapshots are cheap now and expensive to backfill.

Schemas are the source of truth for what `track()` calls must pass and what can be queried later. See `event_schemas.md` in this directory for the full schema catalogue. The `track()` helper validates calls against these schemas at write time.

## Async event writing via Django Tasks

Use the Django Tasks framework for event persistence. Default to `ImmediateBackend` for now (synchronous writes). This gives us the abstraction layer to switch to async writes later without code changes when a production backend is configured.

## Frontend tracking

Deferred to a separate spec — see `spec_dd/0. drafts/frontend_engagement_tracking/`. The backend event schema is JSONB-flexible, so frontend engagement signals can be added later without retrofitting the table.

## Claude skills for future extension

Adding a new `(verb, object_type)` pair touches several files in a specific order (verbs registry, schema catalogue, Pydantic validator, tracker call sites, tests). This is exactly the kind of repeatable, multi-file workflow that should live in a skill rather than as tribal knowledge.

The implementation of this spec **must** include creating skills in `fls-claude-plugin/skills/` so that future contributors (human or AI) can extend the tracking system without re-deriving the full workflow.

### Skill: `experience-api`

Primary skill for anything involving event tracking. Gets triggered when the user mentions tracking, xAPI, events, verbs, "track()", or when adding a new thing to record.

The `SKILL.md` (under ~30 lines, per the existing pattern in this repo) should cover:

- **When to use it** — adding a new event type, emitting a new event from a view/service, querying events, debugging a failed `track()` call
- **Key rules** — the immutability guarantees (no updates, no deletes, snapshot-all-the-things), the nullable-FK-plus-required-snapshot convention, verbs come from `verbs.py` constants
- **Pointer to a resource file** for full detail: `${CLAUDE_PLUGIN_ROOT}/resources/experience_api.md`

The accompanying `resources/experience_api.md` is where the longer workflow content lives, including:

- **The emit-an-event recipe** — how to call `track()` from a view or service, which arguments are inferred vs. must be passed, how to read a schema entry in `event_schemas.md` to figure out what `result` and `context` fields to pass
- **The add-a-new-event-type recipe** — step-by-step:
  1. Define the verb constant in `freedom_ls/experience_api/verbs.py` if it doesn't exist (reuse existing ADL verbs where possible)
  2. Add the `(verb, object_type)` schema entry to `event_schemas.md`, following the required/optional/snapshot conventions
  3. Add the matching Pydantic validator to `freedom_ls/experience_api/schemas.py`
  4. Wire the `track()` call into the originating view/service
  5. Write tests: a validation test (required field missing → raises), an end-to-end test (call site → row in DB with expected snapshots), and a deletion-survival test (delete the referenced object, assert the event still reads cleanly from its snapshot)
- **How to query events** — typed-column lookups (actor, verb, object_type, object_id, timestamp range) vs. JSONB queries into `context.extensions`, and when to reach for each
- **Anti-patterns** — don't emit events with `.objects.create()`, don't add `_id` fields without snapshots, don't use `CASCADE` on any FK into the events table, don't mutate existing events (express corrections as new `VOIDED` events)

### Skill: `xapi-schema-authoring` (optional, can merge into the above)

A narrower skill focused specifically on writing a new schema entry in `event_schemas.md`. If `experience-api` starts feeling too large, split this out. The decision can be deferred to implementation time.

### Writing the skill files

Follow the existing skill conventions in `fls-claude-plugin/skills/`:

- Frontmatter: `name`, `description` (must clearly state when to trigger), `allowed-tools` (usually `Read, Grep, Glob`)
- SKILL.md stays short; deep content goes in `resources/<topic>.md`
- Description starts with a verb and names a concrete trigger ("Work with...", "Use when...")
- Include worked examples from this codebase — real `track()` calls, real schema entries — not invented ones

Including skill creation as a first-class deliverable means the cost of adding the 11th, 20th, 50th event type stays flat instead of growing with tribal knowledge.

## What to track (initial scope)

Backend events:
- Student views a topic → `experienced`
- Student completes a topic → `completed`
- Student starts a form → `attempted`
- Student answers a form question → `answered`
- Student submits/completes a form → `completed` (with scores in result)
- Student registers for a course → `registered`
- Course progress updates → `progressed`

## What NOT to build

- No full LRS REST API
- No statement forwarding to external LRS
- No signed statements or sub-statements
- No OAuth endpoint for xAPI
- No voiding mechanism (yet)

## Course content quality monitoring

We will implement course content quality monitoring later. The xapi data will support seeing where people typically struggle and how long people take to do different things. Keep this in mind when designing the data model.

## Privacy

- Actor identification uses internal user IDs (account-based IFI), not emails
- Must support right-to-erasure (delete all tracking data for a user)
- Tracking data stays internal, no third-party sharing
- Define a retention policy for old event data
