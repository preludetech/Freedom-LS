## Manual code review

### Auth & ownership (OWASP A01)
- All new state-changing student views (`register_for_course`, `form_start`,
  `form_fill_page`, `view_course_item`, `course_finish`,
  `course_form_complete`) are decorated with `login_required`; the existing
  ownership model (queries always scoped by `request.user`) is preserved.
  Tracking calls were added inside these views only.
- `view_topic` emits a VIEWED event but only when `request.user.is_authenticated`
  ‚Äî it does not expose events to anonymous users.
- `Event.objects` uses `_AppendOnlyManager` which composes
  `SiteAwareManager.get_queryset()` ‚Äî site isolation is preserved.
- Admin (`admin.py`) disables add / change / delete on both `Event` and
  `ActorErasure`; all fields are in `readonly_fields`.

### Immutability & integrity (A08)
Four-layer append-only defence is implemented exactly as the spec requires:
1. `Event.save()` rejects updates and rejects creates without the
   `_tracker_authorised` sentinel (models.py).
2. `Event.delete()` raises. `_AppendOnlyQuerySet.update()` and the queryset
   delete method raise at manager level.
3. `pre_save` signal mirrors the save guard for defence in depth.
4. Migration `0002_revoke_mutations` revokes UPDATE/DELETE on both tables
   from the current application role; `0003_erasure_role_read_insert`
   grants only SELECT+UPDATE on events and INSERT on `ActorErasure` to
   the separate `fls_erasure_role` ‚Äî so even the erasure role cannot
   mutate or delete the audit trail.

### Injection (A03)
- Only one `cursor.execute` in production code
  (`erase_actor._perform_erasure`) ‚Äî fully parameterised (`%s`
  placeholders with separate argument list), including the `LIKE` pattern.
  No string interpolation of user data.
- `maintain_event_partitions._table_is_partitioned` is parameterised.
- `importlib.import_module(module_path)` in `erase_actor._check_blockers`
  is driven by the settings-only `EXPERIENCE_API_ERASURE_BLOCKERS` list
  (operator-controlled, never request-derived). Safe and annotated with
  an explanatory comment.
- No use of the Django raw query escape hatches, no dynamic code
  evaluation, no `subprocess`, no shell invocation, no unsafe
  deserialisation anywhere in the new code.

### Crypto & PII (A02)
- `hash_session_key` implements `sha256(SECRET_KEY + session_key).hexdigest()`
  exactly per GAP-2 in the threat model ‚Äî utf-8 encoded, lower-case hex,
  64 chars. Returns `None` when session_key is empty or non-string.
- `build_actor_ifi` uses `<site_homepage>|<user.id>` ‚Äî never email, as
  the threat model mandates.
- `get_ip_address` is gated on `EXPERIENCE_API_CAPTURE_IP` (defaults to
  `False` in `settings_prod.py`), per the privacy design.
- `get_user_agent` truncates to the DB column length (1024) to prevent
  storage-exhaustion via oversized headers.

### Input-validation / DoS (A04)
- Hard per-string byte ceiling (`_enforce_hard_ceiling`) runs **before**
  Pydantic (GAP-4 in threat model) ‚Äî recursively walks nested dicts/lists.
- Pydantic validation per `(verb, object_type)` schema. Strict vs permissive
  handled by `handle_validation_error` (GAP-3).
- Dangling-pointer guard enforces `_id` ‚Üî `_title`/`_slug`/`_name` pairing
  (GAP-5).
- Log hygiene: every log line names field paths only, never values
  (GAP-10). The permissive unknown-keys path logs the dropped key names
  (field identifiers, not user values) ‚Äî acceptable per the threat model.

### CSRF & state changes
- All new state-changing paths are standard Django POST views with session
  auth ‚Äî Django CSRF middleware applies. No CSRF exemptions anywhere in
  the change set.

### Management command safety (V12)
- `erase_actor` requires `--confirm`, refuses with an active-registration
  blocker (configurable), logs acting OS user / hostname / admin user pk
  in the `ActorErasure` row.
- Strict-mode deployments must also pass `--admin-user-id`; non-strict
  deployments print a WARN.
- Uses a transaction on the `erasure` DB alias so the single UPDATE and
  the audit-row INSERT commit atomically.

### App-boundary & bypass guards (V11)
- `Event.save()` requires a `_tracker_authorised` sentinel that is set
  inside `write_event` task and deleted after the save ‚Äî so any direct
  `Event(...).save()` call is rejected.
- `apps.py.ready()` refuses to boot with a non-ImmediateBackend TASKS
  default unless `EXPERIENCE_API_QUEUED_BACKEND_OBSERVABILITY_OK=True`
  (GAP-7 mitigation).
- `test_import_boundaries.py` exists to pin the rule that `experience_api`
  does not import from any domain app.

## Findings

**No blocking issues.** No High or Medium severity issues found. The
implementation materially follows the threat model and spec.

### Informational notes (no action required)

1. **pip / CVE-2026-3219.** Installer-side vulnerability in pip 26.0.1.
   Not a runtime dependency; upgrade at the next routine `uv` bump.
2. **Prod deploy-check warnings `W005` / `W021`** ‚Äî HSTS subdomains and
   preload are opt-in env vars (`HSTS_INCLUDE_SUBDOMAINS`, `HSTS_PRELOAD`).
   Operational choice, not a code issue.
3. **`user.id` enumerability** (threat model V6 sub-note). Already
   acknowledged. No action.
4. **Permissive-mode unknown-extensions drop.** The implementation drops
   the keys and logs their names at WARN. Acceptable per GAP-3.

## Verdict

No blocking security issues. Ship.
