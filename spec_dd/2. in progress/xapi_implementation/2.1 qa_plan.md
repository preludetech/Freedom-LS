# xAPI Event Tracking — QA Plan

Deliberately minimal. One scenario: confirm that a real, logged-in, in-browser action writes exactly one durable `experience_api_event` row that is visible from a separate DB connection. The point is to catch the transaction / `on_commit` / task-backend failure modes unit tests can't see.

Reference: `1. spec.md`, `2. plan.md`. Schemas live next to the code that emits them — for `(VIEWED, Topic)` that's `freedom_ls/student_interface/xapi_events.py`.

---

## Setup

- **Site:** DemoDev (`demodev@email.com` convention). All test data must live on the DemoDev site per the project's standing QA convention.
- **Server:** follow `/do_qa` Step 3 — pick a random port via `.claude/fls/scripts/find_available_port.sh`, run `uv run python manage.py runserver $PORT`, and use `http://127.0.0.1:$PORT/` as the base URL.
- **Login:** admin credentials from `.claude/fls/config.md` (`demodev@email.com` / `demodev@email.com`).
- **DB verification:** queries issued from a **separate** `uv run python manage.py shell` process via the `Event` ORM — **not** from inside the request's process/connection. A separate process means a separate DB connection; this is deliberate, because a query on the same connection would see pending transaction state. The whole point of this QA is to confirm the row is really committed.

## Test-data prerequisites

Ask the `qa-data-helper` agent to ensure the following on the DemoDev site (reuse existing DemoDev seed data if it already provides these; only create what's missing):

1. A non-admin **student** user on DemoDev with a known email and password (e.g. `qa-xapi-student@demodev.local` / `testpass123`).
2. That student is **registered** for at least one DemoDev course.
3. That course contains at least one **Topic** that the student has not yet viewed, reachable via the normal student navigation (`student_interface`) — i.e. the topic detail view resolves and renders for this user without any pre-requisites blocking it.

Record the student's `accounts.User.id` (bigint PK) and the topic's UUID — the verification query needs both.

## Scenario 1: `(VIEWED, Topic)` smoke test

Chosen because it is the cheapest event in the catalogue to trigger: no form interaction, no completion threshold, no progression chain. One page load is one event.

### Playwright steps (`/do_qa` drives this)

1. Before any interaction, capture the current maximum `stored` timestamp in `experience_api_event` for this actor (see "Verification query A"). Record it. Any row with `stored > <that timestamp>` is attributable to this test.
2. Navigate to base URL, confirm the `debug-branch-badge` shows the `xapi_implementation` branch (per `/do_qa` Step 4).
3. Log in as the student user from the prerequisites.
4. Navigate to the topic detail page (via the course page → topic link, mimicking a real learner flow — do not deep-link directly unless the nav is broken, to keep the path representative).
5. Wait for the topic page to fully render (the view's response is what triggers the `track_topic_viewed` call).
6. Take a screenshot: `desktop_1_topic_viewed.png`.
7. Log out (ends the request cleanly so any `on_commit` callbacks fire).
8. Hand off to `qa-data-helper` with the student `user_id`, the topic UUID, and the captured baseline timestamp. Ask it to run the verification queries below on a separate DB connection.

### Verification (performed by `qa-data-helper` on a separate DB connection)

All three queries run in a single `uv run python manage.py shell` session (one process = one fresh DB connection, separate from the runserver process the browser is hitting). Use the `Event` model directly. No raw SQL.

```python
import datetime as dt
from django.db.models import Max
from freedom_ls.experience_api.models import Event

STUDENT_USER_ID = <student_user_id>
EPOCH = dt.datetime(1970, 1, 1, tzinfo=dt.timezone.utc)
```

**Query A — baseline (run before step 2):**

```python
baseline_stored = (
    Event.objects.filter(actor_user_id=STUDENT_USER_ID)
    .aggregate(m=Max("stored"))["m"]
) or EPOCH
```

**Query B — post-action count:**

```python
new_event_count = Event.objects.filter(
    actor_user_id=STUDENT_USER_ID,
    stored__gt=baseline_stored,
).count()
```

**Query C — single-row inspection:**

```python
new_events = list(
    Event.objects.filter(
        actor_user_id=STUDENT_USER_ID,
        stored__gt=baseline_stored,
    )
    .order_by("-stored")
    .values(
        "id", "verb", "verb_display", "object_type", "object_id",
        "actor_ifi", "actor_email",
        "site_id", "site_domain",
        "timestamp", "stored",
        "object_definition",
    )
)
```

Notes:
- `Event` inherits `SiteAwareManager`, but in a `manage.py shell` context there is no request in thread-locals, so `Event.objects` returns unfiltered across sites — no `unfiltered()` escape hatch needed.
- Do **not** wrap these calls in `transaction.atomic()` or set `ATOMIC_REQUESTS`-style behaviour; they must run on the default autocommit connection so a successful read proves the write is durably committed.

### Assertions

The test passes if and only if **all** of the following hold:

- Query B returns exactly `1` (exactly one new row was durably written — not zero, not duplicates).
- The single row from Query C has:
  - `verb = 'http://adlnet.gov/expapi/verbs/experienced'` and `verb_display = 'viewed'` (per `verbs.py` — `VIEWED` maps to the ADL `experienced` IRI with display `viewed`).
  - `object_type = 'Topic'`.
  - `object_id` equals the UUID of the topic that was navigated to.
  - `actor_ifi` has the shape `https://<demodev-site-domain>|<student_user_id>` (account-based IFI per spec §"Privacy"). Crucially, it is **not** the student's email.
  - `actor_email` matches the student's email.
  - `site_domain` matches the DemoDev site's domain (snapshot, not a join).
  - `timestamp` is non-null and within the last few minutes.
  - `stored` is non-null. `stored >= timestamp`.
  - `object_definition` is a non-empty JSONB object containing the topic's slug and title (snapshot survival).
- The row is visible from this separate DB connection **without** any `BEGIN`/`COMMIT` wrapper on the query side — i.e. it is genuinely committed. This catches the "looked written in tests, vanished in prod" class of bug the user is worried about.

If Query B returns `0`, the most likely causes are (in order): the task backend didn't run the write, an outer `transaction.atomic()` rolled back around the view, or the wrapper was never called from the view. The report should name which.

If Query B returns `>1`, the call site is firing twice per page load (e.g. HTMX swap + full page render). The report should include whether the duplicates share a `session_id_hash`.

## Out of scope for this QA

Intentionally not covered — these are unit-test concerns, not smoke-test concerns:

- Other `(verb, object_type)` pairs.
- Validation failures, strict-vs-permissive behaviour, size caps.
- Multi-tenant isolation, erasure, immutability enforcement.
- Async task backends other than `ImmediateBackend`.
