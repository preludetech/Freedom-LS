# Threat model — Lightweight xAPI Event Tracking

Performed against `1. spec.md` using OWASP Top 10:2025 as the mapping framework.

## 1. Assets at risk

| # | Asset | Why it matters |
|---|---|---|
| A1 | **Event rows (statement JSONB + typed columns)** | Append-only audit record. Integrity is the primary security property — the whole value proposition of the log is that it cannot be rewritten after the fact. |
| A2 | **Actor PII snapshots** (`actor_email`, `actor_display_name`, `actor_user` FK, `actor_ifi`) | Learner identity tied to behaviour. High-value target for re-identification; also the subject of GDPR/POPIA erasure obligations. |
| A3 | **`session_id_hash`** | Permits cross-event correlation inside one browser session. Must not be reversible to the raw session key. |
| A4 | **`ip_address` and `user_agent`** | Classical PII / device-fingerprint data. Per-jurisdiction sensitivity. |
| A5 | **Cross-tenant event data** | Events from site A must never leak to users of site B (operators or learners). |
| A6 | **Erasure audit trail (`ActorErasure`)** | Proof that a GDPR/POPIA erasure request was honoured, and by whom. Must itself be tamper-evident. |
| A7 | **`SECRET_KEY`** (used to salt `session_id_hash`) | Loss changes the threat surface for A3 — an attacker with the DB *and* `SECRET_KEY` can correlate sessions with raw keys. |
| A8 | **The database role used by the app** | The REVOKE of UPDATE/DELETE on the `events` table is the last line of defence for immutability. Privilege escalation on this role defeats A1. |
| A9 | **The separately-granted erasure role** | Any compromise of this role permits unlogged mutation of PII — both a privacy breach *and* an integrity breach. |
| A10 | **Content-engine / student-progress identifiers** leaked through `object_definition` and `context.extensions` | Titles and slugs are low sensitivity, but structured references (`cohort_id`, `registration_id`) let an attacker who can read events reconstruct the social graph of a tenant. |

## 2. Threat actors

- **TA1 — Unauthenticated internet attacker.** No account. Interacts via public endpoints. Cannot call `track()` directly unless a view is wired to do so.
- **TA2 — Authenticated learner (student role).** Has a session on one site. Primary vector is crafting input that reaches `track()` via normal flows (viewing topics, answering questions).
- **TA3 — Authenticated educator/admin on site A.** Has elevated read access inside their own tenant. Threat is cross-tenant read/write if isolation fails, or tamper attempts via admin.
- **TA4 — Platform operator with DB shell / server shell.** Trusted but accountable. The immutability and erasure-audit controls exist partly to *deter* this actor from informal edits ("just fix this one row") and to leave a trail when they happen.
- **TA5 — Malicious / compromised internal code.** A bug or a dependency update that calls `Event.objects.create()` directly, bypasses the wrapper, or misuses the async task.
- **TA6 — Compromised erasure role.** Access to the role granted `UPDATE` on the events table. Can anonymise or tombstone any row.
- **TA7 — Log/observability consumer.** Whatever ships stdout/structured logs (Sentry, ELK, cloud logging). Receives error payloads that may include PII if we're careless.
- **TA8 — Future LRS export consumer.** Out of scope today but the `statement` JSONB is explicitly designed to be exportable; anything put in it now leaves via that channel later.

## 3. Attack vectors vs OWASP Top 10:2025

OWASP Top 10:2025 categories: A01 Broken Access Control, A02 Cryptographic Failures, A03 Injection, A04 Insecure Design, A05 Security Misconfiguration, A06 Vulnerable and Outdated Components, A07 Identification & Authentication Failures, A08 Software & Data Integrity Failures, A09 Security Logging & Monitoring Failures, A10 Server-Side Request Forgery.

### V1 — Cross-tenant read (A01)

**Vector.** Educator/admin on site A attempts to read events belonging to site B via admin, ORM, or `track()` return values.

**Required control.** `Event(SiteAwareModel)` default manager filters by `_thread_locals.request.site`; admin uses `SiteAwareModelAdmin`; no unscoped `Event.objects.unfiltered()` callers added; tests assert cross-tenant isolation.

**Status.** Specified (Multi-tenant section, Success criterion 8, test item "Multi-tenant tests"). **OK** — covered by existing `SiteAwareModel` behaviour.

### V2 — Cross-tenant write via spoofed `_thread_locals` (A01, A04)

**Vector.** Code path runs with a stale or wrong `_thread_locals.request`, so a write from site A's request lands with `site_id` for site B.

**Required control.** `CurrentSiteMiddleware` deletes `_thread_locals.request` after the response. Tracker resolves `site_id` *before* enqueuing the async task (the spec calls this out explicitly under "Async write path").

**Status.** Specified. Middleware already clears `_thread_locals`. **OK** — the async-task test mandated by the spec protects the other half.

### V3 — Mutation of persisted events (A08 Integrity Failures)

**Vector.** Application code, an ORM mis-step, or a direct DB user attempts to `UPDATE` or `DELETE` an event.

**Required control.** Four layers, all in the spec:
1. `Event.save()` raises on existing pk.
2. `Event.delete()` and queryset-level `update()`/`delete()` raise.
3. `pre_save` signal mirrors the model-level guard (defence in depth).
4. Migration REVOKEs `UPDATE`/`DELETE` on the events table from the app role.

**Status.** Specified. **OK** — enforced on multiple layers. Immutability tests are mandated.

### V4 — Mutation via the erasure role (A08, A01)

**Vector.** An attacker compromises the erasure DB role, or a developer wires a non-erasure code path through that role, and rewrites events at will.

**Required controls.**
- Erasure role is separately granted; the app never uses it.
- Only the `erase_actor` management command connects as that role.
- Every erasure writes a row to `ActorErasure` (separate table, not itself mutable by the app role).

**Gap identified.** The spec does not explicitly require that `ActorErasure` itself is append-only (no UPDATE/DELETE from the erasure role), nor does it specify logging (who invoked the command, from which host). **GAP-1** below.

### V5 — Session correlation if DB leaks (A02 Cryptographic Failures)

**Vector.** DB backup or SQL dump falls into attacker hands. Attacker attempts to link events to live sessions.

**Required control.** `session_id_hash` = SHA-256(`SECRET_KEY` + session key), not raw hash. Attacker needs both DB *and* `SECRET_KEY` to correlate.

**Status.** Specified in "What the tracker derives automatically" and in Open questions. **OK** — control matches the threat.

**Sub-gap.** The spec says "salt with `SECRET_KEY`" but does not specify concatenation order, encoding, or that the result is lower-case hex. Implementation drift here is a tests-catch-it problem rather than a spec problem, but flag it as **GAP-2** so it doesn't get lost.

### V6 — Re-identification via IFI choice (A02)

**Vector.** An attacker with query access tries to link events to identities using the IFI.

**Required control.** IFI is `account(homePage=<site>, name=str(user.id))` — opaque internal ID, never email. Stored in `actor_ifi` explicitly so the audit record survives erasure of `actor_email`/`actor_user`.

**Status.** Specified ("Privacy → Actor identification (IFI)"). **OK**.

**Sub-note.** `user.id` is `BigAutoField` (sequential). This is enumerable. The threat is low because it's joined with a `homePage` scoped to one site and is already the internal user ID that appears in URL paths etc., but note it here for operator awareness. No spec change recommended.

### V7 — PII leak via `result.response` / schema extensions (A02, A04)

**Vector.** A future schema author adds a field that accepts free-text user input (e.g. `result.response` for a question answer) and the tracker persists it without shape / size limits, allowing PII ingestion beyond what the schema intends. Or — a caller passes an unknown `**context_extensions` key that contains PII; the tracker logs a warning *and persists anyway*.

**Required controls.**
- Pydantic validation per `(verb, object_type)` with required/optional field lists.
- Per-owning-app schemas in `xapi_events.py` modules (e.g. `student_interface/xapi_events.py`, `student_progress/xapi_events.py`), registered with `experience_api` at app-ready time. Each schema is the single source of truth for its `(verb, object_type)` pair.
- Dangling-pointer guard.

**Gap identified.** The spec says "Unknown extra fields → log warning, persist anyway" (Validation behaviour step 2). This is the opposite of a least-surprise default for an immutable audit log: anything persisted here is *unerasable by the normal path* (only actor fields are touched by `erase_actor`). An unknown key containing unexpected PII will sit in the DB forever. **GAP-3** below.

**Further gap.** The spec does not cap the size of any field. A misbehaving caller can write multi-megabyte strings into `result` or `context.extensions`. That's a storage-exhaustion / performance issue as much as a security issue. **GAP-4** below.

### V8 — Injection via JSONB (A03 Injection)

**Vector.** An attacker tries to abuse Postgres JSONB operators through crafted input.

**Required control.** All writes go through the Django ORM / Pydantic validation; no raw SQL. JSONB is a structured type, not interpolated into SQL strings.

**Status.** The project CLAUDE.md prohibits raw SQL without security review, and the spec doesn't introduce any. **OK**.

### V9 — Stored XSS via snapshots surfaced in admin / future views (A03)

**Vector.** A course title containing `<script>` is captured into `object_definition.course_title` and later rendered unescaped in an admin list view or an analytics dashboard.

**Required control.** Django templates auto-escape by default; admin uses the standard template chain. The spec mandates read-only admin (the danger is rendering of untrusted data, not editing it).

**Status.** **OK** for the scope of this spec (admin is read-only with default Django escaping). Worth restating as a note for the future analytics dashboard spec — when it lands, anything that renders `object_definition` or `result` must escape. Not a spec gap.

### V10 — Dangling-pointer abuse / snapshot forgery (A04, A08)

**Vector.** A caller passes a valid `_id` but populates the paired snapshot with misleading text (e.g. `cohort_name="Billing Department"` when `cohort_id` refers to "Learner Cohort 3") to defame or mislead future auditors.

**Required control.** Snapshots at write time are produced by snapshot helpers that live next to the models they walk (`content_engine/xapi_snapshots.py`, `student_management/xapi_snapshots.py`) and are consumed by the domain-app wrappers in `student_interface/xapi_events.py` and `student_progress/xapi_events.py`. Snapshots are built from model reads, not taken from caller input. Where a caller is forced to pass a field explicitly (because relationship-walking returned 0 or ≥2), the wrapper still prefers model-derived values when the record exists and only falls back to caller-supplied strings when the record is gone. `experience_api` itself never walks domain models.

**Status.** Specified (see "What the tracker derives via relationship-walking" and the dangling-pointer guard in validation). **OK**, but see GAP-5 below on caller-supplied overrides.

### V11 — Bypass of the wrapper / `track()` path (A04, A08)

**Vector.** Developer calls `Event.objects.create()` directly, producing a row that didn't go through validation or `_thread_locals` resolution and that carries unknown fields or mis-scoped `site_id`.

**Required controls.**
- The `experience-api` skill and resource doc ("Anti-patterns") say don't do this.
- The `pre_save` signal mirroring the immutability guard could also assert required-field presence.
- Tests.

**Gap identified.** The spec documents the anti-pattern but there is no runtime enforcement. A linter rule or a `pre_save` assertion would catch it. **GAP-6** below.

### V12 — Management command abuse (A01, A07)

**Vector.** `erase_actor` or `maintain_event_partitions` is invoked by the wrong operator, or on the wrong `--user-id`, destroying a learner's history or rotating partitions that still hold live data.

**Required controls.**
- `erase_actor --confirm` gate.
- `erase_actor` refuses if the user still has non-erasure-related history.
- `maintain_event_partitions` is idempotent and defaults `--drop-old` to off (report-only).
- Both write audit / log entries.

**Status.** Specified. **OK**.

### V13 — Async-task failure handling (A09 Logging & Monitoring Failures)

**Vector.** An enqueued `write_event` task fails (schema evolution, DB down, validation regression). With `ImmediateBackend` the caller sees the exception; with a real backend, the event is silently lost.

**Required control.** In production (`STRICT=False`) the tracker "logs at error level and drops" — by design. But that's for validation failures detected *before* enqueue. Post-enqueue failures need their own observability.

**Gap identified.** The spec does not describe what happens to failed tasks under a queued backend: no retry semantics, no dead-letter handling, no monitoring expectation. Today that's moot (`ImmediateBackend`) but it is explicitly called out as a settings-flip in Success criterion 9. **GAP-7** below (low-priority; tie to the "switching to a real backend" future change).

### V14 — Privacy policy drift (A04, GDPR / POPIA compliance)

**Vector.** `EXPERIENCE_API_CAPTURE_IP` is a global setting. A multi-tenant deployment where one tenant operates in the EU and another in ZA could legally require different IP-capture defaults, but the current spec gives them the same knob.

**Required control.** Per-site override.

**Status.** Explicitly deferred — see Open questions. Flagged, not fixed. Calling this **GAP-8** for visibility to the reviewer but acknowledging the spec already marks it as deferred.

### V15 — Actor-role spoofing (A01, A07)

**Vector.** n/a — `actor_role` is not stored. The event records `actor_user`, `actor_email`, `actor_display_name`, `actor_ifi`, and the `site`; role resolution happens at query time from the live permission model, not at write time.

**Status.** **OK** — not applicable.

### V16 — Supply-chain risk: pydantic (A06 Vulnerable and Outdated Components)

**Vector.** Pydantic v2.x is the new dependency. CVEs against pydantic do occur (parser-level issues, etc.).

**Required control.** `uv` adds the dep; the project already has pre-commit hooks and (per the security-audit spec in `3. done/`) dependency scanning patterns. No new tooling needed.

**Status.** **OK**.

### V17 — SSRF (A10)

**Vector.** n/a — this spec does not introduce any outbound HTTP or user-controlled URL-fetching. ADL verb IRIs are constants, not fetched. Statement-forwarding to external LRSs is explicitly out of scope.

**Status.** **OK** — not applicable.

### V18 — Log payloads leak PII (A09, A02)

**Vector.** `TrackingSchemaError` messages, or the "Unknown extra fields → log warning" path, include the raw caller input in log lines. Logs travel to Sentry/ELK and sit longer / with broader access than the DB.

**Required control.** Error messages name the *field*, not the value. The warning-on-unknown-keys case must log the key name, not the value.

**Gap identified.** The spec describes error messages for `TrackingSchemaError` ("pass `course=...` explicitly") but does not explicitly prohibit logging values. **GAP-10** below.

## 4. Gaps summary

These are the items the spec should address before `/plan_from_spec`. Numbering matches the callouts above.

- **GAP-1 — `ActorErasure` is itself append-only, and records invoking operator.** Spec should state that the `ActorErasure` table has the same UPDATE/DELETE-revoked property as `events` for the app role, and that the erasure role can INSERT but not UPDATE/DELETE it either (logical append-only). Command should capture `os.getlogin()` / environment hint / acting admin user pk at the time of invocation.
- **GAP-2 — Specify the session-hash construction.** Exact input order (`SECRET_KEY + session_key`), encoding (utf-8), output format (lower-case hex), so implementation is not up for interpretation.
- **GAP-3 — Change the "unknown extra fields" default.** At minimum in strict mode, unknown extensions should raise `TrackingSchemaError`. Persisting unknown keys into an immutable store that only erases `actor_*` fields is a durable privacy footgun.
- **GAP-4 — Size caps on schema fields.** Pydantic supports `max_length` / `max_items`. Add a default-reasonable cap (e.g. 2KB on string extension fields, 20KB on `result.response`) so no single write can DoS the events table. Hard-coded ceiling at the tracker layer before Pydantic validation — belt and braces.
- **GAP-5 — Note on caller-supplied snapshot overrides.** The spec should clarify that when relationship-walking fails and the caller passes the snapshot explicitly, the tracker still prefers the model-derived values if it can look up the record via the caller-supplied `_id`. Only fall back to caller-supplied strings for `_name`/`_title`/`_slug` when the record no longer exists *at all*.
- **GAP-6 — Runtime guard against bypassing `track()`.** Add an assertion inside `Event.save()` (first-save path) that a specific flag set by `track()` is present. Writes that didn't go through `track()` raise. Flag is a `@classmethod` / private kwarg, not persisted.
- **GAP-7 — Async-backend observability.** When queued-backend support lands, the `write_event` task must have documented retry semantics and a dead-letter/error-sink. Flag this in the spec as a precondition for enabling a non-`ImmediateBackend` backend in production.
- **GAP-8 — Already-deferred: per-site IP-capture toggle.** No new action; simply acknowledging it is a known privacy-compliance gap for multi-tenant deployments crossing jurisdictions.
- **GAP-9 — Specify the `actor_role` derivation rule.** *Closed — no longer applicable.* `actor_role` is not stored on the event; role resolution happens at query time from the live permission model.
- **GAP-10 — No PII in log lines.** State explicitly that `TrackingSchemaError` messages and the unknown-key warning log the field *name* only, never the *value*. Tested in the validation-error tests.

## 5. Summary

- **Vectors considered:** 18 (V1–V18), mapped to OWASP Top 10:2025.
- **Vectors adequately covered by the spec as written:** V1, V2, V3, V5 (with minor GAP-2), V6, V8, V9, V10, V12, V16, V17.
- **Vectors needing spec updates:** V4 (GAP-1), V5 (GAP-2), V7 (GAP-3, GAP-4), V10 (GAP-5), V11 (GAP-6), V13 (GAP-7), V14 (GAP-8, already deferred), V15 (GAP-9), V18 (GAP-10).
- **Total gaps raised:** 10, of which 1 (GAP-8) is already acknowledged as deferred in the spec.

The fundamentals — immutability in depth, multi-tenant scoping, opaque IFI, salted session hash, read-only admin, separate erasure role — are solid. The gaps cluster around *how the edges behave under adversarial or careless input*: unknown fields, oversize payloads, caller-supplied snapshots, log payload hygiene, and the runtime enforcement of "must go through `track()`". Address them in the spec before planning.
