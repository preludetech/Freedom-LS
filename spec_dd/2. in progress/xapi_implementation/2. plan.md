# Implementation plan — Lightweight xAPI Event Tracking

This plan breaks the spec in `1. spec.md` into TDD-ordered tasks. It assumes the reader has read `1. spec.md`, `1.1 threat_model.md`, `erasure.md`, and `structure-concerns.md`.

The plan is organised so that each phase is independently testable. Inside each phase, follow the red-green-refactor cycle: write the smallest failing test, make it green, refactor. Do **not** batch-write tests — write one test, implement, move on.

## Design split

`experience_api` is **generic event-logging infrastructure** and depends only on `accounts` and `site_aware_models`. Domain apps (`student_interface`, `student_progress`) own their event types — they define the Pydantic schemas, the snapshot helpers, and the per-event-type `track_*` wrappers in a local `xapi_events.py` module, and register the schemas at `apps.py.ready()` time.

Schemas are **not** catalogued in a cross-cutting markdown file. Each owning app's `xapi_events.py` is the source of truth for its own event types; the `experience-api` skill documents the conventions and the "add an event type" workflow.

## Phase 1 — `Event` model, `ActorErasure` model, immutability, and admin

**Goal.** Land the storage, the four layers of immutability enforcement, the erasure audit table, and the read-only admin. No tracker, no schema validation, no registry yet.

### Tasks

1. **`models.py` — `Event(SiteAwareModel)`** with all fields from spec §"Storage model → Fields":
   - `id` (UUID PK), `site_domain`, `actor_user` (FK → `settings.AUTH_USER_MODEL`, `null=True`, `on_delete=models.SET_NULL`), `actor_email`, `actor_display_name`, `actor_ifi`, `verb`, `verb_display`, `object_type`, `object_id` (`UUIDField, null=True`), `object_definition` (`JSONField`), `result` (`JSONField, null=True`), `context` (`JSONField`), `statement` (`JSONField`), `timestamp`, `stored` (`auto_now_add=True`), `session_id_hash`, `user_agent`, `ip_address`, `platform`.
   - `site` FK comes from `SiteAwareModelBase` — do **not** redeclare.
   - No `CASCADE` anywhere. Every FK is `null=True, on_delete=SET_NULL` except `site` (inherited `PROTECT`).
   - Indexes from spec §"Indexes":
     - `(site, verb, -timestamp)`
     - `(site, actor_user, -timestamp)`
     - `(site, object_type, object_id, -timestamp)`
     - BRIN on `timestamp`
     - GIN on `context` with `jsonb_path_ops`
     - GIN on `result` with `jsonb_path_ops`
   - Use `django.contrib.postgres.indexes.BrinIndex` and `GinIndex(fields=["context"], opclasses=["jsonb_path_ops"], name=...)`.

2. **`Event.save()` override — immutability rules 1 and 2:**
   ```pseudocode
   def save(self, *args, **kwargs):
       if not self._state.adding:
           raise EventImmutableError("Event rows are immutable")
       if not getattr(self, "_tracker_authorised", False):
           raise EventImmutableError("Events must be created via track()")
       delattr(self, "_tracker_authorised")
       super().save(*args, **kwargs)
   ```
   `self._state.adding` is the reliable "first save" signal; Event uses UUIDs set before save, so `self.pk` is not a reliable "already in DB" signal, and `Event.objects.filter(pk=...).exists()` would be site-filtered and unreliable.

3. **`Event.delete()` override:**
   ```pseudocode
   def delete(self, *args, **kwargs):
       raise EventImmutableError("Events cannot be deleted via ORM")
   ```

4. **Custom manager / queryset** extending `SiteAwareManager`. Override `delete()` and `update()` on the queryset to raise `EventImmutableError`.

5. **`pre_save` signal (defence in depth)** — registered in `apps.py.ready()`. Same logic as `save()` override. The `save()` override deletes `_tracker_authorised` only after `super().save()` succeeds; `pre_save` fires before that, so the flag is present when the signal runs.

6. **`ActorErasure` model** (lands now alongside `Event`):
   - Fields per spec: `id` (UUID PK), `target_user_id` (BigInt), `erased_token` (CharField), `event_count` (Int), `timestamp` (`auto_now_add`), `invoking_os_user` (CharField), `invoking_hostname` (CharField), `invoking_admin_user_id` (nullable BigInt).
   - `save()` override raises `EventImmutableError` if `not self._state.adding`. `delete()` raises. Queryset `update` / `delete` raise.
   - Multi-tenant: also subclass `SiteAwareModel` so erasures are site-scoped for admin browsing.

7. **Migration 0001** — creates the `Event` and `ActorErasure` tables plus indexes. Auto-generated by `uv run manage.py makemigrations experience_api`.

8. **Migration 0002 (RunSQL)** — issue:
   - `REVOKE UPDATE, DELETE ON experience_api_event FROM <app_role>;`
   - `REVOKE UPDATE, DELETE ON experience_api_actorerasure FROM <app_role>;`
   - `CREATE ROLE fls_erasure_role NOLOGIN;` (idempotent `DO $$ IF NOT EXISTS (SELECT 1 FROM pg_roles WHERE rolname='fls_erasure_role') THEN ... END IF; END $$;`)
   - `GRANT UPDATE ON experience_api_event TO fls_erasure_role;`
   - **Do not** grant `UPDATE` or `DELETE` on `actor_erasure` to the erasure role — the audit table is insert-only for everyone.
   - Role is created `NOLOGIN`; a separate login user (provisioned out-of-band) is made a member via `GRANT fls_erasure_role TO <login_user>`. Document in the migration docstring.
   - Reversible: reverse operation re-grants `UPDATE, DELETE` on the events table to the app role. Does **not** drop `fls_erasure_role` (dropping a role that still owns grants is risky).

   > **Resolved:** `docs/deployment-security-checklist.md` §"Database Security" calls out explicitly that the application DB user must not be a superuser (superusers bypass grants).

9. **`admin.py` — `EventAdmin(SiteAwareModelAdmin)`** (spec §"Admin"):
   - `list_display = ("timestamp", "actor_email", "verb_display", "object_type", "object_definition_summary")`
   - `list_filter = ("verb", "object_type")`, `date_hierarchy = "timestamp"`
   - `search_fields = ("actor_email", "actor_display_name")`
   - Read-only: `has_add_permission`, `has_delete_permission`, `has_change_permission` return `False`; `readonly_fields` covers every field.

## Phase 2 — Schema base, registry, context

**Goal.** Land the type-system plumbing that domain apps will use. Still no tracker. No schemas registered yet.

### Tasks

1. **`schema_base.py`** — Pydantic v2 `BaseEventSchema`:
   ```python
   from pydantic import BaseModel, ConfigDict

   STRING_EXTENSION_MAX_LENGTH = 2048
   RESULT_RESPONSE_MAX_LENGTH = 20480
   SNAPSHOT_STRING_MAX_LENGTH = 512
   LIST_EXTENSION_MAX_ITEMS = 100
   HARD_CEILING_BYTES = 65536

   class BaseEventSchema(BaseModel):
       model_config = ConfigDict(extra="forbid")
       object_definition: BaseModel
       result: BaseModel | None = None
       context_extensions: BaseModel
   ```

2. **`registry.py`**:
   ```pseudocode
   REGISTRY: dict[tuple[Verb, str], type[BaseEventSchema]] = {}

   def register_event_type(verb: Verb, object_type: str, schema_cls: type[BaseEventSchema]) -> None:
       key = (verb, object_type)
       existing = REGISTRY.get(key)
       if existing is not None and existing is not schema_cls:
           raise RuntimeError(...)
       REGISTRY[key] = schema_cls

   def get_schema(verb: Verb, object_type: str) -> type[BaseEventSchema]:
       try:
           return REGISTRY[(verb, object_type)]
       except KeyError:
           raise TrackingSchemaError(...)
   ```
   - Idempotent re-registration of the *same* schema is allowed (Django's `ready()` can fire twice under some test configs).

3. **`context.py`** — pure helpers, no DB writes:
   - `get_current_site_and_domain(request) -> tuple[Site | None, str | None]`
   - `hash_session_key(session_key: str | None) -> str | None` — spec §"What the tracker derives automatically":
     ```pseudocode
     if not session_key: return None
     salted = (settings.SECRET_KEY + session_key).encode("utf-8")
     return hashlib.sha256(salted).hexdigest()
     ```
   - `get_ip_address(request) -> str | None` — returns `None` if `settings.EXPERIENCE_API_CAPTURE_IP` is `False`.
   - `get_user_agent(request) -> str | None`.
   - `build_actor_ifi(user: User, site: Site) -> str` — returns `f"{site_homepage(site)}|{user.id}"`.

## Phase 3 — Generic `track()` and `write_event` task

**Goal.** The tracker core in `experience_api.tracking.track()` — validation, composition, persistence — purely generic. No domain-app involvement yet. This phase is testable with a dummy in-test schema registered via `register_event_type`.

### Tasks

1. **`tasks.py`** — async write path:
   ```pseudocode
   from django.tasks import task

   @task()
   def write_event(payload: dict) -> None:
       instance = Event(**payload)
       instance._tracker_authorised = True
       instance.save()
   ```
   The task takes a fully-resolved `payload`. No `_thread_locals` reads.

2. **`tracking.py` — the generic `track()`**:
   ```pseudocode
   def track(
       *,
       actor: User | None,
       verb: Verb,
       object_type: str,
       object_id: UUID | None,
       object_definition: dict,
       result: dict | None = None,
       context_extensions: dict | None = None,
       request: HttpRequest | None = None,
       strict: bool | None = None,
   ) -> Event:
       # 1. Resolve request (explicit > _thread_locals > None)
       req = request or getattr(_thread_locals, "request", None)

       # 2. Resolve site / domain / session / UA / IP
       site, site_domain = get_current_site_and_domain(req)
       session_id_hash = hash_session_key(
           getattr(req.session, "session_key", None) if req else None
       )
       ip = get_ip_address(req) if req else None
       ua = get_user_agent(req) if req else None

       # 3. Length pre-check — any single string > HARD_CEILING_BYTES raises
       enforce_hard_ceiling(object_definition, result, context_extensions or {})

       # 4. Pydantic validation — look up schema, validate composed input
       schema_cls = get_schema(verb, object_type)
       strict_mode = strict if strict is not None else settings.EXPERIENCE_API_STRICT_VALIDATION
       try:
           validated = schema_cls(
               object_definition=object_definition,
               result=result,
               context_extensions=context_extensions or {},
           )
       except ValidationError as exc:
           handled = handle_validation_error(exc, strict_mode, verb, object_type,
                                              object_definition, result, context_extensions)
           # handle_validation_error may re-raise, or return a cleaned schema instance,
           # or return None to signal "drop the event" (permissive mode, unrecoverable error).
           if handled is None:
               return None  # event dropped per permissive-mode contract
           validated = handled

       # 5. Dangling-pointer guard — any `_id` populated but paired `_slug`/`_title`/`_name` empty → raise
       enforce_dangling_pointer_rule(validated)

       # 6. Derive actor fields
       actor_ifi = build_actor_ifi(actor, site) if actor else ""
       actor_email = actor.email if actor else ""
       actor_display_name = getattr(actor, "display_name", "") or (actor.email if actor else "")

       # 7. Compose the full xAPI `statement` JSON
       statement = compose_statement(actor_ifi, verb, validated)

       # 8. Build payload dict — NO _thread_locals reads beyond this point
       payload = {
           "id": uuid4(),
           "site_id": site.id if site else None,
           "site_domain": site_domain or "",
           "actor_user": actor,
           "actor_email": actor_email,
           "actor_display_name": actor_display_name,
           "actor_ifi": actor_ifi,
           "verb": verb.iri,
           "verb_display": verb.display,
           "object_type": object_type,
           "object_id": object_id,
           "object_definition": validated.object_definition.model_dump(),
           "result": validated.result.model_dump() if validated.result else None,
           "context": {"extensions": validated.context_extensions.model_dump()},
           "statement": statement,
           "timestamp": timezone.now(),
           "session_id_hash": session_id_hash,
           "user_agent": ua,
           "ip_address": ip,
           "platform": "backend",
       }

       # 9. Enqueue (ImmediateBackend runs synchronously)
       write_event.enqueue(payload)
       return Event.objects.get(pk=payload["id"])
   ```

3. **`handle_validation_error(...)`** — centralised helper:
   - **Strict mode**: always re-raise as `TrackingSchemaError` with a field-name-only message.
   - **Permissive mode**:
     - If the only errors are unknown-extension keys in `context_extensions`: strip them, re-validate, return the cleaned schema. Log the dropped key names at warning.
     - If there are missing-required / type-mismatch / oversize errors: log at error, return `None` (drop the event).

4. **Log payload hygiene** — helper `log_field_name_only(...)` used by every path that logs. Values never reach the logger.

5. **`compose_statement(actor_ifi, verb, validated)`** — produces the full xAPI-shaped dict (denormalised composition). Pure function, no DB hit.

## Phase 4 — Management commands

**Goal.** `erase_actor` and `maintain_event_partitions`. Both can land before domain-app wiring.

### Tasks

1. **`management/commands/maintain_event_partitions.py`** (spec §"Partition maintenance"):
   - CLI flags: `--lookahead-months N` (default 3), `--retention-months M` (default 24), `--drop-old`, `--dry-run`.
   - Detect partitioning: query `pg_partitioned_table` for `experience_api_event`. If not partitioned, log structured warning and exit 0.
   - If partitioned: create missing future partitions; with `--drop-old`, detach and drop expired partitions.
   - Log every action at `INFO` with structured fields.

2. **`management/commands/erase_actor.py`** (spec §"Right to erasure"):
   - CLI flags: `--user-id <int>` (required), `--confirm` (required), `--admin-user-id <int>` (required when `EXPERIENCE_API_STRICT_VALIDATION=True`).
   - Connect via the erasure role — open a second Django DB connection using `settings.DATABASES["erasure"]`. Refuse if the erasure-user env vars are empty (explicit error naming them).
   - Refuse if the target user has active `UserCourseRegistration` rows — exit non-zero. **Implementation note**: `experience_api` cannot import `student_management`, so the erasure command discovers "still-has-history" via a hook. Simplest: a setting `EXPERIENCE_API_ERASURE_BLOCKERS` listing dotted paths to callables `(user_id) -> bool`; the command calls each and refuses if any returns True. The FLS-specific blocker (check for active `UserCourseRegistration`) lives in `student_management/xapi_erasure.py` and is wired via settings. The portable default is an empty list.
   - In a single transaction on the erasure connection:
     - Generate token `secrets.token_hex(8)`; erased email `f"erased-{token}@example.invalid"`, display name `f"Erased actor {token}"`.
     - Find events: `Event.objects.filter(Q(actor_user_id=user_id) | Q(actor_ifi__endswith=f"|{user_id}"))`. The `__endswith` is deliberate — `actor_ifi` has shape `"<site_homepage>|<user.id>"`, and `__contains="|5"` would false-positive across user IDs 15, 25, 50, 51, 150, etc. `--user-id` is typed as `int` by argparse so no further sanitisation is needed for the LIKE.
     - Execute parameterised raw SQL (`cursor.execute(sql, params)`) through the erasure connection to set `actor_email`, `actor_display_name`, `actor_user_id=NULL`, and `statement->'actor'` via `jsonb_set`. This is the one narrowly-scoped exception to the ORM-only rule; the ORM path is blocked by `Event.save()` by design.
     - Insert one row into `actor_erasure` capturing `target_user_id`, `erased_token`, `event_count`, `timestamp`, `invoking_os_user` (`getpass.getuser()`), `invoking_hostname` (`socket.gethostname()`), `invoking_admin_user_id`.

3. **`EXPERIENCE_API_ERASURE_BLOCKERS` setting** — new in `settings_base.py`: empty list. Dev/prod settings append `"freedom_ls.student_management.xapi_erasure.has_active_registrations"` once Phase 6 lands it.

## Phase 5 — `student_interface/xapi_events.py`: schemas, helpers, wrappers, call-site wiring

**Goal.** `student_interface` owns `(VIEWED, Topic)`, `(ATTEMPTED, Form)`, `(ANSWERED, Question)`, `(COMPLETED, Form)`, `(REGISTERED, Course)`. Land the schemas, snapshot helpers, wrappers, and call-site wiring in one phase.

### Tasks

1. **`content_engine/xapi_snapshots.py`** — pure read helpers for content-engine models. No imports from `experience_api`.
   - `snapshot_topic(topic) -> dict` → `{topic_id, topic_slug, topic_title, topic_type}`.
   - `snapshot_form(form) -> dict` → includes `form_id, form_slug, form_title, form_type, question_count, max_score`.
   - `snapshot_question(question) -> dict` → includes `question_id, question_slug, question_text, question_type, options`.
   - `snapshot_course(course) -> dict` → includes `course_id, course_slug, course_title`.
   - Relationship walks (use `select_related` / `prefetch_related` per CLAUDE.md):
     - `walk_topic_to_course(topic) -> Course | None` — reverse-walk `ContentCollectionItem` to the parent `Course` (directly or via `CoursePart`). Returns the unique match; `None` for zero or multiple (caller must disambiguate).
     - `walk_form_to_topic(form) -> Topic | None`.
     - `walk_form_to_course(form) -> Course | None`.

2. **`student_management/xapi_snapshots.py`** — pure read helpers for registrations / cohorts. No imports from `experience_api`.
   - `resolve_user_course_registration(user, course) -> UserCourseRegistration | None` — single active registration, or `None`.
   - `resolve_cohort_for_user(user, site) -> Cohort | None` — single `CohortMembership`, or `None`.

3. **`student_interface/xapi_events.py`** — Pydantic schemas, wrappers, `register_event_type` calls.
   - Schemas: `ViewedTopicSchema`, `AttemptedFormSchema`, `AnsweredQuestionSchema`, `CompletedFormSchema`, `RegisteredCourseSchema`. Each subclasses `BaseEventSchema`, nests `ObjectDefinition`, `Result`, `ContextExtensions` models, and applies size-cap constants from `experience_api.schema_base`.
   - Field shapes follow the conventions in spec §"Conventions for domain-app schemas". Existing guidance (nullable `_id` paired with required `_slug`/`_title`/`_name`; always include course/cohort context; snapshot all mutable context; size caps) applies to every schema.
   - `register_event_type(VIEWED, "Topic", ViewedTopicSchema)` etc. at module top-level.
   - Wrappers: `track_topic_viewed`, `track_form_attempted`, `track_question_answered`, `track_form_completed`, `track_course_registered`. Each:
     - Takes explicit kwargs for every schema field the call site supplies.
     - Calls the right walk helper(s).
     - Applies the caller-supplied-vs-model-derived preference rule: if the caller passes both `<obj>_id` and `<obj>_slug`/`_title`/`_name` for a record that still exists, use model-derived values and ignore caller's (log at debug when they diverge); only fall back to caller's when the record is gone.
     - Builds the `object_definition` and `context_extensions` dicts.
     - Delegates to `experience_api.tracking.track(...)`.

4. **`student_interface/apps.py`** — `ready()` imports `xapi_events` so the registrations fire.

5. **Call-site wiring** (one edit per call site — keep the wrapper call a single line):

   **Authentication precondition.** Every call site is behind `@login_required` / an equivalent mixin already. Before wiring each, confirm the decorator is in place; if any view is missing it, fix it in the same commit. The tracker dereferences `user.id` in `build_actor_ifi`, which would explode on `AnonymousUser`.

   - `(VIEWED, Topic)` — `views.py::view_topic` (~line 213). At the end before returning: `track_topic_viewed(request.user, topic, request=request, referrer=request.META.get("HTTP_REFERER"))`.
   - `(ATTEMPTED, Form)` — `views.py::form_start` (~line 290): `track_form_attempted(request.user, form, request=request, attempt_number=...)`.
   - `(ANSWERED, Question)` — `views.py::form_fill_page`, inside the answer-save branch, once per question-answer: `track_question_answered(request.user, question, request=request, form_attempt_id=form_progress.id, attempt_number=..., question_position=..., changed_answer=..., result={...})`.
   - `(COMPLETED, Form)` — `views.py::course_form_complete` (~line 417): `track_form_completed(request.user, form, request=request, result={...}, attempt_number=..., answers_changed=..., pass_threshold=..., timed_out=...)`.
   - `(REGISTERED, Course)` — `views.py::register_for_course` (~line 121): `track_course_registered(request.user, course, request=request, result={"success": True}, registered_by="self", cohort=cohort_or_none)`.

6. **Content-engine and student-management apps** — the new `xapi_snapshots.py` modules are additive. `apps.py` in those apps does **not** need to import them; the consumer (`student_interface.xapi_events`) imports them directly.
