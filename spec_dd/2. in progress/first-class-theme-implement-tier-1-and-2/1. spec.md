# Spec: first_class theme ‚Äî implement Tier 1 + Tier 2

## Summary

Refine the `first_class` canary theme so its rendered output matches the
"Modern Altitude" design-system mockups in `./design-system/`. The work has
two halves:

1. **Tier 1** ‚Äî token-level work in **both** `default` and `first_class`
   themes. Adds `--fls-font-mono`, status-light tints, and paired
   `on-*-light` foregrounds; corrects first_class radii drift.
2. **Tier 2** ‚Äî component-class work in **both** `tailwind.components.css`
   (the public default contract) and `first_class/theme.css` (the override
   layer). Introduces `.btn-secondary`, `.btn-ghost`, `.btn-accent`,
   `.chip-info`, `.chip-secondary`, `.chip-muted`, and the `.alert` family;
   removes `.btn-outline` from the public API and migrates all callers.

This spec **supersedes** the in-progress
`themable-implementations-phase-3-component-class-overrides` spec, which
only covered three demo classes. When this work picks up, that older
in-progress folder is archived (no implementation merged from it).

## Guardrail (universal)

> Any new component class lands in **both** `tailwind.components.css`
> (default contract) **and** `first_class/theme.css` (override). Never
> quietly add a class to first_class alone ‚Äî every component class is a
> public API on every theme.

This is the same rule the master `themable-implementations` spec already
sets. Restating because every Tier-2 addition below has to honour it.

## Why this matters

- The first_class canary is the proof that the theming contract works
  end-to-end. If its rendered output drifts from the Modern Altitude
  mockups, the contract has a hole ‚Äî either a missing token or a
  component class that bypasses tokens.
- Tier-1 additions (`*-light`, `on-*-light`, `font-mono`) close concrete
  gaps in the role contract that the mockups already exercise. Without
  them, templates would have to reach for ad-hoc `bg-success/15` opacity
  modifiers or hard-coded hex values, both of which break theme override.
- Tier-2 additions widen the public component API to cover the affordances
  the mockups show (secondary-CTA outline buttons, ghost text buttons,
  accent CTAs, info chips, muted chips, alert callouts). Not adding them
  would force templates to invent local classes per affordance, which
  defeats the contract.

## Tier 1 ‚Äî token additions

### 1.1 Mono font

**Decision:** add `--fls-font-mono` plus a Tailwind alias, exercised by
`type-mono.html` and the technical-spec / timestamp surfaces in the
mockups.

In **`freedom_ls/themes/default/static/themes/default/theme.css`**:

- Add to the `@theme` block:
  ```css
  --fls-font-mono: ui-monospace, SFMono-Regular, Menlo, Consolas,
      "Liberation Mono", "Courier New", monospace;
  ```
- Add to the `@theme inline` block:
  ```css
  --font-mono: var(--fls-font-mono);
  ```

In **`freedom_ls/themes/first_class/static/themes/first_class/theme.css`**:

- Add to the `@theme` block (no inline-alias re-declaration ‚Äî alias is
  inherited from default):
  ```css
  --fls-font-mono: "IBM Plex Mono", ui-monospace, Menlo, Consolas, monospace;
  ```

**Why:** mono is a brand requirement and a real role token. Aliasing it
into `--font-mono` once in default keeps the Tailwind contract centralised
so downstream themes only redeclare the value.

### 1.2 Status-light tints

**Decision:** add `--color-success-light`, `--color-warning-light`,
`--color-error-light`, `--color-info-light` to **both** themes, with
identical hex values pulled from the brand doc:

| Token | Value |
| --- | --- |
| `--color-success-light` | `#F0FFF4` |
| `--color-warning-light` | `#FFFFF0` |
| `--color-error-light` | `#FFF5F5` |
| `--color-info-light` | `#EBF8FF` |

**Why:** these replace today's `bg-success/15`-style opacity modifiers in
chip and alert backgrounds with named tokens. The hexes are also Tailwind
v3's canonical `*-50` values, so they're a sensible default-theme baseline
too ‚Äî not a first_class-specific concession.

**Out of scope:** `--color-secondary-light`. `.chip-secondary` reaches for
`bg-secondary/15` opacity instead, mirroring how `.chip-primary` already
works. Introducing a `secondary-light` token without a `primary-light`
companion would be inconsistent.

### 1.3 `on-*-light` foreground tokens

**Decision:** add `--color-on-success-light`, `--color-on-warning-light`,
`--color-on-error-light`, `--color-on-info-light` to **both** themes, with
identical hex values:

| Token | Value |
| --- | --- |
| `--color-on-success-light` | `#22543D` |
| `--color-on-warning-light` | `#744210` |
| `--color-on-error-light` | `#742A2A` |
| `--color-on-info-light` | `#2A4365` |

**Why:** the existing `--color-on-success: #FFFFFF` etc. are illegible on
the near-white `*-light` tints. Without these, chip/alert templates would
fall back to ad-hoc `text-*` classes and bypass the role contract.
Downstream themes that want a different palette can override per token.

### 1.4 First_class radii correction

**Decision:** realign `first_class/theme.css` radii to the
`design-system/radii.html` mockup spec (currently one step too round):

```css
--fls-radius-sm: 0.375rem;  /* 6px  ‚Äî chips, badges */
--fls-radius-md: 0.5rem;    /* 8px  ‚Äî buttons, inputs */
--fls-radius-lg: 0.75rem;   /* 12px ‚Äî cards, modals */
/* --fls-radius-pill stays inherited from default (9999px ‚âà 999px) */
```

Drop the existing "bumped one step for the brand's rounder card treatment"
comment; replace with a brief note that radii match `radii.html`.

**Why:** the design-system mockups are the visual contract for Tier 2.
The brand-doc text doesn't pin radii ‚Äî only the mockups do ‚Äî so the tokens
must drive the same numbers.

### 1.5 Out-of-scope for Tier 1

- No 50‚Äì900 neutral ramp (deferred by master spec).
- No other token-surface widening.
- No type-scale changes at the token level (heading weight/tracking lives
  in Tier 2 below).
- No `secondary-hover` / `accent-hover` overrides ‚Äî flagged in
  `research_first_class_token_gaps.md` for Tier-2 QA, override per role
  only if QA finds them flat.

## Tier 2 ‚Äî component-class additions to `tailwind.components.css`

Each class below is added to the **default** contract first, with a
first_class override below where the look diverges. None of these classes
exist in `tailwind.components.css` today.

### 2.1 Buttons

**`.btn-secondary`** ‚Äî replaces `.btn-outline`.

- Default: `bg-transparent text-primary border border-primary
  hover:bg-primary/10 focus:ring-primary`.
- Affordance: "secondary CTA". This is the new neutral-secondary
  affordance going forward.
- **`.btn-outline` is removed** from `tailwind.components.css`, both
  themes, and every call site. There is no neutral-grey "cancel" variant
  in the public API anymore.
- Migration: every existing `class="btn btn-outline"` use becomes
  `class="btn btn-secondary"`. Likewise `<c-button variant="outline">`
  becomes `<c-button variant="secondary">`. Audit at the time of
  writing covers `freedom_ls/base/templates/cotton/pagination.html`,
  `freedom_ls/base/templates/cotton/data-table.html`,
  `freedom_ls/student_interface/templates/student_interface/partials/course_list.html`,
  `freedom_ls/student_interface/templates/student_interface/course_topic.html`,
  `freedom_ls/panel_framework/templates/panel_framework/partials/delete_confirmation.html`,
  and the `<c-button variant="outline">` usage example in
  `freedom_ls/base/templates/cotton/button.html`'s docstring. The
  `variant="outline"` in `freedom_ls/icons/templates/cotton/icon.html`
  is the *icon* component's icon-style variant (outline vs solid), not
  a button variant ‚Äî it is **not** affected by this migration.
  Implementation must run a final `grep -rn '\.btn-outline\|variant="outline"'`
  to catch any sites added between spec and merge.

**`.btn-ghost`** ‚Äî lower-emphasis text-button affordance.

- Default: `bg-transparent text-primary hover:bg-primary/10
  focus:ring-primary` (no border).
- Affordance: "tertiary CTA / cancel". Today this pattern is hand-rolled
  as `<a class="text-primary underline">`; landing it as a button class
  brings it into the contract.

**`.btn-accent`** ‚Äî solid Altitude-Orange CTA.

- Default: `bg-accent text-on-accent hover:bg-accent-hover
  focus:ring-accent`.
- Affordance: celebration moments ("Get certified") and accent CTAs.
- Mirrors `.btn-primary` / `.btn-success` / `.btn-error` naming for the
  remaining brand role.

**Why all three:** they close the role-parity gap so every spec token
role (`primary`, `secondary`, `accent`, `success`, `warning`, `error`,
`info`) has a button variant on the public API. `.btn-outline` was the
previous-gen "secondary" affordance with a different name ‚Äî this spec
reconciles the naming with the rest of the role contract and removes the
old class to avoid a confusing dual API.

### 2.2 Chips

**`.chip-info`** ‚Äî info-tinted chip.

- Default: `bg-info/15 text-info`.
- First_class override (see ¬ß3.2): `bg-info-light text-on-info-light`.

**`.chip-secondary`** ‚Äî brand-secondary tag (e.g. "72% scored").

- Default: `bg-secondary/15 text-secondary`.
- No first_class override needed beyond the base `.chip` shape ‚Äî the
  opacity-mix gives a workable wash on Electric Teal without introducing
  a `secondary-light` token (see ¬ß1.2).

**`.chip-muted`** ‚Äî neutral / locked / not-started state.

- Default: `bg-surface-2 text-muted`.
- Reuses existing tokens; no new token required.

**Why:** `chip-info` closes the role-parity gap with the other chip
variants. `chip-secondary` lets the mockup's "Practical" / "72% scored"
chips land without an ad-hoc class. `chip-muted` covers the mockup's
"Locked" affordance ‚Äî distinct semantics from `chip-warning`, so it
deserves its own variant rather than overloading warning.

### 2.3 Alerts (new class family)

**`.alert`** ‚Äî base layout.

- Default: padded box with border, rounded, body type. Concrete recipe to
  be drafted at implementation time but expected shape: `block px-4 py-3
  rounded-md border text-on-surface`.

**`.alert-success`** ‚Äî success variant.

- Default: `bg-success/10 border-success/30 text-on-surface`.
- First_class override (¬ß3.2): `bg-success-light border-success/30
  text-on-success-light`.

**`.alert-error`** ‚Äî same shape, error palette.

**`.alert-info`** ‚Äî same shape, info palette.

**Why:** alerts are a real cross-cutting feedback affordance for the LMS
(quiz feedback, hints, "Up next" callouts). Today's templates either
inline the styling or use `<blockquote>`. Landing them as classes brings
them into the contract.

**Caveat (zero callers today):** unlike the buttons and chips above, the
`.alert` family currently has zero call sites. The class is being added
ahead of demand because the canary mockup demonstrates the role
contract, and templates can adopt incrementally. This is an explicit
exception to "don't build functionality that is not requested" ‚Äî the
canary's job is to demonstrate the role contract, and demonstrating
`.alert` requires the class to exist.

### 2.4 Out-of-scope for Tier 2

- **`.alert-warning`** and **`.alert-cta`** ("Up next" prescriptive
  layout from the mockup) ‚Äî defer until a real call site needs them.
- **`.btn-disabled`** ‚Äî the base `.btn` already applies
  `disabled:opacity-50 disabled:cursor-not-allowed` via the native
  `disabled` attribute. A separate class would duplicate that.

## Tier 2 ‚Äî first_class overrides in `first_class/theme.css`

All overrides live inside one shared `@layer components { ‚Ä¶ }` block.
Concrete `@apply` rules are pre-drafted in `research_tier2_overrides.md`;
the spec expectations summarise them.

Cascade ordering relies on `first_class/theme.css` being `@import`-ed
*after* `tailwind.components.css`. This is already established by the
existing theme-loading mechanism ‚Äî verify with one devtools check after
first deploy.

### 3.1 Existing classes (reopened)

**`.btn`** ‚Äî chunkier brand shape.

- Override expectations: `px-6 py-3 text-sm font-semibold rounded-md`,
  with a `200ms cubic-bezier(0.22, 1, 0.36, 1)` transition on background
  and color (replacing `transition-colors`'s linear curve).
- Why: brand wants 12/24 padding, semibold weight, springier easing.
  Buttons land at the 8px stop per `radii.html` ("8px ¬∑ default ¬∑
  buttons, inputs"), which is `--fls-radius-md` after the ¬ß1.4
  correction ‚Äî so `rounded-md` is the right token here.

  *Note:* `research_tier2_overrides.md` recommends `rounded-sm` because
  it was drafted against the **pre-correction** first_class radii where
  `--fls-radius-sm` happened to equal 8px. After ¬ß1.4 corrects the
  ramp to 6/8/12px, the same 8px target lives at `rounded-md`.

**`.chip`** ‚Äî full pill shape.

- Override expectations: `gap-1.5 px-3 py-1.5 text-xs font-semibold
  rounded-pill`.
- Why: brand chips are full pills, one weight step heavier, smaller text.

**`.chip-primary`, `.chip-warning`, `.chip-success`, `.chip-error`** ‚Äî
flatter Light-tint look using the new `*-light` / `on-*-light` tokens.

- `.chip-primary`: `bg-primary/10 text-primary` (lighter wash).
- `.chip-warning`: `bg-warning-light text-on-warning-light` (text flips
  to dark amber ‚Äî was `text-on-warning` in default).
- `.chip-success`: `bg-success-light text-on-success-light`.
- `.chip-error`: `bg-error-light text-on-error-light`.
- Why: matches the mockup's pastel near-white tints with darker
  hue-matched text. The `*-light` tokens added in Tier 1 carry the load.

**`.surface`** ‚Äî explicit `bg-surface` plus card-radius bump.

- Override expectations: `bg-surface rounded-lg`.
- Why: surfaces lift off the slightly-darker page background a downstream
  brand may choose. Cards land at the 12px stop per `radii.html` ("12px
  ¬∑ lg ¬∑ cards, modals"), which is `--fls-radius-lg` after the ¬ß1.4
  correction ‚Äî so the override has to nudge the radius from the
  default-contract `rounded-md` (8px after correction) up to
  `rounded-lg` (12px) to match the mockup.

  *Note:* `research_tier2_overrides.md` says the radius "already
  matches" with no shape change needed, because it was drafted against
  the **pre-correction** first_class radii where `--fls-radius-md`
  happened to equal 12px. After ¬ß1.4 the 12px target lives at
  `rounded-lg`, so an explicit shape override is required.

### 3.2 Newly-added classes (overrides where first_class diverges)

**`.btn-secondary`** ‚Äî strengthen border to `border-2`.

- Override expectations: `border-2` (default `.btn-secondary` from ¬ß2.1
  plus a thicker outline to match the brand's "thicker outline"
  rationale in `components-buttons.html`).

**`.chip-info`** ‚Äî switches to the new info-light token pair.

- Override expectations: `bg-info-light text-on-info-light`.

**`.chip-secondary`** ‚Äî match the flatter pill treatment.

- Override expectations: lighter wash via `bg-secondary/10 text-secondary`
  (analogous to `.chip-primary` first_class override).

**`.chip-muted`** ‚Äî match the flatter pill treatment.

- Override expectations: confirm `bg-surface-2 text-muted` reads
  correctly in first_class; if not, retune at implementation. Likely no
  override needed.

**`.alert-success`, `.alert-error`, `.alert-info`** ‚Äî switch backgrounds
and text to the `*-light` / `on-*-light` token pairs; tighten border
colour.

- Override expectations: `bg-success-light text-on-success-light
  border-success/30` (and the analogous error / info pairs). Border
  colour stays at the role's mid-tone via the opacity modifier; only the
  surface flips to the pastel.

### 3.3 Tier-1-only classes (no Tier-2 override needed)

The following classes pick up the brand look purely through the Tier-1
token swap and the reopened `.btn` / `.chip` base shape ‚Äî no per-class
override is required:

- `.btn-primary` ‚Äî uses Tier-1 `--color-primary` (Deep Indigo).
- `.btn-success` / `.btn-error` ‚Äî use Tier-1 status colours.
- `.btn-accent` / `.btn-ghost` ‚Äî use Tier-1 accent / primary tokens, plus
  the reopened `.btn` shape.
- `.chip-xs` ‚Äî modifier on the reopened `.chip` base; no override.
- `.alert` ‚Äî base shape only; variants carry the Light-tint palette.

This list is informational ‚Äî no implementation work is needed for these,
but reviewers should know they were considered, not forgotten.

## Tier 2 ‚Äî `@layer base` override in first_class theme.css

**Decision:** match the design-system type scale where it diverges from
the FLS default base layer:

- H2, H3, H4 use `font-semibold` (not `font-bold`).
- H1 and H2 use `tracking-tight`.

Body, mono, and the responsive size scale stay as-is.

Override lives inside an `@layer base { ‚Ä¶ }` block in
`first_class/theme.css`:

```css
@layer base {
    h1 { @apply tracking-tight; }
    h2 { @apply font-semibold tracking-tight; }
    h3 { @apply font-semibold; }
    h4 { @apply font-semibold; }
}
```

**Why:** the brand uses `font-semibold` for H2‚ÄìH4 and tracking-tight on
the largest two headings. This is a genuine brand divergence from the
shared FLS contract that should not be pushed back into
`tailwind.components.css` ‚Äî other themes are entitled to their own
heading weights. Reopening base layer in the theme is the right escape
hatch.

**Why not in Tier 1:** font-weight per heading level is a *component*
concern (it shapes how a tag renders), not a *token* concern. Adding
`--fls-heading-weight` tokens would widen the contract for one canary's
needs.

## Implementation order

This is guidance, not a strict gate ‚Äî but the dependency chain is:

1. **Tier-1 tokens land first** in both themes. Without `--fls-font-mono`
   aliased into `--font-mono`, the `font-mono` Tailwind class is a no-op.
   Without `*-light` tokens, the chip/alert overrides in ¬ß3.2 reference
   undefined CSS variables.
2. **Tier-1 first_class radii correction** lands with the rest of Tier 1.
3. **`tailwind.components.css` contract additions** (¬ß2.1, ¬ß2.2, ¬ß2.3)
   land before any first_class Tier-2 override that reopens them. The
   `.btn-outline` removal happens in the same commit as its callers'
   migration to `.btn-secondary`, so the build is never broken.
4. **first_class Tier-2 component overrides** (¬ß3.1, ¬ß3.2) land last,
   building on the contract from step 3 and the tokens from step 1.
5. **first_class `@layer base` heading override** (¬ß4) lands alongside
   the Tier-2 overrides ‚Äî it shares the same file and the same review
   surface.
