# Make the whole site accessible

Target: WCAG 2.2 Level AA. Motivation is doing the right thing for users, not compliance-driven.

## Scope

This covers both:

1. **Platform accessibility** — templates, components, navigation, HTMX interactions
2. **Content author guidance** — tooling and documentation to help course content creators produce accessible content (alt text, heading structure, video captions, etc.)

## Phase 1: Fix critical issues and set up foundations

### Base template and layout

- Add skip navigation link ("Skip to main content") to the base template
- Add ARIA landmark elements (`<main>`, `<nav aria-label="...">`, `<header>`, `<footer>`)
- Ensure `<html lang="...">` is set
- Ensure every page has a descriptive `<title>`

### Modal component

- Convert the current custom Alpine.js modal (`cotton/modal.html`) to use native `<dialog>` element with `showModal()`
- This gives us automatic focus trapping, Escape key handling, background inertness, and focus restoration for free

### HTMX accessibility infrastructure

- Add a global `aria-live` announcer element (`.sr-only`) to the base template
- Add JS to update the announcer on `htmx:afterSwap` with concise status messages
- Add focus management on `htmx:afterSettle` — move focus appropriately after content swaps
- Add `aria-busy` toggling on `htmx:beforeRequest` / `htmx:afterSwap` for loading states
- Use `hx-disabled-elt` on submit buttons to communicate in-progress state

### Forms

- Ensure all form inputs have associated `<label>` elements
- Add `aria-describedby` linking error messages to their fields
- Add `aria-invalid="true"` on fields with errors
- Use `aria-live="assertive"` on error summary regions
- For multi-page forms: include step progress in headings (e.g., "Step 2 of 4")

### Visual

- Audit and fix color contrast ratios (4.5:1 for normal text, 3:1 for large text)
- Ensure visible focus indicators on all interactive elements
- Ensure color is never the sole means of conveying information (progress, status, errors)
- Ensure content reflows properly at 200% zoom

## Phase 2: Full review and cleanup

### Template audit

- Audit all ~70 templates for:
  - Missing alt text on images
  - Proper heading hierarchy (one `<h1>`, no skipped levels)
  - Semantic HTML (use `<button>` not `<div>` for interactive elements, etc.)
  - HTMX attributes only on semantic interactive elements (`<a>`, `<button>`, `<form>`)
  - Keyboard accessibility of all interactive elements

### Data tables (educator interface)

- Add proper `<th scope="col/row">` and `<caption>` to data tables
- Add `aria-sort` for sortable columns
- Announce filter result counts to screen readers

### Embedded content

- Add `title` attributes to all `<iframe>` elements (PDF embeds, YouTube embeds)
- Provide download links alongside PDF embeds
- Document caption/transcript requirements for video content

### Course content (markdown-rendered)

- Ensure markdown renderer produces semantic HTML (proper headings, lists, `<pre><code>` for code)
- Enforce heading hierarchy in rendered markdown (start at `<h2>`, no skipped levels)
- Post-process markdown tables to add `<th scope>` attributes
- Validate/warn on images with empty alt text in markdown content

### Progress indicators

- Use `role="progressbar"` with `aria-valuenow`, `aria-valuemin`, `aria-valuemax`
- Provide visible text equivalents (e.g., "3 of 10 topics completed")
- Announce progress updates via `aria-live`

## Phase 3: Set up automations to prevent regressions

### Automated testing (axe-core via Playwright)

- Add `axe-playwright-python` to project dependencies
- Create a `check_a11y` pytest fixture
- Gradual adoption strategy:
  - Start in report-only mode (log violations, don't fail)
  - Fix serious/critical violations immediately
  - Tighten to strict mode page by page as violations are fixed
- Add dedicated `tests/e2e/test_accessibility.py` that crawls key pages

### Template linting

- Enable djLint as a pre-commit hook (already installed, has a11y rules for alt, lang, title)

### Claude integration

- Add accessibility conventions to CLAUDE.md
- Add a Claude skill for accessibility review of templates
- Update spec-driven development workflow to include accessibility as a requirement for new features

### Content author documentation

- Document accessibility requirements for course content authors:
  - Alt text requirements for images
  - Heading structure guidelines
  - Video caption/transcript requirements
  - Link text best practices (descriptive text, not "click here")
  - File download descriptions (include type and size)
