# Educator Interface Permissions

## Goal

Secure the educator interface so that only authorised staff members can access it, and each educator only sees the cohorts, courses, and students they have permission to view or manage.

Provide a permission management UI within the educator interface so that admins can configure who has access to what without needing the Django admin.

## Access Control

- Only logged-in users with `is_staff=True` can access the educator interface
- Unauthenticated users are redirected to the login page
- Non-staff authenticated users get a 403

## Permission Model

Use **django-guardian** for object-level permissions. Permissions are granular and assigned per-object (per cohort, per course).

### Cohort Permissions (object-level via guardian)

- `view_cohort` — can see the cohort and its students in the educator interface
- `change_cohort` — can modify cohort settings
- `manage_cohort_students` — can add/remove students from the cohort

### Course Permissions (object-level via guardian)

- `view_course` — can see the course in the educator interface
- A user can be granted blanket "view all courses" via a model-level permission, or granted `view_course` on specific course instances

### Student Visibility

- Students are visible based on cohort access: if you can view a cohort, you can see the students in it
- Student detail access follows from having `view_cohort` on at least one cohort the student belongs to

### Groups for Bulk Assignment

- Use Django's built-in `Group` model for organising educators (e.g. "Science Department", "Year 1 Tutors")
- Permissions can be assigned to groups via guardian — all group members inherit the permissions
- This avoids repetitive per-user assignment when teams of educators share the same access

## Permission Management UI

Build into the educator interface (not the Django admin). Only users with appropriate admin-level permissions can access the permission management features.

### Key Views

1. **Educator list** — see all staff users, what groups they belong to, and a summary of their access
2. **Educator detail** — see and edit a specific educator's cohort/course access and group memberships
3. **Cohort access list** — from a cohort's page, see who has access and at what level, add/remove access
4. **Group management** — create/edit groups, assign permissions to groups in bulk

### UX Principles (from research)

- Use plain language: "Can view students" not "READ student_management"
- Show access from both directions: user → resources they can access, and resource → users who can access it
- Permission changes take effect immediately
- Confirmation for destructive changes (removing access)
- Keep the default experience simple — most of the time an admin is just granting view access to a cohort

## What This Does NOT Include

- Custom permission types beyond the ones listed above
- Student-facing permissions (student interface access is separate)
- Course content editing permissions (content management is a separate concern)
- Audit logging of permission changes (good future enhancement)
