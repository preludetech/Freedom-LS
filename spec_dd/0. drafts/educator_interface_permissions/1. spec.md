# FreedomLS Permission System — Technical Specification

**Version 1.0 · March 2026 · Status: Draft for Review · Classification: Internal**

---

## 1. Overview

### 1.1 Purpose

This document specifies the permission and authorization system for FreedomLS, an open-source Learning Management System built with Django. It is intended as a complete implementation reference for the development team.

The system must satisfy three priorities, in order: compliance and auditability (all permission state must be traceable, version-controlled, and reconcilable); usability (non-technical administrators can manage user access without confusion); and performance (the system scales to tens of thousands of users without degrading response times).

### 1.2 Architectural Summary

The permission system uses a layered architecture with strict separation of concerns:

- **Layer 1 — Role definitions (config files):** Python configuration files define named roles and their associated permissions. Each deployment site has its own config file. These are version-controlled, validated in CI, and immutable at runtime.
- **Layer 2 — Role assignments (database):** Django models record which users hold which roles, scoped to specific objects (courses, sites or system-wide). These are the canonical source of truth for who has what role where.
- **Layer 3 — Permission enforcement (django-guardian):** Guardian stores the computed fine-grained permissions derived from Layers 1 and 2. These are synchronised on role assignment/removal and reconciled on deployment. Guardian permissions are never exposed directly to administrators.

Permission checks at runtime hit Layer 3 (guardian) for speed. Layers 1 and 2 are the sources of truth; Layer 3 is a derived, reconcilable cache.

### 1.3 Key Design Decisions

| Decision | Rationale |
|---|---|
| Guardian for enforcement, not administration | Provides efficient object-level permission checks and QuerySet filtering via `get_objects_for_user()`. Stores `(user, permission, object)` with no role concept. We add the role abstraction above it. |
| Role definitions in config files, not database | Enables version control, CI validation, environment consistency, and compliance audit trails. Eliminates runtime role mutation. |
| Per-site configuration files | Each site gets its own role definitions, supporting full customisation without affecting other deployments. |
| Composable roles | Users can hold multiple roles on the same object. Permissions are the union of all held roles. Small focused roles can be combined with broader roles. |
| Deployment-time sync script | A management command reconciles guardian state with config + assignments on every deployment. Includes dry-run and drift detection. |
| LTI 1.3 role alignment | All roles carry LTI role URIs for standards compliance and interoperability with external tools. |

## 2. Role Configuration System

### 2.1 File Structure

Role configurations are Python modules organised as follows:

```
config/
  permissions/
    __init__.py
    types.py            # PermissionSet, RoleConfig classes
    registry.py         # All valid permission strings
    roles.py            # Default roles shipped with FreedomLS

```

The active configuration is determined by a Django settings dict:

```python
# settings.py (per-site)
FREEDOMLS_PERMISSIONS_MODULES = {
    "Wrend": "config.permissions.sites.wrend",
    "Xxx": "config.permissions.sites.xxx",

}
```

If this setting is absent, the system falls back to `config.permissions.base`.

### 2.3 PermissionSet and RoleConfig Classes

`PermissionSet` is an immutable definition of a single role. `RoleConfig` is a container supporting extension and single-level inheritance.

```python
# config/permissions/types.py

from dataclasses import dataclass


@dataclass(frozen=True)
class PermissionSet:
    """Defines a single role and its capabilities."""
    display_name: str
    permissions: frozenset[str]
    lti_role: str | None = None
    ui_hint: str = "standalone"   # "standalone" | "composable"
    description: str = ""


class RoleConfig:
    """
    Container for a complete role configuration.
    Supports extending a base config with overrides,
    additions, and single-level inheritance.
    """

    def __init__(self, roles: dict[str, PermissionSet]):
        self._roles = dict(roles)

    def extend(self, overrides: dict) -> "RoleConfig":
        """
        Return a new RoleConfig layering overrides onto this one.

        Three override forms:
        1. Full replacement (provide a PermissionSet)
        2. Modification (dict with add_permissions / remove_permissions)
        3. Inheritance (dict with "inherits" key referencing parent)
        """
        new_roles = dict(self._roles)

        for name, spec in overrides.items():
            if isinstance(spec, PermissionSet):
                new_roles[name] = spec
            elif isinstance(spec, dict):
                base_perms = set()

                if "inherits" in spec:
                    parent = spec["inherits"]
                    if parent not in new_roles:
                        raise ValueError(
                            f"Role '{name}' inherits from "
                            f"unknown role '{parent}'"
                        )
                    base_perms = set(new_roles[parent].permissions)
                elif name in new_roles:
                    base_perms = set(new_roles[name].permissions)

                base_perms |= spec.get("add_permissions", set())
                base_perms -= spec.get("remove_permissions", set())

                parent_role = (
                    new_roles.get(spec.get("inherits"))
                    or new_roles.get(name)
                )

                new_roles[name] = PermissionSet(
                    display_name=spec.get(
                        "display_name",
                        parent_role.display_name
                        if parent_role else name,
                    ),
                    permissions=frozenset(base_perms),
                    lti_role=spec.get(
                        "lti_role",
                        parent_role.lti_role
                        if parent_role else None,
                    ),
                    ui_hint=spec.get(
                        "ui_hint",
                        parent_role.ui_hint
                        if parent_role else "standalone",
                    ),
                    description=spec.get("description", ""),
                )

        return RoleConfig(new_roles)

    def __getitem__(self, key):
        return self._roles[key]

    def __contains__(self, key):
        return key in self._roles

    def __iter__(self):
        return iter(self._roles)

    def items(self):
        return self._roles.items()

    def keys(self):
        return self._roles.keys()

    def all_permission_strings(self) -> set[str]:
        result = set()
        for role in self._roles.values():
            result |= role.permissions
        return result
```

### 2.6 Role Configuration Rules

The following rules apply to all configuration files and must be enforced by CI validation:

1. **Single-level inheritance only.** A role may inherit from one parent role. That parent must not itself use inheritance. This prevents resolution complexity and circular dependencies.
2. **All permission strings must exist in the registry.** Any permission referenced by a role must be defined in `config/permissions/registry.py` and have a corresponding Django `auth_permission` entry.
3. **LTI role URIs must be valid IMS Global URIs.** Use the LTI 1.3 role vocabulary. System roles use the `/system/person#` namespace, institution roles use `/institution/person#`, and course roles use `/membership#`. See Section 5 for the complete reference.
4. **The `ui_hint` field must be either `"standalone"` or `"composable"`.** Standalone roles appear as primary choices in the admin UI. Composable roles appear under a separate "Additional Capabilities" section.
5. **Role names must be valid Python identifiers.** Lowercase, underscores, no spaces. These become database values in role assignment records.

### 2.7 Config Loader

A utility function loads and caches the active site configuration. The cache is invalidated on process restart (i.e., deployment), which is exactly when config changes take effect.

```python
# freedomls/permissions/loader.py

from django.conf import settings
from django.contrib.sites.models import Site
from importlib import import_module
from functools import lru_cache


@lru_cache(maxsize=None)
def get_role_config(site_name: str | None = None):
    """
    Load the role config for the given site name.
    Uses FREEDOMLS_PERMISSIONS_MODULES (a dict mapping site names to module paths).
    Falls back to config.permissions.base if no site-specific config exists.
    """
    if site_name is None:
        site_name = Site.objects.get_current().name

    modules = getattr(settings, "FREEDOMLS_PERMISSIONS_MODULES", {})
    module_path = modules.get(site_name, "config.permissions.base")
    module = import_module(module_path)
    return module.ROLES if hasattr(module, "ROLES") else module.BASE_ROLES
```

## 3. Data Model

### 3.1 Overview

The permission system uses four entity groups: sites, courses, users, and role assignments. Role assignments link users to roles scoped to a specific entity. Guardian permission rows are derived from these assignments and must never be created or modified outside the sync service.

### 3.2 Existing Models (No Changes Needed)

The permission system builds on top of existing FreedomLS models. These are **not** new models — they already exist in the codebase:

- **Site** — Django's built-in `django.contrib.sites.models.Site`. FreedomLS uses the Sites framework for multi-tenancy via `SiteAwareModel`. Each site represents a separate deployment/organisation. Site-level permissions are scoped to a Site instance.
- **Course** — `freedom_ls.content_engine.models.Course` (extends `SiteAwareModel`). Courses are automatically scoped to the current site. Course-level permissions are scoped to a Course instance.
- **User** — `freedom_ls.accounts.models.User` (extends `SiteAwareModelBase`, `AbstractBaseUser`, `PermissionsMixin`). Uses email-based login.
- **Student** — `freedom_ls.student_management.models.Student`. Linked to User via ForeignKey.
- **Cohort** — `freedom_ls.student_management.models.Cohort`. Groups of students.

> *Note: The existing `StudentCourseRegistration` model in `student_management` tracks student enrolment in courses. The new `CourseRoleAssignment` model (Section 3.4.3) serves a different purpose: it tracks role-based permission assignments on courses for all user types (instructors, TAs, content developers, etc.). For students, the two models will coexist — `StudentCourseRegistration` handles the student-specific enrolment workflow, while `CourseRoleAssignment` handles the permission layer. A future integration may unify these.*

### 3.3 Role Assignment Models

Three separate models handle role assignments at each scope level. This avoids GenericForeignKey complexity and ensures clean, indexable queries. The `role` field is a `CharField` validated at the application layer against the active site config (since valid roles vary per site).

### 3.4 Role Field Validation

The `role` field on all assignment models is a `CharField`, not a Django `TextChoices` enum. This is deliberate: the valid role set varies per site and is defined by the config file, not by the code. Validation happens at the application layer:

```python
# In the service layer, not the model
from freedomls.permissions.loader import get_role_config

def validate_role(role_name: str) -> None:
    config = get_role_config()
    if role_name not in config:
        raise ValueError(
            f"Unknown role '{role_name}'. Valid roles: "
            f"{list(config.keys())}"
        )
```

## 4. Permission Synchronisation

### 4.1 Design Principles

The sync system ensures guardian's permission tables accurately reflect the combination of role configuration (Layer 1) and role assignments (Layer 2). Two sync operations exist:

| Operation | Trigger | Scope | Expected Duration |
|---|---|---|---|
| Assignment sync | User gains or loses a role | Single user × single object | < 50ms (10–20 DB writes) |
| Deployment sync | Management command on deploy | All assignments system-wide | Seconds to low minutes |

### 4.2 Assignment Sync (On Role Change)

When a user is assigned or removed from a role, the sync service immediately reconciles their guardian permissions for that specific object. This is synchronous within the request/response cycle. It must be called from explicit service functions, not via Django signals (signals are fragile, hard to test, and fail silently).

```python
# freedomls/permissions/sync.py

from django.db import transaction
from guardian.shortcuts import assign_perm, remove_perm, get_perms
from freedomls.permissions.loader import get_role_config


class PermissionSyncService:

    @staticmethod
    def sync_user_object_permissions(user, obj, dry_run=False):
        """
        Compute the desired permission set for a user on an object
        based on ALL roles held, then reconcile with guardian state.
        Returns a dict describing changes made.
        """
        config = get_role_config()
        roles = _get_active_roles_for_user_on_object(user, obj)

        # Compute desired permissions (union of all roles)
        desired = set()
        for role_name in roles:
            if role_name in config:
                desired |= config[role_name].permissions

        # Get current guardian state
        current = set(get_perms(user, obj))

        to_add = desired - current
        to_remove = current - desired

        if not dry_run:
            with transaction.atomic():
                for perm in to_add:
                    assign_perm(perm, user, obj)
                for perm in to_remove:
                    remove_perm(perm, user, obj)

        return {
            "user": user.pk,
            "object": repr(obj),
            "roles": list(roles),
            "added": to_add,
            "removed": to_remove,
        }
```

### 4.4 Deployment Sync (Management Command)

This command runs on every deployment. It reconciles all guardian permissions with the current config and assignment state. Must support `--dry-run` for change management review and `--report-orphans` for compliance.

```python
# management/commands/sync_permissions.py

class Command(BaseCommand):
    help = "Reconcile guardian permissions with role config"

    def add_arguments(self, parser):
        parser.add_argument(
            "--dry-run",
            action="store_true",
            help="Report what would change without writing",
        )
        parser.add_argument(
            "--report-orphans",
            action="store_true",
            help="Report guardian perms with no matching role assignment",
        )

    def handle(self, **options):
        config = get_role_config()
        dry_run = options["dry_run"]

        # Phase 1: Ensure permission objects exist in Django
        self._ensure_permissions_exist(config)

        # Phase 2: Sync all course enrolments
        drift = 0
        for enrolment in (
            CourseRoleAssignment.objects.filter(is_active=True)
            .select_related("user", "course")
            .iterator()
        ):
            result = PermissionSyncService.sync_user_object_permissions(
                enrolment.user,
                enrolment.course,
                dry_run=dry_run,
            )
            if result["added"] or result["removed"]:
                drift += 1
                self._log_drift(result)

        # Phase 3: Sync all SiteRoleAssignment records (same pattern)
        # Phase 4: Sync all SystemRoleAssignment records

        # Phase 5: Detect orphaned permissions
        if options["report_orphans"]:
            self._report_orphans()

        self.stdout.write(
            f"Reconciled {drift} drifted assignments"
            + (" (dry run)" if dry_run else "")
        )
```

### 4.5 Orphan Detection

An orphaned permission is a guardian row that does not correspond to any active role assignment. Orphans indicate a sync bug, a manual grant that bypassed the role system, or a race condition. In a compliance environment, orphaned permissions are audit findings.

The deployment sync command must include an orphan report that lists every guardian permission row not explainable by the current config and assignments. This report must be reviewed as part of the deployment process.

## 5. LTI 1.3 Role Compliance

### 5.1 Role Vocabulary Reference

All roles must carry a valid LTI 1.3 role URI from the IMS Global vocabulary. Composable micro-roles that have no LTI equivalent may set `lti_role` to `None`.

*All URIs below are prefixed with `http://purl.imsglobal.org/vocab/lis/v2`*

| FreedomLS Role | LTI Namespace | LTI Role URI (suffix) |
|---|---|---|
| `system_admin` | System | `/system/person#Administrator` |
| `site_admin` | Institution | `/institution/person#Administrator` |
| `instructor` | Context | `/membership#Instructor` |
| `ta` | Context | `/membership#Instructor#TeachingAssistant` |
| `content_developer` | Context | `/membership#ContentDeveloper` |
| `student` | Context | `/membership#Learner` |
| `observer` | Context | `/membership#Mentor` |
| `auditor` | Context | `/membership#Learner#NonCreditLearner` |

### 5.2 LTI Integration Requirements

**As LTI Platform (tool consumer):** The launch request must include the user's LTI role URI(s) in the `roles` claim. These are read from the `lti_role` field of the user's active role assignments. When a user holds multiple roles, all corresponding LTI URIs are included in the claim array.

**As LTI Tool (provider):** Incoming role claims must be mapped to internal role assignments. The mapping should use the `lti_role` field from the site's role configuration as a lookup key. Unrecognised LTI roles should default to the `student` role with a logged warning.

## 6. QuerySet Filtering and Performance

### 6.1 Guardian QuerySet Filtering

Guardian provides `get_objects_for_user()` for efficient list views. This generates a single SQL query joining user permissions to the target model. Use this for all views where a user needs to see only objects they have access to.

```python
from guardian.shortcuts import get_objects_for_user

# All courses this user can view
visible_courses = get_objects_for_user(user, "course.view", klass=Course)

# All courses this user can grade
gradeable_courses = get_objects_for_user(user, "course.manage_grades", klass=Course)
```

### 6.2 Performance Mitigations

1. **Use Group permissions for high-enrolment courses.** If 500 students share identical permissions on a course, assign guardian permissions to a Django Group rather than to each individual user. This reduces storage from 500 rows to 1. Implement as a Phase 2 optimisation.
2. **Use `ObjectPermissionChecker` for batch checks.** When checking multiple permissions for the same user on the same object (e.g., rendering a course page with edit/grade/announce buttons), use guardian's `ObjectPermissionChecker` to prefetch all permissions in one query.
3. **Request-level caching.** Consider middleware that prefetches the user's course-level permissions at request start and caches them on the request object. Cache per-request only (not in Redis/memcached) to avoid stale permission data.
4. **Monitor guardian table size.** At 10,000 users × 3 courses × 12 permissions, expect ~360K rows. At 100K users, ~3.6M rows. Plan for database `ANALYZE` and potential table partitioning at 5M+ rows.

## 7. Validation and CI Integration

### 7.1 Validation Management Command

A management command validates the role configuration without modifying any state. This must run in CI on any PR that touches files under `config/permissions/`.

```python
# management/commands/validate_permissions.py

class Command(BaseCommand):
    help = "Validate role configuration files"

    def handle(self, **options):
        errors = []
        config = get_role_config()
        registry = load_permission_registry()

        for role_name, role_def in config.items():
            # Rule: Valid role name
            if not role_name.isidentifier():
                errors.append(
                    f"'{role_name}' is not a valid Python identifier"
                )

            # Rule: All permissions exist in registry
            for perm in role_def.permissions:
                if perm not in registry:
                    errors.append(
                        f"Role '{role_name}': unknown permission '{perm}'"
                    )

            # Rule: LTI URI format (if provided)
            if role_def.lti_role is not None:
                if not role_def.lti_role.startswith(
                    "http://purl.imsglobal.org/"
                ):
                    errors.append(
                        f"Role '{role_name}': invalid LTI URI"
                    )

            # Rule: Valid ui_hint
            if role_def.ui_hint not in ("standalone", "composable"):
                errors.append(
                    f"Role '{role_name}': invalid ui_hint '{role_def.ui_hint}'"
                )

        # Check for orphaned role assignments in DB
        assigned = set(
            CourseRoleAssignment.objects.values_list("role", flat=True).distinct()
        )
        orphaned = assigned - set(config.keys())
        if orphaned:
            errors.append(
                f"DB contains assignments for roles not in config: {orphaned}"
            )

        if errors:
            for e in errors:
                self.stderr.write(self.style.ERROR(e))
            raise CommandError(f"{len(errors)} validation error(s)")
        self.stdout.write(self.style.SUCCESS("Role config is valid"))
```

### 7.2 CI Pipeline Integration

| Check | Trigger | Blocks Deploy |
|---|---|---|
| `validate_permissions` | Any change to `config/permissions/` | Yes |
| `sync_permissions --dry-run` | All deployments (pre-deploy) | No (report only) |
| `sync_permissions` | All deployments (post-deploy) | Yes (if errors) |
| Unit tests for sync service | All PRs | Yes |

## 8. Admin UI Guidance

### 8.1 Role Assignment Interface

Administrators interact with roles, never with individual guardian permissions. The admin interface for managing a user's access should present:

1. **Standalone roles as primary options.** Displayed as a dropdown or radio group: Instructor, Teaching Assistant, Student, Observer, Auditor. An admin selects one primary role.
2. **Composable roles as optional additions.** Shown in a separate "Additional Capabilities" section as checkboxes. Clearly labelled as additive. Example: a "Course Announcer" checkbox beneath the main role selector.
3. **Effective permissions as a read-only summary.** After selecting roles, the UI shows a grouped, read-only list of the resulting permissions, computed from the config file. Admins cannot toggle individual permissions through the UI.

### 8.2 Compliance Audit Views

The admin interface must support the following audit queries:

- **Who has access to Course X?** List all users with any role on a course, with role name and assignment date.
- **What can User Y do in Course X?** Show effective permissions for a user on a course, derived from all roles held.
- **Who granted User Y the Instructor role, and when?** Show assignment provenance from `assigned_by` and `assigned_at` fields.
- **What changed in the last deployment?** Archived output of `sync_permissions --dry-run`, stored per deployment.
- **Any orphaned permissions?** Output of `sync_permissions --report-orphans`.

## 9. Risks, Constraints, and Future Work

### 9.1 Known Risks

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Guardian table growth at scale | Medium | Medium | Use Group permissions for high-enrolment courses. Monitor table size. Plan partitioning at 5M+ rows. |
| Sync drift (guardian diverges from config + assignments) | Low | High | Deployment sync with orphan detection. Never grant guardian perms outside the sync service. |
| Role config error deployed to production | Low | High | CI validation gate. Dry-run on staging before production deploy. |
| Role proliferation (too many composable roles) | Medium | Low | Use `ui_hint` to separate standalone vs composable in admin UI. Document naming conventions. |

### 9.2 Explicit Constraints (V1)

- **No per-activity permission overrides.** Permissions scope to Course level, not to individual assignments or modules within a course.
- **No attribute-based access control (ABAC).** The system does not support policies like "users in timezone X" or "students with GPA above 3.0". The architecture supports adding predicate-based checks later without structural changes.
- **No runtime role editing via UI.** Role definitions require a config file change and deployment. This is a feature, not a limitation, for compliance environments.
- **Single-level role inheritance only.** Role A can inherit from Role B, but Role B cannot itself use inheritance.

### 9.3 Future Considerations (V2+)

- **SIS integration:** Automated enrolment from Student Information Systems. Essential at 1,000+ users. The service-layer enrolment API (Section 4.3) is designed to support this.
- **Group-based guardian permissions:** Assign permissions to Django Groups (one per role-per-course) instead of individual users for high-enrolment courses. Reduces storage by up to 100x.
- **SitePermissionOverride table:** Let site admins restrict (but not expand) specific permissions for roles within their site, without modifying config files. Enables admin-level customisation for Tier A changes.
- **Per-activity permissions (V3):** Module-level overrides within a course. Only Moodle implements this natively; most LMS platforms succeed without it.
