# Audio-player cotton component

## Problem

`content_engine.File` has an AUDIO file type, but no cotton component renders
it inline. Audio uploads currently surface as a plain download link via the
`file-download` component, which is a poor learner experience (must download,
open in another app, no scrubbing, no transcript surface).

Existing video (`youtube` cotton component) embeds audio + visual together,
so the SACAA Cat-1 "audio AND visual instructions" clause is already
satisfied by that path. This spec is *not* Cat-1 blocking — it is a
content-engine UX improvement for audio-only assets (podcasts, narrated
explanations, language pronunciation, etc.).

## Direction

- New `audio_player` cotton component using native HTML5
  `<audio controls preload="metadata">`. Native controls are already
  keyboard-accessible and screen-reader-labelled, so no third-party player
  (Able Player or similar) is needed.
- Show filename and duration alongside the player.
- Register as the default renderer for `File.file_type == AUDIO`; keep
  "Download audio" available as a secondary action for low-bandwidth
  learners.
- Optional `transcript` markdown field on `File` so authors can attach a
  text alternative without needing a separate content type. This addresses
  WCAG 1.2.1 (transcript for prerecorded audio).

## Out of scope

- Custom player chrome / branded skin.
- Waveform visualisation, chapter markers, playback-rate UI beyond browser
  defaults.
- Auto-generated transcripts (ASR). Transcripts are author-supplied for
  now.
- Anything related to SACAA Cat-1 compliance — that is handled by the
  existing video path; see `1. next/00. sacaa risk cat 1 requirements/`.
