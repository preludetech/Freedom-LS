The educator interface currently has limited functionality. We need to handle the basics of what an educator will need to do.

## Cohorts CRUD

Cohort create, read, update and delete are already partially implemented (create, list, detail, edit name, delete with confirmation). Extend with:

- Search and filter on the list view
- Show key info at a glance: student count, linked courses, status
- Confirmation on delete should show what will be affected (enrollments, progress data)

## Students CRUD

Educators should be able to both create new student accounts and manage existing ones:

- **Create**: Add individual students (email, name) directly from the educator interface. If a user with that email already exists on the site, link them rather than creating a duplicate.
- **Read**: List students with search/filter. Show cohort memberships and course registrations at a glance.
- **Update**: Edit student details (name, email, active status).
- **Delete/Deactivate**: Deactivation preferred over hard delete to preserve historical data. Make the distinction clear in the UI.

## Bulk operations

- **CSV import for students**: Upload CSV to add multiple students at once. Include a preview/confirmation step before committing (show what will be created vs matched to existing users). Provide a downloadable CSV template.
- **Multi-select with bulk actions**: Checkboxes on list views to select multiple items, then apply bulk actions (e.g., add to cohort, remove from cohort, register for course).

## Adding and removing students from cohorts

- From the cohort detail view: search and add existing students, or create new students inline.
- From the student detail view: add to / remove from cohorts.
- Support both directions for flexibility.
- Removing a student from a cohort should clearly communicate what happens to their course registrations and progress.

## Registering cohorts for courses

- Don't worry about registering individual students directly, we don't need that yet.
- From the cohort detail view: register the cohort for a course (pick from available courses, confirm).
- From the course detail view: link a cohort (pick from available cohorts, confirm).
- Support both directions.
- When a student is added to a cohort, they should automatically be registered for all courses the cohort is linked to ("living link" pattern). And vice versa when removed.

## Permission configuration - adding educators

Full invite flow: enter email, select a role (using named roles like Site Admin, Instructor, TA - not raw permissions), confirm. If the user doesn't exist, create/invite them. If they already exist, assign the role.

- Rely on the role-based permission system implemented by an earlier specification.
- Role assignment should be scoped (e.g., "Instructor for Cohort X" not just "Instructor globally").
- Show role descriptions so non-technical users understand what each role can do.
- Confirmation step before role changes, especially for privilege escalation or removal.

## Permission enforcement across the interface

Be very particular about permissions throughout the entire educator interface:

- **Hide, don't disable** UI elements that are irrelevant to the user's role. Students should never see educator controls. Educators without delete permission should not see delete buttons.
- **Server-side enforcement**: Every action must be permission-checked on the backend. Frontend hiding is UX, not security.
- **Consistent strategy**: Apply the same hide/show approach uniformly across all views.
- **Meaningful errors**: If a permission check fails (e.g., role changed while page was open), show a clear message with what happened, why, and what to do next. Use 403 not 404.
- People should not see buttons they're not allowed to click. They should not have access to information they're not allowed to see. They should not be able to delete what they're not allowed to delete.
- Design each role's view as a complete experience, not a stripped-down version of the admin view.
