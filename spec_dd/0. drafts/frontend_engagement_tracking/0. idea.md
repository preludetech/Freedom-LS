# Frontend Engagement Tracking

Client-side tracking for learner engagement signals that don't produce backend requests. Depends on the xAPI event infrastructure — see `spec_dd/2. in progress/xapi_implementation/` for the backend tracker, verb vocabulary, and event schema.

## Scope

- **Time on page / engaged time**: track how long learners spend on content, distinguishing focused vs idle time using the Page Visibility API
- **Scroll depth**: use IntersectionObserver with sentinel elements to detect how far learners read content

## Implementation approach

- Use the **Beacon API** (`navigator.sendBeacon`) with `visibilitychange` events for reliable data delivery on page leave
- **Event batching**: collect events and flush periodically or on navigation, not per-event
- **HTMX integration**: hook into `htmx:pushedIntoHistory` and `htmx:afterSettle` to handle partial page swaps (reset timers, re-attach observers)
- Keep it lightweight — vanilla JS, no analytics libraries

## Privacy

- Frontend tracking sends no PII — the backend associates events with users via session auth
- Payloads carry only the engagement signal plus the object being engaged with

## What NOT to build

- No keystroke/mouse tracking
- No third-party analytics libraries
