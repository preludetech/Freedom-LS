# Spec: Professional Branded Email Templates

## Overview

Replace allauth's default plain-text emails with professional, on-brand HTML emails that render well across all major email clients. The system should be easy to customise for any project built on FreedomLS.

## Why This Matters

- Plain text emails look unprofessional and reduce trust
- Branded emails reinforce the platform identity and improve user experience
- A well-structured, overridable template system makes FreedomLS easy to customise for downstream projects
- Multi-part emails (HTML + text) improve deliverability (spam filters prefer them) and accessibility

## Requirements

### 1. Base HTML Email Template

Create a base HTML email template that all allauth emails extend.

**Layout:**
- Single-column, table-based layout (no flexbox/grid — these don't work in email clients)
- Maximum width: 600px (industry standard)
- Total email size must stay under 100KB to avoid Gmail clipping
- Mobile-responsive using `max-width: 600px; width: 100%` pattern

**Header:**
- Support an optional logo image via a configurable URL (e.g., a Django setting or context variable)
- Fall back to a styled text header showing the site name when no logo is configured
- Use brand primary color (`#2B6CB0`) as header background

**Content area:**
- Clean white background
- Use system-safe fonts: Arial/Helvetica as primary, sans-serif as fallback (custom fonts don't work reliably in email clients)
- Use brand color tokens for text colors: foreground (`#1A2332`) for body text, muted (`#4A5568`) for secondary text
- Consistent padding (20px)

**Footer:**
- Site name and copyright
- Configurable footer links via a template partial (e.g., privacy policy, terms of service, unsubscribe)
- Muted text styling, smaller font size

**Buttons (CTAs):**
- "Bulletproof button" pattern using a table cell with background color and a styled `<a>` tag
- Use brand primary color for default buttons
- Consistent padding and border-radius

**Why table-based layout:** Email clients (especially Outlook's Word rendering engine and Gmail's aggressive CSS sanitization) don't support modern CSS layout. Tables with `role="presentation"` are the only reliable cross-client layout method.

### 2. CSS Inlining with django-premailer

Use `django-premailer` to allow writing CSS in `<style>` blocks that get automatically converted to inline `style=""` attributes at render time.

**Why:** Gmail strips `<style>` tags, so inline styles are required. A CSS inlining library lets us write maintainable CSS during development while producing email-client-compatible output. This is more maintainable than hand-writing inline styles, especially as the template grows.

**Configuration:**
- Install `django-premailer` (or `premailer` if the Django wrapper is unsuitable)
- Set `allow_network=False` to prevent premailer from fetching external stylesheets
- The `<style>` block in the base template should define classes using the brand colour tokens

### 3. Overridable Template Partials

Use `django-template-partials` (already installed) to create overridable sections in the base email template. This allows downstream projects to customise specific parts without replacing the entire template.

**Required partials:**
- `greeting` — The opening greeting (e.g., "Hi {{ user.first_name }},"). Projects can override this to match their voice (formal, casual, etc.)
- `sign_off` — The closing sign-off (e.g., "Cheers, The {{ current_site.name }} Team"). Projects can override this for their brand voice
- `footer_links` — Optional footer links (privacy policy, terms, unsubscribe). Empty by default — projects add links relevant to their deployment
- `header` — The email header section. Projects can replace the entire header (e.g., to add a logo or change styling)

**Why partials over blocks:** Template partials can be individually rendered and overridden at the file level using the `template_name#partial_name` syntax. This gives downstream projects fine-grained control without needing to copy the entire base template.

### 4. Text Email Fallback

Every HTML email must have a corresponding `.txt` version.

- Override allauth's default `.txt` templates to ensure consistent messaging between HTML and text versions
- Text templates should extend a base text email template with the same greeting/sign-off pattern
- Use `{% autoescape off %}` in text templates to avoid HTML entities

**Why:** Some users prefer text emails, some corporate firewalls strip HTML, and multi-part emails score better with spam filters.

### 5. Allauth Email Template Overrides

Create both `.html` and `.txt` templates for all allauth email types:

| Email Type | Template Prefix | Context Variables |
|---|---|---|
| Email confirmation (signup) | `account/email/email_confirmation_signup` | `user`, `activate_url`, `current_site`, `key` |
| Email confirmation (general) | `account/email/email_confirmation` | `user`, `activate_url`, `current_site`, `key` |
| Password reset | `account/email/password_reset_key` | `user`, `password_reset_url`, `current_site` |
| Password changed | `account/email/password_changed` | `user`, `current_site` |
| Password set | `account/email/password_set` | `user`, `current_site` |
| Email changed | `account/email/email_changed` | `user`, `current_site` |
| Unknown account | `account/email/unknown_account` | `user`, `current_site` |
| Login code | `account/email/login_code` | `user`, `code`, `current_site` |

Each email should:
- Extend the base HTML email template
- Include a clear call-to-action button where appropriate (confirmation, password reset)
- Use concise, friendly copy consistent with the brand voice
- Include the text fallback version

Subject line templates (`_subject.txt`) should also be overridden for consistent branding.

### 6. Documentation

Create `.claude/docs/email_templates.md` documenting:

- The base email template structure and how to extend it
- Available template partials and how to override them
- Context variables available in each email type (table from section 5)
- How to add new email types using the base template
- How CSS inlining works with django-premailer
- How to configure the optional logo URL
- How to preview emails during development (mention the file-based email backend in dev settings)
- Example of creating a custom email template

## Template Location

All email templates should be placed in:
```
freedom_ls/accounts/templates/account/email/
```

The base email template should be at:
```
freedom_ls/accounts/templates/emails/base_email.html
freedom_ls/accounts/templates/emails/base_email.txt
```

This follows the existing pattern where the `accounts` app owns authentication-related templates, and keeps allauth overrides in the standard `account/email/` path that allauth expects.

## Settings / Configuration

- `EMAIL_LOGO_URL` — Optional Django setting. URL to the logo image for the email header. When not set, the header falls back to a styled text header with the site name.

## Success Criteria

1. All allauth emails are sent as multipart (HTML + text)
2. HTML emails render correctly in Gmail, Outlook (desktop and web), Apple Mail, and major mobile email apps
3. Emails use FreedomLS brand colors and styling
4. The greeting, sign-off, header, and footer links are overridable via template partials
5. A downstream project can customise the email appearance without forking the base templates
6. Documentation clearly explains the template system, available context variables, and how to extend/override
7. Email total size stays under 100KB
8. Logo is optional and falls back gracefully to text
9. All existing allauth email flows continue to work (signup confirmation, password reset, etc.)
