# Research: Best Practices for Consolidating Django Models (Student -> User FK Migration)

## Context

The goal is to remove the `Student` model and migrate all foreign keys that currently point to `Student` so they point directly to `User` instead. The `Student` model currently has a FK to `User` plus a few extra fields (`id_number`, `date_of_birth`, `cellphone`). Models that currently FK to `Student` include `CohortMembership`, `StudentCourseRegistration`, `StudentCohortDeadlineOverride`, and others.

---

## 1. Django Migration Strategies for FK Consolidation

### The Multi-Step Approach (Recommended)

The safest way to migrate foreign keys from one model to another is a phased approach across multiple migrations. Doing this in a single migration is risky because it combines schema changes, data changes, and constraint changes in one atomic operation that is difficult to debug or roll back.

**Phase 1 -- Add new FK columns (schema migration)**

Add a new nullable `user` ForeignKey to every model that currently has a `student` FK. This is a pure schema migration generated by `makemigrations`.

```python
# On each affected model, add:
user = models.ForeignKey(
    settings.AUTH_USER_MODEL,
    on_delete=models.CASCADE,
    null=True,       # Temporarily nullable
    blank=True,
)
```

**Phase 2 -- Copy data (data migration)**

Create a data migration (`makemigrations --empty appname`) that populates the new `user_id` column from `student.user_id`:

```python
from django.db import migrations

def forward(apps, schema_editor):
    StudentCourseRegistration = apps.get_model("freedom_ls_student_management", "StudentCourseRegistration")
    for reg in StudentCourseRegistration.objects.select_related("student").iterator():
        reg.user_id = reg.student.user_id
        reg.save(update_fields=["user_id"])

def reverse(apps, schema_editor):
    # Reverse is possible if student FK still exists
    pass

class Migration(migrations.Migration):
    dependencies = [...]
    operations = [
        migrations.RunPython(forward, reverse),
    ]
```

For large tables, use `queryset.update()` or raw SQL via `schema_editor.execute()` for performance:

```python
def forward(apps, schema_editor):
    schema_editor.execute("""
        UPDATE freedom_ls_student_management_studentcourseregistration AS scr
        SET user_id = s.user_id
        FROM freedom_ls_student_management_student AS s
        WHERE scr.student_id = s.id
    """)
```

**Phase 3 -- Make new FK non-nullable, remove old FK (schema migration)**

- Alter the new `user` field to be non-nullable (`null=False`).
- Remove the old `student` FK field.
- Run `makemigrations` to generate this.

**Phase 4 -- Delete the Student model (schema migration)**

- Remove the `Student` model class from `models.py`.
- Run `makemigrations` to generate a `DeleteModel` operation.

**Phase 5 -- Clean up**

- Update admin, views, serializers, factories, and tests.
- Remove any orphaned imports.

### Key Principle: Always Keep the Database Deployable

At every phase, the database should be in a valid state. If a deployment fails partway through, you should be able to roll back to the previous migration without data loss.

## 2. Common Pitfalls

### Data Loss Risks

- **Dropping the old FK before copying data**: If you remove the `student` FK column before the data migration runs, the `student.user_id` mapping is lost. Always add the new column and migrate data *before* removing the old column.
- **Unique constraint violations**: If `Student` had a many-to-one relationship with `User` (multiple Student rows per User), then migrating FKs to User could create duplicate entries or constraint violations. In this codebase, `Student.user` is not `OneToOneField` or `unique=True`, so this must be verified: does any User have multiple Student records? If so, data deduplication is required before migration.
- **Null user_id after migration**: If the data migration is skipped or partially applied, models will have `user_id = NULL`, which will cause `IntegrityError` when the field is made non-nullable.

### Migration Ordering Issues

- **Cross-app dependencies**: If `Student` is in `student_management` but FKs to it exist in `student_progress` or other apps, each app's migration must declare a dependency on the migration that adds the new `user` FK in `student_management`. Use explicit `dependencies` in each migration file.
- **Circular dependencies**: If app A's migration depends on app B's migration which depends on app A's migration, Django will raise `CircularDependencyError`. Break the cycle by splitting operations into separate migrations. Django docs recommend using `SeparateDatabaseAndState` for advanced cases.
- **Running migrations from scratch in tests**: When the test runner creates the database from scratch, it runs all migrations in order. If an early migration references a model that a later migration deletes, the full migration history must still be consistent. This is a common trap -- see the testing section below.

### Other Pitfalls

- **MySQL limitations**: MySQL does not support transactional DDL, so a failed migration leaves the database in a partially-migrated state. PostgreSQL (which this project uses) wraps each migration in a transaction by default, which is safer.
- **Forgetting GenericForeignKey references**: If `Student` is referenced via `ContentType` (GenericForeignKey), those references will also need to be updated -- not just direct FKs.
- **Stale `select_related`/`prefetch_related` calls**: After removing the `student` FK, any queryset that does `.select_related("student")` or `.select_related("student__user")` will raise `FieldError` at runtime, not at import time. These are easy to miss.

## 3. Multi-Step Migration Approach

### Why Not a Single Migration?

A single migration that adds columns, moves data, drops columns, and deletes the model is:
- Impossible to partially roll back.
- Very hard to test incrementally.
- Prone to timeouts on large tables (the data migration holds a lock).
- Not deployable in zero-downtime environments.

### Recommended Sequence

| Step | Migration Type | What Happens | Reversible? |
|------|---------------|-------------|-------------|
| 1 | Schema | Add nullable `user` FK to all affected models | Yes (drop column) |
| 2 | Data | Populate `user_id = student.user_id` for all rows | Yes (set user_id = NULL) |
| 3 | Schema | Make `user` FK non-nullable; remove `student` FK | Partially (can re-add column but data is gone) |
| 4 | Schema | `DeleteModel("Student")` | No (model and table are gone) |

### Deployment Strategy

For production systems, Steps 1-2 can be deployed together since they are additive (no destructive changes). The application code can continue using the old `student` FK during this phase.

Steps 3-4 should be deployed alongside the updated application code that uses `user` instead of `student`. This is the "cut-over" deployment.

### Using `SeparateDatabaseAndState`

In some cases, you may want Django's state (what it thinks the schema looks like) to differ from what actually happens in the database. `SeparateDatabaseAndState` allows this:

```python
migrations.SeparateDatabaseAndState(
    database_operations=[
        # What actually runs against the DB
        migrations.RunSQL("ALTER TABLE ... RENAME COLUMN student_id TO user_id"),
    ],
    state_operations=[
        # What Django thinks happened
        migrations.RemoveField("MyModel", "student"),
        migrations.AddField("MyModel", "user", models.ForeignKey(...)),
    ],
)
```

This is an advanced technique. For most cases, the standard multi-step approach is clearer and safer.

## 4. Handling Methods/Properties on the Removed Model

The `Student` model likely has methods, properties, and manager methods that other parts of the codebase depend on. When removing the model, these need to go somewhere.

### Decision Framework

| Method Type | Where It Should Go | Example |
|------------|-------------------|---------|
| Methods that query the DB for related data (e.g., "get courses for this student") | **Custom Manager or QuerySet on the related model**, or **standalone query functions in a `queries.py` or `utils.py`** | `get_registered_courses(user)` |
| Properties that compute values from the instance (e.g., `full_name`) | **Already on the User model** (or add to User if needed). If FLS cannot modify User, use a **standalone utility function**. | `user.get_full_name()` already exists on Django's User |
| Class methods / static methods | **Module-level functions in `utils.py`** | `get_or_create_student_profile(user)` becomes unnecessary |
| Methods used in templates | **Template tags/filters** or **model properties on User** | `{{ user.full_name }}` |
| Manager methods (custom querysets) | **Custom Manager on the models that previously FK'd to Student**, now FK to User | `StudentCourseRegistration.objects.for_user(user)` |

### Best Practices

1. **Do not add Student-specific fields to the User model.** The spec says "Do not change the user model in any way." Fields like `id_number`, `date_of_birth`, `cellphone` that were on `Student` either need to move to a different model or be handled differently. Consider whether these fields should live on `StudentCourseRegistration` or a lightweight profile model, or if they can be dropped.

2. **Prefer standalone functions over mixins.** Mixins add complexity and can create diamond inheritance issues. A simple function like `def get_student_courses(user: User) -> QuerySet` in a `utils.py` is easier to test and understand.

3. **Use custom QuerySet methods for query logic.** Instead of `Student.objects.active_students()`, use `User.objects.filter(...)` with a custom manager, or more likely, move the query to the models that track enrollment: `StudentCourseRegistration.objects.active().filter(user=user)`.

4. **Keep utility functions close to their domain.** If a function is about course registrations, put it in `student_management/utils.py`. If it is about progress, put it in `student_progress/utils.py`.

## 5. Testing Strategy

### Before Starting the Refactor

1. **Ensure full test coverage of existing behavior.** Run the existing test suite and note the passing tests. Any test that touches `Student` or models with `student` FKs is relevant.
2. **Snapshot the current FK relationships.** Document every model that has a FK to `Student` and every queryset that traverses through `Student`.
3. **Check for multiple Students per User.** Run a query to verify the data invariant:
   ```sql
   SELECT user_id, COUNT(*) FROM student_management_student GROUP BY user_id HAVING COUNT(*) > 1;
   ```
   If any User has multiple Student records, the migration plan must handle deduplication.

### During the Refactor

4. **Test each migration step independently.** After each migration, run the test suite. The tests should pass at every intermediate state (with the possible exception of tests that explicitly test the Student model, which will fail after Step 4).
5. **Test migration reversibility.** Run `migrate appname <previous_migration_number>` to verify that reverse migrations work for Steps 1-2.
6. **Test fresh database creation.** Run `migrate` from scratch (as the test runner does) to ensure the full migration history is consistent. This catches issues where early migrations reference the now-deleted Student model.

### After the Refactor

7. **Update all tests that reference Student.** Replace `StudentFactory` with direct User creation or a helper that creates a User with the necessary registrations.
8. **Search for stale references.** Grep for `Student`, `student_id`, `student__`, `.student` across the entire codebase. Every hit needs to be evaluated.
9. **Test the admin interface.** `StudentAdmin` and all inline admins that reference `Student` will need to be updated or removed. Verify the admin site loads without errors.
10. **Test HTMX views and templates.** Templates that reference `student.user.email` will need to change to `user.email`. These failures are runtime-only, so manual or integration testing is important.
11. **Run the full test suite with `--create-db` flag** (if using pytest-django) to force fresh database creation, ensuring the migration history is clean.

### Migration-Specific Tests

Consider writing a dedicated migration test that:
- Applies migrations up to just before the refactor.
- Creates test data (Students, registrations, etc.).
- Applies the data migration.
- Verifies that all `user_id` columns are correctly populated.
- Applies the remaining migrations.
- Verifies the Student table no longer exists.

Django's `MigrationExecutor` can be used for this pattern, though it adds complexity.
